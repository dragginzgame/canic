# Canic 0.87 Changelog

## [0.87.0] - 2026-07-13 - Failure-Atomic Scaffolding

### 📝 Summary

0.87.0 prevents failed fleet or canister scaffolds from leaving a partially
created local project that blocks a clean retry. Existing project documents
are replaced durably, and one bounded rollback path restores their original
bytes and removes only the new scaffold directory.

Successful generated files, workspace membership, fleet configuration,
package identity, command forms, and deployment behavior are unchanged. The
release does not change or remove `CANIC_*` environment variables.

### 🩹 Fixed

- A failed fleet scaffold removes the new fleet directory instead of leaving
  an incomplete target that the next invocation treats as intentional state.
- A failed canister scaffold restores the original workspace manifest and
  fleet configuration before removing the new role directory.
- Rollback attempts every required cleanup action. If cleanup also fails, the
  reported error retains both the original operation failure and the rollback
  failure.

### 🔧 Changed

- Scaffold workspace-manifest replacement uses the existing host-owned durable
  writer instead of truncating the destination file in place.
- Rollback may recursively remove only a directory that scaffold preflight
  proved did not exist before the command began.
- Existing workspace and fleet documents are restored from the exact bytes
  captured before the first mutation.

### 🧹 Cleanup

- The two affected canister manifests now suppress only dependency-scan false
  positives caused by macro-derived Serde and CDK use; direct Candid use is no
  longer incorrectly ignored.
- No filesystem transaction framework, journal, recovery command, fallback
  writer, compatibility alias, or resumable partial-scaffold state is added.
