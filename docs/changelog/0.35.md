## 0.35.13 - 2026-05-14 - Foreground local install targeting

### 🩹 Fixed

- When a local replica is reachable by HTTP but ICP CLI reports the project
  environment as stopped, `canic install <fleet>` now targets the local gateway
  URL and advertised root key directly for subsequent canister commands instead
  of continuing with `-e local`.
- In that fallback mode root creation uses detached JSON creation and then
  carries the returned principal through the rest of install. Normal
  ICP-managed local environments still create the named `root` canister.
- Root canister creation and root ID lookup now request ICP CLI JSON output
  from `icp canister create --json` and `icp canister status --json`, avoiding
  quiet text parsing for the normal install path while keeping legacy text
  parsing as a compatibility fallback.

## 0.35.12 - 2026-05-14 - Project-root fleet layout and IC-relevant wasm size output

### 🔧 Changed

- Build and list tables now label the artifact size column as `WASM` and show
  the uncompressed `.wasm` size first, matching what the IC installs after
  decompression.
- When both artifacts are available, CLI output shows gzip as secondary context,
  for example `1.52 MiB (gz 811.20 KiB)`. If only the compressed artifact is
  present, the value is `n/a (gz 811.20 KiB)` instead of treating gzip size as
  the primary wasm size.

### 🩹 Fixed

- Canic now hard-cuts fleet config discovery to project-root
  `fleets/<fleet>/canic.toml`. Commands launched from nested directories walk
  up to the discovered Canic project root and write generated `icp.yaml`,
  `.icp/`, and `.canic/` state there.
- Cargo workspace discovery remains separate, so downstream canister builds can
  still use a Rust workspace under `backend/` while Canic/ICP operator state
  stays at the project root above application source directories.
- `canic status` now reads fleet configs from the resolved project root rather
  than re-discovering from the current source directory, so running it from
  `backend/` still reports fleets from the parent `fleets/` directory.
- The temporary alternate-layout support is gone: `backend/fleets/`,
  `--fleets-dir`, and `CANIC_FLEETS_ROOT` are no longer discovery surfaces.
- `canic install <fleet>` now resolves the root canister principal immediately
  after named root creation and uses that principal for reinstall, funding,
  staging, bootstrap resume, readiness, and install-state writes. This avoids a
  later `icp canister install root` failure when ICP CLI can create the named
  canister but does not resolve the name again in the selected environment.
- If named root creation still leaves no usable principal, the install error now
  points at the selected ICP root and expected `.icp/<network>` state instead
  of failing later at root reinstall.

## 0.35.11 - 2026-05-14 - Visible foreground replica readiness

### 🩹 Fixed

- Local replica readiness now falls back to the resolved root's configured HTTP
  gateway `/api/v2/status` endpoint when ICP CLI reports a foreground
  `canic replica start` process as not running. This keeps `canic status`,
  `canic replica status`, and `canic install <fleet>` usable while the replica
  is running in another terminal.
- The foreground fallback is visible instead of silent: text status reports
  show `HTTP reachable; ICP CLI status stopped`, and
  `canic replica status --json` includes `status_source`,
  `icp_cli_running`, and `local_gateway_reachable` fields so automation can
  detect the exact source of truth. `canic install <fleet>` also prints when it
  continues because the HTTP status endpoint is reachable while ICP CLI reports
  the local network stopped.
