## 0.35.3 - 2026-05-13 - TBD

### 🔧 Changed

- Added local replica port visibility and selection: `canic status` and
  `canic replica status` now show the configured local gateway port, and
  `canic replica start --port <port>` updates this project's `icp.yaml`
  `gateway.port` before starting the project-local network.
- Local direct replica queries now read the configured local gateway port
  instead of assuming `127.0.0.1:8000`.
- Tightened local replica ownership diagnostics so Canic distinguishes this
  project's ICP `local` network from a different network/project owner on the
  same port, including the owning ICP network and project path when ICP reports
  them.
- Added `canic fleet sync` to reconcile the Canic-managed `canisters` and
  `environments` sections in `icp.yaml` from discovered fleet configs while
  preserving other top-level ICP project sections such as `networks`.
- `canic fleet create <name>` now syncs `icp.yaml` after creating the fleet so
  the ICP project file stays aligned with Canic fleet configuration.
- Install timing summaries now use the shared Canic table renderer, with
  uppercase `PHASE` / `ELAPSED` headers and right-aligned elapsed values.
- Clarified default top-up opt-ins in the test fleet configs by using inline
  `topup = {}` blocks for roles that should use the default `10T` threshold and
  `5T` top-up amount.
- Changed the default top-up amount from `4T` to `5T`, keeping the default
  threshold at `10T`.

### 🐛 Fixed

- `canic replica status` now reports a stopped local project network as
  `stopped` instead of surfacing the ICP CLI "network is not running" error.
- Foreground `canic replica start` now preserves ICP stderr while streaming
  output, so port-owner conflicts can be classified instead of falling back to a
  generic process status.

### 🧪 Testing

- Added targeted coverage for `topup = {}` default parsing, local ICP config
  port reads/updates, ICP project section reconciliation, replica command port
  parsing, stopped-network diagnostics, local replica endpoint selection, and
  the new `fleet sync` parser path.

## 0.35.2 - 2026-05-12 - TBD

### 🔧 Changed

- Root init and post-upgrade now retain the installing/upgrading root controller
  in the runtime controller set used for newly allocated managed children.
- Renamed the test fleet scaling worker role from `scale` to `scale_replica`,
  renamed the canister config field from `topup_policy` to `topup`, and enabled
  default top-up policy blocks for `app`, `user_hub`, `user_shard`, `scale_hub`,
  and `scale_replica`.
- Slimmed the ICP build hook path: `icp.yaml` now invokes the host
  `build_artifact` entrypoint directly, the Rust builder owns
  `ICP_WASM_OUTPUT_PATH` copying, and the old `scripts/app/build.sh` wrapper
  has been removed.

### 🧭 Migration Notes

- Existing deployed canisters keep their current controller settings until they
  are reallocated, reinstalled, or repaired by a future controller-repair path.

### 🧪 Testing

- Added targeted coverage for root-controller retention in runtime config.

## 0.35.1 - 2026-05-12 - TBD

### ⚠️ Breaking

- Hard-cut managed child controller policy: every newly allocated non-root
  canister now gets its configured controllers, the fleet root, and its direct
  parent as controllers. Reused pool canisters are updated to the same
  controller set before install.

### 🔧 Changed

- Tightened `canic install <fleet>` build output: unset requested profiles are
  hidden, environment-variable labels were replaced with operator labels, same
  workspace/ICP roots are no longer repeated, and the build table now includes
  each generated `.wasm.gz` artifact size. Local root cycle top-ups now print
  the top-up phase, amount, and target instead of the old vague "at least"
  wording.
- Direct parents can now perform controller-authorized maintenance on their
  immediate children, reducing pressure on root for stop/start/snapshot-style
  operations.
- Pool canisters remain parked with configured controllers plus root; the
  selected direct parent is added when a pool canister is allocated into the
  live topology.

### 🧭 Migration Notes

- Existing deployed canisters keep their current controller settings until they
  are reallocated, reinstalled, or repaired by a future controller-repair path.

### 🧪 Testing

- Added targeted coverage for controller-set deduplication and verified the
  provisioning slice with `canic-core` tests.

## 0.35.0 - 2026-05-12 - TBD

### 🔧 Changed

- Added explicit restore-run stop/start phases. Restore apply journals now
  schedule snapshot upload, target canister stop, snapshot load, target
  canister start, and verification operations so `restore run --execute` can
  advance a normal restore without manual stop/start steps.
- Tightened local replica start/stop plumbing so Canic calls the ICP CLI local
  network path directly instead of relying on broader environment selection.
- Changed `canic install <fleet>` build progress output so the Canic build
  context is printed as one key/value per line above the artifact table,
  without the old `Build artifacts:` label interrupting the table.

### 🧹 Cleanup

- Moved the old 0.34 backup/restore design forward into
  `docs/design/0.35-backup-restore/0.35-design.md` and marked the 0.34 draft as
  superseded.
- Split the backup CLI command root into focused `create`, `inspect`,
  `labels`, `model`, `reference`, `status`, and `verify` modules while keeping
  command UX and report labels in the CLI crate.
- Split `canic-backup::plan` into plan building, error, validation, preflight,
  and type modules.
- Split `canic-backup::manifest` into manifest types, validation, summary, and
  error modules.
- Split the executable backup runner into root orchestration, operation
  handlers, manifest construction, and support helpers.
- Split restore planning into mapping validation, member resolution, summaries,
  dependency ordering, types, and errors.
- Split the execution journal into journal orchestration, operation validation,
  receipt handling, and shared validation helpers.
- Split snapshot download into target resolution, topology hashing, manifest
  projection, capture, and support helpers.

### 📊 Audit

- Reran the CLI/host/backup boundary audit after the cleanup. The current
  focused boundary risk is `4/10`: installed-fleet resolution is centralized in
  host, and the remaining risk is concentrated in thin CLI live adapters and
  backup-domain file size.
- Added a cleanliness rerun report for the same slice. The follow-up cleanup
  reduced the broad mixed-responsibility roots: `plan/mod.rs` is now a facade,
  `manifest/mod.rs` is now a facade, and `snapshot/mod.rs` and `runner/mod.rs`
  only own top-level orchestration.

### 🧪 Testing

- Verified the backup boundary and module cleanup with `cargo check` for
  `canic-backup`, host/backup/CLI checks, full `canic-backup` and `canic-cli`
  library tests, targeted plan/manifest/runner/restore-plan/execution/snapshot
  tests, dependency-tree inspection, and clippy for `canic-backup`,
  `canic-host`, and `canic-cli`.
- Verified the restore stop/start journal path with targeted
  `canic-backup restore` and `canic-cli restore` tests.
