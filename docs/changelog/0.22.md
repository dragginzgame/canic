# 0.22.x Detailed Changelog

Technical release notes for the `0.22.x` minor line.

## 0.22.2 - 2026-04-03 - Streaming Wasm Publication Follow-Through

### 📝 Summary

- `0.22.2` continues the downstream `wasm_store` bootstrap fix by removing the last whole-release buffering from the live root/store publication path, switching staged chunked-release verification to incremental hashing, and further reducing the instruction and memory cost of publishing larger managed release sets during bootstrap.

### 🔧 Changed

- `crates/canic-control-plane/src/workflow/runtime/template/publication.rs` now fetches source chunks one at a time during store publication instead of building a full `Vec<Vec<u8>>` for the whole release before the publish loop starts.
- `crates/canic-control-plane/src/workflow/runtime/template/mod.rs` now replays missing store chunks one at a time as well, so store-to-store chunk replay no longer buffers an entire release before re-uploading it into the management chunk store.
- `crates/canic-control-plane/src/ops/storage/template/chunked.rs` now verifies staged chunked payloads with incremental `Sha256` updates instead of concatenating the whole payload into one `Vec<u8>` just to confirm `payload_hash` and `payload_size_bytes`.
- The earlier local-store capacity fixes from `0.22.1` still stand, but the live publication/runtime path now matches that same “incremental, chunk-at-a-time” shape instead of carrying one last whole-release memory spike in downstream bootstrap publication.

### 🧪 Testing

## 0.22.1 - 2026-04-03 - Publication and Audit Follow-Through

### 📝 Summary

- `0.22.1` follows the initial audit baseline by caching the expensive debug small-store reconcile setup, making heavy workspace test runs easier to attribute, recording the first dated `0.22` instruction-footprint report, hardening the wasm audit preflight, preserving the publish lane's one intentional local test-only edge, and trimming managed `wasm_store` publication hot paths so large downstream release sets avoid instruction-limit failures during bootstrap.

### 🔧 Changed

- `root_wasm_store_reconcile` now restores a cached debug small-store baseline after the first setup instead of paying a fresh small-store root install for every reconcile test.
- `scripts/ci/run-workspace-tests.sh` now prints a compact end-of-run timing summary table so the heavy PocketIC suites stay attributable without reading every phase log line.
- `scripts/ci/wasm-audit-report.sh` now fails fast if local `dfx` is not reachable or `dfx canister create --all` fails, instead of silently ignoring setup failure and dying later on the first `dfx build`.
- Managed `wasm_store` publication no longer re-exports and re-accounts the full local store on every chunk publish. The store now does a cheap incremental occupied-byte check in the hot chunk path, which avoids the `IC0522` single-message instruction blowup that large downstream release sets could trigger during bootstrap publication.
- Root-side publication now caches the management-canister `stored_chunks` set once per target store during one publication pass and updates that cache as it uploads new chunks, instead of re-fetching the full chunk-hash list for every template routed into the same store.
- `crates/canic/tests/workspace_manifest.rs` now allows the one intentional local `canic-core` dev-dependency edge on `canic-testkit`, and `crates/canic-core/Cargo.toml` uses that local path again so the publish lane can break the test-only crates.io cycle between `canic-core` and the later-published `canic-testkit`.

### 📊 Audit

- The first dated `0.22` instruction-footprint report is now recorded at [docs/audits/reports/2026-04/2026-04-03/instruction-footprint.md](/home/adam/projects/canic/docs/audits/reports/2026-04/2026-04-03/instruction-footprint.md).
- The first dated `0.22` wasm-footprint rerun is still blocked on a reachable local `dfx` replica. The runner now reports that precondition immediately instead of burning time on a full Cargo release build first.

## 0.22.0 - 2026-04-02 - Audit and Artifact Baseline Reset

### 📝 Summary

- `0.22.0` opens the new audit/wasm-size/perf line by making local `.dfx` artifact reuse aware of build env and profile, pushing more reusable PocketIC root-baseline plumbing into `canic-testkit`, standardizing three wasm build lanes (`debug`, `fast`, `release`) across repo-local and downstream helpers, and removing the last special-case reconcile prebuild glue so future audit work starts from reproducible inputs instead of stale artifact reuse.

### 🔧 Changed

- `canic-testkit::artifacts::dfx` now records a build-env stamp alongside local `.dfx` artifacts and treats those artifacts as reusable only when the requested `DFX_NETWORK`, wasm profile, and extra build env match the last successful build.
- Canic now exposes one explicit wasm-profile contract everywhere: `CANIC_WASM_PROFILE=debug|fast|release`.
- The new `fast` Cargo profile is the middle shrunk lane for PocketIC suites, `make demo-install`, `make test-canisters`, and downstream local fast installs. `release` remains the shipping/install lane, while `debug` stays available for raw large wasm output.
- Repo-local and published downstream builders now agree on the same profile rules. The generated hidden `wasm_store` wrapper defines both `[profile.fast]` and `[profile.release]`, and the installed/package verification probes exercise the `fast` path directly.
- The older `RELEASE` compatibility path is gone. Profile selection now flows only through `CANIC_WASM_PROFILE`.
- The shared root baseline builder in `canic-testkit` now carries per-suite build env through its artifact-readiness and `.dfx` build path, so specialized suites can request their own build flags without bypassing the shared root harness.
- `root_wasm_store_reconcile` now uses the shared root harness directly for its small-store build variant instead of manually prebuilding custom artifacts outside the reusable baseline path.
