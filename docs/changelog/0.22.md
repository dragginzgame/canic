# 0.22.x Detailed Changelog

Technical release notes for the `0.22.x` minor line.

## 0.22.9 - 2026-04-03 - Local Install Path Tightening

### 📝 Summary

- `0.22.9` tightens the local thin-root install path by fabricating cycles only when local root is actually short, building only `root` plus the configured release roles from the root-owning subnet, keeping the normal wait loop quieter, and removing the now-redundant DFX dependency edges from the reference `dfx.json`.

### 🔧 Changed

- [install_root.rs](/home/adam/projects/canic/crates/canic-installer/src/install_root.rs) now probes `dfx canister status` during local installs, skips `dfx ledger fabricate-cycles` when root is already funded, and otherwise fabricates only the missing delta up to the local target floor instead of always minting the full previous amount again.
- The same local install path now builds only `root` plus the configured release roles from the subnet that owns `root`, so extra canisters that happen to exist in `dfx.json` but are not part of the active Canic release topology stop paying local build cost.
- The normal `canic_ready` wait loop no longer prints recent root logs every five seconds, keeping live local-install output readable while preserving the fuller log dump on actual timeout or bootstrap failure.
- [dfx.json](/home/adam/projects/canic/dfx.json) no longer carries the old `user_hub -> user_shard` and `scale_hub -> scale` dependency edges, because the Canic-managed local install path now resolves its build set from `canic.toml` instead of relying on DFX dependency ordering.
- [README.md](/home/adam/projects/canic/README.md) now describes the current local install flow accurately as a config-driven targeted build instead of the older `dfx build --all` behavior.

## 0.22.5 - 2026-04-03 - Init-Time Rollover Import Follow-Through

### 📝 Summary

- `0.22.5` continues the downstream `wasm_store` instruction-limit follow-through by removing a redundant init-time fleet catalog import after managed publication, so root no longer snapshots the just-retired rollover store again before bootstrap can complete.

### 🔧 Changed

- [root.rs](/home/adam/projects/canic/crates/canic-control-plane/src/workflow/bootstrap/root.rs) no longer calls `import_default_wasm_store_catalog()` during the fresh-init `root_create_canisters()` path after `publish_staged_release_set_to_current_store()`.
- Managed publication already mirrors each selected store binding back into root-owned approved manifest state as it publishes, so the extra init-time catalog import was redundant in the normal bootstrap path.
- On downstream rollover, that redundant import immediately re-snapshotted both the new and old managed stores, which could hit the just-retired old store with one more expensive `status` plus `catalog` pass before root ever reached `canic_ready`.
- The reconcile/upgrade path still keeps the explicit catalog import, because that path legitimately needs to rebuild root-owned approved bindings from the live managed fleet rather than from the publication pass that just ran.

## 0.22.4 - 2026-04-03 - Managed Store Install Preflight Follow-Through

### 📝 Summary

- `0.22.4` continues the downstream `wasm_store` instruction-limit follow-through by removing the managed-store chunk-store preflight during install-source resolution, so root no longer asks a freshly published store to enumerate its whole chunk-hash set again right before `install_chunked_code`.

### 🔧 Changed

- `crates/canic-control-plane/src/workflow/runtime/template/mod.rs` no longer calls the managed-store chunk-store preflight during `approved_module_source_from_manifest(...)` for store-backed releases.
- Managed-store publication already uploads missing chunks before promoting a manifest into the store, so install-source resolution now relies on that publication contract instead of paying one more full `stored_chunks(store_pid)` scan first.
- This keeps the managed-store install path aligned with the earlier `0.22.2` and `0.22.3` work: publication remains chunk-at-a-time, store-side byte accounting stays incremental, and the post-publication install path stops doing one last store-sized readback in the common case.

### 📝 Summary

- `0.22.3` finishes the downstream `wasm_store` instruction-limit follow-through by replacing repeated full-store occupied-byte rescans with incremental counters, so one new chunk upload no longer re-serializes every already-stored chunk record just to enforce capacity.

### 🔧 Changed

- `crates/canic-control-plane/src/storage/stable/template/chunked.rs` now keeps incremental occupied-byte counters for the chunk-set and chunk stores, updating them on each upsert and reset instead of recomputing occupied bytes by walking and re-serializing the full stable map on every capacity check.
- `crates/canic-control-plane/src/storage/stable/template/manifest.rs` now does the same for manifest occupied-byte accounting, so all three local store-accounting lanes use the same cached-counter model.
- The downstream hot path in `publish_chunk_in_store_from_input(...)` still performs per-chunk capacity enforcement, but that check now reads cached occupied-byte totals instead of paying a CBOR serialization pass over all already-stored chunks first.

## 0.22.2 - 2026-04-03 - Streaming Wasm Publication Follow-Through

### 📝 Summary

- `0.22.2` continues the downstream `wasm_store` bootstrap fix by removing the last whole-release buffering from the live root/store publication path, switching staged chunked-release verification to incremental hashing, and further reducing the instruction and memory cost of publishing larger managed release sets during bootstrap.

### 🔧 Changed

- `crates/canic-control-plane/src/workflow/runtime/template/publication.rs` now fetches source chunks one at a time during store publication instead of building a full `Vec<Vec<u8>>` for the whole release before the publish loop starts.
- `crates/canic-control-plane/src/workflow/runtime/template/mod.rs` now replays missing store chunks one at a time as well, so store-to-store chunk replay no longer buffers an entire release before re-uploading it into the management chunk store.
- `crates/canic-control-plane/src/ops/storage/template/chunked.rs` now verifies staged chunked payloads with incremental `Sha256` updates instead of concatenating the whole payload into one `Vec<u8>` just to confirm `payload_hash` and `payload_size_bytes`.
- The earlier local-store capacity fixes from `0.22.1` still stand, but the live publication/runtime path now matches that same “incremental, chunk-at-a-time” shape instead of carrying one last whole-release memory spike in downstream bootstrap publication.

### 🔧 Changed

- `root_wasm_store_reconcile` now restores a cached debug small-store baseline after the first setup instead of paying a fresh small-store root install for every reconcile test.

## 0.22.0 - 2026-04-02 - Audit and Artifact Baseline Reset

### 📝 Summary

- `0.22.0` opens the new audit/wasm-size/perf line by making local `.dfx` artifact reuse aware of build env and profile, pushing more reusable PocketIC root-baseline plumbing into `canic-testkit`, standardizing three wasm build lanes (`debug`, `fast`, `release`) across repo-local and downstream helpers, and removing the last special-case reconcile prebuild glue so future audit work starts from reproducible inputs instead of stale artifact reuse.

### 🔧 Changed

- `canic-testkit::artifacts::dfx` now records a build-env stamp alongside local `.dfx` artifacts and treats those artifacts as reusable only when the requested `DFX_NETWORK`, wasm profile, and extra build env match the last successful build.
- Canic now exposes one explicit wasm-profile contract everywhere: `CANIC_WASM_PROFILE=debug|fast|release`.
- The new `fast` Cargo profile is the middle shrunk lane for PocketIC suites, `make demo-install`, `make test-canisters`, and downstream local fast installs. `release` remains the shipping/install lane, while `debug` stays available for raw large wasm output.
- Repo-local and published downstream builders now agree on the same profile rules. The generated hidden `wasm_store` wrapper defines both `[profile.fast]` and `[profile.release]`, and the installed/package verification probes exercise the `fast` path directly.
- The older `RELEASE` compatibility path is gone. Profile selection now flows only through `CANIC_WASM_PROFILE`.
- The shared root baseline builder in `canic-testkit` now carries per-suite build env through its artifact-readiness and `.dfx` build path, so specialized suites can request their own build flags without bypassing the shared root harness.
- `root_wasm_store_reconcile` now uses the shared root harness directly for its small-store build variant instead of manually prebuilding custom artifacts outside the reusable baseline path.
