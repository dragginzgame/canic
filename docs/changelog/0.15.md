# 0.15.x Detailed Changelog

Technical release notes for the `0.15.x` minor line.

## 0.15.0 - 2026-03-12 - Unified Auth Identity Foundation

### 📝 Summary

- `0.15.0` makes delegated-session bootstrap cryptographically strict and formalizes caller-vs-subject boundaries so user-session identity support does not weaken infrastructure trust checks.
- This release focuses on auth model clarity and hardening while preserving existing delegated token, role-attestation, and capability trust semantics.

### 🔧 Changed

- Delegated-session bootstrap now requires a verified delegated token and enforces subject binding (`requested delegated subject` must equal `token subject`).
- Delegated-session expiry is now clamped to the earliest bound across token expiry, configured max TTL, and optional request TTL.
- Legacy delegated-caller helper paths now route through the same token-gated delegated-session bootstrap semantics.
- Added a root-guarded test helper for deterministic delegation proof/key installation in controlled integration scenarios where PocketIC chain-key ECDSA is unavailable.
- Added integration coverage proving delegated sessions affect `authenticated(...)` resolution while caller/topology predicates still consume raw `msg_caller()`.

### 🩹 Fixed

- Prevented delegated-session bootstrap without strong proof.
- Prevented delegated sessions from outliving the bootstrap token validity window.
- Prevented delegated-session identity from accidentally altering raw-caller infrastructure checks through explicit regressions.

### ⚠️ Breaking

- `DelegationApi::set_delegated_session_subject` now requires a bootstrap token and optional requested TTL instead of accepting a standalone TTL.
- `DelegationApi::set_delegated_caller` compatibility helper now requires a bootstrap token and returns `Result<(), Error>`.

### 🧭 Migration Notes

- Treat legacy delegated-caller usage as hardened delegated-session semantics: bootstrap is token-gated and subject-bound.
- Session expiry is clamped to token/config/request TTL bounds; do not rely on unbounded or sticky delegated identity.
- Raw infrastructure predicates (`root`/`controller`/`parent`/`subnet` and related trust checks) remain raw-caller-bound by design.
- Replay/session-binding controls now reject or classify repeated bootstrap material explicitly; update clients to avoid token replay.
- Auth observability now includes delegated-session lifecycle and replay rejection signals; wire dashboards/alerts to the new metrics.

```rust
DelegationApi::set_delegated_session_subject(delegated_subject, bootstrap_token, Some(300))?;
```

### 📚 Documentation

- Added `0.15` design and status tracking docs:
  - `docs/design/0.15-auth-identity-foundation/0.15-design.md`
  - `docs/design/0.15-auth-identity-foundation/0.15-status.md`

### 🧪 Testing

- Added clamp-behavior unit coverage for delegated-session expiry bounds.
- Added regression coverage for:
  - delegated session affecting `authenticated(...)` only
  - role-attestation/capability/root/controller/parent/subnet checks remaining raw-caller bound

## 0.15.2 - 2026-03-16 - Shard Delegation Routing Fix

### 📝 Summary

- `0.15.2` restores shard-initiated delegated token issuance when a shard needs a fresh delegation proof.

### 🩹 Fixed

- Fixed a regression where `DelegationApi::issue_token` on non-root shards could be denied because delegation issuance was evaluated through a root-only local workflow path.
- `DelegationApi::request_delegation` now forwards non-root calls to root via `canic_request_delegation`, so `IssueDelegation` authorization runs in the correct root context while preserving root-only enforcement.

## 0.15.3 - 2026-03-16 - Legacy Compatibility Cleanup and Layer Audit Rerun

### 📝 Summary

- `0.15.3` removes legacy compatibility surfaces that are no longer needed in current environments and records a fresh layer-violations rerun after the cleanup.

### 🗑️ Removed

- Removed legacy delegated-caller compatibility helpers from `DelegationApi` (`set_delegated_caller`, `clear_delegated_caller`, `delegated_caller`) so delegated-session APIs are the only supported auth bootstrap surface.
- Removed legacy `is_authenticated` compatibility aliases from the auth DSL/core access expression path; endpoint guards use `auth::authenticated(...)` only.
- Removed legacy root replay-key compatibility reads (`legacy_root_slot_key`) so replay lookups only use the canonical root replay slot key.
- Removed stable decode alias `delegated_callers` from delegation state storage; only `delegated_sessions` is now recognized.
