# Canic 0.93 Changelog

## [0.93.3] - 2026-07-17 - Bounded Refill Operations and Renewal Diagnostics

### 📝 Summary

0.93.3 removes repeated full-history scans from normal ICP-refill admission,
retry lookup, ID allocation, and metric collection. Durable refill records
remain the sole persistence authority; lifecycle startup rebuilds heap-only
lookup and aggregate indexes, and record mutations keep them synchronized.

Delegated-proof renewal timers now retain typed failure classification in
recent runtime diagnostics instead of discarding failed sweep results. The
release also restores the publication-test layering boundary.

The release does not change public Candid, JSON, configuration, CLI, or stable
record layouts.

### 🩹 Fixed

- Replaces normal ICP-refill record-history scans with derived operation,
  active-refill, maximum-ID, status, error, and per-target aggregate indexes.
- Rebuilds the derived refill indexes synchronously during init and
  post-upgrade. Conflicting operation identities or active refill keys fail
  closed with typed storage causes instead of admitting an ambiguous index.
- Preserves existing metric semantics, including saturating cycle totals and
  completed refills that sent zero cycles, while making projection proportional
  to current metric dimensions rather than durable history.
- Counts every failed delegated-proof renewal sweep and records its bounded
  error class and origin in recent diagnostics without stopping later timer
  attempts.
- Moves publication lifecycle fixtures behind the storage-ops boundary so
  workflow tests no longer construct stable records directly.

### 📊 Audit

- The focused implementation delta spans 16 files, with 960 insertions and 371
  deletions before release-note finalization.
- Durable ICP-refill records remain canonical. The new indexes are derived
  runtime state, add no stable memory allocation, and are rebuilt from the
  complete record set before runtime services become ready.
- Index maintenance validates replacements before mutation and verifies that
  indexed lookups still match canonical records, preserving fail-closed typed
  causes for missing, conflicting, or mismatched state.
- No compatibility alias, duplicate refill path, fallback scan, new public
  surface, or stable-state migration is introduced.

### 🧪 Testing

- Eighty focused ICP-refill tests cover current policy and execution behavior,
  derived-index rebuild, duplicate operation and active-key rejection,
  terminal transitions, metric aggregation, retry identity, and recovery.
- Fifteen renewal tests include typed timer-failure projection, and twenty-two
  control-plane publication tests cover the ops-owned fixture boundary and
  maintained lifecycle behavior.
- Strict all-target Clippy passes for `canic-core` and
  `canic-control-plane`; the layering guard, Cargo Machete, formatting,
  changelog governance, and diff hygiene also pass.

## [0.93.2] - 2026-07-16 - Fail-Closed Lifecycle and Restore Receipts

### 📝 Summary

0.93.2 closes the two P1 safety findings from the post-0.93.1 audit. Wasm-store
retirement now has one generation-bound root lifecycle and a one-way
store-local write boundary. Restore apply receipts no longer infer successful
outcomes or attempt identity from omitted JSON fields.

The release does not change the public Candid schema or stable-state layout.
It intentionally hard-cuts permissive restore-receipt decoding rather than
adding a legacy fallback or compatibility representation.

### 🩹 Fixed

- Binds remote prepare, begin, complete, and finalize results to the same
  retired binding and publication generation observed before the inter-canister
  call. A concurrent ownership transition is returned as a typed conflict
  instead of committing against newer state.
- Serializes root-owned GC/finalize/delete orchestration with a workflow-local
  in-flight guard and treats unexpected persistence outcomes as errors rather
  than discarding them.
- Makes the GC lifecycle one-way for publication authority. Explicit binding,
  automatic placement, release reconciliation, and direct-target publication
  all reject stores whose GC mode is not `Normal`.
- Enforces the same write boundary inside the store canister, so chunk-set
  preparation, chunk publication, and manifest staging fail closed after GC
  starts even if a root-side snapshot becomes stale.
- Requires finalized-store deletion to match the authoritative binding,
  canister principal, unreferenced publication state, and completed local GC
  record both before and after the remote status call.
- Makes restore operation-receipt `outcome` and `attempt` mandatory JSON fields
  and rejects attempt zero with a typed journal-validation cause.

### ⚠️ Breaking

- Restore apply journals containing operation receipts without `outcome` or
  `attempt` no longer deserialize. Receipts using attempt zero no longer pass
  validation.
- Publication requests targeting a store in `Prepared`, `InProgress`,
  `Clearing`, or `Complete` GC state now return a conflict instead of allowing
  the store to regain or retain publication authority.
- No compatibility alias, inferred receipt default, legacy decoder, or
  alternate publication path is retained.

### 🧭 Migration Notes

- Regenerate malformed restore journals or provide the explicit durable
  receipt fields: `outcome` and an `attempt` value starting at one.
- Complete retirement and deletion for a store that has entered GC; publish to
  a different `Normal` store rather than trying to reactivate it.

### 📊 Audit

- The focused source-and-test delta spans 15 files, with 639 insertions and 53
  deletions before release-note finalization.
- The Wasm-store fix covers explicit selection, automatic placement,
  reconciliation, direct publication, store-local mutations, asynchronous GC
  commits, finalization, and destructive deletion.
- The restore fix removes both permissive defaults and proves decode-time and
  validation-time rejection boundaries.

### 🧪 Testing

- Twenty-one focused control-plane tests cover publication selection,
  generation drift, in-flight lifecycle rejection, exact deletion identity,
  typed error projection, and the store-local write gate.
- Thirty-four restore apply-journal tests cover positive receipt behavior,
  missing-field rejection, zero-attempt rejection, duplicates, transition
  ordering, command previews, and journal invariants.
- Targeted package checks, strict all-target Clippy for the control-plane and
  backup packages, formatting, changelog governance, and diff hygiene pass.

## [0.93.1] - 2026-07-16 - Host ICP Surface Hard Cuts

### 📝 Summary

0.93.1 removes an orphaned group of public `canic-host` ICP command helpers
identified by the 0.93 reachability audit. Production callers continue to use
explicit Candid arguments, project-rooted replica operations, typed JSON
snapshot-create receipts, and `canic-backup` restore orchestration.

This is a pre-1.0 hard cut. It adds no aliases, compatibility wrappers,
fallback snapshot-ID parsing, or deprecated surfaces.

### 🗑️ Removed

- Removes uncalled canister call/query convenience wrappers and dry-run
  renderers that duplicated the maintained explicit-argument and
  Candid-aware command paths.
- Removes non-rooted local-replica start, status, ownership, and stop wrappers
  plus their unused display helpers. Maintained replica lifecycle operations
  require an explicit project root.
- Removes host-side snapshot upload and restore wrappers, the upload receipt
  type, and their dry-run renderers. Restore execution remains owned by
  `canic-backup`.
- Removes the legacy text snapshot-ID parser and its unreachable
  `SnapshotIdUnavailable` error. Snapshot creation continues to decode the ICP
  CLI JSON receipt directly.

### 📊 Audit

- The focused source-and-test delta spans eight files, with 9 insertions and
  394 deletions before release-note finalization.
- Workspace call-site tracing found no production caller for any removed
  helper, receipt, parser, display path, or error variant.
- Receipt-backed artifact promotion, the required `finish!` marker, and
  intentional feature-conditioned suppressions remain reachable and intact.

### ⚠️ Breaking

- Downstream Rust code importing the removed `canic-host::icp` helpers,
  `IcpSnapshotUploadReceipt`, `parse_snapshot_id`, or
  `IcpCommandError::SnapshotIdUnavailable` must move to the maintained paths.
- Replica lifecycle callers must supply an explicit project root through the
  rooted methods. Snapshot restore callers must use the backup restore
  workflow rather than the deleted host wrappers.
- There is no compatibility alias or fallback text parser for these removed
  pre-1.0 APIs. Operator-facing Canic CLI behavior is unchanged because no
  production command called them.

### 🧭 Migration Notes

- Decode `snapshot create --json` as `IcpSnapshotCreateReceipt` instead of
  extracting an identifier from human-readable output.
- Use explicit-argument/Candid-aware canister helpers where applicable and the
  `*_in` replica methods for project-scoped local replica operations.
- Remove exhaustive error-match arms for `SnapshotIdUnavailable`; JSON decode
  failures continue to use `IcpCommandError::Json`.

### 🧪 Testing

- Focused host ICP tests cover maintained Candid-aware calls, rooted command
  construction, JSON snapshot creation, status parsing, and CLI version
  enforcement.
- The dependent CLI compiles and its medic error classification remains
  covered after removal of the unreachable error variant.
- Strict all-target Clippy for the affected host and CLI packages, formatting,
  changelog governance, removed-symbol scanning, and diff hygiene pass.

## [0.93.0] - 2026-07-16 - Pre-1.0 Surface Hard Cuts

### 📝 Summary

0.93.0 removes stale and unreachable surfaces found by a repository-wide
read-only audit. The hard cuts cover an obsolete issuer-renewal authority,
replay state with no production writer, unused compatibility APIs, and
serialized variants that current producers cannot emit.

Maintained behavior is now backed by one production authority per concern.
The persisted chain-key batch owns issuer-renewal progress, replay keeps only
live reservation and recovery states, and template publication removes stale
manifests instead of retaining deprecated tombstones.

This release intentionally adds no aliases, fallback decoding, compatibility
records, or anti-resurrection paths for the removed pre-1.0 contracts.

### 🔧 Changed

- Root issuer-renewal status now projects `latest_batch` from the canonical
  persisted chain-key batch instead of projecting a separate
  `active_attempt` record.
- CLI text and JSON report batch status, proof epoch, expiry, installation,
  retry, and failure evidence. A reachable issuer without a usable active
  proof is reported as `proof_unavailable` rather than ready.
- Template publication deletes superseded or unconfigured approved manifests;
  store accounting is updated when those records are removed.
- Chain-key signature verification and delegated root-proof callbacks carry
  only the values used by cryptographic verification. Key identity,
  derivation-path, and certificate-hash checks remain enforced by their
  owning validation stages.

### 🗑️ Removed

- Removes the issuer-renewal attempt model, persisted attempt collection,
  attempt-only metrics, outcome and failure counters, and the old status
  projection. Obsolete attempt data is neither migrated nor exposed.
- Removes replay `TerminalFailed` state and its bounded terminal-error model,
  commit path, classifications, and workflow branches. Production never wrote
  that state.
- Removes the Wasm-store `Deprecated` template-manifest state and the obsolete
  aggregate chunk-set request and storage path.
- Removes backup authority, snapshot-read, quiescence, and proof-source
  variants that no current plan or preflight producer emits, plus the
  unproduced `InstalledModuleHash` artifact-digest source.
- Removes uncalled root-capability, endpoint-performance, lifecycle,
  topology-query, RPC, pool, ledger, CLI, and private compatibility surfaces
  identified by the audit.

### 📊 Audit

- The release delta from `v0.92.14` spans 81 files, with 776 insertions and
  1,739 deletions after release-note finalization.
- Every removed serialized variant was traced through constructors,
  validation, rendering, persistence, and tests before deletion.
- Intentional feature-conditioned stable schemas, canonical auth snapshots,
  delegated-auth metrics, and the required `finish!` marker remain in place.
- Receipt-backed promotion inputs remain supported because operator-provided
  promotion JSON reaches and validates both maintained receipt kinds.

### ⚠️ Breaking

- `RootIssuerRenewalStatusResponse` replaces `active_attempt` with
  `latest_batch` and removes attempt-era state fields. CLI JSON consumers must
  update to the current batch projection.
- `TemplateManifestState::Deprecated` is no longer accepted by the Wasm-store
  Candid or stable-record contract. Persisted stores must not contain that
  state before upgrade; 0.93.0 intentionally provides no compatibility
  decoder or migration record.
- Backup JSON using removed authority, quiescence, proof-source, or artifact
  digest variants must be regenerated with the current CLI.
- Manually injected replay receipts using `TerminalFailed` are unsupported;
  live production replay states retain their existing stable encoding.

### 🧭 Migration Notes

- Update automation that parses issuer-renewal status to consume
  `latest_batch` and its batch fields instead of `active_attempt`,
  `last_outcome`, `consecutive_failures`, or `active_attempt_id`.
- Regenerate persisted backup plans or receipts that were authored with
  unsupported enum variants.
- Verify Wasm-store manifest state before upgrading. Superseded and
  unconfigured approved manifests are removed by current publication flows
  rather than converted to deprecated tombstones.
- No application configuration or source migration is required when these
  removed serialized states and response fields are not consumed.

### 🧪 Testing

- Pins the exact stable bytes of a live replay recovery receipt, proving that
  removal of the unwritten terminal variant does not alter maintained replay
  encoding.
- Current-behavior coverage exercises issuer-renewal batch projection,
  delegated proof verification, template replacement and pruning, backup plan
  preflights, CLI rendering, and host adoption.
- Strict package-scoped linting, formatting, Candid contract checks,
  changelog governance, and diff hygiene pass for the affected surfaces.
