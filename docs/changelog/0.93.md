# Canic 0.93 Changelog

## [0.93.26] - 2026-07-18 - Bounded Delegated-Token Preparation

### 📝 Summary

`0.93.26` closes an availability finding on the public delegated-token
preparation endpoint. Sequential successful calls previously committed their
stable replay receipts and therefore escaped the pending-only replay quota,
while prepared-token and issuer-proof metadata accumulated without an enforced
retention cap. An unauthenticated caller could repeatedly choose new request
identities and consume stable memory, heap, certified-signature work, and
cycles.

Fresh delegated-token preparation now has the same explicit limits at both
retention boundaries: 64 active entries per caller and 512 globally. Expired
state releases capacity before admission, overload fails closed with typed
`ResourceExhausted`, and an exact committed replay still returns its original
response even when fresh work is saturated.

### 🩹 Fixed

- Makes command-specific replay retention count every active unexpired
  receipt, including committed responses, rather than only pending external
  effects.
- Purges a bounded batch of expired receipts for the exact delegated-token
  command before applying its per-caller and global limits.
- Bounds caller-owned prepared-token metadata with the same limits and prunes
  it at the exact one-minute retrieval boundary.
- Preserves deterministic replay semantics at saturation: matching committed
  work succeeds, while actor mismatch, payload mismatch, expiry, and storage
  failures retain their existing typed causes.

### 🧹 Cleanup

- Removes the separate pending issuer-proof metadata map. One retained-token
  record now owns prepared claims, caller binding, and retrieval expiry.
- Keeps the third-party signature map focused on certified cryptographic
  witnesses and relies on its bounded expiry pruning during additions.
- Keeps the generic replay decision model unchanged; the new retained-response
  result and limits remain crate-private and apply only where explicitly
  selected by the delegated-token workflow.

### 📚 Documentation

- Records the public preparation capacity, exact-replay behavior, retrieval
  boundary, and single metadata authority in the authentication architecture
  and delegated-signature contract.

### 🧪 Testing

- Focused core tests cover committed-response accounting, per-caller and
  global saturation, exact-boundary pruning, query expiry, and exact committed
  replay at capacity.
- A public PocketIC regression prepares 64 unique tokens, observes typed
  `ResourceExhausted` on the 65th request, and proves that the first request
  remains exactly replayable at saturation.
- Corrects the reviewed cost-flow inventory to count both maintained
  non-root cycles recovery sites; the pre-effect recovery added in `.20` was
  present in the published `.25` source while its guard still expected one
  site.
- Feature-enabled core compilation, strict affected-package Clippy, layering,
  formatting, changelog governance, residue checks, and diff hygiene pass.

No Candid, JSON, stable-record schema, or Cargo package-version surface changes
in this patch.

## [0.93.25] - 2026-07-18 - Canonical ICP Environment Vocabulary

### 📝 Summary

`0.93.25` corrects the target vocabulary introduced in `.24`. `staging` is an
ICP environment selected through `-e/--environment`; it is not itself the
backing network declared by that environment. Canic now calls this selected
target `environment` consistently from public CLI admission through command
routing, deployment state, artifact selection, evidence, diagnostics, backup,
and restore.

The actual network referenced by an `icp.yaml` environment remains a backing
network, and `build_network` remains the resolved `local`/`ic` compile class
baked into Wasm. `artifact_environment` identifies the exact
`.icp/<environment>/canisters` namespace. `runtime_variant` remains separate
materialization identity.

### 🔧 Changed

- Hard-cuts the public Canic selector from `--network <name>` to
  `--environment <name>` and gives hidden command forwarding the same name.
- Renames selected-target Rust fields and parameters to `environment`,
  including `IcpCli`, workspace build context, install/deployment requests,
  deployment truth, release staging, live queries, and operator diagnostics.
- Renames exact artifact lookup from `artifact_network` to
  `artifact_environment` without changing the exact-root, no-fallback
  behavior introduced in `.24`.
- Renames backup source JSON to `source.environment` and restore plan identity
  to `source_environment`.
- Renames evidence-envelope targets, build-provenance inputs, deployment
  reports, install state, medic/status output, and other selected-target JSON
  fields to `environment`.
- Exposes the canonical `artifact_root_path` projection to CLI consumers and
  routes Candid and Wasm-size lookup through it instead of rebuilding `.icp`
  paths or trying selected/local artifact candidates.
- Keeps `network` only where it means the configured backing network, direct
  local replica network, runtime/auth build network, or an upstream API field.

### 🩹 Fixed

- Named readiness probes now render `icp network ping -e staging` instead of
  treating `staging` as a positional network name.
- Maintainer test-canister calls now select `ICP_ENVIRONMENT` through ICP CLI's
  environment selector instead of its direct network selector.
- Local replica lifecycle commands always target the actual `local` network;
  an environment carried by another ICP command context cannot be reinterpreted
  as a local network name.

### ⚠️ Breaking

- `canic --network <name>` is removed. Use
  `canic --environment <name>`; no alias or compatibility forwarding remains.
- Rust callers must replace selected-target `network` fields with
  `environment`, and artifact lookup fields with `artifact_environment`.
- Consumers must regenerate `.24` build provenance, evidence envelopes,
  deployment/install records, backup manifests, restore plans, and CLI JSON.
  The removed `network`, `artifact_network`, `source.network`, and
  `source_network` keys are not accepted as fallback inputs.
- Medic check codes and the selected-target category use environment
  terminology. Automation must consume the new current codes rather than
  matching the removed `.24` names.

### 🧭 Migration Notes

- `staging`, `local`, and `ic` are environment names. A custom environment may
  reference a separately named backing network in `icp.yaml`.
- Keep `build_network` when referring to the `local`/`ic` Wasm compile class.
  The upstream `ICP_ENVIRONMENT` child-build variable carries that class and
  is not renamed by Canic.
- Keep direct `-n/--network` only for the explicit local replica URL and root
  key path owned by the ICP CLI adapter.
- This is a pre-1.0 hard cut: no aliases, shims, duplicate JSON keys,
  deprecated constructors, or legacy fallback paths are retained.

### 🧪 Testing

- Focused host, CLI, and backup compilation covers the complete renamed Rust
  surface, including examples and unit-test targets.
- All 814 host, 617 CLI, and 198 backup unit tests pass. Focused tests cover
  global environment forwarding, named ICP command
  rendering, environment-to-build-network resolution, exact artifact
  environments, deployment/install state, evidence, backup manifests, and
  restore command selection.
- Residue scans distinguish the maintained backing-network, build-network,
  runtime/auth, and local-replica uses from removed selected-target names.
- Strict all-target Clippy passes for the affected packages, with formatting,
  changelog governance, shell syntax, CLI help, and diff hygiene also clean.

## [0.93.24] - 2026-07-18 - Canonical Network and Artifact Authority

### 📝 Summary

`0.93.24` fixes an artifact-safety gap exposed by the post-`.23` end-to-end
path audit. Normal root installation always materializes its complete build
under `.icp/local/canisters`, including when the deployment targets a named
network such as `staging`. Deployment-truth planning and inventory instead
derived their artifact directory from the deployment network. When that
selected-network directory did not exist, both sides omitted every
artifact row, leaving only warnings and allowing the install artifact gate to
continue without verifying the build it had just produced.

Deployment identity and artifact location are now separate explicit inputs.
Normal installs observe the exact local artifact network, while supplied
deployment plans and standalone deployment planning continue to use the exact
selected network. A missing exact root retains configured role projections, so
comparison produces deterministic missing-artifact hard failures rather than
comparing two empty evidence sets.

The accompanying naming pass hard-cuts a partial target-model transition. Canic
now uses `network` for the selected target, `build_network` for the resolved
`local`/`ic` compile class, and `artifact_network` for the exact artifact
namespace. `runtime_variant` remains a distinct materialization input. The
upstream `environments` configuration key, `-e/--environment` ICP selector, and
`ICP_ENVIRONMENT` child-build variable retain their upstream names only at that
adapter boundary.

### 🔧 Changed

- Adds required `artifact_network` authority to the public host-side local
  plan, inventory, artifact-manifest, and deployment-check request types.
- Records only the exact artifact root owned by the selected install mode in
  current execution context evidence.
- Projects configured artifact roles even when their exact directory is
  missing, preserving expected paths and role identity for fail-closed
  comparison.
- Makes `canic deploy plan --json` retain those configured `role_artifacts` and
  report missing-artifact blockers instead of returning an empty artifact set
  with warnings when the exact root is absent.
- Routes normal canister builds, bootstrap Wasm-store output, root-install
  planning, deployment observation, and CLI deployment planning through the
  canonical artifact-root projection.
- Gives `IcpCli` one selected-network field and maps every named Canic network
  consistently to ICP CLI's environment selector.
- Renames the build-context selected target to `network`, the build-class enum
  and resolver to `IcpBuildNetwork` and `resolve_icp_build_network_from_root`,
  and the matching provenance input to `build_network`.
- Records backup source identity as `source.network` and restore plans as
  `source_network`, matching the existing backup, deployment, and CLI models.
- Documents the canonical distinction between selected, build, artifact, and
  backing networks and the separate `runtime_variant` materialization field.

### 🩹 Fixed

- Named-network root installs now verify the fresh local artifacts produced
  by their complete build while retaining the named deployment network and its
  `ic` build-time configuration.
- A missing artifact root can no longer erase both planned and observed roles
  and thereby reduce a required artifact check to warnings only.
- Inspect, auth, blob-storage, metrics, cycles, token, snapshot, backup, medic,
  installed-deployment, and restore paths no longer interpret the same named
  Canic network through the direct `-n` selector while install and deployment
  truth use `-e`.
- Restore status and mutation commands now select named networks through the
  same ICP environment mechanism as the rest of Canic.

### 🗑️ Removed

- Removes the duplicate deployment-observation artifact-root resolver.
- Removes local artifact-path constants and direct production path assembly
  that competed with the release-set path authority.
- Removes `IcpCli`'s competing optional environment/direct-network target slots
  and the unused direct named-network rendering path.
- Removes the partial `environment`, `build_environment`, and
  `artifact_environment` Canic names without aliases or compatibility fields.

### ⚠️ Breaking

- Rust callers constructing `LocalDeploymentPlanRequest`,
  `LocalDeploymentCheckRequest`, `LocalInventoryRequest`, or
  `LocalArtifactManifestRequest` must provide `artifact_network` explicitly.
  Use the deployment network for exact prebuilt or promoted artifacts and
  `local` for artifacts produced by Canic's normal complete-build flow.
- Rust callers must use `WorkspaceBuildContext.network`, `IcpBuildNetwork`,
  `resolve_icp_build_network_from_root`, the two-argument `IcpCli::new`, and the
  three-argument `add_target_args` surface.
- Build-provenance consumers must read the `build_network` input with schema
  `canic.build_network` and note fields `network` plus `build_network`.
- Backup/restore JSON consumers must read `source.network` and
  `source_network`; the former `environment` fields are not accepted.
- No alias, default, selected-network fallback, or compatibility constructor is
  retained.

### 🧭 Migration Notes

- The maintained CLI supplies the field from its exact operation mode; no CLI
  flag or configuration change is required.
- Keep using `canic --network <name>`. Named networks, including `staging`, are
  now routed consistently; callers do not select `-e` or `-n` themselves.
- Regenerate pre-`.24` build-provenance and backup/restore JSON rather than
  translating the removed pre-1.0 field names.
- The deployment-plan JSON schema is unchanged, but missing-root output now
  contains projected `role_artifacts` and a blocked result rather than empty
  artifact evidence. No Candid, stable-state, package-version, or Cargo feature
  shape changes are introduced.

### 📊 Audit

- The host ICP adapter has one optional named target instead of two competing
  string slots, and named command rendering has one `-e` path instead of
  command-dependent `-e`/`-n` behavior.
- Production code has zero remaining `artifact_environment`,
  `build_environment`, `IcpBuildEnvironment`, `source_environment`, or
  `source.environment` owners.
- The only retained direct `-n` construction is the explicit local replica URL
  plus root-key path; upstream `environments` and `ICP_ENVIRONMENT` names remain
  confined to ICP configuration and process adapters.

### 🧪 Testing

- Focused deployment-truth tests cover exact selected-network lookup,
  retained role projections for a missing root, hard-failure generation, and
  a staging deployment observing local artifacts without fallback.
- Focused install tests exercise the actual named-network safety-check
  entrypoint, and focused build/bootstrap tests preserve canonical output paths.
- Focused host ICP-command tests pin named and implicit networks to `-e`, while
  backup/restore tests pin the hard-cut source schema and matching command
  selection.
- Strict all-target, all-feature Clippy passes for the affected host, CLI, and
  backup packages.

## [0.93.23] - 2026-07-18 - Pure Release-Set Path Authority

### 📝 Summary

`0.93.23` removes a misleading release-set path flow found during the
post-`.22` hard-cut audit. The public root-manifest path helper performed
directory creation and returned an I/O result even though its actual
responsibility was deterministic path projection. Existing-deployment
registration then called that helper through two fallback branches which all
computed the same final path while discarding discovery or directory errors.

Release-set artifact and manifest paths now come from one pure selected-network
projection. Build and deployment planning calculate paths without mutating the
filesystem; the durable writer remains the sole owner of parent creation,
staging, synchronization, publication, and typed I/O failure.

### 🔧 Changed

- Makes selected-network artifact-root projection the shared internal
  authority used by checked artifact discovery and deployment registration.
- Makes root release-set manifest path projection pure and infallible.
- Leaves required parent-directory creation to the existing durable write
  authority at the point where bytes are actually published.

### 🗑️ Removed

- Removes deployment-registration fallback branches that recomputed the exact
  same artifact and manifest paths after ignored errors.
- Removes filesystem mutation from release-set path calculation.

### ⚠️ Breaking

- The public Rust `root_release_set_manifest_path` helper now returns `PathBuf`
  directly instead of `Result<PathBuf, Box<dyn Error>>`. Callers should not use
  path projection as an artifact-existence or directory-writability check.

### 🧭 Migration Notes

- Remove `?`, `unwrap`, or error matching around
  `root_release_set_manifest_path`. Use `resolve_artifact_root` when existing
  selected-network artifacts are required, and use the durable write API when
  publishing the manifest.
- No CLI, JSON, Candid, configuration, package-version, or stable-state shape
  changes are introduced.

### 🧪 Testing

- Focused release-set tests prove selected-network discovery still fails
  closed and path projection creates no directories.
- Focused registration and root-verification tests preserve existing explicit
  registration, validation, and state-transition behavior.

## [0.93.22] - 2026-07-18 - Replay Receipt Identity Binding

### 📝 Summary

`0.93.22` closes a stale-continuation race in shared replay receipts. A receipt
token previously addressed only the stable storage slot derived from command
kind and operation ID. If a reserved receipt expired while an asynchronous
pre-effect step was still running, another request could purge and reuse that
slot. The older continuation could then mark, stage, settle, commit, or abort
the newer request's receipt.

Every token-backed replay operation now verifies the complete immutable
receipt identity before reading or mutating the persisted record. The expiry
collector also removes ordinary receipts at the same exact boundary where
replay admission and quota accounting already consider them expired.

### 🔧 Changed

- Binds replay tokens to schema version, command kind, operation ID, actor,
  payload-hash version and value, creation time, and expiry time.
- Routes reserved and uncommitted aborts through the same identity-checked
  receipt authority used by marking, staging, settlement, and commit.
- Revalidates replay ownership after asynchronous ICP-refill context lookup and
  delegated-token lazy repair, before either flow performs its next local
  operation mutation or proof preparation.
- Uses one auth response-commit path and one auth receipt-error projection for
  delegated-token and role-attestation preparation.
- Treats `expires_at_ns == now_ns` as collectible rather than retaining an
  already-expired receipt for one additional nanosecond.

### 🩹 Fixed

- A stale asynchronous continuation can no longer mutate or delete a newer
  receipt that reused the same command and operation slot.
- Actor or payload replacement cannot be overwritten by an earlier token even
  when both requests resolve to the same stable key.
- A stale pre-effect continuation can no longer create an ICP-refill operation
  or prepare delegated-token material after its replay reservation was replaced.
- Global replay capacity no longer counts ordinary receipts at the exact
  boundary where they have already stopped counting as active or pending.

### 🧭 Migration Notes

- No Candid, public DTO, configuration, package-version, or stable-record shape
  changes are introduced. Invalid stale-token transitions now fail closed with
  a typed replay-store cause.

### 🧪 Testing

- Focused replay tests cover every token-backed mutation, settlement lookup,
  terminal promotion, and abort against a reused slot and prove the replacement
  record is unchanged.
- Delegated-token lazy-repair tests prove replay ownership is revalidated after
  repair and before a second local preparation attempt.
- Stable replay-store tests cover active, pending, and collection behavior at
  the exact expiry boundary.

## [0.93.20] - 2026-07-18 - Serialized Child Funding

### 📝 Summary

`0.93.20` closes a concurrency gap in child cycles funding. Funding temporarily
reserves the child budget before awaiting `deposit_cycles`, then restores the
prior ledger snapshot if the external call fails. A second operation admitted
after the cooldown elapsed could previously update that same ledger while the
first call was still unresolved, allowing the older failure to erase newer
budget and cooldown accounting.

The existing replay receipt store is now the sole durable in-flight authority.
Authorization excludes the current operation ID and rejects any other pending
`RequestCycles` receipt for the same child and command.

The transfer path also settles its cost guard before returning if it cannot
persist the `deposit_cycles` effect marker. This failure is known to occur
before the management call, so the original typed replay error is preserved
while quota accounting is committed and the unused cycle reservation is
released.

### 🔧 Changed

- Serializes pending cycles-funding operations per child through their existing
  replay actor, command kind, and operation identity.
- Treats reserved, external-effect-in-flight, and recovery-required receipts as
  active funding ownership while allowing expired pre-effect reservations to
  stop blocking new work.
- Adds a bounded `operation_in_progress` cycles-funding denial metric and a
  public `Conflict` result for competing requests.
- Gives the request-cycles effect marker the same pre-effect cost-recovery
  contract as the other root value-transfer and deployment paths.

### 🩹 Fixed

- A failed older funding call can no longer restore a stale whole-ledger
  snapshot over a newer reservation or successful grant.
- Concurrent funding cannot bypass per-child cumulative limits or leave the
  cooldown timestamp inconsistent with the cycles actually transferred.
- A missing or corrupt replay receipt at the effect-marking boundary can no
  longer strand pending quota and cycle-reservation intents before
  `deposit_cycles` is called.

### ⚠️ Breaking

- A child with another pending `RequestCycles` operation now receives
  `Conflict` instead of starting a concurrent transfer. The original operation
  ID remains the recovery authority.

### 🧭 Migration Notes

- No public DTO, Candid, configuration, or stable-memory schema changes are
  introduced. Existing replay receipts provide the serialization boundary.

### 🧪 Testing

- Focused replay-store tests cover exact current-operation exclusion, command
  isolation, competing operation detection, and expiry.
- Focused funding tests cover valid positive-cooldown configuration, typed
  conflict projection, unchanged ledger state, and denial diagnostics.
- Effect-marker failure coverage proves typed-cause preservation and complete
  settlement of both pending cost intents without executing the transfer.

## [0.93.19] - 2026-07-18 - Bounded Recovery Integrity

### 📝 Summary

`0.93.19` makes pool reset recovery fair and makes sharding assignment
accounting fail closed. The pool worker now walks one cursor-bounded ordered
sweep rather than repeatedly selecting the same oldest pending rows. Sharding
assignment and release operations validate every referenced record and counter
transition before changing stable state.

### 🔧 Changed

- Replaces the duplicate oldest-pending pool selector with one internal
  read-only page contract ordered by creation time and principal.
- Carries only an in-memory cursor between immediate pool reset batches; after
  one complete sweep, still-blocked entries wait for the configured periodic
  retry unless new work explicitly requests another run.
- Uses checked sharding assignment counter transitions and validates both sides
  of reassignment before committing the mapping or either derived count.

### 🩹 Fixed

- Ten locally non-importable or otherwise blocked oldest pool rows can no
  longer starve every later pending reset or hold the canister in a zero-delay
  timer loop.
- Reassignment no longer hides a missing old shard or a zero old count, and a
  saturated target count no longer silently remains unchanged while accepting
  another mapping.
- Release no longer removes a partition assignment before discovering that
  its referenced shard is missing or its derived count cannot be decremented.

### ⚠️ Breaking

- Corrupt sharding assignment state that was previously normalized or partly
  mutated now returns a typed ops failure and preserves the original evidence.
- The internal non-paginated pending-reset selector is removed; the cursor page
  is the sole pending pool scan contract.

### 🧭 Migration Notes

- No public DTO, Candid, configuration, or stable-memory schema changes are
  introduced. The pool cursor is runtime-only and restarts from the oldest
  pending record after init or upgrade.

### 🧪 Testing

- Focused pool tests cover stable pagination across equal timestamps, bounded
  page edges, non-destructive selection, and the no-spin scheduling decision.
- Focused sharding tests cover successful assignment/release plus missing
  references, decrement underflow, increment overflow, and mutation-free
  rejection.

## [0.93.18] - 2026-07-18 - Interruption-Safe Intent Recovery

### 📝 Summary

`0.93.18` closes the recovery gaps found after the reserved intent namespace
shipped. Placement finalization no longer waits on root acknowledgement after
durable domain membership changes. Pool reset recovery can find and settle the
sole pending immediate-import intent from its canonical resource identity.
Intent cleanup uses the same checked abort authority as ordinary rollback.

The `0.93.17` cost namespace remains a hard cut. No old cost key is read,
aliased, or migrated; the direct-upgrade precondition for pre-`0.93.17`

### 🔧 Changed

- Makes placement registration and intent settlement synchronous, then hands
  root-receipt release exclusively to the durable bounded acknowledgement

- Persists sharding registry and active-set membership before placement intent
  settlement, leaving no acknowledgement await between those transitions.
- Lets pool reset recovery resolve the unique pending immediate-import intent
  by its canonical `canic:pool_import:` resource key.
- Keeps pending pool state retryable when recovered intent settlement fails and
  preserves the original typed reset failure when abort also fails.
- Routes expiry cleanup through the checked intent abort operation instead of

### 🩹 Fixed

- An interrupted shard allocation can no longer leave a registered shard

- An interrupted immediate pool import no longer leaves a non-expiring pending

### 🧭 Migration Notes

- Direct upgrades from `0.93.16` or earlier must also satisfy the placement
  cleanup requirement documented for `0.93.17`; no old placement identity is

- No public DTO, Candid, configuration, or stable-memory schema changes are

### 🧪 Testing

## [0.93.17] - 2026-07-18 - Reserved Intent Authority

### 📝 Summary

`0.93.17` separates application-owned intent resources from Canic runtime
authority. Every Canic-owned local or receipt-backed intent now uses the
reserved `canic:` resource namespace. Consumer APIs reject that namespace on
begin and reject load, settlement, commit, or rollback when an existing intent
belongs to Canic.

Placement acknowledgement no longer classifies queue records from an ordinary
string prefix. Only the canonical `canic:placement:<64 lowercase hex>` resource
shape is eligible for root-receipt acknowledgement and terminal cleanup.
Cost-guard and pool-import intents use the same reserved authority boundary.

### 🔧 Changed

- Reserves resource keys beginning with `canic:` for Canic runtime-owned
  intents; application keys remain unrestricted outside that prefix.
- Hard-cuts placement resources from `placement:<digest>` to
  `canic:placement:<digest>`.
- Moves cost quota and cycle-reservation resources under `canic:cost:` and
  pool-import resources under `canic:pool_import:`.
- Gives consumer and Canic-owned receipt settlement separate internal workflow
  entry points while retaining one storage implementation and metric path.
- Adds exact local-intent lookup at the ops boundary so public commit and
  rollback validate resource ownership before mutation.

### 🩹 Fixed

- An application terminal receipt-backed intent whose resource key began with
  `placement:` can no longer be mistaken for Canic's acknowledgement queue and
  deleted by the placement timer.
- Application reservations can no longer collide with internal placement,
  cost-guard, or pool-import capacity aggregates through a shared resource key.
- Consumer receipt APIs can no longer load or settle a Canic-owned operation,
  and consumer local-intent APIs can no longer commit or roll back a
  Canic-owned numeric intent ID.

### ⚠️ Breaking

- `LocalIntentApi::begin` and `ReceiptBackedIntentApi::begin_or_load` now return

- Binds remote prepare, begin, complete, and finalize results to the same
  retired binding and publication generation observed before the inter-canister
  call. A concurrent ownership transition is returned as a typed conflict
  instead of committing against newer state.
- Serializes root-owned GC/finalize/delete orchestration with a workflow-local
  in-flight guard and treats unexpected persistence outcomes as errors rather
  than discarding them.
- Makes the GC lifecycle one-way for publication authority. Explicit binding,
  automatic placement, release reconciliation, and direct-target publication
  all reject stores whose GC mode is not `Normal`.
- Enforces the same write boundary inside the store canister, so chunk-set
  preparation, chunk publication, and manifest staging fail closed after GC
  starts even if a root-side snapshot becomes stale.
- Requires finalized-store deletion to match the authoritative binding,
  canister principal, unreferenced publication state, and completed local GC
  record both before and after the remote status call.
- Makes restore operation-receipt `outcome` and `attempt` mandatory JSON fields
  and rejects attempt zero with a typed journal-validation cause.

### ⚠️ Breaking

- Restore apply journals containing operation receipts without `outcome` or
  `attempt` no longer deserialize. Receipts using attempt zero no longer pass
  validation.
- Publication requests targeting a store in `Prepared`, `InProgress`,
  `Clearing`, or `Complete` GC state now return a conflict instead of allowing
  the store to regain or retain publication authority.
- No compatibility alias, inferred receipt default, legacy decoder, or
  alternate publication path is retained.

### 🧭 Migration Notes

- Regenerate malformed restore journals or provide the explicit durable
  receipt fields: `outcome` and an `attempt` value starting at one.
- Complete retirement and deletion for a store that has entered GC; publish to
  a different `Normal` store rather than trying to reactivate it.

### 📊 Audit

- The focused source-and-test delta spans 15 files, with 639 insertions and 53
  deletions before release-note finalization.
- The Wasm-store fix covers explicit selection, automatic placement,
  reconciliation, direct publication, store-local mutations, asynchronous GC
  commits, finalization, and destructive deletion.
- The restore fix removes both permissive defaults and proves decode-time and
  validation-time rejection boundaries.

### 🧪 Testing

- Twenty-one focused control-plane tests cover publication selection,
  generation drift, in-flight lifecycle rejection, exact deletion identity,
  typed error projection, and the store-local write gate.
- Thirty-four restore apply-journal tests cover positive receipt behavior,
  missing-field rejection, zero-attempt rejection, duplicates, transition
  ordering, command previews, and journal invariants.
- Targeted package checks, strict all-target Clippy for the control-plane and

## [0.93.0] - 2026-07-16 - Pre-1.0 Surface Hard Cuts

### 📝 Summary

0.93.0 removes stale and unreachable surfaces found by a repository-wide
read-only audit. The hard cuts cover an obsolete issuer-renewal authority,
replay state with no production writer, unused compatibility APIs, and
serialized variants that current producers cannot emit.

Maintained behavior is now backed by one production authority per concern.
The persisted chain-key batch owns issuer-renewal progress, replay keeps only
live reservation and recovery states, and template publication removes stale
manifests instead of retaining deprecated tombstones.

This release intentionally adds no aliases, fallback decoding, compatibility
records, or anti-resurrection paths for the removed pre-1.0 contracts.

### 🔧 Changed

- Root issuer-renewal status now projects `latest_batch` from the canonical
  persisted chain-key batch instead of projecting a separate
  `active_attempt` record.
- CLI text and JSON report batch status, proof epoch, expiry, installation,
  retry, and failure evidence. A reachable issuer without a usable active
  proof is reported as `proof_unavailable` rather than ready.
- Template publication deletes superseded or unconfigured approved manifests;
  store accounting is updated when those records are removed.
- Chain-key signature verification and delegated root-proof callbacks carry
  only the values used by cryptographic verification. Key identity,
  derivation-path, and certificate-hash checks remain enforced by their
  owning validation stages.

### 🗑️ Removed

- Removes the issuer-renewal attempt model, persisted attempt collection,
  attempt-only metrics, outcome and failure counters, and the old status
  projection. Obsolete attempt data is neither migrated nor exposed.
- Removes replay `TerminalFailed` state and its bounded terminal-error model,
  commit path, classifications, and workflow branches. Production never wrote
  that state.
- Removes the Wasm-store `Deprecated` template-manifest state and the obsolete
  aggregate chunk-set request and storage path.
- Removes backup authority, snapshot-read, quiescence, and proof-source
  variants that no current plan or preflight producer emits, plus the
  unproduced `InstalledModuleHash` artifact-digest source.
- Removes uncalled root-capability, endpoint-performance, lifecycle,
  topology-query, RPC, pool, ledger, CLI, and private compatibility surfaces
  identified by the audit.

### 📊 Audit

- The release delta from `v0.92.14` spans 81 files, with 776 insertions and
  1,739 deletions after release-note finalization.
- Every removed serialized variant was traced through constructors,
  validation, rendering, persistence, and tests before deletion.
- Intentional feature-conditioned stable schemas, canonical auth snapshots,
  delegated-auth metrics, and the required `finish!` marker remain in place.
- Receipt-backed promotion inputs remain supported because operator-provided
  promotion JSON reaches and validates both maintained receipt kinds.

### ⚠️ Breaking

- `RootIssuerRenewalStatusResponse` replaces `active_attempt` with
  `latest_batch` and removes attempt-era state fields. CLI JSON consumers must
  update to the current batch projection.
- `TemplateManifestState::Deprecated` is no longer accepted by the Wasm-store
  Candid or stable-record contract. Persisted stores must not contain that
  state before upgrade; 0.93.0 intentionally provides no compatibility
  decoder or migration record.
- Backup JSON using removed authority, quiescence, proof-source, or artifact
  digest variants must be regenerated with the current CLI.
- Manually injected replay receipts using `TerminalFailed` are unsupported;
  live production replay states retain their existing stable encoding.

### 🧭 Migration Notes

- Update automation that parses issuer-renewal status to consume
  `latest_batch` and its batch fields instead of `active_attempt`,
  `last_outcome`, `consecutive_failures`, or `active_attempt_id`.
- Regenerate persisted backup plans or receipts that were authored with
  unsupported enum variants.
- Verify Wasm-store manifest state before upgrading. Superseded and
  unconfigured approved manifests are removed by current publication flows
  rather than converted to deprecated tombstones.
- No application configuration or source migration is required when these
  removed serialized states and response fields are not consumed.

### 🧪 Testing

- Pins the exact stable bytes of a live replay recovery receipt, proving that
  removal of the unwritten terminal variant does not alter maintained replay
  encoding.
- Current-behavior coverage exercises issuer-renewal batch projection,
  delegated proof verification, template replacement and pruning, backup plan
  preflights, CLI rendering, and host adoption.
- Strict package-scoped linting, formatting, Candid contract checks,
  changelog governance, and diff hygiene pass for the affected surfaces.
