# Canic 0.93 Changelog

## [0.93.12] - 2026-07-17 - Durable Replay Recovery and Complete Install Failure Evidence

### 📝 Summary

`0.93.12` closes the recovery gap between a successful external effect and its
terminal accounting or replay commit. The replay receipt is now the canonical
durable authority for that boundary: it records the cost-intent identity and a
staged terminal response before settlement, then lets an identical retry finish
accounting and commit the response without repeating the external operation.

Unresolved post-effect receipts are no longer expired or purged by request TTL.
Root-install phase failures also retain complete typed evidence when both the
operation and the attempt to persist its failure receipt fail. Public successful
Candid, JSON, and CLI response shapes are unchanged.

### 🔧 Changed

- Extended replay receipts with optional staged response data and durable cost
  guard settlement identity. Existing receipts whose fields are absent retain
  their exact stable encoding.
- Made terminal cost settlement reconstructible and idempotent from persisted
  intent identities.
- Added accounting-only recovery for ICP refill, pool creation, governed root
  provisioning, and governed root upgrade.
- Staged governed root responses before terminal settlement, and made root
  upgrade persist its installed-module projection before committing replay.
- Kept `ExternalEffectInFlight` and `RecoveryRequired` receipts active across
  expiry and cleanup; only pre-effect reservations and completed receipts keep
  request-TTL behavior.
- Added typed recovery reasons for failed cost settlement and failed state
  projection so diagnostics preserve the failed boundary.
- Added `InstallPhaseFailureError`, whose accessors expose both the typed phase
  operation error and the typed failure-receipt write error.

### 🩹 Fixed

- An external effect followed by failed terminal cost settlement can no longer
  lose its cost intent and remain permanently unrecoverable.
- Request cleanup can no longer remove evidence for an unresolved external
  effect and permit a later duplicate operation.
- Root-install failure handling no longer relegates a simultaneous
  failure-receipt write error to log-only evidence.

### ⚠️ Breaking

- Identical retries for recoverable cost-settlement failures now finish the
  pending accounting and return the staged response. They do not re-run the
  external effect or return the former generic in-flight outcome.
- Post-effect and recovery-required replay receipts intentionally ignore request
  TTL until they reach a terminal state. Operators must resolve them rather than
  relying on cleanup to discard them.
- When a root-install phase operation and its failure-receipt write both fail,
  the returned source is now `InstallPhaseFailureError`; callers that downcast
  errors should inspect its two typed causes.
- No aliases, compatibility commands, fallback replay paths, or deprecated
  surfaces were added.

### 🧭 Migration Notes

- After a terminal settlement failure, retry with the exact same request ID,
  caller, and payload. Do not create a replacement request or repeat the
  external operation manually.
- `ExternalEffectStatusUnknown` and `StateProjectionFailed` remain fail-closed
  manual-recovery conditions; only a recorded cost-settlement failure is
  completed automatically.
- Consumers that inspect root-install failures should use
  `InstallPhaseFailureError::operation_error()` and
  `InstallPhaseFailureError::receipt_write_error()` when the composite error is
  returned.
- The stable replay schema change uses optional fields and requires no operator
  migration command.

### 📊 Audit

- Traced representative ICP refill, pool-create, root-provision, and root-upgrade
  operations across admission, cost reservation, external effect, replay
  persistence, settlement, recovery, diagnostics, and response projection.
- Verified that recovery authority remains in the replay store and cost-intent
  store rather than introducing a second orchestration path.
- Verified exact stable receipt bytes remain unchanged when the new optional
  recovery fields are absent.
- Kept the change scoped to replay/accounting recovery and root-install failure
  evidence; no public success DTO or unrelated deployment behavior changed.

### 🧪 Testing

- Targeted replay tests cover recoverable settlement failures, accounting-only
  retries, expiry boundaries, cleanup retention, and stable-state round trips.
- Targeted ICP, pool, provision, and upgrade tests cover pre-effect rollback,
  post-effect recovery marking, idempotent completion, and non-reexecution.
- Targeted host tests cover ordinary typed install failures and the composite
  operation-plus-receipt-write failure.
- `canic-core` and `canic-host` pass targeted all-target Clippy with warnings
  denied; formatting, layering guards, changelog governance, and diff hygiene
  also pass.

## [0.93.11] - 2026-07-17 - Fail-Closed Cost Settlement and Typed Dispatch Cleanup

### 📝 Summary

0.93.11 closes the next three findings from the post-0.93.10 code review.
Cost-guard reservation rollback now reports cleanup failure explicitly, and
ICP-refill plus empty-pool creation settle durable quota/cycle accounting
before committing replay success. A settlement failure therefore returns to
the caller while the existing in-flight replay receipt continues to prevent a
duplicate external effect.

The retired Wasm-store protected-internal-proof model is removed completely.
The maintained client uses canonical method names and protocol manifests
directly, while endpoint-owned root-principal authorization remains unchanged.

Backup-manifest, fleet-create, fleet-config, and config-list dispatch preserve
their concrete child-command and fleet-configuration errors rather than
converting them to display strings. Public Candid, JSON schemas, stable-state
records, successful CLI arguments, and successful CLI reports are unchanged.

### 🔧 Changed

- Adds a typed cost-guard reservation rollback failure containing both the
  original reservation error and the quota-cleanup error. Cleanup failure is
  internal and cannot be projected as an ordinary quota rejection.
- Makes resumable ICP-refill cleanup remove its replay receipt only after cost
  recovery succeeds.
- Makes terminal ICP-refill and empty-pool creation complete cost settlement
  before storing a committed replay response.
- Leaves the authoritative replay receipt in-flight with no cached response
  when settlement fails, so the external operation cannot be repeated under
  the same identity.
- Makes backup and fleet dispatch wrap `ManifestCommandError`,
  `ScaffoldCommandError`, and `ListCommandError` directly.
- Makes config-list projection retain `FleetConfigError` and removes the
  generic display-based config error mapper.
- Keeps the canonical Wasm-store root-update and structural-query method
  manifests as the sole client/protocol classification surface.

### 🗑️ Removed

- Removes `canic_wasm_store_method_requires_internal_proof` from
  `canic-core` and the public `canic::protocol` facade.
- Removes the duplicate Wasm-store endpoint wrapper, identical root-update and
  structural-query constructors, always-false proof classification, debug
  assertion, and stale proof-absence tests.
- Removes log-only cost settlement and recovery branches from ICP-refill and
  empty-pool creation.
- Removes string-backed manifest, fleet-create, fleet-config, and config-list
  error projections.
- No alias, compatibility wrapper, deprecated API, duplicate path, fallback,
  or removed-behavior regression test is retained.

### ⚠️ Breaking

- The public `canic_wasm_store_method_requires_internal_proof` helper is
  removed. It represented a deleted protocol and always returned `false`.
- `CostGuardReserveError` adds the fail-closed `ReservationRollback` variant.
  Exhaustive Rust matches must handle it.
- `BackupCommandError::Manifest`, `FleetCommandError::Create`, and
  `FleetCommandError::Config` now contain their concrete child error instead
  of `String`; `ListCommandError::Config(String)` is replaced by
  `ListCommandError::FleetConfig(FleetConfigError)`.
- ICP-refill and empty-pool operations now return an error rather than success
  when durable cost settlement fails after the external effect. Their replay
  identity remains non-replayable until recovery.

### 🧭 Migration Notes

- Delete calls to the removed proof classifier. Use the canonical Wasm-store
  method manifests only when method inventory is required; authorization
  remains enforced by the endpoint caller guard.
- Match or traverse the nested CLI error variants instead of parsing display
  text. Successful command consumers require no migration.
- Propagate cost settlement failures and treat the retained in-flight replay
  receipt as recovery authority. Do not cache success or repeat the external
  effect when settlement did not complete.

### 📊 Audit

- The focused source-and-test delta spans 19 files, with 320 insertions and
  182 deletions before changelog and status finalization.
- Production scans find no remaining internal-proof classifier or duplicate
  Wasm-store endpoint model, and the affected CLI dispatchers contain no
  child-error `to_string()` conversion.
- ICP-refill and empty-pool replay response commits now have one visible
  ordering: durable cost settlement first, committed replay response second.

### 🧪 Testing

- Seventeen focused cost-guard tests plus three boundary-guard tests cover
  reservation rollback, rollback failure, completion/recovery rejection,
  quota behavior, and ICP-refill replay preservation.
- The focused empty-pool negative case proves settlement failure leaves the
  replay receipt in-flight without response bytes.
- The maintained control-plane endpoint-manifest test and public protocol
  manifest test pass after the proof-model hard cut.
- Twenty focused CLI tests cover nested manifest, scaffold, list, fleet-config,
  and adjacent typed-cause preservation.
- Targeted checks and strict all-target Clippy pass for `canic-core`,
  `canic-control-plane`, `canic`, and `canic-cli`. The layering guard,
  changelog governance, formatting, and diff hygiene also pass.

## [0.93.10] - 2026-07-17 - Exact Artifact Authority and Fail-Closed Replay Finalization

### 📝 Summary

0.93.10 removes the remaining ambiguity in install artifact selection and
replay receipt finalization. Deployment-plan artifacts resolve only from the
selected network, while the maintained direct build/bootstrap path uses its
one canonical local artifact root. A missing selected-network artifact set no
longer falls back to unrelated local output.

Replay commit, external-effect, recovery, and abort transitions now require an
authoritative persisted receipt. Missing or corrupt records fail without
mutation or reconstruction from a stale reservation token, and response
commit failures propagate to the caller after recovery state is recorded.

Root install exposes a typed phase/cause boundary. CLI commands retain the
host-owned installed-deployment error rather than translating it through
parallel command-specific variants. Public Candid, JSON schemas, stable state,
successful command arguments, and successful command reports are unchanged.

### 🔧 Changed

- Makes selected-network release artifact lookup return a typed missing-root
  error and require the exact network directory.
- Keeps ordinary direct root builds and generated root bootstrap data on the
  canonical local artifact root, independent of the deployment environment
  used to compile IC behavior into Wasm.
- Makes replay receipt commit, external-effect marking, recovery marking, and
  abort operations return typed store failures.
- Distinguishes missing receipts from decode failures and preserves both until
  the owning workflow projects its error.
- Makes root RPC, non-root cycles, ICP-refill, pool creation, delegated-token,
  and role-attestation workflows stop when replay state cannot be finalized.
- Adds a stable root-install phase classification while retaining the original
  source error chain for diagnostics and downcasting.
- Uses the host installed-deployment error as the sole target-resolution cause
  across inspect, auth, blob-storage, cycles, metrics, token, list, snapshot,
  backup, and environment-export commands. Command-specific exit codes and
  structured blob-storage codes remain presentation policy.

### 🗑️ Removed

- Removes selected-network-to-local artifact fallback from release-set and
  deployment-plan resolution.
- Removes stale-token replay reconstruction when the persisted reservation is
  missing and lossy decode handling during terminal transitions.
- Removes command-local installed-deployment mapping helpers and their
  duplicate missing-state, lost-root, registry, I/O, and replica-query error
  variants where the canonical host error now owns the cause.
- Normalizes the evidence command to the required directory-module layout.
- No alias, compatibility wrapper, deprecated API, duplicate path, or legacy
  fallback is retained.

### ⚠️ Breaking

  selected-network artifact directory even when local artifacts exist.
- `install_root` now returns `InstallRootError`; callers can inspect its
  `InstallRootPhase` and traverse the original source chain.
- Replay terminal-transition helpers now return `Result` and callers must
  handle `ReplayReceiptStoreError::ReceiptMissing` and
  `ReplayReceiptStoreError::ReceiptDecodeFailed`.
- Public CLI command-error enums now wrap `InstalledDeploymentError` instead
  of exposing parallel command-specific variants. Exhaustive Rust matches must
  be updated, and target-resolution error text is now the canonical host text.

### 🧭 Migration Notes

- Materialize deployment-plan artifacts below the exact selected network
  before installation; do not rely on `.icp/local` as a fallback for another
  network.
- Propagate replay terminal-transition failures. Do not ignore them or rebuild
  a missing receipt from an earlier token.
- Match `InstallRootError::phase()` for install-stage handling and inspect its
  source chain for the concrete failure.
- Replace command-local installed-deployment error matches with the nested
  `InstalledDeploymentError` variants. Successful CLI and serialized-data
  consumers require no migration.

### 📊 Audit

- The focused source, test, documentation, and module-layout delta spans 51
  files, with 811 insertions and 685 deletions before release-note
  finalization.
- Production scans find no installed-deployment mapping helper, selected-root

- Artifact selection, replay receipt lifecycle, root-install error ownership,

### 🧪 Testing

## [0.93.9] - 2026-07-17 - Fail-Closed Workspace and Canister Manifest Discovery

### 📝 Summary

0.93.9 makes Cargo workspace, ICP project, and canister-manifest discovery
typed and fail closed. Filesystem inspection, Cargo manifest parsing, path
canonicalization, Cargo metadata, missing role, and ambiguous role failures
now retain their concrete cause through host and CLI boundaries.

Current-context commands resolve the Cargo workspace from the current
directory. An explicit config path no longer selects another workspace or
falls back to the workspace containing the caller when discovery fails.
Split-root builds remain explicit through `--workspace` and `--icp-root`.

Successful command arguments and reports, fleet configuration syntax, public
Candid, JSON schemas, and stable state are unchanged.

### 🔧 Changed

- Adds typed workspace-discovery failures for current-directory resolution,
  path inspection, invalid project markers, Cargo manifest reads and parsing,
  and canonicalization.
- Canonicalizes the discovery starting point before walking its ancestors.
  File symlinks remain valid project markers, while dangling links and
  non-file markers are rejected instead of being treated as absent.
- Makes canister-role manifest resolution require canonical workspace and
  canister roots and return deterministic typed missing or ambiguous-role
  errors. Ambiguous errors retain the sorted manifest paths as structured
  data.
- Makes the Cargo metadata cache reject an unresolvable workspace root rather
  than caching or querying through an uncanonicalized path.
- Preserves discovery causes through ICP configuration, install state, build,
  deploy, and deployment-truth error boundaries.

### 🗑️ Removed

- Removes the public `workspace_root_from` helper and its
  explicit-path-to-current-workspace fallback.
- Removes optional-return discovery that conflated missing roots with
  inspection, parsing, and canonicalization failures.
- Removes string-backed canister-manifest resolution failures and the fallback
  that continued below an uncanonicalized canister root.
- No alias, compatibility wrapper, deprecated helper, duplicate discovery
  path, or legacy fallback is retained.

### ⚠️ Breaking

- `discover_workspace_root_from` and `discover_icp_root_from` now return
  `Result<Option<PathBuf>, WorkspaceDiscoveryError>` instead of `Option<PathBuf>`.
- `workspace_root` and `icp_root` now return `WorkspaceDiscoveryError`, while
  `root_manifest_path` and `canister_manifest_path` return
  `CanisterManifestError` instead of boxed or string errors. Exhaustive error
  matches must handle the new typed variants.
- `workspace_root_from` is removed. Builds without `--workspace` resolve from
  the current directory rather than inferring a workspace from `--config`.
- Malformed or unreadable Cargo manifests and invalid project-file markers now
  terminate discovery instead of being skipped while searching an ancestor.

### 🧭 Migration Notes

- Use `workspace_root()` for the current Cargo context. Pass `--workspace`
  explicitly when the selected workspace is not the one containing the
  current directory.
- Propagate and inspect `WorkspaceDiscoveryError` and
  `CanisterManifestError`; do not convert discovery failures to absence or
  reintroduce current-directory fallback logic.
- Ensure each canister role is declared by exactly one Cargo package below the
  selected canister root. Successful CLI and serialized-data consumers require
  no migration.

### 📊 Audit

- The focused source-and-test delta spans 17 files, with 496 insertions and
  104 deletions before release-note finalization.
- Production scans find no remaining reference to the removed
  `workspace_root_from` surface and no lossy canonicalization fallback in
  workspace, ICP, or canister-manifest discovery.
- Workspace selection now has one current-context authority; alternate roots
  require the maintained explicit command options.

### 🧪 Testing

- Forty-five focused host and CLI tests cover valid and invalid discovery,
  file and symlink boundaries, malformed Cargo manifests, missing and
  ambiguous role metadata, and typed cause propagation through build, deploy,
  and ICP configuration.
- Package checks and strict all-target Clippy pass for `canic-host` and
  `canic-cli`; formatting and diff hygiene also pass.

## [0.93.7] - 2026-07-17 - Fail-Closed Local Query Authority

0.93.7 gives local cycle-balance and root bootstrap-status queries one
transport authority. When the selected network is local, Canic now returns the
direct replica query result or its typed failure. It no longer discards that
failure and retries the same observation through ICP CLI.

Non-local targets continue to use the pinned ICP CLI response contract.
Successful cycle and bootstrap values, CLI report schemas, public Candid,

  observations, including callers without an explicit ICP project root.

- Derives the cycle-balance error display, source, and conversions from its
  typed variants instead of maintaining equivalent manual implementations.

### 🗑️ Removed

- Removes the success-only local cycle probe and its implicit ICP CLI retry.
- Removes the optional local bootstrap-status adapter that erased every direct
  query failure before invoking the second transport.
- No transport alias, retry shim, compatibility error, or fallback query path
  is retained for these authoritative observations.

### ⚠️ Breaking

- Local direct-replica failures now remain failures instead of being retried
  through ICP CLI. Operators may therefore see the concrete local transport,
  Candid, canister, or rejection cause earlier.
- `CycleBalanceQueryError` adds the typed `Replica` variant. Downstream Rust
  callers with exhaustive matches must handle it.

### 🧭 Migration Notes

## [0.93.5] - 2026-07-17 - Canonical Typed Responses and Registry Authority

### 📝 Summary

0.93.5 gives host and CLI canister queries one response-decoding contract. ICP
CLI JSON must contain the top-level hexadecimal `response_bytes` field, which
is decoded directly into the endpoint's canonical Candid DTO. Invalid JSON,
missing bytes, invalid hexadecimal, incompatible Candid, and typed endpoint
rejections remain distinct failures.

Subnet-registry discovery now carries validated typed entries from transport
to consumers instead of converting direct-replica responses into synthetic
JSON and parsing them again. Metadata, readiness, bootstrap diagnostics,
cycles, metrics, and blob-storage observations use the same typed boundary.

Public Candid, configuration, stable state, and successful CLI JSON reports
are unchanged. The public `canic-host` Rust parsing and registry-query surface
is intentionally hard-cut as described below.

### 🔧 Changed

- Adds one host-owned decoder for the pinned ICP JSON envelope and both plain
  Candid values and canonical `Result<T, CanicError>` responses.
- Makes readiness, bootstrap status, metadata version, cycle balance, metrics,
  cycle history, blob-storage status, and blob funding decode their declared
  endpoint DTOs before projection.
- Makes the canonical subnet-registry DTO the sole registry wire authority.
  Redundant principal or role disagreement now fails with a typed cause rather
  than being silently repaired or dropped.
- Carries typed registry entries through installed-deployment, backup,

## [0.93.2] - 2026-07-16 - Fail-Closed Lifecycle and Restore Receipts

### 📝 Summary

0.93.2 closes the two P1 safety findings from the post-0.93.1 audit. Wasm-store
retirement now has one generation-bound root lifecycle and a one-way
store-local write boundary. Restore apply receipts no longer infer successful
outcomes or attempt identity from omitted JSON fields.

The release does not change the public Candid schema or stable-state layout.
It intentionally hard-cuts permissive restore-receipt decoding rather than
adding a legacy fallback or compatibility representation.

### 🩹 Fixed

- Binds remote prepare, begin, complete, and finalize results to the same
  retired binding and publication generation observed before the inter-canister
  call. A concurrent ownership transition is returned as a typed conflict
  instead of committing against newer state.
- Serializes root-owned GC/finalize/delete orchestration with a workflow-local
  in-flight guard and treats unexpected persistence outcomes as errors rather
  than discarding them.
- Makes the GC lifecycle one-way for publication authority. Explicit binding,
  automatic placement, release reconciliation, and direct-target publication
  all reject stores whose GC mode is not `Normal`.
- Enforces the same write boundary inside the store canister, so chunk-set
  preparation, chunk publication, and manifest staging fail closed after GC
  starts even if a root-side snapshot becomes stale.
- Requires finalized-store deletion to match the authoritative binding,
  canister principal, unreferenced publication state, and completed local GC
  record both before and after the remote status call.
- Makes restore operation-receipt `outcome` and `attempt` mandatory JSON fields
  and rejects attempt zero with a typed journal-validation cause.

### ⚠️ Breaking

- Restore apply journals containing operation receipts without `outcome` or
  `attempt` no longer deserialize. Receipts using attempt zero no longer pass
  validation.
- Publication requests targeting a store in `Prepared`, `InProgress`,
  `Clearing`, or `Complete` GC state now return a conflict instead of allowing
  the store to regain or retain publication authority.
- No compatibility alias, inferred receipt default, legacy decoder, or
  alternate publication path is retained.

### 🧭 Migration Notes

- Regenerate malformed restore journals or provide the explicit durable
  receipt fields: `outcome` and an `attempt` value starting at one.
- Complete retirement and deletion for a store that has entered GC; publish to
  a different `Normal` store rather than trying to reactivate it.

### 📊 Audit

- The focused source-and-test delta spans 15 files, with 639 insertions and 53
  deletions before release-note finalization.
- The Wasm-store fix covers explicit selection, automatic placement,
  reconciliation, direct publication, store-local mutations, asynchronous GC
  commits, finalization, and destructive deletion.
- The restore fix removes both permissive defaults and proves decode-time and
  validation-time rejection boundaries.

### 🧪 Testing

- Thirty-four restore apply-journal tests cover positive receipt behavior,

This release intentionally adds no aliases, fallback decoding, compatibility
records, or anti-resurrection paths for the removed pre-1.0 contracts.

  store accounting is updated when those records are removed.

  attempt-only metrics, outcome and failure counters, and the old status
  projection. Obsolete attempt data is neither migrated nor exposed.

- Removes the Wasm-store `Deprecated` template-manifest state and the obsolete
  aggregate chunk-set request and storage path.
- Removes backup authority, snapshot-read, quiescence, and proof-source
  variants that no current plan or preflight producer emits, plus the
  unproduced `InstalledModuleHash` artifact-digest source.
