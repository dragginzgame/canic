# Canic 0.93 Changelog

## [0.93.17] - 2026-07-18 - Reserved Intent Authority

### 📝 Summary

`0.93.17` separates application-owned intent resources from Canic runtime
authority. Every Canic-owned local or receipt-backed intent now uses the
reserved `canic:` resource namespace. Consumer APIs reject that namespace on
begin and reject load, settlement, commit, or rollback when an existing intent
belongs to Canic.

Placement acknowledgement no longer classifies queue records from an ordinary
string prefix. Only the canonical `canic:placement:<64 lowercase hex>` resource
shape is eligible for root-receipt acknowledgement and terminal cleanup.
Cost-guard and pool-import intents use the same reserved authority boundary.

### 🔧 Changed

- Reserves resource keys beginning with `canic:` for Canic runtime-owned
  intents; application keys remain unrestricted outside that prefix.
- Hard-cuts placement resources from `placement:<digest>` to
  `canic:placement:<digest>`.
- Moves cost quota and cycle-reservation resources under `canic:cost:` and
  pool-import resources under `canic:pool_import:`.
- Gives consumer and Canic-owned receipt settlement separate internal workflow
  entry points while retaining one storage implementation and metric path.
- Adds exact local-intent lookup at the ops boundary so public commit and
  rollback validate resource ownership before mutation.

### 🩹 Fixed

- An application terminal receipt-backed intent whose resource key began with
  `placement:` can no longer be mistaken for Canic's acknowledgement queue and
  deleted by the placement timer.
- Application reservations can no longer collide with internal placement,
  cost-guard, or pool-import capacity aggregates through a shared resource key.
- Consumer receipt APIs can no longer load or settle a Canic-owned operation,
  and consumer local-intent APIs can no longer commit or roll back a
  Canic-owned numeric intent ID.

### ⚠️ Breaking

- `LocalIntentApi::begin` and `ReceiptBackedIntentApi::begin_or_load` now return
  `InvalidInput` for resource keys beginning with `canic:`.
- `ReceiptBackedIntentApi::load` and `settle_if_pending`, plus
  `LocalIntentApi::commit` and `rollback`, return `InvalidInput` when the target
  record is Canic-owned.
- Pre-`0.93.17` placement resource identities are not aliased, migrated, or
  accepted by the new internal placement owner.

### 🧭 Migration Notes

- Rename any application resource key beginning with `canic:` before upgrading
  and complete its pending operation under the old build. There is no legacy
  namespace fallback after the hard cut.
- Complete or explicitly resolve every `0.93.16` in-flight placement operation
  and retained root placement receipt before upgrading. New attempts use the
  reserved placement resource identity.
- Application keys such as `placement:jobs`, `cost:budget`, and
  `pool_import:queue` remain consumer-owned because only `canic:` is reserved.

### 🧪 Testing

- Namespace tests prove consumer begin rejects reserved local and
  receipt-backed resources while ordinary application resources remain valid.
- Ownership tests prove consumer load, settlement, commit, and rollback cannot
  mutate Canic-owned records and leave their state unchanged.
- Placement identity tests prove only the canonical reserved digest shape is
  eligible for acknowledgement cleanup.
- Focused placement, cost-guard, pool-import, intent workflow, and storage tests
  pass with all features enabled.

## [0.93.16] - 2026-07-18 - Receipt-Backed Placement Allocation

### 📝 Summary

`0.93.16` gives scaling, directory, and sharding one child-allocation
authority. A deterministic placement identity now binds local capacity,
the root placement replay request, the returned child, and the final registry
transition. Concurrent or retried allocations therefore recover the same
effect instead of creating through unrelated request IDs.

The existing receipt-backed intent store owns local reservation and durable
uncertainty. The existing root replay store owns the remote provision result.
Placement registries continue to own membership only; no duplicate receipt or
claim store is added.

Committed root placement receipts now survive request expiry until the caller
acknowledges that it has durably registered or disposed the returned child.
The capability envelope carries the exact 32-byte placement operation ID, so
retries and acknowledgement address the same root receipt. Terminal placement
intents form a bounded durable acknowledgement queue that is resumed after init
and upgrade, then removed only after the root receipt is released.
Directory claims without a known provisional child are never released merely
because they are old. A tracked claim resumes its exact root request; an
untracked pre-cut claim remains fail closed rather than guessing and possibly
creating a duplicate child.

### 🔧 Changed

- Adds one internal `PlacementAllocationWorkflow` for deterministic identity,
  receipt-backed admission, root provision replay, registry-backed settlement,
  disposal rollback, and provision-receipt acknowledgement.
- Migrates scaling worker creation, directory instance creation/recovery, and
  shard allocation to the shared workflow as a hard cut.
- Derives scaling generations from committed intent history while pending
  callers reuse the current generation, then translates current registry
  availability into the reservation limit. Retries therefore converge on one
  root operation without preventing later replacement workers.
- Binds directory allocation identity to the durable claim ID and logical key;
  explicit release creates a new claim identity while callback-loss recovery
  retains the original one.
- Binds shard allocation identity and capacity to owner, pool, slot, generation,
  role, and install argument while keeping each logical slot independently
  capacity fenced.
- Makes root placement acknowledgement an authorized response-idempotent
  capability. It does not create a replay receipt or consume replay capacity;
  repeated acknowledgement of an already removed receipt succeeds.
- Hard-cuts capability request metadata from a truncated 16-byte request ID
  plus generated 16-byte nonce to the exact 32-byte durable request ID.
- Uses terminal placement intent records as the durable acknowledgement queue,
  scans it in bounded stable-map pages, restarts the drain after init or
  upgrade, and removes exact terminal records without changing settled totals.
- Gives receipt-backed placement its own unversioned
  `root.allocate_placement_child` command; generic root provision keeps normal
  request expiry and does not require placement acknowledgement.
- Retains committed root placement receipts with recorded create effects past
  TTL without counting them as pending work; acknowledgement validates caller,
  committed state, and effect type before removal.
- Makes directory storage retain every existing pending claim until explicit,
  evidence-backed recovery. Automatic stale-claim overwrite is removed.

### 🩹 Fixed

- Concurrent scaling calls can no longer pass the same pre-await worker count
  and create unrelated extra workers beyond the configured live capacity.
- A directory callback loss can no longer age into release while the created
  child PID remains unknown.
- A tracked directory recovery now replays the exact root provision operation
  and binds its cached child result.
- A stale provisional directory child is settled only after it is repaired into
  membership or disposed and explicitly rolled back.
- Directory repair and cleanup that prove an existing provisional child no
  longer attempt to acknowledge a root receipt that the local recovery path
  did not create; their terminal allocation intent is still removed exactly.
- Replacement workers and shard generations are no longer blocked by cumulative
  committed intent history after earlier members leave their registries.
- Root placement results required for local recovery can no longer disappear at
  the five-minute request TTL before the caller records membership.
- Retrying the same placement operation can no longer derive a different root
  replay key or acknowledge a different key from the stored provision receipt.
- Failed post-settlement acknowledgement can no longer become permanently
  unreachable after the placement intent reaches a terminal state.
- A nonexistent acknowledgement can no longer occupy an emergency replay slot
  or create an acknowledgement receipt of its own.

### ⚠️ Breaking

- The previous scaling, directory, and sharding create paths are removed; no
  alias, fallback, duplicate helper, or compatibility replay identity remains.
- The root provision replay command is hard-cut from `root.provision.v1` to
  unversioned `root.provision`; pre-cut in-flight receipts are not mapped or
  accepted by the new command.
- Shard operations admitted under the `0.93.15` allocation hash are not mapped
  to the new shared allocation identity.
- Directory pending claims are no longer automatically replaced or released
  after `PENDING_TTL_SECS` when no provisional child is known.
- `DirectoryRecoveryResponse` adds `ResumedToBound` for exact tracked recovery.
- Root request DTOs add internal `AllocatePlacementChild` and
  `AcknowledgePlacementReceipt` capability families; the response DTO adds the
  matching acknowledgement result.
- `CapabilityRequestMetadata` now contains one 32-byte `request_id` and no
  `nonce`; callers built against the previous envelope shape must rebuild with
  `0.93.16` rather than relying on an alias or decoder fallback.
- An untracked directory claim with no provisional PID returns a conflict and
  remains installed for operator review; the hard cut does not infer that its
  earlier external effect was absent.

### 🧭 Migration Notes

- Retry tracked directory uncertainty through `recover_entry`; do not delete or
  recreate the key manually while its pending claim remains authoritative.
- Review and resolve any pre-`0.93.16` pending directory row with no provisional
  PID before expecting automated recovery. Canic deliberately cannot associate
  that row with the old random root request.
- Complete or explicitly resolve in-flight root provision operations before
  upgrading; `0.93.16` does not retain the old versioned replay namespace.
- Rebuild every canister that constructs a root capability envelope so it sends
  the full durable operation ID in the new metadata shape. Mixed old/new
  capability envelope traffic is intentionally unsupported.
- Consumers adding another child-placement strategy must provide a deterministic
  operation identity, one capacity resource, a durable membership transition,
  and validated disposal evidence. Value transfers, upgrades, recycling, and
  domain-specific cost/replay owners should not use this adapter.
- No Cargo dependency, configuration, package-feature, or stable-memory schema
  migration is introduced; the new lifecycle uses the existing intent and
  replay allocations.

### 🧪 Testing

- Identity coverage binds placement kind, owner, pool, logical subject,
  generation or claim, role, and install argument while proving the intended
  resource scopes.
- Intent coverage proves exact pending replay, concurrent capacity rejection,
  cumulative-history replacement capacity, committed settlement, disposal
  rollback, and rejection of untracked recovery before root RPC.
- Directory coverage proves unknown outcomes remain claimed, stale claims
  cannot be stolen, tracked repair/cleanup settles deterministically, and
  untracked recovery fails closed.
- Replay coverage proves committed provision results survive expiry, do not
  count as pending, reject wrong callers and invalid states, and are removed
  idempotently by acknowledgement.
- Capability coverage proves transport preserves the full operation ID,
  acknowledgement is response-idempotent with no replay reservation, and
  terminal cleanup is paginated, exact, and aggregate-preserving.
- Focused `canic-core` tests, all-feature package check and Clippy, formatting,
  Candid/DTO boundary guards, and diff hygiene pass for the changed surfaces.

## [0.93.15] - 2026-07-18 - Fail-Closed Verification, Cascades, and Shard Allocation

### 📝 Summary

`0.93.15` closes three failure-boundary gaps found by the continuing 0.93 code
audit. Delegated-token verification now requires the verifier's local subnet
identity instead of silently dropping unavailable environment state. State
cascades still attempt every direct child, but return the first typed child
failure after fanout rather than reporting aggregate success.

Shard allocation now validates all locally decidable registry and assignment
bounds before requesting a canister. Each owner, pool, and slot also derives
one stable root-request identity, making the existing root replay receipt the
sole durable create authority for concurrent calls and retries. Repeated local
admission of the same created shard is idempotent and preserves its counters
and original creation metadata.

### 🔧 Changed

- Makes the verifier subnet principal required runtime context for every
  delegated-token audience; missing environment state records an invalid-state
  metric and returns its typed ops cause.
- Keeps state fanout best effort across all direct children, then returns the
  first typed failure with later failures attached as diagnostic context.
- Records partial root and non-root state fanout as failed rather than
  completed, while retaining the bounded `PartialFailure` metric reason.
- Preserves typed transport and public error projections through state and
  topology child-send diagnostics.
- Validates sharding pool, partition-key, role, capacity, and slot record
  shapes before the root create-canister effect.
- Binds shard creation to a domain-separated operation ID derived from the
  allocating canister, pool, and slot, reusing the canonical root replay
  receipt instead of adding a second reservation store.
- Makes repeated admission of the same shard identity a no-op and rejects a
  conflicting registry identity without overwriting existing state.

### 🩹 Fixed

- Canister- or project-audience delegated tokens can no longer verify while
  required local subnet state is unavailable.
- Lifecycle and state-propagation callers can no longer observe success when
  one or more child state cascades failed.
- An oversized partition key can no longer create a shard before deterministic
  local assignment validation rejects it.
- Concurrent creation plans for one shard slot can no longer use unrelated
  root request IDs and create duplicate canisters.
- A replayed shard admission can no longer reset an existing shard's load
  count or creation timestamp.

### ⚠️ Breaking

- Delegated-token verification now rejects incomplete verifier environment
  state even when the token's audience would not otherwise inspect the subnet.
- Root and non-root state cascades now return a typed error after partial child
  failure; callers that previously treated the logged warning as success must
  handle the failure result.
- Partial state-fanout metrics now use a failed outcome instead of a completed
  outcome with a failure reason.
- Successful Candid, JSON, CLI, configuration, dependency, and stable-state
  contracts are unchanged.

### 🧰 Testing

- Delegated-token context coverage proves a missing local subnet fails closed
  with the original ops classification and no invented public projection.
- Cascade coverage proves aggregate fanout preserves the first error's class,
  origin, and public code while retaining later failures diagnostically.
- Sharding coverage proves assignment bounds reject early, operation IDs are
  stable and resource-bound, caller-owned IDs reach root metadata, repeated
  admission preserves state, and conflicting admission rejects.
- Focused `canic-core` tests with sharding enabled, package check and Clippy,
  formatting, and diff hygiene pass for the changed surfaces.

## [0.93.14] - 2026-07-17 - Failure-Cause and Pool-Recovery Integrity

### 📝 Summary

`0.93.14` makes replay failure handling preserve the operation or policy error
that caused the request to fail. If removing a pre-effect reservation or
recording a post-effect recovery marker also fails, that secondary storage
failure is retained as diagnostic context instead of replacing the primary
typed cause.

The rule now applies consistently across root RPC execution and authorization,
cycles funding, empty-pool creation, ICP refill, and authentication
preparation. Successful public responses and stable replay records are
unchanged.

Pool recovery now keeps each pending reset record authoritative while it
performs asynchronous admission and reset work. A recycled canister therefore
remains visibly in progress, duplicate work remains rejected, and its role,
parent, and module hash survive both successful and failed recovery
transitions.

### 🔧 Changed

- Adds one workflow-owned failure-preservation boundary for replay reservation
  cleanup and recovery-marker persistence.
- Makes root capability and direct `RequestCycles` reservation cleanup retain
  the original authorization or execution failure.
- Makes pool creation, ICP refill, delegated-token preparation, and
  role-attestation preparation retain their primary failure when reservation
  cleanup also fails.
- Makes uncertain external-effect handling for root provisioning,
  `RequestCycles`, pool creation, and ICP refill preserve the call failure when
  the recovery marker cannot be written.
- Keeps secondary replay-store failures available in internal diagnostics
  without changing the primary error class, origin, or public projection.
- Replaces destructive pending-reset selection with a deterministic, bounded,
  non-removing recovery snapshot.
- Makes the existing pending record the sole authority throughout scheduler
  admission and reset, so state transitions update it in place and preserve
  recycle metadata.
- Makes synchronous topology removal during recycle fail closed if the record
  observed immediately beforehand cannot be removed.

### 🩹 Fixed

- A corrupt or missing replay receipt can no longer replace a more useful
  policy, operation, or infrastructure error during failure cleanup.
- A failed recovery-marker write can no longer hide the typed cause of an
  uncertain external effect.
- Background recovery can no longer erase recycled canister metadata by
  deleting the pending record before an asynchronous reset.
- A scheduler reset no longer creates a temporary absent-record window in
  which a duplicate import or recycle workflow can be admitted.

### ⚠️ Breaking

- Failure paths that previously surfaced the secondary replay-store error now
  surface the primary typed error. Callers matching public error codes receive
  the original policy or operation projection instead of cleanup internals.
- The internal destructive pending-reset pop operation is removed as a hard
  cut; recovery integrations select a bounded non-removing snapshot instead.
- Successful Candid, JSON, and CLI contracts, package features, dependencies,
  and stable-state schemas are unchanged.

### 🧭 Migration Notes

- Match the returned primary typed or public error as the request outcome.
  Secondary replay persistence failures remain visible through internal
  diagnostics for recovery and debugging.
- Internal pool recovery code must leave pending records installed until a
  terminal or explicit rejection transition updates or removes them.
- No configuration, stable-state, DTO, or command migration is required.

### 📊 Audit

- Reviewed every production replay-reservation abort that follows an existing
  failure and every uncertain-effect marker path in the affected workflows.
- Kept replay mutation in ops and error-preservation orchestration in workflow;
  no duplicate storage or recovery authority was introduced.
- Traced immediate import, queued import, recycle, scheduler selection,
  admission, reset, pool persistence, and public pool projection end to end.
- Confirmed that the discarded Wasm-store GC rollback result is an
  unreachable secondary failure: its transition is synchronous and follows a
  state written by the same non-reentrant execution segment.

### 🧪 Testing

- Corrupt-receipt tests prove cleanup and recovery-marker failures preserve the
  original public error projection.
- Pool regressions prove pending selection is ordered, bounded, and
  non-destructive and that recycled metadata survives the Ready transition.
- Focused root RPC, cycles, pool, ICP-refill, and auth replay tests pass, along
  with the cost-boundary inventory, all-target `canic-core` Clippy, formatting,
  layering guards, and diff hygiene.

## [0.93.13] - 2026-07-17 - Canonical Staged-Response Recovery

### 📝 Summary

`0.93.13` completes the staged-response recovery model introduced in
`0.93.12`. Every replay-protected completion flow now records its terminal
response before its final replay transition, so an identical retry can finish
accounting or response promotion without repeating an external effect.

`RequestCycles` now persists its cost identity before depositing cycles and
uses the same canonical replay authority as the other governed root flows. The
duplicate cycles-specific evaluator and direct response-commit helpers are
removed as a pre-1.0 hard cut. Public successful Candid, JSON, and CLI shapes
and the stable replay schema are unchanged.

### 🔧 Changed

- Makes `RequestCycles` persist its cost-guard permit identity before the
  cycles deposit, stage its response after the deposit, and recover failed
  settlement through an accounting-only identical retry.
- Routes root provisioning, upgrade, recycling, and `RequestCycles` through
  one staged-response promotion path.
- Applies the same response-staging invariant to ICP refill, empty-pool
  creation, delegated-token preparation, and role-attestation preparation.
- Recovers a staged response from `ResponseCommitFailed` as well as
  `CostSettlementFailed`; a missing staged response remains fail closed and
  transitions deterministically to `ResponseCommitFailed`.
- Settles post-effect cost accounting even when response preparation or
  staging fails, preventing a serialization failure from incorrectly rolling
  back consumed quota or cycles.
- Registers a newly created pool canister as ready before response
  serialization, so the durable projection does not depend on the response
  encoding step.
- Preserves the original typed failure when recording a recovery marker also
  fails, while attaching the marker-write failure as diagnostic context.
- Adds an executable inventory guard for the five costed replay owners: ICP
  refill, empty-pool creation, root provisioning, root upgrade, and
  `RequestCycles`.

### 🗑️ Removed

- Removes the cycles-specific replay evaluator, response decoder, request
  mapping, and metric mapping from `RequestCycles`.
- Removes direct replay-response commit helpers and the separate non-costed
  root external-effect marker path.
- Leaves no alias, shim, fallback, deprecated API, duplicate path, or legacy
  behavior test for the removed internal surfaces.

### ⚠️ Breaking

- An identical `RequestCycles` retry after successful deposit but failed cost
  settlement now completes accounting and returns the staged success without
  depositing cycles again.
- A replay receipt with a staged response and `ResponseCommitFailed` now
  promotes that response instead of remaining a permanent conflict.
- Internal code that previously committed a terminal response directly must
  stage the response and use the canonical promotion transition.
- Successful public response contracts are unchanged; the potentially
  breaking impact is limited to recovery behavior and removed internal Rust
  surfaces.

### 🧭 Migration Notes

- Retry a failed post-effect operation with the exact same request ID, caller,
  and payload. The authoritative replay receipt decides whether accounting or
  response promotion remains; callers must not repeat the external operation
  manually.
- A recovery receipt without a staged response remains a fail-closed operator
  condition. The deterministic `ResponseCommitFailed` reason identifies the
  missing boundary evidence.
- Internal replay integrations must stage terminal response bytes before
  settlement or commit. External consumers require no DTO, configuration,
  stable-state, or CLI migration.

### 📊 Audit

- Inventoried every production cost-guard boundary and verified exact reserve,
  complete, pre-effect recovery, and post-effect recovery ownership for all
  five costed flows.
- Traced the non-costed root-recycle and authentication-preparation flows to
  ensure the response-staging invariant also covers replay completion outside
  cost accounting.
- Verified that replay receipts remain the sole recovery authority and that no
  second direct-commit or cycles-specific decision path remains.
- Kept the change scoped to replay recovery, cost settlement, and their focused
  regression coverage; dependencies, package versions, public response shapes,
  and stable records did not change.

### 🧪 Testing

- Focused `RequestCycles` tests cover persisted cost identity, settlement
  failure, accounting-only retry, response promotion, and non-reexecution.
- Focused replay tests cover staged recovery for root recycling, ICP refill,
  pool creation, delegated tokens, and role attestations, including the missing
  staged-response boundary.
- Receipt tests cover staging and promotion transitions, while the inventory
  guard enforces the exact production cost-boundary owner set and call counts.
- `canic-core` passes targeted all-target Clippy with warnings denied;
  formatting, layering guards, changelog governance, and diff hygiene also
  pass.

## [0.93.12] - 2026-07-17 - Durable Replay Recovery and Complete Install Failure Evidence

### 📝 Summary

`0.93.12` closes the recovery gap between a successful external effect and its
terminal accounting or replay commit. The replay receipt is now the canonical
durable authority for that boundary: it records the cost-intent identity and a
staged terminal response before settlement, then lets an identical retry finish
accounting and commit the response without repeating the external operation.

Unresolved post-effect receipts are no longer expired or purged by request TTL.
Root-install phase failures also retain complete typed evidence when both the
operation and the attempt to persist its failure receipt fail. Public successful
Candid, JSON, and CLI response shapes are unchanged.

### 🔧 Changed

- Extended replay receipts with optional staged response data and durable cost
  guard settlement identity. Existing receipts whose fields are absent retain
  their exact stable encoding.
- Made terminal cost settlement reconstructible and idempotent from persisted
  intent identities.
- Added accounting-only recovery for ICP refill, pool creation, governed root
  provisioning, and governed root upgrade.
- Staged governed root responses before terminal settlement, and made root
  upgrade persist its installed-module projection before committing replay.
- Kept `ExternalEffectInFlight` and `RecoveryRequired` receipts active across
  expiry and cleanup; only pre-effect reservations and completed receipts keep
  request-TTL behavior.
- Added typed recovery reasons for failed cost settlement and failed state
  projection so diagnostics preserve the failed boundary.
- Added `InstallPhaseFailureError`, whose accessors expose both the typed phase
  operation error and the typed failure-receipt write error.

### 🩹 Fixed

- An external effect followed by failed terminal cost settlement can no longer
  lose its cost intent and remain permanently unrecoverable.
- Request cleanup can no longer remove evidence for an unresolved external
  effect and permit a later duplicate operation.
- Root-install failure handling no longer relegates a simultaneous
  failure-receipt write error to log-only evidence.

### ⚠️ Breaking

- Identical retries for recoverable cost-settlement failures now finish the
  pending accounting and return the staged response. They do not re-run the
  external effect or return the former generic in-flight outcome.
- Post-effect and recovery-required replay receipts intentionally ignore request
  TTL until they reach a terminal state. Operators must resolve them rather than
  relying on cleanup to discard them.
- When a root-install phase operation and its failure-receipt write both fail,
  the returned source is now `InstallPhaseFailureError`; callers that downcast
  errors should inspect its two typed causes.
- No aliases, compatibility commands, fallback replay paths, or deprecated
  surfaces were added.

### 🧭 Migration Notes

- After a terminal settlement failure, retry with the exact same request ID,
  caller, and payload. Do not create a replacement request or repeat the
  external operation manually.
- `ExternalEffectStatusUnknown` and `StateProjectionFailed` remain fail-closed
  manual-recovery conditions; only a recorded cost-settlement failure is
  completed automatically.
- Consumers that inspect root-install failures should use
  `InstallPhaseFailureError::operation_error()` and
  `InstallPhaseFailureError::receipt_write_error()` when the composite error is
  returned.
- The stable replay schema change uses optional fields and requires no operator
  migration command.

### 📊 Audit

- Traced representative ICP refill, pool-create, root-provision, and root-upgrade
  operations across admission, cost reservation, external effect, replay
  persistence, settlement, recovery, diagnostics, and response projection.
- Verified that recovery authority remains in the replay store and cost-intent
  store rather than introducing a second orchestration path.
- Verified exact stable receipt bytes remain unchanged when the new optional
  recovery fields are absent.
- Kept the change scoped to replay/accounting recovery and root-install failure
  evidence; no public success DTO or unrelated deployment behavior changed.

### 🧪 Testing

- Targeted replay tests cover recoverable settlement failures, accounting-only
  retries, expiry boundaries, cleanup retention, and stable-state round trips.
- Targeted ICP, pool, provision, and upgrade tests cover pre-effect rollback,
  post-effect recovery marking, idempotent completion, and non-reexecution.
- Targeted host tests cover ordinary typed install failures and the composite
  operation-plus-receipt-write failure.
- `canic-core` and `canic-host` pass targeted all-target Clippy with warnings
  denied; formatting, layering guards, changelog governance, and diff hygiene
  also pass.

## [0.93.11] - 2026-07-17 - Fail-Closed Cost Settlement and Typed Dispatch Cleanup

### 📝 Summary

0.93.11 closes the next three findings from the post-0.93.10 code review.
Cost-guard reservation rollback now reports cleanup failure explicitly, and
ICP-refill plus empty-pool creation settle durable quota/cycle accounting
before committing replay success. A settlement failure therefore returns to
the caller while the existing in-flight replay receipt continues to prevent a
duplicate external effect.

The retired Wasm-store protected-internal-proof model is removed completely.
The maintained client uses canonical method names and protocol manifests
directly, while endpoint-owned root-principal authorization remains unchanged.

Backup-manifest, fleet-create, fleet-config, and config-list dispatch preserve
their concrete child-command and fleet-configuration errors rather than
converting them to display strings. Public Candid, JSON schemas, stable-state
records, successful CLI arguments, and successful CLI reports are unchanged.

### 🔧 Changed

- Adds a typed cost-guard reservation rollback failure containing both the
  original reservation error and the quota-cleanup error. Cleanup failure is
  internal and cannot be projected as an ordinary quota rejection.
- Makes resumable ICP-refill cleanup remove its replay receipt only after cost
  recovery succeeds.
- Makes terminal ICP-refill and empty-pool creation complete cost settlement
  before storing a committed replay response.
- Leaves the authoritative replay receipt in-flight with no cached response
  when settlement fails, so the external operation cannot be repeated under
  the same identity.
- Makes backup and fleet dispatch wrap `ManifestCommandError`,
  `ScaffoldCommandError`, and `ListCommandError` directly.
- Makes config-list projection retain `FleetConfigError` and removes the
  generic display-based config error mapper.
- Keeps the canonical Wasm-store root-update and structural-query method
  manifests as the sole client/protocol classification surface.

### 🗑️ Removed

- Removes `canic_wasm_store_method_requires_internal_proof` from
  `canic-core` and the public `canic::protocol` facade.
- Removes the duplicate Wasm-store endpoint wrapper, identical root-update and
  structural-query constructors, always-false proof classification, debug
  assertion, and stale proof-absence tests.
- Removes log-only cost settlement and recovery branches from ICP-refill and
  empty-pool creation.
- Removes string-backed manifest, fleet-create, fleet-config, and config-list
  error projections.
- No alias, compatibility wrapper, deprecated API, duplicate path, fallback,
  or removed-behavior regression test is retained.

### ⚠️ Breaking

- The public `canic_wasm_store_method_requires_internal_proof` helper is
  removed. It represented a deleted protocol and always returned `false`.
- `CostGuardReserveError` adds the fail-closed `ReservationRollback` variant.
  Exhaustive Rust matches must handle it.
- `BackupCommandError::Manifest`, `FleetCommandError::Create`, and
  `FleetCommandError::Config` now contain their concrete child error instead
  of `String`; `ListCommandError::Config(String)` is replaced by

## [0.93.2] - 2026-07-16 - Fail-Closed Lifecycle and Restore Receipts

### 📝 Summary

0.93.2 closes the two P1 safety findings from the post-0.93.1 audit. Wasm-store
retirement now has one generation-bound root lifecycle and a one-way
store-local write boundary. Restore apply receipts no longer infer successful
outcomes or attempt identity from omitted JSON fields.

The release does not change the public Candid schema or stable-state layout.
It intentionally hard-cuts permissive restore-receipt decoding rather than
adding a legacy fallback or compatibility representation.

### 🩹 Fixed

- Binds remote prepare, begin, complete, and finalize results to the same
  retired binding and publication generation observed before the inter-canister
  call. A concurrent ownership transition is returned as a typed conflict
  instead of committing against newer state.
- Serializes root-owned GC/finalize/delete orchestration with a workflow-local
  in-flight guard and treats unexpected persistence outcomes as errors rather
  than discarding them.
- Makes the GC lifecycle one-way for publication authority. Explicit binding,
  automatic placement, release reconciliation, and direct-target publication
  all reject stores whose GC mode is not `Normal`.
- Enforces the same write boundary inside the store canister, so chunk-set
  preparation, chunk publication, and manifest staging fail closed after GC
  starts even if a root-side snapshot becomes stale.
- Requires finalized-store deletion to match the authoritative binding,
  canister principal, unreferenced publication state, and completed local GC
  record both before and after the remote status call.
- Makes restore operation-receipt `outcome` and `attempt` mandatory JSON fields
  and rejects attempt zero with a typed journal-validation cause.

### ⚠️ Breaking

- Restore apply journals containing operation receipts without `outcome` or
  `attempt` no longer deserialize. Receipts using attempt zero no longer pass
  validation.
- Publication requests targeting a store in `Prepared`, `InProgress`,
  `Clearing`, or `Complete` GC state now return a conflict instead of allowing
  the store to regain or retain publication authority.
- No compatibility alias, inferred receipt default, legacy decoder, or
  alternate publication path is retained.

### 🧭 Migration Notes

- Regenerate malformed restore journals or provide the explicit durable
  receipt fields: `outcome` and an `attempt` value starting at one.
- Complete retirement and deletion for a store that has entered GC; publish to
  a different `Normal` store rather than trying to reactivate it.

### 📊 Audit

- The focused source-and-test delta spans 15 files, with 639 insertions and 53
  deletions before release-note finalization.
- The Wasm-store fix covers explicit selection, automatic placement,
  reconciliation, direct publication, store-local mutations, asynchronous GC
  commits, finalization, and destructive deletion.
- The restore fix removes both permissive defaults and proves decode-time and
  validation-time rejection boundaries.

### 🧪 Testing

- Twenty-one focused control-plane tests cover publication selection,
  generation drift, in-flight lifecycle rejection, exact deletion identity,
  typed error projection, and the store-local write gate.
- Thirty-four restore apply-journal tests cover positive receipt behavior,
  missing-field rejection, zero-attempt rejection, duplicates, transition
  ordering, command previews, and journal invariants.
- Targeted package checks, strict all-target Clippy for the control-plane and
  backup packages, formatting, changelog governance, and diff hygiene pass.

## [0.93.0] - 2026-07-16 - Pre-1.0 Surface Hard Cuts

### 📝 Summary

0.93.0 removes stale and unreachable surfaces found by a repository-wide
read-only audit. The hard cuts cover an obsolete issuer-renewal authority,
replay state with no production writer, unused compatibility APIs, and
serialized variants that current producers cannot emit.

Maintained behavior is now backed by one production authority per concern.
The persisted chain-key batch owns issuer-renewal progress, replay keeps only
live reservation and recovery states, and template publication removes stale
manifests instead of retaining deprecated tombstones.

This release intentionally adds no aliases, fallback decoding, compatibility
records, or anti-resurrection paths for the removed pre-1.0 contracts.

### 🔧 Changed

- Root issuer-renewal status now projects `latest_batch` from the canonical
  persisted chain-key batch instead of projecting a separate
  `active_attempt` record.
- CLI text and JSON report batch status, proof epoch, expiry, installation,
  retry, and failure evidence. A reachable issuer without a usable active
  proof is reported as `proof_unavailable` rather than ready.
- Template publication deletes superseded or unconfigured approved manifests;
  store accounting is updated when those records are removed.
- Chain-key signature verification and delegated root-proof callbacks carry
  only the values used by cryptographic verification. Key identity,
  derivation-path, and certificate-hash checks remain enforced by their
  owning validation stages.

### 🗑️ Removed

- Removes the issuer-renewal attempt model, persisted attempt collection,
  attempt-only metrics, outcome and failure counters, and the old status
  projection. Obsolete attempt data is neither migrated nor exposed.
- Removes replay `TerminalFailed` state and its bounded terminal-error model,
  commit path, classifications, and workflow branches. Production never wrote
  that state.
- Removes the Wasm-store `Deprecated` template-manifest state and the obsolete
  aggregate chunk-set request and storage path.
- Removes backup authority, snapshot-read, quiescence, and proof-source
  variants that no current plan or preflight producer emits, plus the
  unproduced `InstalledModuleHash` artifact-digest source.
- Removes uncalled root-capability, endpoint-performance, lifecycle,
  topology-query, RPC, pool, ledger, CLI, and private compatibility surfaces
  identified by the audit.

### 📊 Audit

- The release delta from `v0.92.14` spans 81 files, with 776 insertions and
  1,739 deletions after release-note finalization.
- Every removed serialized variant was traced through constructors,
  validation, rendering, persistence, and tests before deletion.
- Intentional feature-conditioned stable schemas, canonical auth snapshots,
  delegated-auth metrics, and the required `finish!` marker remain in place.
- Receipt-backed promotion inputs remain supported because operator-provided
  promotion JSON reaches and validates both maintained receipt kinds.

### ⚠️ Breaking

- `RootIssuerRenewalStatusResponse` replaces `active_attempt` with
  `latest_batch` and removes attempt-era state fields. CLI JSON consumers must
  update to the current batch projection.
- `TemplateManifestState::Deprecated` is no longer accepted by the Wasm-store
  Candid or stable-record contract. Persisted stores must not contain that
  state before upgrade; 0.93.0 intentionally provides no compatibility
  decoder or migration record.
- Backup JSON using removed authority, quiescence, proof-source, or artifact
  digest variants must be regenerated with the current CLI.
- Manually injected replay receipts using `TerminalFailed` are unsupported;
  live production replay states retain their existing stable encoding.

### 🧭 Migration Notes

- Update automation that parses issuer-renewal status to consume
  `latest_batch` and its batch fields instead of `active_attempt`,
  `last_outcome`, `consecutive_failures`, or `active_attempt_id`.
- Regenerate persisted backup plans or receipts that were authored with
  unsupported enum variants.
- Verify Wasm-store manifest state before upgrading. Superseded and
  unconfigured approved manifests are removed by current publication flows
  rather than converted to deprecated tombstones.
- No application configuration or source migration is required when these
  removed serialized states and response fields are not consumed.

### 🧪 Testing

- Pins the exact stable bytes of a live replay recovery receipt, proving that
  removal of the unwritten terminal variant does not alter maintained replay
  encoding.
- Current-behavior coverage exercises issuer-renewal batch projection,
  delegated proof verification, template replacement and pruning, backup plan

- Strict package-scoped linting, formatting, Candid contract checks,
