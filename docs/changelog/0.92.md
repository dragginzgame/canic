## [0.92.10] - 2026-07-15 - Canonical Layering Closure

### 📝 Summary

0.92.10 completes the D11 canonical-layering slice and fixes the final active
layering finding. The remaining 18 production ops-to-policy dependencies are
removed, leaving the executable layering guard at zero violations.

Shared environment, cycles-funding, topology, placement, and metric decision
values now belong to model. Policy remains pure decision logic, ops performs
mechanical state access and approved single-step effects, and workflow owns
multi-step orchestration.

Root delegation-proof admission now occurs in workflow before proof-epoch
advancement or batch insertion. Ops selects due templates, validates the
ordered approved material, builds and stores the admitted batch, signs the
exact batch, and retrieves the resulting proof. Malformed approvals reject
without advancing the epoch or mutating batch state.

This is an internal architectural hard cut. It adds no alias, compatibility
wrapper, deprecated path, duplicate flow, or fallback behavior. Public Rust
and Candid APIs, stable-state records, configuration schemas, metric labels,
and Cargo dependencies are unchanged.

- Keeps scaling, sharding, and pool metrics mechanical by passing model-owned
  reason values or workflow-classified outcomes into metrics ops.
- Centralizes topology policy-error projection at the internal/public error
  boundary while preserving the existing public codes and fallback behavior.
- Makes root delegation-batch preparation an explicit workflow sequence of
  plan, policy admission, commit, exact-batch signing, and proof retrieval.
- Validates approval count, order, issuer identity, and shared expiry before
  proof-epoch advancement or insertion of a new delegation batch.

### 🗑️ Removed

- Removes policy-owned input wrappers for topology, registry, index, scaling,

## [0.92.8] - 2026-07-15 - Release Execution Integrity

### 📝 Summary

0.92.8 makes release execution reproducible and fail-closed at its external
tool boundary. Every active GitHub Action execution now uses a reviewed full
commit identity, every downloaded executable is bound to an exact version and
repository-owned platform checksum, and Cargo-installed CI and maintainer
helpers use exact crate versions with locked dependency resolution.

`tool-versions.env` is the sole executable-tool identity authority. Installers
load it directly, reject alternate version arguments and environment
overrides, verify downloaded content before extraction or execution, and
confirm the installed binary reports the pinned version. A data-driven CI
guard validates the maintained version/checksum surface and every installer
reference without maintaining a competing pin ledger.

Release mutation is transactional as well. A failed lockfile regeneration,
surface synchronization, or tag-collision check restores every Cargo manifest,
the lockfile, and the installed-CLI default to their pre-bump contents.
One-shot release targets invoke bump, stage, commit/tag, and push sequentially
even under parallel Make, while the push phase requires a clean exact release
commit and annotated tag before performing one atomic branch/tag update.

This is an operator and CI hard cut. The npm `ic-wasm` wrapper, upstream ICP
CLI shell-installer execution, mutable Action tags, floating runner label, and
implicit broad updates are removed without aliases, shims, deprecated paths,
or fallback installation flows. Runtime canister behavior, Candid and Rust
APIs, stable state, configuration schemas, and Cargo dependency resolution are
unchanged.

### 🔧 Changed

- Pins all 13 active external Action executions to full 40-hex commit
  identities and selects Ubuntu 24.04 explicitly for every CI job.
- Pins actionlint 1.7.12, ShellCheck 0.11.0, ICP CLI 1.0.2, `ic-wasm` 0.9.11,
  and PocketIC 14.0.0 with platform-specific repository-owned digests.
- Pins CI and maintainer Cargo helpers to exact crate versions and installs
  them with locked dependency resolution.
- Verifies PocketIC's compressed archive, decompressed binary, and restored
  cache content before execution.
- Makes the shared IC prerequisite gate consume repository pins
  unconditionally and reject mismatched `icp` or `ic-wasm` versions.
- Makes `make update-dev` synchronize repository-owned tool versions without
  updating unrelated Rust toolchains or workspace dependencies.
- Executes one-shot patch, minor, and major release phases in explicit order
  rather than modeling state-changing phases as parallelizable prerequisites.
- Validates that release HEAD is clean, has the exact `Release <version>`
  subject, and is identified by the matching annotated tag before push.
- Clears Cargo build artifacts before the network operation so no fallible
  cleanup command can report an error after a successful release push.
- Pushes the release branch and followed tags atomically so the remote cannot
  accept only part of the release ref update.
- Establishes one canonical support matrix: Ubuntu 24.04 x86_64 with
  `x86_64-unknown-linux-gnu` and `wasm32-unknown-unknown` is the sole
  release-supported host/target cell.
- Classifies Linux AArch64 and macOS installer branches as install-capable but
  not release-supported rather than allowing installer availability to imply
  release coverage.

### 🗑️ Removed

- Removes mutable GitHub Action tag execution; release tags remain comments
  only and do not select executable workflow code.
- Removes Node setup and the npm `ic-wasm` package/wrapper path in favor of the

### 🩹 Fixed

## [0.92.7] - 2026-07-15 - Reproducible Root Artifacts

### 📝 Summary

0.92.7 removes build-machine path identity from shipped root artifacts. Root
bootstrap runtime records and lifecycle diagnostics now retain only stable
role, artifact-kind, size, and content-hash evidence. Two isolated offline
build lanes consequently produce byte-identical root and bootstrap-store raw
and gzip artifacts, plus identical semantic build provenance.

The host builder now records every considered optional artifact transform with
its canister role, transform kind, admission mode, tool, tool version, and
outcome. An unavailable optional tool remains nonfatal but is explicit; a
present tool with an invalid version identity or a failed invocation rejects
the build.

This is a pre-1.0 serialized hard cut. The current
`canic.build_provenance.v1` payload requires transform evidence and does not
accept the older shape through a default, alias, alternate schema, or fallback
decoder. Root Wasm bytes and hashes also change because the former absolute
path fields are removed.

### 🔧 Changed

- Keeps absolute bootstrap source and generated asset paths as build-time
  inputs only; they no longer become runtime record fields in root Wasm.
- Projects stable root bootstrap diagnostics using role, artifact kind,
  compressed/decompressed sizes, and content hashes.
- Records bootstrap-store transforms in root provenance before the final root
  transforms because those transformed bytes are embedded into root.
- Records `shrink` and `candid_metadata` decisions as `applied`,
  `tool_unavailable`, or `not_requested` under the sole host-builder owner.
- Requires applied transforms to include a non-empty tool version and forbids
  unavailable or unrequested transforms from claiming one.
- Preserves the existing optional-tool policy: absence is recorded and does
  not fail a build, while a present but failing tool remains fail-closed.

### 🗑️ Removed

- Removes the absolute source-artifact and generated-asset path fields from
  embedded root bootstrap runtime records.
- Removes the corresponding path-bearing root lifecycle log material.
- Removes acceptance of build-provenance payloads that omit transform
  evidence. No compatibility decoder or legacy DTO is retained.

### 🩹 Fixed

- Fixes `CANIC-092-BUILD-001`: identical declared root inputs now produce
  identical root Wasm and gzip bytes across isolated Cargo and ICP roots.
- Fixes `CANIC-092-BUILD-002`: stable provenance identifies whether optional
  artifact transformations were applied, unavailable, or not requested, and
  records the exact tool version when applied.
- Removes build-machine paths from the runtime bytes used to identify and
  publish root artifacts.
- Keeps transform failure deterministic: failed shrink output cannot replace
  the original artifact, and missing or inconsistent provenance cannot be
  presented as a successful build record.

### 📊 Audit

- Advances the current mandatory-trace ledger from 9 pass and 1 fail to 10
  pass and 0 fail; `TRACE-DEPLOY-001` now passes without rewriting the frozen
  Phase C result.
- Reduces the live reviewed finding index from 13 to 11: 4 P1, 6 P2, and 1 P3
  remain.
- Reproduces root raw Wasm, root gzip, bootstrap-store raw Wasm,
  bootstrap-store gzip, and normalized provenance across two isolated offline
  release lanes.
- Confirms that neither final root Wasm contains either temporary build root or
  copied bootstrap asset path.

### ⚠️ Breaking

- `BuildProvenanceV1` now requires its `transforms` collection. Stored JSON
  using the prior field set must be regenerated rather than loaded through a
  compatibility fallback.
- Rust code constructing `CanisterArtifactBuildOutput` or
  `BuildProvenanceV1` directly must supply the maintained transform evidence.
  Prefer consuming output from the canonical host builder.
- Root artifacts built before 0.92.7 have different bytes and hashes even when
  the functional source/configuration inputs are otherwise unchanged.
- There is intentionally no alias for the removed bootstrap path fields and no
  decoder for the former provenance shape.
- No Candid endpoint, stable-state schema, fleet configuration, deployment
  truth, or install authority changes.

### 🧭 Migration Notes

- Rebuild root artifacts and any release manifests or registries that bind the
  prior root artifact hash before promotion or installation.
- Regenerate saved build-provenance envelopes with 0.92.7. Do not reuse an
  older `canic.build_provenance.v1` envelope that lacks transform evidence.
- CI policies comparing artifacts should compare transform outcome and tool
  version before evaluating byte hashes from separate builds.
- Install `ic-wasm` when transformed-byte parity is required across builders;
  otherwise the canonical build remains valid and records
  `tool_unavailable` explicitly.
- No canister reinstall, stable-state conversion, configuration migration, or
  operator recovery action is required solely by this release.

### 🧪 Testing

- Seven artifact-transform cases pass across applied, unavailable, failed, and
  unreportable-version boundaries.
- Fifteen build-provenance and policy-gate cases pass, including rejection of
  inconsistent transform/version evidence.
- Twelve release-set manifest cases preserve positive ordering, fail-closed
  admission, and unchanged prior manifest bytes on rejection.
