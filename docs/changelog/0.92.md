# Canic 0.92 Changelog

## [0.92.4] - 2026-07-15 - Blob-Billing Workflow Ownership

### 📝 Summary

0.92.4 gives blob-storage billing one canonical execution path. Authenticated
endpoints delegate through the existing API projection to a billing workflow;
the workflow owns Cashier sequencing, funding-guard lifetime, retry recovery,
gateway synchronization, and readiness observation; pure policy owns only
deterministic validation and decisions.

This is a structural ownership correction. Billing protocol, prices, public
DTOs, Candid shapes, stable records, successful behavior, and existing error
codes are unchanged. No alias, shim, fallback, deprecated API, duplicate
billing path, or migration layer is introduced.

### 🔧 Changed

- Moves billing configuration sequencing, Cashier balance and top-up calls,
  gateway-principal replacement, reserve-aware funding, and status assembly
  from the API layer into one workflow owner.
- Places configuration, funding-attachment, and readiness decisions in pure
  policy without DTO, storage, async, timer, or IC dependencies.
- Keeps ops responsible for Candid-number conversion, individual Cashier and
  IC calls, funding-guard access, and single-step state mutation.
- Makes the API a thin delegate that maps typed workflow failures once into
  the existing public error contract.

### 🩹 Fixed

- Removes the competing API-owned blob-billing orchestration identified by
  `CANIC-092-LAYERING-001`.
- Preserves all-or-nothing reserve admission: a request that would cross the
  project reserve returns the existing skipped report without partially
  attaching cycles or calling Cashier.
- Preserves deterministic transient-failure recovery by holding the funding
  guard across the Cashier operation and releasing it on every return path so
  a later retry can proceed.
- Keeps malformed Cashier balances distinct from unavailable balances and
  retains the existing readiness blocker and warning ordering.

### 📊 Audit

- Fixes `CANIC-092-LAYERING-001` and moves the current `TRACE-BLOB-001` rerun
  from fail to pass without rewriting the frozen Phase C result.
- Advances the current mandatory-trace ledger to 9 pass and 1 fail; the frozen
  Phase C aggregate remains 6 pass and 4 fail.
- Leaves 16 unresolved reviewed findings: 5 P1, 10 P2, and 1 P3.
- Leaves the separate `CANIC-092-LAYERING-005` inventory unchanged at 18
  ops-to-policy files; those subsystems remain separately reviewed work.

### 🧪 Testing

- All 878 all-feature `canic-core` library tests pass, including stable billing
  round trips and existing blob lifecycle regressions.
- Fifty focused blob-storage tests and four policy/DTO boundary guards pass.
- Nineteen public protocol-surface tests confirm no Candid or public-placement
  change.
- Disposable PocketIC cases pass for reserve refusal, transient Cashier
  failure and retry after guard release, the readiness status matrix, and
  configured or missing-config upgrade persistence.
- Strict all-feature/all-target `canic-core` Clippy, targeted formatting,
  changelog governance, layering detector fixtures, and diff hygiene pass.

## [0.92.3] - 2026-07-15 - Root-Issuer Admission Ownership

### 📝 Summary

0.92.3 gives root-issuer policy and renewal-template admission one canonical
path: authenticated endpoints delegate to workflow, workflow invokes pure
policy before mutation, ops converts and persists, and model owns the
authoritative issuer and renewal state shapes.

This is a structural hard cut rather than a compatibility migration. The
former ops-owned validators and policy-owned persisted shapes are removed
without aliases, shims, duplicate admission paths, or deprecated re-exports.
Public DTOs, Candid encodings, stable records, and successful behavior are
unchanged.

### 🔧 Changed

- Moves root issuer policy, renewal template, renewal state, proof reference,
  attempt, status, and outcome shapes from pure policy into model ownership.
- Makes workflow the sole orchestrator for root policy and renewal-template
  admission, including policy lookup, pure validation, commit, and renewal
  timer start.
- Restricts root-issuer ops to DTO/model conversion, persistence, status
  projection, and existing single-step renewal bookkeeping.
- Preserves the previous renewal ordering: successful admission commits before
  requesting timer start, while rejected admission performs neither action.

### 🩹 Fixed

- Removes duplicate request validation from ops and places every root policy
  and renewal-template admission decision in pure policy.
- Rejects zero or full refresh ratios, zero TTLs, missing enabled audiences or
  grants, unregistered renewal issuers, and policy-widening templates before
  changing policy, template, registry epoch, or timer state.
- Preserves the existing typed public codes and renewal TTL diagnostic text at
  the workflow projection boundary.

### 📊 Audit

- Fixes `CANIC-092-TEST-001` with direct positive, rejection, boundary, and
  unchanged-state proof.
- Removes seven root-issuer/model ownership violations from
  `CANIC-092-LAYERING-005`, reducing the live ops-to-policy detector result
  from 25 files to 18.
- Keeps `CANIC-092-LAYERING-005` open for separately reviewed subsystem slices;
  this patch does not claim a bulk layering cleanup.
- Leaves 17 unresolved reviewed findings: 5 P1, 11 P2, and 1 P3.

### 🧪 Testing

- All 881 all-feature `canic-core` library tests pass, including stable auth
  round trips and renewal/provisioning regressions.
- Thirteen focused root-issuer tests prove successful policy/template upsert,
  every request rejection boundary, unchanged state and registry epoch, and
  skipped timer start.
- Four pure-policy/passive-DTO boundary guards and 19 public protocol tests
  pass.
- Strict all-feature/all-target `canic-core` Clippy, formatting, changelog
  governance, and diff hygiene pass.
- Disposable PocketIC issuer-readiness and scheduled-renewal regressions pass.

## [0.92.2] - 2026-07-15 - Typed Auth Causes and Canonical Layer Authority

### 📝 Summary

0.92.2 preserves exact delegated-auth proof and issuer-provisioning failures
through the existing public error envelope. Pending, expired, stale, invalid,
transport, malformed-response, and issuer-application failures no longer
collapse into string-backed or generic unavailable/internal outcomes.

The release also establishes `AGENTS.md` as the sole active architecture
authority. Public core documentation, module guidance, and the architecture
contract now agree on `endpoints -> workflow -> policy -> ops -> model`, with
model owning authoritative state and storage invariants.

### 🔧 Changed

- Retains typed proof verification causes until the existing public auth error
  projection boundary.
- Returns an issuer canister's exact application rejection from explicit root
  proof provisioning instead of converting every failed install to
  `Unavailable`.
- Distinguishes root-to-issuer transport failures from request-encoding and
  invalid-response failures while retaining their diagnostic causes.
- Keeps renewal-ledger install outcomes coarse while the explicit root facade
  returns the first exact typed failure.
- Makes endpoint-to-workflow delegation the only documented API boundary and
  assigns authoritative state and storage invariants to model. Storage remains
  a passive persisted representation accessed and converted by ops.

### 🩹 Fixed

- Removes string flattening from active-proof, token, role-attestation, and
  chain-key proof verification paths.
- Rejects wrong-issuer, expired, and cryptographically invalid active proofs
  with their specific existing error codes before replacing installed proof
  state.
- Removes active wording that allowed endpoints to delegate directly to ops or
  assigned storage ownership to a competing layer.

### 📊 Audit

- Fixes `CANIC-092-ERROR-001`, `CANIC-092-LAYERING-003`, and
  `CANIC-092-DOCS-001` through bounded Phase D slices D2 and D3.
- Leaves 18 unresolved reviewed findings: 5 P1, 12 P2, and 1 P3.
- Keeps the 25 existing product-code ops-to-policy violations visible for later
  finding-backed slices; this release does not claim that documentation changes
  resolved those implementation findings.

### ⚠️ Breaking

- Callers that treated proof or issuer-install failures as generic
  `Unavailable`, `Internal`, or display text may now receive the existing
  specific codes `AuthProofPending`, `AuthProofExpired`, `AuthMaterialStale`,
  `AuthTokenExpired`, or `InvalidInput`, or the issuer's exact existing error.
- No public Candid shape, token or proof format, stable-state schema, package
  feature, or success response changes.

### 🧭 Migration Notes

- Match auth and provisioning failures by typed error code rather than display
  text or a generic unavailable/internal fallback.
- No state migration, canister reinstall, configuration change, or proof/token
  format conversion is required.

### 🧪 Testing

- All seven retained authentication methods pass, including the current
  mandatory authentication trace and nonempty audience/replay selections.
- Focused core validation passes 263 auth tests plus strict warning-as-error
  Clippy and package verification.
- PocketIC covers role attestation, capability checks, root provisioning,
  session bootstrap, renewal, and invalid-proof rejection without active-state
  mutation.
- Architecture scans and layering detector fixtures pass. The full guard
  reports the same 25 known implementation violations, and warning-as-error
  rustdoc retains only the separately tracked D10 private-link finding.

## [0.92.1] - 2026-07-15 - Holistic Baseline and Publication Safety

### 📝 Summary

0.92.1 completes the frozen holistic product baseline: all 22 retained audit
definitions have valid results, and all ten mandatory end-to-end traces have
complete execution evidence. The resulting baseline is deliberately a valid
`fail`, not a release-readiness claim: six traces pass and four retain product
findings for bounded Phase D work.

The first accepted product slice makes durable root Wasm-store publication
fail closed. Bootstrap publication, controller-requested publication, and
post-upgrade reconciliation now reserve quota and cycle headroom through one
workflow-owned permit before any store or management effect. Publication
failures retain distinct typed causes and existing public error codes instead
of collapsing into generic internal failures.

### 🔧 Changed

- Gives durable Wasm-store publication one canonical quota/cycle admission and
  settlement path across bootstrap, administrator publication, and upgrade
  reconciliation.
- Carries an unforgeable publication permit through placement, store creation,
  inventory, upload, staging, promotion, and root mirroring effects.
- Commits consumed quota on failed work while releasing its cycle reservation;
  successful work settles both intents.
- Distinguishes administrator, bootstrap, and reconciliation command kinds so
  retries and accounting retain their actual origin.
- Projects publication conflict, capacity, missing manifest, hash mismatch,
  invalid state, missing store, resource exhaustion, and unavailable transport
  through the existing typed public error envelope.
- Removes declaration-only durable-publication labels from direct staging and
  internal store-effect endpoint metadata. Binding and garbage collection
  remain separate commands outside this cost boundary.

### 🩹 Fixed

- Rejects an over-quota or underfunded publication before fleet or Wasm-store
  mutation rather than discovering the condition after effects begin.
- Preserves exact conflict and store-capacity causes through the public API,
  including unchanged-state rejection and interrupted-publication recovery.
- Removes the duplicate control-plane cost-reservation error mapper in favor
  of the canonical core projection.
- Completes the previously missing delegated-session, generated-guard, replay,
  target-committed, and root-unmirrored execution evidence for the mandatory
  authentication and control-plane traces.

### 📊 Audit

- Attempts all 22 retained frozen methods against the published `v0.92.0`
  snapshot and retains valid results for every active definition.
- Corrects method defects in the dependency, build, authentication,
  layering, capability-surface, complexity, change-friction, instruction, and
  Wasm audits. Superseded results remain invalid history and are not compared
  with the corrected methods.
- Executes all ten mandatory traces through the versioned, fingerprinted trace
  protocol. Six frozen-baseline traces pass; deployment, authentication,
  control-plane publication, and blob-storage traces fail on indexed product
  findings with no partial or blocked trace remaining.
- Fixes `CANIC-092-COST-001` and `CANIC-092-ERROR-002` in the first bounded
  Phase D slice. The current control-plane trace rerun passes without rewriting
  its frozen Phase C result.
- Leaves 21 reviewed product findings for later bounded work: 7 P1, 13 P2, and
  1 P3. This patch does not claim 0.92 closeout or 1.0 readiness.

### ⚠️ Breaking

- Durable root publication may now reject before effects with
  `ResourceExhausted` when its quota window or cycle headroom is exhausted.
- Automation that treated publication conflict, capacity exhaustion, missing
  releases, invalid state, hash mismatch, or transport failure as a generic
  internal error must handle their existing specific error codes.
- No Candid shape, public success response, stable-state schema, binding
  behavior, or garbage-collection behavior changes.

### 🧭 Migration Notes

- Treat publication rejection codes as typed outcomes. Retry quota exhaustion
  after its window resets and retry cycle exhaustion only after replenishing
  cycle headroom.
- Do not translate specific conflict, capacity, manifest, hash, state, store,
  or transport failures back into a generic internal failure.
- No state migration or configuration change is required.

### 🧪 Testing

- Eighteen focused publication unit tests cover permit ownership, settlement,
  typed cause projection, conflict, capacity, missing-release, and recovery
  behavior.
- Thirty replay/capability tests and focused security-ordering checks preserve
  authentication and replay boundaries.
- PocketIC proves ten same-window publications are admitted, the eleventh is
  rejected before fleet mutation, and low-cycle, conflict, capacity,
  authorization, and interrupted-recovery cases retain their expected state
  and typed outcomes.
- Package-scoped checks and strict Clippy pass for the changed core and
  control-plane surfaces. The layering guard reports the same 25 existing
  ops-to-policy findings and no new publication violation.

## [0.92.0] - 2026-07-14 - Holistic Audit-System Foundation

### 📝 Summary

0.92.0 establishes the reproducible audit system that the remainder of the
0.92 holistic product review will use. It first inventories the existing audit
machinery at the immutable 0.91.6 release anchor, records six confirmed
audit-system defects, then replaces competing or stale authority with one
versioned catalog, shared execution contract, and fingerprinted method set.

This release changes audit and operator-validation governance only. It does not
change runtime behavior, public Rust or Candid APIs, JSON or configuration
schemas, stable state, dependencies, package surfaces, deployment semantics, or
Wasm artifacts. It also does not claim that the holistic product baseline,
finding-backed product hardening, or 0.92 closeout has completed; those steps
follow after this commit becomes the frozen method snapshot.

### 🥾 Governance

- Establishes one canonical catalog for 22 active methods: 14 system methods,
  7 authentication invariants, and 1 manual-only module-surface review.
- Gives every active definition a stable audit ID, method version, owner,
  profile, trace mode, runtime/cost expectation, prerequisites, false-positive
  boundary, and shared run contract.
- Separates definition disposition, run result, result validity, finding
  status, and closeout verdict into independent state domains.
- Defines immutable run identities, authority precedence, deterministic finding
  identity, parent-versus-baseline comparisons, constrained P1 waivers, and a
  mandatory post-freeze method-defect protocol.
- Adds a versioned product-tree scope so audit-method preparation cannot hide a
  product, operator, release, or CI behavior change.

### 🔧 Changed

- Corrects instruction-footprint baselining: the first UTC run records no
  baseline, while same-day reruns compare to that day's unnumbered report.
- Makes instruction and Wasm audit execution fail closed against the IC
  environment, isolate build and temporary state, disable Cargo network access,
  detect source-tree mutation, and record full method identities.
- Requires compact evidence manifests with commands, exit status, baseline and
  method identities, tool versions, timestamps, retention/redaction
  declarations, and SHA-256 hashes.
- Retains one canonical Wasm size baseline instead of parallel aggregate JSON
  and TSV measurement projections.
- Assigns explicit owners for dependency advisories and licenses, lockfile
  integrity, build scripts, procedural macros, generated-code trust, unsafe
  code, reproducible Wasm, CI permissions, action pinning, secret scanning,
  release provenance, checksums, and supported host/target coverage.
- Replaces standing readiness conclusions with current executable gates, dated
  audit reports, current status, and an explicit release-line closeout.

### 🗑️ Removed

- Merges the separate access-purity, ops-purity, and workflow-purity
  definitions into the canonical layering method. No duplicate decision path,
  compatibility definition, or alias remains.
- Removes standing diagnostic-consistency, upgrade-state-compatibility, and
  RC-readiness audit documents whose conclusions were fixed to the 0.62 line.
- Removes their three literal CI guards and all remaining 0.62 filename
  anti-resurrection sentinels from active operator guards.
- Retires the module cleanup runner as an independent audit verdict; it remains
  only as a finding-backed implementation workflow.

### 📊 Audit

- Phase A inventories 15 recurring system definitions, 7 authentication
  invariants, 2 modular candidates, 3 operational audit candidates, 6
  executable helpers, release-line records, and retained evidence at the full
  0.91.6 commit.
- The inventory reports six confirmed P1 audit-system findings: missing method
  identity, competing catalogs/contracts, invalid instruction baselining,
  active historical verdict authority, missing holistic coverage owners, and
  unenforced execution/evidence safety.
- Phase B prepares and validates corrections for all six findings. Their final
  status remains tied to the immutable commit and product-tree verification,
  rather than being inferred from a dirty worktree.
- The 0.91.6 product-tree baseline is
  `8fce43e41ce430d9b505e19f8d596ed440b291d4c6ecb19c4a1cfdf71656a9b6`.
  The prepared method manifest is
  `fa92c4102efe74391c51f1f829aec7ac9c0b64941da73ee6dad1ebf2b292df07`.

### ⚠️ Breaking

- Custom CI or documentation that invokes the removed diagnostic, upgrade-state,
  or RC-readiness guard paths must move to the canonical audit-method catalog,
  current operator validation gates, and dated closeout evidence. No wrapper or
  fallback command is retained.
- Historical audit method labels and results are not automatically comparable
  with the new versioned methods. A comparison requires the same method
  identity and fingerprint or must be reported as non-comparable.
- Runtime, Candid, CLI, JSON, configuration, stable-state, dependency, and
  package contracts are unchanged.

### 🧭 Migration Notes

- Select reusable audits through the canonical method catalog and apply the
  shared run/evidence contract before treating a result as release evidence.
- Replace references to removed standing audit verdicts with the relevant
  current method, operator validation procedure, dated report, or release-line
  closeout.
- Commit the complete 0.92.0 batch before running the holistic product suite.
  The resulting full commit is the method-freeze candidate; instruction/Wasm
  baselines and product findings must not be produced from the dirty
  preparation tree.

### 🧪 Testing

- The catalog guard verifies all 22 active definitions, unique IDs, required
  contract fields, canonical catalog membership, and exact SHA-256
  fingerprints.
- Focused instruction tests prove method identity serialization and the
  first-run/same-day baseline rule.
- Bash syntax, current release-matrix, recovery, package/install, and workflow
  guards pass, together with action linting.
- Package-scoped formatting, changelog governance, product-tree baseline
  reproduction, and diff hygiene pass.
- High-cost instruction/Wasm baselines and broad workspace, PocketIC,
  deployment, publish, and release gates are intentionally deferred until the
  committed method freeze.
