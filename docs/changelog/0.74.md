# 0.74 Changelog

## 0.74.7 - 2026-06-27 - Renewal expiry and prepare failure hardening

### 📝 Summary

- Hardens root-managed delegated-auth renewal around bridge outages and late
  scheduled work.

- Keeps the long-term model root-managed: root records per-issuer prepare
  failures, prunes expired batch transport records, and blocks expired
  provisioner installs before issuer calls are attempted.

### 🔧 Changed

- Renewal sweeps now record prepare-stage failures on each due issuer's
  renewal state. Resource exhaustion is surfaced as `QuotaExceeded`; other
  prepare-stage failures are surfaced as `PolicyRejected`.

- Prepare failure recording preserves the issuer's last installed proof
  metadata, clears active scheduled work, increments consecutive failures, and
  schedules retry backoff.

- Renewal sweeps now prune expired scheduled batch transport records before
  preparing fresh work. Per-issuer renewal attempts remain stored so expiry and
  historical status stay observable.

- Renewal provisioner installs now validate the scheduled batch deadline before
  entering the issuer-call workflow. Late scheduled work is rejected at root,
  expired issuer attempts are marked `Expired` with
  `InstallDeadlineExpired`, and the active issuer state is cleared.

### 🧪 Testing

- Added focused renewal ops coverage for prepare-stage quota failures,
  prepare-stage policy failures, stale active attempt expiry with batch pruning,
  live scheduled install gating, and late scheduled install rejection.

- Added auth storage coverage for pruning expired renewal batch transport
  records while preserving still-live batches.

## 0.74.6 - 2026-06-27 - Manual install renewal repair

### 📝 Summary

- Updates root-managed renewal bookkeeping after successful controller/manual
  proof installs when the installed proof exactly matches the issuer's enabled
  renewal template.

- Keeps scheduled renewal installs on their existing issuer-attempt path while
  letting manual controller repair re-anchor root renewal state without waiting
  for the next scheduled prepare.

### 🔧 Changed

- Successful non-scheduled `canic_install_delegation_proof_batch` installs now
  compare the installed proof against the current enabled renewal template,
  issuer policy, audience, grants, TTL, cert hash, and computed refresh
  threshold before mutating renewal state.

- Matching manual installs update the issuer's root-managed renewal state with
  `last_installed_cert_hash`, expiry, refresh threshold, `Installed` or
  `AlreadyInstalled` outcome, reset failure count, and next attempt time.

- Manual installs that do not match the enabled renewal template remain valid
  manual proof installs but do not mutate renewal scheduling state.

### 🧪 Testing

- Added focused renewal ops coverage proving a matching successful manual
  install records root-managed renewal state and computes the same refresh
  threshold as the issuer policy.

## 0.74.5 - 2026-06-27 - Renewal template disable state

### 📝 Summary

- Makes disabling a root issuer renewal template an explicit renewal state
  transition instead of leaving an already scheduled issuer attempt active.

- Preserves operator visibility into the cancelled attempt while ensuring the
  issuer no longer advertises active scheduled work after the template is
  disabled.

### 🔧 Changed

- Disabling a renewal template now marks any active `Prepared`, `Installing`,
  or `FailedRetryable` scheduled issuer attempt as `Disabled` with
  `TemplateDisabled`.

- The issuer renewal state now clears `active_attempt_id`, records
  `TemplateDisabled`, updates the template fingerprint and timestamp, and
  preserves the existing consecutive failure count.

- Template upsert logging now records enabled/disabled template changes through
  `Topic::Auth` without including proof or token material.

### 🧪 Testing

- Added focused renewal ops coverage proving disabled templates clear active
  attempts, leave the cancelled attempt observable as `Disabled`, and do not
  increment the failure count.

## 0.74.4 - 2026-06-27 - Renewal provisioner CLI

### 📝 Summary

- Adds an operator-facing CLI surface for the constrained delegated-auth
  renewal provisioner ACL that root introduced in 0.74.0.

- Keeps the long-term renewal model intact: provisioners can be managed by
  controllers, while provisioner identities remain limited to completing
  root-scheduled renewal work.

### ➕ Added

- `canic auth renewal provisioner list <deployment>` queries root for the
  configured renewal provisioners and renders their principal plus enabled
  state.

- `canic auth renewal provisioner enable <deployment> <principal>` and
  `canic auth renewal provisioner disable <deployment> <principal>` update one
  provisioner entry through the existing root ACL endpoint.

- `--json` output is stable for automation:
  `auth_renewal_provisioners` for list responses and
  `auth_renewal_provisioner_upsert_result` for enable/disable responses. Both
  include schema version, deployment, network, root target, principal, and
  enabled state.

Command surface:

```bash
canic auth renewal provisioner list <deployment>
canic auth renewal provisioner list <deployment> --json
canic auth renewal provisioner enable <deployment> <principal>
canic auth renewal provisioner disable <deployment> <principal>
```

### 🔧 Changed

  the new renewal provisioner commands alongside `auth renewal run-once` and
  `auth renewal status`.

- Retained packaged/downstream and installed-CLI proofs now exercise the
  provisioner list, enable, and disable flows through the shared auth-renewal
  fixture, including the stable JSON `kind` and enabled-state fields.

### 🧪 Testing

- Added CLI parsing, scripted runtime, JSON/Candid response parsing, endpoint

## 0.74.2 - 2026-06-27 - Renewal drift diagnostics

### 📝 Summary

- Adds passive root-vs-issuer drift reporting for delegated-auth renewal
  status. Root-managed renewal state remains authoritative for scheduling, and
  issuer status is observational only.

- Extends targeted medic diagnostics so operators can surface renewal drift
  without knowing the lower-level auth command first.

### ➕ Added

- `canic auth renewal status <deployment> --issuer <principal> --json` now
  emits schema version `2` with `issuer_observation`, including observation
  availability, issuer active-proof status, cert hash, expiry, refresh
  threshold, drift flag, and bounded unavailable reasons.

- Plain `canic auth renewal status` output now includes issuer observation
  status, issuer cert hash/expiry when available, and an unavailable reason
  when local registry/Candid/transport parsing prevents issuer comparison.

- `canic info medic <deployment> --auth-renewal <issuer-principal>` runs a
  targeted delegated-auth renewal diagnostic and reports the same drift signal
  through the existing medic block output.

Command surface:

```bash
canic auth renewal status <deployment> --issuer <principal>
canic auth renewal status <deployment> --issuer <principal> --json
canic info medic <deployment> --auth-renewal <issuer-principal>
```

### 🔧 Changed

- `auth_renewal_run_once_result` remains schema version `1`; only
  `auth_renewal_status` moved to schema version `2` because the status JSON
  shape gained issuer observation fields.

- Issuer observation is best effort and non-mutating. If the issuer is not
  locally resolvable through installed deployment metadata, the status command

## 0.74.0 - 2026-06-27 - Root-managed delegation renewal

### 📝 Summary

- Starts root-managed renewal for delegated-auth issuer proofs. Root now owns
  renewal intent, scheduled issuer attempts, constrained proof retrieval, and
  install validation while preserving the existing direct-query certificate
  requirement.

- Keeps the bridge narrow. The bridge can courier only root-scheduled renewal
  work and cannot choose arbitrary issuers, audiences, grants, TTLs, or manual
  proof references.

### ➕ Added

- Root issuer renewal templates, issuer renewal status, scheduled renewal
  attempt/batch state, and stable storage records for root-managed renewal.

- A scheduled renewal work endpoint, a scheduled proof-retrieval endpoint, and
  install handling that records per-issuer renewal outcomes for scheduled
  proofs.

- Renewal provisioner registration and authorization for scheduled renewal
  work only.

- Root renewal sweep startup, expired-attempt cleanup, retry/backoff state, and
  bounded delegated-auth metrics plus auth logs for sweeps, retrieval,
  install, and provisioner operations.

- `canic auth renewal run-once <deployment>` as the first CLI bridge command,
  with JSON output available for automation.

- A 0.74 design note documenting the root-managed renewal model, the
  direct-query bridge requirement, threat-model invariants, and deferred
  issuer-drift/medic follow-up.

Command surface:

```bash
canic auth renewal run-once <deployment>
canic auth renewal run-once <deployment> --json
```

### 🔧 Changed

- Root delegated-auth protocol exports now include the renewal template,
  status, scheduled work, scheduled proof retrieval, provisioner, and batch
  install DTOs needed by the renewal bridge.

- Root install preflight now validates scheduled batch identity and pending
  proof metadata before accepting scheduled renewal proof installs.

- Replay-policy coverage now classifies the renewal template and provisioner
  upserts as snapshot-convergent root auth configuration endpoints.

### 🧪 Testing

- Added unit coverage for renewal template policy checks, state conversion,
  scheduled attempt lifecycle, expired-attempt cleanup, provisioner
  authorization, renewal metrics, protocol surface shape, and CLI bridge
  parsing/execution.

- Added a PocketIC scheduled-renewal bridge scenario and retained compile
  coverage for it. At 0.74.0 closeout, a clean local runtime execution had not
  yet been retained because the sandboxed PocketIC server could not bind to
  `127.0.0.1:0` in this workspace.
