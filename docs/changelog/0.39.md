# Canic 0.39

## 0.39.6 - 2026-05-18 - explicit genesis recovery boundary

### ➕ Added

- Added `LedgerCommitStore::recover_or_initialize(...)` so framework adapters
  can explicitly seed a genesis ledger when the protected physical commit store
  is empty.
- Added `LedgerCompatibility`, `LedgerCompatibilityError`,
  `CURRENT_LEDGER_SCHEMA_VERSION`, and `CURRENT_PHYSICAL_FORMAT_ID` so decoded
  ledger state is checked against supported logical and physical format
  versions before it can become authoritative.
- Added `AllocationLedger::validate_integrity(...)` and `LedgerIntegrityError`
  so decoded ledgers must preserve structural allocation-history invariants:
  unique stable keys, unique allocation slots, valid lifecycle metadata,
  bounded schema metadata history, and valid generation records.
- Added `AllocationBootstrap::initialize_validate_and_commit(...)`, the generic
  bootstrap path that initializes an empty ledger store from a supplied genesis
  ledger, validates declarations, commits the staged generation, and publishes
  validated allocations.
- Added `AllocationBootstrap::reserve_and_commit(...)` and
  `initialize_reserve_and_commit(...)` so explicit reservation operations run
  through policy checks, ledger-history checks, and the protected commit path.
- Added `AllocationBootstrap::retire_and_commit(...)` so explicit retirement
  operations stage tombstones and commit them through the same protected ledger
  path as declarations and reservations.
- Added protected commit recovery diagnostics with per-slot presence,
  generation, validity, authoritative generation, and recovery-error fields.
- `DiagnosticExport` can now include the protected commit recovery diagnostic
  alongside ledger records and generation history.

### 🔧 Changed

- Empty physical commit state is now distinguished from corrupt or partially
  written commit state. Genesis initialization is allowed only for an
  uninitialized store; corrupt stores still fail closed even when a caller
  supplies a genesis ledger.
- `LedgerCommitStore::recover(...)` and `commit(...)` now validate compatibility
  by default. Incompatible ledgers fail closed on recovery, and incompatible
  ledgers are rejected before physical commit writes.
- `LedgerCommitStore::recover(...)` and `commit(...)` now validate ledger
  integrity by default. Malformed ledgers fail closed on recovery, and malformed
  staged ledgers are rejected before physical commit writes.
- Reservation and retirement bootstrap helpers reject policy or ledger-history
  failures before committing, preserving the previous authoritative generation.

## 0.39.5 - 2026-05-18 - generic allocation lifecycle mechanics

### ➕ Added

- `ic-memory` now includes a substrate-neutral dual-slot physical commit model
  for encoded ledger generations. It recovers the highest valid generation,
  ignores corrupt newer slots, and fails closed when no committed slot
  validates.
- Added a generic `LedgerCommitStore` and `LedgerCodec` boundary so
  `ic-memory` can own generation-scoped commit mechanics without owning a
  concrete serialization format or stable-memory IO.
- Added generic reservation staging so reserved allocation records become
  durable ledger facts before a store is attached, and matching declarations
  transition those records from `reserved` to `active` without changing
  `stable_key -> allocation_slot` identity.
- Added explicit retirement staging through `AllocationRetirement`, requiring
  both the stable key and allocation slot and committing the tombstone as a
  generation-scoped ledger operation.
- Added `AllocationBootstrap` as the generic lifecycle pipeline for recovering
  the ledger, validating a sealed declaration snapshot against policy/history,
  staging a generation, committing it through the protected ledger store, and
  returning validated allocations tied to the committed generation.

## 0.39.4 - 2026-05-18 - publishable local extraction boundary

### 🔧 Changed

- Kept `ic-memory` as an unpublished local workspace extraction crate by making
  its workspace dependency path-only.
- Removed the compile-time `ic-memory` dependency from `canic-memory`, keeping
  `canic-memory` self-contained for crates.io publishing until `ic-memory` has
  an explicit publish order.
- Restored the stable-key and schema-metadata validation used by
  `canic-memory` into the publishable crate while leaving the generic
  equivalents in `ic-memory`.
- Added a workspace manifest guard that rejects runtime or build dependencies
  from explicitly publishable crates to workspace crates marked
  `publish = false`, catching unpublished local extraction dependencies before
  package or publish time.
- `scripts/ci/publish-workspace.sh` now runs that manifest-boundary guard
  before any publish attempt.

### 🧭 Migration Notes

- `0.39.3` was published out of sequence while the local `ic-memory` dependency
  boundary was still being corrected. Treat `0.39.4` as the packaging
  correction for downstream users.

## 0.39.2 - 2026-05-18 - local `ic-memory` extraction hardening

### 🔧 Changed

- `ic-memory` now keeps sealed declaration snapshots and validated allocation
  sets behind accessor APIs, so downstream code cannot fabricate a validated
  session by struct literal before policy and ledger-history validation.
- Runtime fingerprints carried by a declaration snapshot are now preserved in
  the staged generation record for allocation diagnostics.
- `ic-memory` now builds `DiagnosticExport` values from generic allocation
  ledgers.
- `canic-memory` remains self-contained and publishable while `ic-memory` is a
  local unpublished extraction crate. The generic bridge will be wired into
  `canic-memory` only after the publish order for `ic-memory` is explicit.

## 0.39.1 - 2026-05-18 - AppIndex role predicate

### ➕ Added

- Added `caller::has_app_role(role)` for internal endpoints that need to
  accept the canonical app canister for a configured role.
- The endpoint macro accepts either a role constant such as
  `canister::PROJECT_HUB` or a string literal such as `"project_hub"`.

### 🔧 Changed

- The new predicate resolves through the local AppIndex and compares the raw
  transport caller to the canonical role pid. It does not use delegated user
  identity and does not reopen internal system endpoints to arbitrary callers.
- AppIndex-backed caller checks are internal-only at macro validation time,
  matching the existing protection around topology-sensitive caller predicates.

### 🧪 Testing

- Ran `cargo test -p canic-core -p canic-macros`.

## 0.39.0 - 2026-05-18 - `ic-memory` extraction start

### ➕ Added

- Added the new root `ic-memory` crate as the staging boundary for extracting
  durable allocation-governance infrastructure out of Canic.
- Added generic stable-key parsing, allocation-slot descriptors, schema
  metadata, declaration collection/sealing, policy traits, substrate traits,
  validated allocation sessions, generation commit shapes, ledger records, and
  read-only diagnostic export shapes.
- Added generic allocation-history validation so runtime declarations are
  checked against policy and historical ledger facts before a validated session
  can open storage.
- Added pure logical generation staging so validated declarations can be
  recorded as the next ledger generation without coupling the generic crate to
  any physical stable-memory commit protocol.

### 🔧 Changed

- Started the 0.39 line around `stable_key -> allocation_slot forever`, with
  current `MemoryManager` virtual IDs modeled as one substrate-specific slot
  type rather than the generic ABI identity.
- Kept Canic-specific namespace, range, controller, endpoint, and lifecycle
  policy out of the generic crate. The existing `canic-memory` crate remains
  the Canic integration surface until adapter and migration work is explicit.
