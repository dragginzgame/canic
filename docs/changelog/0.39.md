# Canic 0.39

## 0.39.16 - 2026-05-18 - memory boundary cleanup

### 🔧 Changed

- Expanded Canic's persisted memory authority record for `ic-memory` from the
  ID `0` ledger slot to the full `0-9` allocation-governance reserve.
- Kept ID `0` as the only active canonical ledger self-record; IDs `1-9`
  remain empty per-ID slots under the `ic-memory` authority range until a
  future allocation-governance design explicitly assigns them.
- Normalized the authority owner label to `ic-memory` in tests and diagnostics.
- Changed macro-backed stable-memory opening to validate by explicit stable key
  through `ic-memory::AllocationSession`, instead of opening by MemoryManager
  ID after a bootstrap-ready boolean check.
- Removed the remaining implicit-key declaration and registration helpers from
  `canic-core::memory`.
- Shrunk `canic-core::memory::api` to the ledger diagnostic facade; declaration
  collection, bootstrap validation, and memory opening now stay on the runtime
  adapter path.
- Removed unused registry grouping and historical range/authority export
  helpers that duplicated data already available through the ledger diagnostic.

### ⚠️ Breaking

- This is an intentional hard cut for the current 0.39 line: persisted
  authority records using the older `ic_memory.internal` owner label or an
  ID-only `0-0` internal range are not repaired forward by Canic.
- Direct callers of the removed `MemoryApi` declaration/registration/inspection
  convenience methods must use the managed memory macros or the controller
  ledger diagnostic surface instead.

## 0.39.14 - 2026-05-18 - dual protected store adapter

### 🔧 Changed

- Added `ic-memory::DualProtectedCommitStore`, a generic trait for physical
  stores that expose two protected generation slots.
- Moved authoritative-slot recovery and inactive-slot selection onto the
  generic trait, so adapters get one shared recovery rule without exposing
  their persisted encoding.
- Updated `ic-memory::DualCommitStore` to implement the new trait.
- Updated Canic's physical ABI ledger record to implement the new trait. Canic
  still owns stable-memory IO, header validation, and the current persisted
  encoding, but dual-slot recovery mechanics now come through `ic-memory`.
- Moved protected commit-slot diagnostics onto the generic
  `CommitStoreDiagnostic::from_store(...)` path so adapters expose slot
  presence, generation, validity, and recovery failures through one
  `ic-memory` diagnostic model.
- Added the protected commit recovery diagnostic to Canic's ledger snapshot
  response, making controller diagnostics report which physical slot won
  recovery and whether either slot is corrupt.

### ⚠️ Breaking

- The Canic memory-ledger diagnostic response now includes a
  `commit_recovery` field. This is diagnostic-only state, but clients that
  deserialize the full response shape may need to accept the additional field.

## 0.39.12 - 2026-05-18 - generic allocation validation in core bootstrap

### 🔧 Changed

- Routed Canic's current runtime memory declarations through
  `ic-memory::DeclarationSnapshot`, so duplicate stable keys, duplicate
  allocation slots, and exact duplicate declarations are rejected by the

- Added a production `CanicMemoryManagerPolicy` adapter in `canic-core` that

  `ic-memory::validate_allocations(...)` and the projected historical

  historical error checks.
- Kept the persisted Canic ledger format unchanged in this slice. The new
  projection is an adapter boundary: Canic still owns stable-memory IO and the
  current physical layout, while `ic-memory` owns generic allocation snapshot
  and historical allocation validation mechanics.
- Published the validated allocation set from bootstrap and changed Canic
  memory opening to use `ic-memory::AllocationSession` over the current
  MemoryManager substrate. Opening still returns the same virtual-memory handle
  type, but the path now requires the sealed validated allocation capability.
- Adjusted generic duplicate detection in `ic-memory` so duplicate allocation
  slots are reported before duplicate stable keys. This keeps exact duplicate
  runtime declarations classified as aliasing the same slot in the
  MemoryManager adapter.
- Added coverage proving the Canic physical ledger projects generation and

## 0.39.6 - 2026-05-18 - explicit genesis recovery boundary

### ➕ Added

- Added `LedgerCommitStore::recover_or_initialize(...)` so framework adapters
  can explicitly seed a genesis ledger when the protected physical commit store
  is empty.
- Added `LedgerCompatibility`, `LedgerCompatibilityError`,
  `CURRENT_LEDGER_SCHEMA_VERSION`, and `CURRENT_PHYSICAL_FORMAT_ID` so decoded
  ledger state is checked against supported logical and physical format
  versions before it can become authoritative.
- Added `AllocationLedger::validate_integrity(...)` and `LedgerIntegrityError`
  so decoded ledgers must preserve structural allocation-history invariants:
  unique stable keys, unique allocation slots, valid lifecycle metadata,
  bounded schema metadata history, and valid generation records.
- Added `AllocationBootstrap::initialize_validate_and_commit(...)`, the generic
  bootstrap path that initializes an empty ledger store from a supplied genesis
  ledger, validates declarations, commits the staged generation, and publishes
  validated allocations.
- Added `AllocationBootstrap::reserve_and_commit(...)` and
  `initialize_reserve_and_commit(...)` so explicit reservation operations run
  through policy checks, ledger-history checks, and the protected commit path.
- Added `AllocationBootstrap::retire_and_commit(...)` so explicit retirement
  operations stage tombstones and commit them through the same protected ledger
  path as declarations and reservations.
- Added protected commit recovery diagnostics with per-slot presence,
  generation, validity, authoritative generation, and recovery-error fields.
- `DiagnosticExport` can now include the protected commit recovery diagnostic
  alongside ledger records and generation history.

### 🔧 Changed

- Empty physical commit state is now distinguished from corrupt or partially
  written commit state. Genesis initialization is allowed only for an
  uninitialized store; corrupt stores still fail closed even when a caller
  supplies a genesis ledger.
- `LedgerCommitStore::recover(...)` and `commit(...)` now validate compatibility
  by default. Incompatible ledgers fail closed on recovery, and incompatible
  ledgers are rejected before physical commit writes.
- `LedgerCommitStore::recover(...)` and `commit(...)` now validate ledger
  integrity by default. Malformed ledgers fail closed on recovery, and malformed
  staged ledgers are rejected before physical commit writes.
- Reservation and retirement bootstrap helpers reject policy or ledger-history
  failures before committing, preserving the previous authoritative generation.

## 0.39.5 - 2026-05-18 - generic allocation lifecycle mechanics

### ➕ Added

- `ic-memory` now includes a substrate-neutral dual-slot physical commit model
  for encoded ledger generations. It recovers the highest valid generation,
  ignores corrupt newer slots, and fails closed when no committed slot
  validates.
- Added a generic `LedgerCommitStore` and `LedgerCodec` boundary so
  `ic-memory` can own generation-scoped commit mechanics without owning a
  concrete serialization format or stable-memory IO.
- Added generic reservation staging so reserved allocation records become
  durable ledger facts before a store is attached, and matching declarations
  transition those records from `reserved` to `active` without changing
  `stable_key -> allocation_slot` identity.
- Added explicit retirement staging through `AllocationRetirement`, requiring
  both the stable key and allocation slot and committing the tombstone as a
  generation-scoped ledger operation.
- Added `AllocationBootstrap` as the generic lifecycle pipeline for recovering
  the ledger, validating a sealed declaration snapshot against policy/history,
  staging a generation, committing it through the protected ledger store, and
  returning validated allocations tied to the committed generation.
