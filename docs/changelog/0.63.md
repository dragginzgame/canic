# 0.63 Changelog

## 0.63.3 - 2026-06-09 - CI hardening and script simplification

### Summary

- Simplifies CI and maintainer-script validation after the external tool pin
  cleanup by removing duplicated tag-check workflow setup and sharing
  operations-doc guard helpers.

- Updates first-party GitHub Actions workflow pins to Node 24-compatible majors
  after CI annotations warned about Node 20 action runtime deprecation, and
  adds an optional local maintainer helper for inspecting recent GitHub Actions
  runs and failed-step logs through an authenticated GitHub CLI session.

- Reclassifies the historical post-46 backlog out of numbered release planning:
  archived source material remains archived, optional feature-shaped follow-ups
  now live under design ideas, and the 0.64 backlog draft was removed.

- Keeps runtime behavior, CLI output, Candid, package versions, dependencies,
  lockfiles, generated artifacts, and release-script default versions
  unchanged.

### Changed

- GitHub Actions tag validation now runs through the main `checks` job instead
  of a duplicated tag-only job, while preserving the tag-specific release
  workspace build.

- GitHub Actions now uses `actions/checkout@v6`, `actions/setup-node@v6`, and
  `actions/cache@v5`.

- Operations documentation guard scripts now source a shared shell helper for
  required-file checks, required-text checks, old operations-path rejection, and
  stale git-reference rejection.

- `make install-dev` and `make update-dev` still use root `tool-versions.env`,
  while `scripts/dev/install_dev.sh` now delegates the pinned actionlint and
  ICP CLI versions to the shared installer scripts instead of passing duplicate
  version arguments.

- `scripts/dev/gh-ci.sh` can list recent workflow runs, summarize the latest
  branch run or latest failed branch run, and print failed-step logs when the
  local GitHub CLI session is authenticated.

- The root topology test helper now treats `canic_memory_ledger` as absent from
  the default root bundle, matching the `diagnostics.memory_ledger = true`
  opt-in contract added in 0.63.1.

- Historical post-46 source material stays archived under `docs/design/archive`,
  while optional deployment-group/readiness, active-adoption/import,
  wasm-store-registry, CI-lock/signing, DR-clone, teardown/test-lifecycle, and
  deploy-verification ideas now live under `docs/design/ideas`.

### Cleanup

- Consolidated duplicate cycles installed-deployment error mapping through one
  cycles-module helper.

- Inlined the list command's trivial subtree-anchor accessor.

## 0.63.2 - 2026-06-08 - NNS topology join coverage

### Summary

- Adds joined-topology coverage metrics to `canic nns topology summary` so
  operators can see whether cached nodes and node operators resolve back to
  known node-provider, node-operator, and data-center rows.

- Centralizes external dev-tool pins in root `tool-versions.env` so local setup
  and CI install the same pinned `actionlint` and ICP CLI versions while still
  warning when GitHub advertises a newer ICP CLI release.

- Keeps NNS component cache schemas, runtime canisters, Candid, package
  versions, dependencies, and lockfiles unchanged.

### Changed

- `canic nns topology summary` text output now includes a `join_coverage`
  table with known and unknown counts for node-to-provider, node-to-operator,
  node-to-data-center, operator-to-provider, and operator-to-data-center joins.

- The topology summary JSON report schema is now version `2` and includes
  explicit `nodes_with_known_*`, `nodes_with_unknown_*`,
  `node_operators_with_known_*`, and `node_operators_with_unknown_*` coverage
  counters.

- `make install-dev`, `make update-dev`, GitHub Actions, `require_icp.sh`, and
  the actionlint/ICP CLI installers now read external tool pins from
  `tool-versions.env` instead of carrying duplicate version literals.

- `scripts/ci/install-icp-cli.sh` installs the pinned ICP CLI version, verifies
  that the resolved `icp --version` matches the pin, honors `CARGO_HOME`, and
  emits a non-fatal warning when GitHub's latest `dfinity/icp-cli` release tag
  differs from the pin.

## 0.63.1 - 2026-06-08 - NNS topology refresh and memory ledger opt-in

### Summary

- Adds `canic nns topology refresh` as a one-shot refresh command for every
  cached mainnet NNS report consumed by `canic nns topology summary`.

- Makes the controller-only `canic_memory_ledger` diagnostic opt-in per
  canister role through `diagnostics.memory_ledger`.

- Keeps NNS component cache schemas, package versions, release scripts,
  dependencies, and lockfiles unchanged.

### Added

- `canic nns topology refresh` refreshes cached subnet, node, node-provider,
  node-operator, and data-center reports in one operator command.

- `canic nns topology refresh --dry-run` fetches and validates every topology
  component without replacing component caches.

- `canic nns topology refresh --format json` emits an aggregate refresh report
  with per-component cache path, registry version, fetched timestamp, item
  count, write status, and replacement status.

- `canic nns topology refresh --source-endpoint https://icp-api.io` forwards a
  custom IC API endpoint to each NNS topology component refresh.

- `diagnostics.memory_ledger = true` under
  `[subnets.<name>.canisters.<role>]` enables the memory-ledger recovery
  diagnostic for that role.

### Changed

- README and CLI README NNS inspection examples now advertise
  `canic nns topology refresh` alongside `canic nns topology summary`.

- The shared runtime and wasm-store endpoint bundles now emit
  `canic_memory_ledger` only when the build config emits
  `canic_memory_ledger_enabled`.

- `crates/canic-wasm-store/wasm_store.did` now reflects the default-off
  memory-ledger diagnostic surface.

## 0.63.0 - 2026-06-08 - NNS topology inspection

### Summary

- Adds `canic nns topology summary` as a cached mainnet NNS topology summary
  view.

- Adds filtered `canic nns node list` reads for subnet, subnet kind,
  data-center, node-provider, and node-operator selection.

- Keeps this as a CLI/host inspection expansion: no Cargo package version,
  release-script default version, Candid, runtime canister, package artifact,
  lockfile, or node-cache schema changes.

### Added

- `canic nns topology summary` composes the existing cached subnet, node,
  node-provider, node-operator, and data-center reports into aggregate counts,
  subnet-kind distribution, and per-source registry-version provenance.

- `canic nns node list --subnet <subnet|subnet-prefix>` filters cached nodes
  by assigned subnet principal or prefix.

- `canic nns node list --kind <application|system|unknown>` filters cached
  nodes by assigned subnet kind.

- `canic nns node list --data-center <data-center|data-center-prefix>`
  filters cached nodes by data-center id or prefix.

- `canic nns node list --node-provider <node-provider|node-provider-prefix>`
  filters cached nodes by node-provider principal or prefix.

- `canic nns node list --node-operator <node-operator|node-operator-prefix>`
  filters cached nodes by node-operator principal or prefix.

### Changed

- The node-list filters are applied to the report before text or JSON
  rendering, so automation receives the same filtered node set in both output
  formats.

- README and CLI README NNS inspection examples now advertise
  `canic nns topology summary` and representative filtered
  `canic nns node list` commands.
