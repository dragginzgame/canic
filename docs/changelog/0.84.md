# Canic 0.84 Changelog

## [0.84.10] - 2026-07-11

### 📝 Summary

0.84.10 makes security and runtime financial-history state declarations
truthful. Auth state, replay receipts, cycle top-up events, child funding
ledgers, and ICP-refill records now have exact canonical `*Data` snapshots with
named rows that preserve their stable keys.

This changes internal Rust snapshot and projection types only. Memory IDs,
stable keys, record serialization, auth and replay behavior, refill behavior,
endpoint DTOs, Candid, JSON reports, and persisted bytes are unchanged.

### 🔧 Changed

- Auth singleton state exposes `AuthStateData` around its persisted
  `AuthStateRecord`.
- Replay receipts expose `ReplayReceiptsData` composed of named
  `ReplayReceiptEntryRecord` rows preserving stable slot keys.
- Cycle top-up history and child funding ledgers expose
  `CycleTopupEventsData` and `CyclesFundingLedgerData` with named keyed rows.
- ICP-refill history exposes `IcpRefillRecordsData` with named rows preserving
  stable record keys.
- Cycle and ICP-refill operational projections use the named row forms instead
  of tuples.
- All affected descriptors compile against owner-defined record and snapshot
  name constants.
- Auth, replay, cycle, funding, and refill snapshot restoration is exercised
  only by focused test helpers; no production whole-state import or migration
  surface is added.

### ⚠️ Breaking

- The aspirational singular `ReplayReceiptData` snapshot name is hard-cut in
  favor of the collection-accurate `ReplayReceiptsData` form.
- Internal cycle top-up and ICP-refill entry projections now return named
  records rather than `(key, record)` tuples. Maintained consumers have moved
  without aliases or compatibility wrappers.

### 🧪 Testing

- Targeted descriptor and exact snapshot round-trip coverage passes for auth,
  replay, cycle top-ups, funding ledgers, and ICP-refill records.
- Targeted ICP-refill resume behavior and `canic-core` Clippy pass, along with
  changelog governance, formatting, and patch-hygiene checks.

## [0.84.4] - 2026-07-11

### 📝 Summary

0.84.4 makes the selected ICP environment and the network class baked into a
Wasm artifact separate, explicit values. Named environments now resolve their
declared `icp.yaml` network before Cargo runs, so an environment such as
`staging` backed by the implicit `ic` network builds with mainnet auth settings
instead of silently building as local.

The patch also turns incomplete registered topology into operator-visible
deployment and auth failures. It does not restore legacy state, seed missing
environment records, self-heal registries, adopt retained children, or add a
manual delegation-proof path. Destructive reinstall remains the supported
hard-cut recovery boundary.

### 🩹 Fixed

- The CLI and host install path resolve named ICP environments from
  `icp.yaml`. The implicit `local` and `ic` environments remain available
  without project config; undeclared, duplicate, or incomplete named
  environments fail before compilation.
- Environments backed by `ic` build with the `ic` runtime class. Environments
  backed by `local`, managed, or connected non-mainnet networks build with the
  `local` runtime class.
- Cargo subprocesses receive the resolved build class while the parent process
  retains the selected environment for config, deployment, and artifact
  targeting.
- Local deployment plans require configured bootstrap roles directly attached
  to the subnet. Dynamic placement children remain represented by their pool
  expectations instead of being treated as bootstrap canisters.
- An observed existing deployment with a missing bootstrap registry role now
  fails with `subnet_registry_role_missing`. An unavailable registry query
  remains an observation gap and does not claim registry corruption.
- Auth renewal status now reports `issuer_unregistered` when the issuer is
  absent from the root subnet registry. Medic directs operators to restore the
  affected reinstall dependency closure instead of manually injecting a
  delegation proof.

### CLI

- Build output reports both `environment` and `build network` rather than
  flattening them into one `network` line.
- Build provenance retains the selected environment as its target network and
  adds a `build_environment` input recording both the selected environment and
  resolved build network. The evidence-envelope and provenance payload schema
  versions are unchanged.
- `canic deploy check` can now block an existing verified deployment when a
  successfully observed root registry omits a configured bootstrap role.
- `canic auth renewal status` adds the `issuer_unregistered` aggregate status.

### ⚠️ Breaking

- Automation parsing the human-readable build context must consume the new
  `environment` and `build network` labels instead of the former `network`
  label.
- Consumers that exhaustively match auth renewal statuses must accept
  `issuer_unregistered`, and provenance consumers must tolerate the added
  `build_environment` input.
- Registry incompleteness that previously produced an unobserved-canister
  warning now blocks an existing verified deployment once the registry has
  been observed successfully.

### 🧭 Migration Notes

- Reinstalling a root requires a fresh deployment of its root-owned fleet.
- Reinstalling a hub or placement manager requires reinstalling the children
  whose placement and lifecycle state it owned. Retained children are not
  adopted by injecting their principals through `subnet_index`.
- Reinstalled auth issuers receive a fresh proof through normal root renewal or
  issuer lazy repair after registered topology is restored.

### 🧪 Testing

- Build coverage pins implicit and named environment resolution, mainnet and
  non-mainnet build classes, malformed config rejection, scoped Cargo
  environments, visible context labels, and provenance inputs.
- Deployment and auth coverage distinguishes observed registry omissions from
  observation gaps and pins the new topology-specific status and guidance.

## [0.84.3] - 2026-07-11

### 📝 Summary

0.84.3 hardens two operator boundaries: host-owned single-file persistence and
live observation reporting. Generated Candid artifacts, deployment state and
receipts, release-set manifests, and single-file fleet role mutations now use
one durable sibling-staging write path. Metrics, cycles, and live-list fan-out
retain an explicit result for every requested canister, while malformed
metrics, cycle-history, blob-storage, and auth responses identify the failing
field.

The durability scope remains intentionally bounded. It does not add a journal,
general transaction layer, startup recovery, or multi-file crash recovery.
Persisted canister state, stable-memory IDs and encodings, command forms, and
report schema versions are unchanged.

### 🔧 Changed

- Host-owned single-file outputs are staged beside their destination, synced,
  atomically renamed, and followed by a parent-directory sync.
- Generated Candid artifacts, deployment install state, deployment receipts,
  release-set manifests, and fleet role declaration/attachment use the shared
  durable writer.
- Role rename writes the fleet config before the package manifest and restores
  the original config through the same durable writer if the second write
  fails.
- Metrics and cycles fan-out convert worker panics into per-canister error rows
  instead of omitting the affected canister.
- Cycle reports retain successful tracker samples when live-balance or top-up
  observation fails and expose the supplemental failure in report status and
  error detail.
- Live-list cycle-balance and Canic-version query failures render as `error`;
  legitimate absence of Canic metadata continues to render as `-`.
- Metrics, cycle-history, blob-storage, and auth response parsers distinguish
  invalid JSON, invalid payloads, and malformed fields through typed failures.
  Blob-storage and auth command errors retain the specific parse detail while
  preserving their existing error categories and exit codes.

### ⚠️ Breaking

- The unused `canic_host::cycle_balance::query_cycle_balance_optional` helper
  is removed without an alias; callers use the typed `query_cycle_balance`
  result directly.
- Blob-storage and auth response-parse command error variants now carry parse
  detail instead of being unit variants. No compatibility constructor is
  retained.

### 🧪 Testing

- Host coverage verifies complete replacement writes, parent creation, and
  restoration of the original fleet config when the package-manifest phase of
  role rename fails.
- CLI coverage verifies worker panics remain visible, supplemental cycle-query
  failures preserve tracker data, and malformed operator responses identify
  their exact field.
- Full host and CLI library suites, strict Clippy, formatting, changelog
  governance, and whitespace validation pass for the prepared patch.

## [0.84.2] - 2026-07-10

### 📝 Summary

0.84.2 corrects role-contract applicability across the complete stable-memory
map. Shared topology, environment, replay, cycle/log, funding, and intent state
now resolves for every managed runtime. Auth state, ICP-refill state, and
placement state resolve only for the capabilities and features that use them.

The memory-map audit also restores IDs 86-99 as unallocated Canic
control-plane reserve, matching the established application boundary at ID
100, and makes allocation-owner range validation fail closed. As a pre-1.0
hard cut, replay receipts move from ID 21 to ID 20 and ID 21 becomes
unassigned. The allocation lifecycle/tombstone model and removed-state report
surface are deleted rather than retained as compatibility machinery.

State manifest and audit JSON move to schema version 2. Existing canisters that
carry the old ID-20/21 allocation ledger require a destructive reinstall; no
migration, alias, dual read, or fallback is provided. All other active IDs,
persisted records, encodings, and restore behavior are unchanged.

### 🩹 Fixed

- Root contracts retain canister-pool ID 49 but no longer claim scaling ID 52
  or directory ID 55 unless those capabilities are configured on the root.
- Declared roles and the built-in wasm store now report shared runtime IDs
  11-13, 15-18, 20, 29-32, 34, and 39-42 instead of omitting state used by their
  lifecycle and default endpoint surface.
- Root contracts retain their root-only app-registry ID 14.
- Auth-capable roles report auth-state ID 19. ICP-refill configuration now
  requires the `icp-refill` package feature, and that effective feature reports
  resumable-operation ID 33.
- Scaling roles resolve IDs 49 and 52; directory roles resolve IDs 49 and 55;
  sharding roles resolve IDs 49, 53, 54, and 56.
- Shared pool selection merges capability and feature provenance when more
  than one valid rule selects it.
- State descriptors are keyed separately for the pool, scaling registry, and
  directory registry, allowing strict joins to materialize exact role state.
- Canonical allocation validation now rejects `canic-core` definitions outside
  IDs 11-79 and `canic-control-plane` definitions outside IDs 80-99.
- Downstream application keys start at ID 100; IDs 86-99 are no longer
  accepted as application allocations.
- Allocation definitions now describe active state only. Lifecycle variants,
  retired allocation keys, inactive-allocation findings, removed-state
  descriptors, manifest rows, audit categories, and CLI rendering are removed.

### ⚠️ Breaking

- `StateAllocationKey::CoreRootCapacity` is hard-cut without an alias and
  replaced by `CanisterPool`, `ScalingRegistry`, and `DirectoryRegistry`.
- Root-named core allocation keys are hard-cut into exact runtime/auth/refill
  keys: `CoreRuntimeTopology`, `CoreRuntimeEnvironment`, `CoreAuthState`,
  `CoreReplayReceipts`, `CoreRootAppRegistry`, `CoreRuntimeObservability`,
  `CoreIcpRefillRecords`, and `CoreRuntimeIntent`.
- Replay receipts move from ID 21 to ID 20, reclaiming the old root-replay
  slot; ID 21 is unassigned. Existing canisters with the previous map must be
  destructively reinstalled. There is no migration or compatibility shim.
- State manifest and audit move to schema version 2. The `removed_state` field,
  removed-state audit category, and associated check codes are hard-cut.
- Corrected role contents add shared runtime state to non-root roles, add
  capability-selected auth/refill state, add placement state to non-root hubs,
  and remove unconfigured scaling, directory, or ICP-refill state from roots.
- Application declarations using IDs 86-99 are rejected without a fallback or
  migration shim. Those IDs were accidentally exposed when the framework
  authority was split in 0.40.2 and are restored to Canic reserve before 1.0.

### 🧪 Testing

- Pure policy coverage pins shared runtime, auth, ICP-refill, placement, and
  merged-provenance allocation IDs, including replay receipts at ID 20 with ID
  21 absent from the active map.
- Real package/config resolution verifies `user_hub`, `scale_hub`, and
  `project_hub` materialize only their exact placement-owned domains.
- The built-in wasm-store fixture proves shared runtime IDs plus template/GC
  IDs materialize together.
- Full core, control-plane, host, and CLI library suites pass with the hard cut.

## [0.84.1] - 2026-07-10

### 📝 Summary

0.84.1 removes Canic-owned control flow based on human-readable error and
description fragments. Auth-expiry classification, wasm-store rollover,
registry policy codes, deployment-state safety classification, install blocks,
and operator error categories now consume typed variants or stable machine
codes. Internal PocketIC artifact builds also validate Canic role packages
before receiving the canonical build marker.

The role contract, capability derivation, and package dependency rules are
unchanged; the internal test harness now enforces the existing package
validator. Stable-memory IDs, persisted records, and migrations are unchanged.

### 🔧 Changed

- Delegated token expiry and delegation-certificate expiry remain distinct
  through verifier, access, and public error conversion. Applications can
  inspect the access kind without parsing the denial message.
- Pending chain-key delegation-proof generation has a dedicated
  `AuthProofPending` public code, so retry decisions no longer require its
  operator message.
- Registry policy variants map directly to their existing public policy codes
  before generic internal-error conversion.
- Wasm-store capacity, missing-chunk, hash-mismatch, and missing-manifest
  failures have stable public codes used by managed rollover and metric reason
  selection.
- Deployment planning represents missing local state, installed-network drift,
  and state-read failure with distinct owner-defined assumption kinds.
- Install safety and execution-preflight blocks retain typed reasons through
  the host/CLI boundary.
- Blob-storage invalid cycle values and medic missing-deployment state are
  classified from typed errors rather than their rendered messages.
- Direct local replica destination failures use the structured reject code;
  external ICP CLI stderr matching remains an isolated diagnostic adapter.
- Canister-signature certificate absence is matched through the dependency's
  typed error variant for both root and issuer proof retrieval.
- Deployment conflict/artifact classifiers use exact owner-defined codes;
  observation sources use owner labels; and assumption diagnostics use exact
  key namespaces rather than word fragments.
- Cycle top-up variants, replica query status, HTTP status, Cargo workspace
  manifests, policy exit classes, backup verification kinds, and bootstrap
  artifact kinds are parsed or classified structurally.
- Removed command forms, endpoint options, config fields, and install-state
  fields no longer have anti-resurrection tests. Current behavioral tests use
  typed errors, public codes, or observable state instead of `Display` text.
- The developer installer no longer detects or uninstalls the retired npm ICP
  CLI wrapper; current installation owns only the supported Cargo-installed
  ICP CLI path.
- Active contracts, getting-started guides, release validation, and recurring
  audit definitions now describe only the maintained delegated-token,
  role-attestation, and structural caller-authorization surfaces.
- Internal PocketIC artifact builds validate every Canic-declared role package
  through the host package validator before setting the private canonical build
  marker. Generic non-Canic fixture stubs remain outside role validation.

### ⚠️ Breaking

- Public `ErrorCode` adds `AuthProofPending`, `AuthTokenExpired`,
  `WasmStoreCapacityExceeded`, `WasmStoreChunkMissing`,
  `WasmStoreHashMismatch`, and `WasmStoreManifestMissing`.
- The overloaded deployment assumption key `local_state.root_canister_id` is
  replaced by `.missing`, `.network_mismatch`, and `.read_failed` keys. There
  is no compatibility alias.
- `AccessError` adds distinct delegated-token and delegation-certificate expiry
  variants plus a machine-readable kind.

### CLI

- Blob-storage JSON failures now identify every invalid `--cycles` value with
  the existing invalid-cycles command error code, including zero, malformed,
  negative, and overflowing input.
- Deployment-plan JSON exposes the new exact local-state assumption keys.
- Ordinary diagnostic wording is retained where practical.

### 🧪 Testing

- Typed conversion coverage pins auth expiry, registry policy, wasm-store,
  deployment assumption, install block, blob-storage input, medic state, and
  replica reject classifications.
- Repository-wide scan coverage pins exact finding codes, observation-source
  labels, cycle statuses, numeric HTTP status parsing, TOML workspace
  detection, policy exit-class labels, and backup verification labels.
- Text assertions remain only for maintained renderers, proc-macro/compiler
  diagnostics, source-shape contracts, and isolated external-tool adapters.
- Current-facing source and documentation scans have no references to the
  deleted protected-internal client, predicate, or guard-test surface.
- Canonical wasm-store Candid parsing and the built-in wasm-store build include
  the expanded public error-code surface.
- Host coverage proves internal PocketIC role packages are validated before the
  build marker is granted, and the full PocketIC workspace lane passes with the
  validation enabled.

## [0.84.0] - 2026-07-10

### 📝 Summary

0.84.0 introduces role-aware state contracts. Each canister role is resolved
from its exact configured Cargo package into one typed contract covering
runtime Canic features, configuration capabilities, stable-memory allocations,
and owner-provided state descriptors.

Build, medic, release capability views, state manifest, and state audit now
project the same resolved contract instead of independently reconstructing role
or feature policy.

### ➕ Added

- A single typed role-contract authority for capabilities, public Canic feature
  implications, allocation selection, permanent stable-memory IDs, and
  fail-closed resolution.
- A package-scoped wasm runtime validator supporting one direct,
  unconditional, non-optional normal Canic dependency, including dependency
  renames, workspace inheritance, explicit features, and default-feature
  selection.
- Allocation-keyed state descriptors from core and control-plane owners, plus
  complete registry validation for missing or duplicate descriptors, canonical
  ID agreement, owner agreement, and retired-state bindings.
- Exact optional-state manifests for sharding, blob storage, blob-storage
  billing, root control-plane state, and the built-in wasm store.

### 🔧 Changed

- Canonical Canic builds validate the role contract before Cargo compilation
  and pass one private validation marker to the wasm build.
- State manifest and audit commands resolve local package evidence and return no
  partial manifest for unknown, ambiguous, or rejected roles.
- Medic uses the same package, feature, allocation, and descriptor resolution
  as build and state reporting.
- Release capability labels retain their existing strings but are projected
  from typed resolved capabilities.
- Runtime state summaries join registered memory IDs to owner descriptors
  instead of selecting state from a local role-name table.
- Exported opt-in endpoint macros evaluate features at the Canic facade
  definition boundary, removing the need for mirrored caller-package features.

### 🗑️ Removed

- Independent build-support and medic Canic dependency-feature parsers.
- Local state-manifest role selectors and the raw release capability mapper.
- Redundant blob-probe package-feature forwarding.
- The retired role feature-requirement helper and all replaced compatibility or
  fallback paths.

### ⚠️ Breaking

- Direct authoritative canister wasm Cargo builds are rejected; use
  `canic build <fleet> <role>`.
- Optional, target-specific, transitive, multiply resolved, or
  package-feature-forwarded Canic dependency shapes are rejected rather than
  partially interpreted.
- Medic's missing required-feature failure code changes from
  `role_required_canic_feature_missing` to
  `role_contract_required_feature_missing`.
- State audit may emit blocking `role_contract_*` checks when exact package or
  descriptor resolution fails.

### 🧭 Migration Notes

- Keep one direct normal runtime dependency on `canic` in each role package and
  declare required Canic features on that dependency.
- Replace direct wasm Cargo build automation with the canonical `canic build`
  command.
- Update automation matching the retired medic missing-feature code.
- No stable-state migration is required. State manifest and audit remain schema
  version 1, and stable-memory IDs, persisted record encodings, and migration
  behavior are unchanged.

### 🧪 Testing

- Catalog parity is checked against the actual Canic Cargo feature definitions.
- Supported and rejected package dependency shapes are covered with focused
  host tests.
- Every active allocation is required to have exactly one valid descriptor,
  including optional and built-in state.
- Broad core, control-plane, host, CLI, Clippy, canonical root build, canonical
  blob-storage build, and built-in wasm-store audit validation pass.
