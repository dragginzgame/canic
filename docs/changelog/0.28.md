## 0.28.2 - 2026-04-28 - Delegation Lifecycle Regression

### 📝 Summary

- `0.28.2` locks in the delegated-auth lifecycle repair with focused regression coverage for verifier proof-cache loss.

### 🧪 Testing

- Added a PocketIC regression where a verifier is reinstalled, loses its cached delegation proof, rejects the old token with a proof miss, then accepts a fresh token after signer post-upgrade prewarm re-fans out the proof.
- Moved the verifier reinstall and signer upgrade mechanics into the delegation root harness, keeping the regression readable and the lifecycle reset helpers test-only.
- Recorded the accepted implementation status:
  [docs/design/0.28-delegated-auth-lifecycle/0.28-status.md](../design/0.28-delegated-auth-lifecycle/0.28-status.md)

### 🧹 Cleanup

- Reused the derived role index during verifier-target derivation so role-scoped audiences do not rebuild the registry map for each requested role.
- Marked the delegated-auth lifecycle design as accepted and moved implementation state into the separate status document.

## 0.28.1 - 2026-04-28 - Delegation Verifier Cache Repair

### 📝 Summary

- `0.28.1` closes a delegated-auth lifecycle gap where a signer could keep reusing a valid proof while a verifier had lost the matching cached proof.

### 🩹 Fixed

- Signer runtime prewarm now refreshes root provisioning and verifier fanout even when the local signer proof is still reusable.
- Non-root init and post-upgrade bootstrap now use the same signer prewarm flow, so upgraded signer canisters repopulate verifier caches before marking ready.
- Signer prewarm now uses an internal lifecycle plan instead of constructing token claims as a stand-in for proof provisioning.
- Root now derives delegation verifier fanout from `DelegationAudience` and the registry instead of accepting caller-supplied verifier targets on `DelegationRequest`.
- Root now fails canonical delegation issuance when any required verifier fanout target is missing or failed, and root always caches its own verifier proof copy.
- Removed `include_root_verifier`, unused signer-target provisioning, and the old `DelegationProvisionRequest` shape from the canonical delegation path.
- Added the signed-off delegated-auth lifecycle design note:
  [docs/design/0.28-delegated-auth-lifecycle/0.28-design.md](../design/0.28-delegated-auth-lifecycle/0.28-design.md)
- Removed the need for canister-local lifecycle hooks to call a delegated-auth repair API; signer canisters get the repair through Canic's central lifecycle bootstrap.

## 0.28.0 - 2026-04-27 - Delegation Audience Hard Cut

### 📝 Summary

- `0.28.0` reconciles the issuer-side token reissue API with the new audience model and makes delegation public-key delivery explicit.

### ⚠️ Breaking

- Replaced delegated-token `aud` fields with `DelegationAudience`, using `DelegationAudience::Any` for wildcard verifier audience and `DelegationAudience::Roles(Vec<CanisterRole>)` for role-scoped audience.
- Changed `DelegationApi::ensure_token(...)` and `DelegationApi::reissue_token(...)` to accept `DelegationAudience` directly instead of an iterator of principals.
- Made `shard_public_key_sec1` required on `DelegationRequest`, `DelegationProofInstallRequest`, and `DelegationProvisionRequest`; custom delegation callers must populate it before calling root.

### 🔧 Changed

- Root now forwards the provided shard signer public key during verifier proof installation and caches the same key for its own verifier cache when requested.
- Repair/prewarm verifier pushes now resolve both root and shard public keys before encoding the proof-install payload, so verifier targets do not need to synthesize missing shard key material.
- Audience hashing, replay compaction, stable proof records, and test fixtures now use the same `DelegationAudience` shape.

```rust
let token = DelegationApi::ensure_token(
    existing_token,
    DelegationAudience::Roles(vec![CanisterRole::new("project_hub")]),
)
.await?;
```
