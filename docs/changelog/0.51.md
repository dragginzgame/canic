# 0.51 Changelog

## 0.51.3 - 2026-05-31 - envelope comparison

### Added

- Added a CI-friendly stable envelope comparison command:

  ```text
  canic evidence compare --left <path> --right <path>
  ```

- `canic evidence compare` compares stable `EvidenceEnvelopeV1` fields,
  including schema identity, command provenance, target identity, input
  fingerprints, payload schema, payload SHA-256, structured summary, and
  `exit_class`.
- The compare command supports text output by default and machine-readable JSON:

  ```text
  canic evidence compare --left <path> --right <path> --format json
  ```

### Safety

- Envelope comparison ignores `generated_at`, `canic_version`, and the nested
  command-specific `payload` body. It still compares `payload_sha256`, so CI can
  detect payload identity changes without parsing unstable payload DTO fields.
- The command is read-only: it reads existing envelope files and does not
  inspect live deployments, mutate config, or write artifacts.

### Testing

- `cargo test -p canic-cli evidence --lib --locked`
- `cargo test -p canic-cli usage_lists_command_families --lib --locked`
- `cargo test -p canic-cli command_family_help_returns_ok --lib --locked`

## 0.51.2 - 2026-05-31 - exit-class consistency

### Changed

- Evidence envelope exit-class precedence is now centralized in
  `canic-host::evidence_envelope`.
- Adoption-report and deployment-check envelope emitters now derive
  `exit_class` from the same structured-summary precedence.
- Deployment-check envelopes now report `evidence_conflict` when blocked safety
  findings are caused by contradictory evidence, instead of reducing those cases
  to generic `blocked_by_policy`.
- Evidence-envelope architecture docs now include the normative exit-class
  precedence order and CI policy guidance for warnings, blockers, missing
  required evidence, and conflicts.

### Safety

- `success_with_warnings` remains process-success by default. CI policy can
  still choose to fail warnings outside Canic.
- `missing_or_stale_evidence` remains a review signal unless a command
  explicitly classifies it as `missing_required_evidence`.

### Testing

- `cargo test -p canic-host evidence_envelope --lib --locked`
- `cargo test -p canic-cli deployment_check_envelope --lib --locked`

## 0.51.1 - 2026-05-31 - evidence fingerprint hardening

### Changed

- Shared evidence file fingerprinting now lives in
  `canic-host::evidence_envelope`, so adoption-report and deployment-check
  envelopes use the same path, hash, size, timestamp, and schema handling.
- `InputFingerprintV1` now includes `path_display`, allowing automation to
  distinguish relative paths from absolute paths that Canic intentionally
  redacted.
- Evidence inputs under the selected config/project root render as relative
  paths. Absolute paths outside that root are omitted from envelope output
  instead of being copied into CI artifacts.
- Normalized command provenance records redactions when an evidence path was
  outside the selected config root.
- Payload SHA-256 generation now uses the shared envelope helper for both
  adoption reports and deployment checks.
- Fleet role declarations now require an explicit package path:

  ```toml
  [roles.store]
  kind = "canister"
  package = "canisters/store"
  ```

- Generated standalone configs still remain declared-only, but now declare the
  standalone package explicitly with `package = "."`.
- Checked-in `canic.toml` role declarations must point at a real package
  manifest. Workspace governance rejects `[roles.<role>] package` paths that do
  not contain `Cargo.toml`.
- Test and special-purpose configs now use concrete package paths instead of
  placeholder role-name directories.

### Safety

- The hardened fingerprint path does not change command mutation behavior:
  adoption reports and deployment checks remain passive, and envelope output
  still does not mutate config, topology, controllers, artifacts, deployment
  state, package manifests, inventories, or canister lifecycle.
- Package-less role declarations now fail config validation instead of being
  treated as non-package or implicit roles.
- Role declaration package paths now fail governance if they drift to missing
  or placeholder package directories.
- Adoption report package state no longer exposes a `not-package-backed` role
  state. Observed-only findings now report `undeclared-role`, keeping
  package-backed role declarations as the only configured role model.

### Testing

- `cargo test -p canic-host evidence_envelope --lib --locked`
- `cargo test -p canic-cli adoption_report --lib --locked`
- `cargo test -p canic-cli deployment_check_envelope --lib --locked`

## 0.51.0 - 2026-05-31 - evidence envelope foundation

### Added

- Added `canic-host::evidence_envelope` with the stable outer envelope model:
  `EvidenceEnvelopeV1`, `ExitClassV1`, `EvidenceTargetV1`,
  `PayloadSchemaRefV1`, `InputFingerprintV1`, `CommandProvenanceV1`,
  `EvidenceSummaryV1`, and `EvidenceMessageV1`.
- Added stable envelope JSON output for passive adoption reports:

  ```text
  canic fleet adoption report <fleet> --profile <profile> --format envelope-json
  ```

- `--format envelope-json` wraps the raw adoption payload with fleet/profile
  target identity, command provenance, source config/input fingerprints,
  experimental payload schema identity, payload hash, structured
  warnings/blockers/evidence gaps/conflicts, and an automation-friendly exit
  class.
- Added stable envelope JSON output for passive deployment checks:

  ```text
  canic deploy check <deployment> --format envelope-json
  ```

- `--format envelope-json` wraps the raw `DeploymentCheckV1` payload in
  `EvidenceEnvelopeV1` with deployment/fleet target identity, command
  provenance, config fingerprint metadata, internal payload schema identity,
  payload hash, structured safety warnings/blockers/evidence gaps/conflicts,
  and an automation-friendly exit class.
- Added the maintained architecture note
  `docs/architecture/evidence-envelopes.md`.

### Changed

- `canic fleet adoption report ... --format json` remains the raw experimental
  `AdoptionReportV1` payload. It is not silently wrapped.
- Existing deployment-check JSON remains raw `DeploymentCheckV1`.
- Added an explicit attached `payload_limit_probe` test fleet config so the
  PocketIC payload-limit fixture no longer depends on generated declared-only
  standalone config.
- Updated root-suite fixture configs to declare the current `[fleet]` and
  `[roles.<role>]` shape required by the 0.49 role lifecycle hard cut.
- Internal PocketIC Wasm artifact builders now invoke Cargo with `--locked` so
  release validation cannot silently drift `Cargo.lock`.

### Safety

- Envelope output does not make adoption reporting active. It does not mutate
  fleet config, package manifests, topology, deployments, controllers,
  artifacts, adoption state, or canister lifecycle, and it does not perform
  additional live discovery.
- Envelope output does not change the deployment-check mutation boundary. The
  command remains passive and does not install Wasm, update controllers, write
  deployment state, attach topology, or register artifacts.
