# 0.81 Changelog

## 0.81.6 - 2026-07-04 - Runtime metadata coverage

### 📝 Summary

- Adds live PocketIC coverage for decoded `canic_runtime_status` feature,
  timer, and state metadata on root and child canisters.

- Expands `canic inspect` renderer coverage so text and JSON tests pin decoded
  timer, state-domain, and recent-failure sections instead of only the top-level
  runtime status.

- Makes `canic inspect` text output show decoded metadata rows for enabled
  features, timers, state domains, and recent failures without dumping failure
  summaries or persisted state values.

### 🔧 Changed

- Root and child runtime-status assertions now verify that feature rows are
  non-empty, deterministic, `operator_only`, and sourced from
  `compile_feature`.

- Live runtime-status assertions now check timer rows for deterministic
  subsystem/name ordering, enabled/registered metadata, and sanitized text.

- Runtime state summaries are now asserted as metadata-only live output:
  schema version, declared domains, stable-memory IDs for stable domains, and
  no value/count-style memory detail.

- `canic inspect` tests now cover decoded timer count, recent-failure count,
  state-domain count, JSON timer metadata, JSON state-domain metadata, and the
  recent-failure redaction marker.

- `canic inspect` text output now includes metadata-only rows such as
  `enabled_features`, `timer`, `state_domain`, and `recent_failure` after a
  guarded runtime-status response decodes successfully.

### 🧭 Migration Notes

- This slice does not add new runtime endpoint surfaces or public health
  fields. It keeps the new inspect text rows sourced from the guarded
  `canic_runtime_status` response.

### 🧪 Testing

Validated with:

```bash
cargo clippy --locked -p canic-tests --test root_suite -- -D warnings
```

## 0.81.5 - 2026-07-04 - Runtime feature status projection

### 📝 Summary

- Adds a deterministic guarded feature inventory to `CanicRuntimeStatus`.

- Mirrors decoded runtime feature inventory in `canic inspect` reports so text
  and JSON consumers can see whether the queried canister was built with known
  Canic feature gates.

### 🔧 Changed

- `canic_runtime_status` now populates `features` from a centralized
  compile-feature registry for the known `canic-core` feature gates:

```text
auth-chain-key-ecdsa
auth-chain-key-root-sign
auth-delegated-token-verify
auth-issuer-canister-sig-create
auth-issuer-canister-sig-verify
auth-root-canister-sig-create
auth-root-canister-sig-verify
blob-storage
blob-storage-billing
sharding
```

- Each feature row is emitted with deterministic ordering,
  `visibility = operator_only`, and `source = compile_feature`.

- `canic inspect` text output now includes a `features: <count>` line when the
  runtime status response decodes successfully. JSON output continues to carry
  the decoded typed runtime status, including the feature rows.

### 🧭 Migration Notes

- The feature inventory is runtime-observed compile metadata. It is
  diagnostic-only and does not replace medic's config-driven feature checks or
  authorize any mutation.

- The feature list is guarded by the existing `canic_runtime_status` endpoint
  policy; it is not exposed through public health.

### 🧪 Testing

- Focused tests pin feature ordering, enabled bits from `cfg!(feature = ...)`,
  `operator_only` visibility, `compile_feature` source attribution, text
  feature counts, and JSON feature-row preservation.

## 0.81.4 - 2026-07-04 - Runtime introspection privacy hardening

### 📝 Summary

- Completes the runtime-status field visibility table so every top-level
  `CanicRuntimeStatus` field has an explicit visibility classification.

- Bounds and sanitizes runtime-facing diagnostic labels before they reach
  guarded introspection DTOs.

- Removes raw rendered Candid fallback text from `canic inspect` reports while
  preserving explicit presence metadata for diagnostics and automation.

### 🔧 Changed

- `CanicRuntimeStatus.visibility` now includes entries for schema/version,
  identity, build, feature, topology, timer, state, recent-failure, readiness,
  status, and visibility fields. Public-safe fields stay limited to schema and
  observation timestamp metadata; topology remains controller-only.

- Timer labels projected into `CanicTimerStatus` are sanitized for control
  characters and bounded before rendering through `canic_runtime_status`.
  Metrics storage remains unchanged so low-cardinality metric keys preserve
  their existing behavior.

- Recent failures now bound and sanitize `subsystem` and `code` in addition to
  summary and correlation-id fields. The `redacted` marker is set when any of
  those fields is sanitized or truncated.

- `canic inspect` JSON/text reports now expose
  `runtime_status.response_candid_present` instead of the raw
  `runtime_status.response_candid` text. Typed decoding from `response_bytes`
  is unchanged.

### 🧭 Migration Notes

- Consumers that read `canic inspect --json` should use
  `runtime_status.response_candid_present` as the fallback-presence signal.
  Raw rendered fallback Candid is no longer part of the inspect report.

- Runtime status timer and recent-failure output is still diagnostic metadata
  only; these changes do not add repair, renewal, state inspection, or mutation
  behavior.

### 🧪 Testing

- Focused tests pin full runtime-status visibility coverage, timer-label
  bounding, recent-failure metadata redaction, inspect JSON/text response
  metadata, and the absence of raw fallback Candid in inspect reports.

Validated with:

## 0.81.2 - 2026-07-04 - Typed runtime status decoding

### 📝 Summary

- Makes runtime-introspection DTO enums compatible with the Candid wire labels
  emitted by `CandidType` while preserving the snake_case JSON contract.

- Teaches `canic inspect` to decode typed `CanicRuntimeStatus` from ICP
  `response_bytes` when present, so report status and exit behavior come from
  the runtime DTO instead of rendered Candid text.

### 🩹 Fixed

- Runtime-introspection enums now accept their Rust/Candid variant labels during
  deserialization. This fixes typed Candid roundtrips for health, readiness,
  runtime status, timer status, failure severity, check status, diagnostic
  severity, field visibility, and state-domain status enums.

- `canic inspect` now rejects invalid `response_bytes` hex instead of treating
  malformed bytes as a successful metadata-only response.

### 🔧 Changed

- `canic inspect` embeds the decoded runtime status in JSON/text reports when
  `response_bytes` is available, while still preserving `response_format`,
  `response_bytes_present`, and optional `response_candid` metadata.

- The top-level inspect report status now reflects the decoded runtime status.
  A decoded `runtime_status = failing` renders the report, exits with code `1`,
  and suppresses duplicate stderr.

- `degraded`, `ok`, and `unknown` runtime statuses remain successful report
  retrievals for the 0.81 CLI contract.

### 🧭 Migration Notes

- The JSON enum output remains snake_case. The Candid compatibility change only
  broadens deserialization to accept the labels already emitted by the Candid
  wire representation.

- `response_candid` remains fallback/display metadata. It is not parsed to
  infer status.

### 🧪 Testing

- Focused tests cover runtime enum Candid roundtrips, typed inspect decoding
  from `response_bytes`, invalid hex rejection, report status rendering,
  failing-status exit behavior, and top-level duplicate-stderr suppression.

Validated with:

## 0.81.1 - 2026-07-04 - Runtime inspection handoffs

### 📝 Summary

- Connects deployment-scoped medic diagnostics to the new runtime inspection
  surface without making medic perform hidden runtime fanout.

- Tightens `canic inspect` report metadata so automation can distinguish
  runtime-observed Candid response bytes from optional rendered Candid text.

### 🔧 Changed

- Deployment registry observations in `canic medic deployment <deployment>` now
  point to one explicit runtime inspection command when a deployment registry is
  already observed.

- Medic prefers `canic inspect deployment <deployment> --role root` when a root
  role is present, falls back to the first deterministic role otherwise, and
  falls back to `canic inspect canister <principal>` when registry entries do
  not carry role metadata.

- `canic inspect` runtime-status payloads now include:

```text
response_format = "candid"
response_bytes_present = true|false
response_candid = optional rendered Candid text
```

- `canic inspect` now accepts ICP JSON responses that contain
  `response_bytes` even when `response_candid` is absent, preserving the
  Candid boundary without fabricating a decoded local status.

### 🧭 Migration Notes

- Medic remains diagnostic-only. The new runtime-inspection handoff is a
  `next` action only; medic still does not call runtime introspection endpoints
  or fan out across deployment roles.

- The inspect report remains runtime-observed output from one explicit target.
  `response_bytes_present` is metadata about the endpoint response, not a local
  revalidation or deployment-safety claim.

### 🧪 Testing

- Focused tests cover medic next-action selection for role-based and
  direct-canister runtime inspection, Candid fallback extraction,
  `response_bytes` presence reporting, text rendering, and JSON report field
  stability.

Validated with:

```bash

cargo test --locked -p canic-cli medic
```

## 0.81.0 - 2026-07-04 - Runtime introspection

### 📝 Summary

- Starts the 0.81 runtime-introspection line with a guarded, typed, read-only
  status surface for deployed Canic canisters.

- Keeps runtime observation separate from deployment planning, deployment truth,
  repair, renewal, stable-memory inspection, and mutation.

### ➕ Added

- Added `canic_health`, `canic_readiness`, and `canic_runtime_status` endpoint
  constants and endpoint macro generation for Canic-owned runtime status
  queries.

- Added typed runtime DTOs for health, readiness, runtime status, build/runtime
  identity, feature visibility, timer status, state summary metadata, and
  redacted recent failures.

- Added guarded runtime status construction that projects existing timer metric
  registrations and 0.80 state declarations into deterministic metadata-only
  status sections.

- Added a bounded heap-only recent-failure ring for redacted runtime status
  summaries. It is not stable state and is not a durable log.

- Added top-level CLI inspection forms:

```bash
canic inspect canister <principal>
canic inspect canister <principal> --json
canic inspect deployment <deployment> --role <role>
canic inspect deployment <deployment> --role <role> --json
```

- Added CLI report source labels so runtime-observed status remains distinct
  from direct CLI targets and installed-deployment target resolution.

### 🔧 Changed

- Query-surface and replay-policy manifests now include the runtime
  introspection endpoints as read-only query endpoints.

- CLI help now distinguishes top-level `canic inspect` live runtime status from
  `canic deploy inspect` raw deployment-truth artifact inspection.

### 🧭 Migration Notes

- This patch is read-only runtime introspection. It does not install, upgrade,
  create canisters, change controllers, register topology, write deployment
  truth, update installed deployment records, mutate stable memory, run
  migrations, repair timers, renew auth proofs, rotate keys, or change feature
  flags.

- Runtime status exposes metadata only. It does not expose secrets, session
  material, token material, private keys, raw application records, or
  stable-memory values.

### 🧪 Testing

- Focused tests cover runtime endpoint protocol surface, query/replay
  classification, CLI inspect parsing and rejected forms, runtime-observed
  report rendering, command-family help, and the live-inspect versus
  deployment-artifact help distinction.
