# 0.81 Changelog

## 0.81.1 - 2026-07-04 - Runtime inspection handoffs

### 📝 Summary

- Connects deployment-scoped medic diagnostics to the new runtime inspection
  surface without making medic perform hidden runtime fanout.

- Tightens `canic inspect` report metadata so automation can distinguish
  runtime-observed Candid response bytes from optional rendered Candid text.

### 🔧 Changed

- Deployment registry observations in `canic medic deployment <deployment>` now
  point to one explicit runtime inspection command when a deployment registry is
  already observed.

- Medic prefers `canic inspect deployment <deployment> --role root` when a root
  role is present, falls back to the first deterministic role otherwise, and
  falls back to `canic inspect canister <principal>` when registry entries do
  not carry role metadata.

- `canic inspect` runtime-status payloads now include:

```text
response_format = "candid"
response_bytes_present = true|false
response_candid = optional rendered Candid text
```

- `canic inspect` now accepts ICP JSON responses that contain
  `response_bytes` even when `response_candid` is absent, preserving the
  Candid boundary without fabricating a decoded local status.

### 🧭 Migration Notes

- Medic remains diagnostic-only. The new runtime-inspection handoff is a
  `next` action only; medic still does not call runtime introspection endpoints
  or fan out across deployment roles.

- The inspect report remains runtime-observed output from one explicit target.
  `response_bytes_present` is metadata about the endpoint response, not a local
  revalidation or deployment-safety claim.

### 🧪 Testing

- Focused tests cover medic next-action selection for role-based and
  direct-canister runtime inspection, Candid fallback extraction,
  `response_bytes` presence reporting, text rendering, and JSON report field
  stability.

Validated with:

```bash
cargo fmt --all
cargo test --locked -p canic-cli inspect
cargo test --locked -p canic-cli medic
cargo clippy --locked -p canic-cli --lib -- -D warnings
git diff --check
```

## 0.81.0 - 2026-07-04 - Runtime introspection

### 📝 Summary

- Starts the 0.81 runtime-introspection line with a guarded, typed, read-only
  status surface for deployed Canic canisters.

- Keeps runtime observation separate from deployment planning, deployment truth,
  repair, renewal, stable-memory inspection, and mutation.

### ➕ Added

- Added `canic_health`, `canic_readiness`, and `canic_runtime_status` endpoint
  constants and endpoint macro generation for Canic-owned runtime status
  queries.

- Added typed runtime DTOs for health, readiness, runtime status, build/runtime
  identity, feature visibility, timer status, state summary metadata, and
  redacted recent failures.

- Added guarded runtime status construction that projects existing timer metric
  registrations and 0.80 state declarations into deterministic metadata-only
  status sections.

- Added a bounded heap-only recent-failure ring for redacted runtime status
  summaries. It is not stable state and is not a durable log.

- Added top-level CLI inspection forms:

```bash
canic inspect canister <principal>
canic inspect canister <principal> --json
canic inspect deployment <deployment> --role <role>
canic inspect deployment <deployment> --role <role> --json
```

- Added CLI report source labels so runtime-observed status remains distinct
  from direct CLI targets and installed-deployment target resolution.

### 🔧 Changed

- Query-surface and replay-policy manifests now include the runtime
  introspection endpoints as read-only query endpoints.

- CLI help now distinguishes top-level `canic inspect` live runtime status from
  `canic deploy inspect` raw deployment-truth artifact inspection.

### 🧭 Migration Notes

- This patch is read-only runtime introspection. It does not install, upgrade,
  create canisters, change controllers, register topology, write deployment
  truth, update installed deployment records, mutate stable memory, run
  migrations, repair timers, renew auth proofs, rotate keys, or change feature
  flags.

- Runtime status exposes metadata only. It does not expose secrets, session
  material, token material, private keys, raw application records, or
  stable-memory values.

### 🧪 Testing

- Focused tests cover runtime endpoint protocol surface, query/replay
  classification, CLI inspect parsing and rejected forms, runtime-observed
  report rendering, command-family help, and the live-inspect versus
  deployment-artifact help distinction.
