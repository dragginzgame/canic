## 0.34.4 - 2026-05-11 - Runtime timer consistency

### 🩹 Fixed

- Fixed non-root cycle-tracker startup so the immediate startup sample only
  records the local cycle balance. The first auto-top-up policy check now runs
  from the delayed workflow timer, preventing cycle requests from being issued
  during IC init mode.

### 🔧 Changed

- Standardized delayed background workflow startup on the shared 30-second
  init delay for log retention, intent cleanup, cycle top-up policy evaluation,
  and pool scheduling.
- Removed the separate pool init-delay constant and routed pool scheduling
  through the shared background workflow init delay.
- Moved the cycle tracker endpoint emitter out of topology wiring and into a
  dedicated `canic_emit_cycle_tracker_endpoints!()` macro, with shared runtime
  and `wasm_store` bundles opting into the standard cycle tracker directly.

### 🧪 Testing

- Rebuilt the CLI and reran the local demo install path to confirm the init-mode
  cycle request regression is gone.

## 0.34.3 - 2026-05-11 - Endpoint JSON and module hash visibility

### ⚠️ Breaking

- Changed `canic endpoints --json` from argument strings to structured endpoint
  DTOs with Candid declarations, endpoint modes, arguments, returns,
  cardinality, and resolved named record/variant shapes. Consumers that parsed
  the old `arguments: string[]` field should switch to the new nested type tree.
- Removed the public `canic build <canister-name>` command. Use fleet-aware
  `canic install <fleet>` for normal operator builds, or `icp build --all` for
  project build-hook execution.
- Removed the unused `source_snapshot.wasm_hash` manifest output and restore
  summary wasm-hash completeness fields. Use `source_snapshot.module_hash` for
  IC-reported module identity and `source_snapshot.checksum` for durable backup
  artifact integrity.

### ➕ Added

- Added structured endpoint type JSON from `canic endpoints --json`, including
  endpoint modes, arguments, returns, cardinality, and resolved named
  record/variant shapes.
- Added module identity results to `canic backup inspect` as
  `MODULE_HASH`/`expected_module_hash`, so restore planning can show the IC code
  identity it expects to operate on.
- Added a live module identity result to `canic list <fleet>`: default output
  shows an 8-character `MODULE` prefix before `CANISTER_ID`, while `--verbose`
  shows the full `MODULE_HASH`; terminals subtly color rows whose module differs
  from the first visible app/service hash.
- Added the first executable `canic backup create <fleet>` path. Without
  `--dry-run`, the command now writes the backup plan and execution journal,
  proves direct operator authority with ICP canister-status preflights, runs the
  ordered stop/snapshot/start/download/verify/finalize phases, and writes the
  final manifest plus durable artifact journal.
- Added `canic metrics <fleet>` for fleet-wide cycle tracker summaries. The
  command queries `canic_cycle_tracker` on every registered canister, compares
  the newest sample with the best available baseline for the requested window,
  and reports partial coverage when the tracker log does not contain a true
  boundary sample.
- Added the standard `canic_cycle_tracker` query to the canonical
  `wasm_store` endpoint bundle and checked-in `wasm_store.did`, so fleet
  metrics can include the store without a role-specific fallback.

#### CLI surface

- `canic backup create <fleet>` now executes by default; pass `--dry-run` to
  keep the previous plan-only behavior.
- `canic backup create` summary output now includes `MODE`, `STATUS`, and
  `EXECUTED` columns alongside the existing fleet, scope, target, operation,
  and output-directory fields.
- `canic backup status` prefers the persisted backup execution journal for
  plan-backed layouts, so in-progress executable backups report execution
  progress instead of falling through to the lower-level artifact journal.
- `canic backup list` can show plan-backed layouts as `dry-run`, `running`,
  `failed`, or `complete` depending on execution-journal and manifest state.
- `canic metrics <fleet>` defaults to `--since 24h` and prints `ROLE`,
  `CANISTER_ID`, `STATUS`, `SAMPLES`, `COVERAGE`, `WINDOW`, `LATEST`, `DELTA`,
  and `PER_H`.
- `canic metrics <fleet> --json` emits the structured fleet report, including
  requested window, generated timestamp, per-canister sample counts, latest and
  baseline timestamps/cycles, delta, hourly rate, coverage status, and per-row
  errors.
- `canic metrics <fleet> --since <duration>` accepts compact durations such as
  `30m`, `1h`, `6h`, `24h`, and `7d`; `--limit <entries>` controls the maximum
  cycle tracker samples fetched per canister.

### 🔧 Changed

- Made plain `canic endpoints` output valid per-method Candid declarations
  instead of compact method-name/argument-only signatures.
- Parsed endpoint interfaces through the Candid parser so aliases, variants,
  records, options, vectors, return values, and query/update modes are available
  without downstream string parsing.
- Preserved registry-reported wasm module hashes in backup topology records and
  dry-run targets so operator-facing backup and restore summaries carry IC code
  identity alongside snapshot artifact checksums.
- Made root installs pass the root wasm module hash into root init so the root
  subnet registry entry can expose module identity without `canic list` falling
  back to slow management-status calls.
- Changed install artifact builds to call the host builder directly instead of
  re-entering the public CLI through `canic build`.
- Changed test/bootstrap wasm-store build helpers and packaged downstream
  verification to call the host artifact builder directly instead of depending
  on an internal `canic build` command.
- Centralized SHA-256 byte, hex, wasm-hash, and hex-decode helpers under
  `canic-cdk::utils::hash`, with the older wasm helper now delegating to that
  shared implementation.
- Kept backup artifact checksums separate from module hashes: module hashes
  identify installed IC code, while checksums verify downloaded snapshot
  artifacts before restore.
- Kept the new live backup execution path honest about authority: until
  root-mediated backup endpoints exist, live `canic backup create` uses direct
  operator control with `CrashConsistent` quiescence instead of claiming
  root-coordinated authority.

### 🧪 Testing

- Verified structured endpoint parsing and JSON rendering with targeted
  `canic-cli` endpoint tests.
- Verified module-hash propagation with `canic-backup` library tests and
  `canic-cli` backup command tests.
- Verified the new backup runner with a fake host executor that drives the full
  plan through manifest finalization and layout integrity verification.
- Verified the metrics duration parser, JSON parser, Candid text parser, and
  partial-window summarization with focused `canic-cli` unit tests.
- Verified the canonical `wasm_store.did` cycle-tracker surface with a

## 0.34.0 - 2026-05-10 - Backup plan foundation

### 📝 Summary

- Started the topology-aware backup/restore rework from the 0.34 design by
  adding `canic-backup::plan`.
- Added typed backup plan, target, operation, authority, quiescence, and receipt
  DTOs so backup execution can move away from the old canister-loop shape.
- Split backup plan validation from execution readiness: dry-run/planning can
  carry `Proven`, `Declared`, or `Unknown` authority evidence, while mutating
  execution requires proven control and snapshot read authority for every
  selected target.
- Added target-scoped control and snapshot-read authority preflight receipts
  that can upgrade declared/unknown plans into executable plans only when every
  selected target has proven authority.
- Added typed control and snapshot-read authority preflight request DTOs derived
  from `BackupPlan`, giving future root coordination and host authority adapters
  a stable input contract.
- Added typed topology and quiescence preflight request/receipt DTOs plus
  execution-gate validation for topology drift, target-set changes, policy
  mismatches, and rejected quiescence.
- Added a full execution preflight receipt bundle so future backup execution can
  apply authority receipts and validate topology/quiescence gates through one
  typed boundary.
- Added `preflight_id`, `validated_at`, and `expires_at` to preflight receipts
  and the execution preflight bundle, and made receipt application reject stale,
  not-yet-valid, or cross-preflight evidence.
- Added `canic-backup::execution` with a model-only backup execution journal
  built from `BackupPlan` phases, including preflight acceptance, ordered
  operation transitions, durable operation receipts, retryable failures, resume
  summaries, and `restart_required` tracking after stops.
- Added typed preflight receipt-bundle acceptance to the execution journal so
  mutation cannot be unblocked by a bundle from a different plan.
- Added `BackupLayout` read/write support for
  `backup-execution-journal.json`, keeping phase execution progress separate
  from the existing artifact download journal.
- Added `BackupLayout` read/write support for `backup-plan.json` so future
  backup runners can resume against the exact validated plan instead of
  reconstructing the operation graph.
- Added execution-layout integrity verification that rejects a persisted
  execution journal when its plan/run ids or operation graph no longer match
  the stored `backup-plan.json`.
- Added the first `canic backup create <fleet> --dry-run` CLI path, including
  optional `--subtree <role-or-principal>` planning, installed-fleet registry
  discovery, persisted `backup-plan.json`, persisted
  `backup-execution-journal.json`, and a compact dry-run summary table while
  keeping real mutation disabled.
- Made `canic backup list` include plan-only dry-run directories as
  `STATUS=dry-run`, using the persisted plan id as `BACKUP_ID` and planned
  target count as `MEMBERS`.
- Made `canic backup status --dir <dry-run-dir>` understand dry-run
  `backup-plan.json` plus `backup-execution-journal.json` layouts and report
  execution-journal progress while `--require-complete` still rejects them as
  non-backups.
- Added `canic backup inspect --dir <dry-run-dir>` with table and JSON output
  for plan metadata, selected targets, authority evidence, operation order, and
  execution-journal state.
- Added a `#` column to `canic backup list` so operators can refer to visible
  backup rows by a short ordinal as well as by `BACKUP_ID`.
- Added fail-closed validation for topology scope, root exclusion, selected
  subtree resolution, execution authority, snapshot read authority, and
  preflight-before-mutation ordering.
- Added a registry-backed plan builder that expands explicit subtrees and
  non-root fleet scopes into ordered phases: preflights, top-down stop,
  snapshot creation, bottom-up start, artifact download, verification, and
  manifest finalization.
- Added backup selector resolution for principals and unambiguous roles, with
  fail-closed handling for missing or ambiguous role selectors.
- Kept live `canic snapshot download` behavior unchanged for this slice; the new
  model is a contract for the next CLI/root-coordination implementation work.
