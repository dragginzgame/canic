# 0.61 Changelog

## 0.61.19 - 2026-06-06 - ICP refill shared replay reservation decisions

### Summary

- Wired fresh ICP refill execution through shared replay receipt reservation
  decisions.

- `canic_icp_refill` remains a release blocker until refill transfer and notify
  effects are marked in the shared receipt before external calls.

### Changed

- `execute_manual_refill` now reserves a shared replay receipt before creating
  or advancing a fresh refill record.

- Committed ICP refill replay receipts now decode and return the cached
  `IcpRefillResponse` instead of re-entering the refill transfer/notify path.

- Actor mismatch, payload mismatch, in-progress, expired, recovery-required,
  and terminal-failed shared receipt decisions now map to public conflict
  errors for ICP refill requests.

- Terminal refill records commit their response into the shared replay receipt.

- Resumable refill records, such as requested transfer state or retryable
  transfer/notify failures, abort the temporary shared receipt so existing
  retry behavior remains unchanged until external-effect marking and recovery
  semantics are added.

### Notes

- No CLI commands, flags, output columns, or JSON report shapes change in this
  patch.

- This patch intentionally does not graduate `canic_icp_refill` in the replay
  manifest. Remaining work is to mark ledger transfer and CMC notify effects
  in flight before the external calls, preserve uncertain outcomes as recovery
  required, and then reclassify the endpoint when the full replay lifecycle is
  covered.

### Testing

## 0.61.10 - 2026-06-05 - Root auth-material replay recovery

### Summary

- Root role-attestation and internal-invocation proof issuance now preserve
  replay receipts when a threshold-ECDSA signing boundary has been crossed and
  the result becomes uncertain.

- `canic_request_role_attestation` and
  `canic_request_internal_invocation_proof` are now marked implemented in the
  replay policy manifest.

### Changed

- Split auth-material signing into prepare/sign phases. Preparation resolves
  the attestation key, warms the cached public key, and computes the message
  hash before cost reservation and before the external signing boundary.

- Prepared role-attestation and internal-invocation proof signing still require
  `CostGuardPermit`.

- Root auth-material execution now marks a `ThresholdEcdsaSign` external effect

  receipt `RecoveryRequired(ExternalEffectStatusUnknown)`.

  replay receipt `RecoveryRequired(ResponseCommitFailed)`.

- Generic root replay abort now removes only receipts that are still
  `Reserved`. Receipts already marked in-flight or recovery-required survive
  the generic execution-error path.

## 0.61.8 - 2026-06-05 - Root replay cross-variant guard

### Summary

- Tightened shared root replay receipt handling so a caller cannot reuse the
  same request id across different root capability command kinds.

- Root replay now checks for existing non-expired receipts with the same replay
  actor and operation id before evaluating the current command kind. If such a
  receipt belongs to another command kind, the new request is rejected as a
  replay conflict instead of being treated as a fresh operation.

### Changed

- Added a replay receipt store query for receipts matching actor and operation
  id while excluding the current command kind.

- The root replay workflow uses that query before normal same-command replay
  evaluation. Live cross-command matches return duplicate conflict; expired-only
  cross-command matches preserve the expired replay decision.

- Added a non-reserving existing-replay probe for auth-first root capability
  paths, so those paths can return cached/conflicting replay state before
  authorization while still leaving truly fresh requests unreserved until after
  authorization succeeds.

- Added tests for cross-variant root request-id reuse and for the stable-store
  cross-command receipt query.

### Notes

- No CLI commands, flags, output columns, or JSON report shapes change in this
  patch.

- This strengthens the root capability replay invariant introduced earlier in
  0.61 without changing committed same-command replay behavior.

## 0.61.6 - 2026-06-05 - Pool CreateEmpty replay guard

### Summary

- Continued Slice D from
  `docs/design/0.61-replay-protection/0.61-design.md` by moving
  `PoolAdminCommand::CreateEmpty` onto shared replay receipts and deployment
  cost guards.

- `PoolAdminCommand::CreateEmpty` now carries `CreateEmptyPoolRequest`
  metadata. Direct pool create-empty calls must include a 32-byte request id
  and bounded replay TTL before Canic will reserve or execute the operation.

- Fresh create-empty requests reserve a `pool.create_empty.v1` replay receipt,
  reserve deployment quota and an in-flight cycle budget, mark the management
  `create_canister` effect in flight, create the pool canister through a
  `CostGuardPermit`, register the created canister, and commit the created
  principal as the replay response.

- Duplicate requests with the same request id, caller, and payload return the
  originally created pool canister principal without calling management
  `create_canister` again.

### Changed

- Added `CreateEmptyPoolRequest` to the pool DTO boundary with optional
  `RootRequestMetadata`. The workflow rejects missing metadata, zero TTL, and
  TTL values above the 300-second pool replay maximum.

- Added the explicit replay command kind `pool.create_empty.v1`.

- Pool create-empty payload hashing binds the command kind and direct caller,
  while the empty create payload intentionally excludes replay metadata.

- Management create failures after the external-effect boundary now recover
  the cycle reservation and mark the replay receipt `RecoveryRequired` with
  `ExternalEffectStatusUnknown`, so a retry cannot silently create another
  canister for the same operation id.

- Response-encoding failures after management create mark the receipt
  `RecoveryRequired` with `ResponseCommitFailed` and recover the cycle
  reservation before returning the workflow error.

- Added `MgmtOps::create_canister_with_permit` for guarded management create
  paths. The existing ungated management adapter remains for surfaces that have
  not yet moved onto deployment cost guards.

### Notes

- No CLI commands, flags, output columns, or JSON report shapes change in this
  patch.

- The replay policy manifest still leaves `canic_pool_admin` as a release
  blocker because the endpoint also carries non-CreateEmpty variants. This
  patch only claims the `CreateEmpty` command path.

  so the root signing path cannot call threshold ECDSA through that adapter
  without first crossing the guard.

- The replay policy manifest now marks `canic_request_delegation` as
  implemented. Other costed endpoint entries remain release blockers until
  their replay/cost-guard paths are wired.

### Changed

- Added `ops::cost_guard`, backed by the existing durable intent store.
  The guard records:
  - a per-command, per-subject quota reservation in a deterministic time
    bucket;
  - a per-cost-class, per-payer cycle reservation that counts outstanding
    in-flight work against the local liquid-cycle balance.

- Fresh delegation proof requests reserve cost state only after replay preflight
  returns a fresh receipt. Committed replay responses still return the stored
  proof without consuming signing quota or checking current cycle reserve.

- Delegation proof signing uses a 60-second quota window, allows 60 fresh
  signing operations per shard/caller per window, and reserves
  `1_000_000_000` cycles while requiring `1_000_000_000` cycles to remain
  after the reservation.

- Signing failures after the cost barrier consume the signing quota and release
  the in-flight cycle reservation through recovery handling. Cheap validation
  failures before the cost barrier abort the replay receipt and do not consume
  quota.

### Notes

- No CLI commands, flags, output columns, or JSON report shapes change in this
  patch.

- The cost-guard module is intentionally small in this slice. It establishes
  the unforgeable permit and durable quota/reserve mechanics needed by later
  deployment and token-signing guard work.

- Pool `CreateEmpty`, root role/internal proof signing, delegated-token minting,
  and broader deployment/value-transfer cost guards remain later 0.61 work.
