# 0.61 Changelog

## 0.61.3 - 2026-06-04 - Root replay receipt migration

### Summary

- Continued Slice B from
  `docs/design/0.61-replay-protection/0.61-design.md` by moving root RPC replay
  checks, reservations, commits, aborts, capacity checks, and TTL expiry onto
  the shared replay receipt store.

- Added explicit root replay command kinds for the existing root capability
  surfaces:
  `root.provision.v1`, `root.upgrade.v1`, `root.recycle_canister.v1`,
  `root.request_cycles.v1`, `root.issue_role_attestation.v1`, and
  `root.issue_internal_invocation_proof.v1`.

- Extended shared replay receipts with optional expiry so root replay can keep
  its existing TTL behavior while future non-expiring receipt users can leave
  the field unset.

- Added the first broad NNS inspection foundation command:
  `canic nns registry version`.

- The command queries the canonical mainnet NNS registry canister for the
  latest registry version and prints text or JSON output.

- Added cached NNS node-operator inspection:
  `canic nns node-operator list`,
  `canic nns node-operator info <node-operator-prefix>`, and
  `canic nns node-operator refresh`.

- Node-operator rows are derived from mainnet subnet membership,
  `node_record_<node>`, and `node_operator_record_<operator>` registry values.

### Changed

- Root replay preflight now prepares shared receipt tokens, checks receipt
  capacity, explicitly reserves fresh receipts, returns committed receipt
  responses for duplicate requests, and aborts reserved receipts on policy or
  execution failure.

- The non-root cycles replay-first path now uses the same shared receipt-backed
  root replay machinery.

- Removed the active legacy `RootReplayOps` and root replay slot-key module.
  The old `RootReplayRecord` binary encoding remains test-only so its historical
  stable shape stays documented, but root replay no longer reads or writes the
  legacy `canic.core.root_replay.v1` map.

- Shared receipt storage now supports removal, active actor counts, and expired
  receipt purging for root replay capacity and TTL semantics.

- Kept the new NNS registry inspection implementation split by ownership
  boundary: `canic-ic-registry` owns the raw NNS registry query, `canic-host`
  owns the report shape and text rendering, and `canic-cli` owns the Clap
  command surface.

- Kept node-operator inspection on the same NNS ownership boundary:
  `canic-ic-registry` owns registry traversal and protobuf decoding,
  `canic-host` owns cache/report/text shaping, and `canic-cli` owns the Clap
  surface.

### CLI Surface

```text
canic nns registry version
canic nns registry version --format json
canic nns registry version --source-endpoint https://icp-api.io
canic --network ic nns registry version
canic nns node-operator refresh
canic nns node-operator list
canic nns node-operator list --verbose
canic nns node-operator info <node-operator-prefix>
canic nns node-operator list --format json
```

Text output includes:

```text
network
registry_canister_id
registry_version
fetched_at
source_endpoint
fetched_by
```

JSON output uses `NnsRegistryVersionReport` with schema version `1`.

Node-operator JSON output uses schema version `1` reports with registry
provenance and per-operator fields:

```text
node_operator_principal
node_provider_principal
node_allowance
data_center_id
node_count
```

### Notes

- This finishes the Slice B root replay migration requirement. Delegation-proof
  issuance and pool `CreateEmpty` still need their own replay-protected request
  DTOs and cost guards in later 0.61 slices.

- `canic nns registry version` is a live read and does not write a cache file.
  The value is small and acts as a freshness/provenance probe for later cached
  NNS views.

- The command remains mainnet-only. Local replica NNS registry discovery is not
  implemented in this slice.

- `canic nns node-operator list` and `info` are cache-first and populate
  `.canic/node-operator/ic/operators.json` on a first cache miss. `refresh`
  force-refreshes through `.canic/node-operator/ic/refresh.lock` and atomic
  cache replacement.

- Node counts are assigned mainnet subnet nodes for each operator, not spare or
  unassigned node inventory.

- The nested `version` command is intentionally distinct from the CLI
  `--version` flag. `canic nns registry version` performs the registry query,
  while `canic nns registry --version` prints the Canic CLI version.

## 0.61.2 - 2026-06-04 - Shared replay receipt store

### Summary

- Continued Slice B from
  `docs/design/0.61-replay-protection/0.61-design.md` by adding the shared
  stable replay receipt store and reserve/replay/commit API.

- Reserved stable memory ID `21` for shared replay receipts under the auth
  memory range and added a dedicated `canic.core.replay_receipts.v1` stable
  map.

- Added the stored receipt record shape for shared replay receipts, including
  command kind, operation ID, replay actor, payload hash schema/version,
  receipt status, committed response bytes, and external-effect descriptor.

- Added mechanical receipt storage ops with command-kind-scoped slot-key
  derivation so the same operation ID is isolated per replay-protected command
  kind.

### Changed

- Added `ops::replay::receipt`, the first shared replay receipt service API.
  It can reserve a fresh receipt, classify duplicate requests, return committed
  responses, report actor/payload mismatches, report pending or in-flight
  operations, mark external effects in flight, commit successful responses,
  commit bounded terminal failures, and mark recovery-required receipts.

- Receipt storage encodes records with CBOR at the stable-store boundary while
  converting back into the shared replay model before policy decisions.

- Terminal-failure commits apply the shared
  `MAX_REPLAY_TERMINAL_ERROR_BYTES` bound before storing error payload bytes.

### Notes

- This is still branch-only Slice B foundation work. Existing root RPC replay
  continues to use the legacy root replay store in this patch; migrating root
  replay onto the shared receipt primitive remains required before any 0.61
  release candidate.

- No CLI commands, flags, or output formats change in this patch.

## 0.61.1 - 2026-06-04 - Shared replay model foundation and NNS node-provider enrichment

### Summary

- Started Slice B from
  `docs/design/0.61-replay-protection/0.61-design.md` by adding the shared
  replay model that later command migrations will use.

- Added the core replay vocabulary:
  `OperationId`, `CommandKind`, `ReplayActor`, `ReplayReceiptKey`,
  `ReplayReceipt`, `ReplayReceiptStatus`, `ExternalEffectDescriptor`,
  `EcdsaPurpose`, and common replay error classifications.

- Added versioned replay payload hashing support through
  `REPLAY_PAYLOAD_HASH_SCHEMA_VERSION` and `ReplayPayloadHasher`.

- Added bounded terminal-error bytes with
  `MAX_REPLAY_TERMINAL_ERROR_BYTES = 4096` so later terminal-failure receipts
  cannot persist unbounded error payloads.

- Enriched `canic nns node-provider list` and
  `canic nns node-provider info` with registry-derived assigned-node counts.

- Node counts are derived by following mainnet subnet membership through
  `node_record_<node>` and `node_operator_record_<operator>` registry values to
  each node provider principal.

- Made `canic nns node-provider list` and `info` cache-first. On the first
  cache miss they populate `.canic/node-provider/ic/providers.json`; subsequent
  reads use the cached enriched report.

- Added `canic nns node-provider refresh` to force-refresh the native NNS
  governance/registry-backed cache, with `--dry-run`, `--output`, and refresh
  lock handling matching the subnet metadata surface.

### Changed

- The existing root replay guard and key derivation now accept the shared
  32-byte `OperationId` type instead of raw `[u8; 32]` request IDs at the
  replay-core boundary.

- Root RPC replay behavior is otherwise preserved in this slice. The existing
  root DTO metadata still carries `[u8; 32]` and is converted to `OperationId`
  before entering the replay guard.

- Added minimal local protobuf decoding for NNS registry `NodeRecord` and
  `NodeOperatorRecord`.

- Added bounded concurrent registry reads in `canic-ic-registry` so provider
  enrichment does not perform every node/operator lookup sequentially.

- Added registry provenance to node-provider host reports:
  `registry_canister_id` and `registry_version`.

- `canic nns node-provider list --verbose` now includes the registry version
  alongside full provider principals and reward-account detail.

- Node-provider refresh writes through `.canic/node-provider/ic/refresh.lock`
  and replaces `providers.json` atomically, so repeated `list`/`info` calls no
  longer redo the expensive node/operator registry enrichment pass.

- Factored the shared host refresh-lock, atomic cache-write, refresh-output,
  and positive-duration parsing helpers used by the cached NNS subnet and
  node-provider surfaces.

### Notes

- This is a foundation slice. It does not yet migrate stable root replay
  records to the final shared receipt shape, and it does not yet add replay
  metadata to delegation-proof issuance or pool creation requests.

- The count is assigned mainnet subnet nodes, not a spare-node or unassigned
  inventory count.

- The native NNS governance and registry sources still do not expose a
  canonical human-readable provider name. Dashboard pages expose display names,
  but this slice intentionally keeps the data source IC-native and does not add
  a dashboard HTTP dependency. The JSON `name` field remains nullable for
  schema room, but text output does not render a `NAME` column.

## 0.61.0 - 2026-06-04 - Replay policy inventory and auth hard cut

### Summary

- Started the 0.61 replay-safety hardening line with the Slice A foundation
  from `docs/design/0.61-replay-protection/0.61-design.md`.

- Added a `canic-core::replay_policy` manifest that classifies Canic-emitted
  update endpoints by replay policy, implementation status, cost class, quota
  policy, and cycle-reserve policy.

- Hard-cut verifier-local delegated-token update consumption. Delegated tokens
  are again TTL-bounded bearer credentials; replay-sensitive mutations must use
  domain operation IDs and replay receipts.

- Made `canic_app` set-style commands response-idempotent. Repeating a
  state-setting request now returns success with `changed = false` instead of
  an already-in-state error.

- Added an API-boundary caller/shard guard for local root delegation-proof
  issuance. `canic_request_delegation` now rejects
  `msg_caller() != request.shard_pid` before delegated-token config lookup,
  proof preparation, or threshold ECDSA signing.

### Changed

- Removed the delegated-token consumed-use runtime path from access auth,
  auth ops, auth storage ops, and stable auth state.

- Removed the stable `auth/token_uses` module and the live
  `delegated_token_uses` field from `AuthStateRecord`.

- Added an upgrade-shape test proving old serialized auth state with historical
  consumed-token markers decodes into the new auth state shape while dropping
  that removed field.

- Added manifest tests that compare Canic macro-emitted update endpoints
  against the replay-policy manifest, require unique entries, and require cost
  metadata for costed endpoints.

- Updated active recurring audit templates so delegated auth is described as
  bearer-token verification and replay protection is assigned to domain replay
  receipts.

- Changed the root `canic_app` endpoint response from `Result<(), Error>` to
  `Result<AppCommandResponse, Error>`. `SetStatus` returns
  `Status(SetStateResponse<AppStatus>)`, and `SetCyclesFundingEnabled` returns
  `CyclesFundingEnabled(SetStateResponse<bool>)`; both responses include
  `previous`, `current`, and `changed`.

- Reused the existing `DelegationCallerShardMismatch` wording for the direct
  delegation-proof issuance guard so local root issuance and root-capability
  authorization diagnostics stay aligned.

### Notes

- This is branch-only foundation work. It intentionally does not yet fix pool
  `CreateEmpty` or delegation-proof issuance response replay/caching.

- `ReleaseBlocker` manifest entries identify later 0.61 slices that must land
  before a public 0.61 release candidate.
