# 0.53 Changelog

## 0.53.5 - 2026-05-31 - manifest duplicate hardening

### Changed

- Project evidence manifests now reject duplicate evidence paths before gate
  evaluation. Duplicate paths are invalid input instead of evaluating the same
  saved evidence envelope more than once.

### Safety

- This preserves the passive policy boundary. It does not run builds, discover
  live state, mutate manifests/evidence/config/topology/controllers, register
  artifacts, or turn policy success into deployment truth.

## 0.53.3 - 2026-05-31 - project evidence manifests

### Added

- Added project evidence manifests to the passive policy gate:

  ```text
  canic evidence gate --policy <path> --manifest <path>
  ```

- Added stable project manifest/report shapes:

  ```text
  ProjectEvidenceManifestV1
  ProjectEvidenceGateReportV1
  ```

- A manifest can group existing evidence envelope files, declare whether each
  file is required, and set expected payload schema plus target identity.
  Required missing files fail with `missing_required_evidence`; optional
  missing files produce `success_with_warnings`; payload schema and target
  mismatches fail with `blocked_by_policy`.

### Safety

- Manifest gates still read existing evidence artifacts only. They do not run
  builds, discover live state, mutate manifests or evidence inputs, update
  config/topology/controllers, register artifacts, or turn policy success into
  deployment truth.

## 0.53.2 - 2026-05-31 - build provenance policy rules

### Added

- Added optional build-provenance rules to the passive policy gate:

  ```text
  canic evidence gate --policy <path> --envelope <path>
  ```

- Policies can now require stable `canic.build_provenance.v1` payload evidence:

  ```toml
  [build_provenance]
  require_clean_source = true
  require_cargo_lock = true
  require_wasm_gzip = true
  require_sha256 = true
  require_package_identity_matches_target = true
  ```

- The gate reports structured failures when build provenance does not prove a
  clean source checkout, omits `Cargo.lock` evidence, lacks a gzip Wasm
  artifact, has invalid artifact SHA-256 evidence, or has package metadata
  `fleet.role` that disagrees with the evaluated envelope target.

### Safety

- Build-provenance policy rules still evaluate one existing
  `EvidenceEnvelopeV1`. They do not run builds, generate provenance, query
  deployments, mutate policy or evidence inputs, update config/topology/
  controllers, register artifacts, or turn policy success into deployment
  truth.

## 0.53.1 - 2026-05-31 - single-envelope policy gate

### Added

- Added the passive single-envelope CI policy gate:

  ```text
  canic evidence gate --policy <path> --envelope <path>
  ```

- The command reads one strict `CiPolicyV1` TOML file and one existing
  `EvidenceEnvelopeV1`, evaluates stable envelope fields only, and emits a
  stable `PolicyGateReportV1`.

- Added structured policy findings, requirement IDs, evaluated envelope exit
  class, gate exit class, and pass/fail policy status. The report preserves
  the evaluated envelope identity instead of rewriting it.

- Added output formats:

  ```text
  canic evidence gate --policy <path> --envelope <path> --format json
  canic evidence gate --policy <path> --envelope <path> --format envelope-json
  canic evidence gate --policy <path> --envelope <path> --output <path>
  ```

- Raw `--format json` emits `PolicyGateReportV1`. `--format envelope-json`
  wraps the report in `EvidenceEnvelopeV1` with `target.kind = "policy_gate"`
  and fingerprints for both the policy file and evaluated envelope.

### Safety

- Policy gates are passive. They do not run builds, query deployments, mutate
  policy or evidence inputs, update config/topology/controllers, register
  artifacts, or turn policy success into deployment truth.

## 0.53.0 - 2026-05-31 - single-envelope policy gate design and CLI cleanup

### Added

- Added the tentative 0.53 CI policy gate and project evidence manifest design:

  ```text
  docs/design/0.53-ci-policy-gates-project-manifests/0.53-design.md
  ```

- The design proposes passive policy evaluation over existing stable evidence,
  starting with a single-envelope gate:

  ```text
  canic evidence gate --policy <path> --envelope <path>
  ```

- The proposed line consumes:
  - 0.51 `EvidenceEnvelopeV1`;
  - 0.51 `ExitClassV1`;
  - 0.52 `canic.build_provenance.v1`.

- The first implementation slice is narrowed to envelope-level semantics:
  strict `CiPolicyV1` parsing, envelope schema checks, payload schema
  identity/stability allow-lists, evaluated exit-class checks, summary
  conflict/blocker/missing-evidence policy, and stable `PolicyGateReportV1`
  output.

- `PolicyGateReportV1` must distinguish the evaluated envelope's exit class
  from the gate result's exit class. Policy failure should preserve the
  evaluated evidence and report the gate's `ExitClassV1` separately.

- `--format json` should emit raw `PolicyGateReportV1`.
  `--format envelope-json` should wrap that report in a new
  `EvidenceEnvelopeV1` with policy-file and evaluated-envelope fingerprints as
  inputs. `--output <path>` should write the requested report artifact before
  returning a non-zero status when practical.

- Build-provenance field rules, including clean source, `Cargo.lock`, package
  identity, gzip Wasm, and SHA-256 requirements, are deferred until after the
  single-envelope gate semantics are proven. Project evidence manifests remain
  a later 0.53.x slice.

### Changed

- Moved fleet config inspection under the fleet command family and removed the
  old top-level command:

  ```text
  canic fleet config <fleet>
  ```

- Moved backup manifest validation under the backup command family and removed
  the old generic top-level manifest command:

  ```text
  canic backup manifest validate --manifest <file>
  ```

- Global network forwarding now reaches every `canic deploy ...` leaf that
  consumes deployment network state, including nested authority, external,
  root-verification, register, install, and resume-report commands.

### Removed

- Removed the top-level fleet-config command family:

  ```text
  canic config <fleet>
  ```

- Removed the top-level backup-manifest command family:

  ```text
  canic manifest validate --manifest <file>
  ```

### Safety

- This is design-only for the policy gate. It does not add runtime policy-gate
  behavior.
- The design explicitly defers deployment locks, signing, provider wrappers,
  artifact registry import, `wasm_store` retention/GC, controller mutation,
  topology mutation, active adoption/import, install/upgrade authority, and
  broad live verification.
