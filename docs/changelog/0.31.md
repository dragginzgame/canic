## 0.31.1 - 2026-05-06 - Backup Restore V1 Trim

### đź§ą Cleanup

- Narrowed the backup v1 CLI surface around `backup verify` and `backup status`; deeper no-mutation restore checks now use the explicit restore command sequence.
- Removed `backup smoke`, `backup preflight`, `backup inspect`, and `backup provenance` from the public `canic backup` subcommand dispatcher.
- Removed the CLI-side `backup smoke` and `backup preflight` wrapper modules after their public commands were dropped.
- Removed the unused `canic-backup` preflight module plus standalone layout inspection/provenance report APIs; `verify_integrity` remains the v1 backup-layout check.
- Removed stale restore apply-journal status and next-operation DTOs that only supported the retired low-level restore commands.
- Removed the remaining public `restore apply-status` and `restore apply-report` report commands. Use `restore run --dry-run` to inspect journal progress and attention status.
- Removed detailed `restore run --require-*` assertion flags for mode, action, counts, receipts, progress, and timestamp markers. The v1 CLI keeps only the operator-facing `--require-complete` and `--require-no-attention` gates.
- Removed zero-valued code-reinstall counters and the old `operation_counts_supplied` compatibility flag from restore JSON now that snapshot restore has one current operation model.
- Simplified the backup manifest and restore plan model to single-canister or subtree backup units, removing generic quiescence, dependency-closure, verification-class, and manual restore-group metadata from the current v1 path.
- Collapsed restore output from phase-wrapped lists into ordered `members` and ordered `operations`, matching the current v1 topology flow.
- Narrowed restore verification to status checks and removed manifest-controlled query method execution from the current v1 runner path.
- Made snapshot capture stop each target canister before creating a snapshot; `--resume-after-snapshot` now only controls whether Canic starts canisters again afterward.
- Centralized backup-root-relative artifact path resolution so integrity checks, restore validation, and upload command rendering reject escaping paths consistently.
- Replaced new host-side `"unknown"` timestamp placeholders with shared `unix:<seconds>` markers for snapshot manifests, download journals, and restore runner state transitions.
- Added a current `0.31` backup/restore checklist that uses `backup verify`, `restore plan`, `restore apply --dry-run`, and `restore run` instead of the removed smoke/preflight wrappers.
- Restore execution now proves stopped-canister status before running snapshot load commands, failing the operation before mutation if `dfx canister status` does not show a stopped target.
- Snapshot restore readiness no longer requires module hash, wasm hash, or code version provenance; those fields remain useful audit metadata, but snapshot load restores code and state from the snapshot itself.
- Removed the redundant `restore status` command and `restore apply --status` input. Restore now flows from plan to apply dry-run/journal to runner, with the apply journal as the progress record.

## 0.31.0 - 2026-05-06 - Snapshot Restore Correctness

### đź“ť Summary

- `0.31.0` starts the snapshot cleanup line by removing normal post-load code reinstall operations from restore planning, making snapshot loads depend on uploaded target snapshot receipts, resolving upload artifact paths through the validated backup root, trimming manual restore-journal CLI primitives and batch assertion gates in favor of `canic restore run`, and beginning the incremental `clap` parser migration.

### đź©ą Fixed

- Restore apply dry-runs now emit `UploadSnapshot`, `LoadSnapshot`, and verification operations for each member, without treating `ReinstallCode` as a normal follow-up step.
- Restore operation summaries now report zero planned code reinstalls for snapshot restores, matching the IC snapshot behavior where `LoadSnapshot` restores code and state together.
- Restore apply journals can now persist upload operation receipts with the uploaded target-side snapshot ID, and `LoadSnapshot` command rendering now uses that receipt instead of the source snapshot ID.
- Load command previews fail closed when the matching completed upload receipt is missing or does not include a target snapshot ID.
- Apply journals now persist the artifact validation `backup_root`, and upload command previews resolve relative artifact paths through that root so validation and execution use the same location.
- Method-based restore verification command previews now render `dfx canister call --query`, keeping verification non-mutating by default.
- Native restore runner mutations now acquire an atomic sidecar journal lock while executing or unclaiming work, preventing two runner processes from advancing the same journal at once.
- Apply journal operation receipts now retain command outcome, status, attempt number, failure reason, and bounded stdout/stderr output so failed restore steps remain inspectable after the runner exits.
- Removed the low-level `canic restore apply-next`, `apply-command`, `apply-claim`, `apply-unclaim`, and `apply-mark` commands. Operators now build journals with `restore apply --dry-run`, inspect them with `restore run --dry-run`, and advance them with `restore run`.
- Removed the public `restore run --require-batch-*` assertion flags and the redundant `batch_summary` runner JSON object. Runner output now relies on `max_steps_reached`, `executed_operation_count`, `progress`, and receipt summaries for v1 status.
- Moved manifest validation summaries into `canic-backup`, leaving `canic-cli` to write command output and enforce CLI flags.
- Removed the design-conformance report and `--require-design` gates from the current v1 flow; restore readiness now stays focused on manifest validity, verified backup layout, and snapshot restore readiness.
- Added a global `--version` flag that reports the installed `canic` CLI version from any command depth.
- Added `make install` as the single local `canic` CLI install target and removed the old `make install-canic` alias.
- Added `clap` to `canic-cli`, moved the top-level command metadata/help onto a `clap` command tree, and converted restore plus backup option parsing away from repeated ad hoc argument loops.
- Removed current compatibility surfaces for restore execution: use `canic restore run` directly instead of `scripts/restore/apply_journal.sh`.
- `canic list` now defaults to standalone local demo inventory from a fixed canister-name roster until fleet state exists, while `--root <name-or-principal>` explicitly selects an installed root registry.
- Added `canic install` as the primary local tree setup command. It defaults to `root`, accepts either a dfx canister name or principal as the root target, and replaces the older direct installer binary path.
- Successful installs now write named fleet state under `.canic/<network>/fleets/`, and `canic list` uses the current fleet when `--root` and `--fleet` are omitted.
- Plain `canic list` now auto-selects the current installed fleet when project-local fleet state exists, while `--standalone` still shows raw dfx project canister IDs.
- `canic list` now renders canister principals as the first column with box-drawing tree branches, keeps non-unique roles as the second column, adds `KIND`, labels the implicit `wasm_store`, and supports `--fleet <name>` plus `--from <name-or-principal>` for saved-fleet subtree views.
- Removed stale `canic list --registry-json`, `canic list --ready`, and `canic snapshot download --registry-json` flags; list is now a live tree command with readiness included by default, and subtree snapshots now require a live root.
- Local `canic list` readiness checks now query the replica directly instead of capturing `dfx canister call` output, avoiding false `error` readiness rows from stale standalone IDs or DFX stdout-capture failures.
- `canic install` now treats `canisters/canic.toml` as the only implicit project default; when it is missing, the command lists discovered config choices and requires `--config <path>` instead of guessing.
- Install config choices are now numbered, path-ordered, and interactive in terminals, so operators can enter a config number instead of retyping a discovered path.
- `canic list` now calls `canic_ready` for each live listed canister and includes a whitespace-aligned `READY` column by default; successful installs now point operators at that wrapped smoke check.
- Root-only standalone listings now tell new operators to run `canic install` and then retry with `canic list`, making the first local setup step explicit.
- `canic install` now captures noisy custom-build logs and renders build/staging progress as compact tables with per-canister progress bars.
- Release staging now keeps one live chunk-upload progress line above a compact canister timing table, while keeping non-interactive logs stable and line-oriented.
- Moved shared registry discovery, snapshot manifest construction, and native restore-runner journal mutation into `canic-backup`, leaving `canic-cli` focused on flags, `dfx` execution boundaries, JSON output, and CLI-only fail-closed gates.
- Added `canic::build_standalone!` for sandbox/probe canisters, replacing redundant one-role `canic.toml` files with generated minimal config while keeping explicit `CANIC_CONFIG_PATH` builds strict.

### đź§Ş Testing

- Updated restore planning, apply dry-run, apply journal, backup, CLI status, list, runner, and generated build-config tests to assert the new upload/load/verify operation sequence, receipt-gated load rendering, uploaded snapshot ID parsing, trimmed CLI surface, Clap-backed parsing, standalone list behavior, top-level command help, and counters.
