### Added

- Added `PromotionWasmStoreIdentityReportV1` and

  staged wasm-store identity from `StagingReceiptV1` without querying
  `wasm_store`.
- Added `PromotionWasmStoreIdentityReportRequest`,
  `promotion_wasm_store_identity_report(...)`,
  `promotion_wasm_store_identity_report_from_staging(...)`, and
  `validate_promotion_wasm_store_identity_report(...)`.
- Added `PromotionMaterializationIdentityReportV1`,
  `RolePromotionMaterializationIdentityV1`, and
  `PromotionMaterializationOutputGroupV1` so source/build materialization
  evidence can be aggregated by role and grouped by materialized output
  identity.
- Added `PromotionMaterializationIdentityReportRequest`,
  `promotion_materialization_identity_report(...)`,
  `promotion_materialization_identity_report_from_evidence(...)`, and
  `validate_promotion_materialization_identity_report(...)`.

  `RolePromotionProvenanceV1` to link passive promotion plans to readiness,
  artifact identity, transform, target execution lineage, wasm-store identity,

  `artifact_promotion_provenance_report(...)`, and
  `validate_artifact_promotion_provenance_report(...)`.
- Added host-owned passive text rendering for wasm-store identity reports,
  materialization identity reports, and artifact promotion provenance reports.

### Changed

- The 0.44 status handoff now treats passive wasm-store identity,
  source/build materialization identity, and promotion provenance report
  modeling as implemented.
- The remaining wasm-store gap is now limited to live `wasm_store` catalog
  lookup and verification beyond staging receipt evidence.
- Source/build materialization identity report surfacing is now modeled in
  host code; remaining work is CLI command wiring rather than report DTO
  design.
- Artifact promotion provenance reports explicitly keep
  `execution_attempted` false and do not replace promotion execution receipts.

### Validation

- Added validation for wasm-store identity report schema, report IDs, staging
  receipt schema drift, duplicate roles, status/blocker consistency, stale
  blockers, blocker severity, non-wasm-store transport, missing locators,

- Added validation for materialization identity report schema, report IDs,
  duplicate roles, duplicate evidence IDs, output-group consistency,
  stale output identity keys, status/blocker consistency, and blocker
  severity.
- Added validation for promotion provenance report schema, required linkage
  IDs, duplicate roles, optional report validity, unknown optional report
  roles, status/blocker consistency, blocker severity, and no-execution
  semantics.

### Testing

- Added JSON round-trip and schema-shape coverage for
  `PromotionWasmStoreIdentityReportV1`,
  `PromotionMaterializationIdentityReportV1`, and
  `ArtifactPromotionProvenanceReportV1`.
- Added wasm-store identity tests for staged locator preservation, non

## 0.44.6 - 2026-05-25 - passive promotion plan envelope

### Added

- Added `PromotionTargetExecutionLineageV1` to bind validated promoted-plan
  lineage to validated target execution preflight evidence without claiming
  execution occurred.
- Added `PromotionTargetExecutionLineageRequest`,
  `promotion_target_execution_lineage(...)`,
  `promotion_target_execution_lineage_digest(...)`, and
  `validate_promotion_target_execution_lineage(...)`.
- Added `ArtifactPromotionPlanV1` as a passive envelope over promotion
  readiness, artifact identity, promoted-plan transform, optional target
  execution lineage, and aggregate blockers.
- Added `ArtifactPromotionPlanRequest`, `artifact_promotion_plan(...)`,
  `validate_artifact_promotion_plan(...)`, and
  `validate_artifact_promotion_plan_for_check(...)`.
- Added host-owned passive text rendering for target execution lineage and
  artifact promotion plans.

### Changed

- Promotion target execution lineage explicitly records `execution_attempted`
  as false and rejects artifacts that claim execution occurred.
- Artifact promotion plans now copy the promotion-plan lineage digest from the
  validated transform and revalidate that copy during archive validation.
- Artifact promotion plans can now be checked against a `DeploymentCheckV1`,
  proving the promoted plan and execution preflight match the deployment truth
  check that would gate later execution.
- The 0.44 status handoff now marks full passive promotion plan modeling,
  target execution lineage identity, and cross-target passive check linkage as
  implemented.

### Validation

- Added validation for target execution lineage schema, required fields,
  sha256 lineage digest shape, nested transform validity, nested preflight
  validity, preflight/promoted-plan linkage, no-execution semantics, and stale
  digest rejection.
- Added validation for artifact promotion plan schema, required IDs,
  status/blocker consistency, nested readiness, nested artifact identity
  report, nested transform, nested target execution lineage, blocker
  aggregation, and copied lineage consistency.
- Added deployment-check validation for artifact promotion plans, including
  missing target execution lineage, mismatched promoted plans, and mismatched
  execution preflight provenance.

### Testing

- Added JSON round-trip and schema-shape coverage for
  `PromotionTargetExecutionLineageV1` and `ArtifactPromotionPlanV1`.
- Added target execution lineage tests for valid generated lineage, wrong
  preflight plan, execution-attempt claims, stale digest, and passive text
  output.
- Added artifact promotion plan tests for valid generated plans, stale lineage
  copies, status/blocker mismatches, mismatched target lineage, target-check
  success, wrong target plans, missing lineage, preflight/check mismatches, and
  passive text output.

## 0.44.5 - 2026-05-25 - promotion lineage and materialization links

### Added

- Added `RolePromotionMaterializationLinkV1` so source/build promotion
  transforms can point at validated materialization evidence.
- Added `PromotionPlanTransformWithMaterializationRequest` for callers that
  need promotion transforms with source/build materialization evidence attached.
- Added `promoted_deployment_plan_transform_from_inputs_with_materialization(...)`

  `BuildMaterializationEvidenceV1` links for source/build roles.

- Added `previous_receipt_lineage_digest` to `RoleArtifactSourceV1` for
  receipt-backed promotion artifact sources.

### Changed

- Source/build promotion transform summaries now record the materialization
  evidence ID, target materialization-input digest, materialized wasm digest,
  materialized compressed wasm digest, installed module hash, and Candid digest.
- Promotion transform text now reports materialization evidence IDs and
  materialization digests for source/build roles.
- Passive promotion transforms now include deterministic lineage over the
  target plan ID, promoted plan ID, promoted plan, and role summaries.
- Receipt-backed artifact sources must now carry source receipt lineage
  identity instead of relying on locator and receipt-kind metadata alone.
- The 0.44 status handoff now treats target execution lineage as remaining
  work while marking source receipt and promotion-plan lineage as implemented.

### Validation

  materialization evidence record per source/build role.
- Added validation rejecting missing, duplicate, unexpected, invalid, or
  target-artifact-mismatched materialization links.
- Added validation that `promotion_plan_lineage_digest` matches the promoted
  plan and role summaries.
- Added validation requiring `previous_receipt_lineage_digest` for
  `PreviousReceiptArtifact` sources, rejecting it on non-receipt sources, and
  checking sha256 digest shape.

### Testing

- Added source/build transform tests for successful materialization links,
  missing evidence, duplicate evidence, and passive text output.
- Added lineage tests proving stale promotion-plan lineage digests are rejected
  and materialization-link changes alter the deterministic lineage digest.
- Added receipt-backed source tests for missing lineage digest, invalid lineage
  digest, and unexpected lineage digest on non-receipt sources.

## 0.44.4 - 2026-05-25 - promotion policy readiness

### Added

- Added `RolePromotionPolicyV1` to record role-scoped promotion policy, including
  allowed promotion artifact levels and required promotion claims.
- Added `PromotionPolicyRequirementV1` for policy requirements such as sealed
  bytes, byte-identical wasm, target config digest, same source revision, and
  same Cargo features.
- Added `PromotionPolicyClaimV1` so promotion inputs can explicitly claim
  byte-identical wasm and target-config-digest handling.
- Added `PromotionPolicyCheckV1` and `RolePromotionPolicyDecisionV1` as passive
  policy-check artifacts for role-level policy decisions.
- Added `PromotionPolicyCheckRequest`, `check_promotion_policy(...)`,
  `promotion_policy_check_from_inputs(...)`,
  `validate_promotion_policy_check(...)`, and
  `validate_role_promotion_policy(...)`.
- Added `PromotionReadinessWithPolicyRequest`,
  `check_promotion_readiness_with_policy(...)`, and
  `promotion_readiness_from_inputs_with_policy(...)` so callers can fold policy
  blockers into the same passive readiness artifact.
- Added host-owned passive text rendering for promotion policy checks.

### Changed

- Promotion policy now distinguishes roles that must reuse sealed bytes from
  roles that may use source/build promotion only when byte identity is later
  proven.
- Promotion readiness can now optionally include role policy blockers directly,
  while preserving the original artifact/target-only readiness path for callers
  that need the narrower check.
- Duplicate policy roles are reported as hard blockers even when the duplicated
  role is not part of the current promotion input.
- The 0.44 status handoff now treats CLI/report surfaces for policy checks as
  remaining work, rather than treating the passive policy model as unimplemented.

### Validation

- Added validation for required policy/check IDs, non-empty allowed promotion
  levels, duplicate allowed levels, duplicate requirements, duplicate policy
  roles, status/blocker consistency, decision consistency, and blocker severity.
- Sealed-byte policies now reject source/build promotion levels.
- Byte-identical source/build policies can pass without implying sealed-byte
  reuse.
- Policy-aware readiness artifacts validate through the same
  `PromotionReadinessV1` validation path as ordinary readiness artifacts.

### Testing

- Added JSON round-trip and field-shape coverage for `RolePromotionPolicyV1`
  and `PromotionPolicyCheckV1`.
- Added policy-check coverage for sealed-wasm success, source/build rejection
  under sealed-byte policy, byte-identity/source-build success, missing
  byte-identity claims, duplicate role policies, and stale policy decisions.
- Added readiness-with-policy tests for sealed-byte blockers, byte-identical
  source/build success, missing policy blockers, and passive readiness text
  output that includes policy blockers.

## 0.44.3 - 2026-05-25 - source build materialization identity

### Added

- Added `BuildRecipeIdentityV1` to record reusable source/build recipe
  identity separately from materialized output bytes.
- Added `BuildMaterializationInputV1` to record target-specific
  materialization inputs, including canonical embedded config, network, root
  trust anchor, and runtime variant.
- Added `BuildMaterializationResultV1` to record concrete output digests for a
  materialized source/build recipe.
- Added `BuildMaterializationEvidenceV1` as a passive linkage artifact that
  carries the recipe, input, output result, computed input digest, and
  consistency flags.
- Added `BuildMaterializationEvidenceRequest`,
  `build_materialization_evidence(...)`,
  `build_materialization_input_digest(...)`,
  `validate_build_recipe_identity(...)`,
  `validate_build_materialization_input(...)`,
  `validate_build_materialization_result(...)`, and
  `validate_build_materialization_evidence(...)`.
- Added host-owned passive text rendering for build materialization evidence.

### Changed

- Source/build promotion now has first-class evidence vocabulary for the three
  distinct things the 0.44 design requires: reusable recipe identity,
  target-specific materialization input, and concrete output result.
- Materialization evidence is explicitly passive. Text output states
  `execution: none` and does not claim that a build, staging operation, or
  install occurred.
- The 0.44 status handoff now treats wiring materialization identity into
  promotion plans and operator reports as remaining work, rather than treating
  the basic materialization identity model as unimplemented.

### Validation

- Added validation for required recipe, materialization input, materialization
  result, and evidence IDs.
- Added validation for recipe build selectors and toolchain/environment
  identity fields.
- Added validation for config, lockfile, feature, input, wasm, compressed wasm,
  installed-module, and Candid digest shape.
- Materialization evidence now recomputes the materialization input digest and
  rejects stale computed digests, stale consistency flags, recipe ID mismatches,
  and result/input digest mismatches.

### Testing

- Added JSON round-trip and field-shape coverage for
  `BuildRecipeIdentityV1`, `BuildMaterializationInputV1`,
  `BuildMaterializationResultV1`, and `BuildMaterializationEvidenceV1`.
- Added passive text rendering coverage for materialization evidence.
- Added negative validation tests for blank recipe revision, invalid config
  digest, invalid output digest, stale computed input digest, stale linkage
  flags, and mismatched result/input digest.

## 0.44.2 - 2026-05-25 - promotion artifact identity reports

### Added

- Added `PromotionArtifactIdentityReportV1` as a passive artifact identity
  report for promotion planning.
- Added `RolePromotionArtifactIdentityV1` to preserve each role's promotion
  level, source locator kind, source locator, digest pins, and embedded-config
  digest.
- Added `PromotionArtifactIdentityKindV1` so reports distinguish sealed wasm,
  sealed compressed wasm, combined sealed identity, source/build identity, and
  deferred identity.
- Added `PromotionArtifactIdentityGroupV1` so roles resolving to the same
  artifact identity are grouped by deterministic identity key.
- Added `PromotionArtifactIdentityReportRequest`,
  `promotion_artifact_identity_report(...)`,
  `promotion_artifact_identity_report_from_inputs(...)`, and
  `validate_promotion_artifact_identity_report(...)`.
- Added host-owned passive text rendering for promotion artifact identity
  reports.

### Changed

- Promotion reporting now separates source locator kind from artifact identity
  semantics. A `local_wasm`, `previous_receipt_artifact`, or wasm-store source
  can resolve to the same sealed artifact identity without being treated as a
  different promotion identity solely because the locator type differs.
- Identity groups preserve all contributing source kinds and locators, so
  operator output can show dedupe/grouping without erasing provenance.
- The 0.44 status handoff now treats identity dedupe beyond passive report
  grouping as later work rather than an unmodeled gap.

### Validation

- Added validation for archived identity reports: schema version, required
  report ID, status/blocker consistency, duplicate role rejection, digest
  shape checks, blocker severity, identity group keys, grouped-role coverage,
  and stale group summaries.
- Invalid role sources are represented as hard-failure blockers in the passive
  report instead of becoming implicit promotion input.

### Testing

- Added JSON round-trip and field-shape coverage for
  `PromotionArtifactIdentityReportV1`.
- Added tests proving source kind and identity kind are distinct.
- Added tests proving different source locators can group under one sealed
  artifact identity.
- Added negative validation tests for duplicate roles, status/blocker mismatch,
  invalid digest shape, stale group keys, and ungrouped roles.

## 0.44.0 - 2026-05-25 - passive promotion readiness

### Added

- Added `RoleArtifactSourceV1` and `RoleArtifactSourceKindV1` for role-scoped
  promotion artifact source declarations.
- Added `PreviousArtifactReceiptKindV1` for deployment/staging receipt-backed
  artifact sources.
- Added `RolePromotionInputV1` and `PromotionArtifactLevelV1` for sealed-wasm
  versus source/build promotion policy.
- Added `PromotionReadinessV1` and `RolePromotionReadinessV1` as the first
  passive promotion readiness artifacts.
- Added `PromotionReadinessRequest` and `check_promotion_readiness(...)` as the
  host-owned passive entry point for future CLI/report consumers.
- Added host-owned passive text rendering for `PromotionReadinessV1`.

### Changed

- Executable promotion override inputs must be digest-pinned before they can be
  treated as usable source identity.
- Receipt-backed artifact sources are limited to deployment receipt and staging
  receipt evidence. Authority dry-run receipts/evidence are not representable
  as artifact sources.
- Sealed-wasm readiness requires embedded-config identity to match the target.
- Source/build readiness permits target config digest changes without weakening
  sealed-wasm checks.
- Readiness reports now surface target-store restage requirements as warnings.

### Validation

- Added validation for artifact source locator requirements, digest pins,
  lowercase sha256 digest shape, and previous-receipt eligibility.
- Added validation for archived `PromotionReadinessV1` artifacts: schema,
  identity fields, status/blocker consistency, duplicate roles, digest shape,
  restage state, and finding severity.

### Testing

- Added JSON round-trip and field-shape coverage for the initial promotion
  source, input, and readiness DTOs.
- Added readiness coverage for ready roles, restage warnings, sealed-wasm
  embedded-config mismatch, source/build target config changes, invalid
  sources, missing target roles, and source-role mismatch.
- Added negative validation tests for status/blocker mismatch, duplicate
  readiness roles, restage inconsistency, invalid digest shape, and finding
  severity mismatch.
