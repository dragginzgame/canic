## 0.58.13 - 2026-06-03 - ICP refill grant ledger reuse

### 📝 Summary

- Reused the existing cycles-funding child grant ledger for successful
  registered direct-child ICP refills. After CMC `notify_top_up` completes,
  direct-child refill cycle totals now feed the same budget and cooldown
  accounting as ordinary child grants.

- Kept the follow-up inside the existing funding model: no refill-specific
  grant store, refill record shape, endpoint shape, CLI shape, or metrics
  family was added.

### ➕ Added

- Added workflow-side grant ledger recording for completed direct-child refill
  records whose registered parent matches the refill source canister.

- Added focused pure helper coverage proving matching parents are recorded,
  non-child or mismatched-parent targets are ignored, and very large CMC cycle
  totals saturate into bounded `u128` ledger accounting.

### 🔧 Changed

- Completed CMC notify totals for registered direct-child refills now pass
  through `CyclesFundingLedgerOps::record_child_grant`, making subsequent
  budget and cooldown checks observe the refill.

- Resolved direct-child grant eligibility through the existing children
  projection. Hub self-refill, manual non-child refills, and unregistered
  targets do not create synthetic child grant ledger entries.

## 0.58.0 - 2026-06-02 - ICP-to-cycles refill primitive design

### 📝 Summary

- Started the 0.58 line with the ICP-to-cycles refill primitive design:

  ```text
  docs/design/0.58-convert-icp/0.58-design.md
  ```

- The line is scoped to a Canic-managed canister-side refill primitive that
  transfers ICP to the CMC top-up account, calls `notify_top_up`, records
  recovery state, and integrates with the existing funding chain.

### 🧭 Design

- Defines the canonical canister-funded flow: explicit source and target
  canisters, ICRC-1 transfer to the CMC top-up account, ledger block capture,
  direct CMC `notify_top_up`, compact record persistence, retry, and bounded
  metrics.

- Keeps identity-funded conversion out of scope so `canic cycles` remains the
  existing thin `icp cycles` wrapper instead of becoming a second native funds
  engine.

- Requires config-aware ICP ledger and CMC ID resolution, canonical mainnet
  defaults, local/custom overrides, runtime fee and decimals queries, checked
  ledger block-index conversion, and direct CMC notification.

- Narrows the 0.58.0 config surface to the MVP refill controls:
  enabled/presence, `min_hub_cycles_before_refill`,
  `max_refill_e8s_per_call`, and optional `min_xdr_permyriad_per_icp`.
  Broader treasury controls stay follow-up work unless implementation proves
  one is required.

- Defines recovery semantics for `Requested`, transferred, processing,
  duplicate, bad-fee, refunded, transaction-too-old, invalid-transaction, and
  completed states.

- Separates `IcpRefillRecord` as authoritative recovery state from
  `CycleTopupEvent` as append-only operator observability, so refill retry does
  not depend on the existing top-up query surface.

- Pins the hub self-refill hook to `CycleTrackerWorkflow` and records that
  timer-driven self-refill may defer to 0.58.1 if it cannot stay inside the
  existing funding interval.

- Defines the composable endpoint macro guard shape and requires fabricated
  dry-run output to state that it does not call the canister refill endpoint.

- Keeps local fabrication explicit and local-only, separate from the
  production canister-side refill endpoint.

- Adds an implementation sequence covering DTOs, storage ops, IC infra,
  `ops::ic`, policy checks, workflow orchestration, opt-in endpoint/macros,
  funding-chain integration, local fabrication, and an optional thin CLI
  trigger.

- Adds a validation plan for subaccount derivation, ledger/CMC ID resolution,
  policy gates, storage transitions, checked ledger block conversion, CMC
  recovery branches, PocketIC endpoint/funding integration, local fabrication,
  and CLI delegation.

- Defers the thin CLI trigger from the 0.58.0 exit gate; if added, it must
  resolve existing cycles targets and call the canister update endpoint rather
  than implementing ledger transfer itself.

### 🧹 Cleanup

- Reframes the existing 0.58 design draft as `0.58.0` and preserves the hard
  boundaries around CLI surface, funding policy, metrics, storage mutation, and
  endpoint layering.

### 🧪 Testing

- Validated the changelog/design update with:

  ```text
  cargo test -p canic --test changelog_governance
  git diff --check
  ```
