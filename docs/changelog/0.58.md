## 0.58.3 - 2026-06-03 - Endpoint macro, hub timer hook, and thin convert trigger

### 📝 Summary

- Exposed the manual ICP refill primitive through an opt-in facade endpoint
  macro, wired configured root hub self-refill into the existing cycle tracker
  timer, and added the retained thin `canic cycles convert` operator trigger.

### ➕ Added

- Added the opt-in `icp-refill` facade feature plus
  `canic_emit_icp_refill_endpoints!(guard = ...)`. Host canisters can now emit
  a guarded `canic_icp_refill` update method that delegates dry-run and live
  requests to the existing workflow and keeps retry on the same endpoint.

- Added the facade API wrapper that returns either a dry-run refill preview or
  a live refill response from the shared workflow.

- Added the root hub self-refill hook on `CycleTrackerWorkflow`. When
  `topup.icp_refill` is enabled and the sampled root balance falls below
  `min_hub_cycles_before_refill`, the existing timer path schedules or resumes
  the ICP refill workflow before child grant fan-out.

- Added `canic cycles convert` as a thin CLI trigger for canister-side ICP
  refill:

  ```text
  canic cycles convert <deployment> <role-or-canister> --source <role-or-canister> --icp-e8s <amount>
  ```

  The command resolves source and target canisters from installed deployment
  state, builds the `IcpRefillRequest` Candid payload, and calls the guarded
  `canic_icp_refill` update method. It does not implement ledger transfer or
  CMC notification locally.

- Added local-only fabrication under the same command family:

  ```text
  canic cycles convert <deployment> <role-or-canister> --fabricate --cycles <amount>
  ```

  Fabrication mode calls local `provisional_top_up_canister`, rejects non-local
  networks, and labels dry-run text/JSON with:

  ```text
  mode=fabricate (does not call canister refill endpoint)
  ```

### 🔧 Changed

- Extended endpoint access-expression parsing so macro-supplied grouped guard
  expressions unwrap cleanly before validation.

- Added an ICP CLI display helper for update calls with explicit Candid
  arguments, so dry-runs can show the exact canister method and payload used by
  `canic cycles convert`.

## 0.58.2 - 2026-06-02 - ICP refill IC helpers and manual workflow

### 📝 Summary

- Added the ICP ledger / CMC helper layer, pure refill policy gates, storage
  transition helpers, and the reusable manual canister-side refill workflow
  for the 0.58 ICP-to-cycles primitive.

### ➕ Added

- Added canonical ICP ledger and CMC canister IDs plus config-aware helper
  resolution that rejects mainnet overrides unless explicitly allowed.

- Added `infra::ic::icp_refill` and `ops::ic::icp_refill` helpers for CMC
  top-up subaccount derivation, ICRC-1 transfer argument construction,
  runtime `icrc1_fee` / `icrc1_decimals` queries, direct `notify_top_up`,
  ICP/XDR rate query, and checked ledger block-index conversion.

- Added pure refill policy gates for manual refill requests and hub
  self-refill eligibility, including enabled/presence checks, per-call caps,
  optional ICP/XDR rate gate enforcement, low-hub-cycle threshold handling,
  and active-operation concurrency denial.

- Added deterministic refill record transitions for requested, transferred,
  duplicate-transfer recovery, bad-fee retry preparation, notify processing,
  completed, refunded, invalid transaction, transaction-too-old, stale
  pre-ledger retry, notify failure, and notify max-attempt terminal states.

- Added the manual canister-side refill workflow skeleton. The workflow now
  persists transfer identity before transfer, reuses persisted transfer
  identity for same-`operation_id` retry, recovers ledger `Duplicate` results,
  blocks automatic pre-ledger retry after the 24-hour ICRC-1 deduplication
  window, retries notify from known `ledger_block_index`, caps manual notify
  attempts at five, validates ICP ledger decimals, and reports dry-run
  estimated cycles from the current ICP/XDR rate.

## 0.58.0 - 2026-06-02 - ICP-to-cycles refill primitive design

### 📝 Summary

- Started the 0.58 line with the ICP-to-cycles refill primitive design:

  ```text
  docs/design/0.58-convert-icp/0.58-design.md
  ```

- The line is scoped to a Canic-managed canister-side refill primitive that
  transfers ICP to the CMC top-up account, calls `notify_top_up`, records
  recovery state, and integrates with the existing funding chain.

### 🧭 Design

- Defines the canonical canister-funded flow: explicit source and target
  canisters, ICRC-1 transfer to the CMC top-up account, ledger block capture,
  direct CMC `notify_top_up`, compact record persistence, retry, and bounded
  metrics.

- Keeps identity-funded conversion out of scope so `canic cycles` remains the
  existing thin `icp cycles` wrapper instead of becoming a second native funds
  engine.

- Requires config-aware ICP ledger and CMC ID resolution, canonical mainnet
  defaults, local/custom overrides, runtime fee and decimals queries, checked
  ledger block-index conversion, and direct CMC notification.

- Narrows the 0.58.0 config surface to the MVP refill controls:
  enabled/presence, `min_hub_cycles_before_refill`,
  `max_refill_e8s_per_call`, and optional `min_xdr_permyriad_per_icp`.
  Broader treasury controls stay follow-up work unless implementation proves
  one is required.

- Defines recovery semantics for `Requested`, transferred, processing,
  duplicate, bad-fee, refunded, transaction-too-old, invalid-transaction, and
  completed states.

- Separates `IcpRefillRecord` as authoritative recovery state from
  `CycleTopupEvent` as append-only operator observability, so refill retry does
  not depend on the existing top-up query surface.

- Pins the hub self-refill hook to `CycleTrackerWorkflow` and records that
  timer-driven self-refill may defer to 0.58.1 if it cannot stay inside the
  existing funding interval.

- Defines the composable endpoint macro guard shape and requires fabricated
  dry-run output to state that it does not call the canister refill endpoint.

- Keeps local fabrication explicit and local-only, separate from the
  production canister-side refill endpoint.

- Adds an implementation sequence covering DTOs, storage ops, IC infra,
  `ops::ic`, policy checks, workflow orchestration, opt-in endpoint/macros,
  funding-chain integration, local fabrication, and an optional thin CLI
  trigger.

- Adds a validation plan for subaccount derivation, ledger/CMC ID resolution,
  policy gates, storage transitions, checked ledger block conversion, CMC
  recovery branches, PocketIC endpoint/funding integration, local fabrication,
  and CLI delegation.

- Defers the thin CLI trigger from the 0.58.0 exit gate; if added, it must
  resolve existing cycles targets and call the canister update endpoint rather
  than implementing ledger transfer itself.

### 🧹 Cleanup

- Reframes the existing 0.58 design draft as `0.58.0` and preserves the hard
  boundaries around CLI surface, funding policy, metrics, storage mutation, and
  endpoint layering.

  ```text
  git diff --check
  ```
