## 0.40.13 - 2026-05-21 - protected-call guard closeout

### Changed

- Aligned the 0.40 attested-call design notes with the current raw-ingress
  protected wrapper model: protected Candid methods expose no typed input,
  while `CanicCall` sends `CanicInternalCallEnvelopeV1` in raw ingress bytes
  for Canic-owned decoding.
- Made `CanicCall` encode the internal-call envelope explicitly and dispatch it
  through `with_raw_args(...)`, keeping the call boundary aligned with the
  no-argument protected wrapper ABI.
- Added a protected-call guard assertion that prevents `CanicCall` from
  regressing back to a typed envelope argument dispatch.
- `CanicCall` now rejects empty target methods and zero effective proof TTLs
  before issuing a root proof request.
- Protected endpoint descriptors and handwritten `CanicCall` role selection
  now reject whitespace-only method/role metadata locally.
- Added focused coverage that generated protected clients reject explicit caller
  roles not accepted by the endpoint descriptor before transport/root-proof
  work begins.
- Clarified the 0.40 design notes that the outgoing internal invocation proof
  cache is heap-only in this release; stable cache persistence is deferred
  because the cache is only a performance optimization.
- Reworked the 0.40 design implementation-state sections so they describe the
  current raw-ingress wrapper, generated-client, root-issuance, cache, and
  endpoint-classification surfaces instead of earlier planning text.
- Strengthened the protected raw-call source guard so it scans full raw-call
  expressions and catches multi-line protected method literals or constants,
  while still allowing external calls and structural query exceptions.
- Hardened protected endpoint discovery in the source guard so nested
  `caller::has_any_role([...])` role arrays do not truncate attribute parsing.
- Tightened raw-call pattern matching so allowed `CanicCall::...` usage is not
  misclassified as forbidden raw `Call::...` usage.

## 0.40.10 - 2026-05-20 - root-owned attestation epochs

### Changed

- Root role-attestation issuance now uses the root's current accepted role
  epoch for the requested role instead of copying the caller-supplied
  `RoleAttestationRequest::epoch`. Internal invocation proofs already used the
  root-generated epoch path; this aligns the older role-attestation issuance
  family with the same authority model.
- Root replay payload identity for role-attestation issuance now ignores the
  legacy request epoch field, matching the fact that root no longer treats it
  as authoritative input.
- Root capability proof hashing now uses the same canonical request payload and
  ignores the legacy role-attestation request epoch as well, so replay identity
  and signature binding agree on which request fields are authoritative.
- Outbound root-response attestation caching now treats the configured local
  role epoch as a floor, not an exact cache key. A root-issued proof at a newer
  epoch is reusable, while cached proofs below the local floor are still
  invalidated.
- Added fixed hash domains to the remaining root request metadata and
  capability nonce streams, so independent per-canister counters in separate
  request families cannot produce identical request or nonce bytes from the
  same timestamp/caller/canister entropy.

## 0.40.1 - 2026-05-20 - outgoing internal-proof cache

### Added

- Added a heap-only outgoing cache for `CanicCall` method-scoped internal
  invocation proofs. The cache is keyed by the exact root, configured
  attestation key name, subject canister, caller role, subnet, audience
  canister, exported method, and requested TTL.
- Cached proofs are reused only while they still bind to the requested call
  edge, remain outside the refresh margin, and meet the local minimum accepted
  role epoch floor. Stale or mismatched entries are evicted and the caller
  requests a fresh root-signed proof.
- Added coarse internal-call auth error codes for retry classification:
  `AuthMaterialStale`, `AuthKeyUnknown`, `AuthProofExpired`, and
  `InternalRpcMalformed`.
- `CanicCall` now inspects protected endpoint `Result` responses, invalidates
  the cached proof, requests fresh proof material, and retries once when the
  callee reports stale role-epoch material or an unknown attestation key.

### Boundary

- The cache is an optimization only. Callee-side wrapper verification remains
  authoritative for signature, caller, role, audience, method, subnet, TTL, and
  epoch checks.
- Retry is intentionally narrow. `CanicCall` does not retry malformed
  envelopes, expired proofs, authorization failures, role/method/audience
  mismatches, or domain handler errors.

### Testing

- Ran `cargo test -p canic-core api::ic::call --lib`.

  Candid arguments only after proof validation succeeded. This typed-envelope
  Candid shape was superseded in `0.40.12` by no-argument raw-ingress protected
  wrappers.
- Protected internal wrappers reject mixed access predicates in this slice, so
  an attested-role endpoint cannot silently drop an extra `requires(...)`
  condition while switching to the envelope verifier path.
- Added the low-level `CanicCall` public API. It Candid-encodes the original
  endpoint arguments, requests a method-scoped internal invocation proof from
  root for the configured caller role, builds a `CanicInternalCallEnvelopeV1`,
  and dispatches through the raw call layer. `Call` remains unchanged for
  external services.

### 📝 Boundary

- This slice only establishes the 0.40 protocol data model and signing domain.
  Existing internal endpoints are not mechanically migrated; only endpoints that
  opt into `caller::has_role(...)` use the protected envelope wrapper path.
- Application handlers keep their natural Rust signatures. `CanicCall` is the
  first low-level caller primitive; generated clients, outgoing proof caching,
  and retry-on-stale-material behavior are still staged follow-up work.
- Root internal-proof issuance authorizes the requesting subject through either
  Root AppIndex role ownership or the subnet registry role. This deliberately
  avoids the old `caller::is_registered_to_subnet()` prefilter so AppIndex
  canisters can request method-scoped proofs.
- Root chooses the signed proof epoch from its current role-attestation epoch
  floor instead of accepting a caller-provided epoch in the internal-proof
  request.
