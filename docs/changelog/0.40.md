# Canic 0.40

## 0.40.3 - 2026-05-20 - protected call guardrails

### Added

- Added a canonical `canic-core::protocol` list for wasm-store update methods
  that require a protected internal invocation proof.
- Added a canonical `canic-core::protocol` list for wasm-store structural query
  exceptions while protected query endpoints remain out of scope.
- Added a first-party source guard test that rejects raw `Call`, `CallOps`, or
  `ic_cdk::call::Call` use with known protected wasm-store method names.
- The same guard now verifies that the wasm-store macro declarations and
  checked-in `wasm_store.did` agree with the protected-method and query-exception
  manifests.
- The macro and `.did` checks are bidirectional: extra protected envelope
  methods or structural query exceptions must be added to the manifest instead
  of appearing only in generated endpoint shape.
- Added a built-in internal endpoint classification manifest for the 0.40
  model. First-party macro-emitted internal endpoints must now be classified as
  protected internal RPC, structural bootstrap, structural query exception,
  existing capability RPC, discovery/standards, or operator/controller raw path.
- Added macro coverage that protected internal endpoint wrappers verify
  invocation proofs against the exported wire method name, including
  `name = "..."` overrides.

### Changed

- Routed the control-plane wasm-store method classifier through the canonical
  protected-method list instead of maintaining a local duplicate match.

## 0.40.2 - 2026-05-20 - protected updates and ic-memory runtime

### Added

- Migrated the local wasm-store update endpoints to the 0.40 protected
  internal-call ABI. The update methods now require a root-signed
  `caller::has_role("root")` internal invocation proof and export
  `CanicInternalCallEnvelopeV1` instead of their former raw argument tuples.
- Routed root control-plane wasm-store update calls through `CanicCall`, so
  prepare, manifest stage, chunk publish, chunk/info reads used by installation,
  and store-local GC updates obtain method-scoped root proofs.
- Aligned the direct root auth RPC decode path with the direct endpoint ABI for
  role attestations and internal invocation proofs. The local root workflow still
  returns the root capability response envelope, while remote direct endpoint
  calls now decode the signed proof payload they actually request.
- Refreshed the canonical `crates/canic-wasm-store/wasm_store.did` to reflect
  the envelope ABI for protected update methods and the new 0.40 auth error
  codes.
- Added wasm-store reconcile coverage for the hard-cut boundary: raw legacy
  update calls no longer dispatch against protected update methods.

### Boundary

- Wasm-store catalog/status query methods remain structural root-query
  exceptions in V1 because protected attested queries are intentionally out of
  scope for 0.40.
- The existing root capability RPC and bootstrap/debug root endpoints are not
  double-wrapped in this slice.
- Canic now consumes the generic `ic-memory 0.5.1` static range/declaration
  registry. `canic-core` and `canic-control-plane` declare their framework
  ranges through `ic-memory`, while Canic keeps only the eager TLS touch helper
  for framework storage wrappers.
- The compatibility `canic_core::ic_memory_key!` macro now registers
  declarations and opens memory through `ic-memory`, with a Canic bootstrap
  readiness guard kept in front so host tests and canister lifecycle paths
  still initialize memory before stable stores are touched.
- Removed the remaining Canic open wrapper/error mapping and the shadow
  `canic-tests` policy adapter, leaving Canic-specific namespace policy tests
  next to the production policy adapter.
- Split the host-side test harness boundary so published `canic-testkit` no
  longer depends on any Canic runtime crate. Generic PocketIC/artifact helpers
  stay in `canic-testkit`, while Canic-specific role/init/readiness helpers now
  live in unpublished `canic-testing-internal`.

## 0.40.1 - 2026-05-20 - outgoing internal-proof cache

### Added

- Added a heap-only outgoing cache for `CanicCall` method-scoped internal
  invocation proofs. The cache is keyed by the exact root, configured
  attestation key name, subject canister, caller role, subnet, audience
  canister, exported method, and requested TTL.
- Cached proofs are reused only while they still bind to the requested call
  edge, remain outside the refresh margin, and meet the local minimum accepted
  role epoch floor. Stale or mismatched entries are evicted and the caller
  requests a fresh root-signed proof.
- Added coarse internal-call auth error codes for retry classification:
  `AuthMaterialStale`, `AuthKeyUnknown`, `AuthProofExpired`, and
  `InternalRpcMalformed`.
- `CanicCall` now inspects protected endpoint `Result` responses, invalidates
  the cached proof, requests fresh proof material, and retries once when the
  callee reports stale role-epoch material or an unknown attestation key.

### Boundary

- The cache is an optimization only. Callee-side wrapper verification remains
  authoritative for signature, caller, role, audience, method, subnet, TTL, and
  epoch checks.
- Retry is intentionally narrow. `CanicCall` does not retry malformed
  envelopes, expired proofs, authorization failures, role/method/audience
  mismatches, or domain handler errors.

- Ran `cargo test -p canic-core api::ic::call --lib`.

## 0.40.0 - 2026-05-19 - internal invocation proof ABI

### ➕ Added

- Added passive Candid DTOs for the 0.40 protected internal-call wire ABI:
  `CanicInternalCallEnvelopeV1`, `CanicInternalCallHeaderV1`,
  `InternalInvocationProofRequest`, `InternalInvocationProofPayloadV1`, and
  `SignedInternalInvocationProofV1`.
- Added a separate internal invocation proof signing domain,
  `CANIC_INTERNAL_INVOCATION_PROOF_V1`, so method-scoped invocation proofs
  cannot be confused with generic role attestations.
- Added hash coverage proving method-scoped internal invocation proofs use a
  distinct signing domain from existing role attestations.
- Added root capability issuance for `InternalInvocationProofRequest`, including
  replay binding, root capability metrics, direct root endpoint routing, and
  signed `InternalInvocationProofPayloadV1` responses.
- Added verifier-side internal invocation proof validation that checks
  signature material with the internal proof domain plus caller, audience,
  exported method, accepted role, subnet, TTL, and epoch bindings.
- Added the `caller::has_role(...)` and `caller::has_any_role([...])` endpoint
  macro predicates for 0.40 protected update endpoints. The generated wrapper
  now exports the `CanicInternalCallEnvelopeV1` ABI, verifies the method-scoped
  proof before dispatch, and decodes the original Candid arguments only after
  proof validation succeeds.
- Protected internal wrappers reject mixed access predicates in this slice, so
  an attested-role endpoint cannot silently drop an extra `requires(...)`
  condition while switching to the envelope verifier path.
- Added the low-level `CanicCall` public API. It Candid-encodes the original
  endpoint arguments, requests a method-scoped internal invocation proof from
  root for the configured caller role, builds a `CanicInternalCallEnvelopeV1`,
  and dispatches through the raw call layer. `Call` remains unchanged for
  external services.

### 📝 Boundary

- This slice only establishes the 0.40 protocol data model and signing domain.
  Existing internal endpoints are not mechanically migrated; only endpoints that
  opt into `caller::has_role(...)` use the protected envelope wrapper path.
- Application handlers keep their natural Rust signatures. `CanicCall` is the
  first low-level caller primitive; generated clients, outgoing proof caching,
  and retry-on-stale-material behavior are still staged follow-up work.
- Root internal-proof issuance authorizes the requesting subject through either
  Root AppIndex role ownership or the subnet registry role. This deliberately
  avoids the old `caller::is_registered_to_subnet()` prefilter so AppIndex
  canisters can request method-scoped proofs.
- Root chooses the signed proof epoch from its current role-attestation epoch
  floor instead of accepting a caller-provided epoch in the internal-proof
  request.
