## 0.25.3 - 2026-04-05 - Delegation Hot-Path Follow-Through

### 📝 Summary

- `0.25.3` continues the `0.25` runtime follow-through with another measured cut on the delegated-auth issuance path, keeping the work focused on local auth/replay overhead instead of broader transport or demo-flow redesign.

### 🔧 Changed

- Replaced the compact successful `DelegationIssued` replay payload path with a small internal binary codec, so the root replay cache stops paying generic serializer overhead for the hottest delegation replay shape.
- Added the same style of compact internal replay encoding for cached `CyclesResponse`, so the root cycles replay path no longer stores and decodes the tiny success shape through the generic `Response` codec.
- Switched delegation cert hashing to direct deterministic field hashing instead of re-encoding the cert before hashing, which removes more avoidable auth-path work from both proof issuance and verifier-proof cache keying.
- Tightened the verifier-local proof cache path by avoiding redundant shard-public-key rewrites and unnecessary proof-record cloning when replacing an existing keyed proof entry.
- Removed the root public-key prewarm from `sign_delegation_cert(...)`, so the root signing path only resolves what it actually needs to sign and leaves verification-side key fetches to the verifier cache path.

### 📊 Audit

- The retained rerun at [instruction-footprint-11.md](../audits/reports/2026-04/2026-04-05/instruction-footprint-11.md) cut sampled `root::canic_request_delegation:fresh-shard` from `1993625` in `instruction-footprint-10` to `1770255`, and `commit_replay` dropped out of the delegation hotspot table after the replay compaction work.
- The follow-up retained rerun at [instruction-footprint-12.md](../audits/reports/2026-04/2026-04-05/instruction-footprint-12.md) cut sampled `root::canic_response_capability_v1` `cycles-request` from `1481137` to `601860`, and the old cycles `commit_replay` hotspot dropped out after the compact cached cycles response path landed.
- The remaining main delegation stages are now `sign_cert`, `encode_install_request`, `decode_response`, and the smaller `cache_root_verifier_proof` path, so the easy replay-side savings are largely exhausted.

## 0.25.2 - 2026-04-05 - Delegation Provisioning Runtime Trim

### 📝 Summary

- `0.25.2` starts the post-audit runtime follow-through by trimming avoidable work in delegated-auth proof provisioning and the `root::canic_request_delegation` hot path, while keeping the functional auth/runtime checks green.

### 🔧 Changed

- Threaded optional shard public key material through canonical delegation provisioning so shard-driven issuance can hand verifier installs the key material they already resolved, instead of forcing the root/verifier lane to rediscover it later.
- Updated the local verifier proof-store path to accept provided shard public keys directly and reuse them when caching proof verification material.
- Changed delegation proof push fanout to pre-encode one `DelegationProofInstallRequest` payload per push batch and reuse raw call args for each verifier target instead of Candid-encoding the same request repeatedly.
- Restored `auth-crypto` on the internal `canister_test` role crate so the stricter demo/test split still supports the auth-heavy root/instruction-audit lanes.

### 📊 Audit

- A fresh retained rerun at [instruction-footprint-4.md](../audits/reports/2026-04/2026-04-05/instruction-footprint-4.md) confirms that the targeted `prepare_call` checkpoint in `workflow/auth.rs` dropped sharply after the shared payload-encoding cut.
- The same rerun also shows that `root::canic_request_delegation` is still the highest shared instruction hotspot overall, so the next real reduction target remains proof issuance / replay / root verifier caching rather than more call-builder cleanup.

## 0.25.1 - 2026-04-05 - Auth Decomposition and Demo/Test Split

### 📝 Summary

- `0.25.1` follows the audit refresh with the first structural reduction pass: the main auth/runtime hotspots are split into smaller modules, and the reference demo canisters are separated more strictly from internal test-only canisters and helper flows.

### 🔧 Changed

- Split the old monolithic `storage/stable/auth.rs` into focused modules for auth records, proofs, sessions, and key state while preserving the existing `DelegationState` facade.
- Split `access/auth.rs` into explicit auth-phase modules so identity/session resolution, token verification, and predicate evaluation are no longer concentrated in one file.
- Split `access/expr.rs` and `ops/runtime/metrics/auth.rs` into directory modules so auth expression evaluation and auth metrics families are easier to reason about and extend without accreting more unrelated logic in one place.
- Moved the `test` role out of `canisters/` and into `crates/canic-core/test-canisters/`, so the default demo/reference topology no longer auto-creates or advertises a test helper canister.
- Split internal correctness fixtures from audit probes by keeping test-only canisters under `crates/canic-core/test-canisters/` and moving audit-only probes under `crates/canic-core/audit-canisters/`.
- Removed the debug-only root helper endpoints from the demo `root` canister and aligned the demo configs, DFX surface, and reference-canister lists with the slimmer topology.
- Added fallible PocketIC setup paths to public `canic-testkit`: downstreams can now use `try_pic()`, `PicBuilder::try_build()`, and `try_install_prebuilt_canister_with_cycles(...)` to skip cleanly when PocketIC cannot start instead of preflighting `POCKET_IC_BIN` themselves.
- Hardened the PocketIC process-lock path so `std::env::temp_dir()` and its parent chain are created before `canic-pocket-ic.lock`, which keeps repo-local `TMPDIR` paths usable without downstream temp-root preflight glue.

### 🧪 Testing

- `make test` passed after the split, covering the workspace gates plus the root hierarchy, reconcile, and PocketIC auth suites against the new module layout and demo/test boundary.
- `cargo clippy -p canic-testkit --all-targets -- -D warnings` and `cargo test -p canic-testkit` passed after the new fallible testkit setup surface and temp-root lock-path coverage were added.

### 🧭 Migration Notes

- The default demo topology is now intentionally smaller: `test` is no longer part of the standard reference canister set and only appears through internal test configs and internal test canister crates.
- Downstream or local scripts that assumed a default root helper like `create_minimal` or a default `test` demo canister should now use explicit internal test fixtures or pass an explicit bench method instead.

## 0.25.0 - 2026-04-05 - Recurring Audit Refresh

### 📝 Summary

- `0.25.0` is a narrow audit-refresh line. The goal is to rerun the recurring audits once, keep retained summary outputs only, and limit fixes to issues the fresh audit results actually justify.

### 📊 Audit

- The `0.25` line is explicitly audit-led rather than feature-led.
- Refreshed retained summaries now cover:
  - `layer-violations`
  - `capability-surface`
  - `wasm-footprint`
  - `instruction-footprint`
  - `bootstrap-lifecycle-symmetry`
  - `change-friction`
  - `complexity-accretion`
  - `subject-caller-binding`
  - `canonical-auth-boundary`
  - `capability-scope-enforcement`
  - `expiry-replay-single-use`
  - `token-trust-chain`
  - `audience-target-binding`
  - `auth-abstraction-equivalence`
- The broad result is that the auth and capability invariants still hold, while the most important remaining pressure is structural complexity in `access/auth`, `access/expr`, `storage/stable/auth`, and `ops/runtime/metrics/auth`.
- The line keeps retained markdown summaries instead of bulky generated artifact trees by default.

### 🔧 Changed

- Root `CHANGELOG.md` now opens the `0.25.x` minor line.
- `0.25` is defined as a measurement-and-correction line rather than another broad runtime or topology refactor.
- The audit follow-through already removed bulky generated output trees in favor of summary-only retained reports for the `0.25` sweep.

### 🧭 Migration Notes

- `0.25.0` does not introduce a new API model by itself.
- The practical goal is to refresh the current audit picture, correct real drift, and hand off anything broader than audit-backed fixes to a later minor line.
- The main handoff from the current sweep is complexity reduction, not a broken invariant: the next meaningful structural work should target auth/runtime complexity hotspots rather than reopen already-green invariants.
