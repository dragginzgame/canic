## 0.33.2 - 2026-05-09 - Auth and replay audit hardening

### 📝 Summary

- Completed the next recurring auth/capability/replay audit slice with concrete
  remediation instead of leaving medium-risk watchpoints.
- Tightened delegated-token, capability-proof, delegated-session, and root
  replay behavior around replay, expiry, and capacity boundaries.

### 🔧 Changed

- Root capability proof validation now returns a typed proof view used by
  verifier dispatch, logs, and metrics, reducing repeated raw proof-mode
  matching across the capability path.
- Root replay records now persist caller attribution as a hard-cut stable replay
  schema change with no legacy decode shim.
- Delegated session and session-bootstrap binding stores now reject per-subject
  or global capacity exhaustion instead of evicting unrelated subjects, limiting
  the blast radius of many wallet principals bootstrapping sessions for the same
  delegated subject.
- The delegated-session saturation guard keys pressure by `delegated_pid`, not
  by the wallet caller, because the abuse path is many wallet principals holding
  valid bootstrap tokens for the same delegated subject.
- Optional `start!` / `start_root!` `init = { ... }` blocks now execute from a
  zero-delay lifecycle timer and then schedule bootstrap/user continuation
  timers, keeping generated IC hooks restore/schedule-only.
- `IntentId` now lives in `canic-core::ids`, while stable-memory encoding stays
  in storage, so production workflow signatures no longer reference stable
  storage modules for a pure identifier.

### 🩹 Fixed

- Delegated-token update guards now reject active token replay by consuming each
  token nonce until the token expires. This is a hard-cut auth-state schema
  change with no legacy decode shim.
- Consumed delegated-token state is keyed by
  `(issuer_shard_pid, subject, cert_hash, nonce)`, pruned at token expiry, and
  bounded to fail closed if the active replay marker set reaches capacity.
- Root capability replay reservations now enforce a per-caller active cap before
  the global replay-store cap, preventing one registered caller from filling the
  shared replay table with unique request metadata.
- The replay saturation guard is aimed at registered caller canisters that are
  malicious, compromised, timer-driven, or user-triggered into issuing many
  fresh root capability requests; ordinary users still cannot write directly to
  the root replay store.
- Capability replay metadata and existing root replay records now reject the
  exact expiry boundary (`now >= expires_at`), matching delegated-token and
  consumed-token freshness semantics.
- Removed public `canic::api::auth::AuthApi::verify_token(...)`; delegated
  session bootstrap now uses a private token-material verifier so downstream
  code cannot mistake partial token verification for full endpoint
  authorization.

### 📊 Audit

- Added recurring reports for canonical auth boundary, capability scope
  enforcement, expiry/replay single-use, subject-caller binding, bootstrap
  lifecycle symmetry, and layer violations with remediation and watchpoints
  captured in the May audit summaries.
- Reduced capability scope residual risk by routing proof validation, dispatch,
  logs, and metrics through the validated proof view.
- Reduced expiry/replay residual risk by aligning capability metadata and root
  replay cache expiry with the exclusive delegated-token boundary.

### 🧪 Testing

- Verified targeted delegated-token replay, consumed-token pruning/capacity,
  capability proof-mode, auth identity, root replay boundary, and storage helper
  tests.
- Verified `canic` macro tests plus focused PocketIC lifecycle, root
  post-upgrade reconcile, and role-attestation fixture tests for lifecycle
  timer symmetry.
- Verified PocketIC root replay expiry and TTL-above-max smokes.
- Verified `canic` and `canic-core` clippy for all targets.
- Verified the core layering guard, focused request-handler/intent tests, and
  `canic-core` clippy after moving `IntentId` out of stable storage.

## 0.33.1 - 2026-05-09 - Operator status

### 📝 Summary

- Added native Canic commands for local replica lifecycle checks and quick
  project status.
- `canic status` is now the top-level "what is going on?" command for local
  development.

### ➕ Added

- Added `canic replica start`, `canic replica status`, and
  `canic replica stop` as local ICP replica lifecycle wrappers.
- Added `--debug` to the replica lifecycle commands for ICP CLI debug logging.
- Added top-level `canic status` as a quick project summary for local replica
  health, configured fleets, and deployed fleet count.
- `canic status` now shows the underlying ICP CLI version from the selected
  `--icp` executable.

### 🔧 Changed

- Local installs now fund newly created root canisters with explicit
  config-derived ICP CLI `create --cycles` funding and remove the old
  post-create cycle fabrication/top-up phase.
- `make update-dev` now checks npm for the latest ICP CLI and `ic-wasm`
  packages, uses those latest versions for the update run when available, and
  falls back to the repo pins when the lookup is unavailable.

### 📚 Documentation

- Added a current build-artifact vocabulary for `.icp`, `.canic`, generated
  Candid sidecars, Cargo wasm outputs, and Canic `.wasm.gz` artifacts.
- Consolidated non-fleet test, audit, and sandbox canister placement guidance in
  `TESTING.md`.
- Marked old 0.30/0.31 DFX-era backup/restore docs as historical.
- Cleaned up recurring audit definitions for domain-scoped layout, lifecycle
  evidence standards, and delegated-token/root replay freshness checks.

### 🩹 Fixed

- `canic status` now verifies recorded local roots against the running replica
  and reports stale install state instead of counting saved state files as live
  deployments.
- Local `canic status` now reports `partial` when the recorded root is live but
  its deployed registry is missing roles from the selected fleet config.
- `canic list` now shows local `.wasm.gz` artifact sizes when available and
  lists configured roles missing from the deployed root registry.

### 🧭 CLI Examples

```bash
canic replica start
canic replica start --debug
canic replica status
canic replica stop
canic status
canic status --network local
```

### 🧪 Testing

- Verified the new command help, focused status rendering tests, the full
  `canic-cli` library test suite, and the `canic-cli`/`canic-host` clippy
  gates.

## 0.33.0 - 2026-05-08 - dfx -> icp-cli

### 📝 Summary

- Hard-cut Canic from DFX project tooling to ICP CLI project tooling.
- `icp.yaml` is now the project-level canister and environment definition.
- Local build/install state now lives under `.icp`, while Canic install state
  remains under `.canic`.

### ⚠️ Breaking

- Removed `dfx.json` and the DFX-backed host adapter from the active CLI flow.
- Removed active DFX setup scripts and CI requirements; supported automation now
  installs and checks `icp` plus `ic-wasm`.
- Replaced DFX network propagation with `ICP_ENVIRONMENT` for build scripts and
  generated Canic build context.
- Updated operator-facing commands and examples from `--dfx <path>` to
  `--icp <path>` where a CLI override is still needed.

### 🔧 Changed

- `canic install <fleet>` now creates, tops up, installs, stages, resumes, and
  diagnoses through ICP CLI commands.
- `canic list <fleet>` and `canic medic <fleet>` now query the installed fleet
  through the ICP local endpoint and fleet-scoped install state.
- `canic snapshot download` and `canic restore run` now render ICP CLI snapshot
  commands for upload, restore, and resumable snapshot handling.
- `scripts/app/build.sh` adapts Canic builds to ICP CLI script builds by copying
  the uncompressed `.wasm` to `ICP_WASM_OUTPUT_PATH` while preserving Canic's
  `.wasm.gz` release artifacts.
- The default `icp.yaml` defines separate `demo` and `test` environments so the
  small demo fleet and fuller test topology can coexist in one repo.
- The minimal demo fleet and new fleet scaffolds explicitly disable delegated
  token auth unless the fleet config opts in.
- Delegated-token and role-attestation defaults now use ICP CLI's local
  threshold ECDSA key name, `key_1`, so auth-enabled fleets bootstrap on the
  local ICP environment without extra key configuration.
- Local developer setup installs ICP CLI npm tools under a user-writable npm
  prefix by default, so `make install-dev` no longer needs root-owned
  `/usr/local` npm permissions.
- `make update-dev` now uses the same user-writable ICP CLI install path and
  clears stale npm staging directories for interrupted `icp-cli`/`ic-wasm`
  installs before retrying.

### 🗑️ Removed

- Removed active DFX project files and helper scripts from the supported path.
- Removed DFX artifact discovery from Canic build support and testkit helpers.
- Removed stale DFX wording from current operator docs outside historical
  changelog entries.

### 🧭 CLI Examples

```bash
icp environment list
icp build root -e demo
canic install demo
canic list demo --network local
canic medic demo
canic restore run --journal restore-apply-journal.json --execute --icp icp
```

### 🧪 Testing

- Confirmed the local ICP smoke path with `icp 0.2.5` and `ic-wasm 0.9.10`.
- Verified `canic install demo`, `canic config demo`, `canic list demo`, and
  `canic medic demo` against the local ICP environment.
