# 0.14.x Detailed Changelog

Technical release notes for the `0.14.x` minor line.

## 0.14.0 - 2026-03-09 - Parent-Funded Cycles Control Plane

### 📝 Summary

- `0.14.0` introduces parent-funded subtree cycles as the default control-plane behavior and removes root-self funding assumptions from request handling.
- It strengthens funding safety with replay-aware execution semantics, explicit policy limits, and parent-scoped observability.

### 🔧 Changed

- Funding authorization now enforces direct-parent ownership for `MintCycles` requests.
- Replay handling now serves cached responses for identical replay requests and avoids false failure reporting when replay commit fails after successful execution.
- Replay conflict handling remains fail-closed for same `request_id` with mutated payloads.
- Added role-configurable funding policy controls via `CanisterTopup`: `max_per_request`, `max_per_child`, and `cooldown_secs`.
- Funding policy is enforced in the mint authorization path and tracked by a per-child funding ledger for cooldown and cumulative budget checks.
- Added an app-state global cycles funding kill switch (`cycles_funding_enabled`) and mapped kill-switch denials to public `Unavailable`.
- Metrics now include parent-emitted cycles funding totals and per-child counters for requested/granted/denied cycles with structured denial reasons.

### 🩹 Fixed

- Prevented successful capability execution from being reported as a hard failure when replay record finalization fails post-execution.
- Prevented idempotent retry paths from degrading into duplicate errors when an identical request already has a committed response.

### 📚 Documentation

- Added `0.14` design/status documentation for subtree cycles funding, including parent-only funding direction and runtime control expectations.

```text
replay guard -> authorization -> kill switch -> policy limits -> deposit_cycles
```
