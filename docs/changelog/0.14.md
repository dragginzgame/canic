## 0.14.3 - 2026-03-10 - Delegated Token Issue Naming Unification

### 📝 Summary

- `0.14.3` standardizes delegated-token issuance naming on `issue` across API, canister endpoints, integration tests, and auth metrics labels.
- It makes `DelegationApi::issue_token` the single app-facing issuance entrypoint and removes low-level issuance naming leaks.

### 🔧 Changed

- `DelegationApi::issue_token` is now the canonical issuance call, with proof refresh behavior handled inside the API.
- Low-level `sign_token` and proof-fetch internals are no longer public on `DelegationApi`.
- Renamed delegation test endpoints to `user_shard_issue_token` and `signer_issue_token`.
- Renamed signer access metric label from `mint_without_proof` to `issue_without_proof`.
- Updated delegation integration tests and contract/audit docs to use `issue` terminology consistently.

### ⚠️ Breaking

- Any direct caller using `user_shard_mint_token` or `signer_mint_token` must switch to `user_shard_issue_token` / `signer_issue_token`.
- Any dashboards or alerts keyed on `mint_without_proof` must switch to `issue_without_proof`.

### 🧭 Migration Notes

- Replace delegated-token API calls with `DelegationApi::issue_token(claims)`.
- Update any remaining endpoint call strings and operational metric filters from `mint` to `issue`.

## 0.14.2 - 2026-03-10 - Unified Metrics Query Endpoint

### 📝 Summary

- `0.14.2` introduces a single `canic_metrics` query surface that dispatches by metric family and returns typed results through `MetricsResponse`.
- It removes the per-metric `canic_metrics_*` endpoint variants and standardizes metrics access through one request/response contract.

### 🔧 Changed

- Added `dto::metrics::{MetricsKind, MetricsRequest, MetricsResponse}` as the boundary contract for metric queries.
- Added `MetricsQuery::dispatch(req)` to route to system/ICC/HTTP/timer/access/delegation/root-capability/cycles-funding/perf/endpoint-health snapshots.
- Added protocol endpoint constant `CANIC_METRICS = "canic_metrics"`.
- Updated endpoint-health dispatch to exclude `canic_metrics` from self-observation counting.
- Removed dedicated per-metric endpoint exports (`canic_metrics_system`, `canic_metrics_icc`, `canic_metrics_http`, `canic_metrics_timer`, `canic_metrics_access`, `canic_metrics_delegation`, `canic_metrics_root_capability`, `canic_metrics_cycles_funding`, `canic_metrics_perf`, `canic_metrics_endpoint_health`).

### ⚠️ Breaking

- Clients must migrate from `canic_metrics_*` calls to `canic_metrics` with `MetricsRequest { kind, page }`.

### 🧭 Migration Notes

- Replace per-metric endpoint calls with `canic_metrics` and set the target metric family via `MetricsKind`.

```text
canic_metrics(record { kind = variant { RootCapability }; page = record { limit = 100; offset = 0 } })
```

## 0.14.1 - 2026-03-09 - Funding Policy Config Removal

### 📝 Summary

- `0.14.1` removes the standalone `funding_policy` config contract and consolidates cycles configuration back to `topup_policy`.
- Parent funding requests now evaluate against default unbounded request limits again, so insufficient-balance denials reflect real parent balance instead of config clamping.

### 🔧 Changed

- Removed `funding_policy` from canister config schema and supporting test/builder scaffolding.
- Updated funding policy resolution to stop reading config-level request caps/cooldowns.
- Retained `topup_policy` as the only cycles-related role config surface (`threshold`, `amount`).

### ⚠️ Breaking

- Config files that still define `funding_policy` will now fail schema validation because unknown fields are denied.

### 🧭 Migration Notes

- Remove any `funding_policy` blocks from `canic.toml` and keep `topup_policy` only.

## 0.14.0 - 2026-03-09 - Parent-Funded Cycles Control Plane

### 📝 Summary

- `0.14.0` introduces parent-funded subtree cycles as the default control-plane behavior and removes root-self funding assumptions from request handling.
- It strengthens funding safety with replay-aware execution semantics, explicit policy limits, and parent-scoped observability.

### 🔧 Changed

- Funding authorization now enforces direct-parent ownership for `MintCycles` requests.
- Replay handling now serves cached responses for identical replay requests and avoids false failure reporting when replay commit fails after successful execution.
- Replay conflict handling remains fail-closed for same `request_id` with mutated payloads.
- Added role-configurable funding policy controls via `CanisterTopup`: `max_per_request`, `max_per_child`, and `cooldown_secs`.
- Funding policy is enforced in the mint authorization path and tracked by a per-child funding ledger for cooldown and cumulative budget checks.
- Added an app-state global cycles funding kill switch (`cycles_funding_enabled`) and mapped kill-switch denials to public `Unavailable`.
- Metrics now include parent-emitted cycles funding totals and per-child counters for requested/granted/denied cycles with structured denial reasons.

### 🩹 Fixed

- Prevented successful capability execution from being reported as a hard failure when replay record finalization fails post-execution.
- Prevented idempotent retry paths from degrading into duplicate errors when an identical request already has a committed response.

### 📚 Documentation

- Added `0.14` design/status documentation for subtree cycles funding, including parent-only funding direction and runtime control expectations.

```text
replay guard -> authorization -> kill switch -> policy limits -> deposit_cycles
```
