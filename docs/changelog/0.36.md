# Canic 0.36

## 0.36.4 - 2026-05-15 - Restore receipt and changelog checks

### 🩹 Fixed

- Restore apply journals now reject duplicate receipts for the same operation
  attempt, preventing hand-edited journals from carrying ambiguous command
  history.

### 🥾 Governance

- Added `make changelog-check` to enforce 88-column prose wrapping for the
  active root minor-line changelog section and its detailed changelog file.

### 🧪 Testing

- Added restore apply-journal coverage for duplicate operation receipt
  attempts.

## 0.36.3 - 2026-05-15 - Restore runner hardening

### 🩹 Fixed

- `canic restore run --execute` now fails an upload-snapshot step when the
  command succeeds but does not return an uploaded snapshot id. The runner no
  longer marks that operation complete without the receipt data required by the
  later load-snapshot step.
- `canic restore run --retry-failed` now moves the next failed restore
  operation back to ready after operator inspection, making failed restore
  steps explicitly recoverable without hand-editing the apply journal.
- Legacy text snapshot-upload parsing now only accepts the uploaded-snapshot
  label, so unrelated successful colon-labelled output cannot be mistaken for
  an uploaded snapshot id.
- `canic restore run` now rejects terminal apply-journal operations that do
  not have matching command receipts, so hand-edited completed or failed
  operations cannot silently steer a restore.

### 🧪 Testing

- Added restore-runner coverage for successful upload commands whose output is
  missing the uploaded snapshot id.
- Added restore CLI coverage for failed-operation retry recovery and the
  machine-readable `failed-recovered` runner receipt.
- Added parser coverage for blank JSON snapshot ids and unrelated legacy text
  output.
- Added restore CLI coverage proving a terminal operation without a matching
  receipt is rejected before any runner mode proceeds.

## 0.36.2 - 2026-05-15 - Backup execution integrity

### 🩹 Fixed

- `canic backup status`, `canic backup inspect`, and `canic backup verify` now
  report the same incomplete-layout error when a manifest-backed layout is
  missing its execution journal.
- Execution integrity now rejects terminal mutating operations that do not have
  matching operation receipts, so a tampered journal cannot mark backup work
  complete without receipt evidence.

### 🧪 Testing

- Added status, inspect, and verify coverage for manifest-backed layouts with
  missing execution journals.
- Added persistence coverage for rejecting completed mutating backup operations
  without matching receipts.

## 0.36.1 - 2026-05-15 - Backup resume safety

### 🩹 Fixed

- `canic backup create --out <dir>` now fails closed when `<dir>` already
  contains a backup plan for a different fleet, network, root, scope, target
  set, or operation graph. Compatible layouts are still preserved as the resume
  boundary.
- Existing backup output layouts now also have to match the requested
  authority and quiescence policy, preventing a dry-run plan from being reused
  as an executable backup layout.
- `canic backup create` output now includes a `LAYOUT` column so operators can
  tell whether the command created a new output layout or resumed an existing
  one.
- `canic backup list` now reports manifest-plus-plan layouts with a missing
  execution journal as `invalid-plan-journal` instead of `dry-run`.
- `canic backup create --out <dir>` now rejects manifest-backed layouts that
  are missing their execution journal instead of recreating one over the top of
  a corrupted backup directory.

### 🧪 Testing

- Added regression coverage proving incompatible existing backup output layouts
  are rejected instead of silently reusing the wrong plan.
- Added regression coverage for rejecting dry-run/executable layout mismatches.
- Added output coverage for the backup create layout-source indicator.
- Extended backup list execution-backed status coverage to missing execution
  journals.
- Added regression coverage for refusing to resume manifest-backed layouts with
  missing execution journals.

## 0.36.0 - 2026-05-15 - Backup/restore proofing begins

### 📚 Documentation

- Added the 0.36 Backup/Restore V1 design track, focused on proving and
  hardening the existing backup and restore execution code instead of starting
  a new design.
- Documented the target operator flow across backup create, list, inspect,
  verify, restore planning, dry-run apply, executable restore run, and status.

### 🧪 Testing

- Changed `canic backup list` to report execution-backed manifest layouts from
  their plan/execution journal state instead of treating every manifest-bearing
  directory as `ok`.
- Changed backup create persistence so an existing `--out` layout is preserved
  as the resume boundary instead of overwriting its plan and execution journal.
- Tightened `canic backup verify` for execution-backed backups so a finalized
  manifest and download journal are not accepted when the execution journal is
  still incomplete or mismatched.
- Tightened `canic backup status --require-complete` so a completed execution
  journal is not accepted until the finalized manifest exists.
- Added backup runner coverage proving a max-step limited backup can resume
  without replaying accepted preflight work or completed mutations.
- Added backup runner coverage proving a failed snapshot operation is durably
  recorded, remains retryable, and resumes from the failed operation without
  repeating the previous stop phase.
- Added backup runner coverage proving journal locks reject concurrent
  execution before any command runs.
- Added backup runner coverage proving preflight failures leave mutating
  operations blocked and do not accept the execution preflight.
- Added backup CLI status coverage proving execution layouts render as
  `running`, `failed`, or `complete` from durable plan/journal/manifest state.
- Kept the existing finalization coverage for durable artifact layout and
  manifest integrity.
