# Canic 0.36

## 0.36.13 - 2026-05-16 - Restore operator polish

### 🔧 Changed

- Updated restore command help so the main command and restore plan, prepare,
  apply, run, and status leaves show the row-reference operator flow first.
- Updated the CLI README, root README, and 0.36 backup/restore design notes so
  the documented happy path uses `backup list` row references, `restore
  prepare`, `restore status`, and guarded `restore run` execution.
- Clarified the `--require-verified` error wording so backup references and
  `--backup-dir` are both described as valid verified restore sources.

### 📚 Documentation

```bash
canic restore help
canic restore prepare 1 --require-verified --require-restore-ready
canic restore status 1 --require-no-attention
canic restore run 1 --execute --max-steps 1 --require-no-attention
```

### 🩹 Fixed

- `canic restore apply <backup-ref> --dry-run` now fails with a prepared-plan
  error when the default `restore-plan.json` is missing, including the exact
  path and `canic restore prepare <backup-ref>` command to run.
- `canic restore run <backup-ref>` and `canic restore status <backup-ref>` now
  fail with the same explicit prepare guidance when the default
  `restore-apply-journal.json` is missing.

### 🧪 Testing

- Added restore CLI coverage for the new row-reference help examples.
- Added prepared-artifact path coverage proving missing default plan and
  journal files return typed operator-action errors instead of raw file IO
  errors.

## 0.36.12 - 2026-05-16 - Restore row references

## 0.36.10 - 2026-05-16 - Operator restore path

### 🩹 Fixed

- Restore runner ICP commands now pass `-n <network>` to the concrete
  canister leaf command instead of placing it between `icp canister` and the
  subcommand, matching ICP CLI's snapshot/status command shape.
- Restore snapshot uploads no longer pass `--resume` for fresh uploads. ICP CLI
  rejects resume uploads when no upload progress file exists, so the generated
  restore command now works for first-attempt operator restores.
- Full non-root fleet backups now split final manifest consistency units by
  connected component after the root canister is excluded. Sibling branches are
  no longer emitted as one disconnected subtree, so plain
  `canic backup create <fleet>` produces a valid manifest.
- `canic backup list` now normalizes unfinished execution-layout timestamps to
  the same `unix:<seconds>` marker shape used by completed manifests. Failed
  and running rows use the earliest execution-journal timestamp when present,
  with legacy run-id date stamps converted before display.

### 🔧 Changed

- `canic info cycles` now reports explicit `BURN/H` and `TOPUP/H` columns in
  its compact default terminal table, with canister id, history, total top-ups,
  and net total available through `--verbose`.
- JSON output now includes `burn_cycles`, `burn_cycles_per_hour`, and
  `topup_cycles_per_hour`. These values are derived over the same tracker
  coverage window as net cycle rate, so top-ups no longer hide gross cycle
  consumption.
- Cleaned the local backup artifact root for this slice so only the verified
  complete `test` fleet backup remains under `backups/`.

### 📚 Documentation

```bash
canic backup create test
canic backup verify 1
canic info cycles test
canic info cycles test --verbose
```

### 🧪 Testing

- Updated restore apply-journal and CLI runner coverage for the corrected ICP
  command arguments.
- Added cycle-summary coverage for explicit burn and top-up rate derivation.
- Added runner coverage proving root-omitted fleet plans finalize as multiple
  valid backup units instead of a disconnected subtree.
- Added backup-list coverage for journal-derived created-at markers and
  run-id stamp conversion.
- Proved the local operator path against the deployed `test` fleet's `app`
  subtree: backup create, status, verify, inspect, verified restore plan,
  apply journal, dry-run preview, stepwise execute/resume, completion
  enforcement, and final readiness check.
- Proved `canic backup create test` against the deployed local `test` fleet
  with six non-root targets, then verified the complete backup layout and
  manifest artifacts.

## 0.36.8 - 2026-05-15 - Restore receipt retry integrity

### 🩹 Fixed

- Restore-runner journal loading now treats the latest receipt attempt for a
  terminal operation as authoritative. A completed or failed operation is
  rejected if the newest receipt has a different outcome, metadata, or state
  timestamp, preventing stale retry receipts from masking hand-edited terminal
  restore state.
- `canic list <fleet> --subtree <name-or-principal>` now accepts a unique role
  name such as `scale_hub` as the subtree anchor. Repeated role names now fail
  with an explicit ambiguity error that asks for a concrete canister principal.
- `canic cycles <fleet> --subtree <name-or-principal>` now uses the same
  role-or-principal subtree selector and filters before querying cycle
  histories, balances, and top-up events.

### 🧪 Testing

- Added restore-runner journal coverage for stale terminal receipts, timestamp
  drift, and successful retry histories whose latest receipt matches the
  terminal operation.
- Added list rendering coverage for role-name subtree anchors and repeated-role
  ambiguity.
- Added cycles option coverage for role-name subtree selectors.

## 0.36.7 - 2026-05-15 - Restore receipts and replica status

### 🩹 Fixed

- Restore apply-journal command receipts now require the durable audit
  envelope written by the runner: update timestamp, command preview, exit
  status, and bounded stdout/stderr payloads.
- Restore command receipt validation now rejects empty command program, args,
  or note fields in persisted journals.
- `canic replica start`, `canic replica status`, and JSON replica status now
  treat ICP CLI local status as stale unless the configured gateway port is
  reachable, preventing a dead configured port from being reported as running.
- Stale ICP CLI status now appears as `icp_cli_stale` in JSON output instead
  of `icp_cli`, with `running` set to `false`.
- Fleet sync now preserves the top-level `networks:` section ahead of
  `environments:`, avoiding needless `icp.yaml` churn during replica start or
  explicit fleet sync.

### 🧪 Testing

- Added restore apply-journal coverage proving hand-edited command receipts
  missing audit fields are rejected without appending the invalid receipt.
- Added replica status JSON coverage for stale ICP CLI status metadata.
- Added ICP config sync coverage proving preserved network config remains
  before rendered environments.

## 0.36.6 - 2026-05-15 - Backup execution integrity

### 🩹 Fixed

- Backup execution integrity now checks the latest durable receipt for each
  terminal mutating operation, preventing stale retry receipts from masking a
  hand-edited terminal state mismatch.
- Backup execution integrity now also requires terminal operation
  `state_updated_at` to match the latest durable receipt timestamp.
- Backup execution journals now validate stored `restart_required` against the
  operation graph, preventing edited journals from hiding a required restart
  after a stop operation completes and before start recovery finishes.
- Backup execution transitions now require non-empty update timestamps, and
  pending or terminal operation states cannot be persisted without
  `state_updated_at`.

### 🧪 Testing

- Added persistence coverage for a failed-then-completed retry history whose
  operation state is later edited back to failed.
- Added persistence coverage proving terminal operation timestamps cannot drift
  from their durable operation receipts.
- Added execution-journal coverage proving restart-required drift is rejected.
- Added execution-journal coverage proving preflight, pending, retry, and
  persisted active-state timestamp gaps fail before they can mutate state.

## 0.36.3 - 2026-05-15 - Restore runner hardening

### 🩹 Fixed

- `canic restore run --execute` now fails an upload-snapshot step when the
  command succeeds but does not return an uploaded snapshot id. The runner no
  longer marks that operation complete without the receipt data required by the
  later load-snapshot step.
- `canic restore run --retry-failed` now moves the next failed restore
  operation back to ready after operator inspection, making failed restore
  steps explicitly recoverable without hand-editing the apply journal.
- Legacy text snapshot-upload parsing now only accepts the uploaded-snapshot
  label, so unrelated successful colon-labelled output cannot be mistaken for
  an uploaded snapshot id.
- `canic restore run` now rejects terminal apply-journal operations that do
  not have matching command receipts, so hand-edited completed or failed
  operations cannot silently steer a restore.

### 🧪 Testing

- Added restore-runner coverage for successful upload commands whose output is
  missing the uploaded snapshot id.
- Added restore CLI coverage for failed-operation retry recovery and the
  machine-readable `failed-recovered` runner receipt.
- Added parser coverage for blank JSON snapshot ids and unrelated legacy text
  output.
- Added restore CLI coverage proving a terminal operation without a matching
  receipt is rejected before any runner mode proceeds.

## 0.36.1 - 2026-05-15 - Backup resume safety

### 🩹 Fixed

- `canic backup create --out <dir>` now fails closed when `<dir>` already
  contains a backup plan for a different fleet, network, root, scope, target
  set, or operation graph. Compatible layouts are still preserved as the resume
  boundary.
- Existing backup output layouts now also have to match the requested
  authority and quiescence policy, preventing a dry-run plan from being reused
  as an executable backup layout.
- `canic backup create` output now includes a `LAYOUT` column so operators can
  tell whether the command created a new output layout or resumed an existing
  one.
- `canic backup list` now reports manifest-plus-plan layouts with a missing
  execution journal as `invalid-plan-journal` instead of `dry-run`.
- `canic backup create --out <dir>` now rejects manifest-backed layouts that
  are missing their execution journal instead of recreating one over the top of
  a corrupted backup directory.

### 🧪 Testing

- Added regression coverage proving incompatible existing backup output layouts
  are rejected instead of silently reusing the wrong plan.
- Added regression coverage for rejecting dry-run/executable layout mismatches.
- Added output coverage for the backup create layout-source indicator.
- Extended backup list execution-backed status coverage to missing execution
  journals.
- Added regression coverage for refusing to resume manifest-backed layouts with
  missing execution journals.

  still incomplete or mismatched.

  journal is not accepted until the finalized manifest exists.

  without replaying accepted preflight work or completed mutations.
- Added backup runner coverage proving a failed snapshot operation is durably
  recorded, remains retryable, and resumes from the failed operation without
  repeating the previous stop phase.
- Added backup runner coverage proving journal locks reject concurrent
  execution before any command runs.
- Added backup runner coverage proving preflight failures leave mutating
  operations blocked and do not accept the execution preflight.
- Added backup CLI status coverage proving execution layouts render as
