## [0.99.13] - 2026-07-24 - Prepared Root Endpoint Fence

### đź”’ Security

- Every `canic_query` and `canic_update` wrapper now runs the Fleet-activation

  `canic_fleet_activation_status` inspection reaches its handler. Ordinary

  bootstrap endpoints remain unavailable until activation reaches `Active`;

## [0.99.12] - 2026-07-24 - Prepared Root Install Handoff

### đź”§ Changed

- Fresh install now finalizes the release build and durably publishes the exact
  `Planned` Fleet-activation journal before resolving or creating the root.
  Root installation receives `CurrentRootInstallIdentity`, atomically enters
  ID-21 `Prepared`, and advances the host authority to `RootInstalled` only
  after the exact module and controller-only activation status are observed.
- A lost install response no longer causes a blind reinstall. The host first
  observes the module, Fleet binding, operation ID, release-build ID, phase and
  empty initial evidence; only that exact `Prepared` result can reconcile the
  operation and produce the canonical durable install receipt.
- Prepared root init restores only its synchronous invariants. It starts no
  runtime services, timers, bootstrap work or application install hook, and
  the old controller bootstrap-resume endpoint rejects until ID 21 is
  `Active`.
- The root lifecycle macro retains the application `canic_install` hook as a
  compile-time contract without polling or scheduling it in `Prepared`, so
  strict downstream dead-code checks remain clean until activation owns its
  execution.
- Current-install preflight now describes only the surviving preparation,
  planning, root-resolution, verified-install and journal-acknowledgement
  phases. It requires Canister-status observation and no longer claims the
  deleted artifact-staging or generic call capabilities.

### âš ď¸Ź Breaking

- The root init Candid argument is now solely
  `CurrentRootInstallIdentity`; `SubnetIdentity`, including its Prime,
  Standard and Manual variants, is removed without an alias or decoder.
- The old install completion path is removed. `canic install` currently stops
  with a typed continuation-required result after durable `RootInstalled`; it
  does not fund, stage, resume bootstrap, wait for readiness, write legacy
  installed-deployment state or emit an install-specific artifact-promotion
  receipt.
- The unused host install timeout and promotion-plan fields, root
  staging/resume helpers and related receipt paths are removed. A ready
  promotion envelope remains accepted as a source of its validated promoted
  deployment plan, but no parallel promotion receipt is published.

### đź§Ş Testing

- Focused host coverage proves exact textual-Candid identity round trips,
  module/status reconciliation, typed mismatch causes, verified-receipt
  admission, journal ordering and the current preflight phase/capability set.
  Core lifecycle coverage proves `Prepared` rejects active-only bootstrap and
  schedules no init-time bootstrap or application hooks.
- Repository PocketIC build helpers now embed and supply one matching
  deterministic release-build identity. No package version, dependency,
  lockfile or stable-memory schema changes are included; the intentional
  surface changes are the root init Candid contract, host Rust API removals,
  current-install execution semantics and preflight phase values.

## [0.99.8] - 2026-07-24 - Root-Install Receipt Authority

### đź”§ Changed

- The fresh-install journal now admits its exact sequence-one
  `RootInstalled` state. The transition revalidates the sequence-zero
  authority under the same network/Fleet lock, atomically replaces it and
  re-reads the durable result. A stale source journal or conflicting receipt
  hash fails closed.
- Root-install receipt admission reads a regular no-follow file and requires

  `install_root` phase. The exact root principal must agree with receipt
  identity, and canonical expected and observed module hashes must match.
- `RootInstalled` stores the raw SHA-256 of those exact receipt bytes.
  Repeating the transition with the same durable receipt is idempotent;
  changing the receipt is rejected. `CanistersPrepared` and later phases
  remain reserved and unencodable.

  `SubnetRole::PRIME` / `subnets.prime` to

  `DirectoryConfig` type are removed.

  one-member placement and runtime Directory behavior.
- Generated compiled configuration, checked-in canister/Fleet configs,
  scaffolding, host role mutation, verbose config diagnostics, CI fixtures,
  and active operator guidance emit the current App/default/binding shape.
- Config-derived diagnostic sources use `app_config`; no `fleet_config`
  source label remains in the changed planning and medic paths.
- The 0.99 and 0.100 contracts now derive future work from completed
