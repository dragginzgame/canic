# Canic 0.99 Changelog

## [0.99.2] - 2026-07-23 - Canonical Network Trust Authority

### ✨ Added

- `CanonicalNetworkId` is one domain-separated 32-byte trust identity with an
  exact 64-character lowercase hexadecimal text and JSON form. Public callers
  can derive it only from Canic's pinned public-IC DER anchor or a validated
  enrolled DER trust anchor.
- `canic network enroll <environment> --root-key <der-file> --fingerprint
  <64-lowercase-hex>` is the sole non-public enrollment command. It accepts
  only a regular, non-symlink DER file and verifies the exact SHA-256
  fingerprint before any durable write.

### 🔧 Changed

- Non-public authority is written to
  `.canic/networks/<canonical-network-id>/trust/root-key.der` and
  `enrollment.json` before
  `.canic/environment-profiles/<environment>/network.json` becomes visible.
  The profile contains only the canonical ID and is revalidated against the
  exact authority on every resolution.
- Exact re-enrollment and recovery after an authority-only partial publication
  are idempotent. Conflicting profile bindings, changed anchors, incomplete or
  contradictory authorities, special files, symlinks, and non-canonical
  fingerprints fail closed. No trust-on-first-use, anchor rotation, alias, or
  compatibility path exists.
- Environment labels remain connectivity/provenance selectors. Named staging
  and production profiles backed by `ic` resolve to the same compiled
  public-IC identity; aliases enrolled with the same non-public anchor share
  one network authority.
- The canonical durable writer now supports create-new publication with
  synced missing parents and rejects symlinked parent directories. Enrollment
  reuses that owner rather than adding another atomic-write flow.

### 🧪 Testing

- Focused core identity tests cover domain separation, canonical text, strict
  parsing, and pinned/enrolled DER equivalence; host document tests prove the
  exact JSON projection.
- Host tests cover successful enrollment and resolution, exact idempotency,
  partial-publication recovery, local and public aliases, fingerprint mismatch,
  non-canonical fingerprints, changed anchors, missing authority, public-IC
  enrollment rejection, symlinks, special files, and symlinked output parents.
- Focused CLI parsing, help, version, rendering, formatting, diff hygiene, and
  strict all-target Clippy pass for `canic-core`, `canic-host`, and
  `canic-cli`.
- No Cargo package version, production dependency, Candid, stable-memory,
  runtime endpoint, Fleet ID, install-state, or deployment JSON contract
  changes in this batch. The new host enrollment/profile JSON documents and
  CLI command are intentional 0.99 surfaces.

## [0.99.1] - 2026-07-23 - App Source Surfaces

### ⚠️ Breaking

- The maintained source tree is `apps/<app>/`; `fleets/` is removed and no
  discovery fallback searches both locations.
- Canister package metadata uses `[package.metadata.canic] app = "<app>"`.
  The old `fleet` key is not read.
- Source commands are `canic app ...`, and artifact builds take
  `canic build <app> <role>`. The old `canic fleet ...` command family is
  removed without an alias.

### 🔧 Changed

- App identity now owns config discovery and snapshots, source role
  declaration/attachment/rename, scaffolding, role-package validation,
  build-time environment labels, artifact provenance, adoption reports, and
  source-oriented evidence selectors.
- The root workspace, demo/test Apps, standalone canisters, role-contract
  fixtures, build scripts, inventory guards, packaged-downstream checks, and
  active source documentation all use the canonical App path and metadata
  vocabulary.
- Evidence targets distinguish `app` from `fleet`; App adoption and build
  provenance emit App identity, and project evidence policies can select it
  independently. Live install/deployment/catalog behavior remains unchanged
  for the later Fleet live-surface slice.
- The private evidence-gate result stores its two report payloads indirectly
  after the additional App selector increased their stack footprint. Report
  JSON and CLI output are unchanged.

### 🧪 Testing

- Focused App CLI, list, Medic, evidence, config discovery, release-set,
  build-provenance, adoption, role-contract, workspace-manifest,
  reference-surface, timer/receipt inventory, and blob-storage inventory
  checks pass. The complete `canic-cli` library suite passes 608 tests with
  only its explicitly disposable-environment recovery test ignored.
- Formatting, diff hygiene, and strict all-target Clippy pass for the affected
  `canic-core`, `canic`, `canic-host`, and `canic-cli` packages.
- No Cargo package version, dependency, lockfile, Candid, stable-memory, or
  live deployment contract changed. Source paths, package metadata, Rust
  source models, CLI commands, and source/evidence JSON intentionally break.

## [0.99.0] - 2026-07-23 - Reinstall-Only Minimum App Config Hard Cut

### ⚠️ Breaking

- 0.99 is fresh-install/reinstall-only. It does not upgrade or migrate 0.98
  Canister stable state, installed-state trees, deployment catalogs, or
  authorization records.
- The top-level `app_index` configuration field is removed. App-declared
  Fleet service roles now use:

  ```toml
  [services.fleet]
  roles = ["project_hub"]
  ```
- `[fleet].name` and `FleetConfig` are removed. Every maintained config now
  declares its required immutable source identity as `[app].name`.
- The source-topology type and reserved slot are hard-cut from
  `SubnetRole::PRIME` / `subnets.prime` to
  `SubnetSlotId::DEFAULT` / `subnets.default`.
- Keyed placement config is `binding`; the old `directory` config key and
  `DirectoryConfig` type are removed.

### 🔧 Changed

- `AppId`, `AppRoleRef` and `AppConfigSnapshot` are the source-config identity
  surfaces. Parsing, validation, build support and auth provenance derive the
  exact App identity from one required field.
- Configuration parsing and validation consume one nested Fleet-service
  declaration and one keyed placement binding while preserving current
  one-member placement and runtime Directory behavior.
- Generated compiled configuration, checked-in canister/Fleet configs,
  scaffolding, host role mutation, verbose config diagnostics, CI fixtures,
  and active operator guidance emit the current App/default/binding shape.
- Config-derived diagnostic sources use `app_config`; no `fleet_config`
  source label remains in the changed planning and medic paths.
- The 0.99 and 0.100 contracts now derive future work from completed
  fresh-install evidence rather than a nonexistent 0.98 migration receipt.

### 🧪 Testing

- Focused Canic-core configuration/bootstrap tests, Canic-host release-set and
  ICP-config tests, Canic CLI scaffold tests, Canic config/reference-surface
  tests, targeted package checks, formatting and strict Clippy pass for the
  changed packages.
- No Cargo package version, dependency, Candid endpoint label, JSON, CLI
  command, or stable allocation changes. The Rust config API and source TOML
  schema intentionally break; reinstall-only means no old slot value or
  config shape is decoded.
