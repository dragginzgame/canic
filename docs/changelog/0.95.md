# Canic 0.95 Changelog

## [0.95.1] - 2026-07-20 - Common Timer Authority

### 📝 Summary

Replaces the competing guarded-slot and fixed-rate timer mechanics with one
heap-owned scheduler for Canic processes, lifecycle deferrals, and application
timers. Scheduling is request-sequenced and generation-bound, asynchronous
recurrence begins after completion, and live runtime status no longer treats a
historical metric row as an active registration.

### ⚙️ Scheduling

- Routes log retention, cycle tracking, delegated-proof renewal, intent
  cleanup, root-pool maintenance and reset, placement-receipt acknowledgement,
  lifecycle deferral, and application timers through one `TimerWorkflow`.
- Gives every built-in process a closed timer key. Application timers use a
  separately allocated opaque identity and a non-cloneable `TimerHandle`.
- Merges requests made during an active callback without arming a competing
  callback. The earliest requested deadline wins unless a later sequenced
  cancellation suppresses rearming; a still later schedule reenables it.
- Replaces fixed-rate asynchronous intervals with one-shot rearming after each
  completed invocation. Immediate continuation yields through a new timer
  message rather than looping inside the callback.
- Checks request-sequence, generation, application-identity, delay, and
  deadline arithmetic. Exhaustion fails closed without overwriting the current
  registration.
- Removes the pool worker's duplicate in-progress, reschedule, and timer-slot
  state; the common authority now retains work arriving while a reset batch is
  active.

### 🧹 Hard Cuts

- Removes `timer_guarded!`, `timer_interval_guarded!`, `TimerSlot`, guarded
  workflow/ops helpers, and direct public `canic::cdk::timers` access.
- Removes the direct IC interval primitive from production. The sole platform
  owner arms and clears one-shot callbacks only.
- Renames the internal lifecycle facade operation to `defer_lifecycle` and
  routes root, non-root, and generated user-hook deferrals through the common
  workflow.
- Removes the obsolete pool `in_progress` metric reason with the manual guard
  it described. No alias, deprecated wrapper, or fallback path remains.

### 📡 Public And Operator Surfaces

- `timer!` and `timer_interval!` now return the maintained opaque
  `canic::api::timer::TimerHandle`.
- `canic::api::timer::cancel(handle)` consumes that handle and reports whether
  the live timer identity existed. Cancelling a scheduled timer releases its
  task and status entry; cancelling a running interval allows the current
  invocation to finish and suppresses recurrence.
- `timer_interval!` is explicitly an after-completion interval. It does not
  replay missed fixed-rate ticks after a time jump.
- The current runtime-introspection timer row hard-cuts the old aggregate
  `status` and `registered` fields. It now reports independent `registration`
  and `condition`, scheduling mode, enabled configuration, generation, next
  deadline, typed last outcome, bounded work count, and counters explicitly
  named `*_since_runtime_start`.
- Timer metrics retain `[mode, label]` as their bounded key. `count` is the
  execution count and `value_u64` is the latest armed delay in milliseconds;
  changing exact deadlines no longer allocates a metric row per duration.

### 🧪 Testing

- Adds deterministic control tests for duplicate registration, earlier
  replacement, stale generations, schedule-during-run retention, earliest
  request merging, cancellation precedence, reenabling, and checked overflow.
- Adds PocketIC proof that a cancelled application timer never executes, a
  one-shot executes exactly once, and an interval rearms once after completion
  without replaying missed fixed-rate ticks.
- The same PocketIC journey proves completed and cancelled one-shots disappear
  from runtime status while the live interval reports
  `scheduled + active + after_completion`.
- Updates the timer inventory and lifecycle-boundary guards for the single raw
  platform owner and common lifecycle facade.

## [0.95.0] - 2026-07-20 - Timer Authority Slice A

### 📝 Summary

Closes the completed 0.94 backup/restore line at released `v0.94.14` and
activates the accepted 0.95 timer-authority design. The audit inventories every
production timer, lifecycle deferral, public timer form, retry scheduler, and
bounded host wait before scheduler implementation begins.

### 📊 Audit

- Confirms `ops::runtime::timer` is the sole production consumer of direct IC
  timer operations.
- Dispositions all 14 timer/process rows and both bounded host waits.
- Reproduces async interval overlap, stale guarded slots, false live status,
  unnecessary idle wake-ups, unrelated full scans, duplicate mechanical and
  lifecycle flows, and unreachable configured root self-refill.
- Measures the empty-root first-hour baseline at ten background callbacks and
  72,303 timer-callback instructions. Seven log/pool wakes had no
  authoritative work.
- Adds a source inventory guard so a new timer wrapper, bypass, or timed wait
  cannot enter silently.

### 🔧 Decisions

- Retains cancellable public one-shot and after-completion interval forms.
  Unused guarded macros, `TimerSlot`, fixed-rate async recurrence, and public
  raw-CDK timer access are hard-cut in the common-authority slice.
- Freezes event/deadline ownership for log age retention, pool resets, intent
  expiry, placement acknowledgement, and delegated-proof renewal.
- Keeps cycle sampling and automatic top-up independent so disabled top-up is
  unregistered and configured root ICP self-refill becomes reachable through
  its existing workflow.
- Freezes per-owner batch, retry, and cadence bounds without adding a global
  heartbeat or receipt-specific state.

### 🧪 Testing

- Uses the existing root PocketIC harness to measure current callback count and
  instruction cost from the released anchor.
- Runs the targeted source inventory and direct-owner guards.
- Adds no production scheduling, Candid, JSON, stable-state, Cargo-version,
  dependency, backup/restore, receipt-reclamation, or compatibility change.
