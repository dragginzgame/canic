# 0.71 Changelog

## 0.71.4 - 2026-06-23 - Operator-loop validation proof

### 📝 Summary

- Closes the 0.71 M6 validation gap for blob-storage operator readiness. The
  CLI now has focused in-process proof that the live operator sequence can
  inspect status, sync gateway principals, inspect post-sync status, fund from
  project cycles, and inspect final readiness without changing protocol shape,
  command grammar, or transport ownership.

- Extends the retained package/install validation probes from surface checks to
  a fixture-backed operator loop. Packaged and installed CLI proofs now run the
  same dry-run and live status/sync/fund/final-status checks through a local
  deployment fixture and stateful fake `icp` transport.

Operator loop covered by the proof:

```bash
canic blob-storage status <deployment> <canister-or-role>
canic blob-storage sync-gateways <deployment> <canister-or-role>
canic blob-storage status <deployment> <canister-or-role>
canic blob-storage fund <deployment> <canister-or-role> --cycles <amount>
canic blob-storage status <deployment> <canister-or-role>
```

### 🔧 Changed

- Adds a private blob-storage CLI runtime seam for target resolution and
  canister-call execution. Production still uses the installed-deployment
  resolver and existing `IcpCli` canister-call path; the seam exists only so
  focused tests can script the operator loop without adding a second transport
  stack.

- Keeps live sync and live fund semantics pinned: the primary mutating call
  succeeds or fails independently from optional post-action status diagnostics.
  If post-status parsing or transport fails after a successful mutation, the
  command returns the successful action result with `post_status_unavailable`
  instead of failing the mutation report.

- Consolidates the packaged and installed blob-storage CLI proof fixture into
  a shared CI helper. The helper owns the fixture deployment metadata, local
  Candid sidecars, stateful fake `icp` responses, proof command sequence, and
  stable JSON assertions used by both package-boundary verifier scripts.

### 🧪 Testing

- Adds a scripted `status -> sync -> status -> fund -> status` unit proof for
  the live result path, including method order, sync post-status gateway count,
  funding report parsing, and final ready-for-upload status.

- Adds focused coverage that live sync/fund return successful action results
  with `post_status_unavailable` when their optional post-status diagnostic
  cannot be parsed.

- Extends the packaged downstream CLI proof and installed CLI proof to assert:
  fixture status JSON, live sync JSON with post-status, live fund JSON with
  funding report and ready post-status, final status JSON, and the retained
  dry-run sync/fund JSON surfaces.

## 0.71.2 - 2026-06-23 - Live provisioning and medic diagnostics

### 📝 Summary

- Completes the live blob-storage provisioning CLI loop for the existing 0.70
  billing endpoints. Operators can now inspect status, explicitly sync gateway
  principals, explicitly fund Cashier from project cycles, and use targeted
  medic diagnostics without putting sync or funding into the product frontend
  hot path.

### ➕ Added

- Adds live `sync-gateways` execution. The command uses the same installed
  deployment target resolution, local Candid sidecar validation, and
  metadata-derived query/update mode as dry-run previews, then runs the
  existing `_immutableObjectStorageUpdateGatewayPrincipals` endpoint.

- Adds live `fund` execution. The command calls the existing
  `_immutableObjectStorageFundFromProjectCycles` endpoint with an explicit
  unsigned base-10 cycle amount and parses the structured top-up report into
  stable CLI JSON/plain output.

- Adds targeted blob-storage medic diagnostics. The targeted medic path reuses
  the blob-storage status parser and readiness model instead of reinterpreting
  raw canister DTOs.

- Adds passive default medic hints from local Candid sidecars when a role
  advertises the full blob-storage billing endpoint trio. Default medic remains
  diagnostic-only and does not call status, sync, or fund.

- Adds the versionless blob-storage billing readiness runbook for endpoint
  inspection, status, dry-run previews, explicit sync, explicit funding, and
  post-upgrade checks.

Operator commands:

```bash
canic blob-storage sync-gateways <deployment> <canister-or-role>
canic blob-storage fund <deployment> <canister-or-role> --cycles <amount>
canic info medic <deployment> --blob-storage <canister-or-role>
```

### 🔧 Changed

- Status next-action hints now point at the live sync and fund commands instead
  of dry-run-only previews.

- Live sync and live fund attach post-action status diagnostics when available.
  A successful mutating call is still reported as successful if the optional
  post-status diagnostic query fails; the result carries a
  `post_status_unavailable` warning.

- Blob-storage `--json` command failures now produce a structured
  `blob_storage_error` payload with `schema_version`, `kind`, target context,
  canonical command error code, message, and exit-code metadata.

### 📚 Documentation

- Adds `docs/operations/blob-storage-billing-readiness.md` and links it from
  the operations index. The runbook documents the CLI operator sequence and
  keeps product backend orchestration separate from frontend upload flows.

### 🧪 Testing

- Validates live sync/fund action result shapes, funding report parsing,
  structured JSON error rendering, top-level error prefix preservation,
  targeted medic parsing/output, passive medic Candid discovery, and the
  terminal-readable medic report layout.

## 0.71.1 - 2026-06-23 - Live blob-storage status

### ➕ Added

- Adds live `canic blob-storage status <deployment> <canister-or-role>` support
  on top of the existing target-resolution and local-Candid validation path.
  The command calls the guarded 0.70 `get_blob_storage_status` endpoint with
  `sync_gateway_principals = false` so status remains a read-only diagnostic
  operation.

- Adds a stable `blob_storage_status` JSON result shape with
  `schema_version`, resolved `target`, configured/Cashier/policy/gateway
  sections, decimal-string cycle values, readiness state, blocker/warning
  codes, funding status, and next-action hints.

- Adds compact terminal rendering for the same status view, including Cashier
  balance, upload-balance policy, project reserve, gateway sync metadata,
  readiness state, blockers, warnings, and next commands.

### 🔧 Changed

- Keeps status interpretation in the CLI view layer rather than exposing raw
  DTOs. The parser maps the existing 0.70 status DTO variants into stable
  operator codes without adding new runtime billing policy.

- Keeps mutation transport out of this patch. At 0.71.1, status next-action
  hints point operators at dry-run sync/fund previews because live mutation
  execution is still deferred.

### 🧪 Testing

- Validates status option parsing, stable JSON shape, plain text rendering,
  funding-needed next-action generation, blocker/warning code mapping, and
  Candid-derived method-mode behavior through focused CLI tests.

## 0.71.0 - 2026-06-23 - Blob-storage operator readiness scaffold

### 📝 Summary

- Starts the 0.71 operator-readiness line for blob-storage billing without
  expanding the 0.70 protocol contract. The release focus is making existing
  blob-storage status, gateway-sync, and funding surfaces easier for operators
  and automation to discover, inspect, and eventually run through Canic CLI.

- Records the 0.71 design contract for first-class `canic blob-storage` CLI
  support, targeted medic integration, stable JSON output, strict dry-run
  behavior, target resolution, Candid metadata requirements, exit-code rules,
  and the separation between CLI operators and product backend callers.

### ➕ Added

- Adds the top-level `canic blob-storage` command group with `status`,
  `sync-gateways`, and `fund` subcommands. The command group accepts the
  existing global ICP executable and network options.

- Adds strict `fund --cycles` parsing for unsigned base-10 integer cycle
  amounts. Human suffixes, separators, decimals, scientific notation, negative
  values, and zero are rejected for the operator funding surface.

- Adds dry-run support for `sync-gateways` and `fund`. Dry-runs resolve the
  installed deployment target, resolve `<canister-or-role>`, require local
  Candid metadata, derive query/update mode from the method signature, and
  render the exact canister-call command that would be used.

- Adds schema-versioned JSON dry-run output for sync and fund actions. The
  output includes `schema_version`, `kind`, `deployment`, resolved `target`,
  an `action` object with method/mode/dry-run/command fields, optional
  decimal-string `requested_cycles`, `post_status: null`, and `warnings`.

- Adds host support for rendering dry-run query calls that take an explicit
  Candid argument, keeping blob-storage CLI previews on the existing Canic host
  canister-call abstraction.

### 🔧 Changed

- Live `blob-storage status`, `blob-storage sync-gateways`, and
  `blob-storage fund` execution remains intentionally gated in this first
  slice. The CLI surface is present so the next slices can add status
  transport, response parsing, live mutation execution, structured errors, and
  medic wiring without changing the command grammar.

- Target resolution now follows the 0.71 design rule that role names are
  resolved through installed deployment metadata before direct principal
  fallback. A direct principal without local deployment metadata cannot satisfy
  Candid validation for this MVP dry-run path.

- Missing, unreadable, malformed, or method-incomplete local Candid sidecars
  fail before any transport call is attempted. No built-in fallback signature is
  introduced in this slice.

### 🧪 Testing

- Validates blob-storage CLI option parsing, required target behavior, strict
  cycle syntax, top-level global option forwarding, dry-run JSON shape, plain
  text rendering, target resolution precedence, direct-principal fallback
  behavior, and method-mode derivation from Candid metadata.

- Validates the new host dry-run query-call renderer with a local Candid
  sidecar.

- Runs focused CLI and host clippy checks with warnings denied, plus formatting
  and whitespace checks.
