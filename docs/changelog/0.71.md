# 0.71 Changelog

## 0.71.0 - 2026-06-23 - Blob-storage operator readiness scaffold

### 📝 Summary

- Starts the 0.71 operator-readiness line for blob-storage billing without
  expanding the 0.70 protocol contract. The release focus is making existing
  blob-storage status, gateway-sync, and funding surfaces easier for operators
  and automation to discover, inspect, and eventually run through Canic CLI.

- Records the 0.71 design contract for first-class `canic blob-storage` CLI
  support, targeted medic integration, stable JSON output, strict dry-run
  behavior, target resolution, Candid metadata requirements, exit-code rules,
  and the separation between CLI operators and product backend callers.

### ➕ Added

- Adds the top-level `canic blob-storage` command group with `status`,
  `sync-gateways`, and `fund` subcommands. The command group accepts the
  existing global ICP executable and network options.

- Adds strict `fund --cycles` parsing for unsigned base-10 integer cycle
  amounts. Human suffixes, separators, decimals, scientific notation, negative
  values, and zero are rejected for the operator funding surface.

- Adds dry-run support for `sync-gateways` and `fund`. Dry-runs resolve the
  installed deployment target, resolve `<canister-or-role>`, require local
  Candid metadata, derive query/update mode from the method signature, and
  render the exact canister-call command that would be used.

- Adds schema-versioned JSON dry-run output for sync and fund actions. The
  output includes `schema_version`, `kind`, `deployment`, resolved `target`,
  an `action` object with method/mode/dry-run/command fields, optional
  decimal-string `requested_cycles`, `post_status: null`, and `warnings`.

- Adds host support for rendering dry-run query calls that take an explicit
  Candid argument, keeping blob-storage CLI previews on the existing Canic host
  canister-call abstraction.

### 🔧 Changed

- Live `blob-storage status`, `blob-storage sync-gateways`, and
  `blob-storage fund` execution remains intentionally gated in this first
  slice. The CLI surface is present so the next slices can add status
  transport, response parsing, live mutation execution, structured errors, and
  medic wiring without changing the command grammar.

- Target resolution now follows the 0.71 design rule that role names are
  resolved through installed deployment metadata before direct principal
  fallback. A direct principal without local deployment metadata cannot satisfy
  Candid validation for this MVP dry-run path.

- Missing, unreadable, malformed, or method-incomplete local Candid sidecars
  fail before any transport call is attempted. No built-in fallback signature is
  introduced in this slice.

### 🧪 Testing

- Validates blob-storage CLI option parsing, required target behavior, strict
  cycle syntax, top-level global option forwarding, dry-run JSON shape, plain
  text rendering, target resolution precedence, direct-principal fallback
  behavior, and method-mode derivation from Candid metadata.

- Validates the new host dry-run query-call renderer with a local Candid
  sidecar.

- Runs focused CLI and host clippy checks with warnings denied, plus formatting
  and whitespace checks.
