# Canic 0.98 Changelog

## [0.98.5] - 2026-07-22 - Shared Hash And Host Error Consolidation

### 📝 Summary

- Host evidence, deployment truth, root verification, build provenance,
  replica decoding, and bootstrap key rendering now reuse the existing
  `canic-core` hash and lowercase-hex authority. Operational canister and root
  install builds also consume `FleetConfigSnapshot` rather than reopening and
  reparsing the selected fleet configuration.

### 🗑️ Removed

- Removed five local lowercase-hex or SHA-256 implementations and the unused
  public host byte-hash helper. No alias or compatibility wrapper remains.
- Removed manual `Display`, `Error`, and conversion implementations where
  equivalent typed `thiserror` declarations preserve the same variants,
  messages, and source chains.
- Narrowed direct readiness, bootstrap-status, and subnet-registry replica
  functions to their proved host consumers. The environment-selection and
  replica-reachability functions used by the CLI remain public.

### 🔧 Changed

- Direct canister builds, complete install-build snapshots, and canonical
  deployment-config hashing now obtain their parsed `ConfigModel` from one
  validated `FleetConfigSnapshot`.
- Query error declarations retain the same typed causes and operator text with
  substantially less handwritten forwarding code.

### 🧪 Testing

- Focused bootstrap, replica-query wire, evidence-envelope, build-provenance,
  canister-build, deployment-truth, install-root, CLI list, and backup-runner
  tests pass. Strict all-target Clippy passes for the four affected packages.
- Every error or error-code type deriving `CandidType` was traced to a Canic
  endpoint response or an external IC protocol payload. All remain necessary;
  no Candid derive or wire shape was changed.

### 📊 Audit

- Implementation footprint: 14 Rust files, 85 insertions, 275 deletions, for
  a net reduction of 190 lines before changelog and status updates.
- Contract impact: this is a pre-1.0 Rust API hard cut for the removed
  `evidence_envelope::sha256_hex` helper and narrowed direct replica-query
  functions. Dependencies, package versions, CLI behavior, Candid, JSON,
  configuration, stable state, and runtime behavior are unchanged.

## [0.98.4] - 2026-07-22 - Host Configuration And Replica Query Consolidation

### 📝 Summary

- Host and CLI operations now load one validated fleet configuration snapshot
  and derive every projection and role-package decision from that immutable
  model. Direct local-replica queries similarly use one optional-root transport
  instead of parallel rooted and discovery-based function families.

### 🗑️ Removed

- Removed thirteen standalone fleet projection functions after moving their
  maintained behavior onto `FleetConfigSnapshot`. No aliases or path-based
  projection wrappers remain.
- Removed the unused installed-deployment and install-state entry points that
  implicitly discovered an ICP root, together with their dead root-discovery
  error branch. Maintained callers provide the selected ICP root explicitly.
- Removed the unused deployment-catalog schema getter; the named schema
  constant remains the sole authority.
- Removed the duplicate `_from_root` local-replica query functions and the
  second subnet-registry response path.

### 🔧 Changed

- Config listing, fleet rendering, Medic, status, deployment truth, ICP config
  discovery, install checks, build provenance, role-package validation, and
  state-manifest resolution now reuse one parsed config within each operation.
  A file change during an operation cannot mix projections from different
  snapshots.
- Split-source config discovery retains one narrow fleet-identity parser so it
  can deduplicate candidates before an incomplete config is operationally
  valid; it does not project roles or bypass snapshot validation.
- Role-package validation and contract resolution accept the already-validated
  config model. The old path-based parse/reopen boundary and `_from_config`
  duplicate naming are hard-cut.
- Local readiness, cycle-balance, bootstrap-status, and subnet-registry queries
  select configured-root or default endpoint discovery inside one transport.

### 🧪 Testing

- Focused configuration tests prove a loaded snapshot remains immutable after
  its source file changes, and all release-set projection and role-contract
  tests exercise the surviving config-model path.
- Focused host replica-query, deployment, provenance, manifest, and CLI command
  suites cover the consolidated consumers.

### 📊 Audit

- Contract impact: this is a breaking pre-1.0 public Rust API hard cut for the
  removed host projection, implicit-root deployment, replica `_from_root`, and
  path-reopening role-contract functions. CLI text/exit behavior, JSON,
  Candid, stable state, configuration syntax, runtime behavior, dependencies,
  and package versions are unchanged.

## [0.98.3] - 2026-07-22 - Deterministic Bootstrap Wasm-Store Builds

### 📝 Summary

- Bootstrap Wasm-store generation now consumes one exact resolved Canic
  package family and one shared build-profile definition. The packaged
  downstream proof exercises that maintained surface without brittle manifest
  text assumptions.

### 🩹 Fixed

- Replaced first-name and filesystem-order package selection with exact
  resolved-package admission. Competing Canic packages fail closed, and a
  missing exact packaged sibling can no longer fall through to an arbitrary
  older cached version.
- Repaired the packaged-downstream release proof to inspect the generated
  manifest structurally and require exactly the maintained `metrics` and
  `wasm-store-canister` runtime features.
- Removed the disagreement between generated, command-line, workspace, and
  documented fast profiles. They now consistently disable incremental output.

### 🔧 Changed

- Upgraded the workspace `ic-cdk` requirement from 0.20.1 to 0.20.2. The lock
  already selected 0.20.2, so this corrects the declared authority without a
  second resolved CDK version.
- Generated wrappers derive exact `ic-cdk` and `candid` versions from the
  selected Canic core dependency graph instead of carrying independent version
  literals.
- Release and fast profile rendering plus Cargo command overrides now consume
  the same settings tables.

### 🧪 Testing

- Focused host tests cover competing Canic packages, exact packaged siblings,
  resolved wrapper dependencies, generated feature selection, and shared
  profile settings.
- Both generated-wrapper and canonical packaged-downstream Wasm-store builds
  pass through the release proof.

### 📊 Audit

- Contract impact: no Candid, stable-state, JSON, CLI, configuration, or public
  Rust surface changes. Bootstrap build admission is stricter and
  deterministic; dependency and profile behavior otherwise retains the
  maintained current values.

## [0.98.2] - 2026-07-22 - Architectural Sediment Closeout

### 📝 Summary

- The maintainer approved combining the original randomness Slice C and the
  bounded repository consolidation before publication. The complete patch is
  compared to immutable `v0.98.1`; no intermediate dirty-worktree fingerprint
  or separate release remains authoritative.

### 🗑️ Removed

- Removed the accepted, defaulted, rendered, and publicly reachable per-role
  randomness schema. Generated `CanisterConfig` values no longer carry a
  randomness field, and the implicit Wasm-store config has no false default.
- Removed the unreachable management `raw_rand` infra and ops adapters, their
  invalid-length error, and both raw-randomness metric variants. No PRNG,
  time-derived seed, reseed timer, deprecated wrapper, or application fallback
  replaces them.
- Removed impossible randomness entries from host configured-role details and
  removed the active configuration guide's source, interval, and behavior
  claims.
- Removed the consumerless host duration parser, unreachable core
  management-canister snapshot operations and metrics, and orphan host ICP
  call/start/stop/snapshot/version/display convenience methods. Backup and
  restore retain the sole typed host snapshot authority.
- Removed the obsolete internal capability verifier/router and singleton root
  and issuer payload-kind taxonomies. The surviving structural proof and
  fixed seed/domain byte families are validated directly.
- Removed the retired testing annotation breadcrumb and narrowed
  control-plane/host helpers to their proved consumers without aliases.

### 🔧 Changed

- `ConfigError::CannotParseToml` now carries a typed `ConfigTomlIssue` in
  addition to diagnostic detail. Nested unknown fields expose their logical
  path and field name without callers parsing TOML/Serde display text.
- An explicit retired randomness table now fails through the maintained
  `ConfigInvalid -> CoreConfig(Project) -> CannotParseToml` chain. Omitted
  configuration and every checked-in active `canic.toml` remain valid.
- Host-only configuration parsing uses `serde_path_to_error 0.1.20`; it is not
  linked into Canic Wasm runtime builds.
- Role build-script validation parses Rust syntax and counts the exact
  `canic::build!` macro, so multiline syntax works while comments and strings
  cannot satisfy the contract. The direct host edge reuses the existing Syn
  3.0.3 package.
- State-manifest resolution carries one parsed configuration through passive
  validation instead of reopening it for every role.
- CycleTracker memory 29 is described as active by the same record/data
  snapshot authority that owns its existing persisted entries.
- The configuration guide now describes the complete strict current schema
  and its exact example is parsed by an executable test.
- The token trust-chain recurring audit is fingerprinted as method v2 with
  search terms for the surviving seed/domain authorities; its superseded v1
  identity remains historical evidence.

### 🧪 Testing

- Typed strict-schema tests prove both the direct core parser and the complete
  host projection preserve the exact nested logical path and unknown field.
- All 15 checked-in active configuration files parse and validate. The
  standalone minimal and root audit profiles compile their generated config,
  and focused core config/bootstrap plus host projection suites pass.
- Canic reference and protocol-surface tests pass, including the checked-in
  Wasm-store Candid contract. Strict all-target Clippy passes for `canic-core`,
  `canic-host`, and `canic` with warnings denied.
- Focused role-contract, state-manifest, state-contract, capability, auth,
  host ICP, CLI state, dependency, protocol, and reference checks pass.
- The recorded cumulative unit and PocketIC selection, strict workspace
  Clippy, formatting, layering, feature/dependency, audit-catalog,
  release-policy, changelog-governance, and diff-hygiene gates pass.

### 📊 Audit

- Footprint: two public config types, one config field, two adapter layers, one
  management error, two metric variants, one host projection branch, and one
  production randomness source file are deleted, together with the bounded
  dead runtime/host modules and intermediate taxonomies listed above.
- The complete consolidation ledger resolves 42 candidates: zero P0, one
  fixed P1, 11 fixed P2 findings, 30 proved notes, and no deferred or
  unresolved item across all 37 workspace packages.
- Contract impact: this is a breaking pre-1.0 TOML and public Rust hard cut;
  exhaustive metric matches and host role-detail values lose the impossible
  randomness cases, and unconsumed public Rust helpers are removed or
  narrowed. Product Candid, stable encodings and memory IDs, maintained JSON,
  CLI commands/output, backup/restore and deployment formats, workspace
  package set, and package versions are unchanged.

## [0.98.0] - 2026-07-22 - Immutable Baseline And Dead Package Hard Cut

### 🔧 Changed

- Upgraded the repository tool contract from `ic-wasm 0.9.11` to `0.10.0`.
  Installation now consumes the platform-specific official GitHub release

  SHA-512 variables were removed without a fallback. Canic uses the binary CLI,

  recovery-required while preserving the primary typed cause.

  role-contract diagnostics.

  example, and documentation inventory against the released post-0.97 tree.

- Removed the consumerless `project-protocol-stub` package, its workspace
