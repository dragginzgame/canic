## 0.30.0 - 2026-05-03 - Fleet Snapshot Backups

### 📝 Summary

- `0.30.0` starts the fleet snapshot backup line with a design-backed backup model, reusable backup primitives, and an operator CLI for capturing canister snapshots from topology-defined units.

### ➕ Added

- Added `canic-backup`, a reusable crate for backup manifests, topology hashes, artifact checksums, resumable journals, and restore dry-run planning.
- Added `canic-cli`, with `canic snapshot download` for downloading snapshots for one canister or a canister plus direct/recursive registry-discovered children.
- Added topology-hash validation so backup manifests can bind discovery and snapshot execution to the same canonical `(pid, parent_pid, role, module_hash)` view.
- Added manifest fields and validation for member verification checks, quiescence strategy names, identity modes, module/wasm provenance, and backup-unit consistency rules.
- Added resumable download journal states: `Created`, `Downloaded`, `ChecksumVerified`, and `Durable`.
- Added restore planning that validates remaps, blocks remapping fixed-identity members, and orders parents before children.

```bash
canic snapshot download \
  --canister <canister-id> \
  --root <root-canister-id> \
  --recursive \
  --out backups/<run-id> \
  --stop-before-snapshot \
  --resume-after-snapshot
```

```bash
canic restore plan \
  --manifest backups/<run-id>/manifest.json \
  --mapping restore-map.json \
  --out restore-plan.json
```

### 🔧 Changed

- Moved the snapshot/backup design into `docs/design/0.30-canister-snapshots/0.30-design.md` and made the topology, directory, quiescence, restore-ordering, verification, and registry-projection rules explicit.
- Kept endpoint macro compatibility after the public endpoint metrics backend removal by preserving the access-layer hook names.
