## 0.30.7 - 2026-05-03 - Backup Preflight Bundle

### 📝 Summary

- `0.30.7` makes `canic snapshot download` produce the canonical manifest, journal, and artifact layout that `backup preflight` expects. It also adds a single no-mutation backup preflight command that writes the standard validation, status, integrity, restore-plan, and summary reports for a backup directory, and keeps changelog examples aligned with the patch that introduced each command or flag.

### ➕ Added

- Added `canic backup preflight --dir <backup-dir> --out-dir <dir> [--mapping <file>]`, which reads the canonical backup layout, requires a complete journal, verifies durable artifact checksums, computes a restore plan, and writes all preflight artifacts into one report directory.
- Added `preflight-summary.json`, a compact readiness summary with the backup ID, source environment, source root, topology hash, readiness flags, member counts, restore-plan member count, and paths to each generated report.

### 🔧 Changed

- `canic snapshot download` now writes `fleet-backup-manifest.json` after a successful non-dry-run capture, so a freshly downloaded backup can be passed directly to `canic backup preflight`.
- Snapshot journals now store artifact paths relative to the backup directory, keeping captured backup layouts portable when archived or moved.

```bash
canic backup preflight \
  --dir backups/<run-id> \
  --out-dir preflight/<run-id> \
  --mapping restore-map.json
```

### 📚 Documentation

- Moved the root changelog command examples directly under the matching patch bullets.
- Kept detailed bash examples inside the specific patch sections where those command forms became available.

## 0.30.5 - 2026-05-03 - Verified Restore Planning

### 📝 Summary

- `0.30.5` lets manifest validation write report files, backup status fail on incomplete journals, and restore dry-run planning require a verified backup layout, closing the gap between validation summaries, resumability checks, `backup verify`, and operator restore runbooks. It also adds end-to-end coverage for the public Access and Perf metric rows emitted by macro-generated endpoints.

### ➕ Added

- Added `canic restore plan --backup-dir <dir> --require-verified`, which runs the canonical manifest, journal, durable artifact, and SHA-256 checksum verification path before restore planning.
- Added `canic backup status --dir <dir> --require-complete`, which still writes the resumable status report and then returns an error when any artifact is not durable yet.
- Added `canic manifest validate --manifest <file> --out <file>`, so validation summaries can be captured alongside backup status, integrity, and restore plan reports.
- Added CLI coverage for the verified planning path, including the case where `--require-verified` is rejected for direct `--manifest` planning because there is no canonical backup layout to inspect.
- Added PocketIC coverage that verifies a denied signer guard increments `canic_metrics(Access)` and a successful signer endpoint dispatch increments `canic_metrics(Perf)`.



## 0.30.3 - 2026-05-03 - Backup Integrity Verification

### 📝 Summary

- `0.30.3` adds backup journal status and layout integrity verification so operators can inspect resumability and prove a downloaded backup is complete before using it for restore planning.

### ➕ Added

- Added `BackupLayout::verify_integrity()`, which reads the canonical manifest and download journal, checks matching backup IDs, requires each manifest member to have a durable journal artifact, rejects extra journal artifacts, and verifies artifact SHA-256 checksums.
- Added `DownloadJournal::resume_report()` plus serializable journal status report types that summarize artifact states, completion status, pending artifact count, and next resume actions.
- Added serializable `BackupIntegrityReport` and `ArtifactIntegrityReport` output types for audit logs and runbooks, including an explicit `verified` flag on successful integrity reports.
- Added `canic backup status --dir <backup-dir> [--out <file>]`, which emits the resumable journal report as JSON.
- Added `canic backup verify --dir <backup-dir> [--out <file>]`, which emits the integrity report as JSON.
- Extended `canic restore plan` with `--backup-dir <dir>` as an alternative to `--manifest <file>`, so restore planning can read the canonical manifest from a verified backup layout.

```bash
canic backup status \
  --dir backups/<run-id> \
  --out backup-status.json
```

```bash
canic backup verify \
  --dir backups/<run-id> \
  --out backup-integrity.json
```

```bash

  --backup-dir backups/<run-id> \
  --mapping restore-map.json \
  --out restore-plan.json
```

## 0.30.0 - 2026-05-03 - Fleet Snapshot Backups

### 📝 Summary

- `0.30.0` starts the fleet snapshot backup line with a design-backed backup model, reusable backup primitives, and an operator CLI for capturing canister snapshots from topology-defined units.

### ➕ Added

- Added `canic-backup`, a reusable crate for backup manifests, topology hashes, artifact checksums, resumable journals, and restore dry-run planning.
- Added `canic-cli`, with `canic snapshot download` for downloading snapshots for one canister or a canister plus direct/recursive registry-discovered children.
- Added topology-hash validation so backup manifests can bind discovery and snapshot execution to the same canonical `(pid, parent_pid, role, module_hash)` view.
- Added manifest fields and validation for member verification checks, quiescence strategy names, identity modes, module/wasm provenance, and backup-unit consistency rules.
- Added resumable download journal states: `Created`, `Downloaded`, `ChecksumVerified`, and `Durable`.
- Added restore planning that validates remaps, blocks remapping fixed-identity members, and orders parents before children.

```bash
canic snapshot download \
  --canister <canister-id> \
  --root <root-canister-id> \
  --recursive \
  --out backups/<run-id> \
  --stop-before-snapshot \
  --resume-after-snapshot
```

### 🔧 Changed

- Moved the snapshot/backup design into `docs/design/0.30-canister-snapshots/0.30-design.md` and made the topology, directory, quiescence, restore-ordering, verification, and registry-projection rules explicit.
