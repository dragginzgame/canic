# 0.21.x Detailed Changelog

Technical release notes for the `0.21.x` minor line.

## 0.21.11 - 2026-04-02 - Installer Profile and Progress Visibility

### 📝 Summary

- `0.21.11` tightens the local thin-root install and long PocketIC smoke path by preserving caller-selected build profiles, keeping the repo-local installer fast by default, hardcoding the wasm staging/install chunk size to the IC-safe `1_048_576` byte value, and making staging/proof phases visibly progress instead of looking hung.

### 🔧 Changed

- `canic-installer` no longer forces `RELEASE=1` during its internal `dfx build --all` step, so downstream callers can explicitly drive debug/local artifact builds with `RELEASE=0` while leaving the unset/default behavior unchanged.
- The repo-local `make demo-install` and `make test-canisters` convenience path now passes `RELEASE=0` by default, so manual local smoke installs use the optimized dev wasm profile unless a caller explicitly opts back into `RELEASE=1`.
- The repo-local serial `make test` and `make test-wasm` paths now run with `--nocapture`, so the new long PocketIC phase markers are visible during slow runs instead of being swallowed by the Rust test harness.
- Canic wasm staging/install chunking is now hardcoded to the canonical `1_048_576` byte value through one shared constant, and there is no env-var or config override surface for chunk size.
- `scripts/app/build.sh` now prints the effective canister build environment before each custom DFX build, including the selected profile, `RELEASE`, `DFX_NETWORK`, `CANIC_WORKSPACE_ROOT`, and `CANIC_DFX_ROOT`.
- The installer-side release staging loop now prints live per-release and per-chunk progress, so long ordinary-release uploads are visible instead of silent between `root` reinstall and `canic_ready` polling.
- Root-side template staging now emits `canic::log!` progress for manifest staging, chunk-set preparation, and chunk publication, so the replica log shows the same chunk-by-chunk staging activity as the installer.
- The long grouped `pic_role_attestation` capability scenarios now emit short phase markers during setup and policy/proof checks, so serial PocketIC runs keep visibly advancing through the quiet capability-path phases.
- The root PocketIC baseline builder now lives in `canic-testkit`, so downstream projects can reuse the same immutable cached-baseline pattern instead of re-owning root install/stage/wait plumbing in each test harness.
- The workspace test runner now prebuilds the root `.dfx` artifacts once before the heavy PocketIC suites and prints per-suite elapsed time, which removes repeated “first root test” artifact builds and makes long phases attributable in normal `make test` output.

## 0.21.10 - 2026-04-02 - Split Workspace and DFX Root Support

### 📝 Summary

- `0.21.10` closes the remaining monorepo path assumption in `canic-installer` by separating the Cargo workspace root from the DFX/project root, so downstream apps can keep one real repo-root `.dfx` even when their Rust workspace lives under `backend/`.

### 🔧 Changed

- `canic-installer` now distinguishes:
  - `CANIC_WORKSPACE_ROOT` for Cargo, `canic.toml`, and canister manifests
  - `CANIC_DFX_ROOT` for `dfx.json`, `.dfx`, emitted artifacts, and the hidden generated bootstrap-store wrapper
- The public ordinary-canister builder, bootstrap-store builder, release-set emitter/stager, and local install path now write artifacts under the DFX root instead of assuming `.dfx` always lives under the Cargo workspace.
- Root build-time embedded bootstrap discovery now honors `CANIC_DFX_ROOT`, so `root` can embed the hidden `wasm_store` artifact from the real DFX tree even when the Cargo workspace is nested elsewhere.
- The repo-local `make demo-install` and `make test-canisters` smoke path now defaults to `RELEASE=0`, so local thin-root installs use the optimized dev wasm profile unless a caller explicitly opts back into `RELEASE=1`.

## 0.21.3 - 2026-04-01 - Fleet Observability and Lifecycle Follow-Through

### 📝 Summary

- `0.21.3` adds the next `wasm_store` hardening pass by making fleet placement and retired-store lifecycle state readable from `root` directly, while extending the PocketIC runtime proofs so the managed-store lifecycle is covered beyond the happy-path publish/reconcile flow.

### 🔧 Changed

- Added a root-facing live publication status surface so controllers can inspect the managed `wasm_store` fleet in terms of preferred binding, slot state, publication candidate order, and exact/conflicting managed-release counts without relying on logs.
- Added a root-facing retired-store status surface that reports the current retired binding, GC readiness, reclaimable bytes, and the live underlying store status while one retired store is moving through cleanup.
- Kept these new fleet diagnostics as controller-only update reads because they resolve live store state through inter-canister calls; they are not static root-state projections like the approved-release overview.

### 🩹 Fixed

- Fixed-target admin publication is now covered by runtime proofs for both conflicting duplicate rejection and capacity-failure rejection, ensuring those explicit-store paths fail closed without mutating the managed fleet view.
- The retired-store lifecycle is now covered end to end at runtime: retire, prepare GC, begin GC, complete GC, finalize, and delete all prove the expected slot transitions and cleanup behavior.
- `lifecycle_boundary` no longer fails spuriously when PocketIC rate-limits repeated `install_code` calls; the test now retries the transient install path and keeps asserting the real lifecycle phase errors.

- Removed the downstream compile hazard where the implicit store preset could be pulled through const contexts that external consumers compile more strictly than the workspace itself.

## 0.21.0 - 2026-04-01 - Implicit Wasm Store and Managed Release Fleet

### 📝 Summary

- `0.21.0` starts the managed release-fleet line by making `wasm_store` an internal system concern instead of a downstream-owned canister crate.
- `root` now bootstraps the canonical `wasm_store` from the build-produced `.wasm.gz` artifact, publishes ordinary releases as gzipped payloads across a tracked store fleet, and preserves the exact approved binding through post-upgrade reconcile.
- The downstream contract is now much closer to the intended product promise: users build with `canic`, install `root`, and do not carry a bespoke `wasm_store` crate or a manual post-install bootstrap flow.

### 🔧 Changed

- Added the canonical publishable `canic-wasm-store` package as the owned source of truth for the implicit `wasm_store` canister.
- `scripts/app/build.sh wasm_store` now resolves the canonical store package from the local workspace, Cargo metadata, sibling packaged sources next to the resolved `canic` package, or a generated hidden wrapper when only packaged `canic` is present.
- `root` now embeds the build-produced `wasm_store.wasm.gz` artifact as the dedicated bootstrap payload and logs explicit provenance for the source path, embedded path, byte size, and hashes during init.
- Ordinary root release bundles now embed `.wasm.gz` payloads for child roles instead of raw `.wasm`, keeping bootstrap publication aligned with the real DFX/custom build artifacts and reducing initial store pressure.
- Root publication now manages a tracked `wasm_store` fleet instead of assuming one current store: it reuses exact existing releases, rejects conflicting duplicates, promotes or creates fresh stores when needed, and keeps publication/import state aligned with the actual active binding.
- Root post-upgrade reconcile now validates approved releases against the exact current bytes present somewhere in the fleet and preserves the current approved store binding instead of conflicting on older copies that remain in older stores.
- The root-owned store overview now reports approved-release projections honestly rather than implying that it is a live physical occupancy report for each store.
- The hidden packaged-downstream build path now works hermetically with only packaged `canic` as the explicit dependency, synthesizing its own wrapper crate under `.dfx/local/generated/canic-wasm-store` and producing the expected local `wasm_store.wasm` and `wasm_store.wasm.gz` artifacts.

### 🩹 Fixed

- Root no longer drifts back to a stale checked-in bootstrap payload when a newer DFX-built `wasm_store` artifact exists.
- Invalid empty or non-wasm gzip payloads now fail at root build time instead of surfacing later as opaque runtime install failures.
- Large release salvos no longer assume one store can hold every approved release; the fleet manager can spill publication onto additional stores as capacity is consumed.
- Root upgrade readiness no longer has the built-in false conflict where an older store keeping a previous role version could break reconcile even when the exact current release still existed elsewhere in the fleet.
- Downstreams no longer need to add their own `wasm_store` crate or manual bootstrap script just to satisfy the implicit bootstrap artifact contract.
