# 0.72 Changelog

## 0.72.0 - 2026-06-26 - Security defaults hardening

### 📝 Summary

- Hardens Canic's security-sensitive defaults after the framework audit. The
  release makes endpoint openness explicit, makes whitelist absence deny by
  default, bounds child cycles funding, persists funding budget state, and
  closes the local wasm-store GC double-completion window.

### ⚠️ Breaking

- Endpoint macros now reject `#[canic_update]` and `#[canic_query]`
  declarations unless the endpoint includes either `requires(...)` or `public`.
  App endpoints that were intentionally open must be marked `public`; sensitive
  endpoints should declare a caller or delegated-auth predicate.

- `public` cannot be combined with `requires(...)`. The access DSL also rejects
  `not(...)` around caller predicates and delegated-auth predicates, so guards
  cannot accidentally invert identity checks.

- `caller::is_whitelisted()` now fails closed when no whitelist is configured.
  Apps that used this predicate as a permissive feature flag must configure a
  whitelist or choose a different guard.

- Child cycles funding no longer defaults to unlimited grants. The default
  per-child-role policy is `5T` per request, `100T` per child, and a `60s`
  cooldown. Deployments that need larger grants should configure
  `cycles_funding` for the child role.

### 🔧 Changed

- Adds `cycles_funding` to canister role config and generated config output.
  The root funding authorization path now resolves the effective policy from
  the current subnet config instead of using a fail-open helper.

- Persists child funding totals and cooldown timestamps in stable memory. Grant
  accounting is recorded before the external deposit call and restored only if
  that call returns an error, so successful deposits cannot escape the budget
  ledger if a later continuation fails.

- Marks intended public framework queries, public token-preparation endpoints,
  blob-storage compatibility helpers, and test canisters with the explicit
  `public` endpoint marker.

- Local wasm-store GC now transitions from `InProgress` to `Clearing` before
  awaiting the clear operation, then either returns to `InProgress` for retry
  on clear failure or advances to `Complete` after a successful clear.

### 🧪 Testing

- Adds focused coverage for endpoint `public` parsing and validation,
  rejected implicit-open endpoints, rejected negated identity predicates,
  whitelist default-close behavior, cycles funding config parsing and
  validation, config-backed request-cycle budget denial, stable funding ledger
  accumulation and restore, and wasm-store GC clearing transitions.
