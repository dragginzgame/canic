# 0.64 Changelog

## 0.64.1 - 2026-06-09 - Service manager runtime policy

### Summary

- Hardens the service/singleton split at runtime by requiring service parents
  for directory, scaling, and sharding manager pools.

- Keeps parent-scoped singleton child creation valid for downstream app shapes
  where an instance owns one child role, such as a ledger.

### Changed

- Registry registration policy now treats only `kind = "service"` as a valid
  manager-pool parent kind for directory, scaling, and sharding children.

- Singleton parents with directory, scaling, or sharding config are rejected by
  runtime registration policy, matching the 0.64 schema-level validation rule.

- Internal registry policy variants and messages now use service terminology
  for manager-pool parent requirements.

- Public Candid error-code names for the existing manager-parent policy errors
  remain unchanged for compatibility, while their messages now describe the
  service-parent requirement.

## 0.64.0 - 2026-06-09 - Service/singleton topology split

### Summary

- Introduces `kind = "service"` as the explicit config kind for root-scoped,
  root-created canister roles.

- Keeps `kind = "singleton"` for parent-scoped child cardinality, including
  downstream shapes where an instance owns one child role such as a ledger.

- Moves root bootstrap and SubnetIndex derivation from all singleton roles to
  service roles only.

- Keeps current AppIndex as an explicit role-keyed V0 surface, but validates
  entries as service roles instead of singleton roles.

- Adds the 0.64 topology design note under
  `docs/design/0.64-service-singleton-topology/`.

### Changed

- `CanisterKind` now includes `Service`, parses `kind = "service"`, and renders
  it through the bootstrap config surface.

- `SubnetConfig::auto_create_roles()` and
  `SubnetConfig::subnet_index_roles()` now derive from service roles.

- Current AppIndex entries must be declared in the prime subnet with
  `kind = "service"`.

- Singleton roles may no longer define directory, scaling, or sharding manager
  pools in validated configs.

- Service roles may own directory, scaling, and sharding manager pools.

- Registry policy treats service roles as role-unique root children, rejects
  service registration under non-root parents, and continues to enforce
  parent-scoped singleton uniqueness by `(parent_pid, role)`.

- Root AppIndex and SubnetIndex builders filter registry entries to direct root
  children before duplicate role detection, so nested stale or same-role child
  records cannot poison root indexes.

- Checked-in fleet configs, audit/test canister configs, scaffolding templates,
  install/status/build fixtures, and packaged downstream CLI fixtures now use
  `kind = "service"` for root-created roles.

### Migration Notes

- Root-owned canisters that should be bootstrapped by root or exposed through
  SubnetIndex should use `kind = "service"`.

- `app_index` entries should remain explicit, but each listed role must now be
  a prime-subnet service.

- Parent-owned singleton children should remain `kind = "singleton"` and should
  be resolved through parent-scoped lookup rather than root indexes.

- Existing singleton examples remain valid only where they are explicitly
  modeling parent-scoped child cardinality or schema validation behavior.
