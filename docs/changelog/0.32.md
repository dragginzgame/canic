## 0.32.4 - 2026-05-07 - Script Cleanup

### 🧹 Cleanup

- Removed `scripts/app/reference_canisters.sh` and older redundant fleet command surfaces.
- Removed the public `canic release-set` command; manifest emission and staging are now owned by `canic build` and `canic install` rather than exposed as an operator command family.
- Centralized command help/version handling in the CLI so command modules no longer repeat their own pre-parse checks.
- Tightened nested CLI help/version handling so subcommands such as `canic backup list help`, `canic restore plan help`, and `canic snapshot download --version` exit cleanly, while option values named `version` are parsed as ordinary values.
- Routed local install and root-staging dfx command construction through the shared host `dfx` helpers instead of open-coding `Command::new("dfx")`.
- Made fleet config discovery and confirmed fleet deletion skip symlinks and stay inside the real `fleets/` tree.
- Kept the `CANIC_REFERENCE_CANISTERS` override for focused PocketIC fixture builds.
- Added `scripts/dev/README.md` to mark `scripts/dev/*` as intentional local developer/operator helpers rather than stale CLI leftovers.

### ⚠️ Breaking

- Made role-attestation `audience` required in `RoleAttestationRequest` and `RoleAttestation`; missing audiences now fail at the DTO boundary instead of through a later policy branch.

### 🔧 Changed

- Top-level help now groups commands by context level: default context commands, current network/fleet commands, and workspace/file commands.
- Top-level help now includes the CLI package version in the title banner.
- Added `canic network` and `canic network use <name>` so operators can select a high-level network default while still using `--network <name>` for one-command overrides; both commands print the selected network as a plain scalar value.
- Added `canic status` to show the current network, selected fleet, root canister, and config path in one human-readable default-context summary.
- Replaced separate `canic fleets` and `canic use` surfaces with `canic fleet`, `canic fleet list`, `canic fleet use <name>`, and `canic fleet delete <name>`; fleet list/use/delete now operate on config-defined fleets rather than requiring prior install state.
- `canic scaffold <name>` now selects the new scaffold as the current fleet for the target network, so the next `canic install` can use the matching scaffold config by default.
- `canic install` now propagates the selected network to dfx subprocesses and Rust build scripts, so a saved `canic network use ic` default no longer builds or queries a root canister as `local`.
- `canic manifest validate` now explains that `--manifest <file>` is required and prints command-specific usage when the path is omitted.
- JSON `--out <path>` writers now create missing parent directories before writing reports.

## 0.32.1 - 2026-05-07 - Canic Executable

### 📝 Summary

- `0.32.1` focuses the `canic` executable into a clearer operator surface for installing, inspecting, backing up, and diagnosing fleets.
- Common `canic` commands are easier to discover and run without remembering internal paths or root IDs.

### ➕ Added

- `canic medic` runs read-only setup diagnostics for the selected network, `dfx ping`, saved fleet state, config path, and root `canic_ready` status.
- `canic scaffold <name>` creates `fleets/<name>/canic.toml` plus a minimal standalone root canister under `fleets/<name>/root`.
- `canic backup list` scans backup directories under `backups/` by default and prints reusable backup paths from validated manifests.
- `canic snapshot download` can now use the current fleet or `--fleet <name>` without requiring the operator to spell out the root canister ID.

### 🔧 Changed

- `canic list` now shows the selected fleet and network above the tree, keeps `--fleet <name>` as the explicit selector for non-default installed fleets, and renders the table with wider spacing plus a header separator.
- `canic snapshot download` now defaults its output directory to `backups/fleet-<name>-YYYYMMDD-HHMMSS`; `--out <dir>` remains available for explicit paths.
- `canic snapshot download --fleet <name> --canister <id>` now verifies that the selected canister belongs to that fleet before any snapshot mutation starts.
- `canic scaffold <name>` now asks for confirmation before creating files; pass `--yes` for noninteractive setup scripts.
- Top-level `canic help` now displays the installed version and groups commands by multi-fleet, single-fleet, and single-canister scope while still deriving the command list from Clap metadata.
- `canic install help`, `canic list help`, `canic fleet help`, and `canic medic help` now render fuller Clap-backed option descriptions and examples instead of terse one-line usage text.
- Config selection now previews up to six canisters before ellipsising longer install choices.
- New fleet scaffolds and implicit install defaults now use top-level `fleets/`.

```bash
canic scaffold my_app
canic install --config fleets/my_app/canic.toml
canic snapshot download --fleet demo
canic backup list
```

### 🧭 Migration Notes

- Use `canic scaffold <name>` for new fleets, then install the generated config with `canic install --config fleets/<name>/canic.toml`.

### 📚 Documentation

- Updated the root README and `canic-cli` README to explain the current `canic` executable surface, show config-owned fleet identity, document `canic medic`, and remove `canic install --fleet` examples.
- Clarified `canic-cli` as the narrow operator binary surface, with host-side install/build helpers in `canic-host` and backup contracts in `canic-backup`.

### 🧪 Testing

- Added and updated tests for fleet-name validation, install-state rejection, config-choice previews, grouped help, command-level help, `canic scaffold`, `canic medic`, `canic list` formatting, fleet-aware snapshot downloads, and backup listing.

## 0.32.0 - 2026-05-07 - Fleet Identity

### 📝 Summary

- `0.32.0` makes fleet identity explicit in `canic.toml` and tightens the first multi-fleet operator flows.

### ⚠️ Breaking

- `canic install` no longer accepts `--fleet <name>` and no longer reads `CANIC_FLEET`; install fleet identity must come from `[fleet].name` in the selected config.
- Existing install state that does not contain a fleet name is rejected instead of being displayed as `default`; reinstall from a config that declares `[fleet].name`.

### 🔧 Changed

- `canic list` and top-level help became clearer for multi-fleet operator workflows.

```toml
[fleet]
name = "demo"
```

### 🧭 Migration Notes

Add a fleet identity to every install config, then reinstall the fleet so `.canic/<network>/fleets/<name>.json` and the current-fleet pointer are written from the config-owned name.
