## 0.32.1 - 2026-05-07 - Canic Executable

### 📝 Summary

- `0.32.1` focuses the `canic` executable into a clearer operator surface for installing, inspecting, backing up, and diagnosing fleets.
- Common `canic` commands are easier to discover and run without remembering internal paths or root IDs.

### ➕ Added

- `canic medic` runs read-only setup diagnostics for the selected network, `dfx ping`, saved fleet state, config path, and root `canic_ready` status.
- `canic scaffold <name>` creates `canisters/<name>/canic.toml` plus a minimal standalone root canister under `canisters/<name>/root`.
- `canic backup list` scans backup directories under `backups/` by default and prints reusable backup paths from validated manifests.
- `canic snapshot download` can now use the current fleet or `--fleet <name>` without requiring the operator to spell out the root canister ID.

### 🔧 Changed

- `canic list` now shows the selected fleet and network above the tree, keeps `--fleet <name>` as the explicit selector for non-default installed fleets, and renders the table with wider spacing plus a header separator.
- `canic snapshot download` now defaults its output directory to `backups/fleet-<name>-YYYYMMDD-HHMMSS`; `--out <dir>` remains available for explicit paths.
- `canic snapshot download --fleet <name> --canister <id>` now verifies that the selected canister belongs to that fleet before any snapshot mutation starts.
- `canic scaffold <name>` now asks for confirmation before creating files; pass `--yes` for noninteractive setup scripts.
- Top-level `canic help` now displays the installed version and groups commands by multi-fleet, single-fleet, and single-canister scope while still deriving the command list from Clap metadata.
- `canic install help`, `canic list help`, `canic fleets help`, `canic use help`, and `canic medic help` now render fuller Clap-backed option descriptions and examples instead of terse one-line usage text.
- Config selection now previews up to six canisters before ellipsising longer install choices.

```bash
canic scaffold my_app
canic install --config canisters/my_app/canic.toml
canic snapshot download --fleet demo
canic backup list
```

### 🧭 Migration Notes

- Use `canic scaffold <name>` for new fleets, then install the generated config with `canic install --config canisters/<name>/canic.toml`.

### 📚 Documentation

- Updated the root README and `canic-cli` README to explain the current `canic` executable surface, show config-owned fleet identity, document `canic medic`, and remove `canic install --fleet` examples.
- Clarified `canic-cli` as the narrow operator binary surface, with host-side install/build helpers in `canic-host` and backup contracts in `canic-backup`.

### 🧪 Testing

- Added and updated tests for fleet-name validation, install-state rejection, config-choice previews, grouped help, command-level help, `canic scaffold`, `canic medic`, `canic list` formatting, fleet-aware snapshot downloads, and backup listing.

## 0.32.0 - 2026-05-07 - Fleet Identity

### 📝 Summary

- `0.32.0` makes fleet identity explicit in `canic.toml` and tightens the first multi-fleet operator flows.

### ⚠️ Breaking

- `canic install` no longer accepts `--fleet <name>` and no longer reads `CANIC_FLEET`; install fleet identity must come from `[fleet].name` in the selected config.
- Existing install state that does not contain a fleet name is rejected instead of being displayed as `default`; reinstall from a config that declares `[fleet].name`.

### 🔧 Changed

- `canic list` and top-level help became clearer for multi-fleet operator workflows.

```toml
[fleet]
name = "demo"
```

### 🧭 Migration Notes

Add a fleet identity to every install config, then reinstall the fleet so `.canic/<network>/fleets/<name>.json` and the current-fleet pointer are written from the config-owned name.
