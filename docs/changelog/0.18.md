# 0.18.x Detailed Changelog

Technical release notes for the `0.18.x` minor line.

## 0.18.5 - 2026-03-29 - Root-Only State Snapshots and Locked Surface Coverage

### 📝 Summary

- `0.18.5` keeps `ICRC-21` behind role-scoped compile-time gating, trims the shared generated endpoint surface by making `canic_app_state` and `canic_subnet_state` root-only, and adds integration coverage that proves those two snapshots remain available on `root` while ordinary child canisters reject them as absent.

### 🔧 Changed

- Updated the generated topology-state bundle so `canic_app_state` and `canic_subnet_state` are exported only when building the `root` canister surface.
- Kept `ICRC-21` behind the existing validated role-config gate so canisters that do not opt in do not export `icrc21_canister_call_consent_message`.

### 🧪 Testing

- `cargo test -p canic --test root_hierarchy`

## 0.18.4 - 2026-03-29 - Root-Owned Wasm Store Overview and Lighter Release Gates

### 📝 Summary

- `0.18.4` gives `root` a single controller-facing wasm-store overview built entirely from root-owned state, so operators can inspect tracked store bindings, publication slots, GC progress, and release/catalog payload totals without direct `wasm_store` calls, and it removes the older split publication/retirement read endpoints in favor of that one overview surface. The same release also smooths release workflow friction by making `make patch` / `make minor` skip PocketIC-heavy tests, requiring `dfx` to already be running, and keeping plain Cargo/clippy builds from failing when `.dfx` bundle artifacts have not been generated yet.

### 🔧 Changed

- Added `canic_wasm_store_overview` on `root` as a controller-facing query that returns publication state plus one per-store overview entry for every tracked runtime-managed wasm store.
- Added a root-owned wasm-store overview DTO surface that includes binding, canister id, publication slot, GC status, template/release counts, and payload-capacity figures derived from root-owned manifest state.
- Added root-side GC lifecycle mirroring for tracked wasm stores so the overview can report `Prepared`, `InProgress`, and `Complete` states without querying the store canister directly.
- Consolidated the root wasm-store read surface down to `canic_wasm_store_overview` by removing the older split `canic_wasm_store_publication_status` and `canic_wasm_store_retirement_status` queries.
- Updated `make patch` and `make minor` to use the lighter `test-release` gate, which keeps clippy, local canister smoke coverage, and fast workspace unit/bin/lib tests while skipping the PocketIC-heavy integration suites under `tests/`.
- Updated `test-canisters` to treat `dfx` as a required external dependency and to rely on an already-running background replica instead of trying to start or stop `dfx` itself.

### 🩹 Fixed

- Fixed the operator visibility gap where store-local `canic_wasm_store_status` was root-only and not directly callable from tooling. Operators can now query `root` for the fleet-wide overview instead of calling private store endpoints.
- Fixed bundle canister `build.rs` behavior so ordinary Cargo/clippy builds skip missing `.dfx/{local|ic}` release artifacts with warnings instead of panicking. The real canister bundle path remains strict.
- Fixed `scripts/app/build.sh` so actual bundle builds explicitly opt into strict artifact enforcement, preserving the config-driven release-table guarantees for deployed canisters while allowing non-bundle Rust builds to compile cleanly.

### 📚 Documentation

- Updated release notes to reflect that wasm-store observability now has a root-owned overview path and that local release/version-bump flows expect `dfx` to already be running rather than being managed by `make`.

### 🧪 Testing

- `cargo fmt --all`
- `cargo build -p canic-core -p canister_root -p canister_wasm_store`
- `cargo clippy -p canic-core -p canister_root -p canister_wasm_store --all-targets --all-features -- -D warnings`

## 0.18.3 - 2026-03-29 - Automatic Bootstrap and Clean Local Test Flow

### 📝 Summary

- `0.18.3` removes the last manual bootstrap step from the reference `root` flow by embedding only the first `wasm_store` payload locally while keeping ordinary child releases store-backed, updates the `canic-memory` eager-init contract so the Canic lifecycle consumes it automatically without extra downstream wiring, and fixes local `dfx` canister tests so `make test` and `make patch` start from a clean replica and no longer rerun the old manual `wasm_store` bootstrap staging path.

### 🔧 Changed

- Updated `root` build generation so it now emits two separate bootstrap inputs: the existing manifest-only release catalog for ordinary non-root roles and a dedicated embedded bootstrap release set containing only `wasm_store`.
- Updated the reference `root` canister init flow to import that bootstrap `wasm_store` payload automatically before loading the ordinary release catalog, which keeps the first-store bootstrap automatic without turning `root` back into the payload warehouse for every child canister.
- Updated the `canic-memory` eager-init API contract so `eager_init!` registers startup work into runtime-owned bootstrap state instead of relying on downstream-generated code to invoke a hidden hook manually.
- Clarified `canic-memory` documentation so the standalone contract and the full `canic::start!` / `canic::start_root!` contract are separated explicitly.

### 🩹 Fixed

- Fixed the `0.18` eager-init contract leak by making `eager_init!` register its work into `canic-memory` runtime and by having Canic lifecycle bootstrap run those registered eager-init hooks centrally before memory registry flush. Downstream Canic users no longer need extra manual startup calls to reserve memory ranges correctly.
- Fixed local build helper behavior so `scripts/app/build.sh` now extracts Candid from the debug wasm on the second, extractor-only pass. This keeps `canic::export_candid!()` debug-only while preserving `candid-extractor` compatibility for local build flows.
- Fixed `test-canisters` / `make test` / `make patch` local reliability by forcing the `dfx` canister test target to start a clean local replica before reinstalling `root`, which avoids management wasm chunk-store exhaustion from stale local state.
- Fixed the local root test/install path to stop staging `embedded:wasm_store` manually after `root` learned to bootstrap that payload itself. This removes the manifest race where the stale chunked staging flow could overwrite the new inline bootstrap contract and fail with missing chunk errors during `wasm_store` install.

### 📚 Documentation

- Updated `crates/canic-memory/README.md` to say explicitly that full Canic lifecycle users do not need extra eager-init wiring, while standalone `canic-memory` users still own that bootstrap order themselves.

### 🧪 Testing

- `cargo fmt --all`
- `cargo test -p canic-memory --lib`
- `cargo test -p canic-core --lib`
- `scripts/app/build.sh root`

## 0.18.2 - 2026-03-29 - Config-Driven Release Tables and Dynamic Store Inventory

### 📝 Summary

- `0.18.2` removes the last handwritten ordinary release-role lists by generating both `root` bootstrap catalog input and `wasm_store` embedded release input directly from `canic.toml`, moves live wasm-store inventory into runtime subnet state so `root` can create and promote stores dynamically instead of depending on one static binding, and smooths local bootstrap by skipping unstaged `auto_create` roles instead of aborting the entire init pass.

### 🔧 Changed

- Added shared build-macro hooks so canister `build.rs` files can validate `canic.toml` once and generate extra source from the validated config, while also rerunning correctly when `DFX_NETWORK` changes between `local` and `ic`.
- Switched `root` bootstrap catalog generation to walk configured subnet canister roles directly from `canic.toml`, hash the built `.dfx/{local|ic}` wasm artifacts for those roles, and emit the manifest-only embedded catalog without a handwritten release-role table.
- Switched `wasm_store` embedded release generation to the same config-driven discovery path, so ordinary release payloads now come from built artifacts selected by config instead of a manually maintained Rust array.
- Added runtime-managed wasm-store inventory to subnet state, including binding, principal, and creation time, so publication, cascade, and config fallbacks can resolve the active/default store from live state rather than one implicit configured binding.
- Taught the publication workflow to sync already-registered stores into runtime state, create the first store when none exist, promote another existing store when the preferred one is inside headroom, and create a fresh store when every known candidate is full enough to require rollover.
- Standardized debug-only Candid export behind `canic::export_candid!()`, which keeps local/dev Candid generation available while removing repeated `#[cfg(debug_assertions)]` wrappers from reference canisters and test stubs.

### 🩹 Fixed

- Fixed root bootstrap failure behavior so one missing approved manifest in `auto_create` no longer aborts the whole registry phase; unstaged roles are now skipped and later roles can still install.
- Fixed bootstrap store reconciliation so `root` first syncs existing registered `wasm_store` canisters into runtime inventory before deciding whether it needs to create the initial store.
- Fixed template-store resolution so approved bindings resolve against runtime inventory, which removes the earlier assumption that one configured store role must always be unique and statically known.

### 🧹 Cleanup

- Removed the remaining handwritten ordinary release-role lists from `root` and `wasm_store` build scripts.
- Removed the last config-side assumption that publication/store selection comes from a single implicit static store binding instead of runtime-managed inventory.

## 0.18.1 - 2026-03-28 - Staged Bootstrap and Release Surface Cleanup

### 📝 Summary

- `0.18.1` completes the staged-bootstrap follow-up by making `root` wait for a controller-staged `embedded:wasm_store` payload instead of carrying the full child store bundle, fixing local `dfx` flows so they stage that payload before expecting readiness, and trimming release-only exports so `root` drops again to `3554964` raw bytes (`delta -597330` vs `0.18.0`).

### 🔧 Changed

- Replaced the last build-baked `wasm_store` bootstrap payload in `root` with a manifest-only embedded catalog plus a staged upload flow of `manifest -> prepare -> publish -> resume`, so local root state owns bootstrap transport while installed subnet stores keep ownership of published release bytes.
- Standardized WasmStore naming around explicit `WasmStore*` APIs, DTOs, publication state, and bootstrap entrypoints, while keeping truly generic template transport/install types generic.
- Collapsed the root mutation fan-out for store lifecycle management into a single `canic_wasm_store_admin` command surface, leaving the bootstrap upload endpoints separate because they are transport steps rather than store lifecycle operations.
- Split protocol naming by responsibility: `canic-core` now keeps only the runtime inter-canister protocol constants it actually calls, while the wider endpoint-name table lives at the `canic` facade for tests and external tooling.

### 🩹 Fixed

- Fixed bootstrap install resolution so chunked store installs verify and upload missing management chunk-store hashes before `install_chunked_code`, instead of failing on “chunk hash missing from local chunk store” during root bootstrap.
- Fixed failed-install cleanup so canisters are stopped before deletion attempts, removing the repeated “must be stopped before it is deleted” failure path after bootstrap/install errors.
- Fixed build-time compact-config embedding so generated `canic.compact.toml` preserves source-compatible values for local/debug installs instead of reserializing defaults into a shape that root/test config init could reject.
- Fixed local `dfx` / `make test-canisters` behavior by adding an explicit staged-bootstrap helper for `wasm_store`, so `root` no longer stalls indefinitely at “waiting for staged wasm_store template” on local installs.

### 🧹 Cleanup

- Removed the old embedded-`wasm_store` bootstrap bundle path and related dead code that assumed `root` would keep a full child payload in its own release artifact.
- Removed release-mode `export_candid!()` from reference canisters and test stubs, keeping runtime Candid export available in debug/local builds while dropping unnecessary release metadata.
- Removed the `root` demo-only `create_minimal` and `stress_perf` endpoints from release builds entirely instead of exporting them and rejecting at runtime.

### 📊 Audit

- Current raw release wasm sizes are `2451358` for `app`, `scale`, `minimal`, and `shard`; `2500260` for `test`; `2511122` for `scale_hub`; `2606868` for `user_hub`; `2611421` for `shard_hub`; `2532998` for `user_shard`; `14107099` for `wasm_store`; and `3554964` for `root`.
- Current deterministic gzip sizes are `699613` for `app`, `699765` for `scale`, `699488` for `minimal`, `698936` for `shard`, `716928` for `test`, `717817` for `scale_hub`, `752586` for `user_hub`, `753946` for `shard_hub`, `729362` for `user_shard`, `12044966` for `wasm_store`, and `1022706` for `root`.
- Relative to `0.18.0`, the important change is `root`: it falls from `4151294` raw bytes to `3554964` after the staged-bootstrap cutover is completed and the remaining release-only Candid/demo surface is removed.

### 📚 Documentation

- Updated release notes and architecture guidance to reflect that `root` now requires a staged `wasm_store` bootstrap payload in local/manual install flows rather than relying on an embedded child bundle.
- Clarified the public protocol split so downstream callers should use `canic::protocol`, while `canic-core::protocol` is treated as the smaller runtime-internal inter-canister call table.

## 0.18.0 - 2026-03-27 - Template Store and Chunked Install Cutover

### 📝 Summary

- `0.18.0` starts the wasm-store cutover by moving ordinary deployable child payload ownership out of `root`, making install and upgrade use subnet-local chunk-store-backed publication through `install_chunked_code`, removing the old inline fallback for every role except bootstrap `wasm_store`, reducing the raw release `root` artifact to `4151294` bytes (`delta -10366542` vs `0.17.3`), simplifying setup with one implicit per-subnet `wasm_store` on a fixed 40 MB / 4 MB IC preset, and refreshing the workspace baseline to Rust `1.94.1` with `ctor 0.8` and `sha2 0.11` while keeping `root` as the only deployment, manifest, selection, and orchestration authority.

```toml
[subnets.prime]
auto_create = ["app", "user_hub", "scale_hub", "shard_hub"]

[subnets.prime.canisters.app]
kind = "singleton"
```

### 🔧 Changed

- Introduced `wasm_store` as the canonical storage/distribution canister for deployable template releases, with root-gated publication, deterministic chunk metadata, deterministic chunk lookup, and subnet-local store placement resolved from root-owned runtime state instead of user-authored store topology.
- Made approved manifest resolution the authoritative install and upgrade contract: publication now uploads chunked releases into the selected store canister’s IC chunk store, workflow verifies the manifest-bound store and chunk metadata, and `root` invokes `install_chunked_code` without acting as a full-wasm relay.
- Moved ordinary child payload ownership out of `root` and into the canonical store layer, keeping only the irreducible bootstrap `wasm_store` payload embedded in `root` while ordinary child releases are seeded and republished from `wasm_store`. Ordinary roles no longer have a generic inline install fallback.
- Replaced user-configured store topology and capacity policy with one implicit mandatory `wasm_store` per subnet, using a fixed IC-tuned preset of `40000000` max bytes and `4000000` headroom bytes so installation stays zero-config on the store dimension.
- Added explicit store lifecycle handling for rollover and retirement without legacy fallback: root-owned active/detached/retired publication binding state, store-local GC phases (`Prepared`, `InProgress`, `Complete`), explicit retired-store wipe, explicit retired-store finalization, and explicit finalized-store deletion only after GC completion and empty-store verification.
- Refreshed the workspace toolchain and dependency baseline by pinning Rust to `1.94.1`, bumping `ctor` from `0.6` to `0.8`, and bumping `sha2` from `0.10` to `0.11` so local bootstrap, CI, and downstream builds use the same current compiler and crate floor.

### 🩹 Fixed

- Removed a regressive finalized-history experiment that pushed root-only lifecycle visibility into shared publication DTO/storage state. Finalization visibility is now root-local only, so the fleet floor stays flat while operators still get current publication/finalization/deletion status.
- Replaced an overbuilt bounded lifecycle-history attempt with a smaller root-local last-deleted snapshot, preserving operator visibility without paying a large `root` wasm penalty for generalized event history.

### 📊 Audit

- Current raw release wasm sizes are `2495019` for `minimal`, `app`, `scale`, and `shard`; `2546676` for `test`; `2556974` for `scale_hub`; `2652306` for `user_hub`; `2657242` for `shard_hub`; `2579963` for `user_shard`; `14200208` for `wasm_store`; and `4151294` for `root`.
- Relative to the final `0.17.3` floor, `0.18.0` changes raw release wasm by `+61089` bytes on `minimal`/`app`/`scale`/`shard`, `+61463` on `test`, `+61217` on `scale_hub`, `+21095` on `user_hub`, `+21083` on `shard_hub`, `-310098` on `user_shard`, and `-10366542` on `root`.
- The architectural split is the important result: `root` is no longer the ordinary child payload warehouse, while `wasm_store` now carries the canonical deployable release mass at `14200208` raw bytes.
- The later `0.18` lifecycle work stayed on the correct boundary. Publication/finalization/deletion visibility and teardown moved only `root`, while `minimal`, `app`, `scale`, `shard`, `test`, `scale_hub`, `user_hub`, `shard_hub`, `user_shard`, and `wasm_store` all remained flat at the current floor once the shared finalized-history regression was removed.

### 📚 Documentation

- Updated the `0.18` design to match the IC-native chunk-store model: same-subnet wasm-store placement, `install_chunked_code` as the steady-state ordinary install path, root-authoritative manifests, an implicit mandatory `wasm_store`, and a fixed IC-oriented store-capacity preset instead of user-managed store config.
- Recorded the full `0.18` implementation trail in `docs/design/0.18-canister-templates/0.18-status.md`, including payload-ownership transfer, store-binding/state evolution, store-local GC/retirement, and the corrected root-local operator visibility boundary.
- Updated workspace documentation so the published Rust version guidance matches the enforced local and CI toolchain pin.
