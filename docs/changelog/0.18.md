# 0.18.x Detailed Changelog

Technical release notes for the `0.18.x` minor line.

## 0.18.7 - 2026-03-29 - Stable Local Build Caching and Quieter Attestation Refresh

### đź“ť Summary

- `0.18.7` fixes the last local build-cache regression in the single-build `dfx` flow by excluding generated `.did` files from shared wasm-build invalidation and by scoping interface freshness checks per canister, stops stale or no-longer-registered non-root canisters from retrying attestation-key refreshes against `root` forever, and shrinks shared capability-wire fan-out by replacing concrete proof trees in the common wire surface with `CapabilityProofBlob`.

### đź”§ Changed

- Tightened `scripts/app/build.sh` cache invalidation so generated crate-local `.did` files no longer count as workspace input changes for the shared release/debug wasm build stamps.
- Narrowed `.did` freshness checks to per-canister interface inputs, so one canister regenerating its cached interface no longer forces unrelated canisters to rebuild the whole workspace on the next `dfx` callback.
- Changed the shared capability wire contract so `CapabilityProof` now carries `CapabilityProofBlob` for role-attestation and delegated-grant proofs instead of exposing the full nested proof records directly in every generated non-root `.did`.
- Kept `canic_response_capability_v1` global while moving concrete capability-proof tree types behind root-side decode/verify paths, which reduces proof-type fan-out across non-root interfaces without removing the shared parent-capability surface.
- Removed the stale `#[expect(dead_code)]` on `TimerWorkflow::clear_guarded` now that runtime attestation refresh uses that helper directly.

### đź©ą Fixed

- Fixed repeated whole-workspace Cargo rebuilds during `dfx build --all`, where each newly generated `crates/canisters/<name>/<name>.did` file made the shared wasm-build stamp look stale for the next canister callback.
- Fixed the noisy local-dev auth loop where stale non-root canisters from older background-replica runs kept calling `root` for attestation-key refresh after they were no longer registered on the subnet. Those canisters now stop their own refresh timer once `root` explicitly denies them as unregistered.

### đź§­ Migration Notes

- If you consume capability-envelope Candid types directly, update callers and tooling to handle `CapabilityProofBlob` inside `CapabilityProof` instead of decoding `RoleAttestationProof` or `DelegatedGrantProof` from the shared wire contract.
- This is an additive size/shape cleanup for the common capability endpoint surface, but it is still a compatibility-relevant wire change for downstream code that constructs or inspects those proof payloads by Candid type.

## 0.18.6 - 2026-03-29 - Single-Build Extraction and Config-Driven Root Staging

### đź“ť Summary

- `0.18.6` removes the remaining env-based eager-init build divergence, keeps release builds single-pass while decoupling `.did` generation from release wasm, restores a single config-driven bootstrap/publication path by staging the full release set into `root` before local smoke flows continue, adds a root-owned bootstrap debug query with human-readable staged wasm sizes and chunk progress, and collapses repeated custom-canister Cargo invocations into one shared wasm build per profile.

### đź”§ Changed

- Removed `CANIC_SKIP_EAGER_INIT` and the associated compile-time build split from the `canic-memory` eager-init contract.
- Simplified `scripts/app/build.sh` so each profile now performs one shared workspace wasm build and then reuses those artifacts across the per-canister `dfx` callbacks instead of recompiling one canister crate per callback.
- Kept debug-only Candid extraction on the same built debug artifact, while release builds now stop after the normal wasm build and satisfy `dfx` by copying a cached source-of-truth `.did` file instead of extracting from the release artifact.
- Added cache invalidation for those source `.did` files so version bumps and other workspace input changes regenerate the interface from the debug artifact before later release builds reuse it.
- Updated the local `test-canisters` flow to stage the full config-defined non-root release set into `root`, resume bootstrap, and let `root` publish configured releases into `wasm_store` before local smoke calls continue.
- Added a config-driven staging helper script for local/dev flows so `root`, `wasm_store`, and `auto_create` all consume the same staged template/publication path instead of relying on a separate manual bootstrap-only script.
- Added `canic_wasm_store_bootstrap_debug` on `root` so operators can inspect whether bootstrap is blocked, which staged releases exist in the root buffer, each staged payload size, and each releaseâ€™s chunk-set progress without scraping logs.
- Added shared byte-size formatting for root-visible wasm-store debug/overview responses and staging logs, so payload/capacity figures now render in IEC units with two decimals instead of raw byte counts.
- Cleaned up wasm publication and canister-creation logs so success paths emit one role-aware line at `Ok`/`Info` instead of duplicate or misleading `Warn` records.
- Tightened bootstrap chunk sizing back to the management-canister chunk-store limit so staged installs remain compatible with PocketIC and local management-canister behavior.

### đź©ą Fixed

- Fixed the remaining eager-init extraction hazard by removing the cycle-balance execution-mode probe and treating eager init as lifecycle bootstrap work only.
- Fixed local `dfx` smoke/bootstrap runs that stalled with `root` waiting for staged `wasm_store` input and then rejecting `test` as unregistered, because the release set had never been published into a newly created store.
- Fixed the local smoke path to call the `test` canister that `root` auto-created and registered, instead of reinstalling a duplicate alias canister that would fail root auth as an unregistered caller.
- Fixed staged bootstrap ordering so `wasm_store` is submitted first but `root` resumes only after the full release set is staged, preventing the earlier race where bootstrap publication could start while later release chunk sets were still incomplete.
- Fixed the local staging helper to wait up to `300s` for root readiness and to include the explicit `wasm_store` bootstrap exception even though that role is not part of the ordinary config-defined release set.
- Fixed bootstrap-debug visibility so staged release inspection no longer exhausts the single-message instruction limit when several staged releases are present; the root debug path now counts chunks per release without cloning all chunk payload bytes.
- Fixed repeated `dfx build --all` compile churn so later custom-canister build callbacks in the same profile reuse the first shared wasm build instead of triggering additional Cargo compilation waves.
- Fixed release-only `dfx build` failures where `.dfx/local/canisters/<name>/<name>.did` was missing by restoring a persistent interface cache that release builds can copy without re-enabling release extraction.

### đź“š Documentation

- Updated `crates/canic-memory/README.md` to describe the single-artifact debug extraction model and to keep eager init framed as lifecycle bootstrap work rather than an inferred runtime mode.

## 0.18.0 - 2026-03-27 - Template Store and Chunked Install Cutover

### đź“ť Summary

- `0.18.0` starts the wasm-store cutover by moving ordinary deployable child payload ownership out of `root`, making install and upgrade use subnet-local chunk-store-backed publication through `install_chunked_code`, removing the old inline fallback for every role except bootstrap `wasm_store`, reducing the raw release `root` artifact to `4151294` bytes (`delta -10366542` vs `0.17.3`), simplifying setup with one implicit per-subnet `wasm_store` on a fixed 40 MB / 4 MB IC preset, and refreshing the workspace baseline to Rust `1.94.1` with `ctor 0.8` and `sha2 0.11` while keeping `root` as the only deployment, manifest, selection, and orchestration authority.

```toml
[subnets.prime]
auto_create = ["app", "user_hub", "scale_hub", "shard_hub"]

[subnets.prime.canisters.app]
kind = "singleton"
```

### đź”§ Changed

- Introduced `wasm_store` as the canonical storage/distribution canister for deployable template releases, with root-gated publication, deterministic chunk metadata, deterministic chunk lookup, and subnet-local store placement resolved from root-owned runtime state instead of user-authored store topology.
- Made approved manifest resolution the authoritative install and upgrade contract: publication now uploads chunked releases into the selected store canisterâ€™s IC chunk store, workflow verifies the manifest-bound store and chunk metadata, and `root` invokes `install_chunked_code` without acting as a full-wasm relay.
- Moved ordinary child payload ownership out of `root` and into the canonical store layer, keeping only the irreducible bootstrap `wasm_store` payload embedded in `root` while ordinary child releases are seeded and republished from `wasm_store`. Ordinary roles no longer have a generic inline install fallback.
- Replaced user-configured store topology and capacity policy with one implicit mandatory `wasm_store` per subnet, using a fixed IC-tuned preset of `40000000` max bytes and `4000000` headroom bytes so installation stays zero-config on the store dimension.
- Added explicit store lifecycle handling for rollover and retirement without legacy fallback: root-owned active/detached/retired publication binding state, store-local GC phases (`Prepared`, `InProgress`, `Complete`), explicit retired-store wipe, explicit retired-store finalization, and explicit finalized-store deletion only after GC completion and empty-store verification.
- Refreshed the workspace toolchain and dependency baseline by pinning Rust to `1.94.1`, bumping `ctor` from `0.6` to `0.8`, and bumping `sha2` from `0.10` to `0.11` so local bootstrap, CI, and downstream builds use the same current compiler and crate floor.

### đź©ą Fixed

- Removed a regressive finalized-history experiment that pushed root-only lifecycle visibility into shared publication DTO/storage state. Finalization visibility is now root-local only, so the fleet floor stays flat while operators still get current publication/finalization/deletion status.
- Replaced an overbuilt bounded lifecycle-history attempt with a smaller root-local last-deleted snapshot, preserving operator visibility without paying a large `root` wasm penalty for generalized event history.
