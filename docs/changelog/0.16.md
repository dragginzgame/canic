# 0.16.x Detailed Changelog

Technical release notes for the `0.16.x` minor line.

## 0.16.2 - 2026-03-25 - Delegated Auth Lifetime Hardening

### 📝 Summary

- `0.16.2` hardens delegated-auth token handling by rejecting malformed or unusable lifetimes at both issuance and verification, making the clock-skew policy explicit, restoring ops-owned proof conversion boundaries, and closing the `0.16` auth/proof line with remaining root/template architecture work handed off to `0.17` and `0.18`.

### 🔧 Changed

- Centralized delegated-token lifetime validation so issuance and verification both apply the same invariant checks before deeper auth work proceeds.
- Made the current zero-skew policy explicit in auth code with a dedicated constant instead of leaving it as implicit timestamp behavior.
- Restored DTO-to-storage proof conversion ownership to `ops::storage::auth`, so API and workflow code no longer reach into storage-owned proof record types.

### 🩹 Fixed

- Verification now rejects `exp < iat` claims before any derived TTL math, so malformed delegated tokens return the correct typed error instead of underflowing or being misclassified as a TTL overflow.
- Signing now rejects expired, future-dated, or internally inconsistent claim lifetimes, preventing canisters from minting tokens that are guaranteed to fail during later verification.
- Proof persistence APIs now expose DTO-facing ops methods instead of storage record helpers, removing the auth-layer coupling that made proof-schema refactors ripple into API and workflow code.

### 🥾 Governance

- Closed the `0.16` delegation-proof evolution line after the auth/proof follow-up audit items were resolved and the full repo validation set was recorded.
- Carried forward long-term root payload ownership and template-canister architecture work into:
  - `docs/design/0.17-wasm-audit/0.17-design.md`
  - `docs/design/0.18-canister-templates/0.18-design.md`

## 0.16.1 - 2026-03-24 - Audience Binding and Typed Auth Metrics

### 📝 Summary

- `0.16.1` tightens delegated-auth audience handling so proof installation and session bootstrap only accept audience claims that are explicitly covered by the delegation proof, while also making rollout metrics safer to evolve during the ongoing `0.16` auth work.

### 🔧 Changed

- Split the `DelegationApi` auth slice into focused `admin`, `proof_store`, and `session` modules so the `0.16` auth work can keep moving without growing one large control-hub file.
- Centralized shared audience rules in `ops::auth::audience`, so verifier install paths, runtime token verification, and delegated-session bootstrap all use the same subset and self-audience checks.
- Replaced string-coupled auth rollout classification with typed auth metric predicates and rollout signals, reducing the risk that producer/query drift silently hides rollout failures.

### 🩹 Fixed

- Explicit verifier prewarm/repair pushes now reject verifier targets that are not present in the proof audience instead of allowing an invalid fanout plan to proceed.
- Root-driven verifier proof install now rejects storage on a local verifier when that verifier is not included in the proof audience, and records a dedicated `target_not_in_audience` rollout failure.
- Delegated-session bootstrap now rejects tokens whose claim audience is empty or exceeds the proof audience, preventing externally supplied bootstrap tokens from widening verifier scope.

### 📊 Audit

- Recorded the `2026-03-24` `complexity-accretion` reruns for the targeted `0.16` auth slice; the reports found moderate structural pressure but no audience-binding correctness break.
- The rerun outcome fixes the recommended order for follow-up work: extract shared invariants first, then decompose modules, while keeping typed rollout metrics in place before the next auth feature slice.
