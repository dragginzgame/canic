# 0.24.x Detailed Changelog

Technical release notes for the `0.24.x` minor line.

## 0.24.2 - 2026-04-05 - Root Capability Replay Follow-Through

### 📝 Summary

- `0.24.2` follows the first root auth reductions by reusing cached root response attestations, carrying cycles authorization through replay/capability execution, trimming replay and registry response work, and keeping only the retained final instruction audit output.

### 🔧 Changed

- Root now reuses still-valid outbound response attestations for repeated RPCs to the same audience instead of reissuing the same signed role attestation on every call.
- Root and non-root `RequestCycles` replay/capability execution now carry an already-authorized grant from preflight into execution, while root keeps resolving children from the authoritative subnet registry and non-root parents keep using their local direct-child cache.
- Replay caches now store canonical response bytes directly and decode them through `canic_memory::serialize`, so replay hits no longer pay an extra Candid decode/encode layer just to rebuild the cached response shape.
- Subnet registry responses now iterate the stable registry directly instead of exporting a whole snapshot first, which trims the read path and keeps the response mapper focused on entry projection.
- In the retained instruction audit, those follow-through changes cut sampled `root::canic_request_delegation` from `3205866` local instructions in `instruction-footprint-20` to `2281850` in `instruction-footprint-29`.
- The same retained `instruction-footprint-29` run records `prepare_call` at `299578`, `decode_response` at `254877`, `cache_root_verifier_keys` at `354872`, `preflight` at `116649`, and `execute_capability` at `256322`, so both the delegated-auth and request-cycles paths now have stable stage-level numbers.
- Removed the redundant intermediate `instruction-footprint-19` through `instruction-footprint-28` generated outputs and the stale `tmp-debug` audit artifact so the audit surface keeps one retained final report instead of a confusing same-snapshot run pile.

## 0.24.1 - 2026-04-04 - Root Auth Key Prewarm

### 📝 Summary

- `0.24.1` follows up the first delegation-path replay/storage reductions by cutting cold root auth work, removing a redundant signer fanout, and tightening the root-local verifier cache path, so the first live delegated-auth request is materially cheaper without changing behavior.

### 🔧 Changed

- Warmed the root delegation and attestation key caches during the reference root canister's `canic_setup` hook through the public auth facade, so the cold root key fetch cost moves out of the first live delegated-auth request path without changing runtime behavior.
- In the current instruction audit, that drops sampled `root::canic_request_delegation` from `4356980` local instructions in `instruction-footprint-17` to `3934327` in `instruction-footprint-18`.
- In the same path, `issue_proof` drops from `882945` to `504032`.
- Stopped root from pushing the newly issued delegation proof back to the requesting signer shard, and instead let the shard store the returned proof locally through the existing delegated-auth setup path.
- In the current instruction audit, that drops sampled `root::canic_request_delegation` again from `3934327` in `instruction-footprint-18` to `3247900` in `instruction-footprint-19`, while cutting `prepare_call` from `645625` to `297555` and `decode_response` from `513391` to `256172`.
- Collapsed the root-local verifier cache path from two `DelegationState` writes into one combined auth-state commit, so a fresh delegated-auth issuance no longer rewrites the same stable auth record twice in succession just to store the shard key and verifier proof.
- In the current instruction audit, that trims sampled `root::canic_request_delegation` one more time from `3247900` in `instruction-footprint-19` to `3205866` in `instruction-footprint-20`, with `cache_root_verifier_keys` dropping from `404812` to `357119`.

## 0.24.0 - 2026-04-04 - Shared Runtime Reduction and Test Boundary Cleanup

### 📝 Summary

- `0.24.0` opens the new minor line after the accidental version bump without changing the current engineering direction: the next work still centers on shrinking the shared wasm floor, keeping public `canic-testkit` support separate from Canic-only self-test harness code, and hardening release-facing surfaces such as the curlable installer path.

### 🔧 Changed

- [CHANGELOG.md](/home/adam/projects/canic/CHANGELOG.md) now opens the `0.24.x` minor line.
- `0.24.x` is the new home for the current follow-through work rather than a broad reset or topology redesign.
- Switched shared `CandidType` source comments from `///` to `//` across the shared DTO and spec files, so Candid runtime docs stop inflating release wasm without changing the exported endpoint surface.
- In the current `minimal` release artifact, that reduces the shrunk wasm from `1764427` to `1741732` bytes and the `.wasm.gz` artifact from `498796` to `486798` bytes.
- Expanded the recurring instruction audit so it now measures real scaling, sharding, replay/cycles, delegated-auth issuance, and verifier-confirmation update paths alongside the local-only query perf probes, with stage-level checkpoint deltas captured for the sampled update flows where internal checkpoints exist.
- Reworked staged chunk storage so chunk bytes now live in a stable payload vector behind a small key-to-slot index instead of sitting directly in the stable chunk BTreeMap value path.
- In the current instruction audit, that drops the sampled `root::canic_template_publish_chunk_admin` single-chunk path from `9713784` local instructions in `instruction-footprint-12` to `390788` in `instruction-footprint-13`, with the old `chunk_store_insert` stage falling from `9518067` to `194978`.
- Replaced replay-store `RootReplayRecord` CBOR persistence with a direct byte layout and swapped replay payload hashing from canonical Candid encode to a structured field hash, so the sampled `root::canic_request_delegation` path drops from `5516827` local instructions in `instruction-footprint-15` to `4356980` in `instruction-footprint-17`.
- In the same delegation path, `commit_replay` drops from `1151412` to `993079` local instructions and `prepare_replay_input` drops from `779044` to `137129`, which means replay prep is no longer the dominant stage inside delegated auth issuance.

### 🧭 Migration Notes

- This minor bump does not signal a new API model by itself.
- The practical focus remains shared-runtime wasm reduction, public/internal testing-boundary cleanup, and release-surface correctness.
