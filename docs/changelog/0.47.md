# 0.47 Changelog

## 0.47.4 - 2026-05-27 - archived root-evidence hardening

### Changed

- Root-verification report validation now rejects malformed SHA-256 digest
  fields before archived reports can pass self-consistency checks.
- Root-verification receipt validation now rejects malformed SHA-256 digest
  fields before archived receipts can pass self-consistency checks.
- Root verification now rejects unsupported source `DeploymentCheckV1` schema
  versions before treating source-check evidence as usable.
- Root verification now recomputes the source check diff from its embedded plan
  and inventory, rejecting stale or tampered `DeploymentCheckV1` artifacts whose
  diff no longer matches.
- Root verification now recomputes the source check safety report from its
  diff, rejecting stale or tampered `DeploymentCheckV1` artifacts whose report
  no longer matches.
- Root-verification report validation now rejects duplicate or unexpected
  identity/evidence check rows so archived root evidence remains schema-stable.

### Validation

- Added tests proving malformed report and receipt digests are rejected.
- Added tests proving forged root-verification check rows are rejected.
- Added tests proving unsupported source-check schemas, stale source-check
  diffs, and stale source-check reports fail root verification.
- Added tests proving duplicate or unexpected root-verification check rows fail
  validation.

## 0.47.3 - 2026-05-27 - receipt transition validation

### Changed

- Root-verification receipt validation now requires local-state digest movement
  to match the claimed state transition:
  - `PromotedNotVerifiedToVerified` must change the local-state digest.
  - `NoStateChange` must preserve the same local-state digest.
- `NotAttempted`, `Blocked`, and `WouldPromoteNotVerifiedToVerified` remain
  invalid transition values for successful root-verification receipts.

### Validation

- Added deployment-truth tests proving no-op receipts with changed local state
  are rejected.
- Added deployment-truth tests proving promotion receipts without a local-state
  digest change are rejected.
- Added JSON round-trip and schema-shape coverage for

## 0.47.2 - 2026-05-27 - root verification no-op hardening

### Changed

- Re-verifying an already verified deployment root with matching
  deployment-truth evidence now emits a `NoStateChange`
  `DeploymentRootVerificationReceiptV1` without rewriting local deployment
  state or updating `updated_at_unix_secs`.
- Root verification now re-reads the final local-state digest for no-op
  receipts so the receipt reflects the final observed state file.

- Verified root replacement attempts remain blocked: presenting a different
  observed root in source-check evidence cannot replace an already verified
  registered root.
- Added source-guard coverage proving verified root-state writes remain limited

## 0.47.1 - 2026-05-27 - receipt-backed root verification

### Added

- Added `DeploymentRootVerificationReceiptV1` as the mutating evidence artifact
  for explicit deployment-root verification. The receipt records deployment
  target identity, fleet template, root principal, source report/check/plan/
  inventory digests, local state digests before and after the write, and the
  root-verification state transition.
- Added receipt digest and validation logic to the deployment-truth root module
  so root-verification receipts are self-consistent public artifacts instead of
  install-local structures.
- Added explicit root verification for registered deployment targets.

```bash
canic deploy root verify demo-local --from-check deployment-check.json
```

```bash
canic deploy root verify demo-local --from-check deployment-check.json --format text
```

### Changed

- `canic deploy root verify` now reads an existing deployment-target
  registration, builds a deployment-root verification report from the supplied
  `DeploymentCheckV1`, validates the report, writes
  `root_verification = verified` only after a local-state digest
  compare-and-swap, and emits a validated
  `DeploymentRootVerificationReceiptV1`.
- Root verification remains local-state scoped: it does not install code,
  mutate canisters, mutate controller state, or claim whole-deployment safety.
- The CLI root namespace now separates passive inspection from the explicit
  mutating local-state transition:
  - `root inspect` renders report evidence without writing state.
  - `root verify` writes verified root state only when source evidence matches.

### Validation

- Added host tests proving registered unverified roots promote only with
  matching deployment-truth evidence, local-state-only evidence is rejected,
  stale local-state digests fail closed, and emitted receipts validate.
- Added deployment-truth tests for root-verification receipt digest stability
  and state-transition validation.
- Added CLI parser, dispatch, help, and source-guard coverage for
  `canic deploy root verify`.

## 0.47.0 - 2026-05-27 - passive root verification evidence

### Added

- Added explicit `DeploymentRootObservationV1` evidence to
  `DeploymentInventoryV1`, including deployment target, network, fleet
  template, root principal, observed canister ID, observation source,
  controller facts, module hash, status, and role-assignment source.
- Added passive `DeploymentRootVerificationRequestV1` and
  `DeploymentRootVerificationReportV1` artifacts. These reports can produce
  `EvidenceSatisfied` when source-check evidence is digest-bound and complete,
  but they do not write verified local state.
- Added validation for root-verification report digest stability, required
  source check/plan/inventory IDs, explicit observed-root evidence, and the
  exact `unverified_deployment_root` hard-failure exception.
- Added read-only CLI inspection for passive root-verification reports.

```bash
canic deploy root inspect --request root-verification.json
```

```bash
canic deploy root inspect --request root-verification.json --format text
```

### Changed

- Local deployment inventory now records the deployment target name from the
  request instead of reusing the fleet-template name.
- Deployment-truth registry observations now resolve installed deployment state
  by deployment target name, keeping fleet templates and live deployment
  targets separate at the root-evidence boundary.
- Root-verification reports treat `LocalDeploymentState` root observations as
  insufficient evidence; root verification must bind to explicit
  deployment-truth evidence instead of local-state echo.
- The passive report allows only the exact single
  `unverified_deployment_root` blocker for
  `local_state.unverified_root_canister_id`; unrelated hard failures keep the
  evidence status blocked.

### CLI

- `canic deploy root inspect --request <file>` reads a
  `DeploymentRootVerificationRequestV1` JSON file and prints a
  `DeploymentRootVerificationReportV1` JSON artifact by default.
- `--format text` renders host-owned passive text that calls out
  `mode: passive`, `execution: none`, and `local_state_write: none`.
- The root inspect path has source-guard coverage proving it does not call
  install, register, dfx, or controller-mutation primitives.

### Validation

- Added host tests for bound root evidence, missing explicit root evidence,
  local-state-only evidence, exact unverified-root blocker handling,
  unrelated hard failures, and digest drift.
- Added CLI tests for root inspect parsing, help wording, command dispatch,
  passive source guards, and host report delegation.

### Testing

- `cargo fmt --all`
- `cargo test -p canic-host root_verification --lib --locked`
- `cargo test -p canic-cli deploy --lib --locked`
