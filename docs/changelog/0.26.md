## 0.26.5 - 2026-04-06 - Delegation Verifier Bootstrap Fix

### 📝 Summary

- `0.26.5` finishes the delegated-auth verifier bootstrap contract by having root push the root delegation public key together with each verifier proof install, so verifier-only canisters can stay off `auth-crypto` for delegation provisioning instead of trying to fetch that key themselves through threshold-ECDSA management calls.

### 🩹 Fixed

- Verifier proof installation now carries `root_public_key_sec1` in the canonical proof-install request, which closes the remaining bootstrap gap after `0.26.3`: verifier canisters no longer need to call `ecdsa_public_key` just to validate the first pushed delegation proof.
- Root-driven delegation provisioning and verifier prewarm/repair now prime both the root delegation public key and the shard key in the pushed install payload, so the verifier-side proof store can complete locally without management-key lookups.

### 🔧 Changed

- Added a small local root-key helper in the delegation ops path and threaded the root public key through the root request-handler and delegation workflow fanout, keeping the fix in the existing canonical proof-install flow instead of adding a second bootstrap endpoint.

## 0.26.4 - 2026-04-06 - Runtime and Testkit Ownership Cleanup

### 📝 Summary

- `0.26.4` keeps the late `0.26` line focused on cleaner ownership instead of more runtime cleverness: shared PocketIC baseline state is less field-bag-shaped, runtime lifecycle ownership is split more explicitly, normal bootstrap builds stop rewriting the checked-in `wasm_store.did` by accident, and the workspace test runner now points the moved PocketIC suites at the right package.

### 🔧 Changed

- Narrowed `CachedPicBaseline` in `canic-testkit` behind explicit accessors, so shared PocketIC baseline lifecycle stays owned by the baseline module instead of leaking raw `pic`, `snapshots`, and metadata fields outward.
- Split more `canic-testkit::pic` and runtime ownership seams into dedicated modules, including PocketIC error/readiness helpers and root vs non-root runtime lifecycle handling, so those public barrels carry less inline implementation.
- Split attestation fixture lifecycle from attestation build/setup plumbing, so the internal role-attestation PocketIC seam is easier to follow and less mixed by concern.
- Ordinary `wasm_store` bootstrap builds now treat the checked-in `crates/canic-wasm-store/wasm_store.did` file as canonical input and only refresh it explicitly under `CANIC_REFRESH_WASM_STORE_DID=1`, so unrelated local builds stop churning that source-controlled contract artifact.

### 🩹 Fixed

- Updated the workspace test runner to execute `pic_intent_race`, `pic_sharding_bootstrap`, and `pic_role_attestation` from the `canic-tests` package, matching where those PocketIC suites live now instead of the stale `canic-core` targets.

### 📚 Documentation

- Clarified the canonical `wasm_store.did` ownership/refresh flow in the workspace and `canic-wasm-store` READMEs so explicit contract refresh is documented where developers actually build and bootstrap the store.

## 0.26.3 - 2026-04-06 - Delegated Auth Build Contract

### 📝 Summary

- `0.26.3` closes the delegated-auth config/build contract gap: if `canic.toml` says delegated auth is in play, Canic now fails fast when the root or signer build is missing the threshold-ECDSA support needed to honor that config.

### 🩹 Fixed

- Root runtime startup now traps immediately when delegated auth is configured anywhere in `canic.toml` but the root build does not include `auth-crypto`, instead of letting verifier canisters discover the mismatch later through repeated attestation-key refresh warnings.
- Non-root delegated signer canisters now trap immediately when `delegated_auth.signer = true` but the canister was built without threshold-ECDSA management support, so signer roles cannot boot into a miscompiled state.
- Verifier-only canisters remain verifier-only: `delegated_auth.verifier = true` still does not require `auth-crypto` on that canister build, because local signature verification continues to use the non-management `k256` path.

### 🔧 Changed

- Added one explicit runtime capability probe in `EcdsaOps` and used it during root and non-root runtime initialization, so config intent and compiled crypto support are checked in one place instead of being inferred from later auth warnings.

### 🧪 Testing

- `cargo fmt --all`

## 0.26.2 - 2026-04-06 - Root Runtime Cleanup Follow-Through

### 📝 Summary

- `0.26.2` keeps the first `0.26` runtime pass focused on cleaner shared root paths instead of transport tricks, with measured follow-through that lowers the retained root cycles and delegation hotspots from the `0.26.0` baseline.

### 🔧 Changed

- Routed replay-first root cycles requests through the dedicated cycles workflow instead of paying the generic root-capability dispatcher path first, so the root `RequestCycles` lane now uses the code that already owns cycles replay, authorization, and execution.
- Simplified root handler ownership by carrying root context directly through delegation validation and non-root cycles planning, removing a few repeated environment and lookup steps from the hot path without changing endpoint behavior.
- Tightened verifier proof-cache writes in the stable auth store so cache upserts stop rescanning the whole proof vector just to rebuild stats after every write, while keeping the cache semantics and eviction policy unchanged.

### 📊 Audit

- The retained root cycles hotspot in [instruction-footprint-10.md](../audits/reports/2026-04/2026-04-06/instruction-footprint-10.md) dropped from the `0.26.0` baseline `596032` to `492195` local instructions after the replay-first root cycles routing cleanup.
- The later same-day retained rerun in [instruction-footprint-18.md](../audits/reports/2026-04/2026-04-06/instruction-footprint-18.md) keeps the shared delegation hotspot below the `0.26.0` baseline, moving `root::canic_request_delegation` from `1728271` to `1682331` local instructions while `cache_root_verifier_proof` drops under `100k`.

### 🧹 Cleanup

- Kept the `0.26` runtime line on the clean side by dropping the trial `.with_cycles(0)` transport tweak once the audit showed it regressed totals, so the retained changes stay aligned with the “simpler first” rule.
- Stopped the pass before pushing deeper into signing or Candid transport complexity, since the remaining large stages are mostly real `sign_cert` and proof-install work rather than obvious waste.
