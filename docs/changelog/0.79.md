# 0.79 Changelog

## 0.79.2 - 2026-07-02 - Deploy-plan preview labels

### 📝 Summary

- Expands `canic deploy plan` future-apply preview labels for plan elements
  that were already present in `DeploymentPlanV1` but not visible in the report
  preview.

- Keeps the labels diagnostic-only: the report still does not create apply
  operation objects, dependency graphs, receipts, locks, deployment truth, or
  canister mutations.

### 🔧 Changed

- Expected pool identities without a known canister id now emit non-executed
  `create_canister` preview labels using the stable
  `<pool>:<role>` subject shape, such as `user_shards:user_shard`.

- Desired authority profiles with configured deployment controllers now emit a
  single deployment-scoped `set_controllers` preview label.

- Focused report tests now cover both pool-aware and controller-aware preview
  labels while continuing to assert the labels remain
  `future_apply_preview` / `not_executed`.

### 🧪 Testing

Validated with:

```bash
cargo fmt --all -- --check
cargo test --locked -p canic-cli deploy::tests::plan
```

## 0.79.1 - 2026-07-02 - Deploy-plan report facts

### 📝 Summary

- Tightens `canic deploy plan` report facts without expanding the command
  surface, adding live observation, writing deployment truth, or introducing
  apply semantics.

- Keeps the report as a wrapper around the embedded `DeploymentPlanV1`: the new
  facts expose values already computed by the host planner instead of
  introducing a second plan schema or recomputing plan identity.

### 🔧 Changed

- `verified_facts` now surfaces additional deterministic plan facts when the
  corresponding inputs are actually resolved:
  - canonical runtime config digest;
  - authority profile hash;
  - role topology hash;
  - pool identity set digest;
  - deployment manifest digest;
  - artifact set digest;
  - per-role observed wasm artifact digests.

- Partial desired-state inputs no longer overclaim resolved facts. For example,
  `artifact_set_resolved` is omitted while local artifact gaps remain, and
  authority/topology facts are omitted when their owning config section is
  unresolved.

- JSON report contract coverage now asserts stable top-level field order for
  `DeploymentPlanReport`, including the diagnostic arrays and future-apply
  preview labels.

### 🧪 Testing

Validated with:

```bash
cargo fmt --all -- --check
cargo test --locked -p canic-cli deploy::tests::plan
cargo clippy --locked -p canic-cli --all-targets -- -D warnings
cargo test --locked -p canic-cli deploy
git diff --check
```

## 0.79.0 - 2026-07-02 - Declarative deployment plan

### 📝 Summary

- Adds the first `canic deploy plan <deployment>` surface for deterministic,
  no-mutation deployment planning.

- Keeps 0.79 deliberately narrow: the command explains Canic's desired
  deployment shape and available local evidence without applying changes,
  creating deployment truth, signing evidence, acquiring locks, writing
  installed deployment records, or querying live IC state.

### ➕ Added

- New `canic deploy plan <deployment>` command that builds a
  `DeploymentPlanReport` from local project config and embeds the existing
  `DeploymentPlanV1` rather than introducing a parallel plan schema.

- Text output and `--json` output for the planning report. JSON emits
  `schema_version = 1` and includes `command`, `target`, `network`,
  `build_profile`, `config_path`, `status`, `comparison_status`, `plan`,
  `blockers`, `warnings`, `assumptions`, `verified_facts`,
  `proposed_operations`, and `next_actions`.

- Safe `--out <path>` support that writes JSON only, fails when the output path
  already exists, and fails when the parent directory does not exist.

- Report diagnostics for:
  - resolved deployment targets;
  - invalid deployment target names;
  - malformed desired config;
  - missing installed deployment state;
  - verified installed root canister facts;
  - unverified installed root blockers;
  - installed-state network drift.

- Deterministic `comparison_status` values for already-available installed
  deployment state: `compared`, `compared_with_warnings`,
  `compared_with_drift`, `not_available`, and `not_requested`.

- Future-apply preview labels that remain explanatory only:
  `create_canister`, `install_wasm`, `upgrade_wasm`, and `verify_topology`.
  These are not apply operation objects and do not carry operation IDs,
  dependency graphs, locks, receipts, rollback actions, or idempotency keys.

### 🔧 Changed

- `canic deploy` now exposes `plan` as the canonical planning subcommand while
  preserving existing deployment-truth, inspect, register, root, promote,
  external, authority, and install surfaces.

- Deployment medic missing-target guidance may point operators at
  `canic deploy plan <deployment>` as a next action, but medic does not execute
  the planner internally.

- Blocked deploy-plan reports return exit code `1` without adding a duplicate
  top-level stderr line after the report has already rendered. Usage and output
  write failures still render diagnostics and return exit code `2`.

### ⚠️ Breaking

- Hard-cut rejected forms are intentionally not supported:

```text
canic plan <deployment>
canic deploy plan
canic deploy plan --deployment <deployment>
canic deploy diff <deployment>
canic apply <deployment>
canic apply <deployment> --plan deployment-plan.json
canic deploy plan <deployment> --apply
canic deploy plan <deployment> --write-truth
canic deploy plan <deployment> --evidence
canic deploy plan <deployment> --format json
canic deploy plan <deployment> --from-check deployment-check.json
canic deploy plan <deployment> --observe-local
canic deploy plan <deployment> --out deployment-plan.json --force
```

### 📚 Documentation

- Refined the 0.79 design into an implementation specification with an explicit
  non-mutation contract, output stream rules, deterministic ordering rules,
  `--out` behavior, status derivation, comparison-status derivation, accepted
  and rejected command forms, exit codes, and acceptance criteria.

- Updated the current-status handoff to make 0.79 the active release line and
  summarize the implemented deploy-plan behavior.
