# 0.13.x Detailed Changelog

Technical release notes for the `0.13.x` minor line.

## 0.13.0 - 2026-03-07 - Distributed Capability Invocation

### 📝 Summary

- `0.13.0` introduces signed capability requests between canisters, so root calls are verified instead of trusted by transport alone.
- It combines earlier replay safety and role-attestation work into one shared request pipeline.

### ➕ Added

- Added a service-scoped capability envelope model with explicit `service`, `capability_version`, `capability`, `proof`, and replay `metadata`.
- Added proof-model separation for `Structural`, `RoleAttestation`, and `DelegatedGrant` flows.
- Added capability-hash binding rules so proofs cannot be replayed against modified capability payloads.
- Added standardized replay keying and digest rules for mutating root capability calls, including 0.11 compatibility behavior.
- Added root-facing `canic_response_capability_v1` and `RpcApi::response_capability_v1(...)` as a compatibility endpoint for versioned envelope rollout.

### 🔧 Changed

- The capability pipeline is now explicit and ordered as `proof -> replay -> policy -> execute`.
- Capability routing is now clearly service-owned, so policy stays local and easier to reason about.
- Root 0.11 capability dispatch remains supported as a compatibility profile during rollout.
- Cycle tracking now defaults to a 60-minute cadence (from 10 minutes) to reduce timer churn.
- Non-root root-RPC callers now build and send versioned root capability envelopes with role-attestation proof and capability-hash binding.
- Root capability endpoint now validates envelope `issued_at`/`ttl_seconds` with skew checks and projects `(request_id, nonce)` into replay metadata before dispatch.
- Root capability endpoint now verifies `Structural` proofs for local topology predicates and verifies delegated-grant proofs (hash, scope, time window, and signature) before policy/execution.
- Root capability endpoint now uses explicit fail-closed header validation for `service`, `capability_version`, and non-structural `proof_version` dispatch.
- Root capability metrics now include envelope/proof-stage outcomes, and endpoint logs now emit structured verification rejections with capability/proof mode context.

### 🗑️ Removed

- Removed legacy root `canic_response` endpoint wiring so generic root dispatch no longer has a caller/topology-only path.

### 🧭 Migration Notes

- Start with one pilot service before broad migration (for example registry-style service endpoints).
- Keep root and non-migrated endpoints operational while capability envelopes are adopted incrementally.
- Preserve cryptographic domain separation between attestations and delegated grants.

### 🧪 Testing

- Added unit coverage for version dispatch and hash-binding checks, including delegated-grant hash-consistency failures.
- Added PocketIC coverage for `canic_response_capability_v1` role-attestation flows (success, audience-mismatch rejection, and expiry rejection).
- Added deny-by-default capability policy coverage for `IssueRoleAttestation` (subject mismatch and missing audience).
- Added epoch-floor and tampered-signature proof-failure coverage for role-attestation verification paths.
- Added delegated-grant unit coverage for scoped-authority checks (`issuer`, `subject`, `audience`, `capability_family`) and domain-separated signature validation (`CANIC_DELEGATED_GRANT_V1`).
- Added fail-closed PocketIC coverage for delegated-grant scope mismatch and `capability_hash` mismatch rejection on `canic_response_capability_v1`.

```text
request envelope -> proof verify -> replay verify -> policy decision -> execution
```

## 0.13.1 - 2026-03-07 - Control-Plane Decomposition

### 📝 Summary

- `0.13.1` breaks very large control-plane files into smaller modules by responsibility.
- This is a structure-only patch: request formats, policy rules, and verification order stay the same.

### ➕ Added

- Added dedicated module boundaries for control-plane ownership:
  - `api/rpc/` + `api/rpc/capability/*`
  - `workflow/rpc/request/handler/*`
  - `ops/auth/*` and `ops/rpc/request/*`
- Added focused helper modules for request metadata/proof/replay validation and auth signing/verification/key-cache helpers.

### 🔧 Changed

- `api/rpc` is now a thin transport/error boundary with capability envelope/proof/replay logic extracted into `api/rpc/capability/*`.
- `workflow/rpc/request/handler` is now decomposed by responsibility (`authorize`, `replay`, `execute`, `capability`, `delegation`) with orchestration retained in `mod.rs`.
- `ops/auth` is now decomposed by responsibility (`delegation`, `token`, `attestation`, `verify`, `keys`, `crypto`, `error`, `types`) with a thin facade.
- `ops/storage/children` now follows directory-module structure (`children/mod.rs`) to align with contributor module-layout rules.

### 📊 Audit

- Hub-pressure rerun snapshot (2026-03-07, post-decomposition):
  - `workflow/rpc/request/handler/mod.rs`: `218` LOC (from `1581`)
  - `ops/auth/mod.rs`: `76` LOC (from `1253`)
  - `api/rpc/mod.rs`: `62` LOC (from `900`)
- Recurring audit rerun indices:
  - Complexity Risk Index: `5.36/10` (from `6.55/10`)
  - Velocity Risk Index: `4.50/10` (from `5.80/10`)

### 🧭 Migration Notes

- No external API contract change.
- Existing capability envelope and delegated-auth behavior remain unchanged.
- Follow-up work can now target decision-surface reduction per module without first touching control-plane topology.

## 0.13.5 - 2026-03-08 - Decision Surface and Metrics Axis Reduction

### 📝 Summary

- `0.13.5` continues the control-plane simplification by reducing large branching surfaces in request handling, access predicates, and root capability metrics.
- Internal behavior is easier to evolve safely because replay commit ownership and metrics ownership are now clearer by layer.

### ⚠️ Breaking

- `RootCapabilityMetricEntry` now uses `event_type`, `outcome`, and `proof_mode` fields instead of the old single `event` field.

### 🔧 Changed

- Moved replay commit mechanics fully behind ops via `ops::replay::commit_root_replay(...)`, so workflow replay code remains control-flow oriented and no longer handles replay record encoding/writing details.
- Reduced `Request` decision fanout by pushing variant construction/access into DTO helpers and removing runtime non-dto variant branching for request dispatch paths.
- Refactored `BuiltinPredicate` evaluation to evaluator-based dispatch so the enum is primarily a construction/selection surface instead of a central evaluation hotspot.
- Replaced monolithic `RootCapabilityMetricEvent` usage with dimensional metrics: `event_type` (pipeline stage), `outcome` (result), and `proof_mode` (when relevant), including proof-mode tagging for API envelope/proof checks.

### 📊 Audit

- Rerun indices after this pass:

- Notable surface reductions:
  - `Request` decision sites: `94 -> 18`
  - `BuiltinPredicate` decision/evaluation sites: `30 -> 16`
  - Monolithic `RootCapabilityMetricEvent` usage: removed in favor of axis enums

## 0.13.8 - 2026-03-09 - Top-Up Validation Hardening and Docs Restructure

### 📝 Summary

- `0.13.8` strengthens top-up safety defaults and validation coverage while restructuring documentation for clearer long-term maintenance.
- It also introduces the `0.14` subtree-cycles design/status documentation with parent-only funding and unified app-level runtime controls.

### 🩹 Fixed

- Config validation now rejects unsafe top-up settings where `topup.amount` is at least half of `topup.threshold`, preventing aggressive refill configurations that can amplify cascade pressure.
- Default top-up configuration was adjusted to satisfy the stricter safety invariant by default.

### 🔧 Changed

- Audit report storage is now organized under month/day directories (`YYYY-MM/YYYY-MM-DD`) with required monthly summary files.
- Audit process guidance now explicitly documents monthly summaries and normalized report layout expectations.
- Design documentation has been restructured into version-scoped directories for consistency with active architecture work.

### 📚 Documentation

- Added `0.14` design and status docs for subtree cycles funding with these core invariants:
  - parent-only funding source for all non-root top-ups
  - no compatibility mode or shim path for root fallback behavior
  - replay handled by the shared RPC replay guard (no separate funding replay table)
  - global cycles funding kill switch in app runtime state, controlled via `canic_app` variant updates and cascaded across canisters

### 🧪 Testing

- Added focused config-schema tests for top-up threshold/amount invariants, including rejection of invalid ratios and validation of updated defaults.
