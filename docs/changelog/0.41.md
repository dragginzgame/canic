# Canic 0.41

## 0.41.5 - 2026-05-22 - design realignment and live root inventory

### Added

- Started live deployment inventory expansion for installed roots. When local
  install state identifies a root canister, deployment truth now attempts a
  read-only `icp canister status --json` observation and records live
  controllers, module hash, and status when available.
- Live root status observation failures are represented as typed
  `live_canister_status.root` observation gaps instead of becoming installer
  errors.
- Added installed module-hash comparison to `DeploymentDiffV1.module_hash_diff`
  so a planned role module hash and observed live root module hash must agree.
- Added `DeploymentExecutionStatusV1`, `RolePhaseReceiptV1`, and
  `RolePhaseResultV1` so `DeploymentReceiptV1` can model partial execution and
  role-scoped outcomes as the installer/executor matures.

### Changed

- Corrected the 0.41 deployment truth model so raw local config SHA-256 values
  are raw diagnostic/local-consistency evidence, not canonical deployment
  manifest identity.
- `deployment_manifest_digest` now remains reserved for the future canonical
  deployment manifest digest instead of being populated from raw config bytes.
- Raw plan-vs-inventory config drift still blocks, but now through a distinct
  `raw_config_digest_mismatch` finding and `raw_config_sha256` diff category.
- Current-install receipts populate the new operation-status and
  role-phase-receipt fields conservatively; richer per-role results remain
  future execution work.

## 0.41.4 - 2026-05-22 - deployment truth receipts and stricter local checks

### Added

- Added lightweight deployment truth receipt helpers for the current-install
  artifact materialization gate. The current installer now constructs a
  `materialize_artifacts` phase receipt from the live deployment truth check,
  but the gate still makes decisions from check evidence rather than receipt
  trust.
- Added `canic deploy diff <fleet>` and `canic deploy report <fleet>` as
  read-only JSON views for the normalized `DeploymentDiffV1` and
  `SafetyReportV1` without requiring operators to parse the full
  `DeploymentCheckV1`.
- Added local deployment config SHA-256 evidence to `DeploymentPlanV1` and
  `DeploymentInventoryV1`.

### Changed

- `canic deploy check <fleet>` now returns a failing exit status when the
  generated `SafetyReportV1` is blocked, while still printing the full
  `DeploymentCheckV1` JSON for automation.
- The normalized deployment diff now blocks when the observed local deployment
  manifest digest disagrees with the plan.
- Local artifact comparison now blocks when the plan and inventory both observe
  a `.wasm.gz` file digest for the same role and those digests disagree.
- The 0.41 design/status notes now clarify that the shipped
  `canic deploy diff <fleet>` command is the local plan-versus-inventory
  shorthand; the two-sided `diff <left> <right>` shape remains the general
  future artifact comparison model.

  `canic deploy check <fleet>` as read-only operator surfaces for local
  deployment truth JSON.

  `InstallRootOptions` into the existing local plan, inventory, diff, and

## 0.41.2 - 2026-05-22 - local deployment truth check

### Added

- Added per-design-line `status.md` logs for the 0.41-0.50 design
  directories so implementation progress and design drift can be recorded next
  to each design without rewriting the design documents themselves.
- Clarified the 0.41-0.43 install safety ladder: 0.41 remains truth/report
  groundwork around the current installer, 0.42 is report-first dry-run
  authority reconciliation, and full plan-driven deploy-install execution
  belongs to 0.43 unless explicitly promoted earlier.
- Added a read-only local deployment plan builder that turns resolved fleet
  config plus the local role artifact manifest into `DeploymentPlanV1` without
  querying IC state or replacing the current installer.
- Added a read-only local deployment check wrapper that builds
  `DeploymentPlanV1`, collects `DeploymentInventoryV1`, derives
  `DeploymentDiffV1`, and returns `SafetyReportV1` without mutating deployment
  state.
- Added a read-only local role artifact manifest builder that maps configured
  fleet roles and materialized `.wasm.gz` files into `RoleArtifactManifestV1`,
  including release-set payload hashes when a root release-set manifest is
  available.
- Missing artifact roots, release-set manifests, or role artifacts are captured
  as artifact observation gaps instead of changing installer behavior.
- Local artifact inventory and role manifests now include observed `.wasm.gz`
  file SHA-256 evidence with explicit `ObservedFileDigest` source metadata,
  keeping those live file hashes separate from release-set payload hashes.
- Added a passive deployment truth evaluator that compares
  `DeploymentPlanV1` with `DeploymentInventoryV1` into `DeploymentDiffV1` and
  renders `SafetyReportV1` without gating or mutating installer behavior.
- The initial evaluator reports missing artifacts, unsafe control classes,
  root/network identity mismatches, canonical config mismatches, unobserved
  verifier readiness, and inventory observation gaps as machine-readable
  findings.
- Deployment diffs now surface observed local artifact file hashes as
  informational `artifact_file_sha256` evidence instead of comparing them as
  release-set payload hashes.
- Added a read-only local inventory collector for `canic-host::deployment_truth`
  that snapshots configured fleet roles, local install-state root identity, and
  materialized local `.wasm.gz` artifacts into `DeploymentInventoryV1` without
  querying or mutating IC state.
- Missing config, role, artifact-root, or artifact observations are recorded as
  inventory observation gaps instead of being treated as installer errors.

### Changed

- Split `canic-host::deployment_truth` into focused module files: public
  exports and the schema version remain in `mod.rs`, passive V1 DTOs live in
  `model.rs`, local observation and artifact-manifest collection live in
  `observe.rs`, diff/safety-report logic lives in `report.rs`, and focused
  unit coverage lives in `tests.rs`.

## 0.41.0 - 2026-05-21 - deployment truth design prep

### Added

- Started the 0.41 deployment truth model line from
  `docs/design/0.41-deployment-truth-model/0.41-design.md`.
- Framed the 0.41 release objective around separating intended deployment
  state, observed inventory, receipts, normalized diffs, and operator-facing
  safety reports before deployment mutation.
- Documented the initial object model names:
  `DeploymentPlanV1`, `DeploymentInventoryV1`, `DeploymentReceiptV1`,
  `DeploymentDiffV1`, and `SafetyReportV1`.

### Notes

- This is design preparation for the 0.41 slice. It does not change runtime
  deployment behavior.
- The active implementation scope remains intentionally narrower than later
  authority reconciliation, backend-neutral execution, artifact promotion,
  user-owned lifecycle, or multi-deployment operations work.

### Testing

- Documentation-only slice; no code validation required beyond diff hygiene.
