# Canic 0.94 Changelog

## [0.94.10] - 2026-07-20 - Durable Artifact And Manifest Reconciliation

### 📝 Summary

The backup runner now reconciles the durable artifact transition and final
manifest publication as one immutable closeout path. Restart verifies the
exact canonical directory against the journal checksum, publishes a manifest
only once, and adopts an existing manifest only when it exactly matches the
current plan and durable journal authority.

### 🩹 Fixed

- Makes persistence the single owner of durable artifact verification and
  reuses that owner for both finalization recovery and complete-layout
  integrity checks.
- Rehashes every artifact already marked `Durable` before manifest creation;
  missing or changed canonical bytes can no longer produce a successful
  manifest from a stronger journal claim.
- Preserves exact canonical directory identity across either journal-write
  crash side while completing the normal finalization receipt once.
- Leaves contradictory durable state explicit and fails closed rather than
  downgrading the journal, redownloading, republishing, or inventing recovery
  evidence.
- Gives manifest publication create-or-adopt semantics: first publication is
  atomic and no-replace, exact recovery is a no-op, and conflicting content is
  preserved and rejected.
- Rejects any manifest appearing before the finalization operation is pending,
  before preflight or another backup operation can execute.

### ⚠️ Breaking

- Finalization now rejects durable journal rows whose canonical artifact is
  missing or checksum-conflicting. This is a pre-1.0 hard cut; no fallback,
  state downgrade, compatibility path, or legacy acceptance remains.
- `BackupLayout::write_manifest` is removed and replaced by
  `BackupLayout::publish_manifest`; callers can no longer request replace
  semantics. The public persistence and runner error enums gain the exact
  `ManifestConflict` and `PrematureManifest` causes. No alias, deprecated
  method, compatibility wrapper, or replace fallback remains.
- CLI syntax, JSON output, Candid, durable document shapes, manifest shape,
  plans, Cargo package versions, and genuine version `1` fields are unchanged.

### 🧪 Testing

- Kills a child before the durable journal rename and after rename plus
  parent-directory synchronization, then proves restart verifies the same
  canonical directory and completes without commands or inode replacement.
- Proves a missing or changed canonical artifact cannot create a manifest and
  preserves the typed missing-artifact or checksum-mismatch cause.
- Kills final-manifest publication immediately before canonical publication
  and after parent-directory synchronization, then proves restart publishes or
  adopts the exact manifest without inode replacement.
- Proves conflicting valid manifest content is preserved and rejected, while a
  premature manifest blocks the runner without changing execution state.
- Passes all 292 `canic-backup` library tests, strict all-target Clippy,
  the complete `canic-cli` library suite and strict all-target Clippy,
  formatting, changelog governance, and diff hygiene.

### 📊 Audit

- Completes both `CANIC-094-B14` and both `CANIC-094-B15` write-side cases,
  bringing the frozen matrix to 38 passing cases with 68 pending.
- Fixes `CANIC-094-BACKUP-009` and `CANIC-094-BACKUP-010`; terminal execution
  receipt boundaries remain owned by `B16`.

## [0.94.9] - 2026-07-20 - Checksum-Bound Artifact Publication

### 📝 Summary

The backup runner now reconciles the checksum-verified journal transition and
canonical directory publication as one checksum-bound recovery path. Restart
either computes the checksum from exact `Downloaded` staging, adopts exact
durable checksum authority, publishes matching private staging, or recovers an
already published canonical directory after independently rehashing it.

### 🩹 Fixed

- Reconstructs the normal verification receipt from an exact durable
  `ChecksumVerified` row without overwriting that row or invoking an external
  command.
- Requires the target, snapshot identity, canonical artifact path, exact
  staging path, recorded checksum, and actual staged bytes to agree before
  durable checksum evidence can advance execution.
- Rejects changed checksum-verified staging with the existing typed checksum
  mismatch while preserving the pending execution and original checksum row.
- Centralizes exact target-row lookup and rejects ambiguous artifact authority
  instead of selecting the first matching row.
- Restricts artifact state advancement to the canonical immediate successor;
  repeated or skipped transitions no longer overwrite durable authority.
- Rehashes the temporary artifact tree immediately before first publication;
  changed bytes can no longer be renamed into the canonical path and reported
  durable under an older journal checksum.
- Reconciles a post-rename canonical directory only when its checksum matches
  the journal, while conflicting temporary/canonical paths fail closed without
  replacing either tree.

### ⚠️ Breaking

- The public backup runner error enum gains
  `ArtifactVerificationStateConflict`, and invalid repeated or skipped
  artifact transitions now reject. This is a pre-1.0 hard cut; no alias,
  fallback, legacy transition, or compatibility adapter was added.
- Persisted document shapes and their genuine version `1` fields are

- Proves changed staged bytes cannot replace durable checksum authority or

- Kills canonical publication immediately before rename and after rename plus

## [0.94.1] - 2026-07-19 - Command Quiescence And Fail-Closed Pending Recovery

### 📝 Summary

Mutating backup and restore operations now carry one restart-visible kernel
lock through the intended external command and its complete descendant tree.
Runner death can no longer make a still-running predecessor look quiescent.

The same trace found that restore's pending-reset mode erased an unknown
external outcome and permitted blind replay. That mode and all supporting
surface are hard-cut; unknown mutations remain pending until an operation-
specific recovery journey can prove their outcome.

### 🩹 Fixed

- Keeps command-lifetime descriptors close-on-exec in the runner and passes
  only the exact operation descriptor through child setup to the maintained
  backup/restore command.
- Requires the direct child and descendants to release that descriptor before
  terminal state or a receipt can be persisted. A surviving tree returns a
  typed command-in-flight cause.
- Distinguishes a quiescent command from a known external outcome. Mutating
  `Pending` work now returns a typed unknown-outcome cause without execution,
  state change, receipt creation, or staging cleanup.
- Allows read-only restore verification left `Pending` by interruption to run
  again through the ordinary receipt path.
- Centralizes no-follow regular-file admission and kernel locking for journal
  and command-lifetime sidecars without merging their ownership semantics.

### ⚠️ Breaking

- Removes `canic restore run --unclaim-pending` and the public
  `restore_run_unclaim_pending` API.
- Removes the restore runner `unclaim_pending` mode flag,
  `pending-recovered` receipt kind, `pending_recovered` receipt-summary field,
  and pending-to-ready journal helper. No alias, deprecated mode, fallback
  reset, synthetic receipt, or version-1 output remains.
- Advances restore runner JSON to `run_version: 2` for the maintained hard-cut
  key set. Persisted restore journals remain version 1 and are unchanged.
- Extends injected backup and restore executor contracts with the exact
  command-lifetime handle required by mutating process adapters.

### 🧪 Testing

- Proves `SIGKILL` of the runner owner leaves the operation lock held until a
  direct child and its descendant exit.
- Proves the real host command setup inherits exactly the requested descriptor
  through exec and descendants.
- Proves both runners reject an active predecessor, reject a quiescent unknown
  mutation without replay, and preserve pending state plus empty receipts.
- Proves pending read-only restore verification completes normally.
- Passes focused backup runner, restore runner, restore option, parallel
  restore CLI, backup create, host inheritance, package check, formatting,
  and strict targeted Clippy validation.

### 📊 Audit

- Fixes `CANIC-094-PROC-001` in the current worktree.

## [0.94.0] - 2026-07-19 - Crash-Safe Journal Ownership

### 📝 Summary

The 0.94 operational-readiness line began by freezing the current backup and
restore transition inventory and probing the real snapshot and process-death
boundaries before constructing the complete recovery harness.

The snapshot capabilities are sufficient for the planned disposable local ICP
journey. The same gate reproduced a journal-lock liveness defect: lock
ownership was represented by path existence and released only by `Drop`, so
abrupt process death could block backup or restore forever.

### 🩹 Fixed

- Makes a retained kernel advisory lock descriptor the sole journal-lock
  authority; process death releases ownership without deleting or guessing
  about the sidecar path.
- Opens the sidecar no-follow and close-on-exec, rejects symlink and special
  entry substitution, and preserves distinct typed live-owner and unsafe-entry
  causes.
- Keeps one stable sidecar inode across owners. A stale regular file is no
  longer mistaken for a live process, and a live descriptor remains exclusive.

### ⚠️ Breaking

- Sidecar file existence and PID-like contents no longer mean that a journal is
  locked. Only a live kernel lock owns the maintained backup and restore
  journal boundary.
- The restore error surface now distinguishes an unsafe journal-lock path from
  an ordinary live lock. No force-delete command, stale-age heuristic, PID
  lease, fallback lock path, alias, or compatibility behavior is retained.

### 🧪 Testing

- Focused lock tests prove live-owner exclusion, descriptor close-on-exec,
  stale-sidecar reacquisition, symlink and directory rejection, and immediate
  reacquisition after the lock-owning process is killed without unwinding.
- Focused backup-runner and restore-CLI tests prove a real competing kernel
  owner rejects before command execution.

### 📊 Audit

- The early capability gate confirms snapshot create/upload identity recovery,
  stopped-target restore observation, and exact repeated restore on disposable
  local ICP.
- One separate P1 command-lifetime finding remains open: a current external
  command child may outlive runner death. Its bounded executor-level fix is the
  next 0.94 slice before the full in-flight-command harness is frozen.
