## 0.27.3 - 2026-04-13 - `directory` Claim Ownership and Recovery

### 📝 Summary

- `0.27.3` hardens keyed `directory` placement under interruption: only the current `Pending` claim holder can finalize async create results, missing provisional children no longer wedge stale recovery, and resolve/recover now share one pending-state classifier instead of drifting apart.

### 🔧 Changed

- Added opaque `claim_id` ownership to `DirectoryEntryRecord::Pending`, so claim state is durable across async create windows instead of relying on a bare “some pending exists” check.
- Updated `DirectoryWorkflow::resolve_or_create` to finalize created children with claim-matching writes only, which prevents late async completions from stealing a key after a stale claim has already been reclaimed.
- Updated stale cleanup so missing provisional children are treated as already cleaned, allowing stale `Pending` claims to be released and reclaimed instead of blocking the key forever.
- Unified stale-entry interpretation behind one shared pending classifier, so `resolve_or_create` and `recover_entry` now follow the same `fresh` / `repairable` / `cleanup-needed` rules.
- Tightened the low-level CAS helpers so late claim losers now fail cleanly without mutating the entry, letting the workflow recycle abandoned children instead of surfacing claim-mismatch errors as the main outcome.

## 0.27.2 - 2026-04-13 - `directory` Placement Runtime

### 📝 Summary

- `0.27.2` is the first full `directory` placement cut: singleton parents can declare keyed instance pools, `instance` children are restricted to those parents, the runtime stores `Pending | Bound` directory entries, and `resolve_or_create` now claims before async create so stale in-flight keys cannot block progress forever.

### ➕ Added

- Added `directory` config support to `CanisterConfig`, including `DirectoryConfig` and `DirectoryPool`, so singleton roles can declare keyed instance-placement pools without overloading `scaling` or `sharding`.
- Added schema validation for directory pools, including target-role existence, target kind checks, bounded pool names, and non-empty bounded `key_name` enforcement.
- Added stable storage for `directory` placement entries, using `DirectoryEntryRecord::{Pending, Bound}` instead of a plain key-to-pid map.
- Added `DirectoryEntryStatusResponse` plus status-aware query/API reads, so callers can distinguish externally resolvable `Bound` entries from in-flight `Pending` claims.
- Added `DirectoryRegistryOps`, `DirectoryQuery`, and `DirectoryApi` for directory bind/lookup/status access through the supported placement surface.
- Added `DirectoryWorkflow::resolve_or_create` and the matching `DirectoryApi::resolve_or_create` surface for keyed instance placement.
- Added repair-first recovery for stale `Pending` entries that still point at a valid direct-child `instance` with the configured role.
- Added focused workflow tests for the key non-duplicate and liveness paths: existing `Bound`, fresh `Pending`, stale provisional-child repair, stale pending without a provisional child, and stale pending with an invalid provisional child.
- Added renderer support for compiled `directory` config and new schema/policy seam tests covering valid and invalid `directory` usage.

### 🔧 Changed

- Tightened registry policy so `instance` children are only valid under singleton parents that actually declare `directory`.
- Renamed the public policy error surface from `PolicyInstanceRequiresSingletonParent` to `PolicyInstanceRequiresSingletonWithDirectory`, and updated the checked-in `.did` surface to match that stricter contract.
- Updated `DirectoryRegistryResponse` so entries now expose `status` rather than a bare pid.
- Updated `DirectoryWorkflow::bind_instance` to finalize through the `Bound` writer instead of the old plain assignment path.
- `resolve_or_create` now claims `Pending` before async canister creation and returns `Pending` status instead of creating duplicate instances when another caller already holds a fresh claim.
- Successful create flows now attach `provisional_pid` immediately after child creation and finalize to `Bound` only after direct-child and role validation succeeds.
- Stale pending entries with no valid recoverable provisional child now fall through to normal reclaim-and-create behavior instead of blocking the key indefinitely.

### 🧭 Migration Notes

- Any new `instance` role now needs a singleton parent with `directory` config; a plain singleton parent is no longer enough.
- `directory` placement is now usable as an actual keyed resolve-or-create workflow rather than only as a manual bind/query registry.
- Callers should handle both `Bound` and `Pending` responses from `resolve_or_create`; a fresh `Pending` means another in-flight creator currently owns the key.

- Ran `cargo test -p canic-core --lib`.
- Ran `cargo build --workspace`.
- Ran `make clippy`.

## 0.27.1 - 2026-04-13 - `instance` Kind Cutover

### 📝 Summary

- `0.27.1` is the full first topology implementation cut after the taxonomy decision: it replaces `tenant` with `instance`, renames the old lookup/export surface from `directory` to `index` across config and runtime APIs, and updates the checked-in configs and `.did` surface to the new vocabulary.

### 🔧 Changed

- Renamed the topology config kind from `tenant` to `instance`, so the canonical serialized/displayed kind now matches the `0.27.0` taxonomy decision instead of carrying the older ambiguous term forward.
- Renamed the singleton-parent policy error from `TenantRequiresSingletonParent` to `InstanceRequiresSingletonParent`, and updated the public `ErrorCode` surface to report `PolicyInstanceRequiresSingletonParent`.
- Removed the temporary `kind = "tenant"` parser path, so `instance` is now the only accepted config spelling for dedicated keyed child roots.
- Renamed the config-facing lookup sets from `app_directory` / `subnet_directory` to `app_index` / `subnet_index`, updated the checked-in `canic.toml` and test/audit configs to the new names, and kept the old config keys only as temporary parse aliases.
- Renamed the runtime/export surface from `canic_app_directory` / `canic_subnet_directory` to `canic_app_index` / `canic_subnet_index`, and updated the matching protocol constants, API re-exports, DTO types, payload fields, validation/reporting names, and checked-in `.did` surface.
- Renamed the internal lookup modules from `directory` to `index` across API, workflow, ops, and stable-storage layers, and renamed the create-parent lookup selector from `CreateCanisterParent::Directory` to `CreateCanisterParent::Index`.

### 🧭 Migration Notes

- `kind = "instance"` is now the intended config spelling.
- `app_index` and `subnet_index` are now the canonical config keys.
- `canic_app_index` and `canic_subnet_index` are now the canonical public query names.
- The only remaining old spellings are the temporary config parse aliases `app_directory` and `subnet_directory`; the runtime/export surface itself has moved to `index`.
- Existing compatibility labels such as `replica` and `shard` are unchanged in this slice.

### 📚 Documentation

- Kept the `0.27` taxonomy and changelog line aligned with the implementation cut so the runtime and config layers now both reserve `directory` for placement semantics rather than lookup views.

## 0.27.0 - 2026-04-13 - Topology Taxonomy

### 📝 Summary

- `0.27.0` starts the topology-taxonomy design line by separating structural canister kind from placement family, defining `scaling`, `sharding`, and `directory` as the three real placement families, and moving the fleet snapshot design to the next design slot so backup/recovery can depend on a clearer topology model.

### 🔧 Changed

- Defined a clearer topology vocabulary around structural node kind, placement family, and derived ownership shape, so future config, policy, and backup design do not keep collapsing different ideas into one label.
- Reserved `directory` for keyed instance placement, where one configured logical key resolves to at most one dedicated instance root that may own a recursive subtree.
- Clarified that the older export-layer lookup surfaces are a narrower concept than placement and should be described as `index` views instead of `directory` views.
- Introduced the intended terminology split `app_directory -> app_index` and `subnet_directory -> subnet_index` at the design level, while leaving room for staged compatibility during implementation.
- Tightened the Toko-style model so users and operators remember a stable configured directory key and value rather than a raw `project_instance` canister id, with Canic responsible for resolving that key to the instance root.

### 🧭 Migration Notes

- `tenant` is now treated as deprecated design vocabulary that should be replaced by `instance` immediately, not as an acceptable long-term public concept.
- New topology work should target `instance` as the intended concept for directory-managed dedicated child roots.
- If the existing code or config still uses `tenant`, it should be treated as migration debt and updated to `instance` immediately.

### 📚 Documentation

- Split the previous `0.27` fleet snapshot draft into a new `0.28` slot so the snapshot/backup design can explicitly build on the `0.27` topology taxonomy instead of carrying unresolved topology questions inline.
