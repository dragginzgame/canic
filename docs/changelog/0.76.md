# 0.76 Changelog

## 0.76.0 - 2026-06-30 - Bridge-free delegated auth

### 📝 Summary

- Replaces delegated-auth renewal's bridge-backed root proof path with
  `RootProof::IcChainKeyBatchSignatureV1`, a chain-key batch proof signed by
  root through management-canister ECDSA.

- Makes the 0.76 delegated-auth cutover a hard cut. In `chain_key_batch` mode,
  retained bridge-backed canister-signature provisioning endpoints reject
  instead of remaining as a compatibility renewal path.

- Potentially breaking for deployments, tooling, or tests that still expect
  bridge-backed root proof renewal, legacy root proof acceptance, or
  delegated-token auth to be implicitly enabled in minimal configs.

### ➕ Added

- Chain-key batch root proof DTOs, canonical signing and verification payloads,
  persisted root delegation batch state, delegated-auth registry/proof epoch
  state, and verifier support for the new proof shape.

- Management-canister ECDSA public-key and signing wrappers for delegated-auth
  proof renewal, with production signing cycles derived from the platform/CDK
  call-time fee helper.

- Root timer orchestration that prepares due issuer proof material, signs one
  batch payload, installs issuer active proofs, and records issuer install
  results without requiring a bridge, CLI worker, direct root query, host
  daemon, or client-side provisioning step.

- Issuer delegated-token lazy repair through an internal root update that
  returns cached chain-key proof material or drives a new batch signature when
  active proof material is missing.

- Explicit local test-fleet `chain_key_batch` trust-anchor policy using the
  PocketIC-exposed `key_1` root-derived public key.

- Config and runtime chain-key policy validation for secp256k1 SEC1 public-key
  bytes before chain-key trust anchors are accepted.

### 🔧 Changed

- Root proof contracts, architecture notes, and delegated-auth documentation now
  describe `RootProof::IcChainKeyBatchSignatureV1` as the current proof
  contract for chain-key delegated auth.

- Root signing normalizes high-s management-canister ECDSA signatures before
  proof persistence while verifier policy continues to reject high-s proof
  signatures.

- Delegated-token auth is opt-in by default, so minimal non-auth configs no
  longer need a chain-key trust-anchor policy. Auth-enabled configs must still
  declare explicit `chain_key_batch` key metadata.

- The 0.76 design records the official management-canister contract used by
  this line: ECDSA signs exactly a 32-byte message hash, secp256k1 signatures
  are raw `r || s`, public-key derivation is bound to canister id, derivation
  path, and key id, signing requires explicit cycles, and unknown or canister
  errors cannot prove that no signature exists.

- The first 0.76 non-Rust verifier fixture decision is closed without adding
  fixtures because the repository has no Motoko or TypeScript verifier
  consumer today.

### 🗑️ Removed

- Passive 0.76 cutover DTO and stable-record states for `LegacyBridge`,
  `DualCode`, `ChainKeyPreferred`, and `ChainKeyOnly`, avoiding dormant
  compatibility state that could be wired accidentally later.

- Ignored root-suite TODO placeholders that no longer represented open 0.76
  acceptance gates.

### 🧪 Testing

- Added focused chain-key verifier negative coverage for wrong public keys, key
  ids, derivation paths, root canisters, helper-key signatures, witnesses,
  grants, certificates, leaf windows, duplicate issuer leaves, and legacy root
  proofs.

- Added fake-signer state-machine coverage for signing retry-after recovery,
  duplicate signing ticks, stale registry changes during signing, expired
  pre-install pruning, partial issuer install retry, and stale callback discard
  before returned signatures are persisted.

- Added live PocketIC acceptance coverage proving the local chain-key trust
  anchor matches management-canister public-key discovery, legacy
  bridge-backed provisioning rejects in hard-cut mode, root timer renewal signs
  and installs issuer proof material without external liveness, issuer
  lazy-repair reuses cached proof material, timer renewal batches multiple
  issuers into one management signing operation, and concurrent missing-proof
  repairs collapse to one signing operation.
