# 0.80 Changelog

## 0.80.7 - 2026-07-03 - State manifest reservation metadata

### 📝 Summary

- Makes allocated stable-memory IDs that are not yet modeled as precise state
  domains visible in `canic state manifest` and `canic state audit`.

- Adds an explicit `not_applicable` storage classification for
  declaration-only state metadata so it is not conflated with heap-only runtime
  state.

### ➕ Added

- State manifests now include a `reserved_memory` array for memory IDs that are
  allocated but not yet represented as active state domains or removed-state
  dispositions.

- The root manifest reserves the raw cycle tracker memory and the two
  stable-log memories until those state shapes have precise record/snapshot
  declarations.

- New state-audit check codes:
  `reserved_memory_id_declared`,
  `reserved_memory_id_duplicate`,
  `reserved_memory_id_collision`, and
  `state_domain_storage_not_applicable`.

### 🔧 Changed

- `canic state manifest` text output now renders `reserved_memory` entries with
  label, memory ID, owner, and reason.

- State-audit reports warn when a reserved memory ID remains a temporary
  reservation, fail if the same reserved memory ID is duplicated, and fail if a
  reservation collides with active or removed state.

- State storage metadata now serializes `not_applicable` as a first-class value
  alongside `stable_memory` and `heap_only`.

### 🧭 Migration Notes

- This patch is diagnostic metadata only. It does not inspect stable memory,
  run migrations, change memory IDs, write generated manifests, or mutate
  canister state.

- `reserved_memory` rows are intentionally temporary audit visibility. A future
  slice can replace each reservation with a precise active state-domain
  declaration or removed-state disposition once the state shape is known.

### 🧪 Testing

- Focused tests cover reserved-memory manifest output, reserved-memory audit
  warning/failure behavior, and the `not_applicable` storage classification.

Validated with:

```bash

cargo test --locked -p canic-host state_manifest
cargo test --locked -p canic-cli state

cargo clippy --locked -p canic-host --all-targets -- -D warnings
cargo clippy --locked -p canic-cli --all-targets -- -D warnings
```

## 0.80.6 - 2026-07-03 - Mutation preview ergonomics

### 📝 Summary

- Makes local filesystem mutation boundaries visible where developers already
  look: the existing fleet and scaffold command help.

- Adds no-write dry-run previews for local source/config mutators while keeping
  Cargo.toml dependency-feature edits manual and developer-owned.

### ➕ Added

- `canic fleet help` now includes mutation notes naming read-only fleet
  commands, local source/config writes, role config/package metadata writes,
  fleet deletion, and the `--dry-run` preview convention.

- `canic fleet role help` now distinguishes read-only `list`/`inspect` from
  mutating `declare`/`attach`/`rename`, and documents that the mutating role
  commands support `--dry-run`.

- `canic scaffold help` now documents that canister scaffolding writes a local
  role crate, may append the workspace `Cargo.toml` member, and declares the
  role in `canic.toml`.

- New no-write preview flags:
  `canic fleet create --dry-run`,
  `canic scaffold canister <fleet> <role> --dry-run`,
  `canic fleet role declare ... --dry-run`,
  `canic fleet role attach ... --dry-run`,
  `canic fleet role rename ... --dry-run`, and
  `canic fleet delete <fleet> --dry-run`.

### 🔧 Changed

- Dry-run role previews use host-side planning helpers that validate the
  requested `canic.toml` mutation without writing files.

- Canister scaffold planning now keeps concrete write paths separate from
  display paths, so output formatting cannot affect the subsequent write path.

- Workspace `Cargo.toml` member appends now parse the manifest, only skip when
  the member is actually present in `[workspace].members`, and validate the
  post-edit manifest before any write.

- Package metadata updates during role rename now parse and validate the
  updated `Cargo.toml` source before writing it.

- Fleet and scaffold preview renderers now share one dry-run footer helper for
  the stable `dry_run: true` / `files_changed: 0` output lines.

### 🧭 Migration Notes

- This patch does not add automatic Cargo.toml dependency feature editing.
  Developers remain responsible for applying dependency-feature fixes surfaced
  by medic/build diagnostics.

- No separate top-level mutation-discovery command is introduced; the
  discoverability path is the existing command-family help.

### 🧪 Testing

- Focused tests cover dry-run option parsing, no-write planning, role mutation
  validation, workspace member append validation, package metadata validation,
  rendered preview text, and command-family help mutation notes.

Validated with:


## 0.80.3 - 2026-07-03 - Auth feature-gate preflight

### 📝 Summary

- Adds config-driven preflight checks for runtime `canic` features required by
  fleet auth settings.

- Moves role-to-feature requirement mapping into shared core bootstrap logic so
  medic diagnostics and build-time guards use the same source of truth.

### ➕ Added

- `canic medic project` now reports `role_required_canic_feature_missing` when
  a role's fleet config requires a runtime Canic feature that the role crate's
  `canic` dependency does not resolve.

- `canic medic project --json` exposes the same check code for automation, with
  `role_required_canic_feature_present` emitted when the role resolves the
  required feature.

- `canic::build!` now fails during ordinary canister builds when the current
  package is missing a required runtime `canic` dependency feature. The guard
  checks both direct `[dependencies].canic.features` and inherited
  `[workspace.dependencies].canic.features`.

### 🔧 Changed

- Role-attestation cache users now get an early diagnostic for
  `auth-root-canister-sig-verify` instead of reaching the runtime trap first.

- Root canisters serving role-attestation cache users are checked for
  `auth-root-canister-sig-create`, and delegated-token issuer/verifier roles
  are checked for their issuer-create and delegated-token-verify features.

### 🧪 Testing

- Focused tests cover shared role-to-feature mapping, direct and workspace
  inherited dependency feature detection, missing-feature build failures, and
  medic pass/fail project-config diagnostics.

## 0.80.2 - 2026-07-03 - Expanded state manifest coverage

### 📝 Summary

- Expands the diagnostic Rust-authored state manifest to cover more
  non-gated root-family `canic-core` stable-memory domains.

- Keeps the state surfaces metadata-only. The new declarations and checks do
  not inspect stable memory values, run migrations, write generated manifests,
  create deployment truth, or mutate canisters.

### 🔧 Changed

- The root-family manifest now declares additional subnet topology/state,
  cycle top-up/funding/refill, intent-store, canister-pool, scaling-registry,
  and directory-registry stable-memory domains.

- The root role manifest construction is split by domain family so future
  declaration work can stay local to topology, environment, auth,
  observability, intent, or capacity state.

### ➕ Added

- `canic state audit` now reports `removed_state_memory_id_reserved` when a
  retired state declaration keeps its stable-memory ID reserved.

- `canic state audit` now fails with `removed_state_memory_id_reclaimed` when
  an active stable-memory domain reuses an ID declared by removed state. This
  keeps retired IDs reserved unless a future explicit migration design handles
  them.

### 🧭 Migration Notes

- Ambiguous multi-memory log declarations, raw cycle-tracker metadata, and
  feature-gated sharding/blob-storage declarations remain deferred until the
  manifest model can represent those shapes precisely.

### 🧪 Testing

- Focused tests cover expanded root manifest memory-ID coverage and the new
  removed-state memory-ID reclaim failure.

## 0.80.1 - 2026-07-03 - Diagnostic state manifest and audit

### 📝 Summary

- Adds the first 0.80 state metadata surfaces:
  `canic state manifest` and `canic state audit`.

- Keeps the slice diagnostic-only. The new commands do not inspect stable
  memory values, run migrations, repair memory IDs, write generated manifests,
  create deployment truth, or mutate canisters.

### ➕ Added

- `canic state manifest` renders the derived Rust-authored manifest to stdout
  in text form, with `--json` emitting the manifest directly and
  `--role <role>` filtering to one declared canister role.

- `canic state audit` emits a schema-versioned report with `schema_version = 1`,
  stable check rows, text/JSON output, `--role <role>` filtering, and exit code
  1 only when failing checks are present.

- The initial static declarations cover the root canister family for the first
  canic-core stable-memory domains and the retired root replay memory ID.

### 🔧 Changed

- Top-level CLI help now lists `state` as a project command, and command-family
  help/version tests include the new command.

- The historical root replay memory ID is now referenced by the removed-state
  declaration, so its stale dead-code lint expectation was removed.

### 🧪 Testing

- Focused tests cover manifest JSON shape, audit JSON shape, text output
  fields, failed-audit exit code behavior, duplicate memory-ID detection,
  unknown-role failure, and the hard-cut rejected command forms
  (`migrate`, `repair`, `explore`, `dump`, positional role shorthands,
  `--format json`, `--out`, and `--force`).

## 0.80.0 - 2026-07-03 - Packaged wasm-store bootstrap build

### 📝 Summary

- Fixes packaged downstream root `fast` builds for the implicit `wasm_store`.

- Keeps downstream projects out of Canic's internal bootstrap-store source,
  wrapper, and Cargo-profile concerns.

### 🩹 Fixed

- Generated fallback wrappers now enable the wasm-store-only Canic facade
  feature instead of the broader root control-plane feature.

- The host-side bootstrap-store builder now supplies Canic's release and fast
  profile settings when invoking Cargo for the hidden `wasm_store` build. This
  lets canonical packaged `canic-wasm-store` sources build under
  `CANIC_WASM_PROFILE=fast` without requiring downstream manifests to define
  Canic profiles.

- The packaged downstream proof now covers both bootstrap source-resolution
  paths: canonical packaged `canic-wasm-store` sibling present, and generated
  fallback wrapper when the canonical crate is absent.

### 📚 Documentation

- Active release-probe documentation now describes the special `wasm_store`
  path as Canic-owned bootstrap behavior, not a downstream wrapper
  responsibility.

### 🧪 Testing

- Focused host tests pin the generated wrapper feature and the host-owned fast
  profile config.

- The packaged downstream wasm-store proof validates both generated-wrapper and
  canonical packaged-sibling builds under `CANIC_WASM_PROFILE=fast`.

Validated with:

```bash
cargo fmt --all
bash -n scripts/ci/verify-packaged-downstream-wasm-store.sh
cargo test -p canic-host bootstrap_store
scripts/ci/verify-packaged-downstream-wasm-store.sh
git diff --check
```
