# 0.42 Changelog

## 0.42.4 - 2026-05-23 - dry-run authority readiness

### Changed

- `AuthorityReconciliationPlanV1.external_actions_required` now contains only
  actual external authority actions. Unknown controller observations remain
  typed observation gaps, and unsafe blockers remain hard failures.
- `AuthorityReceiptV1` now preserves unresolved controller-observation gaps
  directly, so standalone dry-run receipts keep missing-evidence facts without
  overloading external-action records.
- `AuthorityReportV1` now includes `apply_readiness`, a report-only summary
  of automatic-action candidates and typed blockers.
- `AuthorityApplyReadinessV1` uses typed `AuthorityApplyBlockerV1` values for
  hard failures, observation gaps, and external actions.

### Design alignment

- Updated the 0.42 design and status docs to match the post-0.41 dry-run
  release boundary.
- Marked authority apply, pool mutation, remote lock/epoch checks, and
  post-apply verification as promoted-or-later work unless explicitly accepted
  into the line.

### Validation

- Added focused host coverage proving automatic controller candidates do not
  make a report apply-ready while external authority actions remain unresolved.
- Updated authority receipt/report tests for observation-gap preservation and
  typed apply-readiness blockers.

## 0.42.3 - 2026-05-23 - break-glass authority blocking

### Changed

- Authority reconciliation now blocks authority profiles where staging or
  emergency principals overlap the normal expected controller set.
- `AuthorityReportCountsV1` now includes `hard_failures`, allowing automation
  to distinguish hard authority findings from unsafe canister-state counts
  without scanning the full finding list.
- `AuthorityReceiptV1` now preserves hard authority findings alongside
  verified controller observations and unresolved external actions.
- Blocked authority reports now emit separate next-action guidance for unsafe
  canister authority findings and hard authority findings.

### Authority semantics

- Staging and emergency principals remain authority categories, not normal IC
  controllers.
- Overlap between staging/emergency authority and the normal expected
  controller set is reported as an `authority_profile_overlap` hard failure in
  the dry-run authority plan and report.
- A report with hard authority findings is `Blocked` even if the observed
  canister controller state is otherwise already correct.

### Validation

- Added focused host tests for staging/emergency overlap blocking.
- Added focused host tests proving dry-run receipts preserve hard findings.
- Updated blocked-report tests to assert specific next-action guidance for
  unsafe canister findings and hard authority findings.

## 0.42.2 - 2026-05-23 - authority dry-run receipts

### Added

- Added `AuthorityReceiptV1` as the evidence-only receipt for authority dry-run
  reconciliation.
- Added `AuthorityAttemptedActionV1` and
  `AuthorityControllerObservationV1` so receipts can distinguish attempted
  controller mutations from verified controller observations.
- Added `authority_dry_run_receipt_from_plan(...)` to build a completed dry-run
  receipt from an authority reconciliation plan and report.
- Added `AuthorityDryRunEvidenceV1` as a single archiveable JSON bundle
  containing the local authority plan, report, and dry-run receipt.
- Added read-only `canic deploy authority receipt <fleet>` output for
  evidence-only `AuthorityReceiptV1` JSON.
- Added read-only `canic deploy authority evidence <fleet>` output for
  evidence-only `AuthorityDryRunEvidenceV1` JSON.

### Authority semantics

- Authority dry-run receipts explicitly record zero attempted controller
  mutations.
- Verified controller observations are copied from the dry-run reconciliation
  plan.
- Unresolved external actions are copied from the authority report, preserving
  operator action requirements without turning diagnostics into authority.
- `canic deploy authority receipt` and `canic deploy authority evidence` are
  read-only local surfaces and do not apply controller changes.

### Validation

- Added focused host tests for authority dry-run receipt construction and JSON
  round-tripping of the evidence bundle.
- Added focused CLI parser/dispatch coverage for `canic deploy authority
  receipt` and `canic deploy authority evidence`.

## 0.42.1 - 2026-05-23 - authority report and evidence surface

### Added

- Added `AuthorityReportV1` and `AuthorityReportCountsV1` as the
  operator-facing summary view over `AuthorityReconciliationPlanV1`.
- Added `authority_report_from_plan(...)` to summarize already-correct
  canisters, automatic dry-run actions, external-action cases, unsafe blocked
  canisters, and unknown observations.
- Added read-only `canic deploy authority report <fleet>` to print
  `AuthorityReportV1` JSON without applying controller changes.
- Added expected pool canisters from `DeploymentPlanV1.expected_pool` to the
  dry-run authority reconciliation plan.
- Added observed pool canisters from `DeploymentInventoryV1.observed_pool` to
  authority reconciliation when they are not already expected by the plan.
- Added `AuthorityAutomaticActionV1` as the explicit dry-run record for
  authority changes that a later apply path may perform automatically.
- Added `automatic_actions` to `AuthorityReconciliationPlanV1` and
  `AuthorityReportV1`.
- Added report next-action guidance for safe dry-run plans that contain
  automatic authority candidates.
- Added `observation_gaps` to `AuthorityReportV1` so missing controller
  evidence is visible as typed report data instead of being inferred from
  summary counts.
- Added `AuthorityActionCountV1` and `AuthorityReportV1.action_counts` so
  automation can summarize add/remove/replace, external-controller, pool
  adoption, blocked, and unknown-observation cases without scanning every
  action.
- Added `AuthorityControlClassCountV1` and
  `AuthorityReportV1.control_class_counts` so report consumers can see
  deployment-controlled, pool, imported, jointly controlled, user-controlled,
  and unsafe authority inventory at the report level.

### Changed

- Expanded `AuthorityExternalActionV1` so each external-action record carries
  the full evidence needed to act without reopening the raw reconciliation
  plan:
  - subject;
  - canister ID;
  - role;
  - control classification;
  - reconciliation state;
  - action;
  - observed controllers;
  - desired controllers;
  - reason.
- `AuthorityReportV1.external_actions_required` now preserves the same
  self-contained external-action records emitted by
  `AuthorityReconciliationPlanV1`.

### Authority semantics

- Expected pool canisters with matching observed pool inventory but no
  controller evidence are reported as `UnknownObservation` external-action
  records instead of being treated as already reconciled.
- Unplanned observed pool canisters are reported as `AdoptPlanAvailable`
  external-action records, preserving the distinction between passive
  observation and explicit future pool ownership/adoption flows.
- Unsafe observed pool canisters are reported as blocked authority findings.
- Automatic actions are emitted only for `CanApplyAutomatically` canister
  actions with a concrete canister ID.
- Each automatic action carries subject, canister ID, role, action, observed
  controllers, desired controllers, and reason.
- External actions remain separate from automatic actions, so future apply
  logic can consume a narrow candidate set instead of filtering all canister
  actions by convention.

### Validation

- Added focused host tests for safe, warning, and blocked authority reports.
- Added CLI parser coverage for `canic deploy authority report`.
- Added focused authority tests for expected pool controller observation gaps
  and unplanned observed pool canisters.
- Updated focused authority tests to assert that user-controlled drift and
  unknown unsafe canisters include the exact observed/desired controller
  evidence in external-action records.
- Updated focused authority tests to assert that automatic controller deltas
  are preserved in both the plan and report surfaces.
- Updated focused authority tests to assert report observation gaps,
  action-count breakdowns, and control-class count breakdowns.

## 0.42.0 - 2026-05-23 - dry-run authority reconciliation

### Added

- Added passive authority reconciliation DTOs under
  `canic-host::deployment_truth`:
  - `AuthorityReconciliationPlanV1`;
  - `CanisterAuthorityActionV1`;
  - `AuthorityExternalActionV1`;
  - `AuthorityActionV1`;
  - `AuthorityReconciliationStateV1`.
- Added a dry-run authority reconciliation planner that consumes the existing
  0.41 `DeploymentCheckV1` instead of creating a second inventory or drift
  model.
- Added read-only `canic deploy authority check <fleet>` to print
  `AuthorityReconciliationPlanV1` JSON without applying controller changes.

### Authority semantics

- Already-correct controller sets are reported as `AlreadyCorrect`.
- Deployment-controlled controller deltas are reported as
  `CanApplyAutomatically` for a later apply path, but 0.42.0 does not mutate
  controller state.
- Pool, imported, jointly controlled, and user-controlled canisters are
  reported as external-action cases rather than silently treated as managed
  deployment canisters.
- Unknown unsafe canisters are reported as blocked authority findings.

### Validation

- Added focused host tests for already-correct authority, automatic dry-run
  controller deltas, external-action classification, and unsafe authority
  blocking.
- Added CLI parser coverage for `canic deploy authority check`.
