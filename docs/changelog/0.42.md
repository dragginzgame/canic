# 0.42 Changelog

## 0.42.14 - 2026-05-24 - authority closeout hardening

### Changed

- Authority CLI help now documents exit-status scope for dry-run authority
  commands: a successful command means the local authority artifact was
  produced, not that controller state changed or that the whole deployment is
  safe.
- The 0.42 authority design/status docs now clarify that authority `Safe` is
  scoped to authority reconciliation only and does not replace the
  whole-deployment safety report.
- Authority receipts and evidence are now documented as structural
  self-consistency artifacts, not signed, sealed, or tamper-evident proof of
  live controller state.
- Added explicit `Authority*V1` schema-governance rules: existing V1 fields
  and enum variants are stable, incompatible meaning or shape changes require
  a new schema version or V2 artifact, text output is presentation-only, and
  dry-run receipts must remain distinct from future controller-mutating
  receipts.

### Tests

- Added source-scan tests that keep the authority CLI and deployment-truth
  authority paths free of controller mutation primitives.
- Added JSON shape tests for `AuthorityReconciliationPlanV1`,
  `AuthorityReportV1`, `AuthorityReceiptV1`, and `AuthorityDryRunEvidenceV1`
  so public field names and enum strings remain deliberate automation
  surfaces.

## 0.42.13 - 2026-05-24 - authority reconciliation closeout

### Changed

- Added the 0.42 closeout audit report, concluding that the authority

- Updated the current status handoff so the latest 0.42 live patch and

  `git diff --check`.

## 0.42.12 - 2026-05-24 - authority receipt and dry-run surface clarity

### Changed

- `canic deploy authority receipt` now uses a receipt-only host helper instead
  of building a full authority evidence bundle and extracting its receipt.
  The receipt output still uses the same report/check provenance validation
  path and remains read-only.
- Authority dry-run receipt construction now rejects `finished_at` timestamps
  earlier than `started_at` directly, so receipt-only output has the same
  timestamp-order guard previously enforced by full evidence validation.
- The generic authority receipt-from-check helper now takes an explicit report
  ID, leaving only the local-ID convenience wrapper to choose Canic's standard
  local report and receipt IDs.
- Authority receipt and evidence text output now explicitly reports
  `controller_mutation: none_attempted` for dry-run receipts.
- Authority plan, report, evidence, and receipt text output now stamps
  `mode: dry_run`, making the read-only 0.42 authority boundary visible in
  every human-oriented authority surface.
- Top-level `canic deploy authority` help no longer describes the authority
  leaves as JSON-only now that each leaf supports JSON by default and text via
  `--format text`.

### Validation

- Verified with focused host authority text tests, focused CLI authority tests,
  host and CLI clippy, formatting, and `git diff --check`.

## 0.42.11 - 2026-05-24 - helper surface cleanup and receipt validation

### Changed

- Authority dry-run receipt construction now rejects unsupported source schema
  versions, missing source report check provenance, blank receipt identity
  inputs, and missing completion timestamps before emitting standalone receipt
  evidence.
- The reusable PocketIC helper surface has moved out of the Canic workspace as
  the sibling internal `ic-testkit` crate. Canic-specific root/auth fixtures
  remain in `canic-testing-internal`.
- Stable-storage code now imports the upstream `ic_memory` B-tree map
  explicitly as `StableBtreeMap` instead of going through a Canic CDK wrapper.
- The declared workspace MSRV is now Rust `1.88.0`, while the internal pinned
  toolchain and CI development lane remain on Rust `1.95.0`.

### Removed

- Removed the unused SNS-specific CDK surface, including the baked-in SNS
  canister catalog. SNS deployment identities should be discovered from
  live/mainnet sources instead of maintained as static framework data.
- Removed the broad CDK NNS system-canister table. The NNS registry and
  exchange-rate canister principals now live beside the Canic core infra
  adapters that call them.
- Removed the obsolete `canic-cdk::structures::BTreeMap` wrapper.

### Validation

- Verified the focused CDK/core static-canister cleanup with
  `cargo check -p canic-cdk`, `cargo check -p canic-core`, stale-reference
  searches, and `git diff --check`.
- Broader workspace validation remains constrained by the in-progress
  `ic-testkit` workspace split.

## 0.42.10 - 2026-05-24 - authority blocker reporting

### Changed

- Authority apply-readiness blockers now distinguish unsafe canister authority
  from other hard authority findings. Unsafe canister hard-failure evidence is
  still preserved in reports and receipts, but report counts and next-action
  guidance no longer double-count unsafe canisters as separate hard
  authority-profile findings.
- Blocked authority reports now keep external-action and missing-observation
  next actions alongside unsafe/hard blocker guidance instead of hiding that
  follow-up work until blockers are resolved.
- Blocked authority report summaries now mention external-action and
  missing-observation counts when those warning-level findings coexist with
  blocking authority findings.
- Authority reports with blockers now still include next-action guidance for
  automatic dry-run candidates, so reviewable controller changes stay visible
  even when they cannot be applied yet.

### Validation

- Added focused authority-report coverage for unsafe canister blockers,
  mixed unsafe/hard authority blockers, blocked reports with external-action
  and observation-gap follow-ups, and automatic dry-run candidate guidance
  under blocker conditions.
- Added evidence/receipt regression coverage proving mutated unsafe-blocker
  readiness is rejected and unsafe findings remain preserved in dry-run
  receipts without being double-counted as hard readiness blockers.
- Verified with focused host authority tests, host all-target clippy,
  formatting, and `git diff --check`.

## 0.42.9 - 2026-05-24 - authority evidence ownership cleanup

### Changed

- Authority dry-run evidence validation now rejects blank required identity
  fields and full evidence bundles whose nested report or receipt omits source
  check provenance.
- Completed authority dry-run receipts now reject `finished_at` timestamps
  earlier than `started_at`.
- Authority dry-run evidence bundle construction now lives in `canic-host`
  deployment-truth code, keeping the CLI as a consumer of validated host
  evidence rather than the owner of plan/report/receipt assembly.
- Authority report construction from a full deployment check now has a
  host-owned helper, so CLI report output no longer assembles source
  provenance directly.
- Local authority report, receipt, and evidence artifact IDs are now generated
  by the host deployment-truth layer, keeping the CLI from owning authority
  evidence identity conventions.
- The four read-only authority CLI leaves now share one parse/load/render
  helper.

### Validation

- CLI authority tests now cover parsing, JSON default format selection, text
  format selection, format rejection, and host-helper delegation.
- Detailed authority DTO and text-rendering behavior remains covered in
  `canic-host`.
- Verified with focused host/CLI authority tests, host/CLI clippy, formatting,
  and `git diff --check`.

## 0.42.8 - 2026-05-24 - authority evidence validation hardening

### Changed

- Authority dry-run evidence validation now rejects schema-version drift across
  the evidence bundle, reconciliation plan, authority report, and authority
  receipt.
- Authority dry-run evidence validation now recomputes report summaries from
  the reconciliation plan and rejects mutated report status, summary, counts,
  readiness, action/control-class breakdowns, observation gaps, or next
  actions.
- Authority dry-run evidence validation now rejects receipts that no longer
  represent a completed successful dry run, including mutated operation status
  or command-result fields.
- Completed authority dry-run receipts must carry `finished_at`, and evidence
  bundles now reject top-level `generated_at` values that differ from the
  receipt completion time.

### Validation

- Added focused host tests for schema drift, report-derived field mutation,
  invalid receipt status/result, missing receipt completion time, and
  evidence/receipt timestamp mismatch.
- Verified with focused host evidence tests, host all-target clippy,
  formatting, and `git diff --check`.

## 0.42.3 - 2026-05-23 - break-glass authority blocking

### Changed

- Authority reconciliation now blocks authority profiles where staging or
  emergency principals overlap the normal expected controller set.
- `AuthorityReportCountsV1` now includes `hard_failures`, allowing automation
  to distinguish hard authority findings from unsafe canister-state counts
  without scanning the full finding list.
- `AuthorityReceiptV1` now preserves hard authority findings alongside
  verified controller observations and unresolved external actions.
- Blocked authority reports now emit separate next-action guidance for unsafe
  canister authority findings and hard authority findings.

### Authority semantics

- Staging and emergency principals remain authority categories, not normal IC
  controllers.
- Overlap between staging/emergency authority and the normal expected
  controller set is reported as an `authority_profile_overlap` hard failure in
  the dry-run authority plan and report.
- A report with hard authority findings is `Blocked` even if the observed
  canister controller state is otherwise already correct.

### Validation

- Added focused host tests for staging/emergency overlap blocking.
- Added focused host tests proving dry-run receipts preserve hard findings.
- Updated blocked-report tests to assert specific next-action guidance for
  unsafe canister findings and hard authority findings.

## 0.42.2 - 2026-05-23 - authority dry-run receipts

### Added

- Added `AuthorityReceiptV1` as the evidence-only receipt for authority dry-run
  reconciliation.
- Added `AuthorityAttemptedActionV1` and
  `AuthorityControllerObservationV1` so receipts can distinguish attempted
  controller mutations from verified controller observations.
- Added `authority_dry_run_receipt_from_plan(...)` to build a completed dry-run
  receipt from an authority reconciliation plan and report.
- Added `AuthorityDryRunEvidenceV1` as a single archiveable JSON bundle
  containing the local authority plan, report, and dry-run receipt.
- Added read-only `canic deploy authority receipt <fleet>` output for
  evidence-only `AuthorityReceiptV1` JSON.
- Added read-only `canic deploy authority evidence <fleet>` output for
  evidence-only `AuthorityDryRunEvidenceV1` JSON.

### Authority semantics

- Authority dry-run receipts explicitly record zero attempted controller
  mutations.
- Verified controller observations are copied from the dry-run reconciliation
  plan.
- Unresolved external actions are copied from the authority report, preserving
  operator action requirements without turning diagnostics into authority.
- `canic deploy authority receipt` and `canic deploy authority evidence` are
  read-only local surfaces and do not apply controller changes.

### Validation

- Added focused host tests for authority dry-run receipt construction and JSON
  round-tripping of the evidence bundle.

  receipt` and `canic deploy authority evidence`.

## 0.42.1 - 2026-05-23 - authority report and evidence surface

### Added

- Added `AuthorityReportV1` and `AuthorityReportCountsV1` as the
  operator-facing summary view over `AuthorityReconciliationPlanV1`.
- Added `authority_report_from_plan(...)` to summarize already-correct
  canisters, automatic dry-run actions, external-action cases, unsafe blocked
  canisters, and unknown observations.
- Added read-only `canic deploy authority report <fleet>` to print
  `AuthorityReportV1` JSON without applying controller changes.
- Added expected pool canisters from `DeploymentPlanV1.expected_pool` to the
  dry-run authority reconciliation plan.
- Added observed pool canisters from `DeploymentInventoryV1.observed_pool` to
  authority reconciliation when they are not already expected by the plan.
- Added `AuthorityAutomaticActionV1` as the explicit dry-run record for
  authority changes that a later apply path may perform automatically.
- Added `automatic_actions` to `AuthorityReconciliationPlanV1` and
  `AuthorityReportV1`.
- Added report next-action guidance for safe dry-run plans that contain
  automatic authority candidates.
- Added `observation_gaps` to `AuthorityReportV1` so missing controller
  evidence is visible as typed report data instead of being inferred from
  summary counts.
- Added `AuthorityActionCountV1` and `AuthorityReportV1.action_counts` so
  automation can summarize add/remove/replace, external-controller, pool
  adoption, blocked, and unknown-observation cases without scanning every
  action.
- Added `AuthorityControlClassCountV1` and
  `AuthorityReportV1.control_class_counts` so report consumers can see
  deployment-controlled, pool, imported, jointly controlled, user-controlled,
  and unsafe authority inventory at the report level.

### Changed

- Expanded `AuthorityExternalActionV1` so each external-action record carries
  the full evidence needed to act without reopening the raw reconciliation
  plan:
  - subject;
  - canister ID;
  - role;
  - control classification;
  - reconciliation state;
  - action;
  - observed controllers;
  - desired controllers;
  - reason.
- `AuthorityReportV1.external_actions_required` now preserves the same
  self-contained external-action records emitted by
  `AuthorityReconciliationPlanV1`.

### Authority semantics

- Expected pool canisters with matching observed pool inventory but no
  controller evidence are reported as `UnknownObservation` external-action
  records instead of being treated as already reconciled.
- Unplanned observed pool canisters are reported as `AdoptPlanAvailable`
  external-action records, preserving the distinction between passive
  observation and explicit future pool ownership/adoption flows.
- Unsafe observed pool canisters are reported as blocked authority findings.
- Automatic actions are emitted only for `CanApplyAutomatically` canister
  actions with a concrete canister ID.
- Each automatic action carries subject, canister ID, role, action, observed
  controllers, desired controllers, and reason.
- External actions remain separate from automatic actions, so future apply
  logic can consume a narrow candidate set instead of filtering all canister
  actions by convention.

### Validation

- Added focused host tests for safe, warning, and blocked authority reports.
- Added CLI parser coverage for `canic deploy authority report`.
- Added focused authority tests for expected pool controller observation gaps
  and unplanned observed pool canisters.
- Updated focused authority tests to assert that user-controlled drift and
  unknown unsafe canisters include the exact observed/desired controller
  evidence in external-action records.
- Updated focused authority tests to assert that automatic controller deltas
  are preserved in both the plan and report surfaces.
- Updated focused authority tests to assert report observation gaps,
  action-count breakdowns, and control-class count breakdowns.

## 0.42.0 - 2026-05-23 - dry-run authority reconciliation

### Added

- Added passive authority reconciliation DTOs under
  `canic-host::deployment_truth`:
  - `AuthorityReconciliationPlanV1`;
  - `CanisterAuthorityActionV1`;
  - `AuthorityExternalActionV1`;
  - `AuthorityActionV1`;
  - `AuthorityReconciliationStateV1`.
- Added a dry-run authority reconciliation planner that consumes the existing
  0.41 `DeploymentCheckV1` instead of creating a second inventory or drift
  model.
- Added read-only `canic deploy authority check <fleet>` to print
  `AuthorityReconciliationPlanV1` JSON without applying controller changes.

### Authority semantics

- Already-correct controller sets are reported as `AlreadyCorrect`.
- Deployment-controlled controller deltas are reported as
  `CanApplyAutomatically` for a later apply path, but 0.42.0 does not mutate
  controller state.
- Pool, imported, jointly controlled, and user-controlled canisters are
  reported as external-action cases rather than silently treated as managed
  deployment canisters.
- Unknown unsafe canisters are reported as blocked authority findings.

### Validation

- Added focused host tests for already-correct authority, automatic dry-run
  controller deltas, external-action classification, and unsafe authority
  blocking.
- Added CLI parser coverage for `canic deploy authority check`.
