# Canic 0.38

## 0.38.0 - 2026-05-17 - Stable memory ABI hard cut

### ⚠️ Breaking

- Treats stable-memory allocation as durable ABI: the stable key, not owner
  crate, Rust type, or local label, is the persistent memory identity.
- Reserves the Canic framework range as `0-99`, reserves downstream
  application memory as `100-254`, and makes ID `255` permanently invalid as
  the unallocated-bucket sentinel.
- Makes ID `0` the canonical `canic.memory.abi_ledger.v1` ledger self-record.
  IDs `1-99` are framework expansion budget unless later allocated by explicit
  Canic framework declarations.

### 🔧 Changed

- Widened `canic-core`'s bootstrap range to `11-99` while keeping
  `canic-control-plane` in the `5-10` planning subrange.
- Added current runtime declaration-snapshot validation so duplicate memory
  IDs, duplicate stable keys, and exact duplicate declarations fail before user
  ledger records are committed during bootstrap.
- Added historical-ledger preflight for pending bootstrap declarations so a
  later stable-key or ID conflict cannot leave earlier user claims written into
  the ledger during a failed bootstrap.
- Reworked the persisted layout ledger into a generation-framed store with two
  committed slots, generation checksums, header metadata, and highest-valid
  generation selection so corrupted newer slots are ignored.
- Made ledger mutation, validation, and diagnostic snapshot reads fail closed
  when no committed generation validates.
- Tightened stable-key namespace enforcement so non-Canic crates cannot claim
  `canic.*` keys even if they choose IDs inside the framework range.
- Split startup declaration from memory opening in the public `MemoryApi`:
  `declare_with_key(...)` queues metadata before bootstrap, while
  `register_with_key(...)` only opens already-validated slots after bootstrap.
- Split `ic_memory_key!` macro declaration from opening: constructor-time
  declaration descriptors are collected before registry validation, while
  eager stable-store initialization opens virtual memory only after bootstrap
  publishes the validated runtime registry.
- Made macro-backed stable-memory opening enforce bootstrap readiness on host
  and wasm targets, backed by a host-test bootstrap hook and process-wide
  registry initialized state so tests cannot bypass the validation-before-open
  rule.
- Added a `MemoryApi::ledger_snapshot()` diagnostic read path for persisted
  ABI ledger history that does not depend on current runtime registry
  reconstruction.
- Updated `canic-memory` guidance to document the `0-99`/`100-254` split and
  the corrected ID `255` sentinel semantics.

### 🧪 Testing

- Added `canic-memory` coverage for duplicate current declarations, duplicate
  stable keys, exact duplicate declarations, the ID `0` self-record, ID `255`
  rejection, non-Canic attempts to claim the `canic.*` namespace, declaration
  before opening, opening before bootstrap rejection, late declaration rejection
  after bootstrap, and failed bootstrap behavior that leaves non-conflicting
  sibling declarations out of historical ledger state.
- Added host-test bootstrap hooks for `canic-core` and `canic-control-plane`
  so unit tests validate the stable-memory registry before first macro-backed
  store access instead of opening handles directly.
- Added direct ledger recovery tests for highest-valid-generation selection and
  corrupted-newer-slot recovery, plus fail-closed behavior when no committed
  slot validates.
- Added a framework source guard that rejects implicit `ic_memory!` and
  `MemoryApi::register(...)` use in framework-owned crates.
- Extended the framework source guard to reject direct raw stable-memory APIs,
  independent `MemoryManager` access, and `RestrictedMemory` carve-outs in
  framework-owned crates.
- Ran `cargo test -p canic-memory`.
- Ran `cargo test -p canic-core -p canic-control-plane`.
