## 0.48.8 - 2026-05-29 - token trust-chain audit

### Audit

- Reran `token-trust-chain` against the current 0.48 tree.
- Delegated-token verification still requires verifier-local root trust,
  root-certified shard authority, canonical certificate and claims hashes, and
  shard signatures before endpoint authorization can proceed.
- Endpoint authorization still verifies token material before subject binding,
  required-scope checks, update replay consumption, and handler dispatch.
- Role-attestation verification still uses cached trusted keys and refreshes
  only on unknown key id; subject, audience, epoch, expiry, and signature
  failures remain fail-closed without broad fallback.

## 0.48.6 - 2026-05-29 - auth freshness audit cleanup

### Fixed

- Delegated grants now reject at `now_secs >= expires_at` instead of accepting
  the exact expiry timestamp.
- Role attestations and internal invocation proofs now reject at
  `now_secs >= expires_at` instead of accepting the exact expiry timestamp.
- Added exact-boundary regression tests for delegated grants, role
  attestations, and internal invocation proofs.

### Audit

- Reran the oldest tied recurring audit,
  `subject-caller-binding`, against the current 0.48 tree.
- The subject-caller audit remained comparable with the 2026-05-09 baseline
  and found no blocker or code remediation.
- Delegated-token subject binding still routes through the canonical
  access-auth verifier after token verification and before required-scope
  enforcement, update-token consumption, and handler execution.
- Generated endpoint wrappers still preserve the separate transport-caller and
  authenticated-subject lanes in `AccessContext`.
- The private `AuthApi::verify_token_material(...)` helper remains private and
  documented as incomplete for endpoint authorization.
- Reran the next oldest tied recurring audit,
  `expiry-replay-single-use`, against the current 0.48 tree.
- The audit confirmed the earlier replay-boundary fixes still hold for
  delegated tokens, root replay metadata, existing replay records,
  consumed-token state, query-token statelessness, and per-caller replay
  capacity ordering.
- Post-remediation risk is `3 / 10`; no remaining blocker was found.

## 0.48.1 - 2026-05-28 - package metadata role wiring

### Breaking

- `canic::build!` now requires `[package.metadata.canic] role = "..."` in each
  canister crate manifest.
- The old public build macro variants were removed. Canister build scripts now
  use only `canic::build!`.
- Non-root canisters now call `canic::start!()` without a role argument. The
  role comes from the package metadata emitted by `canic::build!`.
- Root canisters now call `canic::start!()` too. `role = "root"` in package
  metadata selects the root lifecycle adapters and root endpoint bundle.
- Local sandbox canisters now call `canic::start_local!()` without a role
  argument for the same reason.

### Changed

- Root vs non-root build behavior is selected from package role metadata:
  `role = "root"` emits root bootstrap artifacts, while ordinary roles emit
  the compile-time role consumed by `start!()`.
- The checked-in test and demo fleets now use shared fleet-level build scripts
  instead of duplicate per-canister `build.rs` files.
- Generated scaffold output now writes package role metadata and the simplified
  `canic::build!` / `canic::start!()` canister shape.
- Local-only endpoint guards in the test fleet now share
  `canic::access::require_local()` instead of repeating the same environment
  check in each canister.
- Build-support tests now cover missing role metadata and metadata role typos.
- The facade crate no longer carries a root-only example that required its
  package build script to emit canister role/config environment.

### Migration Notes

- Add package role metadata to every canister crate:

```toml
[package.metadata.canic]
role = "app"
```

- Replace non-root lifecycle setup like:

```rust
use canic::ids::CanisterRole;

const APP: CanisterRole = CanisterRole::new("app");

canic::start!(APP);
```

with:

```rust
canic::start!();
```

- Replace root lifecycle setup with `canic::start!()` too, while keeping root
  hook signatures:

```rust
canic::start!();

async fn canic_setup() {}
async fn canic_install() {}
async fn canic_upgrade() {}
```

- Replace removed build macro variants with `canic::build!("<config-path>")`.

## 0.48.0 - 2026-05-28 - derived singleton topology

### Breaking

- `subnets.<name>.auto_create` is no longer accepted in `canic.toml`.
- `subnets.<name>.subnet_index` is no longer accepted in `canic.toml`.
- Existing fleet configs must remove those keys. With the hard cut, unknown
  field validation now rejects them instead of treating them as compatibility
  aliases.

### Changed

- Canic now derives root auto-create roles from configured
  `kind = "singleton"` canisters in each subnet.
- Canic now derives subnet-index roles from the same singleton role set.
- Root bootstrap, subnet-index rebuilds, host config inspection, local root
  cycle estimation, scaffold output, checked-in fleet configs, and getting
  started docs now use the derived singleton topology.

### Migration Notes

- To auto-create and index a stable subnet service, declare only the canister:

```toml
[subnets.prime.canisters.user_hub]
kind = "singleton"
```

- `kind = "shard"`, `kind = "replica"`, and `kind = "instance"` roles are not
  auto-created or subnet-indexed directly; their placement managers create
  them.
