# 0.45 Changelog

## 0.45.3 - 2026-05-26 - pending lifecycle reports and receipt proposal linkage

### Added

- Added `ExternalLifecyclePendingReportV1` and
  `ExternalLifecyclePendingActionV1` to summarize directly executable, pending
  external, blocked, and residual-exposure work from lifecycle plan and
  proposal artifacts.
- Added `external_lifecycle_pending_report_from_plan(...)`,
  `validate_external_lifecycle_pending_report(...)`, and
  `validate_external_lifecycle_pending_report_for_plan(...)` so archived
  pending reports reject stale counts, digest drift, duplicate pending
  subjects, and source artifact drift.
- Added host-owned passive text rendering for external lifecycle pending
  reports.
- Added `canic deploy external pending <fleet>` to expose passive pending
  lifecycle reports from the CLI as JSON by default or host-owned text with
  `--format text`.

```bash
canic deploy external pending demo
canic deploy external pending --format text demo
```

### Changed

- Strengthened external-upgrade receipt validation with
  `validate_external_upgrade_receipt_for_proposal(...)`, which verifies that a
  receipt matches the exact proposal ID, proposal digest, subject, before-state
  facts, target verification result, and verification notes it claims to
  satisfy.
- Updated the external lifecycle CLI help and source guards so `pending` is
  documented as a passive, non-mutating report.
- Updated the current status handoff to record the pending report and
  proposal-linked receipt validation as the current 0.45 baseline.

### Validation

- Added CLI parser/help/source-guard coverage for the external lifecycle
  `pending` command.
- Added CLI builder coverage for pending report linkage to lifecycle plan and
  proposal report artifacts.
- Added host-side coverage for pending report summary counts, stale source
  rejection, passive text rendering, and JSON round-tripping.
- Added host-side coverage for receipt validation against mismatched proposal
  sources and stale proposal targets.

### Testing

- `cargo test -p canic-cli external_ --lib --offline`
- `cargo test -p canic-host external_lifecycle_pending_report --lib --offline`

- `cargo clippy -p canic-cli --all-targets --offline -- -D warnings`

## 0.45.1 - 2026-05-26 - passive lifecycle artifact hardening

### Added

- Added deterministic `report_digest` fields for
  `LifecycleAuthorityReportV1` and `ExternalUpgradeProposalReportV1`.
- Added `validate_lifecycle_authority_report(...)` for lifecycle authority
  report schema, required-field, duplicate-subject, counter, and digest
  validation.
- Added `validate_external_lifecycle_plan(...)` and
  `validate_external_lifecycle_plan_for_check(...)` so archived lifecycle
  plans reject stale digests, status drift, duplicate subjects, and source
  deployment-truth drift.
- Added `validate_external_upgrade_proposal_report(...)` and
  `validate_external_upgrade_proposal_report_for_lifecycle_plan(...)` so
  archived proposal reports reject stale report/proposal digests, direct-row
  proposal mistakes, duplicate proposal subjects, and source lifecycle/check
  drift.
- Added passive text renderers for lifecycle authority reports, lifecycle
  plans, external-upgrade proposal reports, and external-upgrade receipts.

### Changed

- Strengthened `validate_external_upgrade_receipt(...)` so external lifecycle
  receipts reject stale receipt digests while still treating live inventory as
  completion truth.
- Updated the 0.45 status handoff to record digest/text parity and source
  linkage validation as the current passive artifact baseline.

### Validation

- Added host-side coverage for lifecycle authority report digest/counter drift,

- Extended the deployment-truth no-controller-mutation source guard to include

## 0.45.0 - 2026-05-26 - lifecycle authority projection

### Added

- Added `LifecycleAuthorityReportV1`, `LifecycleAuthorityV1`,
  `LifecycleUpgradeModeV1`, and `LifecycleVerificationRequirementV1` as the
  first passive external-lifecycle DTO surface.
- Added `lifecycle_authority_report_from_check(...)` to project an existing
  `DeploymentCheckV1` into lifecycle authority rows without introducing a new
  controller-classification model.
- Added `ExternalUpgradeProposalReportV1`, `ExternalUpgradeProposalV1`, and
  `ExternalUpgradeAuthorizationModeV1` as the first passive proposal artifact
  shape for externally controlled lifecycle rows.
- Added `ExternalLifecyclePlanV1` and related role/status DTOs to partition
  lifecycle authority rows into directly executable, externally proposed, and
  blocked upgrades before proposal generation.
- Added `external_lifecycle_plan_from_check(...)` and
  `external_upgrade_proposal_report_from_lifecycle_plan(...)` to bind
  lifecycle authority, current observed module/config facts, target
  artifact/config facts, consent requirements, plan/proposal digests, and
  authorization modes without granting consent or executing upgrades.
- Added `ExternalUpgradeReceiptV1`, `ExternalUpgradeConsentStateV1`, and
  `ExternalUpgradeVerificationResultV1` as the first passive external
  lifecycle receipt shape.
- Added `external_upgrade_receipt_from_observation(...)` and
  `validate_external_upgrade_receipt(...)` to record structural verification
  of external completion evidence while keeping live inventory as truth.

### Changed

- Marked 0.45 as the active design line in the current status handoff.
- Updated the 0.45 status log from not-started to the first passive lifecycle
  authority slice.

### Validation

- Added host-side projection and JSON-shape coverage for
  deployment-controlled, user-controlled, and unknown-unsafe lifecycle
  authority rows.
- Added host-side coverage proving direct deployment-authority rows do not
  produce external proposals and unknown-unsafe rows become blocked subjects.
- Added host-side coverage for verified external completion, mismatched
  external completion, contradictory refused/verified receipts, and receipt
  JSON shape stability.

### Testing

- Not run in this slice; maintainer is running the full validation lane in a
  separate window.
