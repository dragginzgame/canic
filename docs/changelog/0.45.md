# 0.45 Changelog

## 0.45.5 - 2026-05-26 - lifecycle checks and operator handoff packets

### Added

- Added `ExternalLifecycleCheckV1` to summarize lifecycle plan, proposal, and
  pending report evidence into one passive status artifact. It carries
  source artifact IDs/digests, direct/pending/blocked counts,
  residual-exposure counts, summary text, next actions, and a deterministic
  check digest.
- Added `external_lifecycle_check_from_reports(...)`,
  `validate_external_lifecycle_check(...)`, and
  `validate_external_lifecycle_check_for_reports(...)` so archived checks
  reject stale source reports, count drift, missing IDs, and digest drift.
- Added host-owned passive text rendering for external lifecycle checks.
- Added `canic deploy external check <fleet>` to expose passive lifecycle
  checks from local deployment truth as JSON by default or host-owned text
  with `--format text`.
- Added `ExternalLifecycleHandoffV1` and
  `ExternalLifecycleHandoffActionV1` to package pending external proposals
  into passive operator coordination instructions. Handoff packets carry
  check/proposal/pending digests, consent channel and subject facts, target
  verification facts, blocked subjects, residual exposure, and deterministic
  handoff digests without delivering consent or executing external upgrades.
- Added `external_lifecycle_handoff_from_reports(...)`,
  `validate_external_lifecycle_handoff(...)`, and
  `validate_external_lifecycle_handoff_for_reports(...)` so archived handoff
  packets reject stale source evidence, duplicate subjects, missing IDs, and
  digest drift.
- Added host-owned passive text rendering for external lifecycle handoff
  packets.
- Added `canic deploy external handoff <fleet>` to expose passive handoff
  packets from local deployment truth as JSON by default or host-owned text
  with `--format text`.

```bash
canic deploy external check demo
canic deploy external check --format text demo
canic deploy external handoff demo
canic deploy external handoff --format text demo
```

### Changed

- Updated the 0.45 status log and current status handoff to record lifecycle
  checks and handoff packets as passive structural evidence only.
- Strengthened external lifecycle CLI help so `check` and `handoff` are
  documented as non-mutating report surfaces that do not request consent,
  execute external upgrades, install code, or mutate deployment state.

### Validation

- Added host-side coverage for lifecycle check summary counts, stale source
  rejection, passive text rendering, digest validation, and JSON
  round-tripping.
- Added host-side coverage for lifecycle handoff packet generation, stale
  source rejection, passive text rendering, digest validation, and JSON
  round-tripping.
- Added CLI parser/help/source-guard coverage for `check` and `handoff`.
- Added CLI builder coverage proving check and handoff artifacts are derived
  from the same lifecycle plan, proposal report, and pending report chain.

## 0.45.4 - 2026-05-26 - critical fix and external verification evidence

### Added

- Added `CriticalExternalFixReportV1` to summarize directly patchable roles,
  externally blocked roles, dependency-blocked roles, required external
  actions, protected-call implications, residual exposure, and operator next
  steps from lifecycle pending evidence without claiming deployment
  completion.
- Added `critical_external_fix_report_from_pending(...)`,
  `validate_critical_external_fix_report(...)`, and
  `validate_critical_external_fix_report_for_pending(...)` so archived
  critical-fix reports reject stale source evidence, missing IDs, digest
  drift, and count/source mismatches.
- Added host-owned passive text rendering for critical external-fix reports.
- Added `canic deploy external critical-fix <fleet>` to expose passive
  critical-fix reports from local deployment truth as JSON by default or
  host-owned text with `--format text`.
- Added `ExternalUpgradeVerificationReportV1` and
  `ExternalUpgradeVerificationReportRequest` to package validated
  proposal/receipt pairs into digest-pinned passive verification artifacts.
- Added `external_upgrade_verification_report_from_receipt(...)`,
  `validate_external_upgrade_verification_report(...)`, and
  `validate_external_upgrade_verification_report_for_receipt(...)` so archived
  verification reports reject stale source evidence, digest drift, and
  proposal/receipt mismatches.
- Added `canic deploy external verify --request <file>` to build passive
  `ExternalUpgradeVerificationReportV1` artifacts from request-file
  proposal/receipt evidence without live lookup, consent delivery, external
  execution, install, or mutation.
- Added `ExternalUpgradeConsentEvidenceV1` and
  `ExternalUpgradeConsentEvidenceRequest` to keep reported consent/action
  state distinct from verification evidence.
- Added `external_upgrade_consent_evidence_from_receipt(...)`,
  `validate_external_upgrade_consent_evidence(...)`, and
  `validate_external_upgrade_consent_evidence_for_receipt(...)` so archived
  consent evidence rejects stale proposal/receipt linkage and digest drift.
- Added `canic deploy external inspect consent --request <file>` as an
  advanced passive inspection command for consent evidence, avoiding another
  top-level lifecycle workflow command.

```bash
canic deploy external critical-fix --fix-id fix-2026-05 --severity critical demo
canic deploy external critical-fix --fix-id fix-2026-05 --severity critical --format text demo
canic deploy external inspect consent --request external-consent.json
canic deploy external inspect consent --request external-consent.json --format text
canic deploy external verify --request external-verification.json
canic deploy external verify --request external-verification.json --format text
```

### Changed

- Updated the 0.45 status log and current status handoff to record the
  critical-fix, consent-evidence, and verification-report surfaces as passive
  structural evidence only.
- Kept external consent evidence under `deploy external inspect` so the public
  operator workflow stays centered on lifecycle plan/proposal/pending/check
  surfaces rather than every DTO becoming a first-class command.
- Strengthened CLI help to state that verification reports package structural
  proposal/receipt evidence only and that live inventory remains deployment
  truth.

### Validation

- Added host-side coverage for critical-fix report packaging, stale source
  rejection, passive text rendering, and digest validation.
- Added host-side JSON shape coverage for external verification request/report
  artifacts and external consent evidence request/evidence artifacts.
- Added host-side coverage proving consent evidence is separate from
  verification evidence and rejects stale source proposal/receipt linkage.
- Added CLI parser/help/source-guard coverage for `critical-fix`, `verify`,
  and `inspect consent`.
- Added CLI builder coverage proving `verify` and `inspect consent` delegate
  to host proposal/receipt validation rather than echoing request content.

## 0.45.0 - 2026-05-26 - lifecycle authority projection

### Added

- Added `LifecycleAuthorityReportV1`, `LifecycleAuthorityV1`,
  `LifecycleUpgradeModeV1`, and `LifecycleVerificationRequirementV1` as the
  first passive external-lifecycle DTO surface.
- Added `lifecycle_authority_report_from_check(...)` to project an existing
  `DeploymentCheckV1` into lifecycle authority rows without introducing a new
  controller-classification model.
- Added `ExternalUpgradeProposalReportV1`, `ExternalUpgradeProposalV1`, and
  `ExternalUpgradeAuthorizationModeV1` as the first passive proposal artifact
  shape for externally controlled lifecycle rows.
- Added `ExternalLifecyclePlanV1` and related role/status DTOs to partition
  lifecycle authority rows into directly executable, externally proposed, and
  blocked upgrades before proposal generation.
- Added `external_lifecycle_plan_from_check(...)` and
  `external_upgrade_proposal_report_from_lifecycle_plan(...)` to bind
  lifecycle authority, current observed module/config facts, target
  artifact/config facts, consent requirements, plan/proposal digests, and
  authorization modes without granting consent or executing upgrades.
- Added `ExternalUpgradeReceiptV1`, `ExternalUpgradeConsentStateV1`, and
  `ExternalUpgradeVerificationResultV1` as the first passive external
  lifecycle receipt shape.
- Added `external_upgrade_receipt_from_observation(...)` and
  `validate_external_upgrade_receipt(...)` to record structural verification
  of external completion evidence while keeping live inventory as truth.

### Changed

- Marked 0.45 as the active design line in the current status handoff.
- Updated the 0.45 status log from not-started to the first passive lifecycle
  authority slice.

### Validation

- Added host-side projection and JSON-shape coverage for
  deployment-controlled, user-controlled, and unknown-unsafe lifecycle
  authority rows.
- Added host-side coverage proving direct deployment-authority rows do not
  produce external proposals and unknown-unsafe rows become blocked subjects.
- Added host-side coverage for verified external completion, mismatched
  external completion, contradictory refused/verified receipts, and receipt
  JSON shape stability.
