# 0.43 Changelog

## 0.43.8 - 2026-05-24 - current-install runner bridge closeout

### Added

- Added a private `InstallPhaseOperation` boundary for current-install
  activation phases.
- Added `InstallReceiptScope::run_operation(...)` so activation phases execute
  through one phase/action/evidence runner instead of manual `run_phase`
  closure wiring.
- Added source-guard coverage proving activation phases use the private
  operation runner.
- Added source-guard coverage proving current-install activation runs only
  after deployment-truth and execution-preflight readiness have been recorded.

### Changed

- Current-install root wasm installation, root funding, release-set staging,
  bootstrap resume, and readiness waiting now enter receipt writing through
  the private operation runner.
- The 0.43 status now distinguishes guarded current-install behavior from
  future non-current-install backend enforcement.
- The 0.44 artifact-promotion design now depends on the 0.43 runner/gate path
  before promoted plans can execute through the existing install command.
- Receipt-backed promotion language now clarifies that receipts may identify
  artifact bytes but must not prove live deployment state by themselves.

### Notes

- This is the final 0.43 runner bridge slice. It preserves current installer
  behavior and does not add controller apply, a new executor backend, or a
  public execution receipt JSON contract.
- Future non-current-install backends still need their own deployment-truth
  gate enforcement and source guards.

### Testing

## 0.43.5 - 2026-05-24 - deployment receipt status hardening

### Added

- Added shared deployment receipt status classification for generic receipt
  construction.
- Added focused coverage for failed execution status classification:
  `FailedBeforeMutation`, `FailedAfterMutation`, and `PartiallyApplied`.
- Added resume-safety coverage proving receipts with contradictory execution
  status and role-phase evidence are blocked.

### Changed

- `deployment_receipt_from_check(...)` now derives failed execution status
  from the command result and role-phase evidence instead of treating every
  failed command as post-mutation.
- Failed receipts now classify as:
  - `FailedBeforeMutation` when no role-phase evidence shows applied work;
  - `FailedAfterMutation` when role evidence shows applied work but no failed
    role;
  - `PartiallyApplied` when role evidence shows both applied and failed role
    work.
- Receipt-aware resume checks now reject persisted receipts whose claimed
  `operation_status` contradicts the command result and role-phase evidence.

### Notes

- Current-install code can still supply an explicit status for phases whose
  mutation boundary is not fully represented by role-phase receipts yet.
- This is receipt semantics hardening only. It does not add a new executor
  backend, controller mutation, or a public execution-receipt JSON contract.

### Testing

  classifier.

  construction derives `PartiallyApplied` from mixed role outcomes.

  execution status.

- Deployment truth and execution preflight now share current-install input
  resolution for workspace root, ICP root, config path, and fleet name.

  patch does not add a public CLI command, controller mutation, or a new

## 0.43.0 - 2026-05-24 - executor boundary scaffold

### Added

- Added the first backend-agnostic execution context model:
  `DeploymentExecutionContextV1`, `DeploymentExecutorBackendV1`, and
  `DeploymentExecutorCapabilityV1`.
- Added a minimal `DeploymentExecutor` trait and current-CLI capability helper
  so later installer extraction can reason about backend capabilities without
  rediscovering deployment intent.
- Added a concrete current-CLI executor wrapper and routed current-install
  execution context construction through that executor object.
- `DeploymentReceiptV1` can now carry optional execution context metadata,
  letting receipts record the backend, workspace root, ICP root, artifact
  roots, and supported executor capabilities without changing current install
  behavior.
- Added passive `DeploymentExecutionPreflightV1` readiness output over
  `DeploymentPlanV1`, `SafetyReportV1`, `AuthorityReconciliationPlanV1`, and
  executor capabilities, so 0.43 has a plan-shaped execution gate before
  backend operations run.

### Changed

- Current-install deployment truth receipts now attach current-CLI execution
  context metadata when they are written by the install flow. This records the
  workspace root, ICP root, build artifact roots, backend, and backend
  capabilities as receipt evidence without moving mutation behind the executor
  yet.
- Current install now checks that the selected backend advertises the
  capabilities required by the existing install phases before those phases
  begin mutating deployment state.

### Documentation

```bash
canic deploy plan <fleet>
canic deploy check <fleet>
canic deploy authority check <fleet>
```

### Notes

- This opening slice does not move current install mechanics behind the
  executor yet. It establishes the receipt/context vocabulary that later 0.43
  slices will use.
- The current CLI backend remains the only implemented backend.
- `DeploymentExecutionPreflightV1` is passive readiness evidence. It does not
  execute controller changes or replace deployment truth and authority checks.

### Testing

- Added deployment-truth unit coverage for current-CLI execution context JSON
  round-tripping and deterministic missing-capability reporting.
- Added deployment-truth unit coverage for ready and blocked execution
  preflight results using the same `DeploymentPlanV1` shape as deployment truth.
- Added install-truth unit coverage proving receipted current-install phases
  include current-CLI execution context metadata.
- Added install-truth unit coverage for current-install required capability
  reporting.
