# 0.82 Changelog

## 0.82.5 - 2026-07-05 - Memory diagnostic domain ownership

### 📝 Summary

- Continues the 0.82 no-behavior-change DTO/domain boundary cleanup.

- Moves memory diagnostic semantic values into domain ownership while keeping
  memory ledger DTO structs as the serialized query response boundary.

- Preserves public DTO re-export paths and serialized Candid shapes for the
  moved values.

### 🧹 Cleanup

- Memory commit recovery error responses are now owned by the memory domain
  module and re-exported from the memory DTO module.

- Memory range authority modes are now owned by the memory domain module and
  re-exported from the memory DTO module.

- Memory allocation states are now owned by the memory domain module and
  re-exported from the memory DTO module.

- Runtime memory ops now import those semantic values from the domain owner
  while leaving memory ledger response structs in DTO ownership.

### 🧭 Migration Notes

- This patch adds no new operator commands.

- This patch adds no backwards-compatibility aliases, shims, hidden routes, or
  alternate command forms.

- Existing Rust imports through `dto::memory` remain available through
  re-exports.

- This patch does not change Candid method names, endpoint behavior, JSON
  report fields, stable state layout, memory ledger format, deployment truth
  schema, evidence/report schemas, or runtime behavior.

## 0.82.2 - 2026-07-05 - Release safety and DTO boundary ownership

### 📝 Summary

- Adds an interactive command-line confirmation guard before minor and major
  release bump targets run release gates or write version files.

- Continues the 0.82 boundary-hardening line by moving internal runtime value
  semantics out of DTO ownership and into domain ownership.

- Preserves public DTO re-export paths and serialized Candid shapes for the
  moved values.

### 🔧 Changed

- `make minor` now requires typing `minor` before it runs `ensure-clean`,
  formatting, the bump test gate, or the minor version bump.

- `make major` now requires typing `major` before it runs `ensure-clean`,
  formatting, clippy, the full test gate, or the major version bump.

- One-shot release targets that depend on those bump targets inherit the same
  guard.

```bash
make minor
make major
make release-minor
make release-major
```

### 🧹 Cleanup

- ICP refill operation status and error-code values are now owned by the
  domain layer and re-exported from the DTO module.

- Root runtime subnet identity values are now owned by the domain layer and
  re-exported from the DTO module used by the macro/init Candid boundary.

- Internal view, storage, workflow, and metrics code imports those values from
  their domain owner instead of treating DTO modules as semantic owners.

### 🧭 Migration Notes

- This patch adds no new operator commands.

- This patch changes release-tooling behavior for existing minor/major bump
  commands by requiring explicit interactive confirmation.

- This patch does not change Candid method names, endpoint behavior, JSON
  report fields, stable state layout, deployment truth schema, evidence/report
  schemas, or runtime behavior.

- Existing Rust imports through `dto::icp_refill` and `dto::subnet` remain
  available through re-exports.

## 0.82.1 - 2026-07-05 - Pure policy namespace boundary

### 📝 Summary

- Continues the 0.82 boundary-hardening line by making the pure-policy
  namespace explicit.

- Moves core policy modules under `domain::policy::pure` so the code layout
  mirrors the architectural rule that policy is pure decision logic.

- Moves policy-owned input and decision shapes out of `view/`, keeping
  `view/` reserved for internal read-only projections rather than policy input
  contracts.

### 🔧 Changed

- Core policy callers now import pure policy through
  `domain::policy::pure::*` or a local `policy` alias that points at the pure
  namespace.

- Environment validation, topology registry inputs, scaling worker plan
  entries, and sharding placement/partition/plan shapes now live with the
  policy modules that own their semantics.

- Ops and workflow modules continue to perform conversion and orchestration,
  but they now hand policy-owned inputs to policy rather than importing those
  inputs from `view/`.

### 🧹 Cleanup

- Removed obsolete `view/` modules that had become policy input containers
  rather than read-only projections.

- Removed the 0.82 legacy-command sweep artifact and retired-command test
  fixtures so the codebase does not preserve old command vocabulary as a
  compatibility contract.

- Kept the existing active command surface unchanged and did not add
  backwards-compatibility aliases, shims, hidden routes, or alternate command
  forms.

### 🧭 Migration Notes

- This patch adds no new operator commands.

- This patch removes no active operator commands.

- This patch adds no backwards-compatibility aliases, shims, or tests for
  retired command forms.

- This patch does not change serialized DTOs, Candid methods, JSON report
  fields, stable state layout, deployment truth schema, evidence/report
  schemas, runtime endpoints, or runtime behavior.

## 0.82.0 - 2026-07-05 - Boundary hardening foundation

### 📝 Summary

- Starts 0.82 as a controlled boundary-hardening line rather than a broad
  architecture cleanup pass.

- Defines the slice contract, hardening report model, severity rubric,
  behavior-impact labeling, and public/serialized surface freeze rules used to
  keep cleanup work reviewable and no-behavior-change by default.

- Completes the first hardening slices for diagnostic/report ownership and
  policy-purity import ownership.

### 🔧 Changed

- The 0.82 design now names diagnostic ownership hardening as the first slice
  and records cleanup safety rules for future slices: no incidental command
  changes, endpoint changes, JSON/schema changes, stable-state changes, or
  behavior changes unless a separate slice contract declares them.

- Diagnostic ownership is documented in a closeout report so report producers
  own diagnostic semantics while CLI renderers remain formatting and exit-code
  consumers.

- Policy helper modules now expose narrower dependency surfaces:
  deployment-truth policy helpers no longer rely on parent wildcard imports,
  core policy modules import shared pure value types through
  `crate::domain::value`, and policy decisions consume policy-owned input
  structs instead of serialized config schema structs.

- Workflow and test boundaries now map config into policy input shapes before
  calling log retention, cycle top-up, ICP refill, scaling, sharding, and
  topology registry policy.

### 🧹 Cleanup

- Added policy-owned input/rules/shape types for log retention, cycle top-up,
  ICP refill, scaling, sharding, and registry topology decisions.

- Split topology registry policy from `CanisterConfig` / `CanisterKind` by
  introducing pure registry canister shape inputs.

- Kept policy behavior, CLI behavior, endpoint surfaces, JSON shapes, stable
  memory layout, deployment truth, and evidence/report schemas unchanged.

### 🧭 Migration Notes

- This patch adds no new operator commands.

- This patch adds no backwards-compatibility aliases, shims, or alternate
  command surfaces.

- This patch does not change serialized DTOs, Candid methods, JSON report
  fields, stable state layout, deployment truth schema, or runtime endpoint
  behavior.

- The remaining 0.82 follow-up is to replace ad hoc import scans with a robust
  dependency-boundary check before making policy-purity import rules a release
  gate.
