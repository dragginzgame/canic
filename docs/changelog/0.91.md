# Canic 0.91 Changelog

## [0.91.6] - 2026-07-14 - Canonical Live Output and Shared Support Ownership

### 📝 Summary

0.91.6 makes the shipped live `cycles convert --json` behavior agree with the
direct verified-result contract documented for 0.91.5. The live path no longer
retains the old command-context wrapper or serializes the verified result into
an `icp_output` string. It prints the six maintained response fields directly
and renders `cycles_sent` as unformatted base-10 text for stable automation.

The patch also removes redundant internal ownership found in the post-0.91.5
cleanup. Deployment commands now share one JSON/text selection and rendering
path, while backup and restore share canonical lock, timestamp, and artifact
path helpers. No alias, duplicate output path, compatibility wrapper, or
fallback is retained.

### 🩹 Fixed

- Emits successful live conversion JSON as the verified refill response with
  `operation_id`, `status`, `ledger_block_index`, `cycles_sent`, `error_code`,
  and `error_message` at the top level.
- Serializes arbitrary-precision `cycles_sent` values as plain decimal digits
  rather than Candid display text with digit separators.
- Constructs the display-only canister command only for dry runs. Fabrication
  and dry-run output retain their separate maintained schemas.
- Removes a newly created journal sidecar lock if writing its ownership marker
  fails, preserving the underlying I/O error instead of leaving a stale lock.

### 🧹 Cleanup

- Replaces six parallel deployment JSON/text enums, six local flag converters,
  and their repeated stdout branches with one internal selector and renderer.
  The deployment-check evidence-envelope format remains a distinct type.
- Moves backup and restore journal exclusion to one persistence-owned sidecar
  lock while preserving their existing typed public error projections.
- Gives backup timestamps and artifact path segments one owner each. Direct
  snapshot capture and resumable backup execution continue as intentionally
  separate workflows.

### 📊 Audit

- Reduces six two-variant deployment output types to one and retains only the
  genuinely different three-variant deployment-check format.
- Reduces two journal lock implementations, two state-timestamp default
  helpers, and two artifact path sanitizers to one implementation apiece.
- Deletes the obsolete backup-runner support module after moving each of its
  remaining responsibilities to its canonical layer.

### ⚠️ Breaking

- Automation that followed the accidentally retained 0.91.5 live wrapper must
  now consume the direct verified-result fields. `mode`, deployment/request
  context, `command`, and `icp_output` are not emitted for a successful live
  conversion, and no legacy wrapper alias is available.
- This patch does not change command syntax, ICP-refill Candid contracts,
  stable schemas, operation identity, dry-run/fabrication JSON, or public Rust
  backup error variants.

### 🧭 Migration Notes

- Read live conversion fields directly from the top-level JSON object and
  treat `cycles_sent` as a base-10 string. Do not parse nested JSON from
  `icp_output`.
- No backup data migration or deployment command update is required; those
  changes remove internal duplication without changing maintained output.

### 🧪 Testing

- Four focused refill-response tests prove typed rejection, operation binding,
  resumability, and the exact direct JSON projection.
- All 96 focused deployment command tests pass across JSON/text defaults,
  flags, dispatch, and report construction.
- Focused backup and restore tests prove shared lock exclusion and typed lock
  rejection, pending recovery, timestamp derivation, and artifact path
  sanitization.
- Targeted checks, warning-denied all-target Clippy, formatting, changelog
  governance, and diff hygiene pass for the affected packages and documents.

## [0.91.5] - 2026-07-14 - Atomic ICP Refill Recovery

### 📝 Summary

0.91.5 closes concurrency and recovery gaps in ICP-to-cycles refill. Durable
storage now owns the final same-key admission decision, workflow policy is
re-evaluated after asynchronous preflight, and unavailable canonical network
identity or child-funding configuration fails closed instead of selecting or
skipping policy implicitly.

Live `canic cycles convert` now verifies the typed canister result and its
operation ID before deciding whether retry identity may be released. Its
project-local pending log serializes read/modify/write transitions, retains
resumable operations, and reports completion-persistence failures rather than
claiming success without durable recovery state.

The patch does not change ICP-refill Candid contracts, stable record schemas,
command syntax, or operation-key derivation. It intentionally changes live
command output and several previously permissive failure paths.

### 🩹 Fixed

- Makes refill record creation atomically reject a distinct active operation
  for the same source, subaccount, and target. Retryable transfer or notify
  failures remain active until a terminal outcome, while an exact operation-ID
  retry still resumes its stored operation.
- Rebuilds and evaluates manual policy state after asynchronous ledger fee,
  decimals, and conversion-rate reads. Storage revalidates same-key admission
  at record creation, closing the remaining race between preflight and durable
  mutation.
- Rejects an unavailable build network and propagates missing direct-child
  funding policy as typed internal failures instead of defaulting to local or
  silently treating the cooldown as absent.
- Gives runtime and CLI one resumability classification for requested,
  transferred, notify-processing, retryable bad-fee, and retryable notify
  outcomes.
- Serializes pending-operation reservation and completion with a project-local
  advisory lock while retaining atomic replacement and directory sync.
- Decodes live call output from raw Candid bytes, rejects typed endpoint
  errors, unexpected dry-run responses, malformed output, and operation-ID
  mismatches, and releases a generated pending ID only after a matching
  terminal outcome is durably marked complete.

### ⚠️ Breaking

- Live `canic cycles convert --json` output is now the verified refill result
  with `operation_id`, `status`, `ledger_block_index`, `cycles_sent`,
  `error_code`, and `error_message`. The former command/context wrapper and
  opaque `icp_output` field are removed. Human output is likewise normalized
  to response fields rather than tool-specific Candid rendering.
- Typed endpoint rejection, response decoding failure, an operation-ID
  mismatch, an unexpected response variant, or failure to persist terminal
  completion now makes the command fail. No output-parser fallback or
  best-effort completion path is retained.
- A refill runtime without a canonical baked `local` or `ic` identity, or a
  direct-child refill whose funding policy cannot be resolved, now fails closed
  instead of selecting local behavior or omitting the cooldown gate. The
  intentional unset-to-local build default remains unchanged.

### 🧭 Migration Notes

- Update live-convert JSON consumers to read the verified result fields
  directly. Do not expect `mode`, deployment context, request metadata,
  `command`, or `icp_output` in successful live output.
- When the result is resumable or command completion is uncertain, retry from
  the same project root with the same network, deployment, source, target, and
  amount. Preserve `.canic/operations/pending.json`; the matching pending ID is
  the retry identity until a typed terminal result is durably recorded.
- Build refill-capable artifacts through the canonical Canic build path so
  named environments resolve to `local` or `ic`, and ensure every registered
  direct-child refill target has its funding policy configured. No fallback is
  provided for an invalid network identity or missing policy.

### 🧪 Testing

- Core coverage proves atomic same-key conflict rejection, exact retry and
  post-terminal admission, retryable failure activity, fail-closed network and
  funding-policy resolution, and shared resumability boundaries.
- CLI coverage proves locked concurrent reservation, matching pending reuse,
  terminal versus resumable completion, typed endpoint-error preservation,
  operation-ID mismatch rejection, normalized human and JSON rendering, and
  durable completion-write failure propagation.
- The eight-test root wasm-store reconciliation suite passes after isolating
  its canister-deletion lifecycle case from the immutable cached PocketIC
  baseline, removing test-order-dependent snapshot restore failures.

## [0.91.4] - 2026-07-14 - Failure-Path Settlement and Artifact Safety

### 📝 Summary

0.91.4 closes three failure-path gaps found during a repository-wide code
audit. Cost-guard quota and cycle-reservation intents now settle through one
prevalidated stable-state transition, snapshot capture reports requested
restart failures without losing the original capture cause, and artifact
production stops when an installed `ic-wasm` executable fails its shrink pass.

The patch does not change stable record schemas, Candid contracts, command
syntax, or successful build and backup behavior. It intentionally changes the
public Rust snapshot error taxonomy and the handling of a present but failing
optional shrink tool.

### 🩹 Fixed

- Settles cost-guard quota and cycle-reservation intents as one pair. Both
  records, pending indexes, resource totals, and aggregate metadata validate
  before either terminal transition is written, while already-settled records
  remain idempotent on retry.
- Routes failed control-plane creation, root provisioning, child cycles
  funding, pool creation, and canister upgrade operations through one recovery
  helper. A settlement failure is retained as diagnostic context on the
  original typed internal error instead of being discarded.
- Adds distinct snapshot restart and combined capture-plus-restart errors. A
  requested restart is always attempted after capture failure, and callers can
  distinguish a durable capture followed by restart failure from a capture
  failure where restart also failed.
- Keeps an absent `ic-wasm` shrink tool nonfatal, but treats a non-zero exit
  from a present executable as a build failure. The diagnostic includes its
  exit status and stderr, the original WASM remains intact, and partial shrink
  output is removed.

### ⚠️ Breaking

- Exhaustive Rust matches on `SnapshotDownloadError` must handle the new
  `Restart` and `CaptureAndRestart` variants. No compatibility catch-all or
  legacy error remapping is provided.
- Build environments that have a broken or incompatible `ic-wasm` executable
  now fail artifact production instead of silently packaging the unshrunk
  artifact. Environments without the executable retain the existing optional
  tool behavior.

### 🧪 Testing

- Adds cost-guard regression coverage for atomic pair rejection, successful
  quota-commit/reservation-release recovery, typed error preservation, quota
  enforcement, and reservation accounting; all 13 focused tests pass.
- Adds snapshot coverage for capture failure with successful restart,
  successful capture with failed restart, and combined capture/restart
  failure; all 21 focused snapshot tests pass.
- Adds artifact-I/O coverage for missing, successful, and non-zero shrink
  commands; all 4 focused tests pass.
- Package-scoped formatting, warning-denied Clippy across the four affected
  crates, changelog governance, and diff hygiene pass.

## [0.91.3] - 2026-07-14 - Audit Archive Retention

### 📝 Summary

0.91.3 makes Markdown reports and compact comparison baselines the retained
authority for Canic's audit archive. It removes reproducible or duplicate
generated evidence only after repairing historical references and confirming
that the owning reports, aggregate data, and future-comparison inputs remain.

This is an internal audit-governance and tooling cleanup. It does not change
runtime behavior, public Rust or Candid APIs, stable state, deployment
artifacts, or operator command contracts.

### 🧹 Cleanup

- Adds start-here indexes for audit policy, modular playbooks, and the dated
  report archive, including explicit directory ownership and legacy summary
  coverage.
- Defines a bounded retention rule: historical Markdown findings remain
  append-only, compact comparison inputs remain available, and generated
  evidence may be pruned only when it is reproducible or duplicates retained
  report content.
- Removes redundant per-canister `ic-wasm` and `twiggy` output, duplicate
  per-canister size JSON, instruction endpoint-matrix TSVs, and checkpoint
  logs. Historical reports now point only to retained evidence.
- Keeps the 32 `size-metrics.tsv` files used for same-day Wasm comparisons,
  together with aggregate size JSON, per-canister Markdown summaries,
  normalized instruction rows, verification readouts, and diagnostic JSON.

### 🔧 Changed

- The Wasm audit runner now treats raw `ic-wasm info` and `twiggy` output as
  cached analysis input and retains only aggregate data, compact baseline
  metrics, size summaries, and per-canister Markdown evidence in the report
  archive.
- The instruction-footprint generator now embeds its endpoint matrix and
  checkpoint scan in the primary report instead of emitting duplicate TSV and
  log files.

### 📊 Audit

- Deletes 1,768 generated files containing 3,504,256 bytes of duplicate or
  reproducible evidence. The checked-in audit tree falls from 2,904 to 1,139
  files after adding the new indexes.
- The retained archive contains 676 Markdown files, 431 JSON files, and 32 TSV
  baselines totaling 24,819 bytes; it contains no CSV, log, or text dumps.
- Every primary report remains present, and no Markdown reference remains to a
  deleted artifact.

### 🧪 Testing

- All 36 non-runner instruction-audit tests pass; the ignored PocketIC report
  generator target also compiles successfully.
- Targeted warning-denied Clippy, Rust formatting, shell syntax, focused
  ShellCheck, reference scans, and diff hygiene pass.

## [0.91.2] - 2026-07-14 - Allocation Governance and Manifest Path Admission

### 📝 Summary

0.91.2 updates Canic's allocation-governance dependency from
`ic-memory 0.10.0` to `0.11.1` and adopts its explicit current
protected-ledger payload format. The dependency recognizes a pre-0.11 payload
as unsupported and does not provide a legacy decoder or in-place migration.

The patch also moves the existing lexical release-artifact path rule into
canonical manifest admission. Loaded manifests now reject empty, absolute,
dot-component, and traversal paths before passive observation or staging,
while filesystem resolution continues to add canonical root-containment and
symlink-escape checks.

### 🔧 Changed

- Updates `ic-memory` to 0.11.1. Canic's stable application keys and
  `MemoryManager` IDs remain unchanged, but the protected allocation-ledger
  payload format is intentionally incompatible with 0.10.x state.
- Applies one lexical artifact-path validator during release-set manifest
  validation and filesystem resolution, so passive manifest consumers and
  staging agree on path admission without duplicate policy.
- Preserves the release-set JSON shape, successful Canic-generated paths,
  hashes, chunking, entry ordering, staging calls, and public Rust and Candid
  APIs.

### ⚠️ Breaking

- A canister with an allocation ledger written by `ic-memory 0.10.x` cannot
  upgrade in place to this dependency line. Recovery requires destructive
  reinstall of the documented authority and placement dependency closure.
- A loaded release-set manifest with an empty, absolute, dot-component, or
  traversal artifact path now fails at manifest admission rather than being
  accepted until a later filesystem-resolution or staging step. No fallback
  path normalization is added.

### 🧭 Migration Notes

- Treat deployment of this patch over existing 0.10.x allocation-ledger state
  as a destructive reinstall. Reinstall the full dependency closure described
  in the destructive-reinstall runbook and restore only through
  application-owned export/import paths valid for the new installation.
- Restoring an old stable-memory image also restores the incompatible ledger;
  it is not a migration path.
- Regenerate release-set manifests through Canic. Existing manifests using
  canonical relative artifact paths require no changes.

### 🧪 Testing

- All 13 focused `canic-core` memory tests pass against `ic-memory 0.11.1`,
  together with the stable-memory ABI guard and four memory protocol-surface
  tests.
- Ten release-set manifest admission/loading tests cover accepted identity and
  shape plus empty, absolute, dot-component, and traversal path rejection.
- Three artifact-path tests cover canonical in-root resolution, lexical
  rejection, and symlink escape rejection.
- Targeted host and CLI checks, warning-denied core/facade/host Clippy,
  changelog governance, formatting, and diff hygiene pass.

## [0.91.1] - 2026-07-14 - Issuer Proof Readiness

### 📝 Summary

0.91.1 lets an application root establish delegated-auth readiness immediately
after installing or reinstalling an issuer. The new public Rust facade creates
or reuses proof material through the current chain-key batch authority,
installs it on the selected issuer, and records the install outcome before
returning.

This closes the first-login gap for applications that inspect active-proof
status before calling Canic's token-prepare path. It does not add a generated
wire endpoint, caller-supplied proof injection, an external provisioner, or a
compatibility path for the retired proof APIs.

### ➕ Added

- Adds
  `AuthApi::provision_chain_key_delegation_proof_for_issuer_root(issuer_pid)`
  as a root-context-only application facade.
- Reuses the existing cached/singleflight chain-key proof creation path and the
  existing issuer install protocol rather than introducing another signing or
  proof representation.
- Records successful and failed issuer installation outcomes through the
  existing root batch state. An incomplete installation returns the typed
  public `Unavailable` error class.

### 🧭 Migration Notes

- Application roots may call the new facade after the issuer is installed or
  reinstalled and registered in restored topology. Root issuer policy and an
  enabled renewal template must already be configured.
- The application-owned endpoint remains responsible for authenticating and
  authorizing its caller and selecting the issuer principal.
- Replace application-owned proof create/retrieve/install orchestration with
  the single facade call. Do not retain fallbacks to the retired proof APIs.

### 🧪 Testing

- Focused PocketIC coverage proves a freshly installed issuer moves from
  `Missing` to `Valid` active-proof status with the expected root and issuer
  bindings after the facade completes.
- Public-facade signature coverage, focused provisioning tests, formatting,
  diff hygiene, package checks, and warning-denied Clippy checks pass.

## [0.91.0] - 2026-07-13 - Role Admission and Complete-Build Manifests

### 📝 Summary

0.91.0 closes two deployment-artifact authority gaps. Canister roles now pass
one lowercase snake_case admission rule before supplying configuration,
mutation, manifest, or artifact-path identity. Root release-set manifests are
now published only from the exact role-labelled outputs returned by one
complete install build, rather than from the existence of configured files.

The implementation follows the accepted
[role admission and complete-build manifest design][design] and its
[closeout audit][audit]. It adds no build epoch, lock, artifact transaction,
role wrapper hierarchy, path framework, or compatibility path.

[design]: ../design/0.91-role-admission-and-manifest-publication/0.91-design.md
[audit]: ../audits/reports/2026-07/2026-07-13/0.91-closeout.md

### 🔧 Changed

- Gives core configuration one typed canister-role policy: names must be 1 to
  40 bytes, start with a lowercase ASCII letter, and contain nonempty lowercase
  alphanumeric words separated by single underscores.
- Reuses one pure ASCII snake_case grammar across canister-role admission and
  project scaffolding while keeping the role-specific length limit at the role
  boundary.
- Applies role admission consistently to complete configuration validation,
  fleet role declaration, attachment and rename, and loaded release-set
  manifests before role-derived artifact access.
- Resolves one validated configuration-backed snapshot before a normal install
  build. Every required builder receives its admitted package identity and
  exact output paths from that snapshot.
- Retains the exact role-labelled outputs returned by the current invocation.
  The crate-private manifest writer rejects missing, duplicate, unexpected,
  unreadable, or path-mismatched outputs before replacing an existing
  manifest.
- Uses the snapshot's configured member order for successful manifest JSON,
  independent of build completion or map iteration order. Root output proves
  complete-build coverage without becoming an ordinary release-set entry.
- Keeps deployment-plan manifest materialization as the separate explicit
  complete-artifact source.

### 🗑️ Removed

- Removes both filesystem-readiness manifest writers and the public normal
  writers that reloaded configuration and reconstructed completion from paths.
- Removes the obsolete install-target and release-role file projections that
  could recreate a second complete-build selection flow.
- Removes the optional fleet-manifest path from per-role build output.
- Removes all single-role manifest creation, rewriting, reporting, and
  provenance attestation.

### ⚠️ Breaking

- Canister roles using kebab-case, uppercase letters, leading digits or
  underscores, repeated or trailing underscores, separators, dot components,
  whitespace, non-ASCII characters, or more than 40 bytes are rejected. No
  alias, normalization, sanitization, or fallback is retained.
- The removed host manifest-writer functions and optional per-role
  `manifest_path` output field have no compatibility wrappers.
- A direct or failed partial build leaves any prior manifest unchanged rather
  than republishing it. If the partial build replaced a covered artifact, the
  old manifest may no longer validate and is not guaranteed to remain
  deployable.

### 🧭 Migration Notes

- Rename invalid roles to lowercase snake_case in fleet configuration and the
  corresponding package metadata before building. Rebuild the affected
  artifacts under their new role directories; old spellings are not read as
  aliases.
- Use the complete install-root flow when a new root release-set manifest is
  required. `canic build <fleet> <role>` owns only that role's Wasm, gzip,
  optional Candid sidecar, and optional build provenance.
- Existing lowercase snake_case roles, release-set JSON fields and field
  order, hashing, chunking, staging, receipts, stable state, and canister
  memory layouts are unchanged.
- Concurrent mutating builds against one install root remain unsupported;
  0.91 does not add locking or inter-process consistency.

### 🧪 Testing

- Focused tests cover canonical and invalid role admission, complete config
  rejection, fleet mutation mapping, scaffold grammar reuse, and release-set
  manifest admission.
- Complete-build tests cover missing, duplicate, unexpected, unreadable, and
  path-mismatched outputs; unchanged prior-manifest bytes on rejection; and
  snapshot-defined successful entry order.
- Per-role provenance tests prove that Wasm, gzip, and optional Candid remain
  role-owned without a fleet-manifest artifact.
- Targeted canister-build, release-role, install-operation, Candid/Serde guard,
  formatting, diff hygiene, and warning-denied host and CLI Clippy checks pass.
  Full workspace, broad PocketIC, deployment, and release suites remain
  maintainer-owned.
