# Canic 0.91 Changelog

## [0.91.2] - 2026-07-14 - Allocation Governance and Manifest Path Admission

### 📝 Summary

0.91.2 updates Canic's allocation-governance dependency from
`ic-memory 0.10.0` to `0.11.1` and adopts its explicit current
protected-ledger payload format. The dependency recognizes a pre-0.11 payload
as unsupported and does not provide a legacy decoder or in-place migration.

The patch also moves the existing lexical release-artifact path rule into
canonical manifest admission. Loaded manifests now reject empty, absolute,
dot-component, and traversal paths before passive observation or staging,
while filesystem resolution continues to add canonical root-containment and
symlink-escape checks.

### 🔧 Changed

- Updates `ic-memory` to 0.11.1. Canic's stable application keys and
  `MemoryManager` IDs remain unchanged, but the protected allocation-ledger
  payload format is intentionally incompatible with 0.10.x state.
- Applies one lexical artifact-path validator during release-set manifest
  validation and filesystem resolution, so passive manifest consumers and
  staging agree on path admission without duplicate policy.
- Preserves the release-set JSON shape, successful Canic-generated paths,
  hashes, chunking, entry ordering, staging calls, and public Rust and Candid
  APIs.

### ⚠️ Breaking

- A canister with an allocation ledger written by `ic-memory 0.10.x` cannot
  upgrade in place to this dependency line. Recovery requires destructive
  reinstall of the documented authority and placement dependency closure.
- A loaded release-set manifest with an empty, absolute, dot-component, or
  traversal artifact path now fails at manifest admission rather than being
  accepted until a later filesystem-resolution or staging step. No fallback
  path normalization is added.

### 🧭 Migration Notes

- Treat deployment of this patch over existing 0.10.x allocation-ledger state
  as a destructive reinstall. Reinstall the full dependency closure described
  in the destructive-reinstall runbook and restore only through
  application-owned export/import paths valid for the new installation.
- Restoring an old stable-memory image also restores the incompatible ledger;
  it is not a migration path.
- Regenerate release-set manifests through Canic. Existing manifests using
  canonical relative artifact paths require no changes.

### 🧪 Testing

- All 13 focused `canic-core` memory tests pass against `ic-memory 0.11.1`,
  together with the stable-memory ABI guard and four memory protocol-surface
  tests.
- Ten release-set manifest admission/loading tests cover accepted identity and
  shape plus empty, absolute, dot-component, and traversal path rejection.
- Three artifact-path tests cover canonical in-root resolution, lexical
  rejection, and symlink escape rejection.
- Targeted host and CLI checks, warning-denied core/facade/host Clippy,
  changelog governance, formatting, and diff hygiene pass.

## [0.91.1] - 2026-07-14 - Issuer Proof Readiness

### 📝 Summary

0.91.1 lets an application root establish delegated-auth readiness immediately
after installing or reinstalling an issuer. The new public Rust facade creates
or reuses proof material through the current chain-key batch authority,
installs it on the selected issuer, and records the install outcome before
returning.

This closes the first-login gap for applications that inspect active-proof
status before calling Canic's token-prepare path. It does not add a generated
wire endpoint, caller-supplied proof injection, an external provisioner, or a
compatibility path for the retired proof APIs.

### ➕ Added

- Adds
  `AuthApi::provision_chain_key_delegation_proof_for_issuer_root(issuer_pid)`
  as a root-context-only application facade.
- Reuses the existing cached/singleflight chain-key proof creation path and the
  existing issuer install protocol rather than introducing another signing or
  proof representation.
- Records successful and failed issuer installation outcomes through the
  existing root batch state. An incomplete installation returns the typed
  public `Unavailable` error class.

### 🧭 Migration Notes

- Application roots may call the new facade after the issuer is installed or
  reinstalled and registered in restored topology. Root issuer policy and an
  enabled renewal template must already be configured.
- The application-owned endpoint remains responsible for authenticating and
  authorizing its caller and selecting the issuer principal.
- Replace application-owned proof create/retrieve/install orchestration with
  the single facade call. Do not retain fallbacks to the retired proof APIs.

### 🧪 Testing

- Focused PocketIC coverage proves a freshly installed issuer moves from
  `Missing` to `Valid` active-proof status with the expected root and issuer
  bindings after the facade completes.
- Public-facade signature coverage, focused provisioning tests, formatting,
  diff hygiene, package checks, and warning-denied Clippy checks pass.

## [0.91.0] - 2026-07-13 - Role Admission and Complete-Build Manifests

### 📝 Summary

0.91.0 closes two deployment-artifact authority gaps. Canister roles now pass
one lowercase snake_case admission rule before supplying configuration,
mutation, manifest, or artifact-path identity. Root release-set manifests are
now published only from the exact role-labelled outputs returned by one
complete install build, rather than from the existence of configured files.

The implementation follows the accepted
[role admission and complete-build manifest design][design] and its
[closeout audit][audit]. It adds no build epoch, lock, artifact transaction,
role wrapper hierarchy, path framework, or compatibility path.

[design]: ../design/0.91-role-admission-and-manifest-publication/0.91-design.md
[audit]: ../audits/reports/2026-07/2026-07-13/0.91-closeout.md

### 🔧 Changed

- Gives core configuration one typed canister-role policy: names must be 1 to
  40 bytes, start with a lowercase ASCII letter, and contain nonempty lowercase
  alphanumeric words separated by single underscores.
- Reuses one pure ASCII snake_case grammar across canister-role admission and
  project scaffolding while keeping the role-specific length limit at the role
  boundary.
- Applies role admission consistently to complete configuration validation,
  fleet role declaration, attachment and rename, and loaded release-set
  manifests before role-derived artifact access.
- Resolves one validated configuration-backed snapshot before a normal install
  build. Every required builder receives its admitted package identity and
  exact output paths from that snapshot.
- Retains the exact role-labelled outputs returned by the current invocation.
  The crate-private manifest writer rejects missing, duplicate, unexpected,
  unreadable, or path-mismatched outputs before replacing an existing
  manifest.
- Uses the snapshot's configured member order for successful manifest JSON,
  independent of build completion or map iteration order. Root output proves
  complete-build coverage without becoming an ordinary release-set entry.
- Keeps deployment-plan manifest materialization as the separate explicit
  complete-artifact source.

### 🗑️ Removed

- Removes both filesystem-readiness manifest writers and the public normal
  writers that reloaded configuration and reconstructed completion from paths.
- Removes the obsolete install-target and release-role file projections that
  could recreate a second complete-build selection flow.
- Removes the optional fleet-manifest path from per-role build output.
- Removes all single-role manifest creation, rewriting, reporting, and
  provenance attestation.

### ⚠️ Breaking

- Canister roles using kebab-case, uppercase letters, leading digits or
  underscores, repeated or trailing underscores, separators, dot components,
  whitespace, non-ASCII characters, or more than 40 bytes are rejected. No
  alias, normalization, sanitization, or fallback is retained.
- The removed host manifest-writer functions and optional per-role
  `manifest_path` output field have no compatibility wrappers.
- A direct or failed partial build leaves any prior manifest unchanged rather
  than republishing it. If the partial build replaced a covered artifact, the
  old manifest may no longer validate and is not guaranteed to remain
  deployable.

### 🧭 Migration Notes

- Rename invalid roles to lowercase snake_case in fleet configuration and the
  corresponding package metadata before building. Rebuild the affected
  artifacts under their new role directories; old spellings are not read as
  aliases.
- Use the complete install-root flow when a new root release-set manifest is
  required. `canic build <fleet> <role>` owns only that role's Wasm, gzip,
  optional Candid sidecar, and optional build provenance.
- Existing lowercase snake_case roles, release-set JSON fields and field
  order, hashing, chunking, staging, receipts, stable state, and canister
  memory layouts are unchanged.
- Concurrent mutating builds against one install root remain unsupported;
  0.91 does not add locking or inter-process consistency.

### 🧪 Testing

- Focused tests cover canonical and invalid role admission, complete config
  rejection, fleet mutation mapping, scaffold grammar reuse, and release-set
  manifest admission.
- Complete-build tests cover missing, duplicate, unexpected, unreadable, and
  path-mismatched outputs; unchanged prior-manifest bytes on rejection; and
  snapshot-defined successful entry order.
- Per-role provenance tests prove that Wasm, gzip, and optional Candid remain
  role-owned without a fleet-manifest artifact.
- Targeted canister-build, release-role, install-operation, Candid/Serde guard,
  formatting, diff hygiene, and warning-denied host and CLI Clippy checks pass.
  Full workspace, broad PocketIC, deployment, and release suites remain
  maintainer-owned.
