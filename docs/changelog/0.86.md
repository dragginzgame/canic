# Canic 0.86 Changelog

## [0.86.1] - 2026-07-12 - Project and Role-Contract Medic Ownership

### 📝 Summary

0.86.1 completes the project-focused portion of the Medic structural pass.
Project configuration and state-audit checks now have one focused owner, while
role-package metadata, runtime-feature requirements, resolved role-contract
findings, and state-descriptor admission checks have another.

The Medic parent continues to own project check ordering. CLI commands, exit
classification, diagnostic codes and order, text and JSON reports, Candid,
stable state, and persisted bytes are unchanged.

### 🔧 Changed

- Project fleet discovery, ICP configuration presence, explicit-network
  guidance, and state-audit result classification move to one project module.
- Role-package metadata validation, configuration-derived feature
  requirements, package evidence, resolved contract findings, and descriptor
  admission move to one role-contract module.
- Cargo manifest parsing remains in the existing package-inspection module;
  role-contract policy and package resolution remain with their existing core
  and host owners.

### 🗑️ Removed

- The former project and role-contract check definitions are deleted from the
  Medic parent. No forwarding wrapper, compatibility alias, generic check
  trait, or rule framework replaces them.

### 📊 Audit

- The Medic parent falls from 1,379 lines in `0.86.0` to 944 lines, a further
  reduction of 435 lines.
- Compared with the 1,573-line audit baseline, the parent boundary is 629 lines
  smaller, a 40 percent reduction.
- The moved responsibilities are contained in a 154-line project module and a
  327-line role-contract module. Deployment diagnostics remain in the parent
  for a separately reviewable slice.

### 🧪 Testing

- All 51 focused Medic library tests pass.
- Targeted `canic-cli` library Clippy passes with warnings denied.
- Changelog governance, formatting, and diff-hygiene checks pass.

## [0.86.0] - 2026-07-12 - Medic Check Ownership

### 📝 Summary

0.86.0 starts the bounded structural-maintainability line with a mechanical
Medic split. Auth-renewal and blob-storage diagnostics now have focused module
owners, while the Medic parent retains project/deployment check selection and
ordering.

CLI commands, exit classification, diagnostic codes, text and JSON reports,
Candid, stable state, and persisted bytes are unchanged.

### 🔧 Changed

- Auth-renewal readiness summary and typed-error classification move to one
  Medic auth module.
- Blob-storage billing readiness, target-error classification, and passive
  Candid endpoint detection move to one Medic blob-storage module.
- The Medic parent imports those checks directly and no longer owns their
  implementation details.

### 🗑️ Removed

- The former auth-renewal and blob-storage definitions are deleted from the
  Medic parent. No forwarding wrapper, alias, compatibility path, generic check
  trait, or rule framework replaces them.

### 📊 Audit

- The audit-identified Medic hub falls from 1,573 to 1,379 lines after the
  first extraction, a reduction of 194 lines from the parent boundary.
- The extracted responsibilities are contained in a 76-line auth module and a
  183-line blob-storage module.
- This release does not begin the deferred deploy-plan or state-manifest
  splits; those remain separately reviewable 0.86 slices.

### 🧪 Testing

- All 51 focused Medic library tests pass.
- Targeted `canic-cli` library Clippy passes with warnings denied.
- Changelog governance, formatting, and diff-hygiene checks pass.
