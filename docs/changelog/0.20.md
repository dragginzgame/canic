# 0.20.x Detailed Changelog

Technical release notes for the `0.20.x` minor line.

## 0.20.10 - 2026-03-31 - Store Fleet Manager

### 📝 Summary

- `0.20.10` upgrades root publication from reactive rollover into a real store-fleet manager.
- Automatic publication now plans against the whole tracked `wasm_store` inventory, reuses exact existing releases instead of blindly republishing them, and allocates fresh stores before publication only when no current writable store can accept the next release.

### 🔧 Changed

- Root publication now uses the full approved manifest set as its source of truth instead of assuming the current release set lives in one default `wasm_store`.
- The bootstrap publication path and the later current-release republish path now share the same manager-owned placement flow.
- Exact matching `template_id@version` releases are now reused in-place when the target store already has the same role, hash, and payload size, so publication stops creating duplicate copies just because a later run touched the same release again.
- Conflicting duplicate releases for the same `template_id@version` now fail closed instead of letting root silently choose between mismatched store copies.
- When no existing writable store can accept a release, root now creates a fresh `wasm_store` proactively and places that release there instead of waiting for a store write to fail first.
- Fixed-target republish now also walks the full approved manifest set, so it can migrate a distributed release fleet into one explicitly chosen store without relying on a single current source store.
- Root post-upgrade reconcile now validates the exact currently approved release bytes across the tracked store fleet and rebinds to the right store deterministically instead of conflicting on older role versions that remain in older stores.
- The root-facing store overview now reports approved-release projections explicitly (`approved_payload_bytes`, `approved_templates`, and related fields) instead of implying that the response is a live physical occupancy view of each store.
- Ordinary embedded child releases are now gzip-only across the root bundle path, so even non-scripted root builds stop falling back to raw `.wasm` payloads.
- The hidden `wasm_store` build path now synthesizes its own wrapper crate from the resolved `canic` source when `canic-wasm-store` is not otherwise present, so downstreams do not need to add extra `wasm_store` source or config just to satisfy the implicit bootstrap artifact contract.

### 🩹 Fixed

- Automatic publication no longer depends on a hidden “one current source store” assumption once releases have already been split across multiple `wasm_store` instances.
- Later publication runs no longer republish exact duplicate releases just because the fleet already contains the correct bytes in another writable store.
- Publication state no longer has to churn the active publication slot on every spill decision just to keep multi-store placement working.
- Root readiness after upgrade no longer has a built-in multi-store false negative where older stores keeping previous role versions could make fleet import fail even though the exact current release still existed in the fleet.
- Root’s store overview no longer overstates what it knows about live store occupancy; it now exposes the narrower root-approved-release projection honestly.

### 🧪 Testing

- `cargo test -p canic-tests --test root_hierarchy -- --nocapture`
- `cargo clippy -p canic-control-plane -p canic -p canic-wasm-store -p canister_root --all-targets --all-features -- -D warnings`
- `cargo build -p canic-wasm-store -p canister_root`

## 0.20.9 - 2026-03-31 - Multi-Store Publication Rollover

### 📝 Summary

- `0.20.9` finishes the first-store-capacity follow-up by making root publication retry each release against the current publication store instead of treating the whole batch as a single-store transaction.
- When one selected `wasm_store` fills up, root can now promote or create another store and continue publishing the remaining releases there.

### 🔧 Changed

- The root bootstrap publication flow now evaluates store selection per staged release instead of once for the whole batch.
- The current-store republish path follows the same per-release rollover logic, so later publication runs use the same capacity behavior as bootstrap.
- On `wasm store capacity exceeded`, root now resolves another publication store, promotes it into the active slot, and retries that one release there.
- If a release still exceeds capacity on an empty fresh store, the publication flow still fails closed instead of creating stores forever.
- Catalog import after publication now follows the active publication binding when one was promoted, so later `auto_create` installs resolve their approved manifests from the store that actually received the releases.

### 🩹 Fixed

- Large bootstrap release sets no longer fail just because the first selected store cannot hold the whole salvo.
- Later root-driven installs no longer assume the configured default publication binding won the earlier publication race when root had already rolled over to another store.

### 🧪 Testing

## 0.20.8 - 2026-03-31 - Canonical Wasm Store Package and Gzipped Release Bundle

### 📝 Summary

- `0.20.8` turns the implicit `wasm_store` into a canonical published package instead of a downstream-owned local crate.
- It also fixes the ordinary root release bundle to embed `.wasm.gz` payloads for child roles, which keeps large topologies from exhausting the live store on raw wasm bytes alone during bootstrap publication.
- Root bootstrap and current-store publication now roll over onto additional `wasm_store` canisters when one selected store runs out of capacity mid-publication, so larger release sets can span multiple stores instead of failing after the first store fills.

### 🔧 Changed

- Added the publishable `canic-wasm-store` crate as the canonical `wasm_store` canister source, so downstreams can build the same store from Canic-owned sources instead of carrying a bespoke local `wasm_store` crate.
- `scripts/app/build.sh wasm_store` now resolves the canonical `canic-wasm-store` package automatically from the local checkout, direct Cargo metadata, or the sibling published registry source next to the resolved `canic` package.
- The old local `canisters/wasm_store` package is gone from the workspace; the hidden `dfx` build path now points at the canonical package instead of a reference-topology crate.
- The embedded ordinary root release bundle now prefers build-produced child `.wasm.gz` artifacts, not raw `.wasm`, for roles like `app`, `test`, and `user_hub`.
- The strict root build path now fails closed if a configured ordinary role would fall back to raw `.wasm` bytes, which prevents raw-payload store-capacity regressions from silently reappearing in real builds.
- Root publication now retries individual releases on a newly promoted store when the current publication target returns `wasm store capacity exceeded`, instead of treating the whole publication batch as a single-store transaction.
- Catalog import after bootstrap publication now follows the active publication binding when one was promoted, so later `auto_create` installs resolve the releases from the store that actually received them.

### 🩹 Fixed

- Large downstream release sets no longer pay store capacity on embedded raw child wasm payloads during the normal root bootstrap publication path.
- Large downstream release sets no longer fail just because the first selected store cannot hold the entire initial publication batch; root can spill the release set across multiple stores.
- Downstreams no longer need to keep a local `wasm_store` canister crate in sync just to satisfy the hidden bootstrap artifact contract.

## 0.20.6 - 2026-03-31 - Strict Bootstrap Artifact Validation

### 📝 Summary

- `0.20.6` turns the embedded `wasm_store` bootstrap path into a stricter build-time contract.
- If a downstream root build points at a `.wasm.gz` artifact that is empty, not really gzip, or does not decompress to a valid wasm module, the root build now fails immediately instead of letting bootstrap reach runtime and die inside `install_code`.

### 🩹 Fixed

- Root build-time bootstrap validation now rejects gzip artifacts that decompress to zero bytes or to non-wasm payloads, which closes the failure mode where downstreams could accidentally embed a placeholder or truncated `wasm_store.wasm.gz` and only discover it after reinstalling `root`.
- The bootstrap provenance log now reports both the original DFX-produced source path and the copied embedded path, so downstreams can tell whether the bad payload came from the build artifact itself or from the embedded copy step.

### 🔧 Changed

- `EmbeddedRootBootstrapEntry` now carries both `artifact_path` and `embedded_artifact_path`.
- The root build support path now validates the copied embedded bootstrap artifact before generating the root bootstrap source.
- The runtime provenance log now prints `source_path=...` and `embedded_path=...` alongside the size and hash data for the embedded `wasm_store` payload.

### 🧪 Testing

- `cargo build -p canic -p canister_root`
- `cargo clippy -p canic -p canic-control-plane -p canic-core -p canister_root --all-targets --all-features -- -D warnings`

## 0.20.5 - 2026-03-31 - Bootstrap Artifact Provenance

### 📝 Summary

- `0.20.5` fixes the root bootstrap drift that could install an old embedded `wasm_store` module even after the current DFX build had moved on.
- Root now embeds the current DFX-built `wasm_store.wasm.gz` artifact through an immutable copied build output, and logs the exact embedded bootstrap provenance during init so mismatches are visible immediately.

### 🩹 Fixed

- Removed the live bootstrap path’s reliance on the stale checked-in `crates/canic/assets/bootstrap/wasm_store.wasm.gz` payload, which was still matching the failing `48f6aa...` module hash seen in root bootstrap failures.
- The embedded `wasm_store` bootstrap payload is now copied into `OUT_DIR` during the root build before `include_bytes!`, so later `.dfx` churn cannot silently change the file path after root has already baked in a specific payload.
- Root init now logs the embedded bootstrap artifact path, compression kind, compressed size/hash, and decompressed size/hash before async bootstrap continues, which makes stale bootstrap payloads diagnosable from the canister logs instead of only from a later `IC0536` method-mismatch failure.
- The install path keeps treating embedded bytes as an opaque module payload, so the current DFX-built `.wasm.gz` artifact is passed through unchanged instead of being decompressed inside `canic-core`.

### 🔧 Changed

- `build_root!()` now records dedicated bootstrap provenance for the embedded `wasm_store` release set through `EmbeddedRootBootstrapEntry`.
- The generated root bootstrap source now points at the copied `OUT_DIR/canic.root-wasm-store-bootstrap.wasm.gz` file instead of the mutable `.dfx/.../wasm_store.wasm.gz` path directly.
- The local root build flow still refreshes `wasm_store` first through the normal DFX/custom build path, but the embedded bootstrap source is now explicitly tied to the exact artifact that root saw at build time.

## 0.20.4 - 2026-03-31 - Embedded Root Release Bundle

### 📝 Summary

- `0.20.4` removes the last user-visible ordinary release bootstrap step by embedding the approved child-role release bundle into `root` during the normal `dfx build --all` flow.
- Local installs now return to the simpler contract: build the canisters, reinstall `root`, and let root publish the ordinary roles without a separate staging/resume script.

### 🔧 Changed

- `build_root!()` now emits an embedded ordinary release bundle from the already-built `.dfx/$DFX_NETWORK/canisters/<role>/<role>.wasm` artifacts for the configured non-root release roles.
- `canic::start_root!()` now seeds that embedded bundle into root lifecycle before async bootstrap begins, so root can publish approved ordinary roles into the first live `wasm_store` automatically.
- The root lifecycle and template support path now treat the embedded bundle as the normal source of approved ordinary role manifests and chunks during bootstrap.
- `canic::start_root!()` now falls back cleanly when no embedded bundle is present, which keeps examples and feature-gated builds compiling even outside the full `dfx build --all` path.
- The local reference install flow and PocketIC root harness no longer stage ordinary release roles manually just to get root to READY.

### 🗑️ Removed

- Deleted the old external root release staging helpers:
  - `scripts/canic/bootstrap_root_release_set.sh`
  - `scripts/app/stage_root_release_set.sh`
- The live docs and app install flow no longer point users at a separate release-staging step after reinstalling `root`.

## 0.20.3 - 2026-03-31 - Perf Audit Tooling Stabilization

### 📝 Summary

- `0.20.3` turns the `0.20` instruction audit into a usable repeated baseline by making the perf runner more resilient, adding real production `perf!` coverage across the critical shared flows, and measuring root template-staging admin updates directly instead of only scanning for checkpoint presence.

### 📊 Audit

- The instruction-footprint runner now emits a current baseline report with measured root admin staging endpoints (`canic_template_stage_manifest_admin`, `canic_template_prepare_admin`, and `canic_template_publish_chunk_admin`) instead of limiting the matrix to the earlier query-heavy floor.
- The generated report now reflects the actual scenario set in its preamble and verification artifacts instead of hardcoding the older endpoint scope.
- The recurring perf baseline now has full checkpoint coverage across the six declared critical flow families, including bootstrap/install/publication.

### 🔧 Changed

- Added production `perf!` checkpoints in the root bootstrap/publication workflow so bootstrap/install/publication attribution now comes from live control-plane code rather than the host harness.
- The instruction audit matrix now includes measured root admin update scenarios for staged manifest, prepare, and chunk publication.
- The `dfx` artifact build helper now provisions missing canister ids before `dfx build --all` and isolates its Cargo target directory, which prevents audit reruns from fighting with normal host-side Cargo builds.
- The build script now respects `CARGO_TARGET_DIR`, so `dfx`-driven canister builds and local Cargo test/clippy runs no longer race over the same target tree during audit reruns.

## 0.20.2 - 2026-03-31 - Reference Topology Consolidation

### 📝 Summary

- `0.20.2` makes `wasm_store` an internal root bootstrap detail and removes the old pure-sharding reference lane, so downstream Canic users no longer need a separate `wasm_store` canister and the reference topology keeps one sharding-capable lane instead of two.

### 🧹 Cleanup

- The reference/demo topology no longer exposes `wasm_store` as a separate `dfx.json` canister, staged release artifact, or top-level reference canister.
- The top-level reference topology no longer carries separate `shard` and `shard_hub` canisters.
- The workspace, `dfx.json`, build scripts, and live docs now match the smaller reference set instead of advertising deleted sharding-only roles.

### 🔧 Changed

- `canic::start_root!()` now registers a built-in bootstrap `embedded:wasm_store` module from the published `canic` crate and installs the first live store directly from that embedded payload.
- The generic module-install path now accepts both chunk-store-backed releases and embedded built-in modules, so the first store install and ordinary role installs share the same orchestration flow.
- Root bootstrap now waits for staged ordinary release roles instead of waiting for an externally staged bootstrap `wasm_store` payload.
- Added a generic host-side root bootstrap helper script so downstream Canic projects can stage their own config-defined release set, resume root bootstrap, and wait for `canic_ready` without copying the reference-topology shell logic by hand.
- Root bootstrap, wasm publication, and install logs now report staged releases as `template_id@version`, so operators can see which approved release root selected for each role instead of only seeing the role name.
- Pure sharding bootstrap coverage now runs through `user_hub`, so the reference topology keeps one sharding-capable coordinator instead of two near-duplicate hubs.
- The PocketIC root harness now stages an explicit current release set instead of iterating every leftover `.dfx/local/canisters` artifact, which prevents stale local build output from silently reintroducing deleted roles during bootstrap tests.

## 0.20.0 - 2026-03-31 - Cleanup and Optimization

### 📝 Summary

- `0.20.0` opens the cleanup and optimization line for Canic.
- The goal of `0.20.x` is to make audit-driven reductions in shared wasm size and `perf!` / endpoint instruction counts, while tightening regression detection so future cleanup work stays measurable and comparable.

### 📊 Audit

- `0.20.x` uses the recurring wasm-footprint audit as the canonical release-size baseline for shared canisters such as `minimal`, with `root` and `wasm_store` treated as intentional outliers rather than the default floor.
- `0.20.x` also uses the recurring instruction-footprint audit as the canonical runtime-cost baseline for endpoint and flow-level instruction tracking.
- Optimization claims in this line should be backed by repeatable audit artifacts and comparable method tags, not by one-off local runs.
- When a change materially affects runtime cost, the patch notes in this line should record the measured delta or explicitly say that coverage is still partial.

### 🧹 Cleanup

- Prefer structural reductions that lower the shared runtime floor first: fewer default-lane dependencies, fewer always-on endpoints, less duplicated generated surface, and less control-plane/sharding baggage in ordinary leaf canisters.
- Prefer deleting dead compatibility paths and stale build/test helpers rather than carrying them indefinitely behind dormant scripts or docs.
- Treat demo/reference topology cleanup as in-scope when it reduces duplicated code, duplicated wasm, or repeated orchestration logic without weakening the reference coverage itself.
- Workspace-only PocketIC and release-audit integration tests now live in an unpublished `canic-tests` package instead of the publishable `canic` crate, so `cargo publish -p canic` no longer tries to resolve unpublished helpers such as `canic-internal` from crates.io.

### 🔧 Changed

- The `0.20` line is intentionally audit-led: patch releases in this minor should prioritize measurable cleanup, wasm reduction, and instruction reduction work over additive surface expansion.
- Stable `perf!` checkpoints for important flows are in scope when they materially improve instruction-footprint coverage and regression detection.
- Live test and audit entrypoints now target `canic-tests` for the heavy integration suite, while `canic` keeps only publish-safe manifest checks in its packaged test surface.
- The public `canic` facade now re-exports `TemplateChunkingMode` and `TemplateManifestState` behind `features = ["control-plane"]`, so downstream root/store helpers can stop reaching into `canic-control-plane` just to inspect template manifest state.

### 🥾 Governance

- Use the recurring audit definitions under `docs/audits/recurring/system/` as the source of truth for method, comparability, and required artifacts.
- If a patch in `0.20.x` changes audit method or transport shape, the patch notes should say so explicitly and treat affected deltas as method changes rather than pretending they are directly comparable.
