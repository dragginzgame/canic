# Canic 0.90 Changelog

## [0.90.2] - 2026-07-13 - Receipt Adapter Conformance

### 📝 Summary

0.90.2 closes the Canic-side downstream-enablement slice for receipt-backed
intents. The focused public-facade fixture now exposes exact begin and
settlement decisions and proves the contract a domain adapter must consume
after Canic is published.

This release adds no mint model, receipt validator, resolver, recovery
framework, production API, or stable-state change. Toko retains ownership of
its later mint implementation and domain evidence tests.

### 🔧 Changed

- Hard-cuts the test fixture's optional success/conflict projection to explicit
  created, replayed, binding-conflict, capacity, settled, already-settled,
  missing, and revision-conflict outcomes.
- Makes the test settlement boundary carry the expected revision, terminal
  decision, evidence source, and fingerprint so commit and durable rollback
  follow the same public Canic facade contract.
- Adds a downstream adapter handoff covering begin-before-effect ordering,
  result handling, receipt validation, conservative pending state, and trusted
  terminal evidence construction.
- Separates Canic publication from downstream Toko adoption in the 0.90 design
  and tracker. Toko's caller-scoped receipts and mint proof begin only after a
  compatible Canic release is published.

### 🥾 Governance

- Restricts automated edits to the Canic repository. Named downstream
  integrations are documented as handoffs rather than mutated across sibling
  repositories.

### 🧪 Testing

- The focused PocketIC proof covers exact pending replay, changed payload and
  resource rejection, shared capacity, missing operations, stale revisions,
  settlement binding conflicts, pending-state upgrade recovery, committed and
  rolled-back settlement, terminal replay, contradictory evidence, and
  released rollback capacity.
- The exact-outcome fast fixture Wasm is 1,268,888 bytes, 16,615 bytes above
  the 0.90.1 receipt fixture. The increase is confined to test Candid outcome
  types and branches.
- Targeted warning-denied Clippy passes for the integration target and intent
  authority fixture. Targeted formatting and diff hygiene also pass. Broad
  workspace, deployment, and release suites remain maintainer-owned.

## [0.90.1] - 2026-07-13 - Receipt-Backed Intent Core

### 📝 Summary

0.90.1 implements the minimal Canic half of receipt-backed intent
reconciliation. One exact `OperationId` lookup now provides durable,
idempotent reservation identity; one non-awaiting compare-and-set transition
moves its quantity to committed or rolled back after the consumer supplies
validated terminal evidence.

This release does not add a generic effect engine, receipt protocol, resolver,
background reconciler, or mint-specific model. Toko mint remains the first
consumer and owns all domain request, receipt, validation, retry, cancellation,
and outcome policy.

### ➕ Added

- Adds opaque, versioned payload bindings and bounded terminal evidence to the
  generic intent model.
- Adds exact begin-or-load, read, and compare-and-set settlement operations to
  the core and public Canic facade.
- Adds direct local begin, commit, and rollback operations for effects whose
  outcome is locally decidable.
- Adds one stable receipt-backed operation map on memory ID 43. The map's
  maintained length gates new records without another metadata allocation or
  scan.
- Adds fixed-label local and receipt-backed intent metrics without operation
  IDs, principals, resource keys, or payload data.

### 🔧 Changed

- Local and receipt-backed reservations now share the existing resource
  aggregate as their single accounting authority.
- Receipt-backed pending rows never expire, enter the local TTL index, or
  start or retain the cleanup timer.
- Capacity applies only to absent receipt-backed IDs. Existing pending and
  terminal operations remain readable and settleable when admission is full.
- Unsupported payload or evidence schema versions, changed bindings,
  contradictory evidence, and stale revisions fail without changing counters.
- Expired local reservations remain conservatively accounted until the
  existing cleanup workflow aborts them instead of disappearing from quota
  reads before durable cleanup.

### ⚠️ Breaking

- Removes `CallBuilder::with_intent`, `IntentReservation`, `IntentKey`, and the
  internal automatic reserve/call/commit-or-abort path completely.
- Maintained callers must explicitly begin a local intent, perform their call,
  then commit or roll back according to the decoded domain outcome.
- No alias, compatibility wrapper, fallback, or legacy orchestration path is
  retained.

### 🧭 Migration Notes

- Existing local intent records, totals, pending entries, metadata, encodings,
  and memory IDs 39 through 42 are unchanged.
- Receipt-backed records use only the new memory ID 43. There is no historical
  scan or in-place state migration.
- Consumers of the removed call-builder shortcut must choose direct local
  operations or the receipt-backed operations; transport success alone is not
  terminal evidence for an irreversible external effect.

### 🧪 Testing

- Focused storage tests cover replay, binding conflicts, shared and fixed
  capacity, unsupported schemas, no-TTL behavior, revision conflicts, commit,
  rollback, terminal replay, contradictory evidence, and exact counter deltas.
- Stable-state and role-contract checks prove the new allocation, unchanged
  local allocations, canonical descriptors, and collision-free ownership.
- The focused PocketIC fixture proves overlapping local capacity, receipt
  replay, admission rejection, pending-state preservation across upgrade,
  settlement, and idempotent terminal replay through the public facade.
- The receipt-enabled fast fixture Wasm is 1,252,273 bytes, 70,530 bytes
  (6.0%) above its local-only baseline, with no new call, scan, timer, or
  cleanup entry.
- Targeted warning-denied Clippy, layering, metrics ordering, changelog
  governance, formatting, and diff hygiene pass. Broad workspace and release
  suites remain maintainer-owned.

## [0.90.0] - 2026-07-13 - Receipt-Backed Intent Reconciliation Design

### 📝 Summary

0.90.0 reserves the 0.90 line for the smallest receipt-backed extension to the
existing Canic intent system. Canic owns generic reservation identity and
at-most-once local settlement. Toko owns mint as the first and only 0.90
consumer, including its request, ledger receipt, evidence validation, retry
policy, cancellation, and business outcome.

This is not a mint subsystem in Canic and not a generic transaction or effect
framework. The implementation contract is recorded in the
[0.90 receipt-backed intent design][design].

[design]: ../design/0.90-receipt-backed-intent-reconciliation/0.90-design.md

This entry records design scope only. It declares no runtime, Candid,
stable-state, package-version, or API change yet.

### ⚠️ Breaking

- The implementation will remove `CallBuilder::with_intent` completely. No
  compatibility alias will preserve automatic reserve/commit/abort behavior.
- Maintained local-intent consumers will call local intent operations directly.
- Any Toko mint replay row that can independently change the same local
  counters must be retired or reduced to non-authoritative request/outcome
  cache.

### 🥾 Governance

#### Minimal Canic boundary

- Canic adds `ReceiptBackedIntentRecord`,
  `ReceiptBackedIntentState`, opaque `PayloadBinding`, bounded
  `TerminalEvidence`, and direct `OperationId` storage.
- The only mutating operations are
  `begin_or_load_receipt_backed_intent` and
  `settle_receipt_backed_intent_if_pending`.
- The state is `Pending`, `Committed`, or `RolledBack`. Canic does not model
  submission, recovery, mint, or ledger phases.
- `Pending` is the conservative uncertainty boundary and has no TTL.
- Canic compares identity, binding, revision, and terminal fingerprint. It
  does not interpret external receipts.

#### First consumer

- Toko owns `MintEffectRequest`, `MintReceipt`, `MintOutcome`,
  `execute_mint`, receipt observation, and absent-mint cancellation.
- Toko authenticates and hashes every mint-specific field, validates ledger
  evidence, then converts it to generic Canic `TerminalEvidence`.
- Toko ledger receipts are scoped by actual authorized instance caller and
  mint `OperationId`.
- A deterministic business rejection writes and returns a durable Toko
  no-effect receipt in the original execute response.
- Cancelling an absent mint requires the authenticated complete typed request;
  a caller-provided digest alone is insufficient.

#### Endpoint idempotency and settlement

- Receipt-backed IDs use one documented global `OperationId` namespace per
  canister. Toko owns domain-separated mint-ID derivation.
- Begin-or-load performs the direct ID lookup before admission or reservation.
- Only `Created` reserves. Existing pending and terminal entries perform no
  counter write; changed binding/resource/quantity fails without mutation.
- Settlement reloads the record, checks binding and expected revision,
  precomputes checked counter changes, and performs no `await`.
- Matching terminal evidence is idempotent. Contradictory evidence is an
  invariant failure.

#### One resource authority

- Local and receipt-backed intents use one resource aggregate. No mint or
  receipt-specific counter system is added.
- A maintained expirable-pending count drives the existing local cleanup
  timer.
- Receipt-backed records never enter the TTL pending index and cannot start or
  retain cleanup.
- Admission reads maintained totals and does not scan receipt-backed rows.

#### Recovery

- Normal applied and deterministic no-effect mint responses settle from the
  original Toko call.
- A post-call transport, decode, callback, or settlement failure leaves the
  Canic record `Pending`.
- Toko may perform at most one immediate targeted receipt observation.
- Later completion replays the same mint ID once; explicit cancellation uses
  the Toko-specific authenticated absent-mint fence.
- No reject code or error string is treated as proof that the call was not
  submitted.
- Lazy and admin recovery address one operation. No background reconciliation
  or fleet scan is added.

#### Permanent compact evidence

- Canic retains one fixed-size receipt-backed row per operation because IDs may
  be replayed indefinitely.
- One maintained record count and configured maximum gate new IDs without
  scanning. Existing IDs remain readable and settleable at capacity.
- Toko independently owns public admission and ledger-receipt capacity.
- A future replay horizon or reclamation protocol requires a separate design.

#### Performance invariant

- A normal mint still makes only its existing `execute_mint` call.
- Deterministic no-effect returns from that same call.
- Successful paths make zero receipt-read calls.
- Ambiguous handling makes at most one immediate targeted observation.
- Canic adds one exact begin/load lookup and bounded record/aggregate writes,
  with no submitted-phase write.
- There is no new timer, heartbeat, polling loop, global scan, generic receipt
  store, generic resolver, or high-cardinality metric.
- Added happy-path inter-canister calls, background work, map-wide scans,
  unbounded retry, or unexplained material instruction/Wasm growth block
  acceptance.

#### Explicit non-goals

- Canic does not add `EffectOperationKey`, `EffectKind`, a generic execute
  result, resolver, fence, receipt protocol, saga, or transaction engine.
- Claims, payment/delivery steps, refunds, and vendor reconciliation remain
  outside 0.90.
- Shared protocol abstractions may be reconsidered only after a second real
  consumer demonstrates the same requirement.

### 🧹 Cleanup

- Keep local intents lightweight and direct.
- Keep mint request, receipt, authorization, validation, and recovery logic in
  Toko.
- Close 0.90 after the minimal Canic primitive, Toko mint proof, migration,
  bounded recovery, and focused performance evidence.
