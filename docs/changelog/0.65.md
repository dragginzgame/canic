# 0.65 Changelog

## 0.65.9 - 2026-06-11 - Explicit delegated-token audiences

### Summary

- Replaces the legacy global delegated-token audience with explicit
  `DelegationAudience::Canister`, `DelegationAudience::CanicSubnet`, and
  `DelegationAudience::Project` variants.

- Requires delegated-token verification to match token and cert audiences
  against local verifier context instead of accepting a broad global Canic
  audience.

- Binds opaque signed token `ext` bytes into token-issue replay payload hashes,
  so a reused operation id cannot replay across different application payloads.

### Changed

- Canonical delegated-auth audience encoding now includes the audience kind and
  concrete canister, subnet, or project value.

- Token runtime verification now passes the current canister principal and
  optional local Canic subnet id into delegated-token verification.

- Test and PIC helper metadata mixing now distinguish canister, subnet, and
  project audiences.

## 0.65.7 - 2026-06-11 - Delegated-token signed ext payloads

### Summary

- Adds opaque `ext: Option<Vec<u8>>` application data to delegated-token issue
  requests and delegated-token claims.

- Includes `ext` inside canonical `DelegatedTokenClaims` bytes, so the current
  token signature binds the application payload and later issuer canister
  signatures will bind the same field.

- Bounds active token ext payloads to 4096 bytes.

### Changed

- Delegated-token preparation now copies requested ext bytes into the signed
  claims payload.

- Canonical claims encoding distinguishes absent ext from present ext and
  rejects oversized ext on both mint and verification paths.

- Token serialization benchmarks now include a representative signed ext
  payload.

## 0.65.6 - 2026-06-11 - Delegated-token positive verifier cache

### Summary

- Adds a bounded heap-only positive delegated-token verifier cache for repeated
  reusable tokens.

- Keys cache entries by the exact embedded delegation proof hash, claims hash,
  current token signature hash, and caller principal.

- Keeps the cache authority-neutral: cache values contain only
  `valid_until_ns` and `verified_at_ns`, and cache hits still rerun canonical
  token checks plus local time, audience, grant, subject, and endpoint-scope
  authorization.

### Changed

- Delegated-token runtime verification now performs a positive cache lookup
  before expensive root-proof and token-signature verification.

- The cache expires entries no later than the signed token/cert expiry boundary
  and evicts the oldest verified entries when the bounded capacity is reached.

- The lower delegated-token verifier now exposes a no-signature local-check path
  used only after an exact token/caller tuple has already been positively
  verified.

## 0.65.4 - 2026-06-10 - Zero-management-ECDSA auth closeout scope

### Summary

- Tightens the 0.65 closeout target from delegated-token root-proof-only
  canister signatures to zero-management-ECDSA normal auth.

- Records that root delegation proofs, issuer delegated-token proofs, and role
  attestations must use IC canister signatures before 0.65 closeout.

- Rejects inbound root-capability `RoleAttestationProof` envelopes as retired
  ECDSA capability proof input.

- Adds release-blocking zero-ECDSA cleanup status for issuer token signatures,
  delegated-grant capability proofs, role attestations, feature flags, Candid,
  and verifier purity.

### Changed

- The 0.65 design now states that normal auth must not call
  `management_canister.sign_with_ecdsa`, and that no normal verifier may depend
  on a root, issuer, or management-canister call on the protected request hot
  path.

- Ordinary token issuance uses issuer prepare/get with installed
  `ActiveDelegationProof`; root prepare/get is maintenance, bootstrap, and
  refresh traffic, not signup, ordinary token issuance, or verifier hot-path
  traffic.

- The design/status notes now treat threshold-ECDSA auth features, issuer ECDSA
  token proofs, delegated-grant ECDSA proofs, and old shard signature fields as
  closeout blockers. Issuer token proofs convert to IC canister signatures;
  delegated-grant ECDSA proofs are removed or hard-failed in 0.65.

- The design document has been re-reviewed and simplified around the active
  basic zero-ECDSA protocol: root proof refresh, issuer
  `ActiveDelegationProof`, issuer prepare/get token issuance, local
  delegated-token verification, and minimal role-attestation prepare/get.

### Removed

- Removed the accepted inbound root-capability role-attestation verifier path.
  `CapabilityProof::RoleAttestation` now fails before role-attestation
  signature verification or replay dispatch.

- Removed role-attestation capability proof blob encode/decode helpers and the
  `TryFrom<RoleAttestationProof>` success-path conversion. The DTO remains only
  as wire-decodable rejected input.

- Removed PIC coverage for successful root-capability role-attestation proof
  authorization and replaced it with hard-cut rejection coverage. Structural
  capability proofs and delegated-grant rejection coverage remain active.

### Audit

- Added a zero-ECDSA cleanup report that currently marks 0.65 closeout as not
  clean. Active source scans still show issuer/shard ECDSA signatures,
  threshold-ECDSA auth features, delegated-grant ECDSA authority, generated
  Candid residue, and verifier-purity items to classify or remove.

- Added a design-line status file separating landed root-proof work from
  remaining zero-ECDSA closeout blockers.

## 0.65.1 - 2026-06-10 - Superseded threshold-ECDSA feature split

### Summary

- Historical note: this patch split threshold-ECDSA public-key fetch support
  from threshold-ECDSA signing support during the earlier root-proof-only
  transition.

- The 0.65 closeout target supersedes this state. Normal auth removes
  threshold-ECDSA public-key fetches and signing from root, issuer,
  role-attestation, verifier, and endpoint paths.

- Final 0.65 issuer delegated-token proofs use
  `IssuerProof::IcCanisterSignatureV1`.

### Added

- Temporarily added `auth-threshold-ecdsa-public-key` to `canic-core` and the
  public `canic` facade. The zero-management-ECDSA closeout removes this from
  normal auth.

### Changed

- This temporary split allowed root proof issuers to compile without
  threshold-ECDSA signing support during the earlier transition.

- Final 0.65 root runtime startup for normal auth must not require
  `auth-threshold-ecdsa-public-key`.

- Final 0.65 issuer runtime startup for normal auth must not require
  `auth-threshold-ecdsa-sign`.

- Delegated grants must be removed or hard-failed before 0.65 closeout.

- The delegation root test canister now uses the public-key feature instead of
  the signing feature.

### Documentation

- Active auth architecture, getting-started, facade README, and generated
  Candid/API docs must be updated during the zero-management-ECDSA cleanup pass.

## 0.65.0 - 2026-06-10 - Canister-signature delegated auth

### Summary

- Hard-cuts delegated-token root proofs from root threshold ECDSA to IC
  canister signatures.

- Preserves the external delegated-token contract: endpoint tokens remain
  self-contained, carry the `ext` payload, and can authorize multiple update
  calls until expiry.

- Removes the normal-auth one-shot fresh-proof path that tried to hide root
  proof issuance behind a single update call.

### Breaking

- Delegated-token root proof verification accepts only
  `RootProof::IcCanisterSignatureV1`.

- Existing delegated tokens from the root threshold-ECDSA proof epoch are not
  accepted after upgrading verifiers to this line.

- Delegated-token verification uses configured root canister identity plus raw
  IC root public key bytes. It no longer trusts
  `SubnetState.auth.delegated_root_public_key` for delegated-token root proofs.

### Changed

- Root delegation proof issuance uses an update/query prepare-get flow for
  maintenance, bootstrap, and refresh. Ordinary issuer token issuance is not a
  root-first flow in the 0.65 closeout design.

- Root proof preparation is replay-protected as its own cost class. Proof
  retrieval is a deterministic query over an existing pending proof and is
  bounded by explicit pending-proof metadata.

- Root auth certified data is owned by a single path and uses the exact `"sig"`
  canister-signature tree shape for this line.

- Canister-signature message construction is pinned to the upstream IC helper
  behavior: creation stores the hash of `domain_len || domain || cert_hash`,
  and verification passes those bytes to the IC canister-signature verifier.

- Auth and replay protocol DTO timestamps now use nanoseconds internally and
  canonically. Operator-facing config can stay in seconds, but conversion
  happens at the boundary.

- Public normal-auth features are split by canister-signature role: root
  canister-signature creation/verification, issuer canister-signature
  creation/verification, role-attestation create/verify, and delegated-token
  verification.

- Final 0.65 token issuer proofs use `IssuerProof::IcCanisterSignatureV1`.
  Normal auth has no threshold-ECDSA token issuer path.

- Root capability calls that do not need delegated-token root proofs now use
  structural proofs for root cycles, direct-child provision, direct-child
  upgrade, and direct-child recycle paths.

### Removed

- Normal delegated-token auth no longer has a legacy root threshold-ECDSA
  verifier branch.

- Fresh one-shot root ECDSA role-attestation and internal-invocation proof
  issuance hard-fail before reserving signing cycles.

- The old one-shot fresh-proof `mint_token` flow is removed from the normal auth
  surface. Client and test helpers must perform the explicit prepare/get/issue
  sequence.

### Migration Notes

- Deploy issuers and clients for the prepare/get/issue flow before upgrading
  endpoint verifiers that reject old root ECDSA proof shapes, unless immediate
  invalidation of existing delegated tokens is acceptable.

- Configure endpoint verifiers with the root canister id and raw IC root public
  key for the target network. Local tests must inject the local replica root
  key rather than silently using the mainnet key.

- Do not deploy the root auth signing canister on a subnet whose canister
  signatures are invalid. The deployment truth check rejects `cloud_engine`
  subnets for this role.

- Revocation remains TTL-bound for self-contained tokens. Stronger emergency
  revocation would require a separate root-certified revocation or epoch model
  and is not part of this line.
