# 0.65 Changelog

## 0.65.2 - 2026-06-10 - Hard-cut auth surface cleanup

### Summary

- Removes obsolete normal-client routing for one-shot root ECDSA
  role-attestation and internal-invocation proof issuance.

- Removes the stale outbound protected-internal fresh-proof cache and public
  client surface.

- Keeps root rejection endpoints and protected endpoint descriptors as explicit
  retained failure/metadata surfaces after the hard cut.

### Removed

- Removed normal `AuthApi` RPC routing to one-shot root ECDSA
  role-attestation and internal-invocation proof issuance.

- Removed `CanicCall`, `CanicInternalClient`,
  `CanicInternalCallOptions`, `CanicInternalWaitMode`, and
  `canic_internal_client!`.

- Removed the outbound internal-invocation proof cache and protected-call
  envelope helper that depended on fresh one-shot proof issuance.

- Removed the project hub generated-client fixture dependency on the deleted
  protected-internal client path.

### Changed

- `RootAuthMaterialClient` now lists only the still-needed structural
  bootstrap calls for attestation key-set refresh and delegation proof
  preparation.

- Active docs direct normal parent/shard calls to delegated-token authenticated
  endpoints until a replacement protected-internal proof protocol exists.

## 0.65.1 - 2026-06-10 - Threshold ECDSA public-key feature split

### Summary

- Splits threshold-ECDSA public-key fetch support from threshold-ECDSA signing
  support.

- Keeps root delegated-token proof issuance on canister signatures while still
  allowing root to fetch threshold-ECDSA shard public keys for certified
  delegation certs.

- Keeps threshold-ECDSA signing scoped to shard token issuers.

### Added

- Added `auth-threshold-ecdsa-public-key` to `canic-core` and the public
  `canic` facade.

### Changed

- `auth-threshold-ecdsa-sign` now includes `auth-threshold-ecdsa-public-key`,
  preserving signer builds while allowing root proof issuers to compile without
  threshold-ECDSA signing support.

- Root runtime startup now requires `auth-root-canister-sig-create` plus
  `auth-threshold-ecdsa-public-key` when delegated-token proof issuance is
  configured.

- Non-root delegated token signers still require `auth-threshold-ecdsa-sign`.

- Legacy delegated-grant root public-key publication now skips cleanly when
  threshold-ECDSA public-key fetch support is not compiled.

- The delegation root test canister now uses the public-key feature instead of
  the signing feature.

### Documentation

- Updated active auth architecture, getting-started, facade README, and 0.65
  design docs to describe the public-key/signing feature split.

## 0.65.0 - 2026-06-10 - Canister-signature delegated auth

### Summary

- Hard-cuts delegated-token root proofs from root threshold ECDSA to IC
  canister signatures.

- Preserves the external delegated-token contract: endpoint tokens remain
  self-contained, carry the `ext` payload, and can authorize multiple update
  calls until expiry.

- Removes the normal-auth one-shot fresh-proof path that tried to hide root
  proof issuance behind a single update call.

### Breaking

- Delegated-token root proof verification accepts only
  `RootProof::IcCanisterSignatureV1`.

- Existing delegated tokens from the root threshold-ECDSA proof epoch are not
  accepted after upgrading verifiers to this line.

- Delegated-token verification uses configured root canister identity plus raw
  IC root public key bytes. It no longer trusts
  `SubnetState.auth.delegated_root_public_key` for delegated-token root proofs.

### Changed

- Fresh delegated-token issuance now uses an explicit update/query/update
  choreography: prepare the delegation proof at root, retrieve the IC
  certificate and witness by query, then issue the shard-signed token.

- Root proof preparation is replay-protected as its own cost class. Proof
  retrieval is a deterministic query over an existing pending proof and is
  bounded by explicit pending-proof metadata.

- Root auth certified data is owned by a single path and uses the exact `"sig"`
  canister-signature tree shape for this line.

- Canister-signature message construction is pinned to the upstream IC helper
  behavior: creation stores the hash of `domain_len || domain || cert_hash`,
  and verification passes those bytes to the IC canister-signature verifier.

- Auth and replay protocol DTO timestamps now use nanoseconds internally and
  canonically. Operator-facing config can stay in seconds, but conversion
  happens at the boundary.

- Public auth features are split by role: root canister-signature creation,
  root canister-signature verification, shard secp256k1 verification, shard
  threshold-ECDSA signing, and delegated-token verification.

- Shard token signing remains threshold ECDSA in this line. Root proof issuance
  no longer pays the root threshold-ECDSA cost, but callers should still reuse
  shard-issued delegated tokens for their TTL.

- Root capability calls that do not need delegated-token root proofs now use
  structural proofs for root cycles, direct-child provision, direct-child
  upgrade, and direct-child recycle paths.

### Removed

- Normal delegated-token auth no longer has a legacy root threshold-ECDSA
  verifier branch.

- Fresh one-shot root ECDSA role-attestation and internal-invocation proof
  issuance hard-fail before reserving signing cycles.

- The old one-shot fresh-proof `mint_token` flow is removed from the normal auth
  surface. Client and test helpers must perform the explicit prepare/get/issue
  sequence.

### Migration Notes

- Deploy issuers and clients for the prepare/get/issue flow before upgrading
  endpoint verifiers that reject old root ECDSA proof shapes, unless immediate
  invalidation of existing delegated tokens is acceptable.

- Configure endpoint verifiers with the root canister id and raw IC root public
  key for the target network. Local tests must inject the local replica root
  key rather than silently using the mainnet key.

- Do not deploy the root auth signing canister on a subnet whose canister
  signatures are invalid. The deployment truth check rejects `cloud_engine`
  subnets for this role.

- Revocation remains TTL-bound for self-contained tokens. Stronger emergency
  revocation would require a separate root-certified revocation or epoch model
  and is not part of this line.
