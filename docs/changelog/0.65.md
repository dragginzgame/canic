## 0.65.16 - 2026-06-11 - Role attestation canister signatures

### Summary

- Replaces `SignedRoleAttestation` ECDSA signatures with embedded root
  canister-signature proofs.

- Adds root `canic_prepare_role_attestation` update and
  `canic_get_role_attestation` query endpoints.

- Removes verifier-side ECDSA key refresh from the role-attestation hot path.

### Changed

- `SignedRoleAttestation` now carries `root_proof: RootProof` instead of
  `signature` plus `key_id`.

- Root role-attestation issuance now follows the same update/query mechanics as
  delegation proofs: prepare validates the caller/subject/role/subnet/TTL,
  stores caller-bound pending metadata, commits the `"sig"` certified data, and
  get returns the prepared payload plus its root canister-signature proof.

- `RootPayloadKind` now includes `RoleAttestation` with a distinct
  root seed/domain from `DelegationCert`.

- Local role-attestation verification hashes the attestation payload, verifies
  the embedded root canister-signature proof against the configured root
  canister id and raw IC root key, then applies subject, audience, subnet, time,
  and epoch checks without root or management-canister calls.

- The old direct `canic_request_role_attestation` one-shot endpoint,
  `AuthApi::request_role_attestation_root` wrapper, and role-attestation
  capability-RPC issuance variant are removed.

- Legacy standalone capability proof DTOs and disabled proof-mode branches are
  removed; root capability envelopes now accept only structural proofs.

- The runtime role-attestation key refresh timer is now an inert compatibility
  shell; canister-signature role attestations do not use verifier-local ECDSA
  public-key caches or key-rotation grace windows.

## 0.65.13 - 2026-06-11 - Issuer canister-signature primitive

### Summary

- Adds `auth-issuer-canister-sig-create` and
  `auth-issuer-canister-sig-verify` feature gates to `canic-core` and the
  facade `canic` crate.

- Expands the issuer canister-signature module from seed/domain helpers into
  internal prepare/get/verify primitives for issuer-signed delegated-token
  claims.

- Adds issuer-proof prepare metrics and an issuer canister-signature replay
  cost class for the later public token prepare endpoint.

### Changed

- Issuer proof creation now uses the same heap-only `SignatureMap` pattern as
  root proof creation: add the claim-hash signature input, refresh certified
  data to the exact `labeled_hash(b"sig", SIGNATURES.root_hash())` shape, and
  pin caller-bound pending retrieval metadata with the one-minute retrieval
  window.

- Issuer proof retrieval returns `IssuerProof::IcCanisterSignatureV1` with
  CBOR witness bytes and the issuer canister-signature public key DER for the
  requested issuer canister id and seed.

- Issuer proof verification validates the canister id and seed encoded in the
  canister-signature public key, then verifies
  `domain_len || issuer_domain || claims_hash` with the raw IC root key.

- This patch does not yet replace `DelegatedToken.shard_sig` with
  `issuer_proof`, regenerate Candid for delegated-token issue/get endpoints, or
  remove shard ECDSA normal-auth features.

## 0.65.0 - 2026-06-10 - Canister-signature delegated auth

### Summary

- Hard-cuts delegated-token root proofs from root threshold ECDSA to IC
  canister signatures.

- Preserves the external delegated-token contract: endpoint tokens remain
  self-contained, carry the `ext` payload, and can authorize multiple update
  calls until expiry.

- Removes the normal-auth one-shot fresh-proof path that tried to hide root
  proof issuance behind a single update call.

### Breaking

- Delegated-token root proof verification accepts only
  `RootProof::IcCanisterSignatureV1`.

- Existing delegated tokens from the root threshold-ECDSA proof epoch are not
  accepted after upgrading verifiers to this line.

- Delegated-token verification uses configured root canister identity plus raw
  IC root public key bytes. It no longer trusts
  `SubnetState.auth.delegated_root_public_key` for delegated-token root proofs.

### Changed

- Root delegation proof issuance uses an update/query prepare-get flow for
  maintenance, bootstrap, and refresh. Ordinary issuer token issuance is not a
  root-first flow in the 0.65 closeout design.

- Root proof preparation is replay-protected as its own cost class. Proof
  retrieval is a deterministic query over an existing pending proof and is
  bounded by explicit pending-proof metadata.

- Root auth certified data is owned by a single path and uses the exact `"sig"`
  canister-signature tree shape for this line.

- Canister-signature message construction is pinned to the upstream IC helper
  behavior: creation stores the hash of `domain_len || domain || cert_hash`,
  and verification passes those bytes to the IC canister-signature verifier.

- Auth and replay protocol DTO timestamps now use nanoseconds internally and
  canonically. Operator-facing config can stay in seconds, but conversion
  happens at the boundary.

- Public normal-auth features are split by canister-signature role: root
  canister-signature creation/verification, issuer canister-signature
  creation/verification, role-attestation create/verify, and delegated-token
  verification.

- Final 0.65 token issuer proofs use `IssuerProof::IcCanisterSignatureV1`.
  Normal auth has no threshold-ECDSA token issuer path.

- Root capability calls that do not need delegated-token root proofs now use
  structural proofs for root cycles, direct-child provision, direct-child
  upgrade, and direct-child recycle paths.

### Removed

- Normal delegated-token auth no longer has a legacy root threshold-ECDSA
  verifier branch.

- Fresh one-shot root ECDSA role-attestation and internal-invocation proof
  issuance hard-fail before reserving signing cycles.

- The old one-shot fresh-proof `mint_token` flow is removed from the normal auth
  surface. Client and test helpers must perform the explicit prepare/get/issue
  sequence.

### Migration Notes

- Deploy issuers and clients for the prepare/get/issue flow before upgrading
  endpoint verifiers that reject old root ECDSA proof shapes, unless immediate
  invalidation of existing delegated tokens is acceptable.

- Configure endpoint verifiers with the root canister id and raw IC root public
  key for the target network. Local tests must inject the local replica root
  key rather than silently using the mainnet key.

- Do not deploy the root auth signing canister on a subnet whose canister
  signatures are invalid. The deployment truth check rejects `cloud_engine`
  subnets for this role.

- Revocation remains TTL-bound for self-contained tokens. Stronger emergency
  revocation would require a separate root-certified revocation or epoch model
  and is not part of this line.
