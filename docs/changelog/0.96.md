# Canic 0.96 Changelog

## [0.96.5] - 2026-07-21 - Receipt Capacity Projection

### 📝 Summary

Completes the read-only half of Slice D without adding another stable-state
authority. Application receipt totals, pending/terminal classification,
reserved slots, physical reservation, logical headroom, and the next terminal
eligibility deadline now have one validated internal projection.

### 🔧 Changed

- Uses the application replay B-tree's maintained length as total receipt count
  and reserved terminal slots. Uses the eligibility B-tree's maintained length
  as terminal count and their checked difference as pending count. These are
  constant-time stable-structure counters, not primary-map scans.
- Derives the next eligibility deadline from the first ordered eligibility key.
  The exact schema, operation identity, primary ownership, replay metadata,
  binding, revision, and computed deadline are validated before projection.
- Runs the projection after complete lifecycle index reconciliation so upgrade
  rejects contradictory count or earliest-entry state instead of publishing
  misleading capacity information.
- Rejects a primary receipt count beyond the fixed 100,000-record ceiling
  before lifecycle rebuild mutates placement acknowledgement state.

### 🧹 Simplified

- Does not add a dedicated stable counter cell. Such a cell would duplicate
  values already maintained by the two canonical indexes, add another drift
  invariant, and consume an additional 8 MiB
  `MemoryManager` bucket. The retained design keeps allocation 47 as the final
  application receipt allocation and the measured high-water subtotal at
  4,737 pages, or 296.0625 MiB.

### 🧪 Testing

- Covers pending and terminal count transitions, record headroom, reserved
  slots/pages, exact first-key eligibility, and lifecycle projection.
- Retains the explicit 100,000-row capacity proof and all existing eligibility,
  settlement, snapshot, and corruption tests.

### 📡 Contract Notes

- Public Rust APIs, Candid, configuration, timers, stable schemas, allocation
  IDs, dependencies, and Cargo package versions are unchanged.
- This patch does not delete receipts or register cleanup work. It establishes
  the canonical constant-time observations required by bounded reclamation.

## [0.96.4] - 2026-07-21 - Terminal Eligibility And Settlement Capacity

### 📝 Summary

Establishes the canonical terminal-retention authority without yet deleting a
receipt. Application settlement now produces one exact ordered eligibility row,
and application admission first provisions enough stable memory for every
admitted pending operation to make that terminal transition.

### 🔧 Changed

- Freezes the terminal observation grace at exactly 24 hours, matching the
  maximum admitted replay horizon. Eligibility is the later of the immutable
  replay deadline and the terminal timestamp plus that grace. An operation
  admitted with the maximum replay window and settled at its final instant can
  therefore remain in receipt history for up to 48 hours from admission.
- Adds one ordered application terminal-eligibility authority at allocation 47.
  Its fixed-width key is `(eligible_at_ns, operation_id)`; its value binds the
  schema, operation ID, payload binding, and terminal revision.
- Inserts eligibility before changing the primary receipt or resource totals.
  Deadline overflow, an existing exact eligibility row, or unavailable
  reserved memory rejects without settling the receipt.
- Extends lifecycle reconciliation to validate every expected terminal entry
  before rebuilding placement acknowledgement state. Missing, extra,
  misordered, wrong-schema, wrong-identity, wrong-binding, or wrong-revision
  eligibility fails closed and preserves the previous placement index.

### 📊 Capacity

- Measures the maximum eligibility value at exactly 229 bytes. At 100,000
  rows, the ordered map consumes 726 Wasm pages for ascending insertion and
  626 pages for the deterministic permutation.
- Admission provisions the pinned B-tree maximum-live-node envelope. The
  calculation uses the dependency's minimum five entries per non-root node,
  its exact 2,362-byte node page and 16-byte allocator chunk header, and the
  fixed map/allocator headers. At 100,000 rows the reserved envelope is the
  measured 726 pages rather than a one-node-per-record over-allocation.
- The four-allocation application high-water subtotal—primary receipts,
  replay metadata, terminal eligibility, and maximum distinct resource
  totals—is 4,737 physical Wasm pages, or 296.0625 MiB, through the pinned base
  `MemoryManager`.

### 🧪 Testing

- Covers exact grace/deadline selection, timestamp overflow without mutation,
  settlement at logical record saturation, exact eligibility persistence,
  malformed fixed-width keys, canonical snapshot restoration, lifecycle
  rejection before placement-index mutation, state-manifest ownership, and
  the permanent allocation inventory.
- Retains and passes the explicit ignored 100,000-row capacity probe with both
  actual and admission-reserved eligibility memory.

### 🛡️ Remaining Gate

- This patch does not remove terminal receipts, register receipt cleanup with
  the timer authority, or expose capacity diagnostics. Pending records remain
  permanently protected and terminal records remain retained.
- Bounded reclamation, maintained counters, poisoned-earliest-entry behavior,
  warning thresholds, and downstream Toko identity/recovery/rate/resource
  conformance remain later 0.96 work.

### 📡 Contract Notes

- Public Rust APIs, Candid, configuration, dependencies, timers, resource
  accounting semantics, and Cargo package versions are unchanged.
- Allocation 47 is a new version-1 stable contract. There is no old reader,
  alias, fallback, or migration path. The reviewed downstream deployment has
  no Canic application receipt rows, so adoption requires no compatibility
  handling.

## [0.96.3] - 2026-07-21 - Linear Receipt Index Reconciliation

### 📝 Summary

Makes lifecycle reconciliation match the combined receipt authority introduced
in 0.96.2. Application replay metadata and receipt primaries are validated in
one ordered pass, and the placement acknowledgement index is rebuilt only
after every canonical relationship has passed.

### 🩹 Fixed

- Removes the lifecycle path that scanned every receipt while performing one
  application-metadata point lookup per primary, then scanned every metadata
  row while performing a primary point lookup in the other direction. At the
  100,000-row limit that path could add up to 200,000 stable-tree point lookups
  to the two full scans during init or upgrade.
- Replaces the placement-only method name and diagnostic with the hard-cut
  `reconcile_receipt_indexes` authority. Its name now covers both canonical
  application-adjunct validation and the derived placement rebuild; the old
  method does not remain as an alias.
- Preserves the existing placement acknowledgement index byte-for-byte when
  application metadata is missing, orphaned, wrong-schema, wrong-identity, or
  attached to a Canic-owned receipt. No partial clear or rebuild occurs on a
  rejected reconciliation.

### 🧪 Testing

- Covers mixed application and placement rows, every application-metadata
  contradiction class, non-mutation on failure, and exact successful placement
  index reconstruction after restoring canonical metadata.
- Retains exact per-operation fail-closed validation on begin, load, and
  settlement in addition to lifecycle reconciliation.

### 📡 Contract Notes

- Public Rust APIs, stable schemas and allocation IDs, Candid, configuration,
  timers, dependencies, resource accounting, and Cargo package versions are
  unchanged.
- This correction does not implement terminal eligibility or reclamation and
  does not invent any downstream rate, grace, backlog, or recovery policy.

## [0.96.2] - 2026-07-21 - Replay Deadline Admission Hard Cut

### 📝 Summary

Makes the public application receipt deadline mandatory and immutable. Exact
retained operations remain observable after their deadline, while an absent
expired or overlong operation rejects before record capacity, resource
capacity, reservation, or external-effect admission.

### 🔧 Changed

- Adds required `replay_deadline_ns` to `BeginReceiptBackedIntentInput` with no
  optional field, default, overload, alias, or compatibility wrapper.
- Adds typed `ReplayWindowClosed` and `ReplayWindowTooLong` begin decisions.
  Equality is closed, and the maximum accepted remaining window is exactly 24
  hours, inclusive, derived from the reviewed signed-token ceiling.
- Preserves the exact begin ordering: structural validation, retained lookup,
  immutable binding/deadline comparison, temporal decision, record capacity,
  then resource capacity and mutation.
- Keeps Canic-owned placement on a separate private admission type with no
  application deadline. Its primary record and acknowledgement cleanup remain
  unchanged.

### 🛡️ Stable Authority

- Adds one version-1 application replay adjunct at stable allocation 46. The
  common receipt primary at 43 remains canonical and its record schema is
  unchanged.
- Binds every application metadata row one-to-one to the exact operation ID.
  Missing, orphaned, wrong-schema, wrong-identity, or Canic-owned metadata
  fails closed through typed ops causes.
- Extends the state manifest and test-only canonical snapshots without adding
  a public export/import surface or fallback reader.
- Measures the exact maximum metadata encoding at 124 bytes. At 100,000 rows,
  the map consumes 442 ascending or 381 deterministic-permuted Wasm pages.
  Primary, metadata, and maximum distinct totals still occupy 3,969 physical
  pages in the measured ascending case after MemoryManager bucket rounding.

### 🧪 Testing

- Covers closed equality, inclusive maximum, overlong rejection, retained
  replay at the deadline, immutable deadline conflict, no-mutation rejection,
  malformed metadata, placement separation, stable snapshots, state
  descriptors, and the permanent authority/allocation inventory.
- Updates the artifact-backed intent adapter fixture to carry and project the
  hard-cut deadline decisions.
- Passes the focused PocketIC adapter with real Wasm installation, typed
  closed/overlong rejection, immutable deadline conflict, concurrent capacity
  admission, settlement, upgrade, and retained terminal replay.

### 🛡️ Remaining Gate

- This slice does not settle from time, delete pending or terminal rows,
  reserve terminal eligibility capacity, schedule cleanup, or change resource
  accounting.
- Terminal observation grace, Toko action vectors and recovery, numeric
  capacity bounds, eligibility, reservation, diagnostics, and bounded
  reclamation remain later 0.96 work.

### 📡 Contract Notes

- The public Rust receipt-begin API changes directly. Production Candid,
  configuration, dependencies, timer behavior, and Cargo package versions are
  unchanged.
- Toko remains read-only and has no existing Canic application receipt rows,
  so no migration or compatibility path is required.

## [0.96.1] - 2026-07-21 - Stable Capacity Evidence And Totals Hygiene

### 📝 Summary

Measures the real 100,000-row receipt footprint through the pinned stable-map
and memory-manager implementations. The slice fixes two independent capacity
defects while leaving the public receipt contract, replay behavior, timers,
and downstream policy unchanged.

### 📊 Audit

- Records exact maximum encodings of 441 bytes for pending receipt rows and
  617 bytes for terminal receipt rows under the current schema. The retained
  1,024-byte bound leaves room for the accepted deadline metadata without a
  speculative bound change.
- Measures allocation 43 at 2,707 Wasm pages for ascending insertion and
  1,899 pages for a deterministic permuted order at 100,000 maximum terminal
  rows.
- Measures the placement acknowledgement index at 452/317 pages and 100,000
  distinct maximum resource-total rows at 545/469 pages.
- Measures the three allocations together through the pinned base
  `MemoryManager`: their ascending receipt-capacity subtotal occupies 3,969
  physical Wasm pages, or 248.0625 MiB, including bucket rounding and excluding
  unrelated canister allocations.
- Retains the expensive 100,000-row probe as an explicit ignored capacity
  test so dependency or record-layout changes can be remeasured without
  slowing ordinary unit tests.

### 🩹 Fixed

- Corrects `IntentResourceTotalsRecord`'s stable bound from 64 to its exact
  69-byte maximum. A valid combination of large reserved, committed, and
  pending totals can no longer cross the declared bound and panic during
  stable insertion.
- Removes an exact resource-total row whenever all three totals return to
  zero. Reads remain unchanged because an absent row already projects the
  same zero value, while rollback and abort no longer leave empty stable
  tombstones.

### 🛡️ Remaining Gate

- Nonzero committed totals are authoritative and cannot be deleted with a
  terminal receipt. Toko must therefore bound the maximum durable resource
  cardinality per canister in addition to its operation rate and backlog.
- The eligibility-index shape and physical settlement reservation remain
  gated on the immutable replay deadline and observation-grace contract. No
  generic allocator, second primary store, guessed duration, or compatibility
  path is added.

### 📡 Contract Notes

- The totals record's serialized fields, schema meaning, stable key, and
  allocation remain unchanged; only the previously undersized storage bound
  is corrected.
- Public Rust APIs, Candid, receipt DTOs, timers, dependencies, configuration,
  and Cargo package versions are unchanged.

## [0.96.0] - 2026-07-21 - Receipt Contract And Consumer Inventory

### 📝 Summary

Starts the bounded receipt replay and terminal-reclamation line without
changing runtime behavior. The immutable `v0.95.10` release is the baseline,
the complete in-repository receipt path is inventoried, and product work
remains gated on exact downstream authorization and capacity evidence.

### 📊 Audit

- Confirms one common receipt primary at stable allocation 43, one
  placement-only acknowledgement index at allocation 45, one storage-ops
  mutation authority, and one thin public application facade.
- Confirms the only direct public-facade consumer in this repository is the
  PocketIC intent-authority fixture. A read-only Toko trace confirms its
  current Canic 0.71.3 dependency has no receipt consumer or old receipt rows.
- Freezes Toko's current mint graph: one authenticated project endpoint, one
  parent-only batch ledger effect, and one unused parent-only stack-mint
  bypass. Its signed token expiry is at most 24 hours, but it has no per-mint
  action identity, recovery endpoint, explicit batch bound, or rate limit.
- Traces application begin, load, settlement, shared resource accounting,
  placement acknowledgement, lifecycle rebuild, stable contract declarations,
  and focused tests end to end.
- Records two accepted P1 baseline findings: terminal application rows
  eventually exhaust the 100,000-row admission ceiling, and current rows have
  no replay deadline proving that deletion is safe.

### 🛡️ Implementation Gate

- Freezes direct lookup before temporal/capacity rejection, closed equality at
  `now >= replay_deadline_ns` for absent operations, indefinite pending
  retention, exact terminal reclamation, unchanged resource accounting, and
  separate placement acknowledgement ownership.
- Requires the Toko owner to supply per-action identity vectors, confirm the
  signed-expiry deadline, choose terminal grace, bound batch/rate/backlog,
  provide ledger replay/recovery evidence, and remove or integrate the
  stack-mint bypass before product implementation begins.
- Keeps the hard cut explicit: no optional deadline, alias, overload, fallback
  reader, migration shim, or inferred historical timestamp.

### 🧹 Scope Control

- Adds a permanent source inventory guard for receipt consumers, workflow/ops
  owners, stable-store access, and the existing stable allocations.
- Routes the remaining placement workflow test reset through the existing
  intent ops authority instead of mutating the stable receipt store directly.
- Clarifies that the common primary remains canonical and application replay
  metadata may only be an exact adjunct; placement does not invent an
  application deadline and no second primary store is added.
- Rejects a generic allocator, transaction framework, public export/import
  API, Toko domain model, ledger call, or background pending resolver. Capacity
  work must use the smallest mechanism justified by measured stable behavior.

### 📡 Contract Notes

- Public Rust APIs, Candid, stable records, memory allocations, timers,
  configuration, dependencies, and Cargo package versions are unchanged.
- The 0.95 status and closeout evidence now identify immutable `v0.95.10`
  rather than its pre-release input anchor.
