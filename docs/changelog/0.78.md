# 0.78 Changelog

## 0.78.5 - 2026-07-01 - Medic hard-cut survivorship cleanup

### 📝 Summary

- Removes remaining active-code survivorship for the deleted
  `canic info medic` route after the 0.78 hard cut.

- Keeps the cleanup at the CLI boundary: no canister calls, deployment truth,
  install-state mutation, medic report schema changes, or auth runtime changes.

### 🩹 Fixed

- `canic install` collision guidance now points operators at
  `canic medic deployment <deployment>` instead of the removed
  `canic info medic <deployment>` route.

- Top-level global ICP/network option forwarding no longer treats
  `info medic` as an active `canic info` leaf. The removed route still rejects
  as a usage error, but the dispatch plumbing no longer preserves it as a
  forwardable command shape.

## 0.78.4 - 2026-07-01 - Medic diagnostic-code alignment

### 📝 Summary

- Tightens the post-0.78.3 medic readiness line by aligning operator-facing
  diagnostic codes and target resolution with the 0.78 design contract.

- Keeps the changes diagnostic/CLI-boundary only: no deployment truth writes,
  canister mutation, controller changes, or auth runtime behavior changes.

### 🩹 Fixed

- Medic now reports `icp_cli_missing` when the ICP binary is absent instead of
  folding that setup failure into generic `icp_cli_incompatible` diagnostics.

- Targeted auth-renewal medic now reports `auth_renewal_issuer_invalid` for
  invalid issuer principals before treating other auth-renewal command failures
  as `auth_renewal_drift_fail`.

- Deployment-scoped medic no longer inherits project-only
  `local_network_implicit` / `local_network_explicit` diagnostics before its
  deployment network check.

- Blob-storage target resolution now treats principal text as a canister ID
  before same-text role-name lookup, while still using registry role metadata
  when that principal is a registered canister.

## 0.78.3 - 2026-07-01 - Medic readiness diagnostics

### 📝 Summary

- Expands top-level medic preflight coverage while keeping medic
  diagnostic-only: the new checks read local config, installed-deployment
  state, local deployment-truth receipts, and existing registry observations
  without mutating project files, deployment truth, canisters, controllers, or
  installed state.

- Makes common operator mistakes more explicit before targeted readiness
  checks, especially fleet-template versus deployment-target confusion and
  blob-storage target-resolution failures.

### ➕ Added

- `canic medic project` now validates discovered role package metadata from
  existing `Cargo.toml` files. Matching deployable roles report
  `role_package_metadata_present`; missing or mismatched metadata reports
  `role_package_metadata_missing`; declared-only roles that are not attached
  to deployable topology warn with `declared_role_not_deployable`.

- Missing deployment-target medic runs now warn when the requested deployment
  name exactly matches a known fleet template or role, using
  `fleet_name_deployment_name_conflated` and
  `role_name_deployment_name_conflated`.

- `canic medic deployment <deployment>` now reports the latest local
  deployment-truth receipt completeness. Complete succeeded receipts emit
  `deployment_truth_complete`; missing, unfinished, invalid, inconsistent, or
  partial receipts emit `deployment_truth_incomplete` with severity based on
  whether mutation may already have happened.

- Deployment-scoped medic now smoke-checks installed deployment registry
  observation through the existing installed-deployment resolver, emitting
  `deployment_registry_observed`, `deployment_registry_empty`,
  `deployment_registry_unavailable`, or `deployment_registry_not_evaluated`
  before targeted blob-storage and auth-renewal diagnostics.

### 🔧 Changed

- Targeted blob-storage medic errors now preserve the stable target-resolution
  codes promised by the 0.78 design:

```text
blob_storage_target_missing
blob_storage_target_ambiguous
blob_storage_target_not_blob_storage
```

- Generic blob-storage transport/readiness failures still report
  `blob_storage_billing_unready`, so automation can distinguish target
  selection problems from canister readiness problems.

### 📚 Documentation

- Updated the 0.78 design code list to include the implemented
  deployment-truth completion and deployment registry observation codes.

### 🧪 Testing

Validated with:

cargo test --locked -p canic-cli

## 0.78.2 - 2026-07-01 - Deployment medic state preconditions

### 📝 Summary

- Tightens deployment-scoped medic so local installed-deployment state
  preconditions are reported explicitly before medic attempts a live root
  readiness query.

- Keeps the behavior diagnostic-only: medic still does not repair deployment
  records, mutate local state, create canisters, or write deployment truth.

### 🩹 Fixed

- Deployment medic now reports `deployment_network_match` or
  `deployment_network_mismatch` from the installed deployment record before
  checking the root canister.

- Deployment medic now reports `root_canister_id_present` or
  `root_canister_id_missing` before attempting root readiness.

- When root readiness is skipped because those local deployment-state
  preconditions fail, medic emits `root_readiness_not_evaluated` instead of
  silently omitting the readiness check.

### 🧹 Cleanup

- Extracted installed-deployment follow-up checks out of the deployment medic
  orchestration path, keeping the command flow under the workspace line-count
  lint while making the root-readiness preconditions easier to test.

## 0.78.1 - 2026-07-01 - Medic exit-code hardening

### 📝 Summary

- Tightens the second 0.78 medic preflight slice around the CLI boundary
  contract: rejected hard-cut medic forms are usage errors, failed medic
  reports do not duplicate diagnostics on stderr, and JSON output remains a
  pure machine-readable report.

- Cleans up the current `significant_drop_tightening` warning set after moving
  the lint to warning level, using explicit guard drops where ownership permits
  and narrow expectations where PocketIC fixtures intentionally borrow shared
  setup state for a full test.

### 🩹 Fixed

- `canic info` usage errors now map through the top-level CLI exit-code helper
  as usage failures with exit code `2`, matching the 0.78 command-contract
  rule for invalid arguments and rejected hard-cut forms.

- Rejected medic command shapes now have regression coverage proving they
  return exit code `2` and still render human diagnostics:

```bash
canic medic <deployment>
canic medic --blob-storage <target>
canic medic project --auth-renewal <issuer>
canic info medic <deployment>
```

- Medic error handling now has focused tests for the designed process exit
  codes:

```text
1 = aggregate status is fail
2 = usage error / invalid arguments / rejected hard-cut form
3 = internal report construction or JSON render failure
```

- Medic JSON renderer tests now assert the payload is parseable JSON and does
  not include text-report fragments such as `status:`, `detail:`, or
  `source:`.

### 🧹 Cleanup

- Shortened significant drop lifetimes in host cargo-metadata and root/PocketIC
  test helpers where the final setup guard use was earlier than the end of the
  test body.

- Added narrow `#[expect(clippy::significant_drop_tightening)]` annotations for
  PocketIC role-attestation tests where the local `pic` borrow intentionally
  keeps the cached setup owner alive for the test duration.

### 🧪 Testing

## 0.78.0 - 2026-07-01 - Top-level medic preflight

### 📝 Summary

- Starts the 0.78 operator-readiness line by making `canic medic` the
  canonical preflight and diagnostic surface.

- Keeps medic diagnostic-only: it reports local project and selected
  deployment readiness checks without creating deployment records, writing
  deployment truth, mutating canisters, or running repair actions.

- Hard-cuts away the old `canic info medic` command surface instead of keeping
  a compatibility route.

### ➕ Added

- Added `canic medic` and `canic medic project` for project-scoped checks that
  do not require an installed deployment.

- Added explicit `canic medic deployment <deployment>` for deployment-scoped
  checks, preserving the existing installed-deployment, root readiness, config
  path, local network, ICP CLI, and identity-session diagnostics under the new
  top-level command surface.

- Preserved targeted deployment-only checks under the new command shape:

```bash
canic medic deployment <deployment> --blob-storage <canister-or-role>
canic medic deployment <deployment> --auth-renewal <issuer-principal>
```

- Added schema-versioned text and JSON medic reports with stable
  `schema_version`, `command`, `scope`, `network`, `deployment`, `status`, and
  `checks` fields.

- Added deterministic check ordering by status and category, plus stable check
  fields for `category`, `code`, `status`, `subject`, `detail`, `next`, and
  `source`.

### 🔧 Changed

- `--json` now works on all supported medic forms and renders the structured
  report directly to stdout.

- `canic medic deployment <deployment>` reports the effective deployment
  network while project-only reports leave `network` nullable when no network
  was explicitly selected.

- Medic command failures now use clearer exit behavior: usage errors exit `2`,
  report render failures exit `3`, and failing medic reports exit `1` without
  duplicating report text on stderr.

### 🗑️ Removed

- Removed active `canic info medic` dispatch.

- Rejected shorthand or old command forms instead of accepting aliases:

```bash
canic medic <deployment>
canic medic --blob-storage <target>
canic medic --auth-renewal <issuer>
canic medic project --blob-storage <target>
canic medic project --auth-renewal <issuer>
```

### 📚 Documentation

- Updated active README, install, getting-started, and operations docs to use
  the top-level medic command surface.

- Updated the compact handoff so new sessions treat the first 0.78 medic slice
  as implemented and continue with follow-up preflight hardening.

### 🧪 Testing

cargo test --locked -p canic-cli --quiet
cargo clippy --locked -p canic-cli --lib --tests -- -D warnings
cargo test --locked -p canic-host deployment_truth --lib
cargo test --locked -p canic --test changelog_governance
