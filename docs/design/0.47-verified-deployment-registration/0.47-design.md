# 0.47 Design: Verified Deployment-Target Registration

## Status

0.47 is in progress. The design is intentionally narrow: it closes the
registered-root recovery caveat from 0.46 without becoming a general deployment
verification line.

0.47 closes the main caveat intentionally left by 0.46: explicit recovery
registration can record a known live deployment root, but that root remains
`not_verified` and cannot authorize mutation.

Core goal:

```text
A registered deployment root can become verified only through bound
deployment-truth evidence, without treating operator assertion, fleet-template
identity, stale local state, or passive comparison evidence as deployment
authority.
```

## Implementation Status

Landed:

- explicit `DeploymentRootObservationV1` evidence on deployment inventory;
- passive `DeploymentRootVerificationRequestV1` /
  `DeploymentRootVerificationReportV1` model and validation;
- rejection of local-state-only root evidence for verification;
- install gating that treats `unverified_deployment_root` as a hard blocker;
- read-only guards for `deploy check`, `deploy compare`, and passive
  `deploy root inspect`;
- `canic deploy root verify <deployment> --from-check <path>`;
- `DeploymentRootVerificationReceiptV1` emitted only after successful local
  state transition or verified same-root no-op;
- local-state promotion from `not_verified` to `verified` through explicit
  root verification;
- same-root re-verification as a receipt-emitting no-op;
- rejection of verified-root replacement attempts;
- archived report/receipt validation for malformed digest fields, forged check
  rows, unsupported source-check schemas, stale source-check diff/report
  content, duplicate or unexpected root-verification check rows, and
  observed-root source drift.

Remaining closeout work should be limited to wording cleanup, focused guard
coverage, and final audit. The main functional release gate has landed: an
operator can register a known deployment root as `not_verified`, produce a
deployment-truth check with explicit root evidence, run
`canic deploy root verify <deployment> --from-check <file>`, receive a
`DeploymentRootVerificationReceiptV1`, and record local state as
`root_verification = verified`.

## Current Code Audit

The current repository state after the 0.46 closeout supports this design as
the next narrow line:

- `RootVerificationStatus` already serializes `verified` and `not_verified`.
- `canic deploy register <deployment> --fleet-template <fleet> --root
  <principal> --allow-unverified` writes minimal deployment-target state and
  always records `root_verification = not_verified`.
- Deployment-target state lives at
  `.canic/<network>/deployments/<deployment>.json`; old
  `.canic/<network>/fleets/<fleet>.json` state is rejected as deployment truth.
- Deployment truth surfaces an unverified registered root as
  `local_state.unverified_root_canister_id`, and the install safety gate blocks
  mutation on that finding.
- `canic deploy check` and `canic deploy compare` have source guards that keep
  them read-only.
- `canic deploy compare --left <file> --right <file>` compares archived
  `DeploymentCheckV1` artifacts passively and cannot promote either side's
  root verification state.
- `canic deploy root verify <deployment> --from-check <file>` now promotes a
  registered root from `not_verified` to `verified` only when the source
  `DeploymentCheckV1` carries explicit matching root evidence and the local
  state digest has not changed before write.

Therefore the remaining 0.47 work should harden this explicit verification
transition and its receipt semantics. It should not reopen 0.46's hard cut,
add a broad deployment verification system, or implement
group/catalog/teardown/test-deployment features.

Observed-root evidence check:

- 0.47 starts by adding explicit `DeploymentRootObservationV1` evidence to
  `DeploymentCheckV1.inventory.observed_root`, so root verification can bind to
  deployment-truth facts instead of inferring the root from request paths,
  local state, filenames, fleet-template identity, or deployment names.
- `DeploymentCheckV1.plan.deployment_identity.root_principal` records the
  planned root only when local deployment state is already `verified`.
- `DeploymentCheckV1.inventory.observed_identity.root_principal` records the
  root principal observed from deployment-target state.
- `DeploymentCheckV1.inventory.observed_canisters` can include the root
  canister as `role = "root"` with `canister_id`, `root_trust_anchor`, and a
  `role_assignment_source` such as `icp_canister_status` or
  `local_install_state`.

Only the explicit `observed_root` evidence can satisfy the 0.47
root-verification report. Local-state-only observations remain blockers.

Offline inventory note:

- a deployment check whose observed root source is `LocalDeploymentState` is
  honest inventory, but it is not verification evidence;
- root verification requires explicit observed-root evidence from
  `IcpCanisterStatus`;
- if a development or offline environment cannot produce `IcpCanisterStatus`
  evidence, it can still inspect reports, but it cannot promote root trust.

## Scope Calibration

Release bar:

- verify an explicitly registered deployment root from a bound
  deployment-truth artifact;
- persist verified root state only after strict deployment-target identity
  checks pass;
- keep failed or incomplete verification fail-closed;
- allow mutation paths to distinguish `not_verified` from `verified`;
- keep `deploy check` and passive comparison read-only.

Design hooks:

- future live root verification source;
- future full `canic deploy verify` profile system;
- future deployment catalog trust summaries;
- future group-level readiness that can distinguish verified and unverified
  targets.

Post-47 boundary:

- full `canic deploy verify`;
- live inventory crawler for arbitrary deployment verification;
- wall-clock freshness or observation-age policy;
- metrics/protected-call/runtime profile verification;
- deployment group/catalog UX;
- root rotation or root replacement;
- comparison diffing of `root_verification`;
- generalized verification profiles;
- richer persisted verification evidence objects inside install state;
- teardown/test-deployment orchestration;
- trust-domain migration.

0.47 is not a general live verification line. It is a narrow
root-registration verification line. It does not prove full deployment health;
it verifies only that the registered deployment root is bound to
deployment-truth evidence strongly enough to remove the `not_verified`
recovery blocker.

## Dependency On Earlier Lines

0.46 established the deployment-target identity hard cut:

- fleet templates are reusable topology inputs;
- deployments are concrete live-state targets;
- local state lives under `.canic/<network>/deployments/<deployment>.json`;
- legacy `.canic/<network>/fleets/<fleet>.json` live state is refused;
- supplied-plan install requires exact deployment-target identity;
- explicit `deploy register` recovery writes minimal state only;
- registered roots remain `not_verified`;
- unverified roots block mutation.

0.47 must preserve that boundary. Verification must attach to a deployment
target, not to a fleet template.

0.45 established the external lifecycle verification boundary:

```text
Consent is not completion.
Reported completion is not verification.
Supplied observation evidence is not live verification.
Verification requires deployment-truth inventory evidence.
```

0.47 should follow the same evidence discipline: registration is not
verification, operator assertion is not verification, and passive comparison is
not verification. Deployment-truth evidence is required.

0.45 also already uses a narrow bridge over existing `DeploymentCheckV1`
inventory artifacts when live crawling is not available. 0.47 should reuse that
pattern rather than inventing a broader live verification system.

## Problem

After 0.46, an operator can recover a known live deployment root:

```bash
canic deploy register <deployment> \
  --fleet-template <fleet> \
  --root <principal> \
  --allow-unverified
```

That is safe because it writes minimal local state and records the root as
`not_verified`.

However, the recovered deployment cannot safely authorize install mutation
until Canic has evidence that the registered root actually belongs to the
intended deployment target.

The missing operation is:

```text
registered but blocked
  -> registered, verified by deployment-truth evidence, and eligible for
     guarded mutation
```

0.47 provides that promotion path without weakening 0.46's identity hard cut.

## Design Principles

### 1. Registration Is Not Verification

`deploy register` remains a recovery operation only.

It may create local deployment-target state, but it must not mark the root as
trusted unless a separate verification operation succeeds.

### 2. Verification Is Deployment-Target Scoped

Verification must bind to:

- `network`;
- `deployment_name`;
- `fleet_template`;
- `root_principal`.

A matching fleet template is not enough. A matching root principal is not
enough. The deployment target identity must match exactly.

### 3. Verification Is Evidence-Backed

The first 0.47 implementation should verify from an existing
`DeploymentCheckV1` artifact.

The verification source should be explicit:

```text
DeploymentRootVerificationSourceV1 {
  DeploymentTruthCheck,
}
```

Future lines may add live verification sources, but this line should not imply
live crawling.

### 4. Verification Must Be Auditable

A successful verification should produce a receipt and persist local state.

A failed verification should produce a report but must not update trusted local
state.

`DeploymentRootVerificationReportV1` is passive. It may exist for successful
or failed evidence evaluation, but it never records a trusted transition by
itself.

`DeploymentRootVerificationReceiptV1` is mutating evidence. It exists only
after local deployment-target state has been written successfully.

### 5. Read-Only Commands Stay Read-Only

`canic deploy check` and `canic deploy compare` must not silently update
`root_verification`.

A successful install may write `root_verification = verified` only for a root
created or proven by that install operation's own guarded receipt path. It
must not promote a previously registered `not_verified` recovery root.

The explicit root-verification command is the only path that may promote an
existing registered root from `not_verified` to `verified`.

The following commands must not promote root verification:

- `canic deploy register`;
- `canic deploy check`;
- `canic deploy compare`;
- `canic deploy root inspect`;
- supplied-plan install over a previously registered unverified root.

## Root Verification State

0.46 introduced deployment-target local state with a root verification field.

0.47 should make the state transition explicit:

```text
RootVerificationStateV1 {
  NotVerified,
  Verified,
}
```

The current implementation already has `RootVerificationStatus` with equivalent
serialized strings. 0.47 should avoid unnecessary enum churn and instead harden
validation and receipt generation around the existing shape.

Suggested expanded persisted evidence shape:

```text
DeploymentRootVerificationV1 {
  status,
  root_principal,
  verified_at_unix_secs,
  verification_source,
  verification_receipt_id,
  verification_receipt_digest,
  source_check_id,
  source_check_digest,
  source_deployment_plan_id,
  source_deployment_plan_digest,
  source_inventory_id,
  source_inventory_digest,
  observed_deployment_name,
  observed_fleet_template,
  observed_network,
  blockers,
  warnings,
}
```

For `not_verified`, most source fields may be absent.

For `verified`, source evidence must be present.

## Verification Request

The first request shape should be artifact-backed:

```text
DeploymentRootVerificationRequestV1 {
  deployment_name,
  network,
  expected_fleet_template,
  expected_root_principal,
  source,
  deployment_check,
}
```

Where:

```text
DeploymentRootVerificationSourceV1 {
  DeploymentTruthCheck,
}
```

The request embeds or references a `DeploymentCheckV1` artifact. If the CLI
accepts a file path, the command can read the JSON file and construct the
request internally.

The request must not accept only a root principal and deployment name as proof.

## Verification Report

Before writing local state, Canic should produce a report:

```text
DeploymentRootVerificationReportV1 {
  report_id,
  report_digest,
  evidence_status,
  state_transition,
  deployment_name,
  network,
  expected_fleet_template,
  expected_root_principal,
  observed_deployment_name,
  observed_network,
  observed_fleet_template,
  observed_root_principal,
  observed_root_canister_id,
  observed_root_observation_source,
  source,
  source_check_id,
  source_check_digest,
  source_deployment_plan_id,
  source_deployment_plan_digest,
  source_inventory_id,
  source_inventory_digest,
  identity_checks,
  evidence_checks,
  blockers,
  warnings,
  recommended_next_actions,
}
```

Suggested evidence status enum:

```text
DeploymentRootVerificationEvidenceStatusV1 {
  EvidenceSatisfied,
  VerificationFailed,
  NotApplicable,
}
```

`EvidenceSatisfied` should require every required identity and evidence check
to pass.

`VerificationFailed` should include blockers and next actions.

`NotApplicable` may be used when the deployment is not registered locally, the
state is malformed, or the request is structurally invalid before evidence
evaluation.

Suggested state-transition enum:

```text
DeploymentRootVerificationStateTransitionV1 {
  NotAttempted,
  WouldPromoteNotVerifiedToVerified,
  PromotedNotVerifiedToVerified,
  NoStateChange,
  Blocked,
}
```

`EvidenceSatisfied` is not itself persisted verified state. Only local
deployment-target state carrying `root_verification = verified` together with
a successful `DeploymentRootVerificationReceiptV1` records the trusted
transition.

Read-only inspection reports must use `NotAttempted`,
`WouldPromoteNotVerifiedToVerified`, `NoStateChange`, or `Blocked`; they must
not emit `PromotedNotVerifiedToVerified` because no local-state transition was
performed.

## Verification Receipt

If verification writes local state, it should emit a receipt:

```text
DeploymentRootVerificationReceiptV1 {
  receipt_id,
  receipt_digest,
  deployment_name,
  network,
  fleet_template,
  root_principal,
  previous_root_verification,
  new_root_verification,
  state_transition,
  source_report_id,
  source_report_digest,
  source_report_evidence_status,
  source_report_state_transition,
  source_root_observation_source,
  source_observed_root_canister_id,
  source_check_id,
  source_check_digest,
  source_deployment_plan_id,
  source_deployment_plan_digest,
  source_inventory_id,
  source_inventory_digest,
  verified_at_unix_secs,
  local_state_path,
  local_state_digest_before,
  local_state_digest_after,
  warnings,
}
```

A receipt exists only for a successful write.

A failed verification report must not be serialized as a successful receipt.

## Required Validation

A verification attempt must fail unless all required facts match.

Required local-state checks:

- deployment state exists under
  `.canic/<network>/deployments/<deployment>.json`;
- state records the same `deployment_name`;
- state records the same `fleet_template`;
- state records the same root principal;
- state is not legacy fleet-named state;
- state does not contain stale pre-cut fields such as duplicate `fleet` or
  `installed_at_unix_secs`;
- existing `root_verification` is `not_verified` or an allowed reverify state.

Required source-check checks:

- `DeploymentCheckV1` is structurally valid;
- check digest matches recomputed content;
- deployment plan ID and digest are valid;
- inventory ID and digest are valid if the check exposes both;
- observed deployment name matches expected deployment name;
- observed network matches expected network;
- observed fleet template matches expected fleet template;
- observed root principal matches expected root principal;
- the check was generated from deployment-target state, not legacy fleet state;
- blocked or unsafe check status cannot be laundered into root verification,
  except for the exact `unverified_deployment_root` blocker that this command
  is evaluating;
- the observed root principal is present as deployment-truth evidence in the
  source check. It may not be inferred from the request path, file name, local
  state alone, or the operator-supplied expected root.
- if the current `DeploymentCheckV1` format does not yet carry explicit
  observed root-principal evidence, 0.47 must add that evidence before root
  verification can produce `EvidenceSatisfied`.

Initial policy recommendation:

- any blocked source check blocks root verification unless the only hard
  blocker is the expected `unverified_deployment_root` finding for the same
  registered root;
- the `unverified_deployment_root` exception is allowed only when it is the
  sole hard blocker, names the same deployment target, network, fleet template,
  and root principal, and the source check ID/digest, deployment-plan
  ID/digest, and inventory ID/digest all validate;
- warnings may be preserved on the verification report but do not necessarily
  block verification unless they concern identity, root, controller, inventory,
  or digest evidence.

That policy is conservative and release-safe.

For 0.47, stale evidence means structurally stale or drifted evidence:
mismatched IDs, mismatched digests, legacy fleet-state evidence, missing source
facts, or evidence that no longer binds to the registered deployment target.
0.47 does not add a live crawler or general wall-clock observation-age policy.
If wall-clock freshness is enforced, it must be explicit in the request and
report shape.

## Local State Mutation

Only this explicit command may promote an existing registered root from
`not_verified` to `verified`:

```bash
canic deploy root verify <deployment> --from-check <path>
```

Alternative considered:

```bash
canic deploy register verify <deployment> --from-check <path>
```

Preferred command:

```bash
canic deploy root verify <deployment> --from-check <path>
```

Reason:

- the subject is the deployment root;
- the command does not imply full deployment verification;
- it avoids overloading `deploy register`;
- it keeps future `deploy verify` available for the broader verification
  profile system.

Successful command behavior:

1. Read local deployment-target state.
2. Read archived `DeploymentCheckV1` artifact.
3. Validate source check.
4. Compare local state identity to source evidence.
5. Produce `DeploymentRootVerificationReportV1`.
6. If evidence is satisfied and the transition is allowed, update
   `root_verification` to `verified`.
7. Emit `DeploymentRootVerificationReceiptV1`.

Failed command behavior:

1. Read local deployment-target state if present.
2. Read archived `DeploymentCheckV1` artifact if present.
3. Produce failure report with blockers.
4. Do not update local trusted state.
5. Do not emit a successful receipt.

The mutating command must write verified state only if the local deployment
state digest at write time still equals `local_state_digest_before`. If local
state changes after report construction, verification fails closed and no
successful receipt is emitted.

0.47 allows:

- promotion from `not_verified` to `verified` for the same registered root;
- no-op re-verification of an already verified same root, if all source
  evidence still matches.

0.47 does not allow:

- changing the registered root principal through root verification;
- replacing a verified root with a different root;
- rotating root authority.

Root rotation is deferred to a later line and must have a separate receipt
shape.

## CLI Surface

Primary operator surface:

```bash
canic deploy root verify <deployment> --from-check <deployment-check.json>
```

Optional advanced inspection surface:

```bash
canic deploy root inspect --request <root-verification-request.json>
```

The inspection command may produce a report without writing local state. It is
for advanced review and debugging; it is not the release operator workflow
because it requires a prebuilt `DeploymentRootVerificationRequestV1`.

The mutating command must make mutation explicit in help text.

Help text should say:

```text
Verifies a registered deployment root from a deployment-truth check artifact
and records verified root state only when deployment target identity and source
evidence match.
```

It should not say:

```text
Verifies the deployment.
```

That phrase is too broad and should remain reserved for the later full
verification line.

## Interaction With Install

Install/preflight currently blocks unverified registered roots.

0.47 should preserve that blocker until verified root state exists.

After successful root verification:

```text
root_verification = verified
```

the install path may proceed past the unverified-root blocker, while still
running all ordinary deployment safety checks.

## Meaning Of Verified Root State

`root_verification = verified` means only that the registered root principal
for this deployment target was bound to accepted deployment-truth evidence.

Root verification does not mean:

- the deployment plan is safe;
- all role module hashes are current;
- external lifecycle work is complete;
- protected-call readiness is satisfied;
- promotion provenance is valid;
- the deployment is globally healthy.

It is not a deployment safety verdict, not an install receipt, not a live
health check, not a protected-call readiness result, and not external lifecycle
completion.

## Interaction With Deploy Check

`canic deploy check` remains read-only.

It may report:

```text
root_verification: not_verified
```

or:

```text
root_verification: verified
```

but it must not silently rewrite local state.

If deploy check sees a check artifact or live evidence that appears to verify a
root, it may recommend:

```bash
canic deploy root verify <deployment> --from-check <path>
```

but it must not perform that action implicitly.

## Interaction With Deploy Compare

`canic deploy compare` remains passive.

Comparison may include root verification state as a comparison category:

```text
left.root_verification
right.root_verification
```

but comparison must not promote either side to verified.

A matching comparison result is not verification.

## Interaction With 0.45 External Lifecycle

0.47 must not weaken the 0.45 external lifecycle boundary.

External lifecycle supplied observations, consent evidence, reported external
actions, and passive completion artifacts are not valid root-verification
sources.

A `DeploymentCheckV1` that already includes deployment-truth inventory evidence
may be used if it satisfies the 0.47 root identity requirements.

## JSON And Automation Stability

0.47 should keep public JSON field names explicit.

Prefer:

- `deployment_name`;
- `fleet_template`;
- `root_principal`;
- `root_verification`;
- `verification_source`;
- `source_check_id`;
- `source_check_digest`;
- `source_inventory_id`;
- `source_inventory_digest`.

Avoid ambiguous fields:

- `fleet`;
- `root`;
- `verified`;
- `source`.

Any new enum strings should be stable and low-risk for automation:

- `not_verified`;
- `verified`;
- `deployment_truth_check`;
- `evidence_satisfied`;
- `verification_failed`;
- `not_applicable`;
- `would_promote_not_verified_to_verified`;
- `promoted_not_verified_to_verified`;
- `no_state_change`;
- `blocked`.

If existing serialized values already use a different casing convention, follow
the repository convention rather than introducing a mixed style.

## Security Properties

0.47 should guarantee:

- no operator assertion alone can verify a root;
- no fleet-template identity alone can verify a root;
- no root principal alone can verify a deployment target;
- no old `.canic/<network>/fleets/<fleet>.json` state can become trusted
  deployment state;
- no passive comparison report can verify a root;
- no passive root-verification report can verify a root without a successful
  local-state write and receipt;
- no stale or digest-drifted check can verify a root;
- no failed verification partially updates local state;
- no read-only command updates `root_verification`;
- no supplied-plan install over a previously registered unverified root
  promotes that root to verified;
- no unverified registered root authorizes mutation.

## Non-Goals

0.47 should not:

- implement full `canic deploy verify`;
- crawl live inventory for arbitrary deployments;
- make `deploy register` verify roots by default;
- migrate legacy fleet-named live state automatically;
- treat comparison as verification;
- treat external lifecycle consent or supplied observation as verification;
- add wall-clock freshness or observation-age policy;
- rotate, replace, or migrate root authority;
- add generalized verification profiles or a deployment catalog trust summary;
- implement deployment groups, catalog UX, teardown, or test-deployment flows;
- weaken install blockers for unverified roots.

## Implementation Slices

### Slice 0: Explicit Root Evidence Audit

Confirm that `DeploymentCheckV1` carries explicit observed root-principal
evidence sufficient for root verification. If the existing
`observed_identity.root_principal` and root `observed_canisters` row are not
specific enough, add a narrow explicit root-evidence field before implementing
verification reports.

Do not infer root evidence from:

- CLI arguments;
- file names;
- local state alone;
- `expected_root_principal`;
- fleet-template name;
- deployment name.

### Slice 1: Root Verification Model

Add or harden:

- `DeploymentRootVerificationReportV1`;
- `DeploymentRootVerificationReceiptV1`;
- `DeploymentRootVerificationSourceV1`;
- `DeploymentRootVerificationEvidenceStatusV1`;
- `DeploymentRootVerificationStateTransitionV1`.

Reuse the existing deployment-target local state shape when possible.

### Slice 2: Artifact-Backed Verification Builder

Implement:

- `deployment_root_verification_report_from_check(...)`;
- `validate_deployment_root_verification_report(...)`.

The builder consumes local deployment state plus an existing
`DeploymentCheckV1`.

The validator rejects digest drift, identity mismatch, missing source facts,
and legacy fleet-state evidence.

### Slice 3: Explicit Mutating Command

Add:

```bash
canic deploy root verify <deployment> --from-check <deployment-check.json>
```

The command writes verified state only after report validation succeeds.

It emits a receipt for successful mutation.

### Slice 4: Read-Only Source Guards

Add source guards proving:

- `deploy check` cannot update `root_verification`;
- `deploy compare` cannot update `root_verification`;
- `deploy register` cannot write `verified`;
- successful install may write `verified` only for a root created or proven by
  that install operation's own guarded receipt path;
- only the explicit root-verification command can promote a registered
  `not_verified` root to `verified`.

### Slice 5: Install Integration

Update install/preflight behavior so:

- `not_verified` roots still block mutation;
- `verified` roots pass the specific unverified-root blocker;
- all ordinary safety checks still run.

### Slice 6: CLI And Documentation Closeout

Update help, status, changelog, and recovery guidance.

Every recovery hint should preserve this distinction:

```text
register records an unverified root;
root verify records verified root state from deployment-truth evidence.
```

## Required Negative Tests

Minimum test coverage:

- `deploy register` writes `not_verified`;
- `deploy register` cannot write `verified`;
- unverified registered root blocks install mutation;
- root verification succeeds from exact matching `DeploymentCheckV1`;
- root verification fails on deployment-name mismatch;
- root verification fails on network mismatch;
- root verification fails on fleet-template mismatch;
- root verification fails on root-principal mismatch;
- root verification fails on check digest drift;
- root verification fails on deployment-plan digest drift;
- root verification fails on inventory ID/digest drift;
- root verification fails when source check has unrelated blocked status;
- root verification may resolve a source check blocked only by the exact
  matching `unverified_deployment_root` finding;
- the exact `unverified_deployment_root` exception fails if any unrelated
  blocker exists;
- the exact `unverified_deployment_root` exception fails if the blocker names
  a different root, deployment, network, or fleet template;
- root verification fails when source check lacks deployment-target identity;
- root verification fails if the observed root is inferred only from request,
  file name, local state, or operator-supplied expected root;
- root verification cannot produce `EvidenceSatisfied` until the source check
  carries explicit observed root-principal evidence;
- root verification fails when legacy fleet-named state is present;
- inspection report with `EvidenceSatisfied` does not update local state;
- passive report cannot be used as a verification receipt;
- root verification fails if local state digest changes before write;
- failed verification does not update local state;
- successful verification emits a receipt;
- receipt digest is deterministic;
- successful install cannot promote a previously registered `not_verified`
  root;
- successful install may write `verified` only for an install-created/proven
  root;
- already verified same-root reverify is a local-state no-op and emits an

## Suggested Validation Commands
