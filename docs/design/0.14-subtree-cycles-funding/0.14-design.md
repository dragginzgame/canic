# Canic Design Proposal: Subtree Cycles Funding

- Version: `0.14`
- Status: `Draft`
- Author: `Canic Core`
- Scope: cycles top-up policy for canister trees

## 1. Motivation

Today, automatic cycles top-up always sources cycles from the root canister.

```text
child canister
    -> auto top-up request
        -> root mints cycles
            -> deposit_cycles(child)
```

This works, but has clear drawbacks:
- centralized spending: root becomes the funding bottleneck
- no subtree budgeting: users cannot control spending for their own trees
- weak isolation: a misbehaving child can indirectly drain root cycles
- scaling limits: all funding decisions pass through root

A more natural CANIC model is hierarchical funding where cycles flow along the ownership tree.

## 2. Design Goals

`0.14` should provide:
1. subtree ownership funding: children request cycles from their parent
2. budget isolation: each subtree owner controls spending for its children
3. global parent-only rule: funding source is always the direct parent
4. safe retries: funding requests are idempotent and replay-safe
5. abuse resistance: children cannot freely drain parent cycles

## 3. Non-Goals

`0.14` does not include:
- cross-subnet cycle markets
- automatic cycle purchasing
- dynamic subtree budgeting algorithms

## 4. Funding Model

Cycles funding becomes hierarchical.

```text
User subtree owner
    |- canister A
    |   |- A1
    |   `- A2
    `- canister B
        `- B1
```

Funding path:

```text
child -> parent -> deposit_cycles(child)
```

Parents participate in the same funding model. When a parent's balance falls
below its configured threshold, it requests cycles from its own parent using
the same protocol. This propagates funding requests upward through the
ownership tree until reaching a root-funded canister.

Hard invariant:

```text
all child funding comes from parent, with no root fallback path
```

## 5. Top-Up Policy

Each canister role/type defines amount/threshold limits, but not funding source.

```rust
pub struct CanisterTopup {
    pub threshold: u128,
    pub amount: u128,
}
```

Funding source is fixed by design: `Parent`.

Config invariant:
- `topup.amount` must be strictly less than half of `topup.threshold`
- equivalent rule: `topup.amount * 2 < topup.threshold`

## 6. Parent Funding Endpoint

Subtree owners expose a funding endpoint, for example `canic_request_cycles_v1`.

Request fields:
- `request_id`
- `issued_at`
- `ttl`
- `requested_amount`

Caller: child canister.

Parent verifies:
- caller is a direct child
- request is not replayed
- global funding kill switch is enabled
- requested amount respects policy limits

## 7. Parent Authorization

Required invariant:

```text
caller in direct_children(parent)
```

Example checks:

```text
caller::is_child(caller)
registry::parent_of(caller) == self
```

This blocks arbitrary canisters from requesting cycles.

## 8. Funding Limits

Parent enforces limits regardless of requested amount.

Example policy controls:
- `max_per_request`
- `max_per_child`
- `cooldown_secs`

Example grant logic:

```text
allowed = min(requested_amount, policy.max_per_request, remaining_budget)
```

## 9. Replay Protection

Funding requests reuse the CANIC RPC replay protection system.

Requests include `request_id`, `issued_at`, and `ttl` fields and are processed
through the standard CANIC capability envelope. Replay protection,
payload-hash binding, and TTL enforcement are handled by the RPC replay guard.

Parent funding logic does not maintain a separate replay table.

Replay protection is scoped by the capability payload, which includes caller
identity and request parameters. Replay detection is therefore effectively
scoped per child request.

## 10. Runtime Funding Controls (APP_STATE)

Global funding kill switch is maintained in app runtime state.

```rust
pub struct AppRuntimeState {
    pub cycles_funding_enabled: bool,
}
```

Default:
- `cycles_funding_enabled = true`

When disabled:
- all `canic_request_cycles_v1` funding requests are rejected

Source of truth and propagation:
- root updates `APP_STATE.cycles_funding_enabled` through the existing `canic_app` endpoint
- updated app state is cascaded to all canisters via existing app-state sync/cascade flow
- each canister enforces its local cascaded value when serving funding requests

`canic_app` remains a single control endpoint and uses a variant payload.
Example shape:

```rust
pub enum CanicAppUpdate {
    SetStatus(AppStatus),                 // Active | Readonly | Stopped
    SetCyclesFundingEnabled(bool),        // global funding kill switch
}
```

Execution order is:

```text
RPC replay guard -> parent authorization -> global funding check -> policy limits -> funding execution
```

The kill switch check must run before policy evaluation, to fail fast.
Replay validation still runs when funding is disabled.

Failure semantics when disabled:
- error code: `Unavailable`
- message: `cycles funding disabled`

Operational control endpoint:
- use the existing root `canic_app` endpoint with variant payload updates
- restricted to root/governance callers (same `canic_app` control boundary)

## 11. Execution

If validation succeeds:

```text
MgmtOps::deposit_cycles(child, allowed)
```

Parent returns `cycles_granted`.

Execution invariant:
- deposit target must always equal the caller principal

## 12. Failure Handling

| Condition | Result |
| --- | --- |
| not a child | unauthorized |
| replay request | duplicate |
| expired request | reject |
| global funding disabled | unavailable |
| policy exceeded | deny |
| insufficient cycles | deny |

Replay entries must not be committed for policy denials.
If a parent denies funding due to insufficient cycles, the child must retry
later. No alternate funding path exists.

## 13. Configuration Example

```yaml
canister_roles:
  user_shard:
    topup:
      threshold: 10T cycles
      amount: 4T cycles

  audit_worker:
    topup:
      threshold: 1T cycles
      amount: 400B cycles
```

## 14. Observability

Required metrics:
- `cycles_requested_total`
- `cycles_granted_total`
- `cycles_denied_total`
- `cycles_requested_by_child`
- `cycles_granted_to_child`
- `cycles_denied_to_child`
- `cycles_denied_global_kill_switch`

Child-scoped metrics are emitted by the parent and must include:
- `child_principal`
- `reason` (for denied outcomes)

Example:

```text
cycles_granted_to_child{child_principal=A1}
```

Implementation note:
- Metrics must be added for funding outcomes.
- Metrics may be emitted through existing CANIC metrics infrastructure or a new
  dedicated metrics store.
- This is observability state only and must not be used as replay/idempotency state.

## 15. Security Considerations

- child drain attack mitigation:
  - per-request caps
  - cooldowns
  - per-child budgets
- replay double-spend mitigation:
  - `request_id` replay guard
  - TTL checks
- unauthorized request mitigation:
  - direct-child-only authorization
- incident containment:
  - global kill switch for immediate funding stop

## 16. Migration Strategy

`0.14` is a global breaking behavior change:
- remove root-source funding paths for child top-ups
- route all child top-up requests to direct parent endpoints
- enforce topology invariant: every non-root canister has a parent
- root is the only canister without a parent and does not use child top-up flow
- remove funding-source toggles from app config
- no backward-compatibility mode is provided
- no compatibility shims are provided

## 17. Future Extensions

Potential post-`0.14` improvements:
- budget buckets: `subtree_budget`, `child_budget`
- autonomous top-up when child balances drop below thresholds

## 18. Summary

`0.14` introduces hierarchical cycles funding:
- children request cycles from their parent
- parents enforce budgets and limits
- no root fallback for child funding paths
- replay protection prevents double spending

This enables safer and more scalable funding for canister trees.
