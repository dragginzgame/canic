## Decision

0.98 removes three evidence-proven pieces of architectural sediment:

1. the accepted but unexecuted per-role randomness configuration and its
   unreachable raw_rand runtime adapter;
2. the dead project-protocol-stub package left behind by the protected
   descriptor/client hard cut; and
3. the obsolete LocalIntent external-call race fixture, including
   intent_client, intent_external, and the buy/call_buy/perform path.

The surviving receipt-backed intent conformance remains in intent_authority,
but the PocketIC test is renamed and narrowed so it describes only the current
receipt-backed contract.

This is a hard-cut line, not a general cleanup line. The original scope's only
non-deletion representation requirement is structured unknown-field evidence
inside the existing ConfigError::CannotParseToml category so the config cut is
proved semantically. The accepted amendment adds only the bounded truth and
authority corrections listed below. It introduces no new runtime abstraction,
entropy service, shared protocol crate, transaction framework, recovery
engine, compatibility parser, or compatibility layer.

## Maintainer-Approved 0.98.2 Amendment

Before 0.98.2 was published, a repository-wide consolidation audit found one
P1, 11 P2 findings, and 30 non-finding candidates. The maintainer explicitly
approved folding that bounded work into the existing patch instead of opening
another release line. The immutable comparison baseline for the complete patch
is therefore `v0.98.1`; no intermediate dirty-worktree fingerprint is a release
authority.

The amendment is limited to:

- syntax-aware role build-macro validation and single-parse config
  projection;
- correction and executable validation of current configuration guidance;
- truthful CycleTracker state-contract metadata;
- removal of dead host duration, runtime snapshot, capability-verifier,
  singleton auth-kind, and orphan ICP helper surfaces;
- narrowing public visibility to proved consumers; and
- exact audit/dependency evidence for the surviving authorities.

It adds no new product feature or replacement abstraction. The 42-candidate
ledger is supporting evidence for this amendment; it does not re-count the
three original 0.98 findings. The canonical closeout audit combines both
scopes and records their contract impact.

## Why This Is A Separate Release Line

0.96 changes durable receipt retention and reclamation. 0.97 changes Cargo
dependency ownership, macro plumbing, and a public Rust facade. Mixing the
sediment deletions into either line would obscure their separate contract and
validation boundaries:

| Line | Sole concern |
| --- | --- |
| 0.96 | Receipt replay horizon, eligibility, bounded terminal reclamation, and downstream conformance |
| 0.97 | Role-owned Canic dependency graph, exact features, and canic::cdk removal |
| 0.98 | Delete proven architectural sediment and consolidate the surviving authorities |

0.98 is technically independent of most 0.97 internals, but it follows 0.97
in release order. Its implementation must use the post-0.97 module/import
layout and must not restore canic::cdk or a forbidden Canic dependency merely
to preserve an old fixture.

## Implementation Baseline And Closeout Dependency

The final 0.96 receipt/reclamation contract is immutable at `v0.96.8`, and the
released 0.97 dependency/CDK baseline is immutable at `v0.97.3`. The complete
deletion, reachability, Cargo-package, stable-allocation, fixture, example, and

The rigorous 0.97 closeout audit subsequently found five P1 defects. Those

Any later evidence that changes reachability, storage, package layout, or the
deletion set stops the affected slice until this design is amended.

## Central Invariants

After 0.98:

1. Every accepted Canic configuration field has one current build, host,
   validation, or runtime consumer.
2. Every workspace package has a current consumer or a necessary build,
   runtime, host, procedural-macro, external-contract, or test boundary.
3. Canic's maintained code, fixtures, examples, and documentation use
   LocalIntent only for locally decidable work. Its public contract explicitly
   excludes rollback based solely on an uncertain external-call result: a
   transport error, callback error, decode error, or missing response never
   proves that an external effect did not occur.
4. Tests demonstrate the supported architecture. A passing test cannot retain
   a call shape that the maintained operational contract rejects.
5. Role constants and fixture-only protocol values live with their current
   owner. A shared package is not created for two constants.
6. Removed pre-1.0 surfaces fail directly. They do not parse and ignore,
   forward, warn, or continue under a legacy mode.

## Current Implementation Reality

### Randomness is a complete configuration story without execution

The schema currently defaults every role to:

~~~text
enabled = true
reseed_interval_secs = 3600
source = "ic"
~~~

The active path is:

~~~text
canic.toml
  -> RandomnessConfig / RandomnessSource
  -> validation
  -> generated compiled CanisterConfig.randomness
  -> host role-detail projection
  -> no runtime consumer
~~~

Exact current owners are:

| Surface | Current location | Current effect |
| --- | --- | --- |
| User documentation | CONFIG.md | Promises seeding and periodic reseeding |
| Parsed schema/default | canic-core config/schema/subnet | Accepts and defaults enabled/ic/3600 |
| Validation | canic-core config/validation/subnet | Rejects an enabled zero interval |
| Generated model | canic-core bootstrap/render | Emits the complete subtree into compiled config |
| Host projection | canic-host release_set/config/projection | Emits randomness=off or source/reseed detail |
| Raw IC effect | canic-core infra/ic/mgmt/randomness | Can call management raw_rand |
| Ops/metrics | canic-core ops/ic/mgmt and metric enums | Wraps and labels raw_rand |
| Runtime workflow | none | Never reads the configuration or calls the adapter |

Repository-wide reachability finds no current call to the ops raw_rand
function and no runtime read of CanisterConfig.randomness. No tracked active
canic.toml sets the randomness table. Tests cover defaults, parsing,
validation, rendering, and fixture construction, but no seed, PRNG, timer,
reseed, or randomness-dependent behaviour.

The original implementation was introduced in the 0.6 era with an actual PRNG
and reseed story. The consuming runtime disappeared while its schema and
adapter survived. Historical origin does not justify retaining a false current
contract.

### project-protocol-stub has no consumer

The package contains only:

~~~text
PROJECT_HUB: CanisterRole
PROJECT_INSTANCE: CanisterRole
~~~

It was created for the 0.40 shared protected-endpoint descriptor and generated
client architecture. 0.65.17 hard-cut that architecture:

- canic_protected_endpoint was removed;
- ProtectedInternalEndpoint and internal-call DTOs were removed;
- descriptor-based generated clients were removed; and
- project fixtures moved to delegated-token endpoints.

The package was emptied to two constants but remained in the workspace.
Current source, manifests, tests, build scripts, and active documentation have
no dependency on it. The current delegation root already owns the two role
constants it uses.

### The intent test contains two incompatible architectures

The current pic_intent_race test first runs this old path:

~~~text
two intent_client canisters
  -> intent_authority.buy
  -> LocalIntent reservation
  -> await intent_external.perform
  -> commit on Ok
  -> rollback on any call error
~~~

It then runs the current receipt-backed conformance in the same
intent_authority canister:

~~~text
begin_or_load deterministic operation
  -> Created / retained / conflict / capacity decision
  -> domain-supplied TerminalEvidence
  -> compare-and-set settlement
  -> exact replay and upgrade persistence
~~~

The first path contradicts the maintained adapter contract. LocalIntent is
locally decidable; an IC transport/callback/decode failure after an await
cannot prove that an external effect did not happen. The second path is the
surviving contract and is also the only current in-repository public
application receipt consumer established by the active 0.96 line.

The correct hard cut removes only the first path. LocalIntent remains supported
for runtime_probe and other purely local reservations.

## Supported Architecture After 0.98

### Configuration

CanisterConfig contains only current runtime inputs:

~~~text
role kind
initial cycles
top-up and cycles-funding policy
scaling / sharding / directory
authentication
standards
diagnostics
metrics
~~~

There is no Canic randomness key and no implied default entropy service.

Applications that need randomness make an explicit role-owned decision:

- depend directly on the relevant upstream IC API after 0.97;
- define the security and replay requirements in the application/domain
  boundary;
- own any seed expansion, persistence, scheduling, and reseeding; and
- test the behaviour that consumes the random value.

0.98 does not recommend time as entropy and does not replace the removed
Canic contract with a convenience wrapper.

### Project fixture protocol

Project role values remain next to the fixture that consumes them:

~~~text
delegation_root_stub
  -> PROJECT_HUB_ROLE
  -> PROJECT_INSTANCE_ROLE
  -> embedded test Wasm release set
~~~

No shared package exists without a shared current protocol.

### Receipt-backed intent conformance

The maintained test shape becomes:

~~~text
pic_receipt_backed_intent
  -> install intent_authority with unit init
  -> begin/load receipt-backed operation
  -> assert exact pending and conflict decisions
  -> settle from deterministic test-supplied opaque TerminalEvidence
  -> assert committed and rolled-back replay
  -> upgrade the same canister
  -> assert durable state and deadline behaviour
  -> exercise the frozen final-0.96 receipt assertion manifest
~~~

The deterministic evidence source may be the installed authority principal or
another explicitly constructed principal. The fixture does not pretend that
Canic validates a domain receipt or that an external effect occurred. Canic
tests binding, state, revision, capacity, persistence, replay, and reclamation;
the downstream domain adapter owns actual external receipt validation.

The final `v0.96.8` assertion manifest is frozen in the
[immutable baseline and inventory](../../audits/reports/2026-07/2026-07-22/0.98-immutable-baseline-and-inventory.md).
The surviving real-Wasm test and focused core tests must preserve these exact
areas and decisions:

| Receipt area | Frozen final-0.96 assertion |
| --- | --- |
| Admission | New valid work is `Created`; exact retained work loads before temporal or capacity admission; payload, quantity, and replay-deadline disagreement is `BindingConflict`. |
| Replay window | Equality is closed; a remaining window above 3,600,000,000,000 ns reports `ReplayWindowTooLong` with that exact maximum. |
| Capacity | The installed fixture reports exact current/requested/limit quantity values; focused core proof retains the exact 1,000-row record boundary. |
| Settlement | Missing identity, expected revision, payload binding, decision, source canister, and evidence fingerprint are checked before transition. |
| Terminal replay | Committed and rolled-back states replay exactly; repeated settlement is `AlreadySettled`; contradictory settlement does not mutate state. |
| Upgrade | Pending state, bindings, revision, replay metadata, and capacity accounting survive same-Wasm upgrade. |
| Reclamation | Exact one-hour replay and 15-minute observation boundaries, ordered bounded removal, preserved totals, replay after removal, and fail-closed corruption remain proved. |

`TerminalEvidence` in this fixture remains opaque domain evidence supplied by
the test. Canic proves only evidence binding and receipt state-machine
behaviour; it does not prove that the represented external effect is true.

## Hard-Cut Contract

### Randomness configuration

Before 0.98, this parses:

~~~toml
[subnets.prime.canisters.app.randomness]
enabled = true
reseed_interval_secs = 3600
source = "ic"
~~~

After 0.98, the entire table is an unknown field under the existing
deny_unknown_fields schema. Configuration parsing fails through this existing
typed category chain:

~~~text
FleetConfigError::ConfigInvalid { path: <canic.toml>, ... }
  -> FleetConfigError::CoreConfig {
       operation: FleetConfigOperation::Project,
       ...
     }
  -> ConfigError::CannotParseToml { ... }
~~~

`ConfigError::CannotParseToml` is the existing core category; Canic currently
defines no separate stable error code for that category. 0.98 does not invent
a repository-wide error-code system. It does require the category's typed
cause to expose an unknown-field issue with these semantic evidence fields:

~~~text
logical_path = subnets.<name>.canisters.<role>.randomness
unknown_field = randomness
~~~

The focused test matches the typed category and those fields. It does not
match, parse, or freeze raw Serde/TOML wording. If the final parser cannot
retain this evidence without a randomness-specific compatibility scanner or a
broad parsing framework, Slice C stops and this design is amended. The parser
must also prove that the field is neither ignored nor downgraded to a warning.

The only migration is:

~~~text
Delete the complete randomness table.
~~~

There is no accepted-but-ignored key, serde alias, legacy parser, warning-only
mode, or automatic transfer to an application feature.

Omitted randomness configuration requires no source edit because the old field
was defaulted. Rebuilding regenerates compiled configuration without the
removed field.

### Rust and generated model

The following Rust surfaces are removed:

- RandomnessConfig;
- RandomnessSource;
- CanisterConfig.randomness;
- bootstrap re-exports and generated render functions for those types;
- MgmtInfra::raw_rand and the ops raw_rand wrapper;
- RawRandInvalidLength;
- ManagementCallMetricOperation::RawRand; and
- SystemMetricKind::RawRand.

No replacement Rust type or function is added. Downstream source matching
SystemMetricKind::RawRand or constructing CanisterConfig/RandomnessConfig must
remove that branch/field. Generated role config is rebuilt rather than read
through a compatibility type.

### Host and metrics output

Configured-role details no longer emit:

- randomness=off; or
- randomness=<source> reseed=<seconds>s.

Any structured CLI output that carries the role details list therefore loses
only these impossible-behaviour entries. The rest of the list and its schema
remain unchanged.

The raw_rand system/management metric dimensions are deleted. Because no
supported runtime reaches the adapter, there is no valid current emitted
series to migrate. Dashboards or source code that statically enumerate the
label must remove it.

Accurate negative security documentation stating that authentication does not
use raw_rand remains. Historical changelogs/designs also remain immutable.

### Cargo packages and test artifacts

The following package identities disappear:

- project-protocol-stub;
- intent_client; and
- intent_external.

They are unpublished test packages. Their Cargo.lock rows, workspace member
entries, Wasm build targets, and current test-runner references are deleted.
No renamed or empty placeholder package remains.

### Test Candid

These test-only endpoints disappear:

- intent_authority.buy;
- intent_client.call_buy; and
- intent_external.perform.

intent_authority no longer accepts an external principal as its init argument.
Its maintained unit init and receipt endpoints remain test-only. Product
Candid, the checked-in Wasm-store DID, and generated root Candid do not change.

### Stable state boundary

0.98 distinguishes product and generic stable contracts from state created
only by the obsolete test flow:

- product stable-memory IDs and record schemas do not change;
- generic LocalIntent and ReceiptBackedIntent IDs, records, encodings, and
  upgrade formats do not change;
- obsolete intent_authority test-only LocalIntent state may disappear with the
  removed buy path; and
- the surviving receipt-backed records must remain compatible across the
  PocketIC test's upgrade boundary.

At the provisional anchor, the obsolete flow writes terminal LocalIntent
records and resource totals in Canic's generic LocalIntent format. The
intent_authority fixture declares no fixture-owned MemoryId or persisted
record type; it bootstraps Canic's generic memory registry. The post-0.97
inventory must prove both facts again. If it discovers a fixture-owned ID,
record, or upgrade input, that test-only contract is recorded explicitly
before deletion rather than being folded into a claim that all stable state is
unchanged.

The randomness cut improves security truthfulness: Canic stops claiming that a
role receives IC or time-derived entropy when no such runtime behaviour exists.
It does not assert that direct application use of raw_rand is automatically
secure. That decision remains application-owned and must account for replay,
authorization, persistence, and the value's actual security purpose.

## Failure Handling

### Config removal exposes a hidden consumer

If implementation finds a current runtime read that the audit missed:

1. stop the randomness slice;
2. trace the consumer to a production entry point;
3. determine whether it provides real behaviour and tests;
4. amend the design before choosing retention or deletion.

Do not keep half the schema or add an ignored compatibility field.

### Deleted package has a real consumer

If post-0.97 metadata finds a current consumer of project-protocol-stub,
intent_client, or intent_external:

1. classify the exact consumer;
2. distinguish current product/test need from historical build residue; and
3. amend the affected slice.

Do not replace the deleted package with an alias or copy its obsolete
architecture into a new common crate.

### Receipt conformance depends on the old external fixture

If a maintained 0.96 assertion currently obtains only through
intent_external/client:

1. express the assertion directly through the receipt-backed authority;
2. use deterministic typed evidence without claiming an external effect; and
3. retain the current receipt invariant.

If that cannot be done without changing receipt semantics, stop and return the
issue to the 0.96 owner. Do not preserve unsafe LocalIntent behaviour.

## Alternatives Rejected

### Implement the promised randomness system

Rejected. No current consumer justifies a PRNG/reseed/timer framework. Building
one solely to validate old configuration would turn sediment into new
architecture.

### Parse randomness and ignore it with a warning

Rejected. This preserves a false contract and makes configuration validation
less truthful.

### Deprecate RandomnessConfig and raw_rand

Rejected. Pre-1.0 has no proven compatibility obligation, and deprecation
retains both the public promise and maintenance cost.

### Keep an empty project-protocol-stub package

Rejected. Package identity is not a compatibility contract here. An empty
placeholder would encode the deleted descriptor architecture into the current
workspace.

### Move the two role constants into a new shared crate

Rejected. The values have one current fixture owner and do not justify a
boundary.

### Keep intent_client/external as generic inter-canister test utilities

Rejected. They have one obsolete consumer and encode the wrong rollback
lesson. Current inter-canister infrastructure is tested elsewhere.

### Delete LocalIntent

Rejected. LocalIntent has current legitimate consumers and owns a distinct
locally decidable reservation boundary.

### Add a generic receipt/recovery adapter framework

Rejected. Canic owns generic reservation state; downstream domains own actual
external receipt validation and recovery. 0.98 only removes a misleading
fixture.

### Fold replay operator recovery into 0.98

Rejected. RecoveryRequired observation/reconciliation is safety-sensitive,
potentially additive Candid/CLI work with external-evidence rules. It needs its
own design line.

### Delete icp-refill because no workspace role enables it

Rejected. The audit found an explicit current downstream capability contract.
Lack of an in-workspace role is a missing proof, not deletion evidence.

## Migration Guide

### Canic configuration users

# Delete this entire table.

reseed_interval_secs = 3600

If the application genuinely uses randomness, add and own an explicit
application-level implementation. Do not translate source = "time" into a
security-sensitive entropy source.

- Remove the randomness field from any direct CanisterConfig construction.
