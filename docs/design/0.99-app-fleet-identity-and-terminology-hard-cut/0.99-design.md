# Canic 0.99 Design: App/Fleet Identity and Terminology Hard Cut

Date: 2026-07-23

## Status

- Status: implementation approved. The maintainer selected a reinstall-only
  release boundary; no 0.98-to-0.99 upgrade or host-state migration is
  supported.
- The semantic inventory is complete for terminology, configuration, Candid,
  CLI, JSON, paths, stable state and raw endpoint classification. Its
  structural-upgrade evidence gap is resolved by the reinstall-only decision.
- The first mutating batch is **Implementation Slice 1 — Minimum Config Hard
  Cut**. The design-only inventory is preparation, not an implementation
  slice.
- Immutable implementation baseline: `v0.98.24` at
  `9bc0f016dadf99fb88c3f2fa28855eb666f5660c`.
- Release scope: repository-wide terminology and identity hard cut spanning
  configuration, CLI, Candid, paths, JSON, runtime topology names and stable
  metadata, plus only the minimum configuration shape change needed to express
  the new terms truthfully.
- Runtime scope: preserve the current one-subnet behavior. Multi-subnet Fleet
  installation, membership, synchronization and placement are deferred.
- Schema scope: the terminology cut includes the minimum additive identity,
  Directory-provenance and authentication state needed to make the new terms
  safe. Those additions are wire- and stable-state breaks, not renames.
- Compatibility posture: pre-1.0 hard cut. Removed names receive no aliases,
  fallback parsers, duplicate endpoints, dual-written fields or legacy paths.
- Entry posture: 0.99 Canisters and host state are created by fresh install or
  explicit reinstall only. Existing 0.98 stable memory, installed-state trees
  and deployment catalogs are not read, migrated or upgraded in place.
- Repository scope: Canic only. Downstream migrations are documented but are
  not implemented outside this repository.

## Decision

Canic 0.99 establishes the terminology for this final hierarchy:

~~~text
App
  Fleet
    Authority Subnet
      Fleet Coordinator
      Fleet-authoritative services
    Member Subnet
      Canister
~~~

For Toko:

~~~text
App: Toko
  Fleet: toko-production
    Authority Subnet
      Fleet Coordinator
      authoritative database and other declared authority services
    Member Subnet 1
    Member Subnet 2
    Member Subnet 3
    ...
    Member Subnet 150
  Fleet: toko-staging
  Fleet: toko-canary
~~~

The terms answer different questions:

- **App** answers “what product or application is this?”
- **Fleet** answers “which deployed realization of that App is this?”
- **Fleet Coordinator** answers “which Canister owns Fleet-wide coordination
  and membership state?”
- **Authority Subnet** answers “where do Fleet-wide coordination and declared
  authoritative services live?”
- **Member Subnet** answers “which local root owns this workload canister?”
- **Canister** answers “which concrete runtime member is this?”

A Fleet is a deployed version of one App over one or more member Subnets. It
has one Fleet Coordinator and its explicitly declared Fleet-authoritative
services on one Authority Subnet. **Authority Subnet** is the canonical
forward name for the distinguished-subnet concept that some old `Prime`
language intended. It is not a mechanical rename of the active `prime`
workload slot or current one-member root. The future Authority Subnet is part
of Fleet topology while remaining outside ordinary member lifecycle and
workload placement.

The hierarchy above is forward architecture, not the runtime state installed
by 0.99. The current implementation remains the valid one-member case. Its
single current root retains the existing local `SubnetRegistry` and temporarily
owns both member-local and Fleet-wide responsibilities. 0.99 persists no
resolved Authority Subnet or Fleet Coordinator principal. 0.100 selects the
Authority Subnet, installs the separate Coordinator, transfers Fleet membership
authority and adds synchronization. 0.101 adopts or provisions explicitly
declared authority services.

0.99 does not introduce `AppRelease` as another topology level. Version,
configuration and artifact identity are Fleet metadata backed by existing
release-set and evidence digests.

## Why This Must Precede Multi-Subnet Work

The current terminology collapses the App and Fleet levels:

- the CLI `fleet` is a source/configuration template;
- `[fleet].name` identifies that template;
- `FleetRoleRef` identifies a declared role, not a deployed canister;
- the host `deployment` record identifies one live root target; and
- runtime `App*` state is actually scoped to one deployed root tree.

Those names are tolerable while one config produces one root tree. They become
incorrect when Toko can run across 150 Subnets, when the same role can have an
instance on every Subnet, and when production, staging and canary Fleets can
exist for the same App.

Adding multi-subnet behavior before fixing the vocabulary would make the
single-root assumptions part of new stable state, CLI output and public APIs.
0.99 fixes the boundary first.

## Canonical Terms

### App

An App is the logical product and source-level configuration authority. Toko is
an App.

An App owns:

- its stable name;
- role declarations and package bindings;
- default topology and placement declarations;
- build inputs and release-set construction; and
- default runtime policy carried into a Fleet.

An App does not own runtime subnet principals, root principals or live
canister membership. One App may have multiple Fleets.

### Fleet

A Fleet is one live deployed realization of an App.

A Fleet owns:

- a stable Fleet identity;
- an immutable operator-facing Fleet name;
- exactly one App identity;
- one canonical IC network identity, with host environments remaining lookup
  context;
- the release-set, configuration and artifact digests being deployed;
- one or more member Subnets;
- all member Canisters recorded by their member Subnet registries; and
- Fleet-wide operational state and provenance.

After the later stages, the Fleet additionally has one resolved Authority
Subnet, one Fleet Coordinator and zero or more explicitly declared
Fleet-authoritative services. Those runtime authorities are not persisted by
0.99.

At steady state, a Fleet represents a specific deployed version of its App.
During a rolling upgrade, the desired Fleet version and observed member
versions may temporarily differ. That rollout state does not create an
`AppRelease` topology object and does not require changing Fleet identity.

The same release-set digest may be deployed to more than one Fleet. The digest
is evidence; the Fleet is the live topology and authority boundary.

### Fleet Coordinator and Authority Subnet

In the architecture established from 0.100 onward, every Fleet has exactly one
Fleet Coordinator. A 0.99 Fleet instead has one temporary coordination
authority in its current root. The later Coordinator owns Fleet-wide
coordination state, including the Fleet Registry. 0.101 gives it a separate
operational ownership registry for Canic-managed authority services on the
Authority Subnet; it does not make the Coordinator a member root or merge its
state with a member `SubnetRegistry`.

`Authority Subnet` is the hard-cut forward term for the distinguished runtime
concept previously intended by `Prime Subnet`; it is not a destination for
every active `Prime` surface. The Coordinator-owned authority scope contains
only the Coordinator and explicitly declared Fleet-authoritative services.
The Authority Subnet:

- is a distinguished part of Fleet topology, not a topology rank;
- is outside the ordinary member `Joining`/`Active`/`Draining`/`Removed`
  lifecycle;
- is excluded from ordinary workload placement;
- gives the Coordinator exclusive registry ownership of its authority scope;
  and
- may physically coincide with a member Subnet only when deployment explicitly
  chooses that co-location, without merging their registry scopes.

If the same physical platform Subnet also hosts a member scope, ordinary
workload Canisters remain owned exclusively by that member's root. Physical
co-location never admits those workloads into the Coordinator-owned authority
registry.

Fleet-wide mutable data may therefore have one canonical writable service on
the Authority Subnet. This is an intentional single logical write authority,
replicated by the IC within that Subnet. Member Subnets may host read replicas
or derived copies discovered through their local Directories. The
application-data replication protocol and any explicit failover remain
separate designs.

“App-wide” data is still scoped to one Fleet: production, staging and canary
must receive distinct authority services. The runtime projection is therefore
`FleetDirectory`, not `AppDirectory`.

In 0.99 the current root is only the temporary co-located Fleet authority.
0.100 installs the separate Fleet Coordinator on the selected Authority Subnet
and implements Fleet membership and Registry synchronization. 0.101 adopts or
provisions declared authority services without moving existing application
data.

### Subnet

A Member Subnet is one runtime member of a Fleet. A platform Subnet may host an
authority scope, a member scope or, when explicitly co-located, both without
merging registry ownership. Each member scope is identified by the platform
Subnet identity and has one Canic root for that Fleet.

Each member Subnet root owns the complete local canister topology for that
member Subnet. The future Fleet Coordinator will own membership of member
Subnets in the Fleet; it will not replace any member root's local Subnet
registry. Authority-service ownership is introduced separately by 0.101.

The current `SubnetRole` is not a runtime Subnet identity. 0.99 hard-cuts that
name to `SubnetSlotId`.

A Subnet Slot is an App-declared logical workload-placement slot. The current
required `prime` slot is hard-cut to `default`; it selects the default workload
placement for the one-member implementation and has no Fleet-wide authority.
Other slots remain logical configuration positions until a later design
defines reusable profiles and multi-subnet placement.

The existing `[subnets.<slot>]` TOML structure remains. Its Rust, DTO, generated
config, environment and diagnostic names must say `SubnetSlot` or
`SubnetSlotId`, never `SubnetRole`. The reserved current slot is
`subnets.default`, never `subnets.prime`.

### Canister

A Canister is one concrete runtime member of exactly one Fleet and is
physically hosted on one platform Subnet at a time. In 0.99 every managed
Canister remains in the existing current root's local `SubnetRegistry`. Later,
a member workload/root Canister belongs to one member Subnet registry, while
the Fleet Coordinator and adopted or provisioned authority services belong to
the Coordinator's operational authority scope. A Canister principal remains
its concrete IC identity.

A role does not identify a concrete Canister. A role such as `vault_db` may be
instantiated on 150 Subnets, and a scaling, sharding or binding policy may
produce several Canisters with the same role on one Subnet.

### Role

A role describes behavior, package identity and placement policy. It belongs
to the App definition.

An App-scoped role reference is sufficient for source, build and configuration
work:

~~~rust
pub struct AppRoleRef {
    pub app: AppId,
    pub role: CanisterRole,
}
~~~

It is not sufficient to address a deployed Canister.

### Deployment

Deployment is an operation, plan, observation and receipt family. It is not a
long-lived topology level between App and Fleet.

Names such as `DeploymentPlan`, `DeploymentReceipt` and `canic deploy` remain
correct when they describe deploying or inspecting a Fleet. A persisted
“deployment target” that identifies a live installed root is a Fleet identity
and must use Fleet terminology.

### Environment

An environment selects host-side ICP connectivity and local state location. It
does not replace App, Fleet or Subnet identity.

### Fleet Identity

Fleet security identity is separate from its operator-facing name:

~~~rust
pub struct AppId(CanonicalName);

pub struct CanonicalNetworkId([u8; 32]);

pub struct FleetId([u8; 32]);

pub struct FleetName(CanonicalName);

pub struct FleetKey {
    pub network: CanonicalNetworkId,
    pub fleet_id: FleetId,
}

pub struct FleetBinding {
    pub fleet: FleetKey,
    pub app: AppId,
}
~~~

`AppId` is the existing canonical App-name identity made explicit. Its value is
exactly the validated canonical value of `[app].name`; it uses the existing
`CanonicalName` byte and text rules unchanged. It need not be globally
collision-resistant because every installed use is bound beneath a generated
`FleetKey`. Once installed in a `FleetBinding`, it cannot be renamed,
reinterpreted or rebound to another App.

`FleetId` is generated from 32 cryptographically random bytes and durably
journalled during initial planning before the first deployment mutation.
Existing deployments receive a generated ID in the same pre-mutation phase of
the 0.99 host migration. Its canonical text form is exactly 64 lowercase
hexadecimal characters. It is never accepted from an operator-supplied name.

`FleetName` is the immutable, validated operator-facing label, such as
`toko-production`. Its existing canonical-name limit and text rules apply. A
friendly catalog maps `(CanonicalNetworkId, FleetName)` to one `FleetId`;
that catalog is lookup convenience and not security identity. Independent
operators may choose the same Fleet name without colliding.

`CanonicalNetworkId` is also a 32-byte value with exactly 64 lowercase
hexadecimal characters as its canonical text form. The host derives it before
planning:

~~~text
SHA-256(
  "canic:canonical-network-id\0"
  || network_trust_identity
)
~~~

`network_trust_identity` is exactly 32 bytes:

- for the public IC, it is the SHA-256 digest of Canic's compiled, pinned
  DER-encoded IC root public key;
- for PocketIC or another ephemeral harness that Canic creates and owns, it is
  `SHA-256("canic:ephemeral-network\0" || instance_seed || root_key_digest)`,
  where `instance_seed` is a 32-byte harness-generated value journalled before
  the instance is created and `root_key_digest` is the SHA-256 digest of the
  harness root key; and
- for a pre-existing local replica or another live network, it is the SHA-256
  digest of the explicitly enrolled DER-encoded root trust anchor.

Enrollment of another live network is one explicit host operation. The
operator confirms the root-key fingerprint, and the host durably writes:

~~~rust
pub struct NetworkEnrollmentRecord {
    pub root_key_digest: [u8; 32],
    pub enrolled_at: Timestamp,
    pub source_profile: CanonicalName,
}
~~~

The maintained enrollment command is:

~~~text
canic network enroll <environment>
  --root-key <der-file>
  --fingerprint <64-lowercase-hex>
~~~

It reads one regular, non-symlink DER file, computes its SHA-256 digest and
requires the supplied fingerprint to match before writing anything. The exact
durable files are:

~~~text
.canic/networks/<canonical-network-id>/trust/root-key.der
.canic/networks/<canonical-network-id>/enrollment.json
.canic/environment-profiles/<environment>/network.json
~~~

The first two files are the authoritative trust record and exact anchor bytes.
The profile file is a non-authoritative lookup containing only the
`CanonicalNetworkId`; every resolution revalidates it against the
authoritative digest. Staged-file replacement, file sync, same-filesystem
rename and parent-directory sync complete before the profile lookup becomes
visible. Re-enrolling an exact existing anchor is idempotent. A different
anchor for an existing profile or network directory is rejected rather than
rotated or rebound.

The enrollment is durable before any Fleet planning. An environment profile
resolves either the compiled public IC identity, a Canic-owned harness
identity or one exact enrollment record.
Different environment profiles may resolve to the same
`CanonicalNetworkId` and therefore the same Fleet namespace. Environment
labels, gateway URLs,
`BuildNetwork::{Local, Ic}` and unverified status responses are never identity
inputs. If the public key, Canic-owned harness seed or explicit enrollment
record cannot be proven, planning fails before mutation.

0.99 implements no trust-anchor rotation or continuity-chain protocol. Any
changed enrolled root anchor is a different `CanonicalNetworkId` and requires a
new explicit enrollment. It never silently rebinds an existing Fleet. The
public IC needs no rotation mechanism here because its
[root public key is fixed at genesis](https://docs.internetcomputer.org/concepts/certified-data/).

The initial deployment plan binds:

- `FleetBinding`;
- `FleetName`;
- current one-member root target and expected principal when known; and
- configuration, release-set and artifact evidence.

Initial installation persists the binding in installed host state, current root
state and protected runtime state on every managed Canister. Existing
Canisters receive the same protected binding through the coordinated 0.99
activation step. It is deployment/install identity, not App configuration and
not ordinary mutable cascade state.

Version evidence may advance through later deployments, but `FleetBinding` is
immutable. A Canister rejects installation or activation if protected state
already contains a different binding. Ordinary cascade freshness therefore
cannot change, remove or reinterpret its Fleet identity.

Every subsequent plan, install, upgrade, observation, receipt and live query
must reject a mismatching App, Fleet ID or network before mutation. A Fleet
cannot be renamed or rebound to another App in 0.99. Fleet rename, transfer and
rebinding are outside this release.

Environment remains host connectivity context and local-state selection. It is
recorded as non-authoritative provenance but is not part of canonical Fleet
identity and does not select the host authority path.

### Forward Authority Terms

0.99 reserves the terms `Authority Subnet` and `Fleet Coordinator` so current
source, DTO and operator vocabulary does not freeze the removed `Prime` term.
It does not define a selector, resolve a Subnet, persist a Coordinator
principal or install a Coordinator binding.

0.100 owns the selector, trusted-profile resolution, creation-time capacity
posture, `FleetCoordinatorBinding`, authority epoch and installation. Existing
0.99 Fleets preserve their sole member's physical Subnet through that
extraction. 0.101 owns authority-service adoption, provisioning and
publication. The 0.99 current root remains the temporary one-member Fleet
authority until the 0.100 handoff commits.

## Identity and Membership

Every concrete Canister must have a stable identity that does not include its
current placement:

~~~rust
pub struct FleetCanisterId {
    pub fleet: FleetKey,
    pub principal: Principal,
}
~~~

Network is part of the value, not ambient interpretation. The same
`(FleetId, principal)` on two networks produces two different
`FleetCanisterId` values. An environment label must never substitute for the
canonical network identity.

0.99 does not introduce a parallel `FleetCanisterRecord`.
`SubnetRegistryEntry` remains the single current local registry row boundary,
and `canic_subnet_registry` retains its exact endpoint name and response shape
through the coordinated upgrade. Portable evidence combines the row with the
trusted `FleetBinding` and owning `SubnetId` at the evidence boundary. A later
wire-shape change must evolve or replace the existing projection rather than
store or expose a second overlapping authoritative row type.

An IC canister may retain its principal while moving between Subnets. Such a
move changes the owning Subnet registry and its observed `SubnetId`; it does
not create a new `FleetCanisterId`. Portable receipts, host work and external
evidence that can cross a Canister or network boundary must bind the stable ID
and treat location as observed topology. Canister-local state already scoped
by the immutable protected `FleetBinding` keeps its current principal keys; it
must not duplicate the Fleet key into every local record.

The 0.99 authority model is deliberately smaller:

1. `FleetBinding` identifies the exact network, generated Fleet ID and App.
2. The existing current root retains the one local `SubnetRegistry`.
3. That root temporarily owns both local topology and the one-member
   Fleet-wide responsibilities.
4. 0.99 does not instantiate a separate Coordinator registry or Authority
   Subnet registry.
5. Ordinary Canisters receive only the Directories and state projections they
   need.

The forward ownership invariant is:

> At steady state, every managed Canister has exactly one committed registry
> owner. During a later ownership movement, source and destination may retain
> authority-epoch-bound staged records such as `MovingOut`, `PreparedIn` and
> `Committed`, but two registries must never be committed owners for the same
> ownership epoch.

0.100 introduces the separate Coordinator and Fleet membership registry.
0.101 introduces the authority-service operational registry and implements
the staged ownership handoff. Neither later stage changes
`FleetCanisterId`.

This gives Canic a concept for every Canister in Toko without centralizing and
cascading the complete topology of 150 Subnets.

## Delegation Audience and Protected Runtime Identity

The current `DelegationAudience::{Canister, CanicSubnet, Project}` alternatives
are not compatible with the new hierarchy. `Canister` and `CanicSubnet` use
ambient principals without the canonical network/Fleet boundary, while the
same App can have production, staging and canary Fleets and therefore makes
`Project` too broad.

0.99 hard-cuts the audience to the canonical Fleet identity:

~~~rust
pub enum DelegationAudience {
    Fleet(FleetKey),
}
~~~

There is no retained `Canister`, `CanicSubnet` or `Project` variant, string
fallback or dual verifier. Canister, role and Subnet restrictions belong in
the signed claims, grants and protected runtime bindings inside one Fleet; they
are not alternative audience authorities.
Audience verification reads the immutable protected `FleetBinding` stored
locally on the verifying Canister. `prepare_delegated_token` may still receive
the requested audience, but preparation and verification must reject any
audience that differs from the local Fleet key.

The protected binding is established before authentication resumes:

1. Fresh installs receive it from deployment/install input.
2. The 0.99 upgrade journal records the expected binding before any Canister
   upgrade.
3. Each upgraded Canister receives the exact `FleetUpgradeArgs` through the
   controller-only management-Canister install argument while the Fleet
   remains quiesced.
4. Activation persists the binding once and rejects overwrite or mismatch.
5. Ordinary topology cascades may carry the binding for consistency checks,
   but they are never its authority and cannot update it.

All persisted authorization material whose meaning included any old
`DelegationAudience` variant is invalid under the cut. The upgrade must
invalidate or replace delegation audience records, issuer policies, renewal
templates, active proofs, retained or prepared tokens, positive verification
caches and related replay state. No old Canister-, CanicSubnet- or
Project-scoped credential is accepted after its Canister runs 0.99. Fresh
Fleet-scoped proofs and certificates must be installed and verified before
application authentication traffic resumes.

## Registry, Directory and Index

0.99 supersedes the 0.27 taxonomy that reserved `directory` for keyed
placement and used `index` for cascaded topology lookups.

The canonical meanings are now:

| Term | Meaning | Write authority |
| --- | --- | --- |
| Registry | Complete authoritative records within a declared scope | Owning root |
| Directory | Derived, read-only, consumer-shaped projection of registry state | None; its source Registry remains authoritative |
| Index | Internal lookup or storage acceleration only | Never a public domain authority |

The runtime topology names are:

| Surface | Completeness and source | Read visibility | Distribution |
| --- | --- | --- | --- |
| `canic_fleet_directory` | Consumer projection derived by the root from current Fleet-scoped sources | Public | Queried and cascaded |
| `canic_subnet_registry` | Complete local Canister membership owned by the Subnet root | Public | Queried; not cascaded as complete records |
| `canic_subnet_directory` | Consumer projection derived by the root from the local Subnet registry/configuration | Public | Queried and cascaded |

A Registry may contain lifecycle state, provenance, tombstones and
administrative metadata that is intentionally absent from its Directory. A
Directory has no mutation authority and is shaped for consumers, but is not
necessarily less visible and need not contain fewer rows in every
representation. Public visibility therefore does not distinguish Registry
from Directory; completeness, canonical writer and derivation do.

0.98.24 removes the write-only `canic_app_registry` placeholder and retires
its stable-memory ID. 0.99 does not rename or replace it. The first real
Fleet Registry, its mirror, synchronization, and recovery authority belong to
0.100. The 0.100 bootstrap seed is derived from protected `FleetBinding`,
trusted observed Subnet identity, and the surviving local Subnet Registry
rather than a retained placeholder registry.

Every Directory response and cascaded Directory input identifies its
derivation source with:

~~~rust
pub struct DirectoryProvenance {
    pub fleet: FleetBinding,
    pub source_root: Principal,
}
~~~

This is the minimum one-subnet provenance contract. It identifies which Fleet
and root produced the projection but does not claim freshness, revision
ordering or multi-subnet synchronization. 0.100 adds the richer
epoch/revision/content evidence required for mirrored Fleet Registry state.

The provenance acceptance rules are exact:

- a cascaded Directory input is accepted only when
  `caller == provenance.source_root`;
- `source_root` must equal the protected current-root authority for the exact
  local `FleetBinding`;
- the transport byte bound and Directory-entry count bound are enforced while
  the record is structurally decoded;
- `provenance.fleet` must equal the immutable protected binding before entries
  are semantically validated or activated;
- Directory input can never create, overwrite or reinterpret protected Fleet
  identity;
- provenance returned by an ordinary public query is descriptive unless the
  response is covered by an explicit certified-response contract; and
- authorization, delegation audience, ownership and mutation decisions never
  rely on an uncertified public Directory response.

The one-subnet cascade is authenticated by caller and protected local state.
The provenance record itself is not a signature, freshness proof or
certificate.

An implementation may use a B-tree index internally. That does not make
`Index` part of configuration, Candid, DTO, state-contract or CLI vocabulary.

### Keyed Placement Terminology

The current keyed placement family also uses `directory`. That conflicts with
the new single meaning and is included in the terminology hard cut.

The keyed placement concept becomes **binding**:

- one logical key claims or binds at most one dedicated instance root;
- `DirectoryRegistry` becomes `PlacementBindingRegistry`;
- directory placement config becomes binding placement config; and
- placement bind, retry and recovery behavior is unchanged.

This is a rename of an existing placement contract, not a redesign of its
state machine.

## Version and Release Evidence

0.99 does not add `AppRelease`.

The version deployed by a Fleet is represented by existing evidence such as:

- Canic version;
- canonical App configuration digest;
- release-set manifest digest;
- role artifact-set digest;
- embedded configuration-set digest; and
- observed module hashes.

0.99 separates pre-build Canister identity from post-build release-set
evidence:

~~~rust
pub struct ReleaseBuildNonce([u8; 32]);
pub struct ReleaseBuildId([u8; 32]);

pub enum ReleaseBuildPlanState {
    Planned,
    Finalized {
        release_set_manifest_digest: [u8; 32],
    },
}

pub struct ReleaseBuildPlanRecord {
    pub nonce: ReleaseBuildNonce,
    pub release_build_id: ReleaseBuildId,
    pub state: ReleaseBuildPlanState,
}
~~~

The host obtains `ReleaseBuildNonce` from its cryptographically secure random
source and creates one `Planned` `ReleaseBuildPlanRecord` before any Fleet
Wasm is built. The record is stored at:

~~~text
.canic/release-builds/<release-build-id>/plan.cbor
~~~

The file is written through staged-file replacement, file sync, same-filesystem
rename and parent-directory sync. Its exact derivation is:

~~~text
ReleaseBuildId =
  SHA-256(
    "canic:release-build-id\0"
    || u64_be(32)
    || ReleaseBuildNonce
  )
~~~

There is no alternate deterministic derivation from source-tree or toolchain
metadata. The nonce and derived ID are immutable inputs to the selected build
plan; retry reuses them, while planning a different build creates a new nonce.
The same exact ID is embedded in every Wasm produced by that release build.
It excludes Wasm bytes, Wasm hashes, release-set manifest bytes and the
release-set manifest digest.

After the final Wasms exist, the host computes the separate
`release_set_manifest_digest` over the canonical release-set manifest
containing their exact hashes, atomically advances the plan from `Planned` to
`Finalized`, and never mutates it again. The finalized record's canonical hash
is `release_build_plan_hash`. A fresh-install recovery authority may be
admitted only from that exact finalized record and stores its hash,
`ReleaseBuildId` and manifest digest. It does not copy the nonce into a second
active authority. The host owns validation of the complete post-build
evidence. A Canister compares only the journalled `ReleaseBuildId` in its
install input with its embedded `ReleaseBuildId`.

The release-set manifest digest is never embedded in a Wasm whose hash
contributes to that manifest. This avoids a self-referential digest while
still binding the host to exact final artifacts and every Canister to the
pre-build identity that produced them.

A crash while the plan is `Planned` resumes or abandons only that build; no
Fleet or Canister mutation has occurred. A `Finalized` record may be reused by
more than one Fleet deployment, but each deployment has its own Fleet-bound
install authority. Missing, non-canonical, hash-mismatching or
partially written build-plan evidence fails before deployment mutation.

The Fleet has desired evidence and observed per-member evidence. A later
rollout design may add typed rollout state, but version remains Fleet metadata
rather than a parent or child topology object.

No manually maintained `version = "..."` field is added to `canic.toml` in
0.99.

## Pure Rename and Hard-Cut Mapping

The implementation must classify by semantic ownership rather than perform a
blind text replacement.

| Current surface | Canonical 0.99 surface | Reason |
| --- | --- | --- |
| Source/config `fleet` | App | It describes roles, packages and build inputs |
| `[fleet].name` | `[app].name` | The checked-in config identifies the App |
| `FleetConfigSnapshot` | `AppConfigSnapshot` | It is a validated source App config |
| `FleetRoleRef` | `AppRoleRef` | It identifies a declared role, not a live member |
| `SubnetRole` | `SubnetSlotId` | It identifies an App-declared logical slot, not a platform Subnet |
| Draft/future runtime `Prime Subnet` selector or authority concept | Removed from 0.99; 0.100 introduces `Authority Subnet` | It is the later resolved scope for the Fleet Coordinator and declared authority services |
| Draft `Fleet Authority Canister` | `Fleet Coordinator` | It coordinates Fleet membership without being the application-data service itself |
| Reserved `subnets.prime` slot | Reserved `subnets.default` slot | It is the current default workload placement, not Fleet authority |
| `fleet_template` | `app` or `app_name` | A live Fleet is created from an App config |
| Live `deployment_name` | `fleet` or `fleet_name` | The persisted target is the Fleet |
| Deployment catalog | Fleet catalog | It lists live installed Fleet identities |
| `AppIndex` | `FleetDirectory` | It is a consumer-shaped Fleet-wide cascaded lookup projection, including declared Fleet-authoritative service instances |
| `SubnetIndex` | `SubnetDirectory` | It is a consumer-shaped local cascaded lookup projection |
| `AppState` family | `FleetState` family | The cascaded operational state belongs to one Fleet |
| Keyed placement directory | Placement binding | `Directory` is reserved for cascaded projections |

The old word `prime` conflated two concepts and therefore has two deliberate
destinations:

- genuinely future distinguished-Subnet semantics become **Authority Subnet**
  only when 0.100 resolves that Subnet; and
- the App-declared workload slot becomes `subnets.default`.

The current sole-member root is not renamed to an Authority-Subnet root. It is
the current Fleet root and temporary Fleet-wide authority until 0.100 installs
the separate Coordinator. The active `Prime` surfaces therefore have these
exact destinations:

| Current active surface | 0.99 destination |
| --- | --- |
| `SubnetRole`, `SubnetRole::PRIME`, `is_prime()` and serialized slot value `prime` | `SubnetSlotId`, `SubnetSlotId::DEFAULT`, `is_default()` and value `default` |
| `subnets.prime` configuration and generated topology paths | `subnets.default` |
| `SubnetIdentity` root-install DTO | `CurrentRootInstallIdentity` |
| `SubnetIdentity::Prime` | `CurrentRootInstallIdentity { expected_module_hash: None }` |
| `SubnetIdentity::PrimeWithModuleHash(bytes)` | `CurrentRootInstallIdentity { expected_module_hash: Some(bytes) }` |
| `SubnetIdentity::Standard` and `SubnetContextParams` | Removed; no production constructor survives and 0.100 owns the future member-root install contract |
| `SubnetIdentity::Manual` | Removed; tests use `CurrentRootInstallIdentity` with no expected module hash |
| `prime_root_pid` in environment state, DTOs and policies | `fleet_root_pid`, meaning the current one-member Fleet root, not a resolved Authority Subnet or Coordinator |
| `is_prime_root` access/runtime predicate | `is_fleet_root` |
| `is_prime_subnet` access/runtime predicate | `is_default_subnet` |
| `PrimeRoot*` errors, subject rejections and diagnostics | Corresponding `FleetRoot*` term |
| Draft or future distinguished-Subnet selectors | Removed from 0.99; introduced only as `AuthoritySubnet*` by 0.100 |

`CurrentRootInstallIdentity` is root-fresh-install input, not the argument to
an upgrade and not a topology level. Fresh non-root installation continues to
use the one canonical `CanisterInitPayload`. The two fresh-install payloads
carry the same Fleet activation fields described below; breaking upgrades use
`FleetUpgradeArgs`. This distinction preserves the information required by
the existing role-specific init paths without retaining `Prime` vocabulary.

This is not contradictory: Authority placement is future live deployment
input, `subnets.default` is App workload-topology declaration, and
`fleet_root_pid` identifies only the temporary current one-member root.

These are semantic source/API renames. A Rust type-name change alone does not
change Candid's structural wire format. Endpoint names, record field labels and
variant labels do.

## Additive and Breaking Schema Work

The following items are required by the terminology contract but are not
renames:

| Schema change | Boundary | Hard-cut consequence |
| --- | --- | --- |
| `CanonicalNetworkId`, generated `FleetId`, `FleetName`, `FleetKey` and `FleetBinding` | Plans, receipts, host state, protected runtime state and portable evidence | Existing deployments receive one generated ID; names remain labels and no ambient-network interpretation remains |
| Protected local Fleet binding | Every managed Canister | Fresh install or coordinated activation must complete before auth resumes |
| `FleetCanisterId.fleet: FleetKey` | Portable host work, receipts and external evidence; not redundant Canister-local keys already protected by `FleetBinding` | Same Fleet/principal on different networks no longer collides |
| Reserved Subnet Slot value `prime` → `default` | Config, generated environment, install/bootstrap DTOs and any persisted slot identity | Existing one-member state migrates once; `prime` is never accepted after activation |
| `DirectoryProvenance` and Directory input/response envelopes | Public Directory queries, install payload and state cascade | Candid and stored Directory projections break atomically |
| `DelegationAudience::Fleet(FleetKey)` | Delegation preparation, persistence and verification | All three old audience variants and their credentials/caches are invalidated |

renamed tuple wrappers:

The existing `PageRequest`, `Error` envelope and public access level are
unchanged. `limit` is clamped to the existing `1_000`-entry maximum. The two
response records above are the complete public Candid page-response contracts:
`entries` is the requested bounded page and `total` is the complete entry count
for the same captured local snapshot.

Snapshot consistency is per query call only. One call captures the current
local Directory once, then derives both its page and `total` from that value.
Two separate offset-page calls may observe different local Directory states;
0.99 supplies no revision, cursor or cross-call snapshot pin. The
`DirectoryProvenance` identifies the Fleet and source root but proves neither
freshness nor revision equality. Revisioned pagination belongs to the 0.100
Registry-mirror protocol.

The cascade payload reuses the existing 16,384-byte update payload limit and
adds the exact 1,000-entry vector bound. The generated update export adapter
must read the scalar System API argument size before copying argument bytes or
calling Candid decoding. It rejects a value above 16,384 bytes immediately,
including for inter-Canister calls. The shared `canister_inspect_message` hook
is updated to use the same scalar size check and remains the earlier ingress
gate, but it is not relied on for root-to-child cascades. Only after the
raw-byte check may either path copy and decode the payload.

`FleetDirectoryInput` and `SubnetDirectoryInput` use a bounded vector decoder
whose sequence visitor rejects a declared length above 1,000 before reserving
the `Vec`, and rejects the 1,001st element when no trustworthy length hint is
present. Candid decoding quotas remain an additional defense, not the primary
bound. Provenance and entry semantics run only after both structural bounds
pass. Public page construction can never return more than 1,000 entries.

The implementation must evolve the existing `AppIndexArgs`,
`SubnetIndexArgs`, `IndexEntryResponse` and paged query projections in place;
it must not introduce parallel provenance-free Directory shapes.

The frozen endpoint hard cut is:

~~~text
canic_app            -> canic_fleet_admin
canic_app_index      -> canic_fleet_directory
canic_subnet_index   -> canic_subnet_directory
canic_app_state      -> canic_fleet_state
~~~

The complete active endpoint and access inventory is frozen in the
[semantic inventory](0.99-semantic-inventory.md). Removed pre-1.0
endpoints receive no forwarding aliases.

`canic_subnet_registry`, `canic_sync_state` and `canic_sync_topology` retain
their endpoint names. In particular, `canic_subnet_registry` must not be
renamed for symmetry: its exact response signature is the host's stable
transition, readiness and recovery read surface during the coordinated
upgrade.

Names such as `ApplicationReceipt` or a database settlement index are not
renamed merely because they contain an English word from this taxonomy. The
hard cut applies only where the term denotes App, Fleet, topology Directory or
topology Registry ownership.

Likewise, repository/ICP “project” terminology and the external immutable
object-storage `ProjectCycles` protocol remain unchanged. They identify a
workspace or an upstream protocol contract, not the removed Canic auth
audience or a live topology level.

## Minimum Configuration Update

`canic.toml` is an App definition. It must identify the App, not a live Fleet.
A Fleet name belongs to deployment input, plans, receipts and installed Fleet
state.

The current shape:

~~~toml
app_index = ["project_hub"]

[fleet]
name = "toko"

[app]
init_mode = "enabled"
~~~

becomes:

~~~toml
[app]
name = "toko"
init_mode = "enabled"

[services.fleet]
roles = ["project_hub"]
~~~

Existing `[app.whitelist]` remains nested under `[app]`:

~~~toml
[app.whitelist]
principals = ["aaaaa-aa"]
~~~

This is the minimum schema change:

1. Add required `name` to the existing `AppConfig`.
2. Remove `FleetConfig` from `ConfigModel`.
3. Replace `app_index` with `[services.fleet].roles`. This is App authoring
   input declaring which roles are intended to become singleton
   Fleet-authoritative services. It is not runtime ownership evidence and
   does not create a Directory. 0.99 preserves the current one-member
   service principals, placement and cardinality. 0.101 compiles, adopts or
   provisions and commits the declaration after 0.100 installs the
   Coordinator.
4. Rename keyed placement `directory` config to `binding` without changing its
   behavior.
5. Rename `SubnetRole` to `SubnetSlotId` while keeping the existing
   `[subnets.<slot>]` TOML shape and hard-cutting the required
   `subnets.prime` slot to `subnets.default`.
6. Keep the existing role declarations and topology template behavior.
7. Add no runtime subnet principal, root principal or complete Fleet membership
   list to `canic.toml`.
8. Add no manual App or Fleet version string.

The App name uses the current Fleet-name validation contract. It remains a
bounded canonical source identity suitable for paths, package metadata and
diagnostics.

### Hard-Cut Parsing Rules

After 0.99:

- `[app].name` is required wherever `[fleet].name` is required today;
- `[fleet]` is an unknown table and fails parsing;
- `app_index` is an unknown field and fails parsing;
- keyed placement `directory` is an unknown field and fails parsing;
- `subnets.prime` is an unknown slot and fails validation;
- `subnets.default` is required for the current one-member topology;
- `[services.fleet].roles` retains the current default-service validation,
  rejects scaling, sharding, keyed binding and repeatable placement for
  selected roles, and creates no runtime authority in 0.99; and
- no parser accepts both old and new names.

Tracked configs, examples, scaffolds, generated configuration, fixtures and
active documentation must move atomically with the schema.

## CLI and Host Boundary

The source/config command family becomes App-scoped:

~~~text
canic fleet create <name>                 -> canic app create <app>
canic fleet list                          -> canic app list
canic fleet config <name>                 -> canic app config <app>
canic fleet role ...                      -> canic app role ...
canic build <fleet> <role>                -> canic build <app> <role>
fleets/<name>/canic.toml                  -> apps/<app>/canic.toml
package.metadata.canic.fleet              -> package.metadata.canic.app
~~~

Live target identity becomes Fleet-scoped:

~~~text
deployment target name                    -> FleetName lookup
.canic/<environment>/deployments/<name>   -> .canic/networks/<canonical-network-id>/fleets/<fleet-id>/
deployment catalog entry                  -> Fleet catalog entry
installed deployment inventory            -> Fleet inventory
~~~

Environment profiles resolve to the canonical network directory. A friendly
catalog may map `FleetName` to `FleetId` within that directory, but installed
state, journals, receipts and authority are keyed only by
`(CanonicalNetworkId, FleetId)`. Two environment aliases for the same network
therefore cannot create independent host authorities.

Deployment remains the verb and evidence workflow:

~~~text
canic install <app> <fleet>
canic deploy plan <fleet> --app <app>
canic deploy install <fleet> --plan <file>
DeploymentPlan
DeploymentReceipt
~~~

Fresh `canic install` always requires both identities; it never infers the App
from the Fleet name or requires them to be equal. `canic deploy install` reads
the App identity from the exact supplied plan and rejects a positional Fleet
that differs from it. Those plans and receipts must carry both App and Fleet
identity. A plan may
continue to describe one current member root in 0.99; it must not claim that
the Fleet already spans several installed Subnets or that the root's Subnet is
intrinsically privileged. 0.99 exposes no Authority Subnet selector or related
CLI option; 0.100 owns that complete contract.

`canic status` stops joining source configuration to installed state by equal
names. Its maintained text output has two independent sections:

~~~text
APPS:   APP | CONFIG | CANISTERS
FLEETS: FLEET | APP | NETWORK | DEPLOYED | ROOT
~~~

The App section is discovered from `apps/<app>/canic.toml`; the Fleet section
is discovered only from the canonical-network Fleet catalogs. An App without
a Fleet and a Fleet whose source App is not present locally remain distinct
rows. JSON-producing status/inspection surfaces use the standard identity
projection below rather than reconstructing either section from display text.

The complete command inventory is frozen in the semantic inventory:
config-template commands move to `canic app`, live-state commands use Fleet
terminology, and deployment-operation commands remain under `canic deploy`.
No command alias preserves the old meaning of `fleet` or `deployment target`.

## Rejected Alternative: In-Place 0.98 Host Migration and Fleet Upgrade

The protocol in this section was evaluated and rejected for 0.99. The
immutable 0.98.24 public diagnostics cannot prove all structural replay,
receipt-backed intent and durable child-retry owners terminal before the first
breaking mutation. The maintainer selected reinstall-only operation, so none
of the migration journal, mixed-version, legacy decoder, quiescence, abort or
legacy-path-retirement machinery below is implementation scope.

This retained rejected-alternative analysis documents why an in-place upgrade
must not be introduced piecemeal. It is not an active contract, validation
requirement or permission to preserve a 0.98 compatibility path.

Host migration and the breaking Canister upgrade are one recovery protocol,
not independently resumable procedures.

The host persists exactly one bounded journal:


The first migration generation is created only after the referenced
`ReleaseBuildPlanRecord` is `Finalized`. Its three release-build fields must
equal that record and its canonical hash. The pre-build record is therefore
the only owner while artifacts are being built; the Fleet migration journal
becomes the sole active deployment-recovery authority only after final
artifact evidence exists. There is no placeholder digest, zero sentinel or
partially initialized migration journal.

`FleetActivationHostRecord` is frozen with the Canister-local activation
types below. `FleetStructureFencing` may populate a candidate Canister vector
from the reconciled root Registry so it can journal the non-root stop intents.
`FleetStructureFenced` freezes that vector as the complete bounded
managed-Canister inventory in raw-principal order after a second Registry
capture agrees exactly; later generations may update entry progress but may
not add, remove or reorder principals. Principal uniqueness is validated
before any stop and again before `FleetStructureFenced`. `old_module_hash` is
the observed 0.98 module;
`expected_module_hash` is the exact selected 0.99 artifact;
`expected_controllers_hash` commits the canonical preflight controller set;
and `expected_legacy_protected_state_hash` commits the unchanged recovery
diagnostic needed by a pre-quiesced abort.
The vector uses `MAX_FLEET_ACTIVATION_CANISTERS`, and a canonical generation
over `MAX_FLEET_MIGRATION_JOURNAL_BYTES` fails before it can become active.

The legacy protected-state hash uses only evidence obtainable from the
immutable `v0.98.24` baseline before mutation. Its canonical input is the
Canister principal, observed module hash, canonical controller-set hash and
the `LegacyProtectedIdentityProjection` defined below. The host queries and
journals those values during preflight. After a pre-quiesced abort, it queries
`canic_env` again, constructs the same projection and requires the same
canonical hash. Module identity binds the unchanged embedded configuration;
no new endpoint on the old Wasm is assumed.

Every phase transition, operation intent, status observation and per-Canister
progress update creates a new immutable journal generation before the next
external effect. Sequence `0` has `previous_journal_hash = None`; every later
generation increments with checked arithmetic and names the exact preceding
generation hash. Overflow fails closed. A generation is canonical-CBOR
encoded, hashed under `canic:fleet-upgrade:journal-generation\0`, written and
synced under its content-addressed name, and only then may `active.cbor`
advance through a staged-file, file sync, same-filesystem rename and
parent-directory sync. A crash before the pointer rename leaves the preceding
generation authoritative; an unreferenced later generation is inert evidence.
`pending` is set before issuing the named management call and is cleared only
in the same durable update that records its reconciled outcome:

- `Stop` records the newest management-status evidence while the Canister is
  `running`, `stopping` or `stopped`;
- `Install` binds the exact selected Wasm hash and the domain-separated hash of
  the exact `FleetUpgradeArgs`; and
- `Start` binds the exact module expected after starting: the old hash during
  a pre-quiesced abort or the new hash during forward upgrade.

On recovery, a pending `Stop` is reconciled through `canister_status` and may
repeat the idempotent stop request. A pending `Install` accepts only the old
module hash or the exact expected new hash: the former repeats the exact
install, the latter must still be `stopped` and records
`UpgradedStopped`, and any third hash fails closed. A pending `Start` accepts
only the module hash stored in the intent and repeats or reconciles start until
status is `running`; forward upgrade then requires `Prepared` diagnostics that
match the journalled args, while abort requires the protected legacy-state
proof. A timeout is never interpreted as failure or success without this
reconciliation.

The journal has one catalog-independent discovery location:

~~~text
.canic/recovery/fleet-migration-upgrades/
  <canonical-network-id>/<fleet-id>/
    active.cbor
    attempts/<upgrade-id>/
      journals/<20-digit-sequence>-<journal-hash>.cbor
      evidence/<evidence-hash>.cbor
      terminal.cbor
~~~

`active.cbor` is the single recovery authority for one Fleet and contains
`FleetMigrationUpgradeAuthority`. While active, its pointer identifies the
exact `upgrade_id`, sequence and immutable journal-generation hash. It is
installed atomically only when no active authority exists. Host startup and
every deploy/recovery command validate this fixed record before consulting
either the legacy or Fleet catalog. A second active attempt is rejected.

An `Active` pointer is advanced by atomically replacing only `active.cbor`;
the immutable generation it names never changes. No operation requires a
journal file and its pointer to change atomically. Recovery validates the
generation filename, its canonical bytes, sequence, hash, Fleet/upgrade
identity and complete predecessor chain before using it.

The path components are canonical lowercase hexadecimal encodings. Completion
and abort use one terminal protocol:

1. deterministically encode the candidate final journal generation and its
   hash without advancing `active.cbor`;
2. write and sync that immutable candidate plus a domain-separated terminal
   receipt whose canonical bytes include its sequence/hash;
3. perform and prove the terminal side effect, then atomically advance
   `active.cbor` to the exact candidate generation;
4. write and sync immutable `terminal.cbor` containing the exact
   `FleetTerminalReceiptRef`;
5. atomically replace `active.cbor` with the `Terminal` value; and
6. atomically remove `active.cbor` only after startup can validate
   `terminal.cbor` and its receipt.

For completion, the candidate is `LegacyPathRetired`; its receipt is durable
before the obsolete operational path is removed, but neither becomes
authoritative until removal is proven and step 3 advances the pointer. For
abort, the legacy Fleet and pre-fence App status are restored before the
candidate is activated. Prewritten, unreferenced terminal artifacts have no
authority.

An interruption before step 5 resumes the last active generation and
idempotently recreates the same terminal artifacts. An interruption after step
5 performs only terminal cleanup. The terminal receipt, not a hash field
inside a mutable journal, binds the final journal sequence/hash. Completion
and abort therefore share one typed receipt reference; there is no separate
outcome-specific hash field or untracked completion-receipt hash.

The terminal attempt generations and receipt remain immutable evidence under
`attempts/` but are never discovered as resumable authority after the active
record is removed. A missing, malformed or conflicting active or terminal
record fails closed. Catalog mutation can therefore neither hide nor create
recovery authority, and retained terminal journals cannot compete with a
later attempt.

Every hash referenced by the journal resolves to the exact canonical bytes in
the sibling content-addressed `evidence/` directory. An evidence object is
written and synced under its lowercase hash name before a journal update may
reference it. A missing object, filename/hash mismatch, non-canonical bytes or
wrong domain fails closed. This store retains the actual controller sets,
management statuses, source/target manifests, structure-fence evidence,
terminal receipt, activation evidence and credential-manifest generations
rather than asking an operator to reconstruct them from a digest. Operation
intents remain directly encoded in the journal.

`source_manifest_hash` and `target_manifest_hash` are content-manifest hashes,
not hashes of path strings. For each tree, the canonical manifest is a
definite-length array of:

~~~text
[normalized_relative_path_bytes, file_length_u64, sha256(file_bytes)]
~~~

It contains every regular file below the tree except the independent recovery
journal, uses `/` separators, rejects absolute paths, `.`, `..`, symlinks and
special files, and is sorted by raw normalized path bytes. Empty directories
carry no authority and are omitted. The source and target use distinct domain
separators, so identical file sets do not produce interchangeable evidence.

The migration maps:

~~~text
fleet_template   -> AppId
deployment_name  -> FleetName
generated bytes  -> FleetId
environment      -> host provenance plus resolved canonical IC network
~~~

The inventory is frozen by an explicit `v0.98.24` structural fence before the
target tree or either content manifest becomes authoritative. The fence uses
only current surfaces:

1. commit `FleetStructureFencing` and a durable host mutation fence keyed by
   the legacy deployment identity; every host operation that can provision,
   attach, detach, place, recycle, upgrade, register, change topology or
   mutate installed authority for that deployment then rejects except the
   exact fence-resume/abort workflow;
2. call the current root's controller-only
   `canic_app(SetStatus(Readonly))` and require its completed current cascade,
   so ordinary 0.98 update dispatch and application-owned structural
   initiators stop admitting new work while current queries remain available;
3. reconcile every already durable root-provision replay, placement
   allocation/acknowledgement, pool import/reset, bootstrap/publication and
   host provisioning operation through its existing 0.98 recovery owner until
   each is terminal; any operation whose outcome or resulting principal
   cannot be proven fails closed;
4. capture the reconciled candidate `canic_subnet_registry`, populate the
   journal entries, then stop every managed non-root through the exact durable
   `Stop` intents above; the current root stays running only to expose the
   unchanged recovery/Registry query surfaces;
5. prove from the pre-implementation endpoint/timer inventory that the root has no
   self-originated structural producer after the existing bootstrap,
   publication, pool and placement owners are terminal. All registered
   non-root callers are stopped, so the current internal
   `canic_response_capability_v1` path cannot admit a new structural request;
6. recapture the complete `canic_subnet_registry`, host installed inventory,
   controller sets, module hashes and all membership-bearing receipts, require
   exact agreement with the candidate inventory, and commit their hashes plus
   the empty pending-operation set as `structure_fence_evidence_hash`; and
7. commit `FleetStructureFenced`, then build the source/target content
   manifests, target tree and journal Canister vector only from that frozen
   evidence before committing `HostTargetPrepared`.

The 0.98 `Readonly` state is a conservative universal qualification fence; it
may pause application mutations as well as structural mutation. Application
stable bytes are not part of either content manifest or the frozen inventory.
Callbacks already accepted before the completed cascade may finish while the
pending-operation drain runs and may change application data without
invalidating the migration. Non-root application data may continue changing
until its owning Canister is stopped; the Wasm upgrade preserves the resulting
current stable application state. 0.99 performs no application-data
migration.

The authority rules are exact:

- `LegacyActive`: the legacy tree and catalog remain operational authority.
- `FleetStructureFencing`: the durable host mutation fence and propagated 0.98
  `Readonly` state block new ordinary structural work, pending operations are
  reconciled and non-roots are stopped under journalled intents. Partial
  availability is explicit. Recovery may only finish or abort this exact
  fence.
- `FleetStructureFenced`: the final managed-Canister inventory and structural
  evidence are immutable, every non-root is stopped and the current root
  remains `Readonly` for unchanged recovery/Registry queries. The legacy tree
  remains read authority, but no host or Canister path may mutate membership,
  topology or installed authority.
- `HostTargetPrepared`: the complete new Fleet tree, identity and recovery
  journal are durable, but the legacy catalog remains the read authority
  because no Canister identity changed. The unused target may still be
  abandoned only through the exact fence-abort protocol that restores the
  preceding App status and proves the frozen inventory unchanged.
- `FleetQuiescing`: the current-root `Stop` intent has been committed. The new
  journal is the only recovery authority, but the exact abort protocol below
  may still restore the unchanged legacy Fleet.
- `FleetQuiesced` through `CanistersActivated`: the new recovery journal is the
  only recovery authority and recovery is forward-only. The legacy tree
  remains preserved and readable only by this migration; ordinary legacy
  operations are rejected.
- `HostAuthorityCommitted`: the Fleet catalog resolves the exact
  `(CanonicalNetworkId, FleetId)` target and normal readers use only the Fleet
  tree. The candidate final generation and terminal completion receipt become
  durable before legacy-path retirement.
- `LegacyPathRetired`: the obsolete operational tree is removed only after its
  immutable receipts and evidence are preserved byte-for-byte in the Fleet
  evidence archive.

The Fleet catalog switches exactly once, after every managed Canister is
`Active`. It never points to a target whose protected Canister identities are
only partially activated.

The host prepares the target by:

1. validating every legacy record and rejecting App, Fleet, network or path
   collisions;
2. generating and durably journalling the new `FleetId` and `upgrade_id`
   before mutation;
3. committing the host mutation fence, propagating the current 0.98
   `Readonly` state, reconciling all pending structural operations and
   freezing the complete final registry/inventory evidence;
4. constructing and hashing the complete canonical source and target content
   manifests from that frozen state and binding them to the exact
   `(CanonicalNetworkId, FleetId, FleetName, AppId)`;
5. preserving immutable receipts and evidence without rewriting their signed
   or digested bytes;
6. writing the complete target as a sibling staged directory, syncing its
   files and directories, renaming it on the same filesystem and syncing the
   parent directory; and
7. committing `HostTargetPrepared` before current-root stopping or any
   breaking install begins.

Normal host commands encountering `FleetStructureFencing`,
`FleetStructureFenced`, `HostTargetPrepared`, `FleetQuiescing`,
`FleetQuiesced`, `CanistersPrepared` or `CanistersActivated` do not mutate
through either catalog. They report the journal and require its exact resume
or permitted abort operation.

Once `FleetQuiesced` commits, recovery is forward-only to the journalled
`FleetBinding`, `upgrade_id`, `ReleaseBuildId` and release-set manifest
digest. No recovery action may reactivate the legacy deployment or assign a
different Fleet identity. Regenerating or discarding installed state is not
recovery.

Before the current-root `Stop` intent exists, abort first reconciles every
non-root stop, starts every stopped non-root under exact `Start` intents,
proves the complete Fleet is running its old modules and protected legacy
identity, restores the pre-fence App state through the current `canic_app`
command, re-queries the complete registry/inventory, proves it still equals
the frozen evidence, removes the host mutation fence, records a terminal abort
receipt and returns to `LegacyActive`. Failure to restore or prove any part
leaves the attempt active and fails closed.

An abort from `FleetQuiescing` is a protocol, not a catalog rollback:

1. stop issuing new root-stop work and reconcile every pending intent through
   `canister_status`;
2. prove every Canister still has `old_module_hash`, the exact canonical
   controller set and no accepted 0.99 `FleetBinding`;
3. start every stopped or stopping Canister, persisting `Start` intent before
   each call;
4. prove every Canister is `running` with its old module and exact
   `expected_legacy_protected_state_hash`;
5. restore the exact pre-fence App state, prove the frozen inventory remains
   unchanged and remove the host mutation fence;
6. commit the final `LegacyActive` journal generation and write the canonical
   terminal abort receipt covering its sequence/hash, both content-manifest
   hashes and every final status-evidence hash; and
7. remove the unused staged target and retain every journal generation and
   the abort receipt in the fixed recovery location.

Failure of any proof leaves the journal in `FleetQuiescing` and fails closed.
No abort exists from `FleetQuiesced` onward.

### Canonical recovery and activation evidence

All new 0.99 host-manifest, journal-intent, cascade, credential and activation
hashes use deterministic CBOR. Records are fixed arrays in the field order
shown in this document; integers use shortest encoding; principals, identities
and digests use raw byte strings; vectors are definite-length; optional values
use CBOR null; and maps, text keys, floats and indefinite-length values are
forbidden. Principal sets are sorted by raw principal bytes. A decoded
non-canonical order is rejected rather than silently reordered at an evidence
boundary.

Every evidence hash is:

~~~text
SHA-256(
  domain_separator
  || u64_be(canonical_byte_length)
  || exact_canonical_bytes
)
~~~

The exact domain separators are:

~~~text
canic:release-build:plan\0
canic:fleet-upgrade:host-source-manifest\0
canic:fleet-upgrade:host-target-manifest\0
canic:fleet-upgrade:journal-generation\0
canic:fleet-upgrade:structure-fence\0
canic:fleet-upgrade:controller-set\0
canic:fleet-upgrade:management-status\0
canic:fleet-upgrade:legacy-protected-state\0
canic:fleet-upgrade:args\0
canic:fleet-upgrade:terminal-receipt\0
canic:fleet-activation:state-snapshot\0
canic:fleet-activation:topology-snapshot\0
canic:fleet-activation:cascade-manifest\0
canic:fleet-activation:credential-root-policy-set\0
canic:fleet-activation:credential-renewal-template-set\0
canic:fleet-activation:credential-key-identity\0
canic:fleet-activation:credential-proof\0
canic:fleet-activation:credential-bundle\0
canic:fleet-activation:credential-manifest\0
canic:fleet-activation:activation-evidence\0
canic:fleet-install:activation-journal\0
~~~

The upgrade-only separators bind host migration and management reconciliation.
The activation separators are shared by fresh installation and coordinated
upgrade through `FleetActivationIdentity`.

`ReleaseBuildPlanRecord` canonical bytes are
`[nonce, release_build_id, state]`, where state is `[0]` for `Planned` and
`[1, release_set_manifest_digest]` for `Finalized`. The finalized record's
domain-separated hash is the `release_build_plan_hash` stored by deployment
recovery.

Journal-generation bytes are the fixed array of every
`FleetMigrationUpgradeJournal` field in the struct order shown above.
`FleetMigrationUpgradePhase` discriminants are the declaration order
`0..=9`. `FleetCanisterUpgradeProgress` and
`FleetCanisterOperationIntent` use their declaration order from `0`.
`FleetMigrationUpgradeAuthority` encodes as
`[0, upgrade_id, sequence, journal_hash]` for `Active` and
`[1, outcome, upgrade_id, final_sequence, final_journal_hash, receipt_hash]` for
`Terminal`; terminal outcomes are `0 = Completed`, `1 = Aborted`.
The 20-digit filename sequence is zero-padded ASCII decimal and is checked
against the decoded `u64`.

`FleetInstallActivationJournal` canonical bytes are the fixed array of every
field in declaration order. Phase discriminants are declaration order `0..=4`.
Its domain-separated journal hash binds the exact current sequence and fields;
it is retained as install evidence but is not a second mutable pointer.

The controller-set bytes are one array of sorted raw principals. Management
status bytes are:

~~~text
[
  canister_principal,
  status_discriminant,
  canister_version,
  module_hash_or_null,
  [controller_principals]
]
~~~

The status discriminants are `0 = running`, `1 = stopping` and `2 = stopped`.

The legacy projection is exact:

~~~rust
pub struct LegacyProtectedIdentityProjection {
    pub prime_root_pid: Option<Principal>,
    pub subnet_role: Option<SubnetRole>,
    pub subnet_pid: Option<Principal>,
    pub root_pid: Option<Principal>,
    pub canister_role: Option<CanisterRole>,
    pub parent_pid: Option<Principal>,
}
~~~

It is decoded from the existing `v0.98.24` `canic_env` response and encoded as
the fixed array
`[prime_root_pid, subnet_role, subnet_pid, root_pid, canister_role,
parent_pid]`. The pre-implementation inventory must prove that each projected
field is protected
installation identity and cannot change through App-status cascade,
management stop/start or application-data mutation. No response envelope,
future diagnostic field or mutable health/status field participates.
Legacy protected-state bytes are
`[canister_principal, old_module_hash, expected_controllers_hash,
LegacyProtectedIdentityProjection]`.

Every value is available from management status and the existing
`v0.98.24` `canic_env` query before mutation. This is the immutable legacy
identity and installation evidence needed to prove that stop/start did not
accept a new Fleet binding; it is not a hash of mutable application data,
whose already accepted callbacks may legitimately finish while a Canister is
stopping.
`FleetUpgradeArgs` bytes are
`[FleetBinding, upgrade_id, ReleaseBuildId]`. Operation-intent
bytes are `[canister_principal, operation_discriminant, operation_payload]`
where `0 = Stop`, `1 = Install`, `2 = Start`; the payload is the exact fields
of the corresponding enum variant above. The source and target manifest bytes
are the exact arrays defined above.

Structure-fence bytes are
`[FleetBinding, upgrade_id, prior_app_status, readonly_status_response_hash,
subnet_registry_hash, host_inventory_hash, membership_receipt_set_hash,
[]]`; the final empty array is the required pending structural-operation hash
set after reconciliation. The terminal receipt bytes are
`[outcome_discriminant, FleetBinding, upgrade_id, final_sequence,
final_journal_hash, source_manifest_hash_or_null, target_manifest_hash_or_null,
committed_fleet_catalog_hash_or_null,
[[principal, final_status_evidence_hash]]]`, with `0 = Completed`,
`1 = Aborted` and the last array in raw-principal order. Completion requires
both manifests and the committed Fleet-catalog hash. A pre-stop abort may use
null for a manifest not yet constructed and always uses null for the catalog;
a `FleetQuiescing` abort requires both manifest hashes.

## Fresh-Install Fleet Activation

0.99 deliberately breaks public query names, Candid record labels, Directory
envelopes and delegation variants. It installs only into empty Canister and
host state. Mixed 0.98/0.99 operation and in-place upgrade are unsupported.

The internal cascade endpoint names stay fixed:

- `canic_sync_state` carries the breaking state payload;
- `canic_sync_topology` and its `TopologySnapshotInput` field labels remain
  unchanged; and
- renaming a Rust DTO type without changing its structural Candid shape is not
  itself a wire break.

The known record-field cuts are:

| Record | Removed field | Current field |
| --- | --- | --- |
| `StateSnapshotInput` | `app_state` | `fleet_state` |
| `StateSnapshotInput` | `app_index` | `fleet_directory` |
| `StateSnapshotInput` | `subnet_index` | `subnet_directory` |
| `CanisterInitPayload` | `app_index` | `fleet_directory` |
| `CanisterInitPayload` | `subnet_index` | `subnet_directory` |
| `CanisterInitPayload` | absent | `fleet`, `install_id`, `release_build_id` |
| `EnvBootstrapArgs` | `subnet_role` | `subnet_slot` |
| `EnvSnapshotResponse` | `subnet_role` | `subnet_slot` |
| `InstallActiveDelegationProofRequest` | `proof` | `credential`, `bundle` |
| `DelegationAudience` | `canister`, `canic_subnet`, `project` variants | sole `fleet` variant |

`AppIndexArgs` and `SubnetIndexArgs` travel inside
`StateSnapshotInput` through `canic_sync_state`; they are not part of
`TopologySnapshotInput`. Their type-name renames are free on the wire, while
the enclosing field-label changes and new provenance envelopes are breaking.
The `EnvBootstrapArgs.subnet_slot` and
`EnvSnapshotResponse.subnet_slot` values also hard-cut the reserved slot from
`prime` to `default`; this is a data-value migration in addition to the field
rename.
The semantic inventory confirms this as the complete changed Candid
field/variant set. Any later implementation discovery that contradicts it
requires a design correction before code changes continue.

The bounds are admission bounds, not post-decoding assertions. The raw update
adapter rejects a payload over its endpoint bound before an unbounded Candid
allocation; bounded visitors reject more than
`MAX_FLEET_ACTIVATION_CANISTERS` cascade/host entries or
`MAX_FLEET_CREDENTIAL_MANIFEST_ENTRIES` credential entries before reserving.
The deterministic-CBOR bytes are then checked against the byte constants
above. A single canonical credential bundle over
`MAX_FLEET_CREDENTIAL_BUNDLE_BYTES`, a retained bundle-generation set over
`MAX_FLEET_CREDENTIAL_BUNDLE_SET_BYTES`, or the complete ID-21

`MAX_FLEET_ACTIVATION_HOST_RECORD_BYTES`.

`MAX_FLEET_CREDENTIAL_INSTALL_PAYLOAD_BYTES` before decoding and still
enforces the smaller canonical bundle bound after decoding.

Stable allocation ID 21 owns the one `FleetActivationRecord` on every managed
Canister. On the root, it owns the complete cascade-manifest vector and the
current/previous prepared credential manifests; on a non-root those
root-source vectors are empty and the state owns only its applied evidence and
accepted generation reference. The existing `AUTH_STATE_ID` allocation owns
the exact root proof bundles and their delivery/retry records, keyed by
activation identity and credential generation. ID 21 does not duplicate proof
bytes, and auth state does not duplicate the Fleet binding, cascade manifest
or credential-manifest bytes.

The host uses the exact `FleetActivationHostRecord` as its durable activation
owner. Fresh installation has this single catalog-independent recovery
authority:

~~~rust
pub enum FleetInstallActivationPhase {
    Planned,
    RootInstalled,
    CanistersPrepared,
    CanistersActivated,

    pub committed_fleet_catalog_hash: Option<[u8; 32]>,
~~~

The journal is stored at:

~~~text
.canic/recovery/fleet-install-activations/
  <canonical-network-id>/<fleet-id>/<operation-id>/journal.cbor
~~~

Planning is serialized per `(CanonicalNetworkId, FleetName)` by a
kernel-owned advisory lock opened with close-on-exec at
`.canic/recovery/fleet-install-activation-locks/<canonical-network-id>/<fleet-name>.lock`.
The lock file is only a live-process exclusion mechanism: owner death releases
the lock, and neither its bytes nor its presence are durable recovery
authority. There is no active pointer or second journal index.

While holding that lock, the host discovers authority by sorting and scanning
the canonical Fleet-ID and operation-ID directories for the complete canonical
network subtree. Every published journal is decoded and path-bound before the
requested Fleet is selected. A non-canonical, symlinked, special,
path-mismatched or contradictory entry fails closed. An operation directory
without a published `journal.cbor` is inert evidence of a crash before atomic
publication and is not an activation authority.

No matching active journal allocates fresh Fleet and operation identities. One
matching journal with the exact requested App and finalized release-build
identity, plan hash and manifest digest is resumed idempotently. A different
App or release build, multiple active journals for one Fleet name, or one
Fleet ID claimed by multiple active journals blocks planning with a typed
conflict. Terminal journals remain retained evidence but are excluded from
resumable discovery once their terminal transition is implemented.

The `Planned` journal is created only from a finalized
`ReleaseBuildPlanRecord`, is synced before the first Canister mutation and
must contain the same plan hash, build ID through `activation.identity`, and
manifest digest. Each later phase increments `sequence` with checked
arithmetic and replaces the file through the same staged-file durability rule.
A crash before the rename leaves the preceding complete sequence
authoritative.

`RootInstalled` requires the exact durable root-install receipt. The host
admits only the canonical JSON bytes written by the maintained receipt owner
for one completed successful `install_root` phase. Its verified postcondition
must contain one valid root principal plus canonical expected and observed
module hashes that are equal. A command-success result without that observed
module evidence cannot advance the journal. `root_install_receipt_hash` is the
raw SHA-256 of the exact durable receipt bytes; a different receipt cannot
replay or replace the transition.

`CanistersPrepared` requires the complete bounded activation inventory and
every Canister in `Prepared`; `CanistersActivated` requires every Canister in
`Active`; and `HostAuthorityCommitted` requires the installed Fleet catalog
and deployment receipt to be durable and records their exact catalog hash.
Fields that are not yet legal for a phase are `None`; a field that is required
by a phase is never represented by a zero sentinel. The completed journal is
retained as evidence but is not resumable authority. A workflow has exactly
one fresh-install activation owner.

Fresh root and non-root inputs normalize `install_id` to `operation_id`.
This internal identity is the one canonical activation model; stable state
does not keep parallel install/upgrade variants.

The 0.99 `init` path initializes only the current schema, installs the exact
binding and atomically commits `Prepared` in the same message. It rejects
non-empty 0.98 state rather than decoding it. A trap aborts the management
operation; a successfully installed 0.99 module is never durably `Unbound`.
This relies on the management Canister's controller-only argument and
[atomic install contract](https://docs.internetcomputer.org/references/management-canister/).
The presence of `Prepared { identity, .. }` is the proof that the exact
`FleetBinding` was installed atomically; there is no redundant
`fleet_binding_installed` boolean that could disagree with the enum state.

The new activation/recovery endpoints are exact:

~~~text
canic_fleet_activation_status()
  -> Result<FleetActivationStatusResponse, Error>

canic_prepare_fleet_activation()
  -> Result<FleetActivationStatusResponse, Error>

canic_resume_fleet_activation(FleetActivationResumeRequest)
  -> Result<FleetActivationStatusResponse, Error>

canic_prepare_fleet_credential_generation(FleetCredentialGenerationRequest)
  -> Result<FleetActivationStatusResponse, Error>

canic_activate_fleet(FleetActivationRequest)
  -> Result<FleetActivationStatusResponse, Error>
~~~

`canic_fleet_activation_status` is a controller query on every managed
Canister. The root-only controller update `canic_prepare_fleet_activation`
constructs and persists the exact cascade/credential evidence and may complete
the state/topology cascades, but it does not deliver credentials or activate a
Canister. A lost response is recovered through the status query, whose root
projection includes the protected current credential-manifest bytes as well
as their generation/hash; non-root responses always return
`cascade_manifest = None` and `credential_manifest = None` but do return their
applied cascade evidence and accepted generation reference. The root returns
its complete bounded cascade manifest as well as the current credential
manifest, so either lost response is recoverable from ID 21.

The root-only controller update `canic_resume_fleet_activation` requires the
exact operation ID and host-journalled credential generation, delivers the
stored credential bundles and drives non-root activation.
`canic_prepare_fleet_credential_generation` is the internal non-root update
that accepts only the protected current root and durably installs or advances
the exact generation reference without activating. The root calls it for
every non-root, including configured issuers, before proof delivery and final
activation. `canic_activate_fleet` accepts only the protected current root with
exact operation/evidence identity and requires the generation already stored.
Once every non-root reports `Active`, the resume workflow commits the root
`Active` last. All five endpoints are idempotent for their exact identity and
reject any other identity or phase.

While `Prepared`, a Canister accepts only:

- controller/root activation diagnostics and recovery inspection;
- the exact `canic_sync_state` call from the protected current root;
- the exact `canic_sync_topology` call from that root;
- on the root, controller installation of the exact journal-bound
  `canic_upsert_root_issuer_policy` and
  `canic_upsert_root_issuer_renewal_template` sets;
- the root's exact prepared-mode credential prepare, manifest inspection,
  generation-bound resume and existing proof-install workflow;
- on every non-root, installation or checked advancement of the exact
  credential-generation reference by the protected current root through
  `canic_prepare_fleet_credential_generation`, even when that Canister is not
  a configured issuer;
- on a configured issuer, `canic_install_active_delegation_proof` from the
  protected current root and `canic_active_delegation_proof_status` for
  reconciliation;
- the idempotent exact activation call; and
- on the root only, the controller-only install-resume path that builds,
  persists and sends those two current cascades.

Every other Canic-managed query, update, delegated-authentication path,
provisioning action, topology mutation, ordinary cascade, timer owner and
Canic control-plane operation remains fenced. The fence is enforced in the
shared `canic_query`/`canic_update` dispatch path.

Qualification requires every application endpoint exported by a managed
Canister to use `canic_query` or `canic_update`. Raw `#[ic_cdk::query]`,
`#[ic_cdk::update]` and equivalent direct exports bypass Canic dispatch and
are outside the qualified managed-Canister contract. The pre-implementation
inventory classifies them;
implementation closeout rejects any raw export in a managed Fleet Canister.
`canister_inspect_message` is not treated as a query, inter-Canister or
management-call fence.

“Abort” can record a terminal fail-closed diagnostic but cannot return an
installed Canister to an unbound identity.

For every managed non-root, the prepared root constructs the exact outgoing
`StateSnapshotInput` and target-specific `TopologySnapshotInput`. It encodes
each logical DTO under the deterministic CBOR rules above and hashes the exact
canonical bytes under the state- or topology-snapshot separator. The root
persists the complete `FleetCascadeManifestEntry` vector in raw-principal
order, then stores its cascade-manifest hash as `Source` evidence before
sending anything. It never hashes an internal “local snapshot” as a substitute
for the outgoing DTO.

Each non-root reconstructs the same canonical bytes from the exact accepted
DTO and stores their hashes as `Applied` evidence before acknowledging the
cascade. Its two hashes must equal its root-manifest entry. Root and child
top-level cascade evidence are deliberately not equal: the root stores the
complete source-manifest hash, while a child stores its two applied payload
hashes. Payload hashes for different children are also not required to match;
topology snapshots are target-specific, and even equal current state payloads
are compared only with their own manifest entries.

`FleetActivationEvidence` canonical bytes are
`[FleetActivationIdentity, cascade_evidence_or_null,
credential_generation_ref_or_null]`.
`Source` uses discriminant `0` followed by the cascade-manifest hash; `Applied`
uses discriminant `1` followed by the state and topology hashes. The
`activation_evidence_hash` stored in the host journal is the domain-separated
hash of these exact bytes. `Active` requires both fields to be present for the
same `FleetActivationIdentity`, and `FleetActivationRequest.credential` is
therefore required even for a non-issuer. The receiver requires that exact
reference already be stored, reconstructs its canonical activation evidence
and accepts the request only when the resulting hash equals
`activation_evidence_hash`.
Non-roots activate first and the root activates only after every managed
non-root reports evidence matching its source manifest and the same credential
generation.

Install hooks do not start runtime timers. Timers are established only after
the durable transition to `Active`, through the normal zero-delay lifecycle
adapter. Restart in `Prepared` therefore cannot silently start periodic work.

### Authorization-state initialization

The current authorization allocation is the single `AUTH_STATE_ID` cell.
A 0.99 fresh install atomically creates the current record with:

- `delegated_sessions`,
  `delegated_session_bootstrap_bindings`, `root_issuers`,
  `root_issuer_renewal_templates`, `root_issuer_renewal_states` and
  `chain_key_root_delegation_batches` empty;
- `fleet_credential_bundle_generations` empty;
- `active_delegation_proof` absent; and
- `delegated_auth_registry_epoch` and `delegated_auth_proof_epoch` initialized
  to their current empty-state values.

Heap and certified auth caches start empty. There is no old authorization
decoder and no old session, proof, issuer policy, renewal state or batch is
copied into Fleet-scoped authority.

### Fleet credential bootstrap

0.99 reuses the existing chain-key root delegation issuance workflow; it does
not introduce another credential authority. The prepared root uses
`RuntimeAuthWorkflow::get_or_create_chain_key_delegation_proof_for_issuer_root`
to prepare or reuse the exact proof for each configured issuer, followed by the
existing batch installation path used by
`RuntimeAuthWorkflow::provision_chain_key_delegation_proof_for_issuer_root`.
That path calls `canic_install_active_delegation_proof` on the subject issuer.
The end-user `canic_prepare_delegated_token` and
`canic_get_delegated_token` endpoints remain fenced until `Active`; they are
not bootstrap mechanisms.

The host compiles the exact Fleet-audience `RootIssuerPolicyUpsertRequest` and
`RootIssuerRenewalTemplateUpsertRequest` sets from the selected App
configuration before `HostTargetPrepared`. It sorts each set by raw issuer
principal, rejects duplicate issuers, hashes their canonical bytes under the
root-policy-set and renewal-template-set separators, and retains those exact
bytes in the target tree covered by `target_manifest_hash`.

Root-policy-set bytes are:

~~~text
[
  [
    issuer_pid,
    enabled,
    [allowed_audiences],
    [allowed_grants],
    max_cert_ttl_ns,
    refresh_after_ratio_bps
  ]
]
~~~

Renewal-template-set bytes are:

~~~text
[[issuer_pid, enabled, audience, [grants], cert_ttl_ns]]
~~~

Each outer set is ordered by raw issuer principal. Audience and grant vectors
are ordered by their raw canonical encodings and reject duplicates. Every
enabled policy/template audience is exactly `Fleet(FleetKey)` for this
activation; no old or ambient-network audience value can enter the prepared
root.

After the root reaches `Prepared`, the host controller may use only the
prepared-mode forms of `canic_upsert_root_issuer_policy` and
`canic_upsert_root_issuer_renewal_template` to install those exact sets. The
root verifies every audience equals its protected `FleetKey`. This restores
the root's issuance policy before proof creation. The root does not need an
active child-issued proof for itself: its existing authoritative chain-key
signing workflow, configured key identity and the newly installed Fleet-scoped
policy are the issuance authority.

The exact child delivery bundle and its manifest are:

~~~rust
pub struct FleetCredentialBundle {
    pub fleet: FleetKey,
    pub activation_id: [u8; 32],
    pub generation: u64,
    pub root_issuer: Principal,
    pub subject_canister: Principal,
    pub not_before_ns: u64,
    pub expires_at_ns: u64,
    pub key_identity_hash: [u8; 32],
    pub cert_hash: [u8; 32],
    pub proof_hash: [u8; 32],
    pub proof: DelegationProof,
}

pub struct InstallActiveDelegationProofRequest {
    pub credential: FleetCredentialGenerationRef,
    pub bundle: FleetCredentialBundle,
}

pub struct FleetCredentialManifestEntry {
    pub root_issuer: Principal,
    pub subject_canister: Principal,
    pub not_before_ns: u64,
    pub expires_at_ns: u64,
    pub key_identity_hash: [u8; 32],
    pub cert_hash: [u8; 32],
    pub proof_hash: [u8; 32],
    pub bundle_hash: [u8; 32],
}

pub struct FleetCredentialManifest {
    pub fleet: FleetKey,
    pub activation_id: [u8; 32],
    pub generation: u64,
    pub root_policy_set_hash: [u8; 32],
    pub renewal_template_set_hash: [u8; 32],
    pub entries: Vec<FleetCredentialManifestEntry>,
}
~~~

`FleetCredentialGenerationRef` canonical bytes are
`[generation, manifest_hash]`. `manifest_hash` is the domain-separated hash of
the exact manifest bytes below, and both root and host recompute it rather than
trusting a supplied value.

Bundle bytes are the fixed array of fields shown above in that order, with the
exact canonical `DelegationProof` bytes last. `proof_hash` is the
domain-separated hash of those proof bytes. `key_identity_hash` is the
domain-separated hash of
`[algorithm, key_id, derivation_path_hash, sha256(public_key)]` from the
proof's root-key evidence. `cert_hash`, `not_before_ns` and `expires_at_ns`
must equal the corresponding verified proof values. `bundle_hash` is the
domain-separated hash of the exact bundle bytes. Every nested hash is
recomputed during preparation and installation; disagreement fails closed.
Manifest bytes are:

~~~text
[
  FleetKey,
  activation_id,
  generation,
  root_policy_set_hash,
  renewal_template_set_hash,
  [
    [
      root_issuer,
      subject_canister,
      not_before_ns,
      expires_at_ns,
      key_identity_hash,
      cert_hash,
      proof_hash,
      bundle_hash
    ]
  ]
]
~~~

Manifest entries are ordered by raw subject principal and contain exactly the
configured delegated-token issuers. Generation starts at `1`; `0` is invalid.
The root constructs and verifies every bundle, stores the exact bundles in the
existing auth allocation, and stores the complete manifest in ID 21 before
any child delivery. Prepared diagnostics return the manifest bytes and its
`FleetCredentialGenerationRef` to the host. The host verifies them against
the root-policy/template hashes, persists the complete manifest and reference
in its sole `FleetActivationHostRecord`, and only then resumes the prepared
root with that exact reference.

The root rejects a resume reference that differs from its protected manifest.
It then delivers the already stored proof from each exact bundle through
`canic_install_active_delegation_proof`. Only the protected current root may
make that call, it must still be a controller of the subject, and the subject
must verify the proof's root, issuer, Fleet audience, validity, certificate and
key identity against its bundle, generation reference and protected
`FleetBinding`. The existing
`InstallActiveDelegationProofRequest` evolves in place to the exact request
above, so the subject also records the same generation reference; there is no
parallel legacy request shape.

An unknown install outcome is reconciled using the subject's existing
`canic_active_delegation_proof_status` certificate hash and the root's durable
chain-key batch/install record. A missing proof repeats delivery of the exact
stored bundle; the exact certificate hash for the same generation is
idempotent. Every managed Canister records the same journalled generation
reference in `FleetActivationEvidence`; a configured issuer additionally
proves its exact manifest entry is installed. The root activates only after
all manifest entries and every Canister's recorded reference agree.

Credential expiry during a long recovery is handled by one bounded
generation protocol, not by abandoning or changing the activation:

1. a replacement remains bound to the same `FleetActivationIdentity`, Fleet,
   policy-set hash and renewal-template-set hash;
2. before preparing generation `n + 1`, the host and root reconcile every
   delivery attempt for `n` to a known installed, absent or expired result;
3. the root increments `generation` with checked arithmetic, creates fresh
   exact bundles and one new manifest, persists both durable owners, and the
   host persists the new manifest/reference before delivery;
4. a `Prepared` Canister may replace `n` only with exact `n + 1` from its
   protected root for the same activation identity; equal generation/hash is
   idempotent, a skipped, lower, same-number/different-hash or unrelated proof
   fails closed;
5. every non-root receives the new reference through
   `canic_prepare_fleet_credential_generation`; issuers then receive the exact
   proof bundle for that reference, and no `FleetActivationRequest` is sent
   until every managed Canister acknowledges the same generation;
6. the root retains at most the current and immediately preceding prepared
   manifest in ID 21 and bundle generation in auth state. It discards the
   preceding generation only after every managed Canister durably acknowledges
   the current one; and
7. activation requires one generation across every managed Canister and then
   freezes it in `Active`. Prepared-mode generation replacement is
   unavailable after activation.

This bounds retained state to
`MAX_RETAINED_PREPARED_CREDENTIAL_GENERATIONS` without bounding recovery time
or leaving a forward-only migration permanently stuck on expired proof bytes.
Checked generation overflow fails closed before preparing a new proof.

No `canic_activate_fleet` call is issued until every managed Canister has
durably acknowledged the same generation and every configured issuer has
verified and stored its exact bundle while valid. Once the first activation
call is issued, generation replacement is closed for the activation and
recovery only resumes the exact activation calls. If interruption then
outlives the certificate, the already verified expired proof remains
non-authorizing; activation may rely on its durable installation/certificate
evidence, but delegated authentication continues to fail closed until the
normal post-`Active` renewal workflow installs a currently valid Fleet proof.
It never replays an expired bundle or changes activation generation after the
first activation transition.

The auth-allocation addition is exact:

~~~rust
pub struct FleetCredentialBundleGenerationRecord {
    pub identity: FleetActivationIdentity,
    pub credential: FleetCredentialGenerationRef,
    pub bundles: Vec<FleetCredentialBundle>,
}
~~~

`AuthStateRecord` owns a vector of at most two such records, ordered by
increasing generation; each record's bundles are ordered by raw subject
principal. It contains at most 64 bundles, rejects duplicate subjects, enforces
the per-bundle and aggregate byte bounds above and must match the corresponding
ID-21 manifest exactly. Fresh authorization initialization creates the vector
empty. It is not a second binding authority: every read verifies
`identity == FleetActivationRecord.state.identity()` and derives the Fleet
only from that protected activation identity.

### Rejected Alternative: 0.98 Quiescence and Upgrade Sequence

This sequence belongs to the rejected in-place migration alternative above.
0.99 does not implement it or test it as a supported journey.

0.99 uses management-Canister stop/status evidence rather than an informal
traffic pause. Before `HostTargetPrepared`, preflight proves that the selected
deployment controller can directly stop, inspect, install and start every
managed Canister. A Fleet lacking that controller authority is not qualified
for this upgrade and fails before mutation.

The [management-Canister contract](https://docs.internetcomputer.org/references/management-canister/)
permits `stop_canister` to time out while the transition may already be in
progress; `canister_status` exposes `running`/`stopping`/`stopped`, module hash
and controller settings. That is why an operation intent precedes every call
and timeout recovery always observes state instead of guessing.

The one-Subnet sequence is:

1. Preflight controller authority, selected artifacts, the Candid cut,
   stable migrations and the Fleet-scoped root policy/template sets without
   treating the first observed inventory as final.
2. Commit `FleetStructureFencing`, install the durable host mutation fence,
   propagate the current 0.98 `Readonly` state and reconcile all pending
   structural operations.
3. Capture the candidate complete registry/inventory, stop every managed
   non-root under its journalled intent, recapture the root Registry and
   validate exact agreement across inventory, controller sets, module hashes,
   protected legacy identity and membership-bearing receipts; commit
   `FleetStructureFenced` with that final evidence.
4. Build the source/target manifests, target tree and complete journal
   Canister vector only from that frozen evidence, then commit
   `HostTargetPrepared`.
5. Commit `FleetQuiescing`, persist the current root's `pending = Stop`, issue
   `stop_canister`, and reconcile through `canister_status`, updating the
   intent while status is `stopping` and committing `LegacyStopped` only with
   exact stopped-status, controller and old-module evidence. Until the next
   phase commits, the explicit abort protocol remains available.
6. Commit `FleetQuiesced` only after every entry is `LegacyStopped`. The legacy
   tree is now recovery input only and recovery is forward-only.
7. For every stopped non-root, persist the exact `Install` intent, issue the
   upgrade, reconcile `UpgradedStopped`, persist the exact new-module `Start`
   intent, start it, and verify matching `Prepared` diagnostics.
8. Perform the same intent-driven install/start sequence on the stopped current
   root last.
9. Reconcile an unknown management result only through status, the intent's
   old or expected module hash, activation diagnostics and unchanged recovery
   surfaces. Never restart the legacy deployment.
10. Commit `CanistersPrepared` only when the complete journal inventory is
   prepared under the same binding, upgrade ID and `ReleaseBuildId`, while the
   host has independently validated the journalled release-set manifest
   digest.
11. Through the root's prepared-mode resume path, construct and persist the
   exact per-Canister cascade manifest, send the corresponding canonical
   `StateSnapshotInput` and `TopologySnapshotInput` values, and reconcile every
   non-root's applied hashes.
12. Install the exact root policy/template sets, prepare and persist one
    bounded Fleet credential generation, journal its complete manifest and
    reference on the host before delivery, then deliver and reconcile the same
    bundles and reference. If the proofs expire, advance through the exact
    generation protocol above. Make every Canister converge on one generation.
13. Persist every non-root's complete activation-evidence hash, activate
    non-roots idempotently, activate the root last and commit
    `CanistersActivated`.
14. Write the complete replacement Fleet catalog to a sibling staged file,
    sync it, rename it on the same filesystem, sync the parent directory,
    then commit `HostAuthorityCommitted`.
15. Preserve legacy immutable evidence, prewrite the candidate
    `LegacyPathRetired` generation and its terminal completion receipt, then
    remove the obsolete operational path and advance `active.cbor` to that
    exact generation only after removal is proven.
16. Advance `active.cbor` to its typed terminal reference and remove the
    active pointer only after the terminal artifacts are durable and
    self-consistent.

An interruption resumes the exact phase and per-Canister entry from the new
recovery journal. Mixed-version state is a transitional journal state, never
an operational mode.

During mixed state, host transition, readiness, diagnostics and recovery logic
may call only an unchanged wire surface or a target the journal proves already
runs the expected release. The invariant is satisfied for local registry
recovery by keeping the exact `canic_subnet_registry` endpoint and response
signature unchanged. Every Canister is stopped before the first breaking
upgrade, so the old root sends no old cascade to new code. The new root starts
its current-only cascade only after every non-root is prepared, so it never
sends the new shape to old code.

PocketIC must qualify interruption and restart at every durable boundary. A
compatibility endpoint, compatibility field, old delegation variant or
version-sniffing fallback is not an acceptable recovery mechanism.

## Runtime and Stable-State Boundary

The terminology hard cut preserves topology behavior while deliberately
changing the minimum identity, provenance and authorization schemas described
above.

0.99 must preserve:

- current root and child behavior;
- current registry and directory contents;
- current cascade ordering and authorization;
- current role and service cardinality; and
- current one-subnet install behavior.

For a pure source-name change:

- its existing stable-memory allocation ID remains owned by the renamed
  canonical structure;
- its existing stable key literal remains byte-for-byte unchanged, even if the
  opaque allocation label contains an old source term;
- the serialized map keys, values and record field bytes remain unchanged;
- there is one reader and one writer after the cut;
- no parallel old/new stable map is introduced; and
- state-contract metadata and allocation evidence are updated consistently.

Stable allocation IDs and key literals are ABI identities, not public
terminology aliases. Changing them would create an unnecessary data migration,
so they are retained and documented as allocation-preserving source renames.

## Authority Model After 0.99

0.99 ends with one qualified one-member Fleet. Future authority names are

## Hard-Cut Completion Criteria

0.99 is complete when:

1. Active Canic surfaces consistently use `App` for Toko-like source identity.
2. Active Canic surfaces consistently use `Fleet` for one live deployed App
   over one or more Subnets.
3. The current one-root installation is represented truthfully as a
   one-member Fleet whose current root acts as both member root and temporary
   Fleet-wide authority, without persisting a final Coordinator or Authority
   Subnet binding.
4. Every concrete Canister has a stable
   `(CanonicalNetworkId, FleetId, principal)` identity independent of its
   observed Subnet, without relying on role uniqueness or ambient network
   context, while `AppId` is exactly the immutable canonical `[app].name`.
5. Registry means complete authoritative records within its scope.
6. Directory means a derived, read-only, consumer-shaped projection that is
   cascaded and queried with explicit Fleet/source-root provenance.
7. Index is absent from public topology terminology.
8. Keyed placement uses Placement Binding terminology rather than Directory
   terminology.
9. `canic.toml` identifies the App through `[app].name` and contains no live
   Fleet identity or fabricated runtime membership.
10. App-declared logical topology uses `SubnetSlotId`; `SubnetId` is reserved
    for resolved platform identity, and `subnets.default` carries no authority
    meaning.
11. Initial installation or coordinated activation binds immutable Fleet, App
    and canonical-network identity in host state and protected state on every
    managed Canister, and later operations enforce it.
12. Delegation audiences are Fleet-scoped, with no accepted Canister,
    CanicSubnet or Project variant or credential after the cut.
13. No compatibility parser, endpoint, Candid field, delegation variant, path
    or JSON field preserves the old
    terminology in normal operation.
14. 0.99 has no active distinguished-Subnet selector or identity; forward
    architecture says Authority Subnet, while active old `Prime` surfaces
    resolve individually to `default`, current-root/Fleet-root terms or
    removal.
17. `[services.fleet].roles` is service-owned authoring input and does not make
    a Directory authoritative or activate service ownership in 0.99.
18. No multi-subnet runtime capability is claimed or implemented by this
    terminology release.
19. Every managed Canister remains locally mutation-fenced in `Prepared` until
    its exact cascade evidence and one journalled Fleet credential-generation
    reference are durable for one activation operation; timers resume only
    after `Active`, and the root activates last.

## Follow-On Multi-Subnet Design

After 0.99, the
[Canic 0.100 multi-subnet design](../0.100-multi-subnet-fleet-coordinator-and-registry-synchronization/0.100-design.md) can
implement the Fleet behavior that the original 0.99 draft attempted to include:

~~~text
0.99
  “What is an App, Fleet, Subnet, Canister, Registry and Directory?”

Follow-on Fleet membership
  “How does the Fleet Coordinator register, activate, drain and remove member
   Subnets?”

Follow-on synchronization
  “How do member roots mirror the Fleet Registry and cascade the Fleet
   Directory?”

Follow-on authority services
  “How are existing services adopted in place, missing services provisioned
   on the Authority Subnet and the complete committed Directory published?”
~~~

0.100 owns bounded membership snapshots, revision and freshness evidence,
interruption, root-only inter-subnet calls, durable mirror activation and
member lifecycle. 0.101 owns authority-service adoption, provisioning,
ownership handoff and complete Directory publication. Neither behavior is part
of 0.99; 0.99 carries only immutable identity, the one-member authority and
one-subnet Directory source provenance.

The forward request direction is fixed without implementing it here:

~~~text
Member request with idempotency key and expected authority revision
  → authority validates
  → authority commits the canonical change
  → Fleet revision advances
  → member roots synchronize
  → local cascades distribute the derived projection
~~~

Receipt of a request is never commitment. Only the authority's revisioned
result becomes Fleet truth. Fleet Registry synchronization remains distinct
from mutable application-data replication.

## Final CLI Command-Surface Closeout

The terminology inventory above is necessary but not sufficient. Before 0.99
is complete, the CLI must also prove that every maintained command still has a
current purpose.

The implementation must extend the 0.99 semantic inventory with every
top-level command and every nested leaf accepted by the immutable predecessor,
plus every command introduced or renamed during 0.99. Each leaf receives
exactly one disposition:

| Disposition | Required result |
| --- | --- |
| `keep` | Name one current owner, consumer and behavior that remains necessary after the 0.99 hard cut |
| `rename` | Introduce the one current App/Fleet name and delete the predecessor spelling, help, parser and tests |
| `move` | Establish one canonical command path and delete the old path rather than retaining two entrypoints |
| `delete` | Remove dispatch, parsing, implementation, help, examples, tests and active documentation |

There is no `defer`, `legacy`, `alias` or `compatibility` disposition for a
current pre-1.0 command. A command needed as groundwork for 0.100 or 0.101 is
still classified as `keep` and must name that exact future owner; anticipated
use alone is not evidence.

The inventory is recursive. At minimum, it covers the complete proposed
post-cut top-level surface:

~~~text
status
medic
state
app
scaffold
replica
network
install
inspect
blob-storage
auth
build
deploy
evidence
cycles
token
info
backup
restore
~~~

It also covers every nested leaf, including the complete current `deploy`
family:

~~~text
deploy authority
deploy external
deploy promote
deploy root
deploy plan
deploy install
deploy register
deploy check
deploy inspect
~~~

The following overlaps require an explicit decision rather than an automatic
`keep`:

- top-level `install` versus `deploy install`;
- `status`, `state`, `medic`, `info` and `inspect`;
- `deploy inspect` and its catalog, root and recovery projections;
- `deploy register` versus the canonical-network Fleet catalog and future
  Coordinator membership authority;
- `deploy root` after Fleet-wide coordination moves to the Coordinator;
- `deploy authority`, whose current controller-reconciliation meaning can be
  confused with the Authority Subnet and Fleet authority;
- `deploy promote`, whose artifact-promotion meaning can be confused with
  replica promotion, which remains excluded; and
- `deploy external` and `evidence`, where passive reports must have one current
  operator consumer rather than surviving as historical audit machinery.

The review must inspect actual dispatch, implementation and consumers. Command
help or tests that exercise only the command itself do not prove that the
command is needed. Conversely, apparent naming overlap is not enough to delete
a command whose behavior and operator consumer remain distinct.

0.100 and 0.101 should reuse the smallest surviving command family. They must
not add a new top-level command merely because the internal authority owner
changed. Any genuinely new operator mutation requires its owning design to
freeze authorization, durable recovery, output and interruption behavior
before adding the command.

CLI closeout is complete only when:

1. every predecessor command and nested leaf has one reviewed disposition;
2. every `keep` names a current owner, consumer and non-overlapping purpose;
3. every rename or move has one surviving spelling and no alias;
4. every deletion removes the whole command path and its active docs;
5. top-level help, leaf help, dispatch and implementation expose the same
   inventory;
6. rejected old forms fail through the ordinary unknown-command boundary,
   without anti-resurrection handlers or tests; and
7. targeted CLI help, dispatch and packaged-binary proofs exercise only the
   surviving current surface.

0.99 is not CLI-complete merely because the old source-oriented `canic fleet`
family has become `canic app`. It is CLI-complete only after this full command
disposition proves that obsolete operator machinery has also been hard-cut.
