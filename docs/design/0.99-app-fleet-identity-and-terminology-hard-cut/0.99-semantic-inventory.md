# Canic 0.99 Pre-Implementation Semantic Inventory

Date: 2026-07-23

## Status and Gate

- Status: terminology and surface inventory complete; implementation is
  approved under a reinstall-only release boundary.
- Inventory baseline: immutable `v0.98.24` at
  `9bc0f016dadf99fb88c3f2fa28855eb666f5660c`.
- Inventory method: direct source inspection, active endpoint/config/storage
  tracing and repository-wide stale-term searches.
- Product changes in this preparation pass: none.
- Structural-upgrade verdict: the required v0.98.24 terminal-state evidence is
  unavailable, so the maintainer selected fresh install/reinstall only.
  No 0.98 host-state or stable-state migration is implementation scope.

The inventory is authoritative for semantic destinations. The
[0.99 design](0.99-design.md) remains authoritative for behavior, recovery,
evidence encoding and acceptance criteria. A later implementation discovery
that contradicts this inventory requires a design correction before code is
changed; it is not permission to add a compatibility path.

## Network and Release Identity

Environment profiles are routing/configuration inputs. Each resolves one
`CanonicalNetworkId`:

- the compiled pinned public-IC trust root resolves the public IC;
- a PocketIC or other harness created and owned by Canic resolves from its
  pre-created journalled seed plus root-key digest; and
- a pre-existing local replica or other live network requires explicit
  enrollment of its root trust anchor.

Two profiles such as staging and production may resolve to the same public IC
and therefore share one canonical network/Fleet namespace. Profile labels,
gateway URLs and `BuildNetwork::{Local, Ic}` are never identity inputs. A
pre-creation instance seed cannot be invented for a replica Canic did not
create.

Release evidence has two non-interchangeable identities:

| Identity | Time and owner | Inputs |
| --- | --- | --- |
| `ReleaseBuildNonce` | host generates and journals before Wasm build | 32 cryptographically random bytes |
| `ReleaseBuildId` | host derives before Wasm build; exact value embedded in each Wasm | `SHA-256("canic:release-build-id\0" || u64_be(32) || ReleaseBuildNonce)` |
| `release_set_manifest_digest` | host computes after final Wasms exist | canonical manifest containing the exact final Wasm hashes |

The canonical pre-/post-build owner is:

~~~rust
pub enum ReleaseBuildPlanState {
    Planned,
    Finalized {
        release_set_manifest_digest: [u8; 32],
    },
}

pub struct ReleaseBuildPlanRecord {
    pub nonce: ReleaseBuildNonce,
    pub release_build_id: ReleaseBuildId,
    pub state: ReleaseBuildPlanState,
}
~~~

It is durably created at
`.canic/release-builds/<release-build-id>/plan.cbor` in `Planned` before
building, advances once to `Finalized` after exact Wasm hashes exist, and is
then immutable. Fresh-install recovery starts only from the finalized record
and stores its canonical hash, build ID and manifest digest.
There is no placeholder digest or second active nonce owner.

`ReleaseBuildId` excludes Wasm bytes, Wasm hashes, manifest bytes and the
manifest digest. There is no source/toolchain-derived alternative. A retry
reuses the journalled nonce and ID; a new build plan creates a new nonce. A
Canister compares only its embedded `ReleaseBuildId` with the install/upgrade
argument. The host validates the complete post-build release set. The manifest
digest is never embedded in a Wasm whose hash feeds that manifest.

## Terminology Dispositions

| Active meaning on the baseline | 0.99 disposition |
| --- | --- |
| config/source “Fleet” | App |
| one live installed “deployment target” | Fleet |
| deployment plan, action, comparison, promotion or receipt | retain Deployment |
| root-cascaded Fleet-wide `AppIndex` | `FleetDirectory` |
| root-cascaded local `SubnetIndex` | `SubnetDirectory` |
| complete root-owned local membership | retain `SubnetRegistry` |
| keyed-placement `DirectoryRegistry` | `PlacementBindingRegistry` |
| internal B-tree/hash lookup index | retain ordinary implementation “index” |
| Cargo/ICP repository root “project” | retain ordinary workspace “project” |
| application-domain role names containing “project” | retain; not Canic topology |
| immutable-object-storage `ProjectCycles` protocol names | retain exact external protocol contract |
| auth audience `Project`, `CanicSubnet` or `Canister` | remove; sole Fleet audience |
| `ApplicationReceipt` and application-owned intent terminology | retain; not App identity |

The hard cut is semantic, not a blind word replacement. In particular,
`DeploymentPlanV1`, `DeploymentReceiptV1`, `DeploymentCheckV1`, promotion and
comparison types remain because they describe deployment operations or
evidence. Fields inside them that identify a long-lived target change to Fleet
identity.

- fresh root `init` accepts only `CurrentRootInstallIdentity`;
- fresh managed non-root `init` accepts `CanisterInitPayload` followed by the
  existing optional application bytes;
- managed root and non-root 0.98-to-0.99 `post_upgrade` is unsupported; and
- `start_local!` remains a local-only standalone lifecycle and cannot qualify
  an installed managed Fleet Canister.

The fresh lifecycle records normalize into the one internal
`FleetActivationIdentity` by mapping `install_id` to `operation_id`. The
protected state stores that identity inside `Prepared`/`Active`; it does not
store parallel install and upgrade states.

The activation bounds and durable ownership are frozen:

| Object | Exact bound and owner |
| --- | --- |
| activation Canisters/cascade entries | 4,096 entries; ID 21 on the root, mirrored in the sole host activation record |
| canonical cascade manifest | 1,048,576 bytes |
| credential manifest | 64 entries and 262,144 bytes; ID 21 on the root |
| credential bundle | 65,536 canonical bytes; existing auth allocation on the root |
| proof-install raw Candid payload | 131,072 bytes before decoding |
| retained bundle-generation set | 4,194,304 bytes; existing auth allocation |
| retained prepared credential generations | current plus immediately previous, exactly 2 maximum |
| complete ID-21 activation record | 2,097,152 canonical bytes |
| complete host activation record | 2,097,152 canonical bytes |

Credential generation starts at `1` and advances only by checked `n + 1` for
the same activation identity. Every issuer and non-issuer must record the same
`FleetCredentialGenerationRef`; issuers additionally install their exact
bundle. The first activation call is forbidden until that convergence is
durable. `Active` freezes the generation. If a previously verified proof
expires after activation calls begin, it remains non-authorizing; exact
activation resumes from stored evidence and ordinary post-`Active` renewal
restores availability. This is the sole bounded expiry recovery path, not a
second activation or credential authority.

### Directory shapes

`DirectoryProvenance { fleet: FleetBinding, source_root: Principal }`,
`FleetDirectoryInput` and `SubnetDirectoryInput` are the only cascade
envelopes. Each input contains `provenance` and bounded `entries`.
`FleetDirectoryPageResponse` and `SubnetDirectoryPageResponse` each contain
`provenance`, bounded `entries` and `total`. `DirectoryEntryInput` and
`DirectoryEntryResponse` contain only `role` and `pid`.

There is no provenance-free parallel DTO. The fixed bounds are 16,384 encoded
bytes, 1,000 cascade entries and 1,000 page entries.

### Reinstall-only decision evidence

The current controller-only `canic_app(SetStatus(Readonly))` is usable during
the upgrade and its cascade blocks ordinary update dispatch. It is not by
itself a complete structural fence:

- `canic_response_capability_v1` is an internal endpoint and therefore
  deliberately bypasses the App-mode guard;
- its root capability set includes `AllocatePlacementChild` and
  `ProvisionCanister`, both of which can create and register managed
  Canisters; and
- already accepted root replay/intent work may resume after the App-status
  cascade.

The qualified 0.98 fence therefore has five required parts:

1. a durable host fence blocks every host command that can provision, attach,
   detach, place, recycle, upgrade, register, change topology or mutate
   installed authority;
2. after `Readonly` has cascaded, the host drives the current root-replay,
   placement-intent/receipt, pool import/reset, bootstrap/publication and host
   provisioning owners to terminal outcomes;
3. it captures a candidate local Registry, journals and stops every registered
   non-root, and reconciles every stop;
4. with all registered non-root capability callers stopped, it proves the
   established root has no pending bootstrap/publication/pool/placement owner
   and no self-originated structural timer or raw endpoint;
5. it recaptures `canic_subnet_registry` and requires exact equality with the
   candidate Registry and host inventory before freezing the journal vector.

The current root stays running and `Readonly` only for the unchanged Registry
and recovery queries until `HostTargetPrepared`; it is stopped in
`FleetQuiescing`. A missing pending-owner diagnostic, an unqualified root
self-producer, a Registry mismatch or an unknown creation result makes that
0.98 deployment ineligible. This is why `FleetStructureFencing` may already
have stopped non-roots and why abandoning it must start/prove them before the
old Fleet can become operational again.

Direct baseline inspection produces this exact capability map:

| Required proof | v0.98.24 maintained evidence | Qualification result |
| --- | --- | --- |
| ordinary updates fenced | controller `canic_app(SetStatus(Readonly))` plus completed state cascade | available |
| complete local membership | public `canic_subnet_registry()` captured twice | available |
| pool reset/import terminal | public `canic_pool_list()` plus `canic_runtime_status().timers` | available only when every pool row is terminal and the pool timer reports no pending work |
| bootstrap/publication terminal | `canic_bootstrap_status()`, controller `canic_wasm_store_bootstrap_debug()` and `canic_wasm_store_overview()` | available only for roles exporting those baseline surfaces |
| scaling/sharding bindings stable | controller `canic_scaling_registry()` and `canic_sharding_registry()` where enabled | available as current projections |
| management effects quiescent | stop every registered non-root and prove `stopped` through management `canister_status` | available for accepted calls owned by those Canisters |
| exact root replay ledger terminal | no v0.98.24 endpoint exposes every replay receipt, status, actor and operation identity | unavailable |
| exact receipt-backed placement/import intent set terminal | `canic_runtime_status().receipt_capacity` is aggregate and does not identify every Canic-owned record or its state | unavailable |
| absence of a durable child retry after restart | heap timer status is diagnostic only and does not project every stable retry owner | unavailable |

Metrics, elapsed quiet time, two equal Registry reads and a zero recent
work-count are not substitutes for the three unavailable durable-state
proofs. They cannot distinguish absence from an unobserved pending or
recovery-required record.

Therefore an arbitrary installed `v0.98.24` Fleet does **not** qualify for a
coordinated in-place 0.99 upgrade. The maintainer selected fresh
install/reinstall only. An operator assertion, force flag, journal edit,
inference from inactivity, compatibility decoder or new endpoint on an old
Wasm does not reopen that decision.

All corresponding positional value names, help, errors, output headings and

Live commands retain their top-level operation families:

- `install`, `inspect`, `info`, `medic`, `auth`, `blob-storage`, `backup` and

- their positional/option label becomes `fleet`;
- `--environment` remains connectivity/provenance input and resolves a

- `deploy plan/install/check/inspect/register/promote/root/authority/external`
  remain deployment-operation commands.

Fresh installation is exactly:

~~~text
canic install <fleet> --app <app>
~~~

Both values are mandatory. The command never infers the App from the Fleet
name and never requires the two names to match. Plan-driven installation is:

~~~text
canic deploy plan <fleet> --app <app>
canic deploy install <fleet> --plan <file>
~~~

The plan owns the App on the second command, and a positional Fleet mismatch
is rejected before mutation.

The new network-enrollment command is:

~~~text
canic network enroll <environment>
  --root-key <der-file>
  --fingerprint <64-lowercase-hex>
~~~

There is no automatic trust-on-first-use or alias for another network command.

`canic status` renders independently discovered App and Fleet sections:

~~~text
APPS:   APP | CONFIG | CANISTERS
FLEETS: FLEET | APP | NETWORK | DEPLOYED | ROOT
~~~

It never joins an App config to a Fleet because their display names happen to
be equal.

The deployment planning form is
`canic deploy plan <fleet> --app <app>`. There is no `--fleet-template`,
`--deployment-name`, old `canic fleet` source command or command alias.
