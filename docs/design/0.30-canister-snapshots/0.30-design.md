# Canic Fleet Backup & Restore v1

## Status

Draft v1

This design depends on `0.27` topology taxonomy terminology and should be read
after `0.27`.

## Context

The Internet Computer snapshot model is canister-local, but fleet recovery in Canic is a topology problem, not a single-canister problem. A safe product design must account for:

* snapshot create/download/upload/load being separate operations
* upload creating a new snapshot ID by default rather than replacing in place
* snapshot origin/type compatibility not being enforced at upload/load time
* post-load canister status being insufficient to prove semantic correctness
* the practical need to recover a whole Canic-managed fleet, not just one canister

Within Canic, existing code already uses topology- and fleet-like concepts rather than purely isolated canister operations. Public interfaces include app/subnet directory and environment snapshot methods, which are useful for discovering fleet membership and recording restore metadata.

Canic testkit also already contains controller-aware multi-canister snapshot capture and restore helpers. That is a strong signal that the right abstraction is a snapshot set managed under one controller across a collection of canisters.

Separately, the control-plane code already uses a `PublicationStoreFleet` abstraction with placement logic, reserved states, and fail-closed lifecycle checks. That pattern is relevant for a fleet backup design because it shows the codebase already values explicit fleet state modeling, reserved slots, and conservative mutation semantics.

## Problem Statement

We need a v1 fleet backup and restoration system in Canic that:

1. Discovers all canisters that belong to a target fleet.
2. Captures a restorable snapshot for each canister in that fleet.
3. Persists sufficient metadata to restore the fleet later in a deterministic, auditable way.
4. Restores a fleet safely, in the right order, with explicit verification gates.
5. Prevents operators from treating transport-level success as application-level success.

## Goals

### Primary goals

* Provide one operator-facing command/workflow to back up a whole Canic fleet.
* Provide one operator-facing command/workflow to restore a whole Canic fleet.
* Treat backup and restore as first-class fleet operations with a manifest and audit trail.
* Support restoring into the same fleet or a different compatible target fleet.
* Make verification mandatory and explicit.
* Make backup consistency scope explicit for the whole fleet or for smaller coordinated subsets.

### Non-goals for v1

* Incremental or deduplicated snapshot storage.
* Automatic schema/data migrations across incompatible app versions.
* Cross-network disaster recovery automation.
* Continuous background backups.
* Full policy engine for every canister type.
* Automatic rollback-on-verify-failure after a partial restore.

## Design Principles

* **Fleet-first, not canister-first**: operator actions should target a fleet ID / root / environment.
* **Manifest-driven**: every backup produces a canonical fleet backup manifest.
* **Fail closed**: refuse restore when required metadata, role mapping, or verification policy is missing.
* **Semantic verification required**: healthy status is not enough.
* **Ordered restore**: restore topology and dependencies in a deterministic sequence.
* **Artifact portability**: downloaded snapshots are first-class artifacts and may be re-uploaded later.
* **Compatibility is declared, not assumed**: because DFX does not enforce origin/type compatibility, Canic must add its own checks.
* **Execution state is separate from intent**: immutable backup intent belongs in the manifest; mutable download progress belongs in a journal/receipt log.

## User Stories

### Backup

* As an operator, I want to back up every canister in a fleet with one command.
* As an operator, I want the system to record the exact canisters, roles, IDs, controllers, snapshot IDs, and verification policy used.
* As an operator, I want a downloaded copy of a known-good fleet backup.

### Restore

* As an operator, I want to restore a whole fleet from one manifest.
* As an operator, I want to restore into either the original canisters or a different target fleet.
* As an operator, I want Canic to verify app methods after restore before declaring success.

### Audit / Ops

* As an operator, I want to list backups, inspect their manifest, and see the status of restore attempts.
* As an operator, I want partial failure states to be explicit and resumable.

## Proposed v1 Scope

v1 introduces a **Fleet Backup Manager** in Canic, exposed initially through a CLI-oriented orchestration layer and backed by a manifest format plus off-chain artifact storage.

The v1 system has four core pieces:

1. **Fleet discovery**
2. **Backup execution**
3. **Restore execution**
4. **Verification execution**

## High-Level Architecture

```text
Operator / CI

```

## Existing Canic Signals We Should Reuse

### 1) Fleet discovery via Canic index/env interfaces

The current interface surface includes:

* `canic_app_directory` (current compatibility name for `app_index`)
* `canic_subnet_directory` (current compatibility name for `subnet_index`)
* `canic_env`
* `canic_sync_state`
* `canic_sync_topology`

These suggest a natural source of truth for fleet membership and topology metadata. v1 should use these where available instead of asking operators to manually enumerate canisters.

### 2) Snapshot-set semantics already exist in testkit

`capture_controller_snapshots` and `restore_controller_snapshots` already model the right primitive: one controller, many canisters, one snapshot set. v1 should generalize that idea from testkit/PocketIC into live DFX orchestration.

### 3) Fail-closed fleet state management exists in publication workflows

`PublicationStoreFleet` uses explicit reserved-state logic and blocks lifecycle actions that would overwrite important state. v1 restore planning should copy that attitude: no silent overwrite, no implicit restore target selection, no restore into ambiguous topology.

### 4) Registry and cascade projection boundaries

The root subnet registry is the authoritative topology store, but it must not
become the universal wire shape for backup, restore, or topology cascade.

The v1 design should preserve these boundaries:

* root registry storage owns durable topology truth
* root query APIs expose operator/discovery projections
* topology cascade sends only the child-facing projection needed for runtime sync
* fleet backup manifests own backup-run provenance and restore contracts

Extending the subnet registry is acceptable for stable topology facts such as
role, parent, module hash, creation/provision metadata, placement family, or
identity policy when those facts belong to the standing fleet model. Snapshot
IDs, artifact checksums, live wasm hashes, restore mappings, journal state, and
verification receipts do not belong in the subnet registry; they belong in the
backup manifest or journal.

Cascade DTOs should remain explicit projections. They should not reuse the full
registry storage struct. If children need one additional topology fact, add that
field deliberately to the cascade projection and document why the receiving
canister needs it.

## Core Concepts

### Fleet

A logical collection of canisters managed together.

For v1, a fleet is identified by one of:

* root canister ID
* subnet/root environment handle
* explicit manifest import

### Fleet member

A canister in the fleet with metadata:

* canister ID
* logical role
* parent/root/subnet relations when known
* controller expectation
* identity mode
* verification class
* concrete verification checks
* restore ordering group

Identity mode is required for restore planning:

* `fixed`: the member is identity-sensitive and must restore into the same canister ID unless a future migration tool explicitly proves otherwise
* `relocatable`: the member may restore into a different compatible target when mapping, role, topology, controller, and verification checks pass

Root, registry, directory authority, and topology coordinator roles should default to
`fixed`. Fungible workers may be `relocatable` only when the role contract says their
principal identity is not part of application correctness.

Fleet member metadata is a backup/discovery projection, not the storage registry
record itself. It may be assembled from the root registry, config, env queries,
and live management status, but the manifest should persist the projection it
actually used rather than embedding a storage-owned `CanisterRecord`.

### Backup set

A consistent operator-triggered backup operation across all fleet members.

### Backup unit

The smallest set of fleet members that must be quiesced and snapshotted together to achieve the intended consistency guarantee.

Backup units may be:

* the whole fleet
* one control-plane subset
* one app subtree
* one explicitly declared coordinated group

Backup units are validated against topology before any snapshot is created:

* `subtree-rooted` units must be connected and closed under descendants unless an exclusion is explicitly declared
* exclusions must be recorded in the manifest with the reason and the consistency impact
* `flat` units must declare why the peers form a consistency boundary
* `flat` units should declare known dependency closure, or v1 must mark the unit as operator-declared and higher risk
* directory inputs may select a subtree root, but never define membership

### Backup artifact

One downloaded snapshot directory plus metadata for a specific canister snapshot.

### Fleet backup manifest

The authoritative JSON document that ties together all canister-level snapshot artifacts and the fleet topology/verification model.

### Download journal

A mutable execution log that records snapshot creation, download progress, checksum verification, and durable completion for each backup artifact.

The journal is the resume boundary for backup execution. It must record one
state per member artifact:

* `Created`: the live snapshot was created and the source snapshot ID is known
* `Downloaded`: snapshot bytes were downloaded into a temporary artifact path
* `ChecksumVerified`: local checksums were computed and matched the expected artifact contents
* `Durable`: the artifact was atomically moved into its final path and can be skipped on resume

Resume semantics:

* `Durable` artifacts are never downloaded again unless the operator explicitly asks for repair
* `ChecksumVerified` artifacts may be finalized again if the final move did not complete
* `Downloaded` artifacts may be checksum-verified again idempotently
* `Created` artifacts may be downloaded again idempotently
* partial temporary directories may be discarded or overwritten after journal validation

## Storage Model

v1 stores backup artifacts off-chain.

The default storage target is a filesystem directory managed by the `canic` CLI or by an operator-run automation environment. Object-store adapters may be added later, but the v1 backup payload does not live in a Canic-managed canister by default.

The manifest and journal are stored alongside the artifacts:

* `fleet-backup-manifest.json` is the immutable backup description
* `download-journal.json` records mutable execution state and receipts
* `audit.log` records operator-visible event history

This keeps disaster-recovery artifacts outside the fleet being protected and matches the DFX snapshot download/upload model.

## Consistency Model

v1 must declare the intended backup consistency scope before snapshot creation begins.

### Consistency scopes

* `crash-consistent`: snapshots are taken without quiescing the fleet; each canister snapshot is individually valid but cross-canister state may reflect an in-flight boundary
* `quiesced-unit`: a declared backup unit is paused or otherwise made mutation-stable before its members are snapshotted

v1 should support both modes, but it must record which mode was used in the manifest and must not imply stronger guarantees than were actually enforced.

### Topology snapshot stability

Fleet discovery is only reliable if the topology used for membership resolution
does not change between discovery and snapshot creation.

v1 must therefore compute and enforce a canonical topology hash:

1. Resolve authoritative topology records from the root/subnet registry.
2. Canonically serialize sorted records as `(pid, parent_pid, role, module_hash)`.
3. Hash the canonical serialization with SHA-256.
4. Record `discovery_topology_hash` in the journal and manifest draft.
5. Re-read topology immediately before the first snapshot create.
6. Compute `pre_snapshot_topology_hash`.
7. Fail closed unless the two hashes match.

The manifest must record the accepted `topology_hash` and the member list derived
from exactly that topology. If a future implementation also checks after snapshot
capture, it may record `post_snapshot_topology_hash`, but v1 must not treat a
changed post-capture hash as proof of a coordinated backup. It can only report the
backup as having observed topology drift after capture.

The canonical serialization must be stable across platforms and tool versions.
Records must be sorted by textual principal, then role, and all optional fields
must use explicit `null` markers rather than omitted fields.

### Backup units and partial-fleet backup

Some fleets will not need to stop everything at once. v1 should allow operators to declare one or more backup units that must be backed up in unison.

Examples:

* root + directory + registry
* one app coordinator + its worker leaves
* one publication/store subtree

The backup planner must fail closed if a requested subset omits a required consistency peer for the declared unit.

### Hierarchical backup units

Backup units are not always flat role lists. In fleets with nested ownership, a backup unit may be a subtree rooted at one canister.

For these topologies, v1 should support:

* `subtree-rooted` units: one root canister plus all recursively discovered descendants
* `flat` units: one explicit list of peers that must be coordinated together

Recursive membership should be resolved from authoritative topology data, not from ad hoc operator lists, wherever Canic exposes parent/child relationships.

### Toko-style topology example

For a Toko-like deployment shape:

* `root` owns fleet-global infrastructure canisters
* `project_hub` is root-owned
* `project_hub` creates many `project_instance` canisters
* each `project_instance` owns its own downstream canister subtree

The natural backup-unit model is:

* one fleet-core unit for `root` plus whichever root-owned canisters must stay consistent with root-managed topology state
* one subtree-rooted unit per `project_instance`

That means `project_instance_A` and all of its descendants are backed up together, while `project_instance_B` can be backed up in a different run unless the application declares a stronger cross-project consistency requirement.

This distinction matters operationally:

* managed topology systems such as sharding and scaling should use Canic-owned discovery and quiescence rules
* ad hoc app directories such as `project_hub` should not be treated as authoritative topology by default

`project_hub` is special: it is not automatically part of every project subtree unit. Instead, v1 should treat it as one of two cases:

* `project_hub` belongs to the fleet-core unit if it only coordinates provisioning and registry-level metadata
* `project_hub` must join a project subtree backup unit if project-level correctness depends on hub state mutating in lockstep with that instance subtree

Canic should not guess which case applies. The unit definition must declare it.

For the common "directory of instances" case, `project_hub` should be modeled as a metadata index, not as the owner of project backup membership. In that model:

* the project backup unit is rooted at one `project_instance`
* subtree membership comes from authoritative parent/child topology below that instance
* `project_hub` contributes selection hints and per-project directory metadata
* backup quiescence must prevent `project_hub` from mutating the targeted project's directory entry while the subtree backup is running

This allows one project subtree to be backed up safely without snapshotting the entire `project_hub` canister every time.

Strict rule:

> Directories may influence selection, never membership.

A directory may resolve a key such as `project_id` to a subtree root. After that,
authoritative topology defines every member of the backup unit. v1 must reject a
directory-selected backup if the resolved root is not present in authoritative
topology, or if the operator attempts to use directory listings as the recursive
membership source without an explicit unsafe escape hatch.

### Directory-bearing hubs

Some hubs are not topology owners. They hold an application-level directory that points at instance roots but do not define the recursive descendant set under each instance.

For those hubs, v1 should support a lighter contract:

* freeze or fence mutations for one directory entry or project key
* record the hub-side directory metadata for that project in the manifest
* snapshot the actual project subtree separately

This is different from sharding/scaling hubs, where Canic can often derive both membership and quiescence policy directly from managed placement state.

### Directory placement as a first-class topology mode

If an application directory is not merely advisory, it may deserve its own placement family.

One useful model is `directory` placement:

* one named pool of instance roots
* one logical directory key such as `project_id`
* at most one root instance assigned to each key
* the assigned root owns a recursive subtree of descendants

This is not the same as scaling:

* scaling manages interchangeable workers
* directory placement manages stable named instances

This is also not the same as sharding:

* sharding distributes many keys across a shared shard set
* directory placement binds one key to one dedicated instance root with cardinality `0..1`

Operationally, directory placement would give Canic a better contract for backup and restore:

* resolve one instance root from one directory key
* freeze mutations for that key while backup is in progress
* snapshot the resolved instance subtree
* record the hub-side directory entry as part of the manifest

If we go this direction, `project_hub` stops being a special-case app convention and becomes a managed placement authority for project instances.

### Discovery rules for subtree-rooted units

For a subtree-rooted unit, backup membership should be:

1. start from the declared subtree root such as one `project_instance`
2. recursively discover all descendants from authoritative topology state
3. apply exclusion rules only if the unit definition explicitly allows them
4. compute the discovery topology hash from the same records used for membership
5. fail closed if topology changes before snapshot capture

This is stricter than a role query and is the right model for per-project disaster recovery.

### Quiescence

For `quiesced-unit` backups, Canic should own an explicit quiescence phase before snapshot capture:

1. Resolve the backup unit membership.
2. Apply role-specific stop, pause, or drain actions.
3. Verify the unit reached a stable state.
4. Capture snapshots for all members in the unit.
5. Resume the unit only after capture completes or aborts safely.

v1 does not need a universal pause API for every canister type, but it must make
quiescence policy explicit. If a unit has no declared quiescence strategy, it may
only be backed up under an explicitly requested `crash-consistent` mode.
`quiesced-unit` must never silently downgrade to `crash-consistent`. If the
operator requested quiescence and the required strategy is missing or fails,
snapshot creation must not start for that unit.

v1 supports two acceptable quiescence mechanisms:

#### Standard Canic quiescence hooks

Canisters may expose a standard Canic-level contract:

* `canic_pause`
* `canic_resume`
* `canic_is_quiescent`

These hooks are optional for canisters generally, but required for any role that
claims the standard quiescence strategy.

#### Role-bound quiescence adapters

Roles may instead bind to an adapter that performs the same three phases:

* `quiesce(canister_id)`
* `verify(canister_id)`
* `resume(canister_id)`

Adapters may call role-specific methods, stop ingress through a coordinator, or
drain work queues. The manifest must record the strategy name, strategy version,
members covered, and the successful `verify` result before snapshots are
created.

## Backup Flow

### Step 1: Discover fleet

Input options:

* `--root <principal>`
* `--fleet-manifest <file>`
* `--canisters <explicit list>`

Preferred path in Canic-managed deployments:

1. Query root/subnet env methods.
2. Read app/subnet index pages through the current compatibility methods.
3. Build canonical fleet member list.
4. Assign each member a role and restore group.

### Step 2: Preflight validation

Before taking any live snapshots:

* verify operator control/access for every canister
* verify no duplicate roles where uniqueness is required
* verify target artifact directory is empty or versioned
* verify fleet member count is stable across discovery pass
* compute and record `discovery_topology_hash`
* optionally capture current canister versions / module hashes / env snapshots
* resolve requested backup units and consistency scope
* validate every backup unit against topology closure rules
* verify every `quiesced-unit` has a declared and supported quiescence strategy

### Step 3: Quiesce backup units when required

For each backup unit marked `quiesced-unit`:

1. apply the unit's quiescence strategy
2. verify the unit is mutation-stable
3. record quiescence success in the journal

### Step 4: Capture snapshots

Before the first snapshot create, v1 must re-read topology and verify that
`pre_snapshot_topology_hash == discovery_topology_hash`. If the hash differs,
the run stops before creating snapshots.

Snapshot provenance must be captured before or alongside snapshot creation:

* module hash recorded by Canic topology, when available
* live wasm/module hash from the IC management status path, when available
* canister version or application version endpoint, when available

The manifest must store these values per member so restore preflight can detect
mismatched targets and enforce compatibility policy.

For each canister:

1. `dfx canister snapshot create <canister>`
2. record returned snapshot ID and provenance values
3. record the artifact requirement in the manifest

### Step 5: Download artifacts and verify durability

For each created snapshot:

1. `dfx canister snapshot download <canister> <snapshot_id> --dir <DIR.tmp>`
2. compute local checksums for the downloaded directory
3. atomically move the artifact into its final directory
4. record `Downloaded`, `ChecksumVerified`, and `Durable` states in the download journal

If a backup run is resumed, the journal controls what work is repeated. Completed
`Durable` artifacts are skipped, while incomplete states resume from the earliest
idempotent step that can prove correctness.

### Step 6: Persist manifest

Write `fleet-backup-manifest.json` including:

* backup ID
* created_at
* source environment
* root canister / fleet identifier
* consistency scope
* declared backup units
* fleet topology snapshot
* list of member records
* verification policy references
* operator/tool version info

Write `download-journal.json` including:

* per-member snapshot creation receipts
* per-member download state transitions
* checksum verification receipts
* durable finalization receipts

Journal state transitions must be monotonic for each member artifact:

```text
Created -> Downloaded -> ChecksumVerified -> Durable
```

Retries may repeat the operation for the current state, but must not move a
member backward unless the operator explicitly runs a repair command that records
the reason in the audit log.

### Step 7: Mark backup complete

A backup is only marked complete if:

* every required canister snapshot was created
* every required artifact reached `Durable`
* the manifest is durable
* the journal is durable

## Restore Flow

### Step 1: Load manifest

Restore starts from a manifest, not from ad hoc snapshot directories.

### Step 2: Resolve target mapping

Two restore modes:

#### Mode A: In-place restore

Restore each snapshot back into the original canister ID.

#### Mode B: Mapped restore

Restore manifest entries into different canister IDs based on an explicit mapping file.

Example mapping:

```json
{
  "members": [
    { "source_canister": "aaaaa-aa", "target_canister": "bbbbb-bb" }
  ]
}
```

Because DFX allows cross-canister upload/load without semantic compatibility enforcement, v1 must require explicit role-compatible mapping for mapped restore mode.

Mapped restore must also enforce `identity_mode`:

* `fixed` members may only map to the original canister ID
* `relocatable` members may map to a different target when role, topology, controller, and verification checks all pass
* any override for a `fixed` member is outside v1 and must be treated as a migration, not a restore

### Step 3: Restore preflight

Before any load:

* verify every target canister exists
* verify mapping completeness
* verify mapping respects each member's `identity_mode`
* verify target roles match manifest expectations
* verify a known-good pre-restore snapshot is captured for every target canister
* verify no ambiguous duplicate targets
* verify restore plan ordering

### Dry-run restore

Restore must support a no-mutation dry run:

```bash
canic fleet restore --manifest <manifest> --dry-run
```

Dry run performs every preflight step that does not mutate the target fleet:

* load and validate the manifest
* validate mapping completeness and `identity_mode`
* verify target existence and expected roles
* validate provenance compatibility, including module/wasm hashes when required
* compute restore ordering groups and topological order
* validate the verification plan and report members without checks

Dry run must not upload snapshots, load snapshots, create safety snapshots, pause
canisters, resume canisters, or call mutating app methods.

### Step 4: Upload artifacts

For each member artifact:

* `dfx canister snapshot upload <target_canister> --dir <DIR>`
* record new uploaded snapshot ID

v1 should avoid `--replace` by default. New IDs provide cleaner auditability and safer rollback handling.

### Step 5: Load snapshots in order

Restore ordering groups:

1. infra/control-plane primitives
2. root/subnet coordinators
3. shared state / directories / registries
4. app workers / leaves

The exact grouping should be configurable per role class, but v1 can provide a default conservative ordering.

Within each restore group, v1 must topologically order members by `parent_pid`:
parents load before children when both are present in the same restore plan. If
cross-canister dependencies are declared in the manifest, the planner should
order those dependencies before dependents as long as that does not conflict with
the parent-before-child rule. Cycles or conflicting dependencies must fail
preflight rather than falling back to arbitrary ordering.

### Step 6: Run verification plan

After each ordered phase, run phase verification.
After all phases, run full verification.

### Step 7: Finalize restore result

Restore result is one of:

* `Succeeded`
* `SucceededWithWarnings`
* `VerificationFailed`
* `PartiallyRestored`
* `AbortedPreflight`

## Verification Model

This is the most important v1 safety feature.

### Why

A loaded snapshot may boot and respond to `dfx canister status` while still being application-invalid. Therefore v1 must define a semantic verification contract.

### Verification classes

Each fleet member gets a verification class:

#### `basic`

* canister responds
* `canic_ready == true` when applicable
* optional canister version query

#### `directory`

* responds to directory listing methods
* expected role appears in parent/root directory

#### `stateful-app`

* app-specific query/update smoke test
* expected domain object counts / invariants

#### `store`

* catalog/status methods work
* manifest/store invariants hold

The repo already shows `canic_ready`, environment methods, index-style lookup methods, and wasm store status/catalog methods that are natural building blocks for verification.

Verification is mandatory by default. Every fleet member must have at least one
concrete verification check, not only a class label. A restore run may skip
verification only with an explicit `--force-no-verify` flag. That flag must:

* require interactive acknowledgement unless running in a documented noninteractive mode
* mark the restore result as `SucceededWithWarnings` at best
* write an audit event that verification was intentionally bypassed
* print the list of unverified members

### Verification plan file

v1 should support a declarative verification plan embedded in the manifest:

```json
{
  "verification": {
    "fleet_checks": [
      { "kind": "root_ready" },
      { "kind": "subnet_index_contains_roles", "roles": ["root", "wasm_store", "app"] }
    ],
    "member_checks": [
      {
        "role": "wasm_store",
        "checks": [
          { "kind": "call", "method": "canic_wasm_store_status" },
          { "kind": "call", "method": "canic_wasm_store_catalog" }
        ]
      }
    ]
  }
}
```

## Manifest Format

The manifest persists backup projections, not internal storage records. The
`fleet.members` array is the restore contract produced by discovery; it should
not embed the full subnet registry type or any child cascade DTO directly.

```json
{
  "manifest_version": 1,
  "backup_id": "fbk_2026_04_10_001",
  "created_at": "2026-04-10T12:00:00Z",
  "tool": {
    "name": "canic",
    "version": "v1"
  },
  "source": {
    "environment": "prod",
    "root_canister": "aaaaa-aa"
  },
  "consistency": {
    "mode": "quiesced-unit",
    "backup_units": [
      {
        "unit_id": "control-plane-core",
        "kind": "flat",
        "roles": ["root", "directory", "wasm_store"],
        "consistency_reason": "root-managed topology and publication state must be captured together",
        "dependency_closure": ["root", "directory", "wasm_store"],

    "topology_hash_algorithm": "sha256",

      {
        "role": "root",
```

## CLI Proposal

### Backup

```bash
canic fleet backup --root <root_canister> --out backups/prod-2026-04-10/
```

### Restore in place

```bash
canic fleet restore --manifest backups/prod-2026-04-10/fleet-backup-manifest.json
```

### Restore with mapping

```bash
canic fleet restore \
  --manifest backups/prod-2026-04-10/fleet-backup-manifest.json \
```

## Restore Ordering Strategy

v1 needs deterministic order even if snapshot loads are individually independent.

### Default ordering heuristic

* Group 0: emergency pre-restore safety snapshots on targets
* Group 1: root and topology coordinators
* Group 2: registries/directories/shared state
* Group 3: infrastructure workers such as wasm stores
* Group 4: application leaves and workers

Rationale:

* topology owners should come back before dependents
* directory/state providers should be available before dependents are verified
* leaves should be restored after shared control-plane state is in place

The heuristic is only the first pass. The final restore plan must sort
topologically inside each group using the manifest's parent relationships.
Declared dependency edges may refine the order, but v1 must reject dependency
cycles or edges that make the plan ambiguous.

## Concurrency Model

Backup and restore have different concurrency rules.

### Backup concurrency

After topology hash validation and quiescence verification, snapshot creation may
run in parallel within a backup unit when the unit's quiescence strategy allows
it. Downloads and checksum verification may also run in parallel across members.

Concurrency must not cross these safety boundaries:

* do not begin snapshot creation before the accepted `pre_snapshot_topology_hash`
  is recorded
* do not create snapshots for a `quiesced-unit` before quiescence verification succeeds
* do not mark a backup complete until every required member reaches `Durable`
* preserve per-member journal ordering even when operations run in parallel

### Restore concurrency

Restore loads must respect restore groups and topological ordering:

* groups load sequentially
* parents load before children inside a group
* members with no dependency edge between them may load in parallel within the same group
* phase verification runs after each group completes
* final verification runs after all groups complete

Upload operations may run ahead in parallel after preflight, but loading an
uploaded snapshot must still obey the restore plan.

## Safety Controls

### Topology hash invariant

Backup execution must prove that the topology used for membership discovery is
the topology present at snapshot start. If `discovery_topology_hash` and
`pre_snapshot_topology_hash` differ, the backup fails before creating snapshots.

### Mandatory pre-restore snapshot

Before loading anything into a target fleet, capture a fresh known-good target snapshot set. This creates an operator escape hatch.

### No implicit consistency claims

If backup units were not quiesced, the resulting backup must be labeled `crash-consistent` and must not be presented as coordinated or transactionally aligned across canisters.

### No implicit cross-role mapped restore

Because DFX permits cross-canister upload/load, v1 must reject mapped restores unless each target canister is explicitly assigned a compatible role.

### No directory-defined membership

Directories can select a subtree root or provide metadata, but authoritative
topology defines backup membership. Any mode that uses directory listings as
membership must be an explicit unsafe escape hatch and must be labeled as such in
the manifest and audit log.

### Identity continuity

Restore planning must enforce `identity_mode`. `fixed` members restore in place
only; `relocatable` members may be mapped only with explicit compatibility and
verification checks.

### Fail closed on verification gaps

If a fleet member has no verification class or no applicable checks, restore
should stop unless `--force-no-verify` is explicitly set.

### Audit every snapshot ID transition

Record:

* source snapshot ID
* uploaded target snapshot ID
* load result
* verification result

## Error Handling

### Backup errors

* discovery failure
* snapshot create failure
* snapshot download failure
* checksum failure
* manifest write failure

### Restore errors

* invalid manifest
* incomplete mapping
* target preflight failure
* upload failure
* load failure
* phase verification failure
* final verification failure

### Partial restore handling

If restore fails mid-flight:

* stop further loads
* mark run as partial
* preserve audit log
* print exact canisters restored vs not restored
* recommend restoring from pre-restore safety snapshot set

## Compatibility Policy

Because raw snapshot load does not prove semantic compatibility, v1 should define compatibility in Canic terms:

### v1 compatibility inputs

* logical role match
* expected controller/root/subnet relations when known
* optional code/module hash match or allowlist
* successful role-specific verification

v1 should not attempt full structural state introspection before restore. That is better left for current.

## Security / Integrity

* checksum every downloaded artifact
* never mutate the original backup directory during restore
* prefer new uploaded snapshot IDs over `--replace`
* require explicit operator acknowledgement for destructive in-place restore in prod

## Alternatives Considered

### 1) Simple shell script over DFX only

Rejected for primary product shape because it does not capture topology, verification policy, or audit semantics.

### 2) Backup only root plus reconstruct children later

Rejected for v1 because it assumes reconstruction logic and loses strong rollback semantics.

### 3) Treat canister status as restore success

Rejected because transport-level liveness is not equivalent to app correctness.

## Concrete v1 Recommendation

Build this as a **manifest-driven CLI orchestration layer** first, not as a deeply embedded automatic control-plane workflow.

Why:

* the DFX snapshot operations are already external CLI primitives
* local downloaded artifacts are central to safe ops
* operator visibility and auditability are more important than full automation in v1
* it lets us validate fleet discovery, ordering, and verification policy before moving any of it into long-lived control-plane workflows

## Summary

The right v1 for Canic is:

* topology-aware
* manifest-driven
* conservative by default
* explicit about restore mappings
* strict about semantic verification

The central abstraction should be a **Fleet Backup Manifest** representing a controller-scoped snapshot set across all discovered fleet members, with ordered restore and mandatory post-load verification.
