# Canic 0.94 Design: Backup and Restore Operational Readiness

Date: 2026-07-19

## Status

- Status: accepted and active. Both review rounds are incorporated; the frozen
  inventory and early platform capability gate are complete.
- Release anchor: `v0.93.36` at
  `f9c28c48bdc72055d873e8291d201aac1c871f5e`.
- Implementation tracker: [status.md](status.md).
- 0.94 is an operational proof-and-hardening line. It is not another holistic
  audit, structural cleanup line, feature expansion, or 1.0-readiness claim.

## Decision

0.93 closes at `v0.93.36`. Its purpose was to hard-cut stale behavior,
consolidate authority, align naming, and make failure boundaries exact. 0.94
does not continue that open-ended cleanup.

0.94 proves one narrower claim: Canic can complete, interrupt, resume, verify,
and restore a real multi-canister backup without inventing progress, repeating
unsafe effects, accepting corrupt evidence, or reporting partial work as
success.

Known non-blocking structural residue deferred from 0.93: none. The
effect/receipt uncertainty, abrupt-death lock liveness, completed-operation
replay, corruption rejection, and realistic multi-canister recovery evidence
listed here are operational proof gaps that were outside the structural 0.93
finish line. They are explicit 0.94 inputs, not an undisclosed continuation of
the 0.93 cleanup audit.

The line is organized around named crash points and observable restart states,
not speculative implementation slices. Existing behavior is exercised first.
Code changes are admitted only when a required journey produces a reproducible
finding against this design.

The governing sequence is:

```text
freeze the 0.93.36 operational baseline
  -> inventory exact durable transitions and external effects
  -> probe snapshot observation/replay capabilities in a disposable environment
  -> confirm command-child lifetime and quiescence behavior
  -> amend the design if a core capability is unavailable
  -> freeze the executable crash-point protocol and recovery matrix
  -> run deterministic process-restart journeys
  -> run rejection and completed-operation replay journeys
  -> run one realistic multi-canister journey
  -> fix only journey-backed defects
  -> rerun the affected point and the complete matrix
  -> publish one bounded operational closeout
```

## Current Implementation Reality

The release anchor already contains the intended production owners:

- `canic-backup::plan` builds and validates the exact backup target and phase
  projection.
- `canic-backup::execution` owns backup operation state and terminal receipts.
- `canic-backup::journal` owns snapshot artifact progress through `Created`,
  `Downloaded`, `ChecksumVerified`, and `Durable`.
- `canic-backup::runner` persists a pending claim before invoking an external
  effect and persists the terminal receipt afterward.
- `canic-backup::persistence` validates documents and replaces them through a
  synchronized sibling-file write, rename, and parent-directory sync.
- `canic-backup::restore` owns restore planning, exact dry-run projection,
  apply journals, private no-follow artifact staging, commands, receipts, and
  runner recovery.
- `canic-cli` owns current command admission and operator projection for
  `backup create`, `backup verify`, `restore prepare`, `restore status`, and
  `restore run`.

Existing tests cover model transitions, injected executor failures, durable
JSON replacement, artifact publication, checksums, journal integrity, restore
staging, command receipts, and bounded resume cases. They provide a strong
base but do not establish the 0.94 claim because most run inside one process
and many use simulated command outcomes.

The remaining risk is concentrated at command/effect/receipt boundaries:

```text
durably mark pending
  -> spawn external command
  -> command may outlive the runner
  -> external effect may commit
  -> construct terminal receipt in memory
  -> durably persist terminal receipt and state
```

If the process dies after spawn, the command may still be running even though
Canic's lock owner is gone. Restart must establish command quiescence before it
observes target state or launches a competing command. If the process dies
after the effect but before the last write, durable state truthfully shows an
interrupted operation, but it does not by itself prove whether the effect
committed. Snapshot creation, snapshot upload, and snapshot load cannot be
treated as freely repeatable. Stop/start commands, local artifact publication,
and manifest publication also require their actual idempotency or
reconciliation rule to be proven rather than assumed.

The current sidecar journal lock is removed by `Drop`. An abrupt process death
does not run `Drop`, so lock liveness after a real crash is part of the required
matrix rather than assumed from ordinary unit tests.

The archived 0.34 through 0.36 backup/restore designs explain historical
intent, but they are not active authority and their old command examples do
not reintroduce removed surfaces. This design uses only the current 0.93.36
contracts.

## Central Invariant

After every interruption, durable state must describe exactly what Canic has
durably established about local progress and external outcomes. It must never
represent attempted or unknown work as committed. Resumption must continue
safely, reconcile from authoritative evidence, or fail closed with an explicit
recovery disposition.

This means:

- a durable `Completed` state requires its matching durable terminal receipt;
- a durable `Pending` state means the operation was claimed and its effect may
  be unknown, not that the effect definitely did not occur;
- restart never launches a competing command while the prior command may
  still be running;
- an unknown non-idempotent outcome is reconciled from authoritative evidence
  or safely halted with an explicit recovery disposition;
- a completed operation is never re-executed merely because the previous
  process lost its response;
- corrupt, stale, mismatched, missing, or partial evidence never becomes
  successful progress; and
- every failure identifies the affected backup, operation, durable boundary,
  and next safe operator action where one exists.

## Goals

0.94 must:

1. complete and verify a backup, then restore and verify its actual state;
2. interrupt backup on both sides of every authoritative durable transition
   and prove deterministic restart behavior;
3. interrupt restore on both sides of every claim and terminal-receipt
   transition and prove deterministic restart behavior;
4. interrupt read-only verification and prove a clean restart is safe;
5. repeat completed runners and operations and prove no completed external
   effect is repeated;
6. reject corrupt, missing, stale, mismatched, unsafe, and partially published
   evidence without discarding the last valid recoverable state;
7. distinguish a live journal owner from an orphaned lock without stealing
   active work or permanently blocking recovery;
8. prove command-child lifetime, process-group, lock-inheritance, quiescence,
   and timeout behavior after runner death;
9. run at least one multi-canister PocketIC or explicitly disposable ICP
   environment journey across actual filesystem persistence boundaries;
10. retain typed causes and deterministic operator diagnostics throughout the
   tested flows; and
11. close with a finite matrix, explicit limitations, and no implied 1.0
    readiness.

## Non-Goals

0.94 does not:

- perform another repository-wide naming, module, duplication, dependency, or
  CI cleanup;
- update dependencies merely because a newer version exists;
- add cloud, remote, scheduled, parallel, or daemon backup execution;
- add cross-project restore or generalized disaster-recovery orchestration;
- create a generic transaction, saga, receipt, effect, or failpoint framework;
- create a general process supervisor or daemon merely to test command death;
- promise atomic multi-canister restore when the platform exposes operations
  one canister at a time;
- preserve obsolete documents, commands, states, or errors through aliases,
  shims, fallback readers, deprecated APIs, or migration wrappers;
- introduce a new document version merely because the release number changed;
- change Candid, stable memory, JSON, CLI, or public Rust surfaces without a
  journey-backed necessity and explicit design amendment;
- use production/mainnet state, credentials, or destructive authority; or
- claim that operational test evidence replaces the maintainer's required
  real-world use period before 1.0.

## Safety Definitions

### Durable transition

A durable transition is complete only after the owning write returns from the
file sync, atomic sibling rename, and parent-directory sync sequence. An
in-memory mutation or successful serialization is not durable progress.

### External effect

An external effect is committed when the authoritative target observes it,
not merely when a child process was started or returned an exit code. Examples
include a canister entering `Stopped`, a snapshot appearing, state being loaded,
or an artifact directory becoming the canonical published directory.

Command invocation count and authoritative committed-effect count are
different measurements. Both are recorded. A command may be invoked once and
commit no effect, or remain in flight after its runner dies and commit later.

### Command quiescence

Command quiescence means the previous command and any relevant descendants are
conclusively finished or terminated before restart performs reconciliation or
issues another mutating command. Waiting for the old Canic process to disappear
is insufficient. If command lifetime cannot be observed, restart waits for the
operation's bounded timeout and then safely halts; it does not infer quiescence
from unchanged target state.

### Reconciliation

Reconciliation observes authoritative target or artifact state and binds that
evidence to the exact plan, target, operation, attempt, snapshot, or checksum.
It may convert an interrupted operation to terminal state only when the same
proof required for a normal terminal receipt is available.

### Safe replay

Replay is safe only when repeating the operation is proven to produce the same
authoritative result without duplicate resource creation, unrelated state
replacement, or loss of the original effect identity. Convenience or apparent
idempotency is not proof.

### Interruption

An interruption is abrupt process loss, such as `SIGKILL`, with no unwinding,
`Drop`, deferred cleanup, response rendering, or in-memory recovery. Ordinary
returned errors are tested separately and do not substitute for crash tests.

### Recovery disposition

Recovery records one of four dispositions:

```text
resumed | reconciled | operator_action_required | safely_halted
```

`operator_action_required` is an intermediate state. A recovery journey passes
only after the documented action is exercised and the operation subsequently
completes. `safely_halted` is a valid terminal expectation for rejection cases
such as corrupt evidence, but a permanently halted create, upload, or load
interruption does not demonstrate resumability and prevents 0.94 closeout.
These dispositions classify evidence and operator behavior; they do not by
themselves require a new public or persisted product enum.

An operator action may invoke a maintained Canic command or change
authoritative external state through a documented platform operation. It may
not edit journals, synthesize receipts, delete lock evidence, rename artifacts
into place, invent snapshot identities, or manually mark an operation
complete. A new recovery command or supplied-identity surface requires the
normal journey-backed design amendment. The runbook records the exact action,
its preconditions, the resulting evidence, and how Canic independently
validates that evidence.

### Durability scope

0.94 proves process-crash recovery on the release-supported local filesystem
and verifies that Canic uses file sync, atomic sibling rename, and
parent-directory sync at its durable boundaries. It does not empirically prove
remote-filesystem, hardware, kernel, or full-machine power-loss behavior.

### Offline verification

Offline verification reads only the persisted backup layout. It makes no live
target, registry, management API, or network call.

### Maintained-entrypoint containment

Multi-canister restore is not claimed to be platform-atomic. For 0.94,
partial restore is not exposed through the maintained parent or entrypoint:

- all artifacts are admitted before the first destructive load;
- all selected targets are stopped before the first load;
- all loads complete before the first start;
- children/dependents start before their parent or entrypoint, keeping the
  parent unavailable until its dependencies are restored;
- interruption never reports the restore as complete or verified; and
- restart evidence identifies exactly which targets may have changed.

Direct-principal access to a child may remain possible once that child starts.
0.94 does not claim to prevent that access because Canic has no general ingress
or maintenance-mode gate. The operator runbook must state this boundary. If
the maintained parent can expose a mixed fleet before all loads finish, that
is a blocking operational finding. The line may not claim fleet-wide atomicity
or absolute non-exposure.

## Recovery Evidence Protocol

Every crash-point case uses this protocol:

1. Start from a freshly generated, validated plan and durable layout.
2. Record hashes of every authoritative document and artifact plus observable
   target state.
3. Advance to the named barrier in a child process.
4. Have the child signal the exact barrier and wait for a parent
   acknowledgement.
5. Send `SIGKILL` only after that handshake; do not use timing or polling as
   the barrier.
6. For an after-write barrier, signal only after file sync, rename, and parent
   directory sync have all returned.
7. Start a new process against the same filesystem and target environment.
8. Establish command quiescence before target reconciliation or another
   mutating command.
9. Record the surviving lock, temporary files, durable documents, terminal
   receipts, target state, command invocation count, and committed-effect
   count.
10. Invoke the documented resume, retry, or reconciliation path.
11. Assert the final durable state, target state, receipt identity,
    diagnostic, recovery disposition, and both operation counts.
12. Run integrity and verification again after completion.

Every durable transition is tested from both sides:

```text
before:
  terminate immediately before the authoritative write
  -> the prior complete document remains authoritative

after:
  terminate after the authoritative write has fully synchronized
  -> the new complete document remains authoritative
```

External effects receive a third mandatory point:

```text
effect_committed_receipt_missing:
  authoritative target changed
  durable operation remains pending
  -> reconcile, prove safe replay, or fail closed
```

Every mutating external command also receives this mandatory point:

```text
effect_in_flight_owner_dead:
  runner is dead
  external command or descendant may still be alive
  -> establish quiescence before observation, reconciliation, or replay
  -> never issue a competing command while the old execution is unknown
```

The inventory records whether child commands survive parent death, their
process-group ownership, whether the journal-lock file descriptor is inherited,
and what observable evidence proves termination. An unobservable command uses
a bounded timeout followed by `operator_action_required` or `safely_halted`;
an unchanged target does not prove that the command is finished.

The deterministic harness may add only the smallest test-owned handshake
necessary at an existing executor or persistence boundary. It must not add an
environment-controlled production failpoint, general hook framework, hidden
CLI, process supervisor, or production branch that changes command behavior
under test.

Two complementary proofs remain distinct. The normal successful journey uses
the maintained CLI end to end. The real high-risk crash case uses a test-only
child executable around the existing runner/executor boundary so the parent
can perform the deterministic handshake, but that child invokes the real
`canic-host` effect against disposable canisters. It does not return a fake
command outcome or introduce a production control surface.

Counting before/after write sides separately, one committed-effect/receipt-loss

protocol cases. Repeating a case across targets is fixture coverage, not a new

instances from the same inventory.

This inventory is the finite 0.94 boundary. Adding an operation variant,
durable transition, external effect, or crash-case generator requires a design
amendment and an updated expected count before matrix execution continues.

## Early Platform Capability Gate

Before constructing the complete 106-case harness, one small disposable-
environment probe exercises real snapshot and command behavior. It must answer:

- can created and uploaded snapshot identities be authoritatively recovered;
- can load completion be observed while the target remains stopped;
- is repeating the exact load while stopped provably safe;
- can surviving child commands and descendants be conclusively identified and
  brought to quiescence; and
- can PocketIC exercise each required boundary, or is a disposable local ICP
  environment required?

The probe records the authoritative observations available for create, upload,
and load plus the actual child-process lifetime behavior. If any core
capability is unavailable, the design and effect-capability ledger are amended
before the executable matrix and deterministic harness are frozen. 0.94 does
not build the full harness around an assumed capability.

### Capability gate result

The 2026-07-19 gate is recorded in the
[capability report](../../audits/reports/2026-07/2026-07-19/0.94-capability-gate.md).
The repository-managed disposable local ICP environment is the selected real
snapshot boundary because it exercised the maintained ICP CLI create,
download, upload, inventory, and restore path directly. PocketIC remains the
preferred in-process fixture where it reaches the same boundary, but it does
not replace the required real CLI proof.

The gate established:

- created and uploaded snapshot identities are recoverable from target
  inventory under exclusive snapshot activity;
- an inventory delta does not by itself bind an uploaded identity to the
  source artifact checksum;
- restore of the exact snapshot may be repeated while the target remains
  stopped, and changed module state is observable through management status;
- application-state restoration still requires the contained start/query
  phase of the realistic journey; and
- the current `Command::output()` boundary permits a child to survive runner
  death and retains no restart-visible execution identity.

The last result is `CANIC-094-PROC-001`. Its fix remains deliberately bounded:
use the existing host command executor boundary and the release-supported
Linux process facilities to identify the exact execution and prove it and its
descendants quiescent. 0.94 does not add a daemon, general supervisor, lease
service, production failpoint, or hidden CLI. The exact protocol and
deterministic descendant test must be frozen before the complete B18/R14
in-flight harness is built.

### Frozen command-quiescence protocol

`CANIC-094-PROC-001` uses one deterministic command-lifetime lock per journal
operation sequence. This is separate from the journal lock and has a narrowly
scoped execution inheritance rule:

- the journal lock is close-on-exec and is owned only by the Canic runner;
- the command-lifetime lock is acquired before a pending mutating operation
  observes target state, stages command input, or invokes an effect;
- its descriptor remains close-on-exec in the runner, so an unrelated
  concurrently spawned command cannot retain another operation's ownership;
- child setup clears close-on-exec only for the intended external command, and
  that command's descendants inherit the same open-file description;
- the runner retains its descriptor while waiting for the command;
- after the direct child exits, the runner closes its descriptor and proves a
  fresh exclusive lock can be acquired before persisting a terminal receipt,
  allowing only a bounded 250 ms grace for a just-exiting command tree;
- if Canic dies, the inherited open-file description keeps the kernel lock
  held until the external command tree exits; and
- restart that cannot acquire the exact operation lock returns a typed
  command-in-flight cause while leaving `Pending` and its receipts unchanged.

The lock path is derived from the canonical journal path and operation
sequence; file contents, PID text, age, and path existence are not authority.
A no-follow regular-file check rejects substitution. A completed-operation
receipt is impossible until the post-command quiescence probe passes, so a
later operation never overlaps a still-live predecessor through the maintained
runner.

This protocol adds no daemon, timeout service, process scanner, persisted
journal field, command option, or compatibility path. The maintained ICP CLI
is treated as a trusted command child that preserves the explicitly supplied
descriptor without receiving unrelated close-on-exec descriptors. The
deterministic direct-child/descendant test and the required real high-risk
journey must prove that contract. Failure to preserve it is a blocking 0.94
finding, not permission to infer quiescence.

A command-lifetime lock establishes only whether the prior command tree is
quiescent. It does not prove whether a mutating effect committed. A mutating
operation that remains `Pending` after quiescence therefore returns a typed
unknown-outcome cause without executing, changing state, or creating a
receipt. Read-only verification may be repeated. The former restore
pending-to-ready reset path is hard-cut because it erased this distinction and
permitted blind replay; no alias, fallback, synthetic receipt, or manual
journal edit replaces it.

## Backup Crash-Point Matrix

Backup recovery coordinates three authorities: the execution journal for
ordered operations and terminal receipts, the artifact journal for
snapshot/artifact progress, and the final manifest plus published artifacts.
All three must agree.

The execution journal owns operation completion. The artifact journal owns
artifact-stage progress. Published artifacts and the manifest are evidence,
not permission to invent a stronger journal state.

### Representative cross-authority recovery rules

Important disagreement states and their recovery direction are:

| Execution state | Artifact state | Manifest/artifact evidence | Recovery |
| --- | --- | --- | --- |
| `Pending` stop/start | not applicable | manifest absent | Establish command quiescence, observe exact target status, then reconcile or issue one justified command. |
| `Pending` create | exact `Created` | manifest absent | Reconstruct completion only if the artifact row contains every identity and proof required by the normal create receipt; otherwise reconcile externally. |
| `Pending` create | absent | manifest absent | Treat effect as unknown after command quiescence; reconcile snapshot identity or require operator action. |
| `Pending` download | exact `Created`, `Downloaded`, or `ChecksumVerified` | manifest absent | Resume only the canonical next artifact step; never skip checksum or publication. |
| `Pending` download/verify | exact `Durable` | exact published artifact, manifest absent | Reconcile through the canonical journal/receipt path; durable bytes alone do not mark the execution operation complete. |
| `Pending` finalize | all exact `Durable` | manifest absent or exact | Publish or adopt only the exact canonical manifest, then record the normal terminal receipt. |
| `Completed` | missing or mismatched required evidence | any | Reject the stronger completion claim; never normalize weaker evidence to match it. |
| incomplete execution other than `Pending` finalize | any | manifest present | Reject the manifest as premature or conflicting; it cannot complete execution. |
| all required operations `Completed` | all exact `Durable` | exact manifest and artifacts | Return completed with zero effects. |

Any disagreement involving `Pending`, `Completed`, published artifacts, or a
manifest must follow one of these rules or reject. Ordinary `Ready`, retryable
`Failed`, and correctly ordered intermediate combinations remain governed by
the execution and artifact state-machine validators. Cross-document recovery
always moves from independently verifiable evidence into the owning state
machine; weaker evidence never manufactures stronger evidence.

| Point ID | Boundary | Required post-restart result |
| --- | --- | --- |
| `CANIC-094-B01` | Initial execution-journal publication | Either no journal exists and a new exact journal is created, or the complete validated journal exists; partial JSON is never accepted. |
| `CANIC-094-B02` | Preflight-applied plan publication | A preflight-updated plan with an unaccepted execution journal causes safe revalidation only; no mutation starts. |
| `CANIC-094-B03` | Execution-journal preflight acceptance | Accepted state and preflight identity survive together; the first mutation cannot become ready before that write. |
| `CANIC-094-B04` | Pending claim for each post-preflight operation variant | Before the claim, no effect occurred. After the claim, restart treats the effect as unknown until its operation policy is applied. |
| `CANIC-094-B05` | Stop effect | Restart observes the target state before deciding whether another stop command is necessary. |
| `CANIC-094-B06` | Snapshot-create effect | A created snapshot is bound to the exact target and recovered without creating another snapshot. Operator action may be required, but the journey must subsequently complete. |
| `CANIC-094-B07` | `Created` artifact-journal entry | The recorded snapshot identity rebuilds the missing execution receipt only if it contains the complete normal receipt proof; otherwise external reconciliation is required. |
| `CANIC-094-B08` | Start effect | Restart observes the target state and does not turn a completed start into a false failure or unnecessary repeated mutation. |
| `CANIC-094-B09` | Snapshot download into the private temporary path | Partial or uncommitted temporary bytes are discarded or replaced; they never become a durable artifact. |
| `CANIC-094-B10` | `Downloaded` artifact-journal entry | Restart proceeds to checksum verification using the exact expected temporary path and snapshot identity. |
| `CANIC-094-B11` | Checksum calculation | A lost in-memory checksum is safely recomputed without changing authoritative bytes. |
| `CANIC-094-B12` | `ChecksumVerified` artifact-journal entry | Restart publishes only the checksum-bound artifact recorded by the journal. |
| `CANIC-094-B13` | Canonical artifact-directory publication | An already published exact directory is reconciled by checksum; a conflicting destination fails closed and is never overwritten. |
| `CANIC-094-B14` | `Durable` artifact-journal entry | The durable artifact and journal checksum agree; restart does not redownload or republish it. |
| `CANIC-094-B15` | Manifest publication | A manifest is accepted only when it matches the exact plan, journal, and durable artifacts; an existing conflicting manifest is not normalized. |
| `CANIC-094-B16` | Terminal execution receipt and state | Receipt and terminal state appear in the same durable document. Restart skips that operation and appends no duplicate terminal receipt. |
| `CANIC-094-B17` | Final successful response | Loss of the response after complete persistence causes the next run to report the same completed backup with zero external effects. |
| `CANIC-094-B18` | In-flight mutating command after runner death | For `Stop`, `CreateSnapshot`, `Start`, and `DownloadSnapshot`, restart proves the old command is quiescent before observing or invoking the operation again. |

`CANIC-094-B04` and `CANIC-094-B16` apply to each of the six post-preflight
backup operation variants. Preflight acceptance is covered atomically by
`CANIC-094-B03`. The effect-specific points between them prove the additional
authority needed for that operation.

Preflight, stop, snapshot, start, download, checksum, publication, and
finalization failures follow the explicit containment policy below. A failed
or interrupted backup must never silently strand a target because a secondary
cleanup failure replaced the primary cause.

## Verification Interruption Matrix

`backup verify` is read-only with respect to the backup layout. 0.94 does not
add a verification journal merely to remember scan progress. Restarting the
complete verification is acceptable when it is deterministic and bounded.

| Point ID | Boundary | Required post-restart result |
| --- | --- | --- |
| `CANIC-094-V01` | Before document validation | Restart reads the same immutable inputs and produces the same result. |
| `CANIC-094-V02` | During artifact traversal/checksum | No partial pass marker or mutated backup artifact is left; restart recomputes the complete result. |
| `CANIC-094-V03` | After verification result but before output | Loss of output cannot create durable success. A repeated command revalidates all evidence. |

The before/after filesystem inventory must prove verification did not mutate
plans, journals, manifests, receipts, or artifacts. Temporary caches outside
the backup authority are permitted only when they cannot be mistaken for
verification evidence.

## Restore Crash-Point Matrix

Restore operations run in canonical phases: upload every snapshot, stop every
target, load every snapshot, start in reverse topology order, verify members,
then verify the deployment. That ordering is part of the containment proof.

| Point ID | Boundary | Required post-restart result |
| --- | --- | --- |
| `CANIC-094-R01` | Restore-plan publication | The prior complete plan or the new complete validated plan survives; a partial plan cannot produce an apply journal. |
| `CANIC-094-R02` | Initial apply-journal publication | The journal exactly projects the validated dry-run and artifacts before any command can execute. |
| `CANIC-094-R03` | Private upload staging | A killed process may leave only non-authoritative private staging. Restart validates ownership/type, removes or replaces stale staging, and never treats it as a receipt. |
| `CANIC-094-R04` | Pending claim for each restore operation variant | Before the claim, no command ran. After it, restart treats the effect as unknown and applies the operation's reconciliation policy. |
| `CANIC-094-R05` | Stopped-canister precondition | A read or parse failure is a durable typed failure, not permission to load. A confirmed stopped state is bound to the same target attempt. |
| `CANIC-094-R06` | Snapshot-upload effect | The returned uploaded snapshot identity is recovered and bound to the exact source artifact, target, and checksum without duplicate upload. Any required operator action is exercised and the journey completes. |
| `CANIC-094-R07` | Stop effect | Restart observes the target state before any repeated stop and retains the restore's incomplete status. |
| `CANIC-094-R08` | Snapshot-load effect | Restart proves whether the expected snapshot state was loaded before replaying. Any required operator action is exercised and the journey completes; permanent ambiguity fails closeout. |
| `CANIC-094-R09` | Start effect | Restart observes the target state and continues in reverse topology order without starting a parent early. |
| `CANIC-094-R10` | Member verification effect | Verification is safe to repeat and cannot turn a command/parse failure into completion. |
| `CANIC-094-R11` | Deployment verification effect | Completion remains false until every required deployment check has a matching terminal receipt. |
| `CANIC-094-R12` | Terminal receipt and operation state | Receipt attempt, command, target, output, artifact identity, uploaded snapshot ID, timestamp, outcome, and terminal state persist atomically in one journal replacement. |
| `CANIC-094-R13` | Final successful response | Loss of the response after the last receipt causes a subsequent run to return complete with zero mutating commands. |
| `CANIC-094-R14` | In-flight mutating command after runner death | For upload, stop, load, and start, restart proves the old command is quiescent before observing or invoking the operation again. |

`CANIC-094-R04` and `CANIC-094-R12` apply to every restore operation. A
generic `pending -> ready` transition is not sufficient proof that a
non-idempotent effect did not commit.

## Failure Containment And Recovery

Backup and restore have different safe-abort boundaries.

For backup:

- before any successful stop, failure records no cleanup obligation;
- after at least one stop, command quiescence is established before cleanup;
- if snapshot reconciliation or later work cannot continue, the runner may
  execute the already-planned starts for targets proven stopped without
  claiming that the blocked backup operation progressed;
- each cleanup start is durably recorded against its canonical planned
  operation, while the backup remains incomplete and preserves the primary
  failure plus any secondary cleanup failure; and
- the journey does not pass until all selected targets are either proven in
  their intended running state or identified by an exercised, successful
  operator recovery action.

For restore, the first successfully committed load is the point of no safe
automatic abort:

- before the first load commits, no target state was destructively replaced;
  after command quiescence, targets stopped by this attempt may be restarted
  through their canonical operations and the restore may abort incomplete;
- after the first load commits, the selected fleet may contain mixed state;
  automatic restart or rollback is forbidden, targets remain contained, and
  recovery continues by reconciling and completing the remaining loads;
- during the start phase, reverse topology order remains mandatory after every
  restart; a parent or entrypoint cannot start before every selected dependent
  has a completed load and start receipt; and
- if an already-started child is directly addressable, that exposure is
  reported as the maintained-entrypoint limitation rather than hidden behind
  an atomicity claim.

A permanently unrecoverable create, upload, or load result is fail-closed but
does not satisfy operational readiness. It prevents 0.94 closeout rather than
becoming an accepted permanent halt.

## Lock Recovery Requirements

Both runners must satisfy all of the following under abrupt process death:

- the lock is scoped to exactly one authoritative journal/layout;
- the lock path rejects symlinks and special-file substitution;
- a second live owner is rejected deterministically;
- an orphaned lock cannot block the backup or restore forever;
- stale recovery cannot steal a lock from a live process;
- process-ID reuse cannot make an old owner appear authoritative;
- spawned commands cannot inherit journal-lock ownership across `exec`;
- lock recovery leaves an auditable typed outcome; and
- removing or replacing a lock is not treated as proof about the pending
  operation's external effect.

The first and smallest implementation choice is a kernel-owned advisory lock
with close-on-exec behavior on the supported local filesystem. Abrupt owner
death must release ownership immediately. PID/start-time metadata may remain
diagnostic evidence but does not decide ownership. Another mechanism requires
a journey-backed finding that kernel locking is unavailable or insufficient;
0.94 does not build a general lease service. A force-delete alias or
undocumented manual lock removal is not an accepted closeout path.

## Effect Capability And Replay Ledger

| Effect | Authoritative observation and identity proof | Replay rule | Concurrency assumption | Unresolved outcome |
| --- | --- | --- | --- | --- |
| Preflight/status read | Same plan, target, and typed response | Repeat after command quiescence | No mutation assumed | `safely_halted` on unavailable or invalid evidence |
| Stop/start | Exact target status plus canister identity | Observe, then issue at most one justified command | No competing operator mutation | Temporary `operator_action_required`; journey must complete |
| Snapshot create | Snapshot inventory bound to target, operation window, and new snapshot ID | Never blind replay | Exclusive snapshot activity or an unambiguous inventory delta | Reconcile or exercise operator action; permanent ambiguity fails closeout |
| Snapshot download | Exact target and snapshot ID; private path and final checksum | Replace only the expected private temporary path | Exclusive ownership of that temporary path | `resumed` or typed rejection |
| Checksum/verification | Immutable no-follow input plus expected checksum/identity | Recompute completely | Input cannot be replaced during observation | `safely_halted` on mismatch |
| Artifact/manifest publication | Exact type, containment, checksum, plan, and journal projection | Adopt only exact already-published evidence | Exclusive layout lock | `safely_halted` on conflict |
| Snapshot upload | Target snapshot inventory plus exact target, artifact checksum, and uploaded snapshot ID | Never blind re-upload | Exclusive snapshot activity or an unambiguous inventory delta | Reconcile or exercise operator action; permanent ambiguity fails closeout |
| Snapshot load | Exact target and uploaded snapshot ID plus authoritative restored-state evidence | Repeat only when prior completion or safe replay is proven | Target remains stopped; no competing load | Reconcile or exercise operator action; permanent ambiguity fails closeout |
| Completed operation | Matching terminal receipt and owning state | Never execute again | Journal authority is exclusive | Reject any contradiction |

This ledger is a capability test, not an implementation promise. If the
underlying environment cannot provide the stated observation or identity proof,
the baseline run records a finding. Canic does not invent an identifier, infer
success from an empty response, or retry until something appears to work.

## Corruption And Rejection Matrix

| Case ID | Input | Required result |
| --- | --- | --- |
| `CANIC-094-C01` | Truncated, malformed, unknown-field, missing-field, or unsupported-version JSON | Reject before effect; preserve the last independently valid evidence. |
| `CANIC-094-C02` | Plan/journal/manifest/receipt identity or operation mismatch | Reject with the exact conflicting owner and field; do not repair projections. |
| `CANIC-094-C03` | Stale topology, preflight, environment, target, snapshot, or attempt identity | Reject before mutation or reconciliation of unrelated work. |
| `CANIC-094-C04` | Missing artifact, checksum mismatch, unsafe path, symlink, special file, or post-validation replacement | Reject before upload/load and retain no authoritative private stage. |
| `CANIC-094-C05` | Partial temporary download or private restore stage | Treat as non-authoritative; safely replace or reject it without exposing it as durable. |
| `CANIC-094-C06` | Published artifact/manifest with absent or stale journal transition | Reconcile only by exact identity and checksum; otherwise reject without overwrite. |
| `CANIC-094-C07` | Live, orphaned, malformed, or reused-owner lock evidence | Preserve mutual exclusion and recover liveness without guessing. |
| `CANIC-094-C08` | Terminal state without receipt, receipt without matching pending attempt, stale receipt, or duplicate attempt | Reject; never project completion or advance the next operation. |
| `CANIC-094-C09` | Command success with missing uploaded snapshot ID or unparsable required status | Persist a typed failure; never infer success. |
| `CANIC-094-C10` | Persistence failure before rename, after rename, or during cleanup | Preserve either the prior or next complete document; cleanup failure does not replace the primary cause. |

Every rejection case records whether retry is safe, reconciliation is required,
or human/environment change is required. A display string alone is not the
internal cause, although documented CLI JSON, text, and exit behavior may be
asserted at their operator boundary.

## Realistic Multi-Canister Journey

The maintained backup sequence must prove every selected canister is stopped
before the first snapshot-create command is issued. This gives the selected
fleet one quiesced backup interval; it does not provide a distributed
transaction with external or unselected canisters. If a selected target cannot
be authoritatively stopped, no snapshot is admitted and cleanup follows the
backup containment policy. Callers remain responsible for coordinating writes
with systems outside the selected backup topology.

The required realistic fixture contains at least:

- one coordinator/root that exposes the current registry topology;
- two snapshot-restored canisters with a parent/child relationship;
- deterministic queryable application state in both restored canisters; and
- a state change after backup so restore must prove old bytes were actually
  loaded rather than merely proving that commands returned success.

The journey is:

1. Install the disposable topology and write state generation `A`.
2. Create a complete backup through the maintained Canic operator path.
3. Verify the backup without consulting live application state.
4. Mutate both restored canisters to generation `B`.
5. Prepare the restore from the exact completed backup.
6. Execute restore through at least one real process restart.
7. Query both canisters and prove generation `A` returned.
8. Run member/deployment verification and backup integrity again.
9. Re-run the completed backup and restore commands and prove zero repeated
   mutating effects.

The journey must use actual persisted files and a new process after the chosen
interruption. Reloading an in-memory clone does not satisfy it. At least one
high-risk effect/receipt window—snapshot creation, upload, or load—must be
exercised against actual canister state rather than only a fake executor.

PocketIC is preferred where it supports the required snapshot/load behavior.
Otherwise a named disposable local ICP environment is permitted. Production
or mainnet mutation, production credentials, and retained private state are
forbidden. Network access is disabled unless the named disposable environment
requires it and the maintainer explicitly authorizes that run.

## Public And Persisted Contract Policy

The current backup and restore documents remain version 1 because they are
genuine versioned durable contracts. Before Canic 1.0, an accepted hard cut
may change their required shape while the document version remains 1. The old
shape then rejects and must be regenerated; it is not migrated or interpreted
through a second reader. A document that does not benefit from post-1.0 format
versioning should not carry a version field at all.

If a required journey proves an existing public, CLI, JSON, Candid, stable, or
persisted shape cannot represent safe recovery:

1. record the exact finding and the missing information;
2. attempt an internal implementation using the existing contract first;
3. identify every producer, consumer, example, diagnostic, and test affected;
4. amend this design and obtain explicit maintainer acceptance for the exact
   pre-1.0 hard cut;
5. keep the genuine durable contract at version 1, with no parallel model or
   alternate reader; and
6. update all active surfaces together and require affected state to be
   regenerated.

The old shape is removed in that hard cut. No alias, shim, migration or
fallback reader, duplicate command, deprecated variant, hidden repair, or
anti-resurrection test is added. Old documents are rejected and regenerated.
No pre-1.0 hard cut introduces a version 2 merely to preserve superseded
version 1 data.

## Validation Strategy

Validation is layered by evidence value:

### Layer 1 - State-machine and persistence proofs

- exact plan/journal/receipt validation;
- both sides of durable replacement;
- terminal-state/receipt agreement;
- artifact staging/publication identity;
- operation replay counts; and
- typed corruption/rejection paths.

### Layer 2 - Process-death proofs

- child process terminated without unwinding;
- orphaned/live lock behavior;
- filesystem and target state reopened by a new process;
- pending effect reconciliation; and
- response-loss replay after completion.

### Layer 3 - Real environment proof

- actual multi-canister topology;
- actual snapshot create/download/upload/load where supported;
- actual state mutation and restoration;
- maintained CLI/host command path; and
- complete post-restore verification.

Targeted package tests and strict Clippy run for every changed owner. Final
cumulative closeout additionally requires the complete relevant workspace test
set, all-target strict Clippy, formatting, layering guards, changelog/status
governance, diff hygiene, all 106 protocol cases, and the realistic environment
journey. These cumulative gates validate the bounded line; they do not authorize
unrelated CI cleanup. Documentation-only design work uses link, formatting,
and diff hygiene rather than code tests.

## Closeout Verdict

0.94 uses:

```text
journey_result:
  pass | fail | blocked_by_environment

recovery_disposition:
  resumed | reconciled | operator_action_required | safely_halted

finding_status:
  open | fixed | accepted | deferred | rejected | blocked

closeout_verdict:
  pass | pass_with_limitations | fail | blocked
```

The following are non-waivable and prevent `pass` or
`pass_with_limitations`:

- a non-idempotent committed effect can be blindly repeated after receipt
  loss;
- a new mutating command can start while its predecessor may still be running;
- a completed operation can execute again;
- an orphaned lock permanently prevents required recovery;
- a live lock can be stolen;
- corrupt or mismatched evidence can reach mutation or success;
- terminal state can exist without its exact receipt;
- a failed/incomplete restore can report complete or verified;
- a create, upload, or load interruption remains permanently unrecoverable;
- the realistic journey cannot prove actual application-state restoration; or
- any required journey remains `fail` or `blocked_by_environment`.

`pass_with_limitations` is reserved for explicit non-core limitations such as
intentionally repeat-from-start read-only verification, provided every
required journey passes and the limitation does not weaken the central
invariant. An unavailable environment does not become a passing limitation.

## Acceptance Criteria

0.94 is complete only when:

1. the exact backup and restore durable-transition inventory is frozen;
2. the disposable-environment capability gate proves or explicitly resolves
   snapshot identity, load observation/replay, child lifetime, quiescence, and
   environment support before the complete harness is frozen;
3. every named backup and restore crash point passes before and after its
   durable transition;
4. every mutating command owner-death case proves quiescence before
   reconciliation or another invocation;
5. every effect-committed/receipt-missing create, upload, and load case
   ultimately resumes or reconciles, including after an exercised documented
   operator action; permanent safe halt is not accepted;
6. live and orphaned lock behavior preserves both exclusion and liveness;
7. verification interruption is safely repeatable and leaves no durable pass
   marker;
8. re-running completed backup and restore state invokes zero completed
   mutating effects and appends no duplicate receipts;
9. every corruption and partial-publication case rejects deterministically;
10. the realistic multi-canister journey restores exact pre-mutation
   application state through actual persistence and process restart;
11. incomplete multi-canister restore is contained at the maintained entrypoint
    and never reported as complete or verified;
12. all finding-backed fixes pass causal and cumulative reruns;
13. public and persisted surfaces are unchanged or covered by an explicitly
    accepted hard-cut amendment;
14. operator documentation gives the exact successful journey, recovery
    actions, and limitations;
15. all targeted and cumulative closeout gates pass;
16. the tracker contains no open, deferred, or blocked core finding;
17. the final closeout records one explicit verdict and evidence identity; and
18. package version changes remain maintainer-owned.

## Risks And Mitigations

### Risk: the test harness becomes a general failpoint framework

Mitigation: freeze only the named backup/restore barriers and use existing
executor/persistence boundaries. Test-only orchestration stays out of product
CLI and runtime behavior.

### Risk: replay is called idempotent because a command usually tolerates it

Mitigation: assert authoritative target state and exact effect counts.
Non-idempotent snapshot operations reconcile by identity or exercise a
documented operator action; permanent ambiguity prevents closeout.

### Risk: a graceful error test is mistaken for a crash test

Mitigation: required process-death cases terminate without unwinding and
restart in a new process against the same durable files.

### Risk: the realistic journey tests only command success

Mitigation: mutate deterministic application state after backup and prove the
old state returns after restore.

### Risk: partial multi-canister restore is described as atomic

Mitigation: retain topology phase ordering, keep completion false until all
verification passes, record changed targets, and describe the guarantee as
maintained-entrypoint containment. Direct-principal child access remains an
explicit limitation.

### Risk: operational findings reopen general cleanup

Mitigation: every implementation change cites a required journey and crash
point. Unrelated observations remain outside the line.

## Evidence Batches

The bounded work may land in four coherent batches without pre-approving any
product fix:

1. inventory, capability probe, and harness: probe real snapshot and child-
   process capabilities first, amend the design if necessary, then freeze the
   106 cases, deterministic handshake, separate invocation/effect counts,
   command-death proof, and lock proof;
2. backup and verification: run backup authority, crash, replay, corruption,
   and cleanup cases, then fix only reproduced findings;
3. restore: run restore crash, containment, reconciliation, corruption, and
   completed-replay cases, then fix only reproduced findings; and
4. realistic journey and closeout: prove state `A -> B -> A`, run one real
   high-risk interruption, complete cumulative gates, publish the runbook, and
   record the verdict.

These are evidence groupings, not architectural layers or permission for
speculative abstraction.

## Closed Design Decisions

- 0.93 closes at `v0.93.36`.
- 0.94 is backup/restore operational readiness only.
- The central unit of work is a crash point and observable restart state, not
  a broad module slice.
- Both sides of every authoritative durable transition are required.
- Abrupt process death, not only returned failure, is required evidence.
- A prior external command must be quiescent before restart observes or acts.
- Command invocation count and committed-effect count remain distinct.
- Real platform observation and child-lifetime capabilities are probed before
  the complete harness is built.
- Pending means the effect may be unknown; it is not permission for blind
  replay.
- Snapshot create, upload, and load must ultimately reconcile or resume;
  permanent safe halt prevents closeout.
- Operator recovery uses maintained commands or documented platform actions;
  it never edits or fabricates durable evidence.
- Completed operations execute zero additional mutating effects.
- Read-only verification may restart from the beginning; it does not need a
  progress journal.
- Journal locking must preserve both live-owner exclusion and orphan recovery.
- Kernel-owned, close-on-exec advisory locking is the first implementation
  choice; no general lease system is introduced.
- At least one realistic multi-canister actual-state restore is mandatory.
- Backup snapshots begin only after every selected target is proven stopped;
  systems outside that topology are not transactionally covered.
- Restore containment is guaranteed through the maintained entrypoint, not
  against direct-principal child access.
- Genuine durable document versions remain 1 throughout pre-1.0 hard cuts;
  superseded shapes reject and are regenerated without a v2 model, alternate
  reader, migration, or fallback.
- Public-surface changes require a journey-backed design amendment.
- No aliases, compatibility shims, fallbacks, broad cleanup, dependency work,
  speculative abstractions, or 1.0 claim enter the line.
