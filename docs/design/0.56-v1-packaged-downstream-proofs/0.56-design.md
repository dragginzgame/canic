# 0.56 Design: V1 Packaged Downstream Proofs

## Status

- Status: Tentative
- Line: `0.56.x`
- Theme: v1 packaged downstream and installed CLI proof

0.56 is not a new product-feature line. It is the next release-confidence line
after 0.55. 0.55 proves the compact v1 operator story from a repository
checkout; 0.56 should prove that the same compact story survives packaging,
installation, and downstream project use.

## Summary

The compact v1 story is now proven locally:

```text
create fleet -> scaffold role -> attach role -> build with provenance ->
check deployment evidence -> gate saved evidence -> inspect known deployments
```

The remaining v1 confidence gap is distribution. A user does not live inside
the Canic repository. They install `canic`, depend on published crates, and run
Canic from a downstream project with their own project root, ICP root, package
manifests, fleet config, and generated artifacts.

0.56 should make that path boring and testable:

```text
packaged crates + installed CLI + temp downstream project -> v1 proof
```

The output of 0.56 should be release confidence, not broader authority.

If 0.56 passes without discovering a real packaging blocker, it should be the
last pre-v1 confidence line. The next step should be a v1 candidate audit or
release-preparation pass, not another architecture layer.

## Why This Line

0.51 through 0.55 created a stable and small v1 candidate:

- stable evidence envelopes;
- stable build provenance;
- passive policy gates;
- passive deployment catalog;
- v1 readiness docs;
- fast local smoke proof;
- heavier operator proof.

Those checks are necessary, but they mostly exercise the source checkout path.
The next practical release question is:

```text
Can a user install the packaged CLI and use packaged Canic crates in a clean
downstream project without relying on repository-only shortcuts?
```

This is more urgent than groups, locks, signing, registry import, teardown,
provider wrappers, active adoption, or controller mutation because it protects
the v1 user experience that already exists.

## Core Rule

0.56 must not add product surface unless a packaged-v1 proof exposes a blocking
bug in an already shipped surface.

It must not add:

- deployment groups;
- promotion lanes;
- signing or attestations;
- locks;
- provider wrappers;
- artifact registry import;
- `wasm_store` retention or garbage collection;
- active adoption/import;
- controller mutation;
- topology mutation beyond existing role lifecycle commands;
- install or upgrade authority;
- teardown;
- broad live verification;
- new stable public DTO families;
- one-command deployment pipelines.

0.56 may change scripts, docs, tests, and release gates. It may hard-cut stale
scripted command shapes if they no longer match the v1 surface.

0.56 is allowed to delete, rewrite, or hard-cut stale release probes. It is not
allowed to add new user-facing Canic commands unless a packaged proof reveals a
blocker in an already documented v1 command.

## Definition Of Packaged

For 0.56, "packaged" has a strict meaning.

After package archives are created, a packaged downstream proof must not depend
on:

- workspace path dependencies;
- `target/debug/canic`;
- repository-relative crate paths;
- unpublished local crates;
- hard-coded local paths such as `/home/...`;
- repository `.canic` or `.icp` state;
- mutated checked-in fixtures.

The proof may use the repository only to create the package archives and to run
the proof script itself. Downstream execution must use:

- a temporary install root;
- a temporary downstream project root;
- a temporary unpacked package root;
- an installed `canic` binary where the proof is about installed CLI behavior;
- package archives or explicitly unpacked package contents for crate
  resolution.

This is the point of 0.56: the proof should fail if it is only a disguised
repository checkout test.

## V1 Packaged Surface To Prove

The maintained packaged downstream proof should exercise the same current
surface as v1, not legacy compatibility paths:

```text
canic fleet create <fleet>
canic scaffold canister <fleet> <role>
canic fleet role attach <fleet> <role> --subnet <subnet>
canic build <fleet> <role> --provenance <path>
canic deploy check <deployment> --format envelope-json
canic evidence gate --policy <path> --envelope <path>
canic evidence gate --policy <path> --manifest <path>
canic deploy catalog list
canic deploy catalog inspect <deployment>
```

The proof does not need to run all commands in one script. It must, however,
show that the packaged/installed path does not require stale shortcuts such as:

```text
canic build <role>
canic build --dev
canic scaffold canister <fleet> <role> --attach
canic fleet role attach <fleet> <role> --pool <pool>
canic fleet role attach <fleet> <role> --max-shards <n>
```

## Existing Probe Cleanup

0.56 should audit and align existing release probes:

```text
scripts/ci/verify-packaged-downstream-cli.sh
scripts/ci/verify-packaged-downstream-wasm-store.sh
scripts/ci/verify-installed-canic-cli.sh
Makefile
```

The goal is not to multiply scripts. The goal is to make every retained script
answer a clear release question:

- Can the installed CLI run the v1 setup/catalog/evidence story?
- Can packaged crates resolve without unpublished repository paths?
- Can packaged downstream Wasm-store bootstrap behavior still build when it is
  explicitly requested?
- Do packaged probes use current v1 command shapes?
- Do proof scripts write only temp project/proof roots plus normal Cargo
  outputs?

If a script answers no current release question, it should be removed or
folded into a maintained proof.

0.56.0 should also produce a retained-script inventory in docs or the closeout
audit:

| Script / target | Release question | Installed CLI? | Packaged crates? | Temp root? | Network? |
| --- | --- | --- | --- | --- | --- |

If a retained script cannot be explained in that table, it should be removed,
renamed, or folded into a maintained proof.

## Installed CLI Proof

0.56 should keep or add a maintained installed-binary proof.

Required behavior:

- install the CLI from the local package path or packaged source into a temp
  install root;
- run the v1 local smoke through that installed binary;
- avoid mutating repository `.canic`, `.icp`, fleet fixtures, or controller
  state;
- avoid mainnet/network requirements;
- use a temporary downstream project;
- use current v1 command shapes.

The installed CLI proof must not call `target/debug/canic`. If it accepts a
`CANIC_BIN` override for local debugging, the release proof path must still
exercise the installed binary by default.

The proof should isolate local state where practical:

- set `HOME` or avoid using home-owned state;
- set `CARGO_HOME` when doing so is practical without hiding real package
  resolution issues;
- set `CARGO_TARGET_DIR` or use a proof-local target directory when the proof
  should not share repository build output;
- set explicit project, ICP, and proof roots.

Optional heavier coverage:

- build one attached role with provenance through the installed binary;
- emit a deployment-check envelope under a temp proof root;
- expect `blocked_by_policy` when no live install/root verification exists.

## Packaged Crate Proof

0.56 should prove that packaged crates are publish-safe for the supported v1
use cases.

Required behavior:

- create package archives from the workspace;
- unpack them into an isolated temp package root;
- wire only packaged crates through `[patch.crates-io]` when needed;
- run a downstream probe without unpublished local crate dependencies;
- verify `[package.metadata.canic] fleet` plus `role` still resolve through
  packaged build/config helpers;
- verify every package-backed role has explicit `package = "<path>"`;
- fail on stale role-only metadata or topology-only package-backed role
  assumptions.

The packaged proof should not make canister runtime crates into general
downstream Rust dependencies. Canic-managed canister crates remain runtime
artifacts.

This proof is separate from the installed CLI proof. Installed CLI behavior and
packaged crate resolution are related, but they answer different release
questions and should fail independently.

## Wasm Store Bootstrap Proof

The `wasm_store` path is special and should stay special.

0.56 may retain a packaged downstream `wasm_store` probe if it proves the
generated wrapper/build path still works outside the repository. That proof
must remain explicit and separate from ordinary user canister setup.

It must not imply that ordinary users depend on `canic-wasm-store` as an rlib
or import Canic-managed canister crates directly.

## MSRV And Toolchain Boundary

0.56 should verify the published MSRV story where practical.

The design target is:

- published crates declare the supported MSRV consistently;
- repo-local newer toolchains may exist, but they must not be required by
  users of the published crate surface;
- release/package probes should make the MSRV lane explicit when the toolchain
  is available;
- if the MSRV toolchain is unavailable locally, the release audit records that
  limitation instead of silently skipping the question.

0.56 must not bump MSRV or dependency versions unless explicitly requested as
release-preparation work.

If the MSRV lane fails because the local toolchain is unavailable, the audit
may record that limitation. If it fails because a packaged crate actually
requires a newer Rust version than declared, that is a release blocker.

Do not use networked `rustup` installation inside the release proof unless the
repository's release validation policy already permits it for that proof.

## Output Artifacts

Proof scripts should prefer explicit output roots and deterministic paths:

```text
<proof-root>/installed-smoke/
<proof-root>/packaged-cli/
<proof-root>/packaged-wasm-store/
<proof-root>/build-provenance.json
<proof-root>/deployment-check-envelope.json
```

Scripts may leave proof roots in `/tmp` for inspection if documented. Fast
smoke scripts may clean up their temp roots if no artifact inspection is
needed.

## Proposed Slices

### 0.56.0 - Tentative Design And Retained Probe Hard Cut

Create this design and define the packaged downstream proof boundary.

Audit the existing packaged/installed scripts and remove or rewrite stale
command shapes.

Expected outcomes:

- no retained release probe uses `canic build <role>`;
- no retained release probe depends on implicit role-only metadata;
- each retained probe has one documented release question;
- Makefile target names and help text match retained probes.
- retained probes fail if stale v1 command shapes appear in the script body;
- retained probes do not silently fall back to repository checkout paths when
  they claim to prove packaged or installed behavior;
- a retained-script inventory records release question, installed-CLI use,
  packaged-crate use, temp-root use, and network assumptions.

### 0.56.1 - Installed CLI V1 Smoke

Make the installed-binary path run the maintained v1 smoke or an equivalent
temp downstream proof.

This slice should not require network/mainnet or mutate repository fixtures.
It must not call `target/debug/canic` on the default release-proof path.

### 0.56.2 - Packaged Downstream CLI Proof

Align the packaged downstream CLI proof with current v1 commands and package
metadata requirements.

The proof should exercise packaged crates from an isolated package root and a
separate downstream project root.
It must not pass because of repository-relative path dependencies or
unpublished local crates.

### 0.56.3 - Packaged Wasm Store Proof

Keep the special packaged downstream `wasm_store` build path covered, but
document that it is a bootstrap/internal runtime proof rather than ordinary
user canister dependency guidance.

### 0.56.4 - Release Readiness Closeout

Produce a closeout audit that answers:

- which installed/packaged paths are proven;
- which scripts are retained and why;
- whether retained probes use only current v1 command shapes;
- whether packaged crates avoid unpublished repository-only dependencies;
- whether MSRV was verified or explicitly limited by local toolchain
  availability;
- whether 0.56 added no new product surface or mutation authority.

## Acceptance Criteria

0.56 is complete when:

- the packaged/installed proof story is documented in one maintained place;
- retained scripts use current v1 command shapes;
- retained scripts run under temp roots and preserve repository fixtures;
- stale release probes are removed or folded into maintained proofs;
- packaged downstream CLI behavior is verified from packaged crates;
- installed CLI behavior is verified from an installed binary;
- the special `wasm_store` packaged proof is explicit and not presented as
  ordinary canister dependency guidance;
- MSRV/package verification is run or the limitation is recorded;
- installed CLI proof does not call `target/debug/canic`;
- packaged downstream proof does not use repository-local path dependencies
  after package archives are created;
- retained probes fail or audit-fail on stale v1 command shapes;
- retained probes isolate `HOME`, `CARGO_HOME`, `CARGO_TARGET_DIR`, project
  root, proof root, and ICP root where practical;
- package archives are created before downstream use rather than bypassed
  through workspace source paths;
- expected `blocked_by_policy` proof results are documented as successful
  evidence of the boundary, not as release failures;
- no new product surface or mutation authority is added;
- closeout audit verdict is PASS or PASS WITH FOLLOW-UPS with no blocker/high
  findings.

## Non-Goals

0.56 does not:

- add a deployment workflow;
- add one-command install/deploy;
- add deployment groups or promotion lanes;
- add signing, attestations, or locks;
- add provider-specific GitOps integration;
- import artifacts into `wasm_store`;
- import brownfield deployments;
- mutate controllers;
- perform teardown;
- broaden live verification;
- redesign release publishing;
- change package versions, dependency versions, release-script defaults, or
  install URLs without explicit maintainer release-preparation approval.

## Validation

Expected validation should include:

```text
cargo fmt --all --check
cargo test -p canic --test changelog_governance --locked
cargo test -p canic --test workspace_manifest --locked
scripts/ci/v1-readiness-smoke.sh
scripts/ci/v1-operator-proof.sh
make test-packaged-downstream-cli
make test-packaged-downstream-wasm-store
make test-installed-canic-cli
git diff --check
```

If a packaged or installed proof requires a tool not available locally, the
0.56 closeout audit must record the exact command, failure mode, and strongest
passing substitute.
