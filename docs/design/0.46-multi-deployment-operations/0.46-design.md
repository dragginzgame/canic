# 0.46 Design: Multi-Deployment Operations

## Status

Tentative design notes for the 0.46 line.

0.46 turns prod, staging, v2, tenant, and customer deployments into comparable
operational objects. The line also hard-cuts the remaining fleet-named local
install state seam before deployment groups or catalog work can grow.

Core goal:

```text
Prod, staging, v2, and tenant deployments are comparable operational objects.
```

---

## Scope Calibration

Release bar:

- deployment-target local state hard cut;
- exact deployment target identity for plan-mediated install;
- explicit deployment registration with minimal state;
- old fleet-named live state refusal with recovery guidance;
- passive two-target comparison over existing deployment-truth artifacts.

0.46 is not a deployment orchestration line. It is an identity hard-cut plus
passive comparison line. Catalog, readiness, teardown, and test-deployment
command shapes are post-46 backlog topics.

The first foundation is identity correctness:

```text
FleetTemplate = reusable desired topology
DeploymentTarget = concrete live deployment state
DeploymentGroup = related deployment targets
```

0.46 must not build group/catalog features on fleet-named live state. Any
comparison or drift artifact that represents a live deployment must be bound to
a deployment target, not merely to the fleet template that produced it.

Deferred ideas belong in the post-46 backlog, not in a numbered follow-on
minor. 0.46 may leave hooks in the data model, but the release bar is only the
identity hard cut and passive two-target comparison.

## Deployment Target State Hard Cut

0.46 hard-cuts local live state from fleet-template identity to
deployment-target identity.

Rules:

- live deployment state is keyed by deployment target;
- fleet templates remain reusable topology/config inputs;
- old fleet-named install state is not deployment truth;
- no automatic migration from old fleet state;
- no silent fallback from fleet state to deployment state;
- no compatibility bridge in deployment truth;
- plan-mediated install requires an exact deployment target match.

Target local state path:

```text
.canic/<network>/deployments/<deployment>.json
```

Legacy local state path:

```text
.canic/<network>/fleets/<fleet>.json
```

The legacy path may be detected only to fail closed with recovery guidance.
It must not be read as authoritative live deployment truth.

Supplied-plan install must reject any plan whose
`DeploymentPlanV1.deployment_identity.deployment_name` does not match the
requested deployment target. Matching `fleet_template` alone is not sufficient.

### Operator Recovery

No automatic conversion is allowed. A hard-cut recovery path may exist only
when the operator explicitly registers a known live deployment target.

Preferred command shape:

```text
canic deploy register <deployment> --fleet-template <fleet> --root <principal>
```

This command writes minimal deployment-target state directly. It must not
infer or migrate from `.canic/<network>/fleets/<fleet>.json` automatically.

Registration state is deliberately small:

```text
schema_version
deployment_name
fleet_template
network
root_canister_id
root_verification
created_at
updated_at
```

It must not write receipts, artifact hashes, config digests, controller
snapshots, release manifests, role inventories, or `wasm_store` facts. Those
are deployment truth and must be rediscovered by `deploy check` or
`deploy inventory`.

`root_verification` records whether the registered root principal was checked
against the target network. If the network is reachable, registration should
verify the root canister status before writing verified state. If live
verification cannot be performed, registration should require an explicit
operator flag such as `--allow-unverified` and write
`root_verification = "not_verified"`.

Deployment state with `root_verification = "not_verified"` must not authorize
mutation. `deploy check` may report live verification facts, but it must remain
observational and must not silently rewrite local state. Either registration
writes verified state, mutation commands perform live verification before
mutation, or a later explicit verification command updates the state.

If the command is not implemented in the first hard-cut slice, the error for
legacy state detection must include exact recovery guidance:

```text
legacy fleet install state found: .canic/local/fleets/demo.json

Canic 0.46 stores live deployment state by deployment target, not fleet
template. Fleet templates are reusable desired topology; deployments are live
targets.

Create explicit deployment state with:
  canic deploy register demo --fleet-template demo --root <principal>

Or reinstall the deployment with a 0.46 install path that writes
deployment-target state:
  canic install demo

If the old state is obsolete, remove:
  .canic/local/fleets/demo.json
```

This is an operator registration escape hatch, not backward compatibility.

## Dependency On Earlier Lines

0.46 depends on stable, serializable, content-addressable, diffable, and
versioned objects from 0.41 through 0.45:

- deployment plans;
- live inventories;
- receipts;
- safety reports;
- authority reconciliation reports;
- artifact promotion reports;
- external lifecycle proposals and receipts.

If 0.41 emits sloppy receipts or non-normalized diffs, 0.46 becomes painful.

## 0.42.14 Handoff Constraints

Multi-deployment comparison must preserve the authority boundary 0.42 closed:

- Authority `Safe` is authority-scoped and does not replace the deployment
  safety report.
- Authority command success means a local authority artifact was produced; it
  does not imply live controller mutation or whole-deployment safety.
- Authority dry-run receipts/evidence are structural self-consistency
  artifacts, not tamper-evident proof of live controller state.
- `Authority*V1` JSON fields and enum strings are stable automation surfaces.
- Future mutating authority receipts must stay schema-distinct from dry-run
  receipts.

0.46 may compare authority dry-run artifacts as reporting evidence. It must
not treat them as proof that live controller state changed or that a deployment
is globally safe.

## 0.45 Handoff Constraints

Multi-deployment comparison must preserve the external lifecycle boundary 0.45
closed:

- Consent evidence, reported external action, supplied observations, and
  passive completion reports are operator evidence, not live deployment truth.
- Supplied observations remain structural consistency evidence only.
- `DeploymentTruthInventory`-backed verification derived from a bound
  `DeploymentCheckV1` is the only 0.45 external lifecycle artifact that may be
  compared as verified external completion.
- External lifecycle completion remains distinct from direct deployment
  authority. User-controlled, jointly controlled, externally imported, and
  unknown-unsafe roles must not become directly executable because a comparison
  report observes lifecycle evidence.
- Wall-clock observation-age enforcement did not land in 0.45. 0.46 may compare
  check IDs, digests, inventory IDs, observed-at evidence, and artifact
  bindings, but must not imply stronger freshness semantics unless it adds and
  validates them explicitly.

0.46 may compare external lifecycle proposals, receipts, consent evidence,
completion reports, and verification checks as evidence. It must not flatten
proposal, consent, reported completion, supplied observation, and
inventory-backed verification into one "complete" state.

## Comparison Model

0.46 should provide normalized comparison for existing deployment-truth
artifacts:

```text
plan vs plan
plan vs inventory
plan vs receipt
inventory vs inventory
receipt vs receipt
```

Comparison categories should reuse 0.41 `DeploymentDiffV1`:

- controllers;
- module hashes;
- source artifact digests;
- installed module hashes;
- canonical embedded config digests;
- root trust domain;
- authority profile;
- pool identity set;
- role topology;
- role epochs and verifier readiness;
- external/user-owned lifecycle state, including pending consent, reported
  action, supplied-evidence consistency, inventory-verified completion, and
  residual exposure as distinct states;
- `wasm_store` artifact evidence already present in deployment-truth checks.

Installed module hash and live canister inventory remain deployment truth.
`wasm_store` catalog and receipt locators are artifact availability evidence.
0.46 does not add live `wasm_store` catalog lookup.

## Drift Reports

Drift reports should answer:

- what changed?
- where did it change?
- is the change expected?
- is the change safe?
- which operation or receipt last touched it?
- what action is required?

Tentative shape:

```text
DeploymentDriftReportV1 {
  compared_at,
  left_identity,
  right_identity,
  trust_domain_diff,
  artifact_diff,
  embedded_config_diff,
  authority_diff,
  pool_diff,
  verifier_readiness_diff,
  external_lifecycle_diff,
  hard_failures,
  warnings,
  recommended_next_actions,
}
```

## Post-46 Backlog

The following ideas are explicitly outside 0.46:

- deployment groups and promotion lanes;
- deployment catalog UX;
- promotion readiness across groups;
- lane teardown and cleanup planning;
- test deployment create/rebuild/teardown workflows;
- verification baselines and full `deploy verify`;
- live `wasm_store` catalog lookup;
- artifact registry and retention policy;
- adoption profiles;
- CI/GitOps provenance contracts;
- DR clone and trust-domain migration commands.

They live in [Post-46 Deferred Deployment Ideas](../post-46-backlog/deferred-from-0.46.md)
and the post-46 backlog topic docs. They must not be treated as approved
numbered follow-on lines.

## Commands

Tentative operator surface:

```text
canic deploy register <deployment> --fleet-template <fleet> --root <principal>
canic deploy compare --left <file> --right <file>
```

Command names are tentative. The stable parts are the deployment-target state
hard cut and passive comparison model.

`deploy register` is the narrow hard-cut recovery surface. It may be
implemented before group/catalog commands, writes only minimal deployment
target state, and does not imply automatic migration from old fleet state.

---

## Non-Goals

0.46 should not:

- reimplement promotion semantics;
- bypass authority reconciliation for teardown;
- hide trust-domain differences;
- require all deployments in a group to share one root;
- treat fleet-template install state as deployment-target state;
- accept a supplied plan for install because only its `fleet_template` matches;
- silently migrate or read old `.canic/<network>/fleets/*.json` state as live
  deployment truth;
- add deployment group/catalog/teardown/test lifecycle workflows;
- add live `wasm_store` catalog lookup;
- implement full `canic deploy verify`;
- run broad destructive cleanup outside a teardown plan;
- turn deployment group config into a scripting language.

## Implementation Slices

### Slice 0: Deployment Target State Hard Cut

Introduce deployment-target local state and stop using fleet-named install
state as live deployment truth.

Required outcomes:

- write live deployment state under
  `.canic/<network>/deployments/<deployment>.json`;
- carry `deployment_name` and `fleet_template` as separate identities;
- update deployment-truth reads to use deployment-target state;
- reject old `.canic/<network>/fleets/<fleet>.json` state with clear recovery
  guidance;
- add explicit deployment registration with minimal state and root verification
  status;
- require exact deployment target identity for
  `canic deploy install <deployment> --plan <file>`;
- block mutation for unverified registered roots until an explicit verification
  path or the mutation command itself verifies live status;
- remove any supplied-plan fallback where matching `fleet_template` is enough;
- add focused tests proving fleet-template state cannot be used as live
  deployment state.

This slice is required before deployment group, catalog, or drift workflow
expansion.

### Slice 1: Passive Two-Target Comparison

Compare plans, inventories, receipts, and deployments using normalized diff
categories.

---

## Exit Criterion

0.46 is complete when:

```text
Canic stores live local deployment state by deployment target, refuses old
fleet-named state as deployment truth, requires exact deployment identity for
plan-mediated install, and can passively compare two deployment-truth artifacts
without implying live crawling or mutation.
```

The line is not complete until live local state is deployment-target keyed and
fleet-named install state is no longer accepted as deployment truth.
