# 0.46 Design: Multi-Deployment Operations

## Status

Tentative design notes for the 0.46 line.

0.46 turns prod, staging, v2, tenant, and customer deployments into comparable
operational objects.

Core goal:

```text
Prod, staging, v2, and tenant deployments are comparable operational objects.
```

---

## Scope Calibration

Release bar:

- plan/inventory/receipt comparison;
- drift report over 0.41 categories.

Design hook:

- deployment catalog as an operator concept;
- `wasm_store` artifact availability comparisons;
- promotion readiness across lanes;
- lane teardown as a designed command shape;
- verification baselines as comparison inputs, while full `deploy verify`
  stays in 0.50.

Deferred:

- full deployment catalog UX;
- actual `deploy verify` implementation in 0.50;
- DR clone and trust-domain migration commands.

---

## Dependency On Earlier Lines

0.46 depends on stable, serializable, content-addressable, diffable, and
versioned objects from 0.41 through 0.45:

- deployment plans;
- live inventories;
- receipts;
- safety reports;
- authority reconciliation reports;
- artifact promotion reports;
- external lifecycle proposals and receipts.

If 0.41 emits sloppy receipts or non-normalized diffs, 0.46 becomes painful.

---

## Deployment Groups

A deployment group is a set of deployments that share an operational
relationship.

Examples:

```text
toko:
  prod
  staging
  v2
  destructive-rehearsal

tenant-a:
  prod
  staging
```

Tentative shape:

```text
DeploymentGroupV1 {
  name,
  fleet_template,
  deployments,
  promotion_lanes,
  comparison_policy,
}
```

A group does not imply a shared root. Different deployments may be different
trust domains.

---

## Deployment Catalog

0.46 should make deployments addressable as operational objects, not only as
fleet template directory names.

Tentative shape:

```text
DeploymentCatalogEntryV1 {
  deployment_id,
  deployment_name,
  group,
  fleet_template,
  network,
  root_principal,
  trust_domain,
  last_inventory_id,
  last_receipt_id,
  labels,
  owner_hint,
}
```

The catalog is a model and operator concept in 0.46. It does not require a full
search UI before the line can ship.

## Comparison Model

0.46 should provide normalized comparison for:

```text
plan vs plan
plan vs inventory
plan vs receipt
inventory vs inventory
receipt vs receipt
deployment vs deployment
```

Comparison categories should reuse 0.41 `DeploymentDiffV1`:

- controllers;
- module hashes;
- source artifact digests;
- installed module hashes;
- canonical embedded config digests;
- root trust domain;
- authority profile;
- pool identity set;
- role topology;
- role epochs and verifier readiness;
- external/user-owned pending consent;
- `wasm_store` artifact availability.

`wasm_store` comparison should distinguish:

- same sealed artifact digest;
- different sealed artifact digest;
- same source/build identity;
- different canonical embedded config digest;
- artifact present in left store;
- artifact absent from right store;
- artifact present but not installed;
- artifact installed but not present in current store catalog.

Installed module hash and live canister inventory remain deployment truth.
`wasm_store` catalog and receipt locators are artifact availability evidence.

---

## Drift Reports

Drift reports should answer:

- what changed?
- where did it change?
- is the change expected?
- is the change safe?
- which operation or receipt last touched it?
- what action is required?

Tentative shape:

```text
DeploymentDriftReportV1 {
  compared_at,
  left_identity,
  right_identity,
  trust_domain_diff,
  artifact_diff,
  embedded_config_diff,
  authority_diff,
  pool_diff,
  verifier_readiness_diff,
  external_lifecycle_diff,
  hard_failures,
  warnings,
  recommended_next_actions,
}
```

---

## Promotion Readiness Across Deployments

0.46 should make promotion readiness visible across a group.

Example questions:

- Which staging artifacts are eligible for prod?
- Which prod roles differ from staging only by embedded config?
- Which roles changed module hash unexpectedly?
- Which user-owned canisters are pending consent?
- Which deployment has a different root trust anchor?
- Which verifier cascade is incomplete?

This builds on 0.44 promotion reports and 0.45 external lifecycle reports.

---

## Lane Teardown

Teardown is plan-driven and authority-aware.

Rules:

- compare the target deployment identity before mutation;
- do not use production authority by default;
- show authority reconciliation requirements before teardown;
- emit receipts for every attempted cleanup action and verified postcondition;
- preserve enough evidence to explain what was removed and what was left.

Broad cleanup commands are a non-goal. A teardown plan targets a deployment
identity, not "whatever looks related" on the network.

---

## Verification Baselines

0.46 may compare verification artifacts if they already exist:

```text
verification profile id
last verification result
metrics snapshot digest
readiness baseline
```

0.46 does not implement full `canic deploy verify`.

0.50 owns deploy verify profiles, runtime checks, protected-call probes, cycle
floor checks, and metrics snapshot production.

---

## Test Deployment Lifecycle

Because the IC does not provide a public testnet, test deployments are real
deployments.

0.46 should make test deployment operations safer:

- create named test deployment;
- compare to prod baseline;
- teardown with authority checks;
- rebuild from plan;
- preserve receipts for audit;
- prevent production authority from leaking into test deployment plans.

Teardown should be plan-driven and authority-aware. It should not be a broad
destructive cleanup command.

---

## Commands

Tentative operator surface:

```text
canic deploy group list
canic deploy compare <left> <right>
canic deploy drift <deployment>
canic deploy readiness <group>
canic deploy catalog list
canic deploy test create <name>
canic deploy test teardown <name>
```

Command names are tentative. The stable part is the comparison model.

---

## Non-Goals

0.46 should not:

- reimplement promotion semantics;
- bypass authority reconciliation for teardown;
- hide trust-domain differences;
- require all deployments in a group to share one root;
- implement full `canic deploy verify`;
- run broad destructive cleanup outside a teardown plan;
- turn deployment group config into a scripting language.

---

## Implementation Slices

### Slice 1: Deployment Group Model

Represent deployment groups and promotion lanes without changing individual
deployment semantics.

### Slice 2: Comparison Commands

Compare plans, inventories, receipts, and deployments using normalized diff
categories.

### Slice 3: Drift Reports

Render operator-facing drift reports with hard failures, warnings, and next
actions.

### Slice 4: Promotion Readiness

Summarize artifact, embedded config, authority, and external lifecycle
readiness across a deployment group.

### Slice 5: Test Deployment Lifecycle

Add plan-driven create, rebuild, and teardown flows for real IC test
deployments.

---

## Exit Criterion

0.46 is complete when:

```text
Operators can compare prod, staging, v2, and tenant deployments and know what
differs before promoting, rebuilding, or tearing anything down.
```
