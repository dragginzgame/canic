# Canic 0.91 Design: Role Admission and Complete-Build Manifest Publication

Date: 2026-07-13

## Status

- Status: complete; Slices A and B were published in `v0.91.0`. Later 0.91
  maintenance is outside this bounded design scope.
- Audit source:
  [post-0.90 deployment health](../../audits/reports/2026-07/2026-07-13/post-0.90-deployment-health.md).
- Implementation tracker: [status.md](status.md).

## Problem

Canic currently has two bounded deployment-identity gaps.

First, complete config validation checks only the byte length of canister role
names. Fleet mutation commands separately reject characters outside ASCII
letters, numbers, `_`, and `-`. A hand-edited config can therefore pass the
core boundary with a role such as `..` or `../app`, after which the artifact
builder uses that role directly in directory and file paths.

Second, building one role may publish a full fleet release-set manifest. The
writer considers the fleet ready when every configured `.wasm.gz` path exists,
including artifacts left by an older build. A newly built role can therefore
be combined with stale role bytes under the current root package version. The
manifest hashes those exact bytes and is internally valid, so later shape and
hash checks cannot identify the mixed release.

0.91 closes those gaps at their admission and orchestration owners. It does
not claim concurrent-writer safety, symlink containment, cross-platform path
equivalence, or preservation of a previously deployable artifact set.

## Decisions

### Slice A - One role-name admission rule

The Canic core configuration schema owns one typed role-name rule:

```text
role is valid when:
  byte length is 1..=40
  it is lowercase ASCII snake_case
  it begins with an ASCII lowercase letter
  each later word is nonempty and separated by exactly one '_'
  words contain only ASCII lowercase letters and numbers
```

Names such as `app`, `user_hub`, `scale_replica`, and `role_2` are valid.
Kebab-case, mixed-case, leading digits or underscores, trailing underscores,
and repeated underscores are invalid.

Canic does not normalize or equate `-` and `_`. Admitting both would allow two
distinct deployment and artifact identities to collapse when a Cargo or Rust
package context converts a hyphen to an underscore. Lowercase snake_case is
the only accepted spelling, and invalid alternatives fail admission rather
than receiving an alias or rewrite.

One pure core helper owns this ASCII snake_case grammar and is also reused by
the existing scaffold project-name parser. The role policy remains distinct:
it adds the 40-byte limit and typed role issue without imposing that limit on
project names or creating a general name-policy abstraction.

The rule is used by:

- complete `ConfigModel` validation for every canister role occurrence;
- fleet role declare, attach, and rename mutation admission; and
- root release-set manifest validation before a role is trusted as identity or
  used to resolve an artifact.

The rule returns a typed issue so callers preserve their existing error owners
without parsing display text. Core config validation adds local field context.
Fleet mutation maps the issue into `FleetConfigError`. Manifest validation
renders it only at its existing command/report boundary.

`CanisterRole` remains a passive representation and may be constructed during
decoding or storage access. Construction does not constitute admission. Any
value crossing an untrusted configuration, mutation, manifest, or equivalent
deployment boundary must pass the canonical role-name rule before it is used
to derive an artifact path or deployment identity.

Because there is intentionally no validated wrapper type, the implementation
and closeout audit must prove that every production path from externally
supplied, persisted, or loaded role data to an artifact-path sink is dominated
by one of those validation boundaries.

This slice applies to canister roles only. Subnet, fleet, package, pool, and
deployment names are not silently moved under the same policy.

#### Exact hard cut

- Replace the length-only canister-role helpers in core config validation with
  the canonical rule.
- Delete the host mutation `validate_role_name` character implementation; its
  call sites use the core rule and typed mapping.
- Do not retain a second local predicate, alias, sanitizing helper, or fallback.
- Invalid configured, decoded, or loaded-manifest roles are rejected before
  any role-derived artifact path is resolved or accessed.

#### Lexical guarantee

An admitted role contributes exactly one nonempty path segment and cannot
introduce separators, dot components, or alternate relative or absolute path
components into a canonical artifact path.

This is a lexical guarantee. It does not prove safety against symlinked
parents, symlinked role directories, or platform-reserved device names.
Lowercase-only admission removes ASCII case variants from the role namespace;
it does not introduce a general cross-platform path-name policy.

Canic may compile or run other host tooling on filesystems with different name
semantics; that does not extend this deployment-artifact guarantee.
Cross-platform reserved-name admission requires a separate accepted rule
rather than a hidden expansion of this slice.

### Slice B - Complete-build proof is explicit writer input

A single-role build owns one role's Wasm, gzip, optional Candid sidecar, and
optional build provenance. It does not own a fleet release set.

The install-root operation resolves one validated in-memory snapshot before
building. That snapshot fixes:

- the complete required build target set;
- the roles that become release-set entries;
- the exact expected lexical artifact path for each role; and
- the release version used by the manifest.

Every required build is parameterized from that same validated install
snapshot. During a complete-build invocation, per-role builders do not reload
deployment configuration or independently resolve target identity, build
selection, release version, or expected artifact location. The builder
receives its exact expected output path from the snapshot and returns that path
in its current output.

The operation invokes the per-role builder for every required target and
collects one role-labelled `CanisterArtifactBuildOutput` from that invocation
for each target. The label may be carried beside the existing output type; no
public output wrapper is required merely to represent the pair.

Conceptually, the normal writer receives:

```text
write_root_release_set_manifest(
    validated_install_snapshot,
    current_role_build_outputs,
)
```

Before writing, the crate-private manifest owner verifies:

- exactly one current output exists for every required build target;
- no required role is missing;
- no role is duplicated;
- no unexpected role is present;
- every output role and gzip path equals its exact expected value in the same
  snapshot; and
- each manifest-member role is hashed only from the gzip path identified by
  its current output.

Outputs for required targets that are not manifest entries, such as the root
artifact itself, prove complete build coverage but are not serialized as
ordinary release-set entries.

Coverage uses the existing install-root target identity. Manifest roles label
release-set members but do not replace target identity for required
non-member builds. This clarification adds no public build-target key.

Output-path matching is exact snapshot equality. The output gzip path must
equal the expected path value precomputed in the install snapshot. The writer
does not use filesystem canonicalization or equivalence heuristics to make a
mismatched path acceptable.

The writer does not reload configuration, rediscover targets, or treat
configured-path existence as evidence that a build completed. Old files,
including optional Candid or provenance sidecars, cannot satisfy coverage and
are not inferred as current outputs. Optional sidecars remain per-role output;
they are never release-set completion evidence.

If any required build fails, the manifest writer is not invoked. Direct
`canic build <fleet> <role>` likewise never invokes it. The complete-build
owner supplies publication proof as the exact set of successful per-role
outputs from the current invocation; the writer never reconstructs that proof
from filesystem existence.

After exact coverage validation, release-set entries use the existing order
recorded in the snapshot. Builder completion order, output collection order,
and map iteration order cannot change successful manifest JSON bytes.

The deployment-plan override remains separate. Its already validated plan is
the complete snapshot and artifact source for that path, so it does not invent
current build outputs for work it did not perform.

#### Failure before publication

The manifest owner completes output coverage validation, exact path matching,
artifact reads, hash construction, manifest validation, and serialization
before replacing the root manifest. Failure at any of those pre-publication
steps leaves an existing root manifest byte-for-byte unchanged.

The final complete bytes continue through the existing durable single-file
writer. This requirement does not introduce an artifact transaction or claim
that a durability error after atomic replacement can restore the old file.

#### Existing manifest after a partial build

A direct or failed partial build leaves an existing root manifest
byte-for-byte unchanged. It may still replace an artifact referenced by that
old manifest. In that case, normal artifact-hash validation rejects the old
manifest.

0.91 guarantees that a manifest is not rewritten to authorize partial output.
It does not preserve the prior artifact set, restore overwritten artifacts, or
guarantee that the prior manifest remains deployable.

#### Exact hard cut

Delete:

- `emit_root_release_set_manifest_if_ready`;
- `emit_root_release_set_manifest_if_ready_with_config`;
- the unused public convenience writer
  `emit_root_release_set_manifest`;
- the `manifest_path` field on `CanisterArtifactBuildOutput`;
- the single-role artifact builder's manifest emission; and
- the build-provenance branch that treats that optional fleet manifest as a
  role-build metadata artifact.

Keep:

- one crate-private manifest writer that consumes the validated install
  snapshot and current role-labelled outputs;
- the deployment-plan override's explicit manifest materialization;
- the current release-set schema, JSON field order, hash construction,
  artifact path validation, staging, and activation behavior; and
- the install deployment-truth receipt for the explicit manifest phase.

Do not replace the deleted readiness functions with a build ID, epoch file,
sidecar database, mtime comparison, artifact registry, or stale-artifact
compatibility path.

## Ownership

| Concern | Owner after 0.91 |
| --- | --- |
| Canister role text policy | `canic-core::config::schema` |
| Config field context and complete validation | `canic-core::config::validation` |
| Fleet mutation error projection | `canic-host::release_set::config` |
| Per-role Wasm/Candid/gzip build | `canic-host::canister_build` |
| Target/config/release-version/output-path snapshot | `canic-host::install_root` |
| Current output collection and exact coverage | `canic-host::install_root` |
| Release-set shape, validation, and persistence | `canic-host::release_set::manifest` |
| Deployment-plan release materialization | `canic-host::install_root::plan_artifacts` |
| CLI rendering and optional per-role provenance | `canic-cli::build` |

The two slices share deployment identity as a correctness theme. They do not
share an implementation abstraction.

## Preserved Contracts

0.91 preserves:

- existing lowercase snake_case role names up to 40 bytes;
- `CanisterRole` representation, Candid/Serde shape, and stable encoding;
- valid `canic.toml` parsing and successful fleet mutation output;
- per-role Wasm, gzip, and Candid paths and bytes;
- root release-set JSON schema, successful complete-build bytes, and hashes;
- deployment-plan override behavior;
- staging, activation, install state, deployment evidence, and network
  selection; and
- existing stable memory and canister runtime behavior.

Intentional changes are limited to rejecting previously admitted non-snake
role names and removing every fleet-manifest side effect from single-role
builds.

## Explicit Operational Precondition

0.91 assumes a single mutating build or install command per install root.
Concurrent commands writing the same artifact root are unsupported and outside
the release guarantee. This line introduces no inter-process lock, isolated
output root, or other concurrent-writer consistency mechanism.

## Deliberate Non-Abstractions

0.91 does not add:

- a validated-name framework shared by unrelated identifiers;
- a path sandbox or safe-join framework;
- a new `CanisterRole` wrapper or fallible constructor migration;
- cross-platform case-folding or reserved-device-name semantics;
- a build session, epoch, lock, lease, or artifact transaction;
- output-directory versioning or restoration of a previous artifact set;
- a new release-set schema or artifact provenance schema version;
- a compatibility reader, name sanitizer, fallback artifact lookup, or old
  manifest writer;
- a generic build pipeline; or
- deployment behavior outside the two audited findings.

## Acceptance

### Slice A

1. One production predicate defines valid canister role text.
2. Empty, over-40-byte, kebab-case, mixed-case, leading-digit,
   leading/trailing/repeated-underscore, dotted, path-like, non-ASCII,
   whitespace, and other punctuated roles fail complete config validation.
3. Release-set manifest validation applies the same predicate before any
   artifact read.
4. Fleet role declare, attach, and rename retain typed `InvalidName` behavior
   without their own character predicate.
5. Existing lowercase snake_case roles remain accepted; kebab-case is not an
   alias and no normalization path exists.
6. An admitted role contributes exactly one nonempty path segment and cannot
   introduce separators, dot components, or alternate relative or absolute
   components into the canonical artifact path.
7. Decoding or constructing `CanisterRole` does not itself mark the value as
   admitted.
8. Every production path from externally supplied, persisted, or loaded role
   data to artifact-path derivation passes the canonical rule first.
9. `CanisterRole` storage and boundary encodings do not change.
10. Path claims remain lexical and do not imply symlink or reserved-name
    protection.
11. Lowercase-only admission prevents case variants from becoming distinct
    Canic identities.

### Slice B

1. Building one role never creates, rewrites, truncates, deletes, reports, or
   attests a root release-set manifest.
2. `CanisterArtifactBuildOutput` describes only artifacts owned by that role
   build.
3. Per-role build provenance contains no fleet manifest.
4. The normal writer consumes the exact successful role-labelled outputs
   collected by the current install-root invocation.
5. Every required builder receives its deployment-relevant inputs from the
   same validated snapshot later supplied to the writer.
6. No complete-build participant reloads configuration to resolve target
   identity, build selection, release version, or expected output location
   after that snapshot is established.
7. Missing, duplicate, unexpected, path-mismatched, unreadable, or unhashable
   current outputs prevent publication and leave an existing root manifest
   byte-for-byte unchanged, even when matching artifact paths exist on disk.
8. Each current output path must exactly equal its snapshot path before
   hashing; canonicalization and equivalence heuristics cannot admit a
   mismatch.
9. Coverage uses existing install-root target identity, including required
   targets that are not ordinary manifest members.
10. Successful manifest entry ordering comes from the validated snapshot and
    is independent of builder completion, output collection, or map iteration
    order.
11. Stale Candid and provenance sidecars never become current output or
   release-set completion evidence through file existence.
12. A direct or failed partial build leaves an existing root manifest
   byte-for-byte unchanged.
13. If that build replaced an artifact covered by the old manifest, existing
    hash validation rejects the old manifest.
14. The deployment-plan override remains an explicit complete manifest source.
15. Successful complete-build manifest JSON, hashes, staging, receipts, and
    activation remain unchanged.
16. No existence-check readiness writer or replacement build-session
    mechanism remains.
17. Concurrent mutation of one install root is explicitly unsupported.

## Sequence

1. Implement and prove canonical role-name admission.
2. Change install-root build collection and the writer input together, then
   hard-cut every single-role release-set side effect.
3. Run a closeout scan for duplicate role predicates, unvalidated role-to-path
   sinks, retired readiness symbols, optional manifest output, configuration
   reload, and filesystem-existence completion proof.
4. Close 0.91; do not extend the line by accretion.

## Targeted Validation

Run only focused checks for the touched owners.

Slice A cases include:

- valid lowercase snake_case roles with single underscores and optional
  digits after the first character;
- empty, overlong, kebab-case, mixed-case, leading-digit,
  leading/trailing/repeated-underscore, dotted, separator-bearing, non-ASCII,
  and whitespace roles;
- a manifest role such as `../sentinel` rejected before artifact access; and
- config, fleet mutation, and manifest use of the same typed policy.

Slice B cases include:

- stale artifacts for roles A and B with current output only for A;
- duplicate A with missing B;
- unexpected output C;
- a path-mismatched, unreadable, and unhashable current output;
- a stale Candid sidecar absent from the current output;
- a direct build replacing A while an old root manifest exists;
- old-manifest bytes unchanged after partial failure;
- old-manifest hash rejection after a covered artifact changes; and
- one successful complete invocation producing the unchanged manifest shape
  and entry order from differently ordered output collections.

Run targeted core config, host fleet-mutation, release-set manifest,
canister-build, build-provenance, and install-root sequencing tests. Then run:

- `cargo check -p canic-core -p canic-host -p canic-cli`; and
- targeted Clippy for those three packages only if Rust implementation files
  change in the same slice.

Do not run the full workspace, broad PocketIC, deployment, or release matrix.
The maintainer owns those gates during deployment and publication.

## Completion Boundary

0.91 is complete when both hard cuts pass their targeted tests and the retired
paths are absent. Further identifier cleanup, cross-platform path policy,
concurrent-writer safety, artifact restoration, or build-pipeline refactoring
requires new evidence and a separate design.
