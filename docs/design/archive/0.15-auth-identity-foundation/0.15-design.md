# Canic Design Proposal: Unified Auth Identity Foundation

- **Version:** `0.15`
- **Status:** `Draft`
- **Author:** `Canic Core`
- **Scope:** unified caller/subject model, delegated-session hardening, access-boundary identity rules

## 1. Motivation

Canic already supports several authentication-capable mechanisms:

- delegated certificates and delegated tokens
- root role attestations
- root capability proofs / delegated grants
- caller and topology trust predicates

These mechanisms are individually strong, but their identities are easy to conflate at integration boundaries. In particular, the system must clearly distinguish:

- the transport caller supplied by the IC runtime
- the authenticated subject used for application/user authorization
- subjects embedded in signed auth artifacts such as delegated tokens, role attestations, and capability proofs

`0.15` defines one explicit identity model so these layers are no longer ambiguous, and so frontend-driven delegated sessions can be supported without weakening infrastructure trust boundaries.

## 2. Design Goals

`0.15` must provide:

1. explicit identity layers at all auth boundaries
2. strict separation between infrastructure trust checks and application/user auth checks
3. temporary delegated-session support for wallet-driven frontend flows
4. secure delegated-session bootstrap using a verified delegated token or equivalent cryptographic proof
5. bounded, expiring, revocable delegated sessions
6. fail-safe fallback to raw caller when delegated-session state is absent, expired, or invalid
7. regression-proof guard semantics so delegated-session behavior affects only application-authenticated paths

## 3. Non-Goals

`0.15` does not include:

- global replacement of `msg_caller()`
- delegated-subject substitution for infrastructure or topology trust
- rewrites of role-attestation semantics
- rewrites of capability / delegated-grant semantics
- redesign of shard-to-root delegation-routing semantics
- production exposure of test-only auth bootstrap hooks

Any change to role-attestation or capability subject semantics requires a separate design review.

## 4. Canonical Identity Model

Every request identity must be interpreted through explicit layers:

| Layer | Meaning | Typical Source |
| --- | --- | --- |
| `transport_caller` | Raw ingress/inter-canister caller (`msg_caller()`) | IC runtime |
| `authenticated_subject` | Resolved application/user subject | raw caller or delegated session |
| `session_subject` | Delegated session stored subject | delegated-session state |
| `token_subject` | Delegated token `claims.sub` | verified delegated token |
| `attested_subject` | Role attestation payload subject | signed role attestation |
| `capability_subject` | Capability proof / delegated grant subject | capability proof payload |

### 4.1 Core Rule

- `transport_caller` remains authoritative for infrastructure trust.
- `authenticated_subject` is authoritative only for application/user auth where `authenticated(...)` semantics are intended.

### 4.2 Allowed Divergence

`authenticated_subject` may differ from `transport_caller` only in wallet-driven user-authenticated flows, and only when a valid delegated session is active.

No other identity layer is implicitly substituted.

### 4.3 Identity-Source Tagging

Resolved authenticated identity should carry an explicit source tag:

- `RawCaller`
- `DelegatedSession`

This tag exists to improve auditability, logging, and reasoning. It does not change authorization semantics by itself.

## 5. Security Invariants

The following invariants are mandatory.

### 5.1 Raw-Caller Authority Invariants

`transport_caller` remains authoritative for:

- controller checks
- root checks
- parent / child checks
- subnet-registration trust checks
- canister-to-canister infrastructure auth
- proof storage / provisioning trust checks
- capability and topology trust predicates

No delegated session, token subject, attested subject, or capability subject may override `transport_caller` for these checks.

### 5.2 Delegated-Session Invariants

- delegated-session subjects must be user principals only
- delegated-session subjects must never be interpreted as canister principals in any authorization path
- delegated-session subjects must never be interpreted as canister, root, controller, parent, child, subnet, or other infrastructure identities
- delegated sessions must be temporary, bounded, and revocable
- delegated-session bootstrap must be cryptographically gated
- session expiration must never outlive delegated-token validity
- delegated-session resolution must fail safe to raw caller

### 5.3 Artifact-Verification Invariants

Delegated token, role attestation, and capability verification remain cryptographically strict, including:

- issuer / root authority validation
- signature validation
- audience validation
- scope validation
- time-window validation
- current-proof / rotation validation where applicable

`0.15` changes identity resolution boundaries, not cryptographic verification strictness.

## 6. Delegated Session Model

Delegated sessions are wallet-scoped per canister.

### 6.1 Session Semantics

- one active delegated session per wallet principal per canister
- creating a new session replaces the prior session for that wallet atomically
- session persists across upgrades until expiry
- expired sessions are ignored automatically
- explicit clear / logout is supported
- storage must be bounded and pruneable

### 6.2 Minimum Session Record

Required fields:

- `wallet_pid`
- `delegated_pid`
- `issued_at`
- `expires_at`

Optional metadata may include:

- `identity_source`
- `session_id`
- bootstrap provenance / token metadata for auditing

### 6.3 Subject-Class Restriction

Delegated-session subjects must be rejected if they are classified as obvious non-user principals, including at minimum:

- anonymous principal
- management canister principal
- known self/root/parent/subnet/prime-root identities
- known registered canister identities

If principal classification is imperfect, the implementation must choose the safest practical rejection rule available in-repo and document it explicitly.

`0.15` does not require perfect principal taxonomy. It requires conservative rejection of any principal that is known or reasonably inferred to be infrastructure- or canister-associated. When classification is uncertain, the implementation should prefer rejection over acceptance.

## 7. Secure Bootstrap Flow

Delegated-session creation must require strong proof.

### 7.1 Canonical Bootstrap Flow

1. frontend obtains or carries a valid delegated token
2. canister verifies token cryptographically against the current delegation material
3. bootstrap request enforces `delegated_subject == token.claims.sub`
4. delegated subject is checked against user-only subject-class rules
5. bounded session record is stored if all checks pass

Any invalid bootstrap input fails closed.
Token verification before session write is mandatory.

### 7.2 Session TTL Clamp

Session expiry must be clamped as follows:

```text
session_expires_at = min(
  token_exp,
  configured_max_ttl_boundary,
  requested_ttl_boundary_if_present
)
```

Operationally, delegated-session lifetime can never exceed delegated-token lifetime.

### 7.3 Bootstrap Policy Boundary

`0.15` standardizes token-gated session bootstrap.

Additional bootstrap policy tuning, such as dedicated bootstrap scopes or richer provenance metadata, is allowed as a follow-up hardening pass, but the cryptographic gate itself is mandatory in `0.15`.

Delegated-session bootstrap is an application-auth facility for wallet-driven user flows. It must not be used to establish, replace, or reinterpret infrastructure identity.

## 8. Access and Macro Integration

Access context must carry both identities explicitly:

- `caller` => raw transport caller
- `authenticated_caller` => resolved authenticated subject

### 8.1 Guard Semantics

- caller and topology predicates consume raw caller
- `authenticated(...)` consumes authenticated subject

In `0.15`, `authenticated(...)` is the canonical consumer of `authenticated_caller`. Other auth paths remain bound to their existing raw-caller or artifact-specific semantics unless explicitly revised in a later design.
Delegated sessions do not replace delegated-token verification: endpoints guarded by `authenticated(...)` still require a delegated token, and session state only substitutes the resolved authenticated subject.

### 8.2 Hard Boundary

Delegated-session resolution must affect only the application-authenticated subject path. It must not mutate, reinterpret, or shadow raw caller semantics elsewhere.

This preserves infrastructure trust boundaries while enabling delegated user sessions for app-authenticated endpoints.

## 9. Role Attestation and Capability Boundaries

In `0.15`, role-attestation and capability verification remain raw-caller-bound.

Delegated-session resolution must not affect:

- role-attestation subject binding
- capability subject binding
- capability caller / topology checks
- root capability trust predicates

This is a deliberate boundary. Any change to these paths is deferred and must be reviewed independently.

## 10. Hardening Requirements

`0.15` hardening direction includes:

- explicit identity-source tagging (`RawCaller` vs `DelegatedSession`)
- strict delegated-subject class rejection for obvious infra/canister principals
- bounded session storage with expiry-aware pruning behavior
- regression tests proving delegated-session impact is limited to `authenticated(...)`
- regression tests proving raw caller predicates are unchanged
- negative tests for invalid bootstrap inputs:
  - mismatched token subject
  - expired token
  - invalid token
  - rejected delegated subject class
  - excessive requested TTL
- explicit logging and metrics for:
  - session created
  - session replaced
  - session expired
  - session cleared
  - fallback to raw caller
  - rejected delegated subject bootstrap

## 11. Compatibility and Migration Notes

`0.15` is intended to preserve existing auth capabilities while making identity boundaries explicit.

Migration expectations:

- legacy delegated-caller compatibility shims are not retained; `0.15` relies only on delegated-session semantics and must not reintroduce persistent caller override behavior
- raw caller trust predicates must preserve prior behavior
- delegated-session support must not require rewriting existing capability or attestation consumers
- any test-only bootstrap or delegation-material install paths must remain clearly non-production and must be compile-time separated from normal builds (no runtime compatibility endpoint fallback)
- current implementation uses compile-time cfg-gating (`canic_test_delegation_material`) for delegation-material install hooks; PIC integration builds set `CANIC_TEST_DELEGATION_MATERIAL=1`, and normal builds leave it unset so the hook is not compiled

## 12. Open Issues and Deferred Follow-Ups

The following items are intentionally deferred from `0.15` core semantics:

- dedicated bootstrap scope policy, if stricter than token-gating alone
- deeper principal-class policy beyond the current conservative rejection rules
- capability / delegated-grant semantic review
- role-attestation semantic review beyond preserving raw-caller binding
- explicit cleanup of ambiguous shard-to-root delegation-routing semantics
- continued audit to ensure test-only delegation-material hooks remain compile-time separated from normal builds

These items may be addressed in follow-up releases, but agents must not reinterpret them as already approved by `0.15`.

### 12.1 Post-`0.15` Addendum: Tier 1 Delegation Provisioning and Verifier Readiness

This addendum captures the current delegated-auth Tier 1 flow, the confirmed
`project_hub`-class verifier failure mode, and a staged hardening plan.

Status:

- canonical provisioning contract completed (Phase 1)
- hardening baseline completed (Phase 2A/2B/2C):
  - root-side verifier-target validation (registered, non-root, non-shard)
  - role-labeled provisioning push metrics (`signer` / `verifier`)
  - PIC end-to-end test covering issuance -> verifier local verify -> session bootstrap -> authenticated guard success (deterministic test-material path under `canic_test_delegation_material`)
- Phase 3 explicitly deferred to
  `docs/design/0.16-delegation-proof-evolution/0.16-design.md`

Scope note:

- this monorepo does not currently define a concrete `project_hub` canister;
  analysis applies to any non-root canister endpoint guarded by
  `auth::authenticated(...)` and verifying delegated tokens locally.

#### Current flow

Token issuance on `user_shard` (canonical signer path):

1. `DelegationApi::issue_token` calls `ensure_signing_proof`
   (`crates/canic-core/src/api/auth/mod.rs`).
2. Missing/stale proof triggers `setup_delegation`, which builds
   `DelegationRequest` via `delegation_request_from_claims`.
3. `request_delegation` calls root `canic_request_delegation`
   (`protocol::CANIC_REQUEST_DELEGATION`).
4. Root handles request through `RootResponseWorkflow::response` ->
   `execute_issue_delegation`
   (`crates/canic-core/src/workflow/rpc/request/handler/execute.rs`).
5. Root executes `DelegationWorkflow::provision`, which:
   - issues proof (`issue_delegation`)
   - pushes signer proof to `signer_targets`
   - pushes verifier proof to `verifier_targets`
   (`crates/canic-core/src/workflow/auth.rs`).
6. Target canisters receive:
   - `canic_delegation_set_signer_proof`
   - `canic_delegation_set_verifier_proof`
   (generated by `crates/canic/src/macros/endpoints.rs`), both calling
   `DelegationApi::store_proof`.
7. `store_proof` is root-caller-only, verifies proof and stores it as active
   local proof (`DelegationStateOps::set_proof_from_dto`).
8. Signer validates required verifier fanout with
   `ensure_required_verifier_targets_provisioned`, then reloads local proof
   via `require_proof`, then signs token.

Token verification on `project_hub` (or any verifier canister):

1. Endpoint guard `auth::authenticated(...)` is expanded by macros into
   `access::expr` evaluation (`crates/canic-dsl-macros/src/endpoint/expand.rs`).
2. `AuthenticatedEvaluator` calls `access::auth::delegated_token_verified`
   (`crates/canic-core/src/access/expr.rs`).
3. `delegated_token_verified` decodes ingress arg0 as `DelegatedToken` via
   `delegated_token_from_args` (`crates/canic-core/src/access/auth.rs`).
4. `DelegatedTokenOps::verify_token` performs structure/time/scope/audience checks
   and calls `verify_current_proof` (`crates/canic-core/src/ops/auth/token.rs`).
5. `verify_current_proof` compares token proof against locally stored active proof
   only (`DelegationStateOps::proof_dto`), failing closed on mismatch/missing.

Delegated-session bootstrap on target canister:

1. Canister-specific bootstrap endpoint calls
   `DelegationApi::set_delegated_session_subject`.
2. Bootstrap verifies token locally first using `DelegatedTokenOps::verify_token`.
3. On success, bootstrap enforces `delegated_subject == token.claims.sub`,
   applies TTL clamp, replay policy, then stores delegated session.

`authenticated(...)` resolution after bootstrap:

1. Macro access stage resolves identity via
   `resolve_authenticated_identity(msg_caller())`.
2. If a valid delegated session exists, `authenticated_caller` becomes delegated
   subject; otherwise it remains raw caller.
3. `authenticated(...)` still requires a `DelegatedToken` in ingress arg0.
4. Raw transport-caller predicates (`is_root`, `is_parent`, subnet checks, etc.)
   remain bound to raw caller and are not rewritten by delegated session.
5. Delegated sessions do not provide tokenless auth; they only allow bounded
   subject substitution after local token verification.

#### Observed failure mode

Confirmed failure path for `project_hub` Tier 1 auth (pre-fix canonical flow):

1. signer sends canonical request with `aud` but `verifier_targets = []`
   (`include_root_verifier = true`).
2. root provisions signer proof, but audience verifier canister(s) may receive no
   verifier proof push.
3. signer `setup_delegation` historically ignored
   `DelegationProvisionResponse.results`, then retried only local `require_proof`.
4. token issuance succeeds on signer.
5. verifier later receives token and verifies locally; `verify_current_proof`
   fails with:
   - `ProofUnavailable` when no local proof exists
   - `ProofMismatch` when stored active proof is for a different shard/cert.

Related fail-closed verifier denials are still expected and correct:

- `SelfAudienceMissing` when token audience does not include local verifier
  canister.
- token/session bootstrap denial when local proof material is absent or mismatched.

#### Root cause

The system was mixed and inconsistent:

- signer proof path was partially self-healing:
  `ensure_signing_proof -> setup_delegation`.
- verifier proof path was push-only via root fanout
  (`DelegationWorkflow::provision`).
- canonical signer path did not enforce verifier fanout completeness.

Additional structural constraint:

- verifier state is single-active-proof (`proof()` / `set_proof()` overwrite),
  so multi-shard verifier scenarios can produce proof replacement/mismatch unless
  storage model is evolved.

#### Design options

Option A: strict push-only provisioning (no verifier pull)

- keep local verification fail-closed
- require canonical issuance to fail if required verifier pushes fail
- requires external repair/prewarm if verifier misses material

Option B: explicit prewarm/repair flow outside auth path

- keep auth hot path local and fail-closed
- add explicit operator/workflow repair endpoint/job
- clearer trust boundary than implicit verifier-side self-heal

Option C: bounded on-demand verifier refresh on miss

- verifier attempts one refresh on `ProofUnavailable`/selected misses
- improves liveness, increases auth-path complexity and root-call coupling
- must remain bounded and auditable to avoid retry amplification

Option D: hybrid (recommended long-term)

- Phase 1 strict push contract + fail-closed local verify
- Phase 2 explicit repair/prewarm for ops
- Phase 3 optional follow-ups are deferred to
  `docs/design/0.16-delegation-proof-evolution/0.16-design.md`

#### Recommended plan

- files/modules:
  - `crates/canic-core/src/api/auth/mod.rs`
  - `crates/canic-core/src/ops/runtime/metrics/auth.rs`
  - `crates/canic-core/tests/pic_role_attestation.rs`
- changes:
  - emit structured logs for derived required verifier targets and provisioning
    fanout outcomes
  - add explicit metrics for required-target missing/failed statuses
  - add/keep regression proving verifier fails closed when proof missing
- why:
  - confirm the exact failure class before and after rollout
- risks:
  - metric-cardinality drift if labels are too granular
- validation:
  - targeted PIC tests for missing verifier proof
  - metric assertions for provisioning failure counters

Phase 1: minimal correctness fix (canonical contract completion)

- files/modules:
  - `crates/canic-core/src/api/auth/mod.rs`
  - `crates/canic-core/src/api/auth/tests.rs`
  - `crates/canic-core/tests/pic_role_attestation.rs`
  - test canister stubs under `crates/canic-core/test-canisters/*`
- changes:
  - derive `required_verifier_targets` from `aud` with strict validation and
    dedupe
  - exclude root from derived targets (root controlled only by
    `include_root_verifier`)
  - exclude signer self from verifier targets (signer handled through
    signer-target path)
  - fail issuance when any required verifier target result is missing/non-`Ok`
  - remove runtime compatibility fallback for test-only material-install hooks;
    compile these hooks only under `canic_test_delegation_material`
- why:
  - signer success must imply required verifier provisioning completeness
- risks:
  - previously tolerated invalid audiences now fail closed at issuance time
  - rollout can expose misconfigured target registration quickly
- validation:
  - unit tests for derivation + fanout result enforcement
  - PIC test confirming install hooks are not compiled in normal builds
  - workspace/package tests

Phase 2: hardening and cleanup

  - formalize required vs optional verifier-target semantics in request/response
  - add root-side policy validation for verifier targets to mirror signer-side

  - add richer fanout observability (attempted/ok/failed/missing by role)
  - add integration test for cross-canister signer-success/verifier-readiness

- why:
  - make provisioning guarantees explicit and machine-checkable end to end
- risks:
  - DTO compatibility changes if request semantics evolve
  - increased complexity in capability handling and tests

  - compatibility tests for request/response decoding
  - deterministic PIC flow tests across signer and verifier canisters

Phase 3: deferred to `0.16` design scope

- remaining optional follow-ups are tracked in:
  - `docs/design/0.16-delegation-proof-evolution/0.16-design.md`

#### Validation matrix

| Scenario | Expected result | Primary validation |
| --- | --- | --- |
| signer issues token with valid required verifier targets | issuance succeeds | `api/auth` unit + PIC end-to-end issuance |
| `aud` includes invalid canonical verifier target | issuance fails closed (`InvalidInput`) | `derive_required_verifier_targets_*` tests |
| required verifier target push returns `Failed` | issuance fails closed (`Internal`) | `ensure_required_verifier_targets_provisioned_*` tests |
| required verifier target result missing | issuance fails closed (`Internal`) | `ensure_required_verifier_targets_provisioned_*` tests |
| verifier receives token without local proof | auth fails closed (`ProofUnavailable`) | PIC delegated-token verification tests |
| verifier has different active proof | auth fails closed (`ProofMismatch`) | PIC/ops auth verification tests |
| bootstrap token invalid/missing local proof | bootstrap fails closed before session write | PIC delegated-session bootstrap tests |
| delegated session active | affects only `authenticated(...)` subject path | PIC session boundary tests |
| raw caller predicates under delegated session | unchanged and fail/allow by raw caller only | PIC raw-caller guard tests |
| normal build without test cfg | test install hook not compiled | PIC normal-build hook-absence test |

#### Open questions

1. Is `SubnetRegistryOps::is_registered` sufficiently strict for
   `is_valid_verifier_target`, or should verifier eligibility be role-scoped?
2. Do we need first-class optional audiences/optional verifier targets, or should
   all canonical `aud`-derived verifier targets remain required?
3. Should verifier-miss recovery remain explicit prewarm/repair only, or do we
   add bounded on-demand refresh in a later phase?
4. How should single-active-proof storage migrate if multi-shard verifier support
   is required?
5. Is there a retained "session-agent / no-bootstrap" consumer path to support?
   If not, it should be removed from docs/plans to prevent policy ambiguity.

Design constraints affirmed by this plan:

- delegated-token verification remains purely local and fail-closed
- no weakening of trust boundaries for raw caller predicates
- explicit bootstrap remains required per canister for delegated-session state
- root provisioning semantics remain primary; optional bounded refresh is a
  follow-up, not part of minimal correctness

## 13. Phased Delivery

`0.15` implementation should remain phased:

1. identity contract and delegated-session state foundations
2. secure bootstrap gating and expiry clamp
3. integration coverage for caller-vs-subject boundaries
4. hardening of test-only hooks and negative-path validation
5. deferred follow-up audits for routing and capability/attestation semantics

Tier 1 delegation provisioning follow-ups are tracked separately in `12.1`.
For `0.15`, execution scope is Phase 0-2 only; all Phase 3 items are deferred
to `docs/design/0.16-delegation-proof-evolution/0.16-design.md` and should be
implemented without weakening local verification or fail-closed behavior.

## 14. Acceptance Criteria

`0.15.0` is acceptable when all of the following hold:

- delegated-session bootstrap is token-gated
- delegated-session subject must equal verified token subject
- delegated-session subject-class rejection is enforced
- session expiry is correctly clamped
- raw-caller infrastructure predicates remain unchanged
- delegated session affects `authenticated(...)` paths only
- no raw-caller infrastructure or topology predicate is influenced by delegated-session state
- role-attestation and capability flows remain raw-caller-bound
- session storage is bounded, pruneable, and revocable
- documentation and status tracking are versioned and explicit

No further auth-semantic changes are part of `0.15` without a new design update.
