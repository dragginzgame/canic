## Findings

### CANIC-082-CANISTER-DTO-001

Severity: P2
Category: dto_boundary
Owner: `canic-core::ops::ic::mgmt::types` and `canic-core::dto::canister`
Current location:
`crates/canic-core/src/ops/ic/mgmt/types.rs` and
`crates/canic-core/src/dto/canister.rs`
Intended location: `crates/canic-core/src/domain/canister.rs`

Evidence:
Ops and DTO each defined canister status and log-visibility enums with the same
semantic variants, while `ops::ic::mgmt::status_settings` converted between
the duplicate values for canister status responses.

Risk:
The same management-status semantics were duplicated across ops and DTO
boundaries, creating drift risk between runtime management observations and
boundary status responses.

Decision:
Fixed with a no-behavior-change ownership move. The enums now live in
`domain::canister`, while ops and DTO modules publicly re-export them so
existing API paths and serialized shapes remain stable.

Regression test:
`dto::canister::tests::canister_status_enums_roundtrip_candid_through_dto_path`
proves the DTO re-export path round-trips the moved enums through Candid.

## Fixes

- Added `domain::canister` as the owner for canister status and log-visibility
  values.
- Re-exported the same values from existing DTO and management-ops paths.
- Removed duplicate ops-to-DTO conversion helpers for shared canister status
  values.

## Deferred

- Raw management-canister infra types remain separate.
- Canister status DTO structs and lifecycle management behavior remain
  unchanged.
