## Findings

### CANIC-082-CYCLE-DTO-001

Severity: P2
Category: dto_boundary
Owner: `canic-core::dto::cycles`
Current location: `crates/canic-core/src/dto/cycles.rs`
Intended location: `crates/canic-core/src/domain/cycles.rs`

Evidence:
`ops::storage::cycles` used `dto::cycles::CycleTopupEventStatus` to create and
project persisted cycle top-up event records.

Risk:
The DTO module was acting as the semantic owner for an internal event-domain
status, making storage projection code depend on a passive boundary module for
operation-state meaning.

Decision:
Fixed with a no-behavior-change ownership move. The enum now lives in
`domain::cycles`, while `dto::cycles` publicly re-exports it so existing API
paths and serialized shapes remain stable.

Regression test:
`dto::cycles::tests::reexported_topup_status_roundtrips_through_candid` proves
the public DTO path still round-trips the re-owned value through Candid.

## Fixes

- Added `domain::cycles` as the owner for `CycleTopupEventStatus`.
- Re-exported the same value from `dto::cycles`.
- Updated storage ops to import the domain owner.
- Added a Candid roundtrip regression test through the DTO path.

## Deferred

- Cycle tracker and cycle top-up event response structs remain DTO-owned.
- Stable cycle event records remain unchanged.
