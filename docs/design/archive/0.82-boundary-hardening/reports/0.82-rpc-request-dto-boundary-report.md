## Slice Contract

Boundary:
RPC request and response structs/enums are wire DTOs. RPC request ops own typed
dispatch helpers and dispatch errors, not DTO re-export ownership.

Current owner:
`canic-core::ops::rpc::request` re-exported `dto::rpc` request and response
shapes alongside ops dispatch helpers.

Intended owner:
`canic-core::dto::rpc` owns request and response DTOs.
`canic-core::ops::rpc::request` owns `RequestOps`, typed RPC adapters, and
`RequestOpsError`.

Files/modules in scope:

- `crates/canic-core/src/ops/rpc/request/mod.rs`
- `crates/canic-core/src/ops/rpc/request/dispatch.rs`
- `crates/canic-core/src/ops/rpc/mod.rs`
- Workflow placement modules that consumed `CreateCanisterParent` through ops.

Files/modules explicitly out of scope:

- RPC endpoint behavior.
- RPC Candid request/response shapes.
- Capability envelope schemas.
- Replay metadata, replay receipts, or replay policy.
- Public CLI, JSON, deployment-truth, or evidence surfaces.

Public surfaces affected:
None intended. The affected imports are inside `canic-core`, whose `ops` module
is crate-internal.

Serialized surfaces affected:
None. RPC DTO definitions remain unchanged in `dto::rpc`.

Commands/endpoints affected:
None.

Expected behavior change:
None.

Expected non-changes:

- RPC dispatch still attaches root request metadata in the same places.
- Capability metadata and structural capability proof behavior are unchanged.
- Create, upgrade, recycle, and cycles RPC request/response DTOs keep the same
  Candid shapes.
- Placement workflows still call `RequestOps` for canister creation.

Validation:

- `cargo fmt --all -- --check` passed.
- `cargo check --locked -p canic-core --lib` passed.
- `cargo test --locked -p canic-core rpc --lib` passed.
- `cargo test --locked -p canic-core replay --lib` passed.

Rollback plan:
Restore the `dto::rpc` re-exports in `ops::rpc::request`, then move the touched
workflow and ops imports back through the ops request namespace.

Follow-ups:

- Keep `RequestOps`, typed RPC adapters, and `RequestOpsError` in
  `ops::rpc::request`.
- Keep RPC DTO definitions and request-family helpers in `dto::rpc`.
- Avoid using ops modules as DTO namespaces in future placement/workflow code.

## Findings

### CANIC-082-RPC-REQUEST-DTO-001

Severity: P2
Category: dto_boundary
Owner: `canic-core::ops::rpc::request`
Current location: `crates/canic-core/src/ops/rpc/request/mod.rs`
Intended location: `crates/canic-core/src/dto/rpc.rs`

Evidence:
`ops::rpc::request` re-exported RPC request and response DTOs, and placement
workflow modules imported `CreateCanisterParent` through that ops namespace.

Risk:
The ops module looked like the semantic owner of passive RPC wire data,
blurring the boundary between DTO contracts and dispatch mechanics.

Decision:
Fixed with a no-behavior-change import cleanup. RPC DTOs are now imported from
`dto::rpc`, while `ops::rpc::request` exposes only request dispatch helpers and
dispatch errors.

Regression test:
Focused RPC tests continue to cover capability metadata, structural capability
proof decisions, typed request dispatch, request-family classification, and
root capability replay behavior.

## Fixes

- Removed `dto::rpc` re-exports from `ops::rpc::request`.
- Updated RPC dispatch internals to import DTOs directly from `dto::rpc`.
- Updated placement workflow imports for `CreateCanisterParent` to use the DTO
  owner directly.

## Deferred

- RPC DTO schema and Candid shape remain unchanged.
- RPC dispatch behavior and capability envelope handling remain unchanged.
