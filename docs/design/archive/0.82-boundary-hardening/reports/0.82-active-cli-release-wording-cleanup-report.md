## Slice Contract

Boundary:
Active CLI help, active operator error text, and test-only diagnostic examples
should describe current command contracts instead of naming the release line
where the surface was first introduced.

Current owner:
Each command module owns its help and error text. A few active strings still
named the original release line for state audit, deploy plan, and runtime
inspect behavior.

Intended owner:
Current command modules describe the stable command contract directly. Release
numbers remain appropriate in design docs, changelogs, fixture versions, and
comments that intentionally refer to a historical design contract.

Files/modules in scope:

- `crates/canic-cli/src/state/mod.rs`
- `crates/canic-cli/src/deploy/command.rs`
- `crates/canic-cli/src/deploy/plan.rs`
- `crates/canic-cli/src/deploy/tests/plan.rs`
- `crates/canic-cli/src/inspect/mod.rs`

Files/modules explicitly out of scope:

- Changelogs and historical design documents.
- Test fixture package/version values.
- Comments that intentionally reference the design release contract.
- Command parser behavior and accepted/rejected command forms.

Public surfaces affected:
Only human-readable help/error wording. No command forms, flags, exit codes, or
report fields changed.

Serialized surfaces affected:
None.

Commands/endpoints affected:
None semantically. Help/error text is release-neutral.

Expected behavior change:
None.

Expected non-changes:

- `canic state` still rejects `--format json`.
- `canic deploy plan` report structure and status derivation are unchanged.
- `canic inspect deployment` still rejects ambiguous multi-canister role
  resolution.
- No compatibility aliases or command shims are added.

Validation:

- `cargo test --locked -p canic-cli inspect --lib` passed.
- `cargo test --locked -p canic-cli deploy --lib` passed.

- `rg -n "0\.79 operator planning report|0\.79 planner contract|not supported in 0\.81|0\.80 does not" crates/canic-cli/src crates/canic-host/src crates/canic-core/src -S` returned no stale active wording.
- `git diff --check` passed.

Rollback plan:
Restore the previous help/error strings if a maintainer intentionally wants
active CLI output to identify the release line that introduced a command
contract. That is not needed for current operator UX.

Follow-ups:

- Continue leaving historical release references in design docs and changelogs
  where they provide useful provenance.

## Findings

### CANIC-082-ACTIVE-CLI-RELEASE-WORDING-001

Severity: P3
Category: docs_help
Owner: command modules
Current location: active help/error strings
Intended location: release-neutral command-contract wording

Evidence:
Active CLI strings referred to "0.80", "0.79", and "0.81" when describing
current state manifest, deploy-plan, and inspect behavior.

Risk:
Stale release labels in active help/errors can make current supported behavior
look historical, experimental, or tied to an older release line.

Decision:
Fixed with release-neutral help/error wording. Historical references remain in
docs, changelogs, comments, and fixtures where they are intentional.

Regression test:
Covered by focused CLI tests and the stale-wording search.
