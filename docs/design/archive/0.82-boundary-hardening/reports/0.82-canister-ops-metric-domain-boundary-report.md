## Findings

### CANIC-082-CANISTER-OPS-METRIC-OPS-001

Severity: P2
Category: ops_ownership
Owner: `canic-core::ops::runtime::metrics::canister_ops`
Current location: `crates/canic-core/src/ops/runtime/metrics/canister_ops.rs`
Intended location: `crates/canic-core/src/domain/metrics.rs`

Evidence:
Canister-op operation, outcome, and reason values were defined in the
canister-op metrics recorder while lifecycle/provisioning workflows and the
lifecycle facade consumed them as shared metric classification values.

Risk:
Canister-op metric semantics could drift between workflow owners, lifecycle
facades, and public metric projection if the values remain owned by the metrics
recorder.

Decision:
Fixed with a no-behavior-change ownership move. The canister-op metric
dimension values now live in `domain::metrics`, while
`ops::runtime::metrics::canister_ops` publicly re-exports them so existing
imports remain stable. Runtime error-to-reason classification stays in the
metrics module because it depends on internal error categories.

Regression test:
Runtime metric tests continue to prove canister-op labels, counter keys, and
facade recording behavior are unchanged.

## Fixes

- Added canister-op metric dimension values to `domain::metrics`.
- Re-exported the same values from the existing canister-op metrics path.
- Updated lifecycle, provisioning, and management ops to import the domain
  owner directly.

## Deferred

- Canister-op metric storage remains in runtime metrics ownership.
- Canister lifecycle and provisioning behavior remain unchanged.
