## Slice Contract

Boundary:
Control-plane access to Canic's public CDK facade.

Current owner:
`canic_core::cdk` is a public Canic core surface. `control_plane_support::cdk`
also re-exported the whole CDK facade, including `api`, `structures`, `types`,
and `utils`, even though the control-plane crate only needed public CDK types.

Intended owner:
Control-plane code imports public CDK values from `canic_core::cdk` directly.
`control_plane_support` is reserved for mediating crate-private core owners and
for narrow hidden support boundaries.

## Findings

### CANIC-082-CONTROL-PLANE-CDK-SUPPORT-001

Severity: P3
Category: support_boundary
Owner: `canic_core::cdk`
Current location: `control_plane_support::cdk`
Intended location: direct public CDK owner imports

Evidence:
`control_plane_support::cdk` re-exported the full public CDK facade. In-repo
control-plane users only needed `Principal` and `TC`, and other control-plane
modules already imported public `canic_core::cdk` storage/type helpers
directly.

Risk:
A broad hidden support mirror makes the support facade look like an alternate
CDK API and can grow into a compatibility layer for public core surfaces.

Decision:
Fixed with a no-behavior-change support-boundary cleanup. Control-plane code
now imports public CDK types from `canic_core::cdk::types`, and the hidden CDK
support mirror was removed.

Regression test:
Covered by control-plane tests and clippy. No behavior-specific regression is
required because the concrete CDK types and runtime paths are unchanged.
