# 0.16 Trust-Chain Pressure Remediation

Focused follow-up plan for the `2026-03-24` recurring trust-chain and layer audits.

Primary source reports:

- `docs/audits/reports/2026-03/2026-03-24/token-trust-chain.md`
- `docs/audits/reports/2026-03/2026-03-24/layer-violations.md`

## Problem Summary

The current `0.16` auth work does **not** have a trust-chain correctness break.

The recurring score remains elevated because:

- trust-critical verification is still concentrated in a small number of files
- `DelegationProof` remains both a boundary DTO and a heavily reused internal shape
- `DelegatedTokenClaims` still spreads across too many internal helpers
- the trust-chain stage order is correct today, but not yet protected by an explicit regression seam

This is structural risk, not an acceptance of invalid tokens.

## Required Outcome

Reduce trust-chain risk from structural concentration/spread without weakening any existing invariants:

- keep verifier decisions local and fail-closed
- keep stage order fixed: `structure -> current_proof -> delegation signature -> token signature`
- keep proof install / refresh semantics explicitly local-proof-bound
- reduce cross-layer dependency pressure from auth DTOs

## Action Plan

### 1. Seal the Verification Pipeline

Create one explicit trust-chain verification entrypoint in `ops/auth` that owns the canonical order:

1. structure / authority / time / audience checks
2. verifier-local current-proof check
3. delegation signature verification
4. token signature verification

Implementation rule:

- downstream call sites should not reassemble this order ad hoc
- if a future change needs a different order, it must update the sealed entrypoint and its regression tests together

Why:

- this directly attacks the audit’s “stage-order drift” risk
- it makes trust-chain review cheaper and more deterministic

### 2. Add Stage-Order Regression Tests

Add tests that prove:

- `verify_current_proof` cannot be bypassed before signature acceptance
- bogus root/shard relationships fail before handler execution
- proof-miss / proof-mismatch paths stay fail-closed

Why:

- the recurring audit should not rely only on code reading to establish ordering safety

### 3. Stop Using `DelegationProof` as an Internal Default Shape

Introduce a storage-owned persisted proof shape under storage/model ownership and keep:

- DTO -> storage mapping in `ops/storage/auth/*`
- storage -> verifier-local view mapping in ops-owned adapters

Do not keep using `dto::auth::DelegationProof` as the implicit internal truth shape everywhere.

Why:

- this is the largest current cross-layer dependency center
- it aligns with the repo rule that DTOs are transfer-only and conversion belongs in ops

### 4. Narrow `DelegatedTokenClaims` Internal Spread

Where internal helpers need only part of the token claims, replace raw DTO usage with narrower ops-local inputs or internal view structs.

Examples:

- verifier context
- signature material input
- audience-binding input
- bootstrap/session validation input

Why:

- `DelegatedTokenClaims` is acting like an internal domain type in too many places
- this keeps DTO spread from re-forming after the auth split

### 5. Split `verify.rs` by Responsibility

Separate:

- cryptographic verification
- verifier-local proof-state enforcement
- attestation-key validity / related support checks

Do not split just for line count. Split by invariant boundary.

Why:

- trust-critical concentration is currently too high
- this makes ownership and future auditing clearer

### 6. Keep API Trust Boundaries Thin

`api/auth/*` should remain boundary mapping and explicit ingress policy only.

In particular:

- delegated-session bootstrap should delegate to sealed ops verification results
- API modules should not grow new trust-assembly logic
- future install/repair/provisioning work should keep proof-shape conversion in ops

Why:

- otherwise the trust-chain score just migrates from one hotspot file to several boundary files

## Recommended Execution Order

1. Seal verification pipeline.
2. Add stage-order regression coverage.
3. Introduce storage-owned proof shape and ops mappings.
4. Narrow `DelegatedTokenClaims` internal usage.
5. Split `verify.rs` by invariant boundary.
6. Re-run `token-trust-chain` and `layer-violations`.

## Success Criteria

This remediation is complete when all hold:

- trust-chain stage order is enforced through one canonical entrypoint
- regression tests fail if proof-state verification is bypassed
- `DelegationProof` no longer acts as the default internal shape across storage/ops/api/workflow
- `DelegatedTokenClaims` spread is reduced materially from the `2026-03-24` baseline
- trust-critical hotspot concentration in `ops/auth/token.rs` and `ops/auth/verify.rs` is reduced
- recurring `token-trust-chain` score drops below `6`

## Non-Goals

- no auth semantic weakening
- no verifier network refresh path on auth hot path
- no tokenless authenticated path
- no broad public API churn just to lower a score
