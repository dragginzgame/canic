# Canic Design Proposal: Delegation Proof Evolution

- **Version:** `0.16`
- **Status:** `Draft`
- **Author:** `Canic Core`
- **Scope:** deferred Tier 1 delegation Phase 3 work from `0.15`

## 1. Scope Origin

`0.15` completed Tier 1 delegation Phase 1/2 correctness and hardening.
All remaining Phase 3 follow-ups are deferred into this `0.16` design track.

Deferred items originate from:

- `docs/design/0.15-auth-identity-foundation/0.15-design.md` section `12.1`

## 2. Baseline Invariants (Must Not Change)

`0.16` follow-ups must preserve these invariants:

- delegated-token verification stays local and fail-closed
- `authenticated(...)` endpoints remain token-gated (no tokenless mode)
- delegated-session bootstrap remains explicit and verifies token before write
- raw-caller infrastructure predicates remain bound to transport caller
- root provisioning remains the primary readiness path
- token verification must bind to a deterministic verifier-local proof identity
- authentication outcome must be a pure function of `(token, verifier-local proof storage)`
- no external state, network call, or retry path may influence auth correctness for a given verifier decision

## 3. Problem Statement

Current verifier proof state is single-active (`set_proof` overwrite model).
That limits multi-shard verifier reliability:

- proof for shard `A` can be overwritten by later proof for shard `B`
- token from `A` can then fail verifier local checks with `ProofMismatch`

Operational recovery also remains mostly issuance-time push-only; there is no
formalized post-issuance repair/prewarm contract in the current design set.

Failure-mode terminology:

- `proof_miss`: `ProofKey` is absent from verifier-local proof storage
- `ProofMismatch`: `ProofKey` is present, but verification fails against the stored proof material

These failure classes are operationally distinct:

- `proof_miss` indicates prewarm/repair/readiness gaps
- `ProofMismatch` indicates correctness, binding, corruption, or rotation-handling errors

## 4. Goals

1. Define bounded multi-proof verifier storage keyed by shard/cert identity.
2. Define explicit operator/workflow repair-prewarm flow outside auth hot path.
3. Optionally define a bounded single-refresh verifier path only if required.
4. Keep security posture identical to `0.15` fail-closed local verification.
5. Specify rollout/migration and observability requirements for safe adoption.
6. Preserve deterministic behavior under shard churn, cert rotation, and verifier cold start.
7. Reduce trust-chain structural risk called out by the `2026-03-24` recurring audits without weakening verification semantics.

## 5. Non-Goals

- no weakening of local verification into network-dependent verification
- no network proof fetch/refresh on the `authenticated(...)` hot path
- no implicit delegated-session auth without delegated token
- no rewriting role-attestation or capability trust semantics in this phase
- no relaxation of root-only proof install trust boundary
- no unbounded verifier retry loops or auth-time retry amplification

## 6. Design Tracks

### 6.1 Multi-Proof Verifier State (Required)

- evolve verifier proof storage from `Option<DelegationProof>` to bounded map
- key by canonical proof identity `ProofKey = (issuer_shard_id, cert_hash)`
- require verification inputs to include or deterministically derive both `issuer_shard_id` and `cert_hash`
- preserve parallel cert rotation within a shard by allowing multiple proofs for the same shard under different `cert_hash` values
- include bounded capacity and deterministic eviction policy
- preserve strict proof match behavior at verify time

Rationale:

- `issuer_shard_id` alone does not safely support concurrent cert rotation
- `cert_hash` alone is not sufficiently operator-friendly for shard-scoped repair and debugging
- `(issuer_shard_id, cert_hash)` provides exact local lookup while remaining stable and auditable

Bounded-state requirements:

- capacity must be explicit, bounded, and sized as `O(shard_count * rotation_factor)`
- default capacity is `96` proofs
- capacity remains configurable by deployment policy, but must always remain bounded
- capacity is selected from explicit documented profiles, not arbitrary free-form tuning
- initial sizing target should assume approximately `2-3` concurrent proofs per shard
- eviction must be deterministic, local-only, and reproducible from the same sequence of reads/writes
- random eviction is forbidden
- pure time-based expiry as the only retention policy is forbidden
- runtime auto-resizing based on live traffic observations is forbidden
- if shard count is unknown at startup, default to the `standard` profile
- if shard count is unknown at startup, later capacity increase is allowed only through explicit config change

Active-proof definition:

- `active_proof` means a proof that has successfully verified at least one token within a bounded recent window `W`
- default `W` is `10 minutes`
- `W` must be configurable, bounded, and strictly non-zero
- `W` must be long enough to capture normal recent usage and short enough to avoid indefinite pinning of stale proofs

Required cache behavior:

- verifier storage uses a segmented hot/cold policy
- new proof installs enter the cold segment
- successful verification promotes a proof into the hot segment
- eviction removes entries from the cold segment first
- if the cold segment is empty, the least-recent hot entry is deterministically demoted before eviction
- recent successful proofs may use a short-lived protected window, but protection must remain bounded and deterministic
- repeated installation of the same `ProofKey` must result in one logical cache entry only
- repeated installation of the same `ProofKey` must not increase cache pressure beyond a single entry
- repeated installation of the same `ProofKey` must not perturb eviction ordering beyond normal single-entry insertion/update semantics

Eviction determinism scope:

- eviction decisions must be deterministic given the same sequence of proof installs and verification accesses
- eviction must not depend on wall-clock time
- eviction must not depend on concurrency timing
- eviction must not depend on non-deterministic iteration order

Operational constraints:

- cache pressure must not silently degrade into probabilistic auth success
- old and new proofs for the same shard must be able to coexist during cert rotation
- shard-cardinality growth must be observable through occupancy and unique-shard metrics
- configured capacity/profile must be logged at startup and exported as metrics/tags for auditability
- during cert rotation, at least two concurrent proofs per shard (old + new) must be retained until old-cert-bound tokens are guaranteed expired or otherwise no longer in circulation

### 6.2 Explicit Repair/Prewarm Flow (Required)

- add explicit flow to push/repair verifier material outside request auth path
- scope to operator/workflow driven mechanisms with clear authorization
- add observability for attempts, success, failures, and target coverage

Required model:

- repair/prewarm is push-based, not peer-fetched from the auth hot path
- flow must be idempotent for repeated install of the same `ProofKey`
- flow must support targeted install by verifier and by proof identity
- install intent must be explicitly classified (for example `prewarm`, `repair`, `rotation`)

Idempotency semantics:

- installing the same `ProofKey` multiple times must produce identical verifier state
- installing the same `ProofKey` multiple times must not create duplicate entries
- installing the same `ProofKey` multiple times must not amplify cache pressure beyond a single retained entry

Authorization requirements:

- authorization remains root-controlled
- any delegated operator/workflow surface must be explicitly scoped and auditable
- no trust expansion beyond the current root-only proof install boundary

Operational modes:

- `prewarm`: proactive install before rollout, scale-out, verifier activation, or expected shard traffic
- `repair`: reactive install after observed verifier miss, cold-start gap, or rotation drift

Cold-start guarantee:

- a verifier starting from empty proof state must not rely on `authenticated(...)` traffic to converge toward correctness
- verifier readiness must be established by prewarm or explicit repair flows
- auth traffic must observe current verifier readiness; it must not act as a background learning path

Observability requirements:

- `proof_install_attempt_total{result,...}`
- `proof_install_latency`
- `verifier_proof_coverage{shard,...}`
- `proof_cache_size`
- `proof_cache_evictions_total{reason,...}`
- `auth_failures_total{reason=...}`

Failure semantics:

- repair/prewarm install failure must be visible and alertable
- repair/prewarm install failure must not block unrelated system progress outside the targeted verifier-readiness path

### 6.3 Bounded Refresh on Miss (Optional)

- deferred by default; do not implement in initial `0.16` rollout
- only considered if production evidence shows explicit repair is insufficient
- single attempt only for selected verifier-miss classes
- must be auditable, bounded, and non-amplifying
- must not convert `authenticated(...)` into unbounded retry behavior

If this track is enabled later, all must hold:

- trigger only on verifier miss classes equivalent to `ProofMismatch` with absent `ProofKey`
- source of repair material must remain trusted control-plane infrastructure
- exactly one refresh attempt is allowed per auth decision
- no retry loop, background storm, or peer-to-peer verifier fetch path is introduced
- all refresh attempts emit structured metrics/events for auditability

### 6.4 Trust-Chain Pressure Reduction (Required Follow-Up)

The `2026-03-24` recurring `token-trust-chain` audit did **not** find a trust
correctness break, but it did score the current state as moderate structural
risk because trust-critical verification is still concentrated and auth DTOs
still spread across too many internal layers.

This track is therefore required for `0.16` closeout:

- seal trust-chain verification behind one canonical ops entrypoint
- add stage-order regression tests for `structure -> current_proof -> signatures`
- reduce internal dependence on `dto::auth::DelegationProof`
- reduce internal spread of `dto::auth::DelegatedTokenClaims`
- split trust-critical verification modules by invariant boundary, not by size alone

## 7. Rollout and Migration Requirements

- rollout must hard-cut to keyed proof storage only; runtime legacy fallback is forbidden
- deployment sequencing must preserve required proof material before auth traffic is shifted
- upgrade path must be deterministic and reversible by deployment policy, but reversibility must come from deployment control rather than in-process compatibility fallback
- metrics must expose cache occupancy/eviction and miss reasons
- integration tests must cover mixed-shard verifier traffic and rollover
- rollout must avoid regressions from `can verify` to `cannot verify` for already-installed verifier material

Recommended rollout shape:

- Phase 0: install keyed proof material and validate verifier observability in lower environments
- Phase 1: roll keyed-proof-only verifier behavior to production behind the rollout gates below
- Phase 2: remove any deployment tooling or documentation that still assumes the old single-proof model

Observation window requirements:

- lower environments require a minimum observation window of `24h`
- production rollout requires a minimum observation window of `48h`
- stronger production confidence may require `72h` when cert rotation cadence or shard churn justifies it
- the observation window is valid only if rollout thresholds hold continuously throughout the window
- time alone is insufficient; the window must exercise the required shard/rotation/cold-start conditions below

Required coverage within the observation window:

- full shard coverage: `unique_shards_seen == expected_shards`
- at least one cold-start, restart, or scale-out event
- representative steady-state and burst/imbalanced shard traffic
- for stronger production sign-off, at least one cert-rotation boundary or equivalent multi-cert coexistence evidence

Pre-cutover checks:

- proof miss rate remains below `0.01%`
- `ProofMismatch` does not regress relative to `0.15` baseline
- active-proof evictions remain at `0`
- cache `p95` utilization remains below `85%`
- repair success rate remains at or above `99.9%`
- verifier coverage includes the full expected shard set before cutover

## 8. Acceptance Criteria

`0.16` Phase 3 work is complete only when all hold:

- multi-shard verifier traffic does not fail due to single-proof overwrite
- provisioning repair path is explicit, authorized, and observable
- fail-closed local verification behavior is unchanged
- no tokenless path is introduced for `authenticated(...)`
- integration tests prove issuance -> verifier readiness -> bootstrap -> auth
- proof selection is deterministic via `(issuer_shard_id, cert_hash)` binding
- bounded cache behavior is deterministic and tested under rotation/churn pressure
- migration/rollout proves mixed old/new proof coexistence without verifier regression
- required alerts/metrics are present before production enablement

## 9. Resolved Design Decisions

1. Canonical proof identity is `ProofKey = (issuer_shard_id, cert_hash)`.
2. Bounded verifier storage uses deterministic hot/cold segmented retention with cold-first eviction.
3. Verifier capacity is static for a process lifecycle and uses documented profiles:
   - `small = 64` for `<= 16` shards with low rotation overlap
   - `standard = 96` as the default for approximately `<= 48` shards with normal rotation
   - `large = 160` for larger shard sets or higher rotation overlap
4. Profile selection is deterministic:
   - if `shard_count <= 16` -> `small`
   - else if `shard_count <= 48` -> `standard`
   - else -> `large`
5. Capacity override is allowed only upward from the selected profile minimum; downward override is forbidden.
6. Selected capacity/profile must be explicit in config, logged at startup, and exported via metrics.
7. Dynamic capacity resizing based on runtime observations is forbidden.
8. Explicit push-based repair/prewarm is the required operational model for `0.16`.
9. Optional bounded refresh remains deferred unless production evidence proves explicit repair is insufficient.
10. Release rollout requires all of these thresholds:
   - proof miss rate `< 0.01%`
   - no `ProofMismatch` regression
   - zero active-proof evictions
   - cache `p95` utilization `< 85%`
   - repair success rate `>= 99.9%`
   - full expected shard coverage
11. Hard-cut production rollout requires:
   - `>= 24h` continuous healthy observation in lower environments
   - `>= 48h` continuous healthy observation in production
   - at least one cold-start, restart, or scale-out event during the window
   - stronger `72h` production validation when cert rotation coverage is needed
12. `active_proof` means a proof used successfully within bounded recent window `W`, with default `W = 10 minutes`.
13. `proof_miss` and `ProofMismatch` are distinct failure classes and must be measured independently.
14. Cert rotation must retain old and new proof material concurrently until old-cert token validity is safely exhausted.
15. Cold-start verifier correctness must come from prewarm/repair, never from auth-traffic convergence.
16. Authentication correctness is a pure function of `(token, verifier-local proof storage)`.

## 10. Release-Gating Alerts

Hard release gates:

- proof miss spike
- `ProofMismatch` regression
- any active-proof eviction
- repair failure-rate regression
- cache saturation

Operational but non-blocking alerts:

- eviction-rate trend drift
- shard-skew growth
- cold-start miss bursts
- repair latency regression
- shard/cert churn-rate increase

## 11. Operational Defaults

- default capacity profile: `standard (96)`
- capacity profiles:
  - `small = 64`
  - `standard = 96`
  - `large = 160`
- minimum lower-environment observation window: `24h`
- recommended production observation window: `48-72h`
- production observation window must include full shard coverage and at least one cold-start, restart, or scale-out validation
- higher-confidence production sign-off should include at least one cert rotation boundary or equivalent multi-cert coexistence evidence
