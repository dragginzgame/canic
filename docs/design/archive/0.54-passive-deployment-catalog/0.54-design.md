# 0.54 Design: Passive Deployment Catalog

## Status

- Status: Implemented
- Line: `0.54.x`
- Theme: passive deployment catalog
- V1 posture: last small operator-visibility slice before stabilization

0.54 is intentionally narrow. It makes Canic easier to understand before v1,
without adding another future platform layer. The line is closed by:

```text
docs/audits/release-lines/0.54-closeout.md
```

## Summary

Canic already has the core v1 pieces:

- strict package `fleet.role` identity;
- declared and attached role lifecycle;
- metadata-driven `build!` and `start!`;
- deployment-target identity;
- deployment truth checks;
- passive adoption reports;
- stable evidence envelopes;
- build provenance;
- policy gates over saved evidence.

The remaining small operator question is:

```text
What deployment targets does this project already know about?
```

0.54 answers that with a read-only deployment catalog over existing local
deployment-target state:

```text
canic deploy catalog list
canic deploy catalog inspect <deployment>
```

That is all 0.54 should prove before v1.

## Why This Line

0.41 through 0.47 established deployment truth and deployment-target identity.
0.48 through 0.50 made setup and onboarding strict enough to be teachable.
0.51 through 0.53 made evidence stable enough for CI and policy checks.

Those lines created a lot of useful evidence, but they did not give operators a
compact local view of known deployment targets. A simple catalog closes that
gap without opening new authority surfaces.

This line is chosen over locks, signing, registry import, teardown, provider
wrappers, active adoption, or deployment groups because v1 needs a smaller
question:

```text
What does Canic know locally, and what must it not infer?
```

The answer should be visible and boring. If 0.54 grows into groups, lanes,
saved-evidence registries, lock scopes, or promotion semantics, it stops being
a v1 simplification line.

## V1 Product Boundary

Until v1, avoid new stable public DTO families unless they are required by an
implemented command.

For 0.54, the only new stable payload is:

```text
DeploymentCatalogReportV1
```

Catalog entries may exist as nested report data or internal structs, but they
are not promoted as a separate public schema in 0.54.0.

0.54 must not add:

- deployment groups;
- promotion lanes;
- saved evidence reference models;
- `last_*` evidence pointers;
- deployment locks;
- signing or attestations;
- provider-specific CI wrappers;
- artifact registry import;
- `wasm_store` retention or garbage collection;
- active adoption/import;
- controller mutation;
- topology mutation;
- install or upgrade authority;
- teardown;
- broad live verification.

Keep future architecture notes, but do not ship future-facing fields before the
workflow exists.

## Core Rule

The catalog is passive.

It may read:

- existing deployment-target local state;
- local file metadata needed to fingerprint that state;
- minimal project root/config context needed to locate that state.

It must not:

- mutate deployments;
- install or upgrade Wasm;
- attach topology;
- declare, rename, or attach roles;
- import adoption candidates;
- change controllers;
- acquire locks;
- sign evidence;
- register artifacts;
- run builds;
- crawl live deployments;
- create deployment truth;
- infer deployment targets from fleet template names;
- infer groups from names such as `prod`, `staging`, or `v2`.

Allowed writes are limited to an explicitly requested report output file.

## Local-State Source

0.54.0 catalog reports read only the deployment-target local state introduced
by 0.46 and guarded by 0.47:

```text
.canic/<network>/deployments/<deployment>.json
```

That state is the source for:

- deployment target name;
- fleet template, when recorded;
- network;
- root principal;
- root verification state.

The catalog must not infer deployments from:

- fleet template names;
- old fleet-named install state;
- package names;
- topology role names;
- saved evidence file names;
- deployment-like strings inside unrelated reports.

## Missing And Malformed State

Behavior must be deterministic:

- `canic deploy catalog list` with no deployment-target state returns an empty
  catalog with a warning, not an inferred catalog.
- `canic deploy catalog list` with one malformed deployment entry reports a
  warning for that entry and still returns valid entries.
- an unreadable state root fails as `invalid_input` when the directory exists
  but cannot be read.
- `canic deploy catalog inspect <deployment>` fails clearly for an unknown
  deployment target.
- `canic deploy catalog inspect <deployment>` reports malformed selected state
  as `invalid_input` or `blocked_by_policy`; it must not silently treat it as
  verified or fresh.

## Data Model

Stable payload schema:

```text
canic.deployment_catalog_report.v1
```

Suggested shape:

```text
DeploymentCatalogReportV1 {
  schema_version: 1,
  generated_at: string,
  project_root: string?,
  entries: Vec<DeploymentCatalogEntry>,
  warnings: Vec<EvidenceMessageV1>,
}

DeploymentCatalogEntry {
  deployment: string,
  fleet: string?,
  network: string?,
  root_principal: string?,
  root_verification: "unknown" | "not_verified" | "verified",
  local_state_ref: InputFingerprintV1?,
  warnings: Vec<EvidenceMessageV1>,
}
```

Rules:

- entries are sorted by deployment name;
- warnings use stable codes and stable subjects where practical;
- paths follow 0.51 path redaction rules;
- snapshotable output must not expose absolute home-directory paths;
- no field implies saved evidence freshness;
- no field implies install, promotion, teardown, lock, registry, or controller
  authority.

## CLI Shape

0.54.0 commands:

```text
canic deploy catalog list
canic deploy catalog inspect <deployment>
```

Output:

```text
--format text
--format json
--output <path>
```

Rules:

- text is the default operator view;
- raw `--format json` emits `DeploymentCatalogReportV1`;
- `--output <path>` writes only that requested report artifact;
- `--format` controls both stdout and `--output` serialization;
- catalog commands do not refresh live state.

`--format envelope-json` is intentionally not required for 0.54.0. The v1
value is operator clarity. If envelope output is added later, it must be
justified by a concrete CI/policy journey and must not add new catalog-side
evidence models.

## Proposed 0.54.0 Slice

Implement only:

- `DeploymentCatalogReportV1`;
- host-side read-only catalog builder over deployment-target local state;
- `canic deploy catalog list`;
- `canic deploy catalog inspect <deployment>`;
- text output;
- raw JSON output;
- `--output <path>`;
- deterministic sorting by deployment name;
- missing/malformed-state warnings;
- non-mutation tests.

Do not implement groups in 0.54.0.

## Patch Plan

Implemented sequence:

- `0.54.0`: passive deployment catalog list/inspect.
- `0.54.1`: docs, examples, and v1-readiness cleanup for catalog usage.
- `0.54.2`: closeout audit.

After 0.54, prefer v1 stabilization over new feature lines.

## Required Tests

Minimum tests:

- catalog builder lists deployment targets from existing deployment-target
  local state;
- catalog builder ignores old fleet-named install state;
- catalog list with no deployment-target state returns an empty report with a
  warning or documented success state;
- catalog entries are sorted deterministically by deployment name;
- catalog list ignores fleet-template names when no deployment-target state
  exists;
- catalog inspect uses deployment-target identity, not fleet-template identity;
- catalog inspect rejects a fleet-template name that is not a deployment
  target;
- catalog report includes deployment name, fleet, network, root principal, and
  verification state when available;
- unknown or malformed deployment state produces a warning instead of
  pretending success;
- malformed one deployment entry does not make another valid entry disappear;
- `canic deploy catalog list` defaults to text output;
- `canic deploy catalog list --format json` emits
  `DeploymentCatalogReportV1`;
- `--output` uses the selected `--format` serialization;
- `canic deploy catalog inspect <deployment>` succeeds for known deployment
  targets;
- `canic deploy catalog inspect <deployment>` fails clearly for unknown
  deployment targets;
- catalog commands write only explicit `--output` artifacts;
- catalog report does not include absolute home-directory paths in snapshotable
  output;
- catalog commands do not mutate config, topology, controllers, deployment
  state, artifacts, evidence inputs, package manifests, or canister lifecycle;
- command help uses deployment-target terminology, not fleet-template wording.

## Acceptance Criteria

0.54 is acceptable when:

- `canic deploy catalog list` exists;
- `canic deploy catalog inspect <deployment>` exists;
- catalog entries are derived from deployment-target state, not fleet-template
  names;
- text and JSON outputs are tested;
- `--output` follows the selected output format;
- output ordering is deterministic;
- the passive write boundary is tested;
- docs explain that catalog entries are not deployment truth and do not prove
  live freshness;
- no deployment group, promotion lane, saved evidence registry, lock, signing,
  teardown, registry import, active adoption, controller mutation, topology
  mutation, or install authority has been added.

## Deferred Until After V1

These may be useful later, but should not be part of the pre-v1 surface:

- deployment groups;
- promotion lanes;
- saved evidence reference catalogs;
- "latest evidence" selection;
- deployment locks;
- signing;
- registry import and retention;
- teardown;
- active adoption/import;
- broad live verification;
- provider-specific CI integrations.

When any of these return, require a concrete user journey and an implemented
command before adding stable public DTOs.

## Recommended Closeout Wording

0.54 adds a passive deployment catalog. Operators can list and inspect known
deployment targets from existing local deployment-target state. The catalog is
read-only: it does not create deployment truth, refresh live state, infer
deployments from fleet names, mutate topology, change controllers, install
Wasm, register artifacts, acquire locks, sign evidence, import adoption state,
or make saved evidence fresh.
