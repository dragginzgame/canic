# 0.49 Design: Role Lifecycle And Topology Attachment

## Status

Implemented design notes for the 0.49 line.

The implementation is closed by:

```text
docs/audits/release-lines/0.49-closeout.md
```

0.49 separates role declaration from deployment topology attachment so new and
renamed canisters can compile before placement, while deploy, install, check,
and deployment-truth flows remain topology-strict.

Core goal:

```text
A role can be declared and compiled before it is attached to deployment
topology, but only topology-attached roles can become deployment truth or
deployable artifacts.
```

0.49 is not a draft-mode line. It is a role lifecycle line.

## Scope Calibration

### Release Bar

- Introduce a first-class role declaration model.
- Allow non-root role crates to `cargo check` when the role is declared but not
  yet attached to subnet or pool topology.
- Keep `canic::build!()` strict about role identity, but stop requiring
  topology attachment for basic compilation.
- Require explicit topology attachment before `canic build`, deploy check,
  install, or deployment-truth flows treat the role as deployable.
- Add CLI workflow for scaffold, list, attach, and rename role lifecycle.
- Add a shared attached-role derivation so build, CLI role lifecycle,
  governance, and deployment truth agree on attachment state.
- Preserve root strictness.
- Preserve existing deployment safety invariants.

### Deferred

- Broad deployment catalog UX.
- Full live `canic deploy verify`.
- Teardown or test-deployment lifecycle.
- Root rotation.
- Delegated-token multi-role semantics.
- Role placement automation.
- Automatic role-to-pool inference.
- Sharding or scaling redesign.

## Problem

Today, `canic::build!()` performs two jobs at once:

1. Establish compile-time role identity.
2. Prove the role is attached to deployment topology.

The current flow is safe for release and deploy, but painful during incremental
development. A canister crate with:

```toml
[package.metadata.canic]
fleet = "toko"
role = "store"
```

could not `cargo check` before 0.49 unless `toko.store` already existed under
deployment topology in `canic.toml`.

This forces topology-first workflows:

```text
declare topology -> compile crate
```

when the desired development workflow is often:

```text
scaffold crate -> compile crate -> write code -> attach role to topology -> deploy
```

The audit found that this friction appears most sharply when:

- adding a new role after fleet creation;
- renaming an existing role;
- developing sharded service roles before final placement;
- iterating on canister code before topology is ready.

The tempting quick fix is:

```toml
[package.metadata.canic]
draft = true
```

That creates a package-local escape hatch that release, governance, and deploy
paths must police forever. The better model is to represent the real domain
distinction:

```text
role exists != role is attached to deployment topology != role is deployable
```

## Design Principles

### 1. Role Declaration Is Not Topology Attachment

A role declaration means:

- Canic knows this role exists.
- A package may claim this role.
- The crate may compile.

It does not mean:

- this role is placed on a subnet;
- this role is part of deployment truth;
- this role can be installed;
- this role has a shard or pool assignment;
- this role is release-safe.

### 2. Topology Attachment Is The Deployability Boundary

A role becomes deployable only when explicitly attached to deployment topology.

Topology attachment records:

- subnet;
- canister role;
- pool or sharding placement, if any;
- scaling policy, if any;
- deployment participation.

### 3. `build!()` Should Validate Identity, Not Placement

`canic::build!()` should validate that the package role is a declared role. It
should not require the role to be attached to subnet or pool topology merely to
compile.

Deployment artifact builds may still require attachment.

### 4. Deploy, Install, And Deployment Truth Stay Strict

No declared-only role may be treated as deployment truth. No declared-only role
may be installed. No declared-only role may be included in release-safe
deployment artifacts.

### 5. Root Remains Special

The root role defines fleet authority and receives root lifecycle and endpoints.
0.49 should not make root a loose compile-only role by default.

Root packages must remain explicitly configured and topology-valid.

### 6. No Silent Inference

Canic should not infer role identity from:

- package name;
- crate path;
- directory name;
- fleet name;
- scaffold naming convention.

Role identity remains explicit in package metadata and config.

### 7. Role Identity Is Fleet-Scoped

Canonical role identity is:

```text
fleet + role
```

Display and reference output may render this as:

```text
toko.store
```

but the internal/storage model should keep the two fields separate:

```rust
struct FleetRoleRefV1 {
    fleet: FleetName,
    role: CanisterRole,
}
```

In this design, `fleet` means the fleet template identity. It is not a
deployment target name.

The dotted form is display/audit syntax, not the stored role name. Canic should
not collapse `toko.store` into a single role string.

This preserves two important properties:

- runtime role checks can still reason about the local role name, such as
  `store`;
- fleet scope prevents collisions between roles such as `toko.store` and
  `payments.store`.

## Proposed Role Lifecycle

Initial lifecycle states:

```rust
enum RoleLifecycleV1 {
    Declared,
    Attached,
}
```

Future states may include:

- `Deprecated`
- `Draining`
- `Retired`

For 0.49, only declared and attached are required.

### Declared

A declared role:

- has an entry in `canic.toml`;
- has a package path;
- may compile;
- may appear in role listing;
- may be renamed;
- is not deployment truth;
- cannot install;
- cannot produce release deploy artifacts;
- cannot appear in a deployment plan as an installed role.

### Attached

An attached role:

- is declared;
- has subnet, pool, or topology placement;
- may participate in deployment truth;
- may be built as a deployment artifact;
- may be installed, subject to ordinary checks;
- may appear in deployment plans and inventories.

## Config Model

### New Explicit Role Registry

One `canic.toml` defines exactly one fleet template. Canic should not support
multiple fleets in one `canic.toml` file.

Add a role declaration area to `canic.toml` under that single fleet context:

```toml
[fleet]
name = "toko"

[roles.store]
kind = "canister"
package = "crates/toko-store"

[roles.router]
kind = "canister"
package = "crates/toko-router"

[roles.registry]
kind = "canister"
package = "crates/toko-registry"
```

Each declaration maps to a `FleetRoleRefV1`:

```rust
FleetRoleRefV1 {
    fleet: "toko",
    role: "store",
}
```

The role table is top-level because the file has exactly one fleet. It is not
a global role registry across the workspace.

Canic must reject or fail closed on any config shape that attempts to define
multiple fleet templates in one `canic.toml`.

The stable semantic concept is:

```text
role declaration is separate from topology attachment
```

The exact TOML spelling is secondary, but the role identity is not global.
`store` is scoped by the single fleet declared by that `canic.toml`.

### Topology Attachment

Existing topology remains the source for placement:

```toml
[subnets.application.canisters.router]
kind = "singleton"

[subnets.application.canisters.registry]
kind = "singleton"

[subnets.application.canisters.store]
kind = "singleton"
```

In 0.49, any role referenced by topology must exist in the same file's role
registry. A role with topology placement is considered attached.

Recommended rule:

```text
[roles.<role>] declares existence under [fleet] name.
[subnets.<subnet>.canisters.<role>] attaches that same fleet role.
```

Then `attached` can be rendered or derived, not necessarily stored.

Because the file has exactly one fleet, role and topology tables are scoped by
that file's `[fleet] name`.

### Minimal Preferred Shape

To avoid duplicated truth, prefer:

```toml
[fleet]
name = "toko"

[roles.store]
kind = "canister"
package = "crates/toko-store"

[subnets.application.canisters.store]
kind = "singleton"
```

Derived state:

```text
[fleet] name = "toko"
+ roles.store exists
+ topology references store
= attached

[fleet] name = "toko"
+ roles.store exists
+ no topology reference
= declared-only
```

### Package Metadata

Canister packages continue to declare their fleet scope and role explicitly.
In package metadata, `fleet` means fleet template identity, not deployment
target identity:

```toml
[package.metadata.canic]
fleet = "toko"
role = "store"
```

Both `fleet` and `role` are required. `role = "store"` alone is invalid even
when a workspace appears unambiguous. Canic must not resolve unscoped role
metadata against a global role registry, package name, crate path, or current
directory.

0.49 should not require:

```toml
draft = true
```

The declared-only state lives in fleet config, not in package-local metadata.

### Metadata Validation

`canic::build!()` should validate:

- package metadata exists;
- fleet scope is present;
- role is present;
- `fleet + role` resolves to a declared `FleetRoleRefV1`;
- root role remains root-valid within the same fleet;
- package path for that fleet role declaration matches the
  current package.

It should not require:

- subnet placement;
- pool attachment;
- sharding policy;
- scaling policy;
- deployment plan inclusion.

### Standalone Config Source

Declared-only standalone config generation must not synthesize subnet/canister
topology. It may generate:

```toml
[fleet]
name = "toko"

[roles.store]
kind = "canister"
package = "."
```

but it must not generate:

```toml
[subnets.prime.canisters.store]
kind = "singleton"
```

unless explicit attachment is requested.

This keeps the compile-only path honest: declared-only roles must not become
accidentally attached because a synthetic config manufactured topology.

## `canic::build!()` Behavior

### Current Behavior

Current behavior:

```text
package role must exist under subnets.<subnet>.canisters.<role>
```

That means topology attachment is required before compilation.

### New Behavior

New behavior:

```text
package metadata fleet + role must resolve to a declared FleetRoleRefV1
```

Then `build!()` emits role identity and attachment markers.

Suggested emitted environment values:

```text
CANIC_FLEET=<fleet>
CANIC_CANISTER_ROLE=<role>
CANIC_FLEET_ROLE=<fleet>.<role>
CANIC_CANISTER_ROLE_DECLARED=true
CANIC_CANISTER_ROLE_ATTACHED=true|false
```

`CANIC_FLEET_ROLE` is display/audit metadata. `CANIC_CANISTER_ROLE` remains the
local role name used by runtime role checks and non-root lifecycle setup.

Canic-managed artifact build callers must preflight attachment before invoking
Cargo. `canic::build!()` still emits attachment markers, but it does not read a
separate policy environment variable to decide whether declared-only roles are
allowed.

Ordinary `cargo check` can compile a declared-only non-root role, but
Canic-owned artifact build paths fail before Cargo when the role is not
attached. This keeps deploy-artifact strictness mechanical without adding a
second environment-driven policy channel.

Suggested cfgs:

```rust
#[cfg(canic_role_declared)]
#[cfg(canic_role_attached)]
#[cfg(canic_role_declared_only)]
#[cfg(canic_is_root)]
```

| Case | `cargo check` | Deploy artifact build | Notes |
| --- | --- | --- | --- |
| Missing package role metadata | fail | fail | Role identity unknown |
| Package role not declared | fail | fail | Fix config |
| Role declared only | pass | fail by default | Compilable, not deployable |
| Role attached | pass | pass | Subject to normal checks |
| Root role not configured | fail | fail | Root remains strict |

### Root Behavior

For:

```toml
[package.metadata.canic]
fleet = "toko"
role = "root"
```

`build!()` must still require root-valid configuration. A package declaring
`role = "root"` for `fleet = "toko"` must resolve to the configured fleet root
for `toko`.

Root identity is `toko.root`, not global `root`.

Concrete root config shape:

```toml
[fleet]
name = "toko"

[roles.root]
kind = "root"
package = "canisters/toko-root"

[subnets.application.canisters.root]
kind = "root"
```

Root packages receive:

- `canic_is_root`;
- root lifecycle;
- root endpoint bundle.

Root should not become declared-only except for explicitly allowed examples or
test fixtures.

If examples need root behavior, they should either:

- live in a package with proper fleet-scoped metadata; or
- use a documented example-only exception.

## `canic::start!()` Behavior

`canic::start!()` remains the single normal startup macro.

It should use build output:

```text
role = "root" -> root lifecycle + root endpoint bundle
other roles   -> non-root lifecycle + local metadata role
```

0.49 should not reintroduce `start_root!`.

Declared-only non-root roles may still use:

```rust
canic::start!();
```

because startup identity is known. Deployment and install tooling must reject
declared-only roles before they become deployment artifacts.

## CLI Workflow

### Fleet Template Commands Vs Deployment Target Commands

`canic scaffold`, `canic build`, and `canic fleet role ...` operate on fleet
template identity.

`canic deploy ...` commands operate on deployment target identity.

Examples should not use the same bare name for both unless the text explicitly
says the deployment target is intentionally named the same as the fleet
template.

### Create A Fleet

Greenfield projects start by creating the fleet config and ICP project
configuration:

```bash
canic fleet create toko
```

Local installs require an `icp.yaml` that names the fleet roles. The maintained
getting-started guide covers the current greenfield shape:

```text
docs/getting-started/minimal-managed-fleet.md
```

### Create A Role Canister

Implemented command:

```bash
canic scaffold canister toko store
```

Expected behavior:

- creates package or crate;
- writes `[package.metadata.canic] fleet = "toko"` and `role = "store"`;
- writes `build.rs` using `canic::build!()`;
- writes `src/lib.rs` using `canic::start!()`;
- adds the new crate to `workspace.members` when a workspace manifest exists;
- adds `[roles.store]` declaration to `canic.toml`;
- does not attach to subnet or pool topology by default.

0.49.4 implements this declared-only scaffold path for ordinary canister
roles.

After this:

```bash
cargo check -p canister_toko_store
```

should pass.

But these must not treat `store` as deployable until attached:

```bash
canic build toko store
canic deploy check toko-staging
canic deploy install toko-staging --plan plan.json
```

Here `toko` is the fleet template. `toko-staging` is a deployment target.

### Declare An Existing Role Canister

Implemented command:

```bash
canic fleet role declare toko store --package crates/toko-store
```

Expected behavior:

- creates only a role declaration;
- does not create package files;
- does not attach topology;
- records the required package path;
- fails if the selected `canic.toml` is not the config for fleet `toko`.

`scaffold` is file creation plus declaration. `declare` is config-only
declaration for an existing crate.

### Attach After Scaffold

0.49 does not ship scaffold-time attachment flags. Scaffolding creates a
declared-only role. Operators attach the role with an explicit follow-up
command when placement is known.

### List Role Lifecycle

Implemented command:

```bash
canic fleet role list toko
```

Example output:

```text
ROLE         PACKAGE              STATE          TOPOLOGY
toko.root    canisters/root       attached       application/root
toko.router  crates/router        attached       application/router
toko.store   crates/store         declared       -
```

JSON should include:

```json
{
  "fleet": "toko",
  "role": "store",
  "display": "toko.store",
  "package": "crates/store",
  "state": "declared",
  "attached": false,
  "topology": null
}
```

### Inspect Role

Implemented command:

```bash
canic fleet role inspect toko store
```

Should report:

- role declaration;
- package metadata match;
- topology attachment status;
- build eligibility;
- deploy eligibility;
- next action.

For declared-only roles:

```text
next action: canic fleet role attach toko store --subnet ...
```

Output may render the role as `toko.store`, but CLI commands use explicit
`<fleet> <role>` arguments. 0.49 should not add dotted CLI shorthand.

### Attach Role To Topology

Implemented command:

```bash
canic fleet role attach toko store \
  --subnet application
```

Optional direct placement kind:

```bash
canic fleet role attach toko worker \
  --subnet application \
  --kind replica
```

Behavior:

- requires role declaration exists;
- adds direct topology placement under the same fleet;
- fails on duplicate role attachment;
- validates direct placement kind (`singleton`, `shard`, `replica`, or
  `instance`; default `singleton`);
- updates `canic.toml`;
- reports role changed from declared to attached.

0.49.3 implements direct topology attachment. Pool-backed parent attachment for
sharding, scaling, and directory roles remains a later explicit slice; it must
not be inferred from arbitrary pool-like strings.

### Rename Role

Implemented command:

```bash
canic fleet role rename toko hub router
```

Behavior:

- updates `[roles.hub]` to `[roles.router]` in the selected fleet config;
- updates topology references under `toko`;
- updates package metadata if package path is known and editable;
- updates scaffold-managed config if present;
- does not mutate same-named roles in other fleets;
- prints any files it could not update;
- refuses if both old and new role exist.

This directly addresses rename friction.

## Command Strictness

| Command | Declared-only allowed? | Attached required? | Behavior |
| --- | --- | --- | --- |
| `cargo check -p <role-crate>` | yes | no | Compile role code |
| `canic scaffold canister` | yes | no | Creates role declaration |
| `canic fleet role declare` | yes | no | Creates config-only role declaration |
| `canic fleet role list` | yes | no | Reports role state |
| `canic fleet role inspect` | yes | no | Reports eligibility |
| `canic fleet role attach` | source role yes | creates attached | Updates topology |
| `canic build <fleet> <role>` | no by default | yes | Produces deploy artifact |
| `canic deploy check <deployment>` | may report | yes for deployment truth | Declared-only is not deployment truth |
| `canic deploy install <deployment>` | no | yes | Install requires topology |
| release/governance | yes in source | attached for release/deploy artifacts | Declared-only is visible but excluded from artifacts |

## Deployment Truth Interaction

Deployment truth must remain topology-based.

A declared-only role:

- does not appear as an expected live canister;
- does not appear in deployment inventory expectations;
- does not appear in install plans;
- does not count as missing deployment state;
- may appear only as a config warning or role lifecycle note.

An attached role:

- participates in deployment truth;
- may appear in deployment plans;
- may appear in inventories;
- may be checked, installed, compared, and verified.

Deployment truth must reject artifacts that claim a declared-only role is
deployable.

### Attached Role Derivation

Attachment must be derived by one shared function:

```rust
fn attached_fleet_roles(config: &FleetConfig) -> BTreeSet<FleetRoleRefV1>;
```

This derivation is the sole source of truth for whether a role is attached. It
must be used by:

- `canic::build!()`;
- deployment truth;
- `canic fleet role list`;
- `canic fleet role inspect`;
- governance checks;
- release checks.

A role is attached only through exact role-bearing config fields. The helper
must not scan arbitrary strings that look like role names.

Traversal sources:

- direct: `subnets.*.canisters.<role>`;
- sharding child role:
  `subnets.*.canisters.<attached-role>.sharding.pools.*.canister_role`;
- scaling child role:
  `subnets.*.canisters.<attached-role>.scaling.pools.*.canister_role`;
- directory child role:
  `subnets.*.canisters.<attached-role>.directory.pools.*.canister_role`.

Pool names are not role names. A pool attaches another role only when a known
role-bearing field such as `canister_role` names that role.

Traversal should use a worklist plus `BTreeSet` membership so cycles and
duplicates are deterministic and cannot expand forever. Every discovered target
role must exist in `[roles]`; unknown role-bearing references fail closed.

This rule prevents CLI role lifecycle output from disagreeing with deployment
truth, especially when a sharded role is attached through a parent, pool, or
scaling reference rather than a direct singleton topology entry.

`wasm_store` is a fleet-internal bootstrap role unless it is represented as a
normal package-backed role in `[roles]`. Its bootstrap behavior should be
documented separately from ordinary role declaration and attachment.

## Governance And Release Rules

Workspace and governance tests should validate:

- every package metadata `FleetRoleRefV1` is declared;
- every topology-attached `FleetRoleRefV1` is declared;
- no topology references unknown `fleet.role`;
- each `canic.toml` declares exactly one fleet;
- no config attempts to define multiple fleets;
- no package role metadata missing `fleet` is accepted;
- package path matches the declared `fleet.role`;
- root package resolves to the configured root for the same fleet;
- no package claims role `root` unless it resolves to that fleet's root;
- attached role state comes from the shared `attached_fleet_roles(config)`
  derivation;
- declared-only fleet roles are allowed in source;
- release artifacts do not include declared-only roles.

0.49 release policy:

```text
Declared-only roles may exist in source, but release and deploy artifacts must
exclude them and status output must make them visible.
```

This avoids turning declared-only roles into another forbidden transient state.

## Hard-Cut Config Rules

0.49 is a hard cut. It should not keep aliases, alternate config projections,
or topology-only role inference.

Rules:

- A `canic.toml` must resolve to exactly one fleet.
- Every role declaration is package-backed and must have an explicit
  `[roles.<role>]` declaration with `package = "..."`.
- Topology-only role declarations are invalid.
- `[roles.store]` resolves to:

  ```rust
  FleetRoleRefV1 {
      fleet: <current fleet>,
      role: "store",
  }
  ```

- Attempted multi-fleet config shapes are invalid. Canic rejects the config
  before interpreting any unscoped `[roles.store]` table.
- Explicit role declarations under the file's single `[fleet]` are
  authoritative.
- Any conflict between package metadata, role declaration, root identity, and
  topology attachment fails closed.
- There is no topology-only fallback, dotted CLI alias, or development-mode
  artifact-build escape hatch in the 0.49 core design.

## Error Messages

### Package Role Missing From Config

```text
Canic role validation failed.

Package:
  crates/toko-store

Declared package role:
  toko.store

Problem:
  role "toko.store" is not declared in canic.toml.

To create a compilable role declaration:
  canic fleet role declare toko store --package crates/toko-store

To attach the role to deployment topology:
  canic fleet role attach toko store --subnet <subnet>
```

### Role Declared But Not Attached

For deploy, build, and install:

```text
Role "toko.store" is declared but not attached to deployment topology.

Declared roles can compile, but cannot be built as deployment artifacts,
installed, or treated as deployment truth.

Attach it with:
  canic fleet role attach toko store --subnet <subnet>
```

### Root Declared Only

```text
The root role "toko.root" cannot be declared-only.

Root packages define fleet authority and must be configured as the root for the
same fleet.
```

## Security Properties

0.49 should guarantee:

- package role identity remains explicit;
- package role identity is validated against config;
- package role identity is fleet-scoped;
- topology attachment remains explicit;
- declared-only roles cannot be deployed;
- declared-only roles cannot be installed;
- declared-only roles cannot become deployment truth;
- declared-only roles cannot be confused with missing live canisters;
- root remains topology-valid and authority-aware;
- root identity is fleet-scoped rather than global;
- no role identity is inferred from package name;
- no scaffolded role is silently placed without operator intent;
- no release artifact silently includes a declared-only role.

## Non-Goals

0.49 should not:

- introduce package-local `draft = true` as the main model;
- make `build!()` permissive by default;
- infer role identity from crate names;
- silently auto-place scaffolded roles;
- redesign sharding;
- redesign scaling;
- implement deployment catalog or group UX;
- implement full live deploy verify;
- implement teardown or test-deployment lifecycle;
- implement dotted CLI shorthand;
- implement additional role lifecycle commands beyond
  `declare`, `attach`, `rename`, `list`, and `inspect`;
- implement a development-mode artifact-build escape hatch;
- implement root rotation;
- implement multi-role delegated-token certificate semantics;
- reintroduce `start_root!`.

## Implementation Slices

### Slice 1: Fleet Role Identity And Registry

Add `FleetRoleRefV1` and role declaration parsing to fleet config.

Support:

```toml
[fleet]
name = "..."

[roles.<role>]
kind = "canister"
package = "..."
```

Add validation for:

- duplicate roles within the same fleet;
- invalid role names;
- topology references unknown fleet roles;
- any config that attempts to define multiple fleets;
- package metadata missing `fleet`;
- missing explicit `[roles.<role>]` declarations for package-backed roles;
- missing `package = "..."` on any `[roles.<role>]` declaration;
- topology-only package-backed role declarations;
- package metadata role mismatch;
- root special handling.

### Slice 2: Hard-Cut Config Validation

Reject:

- any config that attempts to define multiple fleets;
- topology-only package-backed roles;
- missing explicit `[roles.<role>]` declarations;
- alias, fallback, or alternate config forms.

### Slice 3: Attached Role Derivation

Add the shared `attached_fleet_roles(config)` helper and route attachment
status through it for:

- `canic::build!()`;
- deployment truth;
- role list/inspect;
- governance checks;
- release checks.

The helper must traverse only direct topology canister roles and exact
role-bearing `canister_role` fields under sharding, scaling, and directory pool
configuration reachable from attached topology entries. It must not search
arbitrary strings.

### Slice 4: `build!()` Validation Split And Standalone Config

Change `canic::build!()` to validate:

```text
package metadata fleet + role exists in role declaration
```

instead of:

```text
package role is attached to topology
```

Emit:

```text
CANIC_FLEET
CANIC_CANISTER_ROLE
CANIC_FLEET_ROLE
CANIC_CANISTER_ROLE_DECLARED
CANIC_CANISTER_ROLE_ATTACHED
```

and cfgs for declared/attached status.

Ensure root behavior remains strict. Canic-managed artifact build callers
preflight attachment and reject declared-only roles before invoking Cargo.

Also change declared-only standalone config generation so it emits only `[fleet]`
and `[roles.<role>] package = "."` unless explicit attachment is requested.
Synthetic configs
must not manufacture `[subnets.*.canisters.*]` entries for the compile-only
path.

0.49.8 implements this generated-config boundary. The shared config schema now
accepts topology-less configs only as declared-only non-root role registries;
root declarations and `app_index` still require explicit topology.

### Slice 5: CLI Role Lifecycle

Add or update commands:

```text
canic fleet role declare <fleet> <role> --package <path>
canic scaffold canister <fleet> <role>
```

## Recommended Closeout Wording

0.49 separates fleet-scoped role declaration from topology attachment. A
package declares a `fleet` and `role`, which resolve to a canonical
`FleetRoleRefV1` rendered as `fleet.role`. The role may compile once declared,
but only topology-attached roles can become deployment truth, deployable
artifacts, or install targets.

`canic::build!()` validates package role identity against fleet-scoped role
declarations. Deploy, build, install, and deployment-truth paths remain
topology-strict. Root roles remain fleet-scoped and special: `fleet.root` must
resolve to the configured fleet root. Scaffold and role lifecycle commands now
support the workflow: declare role, compile role, attach role, then deploy.

0.49 does not introduce package-local draft mode, automatic placement, live
deployment verification, group/catalog UX, teardown/test-deployment lifecycle,
or root rotation.
