# Canic Design Proposal: Root Wasm Audit and Cutover Requirements

- **Version:** `0.17`
- **Status:** `Draft`
- **Author:** `Canic Core`
- **Scope:** final audit of deployable wasm footprint, root payload decomposition, IC ceiling headroom, and implementation inputs for the `0.18` hard cut to template canisters

## 1. Continuation Boundary

`0.17` is not a generic size-review exercise.

`0.17` is the final audit line for the current root-embedded-wasm architecture.
Its purpose is to:

1. measure the actual artifact baseline,
2. decompose root into runtime bytes versus embedded payload bytes,
3. prove why root-owned embedded deployable wasm growth is structurally unsustainable at Canic scale,
4. produce the exact implementation inputs required for `0.18`.

The architectural direction is already decided:

- the root-embedded deployable-wasm model is an intentionally expiring design
- template canisters are mandatory
- `0.18` implements the replacement architecture as a hard cut

`0.17` therefore closes the old model.

Hard-cut scope:

- the hard cut applies to ordinary deployable child templates
- the retained bootstrap set is not a compatibility mode
- the retained bootstrap set is part of the new architecture's control-plane bootstrap and recovery basis

## 2. Problem Statement

Current Canic topology makes root responsible for three different concerns:

- deployment authority
- manifest and policy authority
- physical ownership of embedded deployable wasm bytes

The first two responsibilities are durable.
The third is not.

When root physically carries deployable child artifacts:

- root size grows with deployable canister count
- root size grows again when each role's payload grows
- root bundle pressure compounds with ordinary runtime/codegen growth
- deployable artifact accumulation and root executable evolution compete for the same IC wasm ceiling

That scaling law is the failure.

The issue is not whether embedded payloads are intentional.
They are.
The issue is that root payload accumulation is structurally linear in the number
and size of deployable artifacts, while root is also the long-lived control-plane
authority canister.

That combination is not viable as the long-term architecture.

## 3. Audit Objectives

`0.17` must produce:

1. a raw wasm baseline for every deployable canister artifact
2. a canonical shrunk/installable wasm baseline for every deployable canister artifact
3. a debug/dev versus audit-profile comparison for every deployable canister artifact
4. a per-artifact section breakdown
5. a root payload decomposition that separates runtime bytes from embedded deployable payload bytes
6. explicit headroom against the IC wasm ceiling for every artifact
7. a growth model showing why root cannot continue to scale linearly with deployable artifact count
8. the ordered extraction requirements that `0.18` will implement

## 4. Non-Goals

`0.17` does not:

- defend the embedded-root model as a long-term steady state
- remove product features, control-plane capabilities, or operator-observable signals as a wasm optimization tactic
- define compatibility machinery between embedded and template-backed deployment
- define hybrid rollout architecture
- design shims, bridges, dual-mode support, or fallback execution paths
- treat backward compatibility as an architectural goal
- leave template canisters as an open option

Feature-preservation rule:

- `0.17` optimization work must preserve existing runtime capabilities and intended endpoint behavior
- size reduction must come from implementation, linkage, codegen, packaging, and redundant-surface cleanup
- any proposal that removes a user-facing or operator-facing feature is out of scope for `0.17` unless it is explicitly reopened as a separate design decision

## 5. Audit Scope

The audit covers the deployable canister set and the root bundle that currently
owns their payloads.

At minimum, the artifact set includes:

- `canister_root`
- `canister_app`
- `canister_minimal`
- `canister_user_hub`
- `canister_user_shard`
- `canister_scale_hub`
- `canister_scale`
- `canister_shard_hub`
- `canister_shard`
- `canister_test`

If additional deployable canisters exist in the workspace at audit time, they
are in scope as well.

## 6. Baseline Invariants

The audit must preserve these rules:

- raw non-gzipped wasm is the optimization authority
- shrunk/installable wasm is the deployment authority
- `wasm-debug` is a required comparison profile for diagnosis, not the shipping authority
- `minimal` is the canonical minimal leaf-canister baseline for shared-runtime comparison
- `minimal` must remain on the standard non-root Canic runtime surface and must not carry provisioning or other role-specific helpers
- wasm reduction work must not remove intended features or runtime signal merely to improve size numbers
- root executable/runtime bytes and embedded deployable payload bytes are separate accounting categories
- root must no longer be evaluated as a normal peer of leaf canisters
- all conclusions must name the build profile and artifact path they refer to
- the audit must remain reproducible from a clean checkout

## 7. Required Measurements

### 7.1 Artifact Baseline

For every deployable canister:

- raw built `.wasm` bytes
- deterministic `.wasm.gz` bytes
- canonical shrunk/installable `.wasm` bytes
- deterministic shrunk `.wasm.gz` bytes
- built-to-shrunk delta
- raw `wasm-debug` built `.wasm` bytes
- debug-vs-audit delta between `wasm-debug` and the audited profile

### 7.2 Per-Artifact Structure

For every deployable canister:

- function count
- data-section count
- data-section bytes
- exported-method count
- section breakdown including `code`, `data`, and `custom`

### 7.3 Root Payload Decomposition

For `canister_root`, the audit must report separately:

- executable/runtime bytes
- embedded child wasm payload bytes
- metadata/custom-section bytes
- compressed contribution of embedded payloads
- count of embedded deployable artifacts
- largest embedded artifacts by role

The audit must not collapse these categories into one generic "root is large"
statement.

### 7.4 Ceiling Headroom

For every artifact:

- current raw wasm size
- percentage of the IC wasm cap
- remaining bytes of headroom

Warning bands:

- green: `< 60%`
- yellow: `60-80%`
- red: `> 80%`

For `root`, the audit must also model:

- projected headroom after adding one more deployable child role
- projected headroom after expected next-release-line payload growth
- projected headroom under current linear embedded-payload growth

## 8. Root-Specific Audit Model

### 8.1 Root Is Two Artifacts in One

`root` currently behaves as both:

- a control-plane executable canister
- a bundle container for deployable child payloads

These are different byte classes and must be measured separately.

### 8.2 Designed Cost vs Structural Failure

Embedded child payload bytes are intentional deployment payload cost.
They are not accidental code bloat.

That does not save the model.

The failure is structural:

- root runtime growth consumes ceiling headroom
- embedded payload growth consumes the same ceiling headroom
- both growth vectors accumulate in one authoritative canister

The audit therefore distinguishes:

- intentional payload cost
- accidental runtime/codegen bloat
- architecture-level scaling failure from payload accumulation

### 8.3 Root Comparison Model

The audit must use three root baselines:

1. `minimal` canister as the shared-runtime floor
2. `root` runtime-only baseline or synthetic estimate with embedded payloads removed
3. full `root` artifact with embedded payloads included

Without these three baselines, `root` outlier status is obvious but not useful.

## 9. Growth Model

The audit must quantify root growth as:

```text
root_total =
  root_runtime
  + sum(embedded_child_payload_i)
  + root_metadata
```

Where:

- `root_runtime` is root-specific executable/runtime cost
- `embedded_child_payload_i` is each embedded deployable artifact
- `root_metadata` is custom-section and supporting payload overhead

The audit must explicitly show:

- current deployable canister count
- current sum of embedded payload bytes
- marginal root growth from adding one more deployable role
- marginal root growth from increasing existing child payload sizes

This model is the proof obligation for why root-embedded deployable wasm is not
the steady state.

## 10. Hotspot Attribution

The audit must include retained-size attribution for:

- shared runtime baseline
- candid/serialization glue
- logging and metrics
- lifecycle/bootstrap runtime
- auth and crypto
- macro/derive expansion
- root-only runtime cost
- embedded deployable payload cost

`root` retained hotspots must be reported in two lists:

1. runtime/codegen hotspots
2. embedded payload hotspots by child role

## 11. Ordered Extraction Candidates

`0.17` must produce an ordered extraction list for removal from root.

Required ordering:

1. largest non-bootstrap child payloads
2. largest role families with independent deployment cadence
3. payloads that add bytes to root without being required for minimal bootstrap
4. all remaining deployable child payloads except the irreducible bootstrap set

This is not a prioritization exercise for optional cleanup.
It is the required extraction order that `0.18` will execute.

## 12. Minimum Bootstrap and Recovery Set

`0.17` must identify the irreducible bootstrap artifact set that root may retain
after the hard cut.

The chosen position is:

- root retains only the minimum bootstrap set required to establish template-canister availability and seed manifest authority
- root does not retain long-term ownership of ordinary deployable child templates

This retained set does not preserve the old architecture.
It exists only as the irreducible bootstrap and recovery basis of the new
template-canister steady state.

The audit must therefore identify, by artifact and by byte count:

- the bootstrap template payloads that remain irreducible
- the manifest seed material required at bootstrap
- the recovery-critical artifacts that must exist before template-backed install and upgrade orchestration can proceed

Anything outside that set is extraction scope for `0.18`.

## 13. Deliverables

`0.17` is complete only if it produces all of the following:

### 13.1 Baseline Table

A table of every deployable canister artifact with:

- built `.wasm`
- built `.wasm.gz`
- shrunk `.wasm`
- shrunk `.wasm.gz`
- `wasm-debug` built `.wasm`
- debug-vs-audit delta
- built-to-shrunk delta
- percent of IC cap
- remaining headroom

### 13.2 Per-Artifact Section Breakdown

For every artifact:

- section sizes
- function/data/export counts
- outlier notes

### 13.3 Root Payload Decomposition

For `root`:

- runtime bytes
- embedded payload bytes
- metadata bytes
- embedded artifact count
- largest embedded roles

### 13.4 Projection Against the IC Ceiling

For `root`:

- current headroom
- one-more-role projection
- next-release-line growth projection
- structural slope under continued embedded-payload accumulation

### 13.5 Minimum Bootstrap / Recovery Set

An explicit list of:

- artifacts root still must retain after the cut
- artifacts root must stop carrying

### 13.6 Ordered Migration Inputs

An ordered extraction list with:

- role
- current embedded size
- bootstrap criticality
- removal order
- target ownership in `0.18`

## 14. Hard Handoff to 0.18

`0.17` is complete only when it produces the implementation inputs required by
`0.18`.

Required handoff outputs:

1. canonical deployable template inventory
2. canonical root runtime-only baseline or synthetic estimate
3. canonical minimum bootstrap set
4. ordered extraction list from root
5. manifest fields required to bind deployment authority to template payloads
6. size/headroom thresholds that `0.18` must remove from root

`0.17` does not end with "consider template canisters".
`0.17` ends with the data needed to implement them.

## 15. Conclusion

The embedded-wasm-in-root model is not viable as the long-term architecture.

The reason is structural, not incidental:

- root payload growth is linear in deployable artifact count and size
- root runtime evolution and payload accumulation consume the same IC ceiling
- root cannot remain the long-lived control-plane authority canister while also
  acting as the permanent byte container for every deployable child artifact

Template canisters are the required next step.

The required fix is not further root slimming alone; it is removal of ordinary
deployable payload ownership from root.

Root keeps:

- deployment authority
- manifest authority
- policy authority

Root does not keep:

- long-term ownership of all deployable wasm bytes

That architectural cut is the subject of `0.18`.

## 16. Success Criteria

`0.17` succeeds if:

- every deployable artifact has a recorded baseline
- every deployable artifact has a recorded debug-vs-audit comparison or an explicit blocked reason
- root runtime bytes and embedded payload bytes are separately quantified
- root headroom against the IC cap is explicitly measured
- linear embedded-payload growth is shown quantitatively
- the minimum bootstrap/recovery set is explicitly identified
- the extraction order from root is fixed
- the implementation inputs for `0.18` are complete

## 17. Narrow Remaining Questions

The only unresolved questions in `0.17` are implementation-input questions for
`0.18`, not architectural-direction questions:

- exact byte threshold for classifying bootstrap artifacts as irreducible
- exact manifest field set needed for `0.18` install/upgrade orchestration
- exact synthetic method for deriving the root runtime-only baseline when direct
  non-embedded build extraction is impractical
