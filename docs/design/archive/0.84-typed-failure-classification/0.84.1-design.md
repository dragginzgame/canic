# Canic 0.84.1 Design: Typed Failure Classification

Date: 2026-07-10

## Status

- Status: accepted and implemented.
- Target patch: 0.84.1.
- Relationship to 0.84.0: focused interruption after the role-aware state
  contract release; it is not a fourth role-contract slice.
- Primary rule: human-readable text is presentation only and must not drive
  Canic-owned control flow.

## 1. Decision Summary

Canic already produces typed failures in several owner modules, but some
boundaries erase them into `String` and later reconstruct meaning with English
substring matching. That makes wording changes capable of changing retry,
rollover, safety, public-code, metric, and CLI behavior.

0.84.1 preserves or restores machine-readable classification for the owned
paths found by the audit:

- delegated-auth token versus certificate expiry;
- registry policy public error codes;
- wasm-store capacity, missing chunk, hash mismatch, and missing manifest;
- deployment-state missing, network mismatch, and read failure assumptions;
- install preflight and deployment-truth blocks;
- invalid blob-storage cycle input;
- missing installed deployment state in medic;
- direct local-replica destination-invalid rejection;
- typed canister-signature data-certificate errors;
- chain-key proof-pending retry classification;
- deployment finding-code, assumption-key, and observation-source
  classification;
- cycle top-up, replica query/HTTP, policy exit-class, backup verification,
  artifact-kind, and Cargo workspace parsing.

Display messages remain useful operator diagnostics. They are no longer the
machine contract for these decisions.

## 2. Scope Declaration

| Surface | 0.84.1 decision |
| --- | --- |
| Runtime behavior | Yes. Auth expiry classification and wasm-store publication recovery consume typed reasons. |
| CLI behavior or text | Error text is retained where practical; classification is typed. |
| Candid | Yes. `ErrorCode` gains auth-token, auth-proof-pending, and wasm-store failure variants. |
| JSON/output formats | Deployment assumption keys become distinct stable machine keys; existing observation and exit-class labels are preserved. |
| Cargo versions | No automated version edits. |
| Cargo.lock | No change expected. |
| Stable memory | No records, IDs, encodings, or migrations change. |

## 3. Owned Classification Contracts

### 3.1 Delegated auth

`VerifyDelegatedTokenError::{TokenExpired, CertExpired}` map to distinct public
codes before conversion to `InternalError`. Access guards return distinct
`AccessError` variants and expose `AccessErrorKind` for application-owned
endpoint error adapters.

The public mapping is:

| Failure | Public code |
| --- | --- |
| Delegated token expired | `AuthTokenExpired` |
| Delegation certificate expired | `AuthProofExpired` |
| Chain-key delegation proof is still being prepared | `AuthProofPending` |

All other access denials remain non-retryable generic denials unless a later
design assigns them a stable code.

### 3.2 Registry policy

Registry policy variants map to existing policy `ErrorCode` values while the
typed `TopologyPolicyError` is still available. `api::error` no longer searches
the rendered policy message.

The pure policy layer does not own DTO mapping. The conversion lives under
`ops`, which owns cross-layer conversion.

### 3.3 Wasm store

Template owner errors map to these public codes:

- `WasmStoreCapacityExceeded`;
- `WasmStoreChunkMissing`;
- `WasmStoreHashMismatch`;
- `WasmStoreManifestMissing`.

Managed-store rollover and bounded metric reasons consume those codes. Store
capacity includes byte, template-count, and version-retention ceilings.

### 3.4 Deployment truth and install

The previously overloaded `local_state.root_canister_id` assumption key is
hard-cut into three owner-defined keys:

- `local_state.root_canister_id.missing`;
- `local_state.root_canister_id.network_mismatch`;
- `local_state.root_canister_id.read_failed`.

Execution preflight and CLI plan classification use
`DeploymentAssumptionKindV1`, never the assumption description.

Install deployment-truth and execution-preflight blocks retain
`InstallRootBlockedError` and `InstallRootBlockKind` through the boxed host/CLI
boundary.

### 3.5 Repository-wide structural scan

A second full repository scan covered Rust production code, inline test
modules, shell/CI scripts, serialized machine labels, command-output parsers,
and source-shape guards. It removed the remaining avoidable owned substring
decisions:

- deployment evidence conflicts and artifact receipt failures use exact
  owner-defined code registries;
- observation-source decisions use `RoleAssignmentSourceV1` labels and exact
  owner parsing while preserving the open string field and existing JSON;
- deployment assumption diagnostics use defined key namespaces rather than
  searching for English words inside keys;
- cycle top-up variants use exact wire labels;
- direct replica CBOR status is an enum and HTTP success parses the numeric
  status code;
- Cargo workspace discovery parses TOML structure instead of looking for a
  `[workspace]` text fragment;
- policy exit classes own their labels and parsing;
- backup verification and bootstrap artifact kinds use shared owner constants.

Exact matches on documented command names, config values, schema labels,
stable-memory namespaces, and naming-convention suffixes are structured input
validation rather than human-diagnostic classification.

Text assertions remain only where text is part of the maintained observable
contract: renderers, proc-macro diagnostics, source-shape checks, and isolated
external-tool adapters. Behavioral tests assert typed errors, machine-readable
codes, or observable state. Removed command, config, endpoint, and state shapes
have no compatibility paths or anti-resurrection tests.

## 4. External Diagnostic Boundary

ICP CLI diagnostics that currently expose only stderr text remain compatibility
parsers. 0.84.1 does not pretend those external strings are typed Canic
contracts.

Where structured data already exists, it is used. In particular, direct local
replica queries classify a missing destination through reject code 3 instead
of the replica's English message, decode query status as an enum, and parse
HTTP status numerically.

Future external-tool cleanup should prefer stable JSON error codes or direct
agent responses, isolate any unavoidable text matching in one adapter, and
fail closed on unknown wording.

The residual production text adapters found by the audit are:

| Adapter | Text owner | Typed replacement prerequisite |
| --- | --- | --- |
| `canic-cli::list::live` missing-ID and empty-Wasm detection | ICP CLI | Stable structured lookup/status error codes. |
| `canic-cli::metrics::transport` missing-method detection | Replica reject message | A structured reject category that distinguishes method absence. |
| `canic-cli::replica` local-network/project discovery | ICP CLI | Stable machine-readable replica/project diagnostics. |
| `canic-cli::install` already-installed hint fallback | Install transport dependencies | A typed already-installed/install-conflict variant from every transport. |
| `canic-host::install_root::commands` missing canister-ID detection | ICP CLI | Stable structured canister-ID lookup errors. |
| `canic-host::install_root::readiness` missing-method detection | ICP CLI/replica | Structured readiness query rejection categories. |

These adapters are not Canic-owned English contracts. Until their producers
expose structured classifications, changing them would either remove useful
diagnostics or guess semantics from a different text layer.

## 5. Non-Goals

0.84.1 does not:

1. Continue or alter the 0.84 role-contract design.
2. Change stable-memory ownership, records, or migrations.
3. Add broad retry behavior for non-expiry auth failures.
4. Assign a public code to every internal error.
5. Parse all ICP CLI diagnostics into guessed Canic enums.
6. Replace display text, logs, help text, or renderer assertions merely because
   they contain English.
7. Add compatibility aliases for the old deployment assumption key.

## 6. Verification

Focused verification covers:

- delegated token and certificate expiry code/access mappings;
- chain-key proof-pending retry classification;
- registry policy variant-to-code conversion;
- wasm-store code-to-rollover and metric classification;
- deployment assumption kind behavior and install block kinds;
- blob-storage invalid-cycle classification;
- medic missing-deployment classification;
- direct replica reject-code classification;
- exact finding-code, assumption namespace, observation-source, cycle-status,
  HTTP-status, workspace-TOML, policy-label, and verification-kind handling;
- canonical wasm-store Candid parsing and build compatibility;
- formatting, Clippy, package tests, and changelog governance.

## 7. Acceptance Criteria

0.84.1 is complete when:

1. No Canic-owned production decision identified by this audit depends on an
   English error or description fragment.
2. Auth clients can distinguish token expiry, certificate expiry, and
   proof-pending retry state without message matching.
3. Wasm-store rollover and metrics consume stable codes.
4. Deployment safety distinguishes missing state from network drift through an
   owner-defined kind.
5. Existing external-only parsers are explicitly retained as compatibility
   boundaries.
6. Public surface changes are documented and canonical Candid remains valid.
7. Package versions remain untouched by automated implementation.
