# Canic 0.89 Design: Deployment Evidence and Surface Truth

Date: 2026-07-13

## Status

- Status: `0.89.0` is published with all three bounded slices complete, and the
  post-release Slice A conformance correction is published as `0.89.1`. A
  post-release Slice B source-retention correction is implemented for the next
  patch.
- Audit source:
  `docs/audits/reports/2026-07/2026-07-13-post-0.88.md`.
- Implementation tracker: [status.md](status.md).

## Problem

The post-0.88 audit found three bounded truth gaps:

1. install-state persistence returns boxed errors;
2. installed-deployment observation and command projections flatten typed
   install-state and replica-query failures; and
3. four dependency declarations and three RPC adapter visibilities describe
   consumers that do not exist, while one Machete candidate has a generated
   proc-macro consumer.

0.89 closes these findings without introducing a shared application error
system or a broad visibility/dependency cleanup.

## Decisions

### Slice A - Typed install-state persistence

`canic-host::install_root::state` owns deployment install-state validation,
serialization, paths, reads, and writes, so it owns one focused
`InstallStateError`.

- Replace all six boxed dynamic-error signatures in that owner.
- Type invalid deployment/template names, invalid network names, state/network
  mismatch, filesystem operation/path/source, and JSON operation/path/source.
- Keep `InstallState`, schema version 2, JSON field names, paths, and successful
  serialized bytes unchanged.
- Narrow `release_set::icp_root` to its real `io::Error` return contract so the
  current-directory and canonicalization source is not boxed before
  install-state lookup receives it.
- Re-export the concrete error only where public install-state APIs require it.

No migration, alternate reader, legacy error conversion, or general host
persistence error is added.

#### Post-release correction - Install-state admission truth

The `0.89.0` conformance scan found that state reads decoded the current shape
but did not admit it against the canonical path identity. The deployment
catalog duplicated network/deployment checks and did not validate the schema
version.

- One install-state-owned decoder validates the supported schema version,
  deployment name, and network after JSON decoding.
- Named reads and deployment-catalog enumeration use that same decoder.
- Deployment-catalog requests validate the network name before constructing
  the state-directory path; path-like network values cannot escape the
  canonical `.canic` hierarchy.
- Schema, deployment, and network mismatches fail closed with typed
  `InstallStateError` variants.
- The deployment catalog retains its existing malformed-state warning code and
  presentation for the previously checked mismatch cases.
- Writers reject an unsupported schema value before persistence.

This is a correction to Slice A, not a fourth slice. No compatibility reader,
schema negotiation, migration, or alternate catalog parser is added.

#### Post-release correction - Deployment-truth source retention

The closeout scan after `0.89.1` found one passive deployment-inventory path
that immediately rendered `InstallStateError` into a string. The inventory
error now retains the concrete install-state source and preserves its existing
display text.

This is a correction to Slice B, not a fourth slice. It adds no shared error
type, command projector, report field, or behavior change.

### Slice B - Typed installed-deployment evidence

`canic-host::installed_deployment` remains the sole owner of combining install
state, registry observation, and resolved topology.

- Replace `InstalledDeploymentError::InstallState(String)` with the concrete
  `InstallStateError` source.
- Replace `InstalledDeploymentError::ReplicaQuery(String)` with the concrete
  `ReplicaQueryError` source.
- Preserve existing typed ICP, registry, I/O, missing-deployment, and
  lost-local-root variants.
- Command-owned errors retain concrete install-state and replica-query sources
  until their exit, hint, and finding decisions are complete.
- Keep command-specific missing-install and lost-root wording local because the
  recommended next action differs by command.
- Delete direct `.to_string()` conversions of these two source families.

The ten command projectors remain command-owned. Do not replace them with a
generic trait, context enum, global CLI error, or host knowledge of command
presentation. Their repeated variant match is the explicit translation between
one host vocabulary and distinct command policies.

Successful output, JSON, exit codes, hints, Medic source classification, and
network selection remain unchanged.

### Slice C - Manifest and RPC surface truth

Remove exactly the four compilation-proven unused dependencies:

- test root build dependency `sha2`;
- runtime-probe development dependency `ic-testkit`;
- integration-test development dependency `canic-host`; and
- facade development dependency `serde_json`.

Retain the facade development dependency on `ic-cdk`: targeted integration-test
compilation proves that Canic endpoint proc macros generate a direct
`::ic_cdk` reference. Account for that generated consumer with one
package-local Cargo Machete annotation.

Then narrow only the audited RPC request surface:

- remove `CyclesRpc` and `UpgradeCanisterRpc` re-exports;
- make `CyclesRpc`, `UpgradeCanisterRpc`, and `RecycleCanisterRpc` private to
  their dispatch owner; and
- keep `RequestOps` and its methods available across the already-private
  `ops` tree, while exposing `RequestOpsError` only to its RPC owner. The
  private crate-root `ops` module is the effective external boundary; do not
  add redundant `pub(crate)` modifiers inside it.

Do not run a workspace-wide `unreachable_pub` rewrite, restructure the RPC
stack, change request encodings, or add Cargo Machete ignore entries for
dependencies that have no generated consumer.

## Hard Cut

- Replaced boxed install-state returns and string-valued installed-deployment
  source variants are deleted in their slice.
- No aliases, compatibility conversions, fallback strings, or duplicate error
  paths remain.
- Dead dependency declarations and unused adapter exports are removed, not
  ignored or deprecated. The retained generated `ic-cdk` consumer is explicitly
  accounted for rather than misclassified as dead.
- Existing schemas, paths, successful bytes, request encodings, command output,
  and exit behavior do not change.

## Deliberate Non-Abstractions

0.89 does not add:

- a global host or CLI error hierarchy;
- a public error-code schema;
- a generic command-error projector;
- a persistence, path, or project-context framework;
- a workspace dependency-upgrade or advisory-remediation program;
- a broad visibility cleanup;
- deployment, stable-memory, Candid, or persisted-schema changes.

## Acceptance

### Slice A

1. The install-state owner contains no boxed dynamic-error signature.
2. Invalid names, network mismatch, I/O, and JSON failures have typed variants.
3. I/O and JSON variants retain operation, path, and original source.
4. Schema version, paths, successful values, and serialized bytes are
   unchanged.
5. ICP-root discovery retains its original I/O source without a boxed wrapper.

### Slice B

1. `InstalledDeploymentError` retains concrete install-state and replica-query
   sources.
2. No install-state or replica-query error is converted to text before command
   classification completes.
3. Command-specific missing-install and lost-root actions remain local.
4. Existing exit codes, hints, findings, JSON, and successful output are
   unchanged.
5. No generic cross-command projection abstraction is introduced.

### Post-release Slice A correction

1. Every install-state reader validates schema version, deployment identity,
   and network identity in the install-state owner.
2. Deployment-catalog enumeration cannot bypass that admission rule.
3. Unsupported or misplaced state fails closed before its root principal or
   other deployment evidence is consumed.
4. Catalog network selection is validated before filesystem access.
5. Existing valid state bytes, schema version 2, catalog warning code, and
   successful report/command behavior are unchanged.

### Post-release Slice B correction

1. Passive deployment inventory retains `InstallStateError` as its source.
2. No install-state failure is converted to text before deployment-truth
   classification completes.
3. Existing display text, successful inventory output, reports, and exits are
   unchanged.

### Slice C

1. The four proven dead dependency declarations are absent, the one generated
   facade test dependency is retained, and targeted packages still compile.
2. Cargo Machete reports no unused dependency.
3. Concrete RPC command adapters are private and not re-exported.
4. Request bytes, response decoding, replay metadata, and RPC behavior are
   unchanged.
5. No unrelated visibility or dependency change is included.

## Sequence

1. Slice A: typed install-state persistence.
2. Slice B: typed installed-deployment and command evidence.
3. Slice C: exact manifest and RPC surface cleanup.
4. Close 0.89; do not extend the design by accretion.

## Validation Policy

Run only targeted checks:

- install-state read/write, invalid-name, network-mismatch, and exact-byte
  tests;
- installed-deployment host tests and focused auth, backup, blob-storage,
  cycles, environment, inspect, list, Medic, metrics, snapshot, status, and
  token command tests that exercise classification or rendering;
- targeted package and build-script checks for each removed dependency;
- RPC request dispatch unit tests;
- targeted host, CLI, core, facade, and affected fixture Clippy;
- formatting, layering guards, Cargo Machete, changelog governance, and diff
  hygiene.

The maintainer owns full workspace, PocketIC, deployment, Wasm, and publish
validation.
