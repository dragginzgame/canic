# Canic 0.84 Design: Role-Aware State Contracts

Date: 2026-07-10

## Status

- Status: completed and released in 0.84.0.
- Release line: 0.84 role-aware state contracts.
- Implementation: Slices A, B, and C released; 0.84.1 typed failure
  classification is a separate patch interruption, not Slice D.
- 0.84.2 correction: root-only allocation groupings are hard-cut into exact
  shared-runtime, auth, ICP-refill, pool, scaling, directory, and sharding
  allocations so every role reports the stable state its binary can use.
- Primary rule: one typed authority decides a canister role's Canic features,
  state allocations, and stable-memory IDs.
- Scope rule: support one simple Cargo package shape exactly and reject every
  other shape. Do not build a general Cargo feature resolver in 0.84.

## 1. Decision Summary

A state manifest is trustworthy only when these facts are resolved together:

```text
validated role
  + exact role package
  + supported direct runtime Canic dependency
  -> required and effective Canic features
  -> selected state allocations
  -> canonical stable-memory IDs
```

The canonical owner is `canic-core::role_contract`. It owns:

- validated configuration to typed capability derivation;
- capability to required-feature rules;
- public Canic feature defaults and implications;
- feature and built-in-role allocation rules;
- allocation keys and canonical stable-memory IDs;
- pure resolution and rejection policy.

Build support, medic, release-set views, state manifest, and state audit consume
the same result. They may collect evidence or render it, but they may not
reconstruct these decisions.

0.84 deliberately does not support every valid Cargo dependency graph. A role
package must use one direct, unconditional, non-optional normal dependency on
`canic`. Package-feature forwarding, target-specific Canic dependencies, and
transitive runtime contributions are rejected. This makes the effective feature
set exact without implementing a second Cargo resolver.

## 2. Why This Is Safety Work

### 2.1 Stable-memory IDs are persistent schema

A memory ID identifies bytes that survive upgrades. A false ownership claim can
cause a later binary to interpret old bytes through the wrong record layout:

```text
wrong role contract
  -> wrong ownership claim
  -> wrong upgrade assumption
  -> persisted bytes decoded as the wrong schema
```

Unknown roles, missing descriptors, incompatible package shapes, and memory
collisions therefore fail closed.

### 2.2 Workspace features are not role evidence

Workspace test and Clippy lanes compile shared crates with features requested by
many packages. That does not prove one role package requested those features.

```text
workspace compile features != exact role package features
```

The resolver never uses `cfg!(feature = ...)`, the host binary's compiled
features, or workspace-unified metadata features as role evidence.

### 2.3 Consumers must agree

The operator evidence chain is:

```text
configured role
  -> exact package
  -> supported direct Canic dependency
  -> effective features
  -> stable-memory allocations
  -> build / medic / release view / state audit
```

A passing consumer report must be a projection of this one resolution, not an
independent implementation that happens to agree.

## 3. Goals

1. Put feature, capability, allocation, and ID decisions in one core module.
2. Resolve a declared role to one exact package.
3. Support one explicit package dependency shape and reject all others.
4. Account for dependency renames, workspace inheritance,
   `default-features`, direct feature declarations, and Canic public feature
   implications.
5. Derive configuration capabilities exactly once.
6. Keep Cargo defaults and public implication edges in parity with the catalog.
7. Return no consumable contract when any blocking finding exists.
8. Permanently bind each stable-memory ID to one allocation key.
9. Keep records, descriptors, migrations, and lifecycle logic with storage
   owners.
10. Verify the actual built-in wasm_store package or generated wrapper at build
    time.
11. Preserve every current memory ID and persisted record encoding.
12. Hard-cut every replaced helper and unsupported build path.

## 4. Non-Goals

0.84 does not:

1. Implement a general Cargo feature-graph resolver.
2. Support package-feature forwarding into `canic/*`.
3. Support optional, target-specific, or transitive runtime Canic dependency
   shapes.
4. Support multiple runtime paths or versions of `canic`.
5. Support arbitrary `--features`, `--all-features`, or
   `--no-default-features` build selections.
6. Use workspace-unified Cargo features as role evidence.
7. Run nested Cargo from a build script.
8. Read stable-memory values.
9. Generate or execute migrations.
10. Repair or renumber existing stable-memory IDs.
11. Change persisted record encoding.
12. Add runtime state-manifest endpoints.
13. Add compatibility aliases, legacy fallbacks, or parallel resolvers.
14. Change CLI command forms in the first implementation slice.
15. Change JSON schemas or finding codes without an explicit serialized-surface
    decision in the implementing slice.

## 5. Supported Package Contract

### 5.1 Canonical build selection

The only supported 0.84 canister build selection is:

```text
root package: exact selected role package
target: wasm32-unknown-unknown
root package features: package defaults
dependency kind: normal runtime
```

`canic build` currently exposes no arbitrary Cargo feature flags. If such flags
are added later, they require a separate design before they can produce an
authoritative role contract.

### 5.2 Required dependency shape

A declared role package must have exactly one supported Canic dependency:

```text
[dependencies]
canic = { ..., features = [...] }
```

The dependency:

- is normal runtime, unconditional, and non-optional;
- resolves to package name `canic`;
- may be renamed;
- may inherit from `[workspace.dependencies]`;
- may set `default-features = false`;
- may declare Canic public features directly.

The selected role package must not:

- forward any package feature to `canic/*`;
- place Canic under a target-specific dependency table;
- make the Canic dependency optional;
- receive Canic through another normal dependency;
- have more than one normal runtime path to Canic;
- resolve more than one Canic package ID or version.

Unsupported shapes produce a blocking finding. They are not partially
interpreted.

### 5.3 Why restriction is preferable in 0.84

The current repository uses the supported shape everywhere except one redundant
feature-forwarding test package. Hard-cutting that forwarding is smaller and
safer than adding support for every Cargo graph form.

With the restricted shape, the effective feature set is deterministic:

```text
direct dependency features
  + Canic defaults when enabled
  + canonical public implication closure
```

The host still inspects the package-scoped wasm normal-runtime graph to verify
that no forbidden transitive path or second Canic package exists. It does not
derive features from arbitrary graph contributions.

### 5.4 Package identity

The selected runtime graph must contain exactly one Canic Cargo package ID with:

- package name `canic`;
- the exact version supported by the compiled Canic toolchain;
- a public feature definition that passes catalog parity validation.

The source and manifest path are retained for diagnostics. A separate schema
version, catalog digest, source allowlist, or compatibility negotiation layer is
not added in 0.84.

## 6. Single Authority And Crate Direction

### 6.1 Core authority

```text
crates/canic-core/src/role_contract/
  mod.rs          public resolution entry points
  model.rs        typed inputs, outputs, keys, and findings
  allocation.rs   allocation definitions and canonical memory IDs
  catalog.rs      typed feature/capability/allocation tables
  policy.rs       capability derivation and pure resolution
```

This module is safe to compile into canisters. It performs no filesystem access,
Cargo execution, mutation, async work, timers, IC calls, or stable-memory reads.

`catalog.rs` is the only production source allowed to answer:

- which validated configuration fields imply which capabilities;
- which capabilities require which Canic features;
- which public Canic features imply other public features;
- which features or built-in roles select which allocations.

`allocation.rs` is the only production source allowed to answer:

- which active allocation key is assigned which memory IDs.

Together they are one authority boundary.

### 6.2 Host evidence adapter

```text
crates/canic-host/src/role_contract/
  mod.rs       role/package evidence orchestration
  package.rs   supported dependency-shape and graph validation
  descriptor.rs strict allocation-to-descriptor join
```

Cargo, TOML, filesystem, and path logic stay in `canic-host`. These dependencies
do not enter canister-target graphs.

The validator boundary remains deliberately narrow:

```text
RolePackageValidation
  Supported(RolePackageEvidence)
  Unsupported(RoleContractFinding)
```

`RolePackageEvidence` contains only the selected package identity, resolved
Canic package identity, dependency key, default-feature setting, and direct
feature names. Package paths, target predicates, feature-contribution sources,
and partial graph models remain private validation details and never enter core
policy. The validator must not evolve into a general Cargo model during 0.84.

### 6.3 State owners

`canic-core` and `canic-control-plane` storage modules continue to own:

- persisted `*Record` types;
- canonical `*Data` snapshots;
- stable maps;
- conversion ops;
- restore order and invariants;
- migrations and lifecycle behavior.

They import allocation keys and memory IDs from the role-contract authority and
provide descriptors. They do not decide which role receives an allocation and
do not redefine numeric IDs.

The control-plane authority range 80-99 moves to
`canic-core::role_contract::allocation` so `canic-core` never needs to depend
on `canic-control-plane`. Active control-plane IDs remain 80-85; IDs 86-99 are
unallocated reserve.

## 7. Typed Contract

### 7.1 Features and capabilities

```text
CanicFeatureKey
  Metrics
  ControlPlane
  WasmStoreCanister
  Sharding
  BlobStorage
  BlobStorageBilling
  IcpRefill
  AuthChainKeyEcdsa
  AuthChainKeyRootSign
  AuthRootCanisterSigCreate
  AuthRootCanisterSigVerify
  AuthIssuerCanisterSigCreate
  AuthIssuerCanisterSigVerify
  AuthDelegatedTokenVerify
  ...every maintained public Canic feature
```

```text
CanicFeatureEffect
  NoState
  StateBearing
```

Every public feature has an explicit effect. `NoState` is a reviewed catalog
value, not an omission. Capability support is expressed only by
`CapabilityRequirement`; it is not a second feature-effect category.

```text
RoleCapabilityKey
  Runtime
  Root
  RootControlPlane
  WasmStore
  Sharding
  Scaling
  Directory
  IcpRefill
  RoleAttestationSigner
  RoleAttestationVerifier
  DelegatedTokenIssuer
  DelegatedTokenVerifier
  Icrc21
  ...maintained configuration capabilities
```

Raw config consumers do not submit capability lists. The resolver derives them
from the validated `ConfigModel` and requested role through one pure function.
Blob-storage state is selected directly from effective features because the
configuration model has no separate blob-storage capability field.

### 7.2 Allocations

```text
StateAllocationKey
  CoreRuntimeTopology
  CoreRuntimeEnvironment
  CoreAuthState
  CoreReplayReceipts
  CoreRootAppRegistry
  CoreRuntimeObservability
  CoreIcpRefillRecords
  CoreRuntimeIntent
  CanisterPool
  ScalingRegistry
  DirectoryRegistry
  ShardingRegistry
  ShardingAssignments
  ShardingActiveSet
  StoredBlobs
  BlobDeletionPending
  StorageGatewayPrincipals
  BlobStorageBilling
  TemplateManifests
  TemplateChunkSets
  TemplateChunkRefs
  TemplateChunkPayloads
  ControlPlaneSubnetState
  WasmStoreGcState
```

```text
AllocationDefinition
  key
  owner
  memory_ids: static non-empty slice of MemoryId
```

Allocation definitions represent active state only. There is no lifecycle,
tombstone, retired-allocation, or removed-state model. No new collection
dependency is needed. Catalog validation rejects an empty ID slice.

### 7.3 Resolution result

```text
ResolvedStateAllocation
  key
  owner
  memory_ids
  selected_by: set of SelectionProvenance
```

Repeated selection of one allocation key merges provenance. A collision means
two distinct allocation keys claim the same memory ID. Resolved allocations are
active by definition.

```text
RoleContractResolution
  Resolved
    contract: ResolvedRoleContract
  Rejected
    errors: non-empty Vec<RoleContractFinding>
```

Required invariant:

```text
blocking finding => no ResolvedRoleContract exists
```

## 8. Catalog And Cargo Parity

The role-contract catalog is the policy authority. `crates/canic/Cargo.toml` is
Cargo's executable mirror.

A blocking parity test verifies:

1. Every public Canic feature has exactly one `CanicFeatureKey` and effect.
2. Every catalog feature exists in `[features]`.
3. Catalog defaults equal Cargo defaults.
4. Public same-package implication edges equal Cargo implication edges.
5. Every state-bearing feature selects at least one active allocation.

The host applies the same public feature/default/implication check to the exact
resolved Canic manifest it inspects. Drift rejects resolution.

No catalog digest, generated manifest, code generation, or schema negotiation is
needed. The parity check is direct and readable.

The tables remain strongly typed by purpose:

```text
FeatureDefinition
FeatureImplication
CapabilityRequirement
FeatureAllocation
BuiltInAllocation
AllocationDefinition
```

One authority does not require one generic rule struct.

## 10. Resolution Algorithm

`resolve_role_contract` is pure after host evidence collection:

1. Validate the requested role and exact package identity.
2. Validate the supported direct runtime dependency shape.
3. Reject package feature forwarding, optional/target-specific Canic edges,
   transitive Canic paths, multiple paths, and multiple package IDs/versions.
4. Validate the inspected Canic manifest against catalog parity.
5. Resolve direct dependency features and Canic defaults.
6. Compute the catalog public implication closure.
7. Derive typed capabilities once from validated configuration or built-in role.
8. Reject each required feature absent from the effective set.
9. Collect allocation selections with typed provenance.
10. Group by allocation key and merge provenance.
11. Expand each selected key through its canonical active definition.
12. Reject distinct allocation keys that claim the same memory ID.
13. Return `Resolved` only when no blocking findings exist; otherwise return
    `Rejected` without a contract.

The result is independent of:

- unrelated workspace packages;
- the host CLI's compiled features;
- all-feature test and Clippy lanes;
- TOML map iteration order;
- current working directory after explicit config resolution.

## 11. Canonical Memory Map

All current Canic-managed allocations and named reservations in the 11-99
authority ranges live in one `CanicStableMemoryV1` namespace.

The canonical MemoryManager map is:

```text
0-10    ic-memory governance
11-79   canic-core (active allocations plus reserve)
80-85   canic-control-plane active allocations
86-99   canic-control-plane reserve
100-254 downstream applications
255     invalid sentinel
```

Within that namespace:

- one memory ID is assigned to at most one active allocation key;
- the same allocation key may appear in different canister instances;
- distinct active allocation keys cannot share an ID;
- disabling a feature stops current selection but leaves its active allocation
  definition unchanged;
- a future ID is introduced through one new allocation definition.

The catalog carries no historical allocation tombstones. Before 1.0, deleting
an allocation or reassigning an ID is an explicit hard cut. Any canister whose
stable-memory allocation ledger describes the previous map requires a
destructive reinstall; the framework does not add a migration, alias, dual
read, or compatibility reservation for that old map.

Template IDs 80-83 used by root and wasm_store are the same semantic allocation
keys in different canister instances. They are not unrelated numeric reuse.

Catalog validation checks all allocation definitions, not only allocations
selected by current role fixtures, and rejects definitions outside their
owner's authority range.

## 12. Descriptor Materialization

State owners expose descriptors keyed by `StateAllocationKey`. The host validates
the complete registry before materializing any role:

- every active allocation has exactly one descriptor provider;
- descriptor IDs equal canonical allocation IDs;
- descriptor owner agrees with the active allocation definition;
- record, snapshot, version, migration, restore, and invariant metadata satisfy
  the existing state-audit contract.

Then the host performs one strict join:

```text
resolved allocations + owner descriptors -> StateRoleManifest
```

Providers describe state. They never decide applicability.

## 13. Build And Consumer Contract

### 13.1 Canonical build

`canic build` performs the host package/role validation before invoking Cargo.
It sets one private build marker consumed by `canic::build!` so the build script
knows the outer validation ran. The marker carries no second feature model.

Build scripts do not run Cargo and do not retain a private feature resolver.

A direct authoritative canister wasm `cargo build` without the marker is
hard-cut and fails with guidance to use `canic build`. The marker check applies
only when `canic::build!` is building a Canic package for the canister wasm
target. Host compilation, unit tests, non-wasm IDE checks, and ordinary non-wasm
development remain unaffected and do not claim an authoritative wasm role
contract.

No pre-build/post-build graph fingerprint protocol or persisted evidence
envelope is added in 0.84.

### 13.2 Built-in wasm_store

The actual canonical `canic-wasm-store` manifest or generated wrapper is checked
against the same supported dependency shape before its artifact build. The
wrapper generator has a focused test proving it emits that shape and enables
`wasm-store-canister`.

Passive state/medic inspection reads existing package evidence. If downstream
built-in package evidence does not exist yet, the built-in contract is rejected
as unavailable rather than creating a wrapper as a side effect.

### 13.3 Consumers

- Build validates the contract before compilation.
- Medic renders package-shape, required-feature, and allocation findings.
- Release-set capability labels project typed resolved capabilities.
- State manifest and audit join resolved allocations to descriptors.
- CLI parses and renders host results only.

Unknown or rejected roles never receive a manifest.

Command forms remain unchanged initially:

```text
canic state manifest [--role <role>] [--json]
canic state audit [--role <role>] [--json]
canic medic project [--json|--ci]
canic build <fleet> <role>
```

## 14. Findings

Initial blocking findings are intentionally small:

```text
role_contract_role_unknown
role_contract_package_missing
role_contract_package_ambiguous
role_contract_package_metadata_mismatch
role_contract_dependency_shape_unsupported
role_contract_runtime_canic_dependency_missing
role_contract_multiple_canic_packages
role_contract_canic_version_mismatch
role_contract_cargo_catalog_drift
role_contract_required_feature_missing
role_contract_allocation_descriptor_missing
role_contract_allocation_descriptor_duplicate
role_contract_allocation_id_mismatch
role_contract_memory_id_collision
role_contract_builtin_package_unavailable
```

An enabled state-bearing feature that configuration does not currently require
is valid and produces no finding. Its allocations are selected normally. 0.84
does not add a minimal-feature policy or surplus-feature warning.

Finding codes are added only when a consumer surface needs them. Internal error
variants do not automatically become serialized codes.

## 15. Privacy And Mutation Boundary

Resolution may read:

- the selected `canic.toml`;
- the role package and workspace Cargo manifests;
- the selected Canic manifest;
- local package-scoped Cargo graph metadata needed to reject forbidden paths;
- an existing built-in package or wrapper manifest.

Medic and state resolution must not:

- fetch from the network;
- build packages;
- create a generated wrapper;
- read stable-memory contents;
- query a replica;
- write config, manifests, deployment truth, or reports;
- install, upgrade, stop, or mutate canisters.

`canic build` remains a build command; its compilation is not a resolver side
effect.

## 16. Hard-Cut Migration

Hard-cut means the old path is deleted in the same slice that migrates its last
consumer. No aliases, wrappers, fallbacks, dual reads, or deprecated entry
points remain.

The implementation removes:

- `role_required_canic_features` after capability requirements move to the
  catalog;
- the private build-support dependency feature resolver;
- medic's private dependency feature resolver;
- release-set's raw config-to-capability mapping;
- state-manifest role selectors based on local role names or compiled features;
- any fallback that assigns optional allocations to an unknown role;
- direct authoritative canister wasm builds that bypass canonical role
  validation;
- redundant package-feature forwarding in
  `canisters/test/blob_storage_probe/Cargo.toml`.

Exported opt-in endpoint macros gate their definitions on Canic facade features;
they do not inspect the downstream caller package's feature namespace.

The implementation must not add:

- compatibility modules with old names;
- conversion shims between old and new resolver models;
- fallback direct-edge parsing after host validation fails;
- feature aliases or legacy command forms;
- anti-resurrection tests for removed helpers.

Tests cover the maintained contract only.

## 17. Implementation Slices

### Slice A: Core authority

- Add typed features, capabilities, allocations, result, and findings.
- Add one capability derivation, catalog, allocation registry, and pure resolver.
- Move canonical IDs and the 80-99 control-plane authority range without
  changing active allocation values.
- Add Cargo public feature/default/implication parity tests.
- Hard-cut `role_required_canic_features` when its consumers migrate.
- Change no CLI or JSON output.

### Slice B: Package and build boundary

- Add the host supported-package-shape validator.
- Validate the package-scoped wasm normal-runtime graph only to reject forbidden
  paths and versions.
- Add the canonical build marker and hard-cut direct authoritative canister
  wasm builds.
- Validate canonical and generated-wrapper wasm_store packages.
- Remove redundant blob-probe package feature forwarding.
- Hard-cut build-support and medic feature parsers.

### Slice C: State and consumer convergence

- Convert core and control-plane state providers to allocation-keyed
  descriptors.
- Validate the complete descriptor registry.
- Materialize exact role manifests from resolved allocations.
- Migrate medic and release-set views to projections of the same resolution.
- Hard-cut old state selectors and release-set capability mapping.
- Make any necessary serialized finding changes explicit.

Implemented serialized-surface decision:

- `StateManifest` and `StateAuditReport` remain schema version 1 and retain
  their existing successful JSON fields and labels.
- Role contents are now exact projections of resolved allocations, so optional
  sharding and blob-storage domains appear only for packages that enable them.
- A rejected role contract produces no partial manifest; state audit represents
  the rejection with blocking `role_contract_*` checks.
- Medic replaces `role_required_canic_feature_missing` with the canonical
  `role_contract_required_feature_missing` finding code. Successful required
  feature rows retain `role_required_canic_feature_present`.
- Release capability labels retain their existing strings but are projected
  from `ResolvedRoleContract.capabilities`.

0.84.2 serialized-surface correction:

- `StateManifest` and `StateAuditReport` move to schema version 2. The
  `removed_state` manifest field, removed-state audit category, and all
  removed-state checks are hard-cut.
- Every declared role and the built-in wasm store reports shared runtime IDs
  11-13, 15-18, 20, 29-32, 34, and 39-42 through one `Runtime` capability.
- App-registry ID 14 remains root-only.
- Auth-state ID 19 follows root or configured auth capabilities. ICP-refill ID
  33 follows both its configuration capability and effective package feature.
- Ordinary root contracts retain canister-pool ID 49 but no longer claim
  scaling-registry ID 52 or directory-registry ID 55 unless the root is
  configured with those capabilities.
- Scaling roles report IDs 49 and 52, directory roles report IDs 49 and 55,
  and sharding roles report IDs 49, 53, 54, and 56.
- Replay receipts move from ID 21 to ID 20, reclaiming the former root-replay
  slot; ID 21 becomes unassigned. Existing canisters using the old allocation
  ledger require a destructive reinstall. There is no upgrade migration or
  compatibility path.
- All other active stable-memory IDs, records, encodings, and restore behavior
  remain unchanged.
- Root-named allocation groups and `CoreRootCapacity` are removed without
  aliases. Exact runtime, auth, refill, pool, scaling, and directory keys
  replace them.
- Release capability projection adds `icp_refill` when configured; the
  universal internal `Runtime` capability is not rendered as a release label.

Each slice is reviewable and follows changelog governance. Slice names do not
assign package versions.

## 18. Testing Strategy

### Pure policy tests

- capability derivation is deterministic;
- feature implication closure is deterministic and idempotent;
- missing required features reject resolution;
- an enabled state-bearing feature not required by config resolves normally
  with its allocations and no finding;
- blocking findings return no contract;
- repeated selection of one allocation merges provenance;
- allocation definitions and resolved allocations carry no lifecycle state;
- distinct allocation keys sharing one ID fail;
- output ordering is stable.

### Package-shape tests

Accepted:

- direct normal dependency;
- workspace-inherited dependency;
- renamed dependency resolving to package `canic`;
- defaults enabled and disabled;
- direct public feature list.

Rejected:

- package feature forwards `canic/sharding`;
- optional Canic dependency;
- target-specific Canic dependency;
- transitive normal Canic contribution;
- two runtime paths to Canic;
- two Canic package IDs or versions;
- unsupported package feature selection;
- workspace all-feature state used as evidence.

### Catalog and state tests

- Cargo public features, defaults, and public implications match the catalog;
- every public feature is classified as `NoState` or `StateBearing`;
- every state-bearing feature has allocations;
- every allocation has exactly one descriptor;
- every managed role receives the shared runtime allocations its lifecycle and
  default endpoint surface can use;
- auth state follows root or configured auth capabilities;
- ICP refill requires its package feature and selects ID 33;
- root retains its control-plane allocations;
- root retains canister-pool ID 49 without claiming unconfigured scaling or
  directory registries;
- scaling and directory roles select their exact registry plus shared pool;
- wasm_store receives shared runtime state and retains 80-83 and 85;
- sharding selects 49, 53, 54, and 56;
- blob storage selects 62-64;
- billing selects 65 plus implied blob allocations;
- unrelated roles receive none of those optional allocations;
- unknown `missing` remains blocking under workspace all-feature tests;
- replay receipts use ID 20, ID 21 is absent, and all other active IDs plus
  record encodings remain unchanged.

### Consumer-equivalence tests

For one resolved fixture and one rejected fixture, build, medic, release-set
views, state manifest, and state audit compare against the same
`RoleContractResolution`. Tests do not copy mapping tables into each consumer.

PocketIC is required only if an implementation slice changes real install,
upgrade, or migration behavior. Metadata-only slices use unit and host
integration tests.

## 19. Acceptance Criteria

0.84 is complete only when:

1. One core authority owns config capability, feature, allocation, and ID
   decisions.
2. Only the documented direct dependency shape can resolve.
3. Every unsupported Cargo shape fails closed without partial interpretation.
4. Catalog public features, defaults, and implications match Cargo.
5. Blocking findings cannot return a contract.
6. Repeated allocation selection merges provenance; distinct-key ID collisions
   fail.
7. Allocation definitions and resolved allocations have no lifecycle or
   tombstone field.
8. Distinct active allocation keys cannot share a memory ID.
9. Every allocation has exactly one owner descriptor.
10. Storage modules import canonical IDs and do not redefine them.
11. Built-in wasm_store is validated against its actual build package.
12. Build, medic, release-set, state manifest, and state audit use the same
    resolution.
13. Workspace feature unification cannot assign state to another role.
14. Surplus state-bearing features allocate state without producing a warning.
15. The 0.84.2 replay-ID hard cut and destructive-reinstall requirement are
    explicit; all other active IDs and persisted encodings remain unchanged.
16. Every replaced helper and bypass path is hard-cut with no compatibility
    layer.
17. State and medic remain passive, local, and offline.
18. Serialized output changes, if any, are explicitly reviewed.

## 20. Risks And Mitigations

### Risk: package restrictions reject a legitimate future use

Mitigation: fail clearly and add support only through a later focused design.
Do not quietly broaden 0.84.

### Risk: Cargo and catalog drift

Mitigation: direct parity tests against the actual resolved Canic manifest.

### Risk: central authority becomes a general build system

Mitigation: core owns pure tables and policy; host only validates one supported
package shape. No general graph resolution, generated catalog, or schema
negotiation is introduced.

### Risk: moving constants changes layout

Mitigation: exact values and record encodings are pinned by tests. The one
intentional replay-ID reassignment is documented as a destructive-reinstall
hard cut; moving definition ownership alone is not a migration.

### Risk: an old canister uses the pre-hard-cut replay allocation map

Mitigation: fail through the allocation ledger and require destructive
reinstall. Do not add a migration, alias, dual read, tombstone, or fallback.

## 21. Closed Decisions

- 0.84 supports one simple direct Canic dependency shape.
- Unsupported Cargo shapes fail instead of being generalized.
- The canonical build target is `wasm32-unknown-unknown` with package defaults.
- Direct authoritative canister wasm builds outside `canic build` are hard-cut;
  host and non-wasm development paths are unaffected.
- One core module owns configuration capabilities, feature rules, allocations,
  and memory IDs.
- Host-only package evidence stays in `canic-host`.
- Control-plane records stay in `canic-control-plane`; canonical IDs move to
  `canic-core`.
- Cargo parity is checked directly; no digest or schema protocol is added.
- Blocking findings never return a contract.
- Memory IDs are canonical assignments for the active pre-1.0 map. A map
  reassignment is an explicit breaking hard cut requiring reinstall, not a
  retained tombstone or compatibility path.
- Built-in wasm_store is explicit and checked against its actual build package.
- Placement state is capability-scoped: root, scaling, directory, and sharding
  contracts share pool ID 49 where applicable, while IDs 52 and 55 follow only
  their exact configured capabilities.
- Enabled state-bearing features allocate state even when configuration does
  not currently require them.
- All replaced helpers and unsupported paths are removed without aliases,
  fallbacks, or compatibility shims.
- Implementation proceeded only after maintainer acceptance and is complete in
  the current worktree.
