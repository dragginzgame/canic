# Canic 0.81 Design: Runtime Introspection Pack

Date: 2026-07-04

## Status

- Status: implementation specification
- Release line: tentative 0.81 runtime introspection label
- Decision style: security-sensitive runtime observability
- Primary implementation target: expose typed, guarded, no-mutation runtime
  inspection for deployed Canic-managed canisters after installation.
- Current state: Canic has local medic checks, deploy plans, state manifests,
  deployment-truth reports, metrics, logs, auth renewal status, blob-storage
  status, and readiness concepts. These are useful, but there is not yet one
  coherent runtime-observed status surface for a deployed canister's own view
  of itself.

## 1. Summary

0.81 is the post-deploy counterpart to the previous operator-platform releases:

```text
0.78 medic:
  local/deployment readiness diagnosis

0.79 deploy plan:
  desired deployment shape before mutation

0.80 state audit:
  stable-state and migration metadata

0.81 runtime introspection:
  live deployed-canister status after installation
```

The operator should be able to ask:

```text
Is this canister healthy?
Is it ready for its role?
What role and root does it believe it has?
What feature flags are active?
What timers are registered and healthy?
What state schema metadata is active?
What recent failures should I look at first?
```

0.81 crosses from local/no-mutation metadata into live runtime observation.
That makes it security-sensitive. Runtime introspection must not become a
public topology, version, feature-flag, failure, auth, timer, or state
information leak.

## Hard-Cut Rule

0.81 exposes exactly the canonical runtime introspection surfaces listed here.
It does not add compatibility endpoints, fallback method names, command aliases,
runtime-introspection version shims, or alternate accepted report shapes.

The 0.81 JSON payloads include `schema_version = 1`, but code and docs use
unversioned DTO names. This release does not introduce `V1`/`V2` compatibility
layers or dual decoders.

## 2. Goals

1. Define named DTO structs/enums for health, readiness, runtime status, timer
   status, state summary, and recent failures.
2. Define field-level visibility before implementing endpoints.
3. Define endpoint authorization before exposing runtime data.
4. Add `canic_health` for one Canic-owned role.
5. Add guarded `canic_readiness` for one Canic-owned role.
6. Add guarded `canic_runtime_status` for one Canic-owned role.
7. Keep public health intentionally minimal and configurable.
8. Report role, root/topology bindings, build/version metadata, features,
   timers, state metadata, and recent failures only through guarded runtime
   status fields.
9. Add CLI inspection commands that consume typed endpoint responses.
10. Label runtime-observed values separately from local config, deployment
   records, and deployment truth.
11. Let medic point to runtime introspection or consume it only for explicitly
    scoped deployment checks.

## 3. Non-Goals

0.81 does not:

1. Add a hosted dashboard.
2. Replace structured logs or metrics.
3. Expose secrets, session material, token material, private keys, raw request
   payloads, raw application records, or state dumps.
4. Make health, readiness, or runtime status imply deployment safety.
5. Add broad deployment fanout by default.
6. Add a root endpoint that queries every child canister.
7. Add mutation, repair, renewal, rotation, or cleanup behavior to status
   endpoints.
8. Loosen endpoint authentication rules.
9. Require every downstream app to expose every optional introspection field.
10. Add durable recent-failure storage.
11. Add `canic topology` or a separate topology command.
12. Add a state explorer or state dump.

## 4. Field Visibility

0.81 classifies fields, not only endpoints.

Visibility classes:

```text
public_safe
operator_only
controller_only
feature_gated
disabled
```

Default field visibility:

| Field | Visibility | Notes |
| --- | --- | --- |
| `schema_version` | public_safe | Exposed by all DTOs. |
| minimal health `status` | public_safe | Only if public health is enabled. |
| `observed_at_ns` | public_safe | Optional; may be omitted from public health. |
| `canic_version` | operator_only | Version can leak vulnerability posture. |
| role | operator_only | Runtime role is topology metadata. |
| root principal | operator_only or controller_only | Depends on role and deployment policy. |
| network | operator_only | Avoid public environment fingerprinting. |
| topology bindings | controller_only | Includes parent/child/root relationships. |
| feature flags | operator_only | May leak enabled auth/storage surfaces. |
| timer names/status | operator_only | May leak operational cadence/failure shape. |
| recent failures | operator_only | Must be bounded and redacted. |
| state schema versions | operator_only | Metadata only, no persisted values. |
| stable memory pages | operator_only | Metadata only, no record contents. |
| auth renewal status | operator_only or controller_only | Never public. |
| blob-storage billing readiness | operator_only or feature_gated | Never public. |

No endpoint becomes unauthenticated merely because it is "only status."

## 5. Endpoint Surface

0.81 canonical endpoints:

```text
canic_health
canic_readiness
canic_runtime_status
```

Endpoint semantics:

```text
canic_health
  Minimal liveness/health check.
  Public-safe only if explicitly configured.
  No topology, auth, timer, state, feature, version, or failure details.

canic_readiness
  Guarded readiness check for the local canister role.
  Answers whether this canister can perform its configured Canic role now.
  Does not imply deployment safety.

canic_runtime_status
  Guarded operator/controller runtime inspection surface.
  Includes optional sections for role, root, build, features, topology,
  timers, state metadata, auth, blob-storage, and recent failures where
  visibility policy allows them.
```

Deferred endpoints:

```text
canic_topology_status
canic_auth_status
canic_state_status
canic_timer_status
```

0.81 uses optional sections inside `canic_runtime_status` instead of adding
role-specific endpoint sprawl.

## 6. Authorization Matrix

Default authorization:

| Endpoint | Default access | Public fields |
| --- | --- | --- |
| `canic_health` | configurable public-safe or guarded | `schema_version`, minimal `status`, optional `observed_at_ns` |
| `canic_readiness` | operator/controller guarded | none by default |
| `canic_runtime_status` | operator/controller guarded | none |

Rules:

1. Public health is opt-in and must expose only public-safe fields.
2. Public health exposure is controlled by canister/runtime config or a
   compile-time feature declaration, not by endpoint default behavior.
3. Missing configuration must not accidentally expose public health. Default
   public-health behavior is guarded or disabled.
4. Readiness and runtime status require explicit endpoint guards.
5. Unauthorized callers must not learn whether guarded fields exist. The error
   shape must not distinguish "section present but unauthorized", "section
   disabled", and "feature not compiled" unless the caller is authorized.
6. Feature-gated sections are omitted unless the feature is enabled and the
   caller is authorized for the field class.
7. Endpoint guards must be tested with PocketIC when endpoint behavior is
   implemented.

## 7. DTO Model

Use named structs and enums, not boundary type aliases or loose maps.

Top-level DTOs:

```text
CanicHealthStatus
  schema_version
  status
  observed_at_ns
  checks

CanicReadinessStatus
  schema_version
  role
  status
  observed_at_ns
  checks
  blockers
  warnings

CanicRuntimeStatus
  schema_version
  observed_at_ns
  canister_id
  role
  root
  network
  build
  features
  topology
  timers
  state
  auth
  blob_storage
  recent_failures
  visibility
```

Status enums:

```text
HealthStatus:
  healthy
  degraded
  unhealthy
  unknown

ReadinessStatus:
  ready
  degraded
  not_ready
  not_evaluated

RuntimeStatus:
  ok
  degraded
  failing
  unknown

TimerStatus:
  healthy
  delayed
  failing
  disabled
  not_registered
  unknown

FailureSeverity:
  info
  warning
  error
  critical
```

Forbidden status language:

```text
safe
safe_to_deploy
ready_to_apply
production_ready
```

## 8. CLI Surface

0.81 supported forms:

```text
canic inspect canister <principal>
canic inspect canister <principal> --json

canic inspect deployment <deployment> --role <role>
canic inspect deployment <deployment> --role <role> --json
```

0.81 rejected forms:

```text
canic inspect
canic inspect <ambiguous-target>
canic inspect deployment <deployment>
canic inspect deployment <deployment> --all
canic topology <deployment>
canic runtime status <deployment>
```

Targeting rules:

1. Direct canister inspection requires a principal.
2. Deployment inspection requires an explicit deployment and role.
3. 0.81 does not inspect all roles by default.
4. 0.81 does not support ambiguous `canic inspect <target>` shorthand.
5. All structured output uses `--json`, matching 0.78 through 0.80.
6. `canic inspect` calls `canic_runtime_status` by default. Health and
   readiness modes are not exposed as CLI flags in 0.81 unless separately
   designed.

Target-resolution failures:

1. Deployment not found is a command failure.
2. Role not found is a command failure.
3. Role mapping to zero canisters is a command failure.
4. Role mapping to multiple canisters is a command failure unless a later
   disambiguation flag is explicitly designed.
5. Resolved canister lacking the runtime-status endpoint is a command failure
   with a next action.

## 9. CLI Output And Exit Codes

When `--json` is used, stdout contains only JSON. Human-readable diagnostics,
usage errors, authorization failures, transport errors, and internal errors go
to stderr.

Text output writes the rendered inspection report to stdout.

Exit codes:

```text
0:
  report retrieved and rendered successfully with healthy, ready, ok, or
  degraded status

1:
  report retrieved and rendered successfully with unhealthy, not_ready, or
  failing status

2:
  usage error, invalid target, authorization failure, endpoint unavailable,
  invalid response, transport failure, or internal report-construction failure
```

## 10. Inspect Report Shape

CLI report wrapper:

```text
InspectReport
  schema_version
  command
  target_resolution
  endpoint
  status
  health_status
  readiness_status
  runtime_status
  warnings
  next_actions
```

Target resolution:

```text
TargetResolution
  deployment
  role
  canister_id
  source
```

Only one of `health_status`, `readiness_status`, or `runtime_status` is
populated for a given inspect mode. 0.81 `canic inspect` uses
`canic_runtime_status` by default.

## 11. Source Attribution

CLI output must distinguish where each fact came from.

Source values:

```text
runtime_observed
local_config
deployment_record
deployment_truth
cli_arg
unknown
```

If the CLI resolves a canister principal from installed deployment state, that
resolution is a `deployment_record` fact. The runtime status fields returned by
the canister are `runtime_observed` facts. The renderer must not silently fill
runtime-observed fields from local desired config.

Example JSON shape:

```json
{
  "schema_version": 1,
  "command": "canic inspect deployment",
  "target_resolution": {
    "deployment": "demo-local",
    "role": "root",
    "canister_id": "aaaaa-aa",
    "source": "deployment_record"
  },
  "runtime_status": {
    "source": "runtime_observed",
    "status": "degraded"
  }
}
```

## 12. Deterministic Output

Text and JSON output should be deterministic where practical.

Ordering rules:

1. Feature flags are ordered by feature name.
2. Timers are ordered by subsystem, then timer name.
3. State domains are ordered by domain.
4. Recent failures are ordered newest first.
5. Checks, blockers, and warnings are ordered by status/severity, code, and
   subject where practical.

## 13. Timer Status

Timer status is common runtime infrastructure with per-subsystem registrations.

Timer row shape:

```text
CanicTimerStatus
  name
  subsystem
  status
  enabled
  registered
  last_success_at_ns
  last_failure_at_ns
  next_due_at_ns
  consecutive_failures
  last_error_code
  last_error_summary
```

Privacy rules:

1. `last_error_summary` is bounded and redacted.
2. No raw payloads.
3. No token, session, or user identifiers.
4. No principals unless the field is explicitly classified as safe for the
   endpoint guard.

## 14. Recent Failures

0.81 recent failures are heap-only, bounded, redacted, and best-effort.

They are not stable state in the first slice.

Recent failure shape:

```text
RecentFailure
  occurred_at_ns
  subsystem
  code
  severity
  summary
  correlation_id
  redacted
```

Bounds:

```text
max_recent_failures = 16 or 32
max_summary_bytes = small fixed limit
cleared_on_upgrade = true
cleared_on_reinstall = true
```

Recent failures must not include raw request args, token claims, session
material, private keys, full stack traces with secrets, raw application
records, or user-specific payloads.

`correlation_id` must not be user-identifying unless separately classified as
safe for the endpoint guard. `redacted = true` indicates that the summary was
sanitized, truncated, or otherwise privacy-filtered before exposure.

## 15. State Summary

0.81 may reuse 0.80 state declarations, but it must not expose application
data.

Allowed state summary fields:

```text
manifest_schema_version
domain names
domain schema versions
storage mode
memory ID
stable memory pages
migration/invariant status
```

Per-domain shape:

```text
RuntimeStateDomainSummary
  domain
  version
  storage
  memory_id
  status
```

Forbidden in 0.81 unless separately designed:

```text
record values
state dumps
raw stable memory bytes
counts derived from application records
session counts
user counts
billing object counts
application-specific record counts
failed auth attempt counts
```

Allowed counts are limited to metadata such as stable-memory pages and declared
state-domain inventory. User, session, billing, object, auth-attempt, and other
application-record counts are potentially sensitive and forbidden by default.

## 16. Non-Mutation Contract

Runtime introspection endpoints and CLI inspect commands must not:

1. install or upgrade Wasm;
2. create or delete canisters;
3. change controllers;
4. register root or child topology;
5. write deployment truth;
6. create or update installed deployment records;
7. write generated files;
8. modify config files;
9. modify stable memory;
10. run migrations;
11. repair timers;
12. rotate keys;
13. renew auth proofs;
14. change feature flags;
15. clear recent failures.

The endpoints may read in-memory runtime metadata and safe persisted metadata.
They must not become repair endpoints.

## 17. Medic Relationship

0.81 may update `canic medic deployment <deployment>` next actions to point at:

```text
canic inspect deployment <deployment> --role <role>
```

Medic may call runtime introspection only for explicitly selected deployment
and role/canister scopes. Medic must not fan out across all child canisters by
default.

0.81 first slice should prefer next-action integration. Live medic runtime
checks can follow only when the authorization, latency, and failure-mode
contract is explicit.

## 18. Implementation Slices

1. Define DTOs and field visibility classes.
2. Define endpoint authorization matrix.
3. Add minimal `canic_health` to one Canic-owned role.
4. Add guarded `canic_readiness` to one Canic-owned role.
5. Add guarded `canic_runtime_status` to one Canic-owned role.
6. Add timer registry and recent-failure ring for Canic-owned timers.
7. Add state summary from 0.80 declarations where available.
8. Add explicit CLI target forms under `canic inspect`.
9. Add JSON and text renderers with source attribution.
10. Add role/topology section for root and one managed child role.
11. Add medic next-action integration only.
12. Add live medic runtime checks only in a separately scoped follow-up.

Do not start with broad deployment fanout, topology aggregation, or a dashboard.

## 19. Validation

Focused validation should cover:

```text
cargo test --locked -p canic-core runtime --lib
cargo test --locked -p canic-control-plane runtime --lib
cargo test --locked -p canic --test protocol_surface
cargo test --locked -p canic-cli inspect
cargo test --locked -p canic-cli medic
```

PocketIC coverage is required when endpoint behavior, endpoint authorization,
role topology, or inter-canister runtime status is implemented.

Security boundary tests must cover:

1. Public health omits operator-only fields.
2. Missing public-health configuration does not expose unauthenticated public
   health.
3. Readiness requires the intended guard.
4. Runtime status requires the intended guard.
5. Unauthorized error shapes do not reveal whether guarded sections are present,
   disabled, or not compiled.
6. Unauthorized callers cannot access topology, features, timers, state,
   auth, blob-storage, or recent failures.
7. Recent failures are bounded, redacted, and expose `redacted` status.
8. State summary contains metadata only and no application-derived counts.
9. CLI output distinguishes runtime-observed data from local config and
   deployment records.
10. Inspect commands do not mutate state.
11. Medic does not fan out across all children by default.
12. `--json` writes JSON only to stdout and diagnostics to stderr.
13. Inspect exit codes distinguish successful degraded reports from
    unhealthy/not-ready/failing reports and command failures.
14. Deployment-role target resolution failures are deterministic and do not
    fall back to guessed roles or broad fanout.

## 20. Acceptance Criteria

Close 0.81 only when:

1. Common runtime DTOs exist with named structs/enums.
2. Field visibility is declared.
3. Authorization matrix is documented and tested.
4. `canic_health` exists for one Canic-owned role.
5. `canic_readiness` exists for one Canic-owned role and is guarded by default.
6. `canic_runtime_status` or equivalent guarded endpoint exists for one role.
7. Public health, if enabled, exposes only public-safe fields.
8. Runtime status exposes no secrets, session material, token material, private
   keys, raw application state, or state dumps.
9. Timer status exists for at least one Canic-owned timer.
10. Recent failures are bounded, redacted, and heap-only or explicitly
   justified.
11. State summary uses metadata from 0.80 declarations and does not expose
    records.
12. CLI inspect commands consume typed endpoint responses.
13. CLI labels runtime-observed data separately from local config and
    deployment records.
14. CLI JSON/text stream behavior and exit-code behavior are defined and
    tested.
15. CLI output ordering is deterministic for features, timers, state domains,
    recent failures, checks, blockers, and warnings where practical.
16. Medic integration is limited to next actions or explicitly scoped
    deployment checks.
17. No endpoint or CLI path mutates canister state, config, deployment truth,
    installed state, stable memory, timers, auth proofs, keys, or controllers.
18. PocketIC coverage exists for endpoint authorization and at least one runtime
    status call.

## 21. Deferred

Deferred unless separately designed:

- hosted dashboard;
- broad deployment fanout;
- root endpoint querying every child;
- top-level `canic topology`;
- durable recent-failure storage;
- raw metrics export replacement;
- state record counts;
- state explorer or state dump;
- automatic repair or renewal actions;
- unauthenticated detailed readiness;
- downstream-app-required introspection fields.

## 22. Closed Decisions

1. Only minimal health is public-safe, and public health is opt-in.
2. Readiness and runtime status are guarded by default.
3. 0.81 canonical endpoints are `canic_health`, `canic_readiness`, and
   `canic_runtime_status`.
4. First-slice implementation includes minimal guarded `canic_readiness` for
   one Canic-owned role.
5. 0.81 CLI surfaces are `canic inspect canister <principal>` and
   `canic inspect deployment <deployment> --role <role>`.
6. 0.81 `canic inspect` uses `canic_runtime_status` by default.
7. 0.81 uses `--json`, not `--format json`.
8. Public health exposure must come from explicit canister/runtime config or
   compile-time declaration; missing config exposes nothing public.
9. Recent failures are heap-only, bounded, redacted, and best-effort.
10. Timer status is common runtime infrastructure with per-subsystem
   registrations.
11. Runtime-observed values must not be silently filled from local config.
12. Medic integration starts as next-action or explicitly scoped live checks,
   not broad fanout.
