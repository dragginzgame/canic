# Canic Auth v0.65 Design: Basic Zero-ECDSA Protocol

Implementation status for this design line lives in
[status.md](status.md). This file describes only the active Canic 0.65 auth
protocol.

## 1. Decision

Canic 0.65 hard-cuts normal auth away from management-canister threshold ECDSA.
Normal auth must not call `management_canister.sign_with_ecdsa`.

Active Canic 0.65 auth uses IC canister signatures for:

- root-certified issuer authority: `DelegationProof`
- issuer-certified user/session tokens: `IssuerProof`
- root-certified service role proofs: `SignedRoleAttestation`

The preserved external contract is that callers send a self-contained
`DelegatedToken` to protected endpoints. Opaque application data travels inside
the signed token claims, not beside the claims:

```rust
pub struct DelegatedToken {
    pub claims: DelegatedTokenClaims,
    pub proof: DelegationProof,
    pub issuer_proof: IssuerProof,
}
```

Endpoint verification is local. A protected endpoint must not call root, the
issuer, or the management canister while verifying auth.

## 2. Scope

Active Canic 0.65 includes:

- root batch delegation-proof prepare, direct root-query retrieval, and batch
  install for out-of-band issuer authority provisioning
- issuer `install_active_delegation_proof` update
- issuer `prepare_delegated_token` update plus `get_delegated_token` query for
  normal user/session/signup token issuance
- reusable self-contained `DelegatedToken`
- issuer-signed opaque token `ext` bytes inside `DelegatedTokenClaims`
- bounded positive delegated-token verifier cache
- minimal root `prepare_role_attestation` update plus `get_role_attestation`
  query for service-to-service role authorization
- local verifier checks for delegated tokens and role attestations
- legacy threshold-ECDSA and standalone delegated-grant proof success paths are
  absent

Not included in Canic 0.65:
- standalone delegated-grant capability proofs
- internal-invocation proofs
- topology epoch proofs
- emergency revocation or policy epoch proofs
- multi-subtree certified_data
- not-yet-valid delegation cert staging

Canic 0.65 includes `DelegatedRoleGrant` values inside `DelegationCert` and
`DelegatedTokenClaims`. Canic 0.65 does not include standalone delegated-grant
capability proofs. Legacy delegated-grant proof inputs are rejected before
execution.

## 3. Non-Negotiable Invariants

- No normal auth path calls `management_canister.sign_with_ecdsa`.
- Root is not called per signup, per ordinary token issuance, or per protected
  endpoint verification.
- Issuers do not call root while issuing ordinary user/session/signup tokens.
- Protected endpoint verification is local and synchronous.
- A `DelegatedToken` is reusable until TTL, audience, or scope checks reject it.
- `DelegatedRoleGrant` token grants remain active and are checked locally.
- Legacy proof shapes are removed from normal auth success paths.

## 4. Feature Layout

Normal auth features describe canister-signature capabilities. They do not
enable threshold-ECDSA signing.

```toml
[features]
default = []
auth-root-canister-sig-create = [
    "dep:ic-canister-sig-creation",
    "dep:ic-certification",
]
auth-root-canister-sig-verify = ["dep:ic-signature-verification"]
auth-issuer-canister-sig-create = [
    "dep:ic-canister-sig-creation",
    "dep:ic-certification",
]
auth-issuer-canister-sig-verify = ["dep:ic-signature-verification"]
auth-delegated-token-verify = [
    "auth-root-canister-sig-verify",
    "auth-issuer-canister-sig-verify",
]
```

Normal auth removes these feature names:

```text
auth-threshold-ecdsa-public-key
auth-threshold-ecdsa-sign
auth-shard-secp256k1-verify
```

`auth-crypto` is not a public normal-auth umbrella in this hard cut.

Role requirements:

- root auth issuer: `auth-root-canister-sig-create` and
  `auth-root-canister-sig-verify`
- token issuer: `auth-root-canister-sig-verify` and
  `auth-issuer-canister-sig-create`
- endpoint verifier: `auth-delegated-token-verify`
- endpoint verifier for service roles: `auth-root-canister-sig-verify`

## 5. Trust Anchors

Verifier config is shared by delegated-token and role-attestation verification:

```toml
[auth.root_proofs]
root_canister_id = "..."
ic_root_public_key_raw_hex = "..."
network = "mainnet" # mainnet | local | pocketic | testnet
```

Runtime verification resolves operator-facing hex into raw IC root key bytes:

```rust
pub struct RootProofVerifierConfig {
    pub root_canister_id: Principal,
    pub ic_root_public_key_raw: Vec<u8>,
    pub network: RootProofNetwork,
}
```

Verifier startup traps unless required verifier features and trust anchors are
available. Protected endpoints never fetch trust anchors, root canister ids,
issuer state, registry snapshots, or IC root keys during request handling.

Startup validates the configured network/root-key pairing:

- `network = "mainnet"` requires the known mainnet IC root public key
- `network = "local"` or `network = "pocketic"` requires that environment's
  root public key configured explicitly as `ic_root_public_key_raw_hex`
- a mismatch traps at startup
- tests must not verify local or PocketIC certificates with the mainnet root
  key

## 6. Proof DTOs

```rust
pub enum RootProof {
    IcCanisterSignatureV1(IcCanisterSignatureProof),
}

pub enum IssuerProof {
    IcCanisterSignatureV1(IcCanisterSignatureProof),
}

pub struct IcCanisterSignatureProof {
    pub public_key_der: Vec<u8>,
    pub signature_cbor: Vec<u8>,
}

pub struct DelegationProof {
    pub cert: DelegationCert,
    pub root_proof: RootProof,
}
```

Audience DTOs are explicit and remain separate from role or grant authority:

```rust
pub enum DelegationAudience {
    Canister(Principal),
    CanicSubnet(Principal),
    Project(String),
}

pub enum RoleAttestationAudience {
    Canister(Principal),
    CanicSubnet(Principal),
}
```

`cert.aud` covers `claims.aud` when:

- `Canister(a)` covers `Canister(b)` iff `a == b`
- `CanicSubnet(a)` covers `CanicSubnet(b)` iff `a == b`
- `Project(a)` covers `Project(b)` iff `a == b`
- no cross-kind coverage exists in Canic 0.65

`claims.aud` matches the local verifier when:

- `Canister(p)` matches iff `p == this_canister`
- `CanicSubnet(s)` matches iff `s == configured local_canic_subnet`
- `Project(name)` matches iff `name == configured local_project_audience`

`RoleAttestationAudience::Canister(p)` matches iff `p == this_canister`.
`RoleAttestationAudience::CanicSubnet(s)` matches iff
`s == configured local_canic_subnet`.

`local_canic_subnet` and `local_project_audience` are verifier
config/deployment state. They are not fetched during protected verification.

Role remains separate from audience. Audience says where a token or proof may
be used; role, grants, and scopes say what authority it carries.

`DelegationCert` is the root-certified statement that an issuer canister may
issue bounded delegated tokens for a specific audience and grant set:

```rust
pub struct DelegationCert {
    pub root_pid: Principal,
    pub issuer_pid: Principal,
    pub issuer_proof_alg: IssuerProofAlgorithm,
    pub issuer_proof_binding_hash: [u8; 32],
    pub issuer_proof_binding: IssuerProofBinding,
    pub issued_at_ns: u64,
    pub not_before_ns: u64,
    pub expires_at_ns: u64,
    pub max_token_ttl_ns: u64,
    pub aud: DelegationAudience,
    pub grants: Vec<DelegatedRoleGrant>,
}

pub enum IssuerProofAlgorithm {
    IcCanisterSignatureV1,
}

pub enum IssuerProofBinding {
    IcCanisterSignatureV1 {
        seed_hash: [u8; 32],
    },
}
```

`IssuerProofAlgorithm` has a single variant by design. Future issuer proof
algorithms add variants here.

Issuer binding invariants:

```text
seed_hash = sha256(issuer_canister_sig_seed(IssuerPayloadKind::DelegatedTokenClaims))

issuer_proof_binding_hash =
  sha256(
    "canic-issuer-proof-binding-v1" ||
    principal_bytes(issuer_pid) ||
    canonical_bytes(issuer_proof_alg) ||
    canonical_bytes(issuer_proof_binding)
  )
```

The root derives or verifies these fields before certifying `DelegationCert`.
Endpoint verifiers check the binding hash against the certified binding and
the issuer proof public key seed.

Token claims bind to the root-certified cert by hash and carry a subset of the
cert grants:

```rust
pub struct DelegatedTokenClaims {
    pub subject: Principal,
    pub issuer_pid: Principal,
    pub cert_hash: [u8; 32],
    pub issued_at_ns: u64,
    pub expires_at_ns: u64,
    pub aud: DelegationAudience,
    pub grants: Vec<DelegatedRoleGrant>,
    pub nonce: [u8; 16],
    pub ext: Option<Vec<u8>>,
}
```

`DelegatedTokenClaims.nonce` is deterministic issuer-generated uniqueness
material. It makes otherwise identical claims produce distinct `claims_hash`
values without relying on management-canister randomness. It is not secret, not
a replay key, and not an authorization input. Core delegated auth does not
track nonce reuse.

`prepare_delegated_token` must not call the management canister, including
`raw_rand`, to produce `nonce`. The construction is:

```text
nonce = first_16_bytes(
  sha256(
    "canic-token-nonce-v1" ||
    principal_bytes(msg.caller()) ||
    operation_id ||
    principal_bytes(claims.subject) ||
    principal_bytes(issuer_pid) ||
    selected_cert_hash
  )
)
```

Replay protection already makes `(caller, operation_id)` unique for committed
prepares. An implementation may include an issuer-local monotonic counter only
if the counter is stable or replay-receipt-backed enough to avoid reuse across
successful prepares.

Verifier rule:

```text
claims.grants must be a subset of cert.grants
```

Endpoint authorization checks local required scopes against
`DelegatedTokenClaims.grants`.

### Application ext convention, informative

`ext` is issuer-signed opaque application data. It remains inside
`DelegatedTokenClaims` and is included in `canonical_bytes(DelegatedTokenClaims)`,
so it is covered by `claims_hash` and `IssuerProof`.

Core delegated auth does not interpret `ext` for root proof validity, issuer
proof validity, audience checks, grant subset checks, or expiry checks. Endpoint
application code may parse `ext` only after delegated-token cryptographic
verification succeeds.

For Toko application use, the recommended app encoding includes a version byte
or versioned key. Application data may include a stable Toko user id, wallet
principal, tenant or project id, session kind, or onboarding state.

Endpoints that authorize based on `ext` must treat expected keys and values as
local endpoint policy. The core verifier does not make authority decisions from
`ext`.

Active ext limit:

```rust
pub const MAX_TOKEN_EXT_BYTES: usize = 4096;
```

Do not put authority fields in `ext`. Grants, audience, expiry, subject,
issuer, cert hash, and nonce remain first-class claim fields.

Role attestations use the same root-certified wrapper:

```rust
pub struct RoleAttestation {
    pub subject: Principal,
    pub role: CanicRole,
    pub audience: RoleAttestationAudience,
    pub issued_at_ns: u64,
    pub expires_at_ns: u64,
    pub nonce: [u8; 16],
}

pub struct RootCertified<T> {
    pub payload: T,
    pub root_proof: RootProof,
}

pub type SignedRoleAttestation = RootCertified<RoleAttestation>;
```

## 7. Canonical Payloads And Signature Messages

All signed auth DTO timestamps and TTLs are nanoseconds since the Unix epoch.
Protocol fields use `_ns` suffixes. Operator-facing config can remain seconds,
but conversion happens once at the boundary with checked arithmetic.

Root payload kinds active in Canic 0.65:

```rust
pub enum RootPayloadKind {
    DelegationCert,
    RoleAttestation,
}

fn root_canister_sig_domain(kind: RootPayloadKind) -> &'static [u8] {
    match kind {
        RootPayloadKind::DelegationCert => b"canic-root-delegation-cert",
        RootPayloadKind::RoleAttestation => b"canic-root-role-attestation",
    }
}

fn root_canister_sig_seed(kind: RootPayloadKind) -> &'static [u8] {
    match kind {
        RootPayloadKind::DelegationCert => b"canic-root-delegation-cert",
        RootPayloadKind::RoleAttestation => b"canic-root-role-attestation",
    }
}
```

Issuer payload kinds active in Canic 0.65:

```rust
pub enum IssuerPayloadKind {
    DelegatedTokenClaims,
}

fn issuer_canister_sig_domain(kind: IssuerPayloadKind) -> &'static [u8] {
    match kind {
        IssuerPayloadKind::DelegatedTokenClaims => b"canic-issuer-delegated-token",
    }
}

fn issuer_canister_sig_seed(kind: IssuerPayloadKind) -> &'static [u8] {
    match kind {
        IssuerPayloadKind::DelegatedTokenClaims => b"canic-issuer-delegated-token",
    }
}
```

The IC canister-signature creation helper stores:

```text
message_hash = sha256(domain_len || domain || payload_hash)
```

Therefore verifiers pass `domain_len || domain || payload_hash` to
`verify_canister_sig`, not the raw payload hash.

This message hashing is only for canister-signature paths inside the signature
map. It is not the IC hash-tree labeled hash and must not be used to compute
the canister `certified_data` value.

```rust
pub fn root_canister_sig_verification_message(
    kind: RootPayloadKind,
    payload_hash: [u8; 32],
) -> Vec<u8> {
    let domain = root_canister_sig_domain(kind);
    assert!(domain.len() <= 255);

    let mut msg = Vec::with_capacity(1 + domain.len() + 32);
    msg.push(domain.len() as u8);
    msg.extend_from_slice(domain);
    msg.extend_from_slice(&payload_hash);
    msg
}

pub fn issuer_canister_sig_verification_message(
    kind: IssuerPayloadKind,
    payload_hash: [u8; 32],
) -> Vec<u8> {
    let domain = issuer_canister_sig_domain(kind);
    assert!(domain.len() <= 255);

    let mut msg = Vec::with_capacity(1 + domain.len() + 32);
    msg.push(domain.len() as u8);
    msg.extend_from_slice(domain);
    msg.extend_from_slice(&payload_hash);
    msg
}
```

`cert_hash` is:

```text
sha256(canonical_bytes(DelegationCert))
```

`claims_hash` is:

```text
sha256(canonical_bytes(DelegatedTokenClaims))
```

Prepared signature retrieval is bounded:

```rust
pub const SIGNATURE_RETRIEVAL_TTL_NS: u64 = 60_000_000_000;
```

Every prepare/get flow stores `retrieval_expires_at_ns =
prepare_now_ns + SIGNATURE_RETRIEVAL_TTL_NS` with checked arithmetic.

## 8. Root Proof Refresh

Root proof refresh is maintenance, bootstrap, or refresh traffic. It is not a
signup, ordinary token issuance, or endpoint verification path.

Current root proof provisioning uses the 0.68 batch contract:

```text
controller/provisioner -> root canic_upsert_root_issuer_policy update
controller/provisioner -> root canic_prepare_delegation_proof_batch update
controller/provisioner -> root canic_get_delegation_proof_batch direct query
controller/provisioner -> root canic_install_delegation_proof_batch update
root                   -> issuer canic_install_active_delegation_proof update
caller/session         -> issuer canic_prepare_delegated_token update
caller/session         -> issuer canic_get_delegated_token query
```

`canic_prepare_delegation_proof_batch`:

- authorizes the root-controller caller before replay reservation,
  signature-map insertion, certified-data refresh, or pending-proof write
- requires each issuer to be registered through root issuer policy
- validates issuer policy, allowed audiences, allowed grants, cert TTL, and
  refresh timing
- builds one `DelegationCert` per requested issuer
- computes each `cert_hash`
- adds root canister-signature map entries for prepared cert hashes
- refreshes certified data
- stores bounded pending batch metadata
- is request-id keyed and idempotent for the same request

`canic_get_delegation_proof_batch`:

- must be called as a direct root query so root has certificate context
- reads existing pending batch metadata
- requires requested `batch_id`, issuer, and `cert_hash` to match pending
  metadata
- rejects expired pending retrieval
- returns prepared `DelegationProof` material
- does not mutate state
- does not call any canister

`canic_install_delegation_proof_batch`:

- validates submitted proofs against pending metadata
- rejects proof, issuer, or expiry mismatches
- broadcasts valid proofs to issuer `canic_install_active_delegation_proof`
  endpoints
- returns per-issuer install outcomes
- is idempotent for already-installed active proofs

The retired single-proof root `canic_prepare_delegation_proof` and
`canic_get_delegation_proof` endpoints are not part of the active protocol.
Root proof retrieval must not be hidden behind issuer composite-query wrappers,
issuer wrappers, root self-calls, or canister-to-canister calls.

Pending retrieval TTL is not auth TTL. It only controls how long a prepared
signature can be retrieved. Auth validity comes from signed payload fields.

### Replay and cost dependency

Canic 0.65 builds on Canic replay protection for
`canic_prepare_delegation_proof_batch`, `prepare_delegated_token`, and
`prepare_role_attestation`. These are `ReplayProtected` commands and must
follow Canic replay/cost-guard ordering.

Canic/Toko 0.65 inherits Canic 0.61 replay-protection semantics for
`canic_prepare_delegation_proof_batch`, `prepare_delegated_token`, and
`prepare_role_attestation`. These are `ReplayProtected` commands using the
shared replay receipt/cost-guard model, not ad hoc pending-map deduplication.

Duplicate behavior:

- same `(caller, operation_id)` while the first prepare is in flight returns
  `OperationInProgress`
- same `(caller, operation_id)` after commit with identical request returns the
  committed prepared response
- same `(caller, operation_id)` after commit with different request returns
  replay conflict

Required ordering:

1. authorize caller
2. reserve or check replay key
3. enforce cost/quota guard
4. perform first successful signature-map insertion once
5. refresh certified data once
6. store committed replay receipt
7. identical committed replay returns the same prepared response
8. same replay key with different request content returns replay conflict

Cost classes:

```rust
pub enum CostClass {
    RootCanisterSignaturePrepare,
    IssuerCanisterSignaturePrepare,
}
```

## 9. ActiveDelegationProof

Issuer canisters must maintain valid root-certified delegation material or fail
closed.

```rust
pub struct ActiveDelegationProof {
    pub proof: DelegationProof,
    pub cert_hash: [u8; 32],
    pub not_before_ns: u64,
    pub expires_at_ns: u64,
    pub refresh_after_ns: u64,
    pub installed_at_ns: u64,
    pub installed_by: Principal,
}
```

Install API:

```rust
#[update]
pub fn install_active_delegation_proof(
    req: InstallActiveDelegationProofRequest,
) -> Result<InstallActiveDelegationProofResponse, AuthError>;
```

Install rules:

- verify the supplied `DelegationProof` locally
- require root proof verification against configured root canister id plus raw
  IC root key
- require `cert.issuer_pid == this_canister`
- require issuer proof binding hash to match the canonical binding
- require `cert.not_before_ns <= now_ns + AUTH_TIME_SKEW_ALLOWANCE_NS`
- require `now_ns < cert.expires_at_ns`
- reject expired certs with no expiry grace
- reject certs farther in the future than `AUTH_TIME_SKEW_ALLOWANCE_NS`
- reject proofs for another issuer
- do not call root
- do not call the management canister
- do not mutate certified data unless issuer state changes

The emitted issuer install endpoint is controller-gated. In the current MVP,
root broadcasts installs to issuer canisters where root is authorized as a
controller. Ordinary users do not install delegation proofs as part of signup or
token issuance.

### Active proof storage and refresh operations

`ActiveDelegationProof` is stable issuer state and must survive issuer upgrade.
On issuer init and post-upgrade, the issuer reloads stable active proof state
and rebuilds or resyncs certified data for its signature map before serving
signature queries.

`PendingDelegatedToken` is heap-only and ephemeral. If an issuer upgrade occurs
after `prepare_delegated_token` and before `get_delegated_token`, the caller
retries prepare with a new `operation_id`.

Root pending delegation-proof batch metadata is bounded root state with a short
retrieval window. `PendingRoleAttestation` is heap-only and ephemeral unless an
implementation explicitly persists it.

`refresh_after_ns` is local operational metadata, not auth validity. An active
proof remains usable until `cert.expires_at_ns` as long as local verification
still accepts it.

`refresh_after_ns` is computed as `expires_at_ns` minus a configured refresh
margin, plus deterministic issuer/cert jitter bounded to remain before expiry.
Reaching `refresh_after_ns` should emit refresh-due metrics or alerts and may
trigger an authorized refresh workflow, but ordinary token issuance must not
synchronously call root.

`installed_by` is audit metadata and may be emitted in install metrics or logs.
It does not grant future replacement authority by itself.

The provisioner retrieves root batch proof material through a direct root query
and asks root to install it into issuer canisters. Issuer timers may report
refresh need, but issuer timers cannot manufacture root query certificates or
witnesses and must not drive the root batch provisioning flow synchronously
from ordinary token issuance or protected verification.

If active proof material is missing or expired, `prepare_delegated_token` fails
closed with `RootDelegationProofUnavailable`. Issuers increment
`active_delegation_proof_unavailable_count` when issuance fails for this reason.
Operators must alert and follow a runbook whenever
`active_delegation_proof_unavailable_count > 0`: verify refresh actor health,
root proof preparation, issuer install permissions, cert expiry, and per-issuer
installed active proof state before retrying rollout.

Phase A rollout must prove every issuer canister has a valid
`ActiveDelegationProof` installed.

## 10. Issuer Token Issuance

Normal fresh token issuance is issuer-local and uses update/query:

```text
1. caller -> issuer.prepare_delegated_token update
2. issuer selects a valid ActiveDelegationProof
3. issuer prepares IssuerProof::IcCanisterSignatureV1 over DelegatedTokenClaims
4. caller -> issuer.get_delegated_token query
5. caller uses DelegatedToken at endpoint boundary
```

### Sign-in and session token issuance

The default Canic 0.65 sign-in path is session-direct token issuance:

```text
1. wallet -> orchestrator_hub.authenticate(session_principal)
2. orchestrator_hub validates wallet/session/app state
3. orchestrator_hub returns issuer_pid plus approved token request parameters
   or policy
4. session principal calls issuer.prepare_delegated_token
5. session principal calls issuer.get_delegated_token
6. session principal uses DelegatedToken at protected endpoints
```

Normal issuer prepare still requires `req.subject == msg.caller()`. For
sign-in and session tokens, `subject` is the session principal. The wallet
principal or Toko user id may be included only in signed `ext` if application
policy needs it.

The hub does not call root during signup or sign-in. The issuer does not call
root during token issuance. `signup_path_root_call_count` remains zero, and the
integration suite must cover this flow.

Hub-orchestrator token issuance for a foreign subject is not the default Canic
0.65 path. If later needed, it must be a separate explicit endpoint with
authorized hub callers, replay policy, metrics, and tests. Do not leave a vague
admin/service/recovery branch as the only description.

Prepare API:

```rust
#[update]
pub fn prepare_delegated_token(
    req: DelegatedTokenPrepareRequest,
) -> Result<DelegatedTokenPrepareResponse, AuthError>;

pub struct DelegatedTokenPrepareRequest {
    pub metadata: Option<AuthRequestMetadata>,
    pub subject: Principal,
    pub aud: DelegationAudience,
    pub grants: Vec<DelegatedRoleGrant>,
    pub ttl_ns: u64,
    pub ext: Option<Vec<u8>>,
}

pub struct DelegatedTokenPrepareResponse {
    pub claims: DelegatedTokenClaims,
    pub claims_hash: [u8; 32],
    pub retrieval_expires_at_ns: u64,
}
```

Normal `prepare_delegated_token` rules:

- require `req.subject == msg.caller()`
- select a valid `ActiveDelegationProof`
- build `DelegatedTokenClaims`
- set `claims.cert_hash` to selected cert hash
- set `claims.issuer_pid` to this issuer
- clamp `ttl_ns` by issuer policy, `cert.max_token_ttl_ns`, and
  `cert.expires_at_ns`
- require requested audience to be covered by `cert.aud`
- require requested grants to be covered by `cert.grants`
- validate requested `ext` against issuer and application policy
- enforce `ext` size bounds
- copy approved `ext` into `DelegatedTokenClaims.ext`
- derive `claims.nonce` locally from caller, operation id, subject, issuer,
  and selected cert hash
- add issuer canister-signature map entry for `claims_hash`
- refresh certified data
- store pending token metadata
- use replay protection keyed by `(caller, operation_id)`
- do not accept caller-provided nonce bytes
- do not call root
- do not call the management canister
- do not fetch issuer or root keys

```rust
pub struct PendingDelegatedToken {
    pub operation_id: [u8; 32],
    pub prepared_by: Principal,
    pub claims_hash: [u8; 32],
    pub claims: DelegatedTokenClaims,
    pub proof: DelegationProof,
    pub retrieval_expires_at_ns: u64,
}
```

Get API:

```rust
#[query]
pub fn get_delegated_token(
    req: GetDelegatedTokenRequest,
) -> Result<DelegatedToken, AuthError>;

pub struct GetDelegatedTokenRequest {
    pub claims_hash: [u8; 32],
}
```

`get_delegated_token` rules:

- look up pending token by caller and `claims_hash`
- require `pending.prepared_by == msg.caller()`
- reject expired pending retrieval
- return `DelegatedToken`
- include the root `DelegationProof` selected during prepare
- include `IssuerProof::IcCanisterSignatureV1` over the claims hash
- do not call root
- do not call the management canister
- do not mutate state

The pending token returns the root proof selected during prepare. It must not
rebuild from the current active delegation cache during query retrieval.

## 11. Role Attestations

Canic 0.65 keeps role attestations for service-to-service authorization.

```rust
#[update]
pub fn prepare_role_attestation(
    req: RoleAttestationIssueRequest,

pub struct RoleAttestationIssueRequest {
    pub operation_id: [u8; 32],
    pub subject: Principal,
    pub role: CanicRole,
    pub audience: RoleAttestationAudience,
    pub requested_ttl_ns: u64,
}

pub struct PreparedRoleAttestation {
    pub attestation: RoleAttestation,
    pub attestation_hash: [u8; 32],
    pub retrieval_expires_at_ns: u64,
}

#[query]
pub fn get_role_attestation(
    req: GetRoleAttestationRequest,
) -> Result<SignedRoleAttestation, AuthError>;

pub struct GetRoleAttestationRequest {
    pub attestation_hash: [u8; 32],
}
```

Role attestation rules:

- normal issuance requires `req.subject == msg.caller()`
- admin or refresh issuance for another subject must be an explicit authorized
  branch
- root verifies the requested subject, role, subnet, and audience against
  authoritative Canic state
- root adds a canister-signature map entry for `attestation_hash`
- root refreshes certified data
- pending retrieval is keyed by caller and `attestation_hash`
- `get_role_attestation` requires `pending.prepared_by == msg.caller()`
- expired pending retrievals fail
- query retrieval does not mutate state or call any canister

```rust
pub struct PendingRoleAttestation {
    pub operation_id: [u8; 32],
    pub prepared_by: Principal,
    pub attestation_hash: [u8; 32],
    pub attestation: RoleAttestation,
    pub retrieval_expires_at_ns: u64,
}
```

For caller authorization, verifiers require:

- `SignedRoleAttestation.root_proof` verifies against configured root canister
  id plus raw IC root key
- root proof seed/domain is `RootPayloadKind::RoleAttestation`
- `subject == msg.caller()`
- role matches the endpoint requirement
- audience matches the local verifier using `RoleAttestationAudience` matching
  rules
- `issued_at_ns <= now_ns + AUTH_TIME_SKEW_ALLOWANCE_NS`
- `now_ns < expires_at_ns`

The skew allowance accepts a fresh role attestation from a root clock that is
slightly ahead of the verifier clock. It does not extend `expires_at_ns`.

For service-to-service authorization, the default is one subnet-wide
`SignedRoleAttestation` per subject role, Canic subnet, and TTL. Per-verifier
role attestations are allowed only when a narrower audience is required.

Protected role verification never calls root or the management canister.

## 12. Certified Data

Each auth issuer has one certified-data owner. Root and token issuers certify
only the canister-signature map in Canic 0.65:

```rust
use ic_certification::hash_tree::labeled_hash;

set_certified_data(labeled_hash(b"sig", SIGNATURES.root_hash()));
```

`labeled_hash` here is `ic_certification::hash_tree::labeled_hash`. Do not use
`hash_with_domain(domain, message)` for the certified-data root.

`get_signature_as_cbor(..., None)` is valid only for exactly this certified
data shape.

`SignatureMap` is heap-only. `certified_data` is replicated system state and
persists across upgrades. On init and post-upgrade, each auth issuer must
rebuild or reset `SIGNATURES`, then call `set_certified_data(...)` again before
serving signature queries. Prepared tickets that were not retrieved before
upgrade may fail and are retried by preparing again; they are not stable
promises.

Release review audits `set_certified_data` call sites when auth certified-data
ownership changes. Canic 0.65 does not add a permanent source guard for retired
certified-data layouts.

If a root or issuer canister needs certified HTTP assets or another certified
response in the same canister, that is out of scope for Canic 0.65 and requires
a multi-subtree `certified_data` design.

## 13. Local Verification

Delegated-token verification inputs:

- supplied `DelegatedToken`
- `msg.caller()`
- current canister time
- configured root canister id
- raw IC root public key
- local audience, role, and scope policy

Verifier future-skew allowance:

```rust
pub const AUTH_TIME_SKEW_ALLOWANCE_NS: u64 = 60_000_000_000;
```

This allowance applies only to "not from the future" checks. Expiry checks
remain strict: no delegated cert, delegated token, or role attestation gets
post-expiry grace.

Delegated-token verification:

- verifies root proof over `DelegationCert`
- verifies issuer proof over `DelegatedTokenClaims`
- requires root proof public key canister id equals configured root canister id
- requires root proof public key seed equals
  `root_canister_sig_seed(RootPayloadKind::DelegationCert)`
- requires root verifier message equals
  `domain_len || root_canister_sig_domain(RootPayloadKind::DelegationCert) || cert_hash`
- requires issuer proof public key canister id equals `claims.issuer_pid`
- requires issuer proof public key seed equals
  `issuer_canister_sig_seed(IssuerPayloadKind::DelegatedTokenClaims)`
- requires issuer verifier message equals
  `domain_len || issuer_canister_sig_domain(IssuerPayloadKind::DelegatedTokenClaims) || claims_hash`
- requires issuer proof binding seed hash equals
  `sha256(issuer_canister_sig_seed(IssuerPayloadKind::DelegatedTokenClaims))`
- requires `issuer_proof_binding_hash` to match the full certified issuer
  authority context
- requires `claims.cert_hash == sha256(canonical_bytes(DelegationCert))`
- requires `claims.issuer_pid == cert.issuer_pid`
- requires `claims.subject == msg.caller()`
- requires `cert.not_before_ns < cert.expires_at_ns`
- requires `cert.max_token_ttl_ns > 0`
- requires `cert.not_before_ns <= now_ns + AUTH_TIME_SKEW_ALLOWANCE_NS`
- requires `now_ns < cert.expires_at_ns`
- requires `claims.issued_at_ns >= cert.not_before_ns`
- requires `claims.issued_at_ns <= now_ns + AUTH_TIME_SKEW_ALLOWANCE_NS`
- requires `now_ns < claims.expires_at_ns`
- requires `claims.expires_at_ns > claims.issued_at_ns`
- requires token expiry not to exceed cert expiry or `cert.max_token_ttl_ns`
- requires `claims.aud` to be covered by `cert.aud`
- requires `claims.aud` to match local verifier audience policy
- requires `claims.grants` to be a subset of `cert.grants`
- rejects oversized `claims.ext`
- checks endpoint/local required scopes against `claims.grants`
- leaves endpoint-specific `ext` authorization to local endpoint policy after
  cryptographic verification succeeds

`DelegatedToken` is not an on-behalf-of delegation mechanism. A user token is
valid only when presented by `claims.subject` as `msg.caller()`.
Canister-to-canister forwarding of a user's `DelegatedToken` intentionally
fails because the downstream verifier sees the forwarding canister as
`msg.caller()`. Toko frontend flows call protected canisters directly with the
user or session principal. Service-to-service calls must use
`SignedRoleAttestation` or a future explicit on-behalf-of protocol, not
forwarded user tokens.

Verifier failures must be test-addressable with typed errors:

```rust
pub enum DelegatedTokenVerifyError {
    WrongRootCanister,
    WrongRootSeed,
    WrongRootDomain,
    InvalidRootCertificate,
    CertHashMismatch,
    WrongIssuerCanister,
    WrongIssuerSeed,
    WrongIssuerBinding,
    InvalidIssuerCertificate,
    SubjectCallerMismatch,
    CertExpired,
    TokenExpired,
    TokenOutlivesCert,
    TokenTtlOverflow,
    AudienceNotCovered,
    AudienceRejected,
    GrantsNotSubset,
    MissingRequiredScope,
    ExtTooLarge,
    ExtPolicyRejected,
}
```

Each verifier rejection path above should have a negative test asserting the
corresponding typed error.

Time arithmetic must be checked. Never write unchecked
`now_ns + AUTH_TIME_SKEW_ALLOWANCE_NS` or
`claims.issued_at_ns + cert.max_token_ttl_ns` inside auth verification.

```rust
let max_claim_expiry = claims
    .issued_at_ns
    .checked_add(cert.max_token_ttl_ns)
    .ok_or(AuthVerifyError::TimeOverflow)?;

require(claims.expires_at_ns <= max_claim_expiry)?;
```

Verification APIs are local functions. They must not be async and must not
accept root clients, issuer clients, or management-canister adapters.

## 14. Positive Verifier Cache

Canic 0.65 includes a bounded positive delegated-token verifier cache for
production performance.

The cache is an optimization only, not an authority source. Cache miss means
full local verification of the self-contained root and issuer proofs.

The cache is heap-only and not stable.

Cache key:

```text
proof_hash        = sha256(canonical_bytes(DelegationProof))
claims_hash       = sha256(canonical_bytes(DelegatedTokenClaims))
issuer_proof_hash = sha256(canonical_bytes(IssuerProof))

key =
  sha256(
    "canic-delegated-token-cache-v1" ||
    proof_hash ||
    claims_hash ||
    issuer_proof_hash ||
    caller_principal_bytes
  )
```

Cache value:

```rust
pub struct CachedDelegatedTokenProofValidity {
    pub valid_until_ns: u64,
    pub verified_at_ns: u64,
}
```

`verified_at_ns` is diagnostic/metrics metadata only. It may be used to report
cache-hit age and verifier-cache freshness. It is not an authorization input,
not an expiry field, and not a replay field.

Cache config:

```rust
pub struct DelegatedTokenVerifierCacheConfig {
    pub max_entries: usize,
    pub max_entry_ttl_ns: u64,
}
```

`valid_until_ns` is no later than:

```text
min(
  token.expires_at_ns,
  cert.expires_at_ns,
  now_ns + cache_config.max_entry_ttl_ns,
  local verifier policy cache expiry
)
```

The cache key remains `proof_hash + claims_hash + issuer_proof_hash + caller`.
A cache hit may skip root and issuer `verify_canister_sig` only for the exact
tuple. Endpoint-specific authorization still runs after cache hit.

Do not cache parsed certificates, witnesses, proof objects, grants, audiences,
`ext`, endpoint authorization decisions, or `ext` authorization decisions.

Evict expired entries first, then least-recently used entries. Increment
`delegated_token_verifier_cache_eviction_count` on eviction.

A local policy version bump either invalidates the cache or is represented in
`valid_until_ns` through the local verifier policy cache expiry.

The supplied token still carries the claims; the cache key proves those exact
signed claims and proofs were previously verified for the caller.

Because the cache key includes `claims_hash`, different `ext` data naturally
misses the cache. On cache hit, endpoint code may read `ext` from the supplied
token because the cache key binds the exact signed claims that were previously
verified.

## 15. Revocation Model

Canic 0.65 does not implement emergency token revocation before signed expiry.
A leaked token remains valid until `expires_at_ns` unless local checks reject
it. A locally valid self-contained token remains valid until its signed expiry.

A compromised issuer can issue tokens within its current `DelegationCert`
audience and grant ceiling until the cert expires. This is a Canic 0.65
trust-boundary failure. There is no protocol-level emergency revocation for
self-contained tokens before signed expiry.

### TTL and client refresh policy

Operational mitigations:

- short token TTLs
- strict `max_token_ttl_ns`
- short root delegation cert TTLs
- local audience, scope, and role checks on every endpoint call

Production TTL guidance:

- delegated user/session tokens SHOULD NOT exceed 15 minutes
- signup/bootstrap tokens SHOULD NOT exceed 5 minutes
- root delegation cert TTLs should be short enough that an issuer compromise
  has a bounded blast radius
- longer token TTLs require an explicit product/security exception

`requested_ttl_ns` is clamped by issuer policy, `cert.max_token_ttl_ns`, and
`cert.expires_at_ns`. Fleet config and client storage/reissue behavior must
match the configured `max_token_ttl_ns`. If existing clients assume 24-hour
tokens, Phase A or Phase B rollout must explicitly update or exception that
behavior.

## 19. Rollout

Rollout is a two-phase operational state machine. This is not a compatibility
protocol lane: the final Canic 0.65 state still rejects old proof shapes.

Phase A: capability deployment

Preconditions:

- old verification remains deployed temporarily
- root and issuers can create canister-signature proofs
- clients have not been forced to switch yet

Steps:

1. reduce old threshold-ECDSA token TTL if applicable
2. deploy root canister-signature support
3. deploy issuer canister-signature support
4. install valid `ActiveDelegationProof` material into every issuer canister
5. prove every issuer can run `prepare_delegated_token` and
   `get_delegated_token`
6. prove signup/sign-in issuance produces zero root calls
7. keep old verification deployed only as a temporary deployment compatibility
   window

Phase A success criteria:

- every issuer can issue new-shape `DelegatedToken`
- `ActiveDelegationProof` is installed on every issuer canister
- one installed `ActiveDelegationProof` cert contains all
  `DelegatedRoleGrant` targets the issuer will issue, such as user shard,
  project hub, and project instance grants; or multiple active proof rows and
  deterministic selection rules are documented
- no ordinary issuance path calls root
- role-upgrade token issuance produces zero root calls when covered by installed
  certs
- `management_ecdsa_sign_count_normal_auth == 0` for new paths
- `signup_path_root_call_count == 0` for session-direct signup/sign-in tests
- root outage with valid `ActiveDelegationProof` still allows issuer token
  issuance
- root outage with missing or expired `ActiveDelegationProof` fails closed and
  increments `active_delegation_proof_unavailable_count`
- Phase A exit date or flag is documented

Phase B: hard-cut cutover

Preconditions:

- Phase A success criteria are met
- clients are ready to use issuer prepare/get
- operators accept old in-flight tokens may fail after cutover or have waited
  old max TTL

Default Phase B cutover policy:

Canic/Toko 0.65 SHOULD wait `max_old_token_ttl` after clients switch to
new-shape issuer prepare/get before deploying verifier hard-rejection of
old-shape tokens.

Emergency cutover may skip the wait only when the deployment plan explicitly
declares immediate invalidation, expected user impact, and re-authentication
behavior.

Steps:

1. switch clients to issuer prepare/get tokens
2. wait `max_old_token_ttl` after client switch, unless the emergency cutover
   plan explicitly declares immediate invalidation
3. upgrade endpoint verifiers to the new token shape
4. remove old normal-auth ECDSA config and features
5. delete legacy proof push and root-probe paths from fleet docs and code

Phase B success criteria:

- new-shape tokens authenticate
- old-shape tokens have no active success path
- `signup_path_root_call_count == 0`
- `protected_verification_root_call_count == 0`
- `management_ecdsa_sign_count_normal_auth == 0`
- no normal path uses `mint_token`, `setup_delegation` root probes, or proof
  push
- root unavailable with valid `ActiveDelegationProof` still allows issuer token
  issuance and endpoint verification
- generated Candid and docs expose only the new `DelegatedToken` success path
- legacy DTOs and proof-mode branches are removed

Legacy fleet cleanup:

- remove `mint_token` one-shot root proof paths
- remove `setup_delegation` root probes per role target
- remove post-project-mutation delegation refresh inter-canister calls from
  normal auth paths
- remove verifier-target proof push documentation
- generated Candid and docs expose only the new `DelegatedToken` success path
- legacy DTOs and proof-mode branches are removed

Do not add compatibility DTOs for removed auth proof families.

Old verification deployed during Phase A is temporary deployment compatibility
only. The final Canic 0.65 protocol has no legacy acceptance branch. Verifier
upgrade before issuer and client upgrade breaks outstanding old-shape delegated
tokens by design.

## 20. Final Target State

Root:

- canister-signs typed `DelegationCert` and `RoleAttestation` payloads
- never signs arbitrary caller bytes
- never calls `management_canister.sign_with_ecdsa` for normal auth
- handles maintenance/bootstrap proof refresh traffic
- is not called per signup, per endpoint verification, or per ordinary token
  issuance

Token issuers:

- maintain `ActiveDelegationProof`
- canister-sign typed `DelegatedTokenClaims`
- issue delegated tokens from cached root delegation material
- fail closed when no valid cached authority exists
- never synchronously call root from ordinary token issuance
- never call `management_canister.sign_with_ecdsa` for normal auth

Delegated tokens:

- carry claims, root proof, and issuer proof
- carry `DelegatedRoleGrant` values for local scope checks
- carry issuer-signed opaque application `ext` bytes inside claims
- are not forwardable or on-behalf-of credentials
- verify locally at endpoints
- are reusable until TTL, audience, or scope checks reject them

Role attestations:

- use `SignedRoleAttestation = RootCertified<RoleAttestation>`
- verify locally against configured root canister id plus raw IC root key
- authorize callers only when `subject == msg.caller()`
- default to subnet-wide service proofs when possible

Verifiers:

- validate embedded root and issuer canister signatures
- bind token subject to IC caller
- check audience, role, expiry, and `DelegatedRoleGrant` scopes locally
- use bounded positive cache for expensive delegated-token proof validity
- never call root, issuer, or management canister during protected endpoint
  verification

Legacy:

- threshold-ECDSA normal-auth proofs have no active success path
- standalone delegated-grant capability proofs have no active success path
- old delegated tokens expire by hard cut, not migration grace
