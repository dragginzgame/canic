# Canic 0.78 Design: Top-Level Medic Preflight

Date: 2026-07-01

## Status

- Status: tentative design seed
- Release line: tentative 0.78 planning label
- Decision style: operator-readiness consolidation
- Primary implementation target: hard-cut medic into the top-level
  `canic medic` preflight and diagnostic surface for local project and
  deployment readiness.
- Current state: `canic info medic` exists as a local installed-deployment
  diagnostic. It checks local network selection, ICP CLI compatibility,
  identity-session hints, installed deployment state, config path existence,
  root readiness, targeted blob-storage billing readiness, and targeted
  chain-key auth renewal drift. `canic status`, v1 readiness docs, deployment
  check, deployment-truth reports, and backup/restore preflight code already
  contain adjacent readiness concepts.

## 1. Summary

0.78 should hard-cut `canic medic` into a top-level command from the start. The
command should turn Canic's implicit setup rules into explicit PASS/WARN/FAIL
checks without mutating canisters, installing Wasm, changing controllers, or
creating deployment truth.

The existing `canic info medic` command should be removed when the top-level
surface lands. There should be no compatibility command, no shim, and no alias.

The first slice should be practical: consolidate existing local readiness checks
behind a top-level command, add a stable check taxonomy, and prepare the output
shape for CI without trying to solve every future preflight category at once.

## Hard-Cut Rule

`canic medic` replaces the old command surface. The only supported command
forms are the top-level forms listed in this design.

## 2. Goals

1. Promote `canic medic` to the canonical command surface.
2. Preserve current targeted diagnostics for blob-storage billing and auth
   renewal.
3. Add project-level checks that do not require a known installed deployment.
4. Keep deployment-level checks explicit and selected by deployment target.
5. Classify checks by stable category, status, code, subject, detail, and next
   action.
6. Support human-readable output first and structured JSON soon after.
7. Reuse existing deployment-truth, status, install-state, and config-reading
   helpers instead of creating a parallel diagnostic stack.
8. Make local/mainnet mismatches and unsupported deployment shapes visible
   before an operator runs mutating commands.

## 3. Non-Goals

0.78 does not:

1. Keep `canic info medic` as a supported command.
2. Add shorthand aliases for `canic medic deployment <deployment>`.
3. Install, upgrade, register, repair, or delete canisters.
4. Turn a successful medic run into deployment truth.
5. Replace `canic status`, `canic deploy check`, or policy gates.
6. Require live mainnet calls for ordinary local project checks.
7. Add a supervised daemon or background health monitor.
8. Make auth-specific follow-up work part of this release line.

## 4. Command Surface

Canonical and only supported surface:

```text
canic medic
canic medic project
canic medic deployment <deployment>
canic medic deployment <deployment> --blob-storage <canister-or-role>
canic medic deployment <deployment> --auth-renewal <issuer-principal>
```

Hard-cut removals:

1. `canic info medic <deployment>` is removed.
2. `canic medic <deployment>` is not accepted as shorthand.
3. All docs and examples use the explicit top-level surface.

## 5. Check Model

Each check should have:

```text
category
code
status
subject
detail
next
source
```

Suggested statuses:

- `pass`
- `warn`
- `fail`
- `not_evaluated`

Suggested initial categories:

- `environment`
- `project_config`
- `deployment_state`
- `network`
- `artifact`
- `topology`
- `feature`
- `auth`
- `blob_storage`
- `runtime`

The text renderer can continue using compact labels, but JSON should expose the
stable status names.

## 6. Initial Checks

Project checks:

1. ICP CLI is available and compatible.
2. Current ICP project root can be resolved.
3. `icp.yaml` is present enough for Canic local workflows.
4. Canic fleet configs can be discovered.
5. Fleet names, deployment names, and role names are not conflated.
6. Attached roles have package metadata.
7. Declared roles are not silently treated as deployable artifacts.
8. Local network selection is explicit when local-only assumptions are used.

Deployment checks:

1. Installed deployment target exists.
2. Deployment target belongs to the selected network.
3. Recorded config path still exists.
4. Root canister ID is present when required.
5. Local root readiness can be queried when the local replica is reachable.
6. Existing blob-storage and auth renewal targeted diagnostics continue to run.
7. Deployment-truth assumptions are surfaced when a local plan cannot fully
   resolve facts.

Deferred checks:

1. Stable memory ID collision checks.
2. Generated-file staleness.
3. Full feature flag consistency.
4. Unsupported deployment-shape classification.
5. Mainnet authority and controller preflight.

## 7. Output

Text output should stay readable:

```text
project config [pass]
  detail: found 1 fleet config

deployment state [fail]
  detail: no installed deployment target named demo-staging on local
  next: run canic deploy inspect catalog list
```

JSON output should be stable enough for automation:

```json
{
  "schema_version": 1,
  "command": "canic medic deployment",
  "network": "local",
  "deployment": "demo-staging",
  "status": "fail",
  "checks": [
    {
      "category": "deployment_state",
      "code": "deployment_target_missing",
      "status": "fail",
      "subject": "demo-staging",
      "detail": "no installed deployment target named demo-staging on local",
      "next": "run canic deploy inspect catalog list",
      "source": "installed_deployment"
    }
  ]
}
```

## 8. Implementation Slices

1. Add top-level CLI parser for `canic medic`.
2. Remove the `canic info medic` command route and usage examples.
3. Rename internal types from info-specific names to top-level medic names.
4. Add check codes and categories while keeping the text renderer readable.
5. Add `canic medic project` with non-live local checks.
6. Add JSON output after the check model is stable.
7. Move all docs and examples to `canic medic`.

## 9. Validation

Focused validation should cover:

```text
cargo test --locked -p canic-cli medic
cargo test --locked -p canic-cli status
cargo test --locked -p canic-host deployment_truth --lib
```

Broaden when project checks start reading build artifacts, deployment-truth
plans, or live replica state.

## 10. Open Questions

1. Should `canic medic` default to project checks, deployment checks for a
   resolved active deployment, or both?
2. Should JSON use `warn` or `warning` as the stable status value?
3. Should medic output be wrapped in `EvidenceEnvelopeV1`, or should it remain
   diagnostic-only until a separate evidence design approves it?
4. Which checks are warnings locally but failures for mainnet?
