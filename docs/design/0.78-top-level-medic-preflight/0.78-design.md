# Canic 0.78 Design: Top-Level Medic Preflight

Date: 2026-07-01

## Status

- Status: tentative design seed
- Release line: tentative 0.78 planning label
- Decision style: operator-readiness consolidation
- Primary implementation target: hard-cut medic into the top-level
  `canic medic` preflight and diagnostic surface for local project and
  deployment readiness.
- Current state: `canic info medic` exists as a local installed-deployment
  diagnostic. It checks local network selection, ICP CLI compatibility,
  identity-session hints, installed deployment state, config path existence,
  root readiness, targeted blob-storage billing readiness, and targeted
  chain-key auth renewal drift. `canic status`, v1 readiness docs, deployment
  check, deployment-truth reports, and backup/restore preflight code already
  contain adjacent readiness concepts.

## 1. Summary

0.78 should be a consolidation release, not an expansion release.

The release hard-cuts medic into a top-level operator command:

```text
canic medic
```

Medic turns Canic's implicit setup rules into explicit PASS/WARN/FAIL checks.
It remains diagnostic-only. A successful medic run means no blocking preflight
issue was found for the selected scope. It does not mean the deployment is safe,
does not create deployment truth, and does not authorize any mutation.

The existing `canic info medic` command is removed when the top-level surface
lands. There is no compatibility command, no shim, no alias, and no fallback
route.

## Hard-Cut Rule

`canic medic` replaces the old command surface. The only supported command
forms are the top-level forms listed in this design.

Hard-cut removals:

1. `canic info medic <deployment>` is removed.
2. `canic medic <deployment>` is rejected.
3. `canic medic --blob-storage <target>` is rejected.
4. `canic medic --auth-renewal <issuer>` is rejected.
5. `canic medic project --blob-storage <target>` is rejected.
6. `canic medic project --auth-renewal <issuer>` is rejected.

## 2. Command Semantics

Supported forms:

```text
canic medic
canic medic project
canic medic deployment <deployment>
canic medic deployment <deployment> --blob-storage <canister-or-role>
canic medic deployment <deployment> --auth-renewal <issuer-principal>
```

All supported forms accept `--json`.

`canic medic` runs project-level checks only. It does not require an installed
deployment. It does not make live canister calls by default. It may emit
deployment checks as `not_evaluated` with next steps.

`canic medic project` runs the same project-level check set as `canic medic`.
This is an explicit canonical form, not a compatibility alias.

`canic medic deployment <deployment>` runs project-level checks plus
deployment-state checks for the named deployment target. Deployment checks are
never inferred from a guessed active deployment.

`canic medic deployment <deployment> --blob-storage <canister-or-role>` runs
project and deployment checks plus targeted blob-storage billing readiness
checks.

`canic medic deployment <deployment> --auth-renewal <issuer-principal>` runs
project and deployment checks plus targeted chain-key auth renewal drift
checks.

## 3. Command Boundaries

Medic has a separate job from nearby commands:

```text
canic status
  Reports known local project and installed deployment state.

canic deploy check
  Produces passive deployment-check evidence for a selected deployment target.

canic evidence gate
  Evaluates saved evidence against policy.

canic medic
  Diagnoses operator readiness and explains why Canic may not be able to plan,
  inspect, or deploy safely.
```

Medic output should avoid final deployment-safety language. Prefer:

```text
No blocking preflight issues found.
```

Do not say:

```text
Deployment is safe.
```

## 4. Non-Mutation Contract

Medic must not:

1. Write deployment truth.
2. Create deployment records.
3. Update installed deployment state.
4. Install or upgrade Wasm.
5. Create or delete canisters.
6. Change controllers.
7. Register root or child relationships.
8. Modify config files.
9. Write generated files.
10. Perform repair actions.

Medic may read:

1. Config files.
2. Installed deployment records.
3. Deployment-truth reports.
4. Local replica query responses when deployment-scoped checks require them.
5. Existing build or artifact metadata when future checks require it.

## 5. Goals

1. Promote `canic medic` to the canonical command surface.
2. Remove `canic info medic` instead of keeping a compatibility route.
3. Preserve current targeted diagnostics for blob-storage billing and auth
   renewal under the new top-level surface.
4. Add project-level checks that do not require a known installed deployment.
5. Keep deployment-level checks explicit and selected by deployment target.
6. Classify checks by stable category, code, status, subject, detail, next, and
   source.
7. Ship human-readable text output and JSON output in 0.78.
8. Reuse existing deployment-truth, status, install-state, and config-reading
   helpers instead of creating a parallel diagnostic stack.
9. Make local/mainnet mismatches and unsupported deployment shapes visible
   before an operator runs mutating commands.

## 6. Non-Goals

0.78 does not:

1. Keep `canic info medic` as a supported command.
2. Add shorthand aliases for `canic medic deployment <deployment>`.
3. Install, upgrade, register, repair, or delete canisters.
4. Turn a successful medic run into deployment truth.
5. Replace `canic status`, `canic deploy check`, or policy gates.
6. Require live mainnet calls for ordinary local project checks.
7. Add a supervised daemon or background health monitor.
8. Make auth-specific follow-up work part of this release line.
9. Add mainnet authority/controller preflight.
10. Add `--fail-on-warn`.

## 7. Report Model

0.78 includes `--json` output with `schema_version = 1`. The schema is
intentionally small. Later releases may add fields, but the initial fields
should remain stable.

The report model:

```text
MedicReport
  schema_version
  command
  scope
  network
  deployment
  status
  checks
```

`scope` values:

```text
project
deployment
```

JSON always includes `schema_version`, `command`, `scope`, `network`,
`deployment`, `status`, and `checks`. `network` and `deployment` are nullable
when not applicable. Medic must not force a fake `local` network into
project-only output.

Each check has:

```text
category
code
status
subject
detail
next
source
```

Stable status enum:

```text
pass
warn
fail
not_evaluated
```

Do not use both `warn` and `warning`.

Suggested categories:

```text
environment
project_config
deployment_state
network
artifact
topology
feature
auth
blob_storage
runtime
```

Initial `source` values:

```text
command
icp_cli
icp_config
fleet_config
installed_deployment
deployment_truth
local_replica
blob_storage_readiness
auth_renewal
```

Medic output is diagnostic-only in 0.78. Do not wrap it in
`EvidenceEnvelopeV1` until a separate evidence/truth design explicitly says
medic output can become durable evidence.

## 8. Aggregate Status

Aggregate status is derived from checks:

1. `fail` if any check is `fail`.
2. `warn` if no checks fail and at least one check is `warn`.
3. `pass` if all evaluated checks pass.
4. `not_evaluated` only if no checks were evaluated.

`not_evaluated` checks do not downgrade a report by themselves unless every
check is `not_evaluated`.

Example:

```json
{
  "status": "warn",
  "checks": [
    { "status": "pass" },
    { "status": "not_evaluated" },
    { "status": "warn" }
  ]
}
```

## 9. Check Ordering

Checks render in deterministic category order:

1. `environment`
2. `project_config`
3. `network`
4. `deployment_state`
5. `topology`
6. `artifact`
7. `feature`
8. `auth`
9. `blob_storage`
10. `runtime`

Within a category, checks are emitted in implementation-defined but stable
order.

## 10. Network Selection

Medic should reuse Canic's existing network selection model instead of
inventing its own. The intended precedence is:

1. Explicit CLI network flag, if present.
2. Deployment record network, for deployment-scoped checks.
3. Current ICP local network selection, for local project checks.
4. `not_evaluated` if no network can be resolved.

For deployment-scoped checks, a selected network that conflicts with the
deployment record is a `fail`.

For project-only checks, no selected network is acceptable. Checks that require
a network should report `not_evaluated` with next steps.

## 11. Target Resolution

`--blob-storage <canister-or-role>` accepts either a principal text value or a
role name.

Resolution order:

1. If the value parses as a principal, treat it as a canister ID.
2. Otherwise treat it as a role name in the selected deployment target.
3. If a role maps to zero canisters, fail with `blob_storage_target_missing`.
4. If a role maps to multiple canisters, fail with
   `blob_storage_target_ambiguous`.
5. If the resolved canister does not expose blob-storage billing readiness,
   fail with `blob_storage_target_not_blob_storage`.

`--auth-renewal <issuer-principal>` requires a principal. It does not resolve
roles in 0.78.

Targeted options are valid only on `canic medic deployment <deployment>`.
If the operator explicitly asks for a targeted deployment diagnostic, inability
to evaluate that target is usually `fail`, not `warn`.

## 13. Local And Mainnet Severity

Project-level checks should avoid live network assumptions. Deployment-level
checks may be stricter. Mainnet-intent checks fail closed. Local-development
checks can warn when recovery is obvious.

Severity examples:

| Check | Project/local | Deployment/mainnet |
| --- | --- | --- |
| ICP CLI missing | `fail` | `fail` |
| `icp.yaml` missing | `fail` when local workflow requires it | `fail` when deployment command depends on it |
| local replica unreachable | `not_evaluated` or `warn` | `fail` if root readiness was explicitly selected |
| recorded config path missing | `fail` for deployment checks | `fail` |
| network mismatch | `fail` | `fail` |
| root canister ID missing | `fail` for deployment checks | `fail` |
| blob-storage target unresolved | `not_evaluated` by default | `fail` when `--blob-storage` was passed |
| auth-renewal issuer invalid | `not_evaluated` by default | `fail` when `--auth-renewal` was passed |

## 14. Deferred Checks

Keep these out of 0.78 unless the implementation turns out very small and
stays within the consolidation boundary:

1. Stable memory ID collision checks.
2. Generated-file staleness.
3. Full feature flag consistency.
4. Unsupported deployment-shape classification.
5. Mainnet authority and controller preflight.

Mainnet authority and controller preflight is high-stakes and should receive a
separate design before it becomes a hard gate.

## 15. Output

Text output should include an aggregate header:

```text
canic medic project
status: warn
network: not selected

environment [pass] icp_cli_ok
  detail: found compatible ICP CLI
  next: none

project_config [warn] local_network_implicit
  detail: local-only assumptions are used but network selection is implicit
  next: select an explicit local network before deployment checks

deployment_state [not_evaluated] deployment_not_selected
  detail: no deployment target was selected
  next: run canic medic deployment <deployment>
```

JSON output:

```json
{
  "schema_version": 1,
  "command": "canic medic",
  "scope": "project",
  "network": null,
  "deployment": null,
  "status": "warn",
  "checks": [
    {
      "category": "project_config",
      "code": "local_network_implicit",
      "status": "warn",
      "subject": "network",
      "detail": "local-only assumptions are used but network selection is implicit",
      "next": "select an explicit local network before deployment checks",
      "source": "icp_config"
    },
    {
      "category": "deployment_state",
      "code": "deployment_not_selected",
      "status": "not_evaluated",
      "subject": "deployment",
      "detail": "no deployment target was selected",
      "next": "run canic medic deployment <deployment>",
      "source": "command"
    }
  ]
}
```

`--json` is in scope for 0.78 and emits this shape.

## 16. Exit Codes And Streams

Exit codes are the same for text and JSON output:

```text
0 = aggregate status is pass or warn
1 = aggregate status is fail
2 = command usage error, invalid arguments, rejected hard-cut form, or invalid JSON/output option
3 = internal tool error that prevents report construction
```

`warn` does not exit nonzero in 0.78. CI that wants stricter behavior should
parse JSON. A future `--fail-on-warn` flag can receive its own design when
needed.

Stream behavior:

1. JSON output writes only the JSON report to stdout.
2. Text output writes the rendered report to stdout.
3. Human diagnostics about command misuse and internal errors go to stderr.
4. Rejected command forms do not emit a `MedicReport`; they return a usage
   error.

## 17. Implementation Slices

1. Define `MedicCheck`, `MedicStatus`, `MedicCategory`, and `MedicReport`.
2. Move existing deployment-scoped `canic info medic` logic behind the new
   model under `canic medic deployment <deployment>` without changing
   diagnostic outcomes.
3. Add the project-level path for `canic medic` and `canic medic project`.
4. Add the top-level CLI parser for `canic medic`.
5. Remove the `canic info medic` route, docs, and tests.
6. Rename info-specific internal types to top-level medic names.
7. Add project-level non-live checks.
8. Add check codes and categories to existing deployment diagnostics.
9. Add the text renderer.
10. Add the JSON renderer with `schema_version = 1`.
11. Update docs and examples to use only `canic medic`.

Do not route bare `canic medic` through the old installed-deployment medic
behavior. The old deployment-required behavior belongs only under
`canic medic deployment <deployment>`.

## 18. Validation

Focused validation should cover:

```text
cargo test --locked -p canic-cli medic
cargo test --locked -p canic-cli status
cargo test --locked -p canic-host deployment_truth --lib
```

CLI behavior tests should cover rejected command forms:

```text
canic info medic <deployment>
canic medic <deployment>
canic medic --blob-storage <target>
canic medic --auth-renewal <issuer>
canic medic project --blob-storage <target>
canic medic project --auth-renewal <issuer>
```

Renderer and model tests should cover:

1. Text renderer includes category, status, code, detail, and next.
2. JSON renderer emits `schema_version = 1`.
3. Aggregate status is `fail` when any check fails.
4. Aggregate status is `warn` when no check fails and at least one check warns.
5. `not_evaluated` does not force failure.
6. `not_evaluated` is aggregate status only when no checks were evaluated.
7. Check order is deterministic.
8. JSON always includes `schema_version`, `command`, `scope`, `network`,
   `deployment`, `status`, and `checks`.
9. JSON stdout is parseable and contains no human diagnostics.
10. Exit code is 0 for `pass` and `warn`, 1 for `fail`, 2 for usage errors,
   and 3 for report-construction errors.

Broaden validation when project checks start reading build artifacts,
deployment-truth plans, or live replica state.

## 19. Acceptance Criteria

Close 0.78 only when:

1. `canic medic` exists as a top-level command.
2. `canic info medic` is removed, not shimmed.
3. `canic medic` and `canic medic project` run project checks without requiring
   installed deployment state.
4. `canic medic deployment <deployment>` runs deployment checks explicitly.
5. Existing blob-storage billing and auth-renewal diagnostics are preserved
   under the new surface.
6. All checks use stable category, code, status, subject, detail, next, and
   source fields.
7. Text output is readable.
8. JSON output ships with `schema_version = 1`.
9. No medic path mutates canisters, files, deployment truth, controllers, or
   installed state.
10. Rejected shorthand and legacy forms are tested.
11. Exit codes and stdout/stderr behavior match this design.
12. Check ordering is deterministic.
13. JSON fields are always present, with nullable `network` and `deployment`
    when not applicable.
14. Docs and examples use only the top-level command surface.
