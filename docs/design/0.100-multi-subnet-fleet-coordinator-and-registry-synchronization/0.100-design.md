# Canic 0.100 Design: Multi-Subnet Fleet Coordinator and Registry Synchronization

Date: 2026-07-23

## Status

- Status: proposed.
- Scope: multi-subnet coordination and membership topology only.
- Implementation begins with a repository pass to map the canonical
  authorities in this document onto the current root, subnet-state, cascade
  and durable-workflow code.
- This is a hard-cut design. It must not introduce a second competing subnet
  registry, compatibility facade or parallel propagation path.
- Dependency: the [0.99 App/Fleet identity and terminology hard cut](../0.99-app-fleet-identity-and-terminology-hard-cut/0.99-design.md)
  must be complete before any 0.100 implementation begins.
- Compatibility posture: pre-1.0 current schema only. There is no schema tag,
  old-name mapping layer or multi-schema decoder.

## Terminology Alignment With 0.99

This design uses the 0.99 terms directly:

- `FleetCoordinator` is the separately placed Canister that owns Fleet-wide
  coordination and membership state.
- `AuthoritySubnet` is the special Fleet Subnet where the Coordinator is
  hosted. It is outside ordinary member lifecycle and workload placement.
- `FleetRegistry` is the complete Fleet-Coordinator-owned member Subnet/root
  lifecycle authority.
- `FleetRegistryMirror` is the complete, read-only copy on each member root.
- `FleetDirectory` is the consumer-shaped projection cascaded from a root to
  ordinary Canisters.
- `FleetBinding { fleet: FleetKey, app: AppId }` is the immutable runtime
  identity established by 0.99.
- `SubnetSlotId` is a logical App configuration position; `SubnetId` is a
  resolved platform identity.

`Authority Subnet` is the canonical replacement for the former `Prime Subnet`
term. The distinguished-subnet concept remains: every Fleet has one Subnet
selected to host its Fleet Coordinator. Fiduciary is the recommended fresh
live-Fleet placement policy, not the domain name of that concept.

A Registry is the complete authoritative record set within its scope. A
Directory is a derived, read-only, consumer-shaped projection carrying
Registry provenance. `Index` is not public topology vocabulary.

The 0.99 stable Canister identity, fresh-install activation receipt and
current one-member state are prerequisites. This design adds multi-subnet
Fleet behavior; it does not weaken or replace those contracts.

## Summary

Canic currently manages topology inside an individual subnet: that subnet’s
root owns the authoritative local state and distributes it to canisters on the
subnet.

What is missing is the layer above that:

Which managed subnets exist, which root belongs to each subnet, and which
Canister coordinates the Fleet-wide topology?

Install one `FleetCoordinator` on the Authority Subnet and give it one
Fleet-wide `FleetRegistry`.

For a fresh public-IC Fleet, the Authority Subnet defaults to the Fiduciary
subnet and may be selected by named profile or explicit `SubnetId`. An
existing 0.99 Fleet instead preserves its sole member's observed physical
Subnet as the Authority Subnet. Every member root keeps a durable Registry
mirror. Only the Fleet Coordinator and roots communicate across Subnets for
topology synchronization; ordinary Canisters receive member topology through
the existing same-Subnet state cascade.

~~~text
Authority Subnet
  Fleet Coordinator
  owns authoritative FleetRegistry
        |
        | infrequent inter-subnet synchronization
        v
each managed subnet root
  stores a complete FleetRegistryMirror
  owns a local SubnetRegistry
        |
        | existing same-subnet cascade
        v
local canisters
  store consumer-shaped Fleet member topology
~~~

0.100 does not adopt, create, move or publish application service Canisters.
It establishes the one current Fleet Registry schema with an empty
`authority_services` field so 0.101 can publish the complete set without a
schema change. Authority-service operations belong to 0.101.

## Problem

Per-subnet topology is insufficient once one Canic fleet spans several subnets.

Without one canonical Fleet Registry:

- roots cannot reliably discover other managed subnets;
- Canisters may need a hard-coded Fleet Coordinator principal;
- different roots can disagree about Fleet membership;
- adding or removing a subnet has no authoritative lifecycle;
- later placement policies such as `one_per_active_subnet` cannot be evaluated;
- application code may perform unnecessary inter-subnet discovery calls;
- there is no consistent answer to “which Canister owns Fleet membership?”

Later authority-service adoption and replica designs depend on these member
facts already existing. Therefore multi-subnet Fleet Registry management must
come first.

## Existing Authority Preserved

0.100 introduces a global authority without replacing local subnet authority.

| State | Authority | Meaning |
| --- | --- | --- |
| Local Subnet registry | Root of that Subnet | Canisters, roles and topology within one Subnet |
| Fleet Registry | Separately placed Fleet Coordinator | Managed member Subnets, their roots and lifecycle |

The exact existing type may no longer be named `SubnetState`. Implementation
must locate and reuse the current canonical local topology and cascade
authorities.

The two state families must remain semantically distinct:

- local topology changes are owned by the local root;
- Fleet membership changes are owned only by the Fleet Coordinator;
- a member root may mirror the Registry but may not mutate it; and
- an ordinary Canister may consume the Fleet Directory projection but may not
  synchronize topology directly with the Fleet Coordinator.

Physical co-location does not merge logical ownership. A member root and the
Fleet Coordinator may occupy the same physical Subnet while owning disjoint
state and Canister sets. “One root per member Subnet” refers to the member
root; it does not count the Coordinator as a second member root. 0.100 neither
transfers existing local registry rows to the Coordinator nor creates
application-service rows in a Coordinator-owned scope.

### Fleet Registry Access Contract

0.100 creates the first `canic_fleet_registry` boundary. It derives the
genesis row from the immutable 0.99 `FleetBinding`, the qualified
`OneMemberFleetSeed` and the surviving root-owned local `SubnetRegistry`.
0.99 exposes no Fleet Registry, Registry writer or Registry endpoint to
supersede:

- on the Fleet Coordinator, controller-gated `canic_fleet_registry` is the
  complete administrative/read projection of the authoritative Registry;
- member roots expose their complete copy only as a controller-gated
  `canic_fleet_registry_mirror` diagnostic/backup projection, never as
  canonical authority;
- authenticated root synchronization uses the bounded
  `canic_fleet_registry_head` and `canic_fleet_registry_snapshot` endpoints;
- ordinary Canisters and public topology consumers receive only
  `FleetDirectory`; and
- Coordinator backup captures the binding, manifest, complete immutable
  snapshot and protected removed-member qualification evidence, but a backup
  file or restored mirror does not acquire mutation authority.

Host access, controller visibility and backup possession are read/recovery
capabilities; none make the host, a mirror or a backup the canonical writer.
Same-epoch restore of Coordinator mutation authority is constrained separately
below.

## Authority Subnet Selection and Installation

0.100 introduces and resolves the Authority Subnet selector:

~~~rust
pub enum AuthoritySubnetSelector {
    AuthorityDefault,
    Profile(SubnetProfileId),
    Explicit(SubnetId),
}

pub struct FleetCoordinatorBinding {
    pub fleet: FleetBinding,
    pub authority_subnet: SubnetId,
    pub coordinator: Principal,
}
~~~

The selector is host deployment input for a fresh Fleet. `canic.toml` does not
choose the Authority Subnet, and runtime code never interprets a profile name.
For fresh installation, `canic deploy plan` exposes mutually exclusive
`--authority-subnet-profile <profile>` and
`--authority-subnet <principal>` options. Omitting both produces
`AuthorityDefault`.

The placement policy is:

| Fleet path | Authority Subnet rule |
| --- | --- |
| Fresh public-IC live Fleet | Resolve `AuthorityDefault` to the current network `fiduciary` profile |
| Fresh Fleet on another live network | Resolve its configured authority default or fail planning |
| Fresh PocketIC/local/test Fleet | Use the Authority Subnet supplied by the harness |
| Fresh Fleet with developer override | Resolve the exact named profile or explicit `SubnetId` |
| Existing 0.99 Fleet | Use the sole existing member's observed physical `SubnetId`; reject profile and explicit overrides |
| Later relocation | Outside 0.100; requires an epoch-advancing recovery design |

The Fiduciary subnet is suitable as the live default because it is an
application subnet open to user Canister creation and uses a larger committee
than standard application subnets. The stronger fault threshold is valuable
for small, authoritative Fleet metadata. Its higher cycle multiplier is an
explicit tradeoff for a fresh Fleet's Coordinator, not for ordinary workloads.
The Fiduciary Subnet does not give a Canister special system privileges, free
execution or a second writer. Coordinator authority still comes from the
protected `FleetCoordinatorBinding` and single-writer protocol.

Named profiles are open network metadata, not a protocol enum. The public IC
currently documents Fiduciary and European choices; a Swiss or other future
profile becomes selectable only when the chosen network's trusted metadata
actually advertises an eligible Subnet. The design does not reserve or
hard-code speculative names.

The fresh-Fleet installation workflow:

1. Resolve `AuthorityDefault`, `Profile` or `Explicit` against trusted metadata
   for the exact `CanonicalNetworkId`.
2. Verify as a best-effort preflight that the resolved Subnet exists, accepts
   arbitrary user Canister creation and currently appears usable.
3. Persist the selector, exact resolved `SubnetId`, metadata evidence and
   resolution time in the deployment plan.
4. Create the Fleet Coordinator explicitly on that Subnet. The management
   Canister result is the authoritative capacity and eligibility outcome;
   planning metadata is not a capacity proof.
5. Install it with the immutable `FleetBinding`, authority Subnet and initial
   authority epoch.
6. Persist `FleetCoordinatorBinding` in the host receipt, Coordinator and
   every member root's protected bootstrap state.

An unavailable, ineligible or ambiguous default fails planning. A creation
rejection fails installation with the exact typed platform cause. Canic never
silently selects a random application Subnet. The developer can rerun fresh
planning with a named profile or explicit Subnet. System subnets are
ineligible because they do not accept arbitrary user Canister creation.

Changing deployment configuration after installation cannot move or replace
the authority. Authority relocation must fence old history by advancing the
authority epoch through a later explicit recovery design.

## Existing-Fleet Seed

0.100 consumes one exact seed from the qualified 0.99 one-member state:

~~~rust
pub struct OneMemberFleetSeed {
    pub fleet: FleetBinding,
    pub member_subnet: SubnetId,
    pub member_root: Principal,
    pub local_registry_content_hash: [u8; 32],
}
~~~

The 0.99 root and completed fresh-install receipt must agree on these exact
values.
`member_root` is the sole current root, `member_subnet` is its trusted observed
physical Subnet, and `local_registry_content_hash` binds the local Registry
that remains owned by that root throughout 0.100. A missing, ambiguous or
mismatching seed blocks extraction. 0.100 does not reconstruct identity from
an environment name, a role name or an unqualified Registry projection.

The pre-upgrade seed is obtained only from 0.99-compatible evidence: the exact
Active `canic_fleet_activation_status` binding, the current `canic_env`
root/Subnet facts, the unchanged `canic_subnet_registry` projection and the
completed `FleetInstallActivationJournal` plus Fleet catalog receipt. The host
computes the canonical local-Registry hash before mutation. The 0.100
source-root upgrade receives that seed, recomputes the Registry hash from
local stable state and rejects any mismatch. No new endpoint is assumed to
exist on an older Wasm.

The seed's `member_subnet` becomes the Authority Subnet for this extraction.
An existing-Fleet override is rejected. This preserves the physical placement
needed for 0.101 to adopt mutable services without changing their principals
or application data.

## 0.99-to-0.100 Coordinator Handoff

The sole 0.99 root begins as the authority for its immutable one-member seed
and local Registry facts. It has no Fleet Registry and cannot mutate Fleet
membership. 0.100 consumes and permanently fences that seed-attestation
authority while creating Fleet membership authority in the separately
installed Coordinator through one durable handoff:

~~~text
LegacyActive
  → CoordinatorPrepared
  → SourceQuiesced
  → SourceFenced
  → CoordinatorCommitted
  → MirrorActivated
~~~

These are durable workflow states, not inferred host observations:

- `CoordinatorPrepared` binds the exact seed, Coordinator principal, Authority
  Subnet, authority epoch and Fleet-scoped trust bootstrap. It exposes neither
  a current Registry manifest nor mutation authority.
- `SourceQuiesced` records that the old root has stopped seed attestation and
  member-topology publication while retaining authority to write its final
  fence. It does not reinterpret or move retained local Registry services.
- `SourceFenced` is persisted by the old root before acknowledgement. It binds
  a unique handoff ID, seed hash, Coordinator principal and epoch. Once
  persisted, the old seed-attestation authority can never be re-enabled at
  that epoch.
- `CoordinatorCommitted` requires authenticated source-fence evidence from the
  exact old root. Only then may the Coordinator publish the genesis Registry
  manifest and accept membership mutation.
- `MirrorActivated` records that the former source root has activated the
  exact committed Registry as its mirror and completed the member-topology
  cascade.

The irreversible fence evidence is:

~~~rust
pub struct SourceFenceEvidence {
    pub handoff_id: [u8; 32],
    pub fleet: FleetBinding,
    pub source_root: Principal,
    pub coordinator: FleetCoordinatorBinding,
    pub authority_epoch: u64,
    pub seed_hash: [u8; 32],
}
~~~

After persisting `SourceFenced`, the source root calls the prepared
Coordinator's handoff-commit update endpoint itself. The Coordinator requires
`caller == source_root` and an exact match with its prepared handoff. A
host-relayed record without the authenticated source call is not fence
evidence. An unknown call outcome is retried by the fenced source with the
same handoff ID.

For an existing Fleet:

1. Validate and journal the exact `OneMemberFleetSeed`.
2. Create the Coordinator through the current root's same-Subnet creation
   workflow and record its principal before installation proceeds. Do not
   invoke the fresh-Fleet selector.
3. Install the Coordinator with the protected `FleetBinding`, genesis epoch,
   prepared one-member Registry and Fleet-scoped trust bootstrap. Verify
   `CoordinatorPrepared`.
4. Upgrade the existing root to the 0.100 handoff/mirror-capable Wasm with the
   exact seed and protected `FleetRegistryBootstrap`. The install atomically
   enters `SourceQuiesced`, preserves the local `SubnetRegistry`, exposes no
   Fleet Registry writer or active mirror and stops seed attestation/member
   topology publication.
5. Persist `SourceFenced` in the old root and deliver its authenticated fence
   evidence to the prepared Coordinator.
6. Verify the exact fence evidence and commit the genesis Registry containing
   the existing member as `Active`; publish its first current manifest.
7. Activate `FleetRegistryMirror` in the already upgraded root from the exact
   committed manifest; it never gains canonical Registry mutation.
8. Pull the exact committed snapshot, cascade member topology locally, persist
   `MirrorActivated`, verify the Coordinator, mirror and host receipt agree,
   verify that the bound local Registry content hash and rows are unchanged,
   and then remove quiescence.

The current root's temporary authority to attest the exact one-member seed
disappears after `SourceFenced`; installing the 0.100 source-handoff state is
not itself fence evidence. While `SourceQuiesced` it exposes only transition
diagnostics and the authenticated fence operation. After `MirrorActivated` it
exposes only `canic_fleet_registry_mirror`. The new Fleet Coordinator creates
and exclusively exposes the first current `canic_fleet_registry` authority.

The host journal coordinates retries but is not an authority. Before
`SourceFenced`, recovery may abandon the prepared Coordinator and retain the
seed-attestation authority through an explicit rollback inside the already
installed 0.100 root Wasm. That rollback proves the exact seed/local Registry,
re-enters `LegacyActive` behavior and abandons the prepared Coordinator; it
does not reinstall 0.99, decode an old schema or expose a compatibility path.
After `SourceFenced`, rollback is forbidden: recovery must resume preparation
or commitment of the same Coordinator principal. An availability gap is
acceptable; overlapping seed-attestation and Registry-mutation authorities are
not. Unknown create, install, fence-delivery or commit outcomes are reconciled
by exact handoff-ID status reads and idempotent retries, never by creating a
replacement Coordinator.

Fresh 0.100 deployment installs the Fleet Coordinator first, commits an empty
genesis Registry and then joins member roots through the normal `Joining`
lifecycle.

## Goals

0.100 must provide:

1. One authoritative Fleet Registry of managed member Subnets.
2. A unique Canic root for each managed Subnet.
3. A separately placed Fleet Coordinator with a protected Authority Subnet and
   Canister binding.
4. Explicit registration, activation, draining and removal lifecycles.
5. Durable synchronization from the Fleet Coordinator to every member root.
6. Atomic activation of complete Registry snapshots.
7. Same-subnet propagation through the existing root cascade.
8. Local APIs for inspecting authority and member-Subnet topology.
9. Fleet-bound provenance and freshness information.
10. Failure-safe operation while the Fleet Coordinator is temporarily
    unavailable.
11. A bounded inter-subnet call pattern.
12. Durable qualified final inventory and root-local fencing for ordinary
    removal, with explicit unqualified evidence for emergency removal.

## Core Invariants

The implementation must preserve these invariants:

1. For each Fleet and authority epoch, exactly one committed authority
   descriptor value identifies the Authority Subnet and Fleet Coordinator.
2. Every protected or published copy for that Fleet and epoch agrees exactly
   with the committed descriptor value and immutable `FleetBinding`.
3. Every snapshot, manifest, version, acknowledgement, mirror record,
   projection and hash preimage binds the same `FleetBinding`.
4. The Authority Subnet is part of Fleet topology but not an ordinary
   `FleetRegistryEntry`; its binding cannot be joined, drained or removed
   through the member lifecycle.
5. Every member Subnet ID appears at most once in Registry entries. Explicit
   physical co-location with the Authority Subnet does not merge the
   Coordinator and member-root registry scopes.
6. Every root principal in `Joining`, `Active` or `Draining` belongs to at
   most one member Subnet.
7. Only the Fleet Coordinator can commit a new Registry revision.
8. Registry revisions increase monotonically within an authority epoch.
9. A revision identifies exactly one canonical content hash within one
   `FleetBinding` and authority epoch.
10. Evidence from another network, Fleet or App never compares equal or
    activates, even when epoch, revision and entries match.
11. A mirror never activates a partial or unverified snapshot.
12. A mirror never replaces a newer Registry with an older one.
13. A mirror never promotes itself because the Fleet Coordinator is
    unreachable.
14. Ordinary Canisters never need an inter-Subnet call to read Fleet Directory
    state.
15. `Joining`, `Draining` and `Removed` Subnets are not treated as normal
    active placement targets.
16. Registry removal does not falsely claim that the removed root Canister
    has been stopped or deleted.
17. Ordinary workload placement never targets the Authority Subnet.
18. At steady state, every registry-managed Canister has exactly one committed
    registry owner. A future epoch-bound movement handoff may stage source and
    destination records, but two registries are never committed owners for the
    same ownership epoch.
19. Every `Removed` member has exactly one Coordinator-retained
    `Qualified` or `Unqualified` removal record; absence is invalid.
20. A qualified removal binds the exact authority, member, Fleet Registry
    version, final local Registry hash, complete bounded inventory and
    authenticated root-local fence.
21. An unqualified removal never implies an empty member and cannot be used as
    duplicate-absence evidence.
22. A persisted removed-member fence never permits Canic-managed creation,
    attachment, placement or local-registry mutation to resume at the same
    authority epoch.
23. `authority_services` is present in every current Registry snapshot and is
    empty throughout 0.100.

## Data Model

~~~rust
pub struct FleetRegistry {
    pub authority: FleetRegistryAuthority,
    pub revision: u64,
    pub members: Vec<FleetRegistryEntry>,
    pub authority_services: Vec<FleetAuthorityServiceBinding>,
}

pub struct FleetRegistryAuthority {
    pub binding: FleetCoordinatorBinding,
    pub epoch: u64,
}

pub struct FleetRegistryEntry {
    pub subnet: SubnetId,
    pub root: Principal,
    pub status: ManagedSubnetStatus,
}

pub enum ManagedSubnetStatus {
    Joining,
    Active,
    Draining,
    Removed,
}

pub struct FleetAuthorityServiceBinding {
    pub role: CanisterRole,
    pub canister_id: Principal,
}
~~~

Member entries use ascending canonical Subnet identity. Hashing, equality and
publication never depend on insertion order.

`authority_services` is part of the sole current snapshot schema and canonical
encoding from the first 0.100 Registry. It is empty throughout 0.100.
`FleetAuthorityServiceBinding` intentionally stores only the service role and
principal: the exact `FleetBinding`, Coordinator, Authority Subnet and
authority epoch are stored once in `FleetRegistry.authority`. 0.101 may
populate the field only with a complete committed declared set; it does not
replace this type or add another Fleet binding to the public Registry.

The content hash is publication metadata rather than a self-referential field
inside the hashed Registry snapshot:

~~~rust
pub struct FleetRegistryManifest {
    pub authority: FleetRegistryAuthority,
    pub revision: u64,
    pub byte_length: u64,
    pub content_hash: [u8; 32],
}
~~~

A complete Registry identity is therefore:

~~~rust
pub struct FleetRegistryVersion {
    pub authority: FleetRegistryAuthority,
    pub revision: u64,
    pub content_hash: [u8; 32],
}
~~~

The exact encoding and hash construction are frozen below. Otherwise identical
Registries for different Authority Subnets or Coordinators, canonical networks
or Fleets of the same App produce different evidence. Two environment profiles
that resolve to the same canonical public IC network do not create different
Registry evidence merely because their labels differ.

## Authority Epoch

The authority epoch fences Registry history belonging to different Fleet
Coordinators within one immutable `FleetBinding`.

0.100 records the epoch but does not implement automatic authority relocation
or replacement.

Every root is provisioned with a protected bootstrap descriptor:

~~~rust
pub struct FleetRegistryBootstrap {
    pub authority: FleetRegistryAuthority,
    pub local_subnet: SubnetId,
}
~~~

A mirror accepts snapshots only from the configured Fleet Coordinator
and only for the configured authority descriptor and exact protected
`FleetBinding`.

Within 0.100:

- a lower epoch is rejected as stale;
- a different Authority Subnet or Coordinator at the same epoch is rejected;
- an unexpected higher epoch is rejected pending an explicit recovery operation;
- no root derives authority from availability or majority; and
- no mirror can promote itself.

A later design may define a protected authority-relocation workflow that
advances the epoch. Merely including the epoch in 0.100 avoids publishing an
unfenceable format.

## Same-Epoch Restore and Rollback

A matching `FleetBinding`, Coordinator principal and authority epoch do not
prove that a restored Coordinator snapshot is current. Reactivating revision 8
after any member observed revision 12 could fork or reuse committed history.

0.100 therefore permits a Coordinator backup to restore only read-only
diagnostic and recovery state. It does not reactivate Registry mutation
authority at the same epoch. Same-epoch reactivation would require independent
proof that the restored `FleetRegistryVersion` equals the latest committed
version; 0.100 defines no such proof authority.

When latest-version equality cannot be proven, restore remains fenced and
fails closed pending a later epoch-advancing recovery procedure. A member
mirror, host journal, controller assertion or offline backup is not sufficient
on its own. Ordinary canister upgrade with intact monotonic stable state is not
treated as backup restoration.

## Bootstrap Truth

Configuration provides only the minimum facts needed to join the fleet:

- the immutable Fleet binding;
- the expected local subnet identity;
- the Authority Subnet;
- the Fleet Coordinator principal; and
- the expected authority epoch.

It must not contain a second complete list of managed subnets.

These facts are deployment/install input persisted in protected host and root
state. They are not `canic.toml` App configuration. The runtime
`FleetRegistry` becomes the only authority for Fleet membership.

Canic must not claim it has cryptographically discovered which subnet hosts a
root if the Internet Computer does not expose that fact to the canister. The
`(subnet, root)` relationship must come from the existing trusted
deployment/configuration authority and be verified as far as the platform
permits.

## Fleet Coordinator Registry Publication

The Fleet Coordinator stores the authoritative Fleet Registry as an immutable,
revisioned snapshot. It does not own any member's local Canister registry and
0.100 transfers no local Canister rows to it.

A Registry mutation must:

1. Load the current canonical Registry.
2. Validate the requested transition.
3. Construct the complete next snapshot.
4. Serialize it deterministically.
5. Enforce entry-count and byte-size limits.
6. Calculate its content hash.
7. Durably store the complete snapshot.
8. Atomically make its manifest current.

`canic_fleet_registry_head` must never expose a manifest whose snapshot is
absent or incomplete.

~~~rust
async fn canic_fleet_registry_head(
) -> Result<FleetRegistryManifest, FleetRegistryError>;

async fn canic_fleet_registry_snapshot(
    version: FleetRegistryVersion,
) -> Result<Vec<u8>, FleetRegistryError>;
~~~

Both synchronization endpoints are authenticated inter-Canister **update**
calls. They are logically read-only, but 0.100 does not use query or composite
query transport for cross-Subnet synchronization.

## Canonical Snapshot Protocol

0.100 uses complete Registry snapshots, not deltas or an event log.

The protocol constants are:

~~~rust
const GENESIS_AUTHORITY_EPOCH: u64 = 1;
const GENESIS_REGISTRY_REVISION: u64 = 1;
const MAX_MANAGED_SUBNETS: usize = 256;
const MAX_FLEET_REGISTRY_SNAPSHOT_BYTES: usize = 131_072;
const MAX_REMOVED_MEMBER_INVENTORY_ENTRIES: usize = 4_096;
const MAX_REMOVED_MEMBER_INVENTORY_BYTES: usize = 1_048_576;

const MIN_POLL_INTERVAL_SECONDS: u64 = 60;
const DEFAULT_POLL_INTERVAL_SECONDS: u64 = 300;
const MAX_POLL_INTERVAL_SECONDS: u64 = 3_600;
const INITIAL_RETRY_BACKOFF_SECONDS: u64 = 15;
const MAX_RETRY_BACKOFF_SECONDS: u64 = 900;
const MAX_SUPERSEDED_RETRIES_PER_PASS: u8 = 3;
~~~

The genesis Registry is revision 1 in authority epoch 1. It is empty for a
fresh Fleet and contains the sole existing member as `Active` after a
successful 0.99 extraction. `authority_services` is empty on both paths.
Revision increment uses checked arithmetic. An attempt to advance `u64::MAX`
fails before mutation with
`RevisionExhausted`; it never wraps, resets or silently advances the epoch.

Snapshot bytes use deterministic CBOR under RFC 8949 deterministic-encoding
rules. Records are encoded as fixed arrays in the field order shown in this
document; integers use their shortest form; principals and digests are byte
strings; vectors are definite-length; and maps, text keys, floats and
indefinite-length items are forbidden. `ManagedSubnetStatus` encodes as the
unsigned values `0 = Joining`, `1 = Active`, `2 = Draining` and `3 = Removed`.
Nested identity types use their frozen 0.99 raw byte representations rather
than their display text. Members are sorted by canonical raw `SubnetId` bytes
before encoding. Authority-service bindings are sorted by canonical raw role
bytes before encoding; 0.100 rejects a non-empty vector because service
activation belongs to 0.101. `MAX_MANAGED_SUBNETS` counts every retained
member row, including `Removed` tombstones.

The hash is calculated over the exact bytes returned by
`canic_fleet_registry_snapshot`, not over a separately reconstructed logical
preimage:

~~~text
SHA-256(
  "canic:fleet-registry\0"
  || u64_be(snapshot_byte_length)
  || exact_canonical_snapshot_bytes
)
~~~

The snapshot bytes themselves contain the complete authority descriptor,
revision, member vector and authority-service vector. A mutation exceeding an
applicable count or byte bound fails before authoritative state changes.

Chunk transfer is excluded from 0.100. If a Fleet Registry ever becomes too
large for one bounded response, that should be justified by observed use
before adding a generic transfer protocol.

### Snapshot Retention and Head Races

The Coordinator retains only:

- the current committed snapshot and manifest; and
- one non-public staging slot needed to replace them durably.

After a new manifest commits, the previous snapshot is no longer retrievable
and its storage is reclaimed. `canic_fleet_registry_snapshot(version)` returns
typed `VersionSuperseded` unless `version` exactly equals the current
`FleetRegistryVersion`. A root receiving that result restarts from the head
request. One synchronization pass follows at most
`MAX_SUPERSEDED_RETRIES_PER_PASS` such races before recording a retryable
failure and entering bounded backoff.

A join acknowledgement must normally reference the current committed version.
If the Registry advanced first, the Coordinator returns `RegistryAdvanced`
and the joining root synchronizes again. The idempotent exception is a
repeated acknowledgement after that exact root is already recorded as
`Active`; it returns the existing committed Active result even when the
original acknowledgement response was lost.

## Mirror Synchronization

Every member root owns one durable synchronization state:

~~~rust
pub struct FleetRegistryMirrorState {
    pub authority: FleetRegistryAuthority,
    pub active: Option<FleetRegistryVersion>,
    pub last_polled_at: Option<Timestamp>,
    pub last_confirmed_current_at: Option<Timestamp>,
    pub synchronized_at: Option<Timestamp>,
    pub consecutive_failures: u32,
    pub next_poll_at: Option<Timestamp>,
    pub last_failure: Option<FleetRegistrySyncFailure>,
}
~~~

A timer-driven synchronization pass performs:

1. Record the polling attempt and call the configured Fleet Coordinator for
   `canic_fleet_registry_head`.
2. Validate the full Fleet binding and authority descriptor.
3. If the exact version is already active, durably set
   `last_confirmed_current_at`, clear the last failure, reset retry backoff and
   return without cascading.
4. Reject an older epoch or revision.
5. Reject the same epoch and revision with a different hash.
6. Fetch the complete snapshot when behind.
7. On `VersionSuperseded`, restart from the bounded head step.
8. Verify its exact length and hash before decoding.
9. Decode and validate every Registry invariant.
10. Durably stage the snapshot.
11. Atomically activate the mirror and derived Fleet Directory, setting both
    `synchronized_at` and `last_confirmed_current_at`.
12. Enqueue or resume the existing same-subnet member-topology cascade.
13. Clear failure state, reset backoff and record synchronization success.

Revision gaps are permitted because every publication is a complete snapshot.
A mirror can move directly from revision 4 to revision 12.

Polling configuration is clamped to the constants above. Retry backoff doubles
from the initial value to the maximum and resets after either a confirmed
current head or snapshot activation. There is one polling actor per member
Subnet, not one per Canister. Durable timers and work resume through the
existing Canic machinery.

## Interruption and Uncertain Outcomes

All inter-subnet synchronization calls in 0.100 are observational and safe to repeat.

Local activation and cascade must also be idempotent.

If interruption occurs:

- before snapshot activation, the previous Registry remains active;
- after activation but before the success receipt, restart detects the
  already-active `FleetRegistryVersion`;
- during local cascade, the active root Registry remains valid and cascade
  resumes through its existing durable progress; and
- after receiving a response whose local receipt was not persisted, the root
  repeats the head or snapshot request.

No pending state may be reset merely because the previous process tree became quiescent.

## Subnet Registration

Registration is Fleet-Coordinator-authorized and phased.

~~~text
Absent → Joining → Active → Draining → Removed
             └──────────────────────→ Removed
~~~

### Create Joining Entry

An authorized Fleet administrator registers:

- the subnet identity; and
- its expected root principal.

The Fleet Coordinator validates uniqueness and publishes a new Registry
revision with the entry in `Joining`.

A `Joining` subnet:

- is visible for bootstrap and diagnostics;
- is not returned by normal `active_subnets`;
- is not eligible for placement; and
- cannot be treated as a service destination.

### Join Acknowledgement

The joining root:

- starts with its protected bootstrap descriptor;
- pulls the authoritative Fleet Registry;
- verifies that it contains its expected `(subnet, root)` entry;
- activates that snapshot locally; and
- calls the Fleet Coordinator with an idempotent join acknowledgement.

~~~rust
async fn acknowledge_subnet_join(
    applied: FleetRegistryVersion,
) -> Result<JoinStatus, FleetRegistryError>;
~~~

The Fleet Coordinator verifies:

- the caller is the root principal recorded in the `Joining` entry;
- the acknowledged Fleet binding exactly matches protected authority state;
- the reported authority epoch matches;
- the acknowledged revision and hash were published by the Fleet Coordinator;
  and
- the acknowledged snapshot contained that joining entry.

A valid acknowledgement transitions the Subnet to `Active` and publishes the
next revision.

Repeating an acknowledgement after activation returns the existing active result.

### Cancel Failed Join

An authorized Fleet administrator may transition `Joining → Removed` when a
join cannot complete. Cancellation publishes a new Registry revision and
retains the normal removed tombstone. It does not delete, stop or repurpose the
root Canister. A live joining root follows the normal final-inventory and
root-local-fence qualification below. If the root never became callable or an
emergency cancellation cannot obtain that evidence, the Coordinator records
the tombstone as removal-unqualified rather than inventing an empty member.
Reusing that Subnet identity remains outside 0.100.

### Synchronization Caller Access

The Coordinator authenticates `canic_fleet_registry_head` and
`canic_fleet_registry_snapshot` by caller principal and current Registry
status:

- the root recorded for a `Joining`, `Active` or `Draining` member may
  synchronize;
- a `Removed` root, an unknown caller and a principal recorded for another
  member are rejected;
- a controller does not gain synchronization membership merely by being a
  controller; and
- controller-gated administrative Registry reads remain a separate recovery
  capability.

## Draining and Removal

Removal is also phased:

~~~text
Active → Draining → Removed
~~~

### Draining

Draining means:

- do not choose the subnet for new placement;
- retain it for explicit lookup and evacuation;
- retain its root identity; and
- allow existing workloads to be managed according to their domain policy.

The Registry does not decide when application evacuation has finished. An
authorized administrative action starts removal qualification. The normal
transition to `Removed` commits only after the Coordinator retains the
authenticated final inventory and fence evidence described below; emergency
removal records an explicit unqualified tombstone.

### Removed

Removed means:

- exclude the subnet from normal active resolution;
- reject it as a new placement target;
- preserve a tombstone preventing stale state from looking like a new
  registration; and
- reject Fleet-management authority from that root where Registry membership
  is required.

A removed entry is not immediately deleted from the Registry.

0.100 does not permit reusing a removed Subnet identity as a fresh
registration. Replacement or tombstone compaction requires a later explicit
design.

Registry removal does not stop the root Canister, change controllers or erase
local state. Those are deployment operations outside the Registry. A removed
but still-running root may retain stale local knowledge, but active Fleet
members no longer recognize it as active.

Every member Subnet, including one physically co-located with the Authority
Subnet, follows the same lifecycle and may be drained or removed. Removing a
co-located member entry does not move the Fleet Coordinator or remove the
Authority Subnet from the Fleet descriptor. Coordinator relocation is a
separate operation outside 0.100.

### Removed-Member Qualification

Removal must not erase the evidence later needed to prove that no possibly
live managed Canister conflicts with a Fleet-authoritative service.

Before an ordinary `Joining`, `Active` or `Draining` member becomes
`Removed`, the Coordinator requests one final inventory at an exact current
Fleet Registry version. The root atomically:

1. freezes Canic-managed creation, attachment, placement and local-registry
   mutation for that member and authority epoch;
2. exports its complete bounded committed local managed-Canister inventory;
3. persists the final local Registry content hash, canonical inventory bytes
   and inventory hash;
4. persists the removal fence before acknowledging it; and
5. calls the Coordinator's dedicated removal-qualification update endpoint
   itself.

The operational evidence shapes are:

~~~rust
pub struct RemovedMemberIdentity {
    pub subnet: SubnetId,
    pub root: Principal,
}

pub struct FinalManagedCanisterRecord {
    pub canister_id: Principal,
    pub role: CanisterRole,
    pub disposition: FinalManagedCanisterDisposition,
}

pub enum FinalManagedCanisterDisposition {
    Managed,
    Released {
        ownership_epoch: u64,
        receipt_hash: [u8; 32],
    },
    Decommissioned {
        receipt_hash: [u8; 32],
    },
}

pub struct FinalMemberInventory {
    pub authority: FleetRegistryAuthority,
    pub member: RemovedMemberIdentity,
    pub base_registry_version: FleetRegistryVersion,
    pub local_registry_content_hash: [u8; 32],
    pub canisters: Vec<FinalManagedCanisterRecord>,
}

pub struct RemovedMemberFence {
    pub authority: FleetRegistryAuthority,
    pub member: RemovedMemberIdentity,
    pub removal_id: [u8; 32],
    pub base_registry_version: FleetRegistryVersion,
    pub local_registry_content_hash: [u8; 32],
    pub final_inventory_hash: [u8; 32],
}

pub struct RemovedMemberFenceReceipt {
    pub fence: RemovedMemberFence,
    pub fence_hash: [u8; 32],
    pub persisted_at: Timestamp,
}

pub enum RemovedMemberQualification {
    Qualified {
        inventory: FinalMemberInventory,
        fence_receipt: RemovedMemberFenceReceipt,
    },
    Unqualified {
        authority: FleetRegistryAuthority,
        member: RemovedMemberIdentity,
        removal_id: [u8; 32],
        last_registry_version: FleetRegistryVersion,
        reason: RemovalUnqualifiedReason,
    },
}
~~~

`FinalMemberInventory.authority` is the one stored Fleet binding hierarchy;
the evidence does not add a second Fleet identity field. Its member identity,
Fleet Registry revision and local Registry hash are exact. The canonical
inventory includes every committed local managed-Canister row, including
released ownership tombstones. Any record whose disposition is not
`Decommissioned` identifies a Canister that may still exist and must not be
silently treated as absent.

The inventory is limited to
`MAX_REMOVED_MEMBER_INVENTORY_ENTRIES` records and
`MAX_REMOVED_MEMBER_INVENTORY_BYTES` deterministic-CBOR bytes. Records are
ordered by raw Canister principal bytes, then raw role bytes. The inventory
is a fixed array in `FinalMemberInventory` field order, every record is a
fixed array in `FinalManagedCanisterRecord` field order, and dispositions
encode as `0 = Managed`, `1 = Released` and `2 = Decommissioned`.
`Released` payload fields retain their shown order. The inventory hash is:

~~~text
SHA-256(
  "canic:removed-member-inventory\0"
  || u64_be(inventory_byte_length)
  || exact_canonical_inventory_bytes
)
~~~

`removal_id` is 32 cryptographically random bytes generated and durably stored
by the Coordinator before it requests qualification. The root's canonical
fence bytes are a fixed array in `RemovedMemberFence` field order. Its
`fence_hash` is:

~~~text
SHA-256(
  "canic:removed-member-fence\0"
  || u64_be(fence_byte_length)
  || exact_canonical_fence_bytes
)
~~~

The receipt is the fixed array `[fence, fence_hash, persisted_at]`. The root
persists that receipt before returning it through the authenticated
inter-Canister update call. The Coordinator records the authenticated caller,
exact receipt and inventory as one qualification; display projections or a
host assertion cannot substitute for those bytes.

For ordinary removal, the Coordinator accepts qualification only when `caller`
equals the exact recorded root, the member is still `Joining`, `Active` or
`Draining`, every authority/member/version/hash field matches and the
root-local fence receipt authenticates the same bytes. It durably stores the
complete qualified evidence before publishing the `Removed` Registry revision.
Once the root persists the fence, restart cannot re-enable Canic-managed
creation, attachment, placement or registry mutation at that authority epoch.
Read-only diagnostics and the idempotent exact qualification call may remain
available.

If emergency removal must complete while the root cannot provide this
evidence, the Coordinator may publish `Removed` only after durably storing an
`Unqualified` record. It retains the member tombstone, last Registry version,
removal ID and typed reason. It does not invent an empty inventory. A later
exact recovery may replace `Unqualified` with `Qualified`, but the root is not
required to remain live or callable forever and ordinary synchronization
access is not restored. That recovery uses only the dedicated qualification
endpoint, requires `caller` to equal the tombstoned root and must match the
exact stored authority, member, removal ID and last Registry version. It
cannot change membership status, regain Fleet-management authority or submit
an inventory that was not durably fenced at the root.

Qualification state is protected Coordinator operational state, not a public
Fleet Registry field. It is retained for the lifetime of the member tombstone
unless a later explicit compaction design proves it safe to remove.

## Local Propagation

A root must not create a new direct global channel to every Canister.

After activating a Registry revision, the root uses the existing local
topology/state cascade to distribute the relevant Fleet Directory to:

- existing managed Canisters; and
- Canisters joining or being created later.

0.100 evolves the one canonical `FleetDirectory` envelope introduced in 0.99;
it does not add a second companion projection:

~~~rust
pub struct FleetDirectory {
    pub provenance: DirectoryProvenance,
    pub entries: Vec<DirectoryEntryInput>,
    pub subnets: Vec<FleetDirectorySubnet>,
}

pub struct DirectoryProvenance {
    pub authority: FleetRegistryAuthority,
    pub source_root: Principal,
    pub registry_revision: u64,
    pub registry_content_hash: [u8; 32],
    pub synchronized_at: Timestamp,
}

pub struct FleetDirectorySubnet {
    pub subnet: SubnetId,
    pub root: Principal,
    pub status: ManagedSubnetStatus,
}
~~~

`subnets` is the complete consumer-shaped member projection derived from the
active verified `FleetRegistryMirror`. `entries` is the existing service
lookup field, but it is empty throughout 0.100 because
`authority_services` is empty. 0.100 does not reinterpret a local service as
Fleet authority or delete it from its member-local registry. 0.101 populates
`entries` only from the complete committed `authority_services` set.

The existing `fleet_directory` cascade field carries this exact shape as a
deliberate pre-1.0 wire cut. No companion membership or service Directory is
introduced.

The semantic rules are fixed:

- each member root's active verified mirror is the Directory source;
- ordinary Canisters never fetch the Registry from the Fleet Coordinator;
- projection does not give ordinary Canisters mutation authority;
- new Canisters receive the latest locally active Fleet Directory;
- interrupted distribution resumes rather than starting a second cascade; and
- `synchronized_at` records the instant the root atomically activates both the
  verified Registry mirror and its derived Fleet Directory for that exact
  revision.

An up-to-date head confirmation updates root-local operational state but does
not change `synchronized_at` or recascade the Directory. The value is not the
Coordinator publication time, last poll time or cascade-completion time.
Consequently, cascaded provenance makes no claim about current Coordinator
reachability.

## Freshness and Authority Subnet Downtime

If the Fleet Coordinator is temporarily unreachable:

- each root retains its last complete Fleet Registry;
- existing local topology continues operating;
- ordinary Canisters keep their last distributed Fleet Directory;
- local Fleet Directory lookups remain available with stale provenance;
- synchronization retries with bounded backoff;
- subnet registration, activation, draining and removal are unavailable; and
- no mirror becomes authoritative.

Canic exposes root-local time since the last poll and last confirmed current
head. The cascaded Directory exposes only `synchronized_at`, the atomic
mirror-and-Directory activation time for its exact revision. Neither a mirror
nor the Directory claims a stale Registry is fresh, and no mirror promotes
itself.

Callers that require fresh Fleet membership may apply an explicit maximum-age
policy. Generic Directory lookup does not silently reject stale state unless
the operation specifically requires freshness.

## Fleet-Scoped Authorization Transfer

The Coordinator derives administration authority from the 0.99 Fleet-scoped
trust boundary. It does not create a new controller list, administrator list
or Coordinator-specific token audience.

`CoordinatorPrepared` installs one canonical trust bootstrap exported by the
0.99 source root while that root is still the trusted Fleet authority. The
bootstrap contains:

- the exact protected `FleetBinding`;
- the current Fleet-scoped issuer verification material and proof chain;
- the Fleet-administration subject/policy projection needed for membership
  mutation;
- the current revocation, replay and proof-floor state required by the 0.99
  verifier; and
- the handoff ID, source root and canonical content hash binding that material
  to this extraction.

The Coordinator verifies the proof chain and rejects any audience other than
the exact `FleetKey` in its protected binding. It persists the bootstrap before
reporting `CoordinatorPrepared`. A lost preparation receipt is reconciled by
an authenticated status read keyed by handoff ID and content hash; a retry
with different trust material is rejected.

After `CoordinatorCommitted`, Fleet Registry administration uses the same
Fleet-scoped delegated-token verification and policy authority established by
0.99. Controller capability permits install, upgrade and explicitly documented
recovery operations, but controller status alone does not authorize Registry
mutation. Member roots expose no canonical Registry mutation endpoint.

Join acknowledgement additionally requires the caller to equal the recorded
`Joining` root. Head and snapshot calls use the lifecycle access rules above.
Every administrative request, root call and acknowledgement binds the exact
protected `FleetRegistryAuthority`; a snapshot's internal authority claim is
never sufficient.

## Validation Rules

A Fleet Registry snapshot is valid only when:

- its full `FleetBinding` exactly matches protected bootstrap state;
- the authority epoch, Authority Subnet and Fleet Coordinator principal match
  protected bootstrap state;
- its revision is at least `GENESIS_REGISTRY_REVISION`;
- member entries are canonically ordered;
- member Subnet IDs are unique;
- root principals are unique across `Joining`, `Active` and `Draining`;
- `authority_services` is present and empty;
- every status value is recognized;
- all identities pass existing canonical validation;
- serialized size and member count are within the frozen bounds; and
- the content hash matches the received bytes.

The same `(FleetRegistryAuthority, revision)` arriving with a different hash
is not a routine retry failure. It is authority equivocation or corruption
and must be surfaced as a terminal synchronization failure requiring
investigation.

A different `FleetRegistryAuthority` is always rejected before revision
comparison. Matching revision and content from another canonical network,
Fleet, App, Authority Subnet or Fleet Coordinator is not an idempotent retry.

Coordinator removal state is valid only when every `Removed` member has one
matching qualification record. Qualified evidence must pass its independent
authority, member, version, ordering, size, inventory-hash and root-fence
receipt checks. An unqualified record must retain its exact typed reason and
must never be projected as an empty inventory.

base Registry version, final inventory count/hash when qualified and typed
unqualified reason otherwise. The complete inventory remains protected
operational evidence rather than a public Directory or Fleet Registry field.

- downloaded snapshot count and bytes;

Failures must remain typed through internal ownership. Human-readable strings
are projections, not the stored error authority.

## Stable-State and Schema Rules

The implementation must identify:

- the canonical stable owner of the authoritative Fleet Registry;
- the old root's durable handoff/fence owner and the Coordinator's preparation
  and commitment owner;
- the canonical stable owner of each mirror;
- the Coordinator's protected owner of qualified and unqualified
  removed-member evidence and the root's irreversible removal fence;
- the current stable owner of local cascade progress; and
- whether the Fleet Coordinator and mirrors use the same stored snapshot type.

There must not be:

- two stored canonical Subnet lists;
- a configuration list continually dual-written with the runtime Registry;
- both mutable row state and an independently authoritative published snapshot; or
- legacy and new Registry readers running in parallel.

The Coordinator stores one current snapshot/manifest pair plus one staging
slot. The source root stores its irreversible fence independently. The host
journal records progress and receipts but cannot synthesize source-fence
evidence, removed-member qualification, Coordinator commitment or a newer
Registry version. Qualified inventories and unqualified tombstones are retained
for the lifetime of their member tombstones.

If an existing global-subnet concept already exists, 0.100 must migrate or
replace it as a hard cut rather than wrapping it indefinitely.

Stable snapshots, manifests, mirror state, cascade progress and provenance have
one current shape with the full `FleetRegistryAuthority`. Any migration
required for deployed Canisters must be explicit, bounded and tested through
upgrade. After activation there is no schema tag, old decoder or fallback
reader; pre-1.0 internal structures are replaced directly.

- wrong-root, wrong-Fleet, wrong-member, stale-version, hash-mismatched or
  non-canonical final evidence is rejected;
- restart after the root-local fence cannot resume Canic-managed creation,
  attachment, placement or Registry mutation;
- retrying the exact qualification returns the same retained result;
- emergency removal stores `Unqualified` with its typed reason and never
  fabricates an empty inventory;
- every `Removed` member has exactly one qualification record;
- a removed root may disappear permanently without losing
  Coordinator-retained qualified evidence; and
- a later exact recovery may qualify an unqualified tombstone without
  restoring ordinary synchronization access.

### Synchronization

- A mirror with no Registry bootstraps from the Fleet Coordinator.
- An up-to-date mirror persists `last_confirmed_current_at`, clears failure
  state, resets backoff and does not cascade.
- A mirror skips directly across multiple revisions.
- requesting a snapshot superseded after the head read returns
  `VersionSuperseded` and restarts from head;
- more than three superseded races in one pass enter bounded retry;
- only the current committed snapshot is retrievable;
- committing a new snapshot reclaims the previous snapshot after durable
  manifest replacement;
- An older revision cannot replace a newer revision.
- The same revision and hash is an idempotent no-op.
- The same revision with a different hash fails terminally.
- A corrupted or truncated snapshot never activates.
- A snapshot with a different Fleet binding leaves the previous snapshot
  active.
- An authority descriptor mismatch is rejected.
- Unexpected lower and higher authority epochs fail closed.
- `last_polled_at`, `last_confirmed_current_at` and `synchronized_at` advance
  only for their declared events.

### Handoff, Interruption and Recovery

- interruption in `LegacyActive` leaves the 0.99 root authoritative for its
  exact one-member seed and local `SubnetRegistry`; no Fleet Registry exists;
- `CoordinatorPrepared` exposes neither a current manifest nor mutation
  authority;
- interruption in `SourceQuiesced` resumes or explicitly rolls back before
  fencing without downgrading the root Wasm or changing the seed;
- the source persists `SourceFenced` before acknowledging it;
- Coordinator commitment rejects host-only or wrong-source fence evidence;
- after `SourceFenced`, recovery resumes with the same Coordinator principal
  and never restores the old seed-attestation authority;
- an unknown Coordinator commit outcome reconciles by handoff ID without
  creating another Coordinator;
- at no durable boundary can the Fleet Coordinator accept Registry mutation
  while the member root can still commit or attest a different handoff seed;
- loss of availability between fence and commitment is recoverable and does
  not weaken the single-writer rule;
- Interruption before durable snapshot staging retains the old Registry.
- Interruption after staging but before activation retains the old Registry.
- Interruption after activation but before receipt reconciles as complete.
- Upgrade during polling resumes safely.
- Upgrade during local cascade resumes from durable progress.
- Repeated timeouts do not discard the last valid Registry.
- Fleet Coordinator downtime never promotes a mirror.

### Authorization and Restore

- the Coordinator rejects trust material for another Fleet, audience, source
  root, handoff ID or content hash;
- loss of the trust-install receipt reconciles to the same prepared state;
- controllers cannot call Fleet Registry mutation without Fleet-scoped
  authorization;
- a stale same-epoch Coordinator backup may restore read-only recovery state
  but cannot reactivate mutation authority;
- mirror state, a host journal or controller assertion alone cannot prove that
  a restored revision is latest.

### Local Distribution

- Existing Canisters receive empty service `entries` and the complete
  `subnets` projection without changing member-local Registry ownership.
- A newly created Canister receives the root’s current revision.
- No ordinary Canister performs an inter-Subnet Registry call.
- Partial cascade is observable and recoverable.
- Directory provenance carries `synchronized_at`, not a publication, poll or
  cascade-completion time.
- A successful unchanged head poll does not recascade the Directory.

## Non-Goals

0.100 does not implement:

- adoption, provisioning, replacement, ownership transfer, placement or
  Directory publication of Fleet-authoritative services;
- activation of `[services.fleet].roles` as runtime service ownership;
- IcyDB snapshot, change-stream or application-data replication;
- service groups;
- automatic member-replica or `one_per_active_subnet` provisioning;
- health-, latency- or freshness-based service routing;
- automatic Fleet Coordinator election, replacement or relocation;
- root-to-root consensus;
- multi-writer Registry mutation;
- root-to-root mesh synchronization;
- Registry deltas or change-data capture;
- chunked artifact transfer;
- generic distributed transactions;
- proof that two application databases contain identical data;
- subnet health or “best available subnet” selection; or
- physical removal of a subnet, root or canisters.

## Hard-Cut Completion Criteria

0.100 is complete when:

1. The separately placed Fleet Coordinator is the only canonical writer
   of Fleet Subnet membership.
2. Every authoritative and mirrored record and every evidence hash binds the
   exact canonical network, Fleet and App through `FleetBinding`.
3. Fresh public-IC deployment defaults to the dynamically resolved Fiduciary
   profile, fresh tests use their harness Subnet, developer overrides are
   validated and no failure falls back to a random Subnet.
4. The Authority Subnet is a distinguished Fleet topology component bound
   outside ordinary member rows, lifecycle and workload placement.
5. Existing 0.99 Fleets use the sole observed member Subnet without an
   override, the exact one-member seed remains bound through extraction and
   no local Registry row moves or changes ownership.
6. The durable handoff permits zero writers temporarily but never two
   committed writers, and recovery after the source fence uses the same
   Coordinator principal.
7. Every member root durably mirrors a verified complete current Fleet
   Registry.
8. Registry member changes propagate through the `subnets` field of the exact
   current Fleet Directory while service `entries` remain empty.
9. Inter-Subnet topology synchronization is limited to the Fleet Coordinator
   and member roots, bounded and infrequent.
10. Registration, activation, failed-join cancellation, draining and removal
    are explicit and tested.
11. Every removed member has durable qualified final inventory and root-fence
    evidence or an explicit unqualified tombstone that cannot prove absence.
12. Removed-member evidence remains usable after the root becomes permanently
    unavailable.
13. Every maintained distinguished-subnet surface uses Authority Subnet
    terminology; `Prime Subnet` appears only when identifying the removed
    name.
14. No unexplained competing Subnet list or privileged-member authority
    remains.
15. The snapshot protocol has one exact encoding, hash, retention rule,
    bounded race recovery and checked revision sequence.
16. Every snapshot uses the one current schema with an empty
    `authority_services` field and one Fleet binding beneath `authority`.
17. Fleet-scoped trust transfers without creating a controller-derived or
    Coordinator-local administrator authority.
18. Coordinator restore cannot roll back same-epoch mutation authority.
19. Upgrade and interruption tests prove every durable boundary.

## Expected Follow-On

With 0.100 complete, 0.101 can safely adopt and provision Fleet-authoritative
services:

~~~text
0.100
  “Which managed Subnets exist, which roots own them,
   and which Fleet Coordinator owns membership?”

0.101 authority services
  “How are existing mutable services adopted without changing
   their principals or data, how are fresh services provisioned,
   how does registry ownership move, and what is published
   through the Fleet Directory?”

Later replication
  “How does one authoritative service distribute
   application-owned data to its mirrors?”
~~~

0.101 compiles the service-owned `[services.fleet].roles` declaration,
qualifies singleton eligibility, adopts in place or provisions missing
services, transfers registry ownership and publishes the canonical Directory.
Relocation of an existing mutable authority to another physical Subnet remains
separate from adoption in place.
