# Canic 0.71 Design: Blob Storage Operator Readiness

## Status

- Status: draft design note
- Release line: 0.71
- Decision style: operator-readiness and cleanup line
- Primary implementation target: Canic CLI support and operational diagnostics
  for the 0.69/0.70 blob-storage backend.
- Current state: 0.69 delivered the non-billing blob-storage backend protocol
  foundation. 0.70 delivered the Cashier-backed billing MVP, including gateway
  sync, project-cycle funding, read-only billing status, guarded endpoint
  emission, upgrade coverage, and retained audit coverage. The remaining gap is
  operator usability: a developer can test the behavior through targeted
  canister/probe calls, but there is no first-class Canic CLI workflow for
  status, sync, funding, or troubleshooting.

## 1. Summary

0.71 should make the 0.70 blob-storage billing feature usable by operators and
downstream backend developers.

The release should add a small Canic CLI surface for blob-storage billing
readiness, explicit gateway-principal sync, and explicit project-cycle funding.
It should also extend `canic info medic` with blob-storage-specific hints and
clean up the feature boundary where 0.70 intentionally left pressure in
`api::blob_storage`.

0.71 is not a new protocol line. It should not add frontend billing UI,
TypeScript clients, npm packages, generic blob metadata APIs, automatic
funding, or new gateway/Cashier protocol behavior. Its job is to turn the
backend MVP into something a dev can run, inspect, and debug locally.

## 2. Release-Line Boundary

| Release | Owns |
| --- | --- |
| 0.69 | Non-billing blob-storage protocol inventory, gateway-compatible endpoint emission, lifecycle state, gateway-principal storage/checks, and PocketIC scrubber coverage. |
| 0.70 | Cashier integration, gateway-principal sync from Cashier, project-cycle funding, read-only billing status, endpoint guards, billing persistence, and billing failure coverage. |
| 0.71 | Operator CLI, medic diagnostics, runbooks, packaged/downstream validation, and small boundary cleanup around the 0.70 billing MVP. |

0.71 must preserve the 0.70 contract:

1. `get_blob_storage_status` is read-only with respect to gateway sync and
   funding.
2. Gateway sync remains an explicit operator action.
3. Project-cycle funding remains an explicit operator action.
4. Billing configuration is not exposed as a broad public admin endpoint.
5. Browser/product frontends do not become responsible for billing
   provisioning.
6. Product backend canisters, hub canisters, deployment automation, or human
   operators may orchestrate the explicit status/sync/fund sequence when their
   endpoint guards authorize it.

## 3. Goals

1. Add a first-class CLI group for blob-storage operator actions.
2. Add readable plain-text and JSON status output for
   `get_blob_storage_status`.
3. Add explicit CLI wrappers for gateway-principal sync and project-cycle
   funding.
4. Extend `canic info medic` with blob-storage readiness checks and actionable
   next steps.
5. Provide a local runbook that proves the full 0.69/0.70 flow without a
   product frontend.
6. Add CLI parsing, rendering, and transport tests for the new operator
   surface.
7. Add packaged/downstream validation that the installed CLI can inspect a
   blob-storage billing deployment.
8. Reduce or explicitly contain `api::blob_storage` pressure where doing so
   removes real complexity.
9. Document the existing 0.70 programmatic consumer contract so product
   backends can call the same endpoints directly without depending on the CLI.

## 4. Non-Goals

0.71 does not:

1. Add browser upload UI or billing UI.
2. Add TypeScript clients or npm packages.
3. Add automatic funding from status or medic.
4. Add automatic gateway sync from status or medic.
5. Add a public billing-config admin endpoint.
6. Call production Cashier in CI.
7. Change the gateway-owned `_immutableObjectStorage*` method names or Candid
   signatures.
8. Change 0.69/0.70 stable storage schema unless a bug requires it.
9. Move product-specific migration, upload orchestration, or frontend adapter
   logic into Canic.
10. Add a blob-storage workflow module unless orchestration has clearly outgrown
    the API/ops split.
11. Add upload/batch cost estimation, pricelist helpers, or product-specific
    pricing UX.
12. Add split Cashier balances or account-existence fields to
    `get_blob_storage_status` without a separate protocol decision.

## 5. CLI Surface

Add a top-level `blob-storage` command group for operator actions:

```text
canic blob-storage status <deployment> <canister-or-role> [--json]
canic blob-storage sync-gateways <deployment> <canister-or-role> [--json] [--dry-run]
canic blob-storage fund <deployment> <canister-or-role> --cycles <cycles> [--json] [--dry-run]
```

`<canister-or-role>` is required for every MVP command. Omitting it is a CLI
usage error. 0.71 does not add implicit fleet-wide blob-storage scans.

Expected examples:

```text
canic blob-storage status local backend
canic blob-storage status local backend --json
canic blob-storage sync-gateways local backend --dry-run
canic blob-storage fund local backend --cycles 1000000000000
```

Rationale:

- `canic info ...` remains read-only installed-deployment information.
- `sync-gateways` and `fund` are explicit mutating operator actions, so they
  should not live under `canic info`.
- `status` belongs in the same command group because operators naturally run
  status, then sync, then fund, then status again.

`canic info medic <deployment>` should not duplicate the full command group.
It should surface diagnosis and next-step hints that point to the explicit
`canic blob-storage ...` actions.

The CLI should keep operator-facing command names. The gateway protocol method
names may appear in dry-run/debug output, but normal help text should describe
the action in Canic terms.

### 5.1 Programmatic Consumer Contract

The CLI is not the only supported consumer. Product backend canisters,
deployment automation, and test fixtures may call the 0.70 canister endpoints
directly.

Programmatic consumers should target the canister that:

1. was built with `blob-storage-billing`,
2. emits the billing endpoint macro,
3. stores the blob-storage billing configuration, and
4. owns the Cashier payment account for uploads.

In a multi-canister product, this is the blob-storage billing host canister, not
necessarily the hub. A hub may orchestrate billing only by calling that target
canister through guards intentionally configured to accept the hub/parent
caller.

The supported endpoint sequence is:

1. Read status:

   ```text
   get_blob_storage_status(record { sync_gateway_principals = false })
   ```

2. If gateway principals are empty or stale, call:

   ```text
   _immutableObjectStorageUpdateGatewayPrincipals()
   ```

3. If funding is required and policy allows transfer, call:

   ```text
   _immutableObjectStorageFundFromProjectCycles(requested_cycles : nat)
   ```

4. Re-read status and gate upload on `ready`.

Status is an update method in the 0.70 endpoint bundle even when the request is
read-only with respect to sync/funding. Canister-to-canister callers must use
the method mode from the target interface and must not assume it is a query
because the operation is semantically read-only.

Endpoint guards are product configuration:

- `status_guard` controls who can read billing status.
- `sync_gateway_principals_guard` controls who can refresh gateway principals
  from Cashier.
- `fund_from_cycles_guard` controls who can spend target-canister project
  cycles into the Cashier account.

0.71 does not prescribe a product's auth policy. For fleet products, document
the target role and guard policy in product code/docs. Browser frontends should
not call sync or fund directly during normal upload/login flows; product
backends may do so when explicitly authorized.

Programmatic canister calls use the raw Candid types. In particular,
`_immutableObjectStorageFundFromProjectCycles` takes a raw `u128`/Candid `nat`
amount; the CLI's strict decimal-string parser is only a CLI input rule.

Product consumers remain responsible for:

- upload/batch cost estimation
- comparing estimated upload cost with usable billing state
- deciding when to sync and fund
- computing explicit fund amounts
- hub/instance wiring and endpoint guard configuration
- adapting Canic DTOs or CLI JSON into product frontend shapes
- gateway upload through the 0.69 storage-client protocol

Products that use a separate subscription/payment account outside the current
Canic `ProjectAsPaymentAccount` model own that account mapping themselves in
MVP. Canic 0.71 status should report the account named by the 0.70 payment
model; it should not grow split product account semantics without a separate
protocol decision.

### 5.2 Target Resolution

The `<canister-or-role>` argument must resolve predictably:

1. If the value exactly matches an installed deployment role, use that role.
2. If the value parses as a canister id/principal and no exact role match
   exists, use it directly.
3. If a value both matches a role and parses as a principal-like string, prefer
   the exact role match. A future `--canister-id` flag may force principal
   interpretation if that ever becomes ambiguous in practice.
4. If more than one installed role exposes blob-storage endpoints, the required
   target still selects exactly one role/canister. There is no implicit
   multi-target mode in MVP.
5. If the target does not expose the required method in local Candid metadata,
   fail with a clear diagnostic.
6. If local Candid metadata is missing, fail with a clear message for MVP.
   Do not silently build fragile textual calls from guessed signatures.

MVP behavior is a hard failure when required Candid/endpoint metadata is
missing or ambiguous. Do not add a built-in 0.70 signature fallback in 0.71
unless that becomes an explicit follow-up design decision with tests.

Target JSON should preserve how the target was resolved. For a role-based
target:

```json
{
  "input": "backend",
  "role": "backend",
  "canister_id": "aaaaa-aa",
  "candid_source": "installed_deployment"
}
```

For a direct canister id target:

```json
{
  "input": "aaaaa-aa",
  "role": null,
  "canister_id": "aaaaa-aa",
  "candid_source": null
}
```

## 6. CLI Command Semantics

### 6.1 `canic blob-storage status`

`status` resolves `<deployment>` and `<canister-or-role>` through the installed
deployment registry, then calls:

```text
get_blob_storage_status(record { sync_gateway_principals = false })
```

Status must not pass `sync_gateway_principals = true` in 0.71. The CLI should
not imply that status can repair anything. The returned response is parsed into
a host/CLI view and rendered as:

- billing configuration presence
- Cashier canister id when configured
- Cashier balance when available
- min/target upload balance
- project-cycle reserve
- funding readiness and requested cycles
- gateway-principal count
- last gateway sync timestamp
- warnings/readiness blockers
- next suggested command

Plain text should be compact and terminal-readable. JSON should expose stable
machine-readable field names for automation.

Although status is semantically read-only with respect to sync/funding, the
CLI must derive the actual method mode from local Candid or installed endpoint
metadata. It must not assume `query`; if metadata is missing or ambiguous,
status fails before transport like the mutating commands.

Status process exit behavior:

- Exit `0` when the status call succeeds and the response is parsed, even when
  readiness is blocked.
- Readiness state and blocker codes are data, not process failure, unless a
  future explicit `--check-ready` mode is added.
- Exit nonzero only for command usage, target resolution, transport, or
  response parsing failure.

### 6.2 `canic blob-storage sync-gateways`

`sync-gateways` resolves the canister and calls:

```text
_immutableObjectStorageUpdateGatewayPrincipals()
```

The command reports:

- target canister role/principal
- synced gateway-principal count when available
- stable error details on malformed Cashier response, empty gateway list,
  unauthorized caller, or trapped Cashier call

If the endpoint does not return a gateway count, the CLI must not invent one.
It should report that gateway sync completed and point the operator to `status`
to verify the synced principal count.

The command may include post-action diagnostic status when implemented. A
successful mutating call must be reported separately from any failed
post-action diagnostic query. Post-action diagnostic failure must not be
rendered as a failed sync operation unless the sync call itself failed.

Common errors should be normalized into stable codes:

- `not_configured`
- `cashier_unavailable`
- `cashier_response_malformed`
- `gateway_principals_empty`
- `unauthorized`
- `method_unavailable`

### 6.3 `canic blob-storage fund`

`fund` resolves the canister and calls:

```text
_immutableObjectStorageFundFromProjectCycles(<cycles>)
```

The `--cycles` argument is explicit. The CLI may suggest an amount based on
status output, but it must not invent an implicit default.

Help text should describe the command as funding blob-storage Cashier balance
from project cycles.

Accepted cycle input for MVP is only an unsigned base-10 integer string:

```text
--cycles 1000000000000
```

Reject before transport:

- missing values
- zero
- negative values
- separators such as `1_000_000`
- suffixes such as `1T`
- decimals such as `1.5`
- scientific notation such as `1e12`
- values that overflow the command's chosen integer type

No interactive confirmation prompt is required in MVP. The explicit
`--cycles` value is the operator confirmation. If the CLI later adds prompts
for other commands, blob-storage funding can revisit a `--yes` convention then.

The command reports:

- requested cycles
- attached cycles
- accepted cycles when exposed by the endpoint
- skipped cycles when exposed by the endpoint
- reserve cycles
- Cashier total after funding
- project-cycle balance metadata when available
- skipped reason when reserve policy blocks funding

The 0.70 DTO uses `attached_cycles`; the CLI should preserve that field in
JSON. Plain text may describe it as the cycles attached to the funding call,
and may additionally show `accepted_cycles` or `skipped_cycles` only when the
endpoint exposes enough data to do so accurately.

The CLI must not invent fields that the endpoint did not return. Plain text may
omit unavailable values; JSON should use `null` for known-but-unavailable
optional fields.

The CLI does not own reserve computation. It validates CLI input shape, then
passes the requested cycles to the endpoint. The target canister's 0.70 funding
policy remains authoritative for project-cycle reserve checks, attached cycles,
and skipped reasons.

The command may include post-action diagnostic status when implemented. A
successful funding call must be reported separately from any failed post-action
diagnostic query. Post-action diagnostic failure must not be rendered as a
failed funding operation unless the funding call itself failed.

### 6.4 Dry-Run Contract

`--dry-run` is supported for mutating commands. Dry-run may resolve local
deployment metadata and local Candid artifacts, but it must not:

- call the target canister
- call Cashier
- query status before the action
- query status after the action
- sync gateway principals
- attach or transfer cycles

Dry-run output should include:

- resolved deployment
- resolved role and canister id
- method name
- Candid argument
- whether the command would be a query or update
- the exact configured canister-call command line that would be executed

The CLI must derive method mode from local Candid or installed endpoint
metadata. It must not call a method as query merely because the operation is
described as read-only. If metadata is unavailable or ambiguous, MVP behavior
is to fail before transport.

Do not add `status --dry-run` unless a concrete operator use case emerges.

### 6.5 Process Exit Codes

Use stable process semantics for automation:

| Code | Meaning |
| --- | --- |
| 0 | Command completed successfully. For `status`, this includes fetched status with blocked readiness. |
| 1 | CLI usage, input, local config, target resolution, or local Candid error. |
| 2 | Target canister call failed or was rejected. |
| 3 | Response parsing or Candid decode failed. |
| 4 | Reserved for a future explicit check mode where readiness blockers should fail the process. |

Exit `1` covers local discovery/signature failures such as missing Candid,
unreadable Candid, absent required method metadata, and unresolved targets.
Exit `3` covers failures after successful transport such as malformed response
envelopes, response Candid decode failures, and unsupported DTO shapes.

Plain-text output may be friendly, but JSON output must carry enough structured
status for callers to avoid parsing prose.

## 7. Host and CLI Ownership

Use the existing host/CLI split:

- `canic-cli` owns argument parsing, command grouping, text rendering, and JSON
  output.
- `canic-host` owns reusable canister-call command construction, installed
  deployment resolution, response parsing, and any shared render-ready host
  model.
- `canic-core` remains canister runtime code and must not learn about CLI
  commands.

Use the existing Canic host canister-call abstraction for all status, sync, and
fund transport. 0.71 must not introduce a second canister-call transport stack.

Proposed modules:

```text
crates/canic-cli/src/blob_storage/
  mod.rs
  command.rs
  options.rs
  render.rs
  transport.rs
  model.rs
  tests.rs

crates/canic-host/src/blob_storage/
  mod.rs
  response.rs
  tests.rs
```

If the host response parsing is tiny, it may initially live in the CLI module.
Move it to `canic-host` when `medic` and `blob-storage status` both need it.

## 8. JSON and Response Contract

Prefer JSON output from the configured canister-call command, such as
`icp canister call ... --output json`, with local Candid when possible.

The parser should accept the current canister-call JSON envelope and extract the
Canic DTO response. It should not rely on fragile display strings where a JSON
field exists.

For 0.71, define render-ready CLI models rather than exposing raw DTOs as the
CLI contract:

```rust
struct BlobStorageCliStatus {
    schema_version: u16,
    kind: &'static str,
    deployment: String,
    target: BlobStorageCliTarget,
    configured: bool,
    payment_model: String,
    payment_account: Option<String>,
    cashier: BlobStorageCliCashierStatus,
    policy: BlobStorageCliPolicyStatus,
    project_cycles_available_cycles: String,
    gateways: BlobStorageCliGatewayStatus,
    funding: BlobStorageCliFundingStatus,
    readiness: BlobStorageCliReadiness,
    next: Vec<BlobStorageCliNextAction>,
}
```

Exact Rust names may change during implementation, but the output should keep
the same concepts.

### 8.1 Stable JSON Shape

Treat `--json` output as an automation contract from day one. The exact Rust
types may change, but the JSON field names and value kinds should remain
stable within the 0.71 line.

Status JSON should use this shape:

```json
{
  "schema_version": 1,
  "kind": "blob_storage_status",
  "deployment": "local",
  "target": {
    "input": "backend",
    "role": "backend",
    "canister_id": "aaaaa-aa",
    "candid_source": "installed_deployment"
  },
  "configured": true,
  "payment_model": "project_as_payment_account",
  "payment_account": "aaaaa-aa",
  "cashier": {
    "canister_id": "bbbbb-bb",
    "balance_cycles": "100000000000",
    "balance_source": "cashier_total",
    "balance_available": true
  },
  "policy": {
    "min_upload_balance_cycles": "500000000000",
    "target_upload_balance_cycles": "1000000000000",
    "project_cycles_reserve_cycles": "2000000000000"
  },
  "project_cycles_available_cycles": "5000000000000",
  "gateways": {
    "principal_count": 2,
    "last_sync_at_ns": "1710000000000000000",
    "last_sync_at": "2024-03-09T16:00:00Z"
  },
  "funding": {
    "status": "funding_required",
    "requested_cycles": "900000000000"
  },
  "readiness": {
    "state": "blocked",
    "ready_for_upload": false,
    "blockers": ["gateway_principals_empty", "cashier_balance_below_min"],
    "warnings": []
  },
  "next": [
    {
      "action": "sync_gateways",
      "command": "canic blob-storage sync-gateways local backend",
      "reason": "gateway_principals_empty"
    },
    {
      "action": "fund",
      "command": "canic blob-storage fund local backend --cycles 900000000000",
      "reason": "cashier_balance_below_min"
    }
  ]
}
```

Rules:

- `schema_version` starts at `1`.
- `kind` identifies the output shape. MVP values are
  `blob_storage_status`, `blob_storage_sync_gateways_result`,
  `blob_storage_fund_result`, and `blob_storage_error`.
- Cycle amounts are decimal strings, not JSON numbers. They may exceed safe
  integer limits for JavaScript consumers.
- Nanosecond timestamps are decimal strings for the same reason. The rendered
  ISO timestamp is optional but useful.
- If `last_sync_at_ns` is `null`, `last_sync_at` must also be `null`. When
  present, `last_sync_at` is UTC with a trailing `Z`.
- Unknown optional values should be `null`, not omitted, when the field is part
  of the schema.
- Plain text may format numbers for humans; JSON primary fields must stay raw
  and machine-readable.
- Policy values in examples use the Canic V1 default policy for docs and
  runbooks: `min_upload_balance = 500_000_000_000`, `target_upload_balance =
  1_000_000_000_000`, and `project_cycles_reserve = 2_000_000_000_000`.
  Runtime status must report the actual deployed billing configuration.
- `payment_account` mirrors the 0.70 status DTO. For the current
  `ProjectAsPaymentAccount` model, it is the target blob-storage billing
  canister principal.
- `cashier.balance_cycles` is the Cashier account `total` field converted to
  an unsigned cycle value. If Cashier returns `AccountNotFound`, current 0.70
  status represents that as balance `0` and does not expose a separate
  `account_exists` bit.
- Split Cashier balances such as prepaid, promo, ledger, and debt target are
  not part of 0.71 CLI status. Product consumers that require them must call
  Cashier directly or wait for a separate protocol change.
- `funding.requested_cycles`, when present, comes from the 0.70 funding status:
  `target_upload_balance - cashier_balance`, after project-cycle reserve
  policy is considered. `next` fund actions should use this value so operators
  can copy/paste the suggested command.
- `readiness.ready_for_upload` mirrors the 0.70 `ready` field and is the single
  machine-readable upload gate. It is `true` only when there are no readiness
  blockers.

Sync JSON should use this shape:

```json
{
  "schema_version": 1,
  "kind": "blob_storage_sync_gateways_result",
  "deployment": "local",
  "target": {
    "input": "backend",
    "role": "backend",
    "canister_id": "aaaaa-aa",
    "candid_source": "installed_deployment"
  },
  "action": {
    "name": "sync_gateways",
    "method": "_immutableObjectStorageUpdateGatewayPrincipals",
    "mode": "update",
    "dry_run": false,
    "success": true,
    "gateway_principal_count": 2
  },
  "post_status": null,
  "warnings": []
}
```

Funding JSON should use this shape:

```json
{
  "schema_version": 1,
  "kind": "blob_storage_fund_result",
  "deployment": "local",
  "target": {
    "input": "backend",
    "role": "backend",
    "canister_id": "aaaaa-aa",
    "candid_source": "installed_deployment"
  },
  "action": {
    "name": "fund",
    "method": "_immutableObjectStorageFundFromProjectCycles",
    "mode": "update",
    "dry_run": false,
    "success": true,
    "requested_cycles": "1000000000000",
    "attached_cycles": "1000000000000",
    "accepted_cycles": "1000000000000",
    "skipped_cycles": "0",
    "reserve_cycles": "1000000000000",
    "cashier_total_after_cycles": "2000000000000",
    "skipped_reason": null
  },
  "post_status": null,
  "warnings": []
}
```

The primary action result must be distinguishable from diagnostic status.
`post_status` may be `null`, especially in MVP. If a post-action status query
fails after a successful sync/fund call, the action remains successful and the
diagnostic failure is reported through warnings or a diagnostic field.

### 8.2 Readiness Codes and Command Error Codes

Readiness must be computed before rendering. Plain text and medic should not
invent their own policy decisions.

Top-level readiness states:

- `ready`
- `blocked`
- `warning`
- `unknown`

State derivation:

- `blocked`: the 0.70 status DTO has `ready = false` or any blockers.
- `warning`: the 0.70 status DTO has `ready = true` and non-empty warnings.
- `ready`: the 0.70 status DTO has `ready = true` and no warnings.
- `unknown`: the CLI could not confidently map the response into the status
  model.

Canonical readiness, blocker, and warning codes for MVP:

- `not_configured`
- `gateway_principals_empty`
- `gateway_sync_never_run`
- `funding_needed`
- `cashier_unavailable`
- `cashier_balance_unavailable`
- `cashier_balance_below_min`
- `project_cycles_reserve_blocks_funding`
- `cashier_response_malformed`
- `unauthorized`
- `sync_requested_but_status_is_read_only`
- `unknown`

The 0.70 DTO remains the single source of truth for status readiness. CLI and
medic map DTO values into the stable JSON strings:

| 0.70 DTO value | CLI/readiness code | Upload gate effect |
| --- | --- | --- |
| `BlobStorageReadinessBlocker::NotConfigured` | `not_configured` | blocked |
| `BlobStorageReadinessBlocker::GatewayPrincipalsMissing` | `gateway_principals_empty` | blocked |
| `BlobStorageReadinessBlocker::CashierBalanceUnavailable` | `cashier_balance_unavailable` | blocked |
| `BlobStorageReadinessBlocker::CashierBalanceMalformed` | `cashier_response_malformed` | blocked |
| `BlobStorageReadinessBlocker::InsufficientCashierBalance` | `cashier_balance_below_min` | blocked |
| `BlobStorageReadinessBlocker::ReserveWouldBeViolated` | `project_cycles_reserve_blocks_funding` | blocked |
| `BlobStorageBillingWarning::GatewayPrincipalSetEmpty` | `gateway_principals_empty` | warning if no blocker remains |
| `BlobStorageBillingWarning::CashierBalanceUnavailable` | `cashier_balance_unavailable` | warning if no blocker remains |
| `BlobStorageBillingWarning::CashierBalanceMalformed` | `cashier_response_malformed` | warning if no blocker remains |
| `BlobStorageBillingWarning::SyncRequestedButStatusIsReadOnly` | `sync_requested_but_status_is_read_only` | warning |

`gateway_sync_never_run` is a diagnostic warning derived when
`gateway_principal_count == 0` and `last_gateway_principal_sync_at_ns == null`.
It does not replace the primary `gateway_principals_empty` blocker.

`funding_needed` is derived from `BlobStorageFundingStatus::FundingRequired`.
Its recommended fund amount is the DTO's `requested_cycles`. If funding would
violate the project-cycle reserve, expose
`project_cycles_reserve_blocks_funding` and the DTO's transferable/requested
cycle values rather than suggesting an unsafe fund command.

Canonical command/action error codes for MVP:

- `target_resolution_failed`
- `candid_unavailable`
- `method_unavailable`
- `transport_failed`
- `canister_rejected`
- `response_parse_failed`
- `candid_decode_failed`
- `invalid_cycles`
- `unknown`

Canonical next actions:

- `none`
- `status`
- `sync_gateways`
- `fund`
- `inspect_config`
- `retry`

Do not derive readiness from plain-text strings. Host/CLI parsing should produce
stable codes, and renderers should only display those codes.

Readiness codes appear under `readiness.blockers` and `readiness.warnings`.
Command/action error codes appear under an `error` object and map to the
documented exit-code behavior.

Adding, renaming, or removing a JSON `kind`, readiness code, warning code,
next-action code, or command error code requires a matching design/doc update,
changelog note, and schema/renderer test update.

### 8.3 JSON Error Output and Streams

When `--json` is supplied and the command fails before producing a normal
result, the CLI should emit a structured error object to stderr and exit with
the documented process code. Stdout should remain empty on failure unless a
partial successful primary action result is intentionally emitted.

Example:

```json
{
  "schema_version": 1,
  "kind": "blob_storage_error",
  "deployment": "local",
  "target": {
    "input": "backend",
    "role": null,
    "canister_id": null,
    "candid_source": null
  },
  "error": {
    "code": "target_resolution_failed",
    "message": "No installed role named backend was found.",
    "exit_code": 1
  }
}
```

Stream rules:

- Successful plain-text and JSON command results go to stdout.
- Human-readable diagnostics go to stderr.
- Structured `--json` errors go to stderr.
- Dry-run output is a successful command result and goes to stdout.

## 9. Medic Integration

Extend `canic info medic <deployment>` with blob-storage checks when the
deployment has a canister exposing blob-storage status or when the local Candid
artifact includes blob-storage billing endpoints.

For 0.71 MVP, default medic should not perform per-canister blob-storage status
calls. It may passively report that blob-storage-capable canisters exist and
suggest the targeted `--blob-storage <canister-or-role>` option. Full
blob-storage readiness evaluation runs only when the targeted option is
supplied.

In a multi-canister fleet, the targeted medic role should be the same
blob-storage billing host role described in the programmatic consumer contract:
the canister that emits the billing endpoints, stores billing configuration,
and owns the Cashier payment account. If a product uses a hub to orchestrate
instance billing, medic should still target the instance/backend canister that
actually hosts the billing endpoints unless the hub also exposes an intentional
diagnostic wrapper.

Medic should report:

1. blob-storage billing endpoint availability
2. billing configured vs not configured
3. gateway principals present vs empty
4. Cashier balance available vs unavailable/malformed
5. funding needed vs sufficient
6. last gateway sync timestamp present vs missing
7. next operator command

Medic must not call sync or fund. Targeted blob-storage medic may call status
because 0.70 status is read-only with respect to sync/funding.

If future medic scans many blob-storage-capable canisters, it should either:

- summarize counts and show the first few blockers; or
- require a later `--role` / `--canister` option before doing per-canister
  status.

MVP recommendation: add a targeted option:

```text
canic info medic <deployment> --blob-storage <canister-or-role>
```

This keeps the default medic output readable.

Medic must consume the same parsed host model and readiness codes as
`canic blob-storage status`. It should not maintain a separate interpretation
of blob-storage readiness. Its output should include the safest next command,
for example:

```text
Blob storage billing: blocked
Reason: gateway principals are empty
Next: canic blob-storage sync-gateways local backend
```

## 10. Local Runbook

Add an operator runbook for the local dev flow:

1. Install or register a local deployment.
2. Confirm blob-storage endpoints exist:

   ```text
   canic info endpoints <deployment> <canister-or-role>
   ```

3. Check billing status:

   ```text
   canic blob-storage status <deployment> <canister-or-role>
   ```

4. Sync gateway principals:

   ```text
   canic blob-storage sync-gateways <deployment> <canister-or-role>
   ```

5. Fund project cycles explicitly:

   ```text
   canic blob-storage fund <deployment> <canister-or-role> --cycles <amount>
   ```

6. Re-check status and verify readiness blockers are cleared.
7. Upgrade the canister and re-check persisted billing config, gateway sync
   timestamp, and funding status.

The runbook should state clearly that production Cashier is not called by CI.
Local tests use mock Cashier/PocketIC fixtures.

## 11. Boundary Cleanup

0.70 intentionally left the billing status/sync/funding orchestration in
`api::blob_storage`. 0.71 should clean this only where it improves ownership:

Ownership target for 0.71:

| Area | Owner |
| --- | --- |
| Stable blob-storage and billing records | `storage::stable::blob_storage` |
| Record/view/DTO conversion | `ops::blob_storage` or `view::blob_storage` |
| Pure readiness/funding decision helpers | `policy::blob_storage`, if extracted |
| Cashier bounded-wait calls and response decoding | `ops::cashier` |
| Endpoint guard mapping and public error mapping | endpoint/API boundary |
| Product sequencing across status/sync/fund/re-check | product backend, hub, operator, or CLI |
| Upload cost estimation and pricing UX | product backend/frontend or Cashier direct integration |

Allowed cleanup:

1. Move pure billing status decision construction into `policy::blob_storage`
   if it has no async calls, storage access, DTO serialization, or public error
   mapping.
2. Move Cashier response-to-status view projection into ops or view modules if
   it is a deterministic conversion.
3. Keep Cashier wrappers in `ops::cashier`.
4. Keep gateway-principal stable state in `ops::blob_storage`.
5. Keep public error mapping at API/endpoint boundaries.

Do not add `workflow::blob_storage` just to satisfy an audit. Add it only if
there is a real multi-step orchestration owner that needs retry/idempotency
state. If added, workflow may sequence ops-owned steps but must not own stable
records, DTO conversion, Cashier protocol defaults, or endpoint error mapping.

## 12. Safety Invariants

0.71 must preserve:

1. Status does not mutate gateway-principal state.
2. Status does not transfer cycles.
3. Sync does not transfer cycles.
4. Funding does not sync gateway principals.
5. Funding is explicit and requires `--cycles`.
6. Dry-run modes perform no calls.
7. CLI JSON output is stable enough for automation.
8. CLI plain output stays narrow enough for terminals.
9. CLI commands resolve canisters through installed deployment state and local
   Candid artifacts where available.
10. Medic remains diagnostic and never mutates canister state.
11. Readiness decisions are computed once in a shared model and reused by
    plain text, JSON, and medic renderers.
12. Post-action diagnostic failure is reported as diagnostic failure, not as
    sync/fund failure, when the mutating call succeeded.
13. JSON output includes a `kind` discriminator and uses structured error
    objects for `--json` failures.
14. Method mode comes from local Candid or installed endpoint metadata, not
    semantic guesses.
15. Successful command output goes to stdout; diagnostics and structured
    `--json` errors go to stderr.

## 15. Open Questions

1. Should `status` live only under `canic blob-storage`, or should there also
   be a read-only alias under `canic info blob-storage`?

   Recommendation: start with only `canic blob-storage status` to keep all
   blob-storage operator actions discoverable in one group.

2. Should CLI status support `--sync-requested` to show the status DTO's
   read-only skipped action?

   Recommendation: no for MVP. It would confuse operators because status still
   does not sync.

3. Should 0.71 add a billing-config CLI command?

   Recommendation: no, unless a canister already exposes a deliberately guarded
   config endpoint. Do not create a broad public billing-config admin surface
   just for CLI convenience.

4. Should medic scan all canisters by default?

   Recommendation: no for MVP. Use a targeted `--blob-storage <role-or-id>`
   option to keep output readable and avoid surprising update calls.

5. Should funding use interactive confirmation for large values?

   Recommendation: no for MVP. Treat explicit `--cycles` as confirmation and
   keep scripts non-interactive. Use `--dry-run` examples in docs.

## 16. Release Exit Criteria

0.71.0 is releasable when:

1. `canic blob-storage status` works locally and produces plain/JSON output.
2. `canic blob-storage sync-gateways` works locally and has dry-run support.
3. `canic blob-storage fund --cycles` works locally and has dry-run support.
4. JSON output includes `schema_version`, `kind`, stable readiness codes,
   stable command error codes, and decimal-string cycle values.
5. Target resolution and dry-run behavior are documented and tested.
6. Exit-code, stdout/stderr, and structured `--json` error behavior are
   documented and tested.
7. `canic info medic` can point operators at blob-storage blockers without
   mutating state.
8. The local runbook can be followed by a dev without reading test internals.
9. The relevant CLI, host, protocol, and PocketIC tests pass.
10. Programmatic consumer guidance documents the 0.70 endpoint sequence,
    target canister role, guard ownership, status DTO semantics, and
    product-owned orchestration obligations.
11. No public endpoint/Candid/stable-storage compatibility break is introduced.
12. The root changelog and detailed 0.71 notes identify the CLI/operator
   surfaces clearly.
