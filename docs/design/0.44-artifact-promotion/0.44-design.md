# 0.44 Design: Artifact Overrides And Promotion

## Status

Tentative design notes for the 0.44 line.

0.44 makes artifact selection and promotion explicit.

Core rule:

```text
Promote artifact identity, not authority.
```

A promoted artifact enters the target trust domain only after target embedded
config and target authority are validated.

---

## Scope Calibration

Release bar:

- digest-pinned artifact override plan;
- promotion readiness report;
- role-scoped promotion artifact level: sealed wasm vs source/build;
- target canonical embedded config check.

Design hook:

- external wasm and pinned wasm sources;
- explicit config-independent role flags;
- verifier, role epoch, and cascade prerequisites in promotion readiness;
- role-scoped artifact override planning;
- store-backed receipt artifact locators.

Deferred:

- signed promotion attestations;
- cross-deployment artifact registry semantics;
- rollback selection UX.

---

## Dependency On Earlier Lines

0.44 assumes:

- 0.41 records source artifact bytes, installed module bytes, embedded config,
  and receipts;
- 0.42 can reconcile target authority;
- 0.43 can execute a materialized plan without hardcoded local layout
  semantics.

Promotion must be a plan transformation, not an install shortcut.

## 0.42.14 Handoff Constraints

Promotion must preserve the authority boundary 0.42 closed:

- Authority `Safe` is authority-scoped and does not replace the deployment
  safety report.
- Authority command success means a local authority artifact was produced; it
  does not imply live controller mutation or whole-deployment safety.
- Authority dry-run receipts/evidence are structural self-consistency
  artifacts, not tamper-evident proof of live controller state.
- `Authority*V1` JSON fields and enum strings are stable automation surfaces.
- Future mutating authority receipts must stay schema-distinct from dry-run
  receipts.

## Role Artifact Sources

0.44 should support per-role artifact sources:

```text
workspace_package
published_package
local_wasm
local_wasm_gz
previous_receipt_artifact
canonical_wasm_store_default
```

Every executable override must be digest-pinned or materialized into a digest
before execution.

`previous_receipt_artifact` means deployment execution or artifact receipt
evidence. It explicitly excludes `AuthorityReceiptV1` and
`AuthorityDryRunEvidenceV1`. Authority dry-run artifacts prove structural
self-consistency of authority reporting only; they must not be used as source
bytes, artifact identity, promotion evidence, or proof of current live state.

Tentative source shape:

```text
RoleArtifactSource {
  kind,
  locator,
  expected_wasm_sha256,
  expected_wasm_gz_sha256,
  expected_candid_sha256,
  expected_canonical_embedded_config_sha256,
}
```

---

## Promotion Model

Promotion creates a target plan from a source receipt and selected role
artifacts.

Inputs:

```text
source_deployment_receipt
target_deployment
roles_to_promote
target_runtime_variant
target_authority_profile
```

`source_deployment_receipt` must be a deployment execution or artifact
receipt, not an `AuthorityReceiptV1` or `AuthorityDryRunEvidenceV1`.

Output:

```text
DeploymentPlanV1
```

Promotion must:

- explicitly choose a promotion artifact level for each promoted role;
- preserve the promoted identity allowed by that artifact level;
- recompute target canonical embedded config digest when the artifact level
  requires a target-configured build;
- resolve target root trust anchor;
- resolve target authority profile;
- resolve target pool policy;
- validate target verifier-readiness assumptions;
- refuse unresolved executable assumptions.

Promotion must not copy by default:

- source controllers;
- source root principal;
- source authority profile;
- source network;
- source pool canister IDs;
- source deployment operation epoch.

---

## Source Recipe, Materialization Input, And Result

Source/build promotion must not overload one object to mean both "the build
recipe" and "the bytes produced by this build." It also must not treat
target-specific embedded config as part of the reusable source/build recipe.

Use three separate concepts:

```text
BuildRecipeIdentityV1 {
  source_kind,
  source_revision,
  package_or_role_selector,
  cargo_profile,
  cargo_features_digest,
  cargo_lock_digest,
  rust_toolchain,
  builder_version,
}

BuildMaterializationInputV1 {
  build_recipe_id,
  canonical_embedded_config_sha256,
  network,
  root_trust_anchor,
  runtime_variant,
}

BuildMaterializationResultV1 {
  build_recipe_id,
  materialization_input_digest,
  wasm_sha256,
  wasm_gz_sha256,
  installed_module_hash,
  candid_sha256,
}
```

The build recipe identifies the code and build selection. The materialization
input identifies target-specific config deliberately supplied for one
deployment. The materialization result identifies the concrete output produced
from that recipe and input.

For source/build promotion:

```text
source build recipe id == target build recipe id
source materialization input digest may != target materialization input digest
source wasm digest may != target wasm digest
```

For Git-backed sources, `source_revision` should be the full commit SHA and the
receipt should say whether the source tree was clean. A dirty tree can still be
reported as evidence, but it should not be treated as a promotable
source/build recipe unless the dirty content is captured by an explicit source
identity.

## Promotion Artifact Levels

Promotion may operate at one of two levels.

### SealedWasmArtifact

This promotes exact wasm or wasm.gz bytes.

Rules:

- final wasm bytes are byte-identical to the source sealed artifact;
- source and target sealed wasm digests match;
- embedded config identity must already be valid for the target;
- if embedded config is target-specific and differs, the promotion is invalid.

### SourceOrBuildArtifact

This promotes the source/build recipe, not the sealed wasm bytes.

The source/build recipe identity preserves:

- source revision;
- source kind;
- role or package selector;
- Cargo profile;
- Cargo feature set or feature digest;
- relevant toolchain and builder versions.

Target-specific materialization may intentionally replace:

- canonical embedded config digest;
- deployment root or trust-domain config;
- network or runtime variant config.

Rules:

- source/build recipe identity must match the promotion constraints;
- target-specific config input is intentionally recomputed for the target;
- target materialization produces a new sealed wasm artifact;
- target wasm digest and target config digest must be recorded separately;
- the plan must not claim byte identity unless sealed wasm promotion is used.

A promotion plan must explicitly say whether it is promoting sealed bytes or
source/build identity. It must not describe a target-config-recomputed artifact
as byte-identical to the source sealed wasm.

Roles may be declared config-independent only when the artifact metadata proves
that embedded deployment config does not affect their sealed wasm bytes.

## Role Promotion Plans And Policy

Promotion mode is role-scoped and required.

Tentative shape:

```text
RolePromotionPlanV1 {
  role,
  promotion_level,
  source_receipt_id,
  source_sealed_artifact_identity,
  source_build_recipe_identity,
  source_materialization_input_digest,
  target_materialization_input_digest,
  target_materialization_policy,
}

PromotionLevelV1 {
  SealedWasm,
  SourceBuild,
}
```

Minimal role policy:

```text
RolePromotionPolicyV1 {
  role,
  allowed_promotion_levels,
  require_same_source_revision,
  require_same_cargo_features,
  require_target_config_digest,
  require_byte_identical_wasm,
}
```

This is not a general policy language. It is the minimum needed to say, for
example, that one role must reuse sealed bytes while another role may rebuild
the same source recipe with target config.

Validation rules:

```text
SealedWasm:
  target wasm bytes must match source sealed artifact digest
  target embedded config identity must be acceptable for target deployment

SourceBuild:
  source/build recipe must satisfy promotion constraints
  target materialization input may differ from source materialization input
  target config digest is intentionally recomputed and recorded
  target wasm digest is newly materialized and recorded
```

Promotion reports must not call a target-config rebuild "the same artifact"
unless the sealed bytes are actually identical.

## Embedded Config

The common footgun:

```text
tested staging artifact
-> copied into prod
-> still embeds staging root or staging role topology
```

0.44 must prevent this.

The promotion report should explicitly show:

- promotion artifact level;
- source wasm digest;
- target wasm digest;
- source/build identity when applicable;
- source canonical embedded config digest;
- target canonical embedded config digest;
- source/build recipe identity when promotion level is `SourceBuild`;
- source materialization input digest when promotion level is `SourceBuild`;
- target materialization input digest when promotion level is `SourceBuild`;
- target materialization result when promotion level is `SourceBuild`;
- whether the module bytes are byte-identical;
- whether the embedded config is byte-identical;
- whether byte identity is required by the promotion policy.

Byte-identical wasm is allowed only when the embedded config identity is also
acceptable for the target plan.

## Artifact Diff

Artifact identity drift should be first-class, not prose.

Diff categories:

```text
same_source_bytes
different_source_bytes
same_installed_module_hash
different_installed_module_hash
same_embedded_config
different_embedded_config
same_candid
different_candid
unknown
```

These categories should reuse 0.41 `DeploymentDiffV1` rather than inventing
promotion-only semantics.

## `wasm_store`

`wasm_store` participates as a role artifact:

```text
source = canonical_wasm_store_default
source = previous_receipt_artifact
source = local_wasm_gz
```

Promotion must still respect bootstrap ordering:

```text
executor bootstraps wasm_store
later staging may use wasm_store transport
```

Receipt-backed artifact references may include store-backed locators:

```text
source receipt artifact:
  role = ledger
  sealed wasm digest = ...
  canonical embedded config digest = ...
  store locator = wasm_store://<store>/<template_id>/<version>/<payload_hash>
```

Authority dry-run receipts and evidence are excluded from receipt-backed
artifact references.

Promotion readiness should report whether the target store already has the
artifact. Absence from the target store means restage required, not promotion
invalid.

## Commands

Tentative operator surface:

```text
canic deploy promote plan --from <receipt> --to <deployment> --role <role>
canic deploy promote check <plan>
canic deploy promote diff <source> <target>
```

Execution remains normal plan execution:

```text
canic deploy install --plan <promoted-plan>
```

---

## Non-Goals

0.44 should not:

- copy source authority into target deployments by default;
- bypass target authority reconciliation;
- bypass embedded config checks;
- implement user-owned consent workflows;
- make promotion execute implicitly.

---

## Implementation Slices

### Slice 1: Artifact Override Sources

Allow plan role artifacts to be resolved from digest-pinned override sources.

### Slice 2: Receipt Artifact References

Make receipts usable as artifact sources without making receipts authoritative
about current live state.

### Slice 3: Promotion Plan Transformation

Create target plans from source receipts and selected roles.

Promotion plan transformation must emit role-scoped promotion levels and keep
source/build recipe identity separate from target-specific materialization
input and target materialization result.

### Slice 4: Target Embedded Config Recompute

Recompute target embedded config and fail mismatches explicitly.

### Slice 5: Promotion Readiness Report

Render the artifact, config, authority, and trust-domain differences that make
promotion safe or unsafe.

---

## Exit Criterion

0.44 is complete when:

```text
Canic can promote tested sealed bytes or source/build identity into another
trust domain without copying source authority or stale embedded topology.
```
