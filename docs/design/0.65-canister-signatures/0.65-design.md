# Canic Auth v0.65 Design: Hard Cut Root Auth Back to IC Canister Signatures

## Status

- Status: local implementation closeout complete; release preparation pending
- Release line: 0.65
- Decision style: hard cut
- Primary implementation target: delegated-token root proofs
- Current implementation pass: delegated-token `DelegationCert` root proof
  hard cut plus nanosecond-native auth/replay protocol DTO cleanup. Fresh
  one-shot root ECDSA role-attestation and internal-invocation proof issuance
  is removed from normal auth. Delegated-grant capability proofs remain a
  separately identified root ECDSA surface and are not part of delegated-token
  root proof verification.

## 1. Decision

Canic 0.65 hard-cuts root-issued authentication proofs from threshold ECDSA
back to IC canister signatures.

The important preserved contract is the external delegated-token contract:

```rust
pub struct DelegatedToken {
    pub claims: DelegatedTokenClaims,
    pub proof: DelegationProof,
    pub shard_sig: Vec<u8>,
}
```

Callers still supply a self-contained `DelegatedToken` at the endpoint
boundary. Verifiers still validate the embedded proof locally. No verifier may
require root proof fanout, verifier-local proof history, or a root call on the
request hot path.

The hard cut changes the root-proof signing backend and proof DTO:

```rust
pub struct DelegationProof {
    pub cert: DelegationCert,
    pub root_proof: RootProof,
}

pub enum RootProof {
    IcCanisterSignatureV1(IcCanisterSignatureProof),
}
```

There is no legacy threshold-ECDSA acceptance branch in 0.65.

## 2. Problem

The pre-0.65 root proof path was:

```text
AuthApi::issue_delegation_proof
-> AuthOps::prepare_delegation_proof
-> root threshold ECDSA signing of the prepared DelegationCert hash
-> EcdsaOps::sign_bytes
-> management canister sign_with_ecdsa
```

That makes root-issued auth scale with threshold ECDSA calls. Threshold ECDSA is
the wrong cost model for root auth proofs that may be minted repeatedly.

The original security goals remain correct:

- root is the canonical Canic authority
- target canisters verify root-approved auth locally
- root is not called on every endpoint request
- root does not keep a normal private signing key in Wasm memory

IC canister signatures satisfy those goals without per-proof threshold ECDSA.

## 3. Hard-Cut Consequences

0.65 is a protocol cut, not a compatibility bridge.

- Existing delegated tokens from the threshold-ECDSA root-proof epoch stop
  verifying after the 0.65 verifier upgrade.
- `SignatureAlgorithm::EcdsaP256Sha256` is removed or renamed wherever it
  described secp256k1 threshold ECDSA.
- Root ECDSA key publication through delegated `SubnetState` is no longer a
  verifier requirement for root proof verification.
- Root proof endpoints change from one update returning a finished ECDSA
  signature to update-then-query canister-signature issuance.
- Candid surfaces that expose root proofs are regenerated to the hard-cut
  shape. Do not add delegated-auth version lanes for compatibility.

This cut still keeps the external delegated-token shape self-contained. The
proof inside the token changes; the token remains the verifier input.

## 4. Scope

### In Scope

- Replace root threshold ECDSA over `DelegationCert` with IC canister
  signatures.
- Preserve `DelegatedTokenClaims.cert_hash` as the canonical binding between
  token claims and the root-certified delegation certificate.
- Keep `DelegatedToken` self-contained.
- Use a two-phase root API:
  - update prepares the signature-map entry and commits certified data
  - query returns the certificate and hash-tree witness as the canister
    signature
- Centralize root `set_certified_data` ownership.
- Rename current threshold-ECDSA algorithm labels so they say secp256k1, not
  P-256.
- Identify remaining root ECDSA auth surfaces so they are not mistaken for
  completed delegated-token root proof work.

### Out of Scope For This Slice

- Replacing shard token signatures. Current `DelegatedToken.shard_sig` remains
  a shard signature over `DelegatedTokenClaims`. If shard token signing volume
  is also a cycle problem, that is a second hard cut.
- Replacing delegated-grant capability proof signing in the first
  delegated-token proof implementation pass. Role-attestation and
  internal-invocation fresh one-shot issuance is removed from normal auth in
  0.65; any future replacement must use update-then-query root proofs.
- Introducing verifier-local grant state as a required authority source.
- Creating a generic root signing oracle.
- Making root canister-sign every endpoint request.

## 5. Cargo Feature Layout

0.65 must not hide the new crypto split behind the existing broad
`auth-crypto` name. That feature currently means threshold-ECDSA management
support. The hard cut needs feature names that describe the actual capability.

### canic-core features

```toml
[features]
default = []
sharding = []
auth-root-canister-sig-create = [
    "dep:ic-canister-sig-creation",
    "dep:ic-certification",
]
auth-root-canister-sig-verify = ["dep:ic-signature-verification"]
auth-shard-secp256k1-verify = ["dep:k256"]
auth-threshold-ecdsa-sign = []
auth-delegated-token-verify = [
    "auth-root-canister-sig-verify",
    "auth-shard-secp256k1-verify",
]
```

Feature meanings:

- `auth-root-canister-sig-create`: root-side IC canister-signature creation,
  signature-map maintenance, and certified-data refresh.
- `auth-root-canister-sig-verify`: verifier-side IC canister-signature
  validation only.
- `auth-shard-secp256k1-verify`: verifier-side delegated-token shard
  signature verification.
- `auth-threshold-ecdsa-sign`: management-canister threshold ECDSA signing and
  public-key fetching. In 0.65 this remains for shard token signing, shard
  public-key resolution during root cert preparation, and explicit
  non-delegated-token root-auth uses. The management-canister crate is still a
  normal dependency because Canic uses it for other management APIs; the
  threshold ECDSA calls themselves are feature-gated.
- `auth-delegated-token-verify`: convenience verification feature for
  endpoints that accept `DelegatedToken`; it includes root canister-signature
  verification and shard secp256k1 verification.

`auth-crypto` should be removed as a public feature name in this hard cut. If
the implementation keeps it briefly for local transition convenience, it must
not become an umbrella that silently enables both canister signatures and
threshold ECDSA.

### canic facade features

The facade should mirror the lower-level capability names:

```toml
[features]
auth-root-canister-sig-create = ["canic-core/auth-root-canister-sig-create"]
auth-root-canister-sig-verify = ["canic-core/auth-root-canister-sig-verify"]
auth-shard-secp256k1-verify = ["canic-core/auth-shard-secp256k1-verify"]
auth-threshold-ecdsa-sign = ["canic-core/auth-threshold-ecdsa-sign"]
auth-delegated-token-verify = ["canic-core/auth-delegated-token-verify"]
```

Do not add a facade feature that bundles these with `control-plane`, `sharding`,
or `metrics`. Root crates choose the capabilities they need explicitly.

### Role build requirements

Root auth issuer:

```toml
canic = { features = [
    "control-plane",
    "auth-root-canister-sig-create",
    "auth-threshold-ecdsa-sign",
] }
```

`auth-root-canister-sig-create` is the only feature required for the
canister-signature map itself. The current threshold-ECDSA shard-key model also
requires management-canister public-key fetching so root can certify the shard
SEC1 public key. The 0.65 delegated-token root proof path must not call
`sign_with_ecdsa`; root threshold-ECDSA signing remains only in the known
non-delegated-token proof surfaces until the follow-up pass removes or converts
them.

Verifier-only target canister:

```toml
canic = { features = ["auth-delegated-token-verify"] }
```

Delegated token signer shard:

```toml
canic = { features = [
    "auth-delegated-token-verify",
    "auth-threshold-ecdsa-sign",
] }
```

Root canister-signature creation does not require threshold-ECDSA signing.
The current shard-key certification path does require the management-canister
ECDSA public-key call while `DelegatedToken.shard_sig` remains threshold
ECDSA-backed.
Verifier-only canisters do not require `auth-threshold-ecdsa-sign`.

Runtime startup must keep the existing config/build contract style:

- delegated-token verifier config requires `auth-delegated-token-verify`
- root proof issuer config requires `auth-root-canister-sig-create`
- delegated-token signer config requires both `auth-delegated-token-verify`
  and `auth-threshold-ecdsa-sign`
- missing compile support traps during startup, not later during auth warnings

Verifier trust-anchor config:

```toml
[auth.delegated_tokens]
enabled = true
root_canister_id = "..."
ic_root_public_key_raw_hex = "..."
network = "mainnet" # mainnet | local | pocketic | testnet
```

TOML remains ergonomic: `ic_root_public_key_raw_hex` is operator-facing
hex input, but runtime verification resolves to raw bytes:

```rust
pub struct DelegatedTokenVerifierConfig {
    pub root_canister_id: Principal,
    pub ic_root_public_key_raw: Vec<u8>,
}
```

Verifier startup rule:

- if delegated-token verification is enabled on an endpoint canister, startup
  traps unless `auth-root-canister-sig-verify` is compiled
- startup traps unless there is an effective root canister id from
  `auth.delegated_tokens.root_canister_id` or the initialized Canic root env
- startup traps unless there is an effective raw IC root public key from
  `auth.delegated_tokens.ic_root_public_key_raw_hex` or a local/PocketIC test
  root-key provider

`SubnetState::delegated_root_public_key` is not read during 0.65
delegated-token root proof verification. It may remain temporarily for
delegated-grant capability or other unrelated ECDSA-era paths, but it is not
part of the delegated-token root-proof verifier.

## 6. Preserved Delegated-Token Contract

Delegated-token verification remains self-validating.

Verifier inputs:

- the supplied `DelegatedToken`
- configured root principal
- IC root public key for the network being verified
- shard public key embedded in the root-certified `DelegationCert`
- local project id, local canister role, required scopes, and canister time

Verifier must not require:

- local proof presence
- proof fanout from root
- proof history replication
- query-time root key fetching
- registry snapshots for delegated-token audience membership
- a root call on request verification

The hot path remains:

```text
decode first ingress argument as DelegatedToken
verify root proof embedded in token
verify shard signature embedded in token
bind claims.subject to msg.caller()
check audience, local role, required scope, and expiry
```

Implementations may cache successful verification results for performance, but
that cache is not an authority source. Cache miss means re-verify the
self-contained token, not fetch proof state from root.

Verification is non-consuming. A valid `DelegatedToken` is reusable across
multiple update calls until its signed TTL, audience, role, and scope limits
expire or no longer authorize the endpoint. Core delegated auth does not track
token nonce reuse for endpoint authorization.

### Revocation Model

0.65 delegated tokens and delegation proofs are self-contained. A verifier that
has the token and the configured IC root key can verify without online root
state. Therefore, emergency revocation before `expires_at_ns` is not
guaranteed by the protocol unless the verifier also has a separately updated
root-certified revocation epoch.

The 0.65 hard-cut mitigation is short cert/token TTLs, strict
`max_token_ttl_ns`, and shard key rotation. Any stronger revocation model is a
separate protocol addition and would weaken the no-required-verifier-state
contract.

## 7. IC Canister-Signature Model

A canister signature is not ECDSA, Ed25519, or a private key stored in canister
memory.

Root signs by committing an entry to its certified data tree. Verification
checks an IC certificate and hash-tree witness against the IC root key. The
signature proves that the root canister certified a signature-map path for a
specific seed and message.

Operationally, creation is two-phase:

```text
1. update: prepare signature entry and call set_certified_data
2. query: read data_certificate and return certificate + witness as CBOR
```

This is not a drop-in replacement for `sign_bytes()`. An update method cannot
reliably return the final canister signature because the certificate is exposed
to query execution.

Root auth signing must not be deployed on a subnet whose canister signatures
are not valid for verification.

Deployment invariant:

```text
root_auth_canister.subnet_type != cloud_engine
```

This is enforced by deployment tooling, not by relying on the root canister to
discover its subnet type at runtime. If runtime defense is needed too, the
checked subnet type or deployment attestation is passed into root config and
root traps when it says `cloud_engine`.

0.65 deployment tooling preserves `SubnetType::CloudEngine` as
`SubnetKind::CloudEngine` in the cached mainnet subnet catalog and the local
deployment truth check blocks a mainnet root auth signer when the observed root
canister resolves to that subnet kind. Missing or unresolved mainnet subnet
evidence is also a deployment hard failure because the tool cannot prove the
root signer satisfies the canister-signature invariant.

## 8. Proof DTOs

### RootProof

```rust
pub enum RootProof {
    IcCanisterSignatureV1(IcCanisterSignatureProof),
}

pub struct IcCanisterSignatureProof {
    pub public_key_der: Vec<u8>,
    pub signature_cbor: Vec<u8>,
}
```

The verifier parses `public_key_der`, checks that it encodes the expected root
canister id and expected seed for the payload kind, and verifies
`signature_cbor` against the configured `ic_root_public_key_raw`.

### DelegationProof

```rust
pub struct DelegationProof {
    pub cert: DelegationCert,
    pub root_proof: RootProof,
}
```

The old `root_sig: Vec<u8>` field is removed.

### DelegationCert

`DelegationCert` remains the root-certified statement that a shard public key
may mint bounded delegated tokens for a specific audience and grant set.

Hard-cut shape:

```rust
pub struct DelegationCert {
    pub root_pid: Principal,
    pub shard_pid: Principal,
    pub shard_key_id: String,
    pub shard_sig_alg: ShardSignatureAlgorithm,
    pub shard_public_key_sec1: Vec<u8>,
    pub shard_key_hash: [u8; 32],
    pub shard_key_binding: ShardKeyBinding,
    pub issued_at_ns: u64,
    pub not_before_ns: u64,
    pub expires_at_ns: u64,
    pub max_token_ttl_ns: u64,
    pub aud: DelegationAudience,
    pub grants: Vec<DelegatedRoleGrant>,
}

pub enum ShardSignatureAlgorithm {
    IcThresholdEcdsaSecp256k1,
}

pub enum ShardKeyBinding {
    IcThresholdEcdsaSecp256k1 {
        key_name_hash: [u8; 32],
        derivation_path_hash: [u8; 32],
    },
}
```

Removed from the cert:

- `root_key_id`
- `root_key_hash`
- root `alg`

The root public key for canister signatures is derived from:

```text
root canister principal + purpose seed
```

It is not a threshold-ECDSA SEC1 public key cascaded through `SubnetState`.

Shard key hash invariant:

```text
shard_key_hash =
  sha256(
    "canic-shard-key-v1" ||
    shard_sig_alg ||
    shard_public_key_sec1 ||
    canonical_bytes(shard_key_binding)
  )
```

Root must verify or derive the shard public key before certifying it. Endpoint
verifiers do not perform a management-canister public-key lookup; they trust the
root-certified cert and still check internal consistency.

### DelegatedTokenClaims

`DelegatedTokenClaims` keeps its current role:

```rust
pub struct DelegatedTokenClaims {
    pub subject: Principal,
    pub issuer_shard_pid: Principal,
    pub cert_hash: [u8; 32],
    pub issued_at_ns: u64,
    pub expires_at_ns: u64,
    pub aud: DelegationAudience,
    pub grants: Vec<DelegatedRoleGrant>,
    pub nonce: [u8; 16],
}
```

`cert_hash` remains:

```text
sha256(canonical_bytes(DelegationCert))
```

That is how token claims bind to the root-certified shard authority.

All 0.65 auth protocol DTO timestamps are nanoseconds since the Unix epoch.
Auth DTOs must use `_ns` suffixes for absolute times and TTLs. Do not mix
seconds and nanoseconds inside signed auth payloads, replay metadata, pending
proof metadata, or capability proof metadata. `expires_at_ns` is exclusive:

```text
not_before_ns <= now_ns < expires_at_ns
```

External operator config may remain seconds for ergonomics, but conversion
happens exactly once at the config/API boundary with checked arithmetic:

```rust
let max_token_ttl_ns = max_token_ttl_secs
    .checked_mul(1_000_000_000)
    .ok_or(ConfigError::DurationOverflow)?;
```

Do not add `DELEGATED_AUTH_VERSION` or equivalent compatibility epochs for this
cut. The canonical hard-cut DTO shape is the protocol.

The hard-cut ns rule applies beyond delegated-token DTOs:

```rust
pub struct RootRequestMetadata {
    pub request_id: [u8; 32],
    pub ttl_ns: u64,
}

pub struct CapabilityRequestMetadata {
    pub request_id: [u8; 16],
    pub nonce: [u8; 16],
    pub issued_at_ns: u64,
    pub ttl_ns: u64,
}

pub struct RoleAttestation {
    pub issued_at_ns: u64,
    pub expires_at_ns: u64,
    // other fields unchanged
}

pub struct InternalInvocationProofPayloadV1 {
    pub issued_at_ns: u64,
    pub expires_at_ns: u64,
    // other fields unchanged
}

pub struct DelegatedGrant {
    pub issued_at_ns: u64,
    pub expires_at_ns: u64,
    // other fields unchanged
}
```

Config fields such as `max_ttl_secs` may remain seconds. Runtime conversion to
protocol DTOs must use checked multiplication and must not store `*_secs` in
auth protocol structs.

### Shard Signature Compatibility

0.65 does not replace `DelegatedToken.shard_sig`, but shard signature bytes are
still part of endpoint auth and must be canonical.

Shard token signing input:

```text
shard_token_message =
  domain_len ||
  b"canic-shard-delegated-token" ||
  canonical_bytes(DelegatedTokenClaims)

shard_token_hash = sha256(shard_token_message)
```

Threshold ECDSA signs `shard_token_hash`.

Shard public key encoding:

```text
compressed SEC1 secp256k1 public key, exactly 33 bytes
```

Shard signature encoding:

```text
fixed 64-byte secp256k1 ECDSA signature: r || s
```

Verifier rejects:

- noncanonical shard public-key encoding
- malformed shard signature encoding
- noncanonical `DelegatedTokenClaims` encoding

## 9. Canonical Payloads

Do not introduce deterministic CBOR for auth payloads in 0.65. Use the existing
Canic auth canonical encoding path and extend it for the hard-cut DTOs.

Current canonical rules remain important:

- vectors are already sorted and duplicate-free
- role, scope, and project strings are strict ASCII labels
- verifiers reject noncanonical data instead of normalizing it
- one semantic token has one valid canonical encoding

Root canister signatures sign typed canonical hashes, not arbitrary caller
bytes.

For `DelegationCert`:

```text
cert_id   = cert_hash
cert_hash = sha256(canonical_bytes(DelegationCert))
seed      = b"canic-root-delegation-cert"
domain    = b"canic-root-delegation-cert"
message   = cert_hash
```

Root preparation and verifier verification must use the same helper for
`seed`, `domain`, and `message`. Do not let a proof choose its own semantic
domain. The payload type determines the expected seed and domain.

Equivalent root-proof domains:

```rust
fn root_sig_domain(kind: RootPayloadKind) -> &'static [u8] {
    match kind {
        RootPayloadKind::DelegationCert => b"canic-root-delegation-cert",
        RootPayloadKind::RoleAttestation => b"canic-root-role-attestation",
        RootPayloadKind::InternalInvocationProof => {
            b"canic-root-internal-invocation-proof"
        }
        RootPayloadKind::TopologyEpoch => b"canic-root-topology-epoch",
    }
}

fn root_sig_seed(kind: RootPayloadKind) -> &'static [u8] {
    match kind {
        RootPayloadKind::DelegationCert => b"canic-root-delegation-cert",
        RootPayloadKind::RoleAttestation => b"canic-root-role-attestation",
        RootPayloadKind::InternalInvocationProof => {
            b"canic-root-internal-invocation-proof"
        }
        RootPayloadKind::TopologyEpoch => b"canic-root-topology-epoch",
    }
}
```

For 0.65, root signature seed and domain are equal. They remain separate
helpers so a future proof-kind can change one without making verifiers infer
semantics from the proof bytes.

Creation and verification differ at the final byte boundary. The DFINITY
creation helper stores a signature-map path using:

```text
message_hash = sha256(domain_len || domain || message)
```

`ic_signature_verification::verify_canister_sig` hashes the message bytes
passed to it and looks for:

```text
["sig", sha256(seed), sha256(message)]
```

Therefore the verifier must pass `domain_len || domain || cert_hash`, not raw
`cert_hash`.

```rust
pub fn root_canister_sig_inputs<'a>(
    kind: RootPayloadKind,
    payload_hash: &'a [u8; 32],
) -> CanisterSigInputs<'a> {
    CanisterSigInputs {
        domain: root_sig_domain(kind),
        seed: root_sig_seed(kind),
        message: &payload_hash[..],
    }
}

pub fn root_canister_sig_verification_message(
    kind: RootPayloadKind,
    payload_hash: [u8; 32],
) -> Vec<u8> {
    let domain = root_sig_domain(kind);
    assert!(domain.len() <= 255);

    let mut msg = Vec::with_capacity(1 + domain.len() + 32);
    msg.push(domain.len() as u8);
    msg.extend_from_slice(domain);
    msg.extend_from_slice(&payload_hash);
    msg
}
```

## 10. Root API

The old one-update root proof API cannot remain the fresh-proof issuance path.

### Prepare Delegation Proof

```rust
#[update]
pub fn prepare_delegation_proof(
    req: DelegationProofIssueRequest,
) -> Result<PreparedDelegationProof, AuthError>;
```

Returns:

```rust
pub const ROOT_PROOF_RETRIEVAL_TTL_NS: u64 = 60_000_000_000;

pub struct PreparedDelegationProof {
    pub cert: DelegationCert,
    pub cert_hash: [u8; 32],
    pub retrieval_expires_at_ns: u64,
}
```

`ROOT_PROOF_RETRIEVAL_TTL_NS` matches the upstream
`ic-canister-sig-creation` signature-map retention period. The upstream map
uses a fixed one-minute retention and prunes expired signatures during later
`add_signature` calls. The query path cannot mutate state to prune expired
entries.

Canic therefore tracks pending proof metadata:

```rust
pub struct PendingRootProof {
    pub operation_id: [u8; 32],
    pub cert_hash: [u8; 32],
    pub cert: DelegationCert,
    pub retrieval_expires_at_ns: u64,
    pub prepared_by: Principal,
}
```

Root does:

```rust
let cert_hash = cert_hash(&cert)?;
let now_ns = ic_time_ns();
let retrieval_expires_at_ns = now_ns
    .checked_add(ROOT_PROOF_RETRIEVAL_TTL_NS)
    .ok_or(AuthError::TimeOverflow)?;

let inputs = root_canister_sig_inputs(
    RootPayloadKind::DelegationCert,
    &cert_hash,
);

SIGNATURES.with_borrow_mut(|sigs| {
    sigs.add_signature(&inputs);
});

PENDING_ROOT_PROOFS.insert(
    (caller, operation_id),
    PendingRootProof {
        operation_id,
        cert_hash,
        cert,
        retrieval_expires_at_ns,
        prepared_by: caller,
    },
);

CertifiedDataOps::refresh();
```

### Prepare/Get Replay Model

`prepare_delegation_proof` is replay-protected. `get_delegation_proof` is not
separately replay-protected because it is a query and is deterministic over an
existing prepared cert.

Replay identity:

```text
prepare_key = (prepared_by, operation_id)
```

Prepare rules:

- same `prepare_key` plus the same request returns the same
  `PreparedDelegationProof`
- same `prepare_key` plus a different request returns replay conflict
- first successful prepare calls `SIGNATURES.add_signature` once
- first successful prepare refreshes certified data once
- committed replay must not silently extend `retrieval_expires_at_ns`
- if the prepared proof expired, was pruned, or was lost after upgrade, the
  caller must use a new `operation_id`

The replay receipt stores the committed `PreparedDelegationProof`. The pending
proof index stores enough query-side state to return the proof without
rebuilding or re-certifying it.

The reason for replay protection has changed. In 0.61, replay protection
guarded a threshold ECDSA signing effect. In 0.65, prepare protection guards
signature-map growth, certified-data churn, and duplicate pending proof state.

### Get Delegation Proof

```rust
#[query]
pub fn get_delegation_proof(
    req: GetDelegationProofRequest,
) -> Result<DelegationProof, AuthError>;
```

Request:

```rust
pub struct GetDelegationProofRequest {
    pub cert_hash: [u8; 32],
}
```

Root looks up the prepared cert by `(caller, cert_hash)`, derives the expected
signature inputs, and returns:

```rust
let pending = PENDING_ROOT_PROOFS
    .get_by_caller_and_cert_hash(caller, req.cert_hash)
    .ok_or(AuthError::PreparedProofNotFound)?;

require(pending.prepared_by == caller)?;
require(req.cert_hash == pending.cert_hash)?;
require(now_ns < pending.retrieval_expires_at_ns)?;
```

The query must reject an expired pending proof even if the signature-map witness
is still present. `get_signature_as_cbor` is the witness retrieval primitive,
not the Canic API-level expiry check.

```rust
DelegationProof {
    cert,
    root_proof: RootProof::IcCanisterSignatureV1(IcCanisterSignatureProof {
        public_key_der,
        signature_cbor,
    }),
}
```

### Token Issuance Impact

`AuthApi::issue_token` remains the canonical token minting path because the
caller supplies an explicit self-contained `DelegationProof`.

`AuthApi::mint_token` cannot always fetch a fresh root canister signature from
inside one update call. Inter-canister update execution is not the query phase
that exposes the root data certificate.

0.65 removes single-call fresh-proof `mint_token` from the normal auth surface.
Fresh token issuance is:

```text
1. caller -> root.prepare_delegation_proof update
2. caller -> root.get_delegation_proof query
3. caller -> shard.issue_token update, supplying DelegationProof
4. caller uses DelegatedToken at endpoint boundary
```

A future optimization may add cached-proof `mint_token`, but it is not part of
the 0.65 hard cut. The issued `DelegatedToken` is reusable for multiple update
calls until its signed TTL and grants no longer authorize the request.

## 11. Certified-Data Owner

Root has one 32-byte certified data value. Canic root therefore needs exactly
one module that owns `set_certified_data`.

0.65 uses the simplest certified-data shape: the root auth canister certifies
only the canister-signature map.

```rust
set_certified_data(labeled_hash(b"sig", SIGNATURES.root_hash()));
```

No `http_assets` or `canic_state` certified subtrees share the same certified
data value in the 0.65 hard cut.

If a later slice certifies additional data in the same canister, the refresh
path and query witness path must use the same exact tree shape. The query
method must return:

- the `"sig"` witness for the requested seed/message
- pruned siblings for every other certified subtree
- a tree whose digest equals the canister's current certified data value

Calling `get_signature_as_cbor(..., None)` is valid only when the committed
certified data is exactly `labeled_hash("sig", SIGNATURES.root_hash())`.

On `init` and `post_upgrade`, root must resync certified data from the
signature map or explicitly reset to an empty valid signature root. Prepared
signature tickets that were not retrieved before upgrade are retryable, not
stable promises.

## 12. Verification

Root proof verification for a delegated token:

```rust
fn verify_root_delegation_proof(
    proof: &DelegationProof,
    expected_root: Principal,
    ic_root_public_key_raw: &[u8],
) -> Result<[u8; 32], AuthVerifyError> {
    let cert_hash = cert_hash(&proof.cert)?;

    let RootProof::IcCanisterSignatureV1(root_proof) = &proof.root_proof;

    require(proof.cert.root_pid == expected_root)?;

    let (pk_canister_id, pk_seed) =
        parse_canister_sig_public_key_der(&root_proof.public_key_der)?;
    require(pk_canister_id == expected_root)?;
    require(pk_seed == root_sig_seed(RootPayloadKind::DelegationCert))?;

    let verifier_msg = root_canister_sig_verification_message(
        RootPayloadKind::DelegationCert,
        cert_hash,
    );

    verify_canister_sig(
        &verifier_msg,
        &root_proof.signature_cbor,
        &root_proof.public_key_der,
        ic_root_public_key_raw,
    )?;

    Ok(cert_hash)
}
```

`ic_root_public_key_raw` is the raw IC BLS root public key bytes, not DER. For
mainnet, use the known IC root key in raw form after DER extraction. For local
replica and PocketIC tests, use that environment's local root key. Tests must
not silently verify local certificates with the mainnet root key.

Then delegated-token verification continues as it does now:

- `claims.cert_hash == cert_hash`
- cert and token time windows are valid
- `cert.not_before_ns <= now_ns`
- `now_ns < cert.expires_at_ns`
- `claims.issued_at_ns >= cert.not_before_ns`
- `claims.issued_at_ns <= now_ns`
- `now_ns < claims.expires_at_ns`
- token does not outlive certificate or max token TTL
- `claims.issuer_shard_pid == cert.shard_pid`
- claims audience is a subset of cert audience
- local verifier accepts both token and cert audience
- claims grants are a subset of cert grants
- local role has the required endpoint scopes
- shard signature verifies under `cert.shard_public_key_sec1`
- endpoint guard binds `claims.subject == msg.caller()`

Time checks must use checked arithmetic:

```rust
require(cert.not_before_ns < cert.expires_at_ns)?;
require(cert.max_token_ttl_ns > 0)?;

require(claims.issued_at_ns >= cert.not_before_ns)?;
require(claims.issued_at_ns < cert.expires_at_ns)?;
require(claims.expires_at_ns > claims.issued_at_ns)?;
require(claims.expires_at_ns <= cert.expires_at_ns)?;

let max_claim_expiry = claims
    .issued_at_ns
    .checked_add(cert.max_token_ttl_ns)
    .ok_or(AuthVerifyError::TimeOverflow)?;

require(claims.expires_at_ns <= max_claim_expiry)?;
```

Never write `claims.issued_at_ns + cert.max_token_ttl_ns` inside auth
verification.

### Verification Cache

Positive verification caching is allowed and expected for repeated calls with
the same token. The cache is a performance optimization only.

Use two caches.

Root proof cache:

```text
proof_hash = sha256(canonical_bytes(DelegationProof))

key = proof_hash

value =
  verified cert_hash
  verified shard public key
  verified cert grant ceiling
  cert expires_at_ns
```

Delegated token cache:

```text
proof_hash     = sha256(canonical_bytes(DelegationProof))
claims_hash    = sha256(canonical_bytes(DelegatedTokenClaims))
shard_sig_hash = sha256(shard_sig)

key =
  sha256(
    "canic-delegated-token-cache-v1" ||
    proof_hash ||
    claims_hash ||
    shard_sig_hash ||
    caller_principal_bytes
  )
```

Delegated token cache value:

```text
verified subject
verified cert_hash
verified shard public key
verified grants
verified audience
valid_until_ns
```

Cache hit authorizes the current request only after checking caller, endpoint
scope, local role, audience, and `now_ns < valid_until_ns`. Cache hit does not
consume the token. Cache miss re-verifies the embedded proof and shard
signature. A cache hit must never apply to a different claims hash or a
different shard signature hash, even when the proof, cert hash, expiry, and
caller are the same.

## 13. Root ECDSA Auth Surface Audit

The 0.65 audit identified the following root auth surfaces outside the
delegated-token `DelegationCert` root proof path:

- role attestations
- internal-invocation proofs
- delegated-grant capability proofs

0.65 removes fresh one-shot root ECDSA issuance for role attestations and
internal-invocation proofs from the normal auth surface. The existing
verification DTOs may remain for explicitly supported legacy/non-normal paths,
but root RPC execution must not reserve threshold-ECDSA signing cycles, record a
threshold-ECDSA external effect, or call `sign_with_ecdsa` for these commands.
Delegated-grant capability proofs remain isolated from delegated-token root
proof verification and continue to be tracked as a separate retained surface.

A future replacement, if needed, must use the same update-then-query proof model:

```rust
pub struct RootCertified<T> {
    pub payload: T,
    pub root_proof: RootProof,
}

pub type SignedRoleAttestation = RootCertified<RoleAttestation>;
pub type SignedInternalInvocationProofV1 =
    RootCertified<InternalInvocationProofPayloadV1>;
```

These payloads would also require update-then-query issuance. Any code path that
expects root to return a freshly signed attestation from one update must either:

- use an explicit externally retrieved proof, or
- use a local cache populated before the protected operation

0.65 implementation rule:

- `canic_request_role_attestation` is replay-protected but hard-fails before
  ECDSA signing
- `canic_request_internal_invocation_proof` is replay-protected but hard-fails
  before ECDSA signing
- replay-policy rows for these commands use `CostClass::None`, not
  `ThresholdEcdsaSign`
- delegated tokens are the supported reusable normal endpoint-auth path

Non-root runtime placement RPCs must not use role attestations as a replacement
caller proof. For root-bound management requests issued by a registered
non-root canister, 0.65 uses structural capability proofs:

- `CreateCanister` may use a structural proof only when
  `parent == CreateCanisterParent::ThisCanister`; root resolves the parent to
  the caller and normal registry/config policy enforces whether that caller may
  create the requested child role
- `UpgradeCanister` may use a structural proof only for a registered direct
  child of the caller
- `RecycleCanister` may use a structural proof only for a registered direct
  child of the caller

`RpcOps::execute_response_rpc` must not call
`canic_request_role_attestation` for those structural placement paths. This is
what keeps sharding, scaling, and directory child creation working after the
one-shot role-attestation issuer is disabled.

The old ECDSA attestation key set is not the verification authority after this
follow-up hard cut. Verifiers use the configured root principal plus the IC
root key.

## 14. Root ECDSA Removal Targets

Remove or isolate from normal delegated-token auth in this pass:

- the old prepared delegation-cert root ECDSA signing step
- root delegated ECDSA public-key publication for root proof verification
- any delegated-token root verifier read of
  `SubnetState::delegated_root_public_key`
- replay effect descriptors that classify root auth proof issuance as
  threshold ECDSA

Additional removal targets before claiming zero root threshold-ECDSA signing
across all normal auth:

- root role-attestation `EcdsaOps::sign_bytes`
- root internal-invocation-proof `EcdsaOps::sign_bytes`
- delegated-grant capability proof root public-key trust state, unless that
  capability mode is explicitly retained as a non-delegated-token protocol

Keep `EcdsaOps` for:

- shard token signatures, unless a separate hard cut replaces `shard_sig`
- external-chain interoperability
- explicit non-auth uses

## 16. Shard Signing Cost Model

0.65 fixes root-proof issuance cost only.

If shards mint `DelegatedToken.shard_sig` using threshold ECDSA for every
request, the system still pays threshold ECDSA costs at shard token issuance.
The expected 0.65 usage model is that root `DelegationProof`s and shard
`DelegatedToken`s are reused for their TTL, not minted per endpoint request.

Required metrics:

- `root_threshold_ecdsa_sign_count` must remain zero for normal auth root proof
  issuance. 0.65 also hard-fails one-shot role-attestation and
  internal-invocation proof issuance before ECDSA signing.
- `shard_threshold_ecdsa_sign_count`
- `delegated_tokens_minted`
- `delegated_token_verifications`
- `delegated_token_cache_hits`
- `tokens_minted_per_request`

Benchmarks:

- `delegated_token_candid_encode`
- `delegated_token_candid_decode`
- `delegated_token_encoded_size_bytes`
- `root_canister_sig_verify_invalid_certificate`

The canister-signature verification benchmark uses the exact root verifier
message shape and a structured invalid certificate fixture. A future positive
fixture with a real IC certificate can extend it without changing the 0.65
contract.

### Replay And Cost Classes

Do not classify root canister-signature preparation as threshold ECDSA.

0.65 cost classes:

```rust
pub enum CostClass {
    ThresholdEcdsaSign,
    RootCanisterSignaturePrepare,
    ShardTokenSign,
}
```

Use `RootCanisterSignaturePrepare` for `prepare_delegation_proof`. This class
protects signature-map size and certified-data churn, not a 26B-cycle signing
effect.

Use `ShardTokenSign` for shard token issuance while
`DelegatedToken.shard_sig` still uses threshold ECDSA. This keeps shard token
signing visible as the remaining delegated-token ECDSA cost center.

Suggested quotas:

- `root_canister_signature_prepare_per_principal_per_window`
- `root_canister_signature_prepare_global_per_window`
- `shard_token_sign_per_principal_per_window`
- `shard_token_sign_global_per_window`

## 17. Gotchas

### Update/Query Split

Canister signatures need an update to commit certified data and a query to
retrieve the certificate and witness. Do not hide this behind a `sign_bytes()`
shaped API.

### Self-Contained Token Contract

Verifier-local caches are optimizations only. They must not become required
proof state for delegated-token auth.

### Certified Data Is Global

Only the certified-data owner may call `set_certified_data`.

### Domain And Message Must Match

Root and verifier must use one helper for seed, domain, and message. Golden
tests must catch a domain mismatch.

### Retrieval TTL Is Not Auth TTL

Signature-map retention only controls how long a prepared proof can be
retrieved. Auth validity comes from the signed payload fields.

### Query Results Are Not Trusted By Themselves

The query returns a certificate and witness. Consumers must verify the
canister signature before trusting the payload.

### Upgrades Invalidate Outstanding Tickets

Prepared-but-not-retrieved signatures are retryable. They are not stable
commitments across root upgrade unless root explicitly persists enough
signature-map state to preserve them.

## 19. Hard-Cut Rollout

0.65 has no legacy root ECDSA verifier branch, so deployment order matters.

Hard-cut rollout:

1. Stop issuing old threshold-ECDSA root-proof tokens or reduce their TTL.
2. Deploy root and shard issuers capable of hard-cut issuance.
3. Switch clients to prepare/update, get/query, then issue-token/update.
4. Wait the old max token TTL, or explicitly accept immediate invalidation.
5. Upgrade endpoint verifiers to reject the old `root_sig` proof shape.
6. Remove normal delegated-token root ECDSA config. Keep any remaining root
   ECDSA auth surfaces explicitly listed until their follow-up conversion or
   removal lands.

Verifier upgrade before issuer and client upgrade breaks outstanding old-shape
delegated tokens by design.

## 21. Final Target State

Root:

- canister-signs typed Canic auth payloads
- never signs arbitrary caller bytes
- never calls `sign_with_ecdsa` for normal root auth
- owns `certified_data` through one module

Delegated tokens:

- remain self-contained
- still carry claims, root proof, and shard signature
- still verify locally at endpoints
- are reusable across multiple update calls until TTL/scope/audience limits
  reject them

Verifiers:

- validate the embedded root canister signature
- validate the embedded shard signature
- bind token subject to IC caller
- authorize by local role, audience, scope, and expiry

Legacy:

- threshold-ECDSA root proofs are not accepted in 0.65
