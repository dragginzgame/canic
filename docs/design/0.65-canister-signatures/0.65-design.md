I’ll write this as an implementation-grade Canic design doc focused on returning to canister signatures for root authority proofs, while keeping hot-path authentication local to shards/verifiers.

The key implementation detail is the two-phase signature flow: root commits the signature map in an update, then serves the certificate and witness from a query.

Thought for 7m 35s
Canic Auth v0.65 Design Doc: Return Root Auth to IC Canister Signatures
1. Decision

Replace the current hot-path root sign_with_ecdsa authentication proof with IC canister signatures.

The new design is:

Root canister-signs medium-lived auth/topology/grant certificates.
Shards and target canisters verify those proofs locally and cache the result.
Hot-path request auth uses IC caller principals plus cached grants.
No management-canister threshold ECDSA call in normal auth.

This keeps the important security property that root does not store a private signing key in canister memory, while removing the per-signature threshold ECDSA cost. The IC canister-signature scheme lets a canister “sign” by committing to a certified variable; the public key encodes the signing canister ID plus a seed, and the signature is a CBOR object containing an IC certificate and hash-tree witness. The IC spec verifies the path:

["sig", sha256(seed), sha256(payload)] = ""

inside the certified tree.

The current threshold ECDSA path should remain available only for migration, external-chain interoperability, or rare anchoring. Current ICP cycle-cost docs list production sign_with_ecdsa / sign_with_schnorr at 26,153,846,153 cycles per signature for key_1, charged per signature.

2. Problem

The current auth path effectively does this:

AuthApi::issue_delegation_proof
-> AuthOps::prepare_delegation_proof
-> AuthOps::sign_prepared_delegation_proof
-> EcdsaOps::sign_bytes
-> management canister sign_with_ecdsa

That makes root auth scale with threshold ECDSA calls. This is the wrong cost model for per-request or high-frequency auth. Even if verification is local, minting the auth proof is expensive.

The original design goal was correct:

Root should be the Canic authority.
Other canisters should verify root-approved auth locally.
Root should not be called on every request.
Root should not hold a normal private key in Wasm memory.

The mistake was replacing canister signatures with threshold ECDSA and treating threshold ECDSA like a cheap signing primitive. It is not.

3. Goals
Primary goals
Remove sign_with_ecdsa from normal Canic authentication.
Keep root as the canonical auth/topology authority.
Support authentication across multiple subnets.
Avoid storing a root private key inside root canister memory.
Preserve local verification by shards and target canisters.
Make the implementation robust enough that canister signatures do not fail silently or mysteriously again.
Secondary goals
Keep a migration bridge for existing ECDSA proofs.
Make proof format versioned and explicit.
Cache full root-proof verification results.
Keep the hot path cheap: caller principal checks, local authorization maps, short-lived cached grants.
Fix the current algorithm naming mismatch: if legacy ECDSA remains, call it ThresholdEcdsaSecp256k1, not EcdsaP256Sha256.
4. Non-goals
Do not use root canister signatures for every endpoint request.
Do not create a generic “root signs arbitrary bytes” API.
Do not replace threshold ECDSA with a root-local Ed25519/secp256k1 private key.
Do not require all shards to call root on every request.
Do not rely on uncertified query data for auth.
5. IC canister-signature model

A canister signature is not ECDSA and not Ed25519.

The signing canister commits to a certified data tree. The public key is a DER-wrapped structure containing:

signing_canister_id
seed

The signature is CBOR containing:

certificate
tree witness

Verification checks:

1. The IC certificate is valid against the IC root key.
2. The certificate contains the signing canister's certified_data.
3. The signature witness reconstructs that certified_data.
4. The witness contains ["sig", sha256(seed), sha256(payload)] = "".
5. The signing canister ID is covered by the certificate/subnet delegation.

The IC interface spec also notes that canister signatures from cloud_engine subnets are not valid, so Canic root must not rely on a root auth canister deployed there.

Implementation-wise, creating a canister signature is a two-step flow:

1. update call: prepare signature and set certified_data
2. query call: retrieve certificate + witness

The DFINITY ic-canister-sig-creation crate README describes exactly this update-then-query flow and uses a SignatureMap to bridge the two phases.

The key shift is:

Old:
  Root signs per delegated token with threshold ECDSA.

New:
  Root canister-signs auth certificates.
  Shards/targets cache verified certificates.
  Requests authorize against cache + msg.caller().

Cross-subnet support is preserved because IC principals are global. A caller principal does not change because the callee lives on a different subnet. The canister-signature proof is also subnet-portable because the certificate chain verifies back to the IC root key.

8. Proof format

Replace the current ECDSA-specific proof shape:

pub struct DelegationProof {
    pub cert: DelegationCert,
    pub root_sig: Vec<u8>,
}

with:

pub struct RootCertified<T> {
    pub payload: T,
    pub proof: RootProof,
}

pub enum RootProof {
    IcCanisterSignatureV1(IcCanisterSignatureProof),

    // Migration only.
    ThresholdEcdsaSecp256k1V1(ThresholdEcdsaProof),
}

pub struct IcCanisterSignatureProof {
    pub root_canister_id: Principal,

    // Purpose-specific seed. Example:
    // b"canic-root-role-grant-v1"
    // b"canic-root-shard-authority-v1"
    pub seed: Vec<u8>,

    // Include for debugging and compatibility, but verifier must recompute
    // and require equality.
    pub public_key_der: Vec<u8>,

    // CBOR canister signature: certificate + tree witness.
    pub signature_cbor: Vec<u8>,

    // Key id = sha256("canic-canister-sig-pk-v1" || public_key_der)
    pub kid: [u8; 32],
}

Do not let the proof decide its own semantic domain. The verifier must derive the expected domain from the payload type.

Example:

fn expected_domain(kind: CertKind) -> &'static [u8] {
    match kind {
        CertKind::ShardAuthority => b"canic-root-shard-authority-v1",
        CertKind::RoleGrant => b"canic-root-role-grant-v1",
        CertKind::TopologyEpoch => b"canic-root-topology-epoch-v1",
    }
}
9. Signing payload construction

Use one canonical encoding path everywhere.

pub fn canonical_root_payload<T: Serialize>(payload: &T) -> Vec<u8> {
    // Use existing Canic canonicalization if it is already stable.
    // Otherwise use deterministic CBOR with:
    // - fixed field ordering
    // - no unordered maps unless sorted
    // - explicit version
    // - explicit cert_kind
    // - explicit root_pid
    // - explicit topology_epoch
}

For canister signatures, the DFINITY helper applies domain separation as:

sha256( domain_len || domain || message )

The crate’s CanisterSigInputs::message_hash() calls hash_with_domain(domain, message), and hash_with_domain prefixes the domain length before hashing.

So Canic should standardize this helper:

pub fn canic_canister_sig_message(domain: &[u8], payload: &[u8]) -> Vec<u8> {
    assert!(domain.len() <= 255);
    let mut out = Vec::with_capacity(1 + domain.len() + payload.len());
    out.push(domain.len() as u8);
    out.extend_from_slice(domain);
    out.extend_from_slice(payload);
    out
}

Root preparation uses:

let sig_inputs = CanisterSigInputs {
    domain: expected_domain(payload.cert_kind),
    seed: seed_for(payload.cert_kind),
    message: &canonical_payload,
};

Verifier uses:

let message = canic_canister_sig_message(
    expected_domain(payload.cert_kind),
    &canonical_payload,
);

verify_canister_sig(
    &message,
    &proof.signature_cbor,
    &proof.public_key_der,
    &ic_root_public_key_raw,
)?;

This avoids the common mismatch where root signs hash_with_domain(domain, payload) but the verifier checks sha256(payload).

11. Certified-data owner module

One likely reason this was hard before: a canister has only one 32-byte certified_data value. The System API lets a canister set certified data up to 32 bytes, and query calls can retrieve a certificate for the last value set.

So Canic root needs one module that owns certified data.

pub struct CertifiedDataRoot {
    pub sig_root: [u8; 32],
    pub canic_state_root: Option<[u8; 32]>,
    pub http_assets_root: Option<[u8; 32]>,
}

Commit shape:

root
├── "sig"          -> SignatureMap root
├── "canic_state" -> root topology/state root, optional
└── "http_assets" -> HTTP asset cert root, optional

If root only uses signatures:

set_certified_data(&labeled_hash(b"sig", &SIGNATURES.root_hash()));

If root also certifies other data:

set_certified_data(&combined_certified_root_hash(
    sig_root,
    canic_state_root,
    http_assets_root,
));

Never allow unrelated modules to call set_certified_data directly. They must update their subtree and ask the certified-data owner to recompute the combined root.

13. Auth flows
13.1 User obtains a role grant
1. User calls root.prepare_role_grant(...)
   - update call
   - root checks caller and policy
   - root creates RoleGrantCert
   - root adds canister-signature entry
   - root sets certified_data
   - root returns payload + retrieval ticket

2. User calls root.get_role_grant_signature(...)
   - query call
   - root returns signature_cbor + payload

3. User calls shard/target with RootCertified<RoleGrantCert>

4. Shard/target verifies once and caches

5. Later user calls shard/target normally
   - msg.caller() is the user's principal
   - shard/target authorizes from cache
13.2 Root authorizes shard
1. Root or admin prepares ShardAuthorityCert.
2. Signature is retrieved by query.
3. Shard stores its RootCertified<ShardAuthorityCert>.
4. Target canisters verify/cache the shard authority.
5. Shard-to-target calls use caller principal == shard principal.

No token is needed for the infrastructure caller identity. The IC already authenticates the caller principal. The root proof only says:

this shard principal is authorized by Canic root for these audiences/scopes until this expiry
13.3 Multi-subnet call
Shard A on subnet X calls Target B on subnet Y.
Target B sees caller == Shard A principal.
Target B checks cached ShardAuthorityCert signed by root.
Target B checks method/scope/audience.

No management canister call. No root call. No subnet-local identity problem.

14. Migration plan
Phase 1: introduce proof enum

Add:

enum RootProof {
    IcCanisterSignatureV1(IcCanisterSignatureProof),
    ThresholdEcdsaSecp256k1V1(ThresholdEcdsaProof),
}

Keep old ECDSA verification working.

Phase 2: implement root signature map

Add root module:

root_auth_canister_sig.rs

Responsibilities:

- build seed
- build public_key_der
- add_signature
- recompute certified_data
- retrieve signature_cbor
- reset certified_data on init/post_upgrade

Use the DFINITY ic-canister-sig-creation crate unless there is a strong reason to own the Merkle implementation. Its README explicitly covers preparing a signature, updating certified data, and retrieving the CBOR signature.

Phase 3: add two-step root APIs

Do not try to make the existing update method return the canister signature. It cannot reliably do that because the certificate is available from query execution, not from the update execution path. The System API says data_certificate_present() is available for query/composite-query methods on the target canister of a query call, and returns 0 for update methods or replicated query execution.

Add:

prepare_role_grant
get_role_grant_signature
prepare_shard_authority
get_shard_authority_signature
prepare_topology_epoch
get_topology_epoch_signature
Phase 4: update verifiers

Verifier support order:

1. Accept canister signatures.
2. Accept legacy ECDSA only if feature flag / migration window enabled.
3. Prefer canister signatures when both are present.
4. Emit metrics on legacy ECDSA proof usage.
Phase 5: stop issuing ECDSA auth proofs

Root config:

auth.issue_threshold_ecdsa_proofs = false;
auth.accept_legacy_threshold_ecdsa_until_ns = ...
Phase 6: remove auth hot-path ECDSA

Delete or isolate:

AuthOps::sign_prepared_delegation_proof -> EcdsaOps::sign_bytes

Replace with:

AuthOps::prepare_root_cert_signature
AuthOps::get_root_cert_canister_signature

Keep EcdsaOps only for external-chain or explicit legacy modules.

15. Important gotchas from the previous failed attempt
Gotcha 1: update/query split

Canister signatures require:

update: prepare and set certified_data
query: retrieve data_certificate and witness

Trying to return the signature from the update call is the wrong model.

Gotcha 2: certified_data is global per canister

If another root module calls set_certified_data, it can overwrite the signature map root. Centralize certified-data ownership.

Gotcha 3: exact tree shape matters

The signature tree must be under:

"sig" -> sha256(seed) -> sha256(message) -> ""

If you certify the raw map root instead of labeled("sig", map_root), verification fails.

Gotcha 4: domain mismatch

The DFINITY creation helper hashes:

domain_len || domain || message

Generic verification hashes whatever message bytes you pass. So Canic’s verification helper must pass the same domain-separated bytes.

Gotcha 5: retrieval retention is not auth TTL

SignatureMap may keep prepared signatures only briefly for retrieval. That is not the same as the Canic cert’s expires_at_ns.

The signed certificate itself must contain:

issued_at_ns
not_before_ns
expires_at_ns
topology_epoch
audience
caller binding

Verifier policy enforces these fields.

Gotcha 6: query results are not trusted unless verified

A query call can be fast but must be authenticated with the returned certificate and witness. Do not trust the root query response just because it came from root; verify the canister signature.

Gotcha 7: no cloud_engine root auth signer

The IC spec/changelog says canister signatures from cloud_engine subnets are not valid. Keep Canic root auth signing on a subnet whose canister signatures are valid.

Gotcha 8: upgrades between prepare and retrieve

If root upgrades after prepare_* but before get_*_signature, the in-memory SignatureMap may be gone. Treat prepared signatures as short-lived and retryable. On post_upgrade, explicitly reset the certified signature root and invalidate outstanding tickets.

Gotcha 9: do not make root a signing oracle

Root must only sign typed, policy-checked Canic certificates. No API like:

prepare_signature(bytes: Vec<u8>)

unless it is private and unreachable by users.

16. Security model
Preserved
Root does not store a private signing key.
Proofs are portable across subnets.
Verifiers do not need to call root during hot-path auth.
Verifiers can check proof locally against the IC root key.
Root authorization is bound to audience, subject, caller, scope, and expiry.
Changed from threshold ECDSA
Proofs are larger.
Proof retrieval is update + query instead of one update.
Verification is IC-specific, not generic ECDSA.
Full proof verification is heavier than checking a 64-byte ECDSA signature.

This is why the design caches verified grants and uses caller principals on the hot path.

Not acceptable
Root-local private key as the main root auth key.
Per-request root canister signatures.
Per-request threshold ECDSA.
Uncertified query auth.
Generic root signing endpoint.
17. Cost and performance model
Root proof issuance
Old:
  one threshold ECDSA management call per proof
  ~26.15B cycles/signature for production key_1 today

New:
  one normal root update call
  one root query call returning certificate + witness
  no threshold ECDSA signing fee
Verification

Full canister-signature verification involves CBOR parsing, hash-tree checks, and BLS certificate verification. It is fine for cache fill, topology updates, and grant ingestion. It should not be the inner-loop authorization check for every endpoint.

Hot path

Hot path should be:

msg.caller()
+ local grant cache lookup
+ expiry/audience/scope check

That is the part that should scale linearly with requests.

18. Minimal code skeleton
18.1 Root signature module
use ic_canister_sig_creation::{
    CanisterSigPublicKey,
    signature_map::{CanisterSigInputs, SignatureMap, LABEL_SIG},
};
use ic_cdk::api::set_certified_data;
use std::cell::RefCell;

thread_local! {
    static SIGNATURES: RefCell<SignatureMap> =
        RefCell::new(SignatureMap::default());
}

pub fn seed_for(kind: CertKind) -> &'static [u8] {
    match kind {
        CertKind::RoleGrant => b"canic-root-role-grant-v1",
        CertKind::ShardAuthority => b"canic-root-shard-authority-v1",
        CertKind::TopologyEpoch => b"canic-root-topology-epoch-v1",
    }
}

pub fn public_key_der(root: Principal, kind: CertKind) -> Vec<u8> {
    CanisterSigPublicKey::new(root, seed_for(kind).to_vec()).to_der()
}

pub fn prepare_signature(kind: CertKind, canonical_payload: &[u8]) {
    let seed = seed_for(kind);
    let domain = expected_domain(kind);

    let inputs = CanisterSigInputs {
        domain,
        seed,
        message: canonical_payload,
    };

    SIGNATURES.with_borrow_mut(|sigs| {
        sigs.add_signature(&inputs);
    });

    refresh_certified_data();
}

pub fn get_signature_cbor(
    kind: CertKind,
    canonical_payload: &[u8],
) -> Result<Vec<u8>, AuthError> {
    let inputs = CanisterSigInputs {
        domain: expected_domain(kind),
        seed: seed_for(kind),
        message: canonical_payload,
    };

    SIGNATURES.with_borrow(|sigs| {
        sigs.get_signature_as_cbor(&inputs, None)
            .map_err(AuthError::from)
    })
}

fn refresh_certified_data() {
    SIGNATURES.with_borrow(|sigs| {
        let sig_root = sigs.root_hash();

        // Must be labeled as "sig".
        let certified_root = labeled_hash(LABEL_SIG, &sig_root);

        set_certified_data(&certified_root);
    });
}

If root also certifies assets or Canic state, replace labeled_hash(LABEL_SIG, ...) with the combined certified-data tree described above.

18.2 Verifier
use ic_signature_verification::verify_canister_sig;

pub fn verify_ic_canister_root_proof<T: CanicRootPayload>(
    signed: &RootCertified<T>,
    expected_root: Principal,
    ic_root_public_key_raw: &[u8],
) -> Result<(), AuthVerifyError> {
    let RootProof::IcCanisterSignatureV1(proof) = &signed.proof else {
        return Err(AuthVerifyError::WrongProofScheme);
    };

    let kind = signed.payload.cert_kind();
    let seed = seed_for(kind);

    if proof.root_canister_id != expected_root {
        return Err(AuthVerifyError::WrongRoot);
    }

    if proof.seed != seed {
        return Err(AuthVerifyError::WrongSeed);
    }

    let expected_pk =
        CanisterSigPublicKey::new(expected_root, seed.to_vec()).to_der();

    if proof.public_key_der != expected_pk {
        return Err(AuthVerifyError::WrongPublicKey);
    }

    let canonical = canonical_root_payload(&signed.payload);
    let message = canic_canister_sig_message(expected_domain(kind), &canonical);

    verify_canister_sig(
        &message,
        &proof.signature_cbor,
        &proof.public_key_der,
        ic_root_public_key_raw,
    ).map_err(AuthVerifyError::InvalidCanisterSignature)?;

    Ok(())
}

  signs Canic auth certificates with IC canister signatures
  never signs arbitrary bytes

Shards:

  cache grants/authority
  authorize requests with caller principal + cache
