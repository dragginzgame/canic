# Canic 0.102 Design: Compact Diagnostic Codes

Date: 2026-07-23

## Status

- Status: proposed for maintainer approval.
- Scope: Canic public errors, internal diagnostic identity, host and CLI
  rendering, and diagnostic-derived durable state.
- Sequence: this is the 0.102 design line. It follows the
  [0.99 terminology hard cut](../0.99-app-fleet-terminology/0.99-design.md) and
  [0.100 multi-subnet design](../0.100-multi-subnet/0.100-design.md), but has no
  semantic dependency on multi-subnet synchronization. The intervening 0.101
  line is reserved for Fleet-authoritative service adoption, provisioning,
  ownership transfer, placement and Directory publication.
- Compatibility posture: pre-1.0 hard cut. The current error contract is
  replaced in place; there is no parallel error generation, alias, shim,
  fallback decoder or legacy message field.
- Protocol posture: there is one current diagnostic protocol. Types and fields
  must not use `V1`, `V2`, `version`, `schema_version` or equivalent
  generation names.
- Footprint posture: the wire carries only one `nat16` code. Host-only catalog
  data recovers origin and meaning; a masked failure retains a separate
  operator-visible numeric internal code.
- Repository scope: Canic only. Required downstream migrations are documented,
  not implemented in sibling repositories.
- Measurement posture: the design is motivated by a credible Wasm-retention
  problem, but implementation must establish a fresh Canic baseline before
  claiming a size win.

## Decision

Canic will hard-cut its public and internal error machinery to use compact,
numeric diagnostic identities in canister-reachable code. Rich labels,
messages and remediation text will live in host-only code.

The public Candid error becomes:

~~~rust
pub struct Error {
    pub code: u16,
}
~~~

This directly replaces the current public shape:

~~~rust
pub struct Error {
    pub code: ErrorCode,
    pub message: String,
}
~~~

The current Candid `ErrorCode` variant and `message : text` field are removed.
Every maintained endpoint returns the new numeric record immediately. No
endpoint, decoder or release may support both shapes.

Canister code retains typed errors where policy or workflow decisions require
them, but every failure that crosses a layer boundary has one stable numeric
diagnostic identity. Formatting that identity inside a canister is compact and
numeric. The unique code identifies its class and origin in the host catalog.
Operator-facing origin, prose and remediation are reconstructed by
`canic-host` and surfaced by the CLI.

The goal is not to remove every string from Canic Wasm. It is to remove static
diagnostic prose, enum-label formatting and redundant failure text while
preserving strings that are part of a real protocol, cryptographic domain,
metric, application log or operator-owned data contract.

## Why 0.102

The current public error contract carries both a Candid variant label and an
owned message. Internally, `InternalError` carries another owned message,
supports appended string context and often receives errors through
`to_string()`. Separately, many canister-reachable error enums derive prose-rich
`thiserror::Error` and `Debug` implementations.

That shape has four costs:

1. Static error messages and formatting fragments can be retained in every
   Wasm role that reaches their constructors.
2. Variant names can be retained through derived `Debug`, Candid schema and
   formatting code.
3. The public `message` encourages callers and tests to depend on prose even
   though prose is not a stable machine contract.
4. Error context is flattened to strings, losing the typed identity needed for
   reliable automation and host-side rendering.

A sibling implementation, IcyDB, has shown that compact numeric public errors,
numeric debug formatting and host-owned rendering can remove meaningful Wasm
weight. Its individual cuts ranged from low single-digit KiB to roughly
20–27 KiB of raw Wasm per affected canister, with about 47.5 KiB cumulative in
one compacting sequence. Those measurements demonstrate feasibility, not an
expected Canic result: the linked code, profiles, error fan-out and baseline
are different.

The decision is therefore architectural first and size-sensitive second:

- numeric diagnostics are a better machine contract;
- host-owned prose prevents canister presentation concerns from spreading
  through runtime crates; and
- the implementation proceeds in measured slices so the actual Canic result
  is known.

## Current Canic Baseline

The design inventory was taken against immutable tag `v0.98.14`. It is a
starting point, not the implementation acceptance baseline. The inventory must
be refreshed after any intervening 0.99, 0.100 or 0.101 work.

At that tag:

- `canic_core::dto::error::Error` contains `ErrorCode` plus `String`;
- `ErrorCode` has 25 Candid and Serde variant leaves;
- public `Display` prints the enum through `Debug` and then the message;
- `InternalError` contains class, origin, `String` and an optional complete
  public error;
- `with_diagnostic_context` appends formatted prose;
- conversions frequently classify an error after calling `to_string()`;
- the canonical Wasm-store Candid declares `message : text` and the full
  `ErrorCode` variant;
- approximately 59 generated facade endpoint signatures return
  `::canic::Error`;
- the host response decoder renders `CanicError` through its current
  message-bearing `Display`; and
- CLI inspection, authentication, blob-storage, metrics and cycle paths
  consume that host decoder.

An indicative source scan also found hundreds of internal error constructions
and `#[error("...")]` declarations in canister-reachable crates. Those counts
must not be reported as removable Wasm bytes. Linker reachability, generic
instantiation, profile settings and role fan-out determine what survives.

The latest retained authoritative Canic Wasm audit predates this design and is
not a current comparison baseline. Implementation must use the current
[Wasm footprint audit method](../../audits/recurring/system/wasm-footprint.md)
or its approved successor.

## Goals

0.102 must:

1. Establish one stable numeric identity for every maintained diagnostic leaf.
2. Remove public diagnostic text and Candid error variant labels from canister
   error responses.
3. Make internal diagnostic propagation code-first rather than string-first.
4. Keep rich labels, summaries and remediation guidance in host-only code.
5. Preserve typed distinctions needed by policy, workflow and automation.
6. Keep unknown future codes observable as their lossless raw number without
   inventing fallback behavior.
7. Audit diagnostic-derived durable strings and remove or type them only where
   stable-state safety is proved.
8. Measure raw Wasm, gzip Wasm and retained sections before and after each
   material slice.
9. Leave no dual protocol or pre-1.0 compatibility layer.
10. Produce an explicit downstream migration guide for the one current
    contract.

## Non-Goals

0.102 does not:

- strip all strings from Canic Wasm;
- replace application logs with numeric codes;
- remove IC method names, Candid field names, role names, metric labels or
  cryptographic domain-separation strings;
- define an application-wide localization framework;
- make a generic structured payload for arbitrary error details;
- add telemetry transport or a remote diagnostic catalog;
- preserve callers that match the current Candid `ErrorCode` variant;
- preserve callers that read or parse `Error.message`;
- recover codes by matching historical error text;
- redesign successful endpoint responses;
- change authentication boundaries; or
- modify downstream repositories.

## Naming and Versioning

The maintained vocabulary is:

- `DiagnosticCode`: the numeric leaf identity;
- `DiagnosticClass`: a host-only broad failure classification;
- `DiagnosticOrigin`: a host-only producing layer or subsystem identity; and
- `Error`: the public Candid error record.

`DiagnosticClass` and `DiagnosticOrigin` are native host catalog types. They
are not Candid types, runtime error fields or canister-side lookup tables.

There is no `DiagnosticV1`, `ErrorV2`, `LegacyError`, `schema_version` or
versioned endpoint. The 0.102 number identifies this Canic design line; it is
not part of the runtime protocol.

Code numbers are identities, not protocol generations. A retired number is
never reassigned to a different meaning, but its constructor, parser, catalog
entry, message and behavior are removed. If an old raw number is encountered
later, the current host treats it as unknown.

No range is reserved speculatively. Allocate codes only for diagnostics with a
current producer or for a code introduced in the same coherent implementation
slice.

## Public Error Contract

The Candid wire shape is exactly:

~~~candid
type Error = record {
  code : nat16;
};
~~~

The corresponding Rust field remains a plain numeric wire value. A typed
accessor may convert it to `DiagnosticCode` without changing the wire shape.

### Field Semantics

`code` is the complete public identity. It must be specific enough for the
host catalog to recover:

- the producing layer or subsystem;
- the broad failure class;
- the symbolic label;
- the operator summary and action; and
- the intended machine behavior.

Class and origin are not duplicated on the wire or in the canister-side error.
That redundancy would spend Wasm and payload bytes without adding information
for a known code.

An unknown code remains observable as its raw `u16`, but its origin cannot be
guessed. Diagnosis requires a host catalog from the same maintained Canic
surface. Canic does not add version negotiation, reserved numeric ranges or
fallback metadata to make an unknown code self-describing.

### Public Rust Surface

`canic::Error` remains the facade return type so endpoints do not acquire a
replacement-generation name. Its fields change in place. Convenience methods
may include:

~~~rust
impl Error {
    pub const fn diagnostic_code(&self) -> DiagnosticCode;
}
~~~

The canister-side `Display` and `Debug` forms, if retained, are compact numeric
forms:

~~~text
E1203
~~~

The exact punctuation is chosen once and tested. It must not include the Rust
constant name, class label, origin label, summary or remediation prose.

## Diagnostic Registry

`canic-core` owns only the prose-free runtime identity:

- the `DiagnosticCode(u16)` newtype;
- explicit numeric code constants;
- compact numeric formatting.

Production Wasm must not contain:

- a complete iterable code list;
- code-to-class, code-to-origin or code-to-prose metadata tables;
- public exposure tables;
- human-readable labels;
- complete English messages;
- remediation hints;
- formatted invariant prose;
- documentation copied into compiled string constants; or
- a feature-gated rich renderer.

Feature-gating prose inside `canic-core` is rejected because Cargo feature
unification can make host-selected features reachable in canister builds. The
physical crate boundary is the enforcement mechanism.

An illustrative source shape is:

~~~rust
pub const INPUT_CANISTER_ID_MALFORMED: DiagnosticCode =
    DiagnosticCode::new(101);

#[cfg(not(target_arch = "wasm32"))]
pub const ALL_DIAGNOSTIC_CODES: &[DiagnosticCode] = &[
    INPUT_CANISTER_ID_MALFORMED,
    // Every currently produced public or operator-visible code exactly once.
];
~~~

The implementation may use a local macro to declare numeric constants once and
emit a native-only complete list for coverage tests. The macro must not accept
prose, class or origin arguments. Target gating a numeric test inventory is
permitted; feature-gating a rich renderer in `canic-core` is not.

The resulting Wasm must contain only constants reached by that role. The host
catalog is checked against the native-only complete list so every current code,
including internal codes emitted to operator diagnostics, remains resolvable.

### Registry Invariants

The registry must prove:

1. Every allocated code is nonzero and unique.
2. Every current public producer maps to one registered code.
3. Every internal diagnostic emitted to operator logs is registered.
4. Every registered code has exactly one host class, origin and rendering
   entry.
5. Every internal-only code has an explicit public projection at its runtime
   constructor or boundary conversion.
6. No complete registry or metadata table is retained in release Wasm.
7. No string comparison participates in classification.
8. Unknown raw codes remain round-trippable.
9. Removed codes are not reassigned.
10. There are no placeholder, reserved or padding leaves.

Contiguous numbering is not required. Ownership-based numeric bands may be
used only when they reflect present subsystem ownership and do not reserve
unused leaves.

## Diagnostic Granularity

The current 25 public `ErrorCode` leaves are too broad to reconstruct useful
operator guidance on their own. `InvalidInput`, `Conflict`, `Unavailable` and
`Internal` are classes in practice, not sufficient diagnostic leaves.

0.102 therefore allocates leaf codes at the point where any of these differs:

- caller action;
- retry behavior;
- security exposure;
- owning subsystem;
- operational remediation; or
- machine decision.

Two sites may share a code only when they have the same origin, mean the same
thing and require the same response. Two differently worded messages do not
require different codes when their origin, semantics and action are identical.

Conversely, a generic string slot must not be used to avoid allocating a leaf.
The following is intentionally excluded:

~~~rust
pub struct Error {
    pub code: u16,
    pub detail: String,
}
~~~

When a meaningful value such as an operation ID, principal, limit or retry
time must cross a boundary, it belongs in an existing typed response, view,
receipt or status record. It is not appended to a generic error message.

## Internal Error Model

`InternalError` becomes code-first. Its authoritative state is diagnostic
identity and exposure policy, not a formatted message.

An illustrative shape is:

~~~rust
pub struct InternalError {
    code: DiagnosticCode,
    public_code: DiagnosticCode,
}
~~~

An ordinary public diagnostic stores the same code in both fields. A sensitive
internal diagnostic stores its exact operator code plus a safe public code.
This small error-path value avoids a global runtime metadata or exposure table.

The final names may follow existing model conventions, but the invariants are
fixed:

- there is no owned `message: String`;
- there is no optional complete public `Error`;
- public projection is deterministic and typed;
- sensitive internal identities can map to a safe public leaf;
- the internal code remains available for numeric operator diagnostics when a
  public projection masks it;
- class and origin are not stored in the runtime error; and
- formatting is never used to recover identity.

### Conversion Rules

Conversions from typed internal errors must match typed variants or fields.
They must not call `to_string()` and then classify the result.

`with_diagnostic_context` is removed. A caller that currently appends prose
must do one of the following:

1. preserve the primary diagnostic unchanged;
2. map it to a more specific registered leaf;
3. retain a typed secondary cause for internal control flow; or
4. write dynamic context to an approved structured status or log owner.

There is no generic vector of causes on the public wire and no recursively
nested diagnostic tree. Those shapes would add protocol and allocation cost
without a demonstrated consumer.

### Retained Typed Errors

This design does not flatten every internal error enum into `u16`.

Typed variants remain appropriate where code makes a decision such as:

- retry versus stop;
- absent versus forbidden;
- stale versus expired;
- conflict versus invariant failure; or
- recoverable platform rejection versus corrupt state.

At the layer boundary, those variants map exhaustively to compact diagnostics.
Derived `thiserror::Error` or `Debug` prose is removed when it remains
canister-reachable solely for formatting. Manual numeric formatting is used
where a formatting trait is actually required.

## Public Exposure and Security

Not every internal diagnostic is safe to expose.

Each runtime constructor or typed boundary mapping makes one explicit choice:

- public as-is;
- public through a less-specific safe code; or
- internal-only, with a deterministic safe public projection.

Projection must not leak:

- secret or authentication material;
- cryptographic intermediate values;
- private topology;
- stable-state contents;
- raw platform reject text;
- file or build paths; or
- implementation details that weaken authorization boundaries.

Authentication remains enforced at endpoints. Workflow and ops continue to
assume authenticated input. Compact diagnostics do not replace those
boundaries.

A mapped public diagnostic exposes only the safe public code. The host catalog
reports the safe code's public origin and does not reveal the sensitive
internal origin by accident.

When the internal and public codes differ, the owning workflow or ops path must
have an approved numeric observability owner for the internal code. That may be
a structured diagnostic log, operation receipt or status record. If no such
owner exists, the implementation must allocate a sufficiently specific safe
public code rather than make the true origin permanently undiagnosable.

The internal observability value is the numeric code, not diagnostic prose.

## Host-Owned Rendering

`canic-host` owns the rich diagnostic catalog. This crate is already shared by
host/operator surfaces and is not linked into canister Wasm.

Every current registered code maps to:

- a stable symbolic label for logs and automation;
- a concise operator-facing summary;
- an actionable hint when one exists;
- its host-only class and origin;
- its Canic source owner when that improves developer triage; and
- optional documentation metadata.

This includes internal codes written to approved operator diagnostics, not only
codes returned publicly. An origin names the narrowest stable owner that makes
the failure actionable, such as a specific authentication workflow or
Wasm-store operation, rather than only the broad `workflow` or `ops` layer.

An illustrative entry is:

~~~rust
DiagnosticEntry {
    code: INPUT_CANISTER_ID_MALFORMED,
    label: "INPUT_CANISTER_ID_MALFORMED",
    summary: "The canister ID is malformed.",
    action: Some("Check the principal supplied by the caller."),
}
~~~

The concrete wording is written during the inventory slice. The example does
not allocate that code or mandate that message.

The central host response decoder renders known codes automatically. A normal
CLI command therefore remains useful to a human even though the canister sent
only one numeric field.

Unknown codes render losslessly:

~~~text
E65000 (unknown diagnostic)
~~~

The host must not fail decoding, relabel an unknown code as a known error or
infer its origin, class or retry behavior.

### CLI Lookup

0.102 adds one current lookup surface:

~~~text
canic diagnostic E1203
canic diagnostic 1203
~~~

It prints the numeric identity, symbolic label, class, origin, summary and
action. If the CLI supports JSON for this command, the maintained shape is:

~~~json
{
  "code": 1203,
  "known": true,
  "label": "INPUT_CANISTER_ID_MALFORMED",
  "class": "invalid_input",
  "origin": "canister_input",
  "summary": "The canister ID is malformed.",
  "action": "Check the principal supplied by the caller."
}
~~~

Unknown lookup keeps `code` and `known: false` and omits unavailable catalog
fields. The implementation must use the repository's current CLI output
conventions rather than add a second envelope format.

## Dynamic and External Text

Static diagnostic prose and dynamic operational data are different ownership
classes.

### Static Prose

Static summaries, labels and hints move to the host catalog. They must not be
duplicated in runtime crates for convenience.

### Dynamic Context

Dynamic context remains only when an explicit owner needs it:

- typed IDs and values remain in typed response or status fields;
- application and operator logs may retain bounded messages;
- external platform rejection text may be retained in an approved diagnostic
  log or status record when it is operationally necessary; and
- no decision may depend on parsing that text.

Raw reject text must not be copied into the compact public `Error`. A public
caller receives only the registered code.

### Strings Explicitly Outside This Cut

The following do not become numeric diagnostics merely because they are
strings:

- user and application log messages;
- Candid method and record-field names;
- metric names and label values;
- role, App, Fleet and Subnet identities;
- cryptographic suite names and domain-separation constants;
- configuration paths and user-provided configuration values;
- HTTP content and headers;
- certified data inputs; and
- protocol strings required by the IC or another current external contract.

Any later proposal to compact one of these owners needs its own evidence and
contract analysis.

## Durable-State Audit

Canic currently persists or projects several failure strings. Examples include:

- runtime bootstrap `last_error`;
- ICP refill `error_message`, alongside a typed refill error code;
- cycle top-up event `error`;
- authentication renewal or batch `failure`;
- pool and blob-operation reason strings; and
- runtime detail or validation strings.

The list is illustrative. Implementation must inventory every stable record,
snapshot, DTO projection and lifecycle restoration path before changing one.

General log text is not diagnostic baggage and remains outside this cut.
Diagnostic-derived durable text is evaluated field by field.

### Required Classification

Every durable failure string must be assigned to exactly one class:

1. **Redundant diagnostic prose.** A typed code is already authoritative. Drop
   the text only after proving the exact stable decode and upgrade path.
2. **Advisory transient context.** The value may be cleared and safely rebuilt,
   retried or re-observed during lifecycle restoration. The owner must document
   that behavior.
3. **Recovery-significant state.** The value affects recovery or decisions. It
   must become a typed state field before prose can be removed.
4. **Owned operational text.** A log or status consumer genuinely needs the
   dynamic text. Keep it bounded and prohibit string-based classification.

### Stable Compatibility Boundary

Stable state currently uses Serde CBOR in relevant storage paths. Removing a
field from a Rust record does not by itself prove that an installed canister
can decode its prior stable bytes.

For every changed stable record, implementation must demonstrate one of:

- old bytes decode directly into the new exact record under the repository's
  canonical serializer;
- lifecycle code intentionally clears or reconstructs an advisory field
  without violating invariants; or
- a separately approved one-time migration is required.

0.102 does not authorize a generic legacy reader, dual-written field, fallback
parser or text-to-code migration. If a hard cut cannot preserve an existing
installed-state invariant, that individual stable-field cut is blocked for an
explicit maintainer decision. The public and internal diagnostic cuts may
still proceed independently.

No implementation may infer a numeric code by matching historical failure
text.

## Candid and Generated Surface Cut

The public cut must update every owned schema and generated surface in one
coherent slice:

- canonical `.did` files;
- generated facade endpoint signatures;
- checked-in test and example declarations;
- host response decoding;
- CLI rendering;
- canic-testing helpers; and
- Canic-owned downstream fixtures.

The old `ErrorCode` Candid variant is deleted, not deprecated. The old
`message` field is deleted, not left empty.

An intermediate local commit may introduce registry plumbing before the
contract cut, but no supported release state may expose dual public shapes or
two diagnostic protocols.

## Interaction With 0.99, 0.100 and 0.101

This design uses the canonical App, Fleet, Subnet and Canister vocabulary from
0.99. Its diagnostic origins must not reintroduce old topology terms.

0.100 may add new multi-subnet errors. If implemented first, those producers
join the 0.102 inventory and receive numeric leaves. If 0.102 is implemented
first, later 0.100 work allocates only codes it actually produces.

0.101 may add authority-service adoption, provisioning and ownership-handoff
errors. Those producers follow the same ordering rule: if present before
0.102, they join its inventory; if implemented later, they allocate only their
current diagnostic leaves.

No diagnostic schema generation, negotiation or compatibility field is added
for this sequencing. Both designs target the one maintained pre-1.0 surface.

## Implementation Slices

Implementation should use the following coherent slices. These are work
boundaries, not patch versions or protocol generations.

### Slice A: Baseline and Registry Inventory

1. Build current canonical release artifacts for representative leaf, root and
   Wasm-store roles.
2. Record raw Wasm, gzip Wasm, data sections, function counts and bounded
   retained-size attribution.
3. Inventory every public constructor, internal conversion and host consumer.
4. Define each code's leaf semantics, host class, host origin and public
   projection.
5. Assign an observability owner to every masked internal code.
6. Inventory durable diagnostic strings and classify their ownership.

This slice changes no public behavior.

### Slice B: Prose-Free Registry and Host Catalog

1. Add the compact code newtype and constants to `canic-core`.
2. Add exhaustive host rendering to `canic-host`.
3. Add CLI lookup.
4. Prove code uniqueness, native inventory coverage and host catalog
   consistency.
5. Prove the complete inventory and host metadata are absent from release
   Wasm.

Until the public cut lands, this is internal scaffolding only. It must not
become a second user-visible diagnostic protocol.

### Slice C: Atomic Public Contract Cut

1. Replace `ErrorCode` and `message` with the single `code` field.
2. Update canonical and generated Candid.
3. Update all facade endpoints, host decoders, CLI callers and test helpers.
4. Remove the old types and formatters completely.
5. Measure every representative role again.

This slice is atomic at the supported repository boundary.

### Slice D: Internal Propagation Cut

1. Make `InternalError` code-first.
2. Remove message concatenation and optional complete public errors.
3. Carry explicit internal and public codes where security projection differs.
4. Give each masked internal code a numeric observability owner.
5. Replace string conversions with exhaustive typed mappings.
6. Compact or remove canister-reachable derived formatting.
7. Measure by subsystem so high-fan-out wins and regressions are attributable.

### Slice E: Durable and Observability Owners

1. Apply the durable-state classification field by field.
2. Keep general logs and required dynamic context.
3. Add exact stable decode and lifecycle evidence for every changed record.
4. Verify that metrics, traces and operator statuses still identify failures
   through typed numeric codes.

### Slice F: Closeout

1. Scan runtime crates and release Wasm for residual diagnostic prose.
2. Attribute retained strings rather than deleting by pattern.
3. Record final size deltas and method identity.
4. Update active documentation and migration notes.
5. Remove temporary inventory tooling unless it is admitted as a recurring
   audit owner.

## Measurement Contract

Raw canonical release Wasm is the primary size metric. Builder-produced gzip is
secondary because compression can hide duplicated prose or amplify unrelated
layout changes.

For every measured role and slice, record:

- immutable source identity;
- canonical builder and profile identity;
- raw release Wasm bytes;
- builder-produced gzip bytes;
- data-section bytes;
- function count;
- exported-method count; and
- bounded retained-size evidence for changed hotspots.

At minimum, the roster must include:

- one representative leaf role;
- root;
- Wasm store; and
- any role found to have substantially different diagnostic reachability.

The preferred full closeout is a compatible run of
`CANIC-WASM-001`. Role-scoped development measurements may guide a slice but
must not be presented as the retained audit.

### Decision Thresholds

The architecture and public-contract benefits stand independently of size, but
the measured result determines how strongly Wasm savings justify further
low-level compaction:

| Raw release delta | Interpretation |
| --- | --- |
| Less than 5 KiB per affected role | Not worthwhile for size alone; retain only cuts justified by the cleaner contract and ownership model |
| 5–10 KiB | Modest; continue only through low-risk, high-fan-out paths |
| 10–20 KiB per leaf or at least 25 KiB in root | Worthwhile |
| 30–50 KiB in a role | Strong result |

These are review thresholds, not acceptance promises. A slice that increases
Wasm must be explained and either corrected or accepted explicitly for its
contract value.

IcyDB measurements must never be inserted into Canic audit history as if they
were Canic results.

## Testing Strategy

### Registry Tests

- every numeric code is unique and nonzero;
- every current public or operator-visible internal code appears in the
  native-only inventory;
- every inventoried code has one host class, origin and rendering entry;
- every masked internal code has an explicit public projection and numeric
  observability owner;
- no unproduced placeholder code is allocated;
- compact `Display` and `Debug` contain no symbolic labels; and
- unknown raw values round-trip without panic.

### Mapping Tests

- typed internal variants map to the intended diagnostic;
- internal-only codes project to the intended safe public code;
- host class and origin come only from the host catalog;
- runtime errors do not store or look up class and origin;
- no mapping path parses or compares text; and
- removal of string context does not change retry or policy decisions.

Tests assert typed codes and observable state, not rendered error strings.

### Wire Tests

- Candid `Error` is exactly one `nat16` field;
- `message : text` is absent;
- the Candid `ErrorCode` variant is absent;
- every checked-in `.did` uses the current shape;
- representative facade endpoints decode both success and error results; and
- host response and replica-query paths decode the compact record.

### Host and CLI Tests

- every current public or operator-visible internal code has one catalog entry;
- known codes render the expected origin, class, label, summary and action;
- unknown codes retain only their raw code and are not assigned guessed
  metadata;
- lookup accepts the canonical numeric forms;
- JSON output follows existing CLI envelope conventions; and
- normal command failures automatically use host rendering.

Human-facing host text may be snapshot-tested in the host/CLI owner. Runtime
and domain tests must not assert it.

### Stable-State Tests

For every changed stable record:

- encode the exact pre-cut state;
- restore it through the maintained lifecycle;
- prove authoritative typed failure state and recovery behavior;
- prove no string matching occurs; and
- exercise upgrade behavior with PocketIC when a canister lifecycle is
  involved.

There is no generic “all old error strings still work” test.

### Wasm Tests

- search canonical release Wasm for a bounded set of removed diagnostic
  fragments;
- inspect retained sections and functions for the high-fan-out error paths;
- confirm strings intentionally retained by logs or protocols have documented
  owners; and
- record deltas using the measurement contract.

Source scans are discovery aids. Absence from source or presence in source is
not a substitute for artifact measurement.

## Performance and Resource Invariants

- Success paths must not allocate merely to prepare an unused diagnostic.
- Canister code performs no registry lookup to construct a public error.
- Release Wasm contains no complete code inventory or host metadata table.
- Public error construction allocates no message string.
- Host rendering may allocate because it is outside canister Wasm and occurs
  on error paths.
- No network call is made to resolve a diagnostic code.
- Diagnostic lookup does not read canister stable state.
- Unknown codes remain printable without a catalog search.

## Contract Impact

### Breaking Public Changes

- `canic::Error.code` changes from a Candid enum to `u16`.
- `canic::Error.message` is removed.
- the public Rust and Candid `ErrorCode` enum is removed.
- callers matching named Candid variants must match current numeric codes
  instead.
- callers parsing message text must stop.

### Preserved Behavior

- endpoints continue to return `Result<T, canic::Error>`;
- authentication and authorization remain endpoint-owned;
- host and CLI users continue to receive human-readable diagnostics for known
  codes;
- unknown diagnostics remain visible; and
- typed domain decisions remain typed inside Canic.

### Conditional Stable Changes

The public error cut does not itself change stable state. Individual durable
failure fields change only after the durable-state audit proves their exact
upgrade path.

## Migration Guide

Canister and Candid callers must:

1. regenerate bindings from the current `.did`;
2. replace named `ErrorCode` matches with numeric code constants;
3. stop reading or parsing `message`; and
4. retain the raw code when logging an unknown diagnostic.

Rust callers should import the current `DiagnosticCode` constants or use typed
accessors. They must not duplicate the host message catalog in canister code.

CLI users receive rendered prose automatically. A code copied from a canister
log can be resolved with `canic diagnostic`.

Canic-owned examples and fixtures are updated in the atomic public slice.
Downstream repositories receive a migration report; automated work under this
design does not edit them.

There is no migration period in which both error shapes are accepted.

## Alternatives Rejected

### Keep `message` Beside a Numeric Code

This preserves the main Wasm-retention source and encourages prose-dependent
callers. It is not a hard cut.

### Carry Class and Origin Beside the Code

`{ code, class, origin }` is more self-describing under host/catalog skew, but
class and origin are redundant for every known globally unique code.
The extra fields also require canister-side metadata or repeated constructor
arguments. 0.102 chooses the smaller `{ code }` contract; the current host
catalog recovers exact origin and class.

### Feature-Gate Rich Messages in `canic-core`

Cargo feature unification makes this boundary fragile. Rich rendering belongs
in a crate that canister artifacts do not link.

### Shorten Existing Messages

Shorter prose may save some bytes but leaves the string-first architecture,
variant formatting and unstable machine contract intact.

### Reuse Only the Current 25 Error Codes

The broad variants cannot reconstruct useful action for many failures. They
are suitable class inputs, not a sufficient leaf taxonomy.

### Add a Generic `detail: String`

This recreates the message field under a new name. Required details belong in
typed owner records.

### Add a Versioned Error Type

`ErrorV2`, dual endpoints and version fields are pre-1.0 compatibility
baggage. Canic maintains one current protocol.

### Strip Every Runtime String

Many strings are protocol constants, application data, metrics or logs.
Deleting them would conflate ownership and can break correctness.

### Rewrite Every Error Before Measuring

A blanket rewrite obscures attribution and risks removing operational context
without proving Wasm value. Measured coherent slices preserve evidence.

## Acceptance Criteria

0.102 is complete when:

1. The public Candid error is exactly `{ code : nat16 }`.
2. The old public `ErrorCode` enum and `message` field no longer exist.
3. No supported endpoint or host decoder accepts the old shape.
4. Release Wasm contains compact diagnostic identity but no complete
   inventory, class, origin or rich rendering catalog.
5. Every current public producer has one registered code and explicit public
   projection.
6. Every current public or operator-visible internal code has one host class,
   origin and rendering entry.
7. `InternalError` and boundary conversions are code-first and do not classify
   through strings.
8. Every masked internal code has an approved numeric observability owner.
9. The CLI renders known failures, including their origin, and looks up raw
   codes.
10. Unknown diagnostics remain lossless and non-panicking without guessed
    metadata.
11. Every changed durable failure field has exact stable decode and lifecycle
    evidence.
12. General logs and required protocol strings remain owned and intact.
13. Canonical Wasm measurements exist for baseline, material slices and
    closeout.
14. The final report distinguishes Canic measurements from comparative IcyDB
    evidence.
15. Active docs, examples and migration guidance describe only the current
    numeric protocol.
16. No aliases, shims, legacy fields, fallback parsers, generation names or
    anti-resurrection tests remain.

## Implementation Readiness

The design is ready for maintainer review. Implementation is ready to begin
only after:

1. 0.99 and 0.100 sequencing is known well enough to select the exact current
   source baseline;
2. the fresh error and durable-state inventories are recorded;
3. the first canonical Wasm baseline is reproducible; and
4. the initial numeric allocation and public projection inventory receive
   maintainer approval.

No numeric assignments in examples in this document are authoritative. The
approved inventory created in Slice A becomes the implementation authority.
