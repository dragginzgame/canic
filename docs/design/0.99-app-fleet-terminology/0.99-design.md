# Canic 0.99 Design: App and Fleet Terminology Hard Cut

Date: 2026-07-23

## Status

- Status: proposed for maintainer approval.
- Release scope: repository-wide terminology and identity hard cut spanning
  configuration, CLI, Candid, paths, JSON, runtime topology names and stable
  metadata, plus only the minimum configuration shape change needed to express
  the new terms truthfully.
- Runtime scope: preserve the current one-subnet behavior. Multi-subnet Fleet
  installation, membership, synchronization and placement are deferred.
- Schema scope: the terminology cut includes the minimum additive identity,
  Directory-provenance and authentication state needed to make the new terms
  safe. Those additions are wire- and stable-state breaks, not renames.
- Compatibility posture: pre-1.0 hard cut. Removed names receive no aliases,
  fallback parsers, duplicate endpoints, dual-written fields or legacy paths.
- Repository scope: Canic only. Downstream migrations are documented but are
  not implemented outside this repository.

## Decision

Canic 0.99 establishes this canonical hierarchy:

~~~text
App
  Fleet
    Authority Subnet
      Fleet Coordinator
      Fleet-authoritative services
    Member Subnet
      Canister
~~~

For Toko:

~~~text
App: Toko
  Fleet: toko-production
    Authority Subnet
      Fleet Coordinator
      authoritative database and other declared authority services
    Member Subnet 1
    Member Subnet 2
    Member Subnet 3
    ...
    Member Subnet 150
  Fleet: toko-staging
  Fleet: toko-canary
~~~

The terms answer different questions:

- **App** answers “what product or application is this?”
- **Fleet** answers “which deployed realization of that App is this?”
- **Fleet Coordinator** answers “which Canister owns Fleet-wide coordination
  and membership state?”
- **Authority Subnet** answers “where do Fleet-wide coordination and declared
  authoritative services live?”
- **Member Subnet** answers “which local root owns this workload canister?”
- **Canister** answers “which concrete runtime member is this?”

A Fleet is a deployed version of one App over one or more member Subnets. It
has one Fleet Coordinator and its explicitly declared Fleet-authoritative
services on one Authority Subnet. **Authority Subnet** is the canonical
replacement for **Prime Subnet**: the distinguished-subnet concept remains,
but its name states its function. It is part of the Fleet topology while
remaining outside ordinary member lifecycle and workload placement.

The current implementation remains the valid one-member case for 0.99. Its
single current root still owns the existing Fleet-wide state, so its hosting
Subnet is the current co-located Authority Subnet and it acts as both the
current Fleet Coordinator and member root. 0.100 separates those ownership
roles, provisions the selected Authority Subnet and adds synchronization.

0.99 does not introduce `AppRelease` as another topology level. Version,
configuration and artifact identity are Fleet metadata backed by existing
release-set and evidence digests.

## Why This Must Precede Multi-Subnet Work

The current terminology collapses the App and Fleet levels:

- the CLI `fleet` is a source/configuration template;
- `[fleet].name` identifies that template;
- `FleetRoleRef` identifies a declared role, not a deployed canister;
- the host `deployment` record identifies one live root target; and
- runtime `App*` state is actually scoped to one deployed root tree.

Those names are tolerable while one config produces one root tree. They become
incorrect when Toko can run across 150 Subnets, when the same role can have an
instance on every Subnet, and when production, staging and canary Fleets can
exist for the same App.

Adding multi-subnet behavior before fixing the vocabulary would make the
single-root assumptions part of new stable state, CLI output and public APIs.
0.99 fixes the boundary first.

## Canonical Terms

### App

An App is the logical product and source-level configuration authority. Toko is
an App.

An App owns:

- its stable name;
- role declarations and package bindings;
- default topology and placement declarations;
- build inputs and release-set construction; and
- default runtime policy carried into a Fleet.

An App does not own runtime subnet principals, root principals or live
canister membership. One App may have multiple Fleets.

### Fleet

A Fleet is one live deployed realization of an App.

A Fleet owns:

- a stable Fleet identity;
- exactly one App identity;
- one canonical IC network identity, with host environments remaining lookup
  context;
- the release-set, configuration and artifact digests being deployed;
- exactly one Fleet Coordinator binding;
- one or more member Subnets;
- the Authority Subnet and Fleet Coordinator named by that binding;
- explicitly declared Fleet-authoritative service roles and their Directory
  projection;
- the Coordinator-owned Authority Subnet registry;
- all member Canisters recorded by their member Subnet registries; and
- Fleet-wide operational state and provenance.

At steady state, a Fleet represents a specific deployed version of its App.
During a rolling upgrade, the desired Fleet version and observed member
versions may temporarily differ. That rollout state does not create an
`AppRelease` topology object and does not require changing Fleet identity.

The same release-set digest may be deployed to more than one Fleet. The digest
is evidence; the Fleet is the live topology and authority boundary.

### Fleet Coordinator and Authority Subnet

Every Fleet has exactly one Fleet Coordinator. It owns Fleet-wide
coordination state defined by later designs, including the Fleet Registry, and
acts as the local root for Canic-managed authority services on the Authority
Subnet.

`Authority Subnet` is a hard-cut rename of the runtime concept previously
called `Prime Subnet`; it is not the removal of the concept. The Fleet
Coordinator is installed there together with only explicitly declared
Fleet-authoritative services. The Authority Subnet:

- is a distinguished part of Fleet topology, not a topology rank;
- is outside the ordinary member `Joining`/`Active`/`Draining`/`Removed`
  lifecycle;
- is excluded from ordinary workload placement;
- has a local `SubnetRegistry` owned by the Fleet Coordinator for its
  authority-service children; and
- may physically coincide with a member Subnet only when deployment explicitly
  chooses that co-location, without merging their registry scopes.

Fleet-wide mutable data may therefore have one canonical writable service on
the Authority Subnet. This is an intentional single logical write authority,
replicated by the IC within that Subnet. Member Subnets may host read replicas
or derived copies discovered through their local Directories. The
application-data replication protocol and any explicit failover remain
separate designs.

“App-wide” data is still scoped to one Fleet: production, staging and canary
must receive distinct authority services. The runtime projection is therefore
`FleetDirectory`, not `AppDirectory`.

In 0.99 the current root and Coordinator role remain co-located. 0.100
installs the separate Fleet Coordinator on the selected Authority Subnet,
provisions its declared authority services and implements Registry
synchronization.

### Subnet

A Subnet is one runtime member of a Fleet. It is identified by the platform
Subnet identity and has one Canic root for that Fleet.

Each member Subnet root owns the complete local canister topology for that
member Subnet. The future Fleet Coordinator will own membership of member
Subnets in the Fleet and the Authority Subnet's local authority-service
registry; it will not replace any member root's local Subnet registry.

The current `SubnetRole` is not a runtime Subnet identity. 0.99 hard-cuts that
name to `SubnetSlotId`.

A Subnet Slot is an App-declared logical workload-placement slot. The current
required `prime` slot is hard-cut to `default`; it selects the default workload
placement for the one-member implementation and has no Fleet-wide authority.
Other slots remain logical configuration positions until a later design
defines reusable profiles and multi-subnet placement.

The existing `[subnets.<slot>]` TOML structure remains. Its Rust, DTO, generated
config, environment and diagnostic names must say `SubnetSlot` or
`SubnetSlotId`, never `SubnetRole`. The reserved current slot is
`subnets.default`, never `subnets.prime`.

### Canister

A Canister is one concrete runtime member of exactly one Fleet and is
physically hosted on one platform Subnet at a time. A member workload/root
Canister belongs to one member Subnet registry. The Fleet Coordinator and each
declared authority-service child belong to the Authority Subnet registry,
whose root is the Coordinator. `FleetCoordinatorBinding` separately binds the
Coordinator and Authority Subnet because that Subnet is not an ordinary member
entry. A Canister principal remains its concrete IC identity.

A role does not identify a concrete Canister. A role such as `vault_db` may be
instantiated on 150 Subnets, and a scaling, sharding or binding policy may
produce several Canisters with the same role on one Subnet.

### Role

A role describes behavior, package identity and placement policy. It belongs
to the App definition.

An App-scoped role reference is sufficient for source, build and configuration
work:

~~~rust
pub struct AppRoleRef {
    pub app: AppId,
    pub role: CanisterRole,
}
~~~

It is not sufficient to address a deployed Canister.

### Deployment

Deployment is an operation, plan, observation and receipt family. It is not a
long-lived topology level between App and Fleet.

Names such as `DeploymentPlan`, `DeploymentReceipt` and `canic deploy` remain
correct when they describe deploying or inspecting a Fleet. A persisted
“deployment target” that identifies a live installed root is a Fleet identity
and must use Fleet terminology.

### Environment

An environment selects host-side ICP connectivity and local state location. It
does not replace App, Fleet or Subnet identity.

### Fleet Identity

The operator supplies an immutable, validated `FleetId` when planning the
Fleet's initial installation.

`FleetId` is a canonical bounded name. Its uniqueness scope is one canonical
IC network, not one host environment label. The canonical identity key is
therefore `(network, FleetId)`. Two host environments that address the same IC
network address the same Fleet namespace.

The canonical identity types are:

~~~rust
pub struct CanonicalNetworkId(/* bounded canonical identifier */);

pub struct FleetKey {
    pub network: CanonicalNetworkId,
    pub fleet: FleetId,
}

pub struct FleetBinding {
    pub fleet: FleetKey,
    pub app: AppId,
}
~~~

The host deployment authority resolves the selected ICP network profile to a
`CanonicalNetworkId` before it creates a plan. The value is not an environment
label and is not the current two-way `BuildNetwork::{Local, Ic}`
classification. It must distinguish every trusted local, staging and
production network on which the same `FleetId` could exist. Two host
environments that resolve to the same canonical network produce the same
`CanonicalNetworkId`.

The initial deployment plan binds:

- `FleetBinding`;
- current one-member root target and expected principal when known; and
- configuration, release-set and artifact evidence.

Initial installation persists the binding in installed host state, current root
state and protected runtime state on every managed Canister. Existing
Canisters receive the same protected binding through the coordinated 0.99
activation step. It is deployment/install identity, not App configuration and
not ordinary mutable cascade state.

Version evidence may advance through later deployments, but `FleetBinding` is
immutable. A Canister rejects installation or activation if protected state
already contains a different binding. Ordinary cascade freshness therefore
cannot change, remove or reinterpret its Fleet identity.

Every subsequent plan, install, upgrade, observation, receipt and live query
must reject a mismatching App or network before mutation. A Fleet cannot be
renamed or rebound to another App in 0.99. Fleet rename, transfer and rebinding
are outside this release.

Environment remains host connectivity context and local-state selection. It is
recorded as provenance but is not part of canonical Fleet identity.

### Authority Subnet Selection

Authority placement is deployment input, not App configuration and not
ordinary Fleet member registration. The host-side selector is:

~~~rust
pub enum AuthoritySubnetSelector {
    NetworkDefault,
    Profile(SubnetProfileId),
    Explicit(SubnetId),
}

pub struct FleetCoordinatorBinding {
    pub fleet: FleetBinding,
    pub authority_subnet: SubnetId,
    pub coordinator: Principal,
}
~~~

`SubnetProfileId` is an open, bounded deployment-profile name resolved against
trusted metadata for one canonical network. It is not a closed Rust enum of
today's IC subnet offerings. This lets profiles such as `fiduciary`,
`european` and future network-defined choices be added without changing the
runtime protocol. Profile resolution is entirely host-side: the Canister Wasm
receives no profile strings or profile catalog.

Selection rules are:

| Deployment | `NetworkDefault` |
| --- | --- |
| Public IC live deployment | Resolve the current `fiduciary` profile |
| Other live network | Resolve that network's configured authority default or fail planning |
| PocketIC, local replica or test harness | Use the harness-provided authority Subnet |
| Any deployment with `Profile` or `Explicit` | Use the developer-selected target after validation |

The Fiduciary subnet is the live default because it is an open application
subnet with a larger node committee and stronger fault threshold than a
standard application subnet. Its higher cycle multiplier is an explicit
safety-versus-cost tradeoff for the Coordinator and declared authority
services; ordinary workloads do not inherit that placement. A write-heavy or
storage-heavy App may explicitly select another eligible Authority Subnet.

The resolver must:

1. Resolve the selector using current trusted network metadata at plan time.
2. Verify the exact Subnet belongs to the selected canonical network and
   accepts user Canister creation.
3. Record the selector, resolved `SubnetId`, evidence source and resolution
   time in the plan and receipt.
4. Fail closed if the default/profile is unavailable, ambiguous, stale or at
   capacity.
5. Never fall back silently to a random application Subnet.

Subnet principal IDs and named network offerings can change. Runtime and stable
state therefore persist only the resolved `SubnetId` in
`FleetCoordinatorBinding`; the profile name remains host-side provenance.
Trusted metadata must be a pinned network-profile catalog or equivalent
verifiable deployment source whose identity/digest is recorded in the plan;
Canic must not scrape human-facing dashboard HTML during deployment.
Changing a selector after the authority exists does not move the Canister.
Authority relocation requires an explicit fenced workflow outside 0.99 and
0.100.

0.99 reserves this selection contract and replaces Prime terminology with
Authority terminology. The separate `FleetCoordinatorBinding` becomes
installed runtime authority when 0.100 introduces the Fleet Coordinator.

## Identity and Membership

Every concrete Canister must have a stable identity that does not include its
current placement:

~~~rust
pub struct FleetCanisterId {
    pub fleet: FleetKey,
    pub principal: Principal,
}
~~~

Network is part of the value, not ambient interpretation. The same
`(FleetId, principal)` on two networks produces two different
`FleetCanisterId` values. An environment label must never substitute for the
canonical network identity.

0.99 does not introduce a parallel `FleetCanisterRecord`.
`SubnetRegistryEntry` remains the single current local registry row boundary,
and `canic_subnet_registry` retains its exact endpoint name and response shape
through the coordinated upgrade. Portable evidence combines the row with the
trusted `FleetBinding` and owning `SubnetId` at the evidence boundary. A later
wire-shape change must evolve or replace the existing projection rather than
store or expose a second overlapping authoritative row type.

An IC canister may retain its principal while moving between Subnets. Such a
move changes the owning Subnet registry and its observed `SubnetId`; it does
not create a new `FleetCanisterId`. Receipts, durable work and external APIs
must bind the stable ID and treat location as observed topology.

The authority model is hierarchical:

1. `FleetCoordinatorBinding` identifies the Fleet Coordinator and its
   Authority Subnet.
2. The Fleet Coordinator owns the Registry containing the complete set of
   member Subnet/root bindings.
3. The Fleet Coordinator also owns the Authority Subnet's local
   `SubnetRegistry`, containing itself and explicitly declared authority
   services.
4. Each member Subnet registry owns the complete set of its local
   root/workload Canisters.
5. Every managed Canister appears in exactly one owning Subnet registry at a
   time; a placement move transfers that ownership without changing identity.
6. A complete operator inventory is composed from the Coordinator binding,
   Authority Subnet registry, Fleet membership and member Subnet registries.
7. Ordinary Canisters receive only the directories and state projections they
   need; they do not receive every registry record.

This gives Canic a concept for every Canister in Toko without centralizing and
cascading the complete topology of 150 Subnets.

## Delegation Audience and Protected Runtime Identity

`DelegationAudience::Project(String)` is not compatible with the new
hierarchy. The same App can have production, staging and canary Fleets, so an
App-scoped audience permits a credential created for one Fleet to be presented
to another.

0.99 hard-cuts the audience to the canonical Fleet identity:

~~~rust
pub enum DelegationAudience {
    Fleet(FleetKey),
}
~~~

There is no retained `Project` variant, string fallback or dual verifier.
Audience verification reads the immutable protected `FleetBinding` stored
locally on the verifying Canister. `prepare_delegated_token` may still receive
the requested audience, but preparation and verification must reject any
audience that differs from the local Fleet key.

The protected binding is established before authentication resumes:

1. Fresh installs receive it from deployment/install input.
2. The 0.99 upgrade journal records the expected binding before any Canister
   upgrade.
3. Each upgraded Canister is activated by the trusted deployment/controller
   path while the Fleet remains quiesced.
4. Activation persists the binding once and rejects overwrite or mismatch.
5. Ordinary topology cascades may carry the binding for consistency checks,
   but they are never its authority and cannot update it.

All persisted authorization material whose meaning included
`DelegationAudience::Project` is invalid under the cut. The upgrade must
invalidate or replace delegation audience records, issuer policies, renewal
templates, active proofs, retained or prepared tokens, positive verification
caches and related replay state. No old Project-scoped credential is accepted
after its Canister runs 0.99. Fresh Fleet-scoped proofs and certificates must
be installed and verified before application authentication traffic resumes.

## Registry, Directory and Index

0.99 supersedes the 0.27 taxonomy that reserved `directory` for keyed
placement and used `index` for cascaded topology lookups.

The canonical meanings are now:

| Term | Meaning | Write authority |
| --- | --- | --- |
| Registry | Complete authoritative records within a declared scope | Owning root |
| Directory | Derived, read-only, consumer-shaped projection of registry state | None; its source Registry remains authoritative |
| Index | Internal lookup or storage acceleration only | Never a public domain authority |

The runtime topology names are:

| Surface | Completeness and source | Read visibility | Distribution |
| --- | --- | --- | --- |
| `canic_fleet_registry` | Current root-owned one-subnet Subnet/root bindings; controller-gated while its host/backup semantics remain unresolved | Controllers only | Queried; not cascaded as complete records |
| `canic_fleet_directory` | Consumer projection derived by the root from current Fleet-scoped sources | Public | Queried and cascaded |
| `canic_subnet_registry` | Complete local Canister membership owned by the Subnet root | Public | Queried; not cascaded as complete records |
| `canic_subnet_directory` | Consumer projection derived by the root from the local Subnet registry/configuration | Public | Queried and cascaded |

A Registry may contain lifecycle state, provenance, tombstones and
administrative metadata that is intentionally absent from its Directory. A
Directory has no mutation authority and is shaped for consumers, but is not
necessarily less visible and need not contain fewer rows in every
representation. Public visibility therefore does not distinguish Registry
from Directory; completeness, canonical writer and derivation do.

The current `canic_app_registry` implementation is controller-gated and carries
an explicit unresolved host/backup topology comment. 0.99 renames that current
one-subnet boundary to `canic_fleet_registry`; the rename does not claim that
multi-subnet membership, host semantics or backup semantics are resolved. That
TBD remains visible and is owned by the 0.100 design. The implementation
inventory must confirm the endpoint is complete for its declared current
one-subnet binding scope; if it is not, the boundary must receive an honest
non-Registry name rather than inheriting authority by terminology.

Every Directory response and cascaded Directory input identifies its
derivation source with:

~~~rust
pub struct DirectoryProvenance {
    pub fleet: FleetBinding,
    pub source_root: Principal,
}
~~~

This is the minimum one-subnet provenance contract. It identifies which Fleet
and root produced the projection but does not claim freshness, revision
ordering or multi-subnet synchronization. 0.100 adds the richer
epoch/revision/content evidence required for mirrored Fleet Registry state.

An implementation may use a B-tree index internally. That does not make
`Index` part of configuration, Candid, DTO, state-contract or CLI vocabulary.

### Keyed Placement Terminology

The current keyed placement family also uses `directory`. That conflicts with
the new single meaning and is included in the terminology hard cut.

The keyed placement concept becomes **binding**:

- one logical key claims or binds at most one dedicated instance root;
- `DirectoryRegistry` becomes `PlacementBindingRegistry`;
- directory placement config becomes binding placement config; and
- placement bind, retry and recovery behavior is unchanged.

This is a rename of an existing placement contract, not a redesign of its
state machine.

## Version and Release Evidence

0.99 does not add `AppRelease`.

The version deployed by a Fleet is represented by existing evidence such as:

- Canic version;
- canonical App configuration digest;
- release-set manifest digest;
- role artifact-set digest;
- embedded configuration-set digest; and
- observed module hashes.

The Fleet has desired evidence and observed per-member evidence. A later
rollout design may add typed rollout state, but version remains Fleet metadata
rather than a parent or child topology object.

No manually maintained `version = "..."` field is added to `canic.toml` in
0.99.

## Pure Rename and Hard-Cut Mapping

The implementation must classify by semantic ownership rather than perform a
blind text replacement.

| Current surface | Canonical 0.99 surface | Reason |
| --- | --- | --- |
| Source/config `fleet` | App | It describes roles, packages and build inputs |
| `[fleet].name` | `[app].name` | The checked-in config identifies the App |
| `FleetConfigSnapshot` | `AppConfigSnapshot` | It is a validated source App config |
| `FleetRoleRef` | `AppRoleRef` | It identifies a declared role, not a live member |
| `SubnetRole` | `SubnetSlotId` | It identifies an App-declared logical slot, not a platform Subnet |
| Runtime `Prime Subnet` | `Authority Subnet` | It hosts the Fleet Coordinator and declared authority services; Fiduciary is the recommended live placement |
| Draft `Fleet Authority Canister` | `Fleet Coordinator` | It coordinates Fleet membership and owns the Authority Subnet registry without being the application-data service itself |
| Reserved `subnets.prime` slot | Reserved `subnets.default` slot | It is the current default workload placement, not Fleet authority |
| `fleet_template` | `app` or `app_name` | A live Fleet is created from an App config |
| Live `deployment_name` | `fleet` or `fleet_name` | The persisted target is the Fleet |
| Deployment catalog | Fleet catalog | It lists live installed Fleet identities |
| `AppRegistry` | `FleetRegistry` | Its current one-subnet bindings belong to one deployed Fleet; the existing host/backup semantics TBD remains |
| `AppIndex` | `FleetDirectory` | It is a consumer-shaped Fleet-wide cascaded lookup projection, including declared Fleet-authoritative service instances |
| `SubnetIndex` | `SubnetDirectory` | It is a consumer-shaped local cascaded lookup projection |
| `AppState` family | `FleetState` family | The cascaded operational state belongs to one Fleet |
| Keyed placement directory | Placement binding | `Directory` is reserved for cascaded projections |

The old word `prime` conflated two concepts and therefore has two deliberate
destinations:

- the distinguished runtime Subnet becomes **Authority Subnet**; and
- the App-declared workload slot becomes `subnets.default`.

This is not contradictory: Authority placement is live deployment input,
whereas `subnets.default` remains an App workload-topology declaration.

These are semantic source/API renames. A Rust type-name change alone does not
change Candid's structural wire format. Endpoint names, record field labels and
variant labels do.

## Additive and Breaking Schema Work

The following items are required by the terminology contract but are not
renames:

| Schema change | Boundary | Hard-cut consequence |
| --- | --- | --- |
| `CanonicalNetworkId`, `FleetKey` and `FleetBinding` | Plans, receipts, host state, protected runtime state and portable evidence | Existing identity records migrate once; no ambient-network interpretation remains |
| Protected local Fleet binding | Every managed Canister | Fresh install or coordinated activation must complete before auth resumes |
| `FleetCanisterId.fleet: FleetKey` | Durable work, receipts and portable evidence | Same Fleet/principal on different networks no longer collides |
| Reserved Subnet Slot value `prime` → `default` | Config, generated environment, install/bootstrap DTOs and any persisted slot identity | Existing one-member state migrates once; `prime` is never accepted after activation |
| `DirectoryProvenance` and Directory input/response envelopes | Public Directory queries, install payload and state cascade | Candid and stored Directory projections break atomically |
| `DelegationAudience::Fleet(FleetKey)` | Delegation preparation, persistence and verification | Project-scoped credentials and caches are invalidated |

The Directory transport shapes become records carrying provenance rather than
renamed tuple wrappers:

~~~rust
pub struct FleetDirectoryInput {
    pub provenance: DirectoryProvenance,
    pub entries: Vec<DirectoryEntryInput>,
}

pub struct SubnetDirectoryInput {
    pub provenance: DirectoryProvenance,
    pub entries: Vec<DirectoryEntryInput>,
}
~~~

The public `canic_fleet_directory` and `canic_subnet_directory` responses wrap
each requested page with the same provenance. The implementation must evolve
the existing `AppIndexArgs`, `SubnetIndexArgs` and paged query projections in
place; it must not introduce parallel provenance-free Directory shapes.

The expected endpoint direction is therefore:

~~~text
canic_app            -> canic_fleet_admin
canic_app_registry   -> canic_fleet_registry
canic_app_index      -> canic_fleet_directory
canic_subnet_index   -> canic_subnet_directory
canic_app_state      -> canic_fleet_state
~~~

The exact active endpoint inventory must be audited before implementation.
Removed pre-1.0 endpoints receive no forwarding aliases.

`canic_subnet_registry`, `canic_sync_state` and `canic_sync_topology` retain
their endpoint names. In particular, `canic_subnet_registry` must not be
renamed for symmetry: its exact response signature is the host's stable
transition, readiness and recovery read surface during the coordinated
upgrade.

Names such as `ApplicationReceipt` or a database settlement index are not
renamed merely because they contain an English word from this taxonomy. The
hard cut applies only where the term denotes App, Fleet, topology Directory or
topology Registry ownership.

## Minimum Configuration Update

`canic.toml` is an App definition. It must identify the App, not a live Fleet.
A Fleet name belongs to deployment input, plans, receipts and installed Fleet
state.

The current shape:

~~~toml
app_index = ["project_hub"]

[fleet]
name = "toko"

[app]
init_mode = "enabled"
~~~

becomes:

~~~toml
[app]
name = "toko"
init_mode = "enabled"

[directories.fleet]
roles = ["project_hub"]
~~~

Existing `[app.whitelist]` remains nested under `[app]`:

~~~toml
[app.whitelist]
principals = ["aaaaa-aa"]
~~~

This is the minimum schema change:

1. Add required `name` to the existing `AppConfig`.
2. Remove `FleetConfig` from `ConfigModel`.
3. Replace `app_index` with `[directories.fleet].roles`. The declaration
   selects App roles that have one Fleet-authoritative instance discoverable
   through `FleetDirectory`; it is not itself a runtime Directory. 0.99
   preserves current one-member creation and placement behavior. 0.100 places
   those authoritative instances as children of the Fleet Coordinator on the
   Authority Subnet.
4. Rename keyed placement `directory` config to `binding` without changing its
   behavior.
5. Rename `SubnetRole` to `SubnetSlotId` while keeping the existing
   `[subnets.<slot>]` TOML shape and hard-cutting the required
   `subnets.prime` slot to `subnets.default`.
6. Keep the existing role declarations and topology template behavior.
7. Add no runtime subnet principal, root principal or complete Fleet membership
   list to `canic.toml`.
8. Add no manual App or Fleet version string.

The App name uses the current Fleet-name validation contract. It remains a
bounded canonical source identity suitable for paths, package metadata and
diagnostics.

### Hard-Cut Parsing Rules

After 0.99:

- `[app].name` is required wherever `[fleet].name` is required today;
- `[fleet]` is an unknown table and fails parsing;
- `app_index` is an unknown field and fails parsing;
- keyed placement `directory` is an unknown field and fails parsing;
- `subnets.prime` is an unknown slot and fails validation;
- `subnets.default` is required for the current one-member topology;
- `[directories.fleet].roles` retains the current default-service validation
  until a later multi-subnet design changes its shape; and
- no parser accepts both old and new names.

Tracked configs, examples, scaffolds, generated configuration, fixtures and
active documentation must move atomically with the schema.

## CLI and Host Boundary

The source/config command family becomes App-scoped:

~~~text
canic fleet create <name>                 -> canic app create <app>
canic fleet list                          -> canic app list
canic fleet config <name>                 -> canic app config <app>
canic fleet role ...                      -> canic app role ...
canic build <fleet> <role>                -> canic build <app> <role>
fleets/<name>/canic.toml                  -> apps/<app>/canic.toml
package.metadata.canic.fleet              -> package.metadata.canic.app
~~~

Live target identity becomes Fleet-scoped:

~~~text
deployment target name                    -> Fleet name
.canic/<environment>/deployments/<name>   -> .canic/<environment>/fleets/<fleet>
deployment catalog entry                  -> Fleet catalog entry
installed deployment inventory            -> Fleet inventory
~~~

Deployment remains the verb and evidence workflow:

~~~text
canic deploy plan <fleet> --app <app>
canic deploy install <fleet> --plan <file>
DeploymentPlan
DeploymentReceipt
~~~

Those plans and receipts must carry both App and Fleet identity. A plan may
continue to describe one current member root in 0.99; it must not claim that
the Fleet already spans several installed Subnets or that the root's Subnet is
intrinsically privileged. 0.99 does not expose a no-op authority selector.
0.100 adds mutually exclusive `--authority-subnet-profile <profile>` and
`--authority-subnet <principal>` plan options; omitting both selects
`NetworkDefault`.

The exact command inventory must be audited so config-template commands move
to `canic app`, live-state commands move to Fleet terminology, and operation
commands remain under `canic deploy`. No command alias preserves the old
meaning of `fleet` or `deployment target`.

## Host-State Migration

Existing installed state and immutable deployment evidence must not be silently
discarded. 0.99 uses one explicit, bounded and resumable host migration.

The migration maps:

~~~text
fleet_template   -> AppId
deployment_name  -> FleetId
environment      -> host provenance plus resolved canonical IC network
~~~

The migration must:

1. Run in an explicit migration mode before ordinary 0.99 Fleet operations.
2. Read old deployment paths only inside that migration operation.
3. Validate every old record, resolve the canonical network and reject App,
   Fleet or path collisions before changing state.
4. Write a durable migration journal binding source paths and content hashes to
   the target `(network, FleetId, AppId)`.
5. Atomically write the new installed Fleet state and move related mutable
   state beneath `.canic/<environment>/fleets/<fleet>`.
6. Preserve immutable receipts and evidence byte-for-byte in a Fleet evidence
   archive; the migration receipt binds their old hashes to the new identity
   without rewriting signed or digested content.
7. Remove the old active deployment paths only after the new state and
   migration receipt are durable.
8. Resume idempotently after interruption and produce the same final state.

After successful migration, normal readers inspect only Fleet paths and current
schemas. They do not fall back to deployment paths. A failed migration stops
with typed, operator-visible recovery instructions and leaves the last complete
authority intact.

Regenerating or discarding existing installed state is not the default 0.99
policy. An operator may deliberately abandon invalid local state only through
an explicit recovery action that reports which evidence will no longer be
catalogued.

## Coordinated Breaking Fleet Upgrade

0.99 deliberately breaks public query names, Candid record labels, Directory
envelopes and delegation variants. Mixed 0.98/0.99 operation is unsupported.

The internal cascade endpoint names stay fixed:

- `canic_sync_state` carries the breaking state payload;
- `canic_sync_topology` and its `TopologySnapshotInput` field labels remain
  unchanged; and
- renaming a Rust DTO type without changing its structural Candid shape is not
  itself a wire break.

The known record-field cuts are:

| Record | Removed field | Current field |
| --- | --- | --- |
| `StateSnapshotInput` | `app_state` | `fleet_state` |
| `StateSnapshotInput` | `app_index` | `fleet_directory` |
| `StateSnapshotInput` | `subnet_index` | `subnet_directory` |
| `CanisterInitPayload` | `app_index` | `fleet_directory` |
| `CanisterInitPayload` | `subnet_index` | `subnet_directory` |
| `CanisterInitPayload` | absent | `fleet_binding` |
| `EnvBootstrapArgs` | `subnet_role` | `subnet_slot` |
| `EnvSnapshotResponse` | `subnet_role` | `subnet_slot` |

`AppIndexArgs` and `SubnetIndexArgs` travel inside
`StateSnapshotInput` through `canic_sync_state`; they are not part of
`TopologySnapshotInput`. Their type-name renames are free on the wire, while
the enclosing field-label changes and new provenance envelopes are breaking.
The `EnvBootstrapArgs.subnet_slot` and
`EnvSnapshotResponse.subnet_slot` values also hard-cut the reserved slot from
`prime` to `default`; this is a data-value migration in addition to the field
rename.
The implementation inventory must add any other changed Candid field or
variant label to this table before code changes begin.

The one-Subnet upgrade protocol is:

1. Preflight the complete managed Canister inventory, artifact set, exact
   Candid cut inventory, stable migrations and host-state migration plan.
2. Resolve and durably journal the expected `FleetBinding`.
3. Quiesce provisioning, topology mutation, Canic cascades, delegated
   authentication, relevant timers and application traffic that can cross a
   changed interface.
4. Persist the durable upgrade journal before the first Canister upgrade.
5. Upgrade non-root Canisters while the old current root remains quiesced and
   performs no cascade or renamed endpoint call.
6. Through the trusted deployment/controller activation path, persist the
   protected Fleet binding on each upgraded non-root and invalidate its
   Project-scoped authorization state. Migrate any persisted reserved Subnet
   Slot value from `prime` to `default`. It remains quiesced.
7. Upgrade the current root last, persist the same protected binding,
   migrate the same reserved slot identity, invalidate its Project-scoped
   authorization state and resume the journal under 0.99 code.
8. Run one complete `canic_sync_state` cascade with the current field labels
   and Directory provenance, plus the unchanged `canic_sync_topology`
   cascade.
9. Reissue and verify Fleet-scoped delegation proofs and certificates.
10. Verify all managed Canisters and the host recovery read surface, persist
    the completion receipt, then remove quiescence.

An interruption before the current root upgrade resumes the remaining
non-root upgrades while the Fleet stays quiesced. An interruption after the
current root upgrade resumes protected-binding activation, authorization
refresh, the 0.99 cascade and verification from the journal. Mixed-version
state is a transitional journal state, never an operational mode.

During mixed state, host transition, readiness, diagnostics and recovery logic
may call only an unchanged wire surface or a target the journal proves already
runs the expected release. The invariant is satisfied for local registry
recovery by keeping the exact `canic_subnet_registry` endpoint and response
signature unchanged. The old root makes no cascade call after the first
non-root upgrade; therefore it never sends an old `StateSnapshotInput` to a
new Canister. The new root starts its current-only cascade only after every
non-root is upgraded; therefore it never sends the new shape to an old
Canister.

PocketIC must qualify interruption and restart at every durable boundary. A
compatibility endpoint, compatibility field, old delegation variant or
version-sniffing fallback is not an acceptable recovery mechanism.

## Runtime and Stable-State Boundary

The terminology hard cut preserves topology behavior while deliberately
changing the minimum identity, provenance and authorization schemas described
above.

0.99 must preserve:

- current root and child behavior;
- current registry and directory contents;
- current cascade ordering and authorization;
- current role and service cardinality; and
- current one-subnet install behavior.

For a pure source-name change:

- its existing stable-memory allocation ID remains owned by the renamed
  canonical structure;
- its existing stable key literal remains byte-for-byte unchanged, even if the
  opaque allocation label contains an old source term;
- the serialized map keys, values and record field bytes remain unchanged;
- there is one reader and one writer after the cut;
- no parallel old/new stable map is introduced; and
- state-contract metadata and allocation evidence are updated consistently.

Stable allocation IDs and key literals are ABI identities, not public
terminology aliases. Changing them would create an unnecessary data migration,
so they are retained and documented as allocation-preserving source renames.

The new `FleetBinding`, `default` Subnet Slot value, Directory provenance and
Fleet-scoped authorization records are different: they require one explicit,
bounded, coordinated migration or invalidation. The post-cut code has exactly
one current reader and writer for each shape. It never dual-reads the old
record, dual-writes both schemas or keeps a fallback after activation.

## Authority Model After 0.99

0.99 names the future authorities without implementing them:

| State | Canonical authority |
| --- | --- |
| App definition | Checked-in App configuration and package declarations |
| Canonical network resolution | Trusted host deployment/network profile resolved before planning |
| Authority Subnet selector | Host deployment input resolved to an exact eligible `SubnetId`; Fiduciary profile by default for public-IC live deployment |
| Fleet identity binding | Initial deployment plan and receipt, persisted in installed host state, current root state and immutable protected state on every managed Canister |
| Fleet version evidence | Deployment plans, receipts, installed host state and current-root-accepted desired evidence |
| Current Fleet Subnet/root binding view | Current one-member root; controller-only endpoint with the existing host/backup semantics TBD retained |
| Multi-subnet Fleet membership | Future Fleet-Coordinator-owned `FleetRegistry` in 0.100 |
| Local Canister membership | Current Subnet root `SubnetRegistry` |
| Fleet discovery projection | Root-derived `FleetDirectory` with one-subnet source provenance, including declared Fleet-authoritative roles |
| Local discovery projection | Root-derived `SubnetDirectory` with one-subnet source provenance |
| Delegation audience | Protected local `FleetBinding`; Fleet-scoped credentials only |

Current one-subnet state may populate Fleet structures with only its sole
member Subnet. That Subnet is also the current co-located Authority Subnet
because its root owns Fleet-wide state. It must not fabricate additional
membership or pretend multi-subnet synchronization exists.

## Implementation Slices

### Slice 1 — Semantic Inventory

- Classify every active `App`, `Fleet`, `Deployment`, `Directory`, `Index` and
  `Registry` surface by actual ownership.
- Identify names that are ordinary English or unrelated implementation indexes
  and must remain unchanged.
- Freeze the exact config, CLI, Candid endpoint, Candid record-field,
  stable-state, JSON and path migration inventory before edits begin.

### Slice 2 — Minimum Config Hard Cut

- Merge the current Fleet name into `AppConfig.name`.
- Remove `FleetConfig`.
- Replace `app_index` with `[directories.fleet].roles`.
- Treat those roles as the declaration of Fleet-authoritative service
  discovery while preserving current one-member placement behavior.
- Rename `SubnetRole` to `SubnetSlotId` across configuration and generated
  surfaces.
- Rename the required workload slot from `subnets.prime` to
  `subnets.default`.
- Rename keyed placement config from `directory` to `binding`.
- Update validation, rendering, scaffolds, examples and tracked configs.
- Test the maintained current schema without anti-resurrection fixtures.

### Slice 3 — App Source Surfaces

- Rename config discovery, config snapshots, role references, package metadata,
  source paths and source-oriented CLI commands to App.
- Move the tracked source layout from `fleets/` to `apps/` as one atomic,
  buildable workspace cut covering the root workspace member paths, every
  affected package's `package.metadata.canic` key, shared and package-local
  build paths, demo/test crates, tracked test configuration, CI scripts and
  audit scripts. No intermediate commit may leave the workspace manifest
  pointing at the old tree.
- Keep build behavior and artifact contents unchanged except for canonical
  identity labels.

### Slice 4 — Fleet Live Surfaces

- Rename persisted live deployment target identity to Fleet identity.
- Rename installed-state paths, catalog/inventory projections and JSON fields.
- Keep `deploy` for action plans, checks, execution and receipts.
- Make plans, receipts, host state and current root state bind Fleet, App and
  canonical network identity explicitly.
- Add the host-side Authority Subnet selector contract without installing the
  separate Coordinator or authority-service children before 0.100.
- Persist the protected `FleetBinding` on every managed Canister
  through fresh-install or coordinated activation input.
- Reject Fleet rename, App rebinding and network mismatch.

### Slice 5 — Host-State Migration

- Implement the explicit journalled migration from deployment paths to Fleet
  paths.
- Preserve immutable receipts and evidence byte-for-byte while binding them to
  the migrated Fleet identity through new migration evidence.
- Qualify collision handling, interruption, restart and deliberate abandonment.
- Remove all legacy-path reads from ordinary post-migration operation.

### Slice 6 — Runtime Topology Surfaces

- Rename Fleet-scoped `AppRegistry`, `AppIndex` and `AppState` owners.
- Rename `SubnetIndex` to `SubnetDirectory`.
- Rename keyed placement Directory owners to Placement Binding owners.
- Retain the exact `canic_subnet_registry` name and response shape.
- Hard-cut `StateSnapshotInput`, `CanisterInitPayload`,
  `EnvBootstrapArgs` and `EnvSnapshotResponse` field labels using the explicit
  inventory above.
- Add `DirectoryProvenance` to the current Directory cascade and public query
  envelopes.
- Evolve the existing topology DTOs in place; do not add
  `FleetCanisterRecord` or competing provenance-free Directory DTOs.
- Preserve pure-rename stable allocations and bytes while migrating only the
  explicit additive identity/provenance state.

### Slice 7 — Coordinated Upgrade Qualification

- Implement the quiesced non-root-first, root-last upgrade journey as
  specified.
- Activate the immutable Fleet binding and invalidate Project-scoped
  authorization state on each Canister before it becomes available.
- Install and verify fresh Fleet-scoped delegation proofs before auth resumes.
- Qualify restart at every durable journal boundary.
- Complete a final current-only cascade before returning the Fleet to service.
- Prove that transition/readiness/recovery host reads use only unchanged wire
  surfaces or journal-confirmed current targets.

### Slice 8 — Closeout

- Run repository-wide current-surface searches for stale terminology.
- Prove the one-subnet behavior and stable ownership are unchanged.
- Update active architecture, configuration, CLI and operator documentation.
- Record downstream migrations without editing downstream repositories.

## Required Validation

### Configuration

- `[app].name` parses and validates with the existing canonical name rules.
- Missing `[app].name` fails with a typed configuration error.
- `[directories.fleet].roles` retains current role validation.
- Each selected role has one Fleet-authoritative Directory identity, without
  claiming 0.100 placement or replication behavior;
- generated and diagnostic surfaces use `SubnetSlotId`, not `SubnetRole`.
- `subnets.default` is required and drives the maintained one-member workload
  placement.
- keyed placement `binding` preserves the existing placement behavior.
- generated config and scaffold output contain only current names.

### Identity and CLI

- App discovery selects `apps/<app>/canic.toml`.
- duplicate App identities fail deterministically.
- build provenance binds App and role.
- the operator-supplied Fleet ID is validated and immutable.
- the trusted network resolver distinguishes local, staging and production
  networks without treating environment labels or `BuildNetwork` as canonical
  identity.
- live Fleet state binds Fleet, App and canonical IC network in host, current
  root and every managed Canister's protected state.
- public-IC live `NetworkDefault` is specified as the Fiduciary profile,
  local/test default is harness-provided, and explicit/profile selection has
  no silent random fallback.
- every later operation rejects an App or network mismatch before mutation.
- two environment labels for the same network resolve the same Fleet namespace.
- two Fleets may reference the same App without colliding in local state.
- the same Fleet ID may exist on different canonical IC networks without
  collision.
- the same Fleet ID and Canister principal on different networks produce
  different `FleetCanisterId` values and evidence hashes.
- CLI help distinguishes App config commands, Fleet live-state commands and
  deployment operations.
- JSON output contains no ambiguous `fleet_template` or `deployment_name`
  identity fields.

### Host-State Migration

- existing deployment state migrates to Fleet state without losing receipts or
  installed evidence;
- path, Fleet, App or canonical-network collisions fail before authority moves;
- interruption at every journal boundary resumes idempotently;
- immutable evidence bytes and hashes are unchanged by migration;
- a completed migration leaves no ordinary reader of deployment paths; and
- deliberate abandonment requires the explicit recovery action and reports
  uncatalogued evidence.

### Runtime and Stable State

- the current root remains the root of the sole member Subnet, and its hosting
  Subnet is described as the co-located Authority Subnet;
- the Fleet registry contains the same current subnet/root facts as the renamed
  current registry;
- Fleet and Subnet directories contain the same entries as the renamed
  current indexes;
- Fleet Directory entries selected by `[directories.fleet].roles` are
  understood as the Fleet-authoritative instances for those roles;
- every Directory cascade input and query page carries the expected immutable
  Fleet binding and source-root provenance;
- a Directory from another network, Fleet, App or source root is rejected;
- persisted one-member placement identifies `subnets.default`; no current
  state, Candid value or generated surface emits the removed `prime` slot;
- Fleet state has the same mode, status and funding behavior as current App
  state;
- a Canister retaining its principal through a Subnet move retains the same
  `FleetCanisterId` while its observed location and owning registry change;
- binding placement preserves bind, pending, retry and recovery behavior;
- pure source renames preserve the existing stable bytes, allocation IDs and
  stable key literals;
- additive Fleet binding, reserved slot identity and Directory provenance
  state migrate once with only current readers and writers; and
- removed endpoints and state-contract names have no compatibility aliases.

### Upgrade Journey

- a fresh 0.99 one-Subnet Fleet installs and completes a current-only cascade;
- a populated 0.98 one-Subnet Fleet upgrades through the quiesced coordinated
  sequence without losing stable or host evidence;
- interruption before and after the current root upgrade resumes to the same
  completed state;
- each upgraded Canister rejects Fleet-binding overwrite or mismatch;
- old Project-scoped delegation records, proofs, tokens, caches and replay
  state cannot authorize after the cut;
- fresh Fleet-scoped credentials authorize only within the exact
  `FleetKey`;
- `canic_sync_state` uses only the current record labels after the root
  upgrade, while `canic_sync_topology` retains its existing wire shape;
- the exact `canic_subnet_registry` endpoint and response remain usable by
  transition, readiness, diagnostics and recovery code throughout mixed state;
- no mixed-version state is exposed as an operational Fleet; and
- removed endpoints are never required after the corresponding Canister moves
  to 0.99.

For a design-only commit, Markdown, link, whitespace and focused documentation
checks are sufficient. Implementation closeout is a separate gate: it requires
the maintainer-owned full workspace validation plus the focused one-Subnet
PocketIC fresh-install, 0.98-to-0.99 upgrade and interruption journeys above.
Automated agents must still follow repository governance about who runs broad
gates; that ownership does not reduce the release evidence required here.

## Hard-Cut Completion Criteria

0.99 is complete when:

1. Active Canic surfaces consistently use `App` for Toko-like source identity.
2. Active Canic surfaces consistently use `Fleet` for one live deployed App
   over one or more Subnets.
3. The current one-root installation is represented truthfully as a
   one-member Fleet whose current root acts as both member root and Fleet
   Coordinator on the co-located Authority Subnet.
4. Every concrete Canister has a stable
   `(CanonicalNetworkId, FleetId, principal)` identity independent of its
   observed Subnet, without relying on role uniqueness or ambient network
   context.
5. Registry means complete authoritative records within its scope.
6. Directory means a derived, read-only, consumer-shaped projection that is
   cascaded and queried with explicit Fleet/source-root provenance.
7. Index is absent from public topology terminology.
8. Keyed placement uses Placement Binding terminology rather than Directory
   terminology.
9. `canic.toml` identifies the App through `[app].name` and contains no live
   Fleet identity or fabricated runtime membership.
10. App-declared logical topology uses `SubnetSlotId`; `SubnetId` is reserved
    for resolved platform identity, and `subnets.default` carries no authority
    meaning.
11. Initial installation or coordinated activation binds immutable Fleet, App
    and canonical-network identity in host state and protected state on every
    managed Canister, and later operations enforce it.
12. Delegation audiences are Fleet-scoped, with no accepted Project variant or
    credential after the cut.
13. Existing host state and evidence survive the explicit resumable migration
    to Fleet paths.
14. A coordinated, interruption-safe one-Subnet upgrade completes without
    treating mixed 0.98/0.99 state as operational, and its host recovery reads
    use only unchanged wire surfaces or journal-confirmed current targets.
15. No compatibility parser, endpoint, Candid field, delegation variant, path
    or JSON field preserves the old
    terminology in normal operation.
16. Every active distinguished-subnet surface says Authority Subnet rather than
    Prime Subnet.
17. `[directories.fleet].roles` declares Fleet-authoritative service
    discovery without claiming application-data replication.
18. No multi-subnet runtime capability is claimed or implemented by this
    terminology release.

## Follow-On Multi-Subnet Design

After 0.99, the
[Canic 0.100 multi-subnet design](../0.100-multi-subnet/0.100-design.md) can
implement the Fleet behavior that the original 0.99 draft attempted to include:

~~~text
0.99
  “What is an App, Fleet, Subnet, Canister, Registry and Directory?”

Follow-on Fleet membership
  “How does the Fleet Coordinator register, activate, drain and remove member
   Subnets?”

Follow-on synchronization
  “How do member roots mirror the Fleet Registry and cascade the Fleet
   Directory?”

Follow-on placement
  “How are declared authority services provisioned on the Authority Subnet,
   and which member-local replicas are provisioned elsewhere?”
~~~

That follow-on design must address bounded snapshots, revision and freshness
evidence, interruption, root-only inter-subnet calls, durable mirror
activation, lifecycle transitions and PocketIC multi-subnet validation. None
of those behaviors are part of 0.99; 0.99 carries only immutable one-subnet
Directory source provenance.
