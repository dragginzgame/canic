# 0.60 Design: NNS Subnet Inspection

## Status

- Status: Cached catalog, live mainnet refresh, and catalog-derived
  instruction-audit estimate source implemented; custom-network refresh
  deferred
- Line: `0.60.x`
- Theme: host-side IC NNS subnet catalog, inspection CLI, and optional estimate
  resolver source

0.60 is the NNS inspection line. It adds operator-visible IC network subnet
metadata without changing canister runtime perf accounting, cycle funding, or
deployment mutation semantics.

0.59 owns the instruction/cycle terminology hard cut and offline estimate
math:

```text
docs/design/0.59-instruction-accounting-cost-estimates/0.59-design.md
```

0.60 owns the work intentionally deferred from 0.59:

- local NNS subnet catalog;
- cached canister-ID-to-IC-subnet resolution;
- `canic subnet catalog list`;
- `canic subnet catalog info`;
- `canic subnet catalog refresh` once the shared protobuf registry adapter is
  introduced;
- stale-cache policy and registry provenance;
- optional use of the catalog as an estimate source.

## Problem

Operators need a local, scriptable way to inspect where deployed IC canisters
live in the IC subnet topology and what subnet metadata affects pricing
interpretation. That is a real operator feature, not a prerequisite for the
0.59 instruction audit.

The feature must keep three boundaries clear:

- IC network subnets from the NNS registry are not Canic logical fleet subnets.
- Cached NNS metadata is operator observation state, not deployment-owned
  truth.
- NNS metadata can decorate cost estimates, but it must not change measured
  instruction rows or runtime perf counters.

## Relationship To 0.59

0.59 estimates require explicit offline inputs:

```text
--estimate-node-count <n>
--cycles-per-billion-instructions <cycles>
```

0.60 adds a third source after the cached inspection surface is stable:

```text
subnet_source = "nns-registry-cache"
bash scripts/ci/instruction-audit-report.sh --estimate-execution-cycles --estimate-canister-principal <canister-principal>
```

That source is optional. Instruction audits without estimates remain
network-free. Estimate generation must not perform live NNS lookups by default;
it may read a fresh local catalog, refresh only when explicitly requested, or
fall back to explicit flags.

## In Scope

- Host-side local catalog of IC NNS subnet metadata.
- Cached routing-range resolver from canister principal to IC subnet principal.
- Subnet metadata records with node count, subnet kind, curated specialization,
  geographic scope, and charge applicability.
- Read-only `canic subnet catalog list` and
  `canic subnet catalog info <x>` commands.
- Explicit `canic subnet catalog refresh` command for live mainnet registry
  reads after the shared registry adapter lands.
- Atomic catalog writes with previous-cache preservation on refresh failure.
- Deterministic stale-cache policy.
- JSON and text output suitable for operator scripts.
- Optional integration with instruction-audit `execution_cycle_estimate`
  artifacts.

## Out Of Scope

- Canister runtime calls to the NNS registry for estimate decoration.
- Candid as the raw NNS registry transport escape hatch.
- Exact cycle-bill reconstruction.
- Deployment, controller, canister, cycle-balance, or Canic runtime mutation.
- A new `canic cycles ...` profiling namespace.
- Custom-network registry refresh in the first 0.60 release. Custom networks
  continue to use explicit rate inputs until a separate custom-registry design
  exists.
- Treating cached NNS metadata as authority for deployment truth.

## Shared Crate Boundaries

The raw NNS registry refresh path is protobuf-based. The NNS registry transport
proto defines registry methods such as `get_latest_version` and `get_value`,
and subnet records are stored as protobuf registry values. Canic accepts that
dependency only behind a shared registry adapter. The catalog model and CLI
surface remain protobuf-free.

0.60 uses this crate split:

```text
crates/canic-subnet-catalog
crates/canic-ic-registry
crates/canic-host
crates/canic-cli
```

`crates/canic-subnet-catalog` owns the pure domain model:

- `SubnetCatalog`;
- `SubnetInfo`;
- `RoutingRange`;
- `SubnetKind`;
- `SubnetSpecialization`;
- schema versioning;
- serde JSON fixtures;
- cached range resolution over canonical principal bytes.

It must not depend on `ic-agent`, `prost`, generated protobuf types, or live
registry transport code.

`crates/canic-ic-registry` owns live mainnet registry access:

- NNS registry transport calls;
- protobuf request/response encoding;
- protobuf registry value decoding;
- NNS registry key construction;
- conversion into `canic-subnet-catalog` structs.

No protobuf-generated type may cross the `canic-ic-registry` public boundary.
The public API returns Canic-owned catalog structs only.

`crates/canic-host` uses catalog interfaces only:

- cache path resolution;
- cache load/store;
- stale-cache policy;
- refresh orchestration and atomic writes;
- operator-facing report preparation.

It must not directly depend on protobuf-generated registry types. Shelling out
to `icp` is acceptable only as a spike or comparison tool, not as the primary
live refresh implementation.

## Catalog Location

0.60 supports the mainnet `ic` catalog only until a later custom-registry
design exists. The default Canic cache root is the selected ICP root's
`.canic` directory:

```text
<canic-cache-root> = <icp_root>/.canic
```

The mainnet subnet catalog lives outside deployment-owned state:

```text
<canic-cache-root>/subnet-catalog/ic/catalog.json
```

It must not live under:

```text
<icp_root>/.canic/ic/deployments/
<icp_root>/.canic/ic/fleets/
```

The catalog is host/operator observation state. Backup/restore and deployment
truth flows may read it in future slices, but they must not require it as
deployment authority.

A future custom-registry slice must introduce a network-scoped catalog path,
for example:

```text
<canic-cache-root>/subnet-catalog/<network-slug>/catalog.json
```

It must not reuse the mainnet `ic` catalog.

## Catalog Shape

Catalog records must include a schema version. Canic must reject unknown future
catalog schema versions unless a migration exists.

```text
catalog_schema_version
network = "ic"
registry_canister_id = "rwlgt-iiaaa-aaaaa-aaaaa-cai"
registry_version
fetched_at
fetched_by
source_endpoint
resolver_backend
subnets[] = {
  subnet_principal
  subnet_kind
  subnet_kind_source
  subnet_specialization
  subnet_specialization_source
  geographic_scope
  geographic_scope_source
  subnet_label
  subnet_label_source
  node_count
  charges_apply_by_default
}
routing_ranges[] = {
  start_canister_id
  end_canister_id
  subnet_principal
}
```

`subnet_kind` values:

```text
application
system
unknown
```

`subnet_specialization` values:

```text
none
fiduciary
european
unknown
```

`geographic_scope` values:

```text
global
europe
unknown
```

Classification source values:

```text
registry
curated
computed
unknown
```

`subnet_label_source` uses `registry`, `curated`, or `unknown`.

Canic derives base subnet kind from registry data when available. Canic derives
user-facing specializations such as `fiduciary` and `european` from a curated
mainnet annotation overlay when the registry record does not carry that
classification directly.

Each curated annotation must include:

```text
subnet_principal
specialization
label
source
checked_at
reason
```

Initial curated annotations are checked on 2026-06-04 from the ICP subnet type
docs / ICP dashboard:

```text
fiduciary: pzp6e-ekpqk-3c5x7-2h6so-njoeq-mt45d-h3h6c-q3mxf-vpeq5-fk5o7-yae
european: bkfrj-6k62g-dycql-7h53p-atvkj-zg4to-gaogh-netha-ptybj-ntsgw-rqe
```

European subnets are application-kind charged subnets with
`subnet_specialization = european` and `geographic_scope = europe`, not a
separate billing type. This preserves their application pricing behavior while
still exposing the data-residency property operators care about.

The fiduciary subnet is an application-kind charged subnet with
`subnet_specialization = fiduciary`. Its cost multiplier comes from the current
registry-derived node count, not from a hard-coded permanent node count.

If Canic cannot derive `geographic_scope` from a trusted source, it records
`geographic_scope = unknown`. It must not infer geography from a principal
prefix unless that prefix mapping is an explicit curated annotation with
versioned provenance.

`node_count` is the count of subnet membership entries for the subnet at
`registry_version`, after applying Canic's registry decoding rules for active
replica membership. Do not infer node count from `subnet_kind`. Always use
registry membership when available.

## Stale Cache Policy

Default freshness:

```text
default_stale_after = 7d
```

Commands may override this with:

```text
--stale-after <duration>
--allow-stale-subnet-catalog
```

Behavior:

- `list` and `info` show stale status and can display stale cached data.
- Estimate-producing commands omit catalog-derived estimates by default when
  the catalog is stale.
- `--allow-stale-subnet-catalog` permits estimates from stale data and records
  `catalog_stale = true`.
- Once live refresh lands, `canic subnet catalog refresh` updates the cache
  before later list/info/estimate reads.
- A future strict flag may fail instead of omitting estimates, but omission is
  the default safe behavior.

`canic subnet catalog refresh` is introduced with live refresh after `0.60.1`.
Cached-only `0.60.0` and `0.60.1` commands must not advertise refresh flags.

## Resolver

The cached resolver maps:

```text
deployment target -> canister principal -> cached routing range -> IC subnet
principal -> cached node count/kind/specialization -> rate table
```

`canic subnet catalog info <x>` accepts:

1. an IC subnet principal;
2. a canister principal;
3. an existing Canic deployment target / canister name that resolves to one
   canister principal for the selected network.

Resolution is deterministic:

1. parse `<x>` as a principal;
2. if it equals a known `subnet_principal` in the local catalog, resolve it as
   a subnet;
3. otherwise treat the parsed principal as a canister principal and resolve it
   through the cached routing table;
4. if `<x>` is not a principal, resolve it as an existing deployment target;
5. route the resolved canister principal through the cached routing table;
6. if no cached routing range covers the canister principal, return
   `RouteNotFound` with catalog version information;
7. if a deployment target does not resolve to exactly one canister principal,
   return `TargetResolutionFailed`.

If `<x>` parses as a principal, principal interpretation wins. If the parsed
principal is a known subnet, direct subnet lookup wins. Otherwise the principal
is treated as a canister principal.

Routing ranges are inclusive:

```text
start_canister_id <= principal <= end_canister_id
```

Range membership is computed over canonical principal bytes, not textual
principal ordering.

The command may accept:

```text
--as subnet
--as canister
```

to force interpretation in tests and in ambiguous synthetic-principal cases.
`--as subnet` fails unless `<x>` exactly matches a cached subnet principal.
`--as canister` fails unless `<x>` matches a cached routing range.

## Charge Applicability

Catalog inspection records `charges_apply_by_default` for subnet-level
metadata. `info` records the subject-specific fields:

```text
charges_apply_to_subject
charge_applicability_reason
```

`charge_applicability_reason` values:

```text
charged_user_canister_subnet
system_subnet_core_canister
system_subnet_unknown_subject
unknown_subnet_type
explicit_rate_override
not_resolved
```

For `subnet_kind = application`, `charges_apply_by_default = true`. For
`subnet_kind = system`, `charges_apply_by_default = false`. For
`subnet_kind = unknown`,
`charges_apply_by_default = false` and estimate decoration is omitted unless an
explicit operator override is used.

For charge applicability, `subnet_specialization = fiduciary` and
`subnet_specialization = european` both remain chargeable application-kind
subnets. Their node count and geography change interpretation, not whether
ordinary user canister execution charges apply.

System subnets host core IC infrastructure; user canisters cannot be created
there, and estimate decoration should use `charge_model = not_applicable`
unless an explicit operator override is used.

System subnet results must not blindly apply application-subnet pricing. Record
`subnet_kind = "system"` and mark charge applicability as not applicable where
cycles charges do not apply.

## Refresh

`canic subnet catalog refresh` is introduced after the cached inspection
surface. It performs the live mainnet NNS registry fetch and rewrites the
catalog atomically. Network failure leaves the previous catalog intact and
reports the failure clearly.

For `network = "ic"`, refresh uses the canonical NNS registry canister
`rwlgt-iiaaa-aaaaa-aaaaa-cai`. `source_endpoint` records the agent endpoint
used to reach mainnet; it is not the source of catalog authority. Mainnet
registry canister overrides are rejected unless an explicit unsafe/test flag is
supplied.

Refresh paths:

```text
catalog: <canic-cache-root>/subnet-catalog/ic/catalog.json
lock:    <canic-cache-root>/subnet-catalog/ic/refresh.lock
temp:    <canic-cache-root>/subnet-catalog/ic/catalog.json.tmp.<pid>
```

Refresh steps:

1. create `refresh.lock` with atomic `create_new` / `O_EXCL`;
2. if the lock exists and is fresh, fail with `RefreshAlreadyInProgress`;
3. if the lock exists and is stale, remove it and retry once;
4. query the canonical NNS registry canister;
5. fetch the current routing table and subnet records;
6. read subnet membership;
7. derive `node_count` from subnet membership entries at `registry_version`;
8. read subnet kind when available;
9. decode into an in-memory catalog;
10. apply the curated mainnet annotation overlay;
11. validate non-empty subnet list and routing table;
12. write `catalog.json.tmp.<pid>` in the catalog directory;
13. fsync the temp file when supported;
14. atomically rename the temp file to `catalog.json`;
15. fsync the parent directory when supported;
16. remove `refresh.lock`;
17. keep the previous catalog on any failure and remove abandoned temp files on
    a best-effort basis.

The lock file is JSON so tests can assert deterministic behavior:

```text
{
  "schema_version": 1,
  "network": "ic",
  "pid": 12345,
  "started_at_unix_ms": 1780590000000,
  "catalog_path": "<canic-cache-root>/subnet-catalog/ic/catalog.json"
}
```

The catalog lock must be released on every success and failure path.
Stale-lock tests use an injectable clock; do not rely on wall-clock sleeps.

Useful flags:

```text
--format text|json
--output <path>
--dry-run
--source-endpoint <url>
--lock-stale-after <duration>
```

`--output <path>` exports a copy of the fetched and decoded catalog. It does
not redirect the canonical cache path. A cache-path override would require a
future explicit `--cache-path` or test-only flag.

0.60 supports mainnet NNS refresh first. Custom networks require explicit rate
or explicit node count until a later custom-registry slice.

## CLI Surface

Do not add a public `canic cycles ...` profiling command for this line. That
namespace means real cycle balances, minting, transfer, top-up, and ICP
conversion.

Add a read-only subnet catalog namespace:

```text
canic subnet catalog list
canic subnet catalog info <subnet-principal|canister-principal|deployment-target>
canic subnet catalog refresh
```

The `catalog` segment is intentional because the command reads cached subnet
catalog data while `--network` remains the network selector:

```text
canic subnet catalog ...   # IC network subnets from the NNS registry
canic subnet fleet ...     # reserved for possible future Canic fleet topology
```

This namespace reads and refreshes IC network subnet metadata from the local
NNS subnet catalog. It never mutates deployments, canisters, controllers, cycle
balances, role attachments, or Canic runtime state.

`canic subnet catalog ...` is scoped to the mainnet IC subnet catalog in 0.60.
Network behavior is fixed:

```text
canic subnet catalog list
canic --network ic subnet catalog list
canic --network local subnet catalog list
```

- omitted `--network` is treated as `--network ic` for this command group only;
- `--network ic` runs normally;
- `--network local` or any non-`ic` network fails with a usage error.

Use this error shape:

```text

The cached subnet catalog describes the public Internet Computer mainnet.
Local replica subnet discovery is not implemented yet.

Try:
  canic --network ic subnet catalog list
```

Do not silently show mainnet catalog data under a local-network invocation.
`canic subnet catalog refresh`, once introduced, is also allowed only for
`--network ic`.

## Estimate Integration

Estimate source precedence is:

1. `--cycles-per-billion-instructions`;
2. `--estimate-node-count`;
3. fresh `nns-registry-cache` resolution;
4. stale `nns-registry-cache` resolution only with
   `--allow-stale-subnet-catalog`;
5. omit estimate.

Catalog-derived estimates support any positive trusted `node_count`.
Catalog-derived rates do not inherit the 0.59 explicit flag restriction to
`13|34`.

For chargeable application-kind subnets, Canic uses the 13-node application
subnet as the baseline and applies:

```text
scaled_cycles = ceil(base_13_node_cycles * node_count / 13)
```

Use integer/rational arithmetic only, never floating point. For the initial
instruction-component estimate:

```text
base_13_node_cycles_per_billion_instructions = 1_000_000_000
catalog_cycles_per_billion_instructions =
  ceil(base_13_node_cycles_per_billion_instructions * node_count / 13)
```

If `node_count` is missing, zero, or untrusted, Canic omits the
catalog-derived estimate. System subnets also omit catalog-derived estimates;
they must not return zero cycles pretending to be a user-deployment estimate.

The existing explicit sources remain stable:

- `--cycles-per-billion-instructions` accepts the operator-provided rate and
  wins over all other sources;
- `--estimate-node-count` remains the explicit 0.59-style table source unless
  a later release deliberately broadens that flag.

When the catalog is used as an estimate source, reports record:

```text
subnet_source = "nns-registry-cache"
registry_canister_id = <nns-registry-principal>
registry_version = <version>
subnet_principal = <resolved-subnet-principal>
subnet_node_count = <cached-node-count>
subnet_kind = <cached-subnet-kind>
subnet_kind_source = <cached-subnet-kind-source>
subnet_specialization = <cached-subnet-specialization>
subnet_specialization_source = <cached-subnet-specialization-source>
geographic_scope = <cached-geographic-scope>
geographic_scope_source = <cached-geographic-scope-source>
catalog_schema_version = <version>
catalog_stale = false
resolver_backend = "local-nns-subnet-catalog"
```

If the catalog is stale, missing, or cannot map the canister ID to a subnet,
the estimate should be omitted unless the caller supplies an explicit fallback
or asks Canic to refresh the catalog. The instruction report itself should
still complete.

Explicit rate and explicit node count remain valid. Explicit rate wins over
explicit node count and catalog-derived rate. Explicit node count wins over the
catalog. If an explicit rate or explicit node count wins, `subnet_source`
remains `explicit-rate` or `flag`, and catalog provenance is omitted unless the
caller explicitly asks for separate inspection metadata. Catalog-derived
provenance is recorded only when the catalog actually supplies the estimate
source.

## Layering

- `canic-core::perf` remains instruction-only instrumentation.
- Runtime metric DTOs remain instruction DTOs.
- Catalog DTOs and resolver output live in host/operator code, not canister
  runtime code.
- Live NNS calls live behind the shared `canic-ic-registry` adapter.
- CLI adapters parse commands and delegate immediately.
- Estimate decoration may read the catalog, but measured instruction rows do
  not depend on the catalog.

Host-side ownership stays split:

- `canic-subnet-catalog` owns pure catalog data, validation, schema-version
  rejection, JSON fixtures, and cached principal-byte routing resolution.
- `canic-ic-registry` owns raw registry calls, protobuf decoding,
  registry-version handling, and conversion into catalog structs.
- `canic-host` owns cache path resolution, stale-cache policy, refresh
  orchestration, atomic writes, and operator report preparation.
- `canic-cli` owns argument parsing, command delegation, and output rendering.

This keeps runtime layering intact and prevents a reporting feature from
creating canister-side registry dependencies.

## Build Order

1. Add catalog data shapes, parser/validator, schema-version rejection, and
   fixture tests in `canic-subnet-catalog`.
2. Add cached resolver over routing ranges and subnet records.
3. Add `canic subnet catalog list` and `info` over cached data.
4. Add stale-cache evaluation and output fields.
5. Add `canic-ic-registry` with protobuf registry transport and decoding.
6. Add live mainnet NNS refresh with lock and atomic write behavior.
7. Wire catalog-derived estimates as an optional source for instruction-audit
   reports.
8. Run an operator proof that refreshes, lists, resolves a known canister, and
   produces an estimate artifact without changing instruction rows.

- explicit rate and explicit node-count sources win over the catalog;

## Acceptance Criteria

- `canic subnet catalog list`, `info`, and `refresh` are read-only with respect
  to deployments, canisters, controllers, cycle balances, and runtime state.
- The catalog lives under `<canic-cache-root>/subnet-catalog/ic/catalog.json`
  and not under deployment-owned state.
- Mainnet refresh uses registry canister
  `rwlgt-iiaaa-aaaaa-aaaaa-cai`; `source_endpoint` does not override catalog
  authority.
- Live refresh protobuf types remain behind `canic-ic-registry`; no generated
  protobuf type crosses into host, CLI, or catalog APIs.
- Catalog records include `catalog_schema_version` and reject unknown future
  versions unless migrated.
- Catalog classification fields record provenance, and unknown geography stays
  `unknown`.
- `node_count` is derived from registry membership and is not inferred from
  subnet kind or specialization.
- `info <canister-principal>` resolves through cached routing ranges to the
  same subnet metadata the estimate resolver would use.
- `info <deployment-target>` resolves through the existing deployment target
  resolver to exactly one canister principal before cached routing lookup.
- Routing range resolution is inclusive and uses canonical principal-byte
  ordering.
- `canic subnet catalog ...` defaults to mainnet `ic`, accepts `--network ic`,
  and rejects every non-`ic` network in 0.60.
- Stale status is deterministic and visible in text and JSON output.
- Missing cache fails with an actionable error that points operators at
  `canic subnet catalog refresh` after refresh lands.
- Refresh failure keeps the previous catalog intact, including validation,
  write, and rename failures.
- Range output is capped by default in text mode.
- European subnets preserve application-pricing behavior while exposing
  `subnet_specialization = european` and `geographic_scope = europe`.
- Fiduciary subnet metadata is an application-kind subnet with
  `subnet_specialization = fiduciary`, and pricing uses the current
  registry-derived node count.
- System subnet results mark charge applicability as not applicable.
- Instruction reports remain complete when catalog lookup fails.
- Instruction reports without estimates do not read the catalog.
- Catalog-derived estimate artifacts record subnet, registry, catalog, and
  resolver provenance.

## Required Tests

- Catalog schema round-trips through stable JSON.
- Unknown future catalog schema version is rejected.
- Mainnet refresh uses the canonical registry canister ID.
- `source_endpoint` does not override registry authority.
- Custom network refresh is rejected when live refresh is introduced in
  `0.60.2`.
- Empty subnet list or empty routing table is rejected on refresh.
- Cached resolver maps a canister principal through routing ranges to subnet
  principal, node count, subnet kind, subnet specialization, and geographic
  scope.
- `info <x>` resolves `<x>` deterministically as subnet, canister, or unknown.
- `info <deployment-target>` resolves exactly one deployment target to a
  canister principal before cached routing lookup.
- ambiguous deployment targets fail with `TargetResolutionFailed`.
- Routing-range start boundary resolves.
- Routing-range end boundary resolves.
- Principals just before start and just after end do not resolve.
- Resolver uses principal-byte ordering, not textual ordering.
- `--as subnet` fails when the principal is only a canister-range match.
- `--as canister` fails when no routing range matches.
- omitted `--network` uses the mainnet `ic` catalog for `canic subnet catalog`.
- `--network ic` uses the mainnet `ic` catalog for `canic subnet catalog`.
- `--network local` and any other non-`ic` network fail with the pinned usage
  error.
- `list --format json` includes registry version, fetched-at, subnet
  principal, subnet kind, subnet specialization, classification sources,
  geographic scope, subnet label, node count, charge applicability, stale
  status, and rate metadata.
- `list` text output caps routing ranges unless `--show-ranges` is supplied.
- European subnet classification preserves `subnet_kind = application`.
- European subnet classification records
  `subnet_specialization = european`.
- European subnet classification records `geographic_scope = europe`.
- Fiduciary subnet classification preserves `subnet_kind = application`.
- Fiduciary subnet classification records
  `subnet_specialization = fiduciary`.
- Unknown geography records `geographic_scope = unknown`, not `global`.
- System subnet node count comes from registry membership.
- System subnet catalog-derived estimate is omitted unless an explicit
  override is supplied.
- Concurrent refresh attempts are serialized by a lock.
- `refresh` writes a temp file and preserves the previous catalog on failure.
- Validation, temp-file write, and rename failures preserve the previous
  catalog.
- Abandoned temp files are cleaned up on a best-effort basis.
- `--output` exports without changing the canonical cache path.
- Missing or stale catalog omits estimates unless explicit fallback or stale
  allowance is supplied.
- Explicit rate beats explicit node count and catalog.
- Explicit node count beats catalog.
- Fresh catalog is used only when no explicit source is provided.
- Stale catalog is omitted by default and used only with
  `--allow-stale-subnet-catalog`.
- Catalog-derived estimates accept arbitrary positive node counts, including
  13, 28, 31, and 34.
- A missing, zero, or untrusted catalog node count returns
  `EstimateUnavailable`, not a panic.
- 13-node catalog-derived estimates equal the 13-node baseline.
- 34-node catalog-derived estimates pin the chosen integer rounding behavior
  against the previous fiduciary expectation.
- Successful catalog-derived estimate records registry version, subnet
  principal, catalog schema version, and resolver backend.
- Failed live lookup leaves the instruction report complete.
- Instruction reports without estimates do not read the catalog.
- No `canic cycles ...` command is added for profiling or inspection.
