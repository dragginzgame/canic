# Canic 0.74 Design: Root-Managed Delegation Proof Renewal

## Status

- Status: draft design note
- Release line: 0.74
- Decision style: delegated-auth operational hardening
- Primary implementation target: root-managed renewal of issuer active
  delegation proofs before expiry.
- Current state: delegated-token issuer canisters can store and report an
  active root-certified delegation proof. Root can prepare, expose, and install
  proof batches, and 0.74 work-in-progress now has root renewal templates,
  scheduled attempt/batch state, scheduled proof retrieval, and a root renewal
  sweep timer. Scheduled installs now update issuer-level renewal state,
  renewal provisioners are persisted and authorized for scheduled work only,
  scheduled work is listed through a constrained bridge endpoint, renewal paths
  emit bounded delegated-auth metrics plus `Topic::Auth` logs, and
  `canic auth renewal run-once` can courier scheduled proof batches through the
  direct-query/install bridge. The 0.74.1 follow-up adds root-managed issuer
  renewal status output through `canic auth renewal status`. A scheduled-renewal
  PocketIC scenario now retains the full renewal bridge loop; issuer
  self-refresh and medic drift reporting remain deferred.

## 1. Summary

0.74 should turn delegated-token root proof provisioning from a manual
operator sequence into a root-managed renewal workflow.

The important constraint is that root cannot complete the current proof flow
entirely inside a timer or update call. Root proof assembly depends on the IC
data certificate for root certified data, and that certificate is available in
a direct query context, not in the root update that prepares the signature-map
leaf. Therefore the clean 0.74 model is:

```text
root timer         -> schedules and prepares due renewal batches
renewal bridge    -> performs the required direct root query for proof material
renewal bridge    -> submits the retrieved proof batch back to root
root              -> validates pending metadata and installs proofs on issuers
issuer            -> continues issuer-local delegated-token issuance
```

Root remains the authority for issuer selection, policy, requested
audience/grants, certificate TTL, retry state, and install validation. The
bridge is a mechanical certificate courier. Product frontends must not run
delegation-proof provisioning during normal auth.

## 2. Current Contract

The active delegated-token root proof flow is update-then-query-then-update:

```text
controller             -> root canic_prepare_delegation_proof_batch update
controller             -> root canic_get_delegation_proof_batch direct query
controller/provisioner -> root canic_get_delegation_renewal_proof_batch direct query
controller/provisioner -> root canic_install_delegation_proof_batch update
root                   -> issuer canic_install_active_delegation_proof update
caller/session         -> issuer canic_prepare_delegated_token update
caller/session         -> issuer canic_get_delegated_token query
```

The existing protocol already exposes the lifecycle data needed for renewal:

- root issuer policies include `refresh_after_ratio_bps`;
- prepared root delegation proof entries return `expires_at_ns` and
  `refresh_after_ns`;
- issuer status reports `Missing`, `Valid`, `RefreshNeeded`, and `Expired`;
- issuer-local delegated-token prepare fails when the active proof is missing,
  stale enough to outlive the requested token, or expired.

What is missing is root-owned orchestration. Current root startup only starts
log retention, cycle sampling, and pool scheduling. It does not start an auth
renewal service.

## 3. Goals

1. Keep delegated-token issuer active proofs fresh before expiry without
   product frontend involvement.
2. Make root the owner of renewal intent, scheduling, policy selection, batch
   preparation, install validation, and renewal status.
3. Preserve the current self-validating delegated-token verification contract.
4. Preserve the direct-query requirement for root proof retrieval.
5. Add a limited renewal bridge role that cannot choose arbitrary audiences,
   grants, issuers, or TTLs.
6. Keep renewal state bounded and recoverable across root upgrades.
7. Make renewal failures observable through root status, issuer status, medic,
   and CLI/operator output.
8. Prove the full loop under PocketIC: initial install, scheduled renewal,
   direct-query bridge retrieval, root install, and issuer-local token issuance
   after the original proof would have expired.

## 4. Non-Goals

0.74 does not:

1. Hide root proof retrieval behind issuer composite queries.
2. Add a single-call `mint_token` or normal-auth fresh-proof path.
3. Let product browser frontends provision root delegation proofs during login
   or upload flows.
4. Add expiry grace to certificates, delegated tokens, or sessions.
5. Change verifier trust from embedded self-validating token material to online
   root or issuer state.
6. Reintroduce root threshold ECDSA for `DelegationCert`.
7. Make issuer canisters decide their own root-certified audience or grants.
8. Require root to query every issuer's active-proof status during each sweep.
9. Prune root canister-signature map leaves unless a separate storage cleanup
   design owns that behavior.
10. Replace the existing explicit controller prepare/get/install endpoints.

## 5. Hard Constraints

### 5.1 Direct Query Certificate

Root can prepare canister-signature leaves during an update, but it cannot
assemble the root canister-signature proof material in that same update. The
proof assembly step needs the root data certificate from a direct root query.

0.74 must not work around this by:

1. doing root proof retrieval through issuer composite-query wrappers;
2. doing update-time proof assembly without a data certificate;
3. adding verifier fetch-on-verify behavior;
4. changing the token trust chain.

### 5.2 Strict Expiry

Expiry remains strict: `now_ns >= expires_at_ns` means expired. Renewal is
best effort before expiry, not a grace mechanism after expiry.

### 5.3 Bounded State

The current root provisioning bounds still apply unless this design explicitly
changes them:

- at most 64 issuers per prepare batch;
- at most 128 pending batches;
- at most 16 pending root delegation proofs per issuer;
- one-minute retrieval window for prepared proof metadata;
- cert TTL bounded by root issuer policy and
  `auth.delegated_tokens.max_ttl_secs`.

0.74 renewal must reuse these limits or define tighter renewal-specific limits.

## 6. Architecture

### 6.1 Root Renewal Templates

Existing root issuer policy says what an issuer is allowed to receive. It does
not say what root should automatically renew.

0.74 should add a separate root-managed renewal template per issuer:

```rust
RootIssuerRenewalTemplate {
    issuer_pid: Principal,
    enabled: bool,
    aud: DelegationAudience,
    grants: Vec<DelegatedRoleGrant>,
    cert_ttl_ns: u64,
}
```

The template must pass the existing root issuer policy checks before root stores
it and before each renewal prepare. This keeps policy and desired renewal shape
separate:

- `RootIssuerPolicy` answers "what is allowed?"
- `RootIssuerRenewalTemplate` answers "what should root keep installed?"

An issuer with a policy but no enabled renewal template remains manually
provisioned.

### 6.2 Root Renewal State

Root should store renewal state independently from pending proof metadata:

```rust
RootIssuerRenewalState {
    issuer_pid: Principal,
    template_fingerprint: [u8; 32],
    last_installed_cert_hash: Option<[u8; 32]>,
    last_installed_expires_at_ns: Option<u64>,
    last_installed_refresh_after_ns: Option<u64>,
    active_attempt_id: Option<[u8; 32]>,
    last_outcome: RootIssuerRenewalOutcome,
    consecutive_failures: u32,
    next_attempt_after_ns: u64,
    updated_at_ns: u64,
}
```

Root should make due decisions from this state, not from issuer queries on
every sweep. Root already knows the proof it installed successfully. Issuer
status remains the operational source for verifying what an issuer currently
has, but it should not be required for the timer to make progress.

Manual installs through the existing root install endpoint should update this
state when the installed proof matches an enabled renewal template. That keeps
manual repair and automated renewal from diverging.

### 6.3 Renewal Attempts

Decision: renewal attempts are issuer-level. A batch is only a transport
optimization for preparing, retrieving, and installing multiple issuer attempts
together.

Each scheduled prepare creates or reuses one issuer-level attempt per issuer:

```rust
RootIssuerRenewalAttempt {
    attempt_id: [u8; 32],
    issuer_pid: Principal,
    template_fingerprint: [u8; 32],
    batch_id: [u8; 32],
    proof_ref: RootDelegationProofBatchProofRef,
    status: RootIssuerRenewalAttemptStatus,
    prepared_at_ns: u64,
    retrieval_expires_at_ns: u64,
    install_deadline_ns: u64,
    prepared_cert_hash: [u8; 32],
    prepared_expires_at_ns: u64,
    prepared_refresh_after_ns: u64,
    failure: Option<RootDelegationRenewalFailure>,
}
```

Batch state is separate:

```rust
RootDelegationRenewalBatch {
    batch_id: [u8; 32],
    attempt_ids: Vec<[u8; 32]>,
    prepared_at_ns: u64,
    retrieval_expires_at_ns: u64,
}
```

Attempt ids should be deterministic for the same issuer/template/current-proof
window so repeated timer sweeps do not create duplicate pending work while a
retrieval window is still open.

Suggested hash input:

```text
"canic-root-renewal-attempt-v1" ||
issuer_pid ||
template_fingerprint ||
last_installed_cert_hash_or_zero ||
last_installed_refresh_after_ns_or_zero
```

If the retrieval window expires before the bridge retrieves proof material,
root marks the attempt expired, applies backoff, and schedules a fresh prepare
when allowed.

Install results are per issuer. Successful issuer installs update that issuer's
renewal state immediately. Failed issuer installs retain or create issuer-local
retry state. The batch-level outcome is derived from issuer outcomes and is not
authoritative. A bridge retry of the same batch is idempotent for already
installed issuer attempts.

### 6.4 Root Timer

Root startup should ensure the delegated-auth renewal timer is running when:

1. the canister is root;
2. delegated tokens are enabled;
3. at least one renewal template is enabled;
4. the build includes root canister-signature proof creation support.

The timer may stay cheap and idle when no enabled renewal templates exist.
Upserting the first enabled renewal template must call the same
`ensure_renewal_timer_started()` path used by post-upgrade startup. Deleting or
disabling the last enabled template may either stop the timer or leave the idle
timer running, but it must not require a restart or upgrade before newly added
templates can schedule renewal work.

The timer sweep should:

1. prune expired renewal attempts and pending proof metadata;
2. select issuers whose renewal template is enabled and whose state is due;
3. batch selected issuers up to the existing 64-entry prepare limit;
4. prepare root delegation proof batches using the existing root prepare ops;
5. store attempt metadata for bridge retrieval;
6. update status counters and next-attempt times.

Before preparing new renewal work, the sweep must expire old attempts and mark
or release pending proof metadata so a dead bridge cannot permanently consume
per-issuer or global pending quotas.

The current implementation expires stale active issuer attempts during the
root renewal sweep before selecting due templates. Prepared attempts become
`Expired` with `RetrievalExpired`; installing or retryable attempts become
`Expired` with `InstallDeadlineExpired`. The next prepare can then schedule
fresh work for that issuer.

Due criteria:

- no successful installed proof is known;
- `now_ns >= last_installed_refresh_after_ns`;
- the active attempt expired or failed and backoff permits retry;
- the template fingerprint changed.

The sweep should not block on issuer calls. Issuer calls happen only after the
bridge returns proof material through the root install update.

### 6.5 Renewal Bridge

The bridge can be implemented first as a Canic CLI/host/operator worker. It is
not product frontend logic.

Bridge loop:

```text
1. query root renewal status/work
2. choose prepared attempts whose retrieval window is still open
3. call root direct query to retrieve proof material
4. call root install update with the retrieved proof batch
5. report install outcomes
```

The bridge does not decide:

- issuer eligibility;
- audience;
- grants;
- certificate TTL;
- refresh threshold;
- retry/backoff policy.

Those decisions are root-owned and encoded in renewal templates, root issuer
policy, and renewal attempt state.

## 7. State Machine

Issuer renewal state uses one active attempt per issuer. Legal states:

```text
Idle
Due
Prepared
Retrieved
Installing
Installed
FailedRetryable
FailedTerminal
Disabled
Expired
```

`Retrieved` is optional implementation state. Root may skip persisting it if
retrieval is only observed when the bridge submits an install update.

Legal transitions:

| From | Trigger | To |
| --- | --- | --- |
| `Idle` | enabled template has no known installed proof | `Due` |
| `Idle` | `now_ns >= last_installed_refresh_after_ns` | `Due` |
| `Installed` | `now_ns >= last_installed_refresh_after_ns` | `Due` |
| `Installed` | template fingerprint changed | `Due` |
| `Installed` | `now_ns >= last_installed_expires_at_ns` | `Expired` |
| `Due` | timer prepares pending proof metadata | `Prepared` |
| `Prepared` | scheduled proof is retrieved, if root records retrieval | `Retrieved` |
| `Prepared` | matching install submitted before deadlines | `Installing` |
| `Retrieved` | matching install submitted before deadlines | `Installing` |
| `Prepared` | retrieval window expired | `FailedRetryable` |
| `Prepared` | install deadline expired | `FailedRetryable` |
| `Installing` | issuer install succeeds or is already installed | `Installed` |
| `Installing` | transient call failure | `FailedRetryable` |
| `Installing` | issuer rejects because proof is malformed or wrong authority | `FailedTerminal` |
| `FailedRetryable` | backoff permits retry and template still matches | `Due` |
| `FailedTerminal` | controller updates template or manually repairs state | `Due` |
| any | template disabled or deleted | `Disabled` |
| `Disabled` | template re-enabled | `Due` |

Idempotency rules:

1. Repeated timer sweeps must not create another active attempt while one
   issuer attempt is `Prepared`, `Retrieved`, or `Installing`.
2. Re-submitting the same scheduled install must return per-issuer idempotent
   outcomes for attempts already completed as `Installed`.
3. If a template fingerprint changes while an attempt is active, root must
   reject old-template installs with `TemplateChanged`.
4. Disabling a template clears `active_attempt_id` and stops future scheduling
   for that issuer. It does not delete the issuer's currently installed active
   proof.
5. `Expired` is an observable state and does not grant expiry grace. A later
   successful renewal moves the issuer to `Installed`.

## 8. Endpoint Surface

0.74 should keep the current explicit provisioning endpoints and add a small
root-managed renewal surface.

### 8.1 Root Admin

```text
canic_upsert_root_issuer_renewal_template update
```

Authorization: root controller only.

This endpoint mutates renewal templates. Upsert must validate enabled
templates against the existing root issuer policy and global delegated-token
TTL config before storing them. Disabled templates may be staged without a
registered issuer policy, but they do not schedule work.

0.74 starts with disable-as-stop semantics. A delete/reset endpoint can be
added later only if scheduler repair semantics need physical template removal.

When the scheduler slice lands, upserting the first enabled renewal template
must ensure the renewal timer is running. Disabling the last enabled template
may stop the timer or leave it idle.

### 8.2 Root Status

```text
canic_root_issuer_renewal_status query
```

Authorization: root controller. Renewal bridge status reads are expected to go
through CLI/medic surfaces rather than exposing proof material or template
administration to non-controller callers.

Status should include:

- issuer principal;
- template enabled state and fingerprint;
- last installed cert hash;
- last installed expiry and refresh threshold;
- active attempt id, batch id, status, proof ref, and deadlines;
- batch/attempt retrieval expiry;
- last outcome;
- consecutive failure count;
- next attempt timestamp;
- machine-readable readiness code.

This endpoint is operational status. It must not retrieve root proof material.

### 8.3 Root Renewal Provisioners

```text
canic_upsert_delegation_renewal_provisioner update
canic_delegation_renewal_provisioners query
```

Authorization: root controller only.

These endpoints manage the constrained renewal provisioner ACL. The stored
record is intentionally narrow: principal plus enabled state. A provisioner can
retrieve scheduled renewal proof material and submit scheduled installs, but it
cannot create renewal intent, change templates, prepare arbitrary proof
batches, or retrieve arbitrary proof refs.

### 8.4 Root Work Retrieval

Decision: 0.74 adds scheduled-work retrieval:

```text
canic_delegation_renewal_work query
canic_get_delegation_renewal_proof_batch query
```

`canic_delegation_renewal_work` lists only root-scheduled batches whose issuer
attempts are still prepared and inside their retrieval/install windows. It is
available to root controllers and enabled renewal provisioners. The response
contains batch ids, deadlines, attempt counts, and issuer attempt views, but no
arbitrary proof refs beyond the scheduled attempts root already owns.

The bridge supplies only a `batch_id`. Root looks up the scheduled issuer
attempt ids, resolves proof refs internally, and refuses arbitrary proof
references. This endpoint must still be a direct root query.

The generic `canic_get_delegation_proof_batch` endpoint remains the
controller/manual-provisioning surface only. Renewal provisioners may retrieve
only scheduled renewal attempts through
`canic_get_delegation_renewal_proof_batch`.

### 8.5 Root Install

The existing install endpoint should remain the install path:

```text
canic_install_delegation_proof_batch update
```

For 0.74, it accepts either a root controller or an enabled renewal
provisioner. Controllers may continue to use the manual install surface.
Non-controller provisioners may install only batches already marked as
root-scheduled renewal batches, and proof-level preflight rejects submissions
not bound to active scheduled attempts.

Root must continue to preflight every proof against pending metadata before
broadcasting issuer installs.

Install preflight for scheduled renewal work must verify:

1. submitted `batch_id` belongs to an active scheduled issuer attempt;
2. submitted proof hash matches pending metadata;
3. the attempt has not expired;
4. `install_deadline_ns` has not passed;
5. template fingerprint still matches, otherwise reject with `TemplateChanged`;
6. issuer still has an enabled template;
7. proof `expires_at_ns` is still useful at install time;
8. caller has `can_install_scheduled_proofs`;
9. renewal provisioner is installing scheduled work only.

## 9. Authorization

0.74 should introduce a narrow renewal provisioner authorization model instead
of requiring a hot controller identity for unattended renewal.

Implemented model:

```rust
RootRenewalProvisioner {
    principal: Principal,
    enabled: bool,
}
```

Controller-only remains valid for local development and manual repair. The
provisioner role exists so operators can run a daemon with only the permissions
needed to complete root-scheduled renewal work.

Provisioners must not be able to:

1. upsert root issuer policy;
2. upsert renewal templates;
3. prepare arbitrary proof batches;
4. retrieve arbitrary pending proof refs;
5. install batches not marked as active scheduled renewal attempts.

Local development may support controller-only bridge operation. Production-facing
0.74 acceptance requires the renewal provisioner ACL so unattended renewal does
not require a hot controller identity.

## 10. Threat Model and Security Invariants

Security invariants:

1. A renewal provisioner can only complete root-scheduled work.
2. A renewal provisioner cannot create renewal intent.
3. A renewal provisioner cannot widen audience, grants, issuer set, or TTL.
4. A renewal provisioner cannot retrieve arbitrary manual proof refs.
5. A renewal provisioner cannot install proofs not bound to an active scheduled
   attempt.
6. A retrieved proof batch is useless after its retrieval or install deadline.
7. Root preflight is repeated at install time and is not trusted from prepare
   time.
8. Issuer verification remains mandatory; root scheduling success is not enough.
9. The generic manual provisioning endpoints remain controller-only.

The bridge is intentionally untrusted for policy choices. It can observe
scheduled work, perform the direct-query certificate hop, and submit the exact
scheduled proof material back to root. Root and issuer validation must still be
able to reject stale, mismatched, expired, disabled, or policy-invalid proof
material.

## 11. Scheduling and Retry Policy

Default behavior:

1. Prepare when `now_ns >= refresh_after_ns`.
2. Prepare immediately when no successful installed proof is known.
3. Add deterministic jitter per issuer to avoid all issuers renewing in the
   same sweep.
4. Retry expired retrieval windows with exponential backoff capped below the
   remaining certificate lifetime.
5. Mark issuers `RenewalUrgent` when current time is within a configured lead
   window of `expires_at_ns`.
6. Mark issuers `Expired` when `now_ns >= expires_at_ns`; do not add grace.

Default timing:

- root sweep interval: 60 seconds;
- bridge daemon poll interval: 10 seconds;
- retrieval window: keep the current one-minute root proof retrieval window;
- recommended bridge interval: less than half the retrieval window;
- urgent lead window: at least
  `2 * (root_sweep_interval + bridge_poll_interval + expected_install_latency)`.

If cert TTLs are long enough, the urgent lead window may be rounded up to a
larger operator-friendly value. It must never be so small that one missed bridge
poll normally causes expiry.

Fleet selection should be deterministic and fair. If more than 64 issuers are
due in one sweep, root should order due issuers canonically, prepare at most one
batch, advance a persisted sweep cursor, and continue from that cursor on the
next sweep so later issuers are not starved by repeatedly due earlier issuers.

Backoff must never push `next_attempt_after_ns` past expiry when there is still
time for at least one more retrieval/install attempt. If the bridge is down
long enough for expiry, issuer-local token preparation fails until renewal
succeeds.

## 12. Failure Semantics

Root status should preserve the latest structured failure per issuer/attempt:

| Failure | Meaning |
| --- | --- |
| `PolicyRejected` | Template no longer satisfies root issuer policy or global TTL config. |
| `QuotaExceeded` | Pending batch or per-issuer proof quota is exhausted. |
| `RetrievalExpired` | Bridge did not retrieve proof material before retrieval expiry. |
| `InstallDeadlineExpired` | Bridge submitted proof material after the install deadline. |
| `ProofMismatch` | Submitted proof does not match pending root metadata. |
| `TemplateChanged` | Submitted proof belongs to an attempt prepared for an older template fingerprint. |
| `IssuerCallFailed` | Root could not call the issuer install endpoint. |
| `RejectedByIssuer` | Issuer rejected the active proof after local verification. |
| `AlreadyInstalled` | Install was idempotent for an already installed proof. |
| `TemplateDisabled` | Renewal is disabled for the issuer. |

`AlreadyInstalled` is not a failure for the bridge loop. It should clear or
complete the scheduled attempt when the matching proof is already active.

## 13. Observability, Metrics, and Logs

0.74 renewal must be visible in the same operator paths as the existing auth
and delegated-auth counters. Existing delegated-auth metrics already record
root proof prepare, issuer proof prepare, and token verification events. The
renewal workflow should extend that low-cardinality model instead of adding
caller-, subject-, or token-specific labels.

Required root metrics:

- renewal sweep started, completed, and failed;
- scheduled renewal proof retrieval completed and failed;
- renewal provisioner ACL changes;
- issuer renewal install completed;
- issuer renewal install rejected by root preflight;
- issuer renewal install rejected by issuer;
- issuer renewal install call failed;
- issuer renewal disabled or template changed.

Metric labels must be bounded:

- family: existing `delegated_auth`;
- operation: `renewal_sweep`, `renewal_proof_retrieve`, `renewal_install`,
  or `renewal_provisioner`;
- outcome: `started`, `completed`, or `failed`;
- reason: bounded delegated-auth reasons such as `ok`, `invalid_state`,
  `cert_expired`, `issuer_proof_unavailable`, `cert_hash_mismatch`, and
  `disabled`.

Metrics must not include caller principals, token subjects, full proofs,
audience strings, grant strings, operation ids, batch ids, attempt ids, or cert
hashes as labels. Principal- or hash-specific diagnostics belong in status DTOs,
CLI JSON, or logs, not metric labels.

Required `Topic::Auth` logs:

- `Warn` when the renewal timer is skipped because delegated-token auth is
  disabled;
- `Info` when a sweep prepares a batch, including issuer count and batch id;
- `Info` when scheduled proof retrieval succeeds;
- `Info` when a scheduled install completes for an issuer;
- `Warn` when scheduled install fails, including retryable/terminal state;
- `Warn` when an issuer reaches urgent or expired state;
- `Warn` when CLI/medic detects root-vs-issuer drift;
- `Info` when a template is enabled, disabled, or changed.

Logs may include root/issuer principal, batch id, attempt id, cert hash, expiry,
refresh threshold, and bounded reason codes. Logs must not include raw
signature CBOR, public-key DER blobs, delegated tokens, issuer proofs, root
proof payload bytes, token subjects, or application `ext` data.

The root renewal status endpoint remains the primary structured source of
per-issuer state. Metrics answer "how often and why"; logs answer "what just
happened"; status answers "what is root currently trying to keep installed."

## 14. Drift and Repair

Root renewal status is authoritative for scheduling. Issuer status is
observational.

Drift can happen when:

1. root recorded install success, but the issuer later upgraded or reinstalled
   and lost active proof state;
2. manual issuer-side intervention changed proof state;
3. root install returned success but issuer state did not persist as expected;
4. issuer reports a different active cert hash or expiry than root's
   `last_installed_cert_hash` and `last_installed_expires_at_ns`.

The root timer should not query issuers on every sweep and should not mutate
root state from observational issuer status. CLI and medic may query issuer
active-proof status opportunistically and compare it with root records. If they
observe a mismatch, they should report `DriftDetected` with both root-managed
and issuer-observed cert hash/expiry values.

Optional observational fields may be stored only when a CLI/host bridge
performs issuer comparison:

```rust
last_observed_issuer_cert_hash: Option<[u8; 32]>,
last_observed_issuer_expires_at_ns: Option<u64>,
last_observed_issuer_status_at_ns: Option<u64>,
```

These fields are not scheduling inputs. Repair requires an explicit bridge,
CLI, or root controller action.

## 15. Stable State and Upgrade

0.74 adds stable root-managed renewal records. The schema must support:

1. enabled templates;
2. provisioner authorization records;
3. per-issuer renewal state;
4. active renewal attempts;
5. bounded status/failure history.

Post-upgrade behavior:

1. restore renewal records synchronously with other auth state;
2. start the root renewal timer after environment and memory registry restore;
3. recover active attempts whose retrieval windows are still open;
4. mark expired attempts and retry according to policy;
5. never run async renewal work directly in lifecycle hooks.

Non-root issuer active proof records already persist installed proof material
and should remain compatible. Issuer post-upgrade does not self-refresh.

## 16. CLI and Host Worker

0.74 should add a local operator path that can complete the bridge loop without
a product frontend.

MVP CLI surface:

```text
canic auth renewal run-once <deployment> [--json]
canic auth renewal status <deployment> --issuer <principal> [--json]
```

Implemented `run-once`:

1. read root renewal work;
2. retrieve one or more prepared proof batches through direct root queries;
3. submit the retrieved Candid proof batch record back to root install without
   parsing or modifying proof internals;
4. print stable plain text or JSON outcome summaries.

The 0.74.1 status command queries the root controller status endpoint and
renders the root-managed per-issuer renewal view. It does not yet query issuers
or report root-vs-issuer drift.

`daemon` can be a follow-up if long-running process management is not wanted in
the 0.74 MVP. A packaged proof should at least show `run-once` can complete a
scheduled renewal batch from an installed local deployment.

`canic info medic` should surface passive diagnostics:

- root renewal timer configured/not configured;
- issuers missing templates;
- attempts stuck near retrieval expiry;
- issuers near proof expiry;
- root-vs-issuer drift when issuer comparison is available;
- bridge command hints.

Medic must not perform renewal.

## 17. Tests

Focused unit coverage:

1. renewal template validation against root issuer policy;
2. deterministic attempt id and duplicate-sweep suppression;
3. due/urgent/expired status calculations;
4. retry/backoff that does not skip remaining pre-expiry opportunities;
5. provisioner authorization rejects arbitrary prepare/get/install;
6. manual install updates renewal state when it matches a template;
7. expired retrieval windows are retried with fresh prepare metadata;
8. template changes during active attempts reject old-template installs with
   `TemplateChanged`;
9. first enabled template upsert starts scheduling without root restart;
10. renewal metrics use bounded operation/outcome/reason labels.

Protocol and macro coverage:

1. new root renewal endpoints are pinned in protocol tests;
2. renewal proof retrieval remains a query endpoint;

4. scheduled proof retrieval and scheduled install accept the narrow
   provisioner role only for scheduled work;

   issuer proof;

Docs and operator validation:

Each slice should preserve the current manual prepare/get/install flow.

## 19. Long-Term Watchpoints

0.74 is the right long-term renewal shape under the current canister-signature
trust model, but two follow-ups stay important:

1. Root canister-signature map leaf pruning. Automated renewal increases proof
   preparation churn. The current MVP bounds pending metadata but intentionally
   does not prune signature-map leaves. Before high-churn production renewal,
   a separate cleanup design should define safe leaf retention and certified
   data refresh behavior.
2. Bridge availability. CLI `run-once` is a sufficient MVP bridge and retained
   proof target. Hands-off production deployments should eventually use a
   supervised CLI daemon or `canic-host` worker so renewal does not depend on a
   human running the bridge at the right time.

## 20. Open Questions

1. Should 0.74 ship only CLI `auth renewal run-once`, or should `canic-host`
   also grow a managed bridge worker mode?
2. If `daemon` mode ships in the CLI, should it remain a foreground operator
   process, or integrate with any existing host process-supervision surface?

## 21. Acceptance Criteria

0.74 is complete when:

1. a root with an enabled renewal template prepares renewal work without manual
   prepare calls;
2. an authorized bridge can retrieve root proof material only through a direct
   root query;
3. the bridge can submit scheduled proofs and root installs them on issuers;
4. active issuer proof material is renewed before the original proof expires in
   PocketIC full-loop tests;
5. delegated-token issuance remains issuer-local after renewal;
6. missed renewal produces structured root/medic status and does not relax
   expiry;
7. product frontends are not part of the provisioning path;
8. the original explicit controller prepare/get/install flow still works;
9. a renewal provisioner cannot retrieve or install a manually prepared proof
   batch;
10. re-submitting the same scheduled install is idempotent and does not corrupt
    renewal state;
11. partial issuer install failure produces per-issuer outcomes and does not
    roll back successful issuer installs;
12. bridge outage cannot permanently exhaust pending renewal quotas after
    retrieval windows expire;
13. adding the first enabled template after root startup causes renewal
    scheduling without requiring upgrade or restart;
14. template changes during an active attempt are handled deterministically by
    rejecting stale installs with `TemplateChanged`;
15. CLI/medic reports root-vs-issuer drift without product frontend
    involvement;
16. root metrics expose renewal sweep, attempt, install, retry, expiry, and
    drift outcomes with bounded labels;
17. `Topic::Auth` logs capture renewal progress and failures without raw proof,
    token, or application payload material;
18. production-facing unattended renewal uses the renewal provisioner ACL, not a
    hot controller identity.
