# 0.39 Design: `ic-memory` Extraction Architecture

## Status

This document defines the 0.39 architecture for extracting `canic-memory` out
of Canic into a standalone repository named:

```text
ic-memory
```

`ic-memory` is persistent allocation-governance infrastructure for the
Internet Computer. It is not a Canic subsystem and it is not a convenience
wrapper around `ic-stable-structures`.

The extraction separates:

```text
persistent storage ABI governance
```

from:

```text
framework/runtime policy
```

Canic becomes one consumer of the generic allocation-governance layer.

---

## Core Invariant

The durable allocation invariant is:

```text
stable_key -> allocation_slot forever
```

The 0.38 Canic wording used:

```text
stable_key -> memory_id forever
```

That was correct for the current Canic implementation because Canic uses
`ic-stable-structures::MemoryManager` virtual IDs as its allocation substrate.
It is too narrow for a standalone ecosystem crate.

In `ic-memory`, a `memory_id` is only one possible substrate-specific slot
representation. The generic ABI model governs `allocation_slot` ownership.

### Allocation Slot

An `allocation_slot` is the durable physical allocation identity recorded in
the ledger. It identifies the storage region owned by a `stable_key`.

Examples:

```text
memory_manager:v1:0
memory_manager:v1:100
raw_partition:v1:ledger
stable_region:v1:app/users/primary
```

The slot must be:

* canonical
* serializable
* comparable
* stable across upgrades
* validated by policy
* interpreted by a storage substrate

The ledger does not assume that all slots are `u8` virtual memory IDs. A
`MemoryManager` virtual ID is represented as one slot kind:

```text
AllocationSlot::MemoryManagerId(u8)
```

Future substrates may introduce other slot kinds without changing the core
allocation invariant.

### ASD

`ASD` means:

```text
Allocation Slot Descriptor
```

The design uses `allocation_slot` for the conceptual invariant and
`AllocationSlotDescriptor` for the encoded record stored in the ledger or
passed through APIs.

An allocation slot descriptor contains:

* slot kind
* slot version
* canonical slot value
* optional substrate metadata

For current `MemoryManager` integrations, the descriptor is equivalent to:

```text
kind = memory_manager
version = 1
value = <virtual_memory_id>
```

## Architectural Boundary

`ic-memory` owns generic persistent allocation truth. Frameworks own policy and
runtime integration.

### Generic `ic-memory` Mechanics

These belong in the standalone crate:

* ledger persistence
* allocation history
* stable-key grammar
* allocation-slot descriptors
* tombstones
* reservations
* generation staging and commit
* rollback allocation validation
* declaration collection
* declaration snapshot sealing
* duplicate declaration rejection
* historical conflict rejection
* schema metadata persistence
* read-only diagnostics/export
* substrate abstraction
* allocation validation
* generation history

The generic crate enforces:

```text
historical allocation truth
```

It never infers historical ownership from the current binary alone.

### Policy-Owned Concerns

These are supplied by frameworks or applications through policy adapters:

* namespace rules
* allocation authority rules
* range ownership
* reserved slots
* framework/application separation
* framework-specific slot policies
* naming constraints beyond the generic grammar
* allocation capability rules
* destructive format permission rules
* migration authorization rules

For Canic, these are policy-layer rules:

```text
canic.* -> framework authority
MemoryManager slots 0-9 -> ic-memory internal governance
MemoryManager slots 10-99 -> Canic framework
other usable MemoryManager slots -> allowed unless reserved by policy
MemoryManager slot 255 -> invalid
```

They are not hardcoded generic `ic-memory` mechanics.

### Framework/Runtime-Owned Concerns

The generic crate does not own:

* controller authorization
* endpoint dispatch
* install lifecycle
* post-upgrade lifecycle
* runtime orchestration
* IC auth semantics
* user-facing endpoint APIs
* framework bootstrapping policy
* application lifecycle semantics
* schema migrations

Canic, IcyDB, or another framework integrates `ic-memory` into its own
lifecycle and decides who may call diagnostics, format, migration, or repair
operations.

### Current Canic Boundary After 0.39.16

Canic no longer maintains a live local allocation registry. Canic collects
declaration inputs and delegates generic allocation validation to `ic-memory`.
Runtime bootstrap validates declarations against the native `ic-memory` ledger,
publishes `ValidatedAllocations`, and only then allows stable-memory handles to
open.

The current Canic bootstrap order is:

```text
collect declarations
recover native ic-memory ledger
apply Canic policy
validate declarations with ic-memory
publish ValidatedAllocations
open stable-memory handles through the validated session
```

Generic allocation validation now comes from `ic-memory`:

* stable-key grammar
* schema metadata bounds
* `MemoryManager` slot shape and ID `255` rejection
* duplicate declaration stable keys and slots
* historical stable-key movement rejection
* physical slot reuse rejection
* retired/tombstone rejection when represented in the native ledger

Canic still owns:

* `canic.*` namespace policy
* `ic_memory.*` owner restrictions
* Canic framework reserved IDs and application collision rejection
* declaring-crate checks
* native ledger stable-memory IO adapter
* runtime lifecycle ordering and handle opening
* controller diagnostic DTO shaping

## Core Types

The extracted repository should expose a small allocation-governance model.
Names below are architectural names; implementation may use crate-local module
organization as long as the same boundaries are preserved.

```rust
pub struct StableKey(String);

pub enum AllocationSlot {
    MemoryManagerId(u8),
    NamedPartition(String),
    Custom {
        kind: String,
        version: u32,
        value: Vec<u8>,
    },
}

pub struct AllocationSlotDescriptor {
    pub slot: AllocationSlot,
    pub substrate: String,
    pub descriptor_version: u32,
}

pub struct SchemaMetadata {
    pub schema_version: Option<u32>,
    pub schema_fingerprint: Option<String>,
}

pub enum AllocationState {
    Reserved,
    Active,
    Retired,
}

pub struct AllocationDeclaration {
    pub stable_key: StableKey,
    pub slot: AllocationSlotDescriptor,
    pub label: Option<String>,
    pub schema: SchemaMetadata,
}

pub struct DeclarationSnapshot {
    pub declarations: Vec<AllocationDeclaration>,
    pub runtime_fingerprint: Option<String>,
}

pub struct ValidatedAllocations {
    pub generation: u64,
    pub declarations: Vec<AllocationDeclaration>,
}

pub struct AllocationSession<S: StorageSubstrate> {
    substrate: S,
    validated: ValidatedAllocations,
}

pub struct AllocationBootstrap<'store> {
    store: &'store mut LedgerCommitStore,
}

pub struct BootstrapCommit {
    pub ledger: AllocationLedger,
    pub validated: ValidatedAllocations,
}
```

The important rule is that `AllocationSession` is required to open allocation
slots. Declaration collection alone is not sufficient.

## Trait Architecture

### Storage Substrate

The storage substrate owns physical access to the ledger anchor and allocation
slots.

```rust
pub trait StorageSubstrate {
    type Slot;
    type LedgerMemory;
    type MemoryHandle;
    type Error;

    fn open_ledger(&self) -> Result<Self::LedgerMemory, Self::Error>;

    fn open_slot(
        &self,
        slot: &AllocationSlotDescriptor,
    ) -> Result<Self::MemoryHandle, Self::Error>;

    fn describe_slot(&self, slot: &Self::Slot) -> AllocationSlotDescriptor;
}
```

The generic crate calls `open_ledger()` before allocation validation because
the ledger is the source of historical truth. It must not call `open_slot()`
for application/framework allocations until validation succeeds and an
`AllocationSession` has been produced.

### Allocation Policy

Policy validates whether a key may claim a slot. It does not persist history;
history belongs to the ledger.

```rust
pub trait AllocationPolicy {
    type Error;

    fn validate_key(&self, key: &StableKey) -> Result<(), Self::Error>;

    fn validate_slot(
        &self,
        key: &StableKey,
        slot: &AllocationSlotDescriptor,
    ) -> Result<(), Self::Error>;

    fn validate_reserved_slot(
        &self,
        key: &StableKey,
        slot: &AllocationSlotDescriptor,
    ) -> Result<(), Self::Error>;
}
```

Canic implements this trait with its `canic.*` namespace and reserved `10-99`
framework range. Downstream application keys are not assigned to a named Canic
authority range; they are accepted when the slot is usable and not covered by a
reserved infrastructure range. Other frameworks provide their own policies.

### Namespace Authority

Namespace authority answers whether a policy domain owns a stable-key prefix.

```rust
pub trait NamespaceAuthority {
    fn owns(&self, key: &StableKey) -> bool;
}
```

Examples:

```text
canic.*       -> Canic framework policy
app.*         -> downstream application policy
icydb.*       -> IcyDB policy, if used standalone
```

The generic crate validates stable-key grammar. It does not decide which
namespace is framework-owned.

### Range Authority

Range authority is optional and substrate-specific. A substrate with numeric
slots may use it; a named-partition substrate may not.

```rust
pub trait RangeAuthority {
    type Error;

    fn validate_slot(
        &self,
        slot: &AllocationSlotDescriptor,
    ) -> Result<(), Self::Error>;
}
```

Range authority is policy. The ledger records the result as allocation history
only after validation succeeds.

## Stable-Key Grammar

`ic-memory` defines a generic stable-key grammar because key canonicalization is
part of the durable ABI.

```text
stable_key = segment("." segment)* ".v" nonzero_digit digit*
segment    = lowercase_ascii_letter (lowercase_ascii_letter | digit | "_")*
```

Rules:

* ASCII only
* lowercase only
* no whitespace
* no slashes
* no hyphens
* no empty segments
* no leading or trailing dots
* version suffix required
* version suffix must be nonzero
* maximum length: 128 bytes

The `.vN` suffix is the allocation ABI generation. It changes only when the
physical allocation is intentionally replaced.

`schema_version` is not the stable-key version. It is metadata for in-place
schema evolution within the same allocation.

## Ledger Architecture

`AllocationLedger` is the durable root of trust.

```rust
pub const CURRENT_LEDGER_SCHEMA_VERSION: u32 = 1;
pub const CURRENT_PHYSICAL_FORMAT_ID: u32 = 1;

pub struct AllocationLedger {
    pub ledger_schema_version: u32,
    pub physical_format_id: u32,
    pub current_generation: u64,
    pub allocation_history: AllocationHistory,
}

pub struct AllocationHistory {
    pub records: Vec<AllocationRecord>,
    pub generations: Vec<GenerationRecord>,
}

pub struct AllocationRecord {
    pub stable_key: StableKey,
    pub slot: AllocationSlotDescriptor,
    pub state: AllocationState,
    pub first_generation: u64,
    pub last_seen_generation: u64,
    pub retired_generation: Option<u64>,
    pub schema_history: Vec<SchemaMetadataRecord>,
}

pub struct GenerationRecord {
    pub generation: u64,
    pub parent_generation: Option<u64>,
    pub runtime_fingerprint: Option<String>,
    pub declaration_count: u32,
    pub committed_at: Option<u64>,
}

pub struct LedgerCompatibility {
    pub min_ledger_schema_version: u32,
    pub max_ledger_schema_version: u32,
    pub physical_format_id: u32,
}
```

The ledger records:

* stable-key to allocation-slot ownership
* reservations
* active allocations
* retired/tombstoned allocations
* schema metadata history
* generation history
* range/namespace authority facts when supplied by policy

Absence from the current declaration snapshot is not retirement. Retirement
requires an explicit operation or explicit retirement declaration and is
irreversible.

Reserved records are allocation facts, not comments. A matching runtime
declaration may transition a record from `reserved` to `active`, but it must
preserve the same stable key and allocation slot. A reservation cannot be
repurposed to a different future key.

Retirement is also an allocation fact. `AllocationRetirement` names both the
stable key and allocation slot, stages a new generation, and commits the
tombstone through the same protected commit protocol as allocations and
reservations.

Recovered ledgers are validated against `LedgerCompatibility` before they can
become authoritative state. Older or newer unsupported `ledger_schema_version`
values and unsupported `physical_format_id` values fail closed. Implementations
must reject incompatible ledgers before writing them to protected commit slots;
bootstrap must not silently rewrite unsupported ledger formats.

Recovered and staged ledgers are also validated for structural integrity before
use or commit. The generic crate rejects duplicate stable keys, duplicate
allocation slots, invalid retired-state metadata, schema history that is empty
or not generation-ordered, generation records that point beyond the current
generation, and parent-generation relationships that do not precede the child
generation. A malformed ledger is treated like corrupt storage: fail closed
without rewriting allocation history.

## Ledger Anchor

The ledger anchor is the substrate-defined location of the allocation ledger.
It is the only storage region the generic crate may open before allocation
validation.

```rust
pub struct LedgerAnchor {
    pub substrate: String,
    pub descriptor: AllocationSlotDescriptor,
}
```

The substrate exposes the anchor through `StorageSubstrate::open_ledger()`.

For current Canic-on-`MemoryManager` integration:

```text
substrate = "ic-stable-structures.memory_manager"
ledger_anchor = AllocationSlot::MemoryManagerId(0)
```

That is Canic adapter configuration, not a generic invariant.

`ic-memory` may still provide substrate-specific descriptor guards for known
substrates. For the current `MemoryManagerId(u8)` substrate, generic mechanics
validate only the descriptor shape:

* substrate marker is `ic-stable-structures.memory_manager`
* descriptor version is supported
* slot kind is `MemoryManagerId`
* IDs `0-254` are usable
* ID `255` is the invalid unallocated-bucket sentinel
* inclusive ID ranges must stay inside the usable `0-254` space

Namespace ownership, framework/application separation, and reserved
infrastructure ranges remain policy adapter responsibilities.

### MemoryManager Internal Governance Range

The current `MemoryManagerId(u8)` substrate adapter reserves a small concrete
internal governance range:

```text
0-9 ic-memory internal allocation-governance range
```

This prevents the standalone crate from being permanently constrained to one
physical memory structure for every future ledger, recovery, and diagnostic
need. The range is substrate-specific. It is not a generic requirement for all
future substrates.

Genesis records for the `MemoryManagerId(u8)` adapter are:

```text
0 ic_memory.ledger.v1              active
1 ic_memory.generation_log.v1      reserved
2 ic_memory.recovery_scratch.v1    reserved
3 ic_memory.diagnostics_index.v1   reserved
4 ic_memory.maintenance_journal.v1 reserved
5-9                                ic-memory internal expansion reserve
```

Reserved records are concrete ABI claims. A reserved slot may transition to
active only for the same stable key and slot. It may not be repurposed by a
framework policy or application policy.

For Canic, the framework range therefore begins at `10`:

```text
0-9     ic-memory internal
10-99   Canic framework
other usable IDs available unless a policy reserves them
255     invalid MemoryManager sentinel
```

Future substrates may use:

* a root slot
* a dedicated storage partition
* a named ledger region
* a reserved substrate descriptor
* an externally provided memory handle

The generic crate requires only that the anchor be deterministic, durable, and
available before other allocation slots are opened.

## Bootstrap Lifecycle

Declaration collection and allocation opening are separate phases.

The generic lifecycle is:

```text
collect declarations
seal declaration snapshot
open ledger anchor
load ledger
validate declarations against policy + history
stage generation
commit generation
publish validated allocation session
open allocation slots only through the session
```

`AllocationBootstrap` is the generic helper for this sequence. It recovers the
authoritative ledger through a `LedgerCommitStore`, validates the sealed
snapshot with `validate_allocations`, stages a logical generation, commits it
through `LedgerCodec`, and returns `BootstrapCommit` with validated allocations
tied to the committed generation.

The generic crate also provides an explicit genesis bootstrap path:

```rust
AllocationBootstrap::initialize_validate_and_commit(...)
```

This path may initialize the protected ledger store only when the physical
commit store is empty. The genesis ledger is supplied by the framework adapter;
`ic-memory` does not infer genesis records from runtime declarations. If commit
slots are present but corrupt, partially written, or undecodable, initialization
must fail closed instead of overwriting state.

Explicit reservation and retirement operations use the same bootstrap and
protected commit boundary. Reservation operations validate stable keys and
reserved-slot claims through `AllocationPolicy` before staging the reservation
generation. Retirement operations name the historical stable key and slot,
stage a tombstone generation, and commit it atomically. Authorization for these
operations remains framework policy; `ic-memory` enforces only policy adapter
results, historical allocation facts, and protected commit sequencing.

### Declaration Collection

Declarations are data only:

```rust
pub struct DeclarationCollector {
    declarations: Vec<AllocationDeclaration>,
}
```

Collectors may be populated by:

* macros
* generated schema manifests
* framework registration hooks
* explicit runtime bootstrap code
* static configuration

Collection must not open allocation slots.

### Snapshot Sealing

The framework seals collection into a `DeclarationSnapshot`.

During sealing, `ic-memory` rejects:

* duplicate stable keys
* duplicate allocation slots
* same key with different slot
* same slot with different key
* invalid stable-key grammar
* invalid schema metadata

Exact duplicate declarations in one snapshot are rejected. They are not
treated as idempotent because they can hide aliasing bugs.

### Validation

Validation compares the sealed snapshot against:

* generic ledger history
* framework policy
* namespace authority
* range authority, when configured
* substrate slot validity

Validation proves allocation ABI safety only. It does not prove schema
compatibility.

### Session Publication

After validation and generation commit, the crate returns:

```rust
pub struct AllocationSession<S: StorageSubstrate> { ... }
```

Slots can be opened only through the session:

```rust
impl<S: StorageSubstrate> AllocationSession<S> {
    pub fn open(
        &self,
        key: &StableKey,
    ) -> Result<S::MemoryHandle, AllocationError>;
}
```

If bootstrap fails, no validated session is produced and allocation opening
must fail closed.

## Generation Commit Semantics

Stable memory writes are not automatically rolled back on trap. Ledger commits
must therefore be atomic by design.

`ic-memory` supports generation-scoped mutation only. A new allocation,
reservation, tombstone, metadata update, or retirement becomes authoritative
only when its containing generation commits.

Implementations must use a protected commit protocol, such as:

* dual superblocks with generation number, pointer, length, checksum, and
  commit marker
* append-only generation records with checksummed commit footer

The authoritative generation pointer must itself be protected. A torn pointer
write must not be able to select a partially written generation.

Recovery rules:

* reader selects the highest valid committed generation
* partial staged generations are ignored
* corrupt protected commit metadata fails closed
* unsupported ledger schema versions fail closed
* unsupported physical format IDs fail closed
* malformed allocation-history structure fails closed
* empty protected commit state is distinguishable from corrupt state
* interrupted writes leave the previous committed generation authoritative

The generic crate defines commit semantics. The storage substrate supplies the
physical write operations needed to implement them.

The initial generic implementation models this as `LedgerCommitStore` over a
dual-slot physical commit record. `LedgerCodec` is integration supplied, so the
core owns commit ordering and recovery rules without owning a serialization
format or concrete stable-memory IO implementation.

## Rollback Semantics

Rollback validation is allocation-safety validation.

An older binary may run against a ledger containing newer active or retired
allocations if its current declarations do not conflict with historical
ownership.

Rules:

* undeclared historical records remain historically owned
* omitted records are not retired
* rollback never permits slot reuse
* rollback never erases tombstones
* rollback metadata remains visible for diagnostics
* schema compatibility is outside generic allocation validation

If an older binary passes allocation validation but cannot read a newer in-place
schema, the framework or store must fail before mutating that store.

## Schema Metadata

`SchemaMetadata` is persisted for diagnostics and compatibility tooling.

```rust
pub struct SchemaMetadata {
    pub schema_version: Option<u32>,
    pub schema_fingerprint: Option<String>,
}
```

0.39 rules:

* schema metadata does not define allocation identity
* schema metadata changes are not allocation conflicts
* schema metadata changes are recorded per generation
* schema migration hooks do not belong in the generic crate

Recommended fingerprint format:

```text
<algorithm>:<lowercase-hex-digest>
```

The generic crate validates bounded ASCII storage. It does not require one
hash algorithm in 0.39.

## Diagnostics and Export

`ic-memory` exposes read-only machine-readable diagnostics.

Diagnostics may include:

* ledger header
* ledger schema version
* physical format ID
* current committed generation
* allocation records
* tombstones
* reservations
* schema metadata history
* policy facts recorded in the ledger
* generation history
* recovery status
* protected commit slot presence, validity, and selected generation

`DiagnosticExport` may include protected commit recovery diagnostics alongside
the logical ledger records. This is still read-only generic metadata; the
framework decides whether and how to expose it.

The generic crate does not decide who may call diagnostics.

For Canic, controller-only diagnostics remain Canic runtime policy. Another
framework may expose diagnostics differently.

Diagnostic read paths may open the ledger anchor. They must not open
application/framework allocation slots and must not mutate allocation history.

---

## Dynamic Allocation

Dynamic allocation is allowed only as an explicit future ledger operation. It
is not endpoint-time implicit registration.

Dynamic allocation must:

* reserve a slot through policy
* stage an allocation generation
* commit through the protected generation protocol
* preserve historical ownership
* never reuse retired slots unless the policy and ledger explicitly support a
  future non-reuse-preserving slot class

0.39 does not implement automatic slot reclamation.

## Canic Hard-Cut Adoption

`ic-memory` introduces the generic durable allocation ledger. Canic 0.39.16
adopts the `ic-memory` native commit store as the authoritative persistence
path for allocation history.

Canic no longer owns or preserves the older Canic physical memory ledger
format. There is no compatibility projection bridge, no dual-read transitional
mode, and no automatic migration that reconstructs allocation truth from
current source declarations alone. Existing Canic physical ledger bytes are
rejected with an explicit hard-cut error unless a separate audited migration or
destructive format tool is designed later.

### Compatibility Model

Canic provides a policy/substrate adapter for the current hard-cut
`MemoryManagerId(u8)` allocation contract:

```text
ledger anchor: MemoryManager virtual ID 0
slot type: AllocationSlot::MemoryManagerId(u8)
policy: ic-memory internal -> 0-9, canic.* -> 10-99, app keys avoid reserved ranges, 255 invalid
```

The adapter is Canic-owned policy, not generic `ic-memory` behavior. The
0.39.7 proof kept this mapping in unpublished Canic test code first. Once
`ic-memory` is treated as publish-intended infrastructure, `canic-memory` may
consume generic validation mechanics directly while retaining Canic-specific
namespace and range authority in its adapter layer.

The adapter does not infer historical ownership from older Canic source
declarations. Runtime validation consumes the recovered `ic-memory` native
ledger plus the current sealed declaration snapshot.

### Layout Epochs

Canic layout epochs become Canic policy/runtime metadata. The generic
`ic-memory` ledger uses:

* `ledger_schema_version`
* `physical_format_id`
* generation numbers
* optional policy/runtime fingerprints

Changing Canic's layout epoch does not import, adopt, or reinterpret older
ledger state. Any future cross-format import would be a separate design outside
0.39.

* generic allocation governance in `ic-memory`
* Canic policy and lifecycle integration in Canic

## Non-Goals

0.39 does not:

* redesign `ic-stable-structures`
* run schema migrations
* reclaim retired slots
* manage endpoint lifecycle
* define framework namespaces
* manage controllers
* provide Canic-specific runtime APIs
* infer ownership from current declarations
* preserve implicit owner/label identity as a compatibility shim

## Implementation Readiness Checklist

Before implementation starts, the design is considered ready when the following
are true:

* `stable_key -> allocation_slot forever` is the only allocation invariant
* `MemoryManager` virtual IDs are modeled as one substrate-specific slot type
* ledger anchor behavior is explicit
* declaration collection and slot opening are separate phases
* `AllocationSession` is required before opening slots
* policy traits define namespace/range authority without hardcoding Canic
* diagnostics are generic and authorization-free
* Canic controller-only exposure remains Canic policy
* Canic hard-cut adoption does not provide compatibility migration or implicit
  import

This establishes `ic-memory` as persistent allocation-governance
infrastructure for long-lived Internet Computer systems.
