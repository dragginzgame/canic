# Canic Delegated Auth Lifecycle v1

- **Version:** `0.28`
- **Status:** `Draft for signoff`
- **Author:** `Canic Core`
- **Scope:** delegated-token proof issuance, verifier fanout, signer lifecycle prewarm, and verifier-local authentication

## 1. Context

Canic delegated auth uses a root-anchored trust chain:

```text
root canister -> signer shard -> user-bound delegated token -> verifier canister
```

The root canister signs delegation proofs for signer shards. Signer shards use
those proofs to issue user-bound delegated tokens. Verifier canisters authenticate
requests locally from cached proof and key material.

The important operational constraint is that verifier authentication is
intentionally local and fail-closed. If a verifier does not have the exact proof
needed by a token, authentication must fail. It must not fetch proof material
from the network during the auth decision.

`0.28` changed delegated auth from principal-list audiences to
`DelegationAudience`:

- `DelegationAudience::Any`
- `DelegationAudience::Roles(Vec<CanisterRole>)`

This moved the public intent from "push this proof to these principals" toward
"this proof is valid for these verifier roles." That creates a clearer authority
model, but only if root owns verifier target derivation and fanout success.

## 2. Problem Statement

The immediate bug class is a lifecycle gap:

1. A signer shard has a reusable local delegation proof.
2. A verifier is reinstalled, loses proof state, or is added later.
3. Signer issuance reuses the local proof.
4. Root does not re-push that proof to the verifier.
5. The verifier rejects otherwise valid tokens with a proof miss.

The broader design issue is authority split:

- caller-owned request fields historically described fanout targets
- root validated those targets, but did not fully own the derived fanout model
- signer lifecycle code had to know too much about verifier repair
- init and post-upgrade did not always use the same readiness path

The `0.28` design goal is to make root the authority for proof fanout while
keeping verifier auth local and deterministic.

## 3. Goals

1. Keep verifier authentication local, deterministic, and fail-closed.
2. Make `DelegationAudience` the canonical proof intent.
3. Make root derive verifier targets from audience and registry state.
4. Make root fail closed when required verifier fanout does not complete.
5. Make signer lifecycle prewarm re-publish current proof material even when a
   local proof is reusable.
6. Make non-root init and post-upgrade use the same signer auth bootstrap flow.
7. Remove demo-canister lifecycle hooks from the core repair model.
8. Keep explicit admin repair/prewarm as a root-controlled operator path.

## 4. Non-Goals

- No verifier-side network fetch during `authenticated(...)`.
- No background auth-time retry loop.
- No peer-to-peer proof repair.
- No demo canister behavior as normative system design.
- No public "force repair" API that application canisters must call from their
  own lifecycle hooks.
- No weakening of subject binding or endpoint scope checks.

## 5. Actors and Ownership

### Root

Root is the only authority that may issue delegation proofs.

Root owns:

- delegation cert issuance
- verifier target derivation from `DelegationAudience`
- proof fanout to verifier canisters
- root-local verifier cache writes for every issued delegation proof
- explicit admin repair/prewarm commands

Root must not trust caller-provided fanout targets for canonical issuance.

Root must always be capable of providing the delegated-auth services required by
the system. If delegated auth is enabled anywhere in the deployment, root config
must not be allowed to disable the root-side signing, verifier-cache, repair, or
fanout services that make delegated auth correct. Such config is invalid and
should fail fast.

### Signer shard

Signer shards issue user tokens after obtaining a root-signed proof.

Signer shards own:

- local proof cache for signing
- shard ECDSA public key lookup and caching
- token issuance and reissue
- lifecycle prewarm request to root

Signer shards should not own verifier topology decisions.

### Verifier canister

Verifier canisters authenticate incoming delegated tokens.

Verifier canisters own:

- local proof cache
- local root and shard public-key cache
- fail-closed token verification
- audience, scope, lifetime, signature, and subject binding checks

Verifier canisters must not repair themselves during an auth decision.

## 6. Canonical Data Shapes

### `DelegationAudience`

`DelegationAudience` is the proof and token audience intent.

```rust
pub enum DelegationAudience {
    Any,
    Roles(Vec<CanisterRole>),
}
```

Rules:

- `Any` means the audience is not restricted by role.
- `Roles(...)` means only configured verifier roles listed in the audience.
- `Roles(vec![])` is invalid.
- A role-scoped token audience must be a subset of the proof audience.

### `DelegationRequest`

Canonical signer-to-root issuance request:

```rust
pub struct DelegationRequest {
    pub shard_pid: Principal,
    pub scopes: Vec<String>,
    pub aud: DelegationAudience,
    pub ttl_secs: u64,
    pub shard_public_key_sec1: Vec<u8>,
    pub metadata: Option<RootRequestMetadata>,
}
```

Design rule:

- The request describes proof intent and signer key material.
- It must not carry caller-supplied verifier target principals.
- It must not carry a caller-controlled root-cache switch.
- Root derives verifier targets from `aud` and registry state.

### `DelegationCert`

Root-signed proof intent:

```rust
pub struct DelegationCert {
    pub root_pid: Principal,
    pub shard_pid: Principal,
    pub issued_at: u64,
    pub expires_at: u64,
    pub scopes: Vec<String>,
    pub aud: DelegationAudience,
}
```

The cert is root-signed and later used by verifier canisters to validate token
claims.

### `DelegatedTokenClaims`

Signer-issued user token claims:

```rust
pub struct DelegatedTokenClaims {
    pub sub: Principal,
    pub shard_pid: Principal,
    pub scopes: Vec<String>,
    pub aud: DelegationAudience,
    pub iat: u64,
    pub exp: u64,
    pub ext: Option<Vec<u8>>,
}
```

Rules:

- `sub` must match the IC caller at the verifier endpoint.
- `shard_pid` must match the delegation cert.
- `aud` must stay within the cert audience.
- `scopes` must stay within the cert scopes.
- token lifetime must stay within the cert lifetime.

## 7. Root Target Derivation

Root derives canonical verifier targets from:

1. `DelegationAudience`
2. root subnet registry
3. canister delegated-auth config
4. root pid and signer shard pid exclusions

Algorithm:

```text
derive_required_verifier_targets(aud, signer_pid, root_pid):
  if aud == Any:
    scan registry role index
    skip audience role filtering
    keep only canisters whose configured role is delegated_auth.verifier
    append all registered pids for those roles

  if aud == Roles(roles):
    reject empty roles
    for each role:
      require role config exists
      require delegated_auth.verifier == true
      append all registered pids for that role

  exclude signer_pid
  exclude root_pid
  dedupe while preserving deterministic order
```

Root authorization validates the audience is provisionable. Root execution uses
the same derivation to build fanout.

Design rule:

- `Any` does not care which verifier role a canister has.
- `Any` still only provisions currently registered verifier canisters.
- Verifier roles with zero registered canisters produce no fanout targets.
- Root-derived targets are required targets.
- Missing or failed installation for any required target must make the
  delegation request fail.

## 8. Canonical Issuance Flow

Signer-to-root issuance:

```text
signer
  -> build DelegationRequest { shard_pid, scopes, aud, ttl_secs, shard_public_key_sec1 }
  -> call root canic_request_delegation

root authorize
  -> require root context
  -> require caller == shard_pid
  -> require delegated auth enabled
  -> validate ttl, scopes, audience
  -> derive required verifier targets from audience and registry

root execute
  -> derive required verifier targets again from current registry
  -> build DelegationCert
  -> validate cert policy
  -> sign cert with root key
  -> push proof to every required verifier target
  -> cache root verifier proof locally
  -> fail if any required verifier target is missing or failed
  -> return DelegationProvisionResponse

signer
  -> verify returned proof
  -> store signer proof locally
  -> issue or reissue token
```

Important boundary:

- root owns required fanout and fanout success
- signer owns local proof storage and token signing

## 9. Signer Lifecycle Prewarm

Non-root signer lifecycle prewarm exists to close the verifier-cache lifecycle
gap.

Current prewarm plan:

```text
aud = DelegationAudience::Any
scopes = ["canic:verify"]
ttl = delegated_tokens.max_ttl_secs or 900 seconds
shard_pid = current canister
```

Behavior:

1. If delegated auth is disabled, do nothing.
2. If current canister is not a delegated signer, do nothing.
3. Build the signer prewarm plan.
4. If a local proof is reusable, do not return early.
5. Request root provisioning anyway.
6. Store the returned proof locally after verification.
7. Mark runtime ready only after bootstrap succeeds.

Why the reusable-proof early return is forbidden:

- Local signer proof reuse says the signer can still mint tokens.
- It says nothing about verifier proof cache coverage.
- Verifier cache repair requires a root fanout action.

Design rule:

- signer lifecycle prewarm always uses `DelegationAudience::Any`
- lifecycle prewarm is a cache-coverage repair path, not a per-token audience
  policy path
- future signer-role audience restrictions, if needed, must be modeled as
  root-enforced issuance policy instead of configurable lifecycle prewarm
  narrowing

## 10. Init and Post-Upgrade Semantics

Lifecycle hooks remain thin and synchronous at the IC hook boundary. They
schedule asynchronous bootstrap work using zero-delay timers.

Non-root init flow:

```text
init hook
  -> restore config/env/memory invariants
  -> schedule bootstrap_init_nonroot_canister

bootstrap_init_nonroot_canister
  -> bootstrap configured sharding/scaling work
  -> prewarm signer delegation proof
  -> mark ready
```

Non-root post-upgrade flow:

```text
post_upgrade hook
  -> restore config/env/memory invariants
  -> schedule bootstrap_post_upgrade_nonroot_canister

bootstrap_post_upgrade_nonroot_canister
  -> prewarm signer delegation proof
  -> mark ready
```

Design rule:

- init and post-upgrade share the same auth readiness behavior
- differences in placement bootstrap are explicit and unrelated to auth
- readiness must not be marked before required signer auth prewarm completes

## 11. Verifier Authentication Flow

Verifier auth must remain local:

```text
authenticated(required_scope)
  -> decode delegated token from request
  -> find exact proof in local verifier cache
  -> verify cert structure
  -> verify cert signature using cached root public key
  -> verify token signature using cached shard public key
  -> verify token lifetime
  -> verify claims.shard_pid == cert.shard_pid
  -> verify claims.aud subset of cert.aud
  -> verify claims.scopes subset of cert.scopes
  -> verify local verifier role is allowed by claims.aud
  -> verify claims.sub == IC caller
  -> verify required endpoint scope
```

If the exact proof is absent, authentication fails. This is expected and is the
reason lifecycle prewarm and explicit repair exist.

## 12. Explicit Admin Repair and Prewarm

Explicit repair/prewarm is a root-controlled operator path.

Use cases:

- verifier was reinstalled
- verifier cache was cleared
- verifier was added after a proof was issued
- operator wants targeted repair without waiting for signer lifecycle

Rules:

- root-only
- explicit target list is allowed because this is an admin/tooling path
- explicit admin repair/prewarm is principal-targeted only
- targets are normalized and validated before fanout
- repair requires the proof to already exist locally at root
- install intent is tracked as `Prewarm` or `Repair`

This path is not the canonical signer issuance path.

## 13. Failure Semantics

### Token verification

Verifier token verification fails closed.

Expected failure classes:

- proof miss
- proof mismatch
- invalid cert signature
- invalid token signature
- audience mismatch
- scope mismatch
- subject mismatch
- expired token or proof

No verifier auth failure should trigger unbounded network work.

### Delegation issuance

Canonical root issuance should fail if:

- delegated auth is disabled
- caller does not match `shard_pid`
- ttl is invalid
- scopes are empty
- audience is empty or not provisionable
- required verifier target derivation fails
- any required verifier proof push fails
- root proof signing fails

### Lifecycle prewarm

Signer prewarm failure prevents the bootstrap flow from marking the canister
ready. The lifecycle timer logs the error. The IC hook itself is not awaited.

## 14. Layering

Expected ownership by layer:

- `dto`: request/response wire shapes only
- `ops::auth`: pure auth helpers, signing, verification, key lookup/cache
- `ops::storage`: proof and key storage facades
- `workflow::auth`: root proof issuance and proof fanout orchestration
- `workflow::runtime::auth`: runtime/lifecycle auth bootstrap
- `workflow::rpc`: root request authorization, replay, and execution
- `api::auth`: public facade and local signer/verifier convenience surface
- endpoint macros: route calls and enforce root-caller guards

Policy decisions must not move into DTOs or endpoint macros.

## 15. Current 0.28 Cleanup State

Already done in the current working tree:

- signer prewarm no longer returns early for reusable local proof
- non-root init and post-upgrade both run signer auth prewarm before ready
- demo canister hooks are not part of the repair model
- `DelegationRequest` no longer carries caller-controlled verifier targets
- root derives verifier targets from `DelegationAudience` and registry
- replay hashing no longer includes caller-supplied verifier targets
- root execution fails closed when required verifier fanout is missing or failed
- signer-side verifier target derivation/validation has been removed
- `include_root_verifier` has been removed from canonical `DelegationRequest`
- root always caches its own verifier proof for issued delegation proofs
- unused `DelegationProvisionRequest` has been removed
- signer-target fanout has been removed from canonical `DelegationWorkflow::provision`
- verifier-target derivation now returns a typed error instead of a sentinel role
- stale RPC errors for caller-supplied verifier targets have been removed

Still to clean up before this design is fully implemented:

No remaining cleanup items are known for the signed-off `0.28` delegated auth
lifecycle design.

## 16. Signed-Off Decisions

The following decisions are signed off for `0.28`:

1. Root owns verifier target derivation for canonical issuance.
2. Root must fail the canonical delegation request if any root-derived verifier
   target fails installation.
3. Signer code should not independently derive or validate verifier fanout
   success after root owns fail-closed issuance.
4. Root always caches its own verifier proof for issued delegation proofs.
5. Root config that disables delegated-auth services required by an enabled
   delegated-auth deployment is a hard config error.
6. Root-local cache behavior is mandatory and not caller-controlled.
7. Explicit target lists remain valid only for root admin/tooling repair paths.
8. Demo canisters remain examples only and must not carry lifecycle repair
   responsibilities.

## 19. Compatibility

This design changes internal and public DTO shape during the unstable API phase.
Breaking changes are acceptable for `0.28` while the architecture is settling.

Downstream impact:

- canonical callers should pass `aud`, not verifier principal lists
- callers should not depend on caller-controlled root verifier caching
- verifier auth behavior remains fail-closed and local
- explicit admin repair remains root-controlled
