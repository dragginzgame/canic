# 0.42 Design: Authority Reconciliation

## Status

Tentative design notes for the 0.42 line.

0.42 builds on the 0.41 deployment truth model. Its job is to turn controller
policy from metadata into reconciled deployment authority.

This line should be framed as reconciliation, not simple application:

```text
Canic can prove controller state is correct or explain exactly why it cannot
make it correct.
```

---

## Scope Calibration

Release bar:

- dry-run authority reconciliation;
- exact external-action reports;
- controller overlap validation;
- destructive actions represented as unsafe or explicitly confirmable actions.

Design hook:

- imported, jointly controlled, user-controlled, and unsafe canister classes;
- per-spawn controller policy as a design object;
- observe/adopt planning for existing canisters without destructive reset;
- CLI profile and identity-binding semantics.

Deferred:

- full brownfield adoption UX;
- non-destructive pool registration workflow;
- operator-team RBAC product surface.

---

## Dependency On 0.41

0.42 assumes these 0.41 objects exist:

- `DeploymentPlanV1`;
- `DeploymentInventoryV1`;
- `DeploymentDiffV1` / `SafetyReportV1`;
- `DeploymentReceiptV1`;
- `CanisterControlClassV1`;
- role artifact and embedded config identity;
- phase receipts with verified postconditions.

0.42 should not invent a second drift model. It consumes 0.41 inventory and
diffs, then adds authority-specific reconciliation planning and post-apply
verification.

---

## Core Decision

`AuthorityProfile` becomes executable policy.

Tentative shape:

```text
AuthorityProfile {
  root_controllers,
  managed_canister_controllers,
  role_controller_overrides,
  staging_principals,
  emergency_controllers,
  break_glass_policy,
}
```

Important distinction:

```text
staging_principals and emergency_controllers are authority categories,
not normal IC controllers.
```

They must not overlap normal IC controller fields unless an explicit
break-glass flow is selected.

---

## Reconciliation States

For each canister, 0.42 should report one of:

```text
already_correct
can_apply_automatically
requires_external_action
unsafe_blocked
unknown
```

This is more useful than a generic apply failure.

Example report:

```text
root:
  desired controllers: [deploy-principal, ops-principal]
  observed controllers: [deploy-principal]
  action: add ops-principal
  can_apply: yes

root break-glass:
  emergency controller requested: emergency-principal
  state: requires_explicit_break_glass_flow
  can_apply: no in normal reconciliation

pool canister abc:
  desired controllers: [root, pool-manager]
  observed controllers: [unknown-user]
  action: external consent required
  can_apply: no
```

### AuthorityReconciliationPlanV1

```text
AuthorityReconciliationPlanV1 {
  schema_version,
  plan_id,
  inventory_id,
  authority_profile_hash,
  canister_actions,
  hard_failures,
  external_actions_required,
}
```

### CanisterAuthorityAction

```text
CanisterAuthorityAction {
  canister_id,
  role,
  control_classification,
  observed_controllers,
  desired_controllers,
  action,
  can_apply,
  reason,
}
```

Actions:

```text
none
add_controllers
remove_controllers
replace_controller_set
requires_external_controller
blocked_by_policy
unknown_observation
```

### AuthorityReceiptV1

Authority reconciliation should emit receipts with verified postconditions:

```text
AuthorityReceiptV1 {
  operation_id,
  reconciliation_plan_id,
  attempted_actions,
  verified_controller_observations,
  unresolved_external_actions,
}
```

## Controller Policy

### Root

Root controller reconciliation must be explicit.

Canic may:

- create root with intended controllers when possible;
- update root settings when the current operator/root has authority;
- report exact external action when it does not.

Root is the 0.40 protected-call trust anchor. If root changes, that is not a
normal resume. It is a new trust domain or a migration.

### Managed Canisters

Default managed canister controller rule:

```text
configured managed controllers
+ root
+ direct parent
```

Role-specific overrides may refine this rule, but the plan must show the final
desired controller set per canister.

### Pool Canisters

Pool canisters must be reconciled before they are reused.

Pool states:

```text
claimable
already_managed
requires_external_controller
unsafe_import
unknown
```

Rules:

- imported pool canisters are validated before reset/reuse;
- reused pool canisters are updated before install;
- unclaimable pool canisters are blocked before mutation;
- pool reset receipts must include observed controller postconditions.

### Break-Glass Authority

Emergency controllers are not normal controllers by accident.

A break-glass flow must be explicit, visible in the plan, and receipted.

The normal path should fail if emergency or staging principals overlap
ordinary IC controller sets.

---

## Commands

Tentative operator surface:

```text
canic deploy authority check <fleet>
canic deploy authority apply <fleet>
canic deploy authority diff <plan-or-receipt>
```

`check` is dry-run reconciliation.

`apply` runs only actions whose state is `can_apply_automatically`, then
re-inventories and verifies postconditions.

---

## Concurrent Operation Protection

0.42 should add or consume a deployment operation epoch.

Minimum requirement:

- authority reconciliation reads the latest inventory before applying;
- it records the previously observed deployment epoch;
- it verifies the deployment epoch after apply;
- it fails or warns if another operation changed authority state mid-run.

If root exposes a lightweight remote deployment lock by this line, authority
reconciliation should use it. If not, schema support from 0.41 remains
mandatory.

---

## Non-Goals

0.42 should not:

- introduce backend-agnostic execution;
- implement artifact promotion;
- implement user consent workflows;
- make deployment intent a scripting language;
- hide controller changes behind runtime config.

User-controlled lifecycle is deferred, but 0.42 must preserve canister control
classification so unknown or user-controlled canisters are not treated as
normal managed roles.

---

## Implementation Slices

### Slice 1: Authority Reconciliation Plan

Build authority reconciliation from 0.41 plan and inventory.

### Slice 2: Dry-Run Operator Report

Render per-canister controller state and external actions required.

### Slice 3: Root And Managed Apply

Apply controller changes only where Canic has authority.

### Slice 4: Pool Ownership Reconciliation

Validate and reconcile pool canisters before reset or reuse.

### Slice 5: Post-Apply Verification

Re-inventory and verify desired controller state after each applied change.

---

## Exit Criterion

0.42 is complete when:

```text
Canic can prove controller state is correct, apply the safe subset of missing
controller changes, and explain exactly which external action is required for
the rest.
```
