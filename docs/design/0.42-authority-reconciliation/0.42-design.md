# 0.42 Design: Authority Reconciliation

## Status

Active 0.42 design line. See
[`status.md`](status.md) for the implementation log.

0.42 builds on the 0.41 deployment truth model. Its job is to turn controller
policy from metadata into reconciled deployment authority.

This line should be framed as reconciliation, not simple application:

```text
Canic can prove controller state is correct or explain exactly why it cannot
make it correct.
```

---

## Scope Calibration

Release bar:

- dry-run authority reconciliation;
- exact external-action report.

The 0.42 minimum release is report-first: it produces dry-run authority
reconciliation and exact external-action reports. Applying controller changes
is optional or later unless explicitly promoted into the line.

Design hook:

- destructive pool import classified explicitly and requiring confirmation
  later;
- observe/adopt planning for existing canisters without destructive reset;
- imported, jointly controlled, user-controlled, and unsafe canister classes;
- per-spawn controller policy as a design object;
- CLI profile and identity-binding semantics as authority design.

Deferred:

- full brownfield adoption UX in 0.48;
- non-destructive pool registration workflow in 0.48;
- operator-team RBAC product surface.

---

## Dependency On 0.41

0.42 assumes these 0.41 objects exist:

- `DeploymentPlanV1`;
- `DeploymentInventoryV1`;
- `DeploymentDiffV1` / `SafetyReportV1`;
- `DeploymentReceiptV1`;
- `CanisterControlClassV1`;
- role artifact and embedded config identity;
- phase receipts with verified postconditions.

0.42 should not invent a second drift model. It consumes 0.41 inventory and
diffs, then adds authority-specific dry-run reconciliation planning, reports,
receipts, and evidence bundles.

0.42 consumes the passive identity facts that 0.41 actually exposes. It must
not require final promotion-grade target materialization identity or richer
role-scoped embedded-config digests; those remain execution-layer and
0.44-plus work.

## Core Decision

`AuthorityProfile` becomes executable policy.

Tentative shape:

```text
AuthorityProfile {
  root_controllers,
  managed_canister_controllers,
  role_controller_overrides,
  spawned_canister_controller_policy,
  staging_principals,
  emergency_controllers,
  break_glass_policy,
  operator_identity_bindings,
}
```

Important distinction:

```text
staging_principals and emergency_controllers are authority categories,
not normal IC controllers.
```

They must not overlap normal IC controller fields unless an explicit
break-glass flow is selected.

---

## Reconciliation States

For each canister, 0.42 should report one of:

```text
already_correct
can_apply_automatically
requires_external_action
unsafe_blocked
unknown
```

This is more useful than a generic apply failure.

Example report:

```text
root:
  desired controllers: [deploy-principal, ops-principal]
  observed controllers: [deploy-principal]
  action: add ops-principal
  can_apply: yes

root break-glass:
  emergency controller requested: emergency-principal
  state: requires_explicit_break_glass_flow
  can_apply: no in normal reconciliation

pool canister abc:
  desired controllers: [root, pool-manager]
  observed controllers: [unknown-user]
  action: external consent required
  can_apply: no
```

## Data Model

### AuthorityReconciliationPlanV1

```text
AuthorityReconciliationPlanV1 {
  schema_version,
  plan_id,
  inventory_id,
  authority_profile_hash,
  canister_actions,
  automatic_actions,
  hard_failures,
  external_actions_required,
}
```

### AuthorityReportV1

```text
AuthorityReportV1 {
  schema_version,
  report_id,
  reconciliation_plan_id,
  status,
  summary,
  counts,
  apply_readiness,
  action_counts,
  control_class_counts,
  observation_gaps,
  automatic_actions,
  hard_failures,
  external_actions_required,
  next_actions,
}
```

`apply_readiness` is report data only. It summarizes whether a future apply
path would be blocked by hard authority findings, missing observations, or
external authority actions. It does not perform controller mutation.

```text
AuthorityApplyReadinessV1 {
  can_apply_automatically,
  automatic_action_count,
  blockers,
}

AuthorityApplyBlockerV1 {
  hard_failures
  observation_gaps
  external_actions
}
```

### CanisterAuthorityAction

```text
CanisterAuthorityAction {
  canister_id,
  role,
  control_classification,
  observed_controllers,
  desired_controllers,
  action,
  can_apply,
  reason,
}
```

Actions:

```text
none
add_controllers
remove_controllers
replace_controller_set
requires_external_controller
requires_destructive_import_confirmation
observe_only
adopt_plan_available
blocked_by_policy
unknown_observation
```

### AuthorityReceiptV1

Authority reconciliation should emit receipts with verified postconditions:

```text
AuthorityReceiptV1 {
  schema_version,
  operation_id,
  reconciliation_plan_id,
  operation_status,
  started_at,
  finished_at,
  attempted_actions,
  verified_controller_observations,
  hard_failures,
  unresolved_observation_gaps,
  unresolved_external_actions,
  command_result,
}
```

Dry-run receipts record zero attempted controller mutations. They preserve
verified observations and unresolved evidence so a standalone receipt can
explain the dry run without becoming controller authority.

## Controller Policy

### Root

Root controller reconciliation must be explicit.

In 0.42 dry-run mode, Canic reports:

- whether root controllers are already correct;
- which root controller changes would be automatic candidates for a future
  apply path;
- which exact external action is required when the current operator/root lacks
  authority.

If controller mutation is explicitly promoted later, Canic may create root with
intended controllers when possible or update root settings when the
operator/root has authority. That is not required for the 0.42 minimum.

Root is the 0.40 protected-call trust anchor. If root changes, that is not a
normal resume. It is a new trust domain or a migration.

### Managed Canisters

Default managed canister controller rule:

```text
configured managed controllers
+ root
+ direct parent
```

Role-specific overrides may refine this rule, but the plan must show the final
desired controller set per canister.

### Pool Canisters

Pool canisters must be reconciled before they are reused.

Pool states:

```text
claimable
already_managed
requires_external_controller
unsafe_import
unknown
```

Rules:

- imported pool canisters are classified before reset/reuse;
- reused pool canisters that require authority changes are reported before
  install;
- unclaimable pool canisters are blocked before mutation;
- pool reset receipts and observed controller postconditions are required only
  if pool mutation is explicitly promoted into the line.

### Break-Glass Authority

Emergency controllers are not normal controllers by accident.

A break-glass flow must be explicit, visible in the plan, and receipted.

The normal path should fail if emergency or staging principals overlap
ordinary IC controller sets.

## Adoption-Facing Design Hooks

0.42 should expose enough authority state for later adoption workflows without
turning this line into the brownfield adoption product.

Destructive pool import is not a normal reconciliation action. If a reused pool
canister would require controller reset, code uninstall, or another destructive
operation, the authority plan should classify that separately:

```text
action: requires_destructive_import_confirmation
can_apply: no
```

Authority reconciliation may also produce an observe/adopt plan for existing
canisters:

```text
action: observe_only
action: adopt_plan_available
```

Those states let the operator see what would be needed to adopt or observe a
canister, but the full `canic fleet adopt` UX is deferred to 0.48.

Imported or jointly controlled canisters must remain visible through
`CanisterControlClassV1`. They must not be collapsed into normal managed roles
only because their controller set happens to match today.

Spawned canister controller behavior should be represented as policy:

```text
spawned_canister_controller_policy
```

It should not hide in runtime config or be inferred from a parent canister's
current controller set.

CLI identity binding may appear in `AuthorityProfile` and reconciliation
reports:

```text
operator_identity_bindings
```

That lets 0.42 explain when the current operator principal is not allowed to
apply a profile. Productized identity-provider or operator-team RBAC UX can
wait.

## Commands

0.42 minimum operator surface:

```text
canic deploy authority check <fleet>
canic deploy authority report <fleet>
canic deploy authority receipt <fleet>
canic deploy authority evidence <fleet>
```

These commands are read-only local surfaces over deployment truth. They do not
apply controller changes.

Future or explicitly promoted surfaces:

```text
canic deploy authority apply <deployment>
canic deploy authority diff <plan-or-receipt>
```

`<fleet>` matches the current local CLI input shape. `<deployment>` becomes the
preferred spelling once deployment-target live-state naming lands.

`apply` is not part of the 0.42 minimum. If explicitly promoted into the line,
it may run only actions whose state is `can_apply_automatically`, then
re-inventory and verify postconditions.

## Concurrent Operation Protection

Remote operation-epoch or lock protection is required only if controller apply
is explicitly promoted into 0.42.

For dry-run 0.42, authority receipts preserve the reconciliation plan ID,
inventory/check evidence, timestamps, attempted action list, observation gaps,
hard failures, and unresolved external actions. No remote lock or deployment
epoch check is required because no controller mutation occurs.

If apply is promoted, the minimum requirement becomes:

- authority reconciliation reads the latest inventory before applying;
- it records the previously observed deployment epoch;
- it verifies the deployment epoch after apply;
- it fails or warns if another operation changed authority state mid-run.

If root exposes a lightweight remote deployment lock by this line, authority
reconciliation should use it. Otherwise the apply path must remain deferred.

---

## Non-Goals

0.42 should not:

- introduce backend-agnostic execution;
- implement artifact promotion;
- implement user consent workflows;
- implement full brownfield adopt UX, which is deferred to 0.48;
- implement non-destructive pool registration UX, which is deferred to 0.48;
- implement identity-provider or operator-team RBAC product UX;
- make deployment intent a scripting language;
- hide controller changes behind runtime config.

User-controlled lifecycle is deferred, but 0.42 must preserve canister control
classification so unknown or user-controlled canisters are not treated as
normal managed roles.

## Implementation Slices

### Minimum Slice 1: Authority Reconciliation Plan

Build authority reconciliation from 0.41 plan and inventory.

### Minimum Slice 2: Dry-Run Operator Report

Render per-canister controller state and external actions required.

### Minimum Slice 3: Dry-Run Authority Receipt And Evidence

Emit read-only authority receipts and evidence bundles that preserve the plan,
report, zero attempted controller actions, verified observations, unresolved
observation gaps, hard failures, and external actions.

### Minimum Slice 4: Hard Authority Findings

Report break-glass, staging, and emergency authority category violations as
hard findings without attempting controller mutation.

### Minimum Slice 5: Pool/Imported/User-Controlled Classification

Classify pool, imported, jointly controlled, user-controlled, and unknown
canisters as automatic candidates, external-action cases, observation gaps, or
unsafe blockers.

### Promoted Or Later Slice: Root And Managed Apply

Apply controller changes only where Canic has authority. This is not required
for 0.42 unless explicitly promoted.

### Promoted Or Later Slice: Pool Ownership Reconciliation

Validate and reconcile pool canisters before reset or reuse.

### Promoted Or Later Slice: Post-Apply Verification

Re-inventory and verify desired controller state after each applied change.

---

## Exit Criterion

0.42 is complete when:

```text
Canic can produce a dry-run authority reconciliation plan, summarize
already-correct state, automatic controller-change candidates, unsafe blockers,
missing controller observations, and exact external actions, then emit
machine-readable evidence and receipts for that dry run without mutating
controller state.
```

Applying the safe subset is a promoted slice only if explicitly accepted into
0.42. Otherwise it belongs to the later execution boundary.
