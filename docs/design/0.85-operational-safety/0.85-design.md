# Canic 0.85 Design: Operational Safety Hardening

Date: 2026-07-11

## Status

- Status: implementation in progress; Slice A is complete.
- Release line: 0.85 operational safety hardening.
- Audit source:
  `docs/audits/reports/2026-07/2026-07-11/codebase-health.md`.
- Implementation status: [status.md](status.md).
- Primary rule: recovery state must be durable, build authority must be
  explicit, and Canic-owned serialization dependencies must be justified by
  bytes rather than convenience.
- Scope rule: complete the three audit priorities without adding a filesystem
  transaction framework, a generic process runner, codec negotiation, or a
  compatibility layer.

## 1. Decision Summary

0.85 contains three hard-cut changes:

1. Restore plans and execution journals that Canic later consumes as recovery
   authority use one `canic-backup` durable replacement primitive. Mutating
   restore steps never truncate the active journal in place.
2. Canister build/install context is one typed host-owned value applied to
   child commands. Production code no longer mutates the process-global
   environment.
3. Canic separates IC-required CBOR wire semantics from the specific
   `serde_cbor` crate. Host-owned IC CBOR and Canic-owned stable CBOR are tested
   against exact bytes. Canic removes its direct `serde_cbor` dependencies only
   where the replacement is proven safe.

The final architecture is:

```text
restore model
  -> serialize completely
  -> one durable sibling replace
  -> recoverable journal

typed build context
  -> exact Cargo/ICP child command environment
  -> no process-global mutation

IC protocol requires CBOR
  != IC requires serde_cbor

Canic stable/wire fixtures
  -> prove candidate codec bytes
  -> one codec in each final path
```

## 2. Why These Changes Belong Together

All three findings concern implicit process boundaries:

- a restore journal is authoritative only after bytes are durably committed;
- a build is authoritative only when its workspace, configuration, network,
  and role context are passed explicitly to the child process;
- a stable or wire encoding is authoritative only when its exact byte contract
  is known independently of a library name.

The line does not combine unrelated features. It removes three places where
correctness currently depends on an unstated environmental assumption.

## 3. Goals

1. Preserve the previous valid restore journal across interrupted or failed
   writes.
2. Commit restore journal bytes before the next externally mutating operation.
3. Put recovery-document persistence in `canic-backup`, which owns the
   documents and runner.
4. Remove both production `BuildEnvGuard` implementations.
5. Remove production `std::env::set_var` and `remove_var` calls from Canic
   build/install orchestration.
6. Resolve build context once and pass it explicitly to every child command
   that needs it.
7. Distinguish IC CBOR protocol requirements from Canic's codec dependency.
8. Attempt to reuse the `ciborium` generation already present through
   `ic-memory` instead of carrying a second direct CBOR generation.
9. Preserve exact Canic-owned stable bytes if the direct stable codec is
   replaced.
10. Remove direct `serde_cbor` dependencies where exact fixtures pass.
11. Measure the default Wasm dependency-graph reduction rather than claiming
    whole-workspace removal that upstream IC crates cannot provide.
12. Hard-cut replaced paths without aliases, wrappers, fallback decoders, or
    parallel writers.

## 4. Non-Goals

0.85 does not:

1. Add multi-file transactions or a write-ahead-log framework.
2. Make ordinary presentation-only `--out` files transactional.
3. Add cross-filesystem copy/replace fallback behavior.
4. Add restore-journal schema fields or change restore command forms.
5. Add a generic command/process abstraction.
6. Stop reading environment variables that remain legitimate CLI defaults.
7. Remove environment-driven behavior from Cargo build scripts; child Cargo
   commands still receive explicit environment values.
8. Promise that `serde_cbor` disappears from `Cargo.lock` or the full workspace
   graph. `ic-agent`, canister-signature libraries, and other upstream IC crates
   may still select it.
9. Change the IC replica protocol away from CBOR.
10. Add codec identifiers, runtime negotiation, dual writes, or fallback
    decoders.
11. Add a stable-state migration layer.
12. Split medic, deploy-plan, or state-manifest hub modules. Those are lower
    risk structural follow-ups after the three safety findings.
13. Clean every test-only environment mutation unrelated to the build/install
    path.
14. Change package versions as part of implementation slices.

## 5. Hard-Cut Rules

The implementation must end with one maintained path for each responsibility:

| Responsibility | Final owner | Removed path |
| --- | --- | --- |
| Restore recovery-document persistence | `canic-backup` | direct truncating journal/plan writes |
| Canister build context | `canic-host` typed input | CLI and install `BuildEnvGuard` globals |
| Host IC CBOR | host codec adapter | direct `serde_cbor` use in replica query modules |
| Stable serde-backed records | core stable codec adapter | old codec only if fixture proof passes |

There are no deprecated functions, compatibility aliases, environment
fallbacks, legacy journal writers, or old/new codec selection flags.

If codec proof fails, implementation stops at the decision gate described in
Section 8. It must not ship a dual decoder or silently change stable bytes.

## 6. Slice A: Durable Restore Recovery Documents

### 6.1 Current problem

`canic-backup::restore::runner` rewrites `RestoreApplyJournal` with
`fs::write`. That opens and truncates the authoritative journal before the new
JSON is fully committed. The runner persists transitions after operations such
as stop, snapshot load, start, and verification, so a partial write can destroy
the only reliable resume evidence.

Canic currently has three write shapes:

- backup persistence uses an atomic-looking fixed `.tmp` path;
- restore runner journals use direct truncating writes;
- host artifacts use a stronger unique sibling, sync, rename, and directory
  sync primitive.

The final backup path must have one implementation, not three.

### 6.2 Final primitive

`canic-backup` owns one private function with this semantic contract:

```text
durably_replace(path, complete_bytes)
  1. create parent directory
  2. create a unique sibling temporary file with create_new
  3. write all bytes
  4. sync temporary file
  5. close temporary file
  6. rename sibling over target
  7. sync parent directory
  8. remove the sibling on every pre-rename failure
```

Serialization completes into memory before this function is called. A
serialization failure therefore cannot touch the current file.

The sibling location makes rename a same-filesystem operation. Failure is
returned as a typed backup/restore persistence error. Cleanup failure does not
replace the primary error.

### 6.3 Owned public boundary

The raw durable byte helper stays private. `canic-backup` exposes only typed
operations required by its CLI consumer, approximately:

```rust
write_restore_plan(path, &RestorePlan)
write_restore_apply_journal(path, &RestoreApplyJournal)
```

Exact names may follow the existing module vocabulary. `canic-cli` must not
own serialization or persistence semantics for backup-owned recovery types.

The existing backup manifest, plan, and execution-journal persistence paths
reuse the same private primitive. Generic CLI text/JSON output remains a
presentation concern and does not move into `canic-backup`.

### 6.4 Commit ordering

For each restore mutation:

```text
validate current journal
  -> record pending/attempt state durably when required
  -> perform one external operation
  -> record receipt and resulting state durably
  -> continue
```

The implementation must preserve the existing state machine and receipt
semantics. Slice A changes persistence mechanics, not operation ordering or
recovery policy, except where an existing transition currently proceeds before
its prerequisite journal state is committed.

### 6.5 Targeted validation

- replacing an existing document yields only the complete new document;
- a pre-rename failure preserves the previous document;
- concurrent/stale sibling names do not overwrite another staging file;
- successful replace leaves no sibling file;
- runner transition tests still observe the same journal states and receipts;
- restore prepare writes its plan and initial journal through backup ownership;
- no production restore recovery path calls `fs::write` directly.

## 7. Slice B: Explicit Canister Build Context

### 7.1 Current problem

`canic-cli::build` and `canic-host::install_root` each implement a scoped
`BuildEnvGuard`. Both mutate process-global environment through Rust 2024
unsafe APIs, and both depend on a comment-only single-threaded invariant.

The host artifact builder already applies some values directly to the Cargo
`Command`, but it discovers other workspace/network/config inputs by reading
the mutated parent environment. This splits one build contract across global
state and command-local state.

### 7.2 One host-owned value

The host owns one typed build input containing the already-maintained facts:

```text
role
profile
workspace_root
icp_root
config_path
selected ICP environment
resolved build network
optional local replica URL/root key
```

This should extend or replace the current `WorkspaceBuildContext`; it must not
create a parallel context type with overlapping meaning. Construction validates
paths and environment selection once.

### 7.3 Command application

The typed value is passed into artifact building and:

- supplies direct arguments and current directories;
- applies exact environment pairs with `Command::env`;
- clears optional child values with `Command::env_remove` when absence is part
  of the contract;
- supplies the canonical role-contract marker only to the Cargo command that
  is authorized to build the selected canister;
- supplies display/provenance values without rereading mutable global state.

Existing ICP command helpers continue to receive network/root arguments
explicitly. Do not add a generic command wrapper.

### 7.4 Removed paths

Slice B deletes:

- both production `BuildEnvGuard` types;
- their `Drop` restoration logic;
- production unsafe `set_var`/`remove_var` helpers;
- build-context reads that exist only because the parent process was mutated.

No compatibility environment path remains. Environment variables supplied by
the user's shell may still be read once while constructing defaults; Canic
does not mutate them.

### 7.5 Targeted validation

- CLI build and install resolve the same typed context for equivalent inputs;
- named environments preserve the existing local/IC build-network behavior;
- optional local replica URL/root-key values are applied or removed on the
  child command exactly;
- sequential builds cannot inherit the previous request's config/network;
- command construction tests inspect explicit child environments;
- production workspace scans find no `std::env::set_var` or
  `std::env::remove_var` in build/install orchestration;
- existing focused build provenance and packaged feature checks pass.

## 8. Slice C: CBOR Ownership And Dependency Reduction

### 8.1 Protocol clarification

The IC replica API uses CBOR request and response bodies. That is a wire-format
requirement, not a requirement to use the `serde_cbor` Rust crate.

Current ownership is:

| Use | Requirement | Current implementation |
| --- | --- | --- |
| Local replica query/status | IC CBOR wire compatibility | direct host `serde_cbor` |
| Canic serde-backed `Storable` values | Canic stable schema | direct core `serde_cbor` |
| Canister-signature libraries | upstream implementation detail | transitive `serde_cbor` |
| `ic-memory` allocation ledger | upstream `ic-memory` implementation | transitive `ciborium` |

The default `canic-core` Wasm graph currently selects `serde_cbor` only because
Canic directly depends on it. `ciborium 0.2.2` is already in that graph through
`ic-memory 0.9.0`. All-feature auth builds and host tooling may still receive
`serde_cbor` transitively from IC libraries after Canic's direct edges are
removed.

### 8.2 Target codec

The first candidate is the existing workspace `ciborium` generation. Canic
declares it directly if and only if Canic source calls it; it must not rely on
an undocumented transitive re-export.

Two narrow adapters own usage:

- `canic-core` stable serialization adapter;
- `canic-host` replica CBOR adapter.

Do not expose codec-specific `Value` or error types across those module
boundaries.

### 8.3 Stable-byte proof gate

The exported `impl_storable_bounded!` and `impl_storable_unbounded!` macros
currently encode 37 Canic-owned record/key types through the core adapter.
`ReplayReceiptRecord` uses the codec directly. Downstream users may also use
the exported macros.

Before changing the adapter:

1. Freeze exact current bytes as owner-local golden fixtures for every active
   Canic-owned macro-backed type and the direct replay record.
2. Include non-default fields and every maintained enum/option shape that can
   change the encoding of that type.
3. Record the current bound length for bounded records.
4. After switching the candidate codec, require both:
   - golden bytes decode to the expected typed value;
   - that typed value encodes to the exact golden bytes.
5. Keep the fixtures after removing the old dependency. They protect the byte
   contract, not a library implementation.

Decision gate:

- If all Canic-owned fixtures match exactly, hard-cut the core adapter to the
  candidate codec and remove Canic's direct `serde_cbor` core dependency.
- If any fixture differs, stop. Do not add dual decoding, codec tags, or a
  migration. Record the mismatch in `status.md` and revise this design only if
  the maintainer explicitly chooses a destructive pre-1.0 reinstall cut.

The public storable macros switch with the core adapter if proof passes. Since
Canic cannot enumerate downstream private types, the release notes must call
the macro implementation change breaking even when all Canic-owned fixtures
match.

### 8.4 Host IC wire proof

Host replica query/status fixtures must cover:

- anonymous query envelope bytes;
- replied query response;
- rejected query response;
- status root key represented as CBOR bytes;
- malformed and unsupported CBOR rejection.

IC wire semantics must remain unchanged. The host adapter may use
`ciborium::Value` internally for status discovery, but no codec value type
escapes the adapter.

Host direct `serde_cbor` removal is independent of stable-byte proof because
host query bodies are not persisted Canic state. It still requires the wire
fixtures above.

### 8.5 Dependency claims

Success is measured precisely:

- `canic-core` default Wasm graph has no Canic-direct `serde_cbor` edge;
- `canic-host` has no direct `serde_cbor` edge;
- the default Wasm graph no longer selects `serde_cbor` and its unique
  `half 1.8` dependency when no upstream feature requires them;
- all-feature and host graphs may retain `serde_cbor` through upstream IC
  crates;
- `cargo audit` may therefore continue reporting the upstream unmaintained
  crate for the full workspace.

Do not claim lockfile-wide removal unless the actual final graph proves it.

### 8.6 Targeted validation

- stable golden fixtures for every active Canic-owned macro-backed type;
- exact replay receipt golden bytes;
- stable bounds do not increase;
- local replica query/status wire fixtures;
- default and all-feature `canic-core` dependency-tree checks;
- host dependency-tree check;
- focused default and Wasm-target core checks;
- focused host replica-query tests;
- no direct `serde_cbor` source references remain in an owner whose switch
  passed.

## 9. Implementation Slices

### Slice A - Durable recovery persistence

Owner: `canic-backup`, consumed by `canic-cli`.

Deliver:

- one private durable replacement primitive;
- typed plan/journal persistence entrypoints;
- all mutating runner journal transitions routed through it;
- old fixed-temp and truncating recovery writers removed;
- focused durability and runner transition tests.

### Slice B - Explicit build authority

Owner: `canic-host`, consumed by `canic-cli` and install workflows.

Deliver:

- one typed build context;
- command-local environment application;
- both global guards and production unsafe mutations removed;
- focused CLI build, install, provenance, and child-command tests.

### Slice C - CBOR proof and hard cut

Owners: core stable schema owners and host replica-query adapter.

Deliver:

- frozen stable/wire byte fixtures;
- host direct codec replacement;
- core replacement only after the stable-byte gate passes;
- exact dependency-tree report;
- direct old dependencies removed where proven.

Each slice is independently reviewable. Slice names do not assign patch
versions; release preparation remains maintainer-owned.

## 10. Acceptance Criteria

0.85 is complete when:

1. Restore mutation never truncates the active journal in place.
2. Every recovery-document write uses one backup-owned durable primitive.
3. A failed pre-rename write preserves the previous valid document.
4. CLI restore preparation delegates plan/journal persistence to
   `canic-backup`.
5. One host-owned typed value contains canister build authority.
6. CLI and install build paths use that same value.
7. Production build/install code contains no process-global environment
   mutation.
8. No `BuildEnvGuard` compatibility path remains.
9. IC CBOR requirements are covered by exact host wire fixtures.
10. Every active Canic-owned macro-backed stable type has an exact golden
    fixture before a codec switch.
11. Core stable codec replacement occurs only if all golden bytes match.
12. No dual codec, fallback decoder, migration shim, or codec negotiation is
    introduced.
13. Canic's direct `serde_cbor` edges are removed wherever the corresponding
    proof gate passes.
14. Dependency reports distinguish default Wasm, all-feature Wasm, host, and
    full-workspace graphs.
15. Stable-memory IDs, keys, record meanings, DTOs, Candid, JSON schemas, and
    command forms remain unchanged unless a later explicit design revision
    authorizes a destructive codec cut.
16. Targeted tests, checks, Clippy for touched packages, repository guards, and
    documentation validation pass.

## 11. Risks And Mitigations

### Risk: durable rename is assumed to work across filesystems

Mitigation: always create the temporary file as a sibling of the target. Do
not add copy fallback semantics.

### Risk: a journal commit is reordered around an external mutation

Mitigation: retain the current state machine and add focused transition tests
around every write site. Persistence mechanics move first; orchestration
changes require explicit evidence.

### Risk: explicit build context misses an inherited variable

Mitigation: inventory the two existing guards and every Cargo/ICP command,
then test the child command environment as data. Unknown inherited variables
must not become part of Canic's authority contract accidentally.

### Risk: two codecs round-trip values but emit different bytes

Mitigation: require exact byte equality, not only typed round trips.

### Risk: Canic removes a direct dependency but RustSec still reports it

Mitigation: report graph ownership accurately. Upstream IC dependency warnings
are tracked separately from Canic-direct ownership.

### Risk: the codec fixture inventory becomes another state catalog

Mitigation: fixtures live beside storage owners and test only bytes. The 0.84
role/state contract remains the allocation/descriptor authority.

## 12. Closed Decisions

- The line contains exactly three implementation slices.
- Durable persistence is one backup-owned function, not a framework.
- Presentation-only output is not made transactional.
- Build authority is passed explicitly; no global compatibility fallback
  remains.
- IC CBOR does not imply `serde_cbor` crate ownership.
- `ciborium` is the first candidate because it is already selected through
  `ic-memory`; this is a dependency-graph reason, not permission to change
  bytes.
- Stable replacement is gated by exact fixtures.
- A mismatch blocks the replacement until an explicit destructive-cut design
  decision; it does not create a migration or dual decoder.
- Upstream transitive `serde_cbor` is not falsely counted as Canic-direct debt.
- Structural hub splitting and external diagnostic consolidation are not 0.85
  scope.
