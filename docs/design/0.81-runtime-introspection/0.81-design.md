# Canic 0.81 Design: Runtime Introspection Pack

Date: 2026-07-01

## Status

- Status: tentative design seed
- Release line: tentative 0.81 planning label
- Decision style: runtime observability and operator inspection
- Primary implementation target: expose a consistent health, readiness,
  topology, feature, timer, state, and version status surface for
  Canic-managed canisters.
- Current state: Canic already has readiness concepts, metrics/logging helpers,
  auth renewal status, blob-storage status, deployment-truth reports, local
  status, and targeted medic checks. These are useful but not yet one coherent
  runtime introspection pack.

## 1. Summary

0.81 should make deployed Canic-managed canisters easier to inspect after
installation. The operator should be able to ask:

```text
Is this canister healthy?
Is it ready for its role?
What role and root does it believe it has?
What feature flags are active?
What timers are registered and healthy?
What state schema version is active?
What recent failures should I look at first?
```

This is broader than logs. It should be a typed runtime status surface that CLI
commands and `canic medic` can consume.

## Hard-Cut Rule

The introspection pack should expose one canonical endpoint and CLI vocabulary
per status concept. The design does not include compatibility endpoints,
fallback method names, or command aliases.

## 2. Goals

1. Define common runtime status DTOs.
2. Add health and readiness endpoints where appropriate.
3. Report canister role and root/topology bindings.
4. Report build/version metadata.
5. Report active Canic feature flags.
6. Report timer health for Canic-owned timers.
7. Report stable memory usage summaries without leaking application data.
8. Report bounded recent failures for operator triage.
9. Add CLI inspection commands that consume the typed endpoint surface.
10. Feed relevant runtime checks into `canic medic deployment`.

## 3. Non-Goals

0.81 does not:

1. Add a hosted dashboard.
2. Replace structured logs or metrics.
3. Expose secrets, session material, private keys, or raw application state.
4. Make readiness imply deployment safety.
5. Query every child canister from one endpoint unless a role-specific design
   owns that cost and authorization model.
6. Add mutation or repair behavior to status endpoints.
7. Loosen endpoint authentication rules.
8. Require every downstream app to expose every optional introspection field.

## 4. Endpoint Surface

Possible common endpoints:

```text
canic_health
canic_readiness
canic_runtime_status
```

Possible role-specific endpoints:

```text
canic_topology_status
canic_auth_status
canic_state_status
canic_timer_status
```

The final endpoint names should align with existing protocol naming and macro
surfaces before implementation.

## 5. Runtime Status Shape

Example payload:

```json
{
  "schema_version": 1,
  "role": "project_hub",
  "canic_version": "0.81.0",
  "root": "aaaaa-aa",
  "network": "local",
  "features": {
    "delegated_tokens": true,
    "blob_storage": false,
    "control_plane": true
  },
  "timers": [
    {
      "name": "root_renewal",
      "status": "healthy",
      "last_success_at_ns": 1,
      "last_failure_at_ns": null
    }
  ],
  "state": {
    "schema_version": 4,
    "stable_memory_pages": 23
  },
  "recent_failures": []
}
```

The actual DTO should use named structs and enums, not boundary type aliases.

## 6. CLI Surface

Possible CLI surface:

```text
canic inspect <canister-or-deployment>
canic inspect <deployment> --role root
canic topology <deployment>
canic runtime status <deployment>
canic medic deployment <deployment>
```

Open naming choice:

- keep all passive runtime reads under `canic inspect`;
- or add focused top-level nouns such as `canic topology`.

The CLI should avoid inventing topology from local config when it is presenting
runtime-observed state.

## 7. Authorization

Auth remains enforced at endpoints. Workflow and ops should continue assuming
authenticated input.

Public health can be considered only for fields that are safe to expose. More
detailed readiness and runtime status should use explicit guards. The design
should classify each field as:

- public-safe;
- operator-only;
- controller-only;
- disabled unless the feature is enabled.

## 8. Implementation Slices

1. Define common DTOs for health, readiness, runtime status, timer status, and
   state summary.
2. Add one minimal endpoint to one Canic-owned canister role.
3. Add CLI rendering for that endpoint.
4. Add timer and recent-failure summaries for Canic-owned timers.
5. Add stable memory summary from the 0.80 manifest when available.
6. Add role/topology status for root and managed children.
7. Feed runtime status into `canic medic deployment`.

## 9. Validation

Focused validation should cover:

```text
cargo test --locked -p canic-core runtime --lib
cargo test --locked -p canic-control-plane runtime --lib
cargo test --locked -p canic --test protocol_surface
cargo test --locked -p canic-cli inspect
```

PocketIC coverage is required when endpoint behavior, role topology, or
inter-canister runtime status is added.

## 10. Open Questions

1. Which status fields are safe for unauthenticated health checks?
2. Should recent failures be stored in stable state, heap-only state, or logs
   only?
3. Should timer status be a common runtime feature or per-subsystem status?
4. Should CLI `inspect` target deployments, canister principals, roles, or all
   three?
