# Canic 0.80 Design: Stable State And Migration Framework

Date: 2026-07-01

## Status

- Status: tentative design seed
- Release line: tentative 0.80 planning label
- Decision style: upgrade-safety framework
- Primary implementation target: make stable state layout, schema versions,
  memory IDs, and migration coverage explicit and auditable.
- Current state: Canic uses stable records and lifecycle restore paths across
  core/control-plane surfaces. The repo already treats lifecycle adapters,
  stable memory, data snapshots, and PocketIC upgrade tests as important safety
  boundaries, but there is not yet one explicit state manifest and migration
  audit surface.

## 1. Summary

0.80 should introduce a formal state-layout and migration contract for
Canic-managed canisters. The main product value is preventing an upgrade from
silently breaking persisted state.

The first slice should focus on manifesting and auditing state shape:

```text
state domain
schema version
stable memory ID
owning module
upgrade support window
migration availability
```

Migration execution and generated migration code can come later. The early win
is making missing versions, memory-ID collisions, and untested upgrade paths
visible.

## Hard-Cut Rule

The framework should migrate explicitly into the current canonical state schema.
It should not keep dual runtime schemas, compatibility readers, legacy manifest
aliases, or fallback memory-ID interpretations.

## 2. Goals

1. Define a state manifest model for Canic canisters.
2. Record every state domain with an explicit schema version.
3. Record every stable memory ID that Canic owns.
4. Detect duplicated memory IDs inside one canister role.
5. Detect state domains without migration coverage.
6. Provide a local audit command consumed by `canic medic`.
7. Provide test helpers for install-upgrade-query regression tests.
8. Keep lifecycle adapters synchronous for restore and schedule async work
   after invariants are restored.

## 3. Non-Goals

0.80 does not:

1. Generate all migrations automatically.
2. Replace existing `export()` and `import()` canonical snapshot contracts.
3. Add production `cfg(test)` behavior to fake IC management.
4. Change existing stable memory IDs without an explicit migration design.
5. Store DTOs as authoritative state.
6. Let workflow construct or mutate persisted records directly.
7. Add a broad state explorer that leaks sensitive application data.
8. Require every downstream app to adopt the full migration framework in one
   release.

## 4. Manifest Shape

Tentative file or generated payload:

```toml
schema_version = 1
canister_role = "root"

[[state]]
domain = "auth_sessions"
version = 3
memory_id = 1
owner = "canic-core"
record = "AuthSessionRecord"

[[state]]
domain = "project_registry"
version = 2
memory_id = 2
owner = "canic-control-plane"
record = "ProjectRegistryRecord"
```

Open choice:

- source-authored `canic-state.toml`
- generated manifest from Rust declarations
- embedded runtime manifest exposed by an introspection endpoint

The safest path may be Rust declarations first, then generated JSON/TOML for
operator tools.

## 5. Audit Checks

Initial audit checks:

1. Every registered state domain has a version.
2. Every registered state domain has a memory ID or explicitly declares that it
   does not use stable memory.
3. Memory IDs are unique within a canister role.
4. Record names end in `Record`.
5. Canonical snapshot names use `*Data`.
6. `export()` and `import()` are reserved for canonical snapshots.
7. Required migrations exist from explicitly supported old versions.
8. Removed state domains declare whether data is migrated or intentionally
   discarded.
9. Lifecycle restore order is explicit.
10. Post-upgrade invariant checks are declared.

`canic medic` should report these as state/readiness checks, but the framework
should have a narrower state-specific command too.

## 6. Command Surface

Possible initial surface:

```text
canic state audit
canic state audit --role root
canic state manifest --format json
canic medic project
```

`canic medic project` can call the state audit when the project exposes enough
metadata.

## 7. Migration Test Harness

The test harness should make the common safe path easy:

```text
install old wasm
seed state
upgrade to new wasm
query state
assert invariants
```

Canister creation, install, upgrade, and inter-canister migration tests must
use PocketIC.

The harness should encourage:

1. typed state assertions;
2. typed errors;
3. observable state checks;
4. no assertions against error strings;
5. no fake production management-canister behavior.

## 8. Implementation Slices

1. Inventory existing Canic-owned stable state domains.
2. Add an internal manifest model and static declarations for one canister
   family.
3. Add a duplicate-memory-ID audit.
4. Add a schema-version presence audit.
5. Add command output for state manifest and audit report.
6. Add PocketIC helper APIs for install-upgrade-query tests.
7. Expand canister families after the first manifest path proves stable.

## 9. Validation

Focused validation should cover:

```text
cargo test --locked -p canic-core state --lib
cargo test --locked -p canic-control-plane state --lib
cargo test --locked -p canic-tests upgrade --test root_suite
```

Actual package names and filters should be narrowed when the first
implementation slice chooses the owning modules.

## 10. Open Questions

1. Should downstream apps author manifests directly, or should Canic derive
   them from Rust declarations?
2. Which old state versions are explicitly supported by migration gates before
   they are hard-cut?
3. Should memory IDs be globally reserved across a fleet or only within one
   canister role?
4. Should state audit failures block build, deploy plan, medic, or only release
   CI?
