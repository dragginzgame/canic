# Canic 0.80 Design: Stable State And Migration Audit

Date: 2026-07-03

## Status

- Status: implementation specification
- Release line: 0.80 state-safety label
- Decision style: no-mutation upgrade-safety metadata
- Primary implementation target: make stable state layout, schema versions,
  memory IDs, lifecycle restore order, and migration coverage explicit and
  auditable before broad migration runtime work begins.
- Current state: Canic uses stable records and lifecycle restore paths across
  core/control-plane surfaces. The repo already treats lifecycle adapters,
  stable memory, canonical data snapshots, and PocketIC upgrade tests as safety
  boundaries, but there is not yet one explicit state manifest and migration
  audit surface.

## 1. Summary

0.80 introduces a formal state-layout and migration audit contract for
Canic-managed canisters. The release value is preventing upgrades from silently
breaking persisted state by making the implicit state contract visible.

The first slice focuses on manifesting and auditing state shape:

```text
state domain
schema version
stable memory ID
owning module
record type
snapshot type
upgrade support window
migration availability
lifecycle restore order
post-upgrade invariant declaration
```

Migration execution, generated migration code, runtime introspection endpoints,
source-authored manifests, and state explorers are intentionally deferred. The
early win is making missing versions, memory-ID collisions, and untested upgrade
paths visible.

The 0.80 source of truth is Rust-authored static declarations that live next to
the code owning the state. Generated JSON/TOML manifests are derived artifacts,
not canonical input. Runtime manifest introspection is deferred.

## Hard-Cut Rule

Old state versions may be accepted only through declared migration paths. There
must be no dual active runtime schemas, no compatibility readers that treat old
and new layouts as interchangeable, no legacy manifest aliases, no memory-ID
aliases, and no "try old layout, then try new layout" fallback outside a
declared one-way migration function.

The framework migrates explicitly into the current canonical state schema. A
domain that intentionally drops old state must declare removed-state disposition
instead of silently ignoring it.

New 0.80 report wrappers are unversioned in code and docs. JSON payloads include
`schema_version = 1`, but 0.80 does not include `V1`/`V2` compatibility layers,
dual decoders, or alternate accepted report versions.

## 2. Goals

1. Define a Rust-authored state declaration model for Canic canisters.
2. Derive a state manifest from declarations.
3. Record every state domain with an explicit schema version and storage mode.
4. Record every stable memory ID that Canic owns.
5. Detect duplicated memory IDs inside one canister role.
6. Treat cross-role memory-ID reuse as valid because stable memory is scoped to a
   canister role.
7. Detect state domains without migration coverage for supported old versions.
8. Detect removed state without declared disposition.
9. Provide local `canic state manifest` and `canic state audit` commands.
10. Let `canic medic project` summarize or point to state-audit findings without
    owning state-audit logic.
11. Provide test helper APIs for install-upgrade-query regression tests.
12. Keep lifecycle adapters synchronous for restore and schedule async work only
    after persisted invariants are restored.

## 3. Non-Goals

0.80 does not:

1. Generate migrations automatically.
2. Add a general migration execution runtime.
3. Replace existing `export()` and `import()` canonical snapshot contracts.
4. Add production `cfg(test)` behavior to fake IC management.
5. Change existing stable memory IDs without an explicit migration design.
6. Store DTOs as authoritative state.
7. Let workflow construct or mutate persisted records directly.
8. Add a broad state explorer or state dump surface.
9. Expose sensitive persisted application values.
10. Require every downstream app to adopt source-authored manifest files.
11. Require every downstream app to adopt the full migration framework in one
    release.
12. Add runtime manifest introspection endpoints.
13. Add fleet-global memory-ID reservations.
14. Make `canic deploy plan` a global state-policy gate.
15. Wrap state reports in evidence envelopes.
16. Write generated files or report files.

## 4. Manifest Authority

0.80 canonical manifest source:

```text
Rust-authored static declarations
  -> derived StateManifest
  -> text/JSON rendered by canic state manifest
  -> state audit checks consume the same derived manifest
```

Deferred:

- source-authored `canic-state.toml`;
- downstream app-authored standalone manifests;
- embedded runtime manifest introspection endpoints.

Reason: a hand-authored manifest can drift from code. State-safety confidence
requires declarations to live next to the code that owns the state domain.

Initial declaration shape:

```text
StateDomainDeclaration
  canister_role
  domain
  version
  storage
  memory_id
  owner
  record
  snapshot
  min_supported_version
  migration_policy
  restore_order
  post_upgrade_invariant
  migrations
```

Reservations for allocated memory IDs that are not yet precise state domains
are explicit manifest entries, not hidden comments. They are temporary
metadata used to make gaps visible without faking record/snapshot declarations.

## 5. Manifest Model

Rendered manifest shape:

```json
{
  "schema_version": 1,
  "roles": [
    {
      "canister_role": "root",
      "state": [
        {
          "domain": "auth_sessions",
          "version": 3,
          "storage": "stable_memory",
          "memory_id": 1,
          "owner": "canic-core",
          "record": "AuthSessionRecord",
          "snapshot": "AuthSessionsData",
          "min_supported_version": 2,
          "migration_policy": "migrate",
          "restore_order": 20,
          "post_upgrade_invariant": "auth_sessions_invariants",
          "migrations": [
            {
              "from": 2,
              "to": 3,
              "kind": "function",
              "name": "migrate_auth_sessions_v2_to_v3",
              "test": "auth_sessions_v2_to_v3_upgrade_preserves_sessions"
            }
          ]
        },
        {
          "domain": "runtime_metrics",
          "version": 1,
          "storage": "heap_only",
          "memory_id": null,
          "owner": "canic-core",
          "record": "RuntimeMetricsRecord",
          "snapshot": "RuntimeMetricsData",
          "min_supported_version": 1,
          "migration_policy": "not_applicable",
          "restore_order": 90,
          "post_upgrade_invariant": "not_applicable",
          "migrations": []
        }
      ],
      "removed_state": [
        {
          "domain": "legacy_auth_cache",
          "last_version": 1,
          "removed_in_version": 3,
          "memory_id": 7,
          "disposition": "discarded",
          "reason": "ephemeral cache; safe to rebuild",
          "test": "legacy_auth_cache_absence_does_not_break_upgrade"
        }
      ],
      "reserved_memory": [
        {
          "label": "log_index",
          "memory_id": 31,
          "owner": "canic-core",
          "reason": "stable log index memory needs multi-memory domain modeling"
        }
      ]
    }
  ]
}
```

Required storage values:

- `stable_memory`
- `heap_only`
- `not_applicable`

`not_applicable` is for declaration rows that are useful to the state contract
but do not own persisted runtime storage. It is distinct from `heap_only`,
which still represents runtime state that is intentionally not in stable
memory.

Required migration policies:

- `new_domain`
- `migrate`
- `discard_declared`
- `manual_migration_required`
- `not_applicable`

Each state domain declares:

```text
current_version = version
min_supported_version = oldest version accepted by declared support
unsupported_below_version = min_supported_version - 1, by implication
```

For each old version in `[min_supported_version, current_version)`, the domain
must declare one of:

- migration available;
- discarded state with explicit disposition;
- manual migration required;
- not applicable.

`manual_migration_required` satisfies the declaration requirement, but it is not
a passing production-upgrade state by itself. `canic state audit` should report
it as `warn` by default, and release CI or future upgrade planning may treat it
as blocking when the affected old version is in the supported production window.

Unsupported old versions are explicit by being below `min_supported_version`;
they are not accidental parser failures or permanent fallback readers.

## 6. Naming And Data Shape Contract

0.80 uses the existing Canic vocabulary strictly:

- `Record`: persisted internal representation. Must end in `Record`. Must not be
  exposed as public DTO.
- `Data`: canonical snapshot/import/export shape. Must end in `Data`. Used by
  `export()` and `import()`.
- `Dto`: wire/API boundary shape. Must not be authoritative persisted state.
- `State`: in-memory aggregate or module-local storage wrapper. Should not be
  serialized directly unless declared.

`export()` and `import()` are reserved for canonical `*Data` snapshots. Other
helpers should use more specific verbs.

## 7. Non-Mutation And Privacy Contract

`canic state audit` and `canic state manifest` must not:

1. install or upgrade Wasm;
2. create or delete canisters;
3. change controllers;
4. write deployment truth;
5. create or update installed deployment records;
6. write generated files;
7. write report output files;
8. modify config files;
9. modify stable memory;
10. run migrations;
11. repair memory IDs;
12. dump application records;
13. expose sensitive persisted values;
14. call live mainnet;
15. require local replica availability.

The commands may report metadata only:

- canister role;
- state domain;
- schema version;
- storage mode;
- memory ID;
- owner;
- record type name;
- snapshot type name;
- migration coverage declaration;
- test coverage declaration;
- lifecycle restore/invariant declaration;
- removed-state disposition.

## 8. Audit Report Model

`canic state audit` emits a `StateAuditReport`:

```text
StateAuditReport
  schema_version
  command
  scope
  role
  status
  manifest
  checks
  next_actions
```

Each check is a `StateAuditCheck`:

```text
StateAuditCheck
  category
  code
  status
  severity
  subject
  detail
  next
  source
```

Report statuses:

- `pass`
- `warn`
- `fail`
- `not_evaluated`

Check statuses:

- `pass`
- `warn`
- `fail`
- `not_evaluated`

Severities:

- `info`
- `warning`
- `blocked`
- `unsupported`

Categories:

- `manifest`
- `schema_version`
- `memory_id`
- `migration`
- `removed_state`
- `snapshot`
- `naming`
- `lifecycle`
- `invariant`
- `test_coverage`
- `privacy`

Initial check codes:

- `state_manifest_schema_version_supported`
- `state_manifest_schema_version_unsupported`
- `state_role_duplicate`
- `state_domain_registered`
- `state_domain_duplicate`
- `state_domain_missing_version`
- `state_domain_invalid_support_window`
- `state_domain_missing_storage`
- `state_domain_missing_memory_id`
- `state_domain_declares_no_stable_memory`
- `state_domain_storage_not_applicable`
- `memory_id_unique`
- `memory_id_duplicate`
- `record_name_valid`
- `record_name_invalid`
- `snapshot_name_valid`
- `snapshot_name_invalid`
- `reserved_export_import_ok`
- `reserved_export_import_violation`
- `migration_available`
- `migration_missing`
- `migration_declaration_invalid`
- `migration_declaration_duplicate`
- `manual_migration_required_declared`
- `migration_unsupported_declared`
- `removed_state_disposition_declared`
- `removed_state_disposition_missing`
- `removed_state_reason_declared`
- `removed_state_reason_missing`
- `removed_state_test_declared`
- `removed_state_test_missing`
- `removed_state_memory_id_reserved`
- `removed_state_memory_id_reclaimed`
- `reserved_memory_id_declared`
- `reserved_memory_id_duplicate`
- `reserved_memory_id_collision`
- `restore_order_declared`
- `restore_order_missing`
- `post_upgrade_invariant_declared`
- `post_upgrade_invariant_missing`
- `upgrade_test_declared`
- `upgrade_test_missing`

Status derivation:

1. `fail` if any check is `fail`.
2. `warn` if no checks fail and at least one check warns.
3. `pass` if all evaluated checks pass.
4. `not_evaluated` only when no checks were evaluated.

Warnings do not exit nonzero by default. CI can choose stricter policy later.

## 9. Audit Checks

Initial audit checks:

1. The derived state manifest declares the supported manifest schema version.
2. Each canister role is declared at most once in the manifest.
3. Every registered state domain has a version.
4. Each state domain name is declared at most once per canister role.
5. Every registered state domain has a positive `min_supported_version` less
   than or equal to the current version.
6. Every registered state domain has a storage mode.
7. Stable-memory domains have memory IDs.
8. Non-stable-memory domains explicitly declare no stable memory.
9. Memory IDs are unique within one canister role.
10. Cross-role memory-ID reuse is allowed and may be reported as info.
11. Record names end in `Record`.
12. Canonical snapshot names use `*Data`.
13. `export()` and `import()` are reserved for canonical snapshots.
14. Required migrations exist from explicitly supported old versions.
15. Declared migration rows are one-step edges inside the supported version
    window.
16. Duplicate migration edge declarations fail instead of relying on whichever
    row is found first.
17. Missing migration coverage is not hidden behind a passing status.
18. Removed state domains declare migrated/discarded/manual disposition.
19. Lifecycle restore order is explicit.
20. Post-upgrade invariant checks are declared or marked not applicable.
21. Upgrade test coverage is declared for supported migrations.
22. State commands do not expose persisted values.
23. Allocated memory IDs not yet modeled as active or removed state are listed
    under `reserved_memory`.
24. Reserved memory IDs do not collide with active or removed state.

Blocking rules:

- duplicate memory ID within the same role = fail;
- duplicate state domain within the same role = fail;
- duplicate canister role in the manifest = fail;
- unsupported manifest schema version = fail;
- missing schema version = fail;
- missing storage declaration = fail;
- stable-memory state without memory ID = fail;
- invalid upgrade support window = fail;
- invalid or duplicate migration declaration = fail;
- supported old version without migration/disposition declaration = fail;
- removed state without disposition = fail.
- reserved memory ID collides with active/removed state = fail;
- reserved memory ID declared more than once = fail.

Warnings:

- declared migration lacks a declared test;
- post-upgrade invariant missing where not clearly required;
- cross-role memory-ID reuse, if reported, is info/warn only.
- reserved memory ID has no precise active or removed state-domain model yet.

## 10. Command Surface

Supported initial surface:

```text
canic state audit
canic state audit --role <role>
canic state audit --json
canic state audit --role <role> --json

canic state manifest
canic state manifest --role <role>
canic state manifest --json
canic state manifest --role <role> --json
```

`--json` follows the 0.78 medic and 0.79 deploy-plan style. 0.80 does not
introduce `--format json` for this command family.

Rejected forms:

```text
canic state
canic state audit <role>
canic state manifest <role>
canic state migrate
canic state repair
canic state explore
canic state dump
canic state manifest --format json
canic state audit --format json
```

The rejected forms must not be silently routed, shimmed, hidden, aliased, or
accepted as compatibility surfaces.

## 11. JSON And Text Output

`canic state manifest` may render the derived manifest to stdout in text form.
`canic state manifest --json` renders the derived manifest to stdout as JSON.
Neither form writes a manifest file in 0.80.

When `--json` is used, stdout contains only JSON. Human-readable diagnostics,
usage errors, and internal errors go to stderr.

JSON output emits the report or manifest directly with `schema_version = 1`. It
is not an evidence envelope, not deployment truth, and not authorization to
upgrade or mutate state.

Text output writes the rendered report or manifest to stdout. Usage errors and
internal errors go to stderr. Text output should be human-readable and report
metadata only. It must not dump application records or persisted values.

Example text:

```text
canic state audit
status: fail
scope: project

root state manifest [pass]
  detail: 6 state domains registered for root

memory_id [fail] memory_id_duplicate
  subject: root/memory_id/1
  detail: memory id 1 is used by auth_sessions and project_registry
  next: assign a unique memory id or add an explicit migration design

test_coverage [warn] upgrade_test_missing
  subject: root/auth_sessions v2 -> v3
  detail: supported old version 2 has no declared migration test
  next: declare upgrade test coverage or hard-cut min_supported_version to 3
```

## 12. Output Determinism

`canic state audit` and `canic state manifest` must emit deterministic output.

Ordering:

1. roles by `canister_role`;
2. state domains by `domain`;
3. removed-state entries by `domain`;
4. migrations by `from`, then `to`;
5. audit checks by status severity, category, code, and subject where practical;
6. `next_actions` lexicographically unless a later design defines a stronger
   operator-priority order.

Text output writes only the rendered report or manifest to stdout. JSON output
writes only JSON to stdout. Usage errors and internal errors go to stderr.

## 13. Exit Codes

0.80 exit-code rules:

- `0`: report constructed and no checks fail.
- `1`: report constructed with one or more failing checks.
- `2`: usage error, invalid command shape, invalid flag combination, or internal
  report-construction failure.

Warnings do not exit nonzero by default.

## 14. Medic Relationship

`canic state audit` owns state/migration metadata checks.

`canic medic project` may run or summarize state-audit results when local
declarations are available, but medic must not:

- inspect sensitive state values;
- run PocketIC upgrade tests;
- run migrations;
- mutate files or canister state;
- become the owner of state-audit logic.

Medic readiness checks may point operators to:

```text
run canic state audit
```

Initial medic summary check codes:

- `state_audit_pass`
- `state_audit_warn`
- `state_audit_fail`
- `state_audit_not_evaluated`

The medic row is a summary only. Its source is `state_manifest`; it must not
inline the full state-audit report, inspect stable memory, or reinterpret
state-audit findings beyond mapping the aggregate status into medic's
PASS/WARN/FAIL/NOT_EVALUATED model.

## 15. Deploy Plan Relationship

0.80 does not make `canic deploy plan` a global state policy gate.

Deploy plan may eventually surface state-audit warnings when planning an upgrade,
but the first 0.80 slice keeps state audit as a separate command. A later
upgrade-planning or apply design may consume `StateAuditReport`.

Suggested integration policy:

- missing schema version: state audit fails; medic project fails; release CI
  blocks; deploy plan warns unless planning an upgrade.
- duplicate memory ID within role: state audit fails; medic project fails;
  release CI blocks; deploy plan blocks only when that role is in an upgrade
  plan.
- missing migration from supported old version: state audit fails; medic
  project warns or fails depending on release mode; release CI blocks; deploy
  plan blocks only for upgrade plans.
- untested migration path: state audit warns or fails depending on declared
  support; medic warns; release CI blocks for supported production versions.

## 16. Lifecycle Contract

`post_upgrade` must:

1. restore stable state synchronously;
2. validate schema versions;
3. run declared migrations or reject unsupported versions;
4. restore core invariants before scheduling timers or async work;
5. schedule async repair, renewal, and background tasks only after persisted
   invariants are restored.

Lifecycle adapters must not perform async work before state invariants are
restored.

## 17. Migration Test Harness

0.80 may provide helper APIs, but the audit should not try to prove test
execution by parsing arbitrary test names. It checks declared migration/test
coverage; CI runs the tests.

Desired helper shape:

```text
UpgradeScenario::new()
  .install_old_wasm(...)
  .seed_state(...)
  .upgrade_to_new_wasm(...)
  .query(...)
  .assert_invariants(...)
```

Canister creation, install, upgrade, and inter-canister migration tests must use
PocketIC.

The harness should encourage:

1. typed state assertions;
2. typed errors;
3. observable state checks;
4. no assertions against error strings;
5. no fake production management-canister behavior;
6. no direct stable-memory mutation outside test harness utilities.

## 18. Implementation Slices

1. Inventory existing Canic-owned stable state domains.
2. Define state manifest/report/check models.
3. Add Rust static declarations for one canister family.
4. Add `canic state manifest` text/JSON renderer.
5. Add state audit engine for declarations.
6. Add duplicate-memory-ID audit.
7. Add schema-version and storage presence audits.
8. Add record/snapshot naming convention audits.
9. Add migration coverage declaration audit.
10. Add removed-state disposition audit.
11. Add lifecycle restore-order and invariant declaration audits.
12. Add `canic state audit` text/JSON output.
13. Add medic project integration as summary/next-action only.
14. Add PocketIC upgrade helper APIs.
15. Expand to more canister families after the first family is stable.

Do not start with generated migrations or a migration runtime. First make the
state contract visible and auditable.

## 19. Validation

Focused validation should cover:

```text
cargo test --locked -p canic-core state --lib
cargo test --locked -p canic-control-plane state --lib
cargo test --locked -p canic-cli state
cargo test --locked -p canic-cli medic
cargo test --locked -p canic-tests upgrade --test root_suite
```

Actual package names and filters should be narrowed when the first
implementation slice chooses the owning modules.

Close 0.80 only when:

1. `canic state audit` exists.
2. `canic state manifest` exists.
3. Both commands support `--json`.
4. Rust declarations are the manifest source of truth for the first slice.
5. At least one canister family has declared state domains.
6. Every declared state domain has explicit version and storage.
7. Stable-memory domains have memory IDs.
8. Non-stable-memory domains explicitly declare no stable memory.
9. Duplicate memory IDs within one role are detected as fail.
10. Record/snapshot naming checks exist.
11. Migration coverage declarations exist for supported old versions.
12. Removed state domains require explicit disposition.
13. Lifecycle restore order and post-upgrade invariant declarations are
    represented.
14. Output does not leak persisted application values.
15. Commands do not mutate canisters, files, config, deployment truth, installed
    state, stable memory, or generated files.
16. Medic integration, if present, summarizes or links to state audit but does
    not run upgrade tests.
17. PocketIC helper APIs exist or are explicitly deferred.
18. Validation covers core/control-plane state libraries and at least one upgrade
    test path, or records a narrow reason for deferral.

## 20. Deferred

Deferred unless separately designed:

- generated migration code;
- general migration execution runtime;
- runtime introspection endpoint;
- source-authored manifest files;
- downstream app manifest authoring;
- fleet-global memory-ID reservation;
- state explorer or state dump;
- automatic repair of memory IDs;
- applying migrations outside existing lifecycle restore paths;
- build/deploy hard gates;
- evidence wrapping;
- report-version compatibility layers.

## 21. Closed Decisions

1. Rust declarations are canonical for 0.80.
2. Manifest files are output, not source of truth.
3. Runtime manifest introspection is deferred.
4. Commands use `--json`, not `--format json`.
5. State commands are no-mutation and privacy-preserving.
6. Memory IDs must be unique within one canister role, not globally across a
   fleet.
7. Medic integration is summary/next-action only.
8. Deploy plan is not a global state policy gate in 0.80.
9. PocketIC upgrade helper APIs come after declaration/report/audit models.
10. New 0.80 report wrappers are unversioned; `schema_version = 1` is the single
    accepted JSON shape for this hard cut.
