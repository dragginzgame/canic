source must not rely on serde attributes Candid does not support.

Current owner:

Intended owner:

now audits their Candid attribute usage.

## Findings

### CANIC-082-CANDID-SERDE-001

Severity: P2
Category: serialized_surface
Owner: Candid DTO/domain owners
Current location: Canic-owned Candid source modules
Intended location: Same owners, with maintained boundary guard

Evidence:
Candid documents support for explicit `serde(rename)` attributes when deriving
`CandidType`, while `rename_all` and `rename_all_fields` are not supported.
After the runtime enum label hard cut, the first audit found no
`serde(alias)` attributes in standalone form. Tightening the guard to catch
combined serde attributes exposed the shared HTTP method uppercase aliases,
which were removed in the companion hard-cut slice.

Risk:
Future Candid-bearing types could accidentally use serde-only rename helpers or
compatibility aliases, creating a mismatch between JSON expectations and the
actual Candid surface.

Decision:
Add a maintained source-shape guard that scans Canic-owned Candid source roots
and fails if a `CandidType` item uses unsupported `rename_all`/`rename_all_fields`
attributes or `serde(alias)`.

Regression test:
`candid_serde_boundary_guard::candid_types_use_supported_serde_rename_attributes`
guards the boundary.

## Fixes

- Added a focused Candid serde boundary guard test.
- Tightened the guard so `#[serde(rename = "...", alias = "...")]` is also
  rejected on Candid-bearing items.
- Kept the guard scoped to Candid-bearing source items, avoiding unrelated
  serde-only report/config models.

## Deferred

- Serde-only JSON/config/report models remain out of scope unless they derive
  `CandidType`.
