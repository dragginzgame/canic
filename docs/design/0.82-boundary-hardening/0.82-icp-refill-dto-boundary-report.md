## Findings

### CANIC-082-ICP-DTO-001

Severity: P2
Category: dto_boundary
Owner: `canic-core::dto::icp_refill`
Current location: `crates/canic-core/src/dto/icp_refill.rs`
Intended location: `crates/canic-core/src/domain/icp_refill.rs`

Evidence:
`view::icp_refill`, storage ops, workflow execution, runtime cycles workflow,
manual/hub/replay workflows, and metrics tests imported ICP refill
mode/status/error-code values from the DTO module even though those values
drive internal operation state, request routing, retry decisions, and terminal
state classification.

Risk:
The DTO module was acting as the semantic owner for internal operation values,
blurring the boundary between passive transport structs and domain values used
by storage/workflow code.

Decision:
Fixed with a no-behavior-change ownership move. The enums now live in
`domain::icp_refill`, while `dto::icp_refill` publicly re-exports them so
existing API paths and serialized shapes remain stable.

Regression test:
`dto::icp_refill::tests::reexported_status_and_error_code_roundtrip_through_candid`
and `dto::icp_refill::tests::reexported_mode_roundtrips_through_candid` prove
the public DTO path still round-trips the re-owned values through Candid.

## Fixes

- Added `domain::icp_refill` as the owner for `IcpRefillMode`,
  `IcpRefillStatus`, and `IcpRefillErrorCode`.
- Re-exported the same values from `dto::icp_refill`.
- Updated internal view/storage/workflow/metrics imports to use the domain
  owner.
- Added a Candid roundtrip regression test through the DTO path.

## Deferred

- ICP refill request/response structs remain DTO-owned.
- Stable ICP refill records remain unchanged.
