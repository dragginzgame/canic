# Canic 0.82 Design: Boundary Hardening Cleanup

Date: 2026-07-05

## Status

- Status: implementation-spec draft
- Release line: tentative 0.82 planning label
- Decision style: architecture cleanup and auditability
- Primary implementation target: reduce internal boundary leaks, duplicated
  semantics, and half-migrated ownership after the 0.78 through 0.81
  operator-platform slices.
- First slice decision: diagnostic/report ownership hardening.
- Current state: Canic has explicit layering guidance:
  `endpoints -> workflow -> policy -> ops -> model`. DTOs are passive boundary
  data, model owns authoritative state and invariants, ops owns deterministic
  state access and approved single-step platform side effects, policy is pure
  decision logic, workflow owns orchestration, and endpoints/macros
  authenticate and delegate.

## 1. Summary

0.82 is a named cleanup track, not a feature grab bag. Its job is to make
Canic's internal architecture stricter and easier to audit after the
operator-facing surfaces added in 0.78 through 0.81.

This design hardens current repo boundaries. It does not propose a wholesale
module-tree rewrite, package rename, or behavior-changing refactor. Cleanup
slices must be small, explicit, and closed with evidence.

0.82 starts with diagnostic/report ownership hardening because the recent
operator-platform releases introduced multiple report surfaces:

- `canic medic`
- `canic deploy plan`
- `canic state audit`
- `canic state manifest`
- `canic inspect`
- guarded runtime introspection DTOs

Those surfaces carry diagnostic codes, categories, status derivation,
next-action semantics, source attribution, and text/JSON renderers. 0.82 should
make sure those semantics have one owner and are not redefined by CLI renderers
or copied between subsystems.

## 2. Hard-Cut Rule

When a hardening slice removes a legacy boundary, command, module surface, or
schema shape that the slice explicitly owns, the removal should be direct. This
design does not include compatibility shims, aliases, or deprecation wrappers.

Hard cuts apply only to surfaces explicitly owned by the slice. A slice must not
opportunistically remove unrelated surfaces. Unrelated legacy paths discovered
during a slice are recorded as follow-ups unless they must be handled to keep
the current slice safe.

## 3. Public And Serialized Surface Rules

By default, a 0.82 cleanup slice must preserve:

1. Public CLI behavior.
2. Endpoint names.
3. Candid method names.
4. JSON schema fields.
5. DTO serialized shape.
6. Stable state layout.
7. Deployment truth schema.
8. Evidence/report schema.

Any change to one of these surfaces must be declared in the slice contract and
tested explicitly. Incidental public, JSON, Candid, stable-state, deployment
truth, or evidence-shape changes are not allowed.

## 4. Slice Contract

Each 0.82 hardening slice must define:

1. Boundary being hardened.
2. Current owner.
3. Intended owner.
4. Public surfaces affected.
5. Serialized surfaces affected.
6. Commands/endpoints affected.
7. Expected behavior changes, if any.
8. Explicit non-changes.
9. Tests that protect behavior.
10. Follow-up candidates deliberately not handled.

A slice must not expand after implementation starts. Newly discovered issues are
recorded as follow-ups unless they are required to keep the current slice safe.

### Slice Template

```text
Name:
Boundary:
Current owner:
Intended owner:
Files/modules in scope:
Files/modules explicitly out of scope:
Public surfaces affected:
Serialized surfaces affected:
Commands/endpoints affected:
Expected behavior change:
Expected non-changes:
Validation:
Rollback plan:
Follow-ups:
```

### Behavior Impact Label

Every 0.82 slice must label behavior impact before implementation:

```text
no_behavior_change
behavior_change_declared
serialized_surface_change_declared
command_surface_change_declared
```

If behavior changes are declared, the slice must include before/after behavior
and regression tests. If serialized or command surfaces change, the slice must
pin the new surface in tests and changelog notes.

## 5. Goals

1. Keep endpoint authentication at endpoint guards.
2. Keep DTOs passive and boundary-owned.
3. Keep policy pure: no mutation, async, timers, IC calls, DTOs, storage access,
   or serialization.
4. Keep conversions in `ops::*`.
5. Keep workflow focused on multi-step orchestration.
6. Consolidate duplicated config parsing.
7. Consolidate diagnostic code ownership by producer.
8. Remove stale legacy paths from active code as hard cuts only when a slice
   owns that boundary.
9. Make lifecycle/init/post-upgrade ownership clearer.
10. Reduce public visibility that is broader than intended.
11. Keep operator report status and safety semantics owned by report producers,
    not renderers.
12. Record follow-up candidates instead of silently expanding cleanup scope.

## 6. Non-Goals

0.82 does not:

1. Rewrite the module tree for aesthetics.
2. Rename Cargo packages.
3. Change package versions.
4. Change serialized DTOs incidentally inside cleanup slices.
5. Combine unrelated feature work with cleanup.
6. Hide behavior changes inside refactors.
7. Add lint suppressions when visibility or module boundaries can be fixed
   directly.
8. Move tests away from the code they cover unless ownership changes require
   it.
9. Introduce new product features.
10. Turn cleanup slices into performance optimization work.

## 8. Severity Rubric

P0:

- Boundary leak creates an auth bypass, mutation in a read-only path, state
  corruption, lifecycle upgrade risk, or public exposure of guarded
  runtime/state data.

P1:

- Active command, endpoint, or schema violates a hard-cut design.
- Policy performs mutation, async work, IC calls, storage access, timers, or
  serialization.
- Workflow constructs persisted records directly.
- CLI renderer redefines safety/status semantics.
- Lifecycle adapter schedules async work before invariants are restored.
- Serialized DTO/report shape changes incidentally.

P2:

- Duplicated config parsing or diagnostic ownership creates drift risk.
- Visibility is broader than needed but not externally exposed.
- Tests live far from the ownership boundary and obscure regressions.
- Stale active module path remains but is not currently reachable.

P3:

- Naming, comments, small visibility polish, archived-doc clarity, or snapshot
  readability.

## 9. Candidate Tracks

### 9.1 DTO And View Boundaries

Checks:

1. DTOs remain data-only boundary contracts.
2. DTOs do not own validation that depends on local runtime config.
3. DTOs do not own authorization decisions.
4. DTOs do not import ops, workflow, policy, storage, or runtime modules.
5. Request and mutation DTOs do not implement `Default` unless the default is
   semantically neutral and cannot authorize, mutate, enable a feature, or
   select production behavior.
6. Internal read-only projections live under `view/`.
7. `view/` types are internal read-only projections derived from model/state.
   They are not API DTOs unless explicitly converted at the boundary.
8. Persisted storage schema ends in `*Record`.
9. Canonical snapshots use `*Data`.
10. Boundary type aliases do not carry cross-layer semantics.

Vocabulary:

```text
Dto:
  wire/API shape, not authoritative persisted state.

Data:
  canonical snapshot/import/export shape.

Record:
  persisted storage schema.

State:
  in-memory aggregate or module-local storage wrapper.

View:
  internal read-only projection.
```

### 9.2 Policy Purity

Checks:

1. Policy modules do not import storage.
2. Policy modules do not call IC APIs.
3. Policy modules do not serialize DTOs.
4. Policy modules do not spawn timers or async work.
5. Policy outputs use named decision structs or enums.
6. Policy inputs are model/value types or dedicated policy input structs.
7. Policy outputs are decisions, never side effects.

Policy modules must not import:

```text
storage
runtime
cdk
ic_cdk
timers
management canister clients
CLI/config file readers
wire/storage serializers
DTO-specific encoding modules
```

Small model/value types are allowed when they are pure inputs to a policy
decision.

### 9.3 Ops And Workflow Ownership

Definitions:

```text
ops:
  single-step deterministic state access, conversion, and approved platform
  side effects.

workflow:
  multi-step orchestration over ops and policy.

model:
  authoritative state types and invariants.
```

Checks:

1. Record construction and mutation stay in ops.
2. Workflow calls ops and policy but does not construct persisted records.
3. Workflow may assemble use-case inputs, but must not directly mutate storage.
4. Workflow may call ops functions that mutate storage.
5. Workflow must not call `cdk`, management canister, timer, or direct platform
   APIs unless a slice explicitly owns an exception.
6. Single-step platform side effects are isolated in ops.
7. Multi-step orchestration stays in workflow.
8. Error conversion boundaries remain explicit.

### 9.4 Config And Diagnostics

This is the first 0.82 hardening track.

Checks:

1. Config parsing has one owner per config type.
2. Diagnostic codes are owned by the producer that emits them.
3. Shared diagnostic categories are explicit and stable.
4. Text output and JSON output stay derived from the same report model.

Report producers own:

```text
diagnostic codes
status derivation
category meanings
next-action semantics
safety language
source attribution semantics
```

CLI renderers own:

```text
text formatting
serialization mechanics for existing report models
CLI-specific exit-code mapping
terminal wrapping and ordering
```

Report producers also own JSON schema fields and meanings. CLI renderers may
emit JSON, but they must not own or reinterpret the report schema.

CLI renderers must not:

```text
invent checks
reinterpret fail/warn/pass
hide blockers
downgrade safety findings
fill runtime-observed facts from local desired config
redefine deployment-safety language
```

Exit-code mapping may be CLI-owned, but it must consume the report producer's
status and severity without recomputing, downgrading, or hiding blockers.

Not all duplication is a boundary violation. Duplicate strings or shapes should
only be consolidated when they represent the same semantic owner. Similar
output in different report families may remain separate when the meanings
differ.

Do not replace producer-owned diagnostic codes with one global mega-enum unless
a separate design proves shared ownership is better. Shared categories and
statuses are allowed; producer-owned codes remain preferred.

### 9.5 Lifecycle

Checks:

1. `canic::start!` stays thin.
2. Lifecycle adapters restore synchronously.
3. Lifecycle adapters schedule async work without awaiting.
4. User hooks run after Canic invariants are restored.
5. Post-upgrade invariant checks have one canonical owner.
6. Install, reinstall, `pre_upgrade`, `post_upgrade`, and timer scheduling call
   graphs are explicitly audited in lifecycle slices.
7. Lifecycle adapters do not perform async work before invariants are restored.
8. Lifecycle adapters do not call user hooks before Canic invariants are
   restored.
9. Lifecycle adapters do not duplicate state restore logic across roles.
10. Lifecycle cleanup includes PocketIC upgrade tests when behavior changes.

## 10. First Slice: Diagnostic Ownership Hardening

Name:

```text
0.82.1 Diagnostic Ownership Hardening
```

Boundary:

```text
Report producers own diagnostic semantics; CLI renderers only render.
```

Scope:

1. Medic report/check model.
2. Deploy plan report/diagnostics.
3. State audit report/check model.
4. Runtime inspect report/status wrapper.
5. CLI text/JSON renderers for those reports.
6. Shared diagnostic category/status definitions, if any.

Behavior impact:

```text
no_behavior_change
```

0.82.1 must preserve existing command behavior, endpoint behavior, JSON fields,
text output meaning, status derivation results, and exit-code behavior unless a
separate slice contract explicitly declares the behavior change. If
implementation discovers that a behavior change is required, stop and either
record it as a follow-up or open a separate slice with
`behavior_change_declared`.

Report family ownership:

| Report family | Producer | Renderer |
| --- | --- | --- |
| `MedicReport` | Medic/preflight domain | `canic-cli` medic renderer |
| `DeploymentPlanReport` | Deployment-plan report builder | `canic-cli` deploy-plan renderer |
| `StateAuditReport` | State-audit engine | `canic-cli` state renderer |
| `StateManifest` | State declaration/manifest builder | `canic-cli` state renderer |
| `InspectReport` | Runtime inspect client/report builder | `canic-cli` inspect renderer |
| Runtime introspection DTOs | Canister runtime endpoint | `canic-cli` inspect renderer consumes decoded DTOs |

Checks:

1. Each diagnostic code is defined near the producer that emits it.
2. Status derivation lives with the report producer, not the renderer.
3. Text and JSON output derive from the same report model.
4. CLI renderer does not downgrade fail/blocker/unsupported states.
5. Shared categories/status enums are explicit if shared.
6. No duplicate code strings have different meanings.
7. No renderer-only checks exist.
8. Runtime-observed facts remain distinct from local config, deployment records,
   and deployment truth.
9. Exit-code mapping consumes producer status/severity instead of recomputing
   report meaning.

Test ownership:

1. Producer tests assert status derivation, diagnostic codes, source
   attribution, and safety semantics.
2. Renderer tests assert formatting, ordering, JSON emission, and exit-code
   mapping.
3. For no-behavior-change report-ownership slices, existing text/JSON outputs
   should be protected by snapshot or field-level assertions before refactoring.

Non-scope:

1. No schema changes.
2. No command changes.
3. No endpoint changes.
4. No new diagnostics except ownership metadata if needed.
5. No broad renderer rewrite.
6. No deploy/medic/state/inspect behavior changes unless explicitly declared by
   the slice contract.

Expected validation:

```text
cargo fmt --all -- --check
cargo test --locked -p canic-cli medic
cargo test --locked -p canic-cli deploy
cargo test --locked -p canic-cli state
cargo test --locked -p canic-cli inspect
cargo test --locked -p canic --test protocol_surface
git diff --check
```

Use narrower filters if the slice touches only one report family.

## 11. Legacy Command Surface Sweep

Do not start by randomly removing old commands. Use hard-cut specs that already
declared forms dead or rejected.

First sweep candidates:

```text
canic info medic <deployment>
canic medic <deployment>
canic medic --blob-storage <target>
canic medic --auth-renewal <issuer>
canic state audit <role>
canic state manifest <role>
canic state migrate
canic state repair
canic state explore
canic state dump
canic inspect <ambiguous-target>
canic inspect deployment <deployment>
canic inspect deployment <deployment> --all
canic topology <deployment>
canic runtime status <deployment>
```

Classify each occurrence as:

```text
active route
help text only
test fixture only
archived doc only
dead code
safe delete
requires separate slice
```

The first sweep is read-only classification unless the owning slice explicitly
includes removal. It should not delete routes, docs, or tests
opportunistically. Compatibility routes, hidden aliases, and active fallbacks
that contradict recent hard-cut specs are P1 or higher.

## 12. Import-Boundary Audit Inputs

Mechanical import scans should inform each slice:

```text
policy imports storage/runtime/cdk/timer/serde_json
dto imports ops/workflow/storage/runtime
workflow imports storage record constructors directly
cli renderer imports core safety policy directly
lifecycle imports feature-specific repair logic before invariant restore
```

These scans are audit inputs, not automatic rewrite instructions. If a scan
finds a real boundary violation outside the current slice, record it as a
follow-up unless it affects slice safety.

## 13. Future Release Cleanup Rule

0.82 is a dedicated hardening line for cross-cutting boundary issues created or
revealed by the 0.78 through 0.81 operator-platform work.

Future feature releases should include local boundary cleanup only for code they
touch. Cross-cutting cleanup belongs to a named hardening slice.

## 15. Implementation Slices

1. Pick one ownership boundary and fill out the slice contract.
2. Document current imports, public surfaces, serialized surfaces, and owner.
3. Move or narrow only the code needed for that boundary.
4. Keep serialized output unchanged unless the slice explicitly owns a schema
   change.
5. Add targeted tests for behavior that could regress.
6. Run focused validation before broad cleanup.
7. Record follow-up candidates instead of expanding the slice mid-flight.
8. Produce a boundary hardening report before closeout.

Recommended sequence:

1. Config and diagnostics ownership.
2. Policy purity.
3. Lifecycle.
4. DTO and view boundaries.
5. Ops and workflow ownership.

Lifecycle is high-value but higher-risk. Do not start there unless a concrete
lifecycle bug is known.

## 16. Validation

Validation depends on the touched surface. Common focused gates:

```text
cargo fmt --all -- --check
cargo check --locked -p canic-core -p canic
cargo test --locked -p canic --test protocol_surface
cargo test --locked -p canic-cli
cargo test --locked -p canic-host deployment_truth --lib
git diff --check
```

Use narrower package/test filters for small slices and broaden when a shared
boundary, CLI surface, or serialized payload changes.

Lifecycle behavior changes require PocketIC install/upgrade/query coverage.
Command-surface changes require parser/help/rejection tests. Report-shape
changes require JSON/text renderer tests and schema/field assertions.

## 17. Acceptance Criteria

Close 0.82 only when:

1. At least one boundary hardening slice is completed.
2. The slice has a written boundary report.
3. The slice declares public and serialized surface impact.
4. No incidental DTO, JSON, Candid, stable-state, deployment-truth, or evidence
   shape changes occurred.
5. No unrelated feature work was included.
6. Focused validation for the touched boundary passes.
7. Follow-up candidates are recorded instead of silently fixed
   opportunistically.
8. Any removed legacy surface was owned by the slice and tested as removed.
9. Visibility was narrowed where safe instead of suppressed.
10. The hardening report lists remaining known boundary risks.

## 18. Closed Decisions

1. First audited boundary: config and diagnostics ownership.
2. 0.82 is a dedicated hardening line for cross-cutting cleanup.
3. Future releases should perform only local cleanup unless a named hardening
   slice is opened.
4. Legacy command removal starts from recent hard-cut specs and begins as an
   audit.
5. First release-gate lint family to pursue: policy purity imports.
6. `BoundaryHardeningReport` is docs-only in 0.82.
7. Policy-purity import checks are audit inputs in 0.82.1, not release gates.
8. Lifecycle hardening must wait for PocketIC coverage of install/query,
   install/timer registration, upgrade/restore before hooks, upgrade/timer
   scheduling after restore, unsupported state version failure, and reinstall
   stale-state behavior.

## 19. Open Questions

1. Which policy-purity import checks can be enforced by tests without creating
   brittle source-text assertions?
2. Which diagnostic/report ownership checks can be implemented as robust tests
   instead of one-off audit notes?
