## Slice Contract

Boundary:
Backup create should not repair an existing plan-only layout by synthesizing a
missing execution journal. Restore run should consume structured command output
for snapshot upload IDs and stopped-canister preconditions instead of accepting
legacy human text or alternate JSON keys. Backup manifests, backup plans, and
backup execution preflight contracts should reject unknown fields instead of
ignoring stale or misspelled data.

Current owner:
Backup create persistence still synthesized `backup-execution-journal.json`
from an existing `backup-plan.json` when the journal was missing. Restore run
accepted human text upload output and an alternate `uploaded_snapshot_id` key,
and stopped-canister precondition checks accepted human status text. Backup
manifest and plan structs accepted unknown JSON fields through serde defaults,
and an unused plan-facing backup receipt shape remained in active plan types.

Intended owner:
Backup layout persistence requires the current plan-plus-execution-journal
layout. Restore runner command-output parsing requires structured JSON owned by
the maintained helper command contract. Backup manifest and plan contracts are
strict current schemas; execution journals own per-operation backup receipts.

## Findings

### CANIC-082-BACKUP-RESTORE-HARDCUT-001

Severity: P2
Category: legacy_surface
Owner: `canic-cli::backup::create::persistence`
Current location: `crates/canic-cli/src/backup/create/persistence.rs`
Intended location: current backup layout persistence

Evidence:
`persist_backup_create_layout` synthesized an execution journal from an
existing plan when `backup-execution-journal.json` was missing.

Risk:
The fallback let incomplete, hand-edited, or pre-journal layouts become
accepted current layouts without requiring the durable execution journal that
other backup commands now use as the authoritative progress record.

Decision:
Require `backup-execution-journal.json` whenever an existing backup plan layout
is reused.

Regression test:
`cargo test --locked -p canic-cli backup::tests::create --lib`

### CANIC-082-BACKUP-RESTORE-HARDCUT-002

Severity: P2
Category: legacy_surface
Owner: `canic-backup::restore::runner::status`
Current location: `crates/canic-backup/src/restore/runner/status.rs`
Intended location: structured JSON restore upload output

Evidence:
`parse_uploaded_snapshot_id` accepted a legacy `uploaded_snapshot_id` JSON key
and human text output with a colon-separated uploaded snapshot label.

Risk:
The restore runner had two accepted names and a human-output parser for one
current value, which made helper command output harder to audit and encouraged
legacy text output to remain active.

Decision:
Require JSON with the maintained `snapshot_id` field.

Regression test:
`cargo test --locked -p canic-backup restore --lib`

### CANIC-082-BACKUP-RESTORE-HARDCUT-004

Severity: P2
Category: legacy_surface
Owner: `canic-backup::manifest`
Current location: `crates/canic-backup/src/manifest/types.rs`
Intended location: strict current backup manifest schema

Evidence:
Backup manifest data structs used serde's default unknown-field behavior, and
active tests asserted that an old removed source-snapshot field was ignored.

Risk:
Unknown backup manifest fields could hide stale, misspelled, or retired data in
a durable artifact consumed by restore planning and verification.

Decision:
Added `deny_unknown_fields` to backup manifest structs and replaced the stale
ignored-field test with current strict-manifest rejection coverage.

Regression test:
`cargo test --locked -p canic-backup manifest --lib`

### CANIC-082-BACKUP-RESTORE-HARDCUT-005

Severity: P2
Category: legacy_surface
Owner: `canic-backup::plan`
Current location: `crates/canic-backup/src/plan/types.rs`
Intended location: strict current backup plan and preflight schemas

Evidence:
Backup plan and preflight data structs used serde's default unknown-field
behavior, and unused plan-facing `BackupReceipt` and `BackupReceiptStatus`
types remained after execution operation receipts became the maintained durable
receipt path.

Risk:
Unknown backup plan/preflight fields could hide stale or misspelled data before
mutation, and unused receipt types made receipt ownership look split between
plan and execution layers.

Decision:
Added `deny_unknown_fields` to backup plan and preflight structs, added
strict-plan rejection coverage, and removed the unused plan-facing receipt
types.

Regression test:
`cargo test --locked -p canic-backup plan --lib`

### CANIC-082-BACKUP-RESTORE-HARDCUT-003

Severity: P2
Category: legacy_surface
Owner: `canic-backup::restore::runner::precondition`
Current location: `crates/canic-backup/src/restore/runner/precondition.rs`
Intended location: structured JSON restore precondition output

Evidence:
`status_output_reports_stopped` accepted stopped status from either JSON output
or human status text.

Risk:
The human-output parser kept a loose compatibility path in a restore
precondition that should fail closed when helper output is not structured.

Decision:
Require JSON status output for stopped-canister precondition checks.

Regression test:
`cargo test --locked -p canic-backup restore --lib`

## Fixes

- Backup create now rejects plan-only layouts missing
  `backup-execution-journal.json`.
- Restore upload snapshot ID parsing now accepts only JSON `snapshot_id`.
- Restore stopped-canister precondition parsing now accepts only JSON stopped
  status shapes.
- Restore tests and fake helper scripts now emit current JSON upload output.
- Backup manifests, backup plans, and backup preflight contracts now reject
  unknown fields.
- Removed unused plan-facing backup receipt types after execution journals
  became the maintained receipt owner.

## Deferred

- Backup download-journal cleanup remains a separate slice because the restore
  and artifact execution paths still rely on current journal behavior.
- Restore apply journal receipt field names remain unchanged; this slice only
  cuts parser inputs, not durable receipt schema.
