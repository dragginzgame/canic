## Findings

### CANIC-082-POOL-DTO-001

Severity: P2
Category: dto_boundary
Owner: `canic-core::dto::pool`
Current location: `crates/canic-core/src/dto/pool.rs`
Intended location: `crates/canic-core/src/domain/pool.rs`

Evidence:
`workflow::pool::import` and `workflow::pool::recycle` inspected
`dto::pool::CanisterPoolStatus` when deciding whether an import/recycle path
should add, requeue, skip, or treat a recycle as already present.

Risk:
The DTO module was acting as the semantic owner for an internal pool state
value that participates in workflow control flow, blurring the passive
boundary-payload rule for DTO modules.

Decision:
Fixed with a no-behavior-change ownership move. The enum now lives in
`domain::pool`, while `dto::pool` publicly re-exports it so existing API paths
and serialized shapes remain stable.

Regression test:
`dto::pool::tests::reexported_pool_status_roundtrips_through_candid` proves the
public DTO path still round-trips the re-owned value through Candid.

## Fixes

- Added `domain::pool` as the owner for `CanisterPoolStatus`.
- Re-exported the same value from `dto::pool`.
- Updated storage pool mapping and pool workflows to import the domain owner.
- Added a Candid roundtrip regression test through the DTO path.

## Deferred

- Pool command/response structs remain DTO-owned.
- Stable pool records remain unchanged.
