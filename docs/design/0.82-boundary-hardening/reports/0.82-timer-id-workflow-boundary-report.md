## Slice Contract

Boundary:
Timer handles are runtime timer ops values. Runtime timer workflow owns timer
scheduling orchestration, not handle re-export ownership.

Current owner:
`canic-core::workflow::runtime::timer` re-exported
`ops::runtime::timer::TimerId` alongside `TimerWorkflow`.

Intended owner:
`canic-core::ops::runtime::timer` owns `TimerId` and timer ops.
`canic-core::workflow::runtime::timer` owns `TimerWorkflow`.

Files/modules in scope:

- `crates/canic-core/src/workflow/runtime/timer.rs`
- `crates/canic-core/src/api/timer.rs`
- `crates/canic-core/src/workflow/runtime/intent.rs`

Files/modules explicitly out of scope:

- Timer scheduling behavior.
- Runtime timer metric labels.
- Lifecycle timer facade behavior.
- Timer persistence, stable state, or replay behavior.
- Public endpoint, CLI, JSON, Candid, deployment-truth, or evidence surfaces.

Public surfaces affected:
None intended. The removed workflow re-export was an internal `canic-core`
namespace shortcut.

Serialized surfaces affected:
None.

Commands/endpoints affected:
None.

Expected behavior change:
None.

Expected non-changes:

- Timer lifecycle and scheduling behavior remain unchanged.
- Lifecycle facade APIs still return the same timer handle type.
- Intent cleanup timer scheduling still uses `TimerWorkflow`.
- Runtime timer metrics and timer mode reporting remain unchanged.

Validation:

- `cargo fmt --all -- --check` passed.
- `cargo check --locked -p canic-core --lib` passed.
- `cargo test --locked -p canic-core runtime --lib` passed.

Rollback plan:
Restore the `TimerId` re-export in `workflow::runtime::timer`, then move API
and workflow imports back through the workflow namespace.

Follow-ups:

- Keep timer handle ownership in `ops::runtime::timer`.
- Keep workflow timer orchestration in `workflow::runtime::timer`.
- Avoid reintroducing workflow re-exports for passive runtime ops values.

## Findings

### CANIC-082-TIMER-ID-WORKFLOW-001

Severity: P3
Category: workflow_ownership
Owner: `canic-core::workflow::runtime::timer`
Current location: `crates/canic-core/src/workflow/runtime/timer.rs`
Intended location: `crates/canic-core/src/ops/runtime/timer.rs`

Evidence:
`workflow::runtime::timer` re-exported `TimerId`, which is defined by runtime
timer ops. Internal API and intent workflow code imported the timer handle
through the workflow namespace.

Risk:
The workflow namespace looked like the owner of a passive runtime timer handle,
blurring the boundary between scheduling orchestration and timer ops values.

Decision:
Fixed with a no-behavior-change import cleanup. `TimerId` is imported from
`ops::runtime::timer`, while `TimerWorkflow` remains in the workflow namespace.

Regression test:
Focused runtime tests continue to cover timer workflow behavior and runtime
status/timer surfaces.

## Fixes

- Removed the `TimerId` re-export from `workflow::runtime::timer`.
- Updated lifecycle timer API imports to use the runtime timer ops owner.
- Updated intent runtime workflow imports to use the runtime timer ops owner.

## Deferred

- Timer scheduling behavior remains unchanged.
- Runtime timer metric dimensions and labels remain unchanged.
