## Slice Contract

Boundary:
Backup create should not repair an existing plan-only layout by synthesizing a
missing execution journal. Restore run should consume structured command output
for snapshot upload IDs and stopped-canister preconditions instead of accepting
human text or alternate JSON keys.

Current owner:
Backup create persistence synthesized `backup-execution-journal.json` from an
existing `backup-plan.json` when the journal was missing. Restore run accepted
human text upload output, an alternate `uploaded_snapshot_id` key, and human
status text for stopped-canister preconditions.

Intended owner:
Backup layout persistence requires the current plan-plus-execution-journal
layout. Restore runner command-output parsing requires structured JSON owned by
the maintained helper command contract.

Files/modules in scope:

- `crates/canic-cli/src/backup/create/persistence.rs`
- `crates/canic-cli/src/backup/tests/create/rejection.rs`
- `crates/canic-backup/src/restore/runner/status.rs`
- `crates/canic-backup/src/restore/runner/precondition.rs`
- `crates/canic-backup/src/restore/tests/apply_journal.rs`

- `crates/canic-cli/src/restore/tests/options.rs`
- `crates/canic-cli/src/restore/tests/run.rs`

Files/modules explicitly out of scope:

- Backup and restore JSON schema strictness.
- Backup plan, manifest, journal, and restore apply journal field names.
- Backup download-journal migration/removal.
- Operator command names and flags.
- Endpoint method names.
- Candid, deployment truth, evidence/report, and stable-state schemas.

Public surfaces affected:
None.

Serialized surfaces affected:
None.

Commands/endpoints affected:
No command or endpoint names changed. Existing backup create and restore run
commands now fail closed for the retired compatibility inputs described below.

Expected behavior change:

- Existing backup create output layouts with `backup-plan.json` but no
  `backup-execution-journal.json` are rejected instead of repaired.
- Restore upload snapshot ID parsing now accepts JSON with `snapshot_id` only.
- Restore stopped-canister precondition parsing now accepts JSON status output
  only.

Expected non-changes:

- New backup create layouts still write both plan and execution journal files.
- Existing complete backup create layouts still resume through the maintained

- Restore apply journal receipt fields remain unchanged, including the current
  `uploaded_snapshot_id` receipt field.
- Restore run still records uploaded snapshot IDs when the helper command emits
  the maintained JSON shape.
- Restore command surfaces, endpoint surfaces, and durable report schemas are
  unchanged.

Rollback plan:
Restore the old parser and layout-repair paths only if the maintainer

## Findings

### CANIC-082-BACKUP-RESTORE-HARDCUT-001

Severity: P2
Category: legacy_surface
Owner: `canic-cli::backup::create::persistence`
Current location: `crates/canic-cli/src/backup/create/persistence.rs`
Intended location: current backup layout persistence

Evidence:
`persist_backup_create_layout` synthesized an execution journal from an
existing plan when `backup-execution-journal.json` was missing.

Risk:
The fallback let incomplete, hand-edited, or pre-journal layouts become
accepted current layouts without requiring the durable execution journal that
other backup commands now use as the authoritative progress record.

Decision:
Require `backup-execution-journal.json` whenever an existing backup plan layout
is reused.

Regression test:
`cargo test --locked -p canic-cli backup::tests::create --lib`

### CANIC-082-BACKUP-RESTORE-HARDCUT-002

Severity: P2
Category: legacy_surface
Owner: `canic-backup::restore::runner::status`
Current location: `crates/canic-backup/src/restore/runner/status.rs`
Intended location: structured JSON restore upload output

Evidence:
`parse_uploaded_snapshot_id` accepted an alternate `uploaded_snapshot_id` JSON
key and human text output with a colon-separated uploaded snapshot label.

Risk:
The restore runner had two accepted names and a human-output parser for one
current value, which made helper command output harder to audit and encouraged
legacy text output to remain active.

Decision:
Require JSON with the maintained `snapshot_id` field.

Regression test:
`cargo test --locked -p canic-backup restore --lib`

### CANIC-082-BACKUP-RESTORE-HARDCUT-003

Severity: P2
Category: legacy_surface
Owner: `canic-backup::restore::runner::precondition`
Current location: `crates/canic-backup/src/restore/runner/precondition.rs`
Intended location: structured JSON restore precondition output

Evidence:
`status_output_reports_stopped` accepted stopped status from either JSON output
or human status text.

Risk:
The human-output parser kept a loose compatibility path in a restore
precondition that should fail closed when helper output is not structured.

Decision:
Require JSON status output for stopped-canister precondition checks.

Regression test:
`cargo test --locked -p canic-backup restore --lib`

## Fixes

- Backup create now rejects plan-only layouts missing
  `backup-execution-journal.json`.
- Restore upload snapshot ID parsing now accepts only JSON `snapshot_id`.
- Restore stopped-canister precondition parsing now accepts only JSON stopped
  status shapes.
- Restore tests and fake helper scripts now emit current JSON upload output.

## Deferred

- Backup download-journal cleanup remains a separate slice because the restore
  and artifact execution paths still rely on current journal behavior.
- Restore apply journal receipt field names remain unchanged; this slice only
  cuts parser inputs, not durable receipt schema.
