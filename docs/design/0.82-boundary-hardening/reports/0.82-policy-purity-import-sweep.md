## Slice Contract

Boundary:
Policy modules should remain pure decision logic. They must not import storage,
runtime execution, timers, IC management clients, async spawners, or
wire/storage serializers. Imports should make the policy input and output
surface explicit enough to audit.

Current owner:
Policy modules under `canic-core::domain::policy` and deployment-truth policy
helpers under `canic-host`.

Intended owner:
Keep policy dependency surfaces explicit. Value types and report model types may
be policy inputs or outputs, but hidden parent wildcard imports should not
obscure dependencies.

Files/modules in scope:

- `crates/canic-core/src/domain/policy/**`
- `crates/canic-host/src/deployment_truth/lifecycle/authority_plan/policy/mod.rs`

## Findings

### CANIC-082-POLICY-001

Severity: P3
Category: policy_purity
Owner: deployment-truth lifecycle policy helpers
Current location:
`crates/canic-host/src/deployment_truth/lifecycle/authority_plan/policy/mod.rs`

Evidence:
The module imported its grandparent scope with `use super::super::super::*`.

Risk:
Wildcard parent imports make it harder to audit whether a policy helper depends
only on value/report types or also sees side-effect helpers from its parent
module.

Decision:
Fixed by replacing the wildcard import with explicit imports for
`stable_json_sha256_hex` and the required deployment-truth lifecycle model
types.

Regression test:
Covered by `cargo test --locked -p canic-host lifecycle`.

### CANIC-082-POLICY-004

Severity: P3
Category: policy_purity
Owner: `canic-core::domain::policy`
Current location: `crates/canic-core/src/domain/policy/**`

Evidence:
Core policy modules imported pure value types such as
`crate::cdk::types::Principal`, `crate::cdk::types::Cycles`, and
`crate::cdk::types::BoundedString64` directly from the CDK namespace. The scoped
scan found no storage/runtime/timer/async/IC-call imports.

Risk:
The value-type imports are not side effects, but the `cdk::types` path can be
mistaken for a runtime dependency when auditing policy purity.

Decision:
Fixed by introducing `crate::domain::value` as a pure value alias namespace and
switching core policy modules to import `Principal`, `Cycles`,
`BoundedString64`, and test-only `TC` through that namespace. The aliases
re-export the same underlying types, so serialized shapes, equality semantics,
and policy behavior remain unchanged.

Regression test:
Covered by `cargo test --locked -p canic-core policy --lib` and the scoped
policy import scan.

### CANIC-082-POLICY-005

Severity: P2
Category: policy_purity
Owner: `canic-core::domain::policy`
Current location: `crates/canic-core/src/domain/policy/**`

Evidence:
Several policy modules accepted config schema structs such as `LogConfig`,
`CanisterConfig`, `TopupPolicy`, `IcpRefillPolicy`, `ScalingConfig`,
`ScalePool`, `ShardPool`, and `ShardPoolPolicy` as direct inputs. The scoped
scan found no storage/runtime/timer/async/IC-call imports.

Risk:
Config schema inputs may couple policy decisions to config parsing shapes. That
is not a side effect, but it can make future policy-purity gates less precise.

Decision:
Fixed. Log retention, cycle top-up, scaling, sharding, ICP refill, and topology
registry policy now receive dedicated policy input/rules/shape structs mapped
by workflow or tests before policy evaluation.

Regression test:
Covered by `cargo test --locked -p canic-core policy --lib` and the scoped
policy import scan.

### CANIC-082-POLICY-006

Severity: P2
Category: policy_purity
Owner: `canic-core::domain::policy::topology::registry`
Current location: `crates/canic-core/src/domain/policy/topology/registry.rs`

Evidence:
Topology registry policy imported `CanisterConfig` and `CanisterKind` from
`crate::config::schema` to decide root/service/singleton/replica/shard/instance
registration rules.

Risk:
The import is pure, but it couples registry policy to the serialized config
shape. A future automated policy-purity gate would need either an exception or
a smaller registry-policy input shape.

Decision:
Fixed by introducing `RegistryCanisterKind` and `RegistryCanisterShape`.
Workflow provisioning and registry seam tests now map config into the policy
shape before calling registry policy.

Regression test:
Covered by existing registry seam coverage plus `cargo test --locked -p
canic-core policy --lib`.

## Fixes

- Narrowed lifecycle authority policy imports from a parent wildcard to explicit
  model and digest imports.
- Narrowed external-upgrade verification policy imports from wildcard imports to
  explicit schema, digest, validation, error, and model imports.
- Narrowed the promotion policy text renderer import to the single report model
  type it renders.
- Added `crate::domain::value` as the pure value import namespace for policy
  modules and switched core policy imports away from direct `cdk::types` paths.
- Added policy-owned input/rules structs for log retention, cycle top-up,
  scaling, sharding, and ICP refill decisions so those policies no longer
  import serialized config schema types.
- Added policy-owned topology registry canister-shape inputs so registry policy
  no longer imports serialized canister config schema types.
