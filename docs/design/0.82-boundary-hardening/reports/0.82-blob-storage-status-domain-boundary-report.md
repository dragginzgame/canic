## Findings

### CANIC-082-BLOB-STATUS-DTO-001

Severity: P2
Category: dto_boundary
Owner: `canic-core::dto::blob_storage`
Current location: `crates/canic-core/src/dto/blob_storage.rs`
Intended location: `crates/canic-core/src/domain/blob_storage.rs`

Evidence:
Blob-storage billing readiness, warning, funding, payment-model, and gateway
sync status values were defined in the DTO module while the billing API used
them as semantic classification outputs.

Risk:
Blob-storage billing status semantics could drift between the public DTO
boundary and the billing domain status builder.

Decision:
Fixed with a no-behavior-change ownership move. The status values now live in
`domain::blob_storage`, while `dto::blob_storage` publicly re-exports them so
existing API paths and serialized shapes remain stable.

Regression test:
`dto::blob_storage::tests::billing_status_enums_roundtrip_candid_through_dto_path`
proves the DTO re-export path preserves the billing status Candid contract.

## Fixes

- Added `domain::blob_storage` as the owner for blob-storage billing status
  values.
- Re-exported the same values from the existing blob-storage DTO path.
- Updated the blob-storage billing API to import status values from the domain
  owner.

## Deferred

- Cashier request/result DTOs remain in `dto::blob_storage`.
- Blob-storage status response struct and billing behavior remain unchanged.
