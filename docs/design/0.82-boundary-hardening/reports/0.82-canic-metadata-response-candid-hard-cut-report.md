## Slice Contract

Boundary:
Host-side parsing of the `canic_metadata` version response used by live list
views and deployment inventory helpers.

Current owner:
`canic-host::canic_metadata`, with fallback parsing from raw Candid text and
ICP `response_candid` wrapper text.

Intended owner:
`canic-host::canic_metadata` consumes structured JSON values from the ICP query
wrapper. Candid text is not a fallback metadata-version response surface.

Files/modules in scope:

- `crates/canic-host/src/canic_metadata/mod.rs`
- `crates/canic-host/src/canic_metadata/tests.rs`
- `crates/canic-host/src/response_parse/mod.rs`

Files/modules explicitly out of scope:

- The `canic_metadata` endpoint method name and Candid signature.
- CLI list command names, options, and rendering.
- Cycle-balance parsing.
- Root bootstrap readiness parsing.
- Metrics/cycles report parsing, which is covered by a separate hard-cut
  report.
- Deployment truth, evidence, stable-state layout, or generated artifacts.

Public surfaces affected:
Live metadata version discovery no longer accepts raw Candid text or
`response_candid` text fallback output when extracting `canic_version`.

Serialized surfaces affected:
No endpoint, Candid, JSON, deployment-truth, evidence/report, or stable-state
schema changes. The host accepted input wrapper shape is intentionally narrowed
to structured JSON for this metadata response.

Commands/endpoints affected:
No command or endpoint name changes.

Expected behavior change:
ICP query output for `canic_metadata` version discovery must include a
structured JSON `canic_version` field. Raw Candid text and `response_candid`
wrapper text are rejected.

Expected non-changes:

- The `canic_metadata` endpoint Candid signature is unchanged.
- Successful live list rendering and metadata display behavior are unchanged
  when structured JSON is present.
- Cycle-balance and bootstrap-readiness fallback parsers remain out of scope.
- No compatibility fallback, alias, or shim is retained for metadata version
  discovery.

- `cargo test --locked -p canic-host response_parse --lib` passed.

Rollback plan:
Restore the removed raw Candid text parsing and `response_candid` fallback in
`canic_metadata`. This is not recommended unless Canic intentionally

## Findings

### CANIC-082-CANIC-METADATA-RESPONSE-CANDID-001

Severity: P2
Category: host_surface
Owner: `canic-host::canic_metadata`
Current location: `crates/canic-host/src/canic_metadata/mod.rs`
Intended location: structured JSON metadata parsing in
`canic-host::canic_metadata`

Evidence:
The host metadata parser extracted `canic_version` from structured JSON, raw
Candid text, or an ICP `response_candid` wrapper. That kept a rendered Candid
fallback accepted after adjacent typed report hard cuts.

Risk:
Host metadata discovery had multiple accepted response shapes for the same
version fact, which made pre-1.0 report/input ownership less strict and kept a
legacy ICP output path alive.

Decision:
Fixed with a hard cut. `canic_metadata` version parsing now reads only
structured JSON and rejects raw Candid text or `response_candid` wrapper text.

Regression test:
Host metadata tests now assert that raw Candid text and text-only
`response_candid` wrapper output are rejected for version discovery.

## Fixes

- Removed raw Candid text and `response_candid` fallback parsing from
  `canic_metadata` version discovery.
- Removed the now-unused generic Candid quoted-field parser helper.
- Updated host metadata tests to pin rejection of old fallback response
  shapes.

## Deferred

- Cycle-balance and bootstrap-readiness fallback parsers remain untouched and
  should be evaluated as their own explicit hard-cut slices if selected.
- No old-fallback compatibility tests were added; this is an intentional hard
  cut, not a compatibility surface.
