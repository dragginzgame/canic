installation and activation waits.

Current owner:
`canic-host::install_root::readiness::parsing`, with fallback parsing from ICP
`response_candid` wrapper text.

Intended owner:
Local replica calls decode typed Candid bytes through the replica-query path.
ICP CLI fallback output is parsed only from structured JSON values.

Files/modules in scope:

- `crates/canic-host/src/install_root/readiness/parsing/mod.rs`
- `crates/canic-host/src/install_root/tests/readiness_parse.rs`

Public surfaces affected:
Root bootstrap readiness polling no longer accepts `response_candid` text
fallback output when extracting `BootstrapStatusResponse` from ICP CLI output.

Serialized surfaces affected:
No endpoint, Candid, JSON, deployment-truth, evidence/report, or stable-state
schema changes. The host accepted input wrapper shape is intentionally narrowed
to structured JSON for this ICP CLI fallback response.

Commands/endpoints affected:
No command or endpoint name changes.

bootstrap status record or wrapped `Ok` record. `response_candid` wrapper text
is rejected. The local replica fast path continues to decode typed Candid bytes

- Direct local replica bootstrap-status query decoding is unchanged.
- Root bootstrap lifecycle behavior is unchanged when structured JSON or local
  typed bytes are available.
- No compatibility fallback, alias, or shim is retained for the ICP CLI
  bootstrap-readiness parser.

## Findings

### CANIC-082-BOOTSTRAP-READINESS-RESPONSE-CANDID-001

Severity: P2
Category: host_surface
Owner: `canic-host::install_root::readiness::parsing`
Current location:
`crates/canic-host/src/install_root/readiness/parsing/mod.rs`
Intended location:
structured JSON bootstrap-readiness parsing for ICP CLI output

Evidence:
`parse_bootstrap_status_value` decoded structured JSON records and also parsed
rendered Candid text from an ICP `response_candid` wrapper. That kept a
rendered Candid fallback accepted after adjacent typed report hard cuts.

Risk:
Root bootstrap readiness had multiple accepted response shapes for the same
status fact, leaving a legacy ICP output path alive before 1.0.

Decision:
Fixed with a hard cut. ICP CLI bootstrap-readiness parsing now reads only
structured JSON records or wrapped `Ok` records, while the separate local
replica path continues to decode typed Candid bytes directly.

Regression test:
Host readiness parser tests now assert that text-only `response_candid` wrapper
output is rejected for bootstrap-readiness discovery.

## Fixes

- Removed `response_candid` fallback parsing from bootstrap-readiness parsing.
- Removed now-unused host Candid-text helper functions from `response_parse`.
- Updated readiness parser tests to pin rejection of old fallback response
  shapes.

## Deferred

- No old-fallback compatibility tests were added; this is an intentional hard
  cut, not a compatibility surface.
