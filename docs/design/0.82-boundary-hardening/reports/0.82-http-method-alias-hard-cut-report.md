# Canic 0.82 HTTP Method Alias Hard Cut Report

Date: 2026-07-06

## BoundaryHardeningReport

```text
schema_version: 1
slice: 0.82 HTTP Method Alias Hard Cut
boundary: HTTP method Candid/Serde labels are canonical lowercase labels only
status: pass_with_followups
behavior_impact: serialized_surface_change_declared
```

## Slice Contract

Boundary:
HTTP method labels accepted through the shared `HttpMethod` value used by HTTP
DTOs, IC HTTP ops, and HTTP runtime metrics.

Current owner:
`canic-core::domain::http`

Intended owner:
`canic-core::domain::http`, with DTO, ops, and metrics modules re-exporting
the value through existing public paths.

Files/modules in scope:

- `crates/canic-core/src/domain/http.rs`
- `crates/canic-core/src/dto/http.rs`
- `crates/canic-core/tests/candid_serde_boundary_guard.rs`

Files/modules explicitly out of scope:

- Raw IC management-canister HTTP payload types.
- HTTP request/response DTO struct fields.
- Runtime HTTP metric labels.
- Endpoint routes, deployment truth, evidence, stable-state layout, or CLI
  command surfaces.

Public surfaces affected:
Public Rust paths remain unchanged. `HttpMethod` remains available through
`dto::http`, `ops::ic::http`, and `ops::runtime::metrics::http` re-exports.

Serialized surfaces affected:
`HttpMethod` no longer accepts uppercase `GET`, `HEAD`, or `POST` serde
aliases. Canonical labels remain `get`, `head`, and `post`.

Commands/endpoints affected:
None.

Expected behavior change:
Clients decoding `HttpMethod` through Serde/Candid-compatible paths must use
the canonical lowercase labels. Uppercase compatibility aliases are removed.

Expected non-changes:

- `HttpMethod::as_str()` still returns uppercase HTTP wire method text for
  outbound HTTP execution and metrics labels.
- Candid labels remain `get`, `head`, and `post`.
- HTTP request execution, metric recording, endpoint routes, and public Rust
  imports remain unchanged.

Validation:

- `cargo test --locked -p canic-core dto::http --lib` passed.
- `cargo test --locked -p canic-core --test candid_serde_boundary_guard` passed.

Rollback plan:
Restore the uppercase serde aliases on `HttpMethod`. This is not recommended
unless Canic intentionally reintroduces HTTP method compatibility labels.

Follow-ups:

- Keep raw IC management-canister HTTP method payload types separate from the
  shared Canic domain value.

## Findings

### CANIC-082-HTTP-METHOD-ALIAS-001

Severity: P2
Category: serialized_surface
Owner: `canic-core::domain::http`
Current location: `crates/canic-core/src/domain/http.rs`
Intended location: `crates/canic-core/src/domain/http.rs`

Evidence:
`HttpMethod` used explicit lowercase `serde(rename = "...")` labels but also
accepted uppercase `serde(alias = "...")` labels.

Risk:
The alias path kept a second accepted label surface after the release line
started hard-cutting unsupported or unnecessary Candid/Serde compatibility
labels.

Decision:
Remove the uppercase aliases and keep only explicit canonical lowercase labels.
Tighten the Candid serde boundary guard so combined attributes such as
`#[serde(rename = "...", alias = "...")]` are caught.

Regression test:
`dto::http::tests::http_method_deserializes_canonical_lowercase_labels` pins
the canonical lowercase labels, while
`candid_serde_boundary_guard::candid_types_use_supported_serde_rename_attributes`
prevents reintroducing Candid-bearing aliases.

## Fixes

- Removed uppercase `GET`, `HEAD`, and `POST` aliases from `HttpMethod`.
- Replaced the old legacy compatibility round-trip test with positive
  canonical-label coverage.
- Tightened the Candid serde boundary guard to catch alias attributes combined
  with explicit rename attributes.

## Deferred

- No old-label compatibility tests were added; this is an intentional hard cut,
  not a compatibility surface.
