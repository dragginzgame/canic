## Slice Contract

Boundary:
`WasmStoreBootstrapApi` owns root bootstrap helpers used by lifecycle seeding
and endpoint macros. Unused root-local normalization helpers and direct publish
helpers should not remain as public Rust API surfaces.

Current owner:
`WasmStoreBootstrapApi` exposed extra public helpers for root-specific
manifest normalization, root-specific chunk preparation/publishing, and direct
staged-release publication. Active lifecycle code and endpoint macros did not
call those helpers.

Intended owner:
Lifecycle remains on the two embedded release-set helpers. Endpoint macros use
`debug_bootstrap`, `stage_manifest`, `prepare_chunk_set`, and `publish_chunk`.
Root bootstrap publication remains owned by the workflow layer.

Files/modules in scope:

- `crates/canic-control-plane/src/api/template/mod.rs`
- `crates/canic-control-plane/src/support.rs`

Files/modules explicitly out of scope:

## Findings

### CANIC-082-WASM-STORE-BOOTSTRAP-001

Severity: P3
Category: legacy_surface
Owner: `canic-control-plane::api::template::WasmStoreBootstrapApi`
Current location: `crates/canic-control-plane/src/api/template/mod.rs`
Intended location: endpoint/lifecycle-used bootstrap helpers only

Evidence:
Repo-wide search showed the root-specific staging helpers and
`WasmStoreBootstrapApi::publish_staged_release_set_to_current_store` were only
defined, not called by lifecycle code, endpoint macros, canisters, or tests.

Risk:
The extra helper names made it look like there were supported root-local
bootstrap staging paths separate from the endpoint/lifecycle paths that are
actually maintained.

Decision:
Removed the unused public Rust helpers. Kept lifecycle-used embedded
release-set helpers and endpoint-used bootstrap helpers.

Regression test:
Control-plane and `canic` facade tests compile the remaining bootstrap API and
endpoint macro paths.

### CANIC-082-WASM-STORE-BOOTSTRAP-002

Severity: P3
Category: visibility
Owner: `canic-control-plane::support`
Current location: `crates/canic-control-plane/src/support.rs`
Intended location: workflow layer

Evidence:
The support wrapper for direct staged-release publication was only used by the
removed public API helper. The active root bootstrap workflow calls
`WasmStorePublicationWorkflow::publish_staged_release_set_to_current_store`
directly.

Risk:
The wrapper duplicated a workflow operation as a support-layer helper without a
remaining caller.

Decision:
Removed the unused support wrapper and left the workflow owner unchanged.

Regression test:
Control-plane tests compile the bootstrap workflow and publication workflow
after support-wrapper removal.

## Fixes

- Removed unused root-specific bootstrap staging helper methods.
- Removed unused root-specific manifest normalization helper code and binding
  constant.
- Removed the unused bootstrap API staged-release publication helper.
- Removed the now-unused support wrapper for direct staged-release publication.
