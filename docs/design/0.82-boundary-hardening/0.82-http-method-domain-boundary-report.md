## Findings

### CANIC-082-HTTP-DTO-001

Severity: P2
Category: dto_boundary
Owner: `canic-core::dto::http`, `canic-core::ops::ic::http`, and
`canic-core::ops::runtime::metrics::http`
Current location:
`crates/canic-core/src/dto/http.rs`,
`crates/canic-core/src/ops/ic/http.rs`, and
`crates/canic-core/src/ops/runtime/metrics/http.rs`
Intended location: `crates/canic-core/src/domain/http.rs`

Evidence:
DTO, IC HTTP ops, and runtime HTTP metrics each defined an HTTP method enum
with the same semantic variants, while `ops::ic::http` converted between the
duplicate values for request execution and metric recording.

Risk:
The same HTTP method semantics were duplicated across DTO, ops, and metrics
boundaries, creating drift risk between public requests, approved IC outcalls,
and runtime metric projection.

Decision:
Fixed with a no-behavior-change ownership move. `HttpMethod` now lives in
`domain::http`, while DTO, ops, and metrics modules publicly re-export it so
existing API paths and serialized shapes remain stable.

Regression test:
`dto::http::tests::http_method_roundtrips_candid_through_dto_path` proves the
DTO re-export path remains compatible with the previous lowercase Candid method
labels.

## Fixes

- Added `domain::http` as the owner for `HttpMethod`.
- Re-exported the same value from existing DTO, IC HTTP ops, and runtime HTTP
  metrics paths.
- Removed duplicate DTO-to-ops and ops-to-metrics method conversions.

## Deferred

- Raw IC management HTTP infra types remain separate.
- HTTP request/response DTO structs and outcall behavior remain unchanged.
