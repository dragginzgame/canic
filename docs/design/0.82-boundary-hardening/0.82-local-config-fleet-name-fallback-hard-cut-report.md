## Findings

### CANIC-082-LOCAL-CONFIG-FLEET-NAME-FALLBACK-001

Severity: P2
Category: deployment_truth
Owner: `canic-host::deployment_truth::observe::config`
Current location: `crates/canic-host/src/deployment_truth/observe/config/mod.rs`
Intended location: explicit unknown local config identity

Evidence:
`observe_local_config_facts` accepted `fallback_fleet_name` and the local
inventory collector passed `request.deployment_name`. When config parsing
failed, root observations could report the deployment target as the fleet
template.

Risk:
Deployment truth could conflate two different identities in a degraded local
observation path: the deployment target and the fleet template name.

Decision:
Fixed with a hard cut. Missing local config fleet identity now reports the
existing observation gap and uses `unknown` for the required root observation
fleet-template field.

Regression test:
The local inventory test now verifies that missing config plus installed root
state does not copy deployment name into `observed_root.fleet_template`.

## Fixes

- Removed the caller-provided fleet-name fallback from local config
  observation.
- Added an explicit unknown fleet-template marker for missing config identity.
- Updated local inventory coverage for missing config plus root evidence.

## Deferred

- The root observation schema still requires a string fleet-template field.
  Making that field optional should be a separate schema-aware design.
