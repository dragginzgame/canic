`ops::replay::receipt` and `ops::storage::replay` directly for the same
mechanics.

- `crates/canic-core/src/ops/replay/slot.rs`

## Findings

### CANIC-082-ROOT-REPLAY-SLOT-001

Severity: P2
Category: ops_ownership
Owner: `canic-core::ops::replay::slot`
Current location: `crates/canic-core/src/ops/replay/slot.rs`
Intended location: `crates/canic-core/src/ops/replay/mod.rs` and
`crates/canic-core/src/ops/replay::guard`

Evidence:
`ops::replay::slot` only adapted root replay calls onto the shared replay
receipt helpers after the replay receipt consolidation. Its call sites were
limited to `ops::replay` and `ops::replay::guard`.

Risk:
The adapter preserved a legacy root-slot concept in active replay ops even
though shared replay receipts are the authoritative replay storage mechanism.

Decision:
Fixed with a no-behavior-change hard cut. Root replay now calls shared replay
receipt helpers and replay receipt storage directly, and the legacy slot adapter
module was removed.

Regression test:
Focused replay tests continue to cover root replay reservation quotas,
duplicate classification, expiry purge, compact root cycle replay encoding,
terminal failure bounding, and replay-protected workflow behavior.

## Fixes

- Removed the `ops::replay::slot` adapter module.
- Inlined root replay quota checks through `ReplayReceiptOps`.
- Inlined root replay reservation and commit through shared replay receipt
  helpers.
- Changed replay guard expiry purge to call `ReplayReceiptOps::purge_expired`
  directly.

## Deferred

- Replay receipt storage remains unchanged.
- Replay protected workflow behavior remains unchanged.
