## Findings

### CANIC-082-MEMORY-DTO-001

Severity: P2
Category: dto_boundary
Owner: `canic-core::dto::memory`
Current location: `crates/canic-core/src/dto/memory.rs`
Intended location: `crates/canic-core/src/domain/memory.rs`

Evidence:
`ops::runtime::memory` mapped `ic-memory` runtime diagnostics into
`dto::memory::MemoryCommitRecoveryErrorResponse`,
`dto::memory::MemoryRangeAuthorityMode`, and
`dto::memory::MemoryAllocationState` while producing memory ledger responses.

Risk:
The DTO module was acting as the semantic owner for memory diagnostic values
used by runtime ops before the query response boundary.

Decision:
Fixed with a no-behavior-change ownership move. The enums now live in
`domain::memory`, while `dto::memory` publicly re-exports them so existing API
paths and serialized shapes remain stable.

Regression test:
`dto::memory::tests::memory_enums_roundtrip_candid_with_existing_variant_labels`
proves the public DTO re-export path round-trips the moved memory diagnostic
enums through Candid.

`ops::runtime::memory::tests::memory_allocation_record_response_includes_live_backing_memory_size`
continues to prove the memory ops projection uses the same allocation-state
response shape.

## Fixes

- Added `domain::memory` as the owner for memory diagnostic value enums.
- Re-exported the same values from `dto::memory`.
- Updated runtime memory ops code to import the domain owners.

## Deferred

- Memory ledger response DTO structs remain unchanged.
- Stable-memory records, allocation runtime, and ledger format remain
  unchanged.
