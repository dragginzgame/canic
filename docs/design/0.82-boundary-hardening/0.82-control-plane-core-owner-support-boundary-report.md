## Findings

### CANIC-082-CONTROL-PLANE-CORE-OWNER-SUPPORT-001

Severity: P3
Category: support_boundary
Owner: canonical public core DTO/ids/replay-policy modules
Current location: `control_plane_support::{dto, ids, replay_policy}`
Intended location: direct canonical owner imports

Evidence:
`control_plane_support` re-exported `ValidationIssue`, `ValidationReport`,
`BuildNetwork`, `CanisterRole`, and `CostClass` through facade namespaces.
Their in-repo control-plane users can import the canonical public core owners
directly; the `ids` support namespace had no in-repo caller.

Risk:
Support namespaces that only mirror public canonical owners obscure ownership
and can grow into accidental compatibility facades.

Decision:
Fixed with a no-behavior-change support-boundary cleanup. Root bootstrap and
deployment workflow now import public validation DTO, ids, and replay-policy
values from their canonical core owners directly, and the hidden public-owner
support facades were removed.

Regression test:
Covered by control-plane tests and clippy. No behavior-specific regression is
required because the concrete types and behavior are unchanged.
