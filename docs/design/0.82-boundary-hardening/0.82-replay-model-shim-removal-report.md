## Findings

### CANIC-082-REPLAY-MODEL-SHIM-001

Severity: P2
Category: ops_ownership
Owner: `canic-core::ops::replay::model`
Current location: `crates/canic-core/src/ops/replay/model.rs`
Intended location: `crates/canic-core/src/model/replay/mod.rs`

Evidence:
`ops::replay::model` existed only as a compatibility shim while callers moved
to `model::replay`, but active internal replay ops and workflow modules still
imported through the shim.

Risk:
Replay model ownership stayed half-migrated, making `ops::replay` appear to own
stable replay model values that are actually canonical model-layer state.

Decision:
Fixed with a no-behavior-change hard cut. Internal callers now import replay
model values from `model::replay`, the compatibility shim was removed, and the
control-plane hidden support path now exposes `CommandKind` from a model
support namespace.

Regression test:
Focused replay tests cover reservation, duplicate handling, terminal-failure
bounding, recovery, auth prepare replay, ICP refill replay, pool replay, and
root capability replay behavior.

## Fixes

- Removed the `ops::replay::model` compatibility shim.
- Moved internal replay ops and workflow imports to `model::replay`.
- Moved hidden control-plane `CommandKind` support from an ops-shaped support
  namespace to a model-shaped support namespace.

## Deferred

- Public DTO/domain re-export compatibility paths remain intentionally
  preserved.
- Replay receipt stable-state schema and migration policy remain unchanged.
