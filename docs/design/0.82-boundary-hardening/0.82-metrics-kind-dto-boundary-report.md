## Findings

### CANIC-082-METRICS-DTO-001

Severity: P2
Category: dto_boundary
Owner: `canic-core::dto::metrics`
Current location: `crates/canic-core/src/dto/metrics.rs`
Intended location: `crates/canic-core/src/domain/metrics.rs`

Evidence:
`ops::runtime::metrics`, `workflow::metrics::query`, and lifecycle facade tests
used `dto::metrics::MetricsKind` to select metric families for runtime
projection and query dispatch.

Risk:
The DTO module was acting as the semantic owner for a metric-family selector
that participates in runtime projection dispatch, blurring DTO ownership with
internal domain meaning.

Decision:
Fixed with a no-behavior-change ownership move. The enum now lives in
`domain::metrics`, while `dto::metrics` publicly re-exports it so existing API
paths and serialized shapes remain stable.

Regression test:
`dto::metrics::tests::reexported_metrics_kind_roundtrips_through_candid` proves
the public DTO path still round-trips the re-owned value through Candid.

## Fixes

- Added `domain::metrics` as the owner for `MetricsKind`.
- Re-exported the same value from `dto::metrics`.
- Updated runtime metrics projection, metrics workflow query, and lifecycle
  facade tests to import the domain owner.
- Added a Candid roundtrip regression test through the DTO path.

## Deferred

- Metric row DTO structs remain DTO-owned.
- CLI-local metrics parser/transport models remain unchanged.
