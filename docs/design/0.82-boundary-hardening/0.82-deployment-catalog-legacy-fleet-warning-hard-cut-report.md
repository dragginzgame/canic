## Findings

### CANIC-082-DEPLOYMENT-CATALOG-LEGACY-FLEET-WARNING-001

Severity: P3
Category: legacy_surface
Owner: `canic-host::deployment_catalog`
Current location: `crates/canic-host/src/deployment_catalog/mod.rs`
Intended location: current deployment-target state catalog only

Evidence:
`build_deployment_catalog_report` called `append_legacy_state_warning`, which
probed `.canic/<network>/fleets` and emitted
`catalog.legacy_fleet_state_ignored` when the removed path existed.

Risk:
The catalog retained an active read of an obsolete install-state path even
though catalog entries already come only from current deployment-target state.
That kept legacy fleet-state terminology visible in an operator report after
the hard cut.

Decision:
Fixed with a hard cut. The catalog no longer probes the legacy path or emits
the legacy warning.

Regression test:
The legacy-warning test was removed with the production legacy probe; the
existing catalog tests still cover current deployment-target catalog behavior,
missing state, malformed entries, and text rendering.

## Fixes

- Removed deployment catalog legacy fleet-state warning constants.
- Removed the active legacy `.canic/<network>/fleets` directory probe.
- Removed the legacy warning test tied to the removed compatibility-era
  behavior.

## Deferred

- No cleanup command for obsolete local directories was added. That should be a
  separate explicit maintenance design if needed.
