## Slice Contract

Boundary:
Workflow modules should make their direct dependency on boundary value types,
role identifiers, and logging macros visible at the import site. The workflow
prelude remains a legacy convenience for untouched modules, not the owner of
IC/account/timer/logging values.

Current owner:
`canic-core::workflow::prelude` re-exported common Candid, principal, cycle,
account, role, and logging values. Pool and IC workflow modules imported those
values through wildcard prelude imports.

Intended owner:
The concrete owners remain `canic-core::cdk`, `canic-core::ids`, and
`canic-core::log`. Workflow modules import those owners directly when they use
their values.

Files/modules in scope:

- `crates/canic-core/src/workflow/mod.rs`
- `crates/canic-core/src/workflow/pool/*`
- `crates/canic-core/src/workflow/ic/*`
- `crates/canic-core/src/workflow/ic/provision/*`
- `crates/canic-core/src/workflow/ic/icp_refill/{cost_guard,replay}.rs`

Files/modules explicitly out of scope:

- Non-pool, non-IC workflow clusters still using the legacy workflow prelude.
- Ops, storage, DTO, endpoint, and lifecycle module prelude usage.
- Operator command surfaces.
- Candid, JSON, deployment truth, evidence/report, and stable-state schemas.

Public surfaces affected:
Rust namespace only: the stale `workflow::prelude::Account` re-export was
removed after the IC ledger workflow switched to importing `Account` from
`cdk::types` directly. Operator-facing behavior is unchanged.

Serialized surfaces affected:
None.

Commands/endpoints affected:
None.

Expected behavior change:
None.

Expected non-changes:

- Pool import, recycle, reset, scheduler, query, and admin behavior remain
  unchanged.
- IC call, ledger, management, ICP refill, and provisioning behavior remain
  unchanged.
- Pool and IC workflow tests continue to exercise the same behavior.
- Endpoint method names, CLI behavior, Candid, JSON, deployment truth,
  evidence/report schemas, and stable-state layout remain unchanged.

Validation:

- `cargo fmt --all` passed.
- `cargo test --locked -p canic-core workflow::pool --lib` passed.
- `cargo test --locked -p canic-core workflow::ic --lib` passed.
- `cargo clippy --locked -p canic-core --tests -- -D warnings` passed.
- `rg -n "workflow::prelude::\\*|prelude::\\*" crates/canic-core/src/workflow/pool crates/canic-core/src/workflow/ic -S` returned no matches.

Rollback plan:
Restore the wildcard prelude imports in the touched workflow modules and
restore the `Account` prelude re-export if downstream Rust callers still depend
on that internal convenience path.

Follow-ups:

- Continue replacing `workflow::prelude::*` in the remaining workflow clusters
  one bounded slice at a time.
- Keep production-only imports separate from test-only helpers when replacing
  wildcard imports.
- Avoid expanding the workflow prelude with new passive value exports.

## Findings

### CANIC-082-WORKFLOW-PRELUDE-001

Severity: P3
Category: workflow_ownership
Owner: `canic-core::workflow`
Current location: `crates/canic-core/src/workflow/pool/*`
Intended location: concrete `cdk`, `ids`, and `log` owners

Evidence:
Pool workflow modules imported `workflow::prelude::*` to obtain principals,
cycles, role/logging helpers, and test-only values.

Risk:
Wildcard prelude imports hid which workflow modules depended on boundary value
types versus workflow orchestration helpers, making future ownership audits
noisier.

Decision:
Fixed with a no-behavior-change import cleanup. Pool workflow modules now
import concrete values from their owning modules. Test-only `Cycles` imports
live inside local test modules where appropriate.

Regression test:
`cargo test --locked -p canic-core workflow::pool --lib`

### CANIC-082-WORKFLOW-PRELUDE-002

Severity: P3
Category: workflow_ownership
Owner: `canic-core::workflow`
Current location: `crates/canic-core/src/workflow/ic/*`
Intended location: concrete `cdk`, `ids`, and `log` owners

Evidence:
IC call, ledger, management, provisioning, and ICP refill workflow modules
imported `workflow::prelude::*` for Candid, principal/account, role, cycle, and
logging values.

Risk:
The IC workflow cluster appeared to depend on the workflow namespace for
passive value ownership, blurring the boundary between orchestration and
boundary data values.

Decision:
Fixed with a no-behavior-change import cleanup. IC workflow modules now import
`CandidType`, `Principal`, `Account`, `Cycles`, `CanisterRole`, `log`, and
`Topic` from their direct owners where needed.

Regression test:
`cargo test --locked -p canic-core workflow::ic --lib`

### CANIC-082-WORKFLOW-PRELUDE-003

Severity: P3
Category: visibility
Owner: `canic-core::workflow::prelude`
Current location: `crates/canic-core/src/workflow/mod.rs`
Intended location: `canic-core::cdk::types::Account`

Evidence:
After ledger workflow imported `Account` directly, `workflow::prelude` still
re-exported `Account` but no touched workflow module used that re-export.

Risk:
The stale export preserved an unnecessary Rust namespace shortcut and caused an
unused import warning under focused `canic-core` tests.

Decision:
Hard-cut the stale internal prelude re-export. `Account` remains available from
its canonical owner, `cdk::types`.

Regression test:
`cargo clippy --locked -p canic-core --tests -- -D warnings`

## Fixes

- Replaced wildcard workflow prelude imports across pool workflow modules with
  concrete imports.
- Replaced wildcard workflow prelude imports across IC call, ledger,
  management, provisioning, and ICP refill workflow modules with concrete
  imports.
- Moved test-only `Cycles` imports into local pool test modules.
- Removed the stale `workflow::prelude::Account` re-export.

## Deferred
