## Findings

### CANIC-082-RUNTIME-STATUS-DTO-001

Severity: P2
Category: dto_boundary
Owner: `canic-core::dto::runtime`
Current location: `crates/canic-core/src/dto/runtime.rs`
Intended location: `crates/canic-core/src/domain/runtime.rs`

Evidence:
`HealthStatus`, `ReadinessStatus`, `RuntimeStatus`, and `TimerStatus` were
defined in the runtime DTO module while neighboring runtime semantic values
already lived in `domain::runtime`.

Risk:
Runtime status semantics could drift between endpoint DTO ownership and the
runtime domain owner, especially as status builders and inspect renderers grow.

Decision:
Fixed with a no-behavior-change ownership move. The status enums now live in
`domain::runtime`, while `dto::runtime` publicly re-exports them so existing API
paths and serialized shapes remain stable.

Regression test:
`dto::runtime::tests::runtime_enums_roundtrip_candid_with_runtime_variant_labels`
continues to prove the DTO re-export path preserves the runtime enum Candid
contract.

## Fixes

- Added health, readiness, runtime, and timer status enums to
  `domain::runtime`.
- Re-exported the same values from the existing DTO runtime path.
- Updated the runtime status builder to import the domain owner directly.

## Deferred

- Runtime endpoint DTO structs remain in `dto::runtime`.
- Inspect CLI report/status semantics remain unchanged.
