## Slice Contract

Boundary:
Host and CLI parser tests should cover maintained structured JSON and typed
response-byte paths. Retired `response_candid` fallback fixtures should not
remain as anti-resurrection tests. Local readiness JSON parsing should only
treat boolean JSON and explicit `Ok` result shapes as readiness success.

Current owner:
`canic-host` and `canic-cli` tests still carried old `response_candid` fixtures
after the parser hard cuts. `replica_query::parse_ready_json_value` also
accepted the string `"(true)"` as ready and recursively accepted any truthy
object field, which preserved a reachable Candid-text fallback path and
over-broad structured object parsing for readiness JSON.

Intended owner:
Structured JSON owns command-output parsing. Typed response bytes own local
replica query decoding. Readiness JSON accepts booleans through bare values,
explicit `Ok` result objects, and arrays of those maintained result shapes only.

## Findings

### CANIC-082-RESPONSE-CANDID-FIXTURE-HARDCUT-001

Severity: P2
Category: legacy_surface
Owner: `canic-host::replica_query`
Current location: `crates/canic-host/src/replica_query/mod.rs`
Intended location: structured JSON boolean readiness parsing

Evidence:
`parse_ready_json_value` accepted `serde_json::Value::String("(true)")` as
ready and accepted any object with any nested `true` value, which allowed an
old Candid-text readiness shape or malformed object such as `{ "Err": true }`
to satisfy readiness parsing.

Risk:
The string fallback kept a reachable parser compatibility path alive after
structured JSON and typed-byte query paths became the maintained surfaces. The
recursive object fallback also made current structured parsing too broad.

Decision:
Hard-cut readiness JSON parsing to booleans inside maintained JSON result
shapes. Object parsing now only follows the `Ok` field; strings, numbers, null
values, and unrelated object fields now evaluate to not ready.

Regression test:
`cargo test --locked -p canic-host replica_query --lib`

### CANIC-082-RESPONSE-CANDID-FIXTURE-HARDCUT-002

Severity: P3
Category: test_ownership
Owner: `canic-host` and `canic-cli` parser tests
Current location: host and CLI parser test modules
Intended location: maintained structured JSON and typed-byte parser coverage

Evidence:
Metrics, cycles, metadata, cycle-balance, bootstrap-readiness, and readiness
tests still contained `response_candid` or raw Candid fixtures whose only
purpose was to show retired fallback forms were rejected.

Risk:
The stale fixtures kept legacy command-output vocabulary in active tests and
made the parser boundary look like it still owned old Candid fallback behavior.

Decision:
Removed the old fixtures and kept tests for maintained structured JSON success,
malformed structured JSON rejection, and typed response-byte decoding.

Regression test:
Focused `canic-host` and `canic-cli` parser test filters listed above.
