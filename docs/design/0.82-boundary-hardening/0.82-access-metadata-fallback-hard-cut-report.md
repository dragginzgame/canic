# Canic 0.82 Access And Metadata Fallback Hard Cut Report

Date: 2026-07-07

## BoundaryHardeningReport

```text
schema_version: 1
slice: 0.82 Access And Metadata Fallback Hard Cut
boundary: access-expression and metadata Rust APIs expose canonical surfaces only
status: pass
behavior_impact: no_behavior_change
```

## Slice Contract

Boundary:
Access-expression helpers should expose the maintained expression builders
directly. Endpoint macro `requires(...)` grammar remains macro syntax, not a
second Rust helper alias for `all(...)`. Canic metadata endpoints should use the
exporting canister package metadata path.

Current owner:
`access::expr::requires` was an unused Rust alias for `access::expr::all`.
`CanicMetadataApi::metadata` was an unused core-crate fallback helper that
produced metadata from the `canic-core` package instead of the canister crate
exporting the endpoint.

Intended owner:
`access::expr::all` owns all-expression construction. Endpoint macro
`requires(...)` remains the maintained endpoint-declaration grammar.
`CanicMetadataApi::metadata_for` owns endpoint metadata construction with the
exporting crate's package name, version, description, Canic framework version,
and canister version.

Files/modules in scope:

- `crates/canic-core/src/access/expr/mod.rs`
- `crates/canic-core/src/api/metadata.rs`

Files/modules explicitly out of scope:

- Endpoint macro grammar and parsing.
- Endpoint method names.
- Metadata response DTO fields.
- Candid, JSON, deployment truth, evidence/report, and stable-state schemas.
- CLI command surfaces.

Public surfaces affected:
Rust namespace only: `canic_core::access::expr::requires` and
`CanicMetadataApi::metadata` are removed.

Serialized surfaces affected:
None.

Commands/endpoints affected:
None.

Expected behavior change:
None for endpoint macro expansion, endpoint access checks, metadata endpoint
responses, operator commands, Candid, JSON, stable state, deployment truth, or
evidence/report schemas.

Expected non-changes:

- Endpoint attributes continue to use macro `requires(...)` grammar.
- Access expressions still compose through `all(...)`, `any(...)`, `not(...)`,
  and typed built-in predicates.
- `canic_emit_canic_metadata_endpoints!` continues to call
  `CanicMetadataApi::metadata_for(...)` with the exporting canister crate
  metadata.
- The `CanicMetadataResponse` shape is unchanged.

Validation:

- `cargo fmt --all` passed.
- `cargo test --locked -p canic-core access --lib` passed.
- `cargo test --locked -p canic canic_metadata --test canic_metadata` passed.
- `cargo test --locked -p canic-macros endpoint --lib` passed.
- `rg -n "CanicMetadataApi::metadata\\(|pub fn metadata\\(|core crate fallback path|access::expr::requires|expr::requires|pub fn requires" crates canisters -S` returned no active-code matches.
- `cargo clippy --locked -p canic-core --tests -- -D warnings` passed.
- `cargo fmt --all -- --check` passed.
- `cargo test --locked -p canic --test changelog_governance` passed.
- `git diff --check` passed.

Rollback plan:
Restore the removed Rust helper names only if the maintainer intentionally
chooses to support those pre-1.0 convenience surfaces. Do not add compatibility
aliases or anti-resurrection tests by default.

Follow-ups:

- Keep endpoint macro grammar and Rust helper APIs clearly separated.
- Continue removing unused Rust fallback helpers when the canonical owner is
  already explicit and tested.

## Findings

### CANIC-082-ACCESS-METADATA-HARDCUT-001

Severity: P3
Category: legacy_surface
Owner: `canic-core::access::expr`
Current location: `crates/canic-core/src/access/expr/mod.rs`
Intended location: `access::expr::all`

Evidence:
`access::expr::requires` was documented as an alias for `all` and had no
in-repo active callers. Endpoint macro `requires(...)` grammar is parsed and
expanded separately by `canic-macros`.

Risk:
The alias made Rust access-expression composition look like it had two
maintained names for the same operation and blurred the line between macro
declaration syntax and runtime helper APIs.

Decision:
Removed the unused alias. Callers should use `access::expr::all(...)` in Rust
and keep using macro `requires(...)` in endpoint attributes.

Regression test:
`cargo test --locked -p canic-core access --lib` and
`cargo test --locked -p canic-macros endpoint --lib`.

### CANIC-082-ACCESS-METADATA-HARDCUT-002

Severity: P3
Category: legacy_surface
Owner: `canic-core::api::metadata`
Current location: `crates/canic-core/src/api/metadata.rs`
Intended location: `CanicMetadataApi::metadata_for`

Evidence:
`CanicMetadataApi::metadata` was documented as the core-crate fallback path and
had no in-repo active callers. The endpoint macro already calls
`CanicMetadataApi::metadata_for(...)` with the exporting canister crate package
metadata.

Risk:
Keeping the fallback helper preserved an unused metadata construction path that
could emit `canic-core` package metadata instead of the canister crate metadata
expected from the maintained endpoint macro.

Decision:
Removed the fallback helper and its now-unused core package constants.
`metadata_for(...)` remains the canonical metadata construction API.

Regression test:
`cargo test --locked -p canic canic_metadata --test canic_metadata`.

## Fixes

- Removed `access::expr::requires`.
- Removed `CanicMetadataApi::metadata`.
- Removed now-unused core package constants from `api::metadata`.

## Deferred

- `CanisterChildrenApi` remains in place because it is the endpoint-facing
  public facade used by the topology endpoint macro, not a stale compatibility
  alias.
- Standard `help`, `--help`, `-h`, `version`, `--version`, and `-V` CLI forms
  remain ordinary CLI UX, not legacy command compatibility.
