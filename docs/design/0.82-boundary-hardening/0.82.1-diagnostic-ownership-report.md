## Slice Contract

Boundary:
Report producers own diagnostic codes, categories, status derivation,
next-action semantics, safety language, source attribution, and JSON schema
meanings. CLI renderers own formatting, serialization mechanics, terminal
ordering, and CLI-specific exit-code mapping.

Current owner:
Mixed but mostly correct. Medic and deploy-plan derive status in their report
builders, state-audit delegates report construction to `canic-host`, and
inspect derives CLI wrapper status from the decoded runtime endpoint DTO.

Intended owner:
Keep producer-owned report semantics in the report family that emits the
diagnostic. Keep top-level CLI error rendering as a delegate to command-family
errors.

Files/modules in scope:

- `crates/canic-cli/src/medic/mod.rs`

## Findings

### CANIC-082-DIAG-001

Severity: P3
Category: test_ownership
Owner: top-level CLI dispatcher
Current location: `crates/canic-cli/src/tests.rs`
Intended location: top-level CLI dispatcher tests

Evidence:
The top-level CLI error renderer delegates report-status stderr suppression and
exit-code handling to command-family errors for deploy-plan, inspect, medic, and
state. The previous test covered inspect only.

Risk:
Future dispatcher cleanup could accidentally special-case one report family,
causing the top-level renderer to reinterpret producer-owned report status or
emit duplicate stderr after a report has already been printed.

Decision:
Fixed with a no-behavior-change regression test that covers all four current
operator report families with report-status errors.

Regression test:
`report_status_errors_delegate_suppression_and_exit_codes`

## Fixes
