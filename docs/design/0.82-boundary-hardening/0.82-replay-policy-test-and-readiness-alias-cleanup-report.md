## Findings

### CANIC-082-REPLAY-READINESS-CLEANUP-001

Severity: P3
Category: test_ownership
Owner: `canic-core::replay_policy::tests::endpoint`
Current location: `crates/canic-core/src/replay_policy/tests/endpoint.rs`
Intended location: active replay-policy tests assert maintained endpoint rows

Evidence:
One test asserted that four bridge-backed delegation-proof endpoint names were
absent from `ENDPOINT_REPLAY_POLICY_MANIFEST`.

Risk:
Removed-surface guard tests keep retired endpoint names present in active test
code and conflict with the pre-1.0 hard-cut policy against anti-resurrection
tests.

Decision:
Removed the absence test. Active replay-policy tests continue to cover current
endpoint rows and classifications.

Regression test:
`cargo test --locked -p canic-core replay_policy --lib`

### CANIC-082-REPLAY-READINESS-CLEANUP-002

Severity: P3
Category: test_ownership
Owner: `canic-tests::root_cases::auth_076`, `canic::tests::protocol_surface`
Current location: `crates/canic-tests/tests/root_cases/auth_076.rs`,
`crates/canic/tests/protocol_surface.rs`
Intended location: active auth/protocol tests assert maintained surfaces only

Evidence:
The 0.76 auth acceptance tests called retired bridge-backed root-proof method
names only to assert they were not callable. The protocol-surface tests scanned
the root endpoint source to assert the same retired method names and retired
provisioner guard string were absent.

Risk:
Retired method and guard names remained embedded in active tests after the
pre-1.0 hard-cut policy ruled out anti-resurrection coverage for removed
surfaces.

Decision:
Removed the test-only absent-method checks and the retired guard-name source
scan. Positive protocol-surface checks for current issuer policy, renewal,
status, and chain-key lazy-repair endpoints remain.

Regression test:
`cargo test --locked -p canic --test protocol_surface`
`cargo test --locked -p canic-tests --test root_suite --no-run`
`cargo clippy --locked -p canic-tests --test root_suite -- -D warnings`

### CANIC-082-REPLAY-READINESS-CLEANUP-003

Severity: P3
Category: dto_boundary
Owner: `canic-host::install_root::readiness`
Current location: `crates/canic-host/src/install_root/readiness/parsing/mod.rs`
Intended location: `canic_core::dto::state::BootstrapStatusResponse`

Evidence:
Install-root readiness parsing defined `BootstrapStatusSnapshot` as a local
alias to `BootstrapStatusResponse`, then used the alias through readiness and
diagnostics helpers.

Risk:
The alias made a simple DTO consumption path look like a distinct host snapshot
model, adding naming drift without changing behavior.

Decision:
Removed the alias and imported `BootstrapStatusResponse` directly where host
readiness helpers need it.

Regression test:
`cargo test --locked -p canic-host readiness_parse --lib`
