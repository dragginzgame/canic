## Slice Contract

Boundary:
Host-side parsing of `canic_cycle_balance` command output used by live list,
cycles reporting, and root cycle checks.

Current owner:
`canic-host::response_parse`, with fallback parsing from raw Candid text and
ICP `response_candid` wrapper text.

Intended owner:
Local replica calls decode typed Candid bytes through the replica-query path.
ICP CLI fallback output is parsed only from structured JSON values.

Files/modules in scope:

- `crates/canic-host/src/response_parse/mod.rs`
- `crates/canic-host/src/response_parse/tests.rs`

Files/modules explicitly out of scope:

- Local replica typed-byte cycle-balance decoding.
- The `canic_cycle_balance` endpoint method name and Candid signature.
- CLI list/cycles command names, options, and rendering.
- Bootstrap readiness parsing.
- Metrics/cycles report parsing and `canic_metadata` parsing, which are covered
  by separate hard-cut reports.
- Deployment truth, evidence, stable-state layout, or generated artifacts.

Public surfaces affected:
Host cycle-balance parsing no longer accepts raw Candid text or
`response_candid` text fallback output when extracting the cycle balance from
ICP CLI output.

Serialized surfaces affected:
No endpoint, Candid, JSON, deployment-truth, evidence/report, or stable-state
schema changes. The host accepted input wrapper shape is intentionally narrowed
to structured JSON for this ICP CLI fallback response.

Commands/endpoints affected:
No command or endpoint name changes.

Expected behavior change:
ICP CLI output for `canic_cycle_balance` must include a structured JSON `Ok`
value. Raw Candid text and `response_candid` wrapper text are rejected. The
local replica fast path continues to decode typed Candid bytes directly.

Expected non-changes:

- The `canic_cycle_balance` endpoint Candid signature is unchanged.
- Direct local replica cycle-balance query decoding is unchanged.
- Successful live list/cycles/root-cycle behavior is unchanged when structured
  JSON or local typed bytes are available.
- Bootstrap-readiness fallback parsing remains out of scope.
- No compatibility fallback, alias, or shim is retained for the ICP CLI
  cycle-balance parser.

## Findings

### CANIC-082-CYCLE-BALANCE-RESPONSE-CANDID-001

Severity: P2
Category: host_surface
Owner: `canic-host::response_parse`
Current location: `crates/canic-host/src/response_parse/mod.rs`
Intended location: structured JSON cycle-balance parsing for ICP CLI output

Evidence:
`parse_cycle_balance_response` extracted cycle balances from structured JSON,
raw Candid text, or an ICP `response_candid` wrapper. That kept a rendered
Candid fallback accepted after adjacent typed report hard cuts.

Risk:
Cycle-balance discovery had multiple accepted response shapes for the same
numeric fact, leaving a legacy ICP output path alive before 1.0.

Decision:
Fixed with a hard cut. ICP CLI cycle-balance parsing now reads only structured
JSON `Ok` values, while the separate local replica path continues to decode
typed Candid bytes directly.

Regression test:
Host response parser tests now assert that raw Candid text and text-only
`response_candid` wrapper output are rejected for cycle-balance discovery.

## Fixes

- Removed raw Candid text and `response_candid` fallback parsing from
  `parse_cycle_balance_response`.
- Removed the private cycle-balance Candid parser and now-unused leading-number
  helper.
- Updated host response parser tests to pin rejection of old fallback response
  shapes.

## Deferred

- Bootstrap readiness fallback parsing remains untouched and should be handled
  as a separate explicit hard-cut slice if selected.
- No old-fallback compatibility tests were added; this is an intentional hard
  cut, not a compatibility surface.
