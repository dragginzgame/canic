## Findings

### CANIC-082-APP-STATE-001

Severity: P2
Category: dto_boundary
Owner: `canic-core::storage::stable::state::app` and
`canic-core::dto::state`
Current location:
`crates/canic-core/src/storage/stable/state/app.rs` and
`crates/canic-core/src/dto/state.rs`
Intended location: `crates/canic-core/src/domain/state.rs`

Evidence:
Storage and DTO each defined an `AppMode` enum with identical variants, while
`ops::storage::state::mapper` converted between the duplicate values for
app-state snapshots.

Risk:
The same app-mode semantics were duplicated across storage and DTO boundaries,
creating a drift risk between guard behavior, stable records, and boundary
snapshots.

Decision:
Fixed with a no-behavior-change ownership move. `AppMode` now lives in
`domain::state`, while storage and DTO modules publicly re-export it so
existing API paths and serialized shapes remain stable.

Regression tests:
`dto::state::tests::app_mode_roundtrips_candid_through_dto_path` proves the
DTO re-export path round-trips the shared enum through Candid.

`storage::stable::state::app::tests::app_state_record_storable_roundtrips_shared_app_mode`
proves the stable record path round-trips the shared enum through the existing
storable serializer.

## Fixes

- Added `domain::state` as the owner for `AppMode`.
- Re-exported `AppMode` from the existing storage and DTO modules.
- Removed duplicate storage/DTO mode conversion in app-state mapping.

## Deferred

- App command/status DTO values remain DTO-owned.
- App-state record fields and stable-memory layout remain unchanged.
