## Slice Contract

Boundary:
Workflow orchestration modules should import passive boundary values from their
concrete owners. The workflow namespace owns orchestration, not a wildcard
prelude for Candid, principal, cycle, role, or logging values.

Current owner:
The previous slice removed workflow prelude usage from pool and IC workflow
clusters. Remaining env, runtime, auth, cascade, lifecycle, RPC, bootstrap,
topology-index, placement-scaling, and cycle-tracking workflow modules still
used `workflow::prelude::*`.

Intended owner:
`cdk`, `ids`, and `log` remain the owners of those passive values. Workflow
modules import the values they use directly.

Files/modules in scope:

- `crates/canic-core/src/workflow/mod.rs`
- `crates/canic-core/src/workflow/env/mod.rs`
- `crates/canic-core/src/workflow/runtime/*`
- `crates/canic-core/src/workflow/runtime/auth/*`
- `crates/canic-core/src/workflow/cascade/*`
- `crates/canic-core/src/workflow/canister_lifecycle/*`

- `crates/canic-core/src/workflow/placement/scaling/mod.rs`

Files/modules explicitly out of scope:

- RPC request workflow, bootstrap, scaling, and topology query behavior remain

Rollback plan:
Restore `workflow::prelude` and move the touched modules back to wildcard
imports. This is not recommended unless downstream Rust callers still depend on
the removed internal convenience namespace.

Follow-ups:

- Keep future workflow modules importing passive values from concrete owners.
- Audit non-workflow preludes separately; this slice intentionally did not
  remove ops, storage, DTO, or support preludes.

## Findings

### CANIC-082-WORKFLOW-PRELUDE-HARDCUT-001

Severity: P3
Category: workflow_ownership
Owner: `canic-core::workflow`
Current location: remaining `crates/canic-core/src/workflow/**` modules
Intended location: concrete `cdk`, `ids`, and `log` owners

Evidence:
Remaining workflow modules imported `workflow::prelude::*` for `CandidType`,
`Principal`, `Cycles`, `CanisterRole`, `log`, and `Topic`.

Risk:
The wildcard prelude obscured whether a workflow module depended on passive
boundary values, logging, or actual workflow orchestration helpers.

Decision:
Fixed with a no-behavior-change import cleanup. Remaining workflow modules now
import concrete owners directly.

Regression test:
`cargo test --locked -p canic-core workflow --lib`

### CANIC-082-WORKFLOW-PRELUDE-HARDCUT-002

Severity: P3
Category: visibility
Owner: `canic-core::workflow::prelude`
Current location: `crates/canic-core/src/workflow/mod.rs`
Intended location: no workflow prelude

Evidence:
After all workflow modules imported concrete owners directly, no code under
`crates/canic-core/src/workflow` referenced `workflow::prelude`.

Risk:
Keeping an unused public prelude would preserve a stale internal shortcut and
invite future wildcard imports.

Decision:
Hard-cut `workflow::prelude` from the workflow module.

Regression test:
`rg -n "workflow::prelude|prelude::\\*|pub mod prelude" crates/canic-core/src/workflow -S`

## Fixes

- Removed remaining `workflow::prelude::*` imports from runtime, auth, cascade,
  lifecycle, RPC, bootstrap, topology-index, placement-scaling, cycle-tracking,
  and env workflow modules.
- Imported `CandidType`, `Principal`, `Cycles`, `CanisterRole`, `log`, and
  `Topic` from their concrete owners where needed.
- Removed `workflow::prelude` from `workflow/mod.rs`.
- Added a maintained workflow prelude boundary guard over
  `crates/canic-core/src/workflow`.

## Deferred

- Non-workflow preludes remain out of scope for this slice.
- No broader module-tree or ownership rewrite was attempted.
