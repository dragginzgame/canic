## Slice Contract

Boundary:
Control-plane access to private core formatting helpers.

Current owner:
Core formatting helpers live in `canic_core::format` and are mediated to
external sibling crates through hidden support modules. `control_plane_support`
exported `byte_size`, `cycles_tc`, and `truncate`, while control-plane code only
uses `byte_size`.

Intended owner:
`control_plane_support::format` exposes only `byte_size` for the
control-plane crate. Other formatting helpers remain available through their
actual support owners where they are used, without widening the control-plane
support boundary.

Files/modules in scope:

- `crates/canic-core/src/control_plane_support.rs`

Files/modules explicitly out of scope:

- Core formatting helper implementations.
- Host-side formatting helpers.
- Operator text rendered by existing control-plane code.
- CLI, endpoint, JSON, deployment truth, evidence, and stable-state schemas.

Public surfaces affected:
None for operator/runtime behavior. The removed hidden support exports were not
used by `canic-control-plane`.

Serialized surfaces affected:
None.

Commands/endpoints affected:
None.

Expected behavior change:
None.

Expected non-changes:

- Control-plane byte-size labels remain formatted by the same helper.
- Host-side cycle and truncation formatting remain unchanged.
- No compatibility shim or replacement support alias is added.

Validation:

- `cargo fmt --all` passed.
- `cargo test --locked -p canic-control-plane --lib` passed.
- `cargo clippy --locked -p canic-core --tests -- -D warnings` passed.
- `cargo clippy --locked -p canic-control-plane --tests -- -D warnings` passed.
- `cargo fmt --all -- --check` passed.
- `rg -n "control_plane_support::format::(cycles_tc|truncate)|pub use crate::shared_support::format::\{byte_size, cycles_tc, truncate\}" crates/canic-core/src/control_plane_support.rs crates/canic-control-plane/src -S` returned no widened format support exports or call sites.
- `git diff --check` passed.

Rollback plan:
Restore the broader `control_plane_support::format` re-export if
`canic-control-plane` intentionally starts using additional private core
formatting helpers. Prefer adding only the specific helper needed.

Follow-ups:

- Continue keeping hidden support namespaces narrow and purpose-built.

## Findings

### CANIC-082-CONTROL-PLANE-FORMAT-SUPPORT-001

Severity: P3
Category: support_boundary
Owner: `canic_core::format`
Current location: `control_plane_support::format`
Intended location: narrow hidden support export for control-plane needs

Evidence:
`control_plane_support::format` exported `byte_size`, `cycles_tc`, and
`truncate`, but control-plane code only imported and called `byte_size`.

Risk:
Broader hidden support exports make it easier for unrelated sibling crates to
grow accidental dependencies on formatting helpers they do not currently need.

Decision:
Fixed with a no-behavior-change support-boundary cleanup. The control-plane
format support namespace now exports only `byte_size`.

Regression test:
Covered by control-plane tests and clippy. No behavior-specific regression is
required because existing control-plane call sites still use the same byte-size
helper.
