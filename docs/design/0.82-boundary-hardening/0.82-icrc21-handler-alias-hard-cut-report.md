## Slice Contract

Boundary:
The ICRC-21 dispatcher should expose named protocol DTOs and store its runtime
handler shape directly. A public alias for the local handler closure shape does
not own a stable protocol concept before 1.0.

Current owner:
`canic-core::dispatch::icrc21` defined `ConsentHandlerFn` as a public alias for
the registry handler closure type and used it only inside the defining module.
`Icrc21Dispatcher::get_handler` exposed the same local registry implementation
detail as a public Rust helper, but it was only called by
`Icrc21Dispatcher::consent_message`.

Intended owner:
`Icrc21Dispatcher` owns public registration and dispatch. The local registry
stores private `RegisteredConsentHandler` values that wrap the concrete
`Arc<dyn Fn(ConsentMessageRequest) -> ConsentMessageResponse + 'static>`
handler shape.

Files/modules in scope:

- `crates/canic-core/src/dispatch/icrc21.rs`
- `docs/status/current.md`

Files/modules explicitly out of scope:

- ICRC-21 Candid DTO shape.
- ICRC-21 consent response conversion.
- ICRC-21 endpoint or protocol method names.
- Consent handler runtime behavior.
- CLI command names, flags, or help output.
- JSON, deployment truth, evidence/report, and stable-state schemas.

Public surfaces affected:
Rust namespace only:

- `canic_core::dispatch::icrc21::ConsentHandlerFn` is removed.
- `canic_core::dispatch::icrc21::Icrc21Dispatcher::get_handler` is no longer
  public.

Serialized surfaces affected:
None.

Commands/endpoints affected:
None.

Expected behavior change:
None at runtime. External Rust code importing the removed alias must use the
concrete `Arc<dyn Fn(ConsentMessageRequest) -> ConsentMessageResponse +
'static>` handler shape directly. External Rust code calling `get_handler`
should call `consent_message` for dispatch or register its own handler through
`register`.

Expected non-changes:

- `Icrc21Dispatcher::register` keeps accepting handler closures with the same
  input and output types.
- `Icrc21Dispatcher::consent_message` still dispatches the same registered
  handler or returns the same unsupported-canister-call error.
- `ConsentMessageRequest`, `ConsentMessageResponse`, `ConsentInfo`, and
  `Icrc21Error` remain the named ICRC-21 boundary DTO/error types.
- No endpoint, CLI, Candid, JSON, deployment truth, evidence/report, or
  stable-state surface changed.

Validation:

- `cargo test --locked -p canic-core icrc21 --lib`
- `cargo test --locked -p canic --test protocol_surface`
- `cargo clippy --locked -p canic-core --tests -- -D warnings`

Rollback plan:
Restore the alias only if the maintainer intentionally chooses to keep that
Rust namespace before 1.0. Do not add compatibility aliases by default.

Follow-ups:

- Continue auditing public Rust type aliases and keep only aliases with a clear
  semantic owner or protocol value.
- Keep `ConsentMessageRequest` and `ConsentMessageResponse` as named DTOs.

## Findings

### CANIC-082-ICRC21-HANDLER-ALIAS-HARDCUT-001

Severity: P3
Category: dto_boundary
Owner: `canic-core::dispatch::icrc21`
Current location: `crates/canic-core/src/dispatch/icrc21.rs`
Intended location: dispatcher stores the concrete handler closure type directly

Evidence:
`ConsentHandlerFn` was a public alias for the registry handler closure type and
was only used in the local thread-local registry and `get_handler` return type.
`get_handler` itself was only called by `consent_message`.

Risk:
The alias added a public Rust namespace for an implementation-detail handler
shape without owning a distinct protocol concept.

Decision:
Removed `ConsentHandlerFn`, wrapped the concrete shared handler shape in a
private registry value, and made `get_handler` private.

Regression test:
`cargo test --locked -p canic-core icrc21 --lib`
