## Slice Contract

Boundary:
Canister pool status is a domain-owned value. The hidden control-plane support
facade may expose it to the control-plane crate, but it should not make the DTO
namespace look like the semantic owner.

Current owner:
`canic-core::control_plane_support::dto::pool` re-exported
`CanisterPoolStatus`.

Intended owner:
`canic-core::domain::pool` owns `CanisterPoolStatus`.
`canic-core::control_plane_support::domain::pool` exposes that owner to the
control-plane crate.

Files/modules in scope:

- `crates/canic-core/src/control_plane_support.rs`
- `crates/canic-control-plane/src/workflow/bootstrap/root.rs`

Files/modules explicitly out of scope:

- Public pool DTO import paths.
- Pool endpoint behavior.
- Pool storage mapping.
- Pool workflow decisions.
- Public CLI, JSON, Candid, deployment-truth, or evidence surfaces.

Public surfaces affected:
None intended. The changed path is the internal hidden support facade used by
the sibling control-plane crate.

Serialized surfaces affected:
None. Pool DTOs and Candid shapes remain unchanged.

Commands/endpoints affected:
None.

Expected behavior change:
None.

Expected non-changes:

- Pool bootstrap logging still classifies ready, pending-reset, and failed pool
  entries the same way.
- `dto::pool::CanisterPoolStatus` remains available as the public DTO
  re-export.
- Pool storage mapping and workflow decisions continue to use the domain owner.

Validation:

- `cargo fmt --all -- --check` passed.
- `cargo check --locked -p canic-core --lib` passed.
- `cargo check --locked -p canic-control-plane --all-targets` passed.

Rollback plan:
Restore `control_plane_support::dto::pool::CanisterPoolStatus` and move the
control-plane bootstrap import back through the DTO-shaped support namespace.

Follow-ups:

- Keep public DTO compatibility re-exports in `dto::pool`.
- Keep hidden control-plane support paths aligned with semantic owners where
  no public compatibility surface is involved.

## Findings

### CANIC-082-CONTROL-PLANE-POOL-STATUS-001

Severity: P3
Category: dto_boundary
Owner: `canic-core::control_plane_support::dto::pool`
Current location: `crates/canic-core/src/control_plane_support.rs`
Intended location: `crates/canic-core/src/control_plane_support::domain::pool`

Evidence:
The hidden support facade exposed `CanisterPoolStatus` under a DTO-shaped
namespace even though the value is owned by `domain::pool` and public DTO
compatibility is already handled by `dto::pool`.

Risk:
Sibling control-plane code could keep treating the DTO namespace as the owner
of pool state semantics, weakening the domain ownership cleanup from earlier
0.82 slices.

Decision:
Fixed with a no-behavior-change support-facade import cleanup. The hidden
support facade now exposes pool status through `domain::pool`, and root
bootstrap imports it from that owner-shaped path.

Regression test:
Control-plane compile coverage continues to verify the bootstrap import path
and pool status classification code.

## Fixes

- Removed the hidden `control_plane_support::dto::pool` status re-export.
- Added `control_plane_support::domain::pool::CanisterPoolStatus`.
- Updated root bootstrap to import pool status through the domain-shaped
  support namespace.

## Deferred

- Public `dto::pool::CanisterPoolStatus` compatibility remains unchanged.
- Pool behavior, storage, workflows, and endpoint surfaces remain unchanged.
