# Canic 0.13: Distributed Capability Invocation Framework

## 1. Overview

Canic 0.13 extends the 0.11 Root Capability model and 0.12 Role Attestation model
into a generalized, deny-by-default invocation framework for service-to-service
requests.

Core rule:

Any canister may send a request to any canister, but no request carries
authority by transport alone.

Authority is granted only through explicit capability policy plus valid proof.

0.13 standardizes:

- capability envelope shape
- proof separation from capability payload
- capability versioning
- payload-hash proof binding
- replay/idempotency semantics

---

## 2. Goals

### Primary Goals

- Preserve strict capability-based authorization for inter-canister requests.
- Separate transport, capability meaning, and authorization proof.
- Support multiple proof models (`Structural`, `RoleAttestation`, `DelegatedGrant`).
- Keep policy small and service-local through service-specific capability enums.
- Provide backward-compatible evolution via explicit capability/proof versions.
- Standardize replay handling across capability services.

### Non-Goals

- No generic "execute arbitrary operation" endpoint.
- No ambient authority derived from caller identity alone.
- No replacement of 0.11 root dispatcher in 0.13.
- No collapse of cryptographic domains (delegation vs attestation vs grants).

---

## 3. Relationship to 0.11 and 0.12

- 0.11 established single-dispatch capability gating for root-only privileged ops.
- 0.12 established portable caller-role evidence via root-signed attestations.
- 0.13 generalizes request authorization to service-local capabilities while
  preserving 0.11 and 0.12 invariants.

Mandatory continuity invariants:

- Enum-based exact dispatch only.
- Pure policy decision layer.
- Replay protection for mutating operations.
- Cryptographic domain separation.

---

## 4. Architectural Principles

- Ask is open, execute is gated.
- Transport is not authority.
- Capability semantics are service-owned.
- Proof is orthogonal to capability payload.
- Policy decides, workflow executes, ops applies.
- Deny by default on all verification/policy failures.

---

## 5. Core Data Model

### 5.1 Conceptual Envelope

```rust
struct CapabilityEnvelope<C> {
    service: ServiceId,
    capability_version: u16,
    capability: C,
    proof: CapabilityProof,
    metadata: RequestMetadata,
}

struct RequestMetadata {
    request_id: [u8; 16], // u128 logical request identity
    nonce: [u8; 16],      // extra entropy and collision hardening
    issued_at: u64,       // unix seconds
    ttl_seconds: u32,
}
```

```rust
enum ServiceId {
    ProjectRegistry,
    ProjectInstance,
    Cycles,
    CanisterLifecycle,
}
```

### 5.2 Proof Variants

```rust
enum CapabilityProof {
    Structural,
    RoleAttestation(RoleAttestationProof),
    DelegatedGrant(DelegatedGrantProof),
}
```

### 5.3 Wire Contract Rule

The generic representation above is conceptual.
Wire-level DTOs **MUST** use concrete service-specific structs/enums (Candid-stable, versionable).

---

## 6. Capability/Proof Separation (Normative)

- `capability` defines requested action and action parameters only.
- `proof` defines why caller may perform that capability.
- `metadata` defines replay/idempotency only.

`metadata` **MUST NOT** encode business capability fields.
`proof` **MUST NOT** redefine capability payload fields.

---

## 7. Versioning

Each request **MUST** include:

- `capability_version: u16`

Each non-structural proof **MUST** include:

- `proof_version: u16`

Rules:

- Services **MUST** dispatch by exact `capability_version`.
- Unsupported versions **MUST** fail closed with typed errors.
- Version evolution **MUST** not reuse old semantic meaning under same version.

---

## 8. Capability Hash Binding

To prevent proof reuse against modified payloads, proof verification **MUST** bind to:

```text
CapabilityHash = H(
  "CANIC_CAPABILITY_V1" ||
  canonical(target_canister, service, capability_version, capability)
)
```

In this document, `canonical(...)` **MUST** mean Candid canonical encoding of
the exact typed payload listed in the function arguments.

Binding rules:

- `RoleAttestationProof` **MUST** carry `capability_hash`.
- `DelegatedGrantProof` **MUST** carry `capability_hash`.
- Verifier **MUST** recompute hash from incoming capability payload and compare.
- Mismatch **MUST** fail closed.

Canonicalization requirements:

- deterministic canonical serialization (Candid canonical encoding)
- include target canister principal
- include service ID
- include capability version
- include exact capability variant discriminant and fields
- exclude replay metadata
- exclude proof wrapper fields

---

## 9. Service-Specific Capability Enums

Capabilities are per target service.
Examples:

- `ProjectRegistryCapability`
- `ProjectCapability`
- `CyclesCapability`
- `CanisterLifecycleCapability`

Policy modules are service-local:

- `policy::project_registry::*`
- `policy::project::*`
- `policy::cycles::*`
- `policy::canister_lifecycle::*`

Services **MUST NOT** share one global mega-capability enum.

## 10. Proof Models

### 10.1 Structural Mode

`Structural` uses local topology/runtime predicates only (caller-parent-child, same subnet registration, root identity).

`Structural` is a verification mode, not a portable proof object.
It carries no signed payload.
It remains valid only in the local execution context.
Structural verification **MUST NOT** perform signature verification.

### 10.2 Role Attestation Proof

`RoleAttestationProof` wraps 0.12 attestation evidence and adds capability binding:

```rust
struct RoleAttestationProof {
    proof_version: u16,
    capability_hash: [u8; 32],
    attestation: SignedRoleAttestation,
}
```

Role attestation signatures **MUST** remain domain-separated per 0.12:

```text
H("CANIC_ROLE_ATTESTATION_V1" || canonical(RoleAttestation))
```

Verifier **MUST** enforce:

1. signature/key_id resolution
2. subject binding (`subject == ic::caller()`)
3. audience binding (`audience == ic::id()` when present/required)
4. expiry
5. epoch floor
6. capability hash equality
7. subnet binding when required by service policy

Policy checks are evaluated after proof verification in the authorization pipeline.

### 10.3 Delegated Grant Proof

`DelegatedGrantProof` carries explicit signed authority scoped to capability/action:

```rust
struct DelegatedGrantProof {
    proof_version: u16,
    capability_hash: [u8; 32],
    grant: DelegatedGrant,
    grant_sig: Vec<u8>,
    key_id: u32,
}
```

Delegated grant signatures **MUST** be domain-separated:

```text
H("CANIC_DELEGATED_GRANT_V1" || canonical(DelegatedGrant))
```

Grant payload **SHOULD** include:

- issuer
- subject
- audience
- scope (allowed service/capability family)
- capability_hash
- quota
- issued_at/expires_at
- epoch

## 11. Standard Replay / Idempotency

All mutating capability invocations **MUST** include `RequestMetadata`.

Metadata validity:

- `MAX_CLOCK_SKEW_SECONDS` **MUST** be `30`
- `ttl_seconds` **MUST** be > 0 and <= configured max
- `issued_at` **MUST NOT** be in the future beyond allowed skew
- `now` **MUST** satisfy `now <= issued_at + ttl_seconds`
- replay identity **MUST** use the tuple `(request_id, nonce)`

### 11.1 Replay Slot Key

```text
ReplaySlotKey = H(
  REPLAY_SLOT_DOMAIN ||
  caller ||
  target_canister ||
  service ||
  request_id ||
  nonce
)
```

### 11.2 Replay Digest

```text
ReplayDigest = H(
  REPLAY_DIGEST_DOMAIN ||
  canonical(target_canister, service, capability_version, capability)
)
```

Replay behavior:

- existing slot + equal digest -> return cached response
- existing slot + different digest -> reject as replay conflict
- expired entry -> reject; do not resurrect
- bounded store required; expired entries purged first

Replay storage structure (recommended):

```rust
StableBTreeMap<ReplaySlotKey, ReplayEntry>
```

Where `ReplayEntry` contains at least:

- replay digest
- issued_at
- expires_at
- serialized response bytes

### 11.3 Root Compatibility Profile (0.11)

0.11 root replay is a profile of the shared replay system:

- `target_canister = root_canister_id`
- `service = Root`

Unified keying remains:

```text
ReplaySlotKey = H(
  REPLAY_SLOT_DOMAIN ||
  caller ||
  target_canister ||
  service ||
  request_id ||
  nonce
)
```

Compatibility guidance for legacy 0.11 request metadata that does not include
`nonce`:

- implementations **MAY** map missing nonce to `ZERO_16` during migration
- new callers **SHOULD** always send explicit `nonce`

## 12. Authorization Pipeline

Per-service execution order is normative:

1. Receive transport payload.
2. Extract context (`caller`, `self`, `subnet`, `now`).
3. Decode concrete service capability request DTO.
4. Compute canonical capability hash.
5. Verify proof (structural or crypto path, including hash binding).
6. Check replay/idempotency metadata.
7. Evaluate service policy for `(ctx, capability, verified_proof_claims)`.
8. Execute workflow/ops handler.
9. Persist replay record.

Dispatcher behavior:

- dispatcher **MUST** match exact enum discriminants
- dispatcher **MUST NOT** dispatch by strings/reflection
- unknown variants/versions **MUST** fail closed

## 13. Example: Project Registry Add Entry

Use case:

Any `ProjectInstance` canister may request `ProjectRegistry` to add an entry.

Recommended proof:

- `RoleAttestationProof` with role `ProjectInstance`

Registry authorization decision:

- verify role attestation cryptographically
- ensure caller-subject binding
- ensure audience binding to registry canister
- enforce role == `ProjectInstance`
- enforce subnet/epoch/expiry constraints
- evaluate policy constraints for `AddEntry`

This avoids registry-wide topology sync while preserving strict role validation.

---

## 14. Example: Parent-Mediated Cycles

`CyclesCapability::RequestTopup { amount }` can be authorized via:

- `Structural` mode for direct child -> parent funding
- `DelegatedGrant` proof when parent acts under root-issued budget authority

This preserves fast structural paths while enabling controlled delegated treasury models.

---

## 15. Security Invariants

The following are mandatory:

- deny by default on any proof, policy, replay, or decode failure
- no execution without explicit capability enum variant
- no proof accepted without required capability-hash binding
- no mutable capability without replay metadata
- no cross-domain key reuse between attestation and delegation/grants
- no policy logic embedded in endpoint handlers

---

## 16. Migration Strategy

1. Introduce shared capability envelope DTO primitives (without replacing 0.11 endpoints).
2. Implement one service pilot (`ProjectRegistryCapability`) using `RoleAttestationProof`.
3. Add shared replay helpers for non-root service capability endpoints.
4. Add `CyclesCapability` with structural + delegated-grant paths.
5. Expand service-by-service; keep root 0.11 dispatcher intact for root-only capabilities.

Compatibility rule:

0.11 root capability endpoints remain supported during 0.13 rollout.

---

## 17. Performance Considerations

Expected per-request overhead:

- one capability hash computation
- one signature verification for non-structural proofs
- one replay map lookup

These costs are typically small relative to most IC workflow execution paths.
Replay stores remain bounded and short-lived due to TTL expiration.

---

## 18. Testing Requirements

Each capability service **MUST** include:

- unit tests for version dispatch
- unit tests for hash-binding success/failure
- unit tests for proof verification failures (subject/audience/expiry/epoch/signature)
- unit tests for policy deny-by-default paths
- replay tests (duplicate same, duplicate conflict, expired metadata)
- PocketIC integration tests for cross-canister invocation flows

---

## 19. Final Decision

Canic 0.13 introduces a distributed capability invocation framework where:

- request transport is open
- authority is explicit and proof-backed
- capability semantics are service-specific
- replay and payload binding are standardized

This preserves 0.11 and 0.12 security guarantees while enabling flexible,
auditable, cross-canister service interactions.
