# Canic 0.70 Design: Toko Blob Storage Backend Migration

## Status

- Status: draft design note
- Release line: 0.70 candidate
- Decision style: downstream backend migration
- Primary implementation target: make Toko consume Canic blob storage backend
- Current state: depends on 0.68 protocol and 0.69 billing

## 1. Summary

0.70 migrates Toko backend canisters from local blob storage protocol and
billing code to the Canic `blob-storage` backend feature.

Toko remains the owner of its asset domain, upload UI, generated declarations,
and frontend behavior. This design only covers backend canister code,
backend-generated Candid compatibility, stable data migration, and backend
verification.

## 2. Problem

After 0.68 and 0.69, Canic owns the reusable backend protocol and billing
implementation. Toko should stop carrying duplicated backend protocol logic,
but its domain model must not be collapsed into Canic.

The migration needs to preserve:

1. Gateway method names and Candid signatures.
2. Existing Toko asset semantics.
3. Existing mainnet/live blob state.
4. Backend billing behavior.
5. Local development registration behavior.

## 3. Goals

1. Enable Canic `blob-storage` features in the relevant Toko backend crates.
2. Replace Toko's local gateway endpoint implementations with Canic endpoint
   macros.
3. Replace Toko backend Cashier/billing helpers with Canic 0.69 APIs.
4. Preserve Toko `RemoteAsset` and related domain entities.
5. Provide stable migration for existing backend blob records if record
   ownership moves.
6. Regenerate and diff Candid output.
7. Verify backend local, PocketIC, and manual mainnet upload/download behavior.

## 4. Non-goals

0.70 does not:

1. Rewrite Toko frontend upload UI.
2. Introduce a Canic TypeScript client.
3. Publish npm packages.
4. Change Toko `RemoteAsset` semantics.
5. Change generated frontend call sites except through normal declaration
   regeneration.
6. Add generic blob metadata endpoints.
7. Change Cashier production behavior beyond calling through Canic wrappers.

## 5. Migration Inventory

Before edits, produce a migration inventory with:

1. Toko backend files that implement `_immutableObjectStorage*` endpoints.
2. Toko backend files that call Cashier directly.
3. Toko stable/IcyDB entities that store live blobs, pending deletion,
   gateway principals, or billing config.
4. Current generated Candid signatures.
5. Current local-dev behavior for registration without gateway upload.
6. Current mainnet verification commands and expected observable behavior.

This inventory should become the checklist for code deletion. No Toko backend
protocol file should be deleted until its replacement and migration path are
identified.

## 6. Backend Migration Plan

### 6.1 Project Instance Canister

1. Enable Canic `blob-storage` and required billing features.
2. Initialize blob storage protocol config during lifecycle startup.
3. Emit 0.68 protocol endpoints with Toko's existing certificate creation
   guard.
4. Emit 0.69 billing endpoints with Toko's existing parent/operator guards.
5. Replace local blob liveness/deletion helpers with `BlobStorageApi`.
6. Preserve Toko asset creation and validation rules around `RemoteAsset`.

### 6.2 Project Hub Canister

1. Replace hub blob billing status logic with Canic 0.69 status workflow where
   applicable.
2. Preserve hub authorization and project lookup boundaries.
3. Keep hub-specific status aggregation in Toko if it is domain-specific.

### 6.3 Asset Canister

Only migrate asset canister code if the inventory proves it owns generic blob
protocol behavior. Domain asset storage remains in Toko.

## 7. Stable Migration

The migration must choose one of two paths:

1. **No state move:** Toko keeps domain records, Canic records start from
   existing gateway-compatible state through explicit registration/sync.
2. **State move:** Toko's existing blob records are migrated into Canic stable
   records during upgrade.

If state moves, add tests for:

1. Existing live blobs remain live after upgrade.
2. Existing pending deletion entries remain pending.
3. Existing gateway principals remain authorized or are immediately refreshed
   from Cashier.
4. Unknown or malformed historical data fails with a typed migration error.

Do not delete Toko's old records until upgrade tests prove the migration path.

## 8. Candid Compatibility

After macro migration, regenerate declarations and diff:

1. `_immutableObjectStorage*` method names.
2. Method modes: query/update.
3. Request DTO fields and order where Candid is sensitive.
4. Response variants and nested records.
5. Existing non-blob Toko methods.

Any gateway-facing Candid drift is a blocker unless it is first confirmed as
compatible with the gateway contract.

## 9. Local Development Behavior

Toko local development should continue to support backend registration without
requiring the public gateway. The backend migration must preserve the existing
local policy through Canic backend APIs instead of duplicating local branches
inside Toko protocol code.

No CI or local tests should depend on `https://blob.caffeine.ai`.

## 10. Mainnet Manual Verification

Manual mainnet verification should cover:

1. Gateway principal sync from production Cashier.
2. Backend certificate creation.
3. Gateway upload using the existing Toko frontend or existing external tool.
4. Blob liveness query after upload.
5. Download through production gateway.
6. Pending deletion and gateway deletion confirmation if safe on a test blob.
7. Billing status and low-balance top-up behavior.

The verification plan may use existing Toko frontend tooling, but this design
does not modify frontend code.

## 11. Code Removal Rules

Remove Toko backend code only after the Canic replacement is wired and tested.

Expected removals:

1. Local gateway endpoint implementations.
2. Hand-rolled Cashier Candid snippets replaced by Canic typed ops.
3. Duplicate gateway-principal sync helpers.
4. Duplicate blob liveness/deletion protocol helpers.

Expected retained code:

1. `RemoteAsset` and domain asset records.
2. Toko upload UI orchestration.
3. Toko authorization policy definitions.
4. Toko project/hub domain aggregation.
5. Generated declarations after regeneration.

## 12. Tests

Toko backend tests:

1. Project instance emits gateway-compatible endpoints.
2. Existing asset creation still works.
3. Blob liveness helper works through Canic backend API.
4. Deletion helper works through Canic backend API.
5. Billing status calls Canic workflow.
6. Authorization failures match existing backend policy.

Migration/upgrade tests:

1. Old state upgrades without data loss.
2. Live blob state is preserved.
3. Pending deletion state is preserved.
4. Gateway principals are preserved or refreshed.

Integration checks:

1. Regenerated `.did` diff is reviewed.
2. Toko backend test suite passes.
3. Canic relevant blob storage tests still pass.
4. Manual mainnet verification is recorded.

## 13. Acceptance Criteria

0.70 is complete when:

1. Toko backend consumes Canic blob storage protocol and billing APIs.
2. Toko no longer owns duplicated backend protocol logic.
3. Toko asset/domain behavior is unchanged.
4. Gateway-facing Candid is compatible with the pre-migration contract.
5. Stable migration is tested or explicitly avoided with a documented no-state
   move plan.
6. Local development registration still works without production gateway calls.
7. Mainnet manual verification confirms upload/download and billing behavior.

## 14. Deferred Work

Frontend migration, TypeScript client extraction, npm packages, upload UI
changes, and generic metadata endpoints are intentionally left to later work.
