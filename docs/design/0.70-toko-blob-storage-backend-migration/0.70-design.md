# Canic 0.70 Design: Toko Blob Storage Backend Migration

## Status

- Status: draft design note
- Release line: 0.70 candidate
- Decision style: downstream backend migration
- Primary implementation target: make Toko consume Canic blob storage backend
- Current state: depends on 0.68 protocol and 0.69 billing

## 1. Summary

0.70 migrates Toko backend canisters from local blob storage protocol and
billing code to Canic 0.68 `blob-storage` protocol APIs and Canic 0.69
`blob-storage-billing` APIs.

Toko keeps `RemoteAsset`, upload UI, frontend behavior, generated declaration
consumption, domain aggregation, and domain authorization policy. 0.70 verifies
backend Candid compatibility, stable-state behavior, local development
behavior, and mainnet behavior.

Gateway-owned endpoint names stay exactly `_immutableObjectStorage*`.
Canic-owned endpoints use snake_case Canic naming, such as
`get_blob_storage_status`.

## 2. Goals

1. Enable Canic 0.68 and 0.69 blob storage features in relevant Toko backend
   crates.
2. Replace Toko's local gateway endpoint implementations with Canic endpoint
   macros.
3. Replace Toko backend Cashier/billing helpers with Canic 0.69 APIs.
4. Preserve Toko `RemoteAsset` and related domain entities.
5. Select and test a state strategy for each migrated canister.
6. Regenerate and diff Candid output before and after migration.
7. Verify backend local, PocketIC, and manual mainnet behavior.

## 3. Non-Goals

0.70 does not:

1. Rewrite Toko frontend upload UI.
2. Introduce a Canic TypeScript client.
3. Publish npm packages.
4. Change Toko `RemoteAsset` semantics.
5. Add generic blob metadata endpoints.
6. Change Cashier production behavior beyond calling through Canic wrappers.
7. Define new Canic protocol or billing APIs.

## 4. Pre-Implementation Gates

0.70 implementation cannot begin until these artifacts exist:

1. `docs/contracts/BLOB_STORAGE_INVENTORY.md`
2. `docs/contracts/BLOB_STORAGE_CASHIER_INVENTORY.md`
3. `docs/contracts/TOKO_CASHIER_CALL_INVENTORY.md`
4. `docs/contracts/TOKO_BLOB_STORAGE_MIGRATION_INVENTORY.md`

The first two artifacts come from 0.68 and 0.69. The Toko Cashier-call
inventory is deferred until before 0.70 implementation, not a 0.69 v1 blocker.

## 5. Migration Inventory

Required artifact:

```text
docs/contracts/TOKO_BLOB_STORAGE_MIGRATION_INVENTORY.md
```

It must include:

1. Toko files implementing `_immutableObjectStorage*`.
2. Toko files calling Cashier directly.
3. Toko stable/IcyDB entities for live blobs, pending deletion, gateway
   principals, and billing config.
4. Current generated Candid.
5. Current local-dev registration behavior.
6. Current mainnet verification commands.
7. Canic replacement or retained Toko owner for each code path.
8. Selected state strategy for each migrated canister.
9. Deletion gate for each old helper/file.

No Toko backend protocol or Cashier helper file may be deleted until this
inventory identifies its replacement, retained owner, or blocker.

## 6. Endpoint and Guard Mapping

| Endpoint | Target Toko canister | Canic macro/API | Existing Toko guard or authorization source | Notes |
| --- | --- | --- | --- | --- |
| `_immutableObjectStorageCreateCertificate` | Project instance | 0.68 blob-storage endpoint macro | Existing Toko certificate creation guard | Gateway Candid must match 0.68 inventory exactly. |
| `_immutableObjectStorageBlobsAreLive` | Project instance | 0.68 blob-storage endpoint macro | Inventory-defined public/query behavior | Must answer for all existing live blobs before cutover. |
| `_immutableObjectStorageBlobsToDelete` | Project instance | 0.68 blob-storage endpoint macro | 0.68 gateway-principal store after cutover | Requires gateway-principal cutover before use. |
| `_immutableObjectStorageConfirmBlobDeletion` | Project instance | 0.68 blob-storage endpoint macro | 0.68 gateway-principal store after cutover | Requires correct pending-deletion state for existing blobs. |
| `_immutableObjectStorageUpdateGatewayPrincipals` | Project instance | 0.69 billing endpoint macro | Existing Toko sync/admin guard | Writes through Canic 0.68 gateway-principal storage API. |
| `_immutableObjectStorageFundFromProjectCycles` | Project instance | 0.69 billing endpoint macro | Existing Toko funding/admin guard | Explicit top-up path; response remains 0.68-inventoried gateway protocol. |
| `get_blob_storage_status` | Project instance; hub may aggregate | 0.69 billing endpoint macro/API | Existing Toko status/admin guard | Must not transfer cycles or call `account_top_up_v1`. |

Explicit top-up behavior is verified through
`_immutableObjectStorageFundFromProjectCycles`, not through
`get_blob_storage_status`.

## 7. State Strategy

Before endpoint macro cutover, each migrated canister must choose one state
strategy:

| Strategy | Use when |
| --- | --- |
| No existing blob state | The canister has no live blob, pending deletion, gateway-principal, or billing config state to preserve. |
| Read-through adapter to existing Toko records | Canic endpoints can answer protocol queries by reading existing Toko records without moving state. |
| Bulk registration into Canic records | Existing Toko records are registered into Canic records during upgrade or controlled maintenance. |
| State move into Canic stable records | Ownership of compatible blob/billing state moves into Canic stable records during upgrade. |

Do not cut over to Canic endpoint macros unless Canic can answer liveness,
pending deletion, and deletion confirmation correctly for all existing live
blobs.

Do not delete Toko's old records until upgrade tests prove the selected
strategy.

## 8. Gateway Principal Cutover

Before Canic gateway-only endpoints are relied on, the 0.68 gateway-principal
store must be populated by either:

1. Migrating existing Toko gateway principals.
2. A successful 0.69 Cashier sync.

A migrated project canister must not depend on an empty Canic
gateway-principal store immediately after upgrade.

## 9. Billing Config Migration

The migration inventory must identify existing Toko billing config behavior.
0.70 must either:

1. Migrate existing billing config into Canic `BlobStorageBillingConfig`.
2. Document why Canic billing config is initialized fresh.

Tests must cover missing config, valid migrated config, and unsupported
separate payment-account behavior if present historically.

## 10. Candid Diff Artifacts

Required artifacts:

```text
docs/contracts/toko/pre_migration/*.did
docs/contracts/toko/post_migration/*.did
docs/contracts/toko/CANDID_DIFF_0_70.md
```

The diff report must classify every difference as:

1. Identical/expected.
2. Expected addition.
3. Compatible drift confirmed by inventory.
4. Blocker.

Gateway-facing Candid drift remains a blocker unless explicitly confirmed
compatible with the gateway contract.

## 11. Local and CI Behavior

Local registration behavior must be preserved through Canic APIs, not by
retaining duplicate Toko protocol branches.

Local/PocketIC tests may use Canic-supported test/local gateway-principal
seeding only when explicitly configured. Production builds must not authorize
`TestSeeded` principals.

CI and local tests must not call the production Caffeine gateway or production
Cashier.

## 12. Mainnet Rollout Safety

Manual mainnet verification must start with a test/canary project.

Record:

1. Canister IDs.
2. Git SHAs.
3. Wasm SHAs or build identifiers.
4. Candid diffs.
5. Command outputs.

Destructive deletion confirmation may be verified only on a disposable test
blob.

Rollback notes must name the last known working Toko Wasm/git SHA and the state
backup/checkpoint procedure.

## 13. Code Removal Rules

Removal checks must be compile-checkable where practical:

1. After migration, Toko must not define local `_immutableObjectStorage*`
   endpoints outside Canic macros.
2. Direct Cashier calls in migrated paths must be absent unless listed as
   retained Toko-domain-specific calls in
   `docs/contracts/TOKO_CASHIER_CALL_INVENTORY.md`.
3. Old Toko protocol helpers are deleted only after Canic tests and Candid diff
   pass.

Expected retained code:

1. `RemoteAsset` and domain asset records.
2. Toko upload UI orchestration.
3. Toko authorization policy definitions.
4. Toko project/hub domain aggregation.
5. Generated declarations after regeneration.

## 14. Tests

Toko backend tests:

1. Project instance emits gateway-compatible endpoints through Canic macros.
2. Existing asset creation still works.
3. Blob liveness helper works through Canic backend API.
4. Deletion helper works through Canic backend API.
5. Billing status calls Canic workflow and never funds.
6. Explicit funding works through `_immutableObjectStorageFundFromProjectCycles`.
7. Authorization failures match existing backend policy.
8. No endpoint collisions exist between old Toko endpoints and Canic macros.
9. No retained direct Cashier calls exist except documented retained cases.

Migration/upgrade tests:

1. Old state upgrades into the selected state strategy.
2. Live blob state is preserved or correctly registered.
3. Pending deletion state is preserved or correctly registered.
4. Gateway-principal cutover happens before gateway-only calls.
5. Billing config migrates or initializes fresh according to the inventory.
6. Unsupported separate payment-account behavior is reported if present
   historically.

Local/PocketIC tests:

1. Local registration works without production gateway or Cashier calls.
2. Test/local gateway-principal seeding works only when explicitly configured.
3. Production behavior does not authorize `TestSeeded` principals.

Integration checks:

1. `docs/contracts/toko/CANDID_DIFF_0_70.md` is reviewed.
2. Toko backend test suite passes.
3. Relevant Canic blob storage tests still pass.
4. Manual mainnet verification is recorded.

## 15. Acceptance Criteria

0.70 is complete when:

1. All four pre-implementation artifacts exist.
2. `TOKO_BLOB_STORAGE_MIGRATION_INVENTORY.md` classifies every old protocol,
   Cashier, stable-state, and billing path.
3. Each migrated canister has a selected and tested state strategy.
4. Gateway-principal cutover is complete before gateway-only endpoint reliance.
5. Toko backend consumes Canic 0.68 protocol and Canic 0.69 billing APIs.
6. Toko no longer owns duplicated backend protocol logic.
7. Direct Cashier calls in migrated paths are removed or documented as retained
   Toko-domain-specific calls.
8. Gateway-facing Candid is compatible with the pre-migration contract and the
   Candid diff report is recorded.
9. Billing config is migrated or intentionally initialized fresh.
10. Local development registration still works without production gateway or
    Cashier calls.
11. Mainnet manual verification records canister IDs, SHAs, Candid diffs,
    command outputs, and rollback notes.
12. Toko asset/domain behavior is unchanged.

## 16. Deferred Work

Frontend migration, TypeScript client extraction, npm packages, upload UI
changes, generic metadata endpoints, and new Canic protocol/billing design are
intentionally left to later work.
