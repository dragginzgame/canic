# 0.41 Design: Deployment Truth Model

## Status

Tentative design notes for the 0.41 line.

0.41 is not a new deployment executor. It is the truth and safety layer around
the current install path:

```text
Canic tells the truth before it mutates anything.
```

The immediate problems are not lack of orchestration features. They are hidden
deployment assumptions:

- configured controllers can differ from actual IC controllers;
- root may not be an IC controller after install;
- reused pool canisters may not be claimable by the deployment authority;
- `wasm_store` behaves like a hidden build/install role;
- artifacts can be discovered but not actually materialized;
- the same wasm source can embed different deployment config;
- retries can repeat expensive upload phases without proving prior progress;
- after 0.40, root identity and verifier state affect protected internal calls.

0.41 should expose these facts, validate them, and make the current installer
refuse unsafe states. It should not yet reconcile controller policy, abstract
all execution backends, implement promotion, or model user-owned upgrade
consent flows.

---

## Scope Calibration

Release bar:

- read-only inventory;
- role artifact manifest;
- safety report;
- post-build materialization checks;
- lightweight current-install phase receipts with verified postconditions for
  phases the current installer already executes.

Design hook:

- deployment identity stored by deployment target, not only fleet template name;
- inventory-only reads against `ic`;
- host-side config validation before build;
- canonical digest model that later promotion and drift reports can reuse.

Deferred:

- controller reconciliation;
- backend-neutral execution;
- promotion;
- external/user-owned lifecycle workflows;
- multi-deployment operator UX.

---

## Core Decision

0.41 revolves around four separate objects:

```text
DeploymentPlanV1       intended state
DeploymentInventoryV1  observed live/local state
DeploymentReceiptV1    attempted actions plus verified postconditions
DeploymentDiffV1       normalized machine diff
SafetyReportV1         operator-facing report derived from the diff
```

These objects must not blur together.

Receipts are evidence, not truth. A receipt can help skip work only after live
inventory proves the phase postcondition still holds.

Receipts also must not imply that deployment execution is atomic. Current and
future deploy paths may change one role, fail on a later role, and leave later
roles unattempted. The receipt model must be able to say what changed, what
failed, what was skipped, and what still requires live re-inventory before
resume.

Key invariant:

```text
Canic never installs from intent alone.
It installs only after comparing intent against observed deployment truth.
```

On install and resume, Canic compares:

```text
intended plan
vs
last receipt
vs
current live inventory
```

This comparison is the foundation for later authority reconciliation,
backend-agnostic execution, promotion, user-owned lifecycle management, and
multi-deployment drift reports.

## 0.40 Context

0.40 changed the deployment boundary.

Root is now the signing authority for protected internal calls. Root identity,
embedded config, role assignment, verifier key material, role epochs, and
controller state are part of whether protected internal RPC works safely.

Deployment state therefore affects authorization, not just installation.

0.41 must treat these as first-class deployment facts:

- root trust anchor;
- canonical embedded config digest;
- AppIndex and subnet-registry role assignment source;
- current role epoch and verifier-readiness assumptions where observable;
- controller sets that can bypass Canic's protected-call layer through direct
  management canister authority;
- whether a canister is controlled by deployment authority, pool-managed,
  imported, user-controlled, or unknown.

---

## Deployment Identity

A deployment identity is more than a root principal, but the root trust anchor
is central.

Terminology:

```text
FleetTemplate:
  reusable roles, topology, build rules, config defaults, and pool policy shape

DeploymentTarget:
  concrete live root, network, controllers, canister IDs, artifact set,
  receipts, and inventory

DeploymentGroup:
  related deployment targets with comparison or promotion policy
```

The deployment truth model should store identity by deployment target, not only
by fleet template name. `FleetTemplate` is the model term. `canic fleet ...`
is the template/adoption command namespace, distinct from `canic deploy ...`.
Live deployment operations should say `deployment`.

Tentative shape:

```text
DeploymentIdentityV1 {
  deployment_name,
  network,
  root_principal,
  authority_profile_hash,
  role_topology_hash,
  deployment_manifest_digest,
  canonical_runtime_config_digest,
  role_embedded_config_set_digest,
  artifact_set_digest,
  pool_identity_set_digest,
  canic_version,
  ic_memory_version,
}
```

Trust-domain rule:

```text
same deployment:
  same root trust anchor

new deployment:
  different root trust anchor

migration:
  explicit plan connecting old trust domain to new trust domain
```

If root changes, Canic should not treat that as a normal resume. It is either a
new trust domain or an explicit migration.

Digest meanings:

```text
deployment_manifest_digest:
  canonical hash over root, network, runtime variant, role topology,
  authority profile hash, role artifact identities, pool identities,
  and expected verifier assumptions

canonical_runtime_config_digest:
  canonical hash over the resolved runtime config selected by the plan

role_embedded_config_set_digest:
  aggregate hash over each role's canonical embedded config digest

artifact_set_digest:
  aggregate hash over all planned role artifact identities
```

The deployment-level digests must not hide per-role embedded config drift.

---

## Canonical Embedded Config Identity

`canonical_embedded_config_sha256` is computed over
`CanonicalEmbeddedConfigV1`, not raw config file bytes.

Canonicalization includes:

- resolved includes;
- stable map ordering;
- normalized principal text;
- normalized role ordering;
- explicit defaults;
- normalized network, root, and trust-domain references;
- no incidental whitespace;
- schema version in the hash input.

If useful, 0.41 may record both:

```text
raw_config_sha256
canonical_embedded_config_sha256
```

Safety decisions must use the canonical digest. Raw config digests are useful
for diagnostics, but they must not create false drift when two config files
resolve to the same deployment semantics.

## Data Model

### DeploymentPlanV1

The plan is intended state.

It should include:

```text
DeploymentPlanV1 {
  schema_version,
  plan_id,
  deployment_identity,
  trust_domain,
  fleet_template,
  runtime_variant,
  authority_profile,
  role_artifacts,
  expected_canisters,
  expected_pool,
  expected_verifier_readiness,
  unresolved_assumptions,
}
```

Preview plans may contain unresolved assumptions.

Executable plans may not.

### DeploymentInventoryV1

The inventory is observed state.

It should include live IC facts and local artifact facts:

```text
DeploymentInventoryV1 {
  schema_version,
  observed_at,
  observed_identity,
  local_config,
  observed_canisters,
  observed_pool,
  observed_artifacts,
  observed_verifier_readiness,
  unresolved_observations,
}
```

Observed canister facts should include:

- canister principal;
- expected or inferred role;
- control classification;
- actual IC controllers;
- module hash;
- canister status when available;
- root trust anchor used by the canister when observable;
- canonical embedded config digest when observable;
- role assignment source when observable.

Obsolete records such as `.canic/<network>/fleets/<name>.json` are
pre-`DeploymentTarget` state. 0.41 should not read them as authoritative live
deployment identity. Operators should create or select explicit
deployment-target state instead.

### DeploymentReceiptV1

The receipt is evidence of attempted work.

It should include:

```text
DeploymentReceiptV1 {
  schema_version,
  operation_id,
  plan_id,
  operation_status,
  started_at,
  finished_at,
  operator_principal,
  root_principal,
  previous_observed_deployment_epoch,
  phase_receipts,
  role_phase_receipts,
  final_inventory_id,
  command_result,
}
```

Every phase receipt records both:

```text
attempted_action
verified_postcondition
```

A phase is resumable only if live inventory still proves the verified
postcondition.

The operation status must distinguish clean failure before mutation from
failure after one or more mutations:

```text
DeploymentExecutionStatusV1 {
  NotStarted,
  InProgress,
  FailedBeforeMutation,
  PartiallyApplied,
  FailedAfterMutation,
  Complete,
}
```

Role-scoped phase evidence is required wherever one phase can touch multiple
roles or canisters:

```text
RolePhaseReceiptV1 {
  role,
  phase,
  attempted,
  result,
  previous_module_hash,
  target_module_hash,
  observed_module_hash_after,
  artifact_digest,
  canonical_embedded_config_sha256,
  error,
}

RolePhaseResultV1 {
  Applied,
  Failed,
  Skipped,
  NotAttempted,
  VerifiedAlreadyApplied,
}
```

0.41 can populate this lightly for current installer phases, but the shape
must preserve partial deployment truth for 0.43 execution and 0.50 recovery.

### DeploymentDiffV1 / SafetyReportV1

`DeploymentDiffV1` is the normalized machine diff.

`SafetyReportV1` is the operator-facing report derived from the diff. It may
render, filter, and explain diff categories, but it should not invent a second
drift model.

Tentative shape:

```text
DeploymentDiffV1 {
  plan_identity,
  observed_identity,
  artifact_diff,
  controller_diff,
  pool_diff,
  embedded_config_diff,
  module_hash_diff,
  verifier_readiness_diff,
  resume_safety,
  hard_failures,
  warnings,
  resumable_phases,
}
```

This object should be used by later 0.42 and 0.43 work instead of inventing
new drift semantics.

### RoleArtifact

Role artifacts must capture source bytes, installed bytes, and embedded
deployment identity separately.

Tentative shape:

```text
RoleArtifact {
  role,
  source,
  build_profile,

  wasm_path,
  wasm_gz_path,

  wasm_sha256,
  wasm_gz_sha256,
  installed_module_hash,

  candid_path,
  candid_sha256,

  raw_config_sha256,
  canonical_embedded_config_sha256,
  embedded_topology_sha256,

  builder_version,
  rust_toolchain,
  package_version,
}
```

Longer-term artifact identity should separate source/build recipe,
target-specific materialization input, and materialized output, instead of
adding unrelated provenance fields directly to every artifact record:

```text
BuildRecipeIdentityV1 {
  source_kind,
  source_revision,
  package_or_role_selector,
  cargo_profile,
  cargo_features_digest,
  cargo_lock_digest,
  rust_toolchain,
  builder_version,
}

BuildMaterializationInputV1 {
  build_recipe_id,
  canonical_embedded_config_sha256,
  network,
  root_trust_anchor,
  runtime_variant,
}

BuildMaterializationResultV1 {
  build_recipe_id,
  materialization_input_digest,
  wasm_sha256,
  wasm_gz_sha256,
  installed_module_hash,
  candid_sha256,
}
```

For source/build promotion, the source/build recipe can stay the same while
the target-specific materialization input and output change because target
embedded config changes.

Important comparisons:

- Are these the same source bytes?
- Are these the same installed module bytes?
- Are these the same embedded deployment config?

These can differ independently and must be distinct diff categories.

### CanisterControlClassV1

0.41 should classify canisters even though user-owned lifecycle flows are
deferred.

One canonical enum should be reused by 0.41 inventory, 0.42 authority
reconciliation, and 0.45 external lifecycle work:

```text
CanisterControlClassV1 {
  DeploymentControlled,
  CanicManagedPool,
  ExternallyImported,
  JointlyControlled,
  UserControlled,
  UnknownUnsafe,
}
```

`DeploymentControlled` means the canister is controlled by the deployment
authority for a concrete deployment target.

The current installer should stop treating unknown or user-controlled canisters
as normal managed roles.

## `wasm_store`

`wasm_store` should be modeled as a normal role artifact.

It is also part of artifact transport, so the model must avoid circular
semantics:

```text
wasm_store installation is bootstrapped by the executor.
later artifact staging may use wasm_store.
```

Separate these concepts:

```text
ArtifactRole::WasmStore
ArtifactTransport::LocalCli | ArtifactTransport::WasmStore | ArtifactTransport::DirectAgent
```

0.41 does not need to remove all special-case builder code, but it must expose
`wasm_store` in the plan, inventory, artifact manifest, and diff.

Tentative inventory evidence:

```text
ObservedWasmStoreV1 {
  canister_id,
  control_class,
  controllers,
  module_hash,
  catalog_entries,
  status,
  staging_status,
  gc_status,
}
```

Tentative staged artifact locator:

```text
WasmStoreArtifactLocatorV1 {
  store_canister_id,
  template_id,
  version,
  payload_hash,
  payload_size_bytes,
  chunk_hashes,
}
```

These are evidence about artifact availability. They are not deployment truth
by themselves.

The current artifact-path mismatch class should become a plan/materialization
validation error, not a root build panic.

## Required Validations

0.41 should fail executable plans on:

- missing built artifact for any planned role;
- digest mismatch between planned and materialized artifact;
- canonical embedded config digest mismatch for the target runtime config;
- artifact path mismatch for `wasm_store`;
- controller overlap between normal IC controllers and staging or emergency
  principals;
- root trust-domain mismatch on install continuation or resume analysis;
- root missing as an expected controller where the plan requires root control;
- unclaimable reused pool canister;
- observed module hash mismatch when a receipt claims an installed phase;
- unknown or unsafe control classification for a canister the plan wants to
  mutate.

Warnings may include:

- byte-identical wasm with different embedded config;
- observed controllers that are broader than desired but not immediately
  blocking in read-only mode;
- verifier readiness unavailable;
- role epoch or key material state not observable through current endpoints.

## Phase Receipts And Resume Safety

0.41 should pick the resumability model, but automated phase skipping is not
required for the first safe implementation.

Tentative phases:

```text
create_root
build_artifacts
materialize_artifacts
install_root
stage_release_set
resume_bootstrap
wait_ready
post_validate
```

Each phase receipt records:

- operation ID;
- plan ID;
- operation status;
- root principal;
- started and finished timestamps;
- attempted action;
- artifact/config/trust-domain identity used by the phase;
- verified postcondition;
- live observation used to prove the postcondition.

When a phase touches roles or canisters independently, its receipt must also
record role-scoped outcomes:

- role;
- attempted or not attempted;
- skipped because already verified;
- failed before mutation or after mutation when known;
- previous, target, and observed-after module identity when relevant;
- artifact and canonical embedded config identity used.

Resume rule:

```text
A receipt can skip work only after live inventory proves the phase
postcondition still holds.
```

0.41 required behavior:

- write lightweight phase receipts for phases the current installer already
  executes;
- record verified postconditions for those phases;
- render a resume safety report;
- explain which phases would be resumable;
- refuse unsafe continuation.

0.41 optional behavior:

- automated `--resume` that skips completed phases.

Automated resume may only skip work when these match:

- plan ID and exact deployment plan identity;
- root principal;
- network;
- canonical embedded config digests;
- role artifact digests;
- authority profile hash;
- deployment operation epoch when available.

`--resume-from <phase>` may exist after the safety report model has been
exercised, but it must still enforce identity and postcondition checks.

## Concurrent Deployment Detection And Local Locking

0.41 should include local locking, best-effort race detection, and schema
support for future remote protection.

Minimum 0.41 requirement:

- local deployment lock;
- operation ID;
- receipt identity checks;
- post-validation that detects if observed deployment identity changed during
  the operation.

Before every mutating phase, Canic should re-inventory the relevant mutation
base and refuse the phase when any required base condition changed:

- root trust anchor;
- controller fingerprint;
- module hash when the phase depends on installed state;
- pool ownership or pool readiness state;
- role artifact identity;
- canonical embedded config identity;
- deployment epoch when observable.

Receipt fields should reserve:

- `operation_id`;
- `operator_principal`;
- `previous_observed_deployment_epoch`;
- `observed_deployment_epoch_after`;
- `plan_hash`;
- `root_principal`.

Future 0.42 or 0.43 work can add a lightweight root-side operation epoch or
remote deployment lock.

## Commands

The 0.41 model should support these read-only or report-oriented operator
surfaces:

```text
canic deploy inventory <deployment>
canic deploy plan <deployment>
canic deploy check <deployment>
canic deploy diff <left> <right>
canic deploy resume-report <deployment>
```

The current 0.41 local CLI may expose `canic deploy diff <deployment>` as a
shorthand for the deployment's planned local state versus observed local
inventory. The two-sided `diff <left> <right>` shape remains the general model
for comparing explicit deployment-truth artifacts later.

Mutation still flows through the current install path until 0.43 unless a thin
preflight wrapper is explicitly added. 0.41 may add safety checks and receipts
around current install behavior without replacing the executor.

Future or optional operator surfaces:

```text
canic deploy install <deployment>
canic deploy resume <deployment>
```

0.41 does not need to ship every command at once. It should, however, design
the objects so the commands do not need separate semantics later.

`canic deploy install` is a future operator surface, not a 0.41 minimum.
`canic deploy resume` is optional for 0.41. The required 0.41 surface is the
resume safety report shape that says whether phase skipping would be valid.
`canic deploy resume-report` may satisfy that surface as a read-only report
command; it must not perform automated resume. The command may accept an
explicit `--receipt <file>` override, but the 0.41 local path can use the latest
persisted local deployment receipt as its default evidence input.

## Implementation Slices

These are development slices, not patch-version promises.

### Slice 1: Types And Serialization

Add versioned, serializable models for:

- `DeploymentPlanV1`;
- `DeploymentInventoryV1`;
- `DeploymentReceiptV1`;
- `DeploymentDiffV1`;
- `SafetyReportV1`;
- `RoleArtifactManifest`;
- phase receipts.

Define canonical digest inputs before any digest becomes a safety decision.

### Slice 2: Read-Only Inventory

Resolve the current standard deployment into `DeploymentInventoryV1`.

Inventory should observe local config, local artifacts, current root, known
canisters, current module hashes, controllers, pool state, and `wasm_store`
state.

### Slice 3: Artifact Materialization

Build or inspect role artifacts and produce a `RoleArtifactManifest`.

Every configured role must have a materialized artifact or a typed unresolved
status.

### Slice 4: Diff And Safety Report

Compare plan, receipt, and live inventory into `DeploymentDiffV1`.

Render `SafetyReportV1` from that diff for operator-facing refusal, warnings,
and next actions.

Hard failures and warnings should remain machine-readable.

### Slice 5: Receipts From Current Install

Emit phase receipts from the existing install path.

Receipts must record verified postconditions, not just phase completion.

### Slice 6: Resume Safety Report

Minimum: report shape. Above minimum / hardening unless explicitly promoted:
automated resume.

Compare plan, receipt, and live inventory to explain which phases could be
skipped and which changes block continuation.

Automated phase skipping is optional until these reports have been exercised.

### Slice 7: Safe Current Install Gates

Minimum: incremental safety checks around current installer behavior. Above
minimum / hardening unless explicitly promoted: replacing the current executor.

Make the current installer refuse hard failures before expensive or dangerous
phases.

Post-build materialization checks must run before staging.

### Slice 8: Concurrent Operation Detection

Minimum: local and best-effort detection. Above minimum / hardening unless
explicitly promoted: CI/deployment-grade operation locks, which belong later,
especially 0.49.

Add local locking and pre-phase live checks for mutation bases that can change
under another operator or CI job.

---

## Non-Goals

0.41 should not:

- reconcile or mutate controller policy beyond existing install behavior;
- introduce a full backend-agnostic executor;
- implement promotion;
- implement user-owned consent workflows;
- implement multi-deployment UI;
- convert deployment config into a scripting language;
- treat receipts as authoritative without live inventory.

---

## Exit Criterion

0.41 is complete when:

```text
Canic can say exactly what it plans to do, what exists, what differs,
and whether it is safe to continue.
```

The current installer may still be the executor, but it must not silently skip
roles, stage mixed-config artifacts, ignore controller drift, or continue from
local receipts without live postcondition proof.
