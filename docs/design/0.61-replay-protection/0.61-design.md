# Canic 0.61 Design: System Replay Protection

## Status

Draft for 0.61.

0.61 is a replay-safety hardening release. It promotes replay protection from
scattered endpoint behavior into a system invariant for externally callable
mutations.

## Summary

Canic currently has several replay/idempotency mechanisms, but they are not
consistently applied:

- root capability RPCs have replay metadata, payload hashing, reservation, and
  cached response commits;
- ICP refill is keyed by `operation_id`;
- some wasm-store and bootstrap flows are convergent or monotonic;
- delegated-token one-shot update consumption was introduced as a broad
  auth-layer replay workaround, but that behavior drifted from the delegated
  auth design;
- some externally callable system commands still create fresh durable side
  effects when repeated.

The 0.61 goal is to make replay behavior explicit and enforceable across the
codebase.

Every externally callable mutation must be classified as one of:

1. **Response-idempotent**

   Repeating the same request is harmless and returns success or an equivalent
   response.

2. **Replay-protected**

   Repeating the same operation returns the original committed response.

3. **Intentionally non-idempotent**

   Repeating the request intentionally creates a new durable result. This must
   be explicit in the command name, docs, tests, and changelog.

Accidentally non-idempotent system commands are not allowed.

## Motivation

The immediate audit findings are:

- **Medium:** `PoolAdminCommand::CreateEmpty` is not replay-safe. Replaying the
  command can create and register another pool canister.
- **Medium:** delegation-proof issuance is not on the replay-protected root
  capability path. Replaying the request can mint fresh proof material because
  `issued_at = now`.
- **Low:** `canic_app` set-style commands are side-effect idempotent but not
  response-idempotent. Replaying `SetStatus` or
  `SetCyclesFundingEnabled` can return already-in-state errors.

These should not be fixed as local patches only. The larger problem is that
Canic does not currently have one mandatory replay model for all externally
callable mutations.

0.61 introduces that model and also removes the auth-layer single-update token
consumption path that drifted out of the delegated-auth design.

## Goals

### G1. Define A System-Wide Replay Policy

Every public update endpoint and externally callable mutation must have an
explicit replay policy.

### G2. Extract Or Reuse Existing Root Replay Machinery

The existing root capability RPC replay handler is the right foundation. 0.61
should move or generalize that machinery so other system commands can use it.

If full migration is too risky for one slice, root RPC replay can remain in
place while the shared receipt primitive is introduced for new surfaces.

### G3. Fix Pool `CreateEmpty`

Repeating the same pool-create operation must not create a second pool
canister.

### G4. Fix Delegation-Proof Issuance

Repeating the same delegation-proof request must return the originally
committed proof response, including the same `issued_at` and signature/proof
bytes.

### G5. Make Set-Style Commands Response-Idempotent

Commands such as `SetStatus` and `SetCyclesFundingEnabled` should return
success when the requested state is already true.

### G6. Hard-Cut Auth-Layer One-Shot Update Consumption

Core delegated auth returns to TTL-bounded bearer-token semantics:

- delegated token `nonce` is informational entropy only;
- verifier-side token checks validate signature, root/shard binding, TTL,
  audience, subject, and scope;
- verifier-side token checks do not consume nonce/use markers;
- replay-sensitive mutations use domain operation IDs and replay receipts.

The old update-call consumption path must be removed completely from runtime
code. It must not remain as a dormant optional path, feature flag, or
half-retained stable store.

## Non-Goals

0.61 does not attempt to make every command globally transactional across
external management-canister effects.

In particular, `create_canister` has unavoidable external-effect risk: if the
management canister creates a canister and Canic traps before committing the
returned canister ID, Canic may not be able to recover the created ID
automatically.

0.61 must still prevent a retry from creating another canister for the same
operation. The unresolved operation should become `RecoveryRequired` or
`OperationInProgress`, not silently perform the external effect again.

0.61 also does not require all existing convergent/import/sync flows to use
operation IDs if they are already provably response-idempotent or
snapshot-convergent.

## Core Invariant

No externally callable mutation may create, sign, transfer, register, publish,
or otherwise commit a second durable result from an equivalent retry unless the
command is explicitly documented and tested as intentionally non-idempotent.

## Replay Policy Classification

Add a small replay-policy registry for public update endpoints.

Suggested enum:

```rust
pub enum ReplayPolicy {
    QueryOrReadOnly,

    ResponseIdempotent {
        command_kind: CommandKind,
    },

    ReplayProtected {
        command_kind: CommandKind,
        requires_operation_id: bool,
    },

    IntentionallyNonIdempotent {
        command_kind: CommandKind,
        reason: &'static str,
    },
}
```

Every public update endpoint must be classified.

Initial classifications:

| Surface | Policy |
| --- | --- |
| `canic_request_delegation` | `ReplayProtected(auth.issue_delegation_proof.v1)` |
| `canic_pool_admin::CreateEmpty` | `ReplayProtected(pool.create_empty.v1)` |
| `canic_app::SetStatus` | `ResponseIdempotent(app.set_status.v1)` |
| `canic_app::SetCyclesFundingEnabled` | `ResponseIdempotent(app.set_cycles_funding_enabled.v1)` |
| root capability RPC request handling | `ReplayProtected(existing root RPC command kinds)` |
| ICP refill | `ExistingOperationIdProtected`; align naming/errors with replay core |
| wasm-store staging/publication/lifecycle | `ResponseIdempotent` or `MonotonicTransition` |
| cascade sync | `SnapshotConvergent` |
| bootstrap resume | `ResponseIdempotent` or `EnsureStyle` |

Add a test that fails if exported update endpoints are added without a replay
policy.

## Operation ID Model

Replay-sensitive commands require an operation ID.

```rust
pub struct ReplayMeta {
    pub operation_id: OperationId,
}
```

The request must carry only the operation ID. The server computes caller,
command kind, and payload hash itself.

Do not trust caller-provided `requested_by`, `created_at`, or `payload_hash`.

Recommended operation ID shape:

```rust
pub struct OperationId([u8; 16]);
```

or:

```rust
pub struct OperationId([u8; 32]);
```

The CLI may expose this as UUID, ULID, or hex, but stable storage should use
bytes.

Requirements:

- generated IDs have at least 128 bits of entropy;
- operation identity is command-kind scoped;
- operation identity is caller-bound;
- payload hash is checked on every retry;
- the server must not generate operation IDs for replay-sensitive public API
  requests unless the generated ID is returned and persisted before the request
  is sent.

For CLI use, Canic should support:

```text
--operation-id <id>
```

When the CLI generates an ID, it should display/store the generated operation
ID for retry diagnostics.

## Replay Receipt Store

Introduce or extract a shared receipt store in `canic-core`.

Suggested location:

```text
crates/canic-core/src/workflow/replay/
```

or:

```text
crates/canic-core/src/replay/
```

The current root RPC replay code should eventually use this shared primitive
instead of owning a special implementation. If migration risk is too high,
0.61 may keep root RPC replay unchanged while new replay-protected surfaces

```rust
pub struct ReplayReceiptKey {
    pub command_kind: CommandKind,
    pub operation_id: OperationId,

```

### Receipt Value

```rust
pub struct ReplayReceipt {
    pub schema_version: u32,
    pub command_kind: CommandKind,
    pub operation_id: OperationId,

    pub caller: Principal,
    pub payload_hash: [u8; 32],

    pub status: ReplayReceiptStatus,

    pub created_at_ns: u64,
    pub updated_at_ns: u64,

    pub response_schema_version: Option<u32>,
    pub response_bytes: Option<Vec<u8>>,

    pub effect: Option<ExternalEffectDescriptor>,
}
```

### Status

```rust
pub enum ReplayReceiptStatus {
    Reserved,

    ExternalEffectInFlight,

    Committed,

    TerminalFailed {
        error_code: ReplayTerminalErrorCode,
        error_bytes: Vec<u8>,
    },

    RecoveryRequired {
        reason: RecoveryReason,
    },
}
```

### External Effect Descriptor

```rust
pub enum ExternalEffectDescriptor {
    ManagementCreateCanister {
        command_kind: CommandKind,
    },

    ManagementCall {
        canister: Principal,
        method: String,
    },

    IcpTransfer {
        operation_id: OperationId,
    },
}
```

This descriptor is not meant to fully recover every effect. It is meant to
distinguish "safe to retry" from "an external effect may already have happened".

## Replay Algorithm

Every replay-protected command uses this flow:

1. Compute `command_kind`.
2. Read `operation_id` from request.
3. Read actual caller from IC context.
4. Normalize request payload.
5. Compute `payload_hash`.
6. Reserve receipt.
7. If a committed receipt exists with the same caller/hash, return cached
   response.
8. If a receipt exists with a different caller, reject.
9. If a receipt exists with a different payload hash, reject.
10. If a receipt is pending/in-flight, return `OperationInProgress` or
    `RecoveryRequired`.
11. Execute command.
12. Commit response before returning.

Pseudo-code:

```rust
pub async fn run_replay_protected<C: ReplayCommand>(
    ctx: &mut Ctx,
    caller: Principal,
    operation_id: OperationId,
    request: C::Request,
) -> Result<C::Response, C::Error> {
    let payload_hash = C::payload_hash(caller, &request);

    match replay::reserve::<C>(ctx, caller, operation_id, payload_hash)? {
        ReplayDecision::ReturnCommitted(bytes) => C::decode_response(bytes),
        ReplayDecision::Reject(error) => Err(error.into()),
        ReplayDecision::Execute(token) => {
            let response = C::execute(ctx, token, request).await?;
            replay::commit::<C>(ctx, token, &response)?;
            Ok(response)
        }
    }
}
```

For commands with external awaits, the command must mark the external effect
before making the call:

```rust
replay::mark_external_effect_in_flight(
    ctx,
    token,
    ExternalEffectDescriptor::ManagementCreateCanister {
        command_kind: CommandKind::PoolCreateEmptyV1,
    },
)?;

let result = management_create_canister(...).await;
```

If a duplicate request arrives while the receipt is
`ExternalEffectInFlight`, Canic must not execute the effect again.

## Payload Hashing

Use a deterministic, versioned hash input.

```rust
pub struct ReplayPayloadHashInput<T> {
    pub schema_version: u32,
    pub command_kind: CommandKind,
    pub caller: Principal,
    pub normalized_request: T,
}
```

Hash with SHA-256 or the existing project-standard hash primitive.

Rules:

- include command kind;
- include schema version;
- include actual caller;
- include all user-controlled request inputs;
- exclude `operation_id`;
- exclude server-generated fields;
- exclude current time;
- exclude signatures/proofs/generated response bytes.

For delegation proof issuance, the payload hash must include:

- caller;
- requested subject/delegate identity;
- audience;
- scopes/capabilities;
- requested TTL/expiry policy;
- target/root context;
- delegation kind.

It must not include:

- `issued_at`;
- server signature;
- proof bytes;
- current time.

For pool `CreateEmpty`, the payload hash must include:

- caller;
- pool creation parameters;
- controller settings;
- initial cycles/configuration inputs;
- any role/admin inputs.

It must not include:

- created canister principal;
- registration timestamp;
- server-generated metadata.

## Response Caching

A replay-protected command commits the final response bytes.

For `CreateEmpty`, the committed response must include the created pool canister
principal.

For delegation proof issuance, the committed response must include the exact
issued proof material.

A replay of the same operation must return the committed response, not
recompute it.

This is especially important for delegation:

```text
First call:
  issued_at = T1
  signature = S1

Replay at T2:
  issued_at = T1
  signature = S1
```

Do not mint fresh proof material for a replay.

## Error Behavior

Add common replay errors:

```rust
pub enum ReplayError {
    OperationIdRequired,

    OperationAlreadyCommittedPayloadMismatch,

    OperationAlreadyCommittedCallerMismatch,

    OperationInProgress,

    OperationRecoveryRequired,

    OperationIdInvalid,

    ReceiptDecodeFailed,

    ReceiptSchemaUnsupported,
}
```

Recommended user-facing messages:

```text
operation_id is required for this command
operation_id has already been used with different request payload
operation_id has already been used by a different caller
operation is already in progress; retry later with the same operation_id
operation reached an external side-effect boundary and requires recovery;
Canic will not repeat the side effect for the same operation_id
```

## Fix 1: Hard-Cut Delegated-Token Update Consumption

### Current Problem

Core delegated auth was originally designed as TTL-bounded bearer-token
verification. `nonce` was informational entropy only, and applications that
needed replay protection were expected to use separate replay/idempotency
state.

0.33 added verifier-side update-token consumption keyed by:

```text
(issuer_shard_pid, subject, cert_hash, nonce)
```

That drift made every authenticated update token effectively one-shot. It also
introduced auth-state storage that does not model a domain operation, payload
hash, caller-specific command, or committed response.

This is the wrong layer for system-command replay safety.

### New Behavior

Delegated-token verification should:

1. decode the first ingress argument as `DelegatedToken`;
2. validate delegated-token config is enabled;
3. verify root/shard trust, signature, cert hash, TTL, audience, and scope;
4. bind verified subject to the authenticated caller;
5. return the issuer shard principal for metrics.

It must not:

- consume update tokens;
- write verifier-local token-use state;
- reject a valid token solely because the same nonce was already used;
- treat token nonce as an application operation ID.

### Required Runtime Removal

The 0.61 implementation must remove:

- `consume_update_token_once` from `access/auth/token.rs`;
- the call to token consumption after `enforce_required_scope`;
- `AuthOps::consume_delegated_token_use`;
- `AuthStateOps::consume_delegated_token_use`;
- the `DelegatedTokenUse` ops view;
- `storage/stable/auth/token_uses.rs`;
- `DelegatedTokenUseConsumeResult`;
- `DelegatedTokenUseRecord`;
- `delegated_token_uses` from `AuthStateRecord`;
- `DELEGATED_TOKEN_USE_CAPACITY`;
- auth validation errors dedicated only to delegated-token replay store
  saturation/reuse;
- tests whose expected behavior is "second authenticated update with same token
  is rejected".

The replacement tests should assert:

- token verification still enforces signature, TTL, subject binding, audience,
  and required scope;
- queries and updates have the same bearer-token verification semantics;
- no auth-state token-use marker is written by endpoint authentication;
- system replay tests live with domain commands, not delegated auth.

### Stable State Cut

This is a hard cut, not a compatibility shim. The runtime should not retain a
dormant `delegated_token_uses` field or API.

Upgrade handling must verify the selected stable encoding behavior:

- if removing the field cleanly ignores historical data, document that the old
  consumed-token set is dropped during decode;
- if the encoding cannot ignore the removed field safely, add a one-time
  migration that drops it and writes the new auth-state shape.

Either way, the post-upgrade runtime model has no delegated-token use store.

### Documentation And Audit Updates

Update active docs and audit templates so they no longer describe delegated auth
as single-use:

- architecture authentication docs;
- recurring auth/replay audit definitions;
- access-purity/canonical-auth boundary audit commands that mention
  `consume_update_token_once`;
- 0.61 design/changelog notes.

Historical changelog and historical audit reports should remain historical
records; do not rewrite past release evidence unless a maintainer explicitly
requests it.

### Acceptance Criteria

The hard cut is complete when:

- `rg "consume_update_token_once|consume_delegated_token_use|DelegatedTokenUse|delegated_token_uses"`
  finds no live production path except historical docs, if those are left
  intentionally untouched;
- delegated-auth verifier tests prove bearer-token semantics;
- replay-sensitive system-command tests do not depend on auth-token reuse
  rejection;
- the architecture doc again matches runtime behavior: token replay is bounded
  by TTL unless a domain replay store applies.

## Fix 2: Pool `CreateEmpty`

### Current Problem

`PoolAdminCommand::CreateEmpty` has no request ID and dispatches directly to
pool creation. Repeating the same request can call management
`create_canister` again and register another pool canister.

### New API Shape

Change:

```rust
pub enum PoolAdminCommand {
    CreateEmpty,
}
```

to require replay metadata:

```rust
pub enum PoolAdminCommand {
    CreateEmpty(CreateEmptyPoolRequest),
}

pub struct CreateEmptyPoolRequest {
    pub replay: ReplayMeta,
    pub args: CreateEmptyPoolArgs,
}
```

### New Behavior

```text
same operation_id + same caller + same payload
  return the same created pool canister principal

same operation_id + same caller + different payload
  reject payload mismatch

same operation_id + different caller
  reject caller mismatch

pending/in-flight operation_id
  return OperationInProgress or RecoveryRequired

new operation_id
  create a new pool canister
```

### Execution Sequence

1. Reserve replay receipt.
2. Validate request.
3. Mark external effect in-flight.
4. Call management `create_canister`.
5. On success, register pool canister and commit replay response in the same
   callback.
6. Return response.

After successful `create_canister`, do as little fallible work as possible:

- write pool registry state;
- write replay committed response;
- return.

No logging, formatting, optional hooks, or extra validation should happen
between receiving the created canister ID and committing the receipt.

### Recovery Behavior

If the receipt is stuck in `ExternalEffectInFlight`, a replay must not call
`create_canister` again.

Return `OperationRecoveryRequired` or `OperationInProgress`, depending on age
and configured stale threshold.

0.61 does not need to solve automatic discovery of orphaned created canisters.
It must prevent repeated creation under the same operation ID.

## Fix 3: Delegation-Proof Issuance

### Current Problem

`canic_request_delegation` signs directly. `DelegationProofIssueRequest` has no
replay metadata. Root uses `issued_at = now`, so a repeated request can mint
fresh proof material.

### New API Shape

Change:

```rust
pub struct DelegationProofIssueRequest {
    // existing fields
}
```

to:

```rust
pub struct DelegationProofIssueRequest {
    pub replay: ReplayMeta,
    pub args: DelegationProofIssueArgs,
}
```

### New Behavior

```text
same operation_id + same caller + same proof request
  return the original proof response

same operation_id + changed proof request
  reject payload mismatch

same operation_id + changed caller
  reject caller mismatch
```

### Execution Sequence

1. Reserve replay receipt.
2. Validate request.
3. Compute `issued_at = now`.
4. Sign proof.
5. Commit full proof response.
6. Return proof response.

Delegation proof issuance must not rely on broad delegated-token update
consumption for replay safety. Replay safety comes from the operation receipt.

## Fix 4: Set-Style App Commands

### Current Problem

Some app set commands are side-effect idempotent but not response-idempotent.

Example: `SetStatus(active)` when already active does not mutate again, but
returns an error.

### New Behavior

Set commands should return success even when the requested state is already
true.

Suggested response:

```rust
pub struct SetStateResponse<T> {
    pub previous: T,
    pub current: T,
    pub changed: bool,
}
```

Examples:

```text
previous = stopped
current = running
changed = true

previous = running
current = running
changed = false
```

Apply this to:

- `SetStatus`;
- `SetCyclesFundingEnabled`;
- any equivalent set-style command found during the 0.61 endpoint sweep.

These commands do not require operation IDs if they are fully
response-idempotent.

## Storage And Pruning

Replay receipts are durable state.

Pruning must not re-enable duplicate durable effects.

### Pool Create Receipts

Pool creation receipts should be retained indefinitely, or compacted only into a
tombstone that still prevents operation ID reuse.

Minimum retained data:

- `command_kind`;
- `operation_id`;
- `caller`;
- `payload_hash`;
- `created_pool_canister_principal`;
- `status = committed`.

### Delegation Proof Receipts

Delegation proof responses may expire. After expiry, Canic may compact the full
proof response into a tombstone.

However, replaying the same operation ID after compaction must not mint a fresh
proof.

Allowed behavior after compaction:

- return expired committed proof if retained; or
- return `OperationAlreadyCommittedResponseExpired`.

Not allowed:

- issue a new proof with a new `issued_at` or signature.

### Pending Receipts

Pending or in-flight receipts should have deterministic stale behavior.

```text
fresh pending
  OperationInProgress

stale pending before external effect
  may be cleared or retried

stale external-effect-in-flight
  RecoveryRequired
```

Do not retry an external-effect-in-flight operation automatically.

## Quotas And Abuse Protection

Because operation IDs create durable receipts, add simple quotas.

Suggested initial limits:

- max pending replay receipts per caller;
- max pending replay receipts per command kind;
- max operation ID length;
- max retained delegation proof response size.

Reject new replay-protected operations when quota is exceeded.

Committed pool-create receipts should not be pruned merely for quota relief.

## Upgrade Behavior

Replay receipts must survive canister upgrades.

Add schema versioning:

```rust
pub const REPLAY_RECEIPT_SCHEMA_VERSION: u32 = 1;
```

Upgrade requirements:

- committed receipts remain readable after upgrade;
- pending receipts remain pending/recovery-required after upgrade;
- unsupported receipt schema returns `ReceiptSchemaUnsupported`, not a panic.

No backfill is required for operations completed before 0.61.

The delegated-token update-consumption hard cut must also have upgrade
coverage. Old verifier-local token-use markers are not replay receipts and must
not be migrated into the replay receipt store.

## CLI Behavior

For replay-protected commands, CLI should include an operation ID.

Supported interface:

```text
--operation-id <id>
```

If omitted, the CLI may generate one client-side.

When the CLI generates one, it should display it in verbose/error output:

```text
operation_id: 6f4b0b7e-0c0e-49dd-b6b7-5fb4b9d34654
```

For high-value operations such as pool creation, the CLI should persist the
generated operation ID in a local pending operation log before sending the
request.

Suggested path:

```text
<canic-cache-root>/operations/pending.json
```

This is a client convenience. Server correctness must not depend on the CLI
cache.

## Backward Compatibility

Replay-sensitive request DTOs will change.

Old clients that call replay-sensitive commands without operation IDs should
receive:

```text
OperationIdRequired
```

rather than silently using a generated server-side operation ID.

Server-side generated operation IDs do not solve replay safety because a retry
would get a different ID.

For CLI users, Canic should generate operation IDs client-side and show them in
diagnostics.

The delegated-token update-consumption hard cut intentionally changes
authenticated update behavior back to bearer-token semantics. Tokens remain
bounded by TTL, audience, subject binding, and scope. Domain replay protection
is owned by domain commands.

Replay-sensitive commands now require operation IDs and return committed

## Acceptance Criteria

0.61 is complete when:

- every public update endpoint has an explicit replay policy;
- delegated-token update consumption is fully removed from live runtime code;
- pool `CreateEmpty` cannot create a second pool canister for the same
  `operation_id`;
- delegation proof issuance cannot mint fresh proof material for the same
  `operation_id`;
- set-style commands are response-idempotent;
- delegated auth docs describe TTL-bounded bearer-token semantics;
- existing root capability RPC replay tests still pass;
- ICP refill `operation_id` behavior still passes;
- upgrade tests prove committed receipts survive upgrade;
- changelogs document the replay-safety invariant.

## Release Blockers

The two Medium audit findings are release blockers for 0.61:

- `PoolAdminCommand::CreateEmpty` replay safety;
- delegation-proof issuance replay safety.

The delegated-auth hard cut is also release-blocking because retaining the old
auth-layer consumption path would keep the replay model ambiguous.

The Low set-command response-idempotence issue should ship in the same release
because it uses the same policy framework and prevents future ambiguity.

## Design Rule Going Forward

No new externally callable mutation may be added to Canic unless the PR states
one of:

- `Replay policy: Response-idempotent`;
- `Replay policy: Replay-protected with operation_id`;
- `Replay policy: Intentionally non-idempotent, with reason`.

A command that creates canisters, transfers value, signs proof material,
publishes durable artifacts, mutates authority, or registers infrastructure
should default to:

```text
Replay-protected with operation_id
```

unless there is a written proof that it is response-idempotent.
