# Canic 0.61 Design: System Replay Protection

## Status

Draft for 0.61.

0.61 is a replay-safety hardening release. It promotes replay protection from
scattered endpoint behavior into a system invariant for externally callable
mutations.

## Summary

Canic currently has several replay/idempotency mechanisms, but they are not
consistently applied:

- root capability RPCs have replay metadata, payload hashing, reservation, and
  cached response commits;
- ICP refill is keyed by `operation_id`;
- some wasm-store and bootstrap flows are convergent or monotonic;
- delegated-token one-shot update consumption was introduced as a broad
  auth-layer replay workaround, but that behavior drifted from the delegated
  auth design;
- some externally callable system commands still create fresh durable side
  effects when repeated;
- replay protection alone does not bound callers that generate many fresh
  operation IDs for expensive signing or canister-management paths.

The 0.61 goal is to make replay behavior explicit and enforceable across the
codebase.

Every externally callable mutation must be classified as one of:

1. **Response-idempotent**

   Repeating the same request is harmless and returns success or an equivalent
   response.

2. **Replay-protected**

   Repeating the same operation returns the original committed response.

3. **Intentionally non-idempotent**

   Repeating the request intentionally creates a new durable result. This must
   be explicit in the command name, docs, tests, and changelog.

Accidentally non-idempotent system commands are not allowed.

## Motivation

The immediate audit findings are:

- **Medium:** `PoolAdminCommand::CreateEmpty` is not replay-safe. Replaying the
  command can create and register another pool canister.
- **Medium:** delegation-proof issuance is not on the replay-protected root
  capability path. Replaying the request can mint fresh proof material because
  `issued_at = now`.
- **Low:** `canic_app` set-style commands are side-effect idempotent but not
  response-idempotent. Replaying `SetStatus` or
  `SetCyclesFundingEnabled` can return already-in-state errors.

These should not be fixed as local patches only. The larger problem is that
Canic does not currently have one mandatory replay model for all externally
callable mutations.

0.61 introduces that model and also removes the auth-layer single-update token
consumption path that drifted out of the delegated-auth design.

Because 0.61 intentionally returns delegated auth to reusable bearer-token
semantics, it must also add backend cost-abuse guardrails around the expensive
operations that token issuance and canister creation can reach.

## Goals

### G1. Define A System-Wide Replay Policy

Every public update endpoint and externally callable mutation must have an
explicit replay policy.

### G2. Extract Or Reuse Existing Root Replay Machinery

The existing root capability RPC replay handler is the right foundation. 0.61
must move or generalize that machinery so other system commands can use it.

If full migration is too risky for one slice, root RPC replay can remain in
place while the shared receipt primitive is introduced for new surfaces.

### G3. Fix Pool `CreateEmpty`

Repeating the same pool-create operation must not create a second pool
canister.

### G4. Fix Delegation-Proof Issuance

Repeating the same delegation-proof request must return the originally
committed proof response, including the same `issued_at` and signature/proof
bytes.

### G5. Make Set-Style Commands Response-Idempotent

Commands such as `SetStatus` and `SetCyclesFundingEnabled` must return
success when the requested state is already true.

### G6. Hard-Cut Auth-Layer One-Shot Update Consumption

Core delegated auth returns to TTL-bounded bearer-token semantics:

- delegated token `nonce` is informational entropy only;
- verifier-side token checks validate signature, root/shard binding, TTL,
  audience, subject, and scope;
- verifier-side token checks do not consume nonce/use markers;
- replay-sensitive mutations use domain operation IDs and replay receipts.

The old update-call consumption path must be removed completely from runtime
code. It must not remain as a dormant optional path, feature flag, or
half-retained stable store.

### G7. Add Cost-Abuse Guardrails For Expensive Operations

Replay receipts prevent duplicate effects for equivalent retries. They do not
prevent an abusive caller from sending many distinct operation IDs.

0.61 must add backend-enforced quotas and cycle-reserve circuit breakers before
publicly reachable threshold-ECDSA signing and canister-management effects.
These guardrails protect the canister that pays for the operation, not only the
caller identity being authenticated.

## Non-Goals

0.61 does not attempt to make every command globally transactional across
external management-canister effects.

In particular, `create_canister` has unavoidable external-effect risk: if the
management canister creates a canister and Canic traps before committing the
returned canister ID, Canic may not be able to recover the created ID
automatically.

0.61 must still prevent a retry from creating another canister for the same
operation. The unresolved operation should become `RecoveryRequired` or
`OperationInProgress`, not silently perform the external effect again.

0.61 also does not require all existing convergent/import/sync flows to use
operation IDs if they are already provably response-idempotent or
snapshot-convergent.

0.61 does not require product-specific paid-account, invite, CAPTCHA, or manual
verification gates before project or child-canister creation. Canic may expose
typed errors and policy hooks that allow applications to enforce those gates,
but the framework invariant is backend cost control, not a specific commercial
policy.

## Core Invariant

No externally callable mutation may create, sign, transfer, register, publish,
or otherwise commit a second durable result from an equivalent retry unless the
command is explicitly documented and tested as intentionally non-idempotent.

Add a small replay-policy registry for public update endpoints.

The initial table is a minimum inventory, not a narrowing of scope. The
implementation manifest must enumerate every executing update endpoint emitted
by Canic endpoint macros and every fleet-exposed wrapper that reaches Canic
signing, deployment, transfer, publication, or registration behavior.

Canic currently has no intentionally non-idempotent public update commands. The
category exists for future extensions and must remain explicit: adding one
requires a command name, documentation, tests, and changelog entry that state
why repeated calls intentionally create distinct durable results.

## Operation ID Model

Replay-sensitive commands require an operation ID.

```rust
pub struct ReplayMeta {
    pub operation_id: OperationId,
}
```

The request must carry only the operation ID. The server computes replay actor,
command kind, and payload hash itself.

Do not trust caller-provided `requested_by`, `created_at`, or `payload_hash`.

Operation ID shape:

```rust
pub struct OperationId([u8; 32]);
```

The CLI may expose this as 64-character hex or another exact 32-byte textual
encoding, but stable storage uses bytes. 0.61 does not support a second
operation-ID width.

Requirements:

- generated IDs fill all 32 bytes with cryptographic entropy;
- operation identity is command-kind scoped;
- operation identity is replay-actor-bound;
- payload hash is checked on every retry;
- the server must not generate operation IDs for replay-sensitive public API
  requests.

For CLI use, Canic must support:

```text
--operation-id <id>
```

When the CLI generates an ID, it must display/store the generated operation
ID for retry diagnostics.

## Actor, Quota Subject, And Cost Payer

Replay binding, quota accounting, and cycle risk are related but separate
identities.

```rust
pub struct ReplayActor {
    pub effective_principal: Principal,
    pub auth_kind: AuthKind,
    pub issuer_shard: Option<Principal>,
}

pub enum AuthKind {
    DirectCaller,
    DelegatedToken,
    RoleAttestation,
}

pub struct CostQuotaSubject {
    pub effective_principal: Principal,
    pub subject: Option<Principal>,
    pub session_id_hash: Option<[u8; 32]>,
    pub account_id_hash: Option<[u8; 32]>,
    pub issuer_shard: Option<Principal>,
}

pub struct CostPayer {
    pub canister: Principal,
    pub role: CostPayerRole,
}

pub enum CostPayerRole {
    Root,
    Shard,
    UserHub,
    UserShard,
    Project,
    Pool,
    Other(CanisterRole),
}
```

Replay actor answers "is this the same authenticated actor retrying the same
operation?" For direct calls, it is the IC caller. For delegated-auth calls, it
is the verified authenticated subject/caller binding produced by canonical auth,
not a caller-provided request field.

Quota subject answers "who is consuming this operation budget?" Canic-core may
default to principal-based subjects, but product surfaces may bind this to a
durable user, session, account, invite, paid state, or application-defined
verification policy.

Cost payer answers "which canister's cycles are at risk?" Cycle-reserve checks
are evaluated against the cost payer, not necessarily against the replay actor.

## Replay Receipt Store

Introduce or extract a shared receipt store in `canic-core`.

Suggested location:

```text
crates/canic-core/src/workflow/replay/
```

or:

```text
crates/canic-core/src/replay/
```

0.61 has one replay implementation. Existing root RPC replay behavior must
migrate onto this shared primitive before any public 0.61 release candidate.
Keeping a legacy root replay implementation beside the shared receipt store is
not an accepted 0.61 state.

### Receipt Key

```rust
pub struct ReplayReceiptKey {
    pub command_kind: CommandKind,
    pub operation_id: OperationId,
}
```

### Receipt Value

```rust
pub struct ReplayReceipt {
    pub schema_version: u32,
    pub command_kind: CommandKind,
    pub operation_id: OperationId,

    pub actor: ReplayActor,
    pub payload_hash_schema_version: u32,
    pub payload_hash: [u8; 32],

    pub status: ReplayReceiptStatus,

    pub created_at_ns: u64,
    pub updated_at_ns: u64,

    pub response_schema_version: Option<u32>,
    pub response_bytes: Option<Vec<u8>>,

    pub effect: Option<ExternalEffectDescriptor>,
}
```

### Status

```rust
pub enum ReplayReceiptStatus {
    Reserved,

    ExternalEffectInFlight,

    Committed,

    TerminalFailed {
        error_code: ReplayTerminalErrorCode,
        error_bytes: Vec<u8>,
        error_bytes_truncated: bool,
    },

    RecoveryRequired {
        reason: RecoveryReason,
    },
}
```

`error_bytes` is capped by `MAX_REPLAY_TERMINAL_ERROR_BYTES = 4096`. Larger
errors are truncated and marked with `error_bytes_truncated = true`; unbounded
terminal error payloads must never be written to the receipt store.

### External Effect Descriptor

```rust
pub enum ExternalEffectDescriptor {
    ThresholdEcdsaSign {
        key_id_hash: [u8; 32],
        purpose: EcdsaPurpose,
        message_hash: [u8; 32],
    },

    ManagementCreateCanister {
        command_kind: CommandKind,
    },

    ManagementCall {
        canister: Principal,
        method: String,
    },

    IcpTransfer {
        operation_id: OperationId,
    },
}

pub enum EcdsaPurpose {
    DelegationProof,
    DelegatedToken,
    RoleAttestation,
    Other(CommandKind),
}
```

This descriptor is not meant to fully recover every effect. It is meant to
distinguish "safe to retry" from "an external effect may already have happened".
Store hashes or classified purpose fields for signing effects; do not store raw
proof, token, attestation, or application payloads unless the response cache
requires them.

Threshold ECDSA signing is an external cost boundary. Replay-protected signing
commands must mark `ThresholdEcdsaSign` in-flight before calling the signing
adapter. A duplicate request while signing is in-flight must return
`OperationInProgress` or `RecoveryRequired`; it must not call ECDSA again.

## Replay Algorithm

Every replay-protected command uses this flow:

1. Compute `command_kind`.
2. Read `operation_id` from request.
3. Resolve the authenticated replay actor from IC context and endpoint auth.
4. Normalize request payload.
5. Compute `payload_hash`.
6. Look up the replay receipt.
7. If a committed receipt exists with the same actor/hash, return cached
   response.
8. If a receipt exists with a different actor, reject.
9. If a receipt exists with a different payload hash, reject.
10. If a receipt is pending/in-flight, return `OperationInProgress` or
    `RecoveryRequired`.
11. For a new operation, run validation that does not perform expensive
    effects.
12. If the command is costed, check receipt quota, cost quota, and liquid-cycle
    reserve.
13. Atomically reserve the replay receipt and, for costed commands, the cost
    quota slot and in-flight cost reservation.
14. Execute command.
15. Commit response or terminal failure before returning.

Pseudo-code:

```rust
pub async fn run_replay_protected<C: ReplayCommand>(
    ctx: &mut Ctx,
    actor: ReplayActor,
    operation_id: OperationId,
    request: C::Request,
) -> Result<C::Response, C::Error> {
    let payload_hash = C::payload_hash(&actor, &request);

    match replay::lookup::<C>(ctx, &actor, operation_id, payload_hash)? {
        ReplayDecision::ReturnCommitted(bytes) => C::decode_response(bytes),
        ReplayDecision::Reject(error) => Err(error.into()),
        ReplayDecision::ExecuteNew(preflight) => {
            C::validate(ctx, &actor, &request)?;
            let permit = C::reserve_new_operation(ctx, preflight, &actor, &request)?;
            let replay_token = permit.replay_token();
            let response = C::execute(ctx, permit, request).await?;
            replay::commit::<C>(ctx, replay_token, &response)?;
            Ok(response)
        }
    }
}
```

`reserve_new_operation` is the non-awaiting state transition that records the
accepted operation. For non-costed commands, it reserves only the replay
receipt. For costed commands, it reserves the replay receipt, cost quota slot,
and in-flight cost reservation together before returning a permit. Costed
commands expose a `CostGuardPermit` from that permit to lower-level expensive
adapters.

For commands with external awaits, the command must mark the external effect
before making the call:

```rust
replay::mark_external_effect_in_flight(
    ctx,
    replay_token,
    ExternalEffectDescriptor::ManagementCreateCanister {
        command_kind: CommandKind::PoolCreateEmptyV1,
    },
)?;

let result = management_create_canister(...).await;
```

If a duplicate request arrives while the receipt is
`ExternalEffectInFlight`, Canic must not execute the effect again.

## Payload Hashing

Use a deterministic, versioned hash input.

```rust
pub const REPLAY_PAYLOAD_HASH_SCHEMA_VERSION: u32 = 1;

pub struct ReplayPayloadHashInput<T> {
    pub schema_version: u32,
    pub command_kind: CommandKind,
    pub actor: ReplayActor,
    pub normalized_request: T,
}
```

Hash with SHA-256 or the existing project-standard hash primitive.
`schema_version` must equal `REPLAY_PAYLOAD_HASH_SCHEMA_VERSION` for new
receipts, and the receipt stores the hash schema version beside the hash.
Changing the payload-hash input is a state migration, not a compatibility shim:
all committed receipts must remain findable under one active replay model before
a release may ship.

Rules:

- include command kind;
- include schema version;
- include the replay actor;
- include all user-controlled request inputs;
- exclude `operation_id`;
- exclude server-generated fields;
- exclude current time;
- exclude signatures/proofs/generated response bytes.

For delegation proof issuance, the payload hash must include:

- authenticated replay actor;
- requested subject/delegate identity;
- audience;
- scopes/capabilities;
- requested TTL/expiry policy;
- target/root context;
- delegation kind.

It must not include:

- `issued_at`;
- server signature;
- proof bytes;
- current time.

For pool `CreateEmpty`, the payload hash must include:

- authenticated replay actor;
- pool creation parameters;
- controller settings;
- initial cycles/configuration inputs;
- any role/admin inputs.

It must not include:

- created canister principal;
- registration timestamp;
- server-generated metadata.

## Response Caching

A replay-protected command commits the final response bytes.

For `CreateEmpty`, the committed response must include the created pool canister
principal.

For delegation proof issuance, the committed response must include the exact
issued proof material.

A replay of the same operation must return the committed response, not
recompute it.

This is especially important for delegation:

```text
First call:
  issued_at = T1
  signature = S1

Replay at T2:
  issued_at = T1
  signature = S1
```

Do not mint fresh proof material for a replay.

## Fix 1: Hard-Cut Delegated-Token Update Consumption

### Current Problem

Core delegated auth was originally designed as TTL-bounded bearer-token
verification. `nonce` was informational entropy only, and applications that
needed replay protection were expected to use separate replay/idempotency
state.

0.33 added verifier-side update-token consumption keyed by:

```text
(issuer_shard_pid, subject, cert_hash, nonce)
```

That drift made every authenticated update token effectively one-shot. It also
introduced auth-state storage that does not model a domain operation, payload
hash, actor-specific command, or committed response.

This is the wrong layer for system-command replay safety.

### New Behavior

Delegated-token verification must:

1. decode the first ingress argument as `DelegatedToken`;
2. validate delegated-token config is enabled;
3. verify root/shard trust, signature, cert hash, TTL, audience, and scope;
4. bind verified subject to the authenticated caller;
5. return the issuer shard principal for metrics.

It must not:

- consume update tokens;
- write verifier-local token-use state;
- reject a valid token solely because the same nonce was already used;
- treat token nonce as an application operation ID.

### Required Runtime Removal

The 0.61 implementation must remove:

- `consume_update_token_once` from `access/auth/token.rs`;
- the call to token consumption after `enforce_required_scope`;
- `AuthOps::consume_delegated_token_use`;
- `AuthStateOps::consume_delegated_token_use`;
- the `DelegatedTokenUse` ops view;
- `storage/stable/auth/token_uses.rs`;
- `DelegatedTokenUseConsumeResult`;
- `DelegatedTokenUseRecord`;
- `delegated_token_uses` from `AuthStateRecord`;
- `DELEGATED_TOKEN_USE_CAPACITY`;
- auth validation errors dedicated only to delegated-token replay store
  saturation/reuse;
- tests whose expected behavior is "second authenticated update with same token
  is rejected".

The replacement tests must assert:

- token verification still enforces signature, TTL, subject binding, audience,
  and required scope;
- queries and updates have the same bearer-token verification semantics;
- no auth-state token-use marker is written by endpoint authentication;
- system replay tests live with domain commands, not delegated auth.

### Stable State Cut

This is a hard cut, not a compatibility shim. The runtime should not retain a
dormant `delegated_token_uses` field or API.

Upgrade handling must verify the selected stable encoding behavior:

- if removing the field cleanly ignores historical data, document that the old
  consumed-token set is dropped during decode;
- if the encoding cannot ignore the removed field safely, add a one-time
  migration that drops it and writes the new auth-state shape.

Either way, the post-upgrade runtime model has no delegated-token use store.

### Documentation And Audit Updates

Update active docs and audit templates so they no longer describe delegated auth
as single-use:

- architecture authentication docs;
- recurring auth/replay audit definitions;
- access-purity/canonical-auth boundary audit commands that mention
  `consume_update_token_once`;
- 0.61 design/changelog notes.

Historical changelog and historical audit reports should remain historical
records; do not rewrite past release evidence unless a maintainer explicitly
requests it.

### Acceptance Criteria

The hard cut is complete when:

- `rg "consume_update_token_once|consume_delegated_token_use|DelegatedTokenUse|delegated_token_uses"`
  finds no live production path except historical docs, if those are left
  intentionally untouched;
- delegated-auth verifier tests prove bearer-token semantics;
- replay-sensitive system-command tests do not depend on auth-token reuse
  rejection;
- the architecture doc again matches runtime behavior: token replay is bounded
  by TTL unless a domain replay store applies.

## Fix 2: Pool `CreateEmpty`

### Current Problem

`PoolAdminCommand::CreateEmpty` has no request ID and dispatches directly to
pool creation. Repeating the same request can call management
`create_canister` again and register another pool canister.

### New API Shape

Change:

```rust
pub enum PoolAdminCommand {
    CreateEmpty,
}
```

to require replay metadata:

```rust
pub enum PoolAdminCommand {
    CreateEmpty(CreateEmptyPoolRequest),
}

pub struct CreateEmptyPoolRequest {
    pub replay: ReplayMeta,
    pub args: CreateEmptyPoolArgs,
}
```

### New Behavior

```text
same operation_id + same replay actor + same payload
  return the same created pool canister principal

same operation_id + same replay actor + different payload
  reject payload mismatch

same operation_id + different replay actor
  reject actor mismatch

pending/in-flight operation_id
  return OperationInProgress or RecoveryRequired

new operation_id
  create a new pool canister
```

### Execution Sequence

1. Follow the cost guard ordering for a replay-protected costed command.
2. Reserve deployment quota and cycle budget for the cost payer.
3. Mark `ManagementCreateCanister` in-flight.
4. Call management `create_canister` through a `CostGuardPermit`.
5. On success, register pool canister and commit replay response in the same
   callback.
6. Return response.

After successful `create_canister`, do as little fallible work as possible:

- write pool registry state;
- write replay committed response;
- return.

No logging, formatting, optional hooks, or extra validation may happen
between receiving the created canister ID and committing the receipt.

### Recovery Behavior

If the receipt is stuck in `ExternalEffectInFlight`, a replay must not call
`create_canister` again.

Return `OperationRecoveryRequired` or `OperationInProgress`, depending on age
and configured stale threshold.

0.61 does not need to solve automatic discovery of orphaned created canisters.
It must prevent repeated creation under the same operation ID.

### Other Pool Admin Commands

`PoolAdminCommand` variants other than `CreateEmpty` must be classified in the
endpoint manifest before 0.61 ships.

Required classifications:

- `Recycle`: replay-protected unless the implementation proves a fully
  response-idempotent ensure-style recycle transition.
- `ImportImmediate`: replay-protected unless import by pool principal is proven
  response-idempotent and returns an equivalent response on repeat.
- `ImportQueued`: replay-protected unless queue insertion is proven
  response-idempotent by stable queue identity.

Repeating an import or recycle request must not duplicate pool registry entries,
queue entries, recycle records, or management effects. The manifest must record
the chosen policy for each variant and the tests that prove it.

## Fix 3: Delegation-Proof Issuance

### Current Problem

`canic_request_delegation` signs directly. `DelegationProofIssueRequest` has no
replay metadata. Root uses `issued_at = now`, so a repeated request can mint
fresh proof material.

The request also carries `args.shard_pid`. Replay and cost guards do not
authorize a caller to request proof material for an arbitrary shard.
Authorization must bind the transport caller to the requested shard before any
root signing path is reached.

### New API Shape

Change:

```rust
pub struct DelegationProofIssueRequest {
    // existing fields
}
```

to:

```rust
pub struct DelegationProofIssueRequest {
    pub replay: ReplayMeta,
    pub args: DelegationProofIssueArgs,
}
```

### New Behavior

```text
msg_caller() != request.args.shard_pid
  reject DelegationCallerShardMismatch before replay execution, cost guards, or
  ECDSA

same operation_id + same replay actor + same proof request
  return the original proof response

same operation_id + changed proof request
  reject payload mismatch

same operation_id + changed replay actor
  reject actor mismatch
```

### Execution Sequence

1. Resolve `msg_caller()` and `request.args.shard_pid`.
2. Require `msg_caller() == request.args.shard_pid`.
3. Reject mismatch with `DelegationCallerShardMismatch` before replay
   execution, `CostGuardPermit`, or `ThresholdEcdsaSign`.
4. Follow the cost guard ordering for a replay-protected costed command.
5. Reserve signing quota and cycle budget for the cost payer.
6. Compute `issued_at = now`.
7. Mark `ThresholdEcdsaSign` in-flight.
8. Sign proof through a `CostGuardPermit`.
9. Commit full proof response.
10. Return proof response.

Delegation proof issuance must not rely on broad delegated-token update
consumption for replay safety. Replay safety comes from the operation receipt.
Cost guards must not be treated as authorization: only the requested shard, or
an explicitly documented root self-call path, may request proof material for
that shard.

### Authorization Tests

Required tests:

## Fix 4: Delegated-Token Minting

### Current Problem

Canic-provided delegated-token minting can chain root delegation proof issuance
and shard token signing. Fleet wrappers such as `*_issue_token` can expose this
as ordinary update endpoints. If minting is not classified and guarded, a retry
or burst of fresh operation IDs can trigger root ECDSA and shard ECDSA outside
the 0.61 replay model.

### Required Policy

Every Canic-provided endpoint that mints delegated tokens must choose one policy
and record it in the endpoint manifest:

1. **Replay-protected mint**

   One `operation_id` covers the combined mint operation, including any root
   proof request and the final shard token signature. The committed response is
   the final `DelegatedToken` bytes. A retry with the same replay actor and
   payload returns the exact committed token and must not call root ECDSA or
   shard ECDSA again.

2. **Response-idempotent session reuse**

   For the same subject, audience, scope, issuer shard, and still-valid session
   policy, return existing still-valid material instead of minting a fresh
   token. If new signing is required, the request must move to replay-protected
   minting before ECDSA is called.

In both policies, new signing work must pass signing quota, cycle reserve, and
`CostGuardPermit` requirements before root or shard ECDSA. Fleet-specific
`*_issue_token` wrappers are not exempt; they must either delegate to the
Canic-provided replay/cost-guarded mint path or be listed in the manifest with
an equivalent policy.

### Execution Sequence For Replay-Protected Mint

1. Read `ReplayMeta` from the mint request.
2. Resolve replay actor, quota subject, issuer shard, and cost payer.
3. Normalize subject, audience, scopes, issuer shard, TTL, and root context into
   the payload hash.
4. Return committed token bytes on matching committed replay.
5. For a new operation, reserve the replay receipt, signing quota, and cycle
   budget for the combined root-proof and shard-token signing work.
6. Mark `ThresholdEcdsaSign` in-flight for root proof signing before root ECDSA
   if a fresh proof is needed.
7. Mark `ThresholdEcdsaSign` in-flight for shard token signing before shard
   ECDSA.
8. Commit the final `DelegatedToken` response.

The delegated-token nonce is never an operation ID and never substitutes for
the replay receipt.

### Mint Logging

Delegated-token mint and issue paths must use the existing Canic runtime log
system with `crate::log!`.

Use `Topic::Auth` for auth-level replay and signing progress. Required log
points:

- `Info`: fresh replay receipt reserved for `auth.mint_token.v1` or
  `auth.issue_token.v1`.
- `Info`: committed replay receipt returned without re-entering root proof or
  shard token signing.
- `Warn`: actor mismatch, payload mismatch, expired receipt, in-progress
  receipt, recovery-required receipt, or terminal-failed receipt blocks a
  token mint/issue request.
- `Info`: signing cost guard reserved before shard ECDSA.
- `Info`: shard token `ThresholdEcdsaSign` effect marked in flight before ECDSA.
- `Error`: post-effect uncertain signing failure marks recovery required.
- `Ok`: final delegated-token response committed into the replay receipt.

Each log message should include command kind, operation ID, replay actor or
caller, and transition name. It must not log token bytes, proof bytes,
signature bytes, or raw receipt response bytes.

### Mint Tests

Required tests:

- replay-protected mint first call may call fake root ECDSA and fake shard ECDSA;
- replay-protected mint retry with the same `operation_id`, actor, and payload
  returns the committed token and does not call fake root ECDSA or fake shard
  ECDSA again;
- same `operation_id` with changed mint payload rejects payload mismatch;
- same `operation_id` with changed replay actor rejects actor mismatch;
- signing quota exhaustion rejects before root or shard fake ECDSA;
- response-idempotent session reuse returns still-valid material without fresh
  ECDSA;
- every fleet-exposed token mint wrapper appears in the endpoint manifest.

## Fix 5: Set-Style App Commands

### Current Problem

Some app set commands are side-effect idempotent but not response-idempotent.

Example: `SetStatus(active)` when already active does not mutate again, but

### New Behavior

Set commands must return success even when the requested state is already
true.

Suggested response:

```rust
pub struct SetStateResponse<T> {

    pub current: T,
    pub changed: bool,

```

This is a public Candid/API shape change for endpoints that previously returned

Examples:

```text

current = running
changed = true

previous = running
current = running
changed = false
```

Apply this to:

- `SetStatus`;
- `SetCyclesFundingEnabled`;
- any equivalent set-style command found during the 0.61 endpoint sweep.

## Fix 6: ICP Refill Replay Core Integration

### Current Problem

ICP refill already carries an `operation_id` and has a dedicated stable store.
Leaving it as a physically separate replay island would preserve two replay
semantics in a release whose invariant is one replay model.

### New Behavior

ICP refill must migrate to the shared `ReplayReceipt` store and shared replay
errors with refill-specific `command_kind` values. The refill domain may keep
its refill records for business state, transfer metadata, accounting, and
recovery details, but replay identity, actor binding, payload hash mismatch,
pending/in-flight state, committed response lookup, and terminal failure
tombstones are owned by the shared replay core.

### Migration Requirements

- existing refill `operation_id` values map to `OperationId([u8; 32])` through
  a documented one-time migration;
- pending refill operations become pending or recovery-required shared replay
  receipts;
- committed refill operations remain non-repeatable under the same
  `operation_id`;
- refill tests continue to prove transfer/notify behavior while using shared
  replay decisions and errors.

### Logging Requirements

ICP refill replay integration must use the existing Canic runtime log system
with `crate::log!`.

Use `Topic::Cycles` for workflow-level ICP refill replay and refill progress
events. Lower-level ICRC ledger transport may use `Topic::Icrc` only when the
log is emitted from ledger-specific code. Do not introduce a new log topic for
this slice unless the runtime log topic enum is intentionally expanded.

Required workflow log points:

- `Info`: fresh shared replay receipt reserved for `icp.refill.v1`.
- `Info`: committed replay receipt returned without re-entering transfer or
  notify execution.
- `Warn`: actor mismatch, payload mismatch, expired receipt, in-progress
  receipt, recovery-required receipt, or terminal-failed receipt blocks a
  refill request.
- `Info`: resumable refill business record causes the temporary shared receipt
  to be aborted so legacy retry/recovery continues.
- `Info`: ledger transfer external effect is marked in flight before the
  ledger call.
- `Info`: CMC `notify_top_up` external effect is marked in flight before the
  CMC call.
- `Ok`: terminal refill response is committed into the shared replay receipt.
- `Warn`: retryable transfer or notify outcome leaves the refill record
  resumable.
- `Error`: uncertain post-effect failure is preserved as a recovery-required
  shared replay receipt.

Each log message should include enough non-sensitive correlation data for
operator recovery:

- `command_kind`;
- `operation_id` as a stable hex string, or a documented short prefix if full
  IDs are too noisy;
- replay decision or state transition;
- refill record id when available;
- source canister, target canister, and amount;
- external effect kind: ledger transfer or CMC notify;
- final refill status for commits or terminal failures.

Logs must not include raw subaccount bytes, full payload hashes, Candid blobs,
receipt response bytes, or ledger/CMC error payloads beyond bounded summarized
messages already stored in the refill record.

## Storage And Pruning

Replay receipts are durable state.

Pruning must not re-enable duplicate durable effects.

### Pool Create Receipts

Pool creation receipts must be retained indefinitely, or compacted only into a
tombstone that still prevents operation ID reuse.

Minimum retained data:

- `command_kind`;
- `operation_id`;
- `actor`;
- `payload_hash`;
- `created_pool_canister_principal`;
- `status = committed`.

### Delegation Proof Receipts

Delegation proof responses may expire. After expiry, Canic may compact the full
proof response into a tombstone.

However, replaying the same operation ID after compaction must not mint a fresh
proof.

Allowed behavior after compaction:

- return expired committed proof if retained; or
- return `OperationAlreadyCommittedResponseExpired`.

Not allowed:

- issue a new proof with a new `issued_at` or signature.

### Expensive Terminal Failure Tombstones

If an operation reaches an expensive external-effect boundary and then fails
without a committed success response, a tombstone is mandatory. Canic may
compact the full failure payload, but it must retain the tombstone state.

Minimum retained data:

- `command_kind`;
- `operation_id`;
- `actor`;
- `payload_hash`;
- `external_effect_descriptor`;
- terminal failure class.

Replaying that operation ID must not repeat the expensive external effect.
Cheap preflight failures may remain ordinary errors without tombstones. A
failure after threshold ECDSA dispatch, management-canister dispatch, ICP
transfer dispatch, or durable publication dispatch is part of the replay and
cost-abuse boundary and must leave committed failure state or a tombstone.

### Pending Receipts

Pending or in-flight receipts must have deterministic stale behavior.

```text
fresh pending
  OperationInProgress

stale pending before external effect
  may be cleared or retried

stale external-effect-in-flight
  RecoveryRequired
```

Do not retry an external-effect-in-flight operation automatically.

## Quotas And Abuse Protection

Replay protection is not a complete abuse boundary. It prevents duplicate
durable effects for the same operation, but a caller may still request many new
operations. 0.61 therefore treats publicly reachable signing and deployment
paths as costed operations.

Costed operation classes:

- threshold-ECDSA signing, including delegation proof issuance, delegated-token
  minting, and role-attestation signing;
- management-canister deployment effects, including `create_canister`,
  install, upgrade, and project/pool/child-canister creation workflows;
- value-transfer or top-up effects that already use operation IDs but still
  spend canister resources while preparing or dispatching work.

Every public update endpoint that can reach a costed operation must have:

- a replay-policy classification;
- a cost class, or an explicit statement that it reaches no costed operation;
- a quota policy for new operations;
- a liquid-cycle reserve check before the expensive effect.

Equivalent retries with committed receipts return the committed response
without repeating the expensive effect. New operations must pass cost guards
before signing, deploying, transferring, or publishing durable state.

### Cost Guard Ordering

Cost guards are evaluated only for new operations.

If a replay receipt is already committed and the actor/payload hash match,
Canic returns the committed response without consuming quota and without
checking current cycle reserve.

For a replay-protected costed command:

1. Parse command kind and `operation_id`.
2. Resolve authenticated replay actor, quota subject, and cost payer.
3. Normalize payload and compute `payload_hash`.
4. Look up the replay receipt.
5. If committed with the same actor/hash, return cached response.
6. If pending/in-flight with the same actor/hash, return `OperationInProgress`
   or `RecoveryRequired`.
7. If actor/hash mismatch, reject.
8. For a new operation, run validation that does not perform expensive effects.
9. Check receipt quota.
10. Check cost quota.
11. Check liquid-cycle reserve using balance minus outstanding reservations.
12. Atomically reserve:
    - replay receipt;
    - cost quota slot;
    - in-flight cost reservation.
13. Mark the costed external effect before the await.
14. Call threshold ECDSA, the management canister, ICP transfer, or durable
    publication code.
15. Commit response or terminal failure.
16. Release or finalize the in-flight cost reservation.

The accepted operation, quota slot, and cost reservation must be recorded in the
same non-awaiting state transition before the first expensive external await. A
check without an in-flight reservation is not sufficient.

Steps 8 through 12 execute inside one update message before the first `await`,
so they observe a consistent local state view under the IC execution model. The
atomicity requirement is about durably recording the replay receipt and cost
reservation before the later external await, not about protecting against local
state interleaving inside the same pre-await block.

Every costed command must have an explicit cost barrier in code. Validation
above the barrier is cheap preflight. Any failure after crossing the barrier
must leave committed failure state or a tombstone. Prefer a typed transition
over a comment-only marker:

```rust
let barrier = cost_guard::cross_cost_barrier(ctx, preflight)?;
```

No threshold-ECDSA, management-deployment, ICP transfer, or durable-publication
adapter may be called before that transition.

### Cost Guard Permits

Do not rely on each call site remembering to run quota and cycle checks. The
cost guard returns a permit that lower-level expensive adapters require.

```rust
pub struct CostGuardPermit {
    pub cost_class: CostClass,
    pub quota_key: CostQuotaKey,
    pub reservation_id: CostReservationId,
    pub payer: CostPayer,
    _private: (),
}
```

Expensive adapters must require the permit:

```rust
ecdsa_sign_with_permit(permit, request).await
management_create_canister_with_permit(permit, request).await
```

No threshold-ECDSA or management-deployment adapter may be called from a public
update path without a `CostGuardPermit`.

`CostGuardPermit` must be unforgeable outside the cost-guard module. It must
use private construction, and tests that need a permit must obtain one through
the same guard API or through a dedicated test-only fixture that enforces the
same state transitions.

### Receipt Quotas

Because operation IDs create durable receipts, add simple receipt quotas.

Suggested initial limits:

- max pending replay receipts per replay actor;
- max pending replay receipts per command kind;
- max operation ID length;
- max retained delegation proof response size.

Reject new replay-protected operations when quota is exceeded.

Committed pool-create receipts must not be pruned merely for quota relief.

### Signing And Deployment Quotas

Receipt quotas do not limit an attacker that continually creates new operation
IDs. Add operation quotas for expensive classes before calling threshold ECDSA
or the management canister.

Suggested initial limits:

- max delegation-proof signing attempts per quota subject and command kind per
  window;
- max delegated-token signing attempts per subject/session and issuer shard per
  window;
- max role-attestation signing attempts per quota subject and target role per
  window;
- max child-canister creation attempts per quota subject and command kind per
  window;
- max in-flight deployment operations per quota subject and command kind.

Principal-only quotas are acceptable as the Canic-core default, but they are not
strong enough for product surfaces where principals are cheap to rotate. Those
surfaces must bind quota identity to a durable user, session, account,
invite, paid state, or application-defined verification policy before exposing
canister-funded signing or deployment.

Cost quotas use deterministic bucketed windows with injected time in tests.

Each costed operation records:

- accepted attempts;
- in-flight attempts;
- completed successes;
- terminal failures after an expensive effect was attempted.

Validation failures before cost reservation do not consume signing/deployment
quota unless explicitly configured. Failures after threshold ECDSA or
management-canister dispatch consume an attempt quota and must leave either a
committed failure receipt or a tombstone that prevents repeating the same
expensive effect with the same `operation_id`.

### Cycle-Reserve Circuit Breakers

Before a costed operation, check the local liquid-cycle balance against a
configured reserve for that cost class. If the balance is below reserve, return
a typed cost-guard error and do not call threshold ECDSA or the management
canister.

Minimum reserve checks:

- root delegation-proof signing;
- shard delegated-token signing;
- role-attestation signing;
- management `create_canister`;
- install/upgrade calls that attach cycles or dispatch large Wasm payloads;
- project/pool/child-canister creation workflows before the first external
  deployment effect.

The reserve check is not a substitute for replay receipts. For replay-protected
commands with external awaits, still reserve the operation and mark the
external effect in-flight before making the management call. A duplicate request
must never bypass the receipt state and repeat the external effect.

Cycle reserve is a durable in-flight reservation, not only a point-in-time
balance check.

```text
available_liquid_cycles =
  current_canister_cycles - outstanding_cost_reservations
```

A new costed operation may proceed only if:

```text
available_liquid_cycles >= min_reserve_after_operation + estimated_cost_floor
```

When accepted, Canic records an in-flight cost reservation before the external
await. The reservation is released or finalized when the operation commits,
terminally fails, or enters recovery state. This prevents many interleaved
updates from each passing the same balance check and collectively draining the
cost payer.

### Token Reuse

Removing update-token consumption means application wrappers must not mint a
fresh delegated token merely because an update call is about to run. Clients
must reuse a valid token until its TTL, audience, subject binding, or scope no
longer satisfies the endpoint.

Any Canic-provided endpoint that mints delegated tokens or session proof
material must be one of:

1. response-idempotent session reuse: return existing still-valid material for
   the same subject, audience, and scope; or
2. replay-protected issuance: require `operation_id` and return the committed
   minted material on retry.

In both cases, new signing work must pass signing quota and cycle-reserve
guards before threshold ECDSA is called. Token refresh or reissue code must not
use the delegated-token nonce as a domain operation ID.

## Upgrade And Hard-Cut Boundary

Replay receipts must survive canister upgrades.

Add schema versioning:

```rust
pub const REPLAY_RECEIPT_SCHEMA_VERSION: u32 = 1;
```

Upgrade requirements:

- committed receipts remain readable after upgrade;
- pending receipts remain pending/recovery-required after upgrade;
- unsupported receipt schema returns `ReceiptSchemaUnsupported`, not a panic.

Operations completed before 0.61 are not retroactively replay-protected. 0.61
does not backfill historical operation receipts and does not claim replay safety
for pre-0.61 retries. Operators must treat pre-0.61 completed operations as
historical behavior outside the new invariant.

The delegated-token update-consumption hard cut must also have upgrade
coverage. Old verifier-local token-use markers are not replay receipts and must
not be migrated into the replay receipt store.

There is no compatibility shim for old delegated-token consumption state, old
request DTOs without required operation IDs, or legacy replay implementations.
The post-upgrade runtime has one replay model.

## CLI Behavior

For replay-protected commands, CLI must include an operation ID.

Supported interface:

```text
--operation-id <id>
```

If omitted, the CLI may generate one client-side before sending the request.

When the CLI generates one, it must display it in verbose/error output:

```text
operation_id: 6f4b0b7e-0c0e-49dd-b6b7-5fb4b9d34654
```

For high-value operations such as pool creation, the CLI must persist the
generated operation ID in a local pending operation log before sending the
request. Write-before-send is required so an operator can retry safely after a
CLI crash or machine restart.

Suggested path:

```text
<canic-cache-root>/operations/pending.json
```

Server correctness must not depend on the CLI cache, but operator retry
diagnostics do. A high-value command must not send the request until its pending
operation entry is durable locally.

## Hard-Cut Client Boundary

Replay-sensitive request DTOs change in 0.61. There is no compatibility adapter
for old request shapes.

Set-style app command responses also change in 0.61: endpoints that previously
returned `Result<(), Error>` now return `SetStateResponse` or a documented
equivalent with `changed`. Old clients that assume the unit response shape are
outside the hard-cut boundary.

Clients that call replay-sensitive commands without operation IDs must receive:

```text
OperationIdRequired
```

rather than a generated server-side operation ID or a legacy execution path.

Server-side generated operation IDs do not solve replay safety because a retry
would get a different ID, so they are not part of the 0.61 public API.

The delegated-token update-consumption hard cut intentionally changes
authenticated update behavior back to bearer-token semantics. Tokens remain
bounded by TTL, audience, subject binding, and scope. Domain replay protection
is owned by domain commands.

Clients that minted a fresh delegated token for every authenticated update
must switch to reusing a valid token until expiry or scope/audience mismatch.
Fresh token minting remains available when required, but new quota and
cycle-reserve guards may reject abusive or underfunded issuance bursts.

## Acceptance Criteria

0.61 is complete when:

- every public update endpoint has an explicit replay policy;
- root capability RPC replay uses the shared replay receipt primitive, with no
  legacy replay implementation left live;
- replay-sensitive DTOs use 32-byte operation IDs;
- replay payload hashes store and enforce `REPLAY_PAYLOAD_HASH_SCHEMA_VERSION`;
- terminal failure error payloads are bounded and truncation is recorded;
- delegated-token update consumption is fully removed from live runtime code;
- old delegated-token use markers are dropped or migrated away and are not
  accepted as replay state;
- old request DTOs without required operation IDs fail with
  `OperationIdRequired`;
- `canic_request_delegation` rejects `msg_caller() != request.args.shard_pid`
  with `DelegationCallerShardMismatch` before any replay reservation, cost
  guard, `ThresholdEcdsaSign`, or ECDSA call;
- every Canic-provided and fleet-exposed delegated-token mint endpoint is in
  the endpoint manifest with `ReplayProtected(auth.mint_token.v1)` or
  documented response-idempotent session reuse;
- delegated-token mint replay does not double-call root or shard ECDSA;
- pool `CreateEmpty` cannot create a second pool canister for the same
  `operation_id`;
- `canic_pool_admin` subcommands other than `CreateEmpty` have explicit replay
  class and tests in the manifest;
- delegation proof issuance cannot mint fresh proof material for the same
  `operation_id`;
- `canic_app` set commands return `SetStateResponse` or documented equivalent,
  and repeated calls return `changed = false` without error;
- ICP refill uses shared replay receipts and shared replay errors, not a
  separate replay semantics island;
- every public update endpoint that can reach signing, deployment, transfer, or
  publication has a cost class or an explicit no-costed-operation statement;
- replay-protected costed commands follow the required ordering: committed
  replay lookup before current quota/reserve checks, and new-operation quota
  plus cycle reservation before expensive awaits;
- replay actor, quota subject, and cost payer are explicit for costed commands;
- threshold ECDSA signing is represented as an `ExternalEffectDescriptor`;
- public signing and management-deployment adapters require `CostGuardPermit`;
- `CostGuardPermit` cannot be constructed outside the cost-guard module or its
  dedicated test fixture;
- signing and deployment quotas reject new abusive operations before ECDSA or
  management-canister effects are called;
- cycle-reserve guards reject underfunded signing and deployment paths before
  ECDSA or management-canister effects are called;
- post-dispatch expensive failures leave committed failure state or tombstones
  that prevent repeating the same expensive effect with the same `operation_id`;
- authenticated update wrappers reuse valid delegated tokens instead of minting
  a fresh token solely because an update call is about to run;
- high-value CLI operations persist generated operation IDs before sending;
- delegated auth docs describe TTL-bounded bearer-token semantics;
- existing root capability RPC replay tests still pass;
- ICP refill `operation_id` behavior still passes;
- upgrade tests prove committed receipts survive upgrade;
- changelogs document the replay-safety invariant.

## Release Blockers

The two Medium audit findings are release blockers for 0.61:

- `PoolAdminCommand::CreateEmpty` replay safety;
- delegation-proof issuance replay safety.

Delegation proof authorization is release-blocking: root must reject foreign
`shard_pid` proof requests before replay execution or ECDSA.

Delegated-token mint coverage is release-blocking because minting can reach root
and shard ECDSA and is an ordinary application-auth path after the hard cut.

The delegated-auth hard cut is also release-blocking because retaining the old
auth-layer consumption path would keep the replay model ambiguous.

Cost-abuse guardrails for public signing and deployment paths are
release-blocking because 0.61 removes the auth-layer token-consumption behavior
that previously discouraged token reuse and unintentionally throttled some
update flows.

No public 0.61 release candidate may be cut until all replay blockers and all
cost-abuse blockers are implemented and passing.
No slice that removes delegated-token update consumption may be merged or
cherry-picked into a released branch without the full 0.61 replay, cost-guard,
and hard-cut client boundary.

The Low set-command response-idempotence issue must ship in the same release
because it uses the same policy framework and prevents future ambiguity.

## Design Rule Going Forward

No new externally callable mutation may be added to Canic unless the PR states
one of:

- `Replay policy: Response-idempotent`;
- `Replay policy: Replay-protected with operation_id`;
- `Replay policy: Intentionally non-idempotent, with reason`.

If the command can reach a costed operation, the PR must also state one of:

- `Cost class: <class>, with quota and cycle reserve`;
- `Cost class: none, with reason`.

A command that creates canisters, transfers value, signs proof material,
publishes durable artifacts, mutates authority, or registers infrastructure
must default to:

```text
Replay-protected with operation_id
```

unless there is a written proof that it is response-idempotent.
