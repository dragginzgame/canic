# 0.59 Design: Instruction Accounting And Cost Estimates

## Status

- Status: Tentative
- Line: `0.59.x`
- Theme: instruction/cycle terminology hard cut plus optional subnet-aware
  execution-cost estimates

0.59 should fix the unit boundary before Canic grows any more profiling or
cost-reporting surface.

Canic's runtime perf counters measure WebAssembly instructions. They do not
measure cycles. Any cycle number derived from them is an estimate, not an
accounting statement and not a bill reconciliation tool.

## Problem

The Canic instruction-audit lane is mostly named correctly today:

- `canic-core::perf` stores `total_instructions`;
- `QueryPerfSample` returns `local_instructions`;
- generated reports use `total_local_instructions` and
  `avg_local_instructions`;
- the recurring audit definition explicitly says it is not a cycle-cost audit.

The risk is that downstream wording, future JSON fields, or convenience
helpers can still collapse two different concepts:

- instruction counters from `performance_counter` / `countInstructions`;
- cycles charged by the protocol for execution, messages, bytes, management
  operations, storage, reservations, and other resource use.

That collapse is dangerous because a clean instruction regression win can still
leave a user's cycle bill unchanged or even higher if message or platform fees
dominate the operation.

## Core Rule

The measured unit is `instructions`.

The only acceptable names for measured values are instruction names:

```text
local_instructions
total_local_instructions
avg_local_instructions
checkpoint_delta_instructions
instruction_count
```

Do not call these values:

```text
cycles
cycle_cost
cycle_delta
burn
charged_cycles
```

Cycle language is reserved for real cycle balances, transfers, top-ups, CMC
conversion, and explicit estimates derived from instruction counts.

## Counter Semantics

0.59 must document which counter each path uses.

Current Canic runtime perf accounting uses `performance_counter(1)`:

- local canister WebAssembly instructions;
- current call context;
- preserved across `await` points in that call context;
- excludes instructions executed by other canisters;
- not a direct measure of cycles charged.

Counter scope still depends on message kind:

- replicated update execution excludes nested IC calls, including calls to the
  same canister;
- non-replicated composite-query execution can accumulate a broader same-query
  call/callback scope and may include nested composite-query work depending on
  the query shape.

Reports must preserve `sample_origin` and must not compare replicated update
samples, ordinary query samples, and composite-query samples as if they had
identical counter scope.

Motoko `countInstructions` and `performance_counter(0)` are different:

- synchronous current-message instruction counter;
- reset at shared function entry and at `await`;
- useful for local slice profiling;
- not complete endpoint accounting across async resumptions.

The design consequence is simple: Canic should not import a
`countInstructions` mental model into endpoint accounting, and it should not
rename current endpoint accounting to cycles.

## In Scope

- Audit and correct confusing instruction/cycle wording in docs, reports,
  tests, logs, help text, and JSON field names.
- Preserve the instruction-footprint audit as the authoritative measured lane.
- Add an optional execution-cycle estimate layer only where it is explicitly
  requested by a report flag.
- Make every estimate carry its input assumptions: subnet node count, selected
  rate, formula version, and omitted costs.
- Provide an offline explicit-subnet path, a local NNS subnet catalog, and a
  live NNS registry refresh path for subnet sizing.

## Out Of Scope

- Exact bill reconstruction.
- Per-message, byte, storage, reservation, GC, and management-call accounting.
- Replacing `canic info cycles`, `canic cycles`, or the 0.58 ICP refill work.
- Runtime canisters calling the NNS registry just to decorate perf rows.
- Treating query samples as charged update execution.
- Renaming real cycle DTOs used for balances, transfers, top-ups, or funding.

## Terminology Hard Cut

0.59 should start with a terminology pass over these surfaces:

- `crates/canic-core/src/perf.rs`
- `crates/canic-core/src/dto/metrics.rs`
- `crates/canic/src/macros/mod.rs`
- `crates/canic/src/instructions/mod.rs`
- `crates/canic-tests/tests/instruction_audit_support/`
- `docs/audits/recurring/system/instruction-footprint.md`
- generated instruction-footprint reports and artifacts
- CLI/help text if any profiling command or report wrapper is added

Expected outcome:

- no measured instruction field is called cycles;
- no report says cycle cost when it only has instruction counters;
- logs keep the `i` suffix or spell out `instructions`;
- test names and fixtures use instruction wording unless they exercise real
  cycle funding behavior.

Historical changelog entries may remain historical unless they actively mislead
current users. Active docs and generated output matter more than archive churn.

## Report Shape

Instruction rows remain the canonical row model:

```text
subject_kind
subject_label
count
total_local_instructions
avg_local_instructions
scenario_key
scenario_labels
principal_scope
sample_origin
```

If estimates are enabled, they should be a sibling object, not a replacement
for instruction fields:

```text
execution_cycle_estimate: {
  "kind": "instruction_component_only",
  "charge_model": "hypothetical_update_execution_component",

  "local_instructions": 1000000,
  "counter_id": 1,
  "counter_name": "call_context_instruction_counter",
  "counter_scope": "current_call_context",
  "sample_origin": "update",

  "estimated_instruction_cycles": "1000000",
  "execution_base_cycles": null,
  "estimated_execution_cycles": null,

  "cycles_per_billion_instructions": "1000000000",
  "subnet_node_count": 13,
  "subnet_principal": null,
  "subnet_type": "application",
  "geographic_scope": "global",
  "subnet_source": "flag",
  "charges_apply_to_subject": true,
  "charge_applicability_reason": "user_canister_on_charged_subnet",

  "rate_source": "official-ic-cycle-costs-docs",
  "formula_version": "ic-cycle-costs-2025-12-18",
  "rounding": "ceil(local_instructions * rate / 1_000_000_000)",
  "catalog_schema_version": null,
  "catalog_stale": null,
  "registry_canister_id": null,
  "registry_version": null,
  "resolver_backend": null,

  "omitted_costs": [
    "update_message_execution_base_fee",
    "message_transmission_base_fee",
    "payload_bytes",
    "garbage_collection",
    "callee_instructions",
    "management_call_fees",
    "storage_and_reservations"
  ]
}
```

Use integer/rational math only. Prefer `cycles_per_billion_instructions` over
floating-point `cycles_per_instruction`.

Do not use `estimated_cycles` as a field name. It is too easy to read as total
charged cycles. The default derived value is
`estimated_instruction_cycles`.

`subnet_type` values are:

```text
application
fiduciary
system
unknown
```

`geographic_scope` values are:

```text
global
europe
unknown
```

`charge_model` values are:

```text
hypothetical_update_execution_component
not_charged_query_sample
not_applicable
```

`charge_applicability_reason` values are:

```text
user_canister_on_charged_subnet
system_canister_no_cycle_charges
explicit_rate_override
unknown_subject
```

## Estimate Model

The estimate layer answers one narrow question:

```text
Given this many measured local instructions, what is the approximate execution
cycle component under a stated subnet-size/rate assumption?
```

It does not answer:

```text
What did this call cost on the user's bill?
```

Formula:

```text
estimated_instruction_cycles =
  ceil(local_instructions * cycles_per_billion_instructions / 1_000_000_000)
```

If an execution base fee is included, it must be a separate field:

```text
execution_base_cycles
estimated_instruction_cycles
estimated_execution_cycles
```

Do not hide the base fee inside the per-instruction estimate.

If the update execution base fee is omitted, `estimated_execution_cycles` must
be absent or `null`; do not populate it with the instruction component alone.

If the update execution base fee is included, use:

```text
kind = "update_execution_base_plus_instruction"
execution_base_cycles = "5000000"
estimated_instruction_cycles = "1000000"
estimated_execution_cycles = "6000000"
```

for 1,000,000 local instructions on a 13-node application subnet under the IC
cost formula version cited above.

Query and composite-query samples are different. Current IC cost docs list
query calls as free, while update execution has a base fee and per-instruction
fee. Therefore:

- query and composite-query estimates are omitted by default;
- if shown, their `charge_model` must be
  `hypothetical_update_execution_component` or
  `not_charged_query_sample`;
- they must never be presented as charged query costs.

## Subnet Inputs

Cycle estimates must use one of these sources.

`--estimate-execution-cycles` must fail unless one of these estimate sources is
explicitly provided or inferred from a deployed IC canister through a local NNS
subnet catalog. Canic must not silently assume a 13-node subnet.

### Explicit Node Count

The offline and deterministic path:

```text
--subnet-node-count <n>
```

The report records:

```text
subnet_source = "flag"
subnet_node_count = <n>
```

This is the preferred path for CI and local recurring audits because it avoids
network dependence.

### Explicit Rate

The escape hatch for protocol-cost changes or custom test networks:

```text
--cycles-per-billion-instructions <cycles>
```

The report records:

```text
subnet_source = "explicit-rate"
cycles_per_billion_instructions = <cycles>
```

When both node count and explicit rate are provided, explicit rate wins and the
node count is retained only as metadata.

### Local NNS Subnet Catalog

The normal inferred path for deployed mainnet canisters:

```text
deployment target -> canister principal -> cached routing range -> IC subnet
principal -> cached node count/type -> rate table
```

Canic should maintain a host-side local subnet catalog fetched from the NNS
registry. Subnet topology changes rarely, so report generation should not need
a live NNS call on every run.

Catalog shape:

```text
catalog_schema_version
network = "ic"
registry_canister_id
registry_version
fetched_at
fetched_by
source_endpoint
resolver_backend
subnets[] = {
  subnet_principal
  subnet_type
  geographic_scope
  subnet_label
  node_count
  charges_apply_by_default
}
routing_ranges[] = {
  start_canister_id
  end_canister_id
  subnet_principal
}
```

The catalog belongs under host/operator state, not canister runtime state. It
should be safe to refresh independently of any deployment mutation.

Catalog records must include `catalog_schema_version`. Canic must reject
unknown future catalog schema versions unless a migration exists, rather than
silently reading a cache whose routing or subnet-record shape may have changed.

The catalog models subnet billing separately from geography:

```text
subnet_type = application | fiduciary | system | unknown
geographic_scope = global | europe | unknown
```

European subnets are application-style charged subnets with
`geographic_scope = europe`, not a separate billing type. This preserves their
13-node application pricing behavior while still exposing the data-residency
property operators care about.

Default cache freshness:

```text
default_stale_after = 7d
```

Commands may override this with `--stale-after <duration>`.
Estimate-producing commands must treat stale catalogs as unavailable unless
the caller explicitly allows stale data. If stale data is used, the estimate
records `catalog_stale = true`.

Stale-cache behavior:

- instruction audits without estimates never fail because of a missing or stale
  subnet catalog;
- instruction audits with estimates and no explicit rate use the catalog only
  when it is fresh and maps the target canister;
- stale catalogs omit estimates by default and explain
  `canic subnet network refresh`;
- `--allow-stale-subnet-catalog` permits estimates from stale cached data and
  records `catalog_stale = true`;
- `--refresh-subnet-catalog` refreshes before resolving;
- a future `--require-estimates` flag may fail instead of omitting estimates,
  but 0.59 should not make missing catalog data a hard failure by default;
- `canic subnet network list` shows stale status and still displays cached
  data;
- `canic subnet network info` shows stale status and warns when pricing
  metadata may be stale.

When this source is used, reports record:

```text
subnet_source = "nns-registry-cache"
registry_canister_id = <nns-registry-principal>
registry_version = <version>
subnet_principal = <resolved-subnet-principal>
subnet_node_count = <cached-node-count>
subnet_type = <cached-subnet-type>
geographic_scope = <cached-geographic-scope>
catalog_schema_version = <version>
catalog_stale = false
resolver_backend = "local-nns-subnet-catalog"
```

If the catalog is stale, missing, or cannot map the canister ID to a subnet,
the estimate should be omitted unless the caller supplies an explicit fallback
or asks Canic to refresh the catalog. A future strict flag may require
estimates and fail instead of omitting them, but omission is the safe default.

The same catalog should be operator-visible. It is useful beyond estimates and
avoids forcing operators to inspect NNS websites or hand-query registry values
just to answer where a canister lives and what that implies for pricing.

### NNS Registry Resolver

The live path is primarily a catalog refresh/fallback path, not the default
per-report dependency:

```text
--ic-subnet <subnet-principal>
--resolve-ic-subnet-for-canister <canister-principal>
--refresh-subnet-catalog
```

The resolver belongs in host/CLI or test-report code, not canister runtime
workflow. It should:

1. query the NNS registry canister;
2. fetch the current routing table and subnet records;
3. read subnet membership;
4. count active nodes;
5. read subnet type when available;
6. write/update the local NNS subnet catalog;
7. select the corresponding execution-rate formula/table;
8. record the registry canister ID, registry version, subnet principal, subnet
   node count, subnet type, and resolver backend in the report.

For `--resolve-ic-subnet-for-canister`, the resolver must use the NNS routing
table/canister-range data to map canister principal to IC subnet principal. Do
not conflate this with Canic's logical deployment subnet names or root-managed
subnet registry.

System subnet results must not blindly apply application-subnet pricing.
Record `subnet_type = "system"` when available and mark
`charges_apply_to_subject = false` or `kind = "not_applicable"` for
system-canister cases where cycles charges do not apply.

If live lookup fails, the estimate section should be absent unless the caller
also supplied an explicit fallback. The instruction report itself should still
complete.

## Subnet Rate Table

Rate selection is its own small module. It should not be embedded in report
formatting.

Input:

```text
node_count
optional explicit cycles_per_billion_instructions
formula_version
```

Output:

```text
cycles_per_billion_instructions
update_execution_base_cycles
rate_source
notes
```

The module must be easy to update when IC cost constants change. Tests should
pin the current configured table/formula and make rate changes deliberate.

Initial pinned values:

```text
13-node application subnet:
  update execution base fee = 5_000_000 cycles
  1B executed instructions = 1_000_000_000 cycles

34-node fiduciary subnet:
  update execution base fee = 13_076_923 cycles
  1B executed instructions = 2_615_384_615 cycles
```

Rate-table updates are release-preparation items unless they fix a known
incorrect estimate. Each update must cite the current IC cycle-cost docs, update
`formula_version`, and update pinned tests for the 13-node and 34-node values.

The rate module owns one integer derivation rule for node-count-derived rates.
For 13-node and 34-node values, Canic pins the documented published table
values. For arbitrary node counts, derive integer rates with floor division:

```text
cycles_per_billion_instructions =
  floor(1_000_000_000 * node_count / 13)

update_execution_base_cycles =
  floor(5_000_000 * node_count / 13)
```

Estimate calculation then uses the derived integer rate:

```text
estimated_instruction_cycles =
  ceil(local_instructions * cycles_per_billion_instructions / 1_000_000_000)
```

The report records both the displayed integer rate and the derivation/rounding
policy so later rate-table updates stay auditable.

## IC Subnet Catalog CLI And Audit Surface

Do not add a new public `canic cycles ...` profiling command for this slice.
That namespace already means real cycle balances, minting, transfer, top-up,
and 0.58 conversion.

Do add a read-only subnet catalog namespace:

```text
canic subnet network list
canic subnet network info <subnet-principal-or-canister-principal>
canic subnet network refresh
```

The `network` segment is intentional. It keeps command arguments one word each
while making the type boundary explicit:

```text
canic subnet network ...   # IC network subnets from the NNS registry
canic subnet fleet ...     # reserved for possible future Canic fleet topology
```

This namespace reads and refreshes IC network subnet metadata from the local
NNS subnet catalog. It never mutates deployments, canisters, controllers, cycle
balances, role attachments, or Canic runtime state.

`canic subnet network list` should default to cached catalog data and show:

```text
SUBNET       TYPE         GEO     NODES  CHARGES  RANGES  VERSION  FETCHED_AT  STALE
<principal>  application  global  13     yes      4       123456   2026-...    no
<principal>  application  europe  13     yes      2       123456   2026-...    no
<principal>  fiduciary    global  34     yes      1       123456   2026-...    no
<principal>  system       global  13     no       2       123456   2026-...    no
```

Useful flags:

```text
--refresh
--type application|fiduciary|system|unknown
--geo global|europe|unknown
--format text|json
--show-ranges
--range-limit <n>
--range-offset <n>
--stale-after <duration>
--allow-stale-subnet-catalog
```

Default list output must show only a routing-range count. `--show-ranges`
enables range rows, capped by `--range-limit` in text mode. If more ranges are
available, text output should say:

```text
showing 50 of 312 ranges; use --range-limit or --format json
```

`canic subnet network info <x>` should accept either an IC subnet principal or
a canister principal. Resolution is deterministic:

1. if `<x>` exactly matches a `subnet_principal` in the local catalog, resolve
   it as a subnet;
2. else, if `<x>` falls within a cached routing range, resolve it as a
   canister principal;
3. else, return `NotFound` and suggest `canic subnet network refresh`.

The command may accept:

```text
--as subnet
--as canister
```

to force interpretation in tests and in ambiguous synthetic-principal cases.

Expected fields:

```text
input_principal
resolved_as
subnet_principal
subnet_type
geographic_scope
node_count
charges_apply_to_subject
charge_applicability_reason
registry_canister_id
registry_version
catalog_schema_version
fetched_at
catalog_stale
resolver_backend
matched_canister_principal
matched_routing_range
cycles_per_billion_instructions
update_execution_base_cycles
rate_source
formula_version
```

`canic subnet network refresh` performs the live mainnet NNS registry fetch and
rewrites the catalog atomically. Network failure leaves the previous catalog
intact and reports the failure clearly.

Refresh write behavior:

1. fetch registry values;
2. decode into an in-memory catalog;
3. validate non-empty subnet list and routing table;
4. write to a temp file;
5. fsync the temp file when supported;
6. atomically rename over the old catalog;
7. keep the previous catalog on any failure.

Useful flags:

```text
--format text|json
--output <path>
--dry-run
```

0.59.4 supports mainnet NNS refresh only. Custom networks use explicit rate or
explicit node count. A later slice may add custom registry support, but it must
require an explicit custom network and registry canister opt-in.

If a host-facing wrapper is added, keep the instruction noun first:

```text
canic audit instructions ...
```

or keep it internal to:

```text
scripts/ci/instruction-audit-report.sh
cargo test -p canic-tests --test instruction_audit ...
```

Estimate flags must read as opt-in decoration:

```text
--estimate-execution-cycles
--subnet-node-count <n>
--ic-subnet <principal>
--resolve-ic-subnet-for-canister <principal>
--refresh-subnet-catalog
--cycles-per-billion-instructions <cycles>
```

Text output should use a label like:

```text
Execution cycle estimate (instructions only, excludes message/byte/GC/platform fees)
```

JSON output should always keep `local_instructions` present even when estimates
are requested.

## Layering

- `canic-core::perf` remains instruction-only instrumentation.
- DTOs for runtime metrics remain instruction DTOs.
- Subnet resolution and estimate decoration live in host/test/report layers.
- Pure estimate math can live in a small host-side policy/helper module.
- NNS registry calls are infra/ops in the host-side boundary, not workflow
  mutation and not canister runtime policy.

This keeps `endpoints -> workflow -> policy -> ops -> model` intact for
runtime code and avoids dragging network registry lookups into canister-side
performance accounting.

## Migration Plan

1. Audit current active wording and generated artifacts for false cycle labels.
2. Update active docs and report templates to state the counter source and
   exclusions near the first metric table.
3. Add unit tests that reject cycle-named measured instruction fields in the
   instruction-audit artifacts.
4. Add estimate math behind an opt-in report flag.
5. Add offline `--subnet-node-count` / `--cycles-per-billion-instructions`
   support.
6. Add a local NNS subnet catalog reader and cached canister-ID-to-subnet
   resolver for deployed `ic` canisters.
7. Add live NNS registry catalog refresh with fixture tests and a clearly
   skipped or manually gated network proof. Direct registry decoding must not
   block 0.59.2; some registry values are protobuf-shaped and belong in the
   later resolver slice.
8. Run the recurring instruction-footprint audit once with estimates disabled
   and once with estimates enabled to prove the canonical instruction rows are
   unchanged.

## Proposed Slices

### 0.59.0 - Tentative Design

Record the unit boundary, non-goals, estimate model, subnet input choices, and
NNS resolver shape.

### 0.59.1 - Terminology Hard Cut

Fix active wording and generated output so measured values are consistently
instructions. Add focused tests for report labels and JSON artifact keys. Real
cycle funding, top-up, and refill surfaces are explicitly whitelisted and must
not be renamed by this pass.

### 0.59.2 - Estimate Math

Add pure integer estimate helpers and offline report flags. No live network
dependency. This slice must require an explicit rate source, use
`estimated_instruction_cycles` as the default derived value, and pin the current
13-node and 34-node rate-table values.

### 0.59.3 - Local NNS Subnet Catalog

Add the host-side local subnet catalog shape plus cached resolver:

- deployed canister principal to IC subnet principal through cached routing
  ranges;
- IC subnet principal to node count/type through cached subnet records;
- stale/missing catalog behavior that omits estimates unless explicit fallback
  inputs are supplied;
- read-only `canic subnet network list` and
  `canic subnet network info <x>` commands over the cached catalog.

This slice should not require live network access.

### 0.59.4 - NNS Subnet Catalog Refresh

Add the optional live NNS refresh path that populates the local subnet catalog.
Record registry canister ID, registry version, subnet principal, subnet type,
node count, resolver backend, and lookup source in artifacts. This slice owns
`canic subnet network refresh` and `canic subnet network list --refresh`.

### 0.59.5 - Audit Closeout

Run and archive a before/after instruction-footprint report showing that
instruction rows stay comparable and cycle estimates are clearly separated.

## Acceptance Criteria

- Active Canic profiling surfaces call measured values instructions, not
  cycles.
- Any cycle number derived from instructions is opt-in and labeled
  `execution_cycle_estimate`.
- Estimate artifacts include the rate, node count or explicit-rate source,
  formula version, charge model, counter metadata, charges-applicability flag,
  and omitted-cost list.
- The default instruction-footprint audit remains network-free and instruction
  only.
- `--estimate-execution-cycles` fails without an explicit estimate source or an
  inferred source from the local NNS subnet catalog.
- Deployed IC canister estimates can use the local subnet catalog without a live
  NNS lookup on every report run.
- `canic subnet network list` and `canic subnet network info <x>` expose cached
  NNS subnet topology without mutating deployments or runtime state.
- Stale catalog behavior is deterministic: estimates are omitted unless stale
  data is explicitly allowed or the catalog is refreshed.
- European subnets preserve application-pricing behavior while exposing
  `geographic_scope = europe`.
- Query and composite-query samples are not presented as charged query costs.
- NNS lookup failures do not block instruction reports.
- Real cycle funding/top-up/conversion surfaces remain named cycles and are not
  renamed by the terminology pass.

## Required Tests

- Measured row JSON keys in instruction-audit artifacts may include
  `instructions`, never `cycle`, `cycles`, `burn`, or `charged`; the optional
  estimate object is the only place cycle-estimate keys are allowed.
- Real cycle DTOs under funding/top-up/refill paths are whitelisted and
  untouched.
- Generated markdown says instructions, not cycle cost, for measured rows.
- Report metadata includes `counter_id = 1` for endpoint accounting.
- Query samples and update samples preserve `sample_origin`.
- Composite-query samples cannot be silently billed as update execution.
- 1,000,000 local instructions at 1,000,000,000 cycles per billion
  instructions produces 1,000,000 estimated instruction cycles.
- One local instruction rounds up correctly under ceil math.
- Overflow is impossible or explicitly handled with checked integer math.
- Explicit rate wins over node-count-derived rate.
- Local NNS subnet catalog can map a canister principal through cached routing
  ranges to subnet principal, node count, and subnet type.
- Local NNS subnet catalog records `catalog_schema_version` and rejects unknown
  future schema versions unless explicitly migrated.
- `canic subnet network info <canister-principal>` reports the same subnet
  metadata the estimate resolver would use.
- `canic subnet network info <x>` resolves `<x>` deterministically as subnet,
  canister, or unknown and reports `resolved_as`.
- `canic subnet network list --format json` includes registry version,
  fetched-at, subnet principal, subnet type, geographic scope, node count,
  charge applicability, stale status, and rate metadata.
- Range output is capped by default; `--show-ranges` never dumps unbounded
  routing data in text mode.
- `canic subnet network refresh` failures keep the previous catalog intact.
- Missing or stale subnet catalog omits estimates unless explicit fallback is
  supplied.
- `--estimate-execution-cycles` without an explicit or inferred estimate source
  fails.
- 13-node rate pins to 1,000,000,000 cycles per 1B instructions.
- 34-node rate pins to 2,615,384,615 cycles per 1B instructions and
  13,076,923 update execution base cycles.
- Arbitrary node-count rates use the documented floor-division derivation rule.
- Formula-version changes require deliberate snapshot updates.
- Failed NNS lookup leaves the instruction report complete and omits the
  estimate unless an explicit fallback was supplied.
- Successful live lookup records registry version and subnet principal.
- System subnet results mark `charges_apply_to_subject = false` or
  `kind = "not_applicable"`.

## Open Questions

- Where should the host-side NNS subnet catalog live relative to existing local
