# 0.59 Design: Instruction Accounting And Offline Cost Estimates

## Status

- Status: Implemented in `0.59.0`
- Line: `0.59.x`
- Theme: instruction/cycle terminology hard cut plus optional offline
  execution-cost estimates

Implemented follow-ups:

- `0.59.1` tightened the report-input contract so direct environment-driven
  estimate sources require estimate mode, matching the shell wrapper. It also
  fixed workflow linting/setup so GitHub Actions context mistakes are caught by
  pinned `actionlint`.
- `0.59.2` restored CI `RUSTUP_TOOLCHAIN` propagation for nested wasm builds
  after the actionlint cleanup, and cleaned release-gate output by replacing a
  noisy expected `compile_fail` doctest with ordinary coverage of the same
  macro `compile_error!` branch. This did not change the estimate model or the
  ICP-refill endpoint contract.
- `0.59.3` kept the estimate artifact shape stable while making the fixed
  `omitted_costs` list a single pinned report-support contract.

0.59 fixes the unit boundary before Canic grows more profiling or
cost-reporting surface.

Canic's runtime perf counters measure WebAssembly instructions. They do not
measure cycles. Any cycle number derived from them is an estimate, not an
accounting statement and not a bill reconciliation tool.

0.59 deliberately does not add IC NNS subnet inspection. The local subnet
catalog, live NNS refresh path, and `canic subnet network ...` namespace belong
to the separate 0.60 NNS inspection line:

```text
docs/design/0.60-nns-subnet-inspection/0.60-design.md
```

## Problem

The Canic instruction-audit lane is mostly named correctly today:

- `canic-core::perf` stores `total_instructions`;
- `QueryPerfSample` returns `local_instructions`;
- generated reports use `total_local_instructions` and
  `avg_local_instructions`;
- the recurring audit definition explicitly says it is not a cycle-cost audit.

The risk is that downstream wording, future JSON fields, or convenience
helpers can still collapse two different concepts:

- instruction counters from `performance_counter` / `countInstructions`;
- cycles charged by the protocol for execution, messages, bytes, management
  operations, storage, reservations, and other resource use.

That collapse is dangerous because a clean instruction regression win can still
leave a user's cycle bill unchanged or even higher if message or platform fees
dominate the operation.

## Core Rule

The measured unit is `instructions`.

The only acceptable names for measured values are instruction names:

```text
local_instructions
total_local_instructions
avg_local_instructions
checkpoint_delta_instructions
instruction_count
```

Do not call these values:

```text
cycles
cycle_cost
cycle_delta
burn
charged_cycles
```

Cycle language is reserved for real cycle balances, transfers, top-ups, CMC
conversion, and explicit estimates derived from instruction counts.

## Counter Semantics

0.59 must document which counter each path uses.

Current Canic runtime perf accounting uses `performance_counter(1)`:

- local canister WebAssembly instructions;
- current call context;
- preserved across `await` points in that call context;
- excludes instructions executed by other canisters;
- not a direct measure of cycles charged.

Counter scope still depends on message kind:

- replicated update execution excludes nested IC calls, including calls to the
  same canister;
- non-replicated composite-query execution can accumulate a broader same-query
  call/callback scope and may include nested composite-query work depending on
  the query shape.

Reports must preserve `sample_origin` and must not compare replicated update
samples, ordinary query samples, and composite-query samples as if they had
identical counter scope.

Motoko `countInstructions` and `performance_counter(0)` are different:

- synchronous current-message instruction counter;
- reset at shared function entry and at `await`;
- useful for local slice profiling;
- not complete endpoint accounting across async resumptions.

The design consequence is simple: Canic should not import a
`countInstructions` mental model into endpoint accounting, and it should not
rename current endpoint accounting to cycles.

## In Scope

- Audit and correct confusing instruction/cycle wording in active docs,
  reports, tests, logs, help text, and JSON field names.
- Preserve the instruction-footprint audit as the authoritative measured lane.
- Add an optional execution-cycle estimate layer only where it is explicitly
  requested by a report flag.
- Make every estimate carry its input assumptions: explicit rate source,
  formula version, counter ID, and omitted costs.
- Provide deterministic offline inputs:
  `--estimate-node-count` and `--cycles-per-billion-instructions`.
- Keep the default instruction audit network-free and catalog-free.

## Out Of Scope

- Exact bill reconstruction.
- Per-message, byte, storage, reservation, GC, and management-call accounting.
- Replacing `canic info cycles`, `canic cycles`, or the 0.58 ICP refill work.
- Runtime canisters calling the NNS registry just to decorate perf rows.
- Treating query samples as charged update execution.
- Renaming real cycle DTOs used for balances, transfers, top-ups, or funding.
- Local IC subnet catalogs, routing-range indexes, stale-cache policy, live NNS
  refresh, and `canic subnet network ...`; those are 0.60 work.
- Deployed-canister estimate inference through cached NNS routing data; 0.59
  requires explicit estimate inputs.

## Terminology Hard Cut

0.59 should start with a focused terminology pass over active surfaces:

- `crates/canic-core/src/perf.rs`
- `crates/canic-core/src/dto/metrics.rs`
- `crates/canic/src/macros/mod.rs`
- `crates/canic/src/instructions/mod.rs`
- `crates/canic-tests/tests/instruction_audit_support/`
- `docs/audits/recurring/system/instruction-footprint.md`
- generated instruction-footprint templates and new artifacts
- CLI/help text if any profiling command or report wrapper is added

Expected outcome:

- no measured instruction field is called cycles;
- no report says cycle cost when it only has instruction counters;
- logs keep the `i` suffix or spell out `instructions`;
- test names and fixtures use instruction wording unless they exercise real
  cycle funding behavior.

Historical changelog entries and archived audit reports may remain historical
unless they actively mislead current users. Active docs and generated output
matter more than archive churn.

## Report Shape

Instruction rows remain the canonical row model:

```text
subject_kind
subject_label
count
total_local_instructions
avg_local_instructions
scenario_key
scenario_labels
principal_scope
sample_origin
```

If estimates are enabled, they are a sibling object, not a replacement for
instruction fields:

```text
execution_cycle_estimate: {
  "estimate_schema_version": 1,
  "kind": "per_instruction_component_only",
  "charge_model": "hypothetical_update_execution_component",

  "local_instructions": 1000000,
  "counter_id": 1,
  "sample_origin": "update",

  "estimated_instruction_cycles": "1000000",
  "cycles_per_billion_instructions": "1000000000",
  "subnet_node_count": 13,
  "subnet_source": "flag",
  "source_meaning": "operator_supplied_pricing_assumption",

  "formula_version": "canic-0.59-ic-cycle-costs-v1",
  "rate_source": "official-ic-cycle-costs-docs",
  "overrode_node_count_table_rate": false,
  "node_count_table_rate": "1000000000",

  "omitted_costs": [
    "update_message_execution_base_fee",
    "message_transmission_base_fee",
    "payload_bytes",
    "garbage_collection",
    "callee_instructions",
    "management_call_fees",
    "storage_and_reservations"
  ]
}
```

Use integer/rational math only. Prefer `cycles_per_billion_instructions` over
floating-point `cycles_per_instruction`.

Do not use `estimated_cycles` as a field name. It is too easy to read as total
charged cycles. The default derived value is
`estimated_instruction_cycles`.

The 0.59 estimate object should stay minimal. Fields such as
`subnet_principal`, `catalog_schema_version`, `registry_version`,
`resolver_backend`, `routing_range`, and `geographic_scope` must not be
populated in 0.59. Those fields only make sense once 0.60 adds an actual NNS
subnet catalog source.

`subnet_source` values in 0.59 are:

```text
flag
explicit-rate
```

`rate_source` values in 0.59 are:

```text
official-ic-cycle-costs-docs
operator-explicit-rate
```

`charge_model` values in 0.59 are:

```text
hypothetical_update_execution_component
not_charged_query_sample
not_applicable
```

## Estimate Model

The estimate layer answers one narrow question:

```text
Given this many measured local instructions, what is the approximate execution
cycle component under a stated subnet-size/rate assumption?
```

It does not answer:

```text
What did this call cost on the user's bill?
```

Formula:

```text
estimated_instruction_cycles =
  ceil(local_instructions * cycles_per_billion_instructions / 1_000_000_000)
```

Implementation must use checked integer math:

```rust
fn estimate_instruction_cycles(
    local_instructions: u128,
    cycles_per_billion_instructions: u128,
) -> Result<u128, EstimateOverflow> {
    let numerator = local_instructions
        .checked_mul(cycles_per_billion_instructions)
        .ok_or(EstimateOverflow)?;

    Ok(ceil_div(numerator, 1_000_000_000))
}
```

JSON serializes cycle estimates and rates as decimal strings. Instruction
counts remain numeric only when they are already safely represented by the
existing artifact schema.

The formula version identifies Canic's pinned cost table, not necessarily the
latest upstream IC docs at report generation time.

If an execution base fee is included in a later slice, it must be a separate
field:

```text
execution_base_cycles
estimated_instruction_cycles
estimated_execution_cycles
```

Do not hide the base fee inside the per-instruction estimate.

0.59 omits `estimated_execution_cycles` by default because it does not include
message base fees or other billed components. If a later report includes the
update execution base fee, it must set a different `kind` and populate the base
fee separately.

Query and composite-query samples are different. Current IC cost docs list
query calls as free, while update execution has a base fee and per-instruction
fee. Therefore:

- query and composite-query estimates are omitted by default;
- if shown, their `charge_model` must be
  `hypothetical_update_execution_component` or
  `not_charged_query_sample`;
- they must never be presented as charged query costs.

## Estimate Inputs

`--estimate-execution-cycles` must fail unless an explicit estimate source is
provided. 0.59 has no network lookup, local catalog, or silent 13-node default.

### Explicit Node Count

The deterministic table-backed path:

```text
--estimate-node-count <n>
```

In 0.59, `--estimate-node-count` is an operator-supplied pricing assumption.
Canic does not resolve the measured canister's actual IC subnet in this slice.

The report records:

```text
subnet_source = "flag"
subnet_node_count = <n>
source_meaning = "operator_supplied_pricing_assumption"
rate_source = "official-ic-cycle-costs-docs"
```

0.59 supports only node counts with pinned table values:

```text
13
34
```

Other node counts are rejected unless the caller supplies an explicit rate.
This avoids false precision from deriving rates for subnet sizes that are not
part of the pinned cost table.

### Explicit Rate

The escape hatch for protocol-cost changes, custom networks, or unsupported
node counts:

```text
--cycles-per-billion-instructions <cycles>
```

The report records:

```text
subnet_source = "explicit-rate"
cycles_per_billion_instructions = <cycles>
rate_source = "operator-explicit-rate"
```

When both node count and explicit rate are provided, explicit rate wins and the
node count is retained only as metadata. The report must show that the
node-count table rate was overridden:

```text
subnet_node_count = 34
subnet_source = "explicit-rate"
cycles_per_billion_instructions = "123456789"
rate_source = "operator-explicit-rate"
overrode_node_count_table_rate = true
node_count_table_rate = "2615384615"
```

## Subnet Rate Table

Rate selection is its own small module. It should not be embedded in report
formatting.

Input:

```text
optional node_count
optional explicit cycles_per_billion_instructions
formula_version
```

Output:

```text
cycles_per_billion_instructions
node_count_table_rate
overrode_node_count_table_rate
rate_source
notes
```

The module must be easy to update when IC cost constants change. Tests should
pin the current configured table/formula and make rate changes deliberate.

Initial pinned values:

```text
13-node application subnet:
  1B executed instructions = 1_000_000_000 cycles

34-node fiduciary subnet:
  1B executed instructions = 2_615_384_615 cycles
```

If a later release includes update execution base fees, it must add separate
base-fee outputs and tests. 0.59's default estimate object is instruction
component only.

Rate-table updates are release-preparation items unless they fix a known
incorrect estimate. Each update must cite the current IC cycle-cost docs,
update `formula_version`, and update pinned tests for the 13-node and 34-node
values.

## Audit Surface

Do not add a new public `canic cycles ...` profiling command for this slice.
That namespace already means real cycle balances, minting, transfer, top-up,
and 0.58 conversion.

Do not add `canic subnet network ...` in 0.59. That namespace is reserved for
the 0.60 NNS subnet inspection line.

If a host-facing wrapper is added, keep the instruction noun first:

```text
canic audit instructions ...
```

or keep it internal to:

```text
scripts/ci/instruction-audit-report.sh
cargo test -p canic-tests --test instruction_audit ...
```

Estimate flags must read as opt-in decoration:

```text
--estimate-execution-cycles
--estimate-node-count <n>
--cycles-per-billion-instructions <cycles>
```

Text output should use a label like:

```text
Execution cycle estimate (instructions only, excludes message/byte/GC/platform fees)
```

JSON output must always keep `local_instructions` present even when estimates
are requested.

## Layering

- `canic-core::perf` remains instruction-only instrumentation.
- DTOs for runtime metrics remain instruction DTOs.
- Estimate decoration lives in host/test/report layers.
- Pure estimate math can live in a small host-side policy/helper module.
- No canister runtime code reads NNS registry data for estimate decoration.

This keeps `endpoints -> workflow -> policy -> ops -> model` intact for
runtime code and avoids dragging network registry lookups into canister-side
performance accounting.

## Migration Plan

1. Audit current active wording and generated artifacts for false cycle labels.
2. Update active docs and report templates to state the counter source and
   exclusions near the first metric table.
3. Add unit tests that reject cycle-named measured instruction fields in the
   instruction-audit artifacts, with real cycle funding DTOs whitelisted.
4. Add pure integer estimate math behind an opt-in report flag.
5. Add offline `--estimate-node-count` / `--cycles-per-billion-instructions`
   support.
6. Run the recurring instruction-footprint audit once with estimates disabled
   and once with estimates enabled to prove the canonical instruction rows are
   unchanged.

## Proposed Release Shape

### 0.59.0 - Terminology And Offline Estimates

Ship one functional release:

- terminology hard cut in active docs and generated report templates;
- JSON/report linting for instruction rows;
- host/test-side integer estimate math;
- offline estimate flags only;
- minimal `execution_cycle_estimate` object;
- explicit tests proving disabled estimates do not read network or catalog
  state.

If review wants two development batches, land terminology before estimates.
Do not turn the NNS catalog into a required 0.59 patch. The catalog is 0.60.
At the commit level, keep the natural split as terminology/artifact linting
first and offline estimate math/flags second.

### 0.59.x - Follow-Up Cleanup

Post-`0.59.0` patches should remain cleanup or contract-hardening patches for
the same surface. They may:

- tighten validation so shell flags and direct report environment inputs behave
  the same way;
- improve local/CI workflow linting for the existing audit runner;
- preserve toolchain and wasm-target propagation for nested Cargo builds;
- quiet release-gate output when a test proves an expected compile-time error.
- keep repeated estimate metadata as pinned constants when doing so does not
  change the serialized artifact shape.

They must not:

- add NNS, subnet catalog, or routing lookup;
- add a public cycles profiling command;
- rename real cycle funding/top-up/refill surfaces;
- broaden estimate semantics into bill reconstruction.

Release gates should be readable. Expected compiler failures should not appear
as red output during ordinary `make patch` runs. If a macro contract needs to
prove a `compile_error!` branch, prefer quiet unit/source coverage or a
dedicated UI-test harness that reports expected failures cleanly.

## Acceptance Criteria

- Active Canic profiling surfaces call measured values instructions, not
  cycles.
- Any cycle number derived from instructions is opt-in and labeled
  `execution_cycle_estimate`.
- Estimate artifacts include the rate, explicit source, formula version,
  counter ID, charge model, rate source, estimate schema version, and
  omitted-cost list.
- The default instruction-footprint audit remains network-free, catalog-free,
  and instruction only.
- `--estimate-execution-cycles` fails without an explicit estimate source.
- With `--estimate-node-count 13`, estimate output uses
  `subnet_source = "flag"`, `rate_source = "official-ic-cycle-costs-docs"`,
  and requires no catalog fields.
- With `--cycles-per-billion-instructions`, explicit rate wins over node-count
  defaults and uses `rate_source = "operator-explicit-rate"`.
- When explicit rate and node count are both provided, the report shows that
  the node-count table rate was overridden.
- Unsupported arbitrary node counts are rejected unless an explicit rate is
  supplied.
- No 0.59 artifact contains `subnet_principal`, `registry_version`,
  `catalog_schema_version`, `resolver_backend`, `routing_range`, or
  `geographic_scope`; those fields belong to the future 0.60 catalog source.
- Query and composite-query samples are not presented as charged query costs.
- Real cycle funding/top-up/conversion surfaces remain named cycles and are not
  renamed by the terminology pass.
- Release-gate validation remains quiet: expected compile-time failures used to
  prove macro contracts must not be emitted as scary red diagnostics during
  ordinary patch preparation.

## Required Tests

- Measured row JSON keys in instruction-audit artifacts may include
  `instructions`, never `cycle`, `cycles`, `burn`, or `charged`; the optional
  estimate object is the only place cycle-estimate keys are allowed.
- Real cycle DTOs under funding/top-up/refill paths are whitelisted and
  untouched.
- Generated markdown says instructions, not cycle cost, for measured rows.
- Report metadata includes `counter_id = 1` for endpoint accounting.
- Query samples and update samples preserve `sample_origin`.
- Composite-query samples cannot be silently billed as update execution.
- With estimates disabled, the report path performs zero network calls and zero
  subnet-catalog reads.
- `--estimate-execution-cycles` without an explicit source fails.
- `--estimate-node-count 13` pins to 1,000,000,000 cycles per 1B instructions.
- `--estimate-node-count 34` pins to 2,615,384,615 cycles per 1B instructions.
- Unsupported node counts are rejected unless
  `--cycles-per-billion-instructions` is supplied.
- Node-count table estimates use
  `rate_source = "official-ic-cycle-costs-docs"`.
- Explicit-rate estimates use `rate_source = "operator-explicit-rate"`.
- Explicit rate plus node count records that the node-count table rate was
  overridden.
- Estimate objects include `estimate_schema_version = 1`.
- 0.59 estimate objects use `kind = "per_instruction_component_only"`.
- 0.59 estimate artifacts do not contain NNS/catalog-derived fields.
- Cycle estimates and rate values serialize as decimal strings.
- 1,000,000 local instructions at 1,000,000,000 cycles per billion
  instructions produces 1,000,000 estimated instruction cycles.
- One local instruction rounds up correctly under ceil math.
- Overflow is impossible or explicitly handled with checked integer math.
- Explicit rate wins over the node-count table rate.
- Formula-version changes require deliberate snapshot updates.
- Direct environment-driven estimate sources are rejected when estimate mode is
  disabled, matching the shell wrapper's explicit flag contract.
- Macro compile-error contracts touched by this line are covered without
  noisy `compile_fail` doctest output in the release gate.
