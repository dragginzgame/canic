# Canic Design Proposal: Template Canisters as Canonical Deployable Wasm Storage

- **Version:** `0.18`
- **Status:** `Draft`
- **Author:** `Canic Core`
- **Scope:** hard architectural cut from root-embedded deployable wasm to subnet-local wasm-store storage with manifest-bound hash verification and root-retained deployment authority

## 1. Continuation Boundary

`0.18` is the implementation line that replaces the root-embedded deployable-wasm
model established as unsustainable in `0.17`.

This document is not an options memo.
The architecture is chosen:

- wasm stores are mandatory
- wasm stores are the canonical storage and distribution layer for deployable wasm templates
- root no longer owns long-term storage of ordinary deployable child wasm bytes
- root retains deployment authority, manifest authority, selection authority, and orchestration authority

This is a hard architectural cut.

Template stores are a required control-plane subsystem in the steady-state
topology.

There is no steady-state embedded/delegated coexistence.
There is no compatibility architecture.
There is no dual-mode target state.

Hard-cut scope:

- the hard cut applies to ordinary deployable child templates
- the retained bootstrap set is not an embedded fallback mode
- the retained bootstrap set is part of the new control-plane bootstrap and recovery basis

## 2. Problem Statement

The old model bound two incompatible scaling roles into one canister:

- root as long-lived control-plane authority
- root as payload accumulator for every deployable child template

That design fails because root grows with both:

- control-plane/runtime evolution
- deployable artifact accumulation

`0.18` removes the second responsibility from root.

The required fix is not further root slimming alone.
The required fix is removal of ordinary deployable payload ownership from root.

## 3. Chosen Architecture

### 3.1 Root Authority

Root remains authoritative for:

- deployment policy
- manifest approval
- template selection
- target-canister creation
- install orchestration
- upgrade orchestration
- topology and lifecycle coordination

### 3.2 Template-Store Authority

Template stores are canonical for:

- deployable wasm storage
- deterministic chunk-set metadata
- approved template chunk storage
- template payload distribution

Template stores do not decide what gets installed.
They do not create child canisters.
They do not perform lifecycle orchestration.
They store and serve approved template chunks plus deterministic metadata only.

A subnet-local wasm store may hold multiple templates and multiple approved
versions.

There is no architectural requirement that one wasm store correspond to one
deployable wasm.

### 3.3 Chunk-Store-Native Install Model

Ordinary child install and upgrade is chunk-store-backed and
management-canister-native.

The canonical ordinary install path is:

1. root resolves the approved manifest for the target role
2. workflow resolves the approved logical template release:
   - `template_id`
   - `version`
3. workflow resolves the target subnet
4. workflow resolves the correct subnet-local store location for that target subnet
5. workflow resolves deterministic chunk metadata for that template version in that store
6. ops verifies that the approved manifest binds:
   - `template_id`
   - `version`
   - expected `payload_hash`
   - expected `payload_size_bytes`
   - expected store placement binding
   - deterministic chunk metadata sufficient to address the exact stored chunk set
7. root invokes `install_chunked_code`
8. the management canister installs or upgrades the target canister from the
   approved chunk set stored in the same-subnet store canister and enforces
   the final `wasm_module_hash` over the reconstructed module

Root does not act as a large-wasm transit courier between canisters.
Ordinary install and upgrade does not depend on moving the full wasm through
root memory via normal inter-canister payload transfer.

The steady-state install path is chunk-store-backed.

### 3.4 Subnet Topology Consequences

Cross-subnet large-wasm retrieval through ordinary inter-canister messaging is
not the architecture.

`install_chunked_code` requires the store canister and the target canister to
be on the same subnet.

Therefore the steady-state topology requires subnet-local template storage for
the templates installable on that subnet.

The baseline topology is:

- one subnet-local wasm store per subnet per Canic deployment

The supported extension is:

- multiple wasm stores on the same subnet for capacity or isolation

This is not an incidental implementation detail.
Template-store topology is a control-plane dependency.

Root remains the caller for install and upgrade.
Therefore root must control the target canister and must also control the store
canister, or satisfy the equivalent IC controller rule required by
`install_chunked_code`.

### 3.5 Store Location and Placement

Subnet-local wasm-store locations are explicit control-plane
configuration and state.

Store location is not implied.
Store location is not discovered opportunistically at install time.

Install resolution requires:

- the target subnet
- the approved logical template release
- the correct subnet-local store placement for that target subnet

Store placement must be deterministic and must be validated before install.

Multiple Canic deployments may each maintain their own subnet-local template
stores even on the same IC subnet.

### 3.6 Store Capacity Policy

Each subnet-local wasm store has explicit capacity policy.

`0.18` requires:

- `max_store_bytes`

`0.18` may also carry:

- `max_templates`
- `max_template_versions_per_template`
- headroom thresholds for republish and rollover planning

Automatic rollover is not part of the baseline `0.18` architecture.
When a store approaches capacity, additional subnet-local stores are an
explicit control-plane action under root authority, not an implicit runtime
behavior.

`max_store_bytes` counts total occupied store bytes in that wasm store.
This includes stored chunk bytes plus store-owned metadata overhead.
It is not limited to the logical payload size of published releases.

### 3.7 Manifest Authority

Hash-verified manifests are authoritative.

Every install or upgrade decision is bound to:

- a manifest entry approved by root
- an expected template identifier
- an expected template version
- an expected payload hash
- an expected payload size
- an expected target role
- an expected store placement binding

No payload is installed without manifest-bound hash verification.

Root is the canonical manifest authority.
Template stores may expose descriptive metadata, but that metadata is not
authoritative for install or upgrade decisions.

## 4. Non-Goals

`0.18` does not:

- preserve backward compatibility with the embedded-root payload model
- add shim layers or bridge layers
- support dual embedded/template deployment modes as a target state
- keep embedded fallback paths for ordinary deployable child templates
- treat coexistence as a desired rollout shape
- weaken root deployment authority
- weaken install or upgrade integrity checks
- turn wasm stores into lifecycle authorities

## 5. Canonical Model

### 5.1 Root

Root stores:

- approved manifests
- role-to-template selection policy
- deployment and upgrade policy
- manifest approval state
- minimal bootstrap material only

Root does not store:

- the long-term payload bytes for ordinary deployable child templates

### 5.2 Template Stores

Template stores hold:

- multiple logical templates
- multiple approved versions per template
- approved template chunk sets
- deterministic chunk ordering metadata
- payload-size metadata
- payload-hash metadata
- chunk-count metadata

Template stores expose:

- deterministic metadata lookup by `template_id` and `version`
- deterministic chunk lookup by `template_id`, `version`, and chunk index
- deterministic chunk-set metadata needed for `install_chunked_code`

Template bytes are published through a root-governed approval path.
Publishing payload bytes does not make them installable until root approves the
corresponding manifest entry.

### 5.3 Workflow

Workflow owns:

- manifest resolution
- template version resolution
- subnet-local store resolution
- deterministic chunk metadata resolution
- expected chunk-set verification
- install and upgrade orchestration
- retry behavior for metadata resolution and install failures

Workflow does not turn root into a byte-forwarding relay.
Workflow coordinates chunk-store-backed install.

### 5.4 Ops

Ops owns:

- manifest storage and retrieval
- store placement metadata shaping
- template release metadata shaping
- store-capacity checks
- deterministic chunk metadata helpers
- hash helpers
- `install_chunked_code` execution helpers
- chunk-store preparation helpers
- template view and DTO conversion

### 5.5 Policy

Policy decides:

- which template is valid for a given role
- which version is installable for a given role
- which manifest states are installable
- which target canisters are eligible for install or upgrade
- which bootstrap artifacts are irreducible

## 6. What Is Removed

The following are removed from the old model:

- root as long-term owner of ordinary deployable child wasm bytes
- root as the canonical byte source for ordinary child installation and upgrade
- root as a large-wasm relay through ordinary inter-canister transport
- root growth that is linear in the number of deployable child artifacts
- embedded-registry assumptions for ordinary child template distribution

The following remain:

- root deployment authority
- root manifest approval authority
- root orchestration authority

## 7. Manifest, Release, and Placement Model

Approved manifests are authoritative deployment records.

Each approved manifest includes at minimum:

- `template_id`
- `role`
- `version`
- `payload_hash`
- `payload_size_bytes`
- `store_binding`
- `chunking_mode`
- `manifest_state`
- `approved_at`
- `created_at`

The logical release is defined by:

- `template_id`
- `version`
- `payload_hash`
- `payload_size_bytes`
- `chunking_mode`

The approved manifest binds that logical release to:

- an installable `role`
- a root-approved `store_binding`

The approved manifest does not imply one globally unique storage canister for
that release.

`store_binding` is a root-approved placement selector that deterministically
resolves, for a given target subnet, to an installable subnet-local store
containing the approved logical release.

Required states:

- `Staged`
- `Approved`
- `Blocked`
- `Deprecated`

Install and upgrade use `Approved` manifests only.

### 7.1 Logical Release Metadata

Logical template release metadata includes:

- `template_id`
- `version`
- `role`
- `payload_hash`
- `payload_size_bytes`
- `chunking_mode`

This metadata defines the logical release.
It is not the same thing as physical store placement.

`version` is scoped to `template_id`.
Version uniqueness is not global across unrelated templates.

### 7.2 Store Placement Metadata

Store placement metadata binds a logical template release to one or more
subnet-local stores for installable use on specific target subnets.

Placement metadata is explicit.
Placement metadata is deterministic.
Placement metadata is validated before install.

Root remains authoritative for both the approved manifest and the placement
binding used for install selection.

## 8. Trust Model

The trust rule is fixed:

> root authorizes manifests; wasm stores hold approved chunk sets;
> workflow resolves the approved manifest and subnet-local store; ops verifies
> the chunk set and wasm hash binding; root invokes `install_chunked_code`.

Template stores are not trust anchors.
Template stores are not deployment authorities.
Manifest approval in root is the trust anchor.

Before any ordinary install or upgrade:

1. root resolves the approved manifest for the target role
2. workflow resolves the approved logical template release:
   - `template_id`
   - `version`
3. workflow resolves the correct subnet-local store for the target subnet
4. workflow resolves deterministic chunk metadata for that template version in that store
5. ops verifies that the approved manifest and placement binding cover the
   expected chunk metadata, chunk ordering, total size, and subnet-local store
   identity for that install
6. root invokes `install_chunked_code`
7. the management canister reconstructs the wasm from the referenced chunk set
   and enforces the final `wasm_module_hash`
8. only then does install or upgrade proceed

If verification fails:

- install or upgrade fails closed
- no fallback payload source is consulted
- no alternate embedded path is used for ordinary deployable child templates

## 9. Retrieval Semantics

Retrieval for install is chunk-store-native.

Ordinary install and upgrade resolves:

- the approved `template_id`
- the approved `version`
- the approved subnet-local store canister
- deterministic chunk ordering metadata
- deterministic chunk-hash metadata

Required properties:

- stable template identity
- stable version identity
- stable chunk order
- stable total reconstructed module
- stable wasm hash
- stable failure behavior for missing or mismatched chunk sets

Required API shape:

- `get_template_info(template_id, version)` for size, hash, chunk-count, and chunk-order metadata
- `get_chunk_set_info(template_id, version)` or equivalent deterministic chunk-set metadata lookup
- `get_template_chunk(template_id, version, chunk_index)` or equivalent deterministic chunk API

There is no "latest template" retrieval path in the install or upgrade flow.
Selection is manifest-driven, not discovery-driven.

Template stores do not serve authoritative manifests.
Manifest authority remains in root.

`chunk_index` is the logical release view required for deterministic
reconstruction.
The store may use additional internal addressing, including chunk-hash-based
storage, so long as the logical release view remains stable.

Install may use metadata resolution plus management-canister chunk assembly.
The steady-state architecture does not require root to reconstruct the full
ordinary deployable wasm inside normal inter-canister message flow.

## 10. Bootstrap and Recovery

The chosen bootstrap position is decisive:

- root retains only the minimum irreducible bootstrap artifact set
- ordinary deployable child templates are not part of that retained set

The minimum bootstrap set exists only to:

- establish wasm-store availability
- seed manifest authority
- recover the control plane to the point where template-backed install becomes available

Bootstrap does not preserve the old embedded-root registry as a shadow system.

Recovery uses:

- root manifest authority
- minimal bootstrap payloads
- subnet-local wasm-store re-establishment where required

Recovery does not use:

- a full embedded copy of ordinary deployable child templates in root

The retained bootstrap set is part of the new control-plane recovery basis.
It is not a hybrid deployment mode.

## 11. Root Storage After the Cut

After `0.18`, root storage contains:

- approved manifests
- role-selection policy
- deployment state
- minimal bootstrap and recovery payloads

After `0.18`, root storage does not contain:

- the ordinary deployable payload bytes for any deployable child role that is
  not part of the irreducible bootstrap set established in `0.17`

## 12. Failure Modes

### 12.1 Manifest Missing

- install or upgrade fails closed
- no payload resolution occurs
- operational action is manifest publication or approval repair

### 12.2 Subnet-Local Store Missing

- ordinary child install or upgrade fails closed
- no alternate embedded payload is used
- operational action is subnet-local wasm-store repair or republish

### 12.3 Chunk Set Incomplete

- install or upgrade fails closed
- no partial chunk set is treated as installable
- operational action is store repair or republish

### 12.4 Chunk Metadata Missing or Hash Entry Missing

- install or upgrade fails closed
- the template is treated as invalid for that manifest
- operational action is metadata repair or republish

### 12.5 Store and Target Subnet Mismatch

- ordinary child install or upgrade fails closed
- cross-subnet large-wasm relay is not attempted
- operational action is publication into the correct subnet-local store

### 12.6 Controller Relationship Missing

- install or upgrade fails closed
- root does not attempt a degraded path
- operational action is controller repair for the store and target canisters

### 12.7 Hash Mismatch Between Approved Manifest and Reconstructed Wasm

- install or upgrade fails closed
- the chunk set is treated as invalid for that template
- manifest authority remains in root

### 12.8 Bootstrap Artifact Loss

- control-plane recovery uses the retained minimal bootstrap set
- loss of the retained bootstrap set is a control-plane recovery incident, not a
  reason to restore the old embedded-root model

## 13. Operational Implications

`0.18` changes the system topology deliberately:

- wasm stores become subnet-local control-plane dependencies
- root becomes an authority and orchestration canister rather than a payload warehouse
- deployment observability must include manifest resolution, template version
  resolution, subnet-local store resolution, chunk metadata validation,
  `install_chunked_code`, and install execution

Required operational metrics:

- manifest lookup success and failure
- subnet-local wasm-store availability
- chunk-store population success and failure
- deterministic chunk metadata resolution success and failure
- chunk-set or chunk-hash mismatch failures
- store and target subnet mismatch failures
- controller authorization failures on the install path
- `install_chunked_code` success and failure
- install or upgrade execution failures

## 14. One-Way Migration Sequence

This is a cutover plan, not a compatibility design.

### Phase 1: Establish Subnet-Local Template Stores

- create the subnet-local wasm stores
- place them under root control
- treat them as control-plane infrastructure

### Phase 2: Publish Approved Templates Into Subnet-Local Stores

- publish the approved template inventory into the correct subnet-local stores
- publish multiple approved template versions where retention policy requires it
- record deterministic chunk metadata for each approved release
- validate chunk-store completeness and `install_chunked_code` readiness

### Phase 3: Switch Ordinary Install and Upgrade to Chunked Install

- resolve ordinary child templates by approved manifest plus subnet-local store
- resolve approved template versions deterministically
- use `install_chunked_code` as the canonical install path
- remove old assumptions that root is the long-term source of ordinary child
  payload bytes or a byte-forwarding relay

### Phase 4: Reduce Root to the Bootstrap Set

- retain only the irreducible bootstrap and recovery payload set in root
- remove all ordinary embedded child template ownership from root

End state:

- subnet-local wasm stores are canonical
- root is authoritative
- ordinary child lifecycle is chunk-store-backed
- the old embedded-root model is gone

## 15. Layering Boundaries

The architecture must preserve Canic's ownership model:

- `policy` decides manifest validity and template selection rules
- `workflow` coordinates manifest resolution, template version resolution,
  subnet-local store resolution, deterministic chunk metadata validation, and install orchestration
- `ops` performs store-capacity checks, deterministic chunk metadata helpers,
  hash helpers, chunk-store preparation helpers, `install_chunked_code`
  execution helpers, and owns conversions
- `model/storage` owns stored manifest state, template release metadata,
  store placement metadata, and store capacity/accounting metadata
- endpoints remain thin request boundaries

Forbidden:

- policy reading template payload storage directly
- workflow owning manifest or storage schema conversion
- endpoints performing retrieval or install orchestration
- template storage records leaking upward as public control-plane shapes
- wasm stores acting as factories or lifecycle authorities

## 16. Success Criteria

`0.18` succeeds if:

- root no longer scales linearly with ordinary deployable child artifact count
- wasm stores are the canonical storage and distribution layer
- one subnet-local wasm store per subnet per Canic deployment is the baseline topology
- ordinary child install and upgrade uses subnet-local `install_chunked_code`
- every install or upgrade is manifest-bound and hash-verified
- ordinary deployable child roles no longer depend on root-owned embedded payload copies
- bootstrap and recovery operates from the minimal irreducible retained set only
- old embedded-root payload ownership is removed from the steady state

## 17. Rejected Directions

Rejected for `0.18`:

- hybrid embedded/template steady state
- embedded fallback for ordinary deployable child templates
- root retaining long-term ownership of all deployable payload bytes
- one store canister per wasm as a required architectural rule
- wasm stores as factories or deployers for ordinary child lifecycle
- implicit store-location discovery without manifest or config binding
- unbounded store growth without explicit capacity policy
- root as a large-wasm relay through ordinary inter-canister transport
- payload installation without manifest-bound hash verification
- compatibility shims as a long-term architecture

The rejected directions are wrong for architectural and IC-topology reasons:

- lifecycle authority belongs to root, not to storage canisters
- ordinary large-wasm install must respect same-subnet chunk-store constraints
- byte-relay through root is not the steady-state IC-native install model
- store growth must be bounded by explicit capacity policy

## 18. Narrow Remaining Questions

Only implementation-scoped questions remain:

- exact template chunk size and chunk-count limits
- exact store placement schema and config key names that should survive as the stable ABI
- exact release metadata record shape for `template_id` + `version` + subnet-local placement
- exact retention policy for old template versions
- exact chunk-store accounting and GC strategy
- exact minimal bootstrap payload inventory from `0.17`
- exact retry policy for chunk metadata resolution before install failure is surfaced
