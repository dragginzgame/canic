# Canic Design Proposal: Template Canisters as Canonical Deployable Wasm Storage

- **Version:** `0.18`
- **Status:** `Draft`
- **Author:** `Canic Core`
- **Scope:** hard architectural cut from root-embedded deployable wasm to template-canister ownership with manifest-bound hash verification and root-retained deployment authority

## 1. Continuation Boundary

`0.18` is the implementation line that replaces the root-embedded deployable-wasm
model established as unsustainable in `0.17`.

This document is not an options memo.
The architecture is chosen:

- template canisters are mandatory
- template canisters are the canonical storage and distribution layer for deployable wasm templates
- root no longer owns long-term storage of ordinary deployable child wasm bytes
- root retains deployment authority, manifest authority, selection authority, and orchestration authority

This is a hard architectural cut.

Template canisters are a required control-plane subsystem in the steady-state
topology.

There is no steady-state embedded/delegated coexistence.
There is no compatibility architecture.
There is no dual-mode target state.

Hard-cut scope:

- the hard cut applies to ordinary deployable child templates
- the retained bootstrap set is not an embedded fallback mode
- the retained bootstrap set is part of the new control-plane bootstrap and recovery basis

## 2. Problem Statement

The old model bound two incompatible scaling roles into one canister:

- root as long-lived control-plane authority
- root as payload accumulator for every deployable child template

That design fails because root grows with both:

- control-plane/runtime evolution
- deployable artifact accumulation

`0.18` removes the second responsibility from root.

## 3. Chosen Architecture

### 3.1 Root Authority

Root remains authoritative for:

- deployment policy
- manifest approval
- template selection
- install orchestration
- upgrade orchestration
- topology and lifecycle coordination

### 3.2 Template Canister Authority

Template canisters are canonical for:

- deployable wasm storage
- template retrieval APIs
- template payload distribution

Template canisters do not decide what gets installed.
They store and serve approved template payloads.

### 3.3 Manifest Authority

Hash-verified manifests are authoritative.

Every install or upgrade decision is bound to:

- a manifest entry approved by root
- an expected template identifier
- an expected payload hash
- an expected target role

No payload is installed without manifest-bound hash verification.

Root is the canonical manifest authority.
Template canisters may expose descriptive metadata, but that metadata is not
authoritative for install or upgrade decisions.

## 4. Non-Goals

`0.18` does not:

- preserve backward compatibility with the embedded-root payload model
- add shim layers or bridge layers
- support dual embedded/template deployment modes as a target state
- keep embedded fallback paths for ordinary deployable child templates
- treat coexistence as a desired rollout shape
- weaken root deployment authority
- weaken install/upgrade integrity checks

## 5. Canonical Model

### 5.1 Root

Root stores:

- approved template manifests
- role-to-template selection policy
- deployment/upgrade policy
- manifest approval state
- minimal bootstrap material only

Root does not store:

- the long-term payload bytes for ordinary deployable child templates

### 5.2 Template Canisters

Template canisters store:

- deployable wasm template bytes
- template metadata required for deterministic retrieval
- template chunk data when chunking is required

Template canisters expose:

- deterministic manifest lookup by template identifier
- deterministic payload retrieval by template identifier
- deterministic chunk retrieval when chunked transport is required

Template bytes are published through a root-governed approval path.
Publishing payload bytes does not make them installable until root approves the
corresponding manifest entry.

### 5.3 Workflow

Workflow owns:

- manifest resolution
- template retrieval coordination
- hash verification sequencing
- install/upgrade orchestration
- retry behavior for retrieval and install failures

### 5.4 Ops

Ops owns:

- manifest storage and retrieval
- template metadata shaping
- deterministic fetch helpers
- hash computation helpers
- install execution helpers
- template view/DTO conversion

### 5.5 Policy

Policy decides:

- which template is valid for a given role
- which manifest states are installable
- which target canisters are eligible for install/upgrade
- which bootstrap artifacts are irreducible

## 6. What Is Removed

The following are removed from the old model:

- root as long-term owner of ordinary deployable child wasm bytes
- root as the canonical byte source for ordinary child installation and upgrade
- root growth that is linear in the number of deployable child artifacts
- embedded-registry assumptions for ordinary child template distribution

The following remain:

- root deployment authority
- root manifest approval authority
- root orchestration authority

## 7. Template Manifest Model

Template manifests are authoritative deployment records.

Each manifest includes at minimum:

- `template_id`
- `role`
- `version`
- `payload_hash`
- `payload_size_bytes`
- `template_canister_id`
- `chunking_mode`
- `manifest_state`
- `approved_at`
- `created_at`

Required states:

- `Staged`
- `Approved`
- `Blocked`
- `Deprecated`

Install and upgrade use `Approved` manifests only.

## 8. Trust Model

The trust rule is fixed:

> root authorizes manifests; template canisters serve bytes; workflow verifies hash; ops executes install.

Template canisters are not trust anchors.
Manifest approval in root is the trust anchor.

Before any install or upgrade:

1. root resolves the approved manifest for the target role
2. workflow resolves the canonical template canister and template identifier
3. workflow retrieves the payload bytes deterministically
4. ops computes the payload hash deterministically
5. computed hash must equal the manifest hash
6. only then does install or upgrade proceed

If hash verification fails:

- install/upgrade fails closed
- no fallback payload source is consulted
- no alternate embedded path is used

## 9. Retrieval Semantics

Retrieval must be deterministic.

Required properties:

- stable template identity
- stable chunk order when chunked
- stable total length
- stable hash over the reconstructed payload
- stable failure behavior for missing or mismatched payloads

Required API shape:

- `get_manifest(template_id)`
- `get_template_chunk(template_id, chunk_index)` or equivalent deterministic chunk API
- `get_template_info(template_id)` for size/hash/chunk metadata

There is no "latest template" retrieval path in the install/upgrade flow.
Selection is manifest-driven, not discovery-driven.

## 10. Bootstrap and Recovery

The chosen bootstrap position is decisive:

- root retains only the minimum irreducible bootstrap artifact set
- ordinary deployable child templates are not part of that retained set

The minimum bootstrap set exists only to:

- establish template-canister availability
- seed manifest authority
- recover the control plane to the point where template-backed retrieval becomes available

Bootstrap does not preserve the old embedded-root registry as a shadow system.

Recovery uses:

- root manifest authority
- minimal bootstrap payloads
- template canister re-establishment where required

Recovery does not use:

- a full embedded copy of ordinary deployable child templates in root

## 11. Root Storage After the Cut

After `0.18`, root storage contains:

- approved manifests
- role-selection policy
- deployment state
- minimal bootstrap/recovery payloads

After `0.18`, root storage does not contain:

- the ordinary deployable payload bytes for any deployable child role that is
  not part of the irreducible bootstrap set established in `0.17`

## 12. Failure Modes

### 12.1 Manifest Missing

- install/upgrade fails closed
- no payload retrieval occurs
- operational action is manifest publication or approval repair

### 12.2 Template Payload Missing

- install/upgrade fails closed
- no alternate embedded payload is used
- operational action is template-canister repair or republish

### 12.3 Hash Mismatch

- install/upgrade fails closed
- manifest remains authoritative
- payload source is treated as invalid for that template

### 12.4 Template Canister Unavailable

- install/upgrade fails closed unless the target is part of the minimal bootstrap set and the required bootstrap payload is one of the irreducible retained artifacts
- ordinary deployable child roles do not fall back to root-owned embedded copies

### 12.5 Bootstrap Artifact Loss

- control-plane recovery uses the retained minimal bootstrap set
- loss of the retained bootstrap set is a control-plane recovery incident, not a reason to restore the old embedded-root model

## 13. Operational Implications

`0.18` changes the system topology deliberately:

- template canisters become control-plane dependencies
- root becomes an authority/orchestration canister rather than a payload warehouse
- deployment observability must include manifest resolution, payload retrieval, chunk assembly, hash verification, and install execution

Required operational metrics:

- manifest lookup success/failure
- template retrieval success/failure
- chunk retrieval retries/failures
- hash verification failures
- install/upgrade execution failures
- template canister availability

## 14. One-Way Migration Sequence

This is a cutover plan, not a compatibility design.

### Phase 1: Establish Template Canisters

- create canonical template canisters
- publish the approved template inventory
- seed root manifest authority with template locations and hashes

### Phase 2: Move Ordinary Deployable Payloads

- populate template canisters with all ordinary deployable child templates
- verify deterministic retrieval and manifest-bound hash checks
- remove those payloads from root storage and root bundle ownership

### Phase 3: Reduce Root to the Bootstrap Set

- retain only the irreducible bootstrap/recovery payload set in root
- remove all ordinary embedded child template ownership from root

### Phase 4: Remove Old Embedded Registry Assumptions

- remove root-side code paths and data structures that assume root is the long-term source of ordinary child payload bytes
- remove build assumptions that make root scale with ordinary deployable template count

End state:

- template canisters are canonical
- root is authoritative
- the old embedded-root model is gone

## 15. Layering Boundaries

The architecture must preserve Canic's ownership model:

- `policy` decides manifest validity and template selection rules
- `workflow` coordinates retrieval, verification, and install
- `ops` performs deterministic fetch/hash/install execution and owns conversions
- `model/storage` owns stored manifest and template metadata state
- endpoints remain thin request boundaries

Forbidden:

- policy reading template payload storage directly
- workflow owning manifest schema conversion
- endpoints performing retrieval orchestration
- template storage records leaking upward as public control-plane shapes

## 16. Success Criteria

`0.18` succeeds if:

- root no longer scales linearly with ordinary deployable child artifact count
- template canisters are the canonical storage/distribution layer
- every install/upgrade is manifest-bound and hash-verified
- ordinary deployable child roles no longer depend on root-owned embedded payload copies
- bootstrap/recovery operates from the minimal irreducible retained set only
- old embedded-root payload ownership is removed from the steady state

## 17. Rejected Directions

Rejected for `0.18`:

- hybrid embedded/template steady state
- embedded fallback for ordinary deployable child templates
- root retaining long-term ownership of all deployable payload bytes
- payload retrieval without manifest-bound hash verification
- compatibility shims as a long-term architecture

## 18. Narrow Remaining Questions

Only implementation-scoped questions remain:

- exact template chunk size and chunk-count limits
- exact manifest record schema names and DTO boundaries
- exact minimal bootstrap payload inventory from `0.17`
- exact retry policy for template retrieval before install failure is surfaced
