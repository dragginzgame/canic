# Canic 0.66 Design: 0.65 Audit, Testing, and Fixing

## Status

- Status: active planning note
- Release line: 0.66
- Decision style: stabilization
- Primary implementation target: prove and harden the 0.65 auth hard cut
- Current state: no feature expansion committed by this document

## 1. Summary

Canic 0.65 is the normal-auth hard cut from threshold-ECDSA token/root signing
to IC canister-signature delegated proofs and issuer proofs. Canic 0.66 is not
a new protocol feature line. It is the audit, test, and fix line for the 0.65
auth epoch.

0.66 should answer one question: can the 0.65 design be trusted, operated, and
released without hidden compatibility paths, stale assumptions, or incomplete
validation?

The endpoint perf-observability design that was previously parked on 0.66 has
moved to `docs/design/0.67-perf-observability/0.67-design.md`.

## 2. Goals

1. Run a full audit pass over the 0.65 zero-management-ECDSA normal-auth code.
2. Validate update/query prepare-get issuance for root proofs, issuer tokens,
   and role attestations.
3. Prove protected endpoint verification is local and synchronous: no root,
   issuer, or management-canister call on the hot path.
4. Prove delegated tokens remain reusable until TTL, audience, grant, subject,
   or endpoint-local checks reject them.
5. Confirm positive verifier caching is correct, bounded, and never replaces
   endpoint-specific authorization.
6. Reconcile docs, Candid sidecars, generated fixtures, changelogs, and release
   status with the implemented 0.65 surface.
7. Remove stale implementation code that is no longer part of the 0.65 design.
8. Fix defects discovered by audit and validation without adding new auth
   protocol families.

## 3. Non-Goals

0.66 should not add:

1. New delegated-auth proof families.
2. New legacy compatibility lanes.
3. One-shot token or proof issuance.
4. Management-canister ECDSA use in normal auth.
5. Endpoint perf-observability features; those are 0.67.
6. Broad anti-resurrection guard suites whose purpose is to prevent old designs
   from ever being reconsidered.

0.66 may remove stale tests that only guard against designs Canic has already
intentionally left behind.

## 4. Audit Scope

The audit should cover:

1. DTO and canonical encoding shapes for `DelegationCert`,
   `DelegatedTokenClaims`, `DelegatedToken`, `IssuerProof`, and
   `SignedRoleAttestation`.
2. Canister-signature seed, domain, verification-message, and certified-data
   construction.
3. Active delegation proof install, persistence, expiry, refresh runbooks, and
   sharded-issuer handling.
4. Root and issuer prepare/get replay, caller binding, pending proof expiry,
   and pinned-proof retrieval.
5. Verifier startup config: root canister id, raw IC root key, network/root-key
   pairing, and compile-feature requirements.
6. Local verifier checks: cert validity, token validity, audience coverage,
   grant subset, subject binding, endpoint scope, opaque signed `ext` coverage,
   and cache behavior.
7. Role-attestation issuance and verification for canister-to-canister service
   authorization.
8. Fleet stubs, PIC fixtures, generated Candid, wasm-store sidecars, docs, and
   changelog state.

## 5. Validation Plan

The release candidate should pass focused validation first, then broad
workspace validation:

```text
cargo fmt --all -- --check
cargo check --locked -p canic-core -p canic
cargo clippy --locked -p canic-core --lib -- -D warnings
cargo test --locked -p canic-core ops::auth --lib -- --nocapture
cargo test --locked -p canic-core workflow::auth --lib -- --nocapture
cargo test --locked -p canic --test protocol_surface -- --nocapture
cargo test --locked -p canic --test changelog_governance -- --nocapture
cargo check --locked -p canic-tests
make test
git diff --check
```

Long-running PocketIC suites may be split when useful, but 0.66 is not complete
until the broad suites have either passed or every remaining failure has a
specific release-blocking issue.

## 6. Fix Policy

Fixes in 0.66 should be conservative:

1. Prefer small, reviewable corrections to the 0.65 implementation.
2. Remove dead code when it is no longer in the active 0.65 design.
3. Keep token grants, opaque signed `ext`, issuer proofs, role attestations,
   and self-contained verification behavior intact.
4. Treat Candid/API changes as release-blocking unless they are necessary to
   make the 0.65 protocol coherent.
5. Do not reintroduce old ECDSA, proof-push, standalone delegated-grant, or
   one-shot issuance semantics as compatibility fallbacks.

## 7. Exit Criteria

0.66 is done when:

1. The implemented 0.65 auth surface matches the design doc and status notes.
2. The normal-auth path has zero management-canister ECDSA calls.
3. Protected endpoint verification performs no root, issuer, or management call.
4. Delegated-token and role-attestation prepare/get flows pass targeted tests.
5. Broad workspace and PocketIC validation are clean enough for release
   preparation.
6. Remaining known risks are documented as operational notes or explicit
   follow-up tickets, not hidden stale code paths.
