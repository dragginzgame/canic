# 0.52 Design: Source, Build, And Artifact Provenance

## Status

- Status: Proposed
- Line: `0.52.x`
- Theme: source, build, and artifact provenance

This is the proposed next line after 0.51. It builds on stable evidence
envelopes by recording which source state, Cargo inputs, package identity, and
produced artifacts led to a build artifact. It does not expand deployment,
controller, topology, registry, signing, lock, or adoption authority.

## Summary

0.52 should answer this question:

```text
Which source state, build inputs, package identity, and produced artifacts led
to this build evidence?
```

0.51 gave automation a stable outer contract:

```text
EvidenceEnvelopeV1
ExitClassV1
InputFingerprintV1
PayloadSchemaRefV1
```

0.52 should add stable source/build/artifact provenance payloads inside that
existing envelope contract. The line should make build artifacts easier to
audit, archive, and compare in CI without turning provenance into deployment
truth or mutation authority.

Core idea:

```text
stable envelope + stable build provenance payload
```

0.52 is not a signing, locking, import, controller mutation, artifact registry,
provider-integration, deployment, or install line.

## Motivation

0.41-0.50 built the deployment and onboarding evidence base:

- deployment truth;
- authority evidence;
- execution boundaries and receipts;
- artifact promotion evidence;
- external lifecycle evidence;
- deployment-target identity;
- verified root recovery;
- metadata-driven setup;
- fleet-scoped role lifecycle;
- passive adoption reporting.

0.51 made selected passive evidence outputs automation-friendly by adding a
stable envelope, exit classes, input fingerprints, payload schema references,
and envelope comparison.

The remaining gap is source/build/artifact provenance. A CI job can now archive
and compare evidence envelopes, but the envelope alone does not fully explain:

- which Git revision produced the artifact;
- whether the workspace was dirty;
- which `Cargo.lock` was used;
- which Rust/Cargo toolchain built artifacts;
- which package manifest and `[package.metadata.canic] fleet.role` identity
  were built;
- which build profile and target were used;
- which generated Wasm artifact hash corresponds to those source/build inputs.

0.52 should make those facts explicit and stable enough for CI/GitOps policy.

## Core Rule

`canic build <fleet> <role>` is already an active artifact-generation command.
0.52 must not expand build provenance into new deployment, install, topology,
controller, registry, lock, signing, or adoption mutation authority.

For the first slice, the only allowed writes are:

- the normal build outputs that `canic build <fleet> <role>` already writes;
- the explicitly requested provenance artifact.

Provenance output must not:

- mutate deployments;
- install or upgrade Wasm;
- attach topology;
- import adoption candidates;
- change controllers;
- register artifacts into `wasm_store` or any artifact registry;
- acquire deployment locks;
- sign or attest evidence;
- imply stale evidence is fresh;
- turn an envelope, manifest, or provenance record into deployment truth.

Provenance is evidence about inputs and outputs. It is not authority to mutate.

## Relationship To 0.51

0.52 consumes the 0.51 envelope surface instead of introducing a second
automation wrapper.

Rules:

- `EvidenceEnvelopeV1` remains the stable outer automation contract.
- Build provenance output is an `EvidenceEnvelopeV1`.
- Existing `--format json` behavior must remain command-specific/raw unless a
  command already documents otherwise.
- The build provenance payload schema is stable:

  ```text
  canic.build_provenance.v1
  ```

- A nested payload is stable only when its `PayloadSchemaRefV1` says so.
- `canic evidence compare` landed in 0.51.3 and should continue to compare
  stable envelope fields without parsing unstable payload DTOs.

## Proposed First Slice

0.52.0 defines the provenance model and adds explicit build provenance output:

```text
canic build <fleet> <role> --provenance <path>
```

The provenance output is an `EvidenceEnvelopeV1` whose payload schema is:

```text
PayloadSchemaRefV1 {
  id = "canic.build_provenance.v1"
  version = "1"
  stability = "stable"
}
```

Existing build behavior is unchanged unless `--provenance <path>` is
requested.

Provenance is written only after artifacts are successfully produced. Failure
envelopes are a later concern.

## Provenance Envelope Target

0.52.0 should use the existing 0.51 target kind:

```text
target.kind = "artifact"
target.fleet = "<fleet>"
target.role = "<role>"
```

Do not add a new `role_build` target kind in 0.52.0 unless the envelope schema
versioning plan explicitly allows extending `EvidenceTargetKindV1`. If a later
line needs `role_build`, add it deliberately with schema compatibility notes.

## CLI Shape

The first CLI shape is:

```text
canic build <fleet> <role> --provenance <path>
```

Rules:

- Ordinary builds write no provenance artifact.
- Builds with `--provenance <path>` additionally write exactly that requested
  provenance artifact.
- The command should follow existing Canic output-file conventions: create
  parent directories when the output helper already does, and replace the
  requested file according to that convention.
- Do not add `--force` in 0.52.0 unless an existing Canic output-file command
  already requires that pattern.
- Do not start with `--format envelope-json`; build is artifact-producing, not
  currently a report-only stdout command.

## Provenance Model

### `SourceProvenanceV1`

Proposed shape:

```text
SourceProvenanceV1 {
  schema_version: 1,
  vcs: "git" | "unknown",
  revision: string?,
  branch: string?,
  dirty: bool?,
  dirty_policy: "clean" | "dirty_recorded" | "unknown",
  dirty_summary_digest: string?,
  dirty_summary_algorithm: string?,
}
```

Rules:

- `revision` should be the commit hash when available.
- `dirty = true` is not necessarily a failure, but must be explicit.
- Dirty source state must never be reported as clean.
- If dirty builds are allowed, the provenance must say that dirty state was
  recorded.
- `dirty_summary_digest` is optional, but when present it must use:

  ```text
  sha256(git status --porcelain=v1 -z)
  ```

- `dirty_summary_algorithm` must be:

  ```text
  git-status-porcelain-v1-z-sha256
  ```

- The dirty summary digest is a review signal. It does not make a dirty source
  build exactly reproducible and must not be described as a source snapshot.
- Canic should not upload source diffs or file contents into provenance by
  default.

### `CargoProvenanceV1`

Proposed shape:

```text
CargoProvenanceV1 {
  cargo_lock_sha256: string?,
  package_manifest_sha256: string?,
  package_name: string,
  package_manifest: string,
  package_metadata_fleet: string,
  package_metadata_role: string,
  rustc_version: string?,
  cargo_version: string?,
  target: string?,
  profile: string,
  features: Vec<String>,
  default_features: bool?,
  rustflags_digest: string?,
  rustflags_digest_algorithm: string?,
  cargo_config_fingerprints: Vec<InputFingerprintV1>,
  build_script_inputs: "not_recorded" | "recorded" | "unknown",
}
```

0.52.0 must record at least:

- `Cargo.lock` hash when available;
- package manifest path and hash;
- package name;
- package metadata `fleet`;
- package metadata `role`;
- Rust version when available;
- Cargo version when available;
- target when available;
- build profile.

Rules:

- Paths should follow the 0.51 path-safety rule: relative to project/config
  root where practical, redacted or omitted otherwise.
- `package_metadata_fleet` and `package_metadata_role` must come from
  `[package.metadata.canic]`, not crate-name inference.
- Missing package metadata remains a hard failure for Canic-managed canister
  builds.
- Features, default-feature state, rustflags, cargo config fingerprints, and
  build-script inputs may be partial in 0.52.0, but unknowns must be explicit.

### `ArtifactProvenanceV1`

Proposed shape:

```text
ArtifactProvenanceV1 {
  role: string,
  fleet: string,
  artifact_kind: "wasm" | "wasm_gzip" | "candid" | "metadata" | "other",
  path: string?,
  path_display: "relative" | "absolute_redacted" | "omitted",
  hash_algorithm: "sha256",
  sha256: string,
  size_bytes: u64,
  produced_by: string,
}
```

Rules:

- Wasm and gzip Wasm artifacts must carry SHA-256 and byte size.
- `wasm` and `wasm_gzip` are distinct artifact kinds.
- A gzip Wasm hash is a hash of compressed bytes. It must not be presented as
  the same fact as a raw Wasm module hash.
- If both raw Wasm and gzip Wasm artifacts exist, record them separately.
- Candid/debug sidecars should be included only when they are actually emitted
  for the selected environment/profile.
- Production `ICP_ENVIRONMENT=ic` builds must not reintroduce debug Candid
  sidecar bloat.

### `BuildProvenanceV1`

Proposed shape:

```text
BuildProvenanceV1 {
  schema_version: 1,
  generated_at: string,
  canic_version: string,
  command: CommandProvenanceV1,
  build_status: "success",
  source: SourceProvenanceV1,
  cargo: CargoProvenanceV1,
  artifacts: Vec<ArtifactProvenanceV1>,
  warnings: Vec<EvidenceMessageV1>,
}
```

Rules:

- This is a stable payload in 0.52.0.
- 0.52.0 writes build provenance only after artifacts are successfully
  produced.
- Failed-build provenance envelopes can be considered later.
- Warnings are structured evidence messages, not free-form CI policy.

## Where Provenance Should Live

0.52 should avoid inventing a parallel deployment state model.

Preferred placement:

- requested provenance output: owns the enveloped `BuildProvenanceV1`;
- local artifact manifest: may reference artifact provenance summaries later;
- deployment-check envelope: may fingerprint supplied build provenance later;
- adoption report: may consume provenance as optional evidence later;
- receipts: may link to provenance digests later.

Do not store provenance in deployment truth unless the existing deployment
truth object explicitly records it as evidence. A provenance record does not
make a deployment true.

## CI/GitOps Use Cases

0.52 should support:

- archive build provenance next to envelope artifacts;
- compare produced Wasm hash against deployment-check artifact evidence;
- fail CI when artifact provenance does not match package metadata
  `fleet.role`;
- warn or fail when source is dirty, according to project policy;
- confirm `Cargo.lock`, Rust toolchain, package manifest, and build profile
  used for a produced artifact;
- trace an adoption/deployment report back to saved provenance evidence without
  running live discovery.

## Non-Goals

0.52.0 does not implement:

- signing;
- external key management;
- attestation verification;
- CI locks;
- project manifest semantics;
- provider-specific wrappers;
- artifact registry import;
- `wasm_store` retention or GC;
- active adoption/import;
- controller mutation;
- topology mutation;
- teardown;
- broad live deployment verification;
- stable schemas for every nested payload DTO;
- making dirty workspaces fail by default;
- failed-build provenance envelopes.

Later 0.52 slices may consider saved provenance evidence inputs, artifact
manifest linkage, or CI policy guidance. Signing, locks, provider wrappers,
and project manifest semantics should remain future lines unless explicitly
promoted.

## Patch Plan

Tentative patch plan:

- `0.52.0`: `BuildProvenanceV1`, `SourceProvenanceV1`,
  `CargoProvenanceV1`, and `ArtifactProvenanceV1`; add
  `canic build <fleet> <role> --provenance <path>`; output an
  `EvidenceEnvelopeV1` containing stable `canic.build_provenance.v1` payload.
- `0.52.1`: connect build provenance to local artifact manifests or build
  reports without changing deploy/install behavior.
- `0.52.2`: allow deployment-check/adoption envelopes to fingerprint supplied
  provenance evidence.
- `0.52.3`: CI policy docs and examples for dirty source state, Cargo lock
  drift, package metadata identity, and artifact hash matching.
- `0.52.4`: closeout audit.

This plan should shrink rather than grow.

## Required Tests

Minimum tests before closing 0.52.0:

- `canic build <fleet> <role> --provenance <path>` writes exactly one requested
  provenance file in addition to ordinary build outputs.
- Ordinary `canic build <fleet> <role>` writes no provenance file.
- Existing build artifact outputs are unchanged unless `--provenance` is
  requested.
- Provenance file is `EvidenceEnvelopeV1`.
- `payload_schema.id == "canic.build_provenance.v1"`.
- `payload_schema.stability == "stable"`.
- Envelope target uses `artifact` and identifies fleet and role.
- Declared-only roles are still rejected before build/provenance.
- Package metadata fleet/role mismatch fails as before.
- Package metadata fleet and role are read from `[package.metadata.canic]`.
- Package metadata is not inferred from the package name.
- `SourceProvenanceV1` records a clean Git revision when available.
- Dirty source state is recorded and never treated as clean.
- Missing Git metadata maps to `unknown` rather than failing unrelated builds.
- Dirty summary digest, when present, uses
  `git-status-porcelain-v1-z-sha256`.
- `CargoProvenanceV1` records `Cargo.lock` hash when available.
- Package manifest path and hash are recorded.
- Cargo and Rust versions are recorded when available.
- Artifact provenance records SHA-256 and byte size for produced Wasm/gzip
  Wasm.
- Gzip Wasm and raw Wasm hashes are recorded separately when both exist.
- Paths are relative or redacted according to the 0.51 path-display rule.
- No absolute home-directory path appears in snapshotable provenance output.
- Production `ICP_ENVIRONMENT=ic` behavior does not regain debug sidecar bloat.
- Build provenance output does not mutate deployment state, topology,
  controllers, `wasm_store`, artifact registry, adoption state, install state,
  or canister lifecycle.

## Documentation Requirements

0.52 docs should explain:

- the difference between build provenance and deployment truth;
- why `EvidenceEnvelopeV1` remains the automation wrapper;
- why `canic.build_provenance.v1` is stable;
- how dirty source state is represented;
- why dirty-state digest does not prove exact source reproducibility;
- why timestamps alone are not provenance;
- how package metadata `fleet.role` participates in provenance;
- how CI should use artifact hashes without parsing unstable payload DTOs;
- why signing, CI locks, and project manifests are deferred.

## Acceptance Criteria

0.52.0 is acceptable when:

- `BuildProvenanceV1` exists as a stable payload schema.
- `canic build <fleet> <role> --provenance <path>` writes an
  `EvidenceEnvelopeV1` containing `BuildProvenanceV1`.
- Source provenance records Git revision when available and explicit unknowns
  when unavailable.
- Dirty source state is recorded and never treated as clean.
- Cargo provenance records `Cargo.lock` hash, package manifest identity,
  package metadata fleet/role, Cargo/Rust version, target, and profile.
- Artifact provenance records SHA-256 and byte size for produced Wasm/gzip
  Wasm.
- Paths follow the 0.51 path-normalization/redaction rules.
- Existing build behavior is unchanged unless `--provenance` is requested.
- No deployment, controller, topology, registry, lock, adoption, install, or
  canister lifecycle mutation is introduced.

Unknowns are acceptable only when the environment truly cannot provide a fact;
stubbed collection is not acceptable for required 0.52.0 fields.

## Closeout Wording

Preferred closeout wording:

```text
0.52 expands Canic's evidence story from stable envelopes into source, build,
and artifact provenance. A build can record which source revision, dirty-state
summary, Cargo inputs, package metadata identity, build profile, toolchain, and
produced artifact hashes led to the archived build evidence.

The provenance output is an EvidenceEnvelopeV1 containing stable
BuildProvenanceV1. It does not sign artifacts, acquire locks, publish to a
registry, mutate controllers, install Wasm, or become deployment truth.
```
