# Canic 0.69 Design: Blob Storage Billing

## Status

- Status: draft design note
- Release line: 0.69 candidate
- Decision style: backend billing extraction
- Primary implementation target: Cashier integration for blob storage
- Current state: depends on 0.68 blob storage protocol foundation

## 1. Summary

0.69 adds backend Cashier integration for Canic blob storage. It consumes the
0.68 protocol inventory for the two deferred gateway endpoints, adds typed
Cashier wrappers, syncs gateway principals into the 0.68 store, implements
explicit project-as-payment-account funding, and exposes backend billing
status.

0.69 is backend-only. It does not add frontend billing cards, upload UI,
TypeScript clients, npm packages, CLI wrappers, or Toko backend migration.

## 2. Boundaries

| Release | Owns |
| --- | --- |
| 0.68 | Protocol inventory for all six `_immutableObjectStorage*` methods, four non-billing emitted endpoints, blob lifecycle state, stable records, gateway-principal storage/checks, protocol Candid snapshots. |
| 0.69 | Cashier integration, gateway-principal sync from Cashier, `_immutableObjectStorageUpdateGatewayPrincipals`, `_immutableObjectStorageFundFromProjectCycles`, project-as-payment-account funding, project-cycle reserve policy, backend billing status. |
| 0.70 | Toko adoption, Toko endpoint replacement, Toko Cashier helper replacement, Toko stable migration or no-state-move decision, Toko Candid diffing, mainnet manual verification. |

## 3. Goals

1. Add `blob-storage-billing`, off by default and dependent on `blob-storage`.
2. Add a Cashier inventory artifact before billing implementation merges.
3. Implement typed Cashier wrappers for balance, top-up, and gateway-principal
   list.
4. Sync Cashier gateway principals atomically into the 0.68 store.
5. Implement project-as-payment-account funding with reserve preservation.
6. Expose the two 0.69 gateway endpoints using the 0.68 Candid snapshots.
7. Expose `get_blob_storage_status` as a Canic-defined backend update endpoint
   with Candid snapshots and no cycle transfers.
8. Add mock Cashier PocketIC tests for sync, status, funding, failures, and
   authorization.

## 4. Non-Goals

0.69 does not:

1. Define new Candid for `_immutableObjectStorageUpdateGatewayPrincipals`.
2. Define new Candid for `_immutableObjectStorageFundFromProjectCycles`.
3. Reimplement the Cashier canister.
4. Migrate Toko backend code.
5. Add frontend billing cards, upload UI, TypeScript clients, npm packages, or
   CLI wrappers.
6. Call production Cashier in CI.
7. Auto-fund from `get_blob_storage_status`.
8. Fully implement separate payment-account linking or funding.

## 5. Feature Flag

Add:

```toml
blob-storage-billing = ["blob-storage"]
```

The feature must be off by default.

`blob-storage-billing` is required for:

1. Cashier DTOs.
2. Cashier call wrappers.
3. Billing config.
4. Billing workflows.
5. Billing endpoint macro.
6. Billing tests.

The 0.68 `blob-storage` feature alone must not emit billing or sync endpoints.

## 7. Inventory Gates

### 7.1 Cashier Inventory

The Cashier inventory is a hard pre-implementation blocker. No Cashier DTO,
wrapper, billing workflow, billing macro, or billing test may merge before this
artifact exists:

```text
docs/contracts/BLOB_STORAGE_CASHIER_INVENTORY.md
```

The artifact must include:

1. Production Cashier canister ID.
2. Source repo, generated `.did`, or deployment provenance.
3. Commit SHA or equivalent source identifier.
4. Exact Candid for `account_balance_get_v1`.
5. Exact Candid for `account_top_up_v1`.
6. Exact Candid for `storage_gateway_principal_list_v1`.
7. Method modes.
8. Request DTO shapes.
9. Response DTO shapes.
10. Result/error variants.
11. Trap/reject behavior.
12. Cycle-attachment requirements for `account_top_up_v1`.
13. Balance units and integer width.
14. Empty gateway-principal list behavior.
15. Malformed response behavior.
16. Optional Cashier methods and why each is implemented or deferred.
17. Non-blocking notes for 0.70 Toko readiness.

### 7.2 Gateway Protocol Inventory

0.69 must not invent Candid for:

1. `_immutableObjectStorageUpdateGatewayPrincipals`
2. `_immutableObjectStorageFundFromProjectCycles`

Those endpoint signatures must come from the 0.68
`docs/contracts/BLOB_STORAGE_INVENTORY.md` snapshots.

0.69 may define internal sync/funding summaries, but gateway-facing DTOs must
match the 0.68 inventory exactly.

## 8. Surface and DTO Ownership

| Surface | DTO source |
| --- | --- |
| `_immutableObjectStorageUpdateGatewayPrincipals` | 0.68 gateway inventory |
| `_immutableObjectStorageFundFromProjectCycles` | 0.68 gateway inventory |
| Cashier methods | Cashier inventory |
| Internal sync/funding summaries | Canic internal DTOs |
| `get_blob_storage_status` | Canic-defined backend DTO with Candid snapshot |

External gateway DTOs are separate from internal Cashier and billing models.
Internal models must not alter gateway-facing Candid.

## 9. Endpoint Topology

| Endpoint | Canister | Reason |
| --- | --- | --- |
| `_immutableObjectStorageUpdateGatewayPrincipals` | owner/project canister | Mutates the gateway-principal store used by 0.68 gateway-only checks. |
| `_immutableObjectStorageFundFromProjectCycles` | owner/project canister | Transfers that canister's cycles to Cashier. |
| `get_blob_storage_status` | owner/project canister by default | Reads owner/project billing state and may optionally sync gateway principals. It never transfers cycles. |

| Endpoint | Mode | Candid source |
| --- | --- | --- |
| `_immutableObjectStorageUpdateGatewayPrincipals` | inventory-defined, expected update | 0.68 gateway inventory |
| `_immutableObjectStorageFundFromProjectCycles` | inventory-defined, expected update | 0.68 gateway inventory |
| `get_blob_storage_status` | update | 0.69 Canic snapshot |

Hub canisters may aggregate by calling project canisters. A hub status endpoint
must not introduce funding behavior unless a separate design explicitly makes
the hub the payment actor.

Macro/Candid tests must verify method modes.

## 10. Authorization and Macro Shape

Existing Canic endpoint macros use `guard = <access expression>` for a single
guard. 0.69 needs multiple guards, so it should extend that convention with
explicit named guard arguments and compile-error tests:

```rust
canic_emit_blob_storage_billing_endpoints!(
    sync_gateway_principals_guard = <expr>,
    fund_from_cycles_guard = <expr>,
    status_guard = <expr>,
);
```

Guard semantics:

1. `sync_gateway_principals_guard` protects
   `_immutableObjectStorageUpdateGatewayPrincipals`.
2. `fund_from_cycles_guard` protects
   `_immutableObjectStorageFundFromProjectCycles`.
3. `status_guard` authorizes reading `get_blob_storage_status`.
4. If status performs gateway-principal sync, the caller must also satisfy
   `sync_gateway_principals_guard`.
5. Status never invokes funding. `fund_from_cycles_guard` protects only the
   explicit `_immutableObjectStorageFundFromProjectCycles` endpoint.
6. Status must return a structured skipped/unauthorized sync action instead of
   silently mutating when the caller lacks `sync_gateway_principals_guard`.

## 11. Status Request and Response

Status is an update endpoint. It may perform IC calls and, only when requested
and authorized, mutate gateway-principal sync state. It must never attach
cycles or call `account_top_up_v1`.

Request:

```rust
BlobStorageStatusRequest {
    sync_gateway_principals: bool,
}
```

Funding readiness is report-only. Callers that want to fund must invoke the
explicit `_immutableObjectStorageFundFromProjectCycles` endpoint.

Status response uses stable machine-readable variants. Human-readable messages
may be included, but variants are the compatibility surface.

```rust
struct BlobStorageStatusResponse {
    payment_model: BlobStoragePaymentModelStatus,
    cashier_canister_id: Option<Principal>,
    payment_account: Option<Principal>,
    cashier_balance: Option<CyclesAmount>,
    min_upload_balance: Option<CyclesAmount>,
    target_upload_balance: Option<CyclesAmount>,
    project_cycles_reserve: Option<CyclesAmount>,
    gateway_principal_count: u32,
    last_gateway_principal_sync_at: Option<Timestamp>,
    gateway_principal_sync_action: BlobStorageGatewayPrincipalSyncAction,
    funding_status: BlobStorageFundingStatus,
    readiness: BlobStorageReadinessReport,
    warnings: Vec<BlobStorageBillingWarning>,
}

enum BlobStorageGatewayPrincipalSyncAction {
    NotRequested,
    SkippedUnauthorized,
    SkippedConfigMissing,
    SkippedCashierUnavailable,
    Succeeded { principal_count: u32 },
    Failed { reason: BlobStorageBillingErrorCode },
}

enum BlobStoragePaymentModelStatus {
    NotConfigured,
    InvalidConfig,
    ProjectAsPaymentAccount,
    SeparatePaymentAccountUnsupported,
}

struct BlobStorageReadinessReport {
    ready: bool,
    blockers: Vec<BlobStorageReadinessBlocker>,
}

enum BlobStorageReadinessBlocker {
    NotConfigured,
    InsufficientCashierBalance,
    GatewayPrincipalsMissing,
    GatewayPrincipalSyncStale,
    CashierUnavailable,
    FundingRequired,
    ReserveWouldBeViolated,
    UnsupportedPaymentModel,
}

enum BlobStorageFundingStatus {
    NotNeeded,
    FundingRequired { amount: CyclesAmount },
    BalanceUnavailable,
    UnsupportedPaymentModel,
    ReserveWouldBeViolated,
}

enum BlobStorageBillingWarning {
    GatewayPrincipalSyncFailed,
    GatewayPrincipalSetEmpty,
    BalanceReadFailed,
    SeparatePaymentAccountPartialSupport,
    CashierInventoryIncomplete,
}
```

Config-derived response fields are `None` when billing config is missing or
invalid. Do not emit fake principals or zero-value cycle thresholds.

The `get_blob_storage_status` request, response, public error-code enum, and
all nested public status variants must be Candid-snapshotted.

## 12. Billing Configuration

Use stable-safe cycle representation, for example:

```rust
struct CyclesAmount(u128);
```

An existing Canic `Cycles` type may be used only if it is stable-serialization
safe.

Config:

```rust
BlobStorageBillingConfig {
    cashier_canister_id: Principal,
    payment_model: BlobStoragePaymentModel,
    project_cycles_reserve: CyclesAmount,
    min_upload_balance: CyclesAmount,
    target_upload_balance: CyclesAmount,
    gateway_principal_sync_stale_after: Option<Duration>,
    created_at: Timestamp,
    updated_at: Timestamp,
}

BlobStoragePaymentModel {
    ProjectAsPaymentAccount,
    SeparatePaymentAccount {
        payment_account: Principal,
        account_owner: Option<Principal>,
        paid_canister: Option<Principal>,
        paid_canister_limit: Option<CyclesAmount>,
    },
}
```

Validation:

1. `min_upload_balance <= target_upload_balance`.
2. `project_cycles_reserve > 0`.
3. `cashier_canister_id` is not anonymous or the management canister.
4. V1 funding is supported only for `ProjectAsPaymentAccount`.
5. Project-as-payment-account derives the payment account from the current
   project canister principal or receives it through a safe builder.
6. Separate payment accounts are represented for truthful status only and must
   report `UnsupportedPaymentModel` until linking and funding are explicitly
   designed.
7. `gateway_principal_sync_stale_after = None` means staleness is not reported.

If `Duration` enters stable config, it must use a stable-safe encoded
representation.

0.69 provides internal Canic APIs for initializing, reading, validating, and
updating billing config. It does not automatically expose public configuration
endpoints. Consumer canisters may expose their own admin endpoints with their
own authorization policy.

## 13. Production Configuration

Production Cashier ID:

```text
72ch2-fiaaa-aaaar-qbsvq-cai
```

Rules:

1. No `Default` implementation may point to production Cashier.
2. Tests must inject a mock Cashier canister ID.
3. Tests must fail if production Cashier is accidentally used.
4. `production_project_account()` must be explicitly named.
5. `production_project_account()` must either take the project canister
   principal or derive it only in endpoint/workflow code where IC environment
   access is appropriate.

## 14. Gateway Principal Sync

Sync policy:

1. A successful Cashier sync atomically replaces the Cashier-sourced
   gateway-principal subset in the 0.68 store.
2. Per-principal `last_seen_at` and global sync metadata are recorded.
3. If the Cashier call fails, traps, rejects, returns malformed Candid, or
   returns invalid principals, the previous principal set is preserved.
4. Empty list is a sync error that preserves the previous set unless the
   Cashier inventory proves empty list is valid production behavior.
5. Test/local seeded principals must not leak into production sync behavior.
6. Cashier-returned gateway principals are deduplicated before write.
7. Duplicate principals are not errors unless the Cashier inventory requires
   that behavior.
8. The synced set has a Canic maximum size to bound stable-memory and response
   costs, unless the Cashier inventory defines a lower service/protocol limit.
9. Anonymous and management-canister principals are rejected unless inventory
   proves they are valid.

Production effective gateway-principal authorization must not depend on
`TestSeeded` principals. Local/test builds may authorize the union of
`CashierSynced` and `TestSeeded` principals only when explicitly configured.

The sync workflow writes through the 0.68 gateway-principal storage API. 0.69
must not duplicate or fork 0.68 gateway-only authorization logic.

## 15. Funding Policy and Cycle Math

Funding policy is pure. It receives observed facts and returns a decision.
Workflow performs IC calls after policy approval.

Required policy:

1. No top-up when Cashier balance is at or above `min_upload_balance`.
2. Top-up amount is `target_upload_balance - balance`.
3. Top-up is refused if it would drop project cycles below
   `project_cycles_reserve` after estimated call overhead.
4. No partial top-up unless explicitly designed.
5. Checked or saturating arithmetic only.
6. Exact cycle attachment follows the Cashier inventory.
7. No cycles are attached on no-op or reserve refusal.

Add a funding in-flight single-flight guard so concurrent calls cannot
double-top-up from the same stale observed balance. The guard must release on
success, rejection, trap, or decode failure. For v1 the guard is transient and
must not be persisted; an upgrade starts with no funding lock set.

`_immutableObjectStorageFundFromProjectCycles` is the only v1 path that can
transfer cycles. It invokes the funding policy, is explicitly requested, and is
protected by `fund_from_cycles_guard`. 0.69 supports it only for
`ProjectAsPaymentAccount`. Separate-payment-account behavior must be reported
as unsupported/deferred using the inventoried protocol mapping.

The endpoint returns the inventoried 0.68 gateway protocol response. Do not
expose internal `BlobStorageFundingStatus` from this gateway endpoint unless
the 0.68 inventory proves that is the protocol response.

The top-up amount comes from the 0.68 gateway inventory if that protocol request
contains an amount. If the protocol request does not contain an amount, 0.69
uses the configured target-balance policy: `target_upload_balance - balance`.

## 16. Stable Memory and Upgrade Requirements

Use existing Canic stable-record conventions.

Required planning:

1. Billing config namespace.
2. Schema versioning.
3. Upgrade round-trip tests.
4. Preservation of billing config.
5. Preservation of gateway principals through the 0.68 store.
6. Preservation of last successful sync metadata.
7. No persistence of transient balances or the transient funding lock.
8. Bounded funding history only if explicitly added after v1.
9. Safe migration from `blob-storage` enabled to `blob-storage-billing`
   enabled.

## 17. Internal Billing Errors and Mapping

Internal error taxonomy:

1. `BillingConfigMissing`
2. `InvalidBillingConfig`
3. `UnsupportedPaymentModel`
4. `SeparatePaymentAccountLinkingDeferred`
5. `CashierCallRejected`
6. `CashierCallTrapped`
7. `CashierDecodeFailed`
8. `CashierUnexpectedResponse`
9. `GatewayPrincipalSyncFailed`
10. `GatewayPrincipalListEmpty`
11. `GatewayPrincipalListInvalid`
12. `UnauthorizedBillingStatus`
13. `UnauthorizedGatewayPrincipalSync`
14. `UnauthorizedFunding`
15. `InsufficientProjectCycles`
16. `ReserveWouldBeViolated`
17. `FundingAlreadyInProgress`
18. `TopUpFailed`
19. `ProductionCashierUsedInTest`

Mapping rules:

1. Gateway-facing endpoint errors map to the exact 0.68-inventoried protocol
   responses.
2. Cashier call errors stay internal unless exposed through Canic-defined
   status.
3. `get_blob_storage_status` may expose Canic-defined structured errors and
   warnings, but it must have Candid snapshots.
4. `BlobStorageBillingErrorCode` is the stable public/status error-code enum.
   Internal `BlobStorageBillingError` values map into this enum. The public code
   enum must be Candid-snapshotted and should not expose implementation-only
   details unless intentionally part of the status contract.

## 18. 0.70 Readiness Follow-Up

0.69 v1 does not block on Toko migration readiness. Before 0.70 implementation
starts, compare Toko's direct Cashier call inventory against 0.69 wrapper
coverage.

The follow-up artifact is:

```text
docs/contracts/TOKO_CASHIER_CALL_INVENTORY.md
```

This artifact is deferred until before 0.70 implementation starts. It is not a
0.69 v1 merge blocker.

Any direct Cashier call needed for 0.70 must be:

1. Implemented as a typed 0.69 wrapper.
2. Explicitly classified as Toko-domain-specific and retained in Toko.
3. Documented as deferred or blocking for 0.70 implementation.

This prevents 0.70 from discovering too late that Canic 0.69 is not sufficient
to replace Toko billing helpers.

## 20. Merge Checks

0.69 is mergeable only when:

1. `blob-storage-billing` exists, is off by default, and implies
   `blob-storage`.
2. Cashier inventory artifact exists.
3. Required Cashier method Candid is snapshotted.
4. 0.69 gateway endpoint Candid matches 0.68 inventory.
5. Typed Cashier wrappers cover balance, top-up, and gateway-principal list.
6. Billing config is stable-safe and validated.
7. Project-as-payment-account funding is fully implemented.
8. Separate payment account is represented and truthfully reported as
   unsupported/deferred.
9. Gateway-principal sync is atomic and failure-preserving.
10. Funding preserves project cycle reserve and prevents concurrent
    double-top-up.
11. Status endpoint has explicit sync authorization and never transfers cycles.
12. Status response uses structured sync action, readiness report, funding
    status, public error codes, and warnings.
13. Mock Cashier PocketIC tests cover sync, status, funding, failures, and
    unauthorized callers.
14. Production Cashier is never used in CI.

## 21. Deferred Work

0.70 migrates Toko backend to the 0.68/0.69 Canic backend surface.

Deferred beyond 0.69 v1:

1. Status-triggered auto-funding.
2. Separate payment-account linking and funding workflows.
3. `docs/contracts/TOKO_CASHIER_CALL_INVENTORY.md` before 0.70
   implementation starts.

Frontend billing UI, upload clients, TypeScript packages, npm packages, CLI
wrappers, and browser workflows remain outside this backend line.
