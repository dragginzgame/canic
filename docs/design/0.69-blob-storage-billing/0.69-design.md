# Canic 0.69 Design: Blob Storage Billing

## Status

- Status: draft design note
- Release line: 0.69 candidate
- Decision style: backend billing extraction
- Primary implementation target: Cashier integration for blob storage
- Current state: depends on 0.68 blob storage protocol foundation

## 1. Summary

0.69 adds backend Cashier integration for Canic blob storage. It keeps the
0.68 protocol endpoints pinned to the Caffeine gateway contract while adding
typed Cashier calls, gateway-principal sync, project-as-payment-account
funding, and a backend billing status endpoint.

This is still backend-only. No frontend client, npm package, browser upload
workflow, or UI migration is included.

## 2. Problem

The gateway protocol needs billing state and gateway-principal data from the
Cashier canister. Toko currently handles this with application-specific code.
Canic should provide reusable backend primitives and workflows so consumers do
not hand-roll Cashier Candid calls or duplicate billing threshold logic.

## 3. Goals

1. Add typed Cashier DTOs and IC call wrappers.
2. Sync gateway principals from Cashier into the 0.68 gateway-principal store.
3. Fully support project-as-payment-account billing.
4. Add guarded project-cycle top-up for blob storage.
5. Add a backend `get_blob_storage_status` update endpoint for hub-style
   canisters.
6. Preserve an owner/project canister cycle reserve before top-up.
7. Use mock Cashier canisters in PocketIC tests.
8. Keep separate payment-account support explicit but mostly documented unless
   the typed wrappers are already cheap.

## 4. Non-goals

0.69 does not:

1. Implement frontend billing cards or upload gating.
2. Publish a TypeScript package.
3. Migrate Toko backend code.
4. Reimplement the Cashier canister.
5. Call production Cashier in CI.
6. Fully automate paid-canister linking unless the inventory shows the surface
   is small and stable.
7. Add CLI wrappers.

## 5. External Services

Production Cashier canister:

```text
72ch2-fiaaa-aaaar-qbsvq-cai
```

The production canister ID should be provided through an explicit production
configuration helper, not hidden in test defaults. Tests must inject a mock
Cashier canister ID.

Cashier Candid details belong only in `ops/cashier`. Blob storage workflows
consume typed Rust operations and typed errors.

## 7. Configuration

Add `BlobStorageBillingConfig`:

- `cashier_canister_id: Principal`
- `payment_model: BlobStoragePaymentModel`
- `payment_account_owner: Option<Principal>`
- `project_cycles_reserve: Cycles`
- `min_upload_balance: Cycles`
- `target_upload_balance: Cycles`
- `created_at: Timestamp`
- `updated_at: Timestamp`

Default production values should be available through:

```rust
BlobStorageBillingConfig::production_project_account()
```

Tests should use explicit mock constructors.

Initial threshold defaults:

- `project_cycles_reserve`: 2T cycles
- `min_upload_balance`: 0.5T cycles
- `target_upload_balance`: 1T cycles

## 8. Cashier Client

Minimum typed methods:

1. `account_balance_get_v1`
2. `account_top_up_v1`
3. `storage_gateway_principal_list_v1`

Optional typed methods:

1. `pricelist_v1`
2. `pricelist_get_v1`
3. payment-account delegation methods
4. paid-canister linking methods

The minimum methods are required for 0.69. Optional methods should not block
the release line unless Toko backend migration proves they are needed.

## 9. Payment Models

### 9.1 Project-As-Payment-Account

This is the v1 supported model.

Rules:

1. The owner/project canister principal is the Cashier payment account.
2. No paid-canister relationship is configured when Cashier forbids
   `paid_canister == payment_account`.
3. Billing readiness depends on Cashier balance for the owner/project
   principal.
4. `_immutableObjectStorageFundFromProjectCycles` transfers cycles from the
   owner/project canister into Cashier.
5. Funding preserves `project_cycles_reserve`.

### 9.2 Separate Payment Account

0.69 should model this in DTO/config so future consumers do not need a schema
break:

1. A separate payment account is topped up externally.
2. The owner canister may be linked as a paid canister with a limit.
3. Status reads can report that the model is configured.
4. Full account creation/linking wrappers may be deferred.

## 10. Gateway Principal Sync

Workflow:

1. Call Cashier `storage_gateway_principal_list_v1`.
2. Validate returned principals.
3. Replace or reconcile the stored gateway-principal set.
4. Record `last_seen_at` and source Cashier canister ID.
5. Return a typed sync summary.

The sync endpoint should be guard-configurable. Even if Cashier is the source
of truth, the operation mutates authorization-relevant local state.

## 11. Funding Workflow

Workflow:

1. Guard authorizes funding.
2. Read current owner/project canister cycles.
3. Read Cashier balance for the configured payment account.
4. If balance is at or above `min_upload_balance`, return no-op status.
5. Compute top-up amount to reach `target_upload_balance`.
6. Refuse top-up if it would drop the owner/project canister below
   `project_cycles_reserve`.
7. Call Cashier top-up.
8. Return typed status with pre/post observed balances when available.

Funding policy should live under `policy/blob_storage/billing.rs`. Workflow
passes facts into policy and performs IC calls through ops.

## 12. Endpoint Macros

Extend backend macros with billing-specific emission:

```rust
canic_emit_blob_storage_billing_endpoints!(
    sync_gateway_principals_guard = <expr>,
    fund_from_cycles_guard = <expr>,
    status_guard = <expr>,
);
```

Emitted endpoints:

| Endpoint | Mode | Guard | Purpose |
| --- | --- | --- | --- |
| `_immutableObjectStorageUpdateGatewayPrincipals` | update | `sync_gateway_principals_guard` | Sync gateway principals from Cashier. |
| `_immutableObjectStorageFundFromProjectCycles` | update | `fund_from_cycles_guard` | Transfer cycles from owner/project canister into Cashier. |
| `get_blob_storage_status` | update | `status_guard` | Read/sync billing status and optionally auto-fund. |

`get_blob_storage_status` must be an update because it performs
inter-canister calls and may mutate gateway-principal/funding state.

## 13. Status Workflow

`get_blob_storage_status` should:

1. Sync gateway principals.
2. Read Cashier balance.
3. Evaluate readiness against `min_upload_balance`.
4. If configured and authorized, top up to `target_upload_balance`.
5. Return structured backend status for operators and callers.

Status DTO should include:

- `payment_model`
- `cashier_canister_id`
- `payment_account`
- `balance`
- `min_upload_balance`
- `target_upload_balance`
- `project_cycles_reserve`
- `gateway_principal_count`
- `synced_at`
- `funding_action`
- `readiness`
- `warnings`

## 14. Tests

Unit tests:

1. Billing threshold policy.
2. Reserve-preservation math.
3. Cashier DTO mapping.
4. Funding refusal when reserve would be violated.
5. Gateway principal reconciliation.

PocketIC tests:

1. Mock Cashier returns gateway principals.
2. Sync writes gateway principals used by 0.68 gateway-only checks.
3. Status reads balance.
4. Status auto-funds when balance is low and reserve permits.
5. Status refuses funding when reserve would be violated.
6. Unauthorized callers cannot sync or fund.
7. Production Cashier is never called in CI.

Macro tests:

1. Billing macro emits expected methods.
2. Missing guards fail clearly.
3. `get_blob_storage_status` is update-only.

## 15. Acceptance Criteria

0.69 is complete when:

1. Typed Cashier calls cover balance, top-up, and gateway-principal list.
2. Gateway-principal sync populates the 0.68 store.
3. Project-as-payment-account billing is fully implemented.
4. Funding preserves owner/project canister reserve.
5. Backend billing status is exposed as an update endpoint.
6. Mock Cashier PocketIC tests cover sync, status, and funding.
7. Separate payment-account model is represented in config and documented as
   partial/deferred if wrappers are not implemented.

## 16. Deferred Work

0.70 migrates Toko backend to the 0.68/0.69 Canic backend surface.

Frontend billing UI, upload clients, and npm packages remain outside this
backend line.
