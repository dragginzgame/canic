# Canic 0.67 Tentative Design: Endpoint Perf Observability

## Status

- Status: tentative design note
- Release line: 0.67 candidate
- Decision style: exploratory
- Primary implementation target: endpoint performance debugging ergonomics
- Current state: no implementation committed by this document

## 1. Summary

Canic already has useful performance instrumentation:

1. `#[canic_query]` and `#[canic_update]` route endpoint calls through dispatch
   wrappers that record endpoint-level instruction totals.
2. `perf!()` records manual checkpoint deltas into the shared runtime metrics
   table.
3. `MetricsQuery::sample_query(...)` can return same-call query instruction
   samples for query endpoints, where normal metrics writes do not commit.
4. `canic_metrics(MetricsKind::Runtime, ...)` exposes persisted perf rows when
   the deployed canister's metrics profile enables the runtime tier.

The issue is not total absence of instrumentation. The issue is that endpoint
debugging still requires too much local knowledge:

- checkpoint baselines are implicit
- query profiling differs from update profiling
- temporary checkpoint labels can pollute stable runtime metrics
- operators have to inspect broad runtime metrics output to find perf rows
- `count + total` is enough for aggregate trend checks but weak for debugging a
  single slow endpoint

0.67 should make endpoint perf debugging first-class without turning Canic into
a full tracing system.

## 2. Problem

The current manual debugging loop is roughly:

```rust
#[canic::canic_update]
async fn endpoint(req: Request) -> Result<Response, canic::Error> {
    canic::perf!("start");

    let state = load_state(req)?;
    canic::perf!("load_state");

    let result = execute(state).await?;
    canic::perf!("execute");

    Ok(result)
}
```

Then an operator calls:

```bash
canic --network <network> info metrics <deployment> --kind runtime --nonzero
```

This works, but it has several sharp edges.

### 2.1 First checkpoint ambiguity

`perf!()` uses a thread-local `PERF_LAST` snapshot. That is simple and cheap,
but it means the first checkpoint in an endpoint is not self-describing. In a
new call context, the first useful label is often a manual `"start"` marker
whose only purpose is to reset the baseline for later labels.

This makes ad hoc debugging fragile:

- forgetting the start marker shifts every following label
- inserting a checkpoint changes the meaning of later checkpoints
- moving code across helper boundaries can make labels harder to compare

### 2.2 Ordered checkpoint semantics are easy to misuse

Checkpoint rows represent "instructions since the previous `perf!()` in this
thread", not "instructions spent by this block".

That distinction matters. These two edits do not mean the same thing:

```rust
canic::perf!("before_parse");
let parsed = parse(input)?;
canic::perf!("after_parse");
```

```rust
let parsed = parse(input)?;
canic::perf!("parse");
```

The second form records everything since the prior checkpoint, not only
`parse(input)`.

### 2.3 Query endpoints need a separate pattern

Update endpoint perf rows are persisted as metrics. Query endpoint state writes
do not commit, so ordinary metric rows are not a reliable transport for query
profiling. The current supported pattern is
`MetricsQuery::sample_query(value)`, which is correct but discoverability is
poor and the endpoint return type must be changed manually.

### 2.4 Debug labels can become public metric cardinality

Manual checkpoint labels are exported as runtime metrics labels. That is useful
for audits and coarse regression tracking, but temporary labels such as
`"before_loop_2"` or per-user labels can pollute metrics and create accidental
cardinality growth.

### 2.5 Runtime metrics are too broad for this workflow

Perf rows live under the runtime metrics tier. Operators often need only:

```text
perf endpoint update my_endpoint
perf checkpoint my_module my_label
```

Today they have to query runtime metrics and scan or post-process the output.

## 3. Goals

1. Make the first endpoint checkpoint meaningful without requiring a manual
   `"start"` marker.
2. Add block/span-style instrumentation so labels can mean "this block cost X".
3. Keep endpoint aggregate metrics stable and low-cardinality.
4. Preserve cheap instruction-counter based instrumentation.
5. Keep query profiling explicit and useful.
6. Provide a CLI readout focused on perf rows.
7. Avoid requiring a distributed tracing backend.
8. Avoid changing endpoint authorization, replay, or lifecycle semantics.

## 4. Non-goals

0.67 should not attempt to:

1. Measure cycle cost directly.
2. Attribute instructions spent by remote canisters.
3. Replace existing metrics tiers.
4. Add per-request durable traces for every production call by default.
5. Preserve every temporary debug checkpoint forever.
6. Build a general observability platform.

## 5. Current Model

### 5.1 Endpoint totals

Macro-generated endpoints call dispatch wrappers. Dispatch enters an endpoint
perf scope before the handler and exits it afterward.

The resulting runtime metric label shape is:

```text
perf endpoint <call_kind> <endpoint_name>
```

Examples:

```text
perf endpoint query canic_env
perf endpoint composite_query read_remote_state
perf endpoint update mint_token
```

These rows are aggregate counters:

```text
count = number of recorded calls
value_u64 = total local instructions
```

### 5.2 Manual checkpoints

`perf!()` records:

```text
perf checkpoint <module_path> <label>
```

The value is the instruction delta since the previous checkpoint in the same
thread-local checkpoint stream.

### 5.3 Query samples

`MetricsQuery::sample_query(value)` returns:

```rust
pub struct QueryPerfSample<T> {
    pub value: T,
    pub local_instructions: u64,
}
```

This is a same-call query probe. It does not rely on committed metric rows.

## 6. Proposed 0.67 Direction

### 6.1 Reset checkpoint baseline at endpoint entry

Dispatch should reset the checkpoint baseline when an endpoint scope begins.
After this, the first manual checkpoint in an endpoint means:

```text
instructions from endpoint entry to this checkpoint
```

This removes the need for a dummy `perf!("start")` marker.

Sketch:

```rust
pub fn enter_endpoint() {
    let start = perf_counter();
    reset_checkpoint_baseline(start);
    enter_endpoint_at(start);
}
```

Open detail: timers and background tasks also use perf counters. If they do
not enter endpoint dispatch, they should keep their current behavior or gain
their own explicit task-scope baseline helper.

### 6.2 Add synchronous scoped spans

Add a helper macro for block-local measurement:

```rust
let parsed = canic::perf_span!("parse_request", {
    parse_request(bytes)?
});
```

Semantics:

1. Capture `start = perf_counter()`.
2. Run the block exactly once.
3. Capture `end = perf_counter()`.
4. Record `end - start` as a checkpoint/span row.
5. Return the block result unchanged.

This should not depend on `PERF_LAST`, so inserting a span does not alter
neighboring checkpoint deltas.

Possible metric label shape:

```text
perf span <module_path> <label>
```

Or reuse checkpoint shape:

```text
perf checkpoint <module_path> <label>
```

The separate `span` label is clearer and avoids mixing ordered checkpoints with
block-local spans.

### 6.3 Add async scoped spans

Async endpoint debugging needs the same ergonomics:

```rust
let result = canic::perf_async!("execute_plan", async {
    execute_plan(plan).await
})
.await?;
```

Semantics should use `performance_counter(1)`, so the delta includes local
instructions across await resumptions in this canister's call context.

This still does not count remote canister work. If an inter-canister call is
expensive in cycles but cheap locally, perf rows will not explain the cycle
spend. Existing platform/call metrics should remain the source for that.

### 6.4 Keep stable endpoint totals separate from debug spans

Endpoint aggregate rows are stable product telemetry. Debug spans are
investigation tools. The system should make that distinction explicit.

Options:

1. Export spans only when a canister-local debug flag is enabled.
2. Keep all spans but bound the table by label count.
3. Export span rows through a separate debug metrics tier.
4. Add a compile-time feature for span export.

Tentative preference:

- Keep endpoint totals always available in runtime metrics.
- Keep manual spans available by default during development.
- Add a bounded label policy before recommending broad production use.

### 6.5 Add last and max samples

Current perf rows expose:

```text
count
total_instructions
```

For endpoint debugging, these are not enough. A single slow call is hidden by
the aggregate unless the operator computes deltas externally.

Extend internal perf slots to retain:

```rust
count
total_instructions
last_instructions
max_instructions
```

Public metric compatibility options:

1. Add separate rows:

```text
perf endpoint update foo total
perf endpoint update foo last
perf endpoint update foo max
```

2. Add a new metric payload variant.

3. Keep public metrics unchanged and add a dedicated perf diagnostics endpoint.

Tentative preference:

- Avoid changing `MetricValue` in the first pass.
- Add separate labeled rows only if the current metrics consumers tolerate the
  extra label dimension cleanly.
- Otherwise start with CLI delta support before changing runtime DTOs.

### 6.6 Make query profiling first-class

Add a query annotation or helper so temporary query profiling is obvious.

Possible endpoint attribute:

```rust
#[canic::canic_query(sample_perf)]
fn inspect_user(id: UserId) -> UserView {
    load_user_view(id)
}
```

Generated return type would have to change to `QueryPerfSample<UserView>`,
which is too magical for a normal endpoint attribute.

Safer helper:

```rust
#[canic::canic_query]
fn inspect_user_debug(id: UserId) -> canic::dto::metrics::QueryPerfSample<UserView> {
    canic::api::metrics::MetricsQuery::sample_query(load_user_view(id))
}
```

0.67 can document this as the recommended query debug pattern before adding a
new macro surface.

### 6.7 Add CLI perf filtering

Add a focused readout:

```bash
canic info metrics <deployment> --kind runtime --perf
canic info metrics <deployment> --kind runtime --perf --endpoint mint_token
canic info metrics <deployment> --kind runtime --perf --checkpoint execute_plan
```

This is a CLI ergonomics slice. It should not require runtime protocol
changes. The CLI can filter rows whose labels start with:

```text
["perf", ...]
```

The output should show:

```text
kind      call     name             count    total_i      avg_i
endpoint  update   mint_token       10       1_200_000    120_000
span      -        execute_plan     10       900_000      90_000
```

If the runtime tier is disabled or the deployed Wasm predates the change, the
CLI should say so clearly.

## 7. Proposed User Workflow

### 7.1 Debug an update endpoint

Add spans:

```rust
#[canic::canic_update]
async fn update_project(req: UpdateProjectRequest) -> Result<ProjectView, canic::Error> {
    let ctx = canic::perf_span!("validate", {
        validate_update(&req)?
    });

    let plan = canic::perf_span!("plan", {
        build_update_plan(&ctx)?
    });

    let view = canic::perf_async!("execute", async {
        execute_update(plan).await
    })
    .await?;

    Ok(view)
}
```

Call the endpoint, then query:

```bash
canic --network ic info metrics toko --kind runtime --perf --endpoint update_project
```

### 7.2 Debug a query endpoint

Create a temporary debug endpoint:

```rust
#[canic::canic_query]
fn project_view_debug(
    id: ProjectId,
) -> canic::dto::metrics::QueryPerfSample<Result<ProjectView, canic::Error>> {
    canic::api::metrics::MetricsQuery::sample_query(project_view(id))
}
```

Call the query directly and inspect `local_instructions`.

## 8. Implementation Slices

### Slice 1: Baseline reset

- Add `perf::reset_checkpoint_baseline(now: u64)`.
- Call it from endpoint dispatch entry.
- Add unit tests around first-checkpoint behavior.
- Preserve existing endpoint aggregate rows.

### Slice 2: Span macros

- Add `perf_span!`.
- Add `perf_async!` if macro ergonomics are acceptable.
- Add tests that spans return the block result and do not mutate ordered
  checkpoint baseline.
- Decide whether spans use `PerfKey::Checkpoint` or a new `PerfKey::Span`.

### Slice 3: CLI filtering

- Add runtime metrics perf filters to `canic info metrics`.
- Render average as `total / count`.
- Keep JSON output compatible or version the output if needed.

### Slice 4: Query debug documentation

- Document `MetricsQuery::sample_query(...)` as the supported query profiling
  pattern.
- Defer endpoint-attribute sugar until the return-type ergonomics are proven.

### Slice 5: Label policy

- Define production guidance for span labels.
- Reject or warn on dynamic label patterns if a static check is feasible.
- Consider a bounded span table if production use grows.

## 9. Compatibility

The first implementation pass should preserve:

- existing `perf endpoint ...` rows
- existing `perf checkpoint ...` rows
- `canic_metrics` Candid shape
- metrics profile behavior
- endpoint macro call signatures

New span rows are additive if they use a new label family. If the runtime DTO
changes, that must be treated as a separate compatibility decision.

## 10. Open Questions

1. Should span rows use `checkpoint` labels for compatibility or a new `span`
   label for clarity?
2. Should manual span export be always-on, config-gated, or compile-feature
   gated?
3. Is CLI-side delta calculation enough for `last` and `max`, or should runtime
   store those values?
4. Should timers get a separate baseline reset helper?
5. Do we need a small per-call debug trace endpoint, or are aggregate rows plus
   query samples enough?
6. Should `canic::perf!()` remain the primary public macro, or should docs
   steer endpoint debugging toward span macros once they exist?

## 11. Tentative Decision

Proceed, if at all, in this order:

1. Reset checkpoint baseline at endpoint entry.
2. Add explicit span macros.
3. Add CLI perf filtering.
4. Document query profiling with `MetricsQuery::sample_query(...)`.
5. Revisit last/max samples after the simpler workflow is proven.

This keeps the first 0.67 slice small and low-risk while addressing the main
debugging friction: developers need to know where endpoint instructions are
going without remembering the internal checkpoint model.
