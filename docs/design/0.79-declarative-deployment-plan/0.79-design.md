# Canic 0.79 Design: Declarative Deployment Plan

Date: 2026-07-01

## Status

- Status: tentative design seed
- Release line: tentative 0.79 planning label
- Decision style: no-mutation operator planning
- Primary implementation target: expose a deterministic deployment plan before
  any install, upgrade, topology, authority, or registration mutation.
- Current state: `canic-host` already contains `DeploymentPlanV1`,
  local deployment-plan construction, deployment-truth comparison, safety
  reports, local observation plans, promotion plans, backup plans, and restore
  plans. The v1 readiness docs intentionally keep report, policy, and catalog
  boundaries separate.

## 1. Summary

0.79 should turn Canic's desired deployment shape into a readable, stable,
no-mutation plan. The plan should answer:

```text
What does Canic think should exist?
What facts did it verify?
What assumptions remain unresolved?
What would a future apply need to do?
What blockers make mutation unsafe?
```

The first 0.79 slice should not implement `apply`. It should produce a
deterministic plan and compare it against available local or live evidence.
That makes the operator story stronger without adding a new mutation engine.

## Hard-Cut Rule

`canic deploy plan` is the only planned command surface for this feature. The
design does not include fallback command names, legacy plan schemas, or
alternate aliases for the same operation.

## 2. Goals

1. Expose a CLI command for deployment planning.
2. Reuse `DeploymentPlanV1` instead of introducing a separate plan schema.
3. Keep plan construction deterministic and no-mutation.
4. Separate verified facts from unresolved assumptions.
5. Classify planned actions by phase.
6. Show unsupported shapes as explicit blockers.
7. Support text and JSON output.
8. Make plan output consumable by `canic medic`, deployment checks, and CI
   policy evaluation in later slices.

## 3. Non-Goals

0.79 does not:

1. Add `canic apply`.
2. Install or upgrade Wasm.
3. Create canisters.
4. Mutate controllers.
5. Register artifacts in `wasm_store`.
6. Acquire deployment locks.
7. Sign evidence.
8. Turn a plan into deployment truth.
9. Replace backup/restore-specific planning.
10. Hide unresolved assumptions behind a green status.

## 4. Command Surface

Proposed initial surface:

```text
canic deploy plan <deployment>
canic deploy plan <deployment> --format json
canic deploy plan <deployment> --out deployment-plan.json
canic deploy plan <deployment> --config <path>
canic deploy plan <deployment> --build-profile <profile>
```

Possible later surface:

```text
canic deploy diff <deployment> --plan deployment-plan.json
canic apply <deployment> --plan deployment-plan.json
```

The later `apply` command should require a separate design because it changes
the safety model.

## 5. Plan Phases

The text renderer can show a phase-oriented view derived from
`DeploymentPlanV1`:

```text
1. resolve fleet config
2. resolve attached roles
3. resolve expected role artifacts
4. resolve expected root trust anchor
5. resolve expected canister inventory
6. resolve authority profile
7. resolve verifier-readiness expectations
8. compare desired plan with observed inventory when available
```

Future mutation-oriented phases can be described as proposed operations without
executing them:

```text
create root canister
install root wasm
create project hub
install project hub wasm
create user shards
register children with root
apply issuer policies
verify topology
```

Those operation labels must remain explanatory until an apply design owns
ordering, idempotency, locking, and receipts.

## 6. Plan Model

Initial implementation should use the existing deployment-truth model:

- `DeploymentPlanV1`
- `DeploymentIdentityV1`
- `TrustDomainV1`
- `AuthorityProfileV1`
- `RoleArtifactV1`
- `ExpectedCanisterV1`
- `ExpectedPoolCanisterV1`
- `VerifierReadinessExpectationV1`
- `DeploymentAssumptionV1`

If the CLI needs command-specific metadata, it should use a report that embeds
the plan instead of forking the plan data:

```text
DeploymentPlanReportV1
  schema_version
  command
  target
  status
  plan
  blockers
  warnings
  next_actions
```

## 7. Safety Status

Suggested statuses:

- `ready_to_compare`: plan was built, but no live comparison was requested.
- `warning`: plan was built with unresolved assumptions.
- `blocked`: required inputs are missing or inconsistent.
- `unsupported`: project shape is intentionally outside the planner.

The plan should not claim `ready_to_apply` until an apply design exists.

## 8. Implementation Slices

1. Add a CLI route for `canic deploy plan`.
2. Use the existing local deployment-plan builder for the first implementation.
3. Render a stable text summary and JSON payload.
4. Add blocker and warning classification for unresolved assumptions.
5. Add optional comparison against existing deployment-check evidence.
6. Teach `canic medic deployment` to point at `canic deploy plan` when a
   deterministic plan would explain a blocker.
7. Defer mutation receipts and rollback notes to the apply design.

## 9. Validation

Focused validation should cover:

```text
cargo test --locked -p canic-host deployment_truth --lib
cargo test --locked -p canic-cli deploy
cargo test --locked -p canic-cli status
```

If live inventory comparison is added, include targeted local replica or
PocketIC tests for the touched path.

## 10. Open Questions

1. Should `deploy plan` require an installed deployment target, or should it
   also work from a fleet config before anything is installed?
2. Should plan JSON be an evidence envelope from the first slice?
3. What is the minimum operation vocabulary needed before `apply` can be
   designed without repainting the schema?
4. Which deployment shapes are unsupported blockers versus warnings?
