# Canic 0.79 Design: Declarative Deployment Plan

Date: 2026-07-02

## Status

- Status: implementation specification
- Release line: 0.79 planning label
- Decision style: no-mutation operator planning
- Primary implementation target: expose Canic's deterministic deployment planner
  before any install, upgrade, topology, authority, registration, truth, lock, or
  evidence mutation.
- Current state: `canic-host` already contains `DeploymentPlanV1`, local
  deployment-plan construction, deployment-truth comparison, safety reports,
  local observation plans, promotion plans, backup plans, and restore plans. The
  v1 readiness docs intentionally keep report, policy, catalog, and evidence
  boundaries separate.

## 1. Summary

0.79 turns Canic's desired deployment shape into a readable, stable,
no-mutation planning artifact. The command answers:

```text
What does Canic think should exist?
What facts did it verify?
What assumptions remain unresolved?
What would a future apply need to do?
What blockers make mutation unsafe?
```

The release is intentionally narrow. It does not implement apply, mutation,
evidence signing, deployment truth creation, lock acquisition, receipts, or
rollback planning. Its value is making Canic's internal deployment planner
visible and deterministic before any mutation happens.

The core model is:

```text
desired state:
  derived from fleet/project config, deployment config, build metadata, and
  existing deployment-plan builders

observed state:
  optional evidence from installed deployment records, deployment truth,
  deployment-check evidence, or existing local observation artifacts

comparison:
  available only when observed state exists
```

A missing installed deployment is not a blocker when the desired deployment
target and shape are deterministic. It is reported as unresolved observation
evidence, usually with `comparison_status = "not_available"`.

## Hard-Cut Rule

`canic deploy plan <deployment>` is the only planned command surface for this
feature. The design does not include fallback command names, legacy plan schemas,
shorthand deployment inference, apply aliases, compatibility routes, or parallel
report schema families.

The existing deployment-truth model names such as `DeploymentPlanV1` are reused
as-is because they already exist in `canic-host`. New 0.79 report wrapper names
are unversioned in code and docs. The JSON payload still includes
`schema_version = 1`, but 0.79 does not include `V1`/`V2` compatibility layers,
dual decoders, or alternate accepted report versions.

## 2. Goals

1. Expose a top-level deployment-planning command under `canic deploy`.
2. Work before first install when the named deployment resolves from
   fleet/project config.
3. Reuse and embed `DeploymentPlanV1` instead of introducing a separate plan
   schema.
4. Keep plan construction deterministic and no-mutation.
5. Separate verified facts, unresolved assumptions, blockers, warnings, and next
   actions.
6. Keep optional observed state separate from desired state.
7. Classify unsupported desired shapes explicitly.
8. Support text output and `--json` automation output with `schema_version = 1`.
9. Support `--out` as a controlled local report write only.
10. Provide explanatory future-apply operation labels without defining apply
    instructions.
11. Let `canic medic` point operators to `canic deploy plan` in next actions
    without making medic execute the planner internally.

## 3. Non-Goals

0.79 does not:

1. Add `canic apply`.
2. Install or upgrade Wasm.
3. Create or delete canisters.
4. Mutate controllers.
5. Register artifacts in `wasm_store`.
6. Register root or child topology relationships.
7. Acquire or validate deployment locks.
8. Sign evidence.
9. Wrap plan output in `EvidenceEnvelopeV1`.
10. Turn a plan into deployment truth.
11. Create or update installed deployment records.
12. Calculate rollback plans.
13. Produce apply receipts.
14. Define operation IDs, idempotency keys, dependency graphs, lock scopes, or
    receipt kinds.
15. Call live mainnet by default.
16. Make `canic medic` depend on deploy-plan execution.
17. Guarantee that a future apply will succeed.
18. Replace backup/restore-specific planning.
19. Hide unresolved assumptions behind an apply-ready status.

## 4. Non-Mutation Contract

`canic deploy plan` must not:

1. Install or upgrade Wasm.
2. Create or delete canisters.
3. Change controllers.
4. Register artifacts in `wasm_store`.
5. Register root topology relationships.
6. Register child topology relationships.
7. Write deployment truth.
8. Create or update installed deployment records.
9. Acquire deployment locks.
10. Sign evidence.
11. Write generated files.
12. Modify config files.
13. Persist plan output unless `--out` is explicitly provided.

`--out` is the only intentional write in scope. It writes the JSON plan report to
the requested local path and does not write deployment truth, deployment records,
config, generated files, or any Canic-managed state.

For 0.79, `--out` fails if the output path already exists. No `--force`
overwrite mode is included in this release. `--out` also fails when the parent
directory does not already exist; it must not create directories as an implicit
side effect.

## 5. Command Surface

Supported initial surface:

```text
canic deploy plan <deployment>
canic deploy plan <deployment> --json
canic deploy plan <deployment> --out deployment-plan.json
canic deploy plan <deployment> --config <path>
canic deploy plan <deployment> --build-profile <profile>
```

`--json` follows the 0.78 medic style. 0.79 should not introduce
`--format json` for this command unless Canic first adopts that convention
consistently across adjacent operator-report commands.

Rejected forms:

```text
canic plan <deployment>
canic deploy plan
canic deploy plan --deployment <deployment>
canic deploy diff <deployment>
canic apply <deployment>
canic apply <deployment> --plan deployment-plan.json
canic deploy plan <deployment> --apply
canic deploy plan <deployment> --write-truth
canic deploy plan <deployment> --evidence
canic deploy plan <deployment> --format json
canic deploy plan <deployment> --from-check deployment-check.json
canic deploy plan <deployment> --observe-local
canic deploy plan <deployment> --out deployment-plan.json --force
```

The later `diff` and `apply` surfaces require separate designs because they
change the safety model.

## 6. Target Resolution And Observation

`canic deploy plan <deployment>` requires a resolvable desired deployment target,
not an installed deployment target.

The command succeeds when:

1. the deployment name resolves from project/fleet config; and
2. the desired deployment shape can be constructed deterministically.

Installed deployment records, deployment truth, deployment-check evidence, and
existing local observation artifacts are optional evidence. Their absence should
not block first-install planning.

Typical missing-observation diagnostic:

```text
observed inventory [not_evaluated]
  detail: no installed deployment record exists for demo-local
  next: run canic deploy check after installation, or continue treating this as
        a first-install plan
```

Failure rules:

- blocked: deployment target name is not filesystem-safe.
- blocked: deployment name cannot be resolved.
- blocked: config path was explicitly provided but is unreadable.
- blocked: required role package or artifact metadata is missing.
- blocked: the same logical role maps to conflicting desired canister
  identities.
- blocked: desired root trust anchor contains an invalid principal.
- blocked: required authority profile is malformed.
- blocked: installed deployment state records an unverified root canister.
- unsupported: role topology cannot be represented in `DeploymentPlanV1`.
- unsupported: required authority profile shape is intentionally outside 0.79.
- unsupported: pool/canister relationship is outside the planner contract.
- warning: installed deployment record is missing.
- warning: deployment truth report is absent.
- warning: local observation evidence is unavailable when comparison was
  optional.
- warning: verifier readiness cannot be checked without live state.
- warning: artifact hash is unavailable because build output was not inspected.
- warning: observed inventory is stale or partial.

## 7. Plan Phases

The text renderer can show a phase-oriented view derived from
`DeploymentPlanV1`:

```text
1. resolve fleet config
2. resolve attached roles
3. resolve expected role artifacts
4. resolve expected root trust anchor
5. resolve expected canister inventory
6. resolve authority profile
7. resolve verifier-readiness expectations
8. compare desired plan with observed inventory when available
```

Future mutation-oriented phases may be described as proposed operation labels
without executing them:

```text
create_canister
install_wasm
upgrade_wasm
set_controllers
register_child
register_root
upload_artifact
apply_policy
verify_topology
verify_readiness
```

These labels are explanatory only. They are not operation objects and must not
carry operation IDs, dependency edges, idempotency keys, rollback actions, lock
scopes, or receipt kinds in 0.79.

## 8. Plan Model

Initial implementation should use the existing deployment-truth model:

- `DeploymentPlanV1`
- `DeploymentIdentityV1`
- `TrustDomainV1`
- `AuthorityProfileV1`
- `RoleArtifactV1`
- `ExpectedCanisterV1`
- `ExpectedPoolCanisterV1`
- `VerifierReadinessExpectationV1`
- `DeploymentAssumptionV1`

The CLI report embeds `DeploymentPlanV1` instead of forking the plan data.
Report-only interpretation belongs in the wrapper:

```text
DeploymentPlanReport
  schema_version
  command
  target
  network
  build_profile
  config_path
  status
  comparison_status
  plan
  verified_facts
  assumptions
  blockers
  warnings
  proposed_operations
  next_actions
```

`plan` is the embedded `DeploymentPlanV1`. `verified_facts`, `assumptions`,
`blockers`, and `warnings` use a stable diagnostic shape:

```text
PlanDiagnostic
  category
  code
  severity
  subject
  detail
  next
  source
```

Suggested severities:

- `info`
- `warning`
- `blocked`
- `unsupported`

Suggested categories:

- `config`
- `deployment_identity`
- `trust_domain`
- `authority`
- `artifact`
- `inventory`
- `topology`
- `verifier_readiness`
- `observation`
- `comparison`
- `unsupported_shape`

Suggested sources:

- `config`
- `cli_arg`
- `fleet_config`
- `deployment_config`
- `installed_deployment`
- `deployment_truth`
- `deployment_check`
- `local_observation`
- `live_replica`
- `build_profile`
- `deployment_plan_builder`

`proposed_operations` are report-layer preview labels only:

```text
ProposedOperationLabel
  phase = "future_apply_preview"
  label
  subject
  status = "not_executed"
```

## 9. Status And Comparison Status

Plan construction status and comparison status are separate.

`status` values:

- `planned`: desired plan was built without blockers.
- `warning`: desired plan was built, but unresolved assumptions or optional
  observation gaps remain.
- `blocked`: required inputs are missing or inconsistent.
- `unsupported`: desired project shape is intentionally outside the planner.

Status derivation:

1. `unsupported` if any required desired shape is outside the planner contract.
2. `blocked` if any blocker exists and no unsupported condition exists.
3. `warning` if the plan was built, blockers are empty, and warnings or
   unresolved assumptions exist.
4. `planned` if the plan was built and blockers, warnings, and unresolved
   assumptions are empty.

`comparison_status` values:

- `not_requested`: no comparison was requested or attempted.
- `not_available`: comparison could not run because observed state was missing.
- `compared`: comparison ran and found no drift.
- `compared_with_warnings`: comparison ran with non-blocking unresolved evidence.
- `compared_with_drift`: comparison found drift that should be reviewed before a
  future apply design uses the plan.

Comparison status derivation:

1. `not_requested` if no comparison was requested or attempted.
2. `not_available` if comparison was attempted from available evidence but
   observed state was missing.
3. `compared_with_drift` if observed state differs from desired state.
4. `compared_with_warnings` if comparison ran with non-blocking unresolved
   evidence.
5. `compared` if comparison ran without drift or warning.

The report must not claim `ready_to_apply`, `safe_to_deploy`, or equivalent
deployment-safety language until an apply design exists.

## 10. Comparison Behavior

0.79 should not start with live comparison as the first implementation slice.
The first slice should build and render deterministic desired plans.

Default behavior:

- no live mainnet calls;
- no required local replica calls;
- compare only against already available installed deployment records,
  deployment truth, deployment-check evidence, or local observation artifacts
  when they are already present and safe to read.

Do not implement new live observation behavior in 0.79. Only reuse evidence
already available through existing read-only helpers. If comparison support
becomes non-trivial, leave `comparison_status = "not_requested"` or
`comparison_status = "not_available"` and finish deterministic desired-plan
output first.

## 11. Output Contract

Text output should be human-readable and avoid apply/safety claims:

```text
canic deploy plan demo-local
status: warning
comparison: not_available
network: local
config: ./canic.toml
build_profile: dev

desired deployment [pass]
  detail: deployment demo-local resolved from fleet config

roles [pass]
  detail: resolved 4 attached roles

artifacts [warning]
  detail: expected artifact hash unavailable because build was not requested
  next: run canic build or provide a build profile with resolved artifacts

inventory [not_evaluated]
  detail: no installed deployment record exists for demo-local
  next: compare after first deployment or provide deployment-check evidence

future apply preview
  - create_canister root
  - install_wasm root
  - create_canister project_hub
  - install_wasm project_hub
  - verify_topology

next actions
  - resolve warnings before designing apply
  - run canic medic deployment demo-local if operator readiness is uncertain
```

Use phrasing such as:

- `future apply would need to...`
- `proposed operation label`
- `not executed`

Avoid phrasing such as:

- `ready to apply`
- `safe to deploy`
- `will create`
- `will install`

JSON output emits `DeploymentPlanReport` directly:

```json
{
  "schema_version": 1,
  "command": "canic deploy plan",
  "target": "demo-local",
  "network": "local",
  "build_profile": "dev",
  "config_path": "./canic.toml",
  "status": "warning",
  "comparison_status": "not_available",
  "plan": {
    "...": "DeploymentPlanV1"
  },
  "verified_facts": [
    {
      "category": "config",
      "code": "deployment_target_resolved",
      "severity": "info",
      "subject": "demo-local",
      "detail": "deployment target resolved from fleet config",
      "next": null,
      "source": "fleet_config"
    }
  ],
  "assumptions": [
    {
      "category": "artifact",
      "code": "artifact_hash_unresolved",
      "severity": "warning",
      "subject": "project_hub",
      "detail": "artifact hash was not resolved because build output was not inspected",
      "next": "run canic build before apply planning",
      "source": "deployment_plan_builder"
    }
  ],
  "blockers": [],
  "warnings": [
    {
      "category": "observation",
      "code": "observed_inventory_unavailable",
      "severity": "warning",
      "subject": "demo-local",
      "detail": "no installed deployment record exists for comparison",
      "next": "run deployment check after installation or provide saved evidence",
      "source": "installed_deployment"
    }
  ],
  "proposed_operations": [
    {
      "phase": "future_apply_preview",
      "label": "create_canister",
      "subject": "root",
      "status": "not_executed"
    }
  ],
  "next_actions": [
    "resolve warnings before designing apply",
    "run canic medic deployment demo-local if operator readiness is uncertain"
  ]
}
```

The JSON report is not an `EvidenceEnvelopeV1`, not deployment truth, and not an
authorization to mutate.

## 12. Output Determinism And Streams

When `--json` is used, stdout contains only the JSON `DeploymentPlanReport`.
Human-readable diagnostics, usage errors, and internal errors go to stderr.

Report arrays must be emitted in deterministic order so snapshots and CI
consumers are stable. Array ordering:

1. `blockers`
2. `warnings`
3. `assumptions`
4. `verified_facts`
5. `proposed_operations`
6. `next_actions`

Within each diagnostic list, sort by severity, category, code, subject, and
source where practical. `proposed_operations` should be sorted by phase, label,
and subject where practical. `next_actions` should be deterministic and should
not depend on map iteration order or filesystem traversal order.

`--out` always writes JSON `DeploymentPlanReport` only, regardless of whether
stdout uses text or `--json`. It fails if the path already exists or the parent
directory does not exist. 0.79 does not include `--force` or overwrite behavior.

## 13. Exit Codes

0.79 exit-code rules:

- `0`: report constructed and no blockers exist. This includes `planned`,
  `warning`, `compared`, and comparison-unavailable states.
- `1`: report constructed with `blocked` or `unsupported` status.
- `2`: usage error, invalid command shape, invalid flag combination, output-write
  failure, or internal report-construction failure.

Unresolved assumptions and warnings do not exit nonzero unless they are
classified as blockers.

## 14. Medic Relationship

0.79 may update `canic medic deployment` next actions to mention:

```text
run canic deploy plan <deployment>
```

Medic must not call deploy-plan internally in 0.79 unless a separate design
explicitly changes that contract. Medic diagnoses readiness and points to next
actions; deploy plan explains desired deployment shape.

## 15. Implementation Slices

1. Define `DeploymentPlanReport` and `PlanDiagnostic` wrapper types.
2. Add a CLI route for `canic deploy plan <deployment>`.
3. Resolve the deployment target from config without requiring installed state.
4. Call the existing `DeploymentPlanV1` builder.
5. Map `DeploymentAssumptionV1` and builder failures into blockers and warnings.
6. Render stable text output.
7. Render JSON output with `schema_version = 1`.
8. Add `--out` with safe write behavior and no overwrite by default.
9. Add optional thin comparison against already available local evidence only.
10. Add medic next-action hints that point to deploy plan.
11. Update docs and examples.

Do not start with comparison. First make deterministic desired-plan output
solid.

## 16. Validation

Focused validation should cover:

```text
cargo test --locked -p canic-host deployment_truth --lib
cargo test --locked -p canic-cli deploy
cargo test --locked -p canic-cli status
```

If explicit live local inventory comparison is added later, include targeted
local replica or PocketIC tests for that path. No live mainnet validation is in
scope for 0.79.

Acceptance criteria:

1. `canic deploy plan <deployment>` exists.
2. No alternate command aliases exist.
3. The plan works from fleet/project config without requiring installed
   deployment state.
4. Plan construction is deterministic.
5. Implementation embeds `DeploymentPlanV1` instead of forking the schema.
6. The report separates verified facts, assumptions, blockers, warnings, and
   next actions.
7. Text output is readable.
8. JSON output emits `schema_version = 1`.
9. Output never says `ready_to_apply` or `deployment is safe`.
10. The command performs no mutation.
11. `--json` stdout contains only JSON and sends usage/internal errors to
    stderr.
12. `--out` only writes JSON `DeploymentPlanReport` to the requested report file,
    does not overwrite by default, and does not create parent directories.
13. `--force` is not accepted for `--out`.
14. Diagnostic arrays are emitted in deterministic order.
15. Missing observed state is not treated as a blocker when desired plan is
    valid.
16. Already-available installed-state evidence produces deterministic
    `comparison_status` values without live observation or mutation.
17. Unsupported desired shapes produce explicit unsupported/blocker diagnostics.
18. Validation covers `canic-host deployment_truth`, `canic-cli deploy`, and
    `canic-cli status`.
19. Docs clearly say apply is out of scope.
20. 0.79 does not add report-version compatibility layers, alternate accepted
    report versions, or `V1`/`V2` wrapper families.

## 17. Closed Design Decisions

1. `deploy plan` requires a resolvable desired deployment target, not an
   installed deployment target.
2. Plan JSON is `DeploymentPlanReport`, not `EvidenceEnvelopeV1`.
3. 0.79 includes operation labels only, not apply operation objects.
4. Unsupported desired shapes are `unsupported`; missing observed evidence is
   usually `warning` or `comparison_status = "not_available"`.
5. Status and comparison status are separate fields.
6. The command uses `--json`, aligned with 0.78 medic.
7. `--out` writes JSON only and fails if the output path already exists or its
   parent directory does not exist.
8. `--force` is not part of 0.79.
9. Medic may suggest deploy plan but must not execute it internally.
10. New 0.79 report wrappers are unversioned; `schema_version = 1` is the single
    accepted JSON shape for this hard cut.
