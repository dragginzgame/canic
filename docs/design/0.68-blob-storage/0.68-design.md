# Canic 0.68 Design: Blob Storage Protocol

## Status

- Status: draft design note
- Release line: 0.68 candidate
- Decision style: extraction with strict protocol compatibility
- Primary implementation target: backend blob storage protocol support
- Current state: no implementation committed by this document

## 1. Summary

Canic should provide a feature-gated backend `blob-storage` module that extracts
the Caffeine immutable object storage gateway protocol into reusable Canic
infrastructure.

The Canic-facing name should be short: blob storage. The gateway-facing
contract must keep the existing `_immutableObjectStorage*` method names and
exact Candid signatures. Those method names are protocol surface, not Canic
style.

0.68 is only the protocol and lifecycle foundation:

1. Inventory the existing backend/reference contract.
2. Freeze exact Candid signatures.
3. Add feature flags, backend modules, typed data, stable records, and storage
   ops.
4. Emit the gateway-compatible endpoints with app-supplied guards.
5. Test the backend lifecycle with PocketIC and mocks.

Cashier billing workflows are deferred to 0.69. Toko backend migration is
deferred to 0.70. Frontend, TypeScript packages, upload UI, npm packaging, and
browser client behavior are out of scope for this design line.

## 2. Problem

The blob storage protocol is useful outside Toko, but the reusable backend
pieces are currently tied to Toko-specific canisters, asset records, billing
flows, and frontend code.

Canic needs a backend protocol module that lets a fleet opt into Caffeine blob
storage without copying Toko-specific implementation code. The module must be
generic enough for non-Toko fleets while remaining exact enough that the
gateway can call it without compatibility drift.

## 3. Naming

Use `blob-storage` for Canic-owned names:

- Cargo feature: `blob-storage`
- Rust module: `blob_storage`
- API type: `BlobStorageApi`
- Endpoint macro: `canic_emit_blob_storage_endpoints!`
- Contract docs: `docs/contracts/BLOB_STORAGE.md`

Reserve `immutable object storage` for protocol prose and for the externally
fixed gateway method names:

- `_immutableObjectStorageUpdateGatewayPrincipals`
- `_immutableObjectStorageBlobsAreLive`
- `_immutableObjectStorageBlobsToDelete`
- `_immutableObjectStorageConfirmBlobDeletion`
- `_immutableObjectStorageCreateCertificate`
- `_immutableObjectStorageFundFromProjectCycles`

`_immutableObjectStorageFundFromProjectCycles` is listed here for full contract
inventory, but its implementation belongs to the 0.69 billing slice.

## 4. Goals

1. Add a feature-gated `blob-storage` backend module to `canic` and
   `canic-core`.
2. Preserve all gateway-facing `_immutableObjectStorage*` endpoint names and
   Candid signatures.
3. Keep authorization app-defined through macro guard expressions.
4. Add typed `BlobRootHash` validation and normalization.
5. Add stable records for live blobs, deletion candidates, and synced gateway
   principals.
6. Add storage ops and workflow helpers for certificate registration, liveness
   queries, pending deletion, deletion confirmation, and gateway-principal
   authorization.
7. Add endpoint macros that emit only backend canister endpoints.
8. Add PocketIC and unit tests for the full backend lifecycle.
9. Document local test behavior using mocks only.

## 5. Non-goals

0.68 does not:

1. Implement Cashier balance, top-up, or billing-status workflows.
2. Implement Toko migration.
3. Implement frontend upload/download clients.
4. Publish an npm package.
5. Move Toko's `RemoteAsset` model into Canic.
6. Reimplement, host, or replace the Caffeine gateway.
7. Depend on production `https://blob.caffeine.ai` in CI.
8. Add generic app metadata endpoints.
9. Add Model A paid-canister account linking.
10. Add CLI wrappers.

## 6. Phase 0 Inventory Gate

Implementation must not start by guessing protocol signatures. The first slice
must produce an inventory file that records:

1. The source repositories and commit SHAs used for Toko and the reference
   implementation.
2. Exact backend file paths containing gateway endpoint implementations.
3. Exact generated Candid for each `_immutableObjectStorage*` method.
4. Request and response DTO shapes, including nested variants.
5. Known local-dev differences from production gateway behavior.
6. Which methods are implemented in 0.68 and which are deferred to 0.69.

The design currently treats the reference implementation path as an input to
inventory. If the checkout is not present, the inventory must name the source
location before implementation proceeds.

## 7. Architecture

```text
Caffeine gateway
  -> owner canister _immutableObjectStorage* endpoints
  -> Canic endpoint macro
  -> canic-core workflow::blob_storage
  -> canic-core ops::blob_storage
  -> canic-core stable records

Consumer backend code
  -> BlobStorageApi helpers
  -> canic-core workflow/ops
```

The owner canister authorizes certificate creation, tracks live blobs, tracks
pending deletion, and verifies gateway-only calls against the current gateway
principal set.

0.68 may store gateway principals and expose gateway-principal checks, but the
Cashier sync workflow that refreshes those principals is implemented in 0.69.
0.68 tests can seed mock gateway principals directly through storage ops or
test-only helper APIs.

## 8. Feature Flags

Add:

```toml
blob-storage = []
```

Mirror the feature through `canic` and `canic-core` wherever DTOs, stable
records, macros, or APIs are conditionally compiled.

Do not add `blob-storage-metadata` in 0.68. Generic metadata endpoints are a
later optional feature.

Proposed layout:

```text
canic-core/src/

  model/blob_storage/

  storage/stable/blob_storage/
    stored_blob.rs
    deletion_pending.rs
    gateway_principal.rs
  ops/blob_storage/
    certificates.rs
    deletion.rs
    gateway_principals.rs
    hash.rs
    registry.rs
  workflow/blob_storage/
    lifecycle.rs
  api/blob_storage/
    mod.rs

canic/src/
  api/blob_storage.rs
  macros/endpoints/blob_storage.rs
```

Keep conversions in `ops::*`. Workflow must not construct or mutate storage
records directly.

## 10. Data Model

### 10.1 BlobRootHash

`BlobRootHash` is the canonical backend hash type.

Rules:

1. Accept only `sha256:<64 hex chars>`.
2. Accept uppercase or lowercase input.
3. Store normalized lowercase text.
4. Reject malformed hashes at API boundaries before storage mutation.
5. Use the same semantic type in DTOs and stable records.

### 10.2 StoredBlobRecord

Fields:

- `root_hash: BlobRootHash`
- `created_at: Timestamp`
- `created_by: Option<Principal>`
- `num_blob_bytes: Option<u64>`
- `content_type: Option<String>`
- `status: StoredBlobStatus`

Registration of an already-live blob must be idempotent when immutable fields
match. Conflicting metadata for an already-live hash must return a typed error.

### 10.3 BlobDeletionPendingRecord

Fields:

- `root_hash: BlobRootHash`
- `requested_at: Timestamp`
- `requested_by: Option<Principal>`

Deletion confirmation should be idempotent for already-confirmed deletion only
if the contract inventory proves the gateway expects that behavior. Otherwise,
unknown or non-pending confirmation should return a typed error.

### 10.4 StorageGatewayPrincipalRecord

Fields:

- `principal: Principal`
- `last_seen_at: Timestamp`
- `source: GatewayPrincipalSource`

`GatewayPrincipalSource` should distinguish test-seeded principals from
Cashier-synced principals so tests and production sync behavior are explicit.

## 11. Endpoint Macro

Canic should expose:

```rust
canic_emit_blob_storage_endpoints!(
    create_certificate_guard = <expr>,
);
```

0.68 emits these backend endpoints:

| Endpoint | Mode | Guard | 0.68 behavior |
| --- | --- | --- | --- |
| `_immutableObjectStorageBlobsAreLive` | query | none | Return liveness for requested hashes. |
| `_immutableObjectStorageBlobsToDelete` | query | gateway-only | Return pending deletion hashes. |
| `_immutableObjectStorageConfirmBlobDeletion` | update | gateway-only | Confirm gateway deletion and clear live/pending state. |
| `_immutableObjectStorageCreateCertificate` | update | `create_certificate_guard` | Validate/register blob and return certificate DTO. |

The macro must fail clearly when:

1. `blob-storage` is disabled.
2. Required guard arguments are missing.
3. A consumer attempts to emit billing endpoints without the 0.69 billing
   feature.

`get_storage_owner` is not part of the gateway contract. If it is needed for
backend diagnostics, it should be a separate helper macro or ordinary API
method, not bundled into the required protocol macro.

## 12. Workflow

### 12.1 Create Certificate

1. Endpoint guard authorizes the caller.
2. Workflow validates the hash and request DTO.
3. Ops registers or confirms the live blob record.
4. Workflow returns protocol-compatible certificate material.

The exact certificate response type must come from the Phase 0 Candid
inventory.

### 12.2 Liveness Query

1. Endpoint accepts requested root hashes.
2. Ops normalizes and validates each hash.
3. Ops returns a protocol-compatible live/not-live response.

Malformed hashes should not silently return false unless the protocol inventory
proves that is required.

### 12.3 Pending Deletion

Consumer backend code may request deletion by hash through `BlobStorageApi`.
The workflow marks a live blob as pending deletion. The gateway-only query then
returns pending hashes to the gateway/scrubber.

### 12.4 Confirm Deletion

Gateway-only update verifies the caller, validates the hash, confirms pending
state, removes the live record, and clears the pending record.

## 13. Local and Test Behavior

0.68 local tests do not call the production gateway or Cashier canister. Tests
seed gateway principals directly and exercise only owner-canister backend
state transitions.

Local upload registration may be supported as an owner-canister registration
flow, but browser/client upload behavior is outside this design.

## 14. Security

1. Certificate creation must be guard-protected in production.
2. Gateway-only methods must check the caller against the stored gateway
   principal set.
3. Hash validation must happen before mutation.
4. Queries that expose deletion candidates must be documented as
   gateway-facing protocol surface.
5. Logs must not include full certificate material or sensitive billing data.
6. CI must not depend on production gateway infrastructure.

## 15. Tests

Unit tests:

1. Hash validation and normalization.
2. Live blob registration and idempotency.
3. Conflicting re-registration rejection.
4. Deletion state transitions.
5. Gateway-principal authorization.
6. DTO/record conversion ownership in ops.

PocketIC tests:

1. Endpoint macro emits expected gateway names.
2. Certificate creation registers a live blob.
3. Live blob query reports true for registered blobs and false for absent blobs.
4. Pending deletion appears only to gateway principals.
5. Non-gateway caller is denied from gateway-only methods.
6. Gateway deletion confirmation removes live state.

Macro compile tests:

1. Macro compiles with feature and guard.
2. Macro fails clearly without required guard.
3. Feature-disabled usage fails clearly.
4. Generated Candid names match the inventory snapshot.

## 16. Acceptance Criteria

0.68 is complete when:

1. `blob-storage` exists and is off by default.
2. Exact gateway Candid signatures are inventoried and snapshotted.
3. Backend modules follow Canic layering.
4. The protocol endpoint macro emits the 0.68 backend gateway endpoints.
5. Stable records and ops own blob liveness, pending deletion, and gateway
   principal checks.
6. PocketIC tests cover the backend lifecycle without production services.
7. Docs describe the backend protocol and local test model.

## 17. Deferred Work

0.69 owns Cashier billing, gateway principal sync from Cashier, project-cycle
funding, and backend billing status.

0.70 owns Toko backend migration.

Frontend clients, upload UI, TS packages, npm publication, and browser
download helpers are not part of this backend design line.
