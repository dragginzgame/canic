# Canic 0.68 Design: Blob Storage Protocol

## Status

- Status: draft design note
- Release line: 0.68 candidate
- Decision style: extraction with strict protocol compatibility
- Primary implementation target: backend blob storage protocol foundation
- Current state: no implementation committed by this document

## 1. Summary

0.68 adds the backend `blob-storage` protocol and lifecycle foundation for the
Caffeine immutable object storage gateway.

Canic-owned names use `blob-storage`, `blob_storage`, and `BlobStorageApi`.
Gateway-owned names stay exactly `_immutableObjectStorage*`. Those method names
and Candid signatures are external protocol, not Canic style.

0.68 emits exactly four non-billing gateway endpoints:

1. `_immutableObjectStorageBlobsAreLive`
2. `_immutableObjectStorageBlobsToDelete`
3. `_immutableObjectStorageConfirmBlobDeletion`
4. `_immutableObjectStorageCreateCertificate`

0.68 inventories all six gateway methods so later releases do not guess at the
contract. 0.69 owns Cashier sync and billing endpoints. 0.70 owns Toko backend
adoption.

No frontend, TypeScript, npm, browser upload workflow, generic metadata
endpoints, Cashier billing, or Toko migration work belongs in 0.68.

## 2. Goals

1. Add a feature-gated backend `blob-storage` module to `canic` and
   `canic-core`.
2. Create the protocol inventory artifact before implementation merges.
3. Snapshot Candid for every inventoried gateway method.
4. Emit only the four 0.68 non-billing gateway endpoints.
5. Separate external protocol DTOs from internal typed models.
6. Add internal blob lifecycle state, stable records, storage ops, and workflow.
7. Store gateway principals and use them for gateway-only authorization.
8. Test lifecycle behavior with unit tests, macro tests, Candid snapshots,
   upgrades, and PocketIC mocks.

## 3. Non-Goals

0.68 does not:

1. Implement Cashier calls, balance reads, top-up, or reserve policy.
2. Emit `_immutableObjectStorageUpdateGatewayPrincipals`.
3. Emit `_immutableObjectStorageFundFromProjectCycles`.
4. Emit `get_blob_storage_status`.
5. Migrate Toko backend code or Toko stable state.
6. Add generic blob metadata endpoints.
7. Add CLI wrappers.
8. Depend on production Caffeine, Cashier, gateway, or Toko services in tests.
9. Add browser/client/frontend/TypeScript/npm work.

## 5. Method Classification

All six gateway methods must be inventoried in 0.68. Only four are emitted by
the 0.68 macro.

| Method | Inventory required in 0.68? | Emitted by 0.68 macro? | Owning release | Guard/authorization model | Reason |
| --- | --- | --- | --- | --- | --- |
| `_immutableObjectStorageBlobsAreLive` | yes | yes | 0.68 | protocol-defined public query behavior | Non-billing liveness query backed by 0.68 live blob state. |
| `_immutableObjectStorageBlobsToDelete` | yes | yes | 0.68 | gateway-only caller check against stored principals | Non-billing deletion coordination backed by 0.68 pending deletion state. |
| `_immutableObjectStorageConfirmBlobDeletion` | yes | yes | 0.68 | gateway-only caller check against stored principals | Non-billing deletion confirmation backed by 0.68 lifecycle transitions. |
| `_immutableObjectStorageCreateCertificate` | yes | yes | 0.68 | macro `guard = <access expression>` protects only this endpoint | Non-billing registration/certificate protocol entrypoint. |
| `_immutableObjectStorageUpdateGatewayPrincipals` | yes | no | 0.69 | guard-configurable Cashier sync endpoint | Requires `storage_gateway_principal_list_v1`, which belongs to Cashier integration. |
| `_immutableObjectStorageFundFromProjectCycles` | yes | no | 0.69 | guard-configurable funding endpoint | Requires billing/top-up and project-cycle reserve policy. |

0.68 may store gateway principals and use them for gateway-only authorization.
It exposes no production sync endpoint. 0.68 tests seed gateway principals only
through direct storage ops or test-only helpers.

The 0.69 design must not define new Candid for
`_immutableObjectStorageUpdateGatewayPrincipals` or
`_immutableObjectStorageFundFromProjectCycles`; it must consume the 0.68
inventory snapshots for those methods.

## 6. Phase 0 Inventory Gate

The inventory is a hard merge blocker. No endpoint, DTO, Candid snapshot,
macro, or stable-record implementation may merge before this artifact exists:

```text
docs/contracts/BLOB_STORAGE_INVENTORY.md
```

The artifact must include:

1. Source repository URLs or local source identifiers.
2. Commit SHAs.
3. Backend file paths.
4. Method names.
5. Query/update modes.
6. Exact generated Candid signatures.
7. Request DTO shapes.
8. Response DTO shapes.
9. Nested records and variants.
10. Error/result behavior.
11. Trap-vs-result behavior.
12. Unauthorized behavior.
13. Malformed request behavior.
14. Batch ordering semantics.
15. Duplicate-input semantics.
16. Idempotency expectations.
17. Production-vs-local differences.
18. Explicit ownership classification for 0.68, 0.69, and 0.70.

Implementation must preserve the inventory unless a later design explicitly
updates the contract.

## 8. Protocol DTOs vs Internal Models

Gateway endpoints must accept and return protocol DTOs that match the inventory
exactly. Do not force internal typed wrappers into external gateway Candid.

Internal ops should convert protocol DTOs into typed models before validation
and mutation. `BlobRootHash` is an internal model type, not necessarily an
external DTO field type.

Candid snapshot tests must prove typed Rust wrappers do not alter external
gateway Candid.

### 8.1 BlobRootHash

Internal rules:

1. Accept only `sha256:<64 hex chars>` after protocol DTO conversion.
2. Accept uppercase or lowercase hex input.
3. Store normalized lowercase text.
4. Reject wrong prefix, wrong length, non-hex chars, whitespace, missing
   prefix, and control characters when applicable.
5. Never store malformed hashes.

Stable records may use typed internal models only if serialization remains
upgrade-safe and covered by upgrade round-trip tests.

## 9. Lifecycle State Machine

States:

| State | Meaning |
| --- | --- |
| `absent` | No live blob state and no pending deletion state. |
| `live` | Canonical live blob record exists. |
| `pending_deletion` | Live blob exists and deletion has been requested. |
| `deleted` | Optional tombstone state, only if required by the inventory. |

Allowed transitions:

| Transition | Rule |
| --- | --- |
| `absent -> live` | Certificate creation/registration succeeds. |
| `live -> live` | Allowed only for idempotent registration with identical immutable metadata. |
| `live -> pending_deletion` | Consumer backend requests deletion. |
| `pending_deletion -> absent` | V1 default after gateway confirmation when no tombstone is required. |
| `pending_deletion -> deleted` | Only if the inventory requires a tombstone. |
| `deleted -> live` | Allowed only if inventory or explicit Canic API policy permits re-registration. |

V1 should not create tombstones unless the inventory requires them. Without a
tombstone requirement, confirmed deletion returns the blob to `absent`.

0.68 v1 must use one canonical representation for pending deletion. The
implementation must choose and document one of:

1. `StoredBlobRecord.status = PendingDeletion`.
2. `BlobDeletionPendingRecord` keyed by the same canonical hash.

Do not use both as independent canonical state in v1. If a secondary map or
index is needed for efficient `_immutableObjectStorageBlobsToDelete`, it must
be derived or rebuildable from the canonical representation and tested as an
index, not a second source of truth.

Lifecycle invariants:

1. Malformed hashes are never stored.
2. A live blob has exactly one canonical live record.
3. Pending deletion state must refer to a live blob unless inventory proves
   otherwise.
4. Deletion confirmation removes records by default, or tombstones only if
   required by inventory.
5. Re-registration after deletion is explicit, not accidental.
6. Repeated certificate creation is idempotent only when immutable metadata
   exactly matches.
7. `None` followed by `Some(value)` metadata is a conflict unless the reference
   implementation proves enrichment behavior.

## 10. Stable Memory and Upgrade Requirements

Use existing Canic stable-record conventions. Do not introduce a different
storage framework.

Required storage planning:

1. Name stable map/logical namespaces for live blobs, pending-deletion state, and
   gateway principals.
2. Use normalized `BlobRootHash` as the key for blob maps.
3. Version serialized record schemas.
4. Bound unbounded fields such as `content_type`.
5. Define request-size and response-size limits for liveness batches and
   deletion candidate sets.
6. Preserve live blobs, pending deletion state, and gateway principals across
   upgrade round trips.
7. Keep record shapes compatible with a future 0.70 Toko state migration or
   document a no-state-move path.

## 11. Endpoint Macro and Authorization

Existing Canic endpoint macros use `guard = <access expression>` for a single
consumer guard, for example `canic_emit_icp_refill_endpoints!(guard = ...)`.
0.68 should follow that style:

```rust
canic_emit_blob_storage_endpoints!(
    guard = <access expression>,
);
```

The guard protects only `_immutableObjectStorageCreateCertificate`.

Gateway-only methods check `caller()` against stored gateway principals:

1. `_immutableObjectStorageBlobsToDelete`
2. `_immutableObjectStorageConfirmBlobDeletion`

Unauthorized gateway-only behavior must match the inventory exactly. Test-seeded
gateway principals must not create a public production sync path.
`GatewayPrincipalSource::TestSeeded` must be test/local-only, or explicitly
documented as safe before production builds can use it.

0.68 macro emission:

| Endpoint | Mode | 0.68 auth |
| --- | --- | --- |
| `_immutableObjectStorageBlobsAreLive` | inventory-defined query | inventory-defined public/malformed behavior |
| `_immutableObjectStorageBlobsToDelete` | inventory-defined query | gateway-only stored-principal check |
| `_immutableObjectStorageConfirmBlobDeletion` | inventory-defined update | gateway-only stored-principal check |
| `_immutableObjectStorageCreateCertificate` | inventory-defined update | macro `guard = <access expression>` |

The macro must fail clearly when:

1. `blob-storage` is disabled.
2. `guard = <access expression>` is missing.
3. A 0.68 macro attempt emits deferred billing/sync endpoints.

## 12. Internal Errors and Protocol Mapping

Internal error taxonomy:

1. `MalformedBlobRootHash`
2. `BlobAlreadyLiveWithConflictingMetadata`
3. `BlobNotLive`
4. `BlobAlreadyPendingDeletion`
5. `BlobDeletionNotPending`
6. `UnauthorizedGatewayPrincipal`
7. `GatewayPrincipalSetEmpty`
8. `ProtocolInventoryMissing`
9. `CertificateMaterialUnavailable`

Protocol mapping rules:

1. Internal errors must not automatically become external protocol variants.
2. If inventoried Candid uses `Result`, map errors into the exact existing
   protocol error variant.
3. If the reference traps or rejects, preserve that behavior.
4. Do not invent new externally visible gateway error variants in 0.68.

## 13. Certificate Creation

Certificate response construction must come directly from
`docs/contracts/BLOB_STORAGE_INVENTORY.md`. No placeholder or fake certificate
response may be implemented.

Inventory must decide:

1. Request validation before mutation.
2. Whether registration happens before or after certificate generation.
3. Failure behavior if certificate generation fails.
4. Rollback or no-rollback on certificate failure.
5. Repeated certificate creation response behavior.
6. Logging/caching restrictions for certificate material.

Certificate material must not be logged, cached, or persisted unless the
inventory proves that is part of the protocol.

1. Input ordering.
2. Duplicate hash behavior.
3. Malformed hash behavior.
4. Absent hash behavior.
5. Batch limits.
6. Output ordering.

0.68 tests must lock the inventoried behavior.

### 14.2 `_immutableObjectStorageBlobsToDelete`

Inventory must define:

1. Result ordering.
2. Maximum batch size.
3. Repeat-return behavior until confirmation.
4. Non-gateway caller behavior.

0.68 tests must lock the inventoried behavior.

## 15. Tests

Unit tests:

1. Protocol DTO to internal model conversion.
2. `BlobRootHash` uppercase normalization.
3. Malformed hash rejection for wrong prefix, wrong length, non-hex chars,
   whitespace, missing prefix, and control characters when applicable.
4. Idempotency for `None` vs `Some` metadata.
5. Deletion confirmation for unknown, live-but-not-pending, pending, and
   already-confirmed hashes.
6. Gateway principal source handling.
7. Lifecycle invariants, including the single canonical pending-deletion
   representation.
8. Stable record serialization compatibility.

PocketIC tests:

1. Unauthorized behavior matching the inventory.
2. Generated query/update modes.
3. Liveness batch ordering.
4. Duplicate hash behavior.
5. Malformed hash behavior.
6. Gateway-only pending deletion access control.
7. Upgrade round trip preserving live blobs, pending-deletion state, and gateway
   principals.
8. No production gateway, Cashier, Caffeine, or Toko dependencies.

Macro/compile tests:

1. Exact generated method names.
2. Exact Candid snapshot match.
3. Feature-disabled usage failure.
4. Missing guard failure.
5. Deferred billing/sync endpoints not emitted by 0.68.

## 16. Merge Checks

0.68 is mergeable only when:

1. `blob-storage` feature exists and is off by default.
2. `docs/contracts/BLOB_STORAGE_INVENTORY.md` exists and names source SHAs.
3. Candid snapshots exist for every inventoried gateway method.
4. The 0.68 emitted subset is explicit and contains only the four non-billing
   protocol endpoints.
5. Deferred 0.69 methods are not emitted by 0.68.
6. 0.69 is blocked unless the 0.68 inventory contains exact signatures for
   `_immutableObjectStorageUpdateGatewayPrincipals` and
   `_immutableObjectStorageFundFromProjectCycles`.
7. Gateway-facing method names are exact.
8. Generated Candid matches inventory snapshots.
9. External protocol DTOs are separated from internal typed models.
10. Typed wrappers do not alter external Candid.
11. Lifecycle invariants are tested.
12. Stable records survive upgrade tests.
13. Gateway-only authorization behavior matches inventory.
14. PocketIC lifecycle tests pass without production gateway, Cashier,
    Caffeine, or Toko dependencies.
15. Docs explain local/mock behavior and production limitations.

## 17. Unresolved Protocol Questions

These questions must be answered by the inventory artifact, not by guesswork in
implementation:

1. Exact request and response Candid for all six gateway methods.
2. Trap/reject/result behavior for malformed requests and unauthorized callers.
3. Batch ordering and duplicate-input behavior for liveness queries.
4. Pending deletion ordering, batching, and repeat-return behavior.
5. Whether deleted blobs may be re-registered.
6. Whether metadata can be enriched from `None` to `Some(value)`.
7. Whether certificate generation mutates before or after material is produced.
8. Whether deletion confirmation of unknown or already-confirmed hashes is
   idempotent or an error.
