# 0.50 Design: Adoption Profiles And Safe Onboarding

## Status

- Status: Tentative
- Line: `0.50.x`
- Theme: Adoption profiles and safe onboarding

## Summary

0.50 introduces a passive adoption model for helping operators understand how
an existing Internet Computer deployment could be represented in Canic without
immediately transferring control, mutating topology, importing pools, installing
code, upgrading canisters, or rewriting ownership.

The first slice is read-only. It defines adoption profiles, classifies observed
and configured roles, and produces an adoption report with non-executed
recommendations from existing Canic config plus deployment-truth evidence.

The core rule for `0.50.0` is:

> Canic may explain what it sees and recommend safe next steps, but it must not
> take ownership, attach topology, import pools, install Wasm, upgrade Wasm, or
> change controllers.

0.50 builds on the prior deployment-truth and role-lifecycle work:

- 0.41-0.47 established deployment-truth evidence, passive checks, authority
  evidence, external lifecycle vocabulary, artifact promotion,
  deployment-target identity, and registered-root recovery.
- 0.48 cleaned the setup surface so package metadata is the source for
  `build!` and `start!` identity.
- 0.49 established fleet-scoped role lifecycle state, including declared-only
  roles, attached roles, and strict deploy/build/install exclusion for
  declared-only roles.

0.50 uses those foundations to make onboarding safer for existing deployments.

## Motivation

Canic now has enough lifecycle vocabulary to distinguish between:

- resources Canic manages;
- resources Canic merely observes;
- roles declared in configuration but not attached to topology;
- topology-attached roles;
- externally controlled canisters;
- external Wasm lifecycle;
- deployment targets backed by evidence.

That creates the opportunity for a safer onboarding flow.

A new user with an existing deployment needs to answer several questions before
allowing Canic to manage anything:

- Which deployed canisters correspond to Canic roles?
- Which roles are already declared?
- Which declarations are intentionally not attached?
- Which observed canisters are user-controlled or externally controlled?
- Which artifacts are known to Canic, external to Canic, or only observable by
  hash?
- Which canisters could become managed later?
- Which resources should remain outside Canic permanently?
- Which actions would require controller changes?
- Which actions would be destructive or authority-changing?

The purpose of 0.50 is to answer those questions passively. The first adoption
slice should reduce ambiguity without increasing operational risk.

## Goals

0.50 should introduce:

1. A stable adoption profile vocabulary.
2. A read-only adoption report model with non-executed recommendations.
3. Role classification across configured and observed deployment state.
4. Explicit safety boundaries for brownfield onboarding.
5. Integration with 0.49 declared-vs-attached role lifecycle.
6. Clear separation between observation, declaration, topology attachment,
   authority control, and mutation.
7. Testable invariants preventing adoption reporting from silently becoming
   import or takeover behavior.

## Non-Goals

`0.50.0` does not implement destructive or mutating adoption.

The following are explicitly out of scope for the first slice:

- controller mutation;
- controller import;
- automatic topology attachment;
- automatic role declaration writes;
- pool import;
- install;
- upgrade;
- reinstall;
- stop/start lifecycle mutation;
- deletion or teardown;
- CI/GitOps provenance;
- DR clone verification;
- deployment catalog/group semantics;
- expanded `wasm_store` artifact registry behavior;
- automatic migration of old configs;
- automatic claim of externally managed canisters.

These may be future lines or later 0.50 slices, but they must not be implicit
behavior of the initial adoption report.

## Prior-Line Dependencies

### Deployment Truth

0.50 depends on deployment-truth evidence from earlier lines. Adoption reporting
should use observed deployment state as evidence, not guesswork.

Deployment-truth evidence may include:

- known deployed canisters;
- canister IDs;
- role identity where known;
- installed Wasm identity where available;
- controller information;
- deployment target identity;
- artifact evidence;
- local manifests;
- inventories;
- external lifecycle evidence.

The exact source set should follow the repository's existing deployment-truth
model.

### Authority Evidence

Adoption reporting must distinguish between "Canic can see this" and "Canic can
safely manage this."

A canister that is visible in deployment truth is not necessarily managed by
Canic. If controller authority is missing, external, ambiguous, or
user-controlled, the report must say so.

### 0.49 Role Lifecycle

0.50 must preserve the 0.49 hard cut:

- roles are fleet-scoped;
- package metadata must include both `fleet` and `role`;
- declared-only roles are valid lifecycle state;
- declared-only roles remain visible through lifecycle inspection;
- declared-only roles are excluded from deploy/build/install/truth mutation
  surfaces;
- topology attachment remains explicit;
- `canic build <fleet> <role>` remains attached-role strict.

Adoption reporting may recommend role declarations, but discovered roles must
not become topology attachments automatically.

## Adoption Profiles

An adoption profile describes the operator's intended onboarding posture.

Profiles are not authority grants. They are report modes that guide
classification, warnings, and recommendations.

### `brownfield`

The deployment already exists and may not have been created by Canic.

Expected characteristics:

- one or more observed canisters already exist;
- Canic config may be missing, partial, or stale;
- controllers may be external;
- Wasm may be externally produced;
- topology may not match existing Canic expectations.

Default posture:

- observe only;
- classify gaps;
- recommend declarations before attachments;
- flag authority-changing operations as out of scope.

### `partial`

Some resources are intended to be managed by Canic, while others remain external
or user-controlled.

Expected characteristics:

- mixed managed and observed-only roles;
- some roles may already be declared and attached;
- others may be intentionally excluded;
- external lifecycle may coexist with Canic lifecycle.

Default posture:

- preserve existing managed roles;
- classify unmanaged observed roles;
- avoid assuming that every observed canister should become managed.

### `standalone`

The deployment consists of one or more standalone canisters with minimal
topology.

Expected characteristics:

- small deployment surface;
- possibly no existing fleet topology;
- package metadata may already identify a role;
- deployment truth may identify a single deployed target.

Default posture:

- report the smallest safe Canic representation;
- prefer declaration before attachment;
- do not synthesize topology attachment for compile-only or standalone
  canisters.

### `leaf-only`

Canic is intended to manage leaf/application canisters, while root/control-plane
or authority-sensitive canisters remain external.

Expected characteristics:

- root or governance-like canisters are observed but not managed;
- application roles may be candidates for declaration;
- controller authority may be intentionally retained outside Canic.

Default posture:

- treat control-plane roles as observed or user-controlled;
- classify application canisters separately;
- avoid recommending ownership changes for roots or authority hubs.

### `hybrid-external-wasm`

Canic observes deployment state, but some or all Wasm artifacts come from an
external build or release process.

Expected characteristics:

- installed Wasm may be identifiable by hash;
- artifacts may not exist in Canic's local artifact store;
- promotion provenance may be external;
- Canic may still track deployment truth.

Default posture:

- classify external Wasm as observable evidence;
- avoid pretending external artifacts are Canic-built artifacts;
- recommend explicit artifact registration only if a future artifact-registry
  flow supports it.

### `minimal`

The operator wants the lowest-risk summary possible.

Expected characteristics:

- limited config;
- limited trust in local state;
- early exploratory onboarding.

Default posture:

- produce high-level classification only;
- avoid detailed mutation recommendations;
- emphasize unknowns and required evidence.

## Role Classification

Adoption reporting should classify each role or role candidate using explicit
vocabulary. Classifications are not mutually exclusive. They are derived from
separate lifecycle, observation, authority, and artifact dimensions.

At minimum, the report should track:

- declaration state: `undeclared` or `declared`;
- topology state: `unattached` or `attached`;
- observation state: `unobserved`, `observed`, `candidate-match`, or
  `conflicting-match`;
- authority state: `Canic-authorized`, `user-controlled`, `external`, or
  `unknown`;
- artifact state: `Canic-built`, `external-Wasm`, or `unknown`.

The labels below are report classifications built from those dimensions. For
example, a role can be `observed-only` and `user-controlled`, or
`declared-only` and `external-controller-required`.

### `managed`

A role is managed when Canic configuration intends it to participate in managed
lifecycle surfaces. Managed configuration does not by itself prove Canic
currently has controller authority.

Expected evidence:

- fleet-scoped role declaration exists;
- role is topology-attached;
- package metadata matches `fleet.role` where package-backed;
- deployment target identity is known where deployed;
- authority evidence does not contradict management.

Managed roles remain subject to existing deploy/build/install safety rules.
Authority must remain a separate classification dimension. A role with managed
configuration but contradictory authority evidence must be reported as both
`managed` and `external-controller-required`, or as `managed` with a
non-operable authority state.

### `declared-only`

A role is declared in Canic config but not attached to topology.

Expected evidence:

- `[roles.<role>]` declaration exists for the selected fleet;
- package metadata may exist and must include both `fleet` and `role`;
- no topology attachment exists for the role.

Required behavior:

- visible in lifecycle inspection;
- visible in adoption reporting;
- excluded from deployable role selectors;
- excluded from install targets;
- excluded from local artifact manifests;
- excluded from inventories;
- excluded from local deployment plans;
- rejected by attached-role-strict build behavior.

### `observed-only`

A role candidate appears in deployment-truth evidence but is not declared in
Canic config.

Expected evidence:

- canister exists in observed deployment state;
- no matching fleet-scoped role declaration exists;
- no topology attachment exists.

Required behavior:

- report as observed;
- do not mutate config;
- do not attach topology;
- recommend a declaration only when evidence is strong enough.

### `attached-unobserved`

A role is declared and attached in Canic config, but deployment-truth evidence
does not confirm a corresponding deployed canister.

Expected evidence:

- fleet-scoped role declaration exists;
- topology attachment exists;
- no corresponding observed canister evidence is present for the selected
  deployment evidence set.

Required behavior:

- report the missing observation;
- do not infer teardown or deletion;
- do not treat the role as safely deployable without normal existing checks.

### `user-controlled`

A canister or role candidate is controlled by the user or an external principal
rather than Canic.

Expected evidence:

- controller set does not match Canic's expected authority;
- authority evidence identifies non-Canic control;
- config or profile indicates the resource should remain outside Canic
  management.

Required behavior:

- do not recommend takeover by default;
- flag controller changes as out of scope;
- distinguish "visible" from "manageable."

### `external-controller-required`

A role candidate could only become managed if controller authority changes.

Expected evidence:

- observed deployment state exists;
- desired management would require adding or changing controllers;
- Canic lacks authority to perform normal lifecycle actions.

Required behavior:

- report the authority gap;
- do not perform or imply controller mutation;
- recommend explicit future operator action only as a non-executed report item.

### `imported-pool-candidate`

A resource appears to belong to a pool-like deployment pattern that might be
importable in a future line, but no import should occur in `0.50.0`.

This is a resource classification, not a role lifecycle state.

Expected evidence:

- observed canister grouping;
- deployment naming, metadata, or topology suggests pool membership;
- current Canic config does not manage it as such.

Required behavior:

- classify as a candidate only;
- explicitly forbid pool import in the first slice;
- avoid generating topology attachment.

### `evidence-conflict`

A configured or observed resource has contradictory evidence.

Examples:

- package metadata says one `fleet.role`, but deployment truth suggests
  another;
- an observed canister matches a role name but has unexpected controllers;
- a local artifact hash differs from observed installed Wasm;
- topology says attached, but inventory has no deployable target.

Required behavior:

- report the conflicting evidence;
- do not classify the resource as safely manageable;
- require operator review before recommending topology attachment, install,
  upgrade, import, or controller-related action.

## Adoption Report Model

The adoption report is a read-only artifact generated from current config and
deployment-truth evidence.

It should answer:

1. What profile was used?
2. What fleet was evaluated?
3. What roles are declared?
4. What roles are attached?
5. Which attached roles are not observed in the supplied evidence?
6. What canisters are observed?
7. Which observed canisters correspond to known roles?
8. Which observed canisters are candidates for declaration?
9. Which resources have conflicting evidence?
10. Which resources are externally controlled?
11. Which resources require controller changes before Canic could manage them?
12. Which resources are intentionally excluded from mutation?
13. Which next steps are safe and non-destructive?
14. Which next steps are out of scope for this slice?

Tentative report structure:

```text
AdoptionReport
  fleet
  profile
  generated_at
  inputs
  summary
  role_findings[]
  observed_canisters[]
  authority_findings[]
  artifact_findings[]
  recommendations[]
  blocked_actions[]
  warnings[]
```

### Inputs

The report may use:

- selected fleet config;
- role declarations;
- topology attachments;
- package manifests;
- `[package.metadata.canic]`;
- deployment-truth evidence;
- local artifact manifests;
- inventories;
- authority evidence;
- external lifecycle evidence;
- generated config metadata;
- existing handoff or current-line metadata where applicable.

The report must identify missing or stale evidence rather than silently
assuming correctness.

### Role Finding

A role finding should describe how Canic classified a configured or observed
role.

Tentative shape:

```text
RoleFinding
  fleet
  role
  classification[]
  declaration_state
  topology_state
  package_state
  observed_state
  authority_state
  artifact_state
  evidence[]
  recommendations[]
  warnings[]
```

### Observed Canister Finding

A canister finding should describe deployment-truth evidence that may or may not
map to a Canic role.

Tentative shape:

```text
ObservedCanisterFinding
  canister_id
  matched_fleet
  matched_role
  confidence
  classification[]
  controllers
  wasm_evidence
  deployment_target_evidence
  recommendations[]
  warnings[]
```

### Matching Confidence

Adoption reporting must not treat an observed canister as a role match unless
there is explicit evidence. Name similarity alone may produce a candidate
finding, but not a managed or declaration recommendation without corroborating
metadata, deployment-target identity, package metadata, inventory evidence, or
operator-provided configuration.

### Recommendation

Recommendations are non-executed report items. They are not executed by
`0.50.0`.

Tentative shape:

```text
Recommendation
  kind
  severity
  description
  suggested_action
  suggested_action_effect
  suggested_action_support
  suggested_action_availability
  operator_action_requirement
```

The adoption report command itself must always be read-only. A recommendation
may describe a future mutating operator action, but that action must be marked
as unsupported by the `0.50.0` adoption command and must never be executed by
report generation.

## Report Execution Boundary

Adoption reporting is a read-only evaluation pass.

The report command may read configuration, package metadata, deployment-truth
evidence, inventories, artifact metadata, and authority evidence. It must not
modify those inputs.

By default, the report is written to stdout. If an explicit output path is
provided, Canic may write only that report artifact. It must not update
`canic.toml`, package manifests, topology, local artifact manifests,
inventories, deployment plans, controller state, or deployment-truth records.

Recommendations are descriptive. A recommendation may describe a future
operator action, including an action that would mutate state, but report
generation must not execute it. Mutating recommendations must be clearly marked
as outside the executable behavior of `0.50.0`.

## Safety Rules

The adoption model must enforce the following invariants.

### No Destructive Import By Default

Adoption reporting must never silently import a canister, pool, artifact,
topology attachment, controller, or deployment group.

### Discovered Roles Become Declarations First

When an observed canister appears to correspond to a Canic role that does not
exist yet, the safe path is:

1. Report observed-only state.
2. Recommend a fleet-scoped role declaration.
3. Keep the role declared-only.
4. Require explicit topology attachment later.

The report may suggest a declaration command, but it must not run it.

### No Automatic Topology Attachment

Adoption reporting must not synthesize topology attachment.

Topology attachment remains an explicit operator action, such as the 0.49 flow:

```text
canic fleet role attach <fleet> <role> --subnet <subnet>
```

### No Controller Mutation

Controller changes are outside `0.50.0`.

The report may identify that controller authority is missing, but it must not
add controllers, remove controllers, replace controllers, or generate an
executable takeover path as part of the first slice.

### No Install, Upgrade, Reinstall, Or Stop/Start Mutation

Adoption reporting must not trigger lifecycle operations.

This includes:

- install;
- upgrade;
- reinstall;
- stop;
- start;
- delete;
- deploy;
- promote;
- rollback.

### No Pool Import

Pool import is out of scope for the first slice.

The report may classify `imported-pool-candidate`, but that is only a label for
future investigation.

### No Authority Laundering

The report must not describe an externally controlled canister as `managed`
merely because it has a matching name, role, package, Wasm hash, or deployment
target.

Authority evidence must remain a separate dimension.

## Integration With 0.49 Role Lifecycle

0.50 adoption must use 0.49 lifecycle states directly.

### Declared-Only Remains First-Class

A declared-only role is not an error. It is a valid intermediate state.

Adoption reporting should show declared-only roles clearly and explain whether
they are:

- intentionally staged;
- missing topology attachment;
- backed by package metadata;
- observed in deployment truth;
- blocked by authority;
- safe to attach later;
- intentionally excluded.

### Attached Remains Explicit

Attached roles are the only roles eligible for attached-role lifecycle
surfaces.

Adoption reporting must not blur this boundary. A role can be observed,
declared, or recommended without becoming attached.

### Build Strictness Remains Unchanged

The report must not suggest that `canic build <fleet> <role>` should work for
declared-only roles.

If the report recommends preparing a package-backed role before topology
attachment, it should describe that as setup or declaration work, not as an
attached artifact build.

### Package Metadata Remains Fleet-Role Strict

0.50 must preserve the 0.49 metadata hard cut.

Valid package metadata must include both:

```toml
[package.metadata.canic]
fleet = "<fleet>"
role = "<role>"
```

Role-only metadata is not valid compatibility behavior.

## CLI Shape

The first implementation may expose adoption reporting through a read-only
command.

Preferred command shape:

```text
canic fleet adoption report <fleet> --profile <profile>
```

This command is fleet-scoped because adoption reporting is evaluated against a
selected fleet config, role declarations, and topology.

Rejected alternative: `canic adoption report <fleet>`.

This is shorter, but it hides that adoption reporting is evaluated within a
selected fleet.

For `0.50.0`, `--profile` is required. Default profile selection can come later
once report behavior has settled.

Supported profiles:

- `brownfield`
- `partial`
- `standalone`
- `leaf-only`
- `hybrid-external-wasm`
- `minimal`

Possible output flags:

- `--format text`
- `--format json`
- `--output <path>`

For `0.50.0`, JSON output is experimental. Stable public JSON envelopes belong
to the CI/GitOps provenance line unless that line is explicitly promoted.

The command must be read-only.

It must not:

- edit `canic.toml`;
- edit package manifests;
- attach topology;
- mutate controllers;
- install or upgrade canisters;
- import pools;
- write artifact manifests as if roles were deployable;
- rewrite deployment inventories.

If the command writes a report file, that file must be an output artifact only,
not an input mutation.

## Test Strategy

0.50 should include regression tests that prove adoption reporting remains
passive.

Suggested test fixtures:

- declared-only role with no observed canister;
- declared-only role with observed canister and external controller;
- observed-only canister with no role declaration;
- managed attached role with matching package metadata and non-contradictory
  authority;
- attached role with no corresponding observed canister;
- role or canister with conflicting package, inventory, artifact, or authority
  evidence;
- role with package metadata missing `fleet`;
- role with package metadata mismatch;
- brownfield deployment with mixed managed and user-controlled canisters;
- standalone compile-only canister that must not synthesize topology;
- hybrid external Wasm canister with known hash but no local Canic artifact;
- pool-like observed canisters classified only as `imported-pool-candidate`.

Required invariant tests:

- adoption report does not mutate config;
- adoption report does not attach topology;
- adoption report does not produce deployable declared-only roles;
- adoption report does not treat observed-only roles as managed;
- adoption report does not treat external-controller-required roles as
  manageable;
- adoption report reports attached-unobserved roles without inferring teardown
  or deletion;
- adoption report reports evidence conflicts and withholds safe-management
  recommendations;
- adoption report does not treat name similarity alone as a role match;
- adoption report writes only stdout by default;
- adoption report with `--output <path>` writes only that report artifact;
- adoption report preserves 0.49 attached-role strictness;
- adoption report rejects or warns on role-only package metadata.

## Documentation Strategy

0.50 documentation should explain adoption as a safe onboarding process, not an
import command.

Docs should include:

- profile definitions;
- report examples;
- classification vocabulary;
- safety boundaries;
- report write-boundary behavior;
- integration with 0.49 role lifecycle;
- examples showing discovered roles becoming declaration recommendations first;
- examples showing externally controlled canisters remaining external;
- warnings that controller changes and destructive import are not part of
  `0.50.0`.

Docs must avoid implying that Canic can safely take over existing deployments
without explicit operator action and authority review.

## Acceptance Criteria

`0.50.0` is acceptable when:

- adoption profiles are defined;
- role classifications are defined;
- a report model exists;
- report generation is read-only;
- declared-only roles are preserved as non-deployable;
- observed-only roles are not treated as managed;
- external-controller-required roles are not treated as manageable;
- attached-unobserved roles are reported without teardown or deletion
  inference;
- evidence conflicts block safe-management recommendations;
- name similarity alone produces at most a candidate finding;
- the report command writes only stdout by default or the explicit `--output`
  artifact when requested;
- discovered roles are recommended as declarations first;
- topology attachment remains explicit;
- destructive import is forbidden by default;
- controller changes are out of scope;
- tests cover passive behavior;
- docs explain safety boundaries.

## Resolved 0.50.0 Decisions

- CLI namespace: prefer `canic fleet adoption report <fleet> --profile
  <profile>`.
- Profile selection: require `--profile` in `0.50.0`.
- JSON stability: treat JSON as experimental in `0.50.0`.
- Report persistence: default to stdout; allow only an explicit `--output
  <path>` report artifact.
- Confidence: name similarity alone can produce only a candidate finding.
- Suggested actions: allowed only as non-executed recommendations; mutating
  actions must be clearly marked as unsupported by `0.50.0`.
- `imported-pool-candidate`: treat it as a resource classification, not a role
  lifecycle state.

## Open Questions

- Should report generation require explicit deployment-truth evidence, or may
  it run with partial local evidence and warnings?
- Which evidence sources should the first report builder consume directly?
- Should persisted report artifacts have a retention or naming convention
  before the CI/GitOps provenance line defines stable envelopes?

## Deferred Work

The following work should remain outside the first 0.50 slice:

- CI/GitOps provenance;
- DR clone verification;
- `wasm_store` artifact registry expansion;
- deployment catalog/groups;
- teardown;
- pool import;
- controller migration;
- automatic config import;
- automatic topology attachment;
- mutating adoption workflows.

These are valuable, but they depend on a stable understanding of what Canic is
managing versus observing.

## Design Principle

0.50 should make onboarding existing deployments safer by making Canic more
explicit, not more aggressive.

The adoption report should help an operator say:

> Here is what Canic sees, here is what Canic already manages, here is what
> Canic should not touch, and here are the safest explicit steps to continue.

It should not make Canic silently own more than the operator intended.
