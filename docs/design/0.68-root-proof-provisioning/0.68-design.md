# Canic 0.68 Design: Root Proof Provisioning

## Status

- Status: draft MVP design note
- Release line: 0.68 candidate
- Decision style: delegated-token provisioning contract repair
- Primary implementation target: root delegation proof provisioning and signer
  install
- Current state: Canic 0.67 issuer-local provisioning can prepare root proofs
  but cannot reliably retrieve them through issuer composite queries

0.68 is the root proof provisioning repair line. The prior blob-storage 0.68
planning is deferred to a later backend storage line. This document is the
0.68 release focus and must not share the release line with unrelated
blob-storage work.

## 1. Summary

0.68 fixes delegated-token root proof provisioning with the smallest supported
contract:

1. A provisioner asks root to prepare proof material in an update call.
2. The provisioner retrieves prepared proofs through a direct root query.
3. The provisioner asks root to install the proofs on signer canisters.
4. Signers verify and store active root delegation proofs.
5. Normal delegated-token prepare/get remains signer-local after installation.

Root remains the trust anchor and renewal authority. Product frontends do not
call root during normal delegated-token auth.

## 2. Problem

The 0.67 issuer-local flow tries to retrieve root proof material through an
issuer composite-query wrapper:

```text
frontend -> signer composite query -> root query -> get_signature_as_cbor()
```

That shape fails because root must assemble the canister-signature proof in a
query context where `data_certificate()` is available. The nested root query
does not provide that certificate context.

The opposite workaround also fails under the 0.67 contract:

```text
external caller -> root direct query
```

That gives root the certificate context, but the 0.67 retrieval path expects
`msg_caller == issuer_pid`, so any caller other than the issuer is rejected.

The fix is to stop using issuer-caller identity as the retrieval authority.
Issuer authorization belongs to prepare/install policy. Retrieval in the MVP is
authorized by root controllers and must be a direct root query. Controller plus
provisioner ACL authorization remains the eventual automation model.

## 3. MVP Scope

0.68 includes:

1. A root issuer allowlist.
2. Controller-only root batch authorization for the MVP, with provisioner ACL
   retained as the eventual automation target.
3. Batch prepare update.
4. Batch direct-query retrieval.
5. Batch install update that broadcasts to signers.
6. Signer active-proof install verification.
7. Signer active-proof status.
8. Removal of the old single-proof root prepare/get endpoints.
9. Tests for the direct-root-query invariant and absence of the old nested
   retrieval route.

0.68 does not include:

1. Autonomous root timer refresh.
2. Signer self-refresh through root.
3. Product frontend-triggered provisioning.
4. Retrieval tickets.
5. Complex install modes.
6. Full CLI automation.
7. Blob-storage or Cashier billing implementation.
8. Public provisioner ACL administration.

## 4. Hard Invariants

1. `canic_get_delegation_proof_batch` is a root query endpoint called directly
   against the root canister.
2. `canic_get_delegation_proof_batch` must not be called through issuer
   composite-query wrappers, signer wrappers, root self-calls, or
   canister-to-canister calls.
3. If `data_certificate()` is unavailable, retrieval fails with
   `RootDataCertificateUnavailable` or the API's documented trap behavior.
4. Retrieval authorization is root-controller-only in the MVP.
5. Retrieval authorization must not require `msg_caller == issuer_pid`.
6. Signer install must verify that the proof certificate names
   `issuer_pid == self`.
7. Delegated-token prepare/get remains signer-local after active proof install.
8. The eventual target remains controller/provisioner ACL authorization; the
   controller-only MVP must not be mistaken for the long-term automation model.

## 5. Root Policy State

Root owns the minimal policy state required to provision signers.

Issuer record:

```rust
struct RootIssuerRecord {
    issuer_pid: Principal,
    enabled: bool,
    allowed_audiences: Vec<DelegationAudience>,
    allowed_grants: Vec<DelegatedRoleGrant>,
    max_cert_ttl_ns: u64,
    refresh_after_ratio_bps: u16,
}
```

Eventual provisioner ACL:

```rust
struct RootProvisionerRecord {
    principal: Principal,
    enabled: bool,
}
```

Prepare must reject unregistered issuers, disabled issuers, audiences outside
policy, grants outside policy, certificate TTLs above the issuer maximum, and
invalid refresh policy.

Refresh policy validation:

1. `0 < refresh_after_ratio_bps < 10_000`.
2. `cert_ttl_ns <= max_cert_ttl_ns`.
3. `refresh_after_ns` is derived from the certificate validity window and the
   issuer's validated `refresh_after_ratio_bps`.

Signer `ActiveDelegationProof.refresh_after_ns` must come from this validated
root issuer policy, not ad hoc implementation logic.

For v1, root batch endpoints may require controllers only. The provisioner ACL
record remains part of the intended policy model so later deployment
automation can operate without granting broad controller authority, but public
provisioner ACL administration is deferred from the MVP.

## 6. Endpoint Contract

### 6.1 Prepare Batch

Root update:

```text
canic_prepare_delegation_proof_batch
```

Responsibilities:

1. Require root controller in the MVP.
2. Validate every issuer against the root issuer allowlist.
3. Build each canonical delegation certificate.
4. Compute each `cert_hash`.
5. Add each root canister-signature input to root's signature map.
6. Update root certified data.
7. Store pending proof metadata keyed by `batch_id`, `issuer_pid`, and
   `cert_hash`.
8. Return batch metadata.

Minimum prepare response metadata:

```rust
struct RootDelegationProofBatchPrepareResponse {
    batch_id: [u8; 32],
    entries: Vec<RootDelegationProofBatchEntry>,
    retrieval_expires_at_ns: u64,
}

struct RootDelegationProofBatchEntry {
    issuer_pid: Principal,
    cert_hash: [u8; 32],
    expires_at_ns: u64,
    refresh_after_ns: u64,
}
```

Exact Candid may be finalized in implementation and snapshotted, but these
behavioral fields must be present.

Prepare does not return final canister-signature proofs because updates do not
have `data_certificate()`.

Prepare must be replay-protected or request-id keyed. Repeating the same
provision request with the same request id returns the same batch metadata.
Repeating without the same request id may create a new batch, subject to
quotas.

### 6.2 Retrieve Batch

Root direct query:

```text
canic_get_delegation_proof_batch
```

Responsibilities:

1. Require root controller in the MVP.
2. Require direct root query certificate context.
3. Validate `batch_id`, `issuer_pid`, `cert_hash`, and retrieval expiry.
4. Call `get_signature_as_cbor`.
5. Return `DelegationProof` values.

If the call does not have root `data_certificate()`, retrieval fails. This is
the explicit regression guard for issuer nested-query retrieval.

### 6.3 Install Batch

Root update:

```text
canic_install_delegation_proof_batch
```

Responsibilities:

1. Require root controller in the MVP.
2. Accept retrieved `DelegationProof` values plus their batch/proof
   identifiers.
3. Verify each submitted proof matches pending metadata.
4. Call each signer install endpoint.
5. Return per-signer outcomes.

Root must not attempt to assemble proofs during the install update. It validates
submitted proofs against pending metadata and broadcasts them to signers.

MVP outcomes:

```rust
enum RootDelegationProofInstallOutcome {
    Installed,
    AlreadyInstalled,
    RejectedBySigner,
    CallFailed,
    ProofMismatch,
    ExpiredOrSuperseded,
}
```

Repeated install calls must be safe. Reinstalling the same proof should be
idempotent.

## 7. Signer Contract

Signer install verifies:

1. The proof is rooted in the configured trusted root canister and root key.
2. The proof certificate hash matches the payload.
3. The certificate names `issuer_pid == self`.
4. Audience and grants are accepted by signer policy.
5. The proof is not expired.
6. The proof is not too close to expiry unless explicitly allowed.
7. Reinstalling the same proof is idempotent.
8. Installing a newer proof replaces the active proof.
9. Installing an older proof is rejected unless an explicit admin override is
   added later.

Signer active proof state:

```rust
struct ActiveDelegationProof {
    root_canister_id: Principal,
    issuer_pid: Principal,
    cert_hash: [u8; 32],
    proof: DelegationProof,
    installed_at_ns: u64,
    expires_at_ns: u64,
    refresh_after_ns: u64,
}
```

Signer status must expose enough for provisioners to decide refresh:

1. Missing active proof.
2. Current `cert_hash`.
3. Trusted root canister id.
4. `expires_at_ns`.
5. `refresh_after_ns`.
6. Whether refresh is needed.
7. Whether the proof is expired.

For MVP, signer status is a query exposing only non-secret operational state.
It may be public or provisioner-guarded, but the chosen access model must be
documented and tested. It must not expose private keys, signing material, or
retrieval secrets.

Signer proof verification must use explicit trusted root canister and root-key
configuration for the deployment target. No protected verification path may
implicitly fall back to `cdk::api::root_key()`. Local, PocketIC, testnet, and
mainnet root-key handling must be covered by tests or by the existing
delegated-token verifier discipline.

## 8. Canonical Hash

0.68 must keep one deterministic `cert_hash` for root delegation certificates.

The implementation must define:

1. Domain separator.
2. Canonical certificate encoding.
3. Field ordering.
4. Omitted-option behavior.
5. Binding to root canister id, issuer principal, audience, grants, and
   validity window.

Tests must lock deterministic hash computation and malformed certificate
rejection. The detailed encoding can live with the implementation rather than
being fully specified in this design note.

## 9. Quotas and Cleanup

MVP quotas:

1. Maximum issuers per batch.
2. Maximum pending batches.
3. Maximum pending proofs per issuer.
4. Maximum certificate TTL.
5. Maximum retrieval TTL.

Cleanup:

1. Expired pending proof metadata must be pruned.
2. Pending metadata and retrieval quotas must prevent unbounded active
   provisioning state.
3. Signature-map leaf pruning is required only if 0.68 actually removes leaves.
4. If cleanup removes signature-map leaves, root must recompute certified data.
5. Cleanup may run opportunistically during prepare/install or through an
   explicit provisioner-triggered maintenance update.

Autonomous root refresh timers are deferred.

## 10. Compatibility

The 0.67 single-proof issuer prepare/get endpoints are removed in the 0.68 MVP.

They are not retained as compatibility endpoints because issuer-to-root nested
proof assembly cannot obtain root
`data_certificate()`.

Documentation must not direct signers to retrieve root proofs through
composite-query wrappers.

## 11. Operator Runbook

Manual v1 sequence:

1. Query signer active-proof status.
2. If the proof is missing, expired, or `now >= refresh_after_ns`, call root
   `canic_prepare_delegation_proof_batch`.
3. Call root `canic_get_delegation_proof_batch` as a direct root query.
4. Call root `canic_install_delegation_proof_batch`.
5. Verify signer active-proof status.
6. Verify signer-local delegated-token prepare/get.

If a proof is expired, signer issuance should stop until refresh succeeds. If a
proof is only past `refresh_after_ns`, issuance may continue until expiry while
the provisioner refreshes it.

Product frontends must not run this sequence during normal auth.

## 12. Tests

Unit tests:

1. Retrieval authorization is independent of `issuer_pid`.
2. Batch id, issuer, cert hash, and expiry binding.
3. Issuer allowlist policy validation.
4. Controller-only MVP authorization validation.
5. Refresh policy validation.
6. Prepare response includes expiry and refresh metadata.
7. Prepare idempotency/replay behavior.
8. Deterministic certificate hash.
9. Malformed certificate rejection.
10. Signer install rejects wrong issuer.
11. Signer install rejects older or expired proof.
12. Install does not assemble proofs during update.
13. Idempotent reinstall.
14. Newer proof replacement.
15. Signer status reports missing, refresh-needed, expired, and valid states.
16. Signer status shape and access model.
17. Explicit trusted root-key verifier configuration.
18. Expired pending metadata cleanup.

PocketIC tests:

1. Root prepare -> direct root query retrieval -> root batch install.
2. Signer-local delegated-token prepare/get after install.
3. The legacy single-proof root prepare/get route is absent, and proof
   retrieval is only exercised through the direct root batch query.
4. Root unavailable after install does not break signer-local issuance before
   proof expiry.
5. Expired active proof stops new delegated-token issuance.
6. Refresh-needed status after `refresh_after_ns`.
7. Partial batch install failure and retry.

Candid/protocol tests:

1. `canic_prepare_delegation_proof_batch`.
2. `canic_get_delegation_proof_batch`.
3. `canic_install_delegation_proof_batch`.
4. Signer install/status endpoints if added or changed.

## 13. Deferred

1. Retrieval tickets.
2. Autonomous root timers.
3. Signer self-refresh.
4. Full CLI automation.
5. Rich install modes.
6. Public root issuer registry administration endpoints.
7. Signature-map leaf pruning beyond what is required for bounded pending
   metadata.
