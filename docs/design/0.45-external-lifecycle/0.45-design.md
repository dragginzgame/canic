# 0.45 Design: External And User-Owned Lifecycle

## Status

Tentative design notes for the 0.45 line.

0.45 handles canisters Canic cannot unilaterally upgrade.

Core goal:

```text
Canic can coordinate upgrades it does not fully control without lying about
authority.
```

---

## Scope Calibration

Release bar:

- `LifecycleAuthorityV1` projection from existing `CanisterControlClassV1`
  classifications;
- external-upgrade proposal and verification shape.

Design hook:

- partial deployment upgrade semantics for user-controlled roles;
- security patch playbook shape;
- SDK or frontend consent requirements without implementing a frontend
  contract;
- externally completed upgrade receipts.

Deferred:

- complete frontend consent implementation;
- productized customer self-service upgrade UX;
- generic brownfield adoption workflows.

---

## Dependency On Earlier Lines

0.41 and 0.42 should already classify canisters with the canonical
`CanisterControlClassV1` enum:

```text
CanisterControlClassV1 {
  DeploymentControlled,
  CanicManagedPool,
  ExternallyImported,
  JointlyControlled,
  UserControlled,
  UnknownUnsafe,
}
```

`DeploymentControlled` means controlled by the deployment authority for a
concrete deployment target.

0.45 adds lifecycle workflows for the externally controlled categories. It does
not introduce the classification from scratch.

0.45 must project existing `CanisterControlClassV1` observations into
`LifecycleAuthorityV1`. It must not fork the control-class model or create a
second user/external classification path.

## 0.42.14 Handoff Constraints

External lifecycle work must preserve the authority boundary 0.42 closed:

- Authority `Safe` is authority-scoped and does not replace the deployment
  safety report.
- Authority command success means a local authority artifact was produced; it
  does not imply live controller mutation or whole-deployment safety.
- Authority dry-run receipts/evidence are structural self-consistency
  artifacts, not tamper-evident proof of live controller state.
- `Authority*V1` JSON fields and enum strings are stable automation surfaces.
- Future mutating authority receipts must stay schema-distinct from dry-run
  receipts.

---

## Problem

Some roles may be part of the runtime topology but not controlled solely by the
deployment authority.

Example:

```text
project_instance is controlled by the user
```

If the user is the sole IC controller, that canister is in a different trust
domain for upgrade authority. A deployment upgrade cannot push there without
user consent or an explicit delegated mechanism.

Canic must distinguish:

- deployment authority can upgrade;
- deployment authority can propose;
- deployment authority can verify externally completed upgrade;
- deployment authority must not touch.

---

## Lifecycle Authority

0.45 extends lifecycle behavior for the canonical control classes. It should
not rename or fork the classification model.

For each role/canister, the plan should record:

```text
LifecycleAuthorityV1 {
  control_class,
  required_controllers,
  consenting_principals,
  allowed_upgrade_modes,
  verification_requirements,
}
```

---

## Upgrade Coordination

For user-owned or externally controlled canisters, the flow is not:

```text
deployment authority pushes upgrade
```

It is:

```text
plan proposes upgrade
Canic proves artifact/config/trust identity
user or external controller consents or executes delegated step
receipt records result
live inventory verifies postcondition
```

### Proposal

```text
ExternalUpgradeProposalV1 {
  proposal_id,
  deployment_plan_id,
  canister_id,
  role,
  current_module_hash,
  target_role_artifact,
  canonical_embedded_config_sha256,
  required_consent,
  verification_policy,
}
```

### Consent

Consent may be:

- explicit user action through an endpoint;
- controller-side execution of a generated command;
- delegated install authority;
- out-of-band external action recorded and verified by inventory.

0.45 should avoid assuming one consent mechanism for all deployments.

### Receipt

```text
ExternalUpgradeReceiptV1 {
  proposal_id,
  consent_state,
  attempted_by,
  observed_before,
  observed_after,
  verification_result,
}
```

Receipts remain evidence. Live inventory remains truth.

---

## Partial Deployment Upgrade Semantics

User-controlled or externally controlled roles should not block upgrades of
unrelated directly managed roles.

The plan should separate:

```text
directly_executable_role_upgrades
proposed_external_role_upgrades
blocked_role_upgrades
```

A directly managed role may still be blocked by dependencies. Protected-call
readiness, role epochs, topology changes, verifier state, or explicit role
dependencies can require ordering across control classes.

Pending external lifecycle work must remain visible in receipts and inventory.
It should not disappear because the directly managed subset completed.

---

## Consent Surface Requirements

0.45 defines the proposal, receipt, and verification contract.

It does not implement the frontend or SDK consent UX.

External upgrade proposals must include enough identity for a frontend, CLI, or
external controller to present a safe consent request later:

```text
artifact identity
canonical embedded config digest
current module hash
target module hash
root trust anchor
authority profile identity
required external action
verification policy
```

The proposal should make clear whether consent authorizes a direct install,
delegated install, or externally completed upgrade.

---

## Critical Fix Path

Critical bug fixes for user-owned roles need an explicit operational flow.

The plan should say:

- which canisters are affected;
- whether deployment authority has direct authority;
- which external consent is required;
- whether protected-call role epochs or verifier cascades must change;
- which directly managed roles may be patched while user-controlled roles
  remain pending, if dependency checks allow it;
- what remains vulnerable until external action completes.

Unresolved external consent must be reported as residual exposure. This must
not be hidden inside a normal deployment upgrade report.

---

## 0.40 Protected-Call Implications

External/user-owned lifecycle changes can affect protected internal calls.

0.45 must account for:

- AppIndex role reassignment;
- subnet registry role changes;
- role epoch bumps;
- verifier cascade completion;
- stale proof TTL exposure;
- root key/verifier readiness;
- audience canister replacement.

A user-owned upgrade is not complete for protected-call purposes until the
relevant role and verifier state is consistent or explicitly recorded as
pending.

---

## Commands

Tentative operator surface:

```text
canic deploy external plan <deployment>
canic deploy external propose <role-or-canister>
canic deploy external verify <proposal-or-receipt>
canic deploy external list-pending <deployment>
```

The exact user-facing consent surface may be application-specific, but the
Canic plan and receipt shapes should be stable.

---

## Non-Goals

0.45 should not:

- assume all user-owned canisters expose the same consent endpoint;
- bypass IC controller authority;
- silently upgrade user-controlled canisters;
- treat proposal receipts as proof without live verification;
- invent a new promotion model.

---

## Implementation Slices

### Slice 1: Lifecycle Authority Model

Represent lifecycle authority per canister and role.

### Slice 2: External Upgrade Proposal

Generate proposals that bind target artifacts, embedded config, current module
hash, and required consent.

### Slice 3: External Completion Receipts

Record pending, refused, completed, and externally completed upgrade outcomes.

### Slice 4: Verification

Re-inventory and verify module hash, controllers, embedded config, and
protected-call readiness.

### Slice 5: Critical Fix Reporting

Render explicit affected-canister and unresolved-consent reports for critical
fixes.

---

## Exit Criterion

0.45 is complete when:

```text
Canic can project existing user-owned or externally controlled roles into
lifecycle authority, generate upgrade proposals, and verify externally
completed upgrades without pretending it has unilateral authority.
```
