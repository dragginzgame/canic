# 0.45 Design: External And User-Owned Lifecycle

## Status

Tentative design notes for the 0.45 line.

0.45 handles canisters Canic cannot unilaterally upgrade.

Core goal:

```text
Canic can coordinate upgrades it does not fully control without lying about
authority.
```

---

## Scope Calibration

Release bar:

- `LifecycleAuthorityV1` projection from existing `CanisterControlClassV1`
  classifications;
- `ExternalLifecyclePlanV1` partitioning of directly executable, externally
  proposed, and blocked role upgrades;
- digest-pinned external-upgrade proposal and verification shape that reuses
  the 0.44 deployment/artifact/provenance evidence model.

Design hook:

- partial deployment upgrade semantics for user-controlled roles;
- security patch playbook shape;
- SDK or frontend consent requirements without implementing a frontend
  contract;
- externally completed upgrade receipts.

Deferred:

- complete frontend consent implementation;
- productized customer self-service upgrade UX;
- generic brownfield adoption workflows.
- live controller mutation for externally controlled canisters unless a later
  line explicitly promotes a delegated execution path.

---

## Dependency On Earlier Lines

0.41 and 0.42 classify canisters with the canonical
`CanisterControlClassV1` enum:

```text
CanisterControlClassV1 {
  DeploymentControlled,
  CanicManagedPool,
  ExternallyImported,
  JointlyControlled,
  UserControlled,
  UnknownUnsafe,
}
```

`DeploymentControlled` means controlled by the deployment authority for a
concrete deployment target.

0.45 adds lifecycle workflows for the externally controlled categories. It does
not introduce the classification from scratch.

0.45 must project existing `CanisterControlClassV1` observations into
`LifecycleAuthorityV1`. It must not fork the control-class model or create a
second user/external classification path.

0.45 also assumes 0.44 can represent role artifact identity, promotion
provenance, target embedded config, materialization evidence, and promoted
execution as digest-pinned `DeploymentPlanV1` /
`ArtifactPromotionPlanV1` evidence.

External lifecycle plans and proposals must reference those plan, artifact,
embedded config, and provenance identities rather than inventing a parallel
artifact promotion model.

If an externally controlled role is promoted from a source artifact, the
external proposal records the same role-scoped artifact identity and target
embedded-config evidence that 0.44 promotion readiness/provenance would use.

## 0.42.14 Handoff Constraints

External lifecycle work must preserve the authority boundary 0.42 closed:

- Authority `Safe` is authority-scoped and does not replace the deployment
  safety report.
- Authority command success means a local authority artifact was produced; it
  does not imply live controller mutation or whole-deployment safety.
- Authority dry-run receipts/evidence are structural self-consistency
  artifacts, not tamper-evident proof of live controller state.
- `Authority*V1` JSON fields and enum strings are stable automation surfaces.
- Future mutating authority receipts must stay schema-distinct from dry-run
  receipts.

## 0.44 Handoff Constraints

External lifecycle work must preserve the artifact-promotion boundary 0.44
closed:

- Promotion remains a `DeploymentPlanV1` transformation, not an install
  shortcut.
- Role artifact identity, target embedded config, materialization evidence,
  promotion plan lineage, and promotion provenance must be reused instead of
  re-modeled.
- `AuthorityReceiptV1` and `AuthorityDryRunEvidenceV1` are not artifact
  sources, consent artifacts, or proof of live deployment state.
- Receipt-backed artifact sources identify bytes only when their artifact
  evidence is complete and verified independently of live-state claims.
- External lifecycle receipts may cite promotion evidence, but live inventory
  remains truth for completion.

## Problem

Some roles may be part of the runtime topology but not controlled solely by the
deployment authority.

Example:

```text
project_instance is controlled by the user
```

If a user, project hub, governance canister, customer-owned controller,
delegated install canister, or multisig authority controls the canister, that
canister is in a different trust domain for upgrade authority. A deployment
upgrade cannot push there without consent, delegation, or external completion.

Canic must distinguish:

- deployment authority can upgrade;
- deployment authority can propose;
- deployment authority can verify externally completed upgrade;
- deployment authority must not touch.

The design uses "external lifecycle subject" and "consent authority" rather
than assuming every externally controlled canister is owned by one human user.

---

## Lifecycle Authority

0.45 extends lifecycle behavior for the canonical control classes. It should
not rename or fork the classification model.

For each role/canister, the plan should record:

```text
LifecycleAuthorityV1 {
  role,
  canister_id,
  control_class,

  observed_controllers,
  expected_deployment_controllers,
  external_controllers,

  lifecycle_mode,
  allowed_upgrade_modes,

  consent_requirements,
  verification_requirements,

  control_observation_id,
  control_observation_digest,
  observed_at,
  inventory_epoch,

  blockers,
  warnings,
}
```

The lifecycle mode should be explicit:

```text
LifecycleModeV1 {
  DirectDeploymentAuthority,
  ProposalRequired,
  DelegatedInstallRequired,
  ExternalCompletionOnly,
  VerifyOnly,
  MustNotTouch,
  UnknownUnsafeBlocked,
}
```

Controller set classification is not itself consent. A jointly controlled
canister means the lifecycle planner has enough information to determine which
consent or delegation policy is required; it does not automatically mean Canic
may upgrade it.

Default projection:

| Control Class | Default Lifecycle Behavior |
| --- | --- |
| `DeploymentControlled` | Directly executable through the normal guarded deployment path. |
| `CanicManagedPool` | Executable only through pool-aware lifecycle rules; do not treat as ordinary role control unless pool policy allows. |
| `ExternallyImported` | Proposal/verify only by default. No unilateral mutation. |
| `JointlyControlled` | Requires explicit delegated or joint-controller policy before mutation. Otherwise proposal/verify. |
| `UserControlled` | Consent required. Proposal/verify path by default. |
| `UnknownUnsafe` | Must not touch. May report only. |

## Upgrade Coordination

For user-owned or externally controlled canisters, the flow is not:

```text
deployment authority pushes upgrade
```

It is:

```text
plan proposes upgrade
Canic proves artifact/config/trust identity
user or external controller consents or executes delegated step
receipt records result
live inventory verifies postcondition
```

The invariant is:

```text
Consent is not completion.
Reported completion is not verification.
Supplied observation evidence is not live verification.
Verification requires fresh deployment-truth inventory evidence.
```

### Lifecycle Plan

Partial deployment semantics belong in a named passive artifact:

```text
ExternalLifecyclePlanV1 {
  lifecycle_plan_id,
  lifecycle_plan_digest,

  deployment_plan_id,
  deployment_plan_digest,

  lifecycle_authority_rows,

  directly_executable_role_upgrades,
  proposed_external_role_upgrades,
  blocked_role_upgrades,

  dependency_blockers,
  protected_call_implications,
  residual_exposure,

  status,
}
```

`ExternalUpgradeProposalV1` is a per-role/per-canister projection from this
lifecycle plan. It must not be generated as a free-floating role-only artifact.

### Proposal

```text
ExternalUpgradeProposalV1 {
  proposal_id,
  proposal_digest,

  deployment_plan_id,
  deployment_plan_digest,

  lifecycle_plan_id,
  lifecycle_plan_digest,

  promotion_plan_id,
  promotion_plan_digest,
  promotion_provenance_id,
  promotion_provenance_digest,

  canister_id,
  role,

  control_class,
  lifecycle_mode,

  observed_before,
  observed_before_digest,

  current_module_hash,
  target_module_hash,
  target_role_artifact_identity,
  target_canonical_embedded_config_sha256,

  root_trust_anchor,
  authority_profile_identity,

  required_external_action,
  required_consent,
  consent_subjects,
  consent_channel_kind,

  verification_policy,

  expires_at,
  supersedes_proposal_id,
}
```

The proposal digest makes archived proposals reject stale drift. The plan and
provenance digests prevent ID-only linkage from becoming authority. The
observed-before digest binds consent to the state the proposal was generated
against. `expires_at` prevents stale consent from being reused after the
deployment truth changes.

### Consent

Consent may be:

- explicit user action through an endpoint;
- controller-side execution of a generated command;
- delegated install authority;
- out-of-band external action recorded and verified by inventory.

0.45 should avoid assuming one consent mechanism for all deployments.

Consent requirements should be modeled explicitly:

```text
ConsentRequirementV1 {
  consent_subject_kind,
  required_principals,
  required_controller_set_digest,
  consent_channel_kind,
  required_action,
}
```

A consent subject may be a user principal, project hub canister, governance
canister, customer-owned controller, delegated install canister, or multisig
authority.

### Receipt

```text
ExternalUpgradeReceiptV1 {
  proposal_id,
  proposal_digest,

  consent_evidence,
  external_action_evidence,
  verification_evidence,

  observed_before,
  observed_after,

  verification_result,

  receipt_digest,
}
```

The receipt should not rely on an ambiguous `attempted_by` authority field.
Prefer names such as `reported_by`, `external_actor`, `observed_actor`, or
`delegated_executor`, then let verification decide whether the reported action
actually produced the expected live state.

Receipts remain evidence. Live inventory remains truth.

If a single receipt object is kept for the first implementation, its states
must be explicit:

```text
ExternalUpgradeStateV1 {
  Proposed,
  ConsentPending,
  ConsentGranted,
  ConsentRefused,
  ConsentExpired,
  DelegatedInstallAttempted,
  ExternalCompletionReported,
  VerificationPending,
  VerifiedComplete,
  VerificationFailed,
  Superseded,
}
```

## Partial Deployment Upgrade Semantics

User-controlled or externally controlled roles should not block upgrades of
unrelated directly managed roles.

The plan should separate:

```text
directly_executable_role_upgrades
proposed_external_role_upgrades
blocked_role_upgrades
```

A directly managed role may still be blocked by dependencies. Protected-call
readiness, role epochs, topology changes, verifier state, or explicit role
dependencies can require ordering across control classes.

Pending external lifecycle work must remain visible in receipts and inventory.
It should not disappear because the directly managed subset completed.

## Consent Surface Requirements

0.45 defines the proposal, receipt, and verification contract.

It does not implement the frontend or SDK consent UX.

External upgrade proposals must include enough identity for a frontend, CLI, or
external controller to present a safe consent request later:

```text
artifact identity
artifact provenance digest
canonical embedded config digest
current module hash
target module hash
root trust anchor
authority profile identity
required external action
verification policy
```

The proposal should make clear whether consent authorizes a direct install,
delegated install, or externally completed upgrade.

It should also make clear when the proposal is derived from a 0.44 promotion
plan and provenance report.

## Critical Fix Path

Critical bug fixes for user-owned roles need an explicit operational flow.

The plan should say:

- which canisters are affected;
- whether deployment authority has direct authority;
- which external consent is required;
- whether protected-call role epochs or verifier cascades must change;
- which directly managed roles may be patched while user-controlled roles
  remain pending, if dependency checks allow it;
- what remains vulnerable until external action completes.

Unresolved external consent must be reported as residual exposure. This must
not be hidden inside a normal deployment upgrade report.

The passive critical-fix report shape is:

```text
CriticalExternalFixReportV1 {
  fix_id,
  severity,
  affected_roles,
  affected_canisters,

  directly_patchable_roles,
  externally_blocked_roles,
  dependency_blocked_roles,

  required_external_actions,
  protected_call_implications,

  residual_exposure,
  operator_next_steps,

  report_digest,
}
```

A critical fix report may say "directly managed roles patched, externally
controlled role still vulnerable pending external consent." It must not render
the deployment as simply upgraded if unresolved external lifecycle work
remains.

## 0.40 Protected-Call Implications

External/user-owned lifecycle changes can affect protected internal calls.

0.45 must account for:

- AppIndex role reassignment;
- subnet registry role changes;
- role epoch bumps;
- verifier cascade completion;
- stale proof TTL exposure;
- root key/verifier readiness;
- audience canister replacement.

A user-owned upgrade is not complete for protected-call purposes until the
relevant role and verifier state is consistent or explicitly recorded as
pending.

The verification policy should be explicit:

```text
ExternalUpgradeVerificationPolicyV1 {
  require_module_hash,
  require_controller_set,
  require_embedded_config_digest,
  require_role_epoch,
  require_verifier_cascade_complete,
  require_root_key_ready,
  require_audience_canister_match,
  max_observation_age,
}
```

The verification result should carry both expectation and observation:

```text
ExternalUpgradeVerificationResultV1 {
  status,
  observed_module_hash,
  expected_module_hash,
  observed_controllers,
  expected_controller_policy,
  observed_embedded_config_sha256,
  expected_embedded_config_sha256,
  protected_call_readiness,
  blockers,
  warnings,
  verified_at,
  inventory_receipt_id,
  inventory_receipt_digest,
}
```

Verification observations must carry an explicit source:

```text
ExternalVerificationObservationSourceV1 {
  SuppliedObservation,
  DeploymentTruthInventory,
}
```

`SuppliedObservation` may prove that archived evidence is internally
consistent, but it must not produce final live verified-complete lifecycle
status. `DeploymentTruthInventory` may produce verified completion only when
the observation is derived from a deployment-truth check/inventory artifact,
is bound by check and inventory IDs/digests, and satisfies the required module,
controller/control-class, embedded-config, protected-call, and freshness
postconditions.

## Commands

Tentative public operator surface:

```text
canic deploy external plan --deployment <deployment>
canic deploy external check --request external-plan.json
canic deploy external verify --request external-verification.json
canic deploy external list-pending --deployment <deployment>
```

Detailed report builders should stay under an advanced inspection namespace:

```text
canic deploy external inspect lifecycle-authority --request ...
canic deploy external inspect proposal --request ...
canic deploy external inspect consent --request ...
canic deploy external inspect verification --request ...
canic deploy external inspect critical-fix --request ...
```

If a `propose` command is added, it must derive from a checked lifecycle plan:

```text
canic deploy external propose --from-plan external-plan.json --role <role>
```

No free-floating role-only proposals.

The exact user-facing consent surface may be application-specific, but the
Canic plan and receipt shapes should be stable.

---

## Non-Goals

0.45 should not:

- assume all user-owned canisters expose the same consent endpoint;
- bypass IC controller authority;
- silently upgrade user-controlled canisters;
- treat proposal receipts as proof without live verification;
- invent a new promotion model.
- create ad hoc external proposals detached from `ExternalLifecyclePlanV1`.

## Implementation Slices

### Slice 0: 0.44 Evidence Reuse Boundary

Define how 0.45 references `DeploymentPlanV1`, artifact identity, promotion
provenance, embedded config, materialization evidence, and execution receipt
wrappers. No new artifact identity model.

### Slice 1: Lifecycle Authority Projection

Project from `CanisterControlClassV1` into `LifecycleAuthorityV1`. Add source
guards proving no second control-class enum or user/external classification
path is introduced.

### Slice 2: External Lifecycle Plan

Produce:

```text
directly_executable_role_upgrades
proposed_external_role_upgrades
blocked_role_upgrades
```

with dependency blockers, protected-call implications, and residual exposure.

### Slice 3: External Upgrade Proposal

Generate digest-pinned proposals from the lifecycle plan, not ad hoc role-only
proposals.

### Slice 4: External Completion And Verification

Record consent/action evidence, then verify against fresh inventory.
If 0.45 does not own a new live inventory crawler, the narrow acceptable bridge
is to verify against existing `DeploymentCheckV1` inventory artifacts and keep
supplied observations as consistency evidence only.

### Slice 5: Critical Fix And Residual Exposure Reporting

Render explicit affected-canister and unresolved-consent reports for critical
fixes.

### Slice 6: CLI Surface Closeout

Keep the public workflow small and move detailed artifacts under
`canic deploy external inspect ...`.

---

## Exit Criterion

0.45 is complete when:

```text
Canic can project existing CanisterControlClassV1 observations into lifecycle
authority, partition a DeploymentPlanV1 into directly executable, externally
proposed, and blocked role upgrades, generate digest-pinned external upgrade
proposals that reference the same artifact/config/provenance evidence used by
deployment truth, and verify externally completed upgrades against fresh
inventory without treating proposal, consent, or reported completion as
unilateral deployment authority.
```
