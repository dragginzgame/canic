# 0.41 Design: Deployment Flexibility

## Status

Tentative design notes for the 0.41 line.

This document is intentionally exploratory, but it now takes a firmer position
on the first line of work:

```text
0.41 makes deployment explicit, inspectable, comparable, and receipted.
It does not need to replace the install engine yet.
```

The goal is to expose the current deployment assumptions as typed data before
Canic makes that data load-bearing. Execution changes should come only after
the plan and receipt model has survived real use.

0.41 starts from this constraint:

```text
The Internet Computer does not give us a separate testnet.
Canic users need to run multiple real deployments with different roots,
controllers, role versions, wasm sources, configs, and promotion lanes.
```

The current Canic flow is good at the standard path:

```text
one fleet config
one root
one controller set
one role -> one discovered canister package
one release set staged into root
```

That is too rigid for Toko-style operations, test deployments, v2 deployments,
canary roles, per-role wasm experiments, and long-lived IC deployments where
different canisters intentionally run different versions.

---

## Core Decision

0.41 separates these concepts:

```text
TrustDomain  = one root, one root trust anchor
Fleet        = runtime topology template
Deployment   = TrustDomain + Fleet + AuthorityProfile + RoleArtifacts
Plan         = materialized deployment intent
Receipt      = observed deployment result
```

This replaces the earlier overloaded use of "fleet". A fleet does not own a
root by itself. A deployment does.

Examples:

```text
toko runtime fleet template
  -> deployment toko-prod    using prod root trust domain
  -> deployment toko-staging using staging root trust domain
  -> deployment toko-v2      using v2 root trust domain
```

The runtime topology can be shared. The trust domain, controllers, resolved
artifacts, and install receipts are deployment-specific.

---

## Core Question

How can Canic make deployment topology explicit and configurable without
turning runtime config into a general deployment scripting language?

The important split is:

```text
runtime topology config:
  what the fleet means after canisters are installed

deployment intent:
  which trust domain, controllers, artifacts, variants, and versions are used
  for one deployment of that fleet
```

0.41 should keep these concepts separate. Deployment intent may select a named
runtime variant, but it must not define that variant's runtime semantics.

---

## Friction Today

### Single-root mental model

Canic currently assumes a standard root-owned topology. That makes the happy
path simple, but makes it awkward to operate:

- production root
- staging root
- Toko test root
- v2 root
- customer-specific root
- forked root running a candidate wasm
- temporary root for destructive migration rehearsal

Those are not just different `ICP_ENVIRONMENT` values. They may need different
canister IDs, controllers, keys, root wasm, child wasm, config variants, and
promotion history.

### Controller sets are too global

`controllers = []` is currently a broad config-level idea. Real deployments
need more precision:

- root controllers
- non-root managed canister controllers
- operator identities allowed to stage artifacts
- emergency controllers
- canister-specific controller overrides
- test deployment controllers that must not leak into production

Controller policy is deployment authority. It should not be accidentally
coupled to the runtime behavior of every role.

### Role artifacts are not first-class deployment inputs

Host discovery can find packages by workspace metadata or `canister_<role>`
names. That works for a normal workspace, but it is awkward for:

- role `project_instance` running version A in prod and version B in a v2
  deployment
- role `project_hub` using a locally built wasm while everything else uses
  published artifacts
- role `wasm_store` using the canonical default in one deployment and a
  candidate source in another
- one role built with a different Cargo profile or feature set
- one role supplied as a fixed `.wasm.gz` digest for promotion

The deployer needs a role-to-artifact matrix, not only package discovery.

### Runtime config and deploy config are entangled

`CANIC_CONFIG_PATH` drives build-time config embedding. That is necessary for
strict configured canisters, but it makes experiments expensive when the thing
being varied is operational rather than semantic:

- same role, different controllers
- same role, different root canister ID
- same role, different install network
- same role, different artifact source
- same runtime topology, different deployment target

These should not all require copying and editing the whole fleet config.

### Promotion is not a first-class object

Canic can build and stage release sets, but deployment promotion is still too
implicit. Operators need to answer:

- which exact wasm digest was tested?
- which deployment ran it?
- which config digest was embedded?
- which root staged it?
- which controllers were installed?
- which roles intentionally diverge from the production baseline?
- what is safe to promote unchanged?

Without a promotion object, every test deployment becomes a set of local
conventions.

### Test fleet lifecycle is expensive

Because there is no IC testnet, test deployments are real deployments. They
need cheap creation, repeatable teardown, clear naming, and isolation from
production identities. The current standard flow is biased toward one canonical
local/demo topology, not many concurrent remote deployments.

---

## Goals

0.41 should make these operations straightforward:

- describe the current implicit deployment as a typed inventory;
- render a deployment plan before mutating anything;
- record deployment receipts from the existing install path;
- compare planned intent against observed receipts;
- identify unresolved assumptions explicitly;
- separate trust domain, runtime fleet template, authority, and artifacts;
- keep test deployment authority separate from production authority;
- make PocketIC exercise the same plan shape used by real deployments;
- lay the groundwork for later per-role artifact overrides and plan execution.

Future 0.42+ work can then:

- execute from a materialized plan;
- allow per-role artifact overrides safely;
- promote selected artifacts from one deployment receipt to a target plan;
- create new roots or use existing roots from typed plan data;
- manage controller changes through authority profiles.

---

## Non-Goals

0.41 should not:

- introduce multi-root runtime consensus;
- let child canisters trust multiple roots implicitly;
- turn `canic.toml` into a general-purpose deployment script;
- hide controller changes inside runtime config;
- add postpaid billing or storage accounting policy;
- replace backup/restore manifests;
- require every project to adopt the same release workflow;
- make deployment overlays a second runtime config;
- require execute-from-plan before the plan model is proven.

The first design should make deployment intent explicit. It should not make
deployment clever.

## Terminology

### TrustDomain

A trust domain is one root and one root trust anchor:

```text
TrustDomain {
  root_canister: Create | Existing(Principal),
  icp_environment,
  root_key_material,
}
```

The TOML representation may use compact syntax, but the internal model must be
a typed variant. `"create"` must not survive as an untyped magic string beyond
parsing.

Different trust domain means different deployment authority and different
runtime trust state.

### Fleet

A fleet is a runtime topology template:

```text
Fleet {
  name,
  runtime_config,
  declared_runtime_variants,
}
```

The fleet says what roles, subnets, pools, auth policy, metrics, and runtime
behavior mean after install. It is closest to today's
`fleets/<name>/canic.toml`.

A fleet does not, by itself, imply a production root, staging root, controllers,
or deployment target.

### Runtime Variant

A runtime variant is a named runtime config choice declared by the fleet:

```text
RuntimeVariant {
  name,
  role_config_overrides,
  config_digest,
}
```

Deployment intent may select a runtime variant:

```text
runtime_variant = "project-instance-v2"
```

Deployment intent must not define arbitrary role config files that mutate
runtime semantics. Variant contents belong to the fleet/runtime config layer.

### AuthorityProfile

An authority profile is deployment-side controller and operator policy:

```text
AuthorityProfile {
  root_controllers,
  managed_canister_controllers,
  role_controller_overrides,
  staging_principals,
  emergency_controllers,
}
```

This lets a test deployment use test controllers while sharing the same runtime
fleet template.

### RoleArtifact

Every deployable role resolves to one concrete artifact:

```text
RoleArtifact {
  role,
  source,
  build_profile,
  runtime_variant,
  wasm_sha256,
  candid_sha256,
  config_sha256,
}
```

The source is one of:

```text
workspace package
published package
local wasm file
local wasm.gz file
previous deployment artifact
canonical wasm_store default
```

`wasm_store` should be modeled as a normal role artifact with a default source,
not as an invisible special case. The default source may still resolve through
the existing canonical package or generated wrapper path.

### Deployment

A deployment combines runtime meaning, trust, authority, and artifacts:

```text
Deployment {
  name,
  trust_domain,
  fleet,
  runtime_variant,
  authority_profile,
  role_artifacts,
}
```

`toko-prod` and `toko-staging` can share a fleet while using different trust
domains and authority profiles.

### DeploymentPlan

A plan is materialized deployment intent:

```text
DeploymentPlanV1 {
  deployment,
  resolved_inputs,
  unresolved_assumptions,
}
```

It answers "what would Canic install where?" before Canic mutates anything.

Unresolved assumptions are first-class output, not a footer. A preview plan may
contain unresolved assumptions. An executable plan may not.

### DeploymentReceipt

A receipt is observed deployment result:

```text
DeploymentReceiptV1 {
  plan_id,
  deployment,
  root_principal,
  controller_policy_applied,
  role_artifact_digests,
  config_digests,
  installed_canister_ids,
  staging_receipts,
  command_result,
}
```

0.41 can emit receipts from the existing install path before Canic can execute
from a plan.

## Runtime Config vs Deployment Intent

This boundary is non-negotiable:

```text
runtime config owns what canisters mean
deployment intent owns which declared meaning is selected and where it runs
```

Allowed in deployment intent:

- target trust domain;
- target ICP environment;
- existing root or create-root mode;
- authority profile;
- selected runtime variant name;
- role artifact source;
- build profile;
- expected digests;
- promotion source receipt.

Not allowed in deployment intent:

- ad hoc pool policy;
- ad hoc auth policy;
- ad hoc role config files that redefine runtime behavior;
- endpoint policy changes;
- runtime feature switches not declared by the fleet.

If Canic later needs deployment-specific runtime behavior, that behavior should
be declared as a named runtime variant in the fleet config first. The
deployment plan selects the variant by name.

## Proposed Config Shape

This is deliberately tentative.

One possible shape:

```text
fleets/
  toko/
    canic.toml
    deploy.prod.toml
    deploy.staging.toml
    deploy.v2.toml
```

`canic.toml` remains the runtime topology base and declares runtime variants.

Example sketch:

```toml
[fleet]
name = "toko"

[runtime_variants.default]

[runtime_variants.project_instance_v2]
# Runtime-owned variant semantics live here, not in deploy.prod.toml.
```

`deploy.<target>.toml` is host-only deployment intent:

```toml
[deployment]
name = "toko-prod"
fleet = "toko"
runtime_variant = "default"

[trust_domain]
environment = "ic"
root = { existing = "aaaaa-aa" }

[authority.root]
controllers = ["principal-a", "principal-b"]

[authority.managed]
controllers = ["aaaaa-aa"]

[roles.project_hub]
source = { package = "canister_project_hub" }
profile = "release"

[roles.project_instance]
source = { package = "canister_project_instance" }
profile = "release"

[roles.wasm_store]
source = { default = "canonical" }
profile = "release"
```

For a v2 deployment:

```toml
[deployment]
name = "toko-v2"
fleet = "toko"
runtime_variant = "project_instance_v2"

[trust_domain]
environment = "ic"
root = { create = true }

[roles.project_instance]
source = { package = "canister_project_instance_v2" }
profile = "release"
```

For a future candidate or canary, once overrides are safe:

```toml
[roles.project_instance]
source = { wasm_gz = "artifacts/project_instance-v2.wasm.gz" }
expected_sha256 = "..."
```

Open question: whether deployment overlays should live inside `canic.toml`
under `[deploy.<name>]`, or in separate files. Separate files still seem safer
because deployment authority is operational and easy to copy incorrectly.

## Deployment Plan Output

The first useful artifact is not install. It is plan rendering:

```bash
canic deploy plan toko --deployment staging
```

It should emit a stable plan:

```text
deployment: toko-staging
fleet: toko
runtime_variant: default
trust_domain:
  environment: ic
  root: Existing(aaaaa-aa)
authority:
  root controllers: [...]
  managed default controllers: [...]
roles:
  root:
    source: package canister_root
    config_sha256: ...
    wasm_sha256: ...
  project_hub:
    source: package canister_project_hub
    config_sha256: ...
    wasm_sha256: ...
  project_instance:
    source: package canister_project_instance
    config_sha256: ...
    wasm_sha256: ...
  wasm_store:
    source: canonical default
    config_sha256: ...
    wasm_sha256: ...
unresolved_assumptions:
  - ...
```

The same object should be serializable as JSON for CI and audit logs.

This gives Canic a clean approval point:

```text
plan -> inspect -> execute
```

instead of:

```text
run command -> discover what it built and installed afterward
```

0.41 does not need to implement the execute step. It should make the plan
accurate enough that execute-from-plan is a straightforward later change.

---

## Receipt Model

0.41 should emit deployment receipts from the existing install path.

That gives Canic immediate audit value without changing install semantics:

```text
existing install command -> DeploymentReceiptV1
```

The receipt should record:

- deployment name;
- fleet name;
- runtime variant;
- trust domain;
- root principal;
- controller policy observed or applied;
- role artifact digests;
- config digests;
- installed canister IDs;
- staging root/store receipts;
- command result;
- unresolved assumptions that were still present at execution.

Receipts give future promotion logic real data to learn from before promotion
syntax becomes load-bearing.

---

## Promotion Model

Promotion should move selected artifact digests into a target plan. It should
not blindly copy a whole source deployment.

Candidate flow:

```text
render plan for toko-v2
execute existing install path on test deployment
record deployment receipt
promote selected role artifact digests into a toko-prod target plan
re-resolve prod trust domain, authority profile, and runtime variant
recompute target config digests
execute prod plan in a later release
```

Promotion input:

```text
source deployment receipt
target deployment
roles to promote
target runtime variant
target authority profile
```

Important invariant:

```text
Promoted wasm artifacts may be byte-identical.
Embedded config often should not be.
```

If prod and staging use different roots, controllers, subnet IDs, key names, or
runtime variants, the target config digest must be recomputed for the target.
Operators should not assume promotion means byte-identical canister modules
unless the plan proves the embedded config is also identical.

Promotion must never carry source controllers or source root identity into the
target by default.

---

## Root and Trust Boundaries

0.41 must preserve the 0.40 trust model:

- root-signed proofs are bound to one root trust anchor;
- child canisters should not silently accept another deployment's root;
- deployment flexibility must not become multi-root authorization;
- root key material and cascaded state remain explicit.

That implies:

```text
different TrustDomain => different runtime trust state
```

If two deployments use different roots, promotion can reuse wasm artifacts but
not root trust state.

## PocketIC Plan Consumption

PocketIC should consume the same `DeploymentPlanV1` shape as real deployments,
or at minimum a lossless test specialization of it.

This is important because local tests are the cheapest way to keep plan
resolution honest. If PocketIC uses only a lower-level artifact resolver, the
plan model will mostly be tested on real deployments, where mistakes are more
expensive.

Initial target:

```text
existing PocketIC root harness -> render/consume DeploymentPlanV1
```

The harness may still execute through test-only install helpers, but its inputs
should match the plan shape:

- fleet;
- runtime variant;
- trust domain mode;
- authority profile;
- role artifacts;
- unresolved assumptions.

## Toko Deployment Use Cases

### Test Deployment

Toko needs a real IC deployment for testing because there is no public testnet.

Requirements:

- separate trust domain;
- separate controllers;
- same or near-same runtime fleet template;
- cheap rebuild/reinstall;
- clear label in receipts and metrics;
- no accidental production authority.

### V2 Deployment

Toko may run a full v2 topology beside production.

Requirements:

- root can be v2 or stable;
- selected roles can use v2 artifacts;
- selected roles can use named runtime variants;
- production remains untouched;
- comparison against prod baseline is easy.

### Per-Role Canary

Toko may want only `project_instance` on a candidate wasm.

Requirements:

- role artifact override;
- same or separate trust domain depending on risk;
- explicit canary deployment or canary role instance;
- digest-pinned rollback candidate.

### Customer or Tenant Deployment

Some users may need isolated deployment boundaries.

Requirements:

- named trust domain;
- named authority profile;
- shared runtime fleet template;
- independent upgrade cadence.

## 0.41 Implementation Slices

### Slice 1: Deployment Inventory

Add a read-only command that resolves the current standard deployment into an
inventory:

```bash
canic deploy inventory <fleet>
```

It should report:

- selected `canic.toml`;
- inferred fleet;
- inferred deployment name;
- role list;
- discovered package per role;
- build profile;
- root package;
- `wasm_store` role artifact source;
- current controller defaults;
- target ICP environment;
- unresolved assumptions as first-class output.

No install behavior changes.

### Slice 2: Plan JSON

Materialize the inventory as a `DeploymentPlanV1` JSON object.

The plan must include role artifact digests after build, or explicit unresolved
status before build.

No install behavior changes.

### Slice 3: Receipts From Existing Install

Record `DeploymentReceiptV1` from the current install path.

This gives Canic audit and comparison value before the plan controls install
execution.

No install behavior changes beyond writing the receipt.

### Slice 4: PocketIC Plan Consumption

Make the main root/test harness render or consume the same plan shape.

This exercises the model frequently without touching real IC deployments.

No production install behavior changes.

### Slice 5: Plan Comparison

Add read-only comparison:

```text
plan vs plan
plan vs receipt
receipt vs receipt
```

This should highlight artifact digest changes, config digest changes, trust
domain changes, controller changes, runtime variant changes, and unresolved
assumption changes.

No install behavior changes.

## Deferred Execution Slices

These are intentionally not required for 0.41.

### Execute From Plan

Make install consume a materialized plan instead of rediscovering everything
inside the command.

This is the first slice that should materially mutate deployment behavior.

### Authority Profile Application

Apply plan authority profiles during install and upgrade.

This should land with or after execute-from-plan, not before, so controller
policy cannot silently diverge between plan output and actual install behavior.

### Role Artifact Overrides

Allow deployment plans to override artifact source per role:

- package;
- package plus features/profile;
- local wasm;
- local wasm.gz;
- previous deployment artifact.

This should not become executable before install is plan-driven, otherwise a
candidate artifact can be deployed under legacy controller/default assumptions.

### Promotion Command

Create a target plan from a source receipt and selected role artifact digests.

Promotion should be plan generation first. Execution remains a separate step.

---

## Boundary Rules

- `canic-host` owns plan resolution, artifact resolution, receipt writing, and
  local command execution.
- `canic-cli` exposes operator commands and human output.
- `canic-core` should not learn host paths, package names, deployment overlays,
  or local artifact layouts.
- `canic-control-plane` should consume staged artifacts and runtime config; it
  should not parse deployment intent.
- Runtime canisters should receive explicit config/trust state, not infer
  deployment target names.
- Backup/restore remains separate; deployment receipts may reference backup
  plans but should not replace them.

---

## Design Risks

### Risk: deployment overlays become a second runtime config

Mitigation: deployment intent may select a named runtime variant. The variant's
runtime semantics must be declared by the fleet/runtime config layer.

### Risk: root trust becomes ambiguous

Mitigation: every plan names one trust domain. Different roots mean different
trust domains, even if deployments share a fleet and artifacts.

### Risk: role overrides bypass Canic invariants

Mitigation: role overrides remain non-executable until install is plan-driven.
Plan validation should later prove the role is configured, the artifact exports
expected Canic metadata, and the role reported by the canister matches the
planned role after install.

### Risk: test controllers leak into production

Mitigation: authority profiles are target-scoped and promotion never carries
source authority into the target by default.

### Risk: digest pins make normal development painful

Mitigation: allow unresolved plans for preview, but require digests for
execution and receipts.

### Risk: plan-only 0.41 feels incomplete

Mitigation: receipts and comparison make plans useful immediately. They expose
real assumptions before the install path depends on them.

---

## Open Questions

- Should deployment intent live in separate `deploy.<target>.toml` files or
  `[deploy.<target>]` sections inside `canic.toml`?
- How should runtime variants be declared without turning `canic.toml` into a
  templating language?
- Should Canic model a deployment group containing multiple deployments that
  share one fleet?
- Should controller policy be role-based, canister-ID-based, or both?
- How much of plan execution should be resumable after partial failure?
- Can `canic_standards` / metadata provide enough installed-canister role
  verification for role override safety?
- Should promotion ever allow byte-identical module promotion when embedded
  config differs only by target trust domain?

---

## First Recommendation

Start 0.41 with inventory, plan JSON, receipts, PocketIC plan consumption, and
plan comparison.

That gives Canic leverage without committing to install semantics:

```text
current implicit behavior -> explicit plan -> receipts -> comparison -> audited gaps
```

The first success criterion should be modest:

```text
Canic can explain and record the exact trust domain, runtime fleet, runtime
variant, controllers, role artifacts, config digests, and wasm_store source
used for a deployment, before and after the current install path runs.
```

That alone reduces deployment friction and gives future execute-from-plan work
a stable object to iterate on.
