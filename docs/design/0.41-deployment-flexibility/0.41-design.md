# 0.41 Design: Deployment Flexibility

## Status

Tentative design notes for the 0.41 line.

This document is intentionally exploratory. The goal is to name the deployment
friction clearly enough that the first implementation slices can reduce it
without committing Canic to the wrong long-term topology.

0.41 starts from this constraint:

```text
The Internet Computer does not give us a separate testnet.
Canic users need to run multiple real fleets with different roots,
controllers, role versions, wasm sources, configs, and promotion lanes.
```

The current Canic flow is good at the standard path:

```text
one fleet config
one root
one controller set
one role -> one discovered canister package
one release set staged into root
```

That is too rigid for Toko-style operations, test fleets, v2 fleets,
canary fleets, per-role wasm experiments, and long-lived IC deployments where
different canisters intentionally run different versions.

---

## Core Question

How can Canic make deployment topology explicit and configurable without
turning runtime config into a general deployment scripting language?

The important split is:

```text
runtime topology config:
  what the fleet means after canisters are installed

deployment plan:
  which concrete root, controllers, configs, artifacts, and versions are used
  to install or upgrade that fleet
```

0.41 should keep these concepts separate.

---

## Friction Today

### Single-root mental model

Canic currently assumes a standard root-owned topology. That makes the happy
path simple, but makes it awkward to operate:

- production root
- staging root
- Toko test root
- v2 root
- customer-specific root
- forked root running a candidate wasm
- temporary root for destructive migration rehearsal

Those are not just different `ICP_ENVIRONMENT` values. They may need different
canister IDs, controllers, keys, root wasm, child wasm, config, and promotion
history.

### Controller sets are too global

`controllers = []` is currently a broad config-level idea. Real deployments
need more precision:

- root controllers
- non-root managed canister controllers
- operator identities allowed to stage artifacts
- emergency controllers
- canister-specific controller overrides
- test fleet controllers that must not leak into production

Controller policy is deployment authority. It should not be accidentally
coupled to the runtime behavior of every role.

### Role artifacts are not first-class deployment inputs

Host discovery can find packages by workspace metadata or `canister_<role>`
names. That works for a normal workspace, but it is awkward for:

- role `project_instance` running version A in prod and version B in a v2 fleet
- role `project_hub` using a locally built wasm while everything else uses
  published artifacts
- role `wasm_store` using the canonical package in one fleet and a candidate
  package in another
- one role built with a different Cargo profile or feature set
- one role supplied as a fixed `.wasm.gz` digest for promotion

The deployer needs a role-to-artifact matrix, not only package discovery.

### Runtime config and deploy config are entangled

`CANIC_CONFIG_PATH` drives build-time config embedding. That is necessary for
strict configured canisters, but it makes experiments expensive when the thing
being varied is operational rather than semantic:

- same role, different controllers
- same role, different root canister ID
- same role, different install network
- same role, different artifact source
- same runtime topology, different staging fleet

These should not all require copying and editing the whole fleet config.

### Promotion is not a first-class object

Canic can build and stage release sets, but deployment promotion is still too
implicit. Operators need to answer:

- which exact wasm digest was tested?
- which fleet ran it?
- which config digest was embedded?
- which root staged it?
- which controllers were installed?
- which roles intentionally diverge from the production baseline?
- what is safe to promote unchanged?

Without a promotion object, every test fleet becomes a set of local conventions.

### Test fleet lifecycle is expensive

Because there is no IC testnet, test fleets are real deployments. They need
cheap creation, repeatable teardown, clear naming, and isolation from
production identities. The current standard flow is biased toward one canonical
local/demo topology, not many concurrent remote fleets.

---

## Goals

0.41 should make these operations straightforward:

- create a new named fleet without copying a whole workspace;
- install a fleet with an explicit root canister ID or create a new root;
- choose controllers per deployment and per role;
- choose role wasm sources per fleet;
- run role X from a candidate package while role Y stays on prod;
- pin a role to an existing `.wasm` or `.wasm.gz` digest;
- build a deployment plan before executing it;
- compare a fleet against a baseline;
- promote a tested artifact set from one fleet to another;
- keep test fleet authority separate from production authority;
- audit what changed between two deployment plans.

---

## Non-Goals

0.41 should not:

- introduce multi-root runtime consensus;
- let child canisters trust multiple roots implicitly;
- turn `canic.toml` into a general-purpose deployment script;
- hide controller changes inside runtime config;
- add postpaid billing or storage accounting policy;
- replace backup/restore manifests;
- require every project to adopt the same release workflow.

The first design should make deployment intent explicit. It should not make
deployment clever.

---

## Working Model

### Fleet

A fleet is a named deployment boundary:

```text
fleet = root + managed canisters + runtime config + deployment authority
```

Examples:

```text
toko-prod
toko-staging
toko-v2
toko-project-instance-canary
customer-a-prod
```

A fleet name is not enough. A fleet also needs an environment and a root
identity:

```text
FleetTarget {
  fleet_name,
  icp_environment,
  root_canister_id,
  authority_profile,
}
```

### Runtime config

Runtime config remains the data canisters embed and use after install:

```text
roles
subnets
pool policy
auth policy
metrics profile
runtime feature switches
```

This is closest to today's `fleets/<name>/canic.toml`.

### Deployment plan

A deployment plan is host-side, explicit, and auditable:

```text
DeploymentPlan {
  target,
  runtime_config,
  controllers,
  role_artifacts,
  install_mode,
  promotion_source,
}
```

It answers "what will be installed where?" before Canic mutates anything.

### Role artifact

Every deployable role should be able to resolve to one concrete artifact:

```text
RoleArtifact {
  role,
  source,
  build_profile,
  config_source,
  wasm_sha256,
  candid_sha256,
}
```

The source is one of:

```text
workspace package
published package
local wasm file
local wasm.gz file
previous deployment artifact
generated canonical wasm_store wrapper
```

The digest is mandatory once a plan is materialized.

### Authority profile

Controller and operator authority should be named independently from runtime
roles:

```text
AuthorityProfile {
  root_controllers,
  managed_canister_controllers,
  role_controller_overrides,
  staging_principals,
  emergency_controllers,
}
```

This lets a test fleet use test controllers while keeping the same runtime role
topology.

## Proposed Config Shape

This is deliberately tentative. The important decision is to separate runtime
config from deployment overlays.

One possible shape:

```text
fleets/
  toko/
    canic.toml
    deploy.prod.toml
    deploy.staging.toml
    deploy.v2.toml
```

`canic.toml` remains the runtime topology base.

`deploy.<target>.toml` is host-only deployment intent:

```toml
[target]
fleet = "toko-prod"
environment = "ic"
root_canister_id = "aaaaa-aa"

[authority.root]
controllers = ["principal-a", "principal-b"]

[authority.managed]
controllers = ["aaaaa-aa"]

[roles.project_hub]
source = { package = "canister_project_hub" }
profile = "release"
config = "canic.toml"

[roles.project_instance]
source = { package = "canister_project_instance_v2" }
profile = "release"
config = "variants/project-instance-v2.toml"

[roles.wasm_store]
source = { package = "canic-wasm-store" }
profile = "release"
```

For a candidate or canary:

```toml
[roles.project_instance]
source = { wasm_gz = "artifacts/project_instance-v2.wasm.gz" }
expected_sha256 = "..."
```

For local test:

```toml
[target]
fleet = "toko-v2-local"
environment = "local"
root_canister_id = "create"

[authority.root]
controllers = ["current-identity"]
```

Open question: whether deployment overlays should live inside `canic.toml`
under `[deploy.<name>]`, or in separate files. Separate files seem safer for
now because deployment authority is operational and easy to copy incorrectly.

## Deployment Plan Output

The first useful artifact is not install. It is plan rendering:

```bash
canic deploy plan toko --target staging
```

It should emit a stable plan:

```text
fleet: toko-staging
environment: ic
root: create or principal
controllers:
  root: [...]
  managed default: [...]
roles:
  root:
    source: package canister_root
    config_sha256: ...
    wasm_sha256: ...
  project_hub:
    source: package canister_project_hub
    config_sha256: ...
    wasm_sha256: ...
  project_instance:
    source: package canister_project_instance_v2
    config_sha256: ...
    wasm_sha256: ...
```

The same object should be serializable as JSON for CI and audit logs.

This gives Canic a clean approval point:

```text
plan -> inspect -> execute
```

instead of:

```text
run command -> discover what it built and installed afterward
```

## Promotion Model

Promotion should move resolved artifacts and config digests, not vague version
labels.

Candidate flow:

```text
build plan for toko-v2
execute plan on test fleet
record deployment receipt
promote selected role artifacts into toko-prod plan
execute prod plan
```

Promotion input:

```text
source deployment receipt
target fleet
roles to promote
controller policy for target
```

Promotion should not copy test controllers or test root identity into prod.
Those belong to the target authority profile.

---

## Root and Trust Boundaries

0.41 must preserve the 0.40 trust model:

- root-signed proofs are bound to one root trust anchor;
- child canisters should not silently accept another fleet's root;
- deployment flexibility must not become multi-root authorization;
- root key material and cascaded state remain explicit.

That implies:

```text
different fleet root => different trust domain
```

If two fleets use different roots, promotion can reuse wasm artifacts but not
runtime trust state.

## Toko Deployment Use Cases

### Test fleet

Toko needs a real IC fleet for testing because there is no public testnet.

Requirements:

- separate root;
- separate controllers;
- same or near-same runtime topology;
- cheap rebuild/reinstall;
- clear label in receipts and metrics;
- no accidental production authority.

### V2 fleet

Toko may run a full v2 topology beside production.

Requirements:

- root can be v2 or stable;
- selected roles can use v2 artifacts;
- selected roles can use altered config;
- production remains untouched;
- comparison against prod baseline is easy.

### Per-role canary

Toko may want only `project_instance` on a candidate wasm.

Requirements:

- role artifact override;
- same parent/root topology;
- explicit canary fleet or canary role instance;
- digest-pinned rollback candidate.

### Customer or tenant fleet

Some users may need isolated deployment boundaries.

Requirements:

- named root;
- named controller policy;
- same runtime templates;
- independent upgrade cadence.

## Implementation Slices

### Slice 1: Deployment Inventory

Add a read-only command that resolves the current standard fleet into a
deployment inventory:

```bash
canic deploy inventory <fleet>
```

It should report:

- selected `canic.toml`;
- role list;
- discovered package per role;
- build profile;
- root package;
- implicit `wasm_store` source;
- current controller defaults;
- target ICP environment;
- unresolved assumptions.

No install behavior changes.

### Slice 2: Plan JSON

Materialize the inventory as a `DeploymentPlanV1` JSON object.

The plan must include role artifact digests after build, or explicit unresolved
status before build.

No install behavior changes.

### Slice 3: Authority Profiles

Introduce host-side authority profiles for controllers and staging operators.

Use them only in plan output first. Do not change installation until plan
comparison and validation are covered.

### Slice 4: Role Artifact Overrides

Allow deployment plan overlays to override artifact source per role:

- package;
- package plus features/profile;
- local wasm;
- local wasm.gz;
- previous deployment artifact.

Start with one role override in tests.

### Slice 5: Plan Execute

Make install consume a materialized plan instead of rediscovering everything
inside the command.

This is the first slice that should mutate deployment behavior.

### Slice 6: Promotion Receipts

Record deployment receipts with:

- fleet target;
- root principal;
- controller policy;
- role artifact digests;
- config digests;
- installed canister IDs;
- staging root/store receipts;
- command result.

Then add a promotion command that creates a target plan from a source receipt.

## Boundary Rules

- `canic-host` owns plan resolution, artifact resolution, and local command
  execution.
- `canic-cli` exposes operator commands and human output.
- `canic-core` should not learn host paths, package names, or local artifact
  layouts.
- `canic-control-plane` should consume staged artifacts and runtime config; it
  should not parse deployment overlays.
- Runtime canisters should receive explicit config/trust state, not infer
  deployment target names.
- Backup/restore remains separate; deployment receipts may reference backup
  plans but should not replace them.

---

## Design Risks

### Risk: deployment overlays become a second runtime config

Mitigation: overlays may select artifacts, controllers, root IDs, and build
inputs. They should not redefine runtime policies that canisters depend on
after install.

### Risk: root trust becomes ambiguous

Mitigation: every plan names one root trust domain. Different roots mean
different fleets, even if they share artifacts.

### Risk: role overrides bypass Canic invariants

Mitigation: plan validation should prove the role is configured, the artifact
exports expected Canic metadata, and the role reported by the canister matches
the planned role after install.

### Risk: test controllers leak into production

Mitigation: authority profiles are target-scoped and promotion never carries
source authority into the target by default.

### Risk: digest pins make normal development painful

Mitigation: allow unresolved plans for preview, but require digests for
execution and receipts.

---

## Open Questions

- Should deployment overlays be separate files or `[deploy.<target>]` sections
  inside `canic.toml`?
- Should a fleet always map to exactly one root, or should Canic model a higher
  "deployment group" containing multiple roots?
- Should controller policy be role-based, canister-ID-based, or both?
- How should Canic represent "create a new root" versus "use this existing
  root" in a durable plan?
- Should local PocketIC test harnesses consume the same `DeploymentPlanV1`, or
  only the same lower-level artifact resolver?
- How much of plan execution should be resumable after partial failure?
- Should `wasm_store` remain implicit in plan output, or become a normal role
  with a default source?
- Can `canic_standards` / metadata provide enough installed-canister role
  verification for role override safety?

---

## First Recommendation

Start 0.41 with a read-only deployment inventory and plan-rendering slice.

That gives us leverage without committing to install semantics:

```text
current implicit behavior -> explicit plan -> audited gaps
```

Once the current standard deployment can be rendered as a plan, we can add
overrides one at a time and see exactly which assumptions are hardcoded.

The first success criterion should be modest:

```text
Canic can explain the exact root, config, controllers, role artifacts, and
wasm_store source it would use for a named fleet before it mutates anything.
```

That alone reduces a lot of deployment friction and gives future 0.41 slices a
stable object to iterate on.
