# Canic 0.88 Design: Artifact Durability and Typed Config Errors

Date: 2026-07-13

## Status

  is published as `0.88.1`, and Slice C is release-noted as `0.88.2` pending
  the human-owned version bump and push.

## Problem

The post-0.87 audit found three bounded authority gaps:

1. backup journals claim artifact directories are durable immediately after
   unsynced renames;
2. CLI evidence and report files can be observed partially written; and
3. the fleet-config owner returns boxed and string-built failures instead of
   preserving typed causes.

0.88 closes three independent, bounded findings from the post-0.87 audit. They
share a correctness theme but do not share an implementation abstraction.

## Decisions

### Slice A - Durable backup artifact commit

`canic-backup` owns the durability of downloaded snapshot directories.

- Add one private artifact-directory commit function in `canic-backup` and use
  it from direct snapshot capture and backup-runner finalization.
- Require the temporary artifact path to be a unique sibling of the canonical
  path so publication stays on one filesystem.
- Walk without following symlinks or reopening a checked path through a
  following operation. Accept only regular files and directories; reject
  symlinks and every other entry type.
- Sync every regular file, then sync nested directories bottom-up, including
  the temporary artifact root directory itself.
- Publish the verified temporary root under the canonical name with an atomic
  no-replace operation. An existence check followed by ordinary replacing
  rename is not acceptable.
- Sync the owning parent directory after publication.
- The logical order is durable artifact commit, durable
  `ArtifactState::Durable` journal transition, then completed metric. State and
  metric may be prepared in one uncommitted journal copy and durably replaced
  together, but neither becomes observable through the caller's journal if
  that write fails.

Publication can succeed before the durable journal transition. Resume handles
that exact interruption window without trusting an arbitrary destination:

- the persisted journal entry must be `ChecksumVerified`, name the canonical
  path, contain the expected checksum, and its recorded temporary path must be
  absent on disk;
- the canonical directory is walked with the same no-follow rules and its
  checksum is reverified;
- the complete tree and owning directory are resynced; and
- only then may the journal advance durably to `ArtifactState::Durable` and
  record completion.

Any canonical destination that cannot satisfy all of those conditions is
unrelated or unverifiable and fails closed. Earlier `Created` and `Downloaded`
states never adopt a pre-existing canonical destination.

The existing journal schema, artifact directory shape, checksum algorithm,
manifest schema, and error reporting remain unchanged. Resume policy gains
only the explicit verified post-publication recovery rule above.

### Slice B - Publication-atomic CLI file output

The existing host durable byte writer remains the owner of ordinary host/CLI
single-file replacement.

- Serialize complete JSON or text bytes before touching the destination.
- Keep two explicit host entry points: the existing replacing `write_bytes`
  operation and one new no-overwrite `create_new_bytes` operation.
- Delegate both entry points to one private commit implementation that owns
  sibling staging, complete writes, file sync, publication, cleanup, and parent
  directory sync. Do not expose a generic public mode/options API.
- Require every temporary file to be a unique sibling of its final path so
  staging and publication remain on one filesystem.
- Use atomic replacement for `write_bytes` and atomic no-replace publication
  for `create_new_bytes`.
- A destination preflight followed by ordinary replacing rename is not an
  acceptable create-new implementation because another process can create the
  target between those operations.
- Sync the final parent after either publication mode. If the platform cannot
  provide the required atomic publication or directory-sync semantics, return
  an explicit error rather than using a weaker fallback.
- Preserve each existing parent-directory contract: replacing shared CLI
  output continues to create requested parents, while deployment-plan output
  continues to reject a missing parent.
- When replacement creates a parent hierarchy, create it one level at a time
  and sync every new directory plus the parent entry that introduced it before
  returning success.
- Keep stdout rendering unchanged.

Serialization remains in the CLI output/report owners; the host helper accepts
only completed bytes. The slice does not create a general output service,
multi-file transaction, or configurable filesystem API, and it does not widen
into unrelated scaffold creation paths.

For this slice, publication-atomic means:

- before publication, failure leaves the final destination unchanged;
- after publication, a later durability failure may leave the complete new
  file visible, but the operation returns an error; and
- replace destinations contain either the old complete bytes or the new
  complete bytes, while create-new destinations are either absent or contain
  the complete new file. Partial final content is never exposed.

#### Closed Slice B inventory

Replace mode covers every file destination behind the existing shared CLI
`write_pretty_json`, `write_pretty_json_file`, and `write_text` helpers. The
current production families are build provenance, fleet adoption, restore
reports and run output, metrics, environment exports, cycles, deployment
catalogs, backup create/inspect/status/list/prune/manifest output, and evidence
gate output.

Create-new mode covers only `canic deploy plan --out`. No in-scope path may
open its final destination in truncating or incremental-write mode.

Explicitly excluded are stdout-only calls, scaffold-generated files governed
by scaffold's multi-file rollback contract, the cycles pending-operation log
governed by its own recovery protocol, tests and fixtures, and persistence
owned directly by host or backup subsystems.

### Slice C - Typed fleet-config failure boundary

`canic-host::release_set::config` owns configuration reads, projections, and
mutations and therefore owns their failure vocabulary.

- Add one focused `FleetConfigError` type in that subsystem.
- Preserve I/O and core configuration errors as sources. I/O failures retain
  a bounded operation kind and path; core-config failures retain their config
  operation context.
- Represent invalid names/kinds, fleet mismatches, missing declarations,
  mutation conflicts, and rollback failure with bounded typed variants.
- Do not create a separate variant for every rendered sentence.
- `RollbackFailed` preserves both the original mutation failure and the
  rollback failure as typed data. Neither cause may be discarded or embedded
  only in display text. The concrete representation may use a boxed concrete
  error or a smaller typed mutation-cause value; boxed dynamic errors remain
  prohibited.
- Remove boxed dynamic-error returns and string-built error construction from
  the config subtree in the same hard cut.
- Convert to the top-level CLI/host error only after command exit behavior,
  finding classification, and rollback decisions are complete. No caller may
  classify a fleet-config failure by parsing its `Display` output.

Existing validation rules, TOML output, fleet mutations, rollback behavior,
and user-facing meaning remain unchanged.

## Hard Cut

- Replaced artifact finalizers, direct CLI file writers, boxed config errors,
  and string-built config decision paths are deleted in their slice.
- No aliases, compatibility writers, legacy error conversions, or fallback
  durability paths remain.
- A failed durable commit cannot advance the associated durable state.
- No existence-check-plus-rename sequence substitutes for atomic no-replace
  publication.
- A platform missing required no-replace or directory-sync semantics fails
  explicitly rather than weakening the contract.

## Deliberate Non-Abstractions

0.88 does not add:

- a cross-crate filesystem or transaction framework;
- a multi-file journal or recovery daemon;
- a new backup format, manifest version, or restore protocol;
- a global application/host error framework or public error-code schema;
- broad `unreachable_pub` cleanup;
- dependency forks or replacements for transitive IC advisories;
- deployment, stable-memory, Candid, or persisted-schema changes.

Host and backup durable functions remain separate. They own distinct file and
directory persistence boundaries.

## Acceptance

### Slice A

1. Both backup artifact-finalization paths call one backup-owned commit
   function.
2. The temporary directory is a unique sibling of the canonical directory.
3. The no-follow walk accepts only regular files and directories, syncs every
   regular file, syncs nested directories bottom-up, and syncs the temporary
   root itself.
4. Canonical publication is atomic and no-replace, and the final parent is
   synced afterwards.
5. Unrelated or unverifiable existing destinations and every unsupported
   filesystem entry fail closed.
6. An expected artifact published before the durable journal transition can
   resume only after exact journal binding, checksum reverification, and tree
   and parent resynchronization.
7. Failure before a durable journal transition leaves the persisted state
   below `ArtifactState::Durable` and the completed metric unchanged.
8. The completed metric cannot be reported before the durable state transition
   succeeds; a failed journal replacement exposes neither proposed update.

### Slice B

1. `write_bytes` and `create_new_bytes` delegate to one private commit
   implementation; there is no second staging/sync/publication flow.
2. Every staged file is a unique sibling of its final path.
3. No in-scope CLI output opens the final destination in truncating or
   incremental-write mode.
4. Serialization and every pre-publication failure leave an existing
   destination unchanged.
5. Replace mode atomically publishes complete new bytes and syncs the final
   parent.
6. Create-new publication is atomic and no-clobber; it does not implement
   no-overwrite as an existence check followed by replacing rename.
7. A post-publication durability failure may return an error with the complete
   new file visible, but no final path ever contains partial bytes.
8. Handled failure leaves no temporary-file residue.
9. Shared CLI output still creates requested parents, while deployment-plan
   output still rejects a missing parent.
10. Every newly created parent and the entry that introduced it are synced
    before success.
11. JSON, text, stdout, and command-exit contracts are unchanged.

### Slice C

1. The fleet-config subtree contains no boxed dynamic-error signatures.
2. Core-config sources retain operation context; I/O sources retain operation
   and path context.
3. Mutation conflicts and invalid inputs are classified without parsing text.
4. Rollback failure retains both the original mutation failure and rollback
   failure as typed causes.
5. Type erasure occurs only after exit, finding, and rollback decisions.
6. Successful projection values and serialized TOML output remain unchanged.
7. No compatibility conversion or global error abstraction is introduced.

## Sequence

1. Slice A: backup artifact-directory durability.
2. Slice B: CLI single-file output durability.
3. Slice C: typed fleet-config failures.
4. Close 0.88; do not extend this design by accretion.

## Validation Policy

Use bounded, test-local failure injection at the private commit-step boundary;
do not introduce a production mock-filesystem framework.

Slice A focused tests cover unsupported entries, regular-file sync failure,
nested-directory sync failure, temporary-root sync failure, atomic publication
failure, an existing unrelated destination, final-parent sync failure,
interruption after publication but before the journal transition, and durable
journal-write failure. Every failure asserts the persisted journal state and
completed metric. Post-publication cases additionally prove that any visible
canonical tree is complete and checksum-verifiable.

Slice B focused tests cover serialization failure, temporary creation, write,
and sync failures, atomic replacement failure, atomic no-replace conflict,
final-parent sync failure, and new-parent creation or sync failure. Replace
assertions allow only old-complete or new-complete final bytes; create-new
assertions allow only an absent path or complete new bytes. Neither mode may
expose an empty truncation, prefix, or partially serialized JSON document.

Slice C focused tests prove that I/O and core-config sources survive with
context; invalid name, invalid kind, fleet mismatch, missing declaration, and
mutation conflict have typed variants; rollback retains both typed causes; and
classification does not depend on `Display` output.

Also run targeted package Clippy, formatting, layering guards, and diff
hygiene. Slice B does not run scaffold tests because scaffold output is outside
the closed inventory.

The maintainer owns full workspace, PocketIC, deployment, and publish gates.
