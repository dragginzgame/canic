# Canic 0.88 Design: Artifact Durability and Typed Config Errors

Date: 2026-07-13

## Status

- Status: proposed and implementation-ready after the 0.87 closeout
  correction.
- Audit source:
  `docs/audits/reports/2026-07/2026-07-13/codebase-health.md`.
- Implementation tracker: [status.md](status.md).

## Problem

The post-0.87 audit found three bounded authority gaps:

1. backup journals claim artifact directories are durable immediately after
   unsynced renames;
2. CLI evidence and report files can be observed partially written; and
3. the fleet-config owner returns boxed and string-built failures instead of
   preserving typed causes.

They belong in one line because each is a boundary-truth problem: durable
state must actually be durable, written evidence must be complete, and
configuration failures must retain their machine-owned meaning.

## Decisions

### Slice A - Durable backup artifact commit

`canic-backup` owns the durability of downloaded snapshot directories.

- Add one private artifact-directory commit function in `canic-backup`.
- Sync downloaded regular files and nested directories before final rename.
- Rename the verified temporary directory into its canonical artifact path and
  sync the owning directory.
- Reject unsupported filesystem entries rather than following them.
- Use the same function from direct snapshot capture and backup-runner
  finalization.
- Advance `ArtifactState::Durable` and its completed metric only after the
  durable commit succeeds.

The existing journal schema, artifact directory shape, checksums, resume
policy, manifest schema, and error reporting remain unchanged.

### Slice B - Failure-atomic CLI file output

The existing host durable byte writer remains the owner of ordinary host/CLI
single-file replacement.

- Serialize complete JSON or text bytes before touching the destination.
- Route replacing CLI file output through the existing durable writer.
- Add one bounded durable create-new operation for deployment plans and other
  current no-overwrite paths.
- Preserve create-parent behavior and the existing refusal to overwrite a
  create-new target.
- Keep stdout rendering unchanged.
- Route scaffold new-file creation through the create-new operation if it can
  do so without changing scaffold rollback ownership.

The slice does not create a general output service or multi-file transaction.

### Slice C - Typed fleet-config failure boundary

`canic-host::release_set::config` owns configuration reads, projections, and
mutations and therefore owns their failure vocabulary.

- Add one focused `FleetConfigError` type in that subsystem.
- Preserve I/O and core configuration errors as sources.
- Represent invalid names/kinds, fleet mismatches, missing declarations,
  mutation conflicts, and rollback failure with bounded typed variants.
- Do not create a separate variant for every rendered sentence.
- Remove boxed dynamic-error returns and string-built error construction from
  the config subtree in the same hard cut.
- Outer CLI/host orchestration may erase the type only at a boundary that does
  not classify it further.

Existing validation rules, TOML output, fleet mutations, rollback behavior,
and user-facing meaning remain unchanged.

## Hard Cut

- Replaced artifact finalizers, direct CLI file writers, boxed config errors,
  and string-built config decision paths are deleted in their slice.
- No aliases, compatibility writers, legacy error conversions, or fallback
  durability paths remain.
- A failed durable commit cannot advance the associated durable state.

## Deliberate Non-Abstractions

0.88 does not add:

- a cross-crate filesystem or transaction framework;
- a multi-file journal or recovery daemon;
- a new backup format, manifest version, or restore protocol;
- a global application/host error framework or public error-code schema;
- broad `unreachable_pub` cleanup;
- dependency forks or replacements for transitive IC advisories;
- deployment, stable-memory, Candid, or persisted-schema changes.

Host and backup durable functions remain separate. They own distinct file and
directory persistence boundaries.

## Acceptance

### Slice A

1. Both backup artifact-finalization paths call one backup-owned commit
   function.
2. Regular files and directories are synced before the journal becomes
   durable.
3. The final artifact parent is synced after rename.
4. Existing destinations and unsupported entries fail closed.
5. Commit failure leaves the journal below `ArtifactState::Durable`.

### Slice B

1. CLI file output never truncates the final destination in place.
2. Serialization failure leaves an existing destination unchanged.
3. Replace mode publishes complete bytes atomically and durably.
4. Create-new mode preserves no-overwrite behavior and leaves no partial final
   file on failure.
5. JSON, text, stdout, and command-exit contracts are unchanged.

### Slice C

1. The fleet-config subtree contains no boxed dynamic-error signatures.
2. Core config and I/O sources survive through the typed owner boundary.
3. Mutation conflicts and invalid inputs are classified without parsing text.
4. Existing successful config projections and mutations are byte-for-byte
   unchanged.
5. No compatibility conversion or global error abstraction is introduced.

## Sequence

1. Slice A: backup artifact-directory durability.
2. Slice B: CLI single-file output durability.
3. Slice C: typed fleet-config failures.
4. Close 0.88; do not extend this design by accretion.

## Validation Policy

Run only focused package checks and tests for the touched owner:

- focused `canic-backup` artifact/journal tests for Slice A;
- focused `canic-host` durable-I/O and `canic-cli` output/deploy/scaffold tests
  for Slice B;
- focused host config mutation/projection and affected CLI command tests for
  Slice C;
- targeted package Clippy, formatting, layering guards, and diff hygiene.

The maintainer owns full workspace, PocketIC, deployment, and publish gates.
