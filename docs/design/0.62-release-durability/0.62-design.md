# 0.62 Design: Release Durability, Upgrade Confidence, And Operator Recovery

## Status

- Status: Tentative
- Scope: post-0.61 stabilization and release-readiness
- Primary mode: docs, validation, governance, targeted tests, and minimal
  diagnostics
- Runtime behavior changes: not expected unless a concrete release-blocking
  defect is found
- Candid changes: not expected
- CLI output changes: not expected
- JSON/output format changes: not expected
- Dependency or lockfile changes: not expected

## Summary

0.62 is the release-durability line following the 0.61 replay-protection work.

0.61 established replay, delegated-auth, operation-ID, receipt, cost-guard, and
permit-boundary behavior across the replay-sensitive request surface. 0.62 does
not reopen those semantics by default. Instead, it makes the 0.61 work easier
to ship, validate, diagnose, upgrade, and recover from in production-like
release workflows.

The central question for 0.62 is:

```text
Can Canic safely release, upgrade, diagnose, and recover across the
replay-protected flows introduced or hardened by 0.61?
```

0.62 should be treated as a stabilization and release-readiness line, not as
"Replay Protection Part 2."

## Background

The 0.61 line completed a large replay-protection effort. The close-out audit
for 0.61 concluded that implementation slicing should stop and that remaining
work belongs to RC/full release validation.

0.61 covered, among other areas:

- replay-policy inventory and release-blocker manifest gates;
- shared replay receipt core;
- stable replay receipt shape coverage;
- root RPC migration to shared replay handling;
- operation-ID hard-cut behavior for replay-sensitive requests;
- delegated-auth hard cut from verifier-local update-token consumption;
- delegation-proof replay protection and caller/shard binding;
- delegated-token mint/issue replay protection;
- pool lifecycle replay classifications and behavior;
- ICP refill replay integration;
- pending operation logging;
- value-transfer cost guard;
- signing, deployment, value-transfer, and upgrade permit boundaries;
- durable-publication manifest coverage;
- release-candidate aggregate manifest gate;
- final 0.61 documentation containment, making the old slice plan historical
  rather than active work.

The 0.61 close-out result did not mean the release is fully shipped. It meant
the 0.61 implementation slice line is complete. The next step is release
validation and operational confidence.

0.62 exists to capture that next step without scope-creeping back into
implementation hardening.

## Problem Statement

After a large security-sensitive release line, the primary risk shifts.

The main remaining risk is no longer "we need another replay slice." The main
risk is that release, upgrade, diagnosis, or recovery workflows are not yet
boring and repeatable enough.

0.62 addresses that risk by making the following explicit:

- which commands and gates must pass before RC promotion;
- which upgrade and stable-state assumptions are relied on;
- which retry and recovery actions are safe for operators;
- which failure modes can be diagnosed without guessing;
- which work belongs to release validation rather than another implementation
  slice;
- which governance checks prevent stale open-slice language from reappearing.

## Design Principle

0.62 should preserve the 0.61 semantic boundary.

The default answer to new behavior should be:

```text
No, unless a concrete RC/release validation failure proves it is necessary.
```

The default answer to documentation, validation, and runbook improvements should
be:

```text
Yes, if they improve release confidence without changing public behavior.
```

## Goals

### G1. Codify Release Validation

Create a clear, versioned release-validation matrix for the post-0.61 system.

The matrix should identify required local and CI validation for RC promotion,
including formatting, linting, workspace tests, focused replay/cost-guard gates,
changelog governance, Candid compatibility where applicable, and release/package
checks.

The goal is repeatability. A maintainer should be able to determine what "ready
for RC" means without reconstructing it from chat history or prior slice notes.

### G2. Audit Upgrade And Stable-State Confidence

Document and, where practical, test the assumptions required to upgrade across
the 0.61 replay-protected system.

The audit should cover replay receipts, operation IDs, pending operations,
delegated-auth state, permit boundaries, cost-guard accounting, and any
stable/publication state that participates in replay-sensitive flows.

The goal is to know whether upgrade/restart paths preserve the safety properties
established by 0.61.

### G3. Improve Operator Recovery Guidance

Create recovery and retry runbooks for the major replay-protected failure modes.

Operators should be able to distinguish safe retry, duplicate replay, expired
authorization, caller/shard mismatch, stuck pending work, cost-boundary refusal,
and publication/upgrade boundary failures.

The goal is not to make every failure auto-recoverable. The goal is to make
manual recovery safe and explainable.

### G4. Improve Diagnostic Confidence

Audit whether existing logs, errors, tests, and docs make key failure modes
distinguishable.

When a replay-protected operation fails, maintainers should be able to identify
the class of failure without relying on ambiguous behavior.

Any diagnostic changes should be minimal and must not accidentally alter stable
public output formats unless explicitly approved.

### G5. Prevent Scope Creep

Add or reinforce release governance that keeps 0.62 bounded.

Each 0.62 slice should declare whether it changes runtime code, CLI behavior,
Candid, JSON/output shape, dependency manifests, lockfiles, generated artifacts,
fixtures, or snapshots.

Docs-only slices must remain docs-only. Runtime-impacting slices require a
concrete blocker or approved charter item.

## Non-Goals

0.62 is not intended to:

- add new replay semantics;
- redesign delegated authentication;
- redesign operation IDs;
- expand the cost-abuse policy surface;
- add new pool lifecycle behavior;
- change Candid interfaces;
- change CLI output;
- change JSON output formats;
- perform broad refactors;
- upgrade dependencies;
- introduce new release packaging systems unless a release blocker requires it;
- continue the 0.61 slice chain under a new name.

Any exception must name the concrete defect or release blocker that justifies
the change.

## Compatibility Policy

The expected compatibility posture for 0.62 is conservative.

### Runtime Behavior

No runtime behavior changes are expected by default.

A runtime change is allowed only if:

- RC/full validation finds a concrete defect;
- the defect is release-blocking;
- the fix is narrowly scoped;
- the changelog and status docs explicitly describe the impact;
- focused tests cover the corrected behavior.

### Candid

No Candid changes are expected.

Any Candid change requires explicit approval and must include compatibility
notes.

### CLI And JSON/Output Formats

No CLI or JSON/output format changes are expected.

Diagnostics should prefer internal logs, docs, tests, or developer-only tooling
unless a stable output change is deliberately approved.

### Dependencies And Lockfiles

No dependency or lockfile changes are expected.

Any Cargo.lock or manifest churn caused incidentally by validation commands must
be reverted before a slice closes.

## Proposed Workstreams

### Workstream A: Release Validation Matrix

Define the release-validation matrix for RC promotion.

The matrix should distinguish:

- required local gates;
- required CI gates;
- focused replay/auth/cost gates;
- release/package/install checks;
- optional or environment-specific checks;
- checks that are recommended but not blockers.

Candidate required local commands:

```text
cargo fmt --all -- --check
cargo test --locked -p canic --test changelog_governance -- --nocapture
git diff --check
```

Candidate focused gates:

```text
cargo test --locked -p canic-core replay_policy --lib -- --nocapture
cargo test --locked -p canic-core --test cost_guard_boundary_guard -- --nocapture
cargo clippy --locked -p canic-core --all-targets --all-features -- -D warnings
```

Candidate broader gates:

```text
cargo clippy --locked --workspace --all-targets --all-features -- -D warnings
cargo test --locked --workspace
```

Candidate repo-governance gates:

```text
make fmt-check
make clippy
make test
```

Candidate release/package/install gates:

```text
make test-packaged-downstream-cli
make test-packaged-downstream-wasm-store
make package
```

The final matrix should be derived from actual repo capabilities, not invented
expectations. It should also classify expensive or environment-specific checks
as local-required, CI-required, RC-required, or recommended-only instead of
pretending every gate is cheap enough for every slice.

### Workstream B: Upgrade And State Compatibility Audit

Audit the upgrade and stable-state assumptions around replay-sensitive flows.

Areas to inspect:

- replay receipt persistence and shape stability;
- operation-ID durability;
- pending operation state;
- delegated-auth and delegation-proof state;
- caller/shard binding assumptions;
- token mint/issue replay state;
- ICP refill and value-transfer replay state;
- cost-guard accounting;
- signing, deployment, value-transfer, and upgrade permit boundaries;
- durable-publication state;
- pre-upgrade and post-upgrade hook ordering;
- old-stable-state-to-new-binary compatibility;
- stable state schema compatibility, distinct from Candid compatibility;
- downgrade behavior, explicitly treated as a non-goal unless separately
  approved;
- any stable memory or migration boundary that intersects the 0.61 behavior.

The audit should produce one of these outcomes for each area:

- covered by existing tests/docs;
- covered after 0.62 targeted docs/tests;
- known gap, but non-blocking for RC;
- release blocker.

### Workstream C: Operator Recovery Runbooks

Create runbooks for common operational failure modes.

Candidate runbooks:

- safe retry after network/client failure;
- duplicate operation or replay rejection;
- expired authorization or permit;
- delegated-auth caller mismatch;
- delegation shard mismatch;
- stuck pending operation;
- ICP refill pending/replayed/refused state;
- value-transfer cost-boundary refusal;
- durable-publication ambiguity;
- upgrade interrupted near replay-sensitive operation;
- receipt mismatch or unexpected receipt state.

Each runbook should include:

- symptom;
- likely cause;
- safety invariant;
- safe operator action;
- unsafe operator action;
- diagnostic, log topic, or public error to check;
- retry/idempotency rule;
- relevant validation command;
- escalation criteria.

The runbooks should avoid promising automatic recovery unless the implementation
already provides it.

### Workstream D: Diagnostic Consistency Audit

Audit whether current diagnostics distinguish the major replay-protected failure
modes.

Failure classes to distinguish:

- duplicate/replay;
- missing operation ID;
- invalid operation ID;
- expired permit;
- wrong caller;
- wrong shard;
- delegation-proof replay;
- token mint/issue replay;
- pending operation already exists;
- operation completed and receipt replayed;
- value-transfer cost refusal;
- upgrade/permit boundary refusal;
- durable-publication conflict.

Possible outcomes:

- existing diagnostic is sufficient;
- docs clarify interpretation;
- internal log improvement needed;
- test assertion needed;
- public output change proposed separately.

Every diagnostic change must include a public-output impact statement classifying
the change as internal log, developer-only tool output, CLI text, JSON schema,
Candid/public API, metric label, or no public-output impact.

Public output changes should not be included in 0.62 unless explicitly approved.
If a stable or user-visible diagnostic changes, the slice must include
changelog/status coverage and tests for the changed surface.

### Workstream E: Release Governance

Add governance checks or documented rules to prevent stale implementation-slice
language from recurring.

Governance rules:

- each slice must state whether it is docs-only, test-only, tooling-only, or
  runtime-impacting;
- each slice must state whether it changes Candid, CLI output, JSON/output
  formats, manifests, lockfiles, fixtures, snapshots, or generated artifacts;
- docs-only slices must have no runtime delta;
- runtime-impacting slices must name the concrete defect or charter item;
- a release line must have a close-out audit before RC promotion;
- historical slice plans must be clearly labeled historical after close-out;
- status docs must not imply open implementation slices after implementation
  close-out.

Candidate governance search:

```text
rg -n "next slice|future slice|open slice|remaining implementation|post-gate|follow-up implementation|Slice [A-Z]|TODO|TBD" docs CHANGELOG* -S
```

The search result is not automatically a failure. Historical entries are
allowed when clearly framed as historical.

When reviewing 0.61 carryover language, also check stale close-out phrases such
as `Remaining Cost-Abuse`, `Slice G`, `post-gate work`, and `remaining release
blocker`. These terms can be legitimate in historical records, but they must
not imply active 0.62 implementation work.

## Release Readiness Criteria

0.62 is ready to close implementation work when all of the following are true:

- the release-validation matrix is documented;
- required local validation passes;
- required focused replay/auth/cost gates pass;
- full CI/release matrix is either passing or explicitly assigned to RC
  promotion;
- upgrade/state compatibility audit has no unresolved release blockers;
- operator recovery docs exist for the major replay-protected flows;
- diagnostic audit has no unresolved release blockers;
- changelog and status docs accurately describe the 0.62 scope;
- no stale docs imply that 0.62 requires additional implementation slices;
- no accidental runtime, CLI, Candid, JSON/output, dependency, lockfile,
  fixture, snapshot, or generated-output changes are present;
- any test-only or tooling-only changes are clearly identified.

## Close-Out Audit Format

The 0.62 close-out audit should use this structure:

### A. Verdict

Use one of:

- READY TO CLOSE 0.62 IMPLEMENTATION WORK
- NOT READY - BLOCKERS FOUND
- INCONCLUSIVE - NEEDS SPECIFIC VALIDATION

### B. Scope Confirmation

- State whether the final slice is docs-only, test-only, tooling-only, or
  runtime-impacting.
- State whether runtime code, CLI, Candid, JSON/output, Cargo.toml, Cargo.lock,
  fixtures, snapshots, generated outputs, or package manifests changed.

### C. 0.62 Completion Summary

- Summarize completed workstreams based only on repo evidence.

### D. Blockers

- List release-blocking issues.
- If none, say "None found in this audit."

### E. Non-Blocking Release-Readiness Work

- List remaining RC/full-release validation tasks.

### F. Validation Results

- Show commands run and pass/fail results.
- Note skipped or recommended commands.

### G. Recommendation

- State whether to close 0.62 implementation work.
- State whether to move to RC/full validation.
- State whether any further slice is justified.

## Risk Register

### R1. Scope Creep Back Into Replay Semantics

Risk:

- 0.62 could accidentally become an extension of 0.61.

Mitigation:

- Require any semantic change to name a concrete release-blocking defect.
- Otherwise keep work to docs, validation, governance, diagnostics, and
  targeted tests.

### R2. Public Output Drift

Risk:

- Diagnostic improvements could accidentally change stable CLI or JSON output.

Mitigation:

- Treat public output changes as explicitly scoped compatibility events.
- Prefer docs and internal diagnostics unless approved.

### R3. Lockfile Or Dependency Churn

Risk:

- Validation commands or local tooling could create incidental Cargo.lock churn.

Mitigation:

- Use `--locked` where applicable.
- Treat unexplained lockfile changes as blockers.
- Restore incidental churn before close-out.

### R4. False Confidence From Narrow Local Validation

Risk:

- Focused local gates may pass while full CI or packaging fails.

Mitigation:

- Separate implementation close-out from release promotion.
- 0.62 can close implementation work only after local/focused gates pass, but
  RC promotion still requires full release validation.

### R5. Historical Docs Mistaken For Active Work

Risk:

- Old slice plans or changelog entries may look like open work.

Mitigation:

- Mark historical implementation records clearly.
- Use status docs to distinguish active release work from historical notes.

### R6. Upgrade Audit Gaps

Risk:

- Existing tests may prove steady-state replay behavior without proving
  old-state-to-new-binary upgrade behavior.

Mitigation:

- Make upgrade/state audit outcomes explicit per state area.
- Treat untested upgrade assumptions as documented gaps until a focused test or
  existing fixture proves them.

### R7. Recovery Runbooks Overpromise Safety

Risk:

- Runbooks could imply automatic recovery or safe retry where implementation
  only provides manual containment or recovery-required state.

Mitigation:

- Require each runbook to state the safety invariant, retry/idempotency rule,
  unsafe action, and escalation criteria.
- Avoid "retry until it works" guidance unless replay behavior is already
  proven for that operation.

### R8. Diagnostics Leak Sensitive Or Stable State

Risk:

- Better diagnostics could expose operation IDs, principals, payload-derived
  data, or stable metric labels in places where they become de facto public API.

Mitigation:

- Classify diagnostic surfaces before changing them.
- Avoid logging payload bytes or secrets.
- Require explicit approval and tests for stable/user-visible diagnostic
  changes.

## Open Questions

Answered during 0.62.0 or 0.62.1:

- What is the authoritative full CI/release matrix for RC promotion?
  Answered in `docs/operations/release-validation-matrix.md`.
- Which PocketIC, canister, packaging, installer, or downstream checks are
  required before RC?
  Answered in `docs/operations/release-validation-matrix.md`.
- Should the release-validation matrix be enforced by a governance test, a
  script, or documentation only?
  0.62.1 adds a CI script guard for the non-versioned matrix path and required
  command anchors. Rust governance-test enforcement can be considered in a
  later governance slice if a concrete gap appears.

Remaining questions for later 0.62 workstreams:

- Are there existing upgrade tests that should become mandatory gates?
- Are diagnostics considered public API, CLI output, JSON output, internal
  logs, or mixed?
- Which release artifacts require checksum or reproducibility validation?
- Where should operator recovery runbooks live in the docs tree?

## Initial Recommendation

Start 0.62 with a docs-only charter slice.

The first slice should establish this design, add the 0.62 changelog skeleton,
update status, and explicitly state that 0.62 is a bounded release-durability
line.

Do not start 0.62 with runtime changes.

Do not carry over unresolved-looking 0.61 slice language.

Do not begin replay-protection hardening unless RC validation finds a concrete
release-blocking defect.
