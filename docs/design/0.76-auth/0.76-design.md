# Canic 0.76 Design: Bridge-Free Root Delegation Renewal

## Status

- Status: implementation underway. Core DTO, canonical verifier, signing
  wrapper, persisted batch state, root timer prepare/sign/install
  orchestration, and issuer-to-root lazy repair are in the 0.76 worktree.
  PocketIC no-external-liveness acceptance now passes for timer-driven renewal,
  and issuer login-level lazy repair now passes for a single missing-proof
  issuer. Live multi-issuer timer batching now passes with one root signing
  operation for two due issuers. Live concurrent lazy repair now passes with
  simultaneous missing-proof login updates collapsing to one root signing
  operation. Focused fake-signer state-machine coverage now passes for retry,
  in-flight duplicate ticks, stale registry changes during signing,
  expired-batch pruning, partial issuer install retry, and stale callback
  discard. Focused verifier-negative coverage now passes for trust-anchor
  binding, signer identity/key metadata mismatch, invalid witnesses, altered
  certs/grants, duplicate issuer leaves, and legacy proof rejection. The
  official IC platform-spec pass and first-slice non-Rust verifier fixture
  decision are closed.
- Release line: 0.76
- Decision style: delegated-auth liveness hard cut and proof-primitive
  replacement.
- Primary implementation target: bridge-free renewal of issuer active root
  delegation proofs.
- Current state: 0.74 root-managed renewal can schedule and prepare proof
  batches, but `RootProof::IcCanisterSignatureV1` still requires an external
  direct root query to assemble IC canister-signature proof material. The
  2026-06-29 audit records this as an architecture break for the stricter
  product requirement that auth liveness must not depend on any bridge, worker,
  CLI, cron job, host daemon, external signer, or client-side provisioning step.
  0.76 is a hard-cut design: the bridge-backed canister-signature root proof is
  not retained as a compatibility mode. The current 0.76 worktree now includes
  the first production chain-key batch implementation slices. Lazy repair now
  has root-side cached/in-flight signing-volume guards and issuer workflow
  retry/no-repair control-flow coverage; multi-issuer batches have a unit guard
  proving one threshold signature covers multiple issuer proofs. The live
  PocketIC no-external-liveness gate now proves timer renewal through real
  management-canister signing, issuer install, and verifier acceptance without
  the bridge. A live PocketIC lazy-repair gate now proves a missing-proof issuer
  can reach valid local proof material through root repair and that a repeated
  login under that proof adds no root management signing work. A live PocketIC
  multi-issuer gate now proves one timer batch covers two issuer canisters with
  one root signing operation. A live PocketIC concurrent-repair gate now proves
  simultaneous missing-proof token prepares reuse one root signing operation and
  retries/fresh-proof reuse add no additional threshold signatures. No
  design-lock gates remain open; remaining work is implementation close-out and
  any broader maintainer-requested validation.

## 1. Summary

0.76 changes the delegated-auth direction: Canic will not attempt to make the
current `RootProof::IcCanisterSignatureV1` renewal flow bridge-free.

The replacement target is `RootProof::IcChainKeyBatchSignatureV1`: root signs a
canonical delegation batch once per proof epoch, issuer canisters receive and
store issuer-specific proof material from that signed batch, and normal logins
reuse the stored proof. The intended liveness path is:

```text
root timer/update
  -> select due issuer delegation templates
  -> build a canonical delegation batch
  -> call management canister sign_with_ecdsa
  -> verify the returned signature defensively
  -> persist the signed batch
  -> install issuer-specific proof and witness on issuers
  -> issuers verify and store active proof
  -> logins reuse stored proof without calling root signing
```

This preserves Canic's root-authorized issuer, audience, grant, and bounded
validity-window semantics under a different IC threshold-signature primitive.
It must not be described as the same security guarantees. The verifier contract
changes because fields previously bound by IC certificate and certified-data
verification must become explicit signed payload or verifier-config fields in
chain-key mode.

## 2. Decision

Decision:

```text
Canic will not make RootProof::IcCanisterSignatureV1 bridge-free.
```

Reason:

```text
The current proof primitive requires direct root query certificate material.
Repo evidence and IC platform behavior show that timer, update, self-call,
canister-to-canister, and issuer composite-query paths cannot provide the root
data_certificate material needed for the current proof shape.
```

Replacement:

```text
Replace RootProof::IcCanisterSignatureV1 with
RootProof::IcChainKeyBatchSignatureV1 for accepted 0.76 root proofs.
```

Compatibility stance:

```text
0.76 steady state is a hard cut. The accepted steady-state root proof mode is
chain_key_batch only. Delegated-token RootProof no longer exposes a
canister-signature root proof variant in the active DTO/stable shape.
Historical bridge state may remain decode-compatible only where explicitly
marked as archive/upgrade state, but it does not authorize 0.76 delegated-token
issuance, active-proof install, verifier acceptance, bridge repair, or
liveness.
```

Security framing:

```text
The migration preserves Canic's root-authorized issuer/audience/grant
delegation semantics, but it uses a different proof primitive and requires an
explicit verifier contract.
```

Signing invariant:

```text
Root signs delegation batches per proof epoch, never per login, session, token,
user, nonce, or request payload.
```

Acceptance gate:

```text
A PocketIC test must prove root proof renewal completes with no bridge, CLI,
worker, external signer, direct root query, host daemon, or client provisioning
step.
```

## 3. Background

The 0.74 renewal design intentionally kept the IC canister-signature root proof
primitive. It says root can prepare canister-signature leaves during an update,
but cannot assemble root proof material in that update because proof assembly
needs the root data certificate from a direct root query
(`docs/design/0.74-root-managed-delegation-proof-renewal/0.74-design.md:114`).
It also forbids issuer composite-query wrappers, update-time proof assembly
without a data certificate, verifier fetch-on-verify behavior, and token trust
chain changes as workarounds
(`docs/design/0.74-root-managed-delegation-proof-renewal/0.74-design.md:116`).

The 0.68 provisioning design records why issuer composite-query retrieval is
not equivalent to an external direct root query: the nested root query does not
provide the root `data_certificate()` context, and the proof retrieval endpoint
must not be called through issuer wrappers, root self-calls, or
canister-to-canister calls
(`docs/design/archive/0.68-root-proof-provisioning/0.68-design.md:41`,
`docs/design/archive/0.68-root-proof-provisioning/0.68-design.md:96`).

The active delegated-signatures contract documents the current
update-query-update flow and says renewal proof retrieval is a direct root query
(`docs/contracts/AUTH_DELEGATED_SIGNATURES.md:186`,
`docs/contracts/AUTH_DELEGATED_SIGNATURES.md:243`). The implementation matches
that contract:

- `RootProof` currently has only `IcCanisterSignatureV1`
  (`crates/canic-core/src/dto/auth.rs:28`).
- Root preparation adds canister-signature leaves and refreshes certified data
  (`crates/canic-core/src/ops/auth/root_canister_sig.rs:210`,
  `crates/canic-core/src/ops/auth/root_canister_sig.rs:239`).
- Root proof assembly returns `RootProof::IcCanisterSignatureV1` from
  query-time signature-map proof material
  (`crates/canic-core/src/ops/auth/root_canister_sig.rs:258`,
  `crates/canic-core/src/ops/auth/root_canister_sig.rs:294`).
- The renewal CLI bridges scheduled work by querying root for proof material
  and then submitting the batch back to root for install
  (`crates/canic-cli/src/auth/mod.rs:711`).

The 2026-06-29 audit is the evidence base for this design:

- `docs/design/audits/root-delegation-proof-renewal-audit.md`
- `docs/design/0.76-auth/audit-findings.md`

If those files diverge, the retained audit under `docs/design/audits/` is the
source to reconcile first.

## 4. Product Invariant

0.76 adopts this hard invariant:

```text
Root delegation proof renewal must complete using only canister timers, update
calls, inter-canister calls, and management-canister signing. No external
direct root query, off-chain bridge, CLI, cron job, external signer, host
daemon, or client-side provisioning step may be required for auth liveness.
```

Consequences:

1. `chain_key_batch` is the only accepted 0.76 root proof primitive.
2. `canister_signature` is treated as pre-0.76/current-state evidence, not a
   0.76 runtime mode.
3. 0.76 must fail tests if renewal only progresses when
   `canic auth renewal run-once`, a daemon, or an external direct root query is
   present.
4. 0.76 verifiers and issuers must reject `IcCanisterSignatureV1` root proofs
   for all delegated-token issuers in 0.76 steady state.
5. Existing installed active proofs must be replaced, not grandfathered.
6. Normal login/token issuance must remain issuer-local after an active root
   proof is installed.
7. Management-canister threshold signing is not considered an external signer
   for this invariant because it is a protocol service callable by canisters.

## 5. Goals

1. Remove external liveness dependencies from root proof renewal.
2. Introduce a root proof primitive that can be refreshed entirely by root
   timers, updates, inter-canister calls, and management-canister threshold
   signing.
3. Sign stable delegation authorization state, not login/session/token state.
4. Prefer one root threshold signature per batch per proof epoch.
5. Require Merkle batch proofs. A batch containing one issuer is valid, but
   per-issuer per-epoch signing is not an acceptable 0.76 fallback.
6. Make the verifier contract explicit for root key metadata, signature
   payloads, issuer inclusion, audience, grants, registry binding, and expiry.
7. Add verifier support before signer support so security semantics are not
   accidentally defined by the signing implementation.
8. Make the 0.76 implementation fail closed until a chain-key batch proof is
   installed; do not accept legacy root proofs as a fallback.
9. Prove through tests that repeated logins under a fresh proof do not call
   `sign_with_ecdsa`.

## 6. Non-Goals

0.76 design lock does not:

1. Implement production chain-key signing.
2. Reintroduce root threshold signing on the login hot path.
3. Make root sign user principals, login nonces, session ids, delegated token
   claims, per-login timestamps, request ids, or endpoint payloads.
4. Add online verifier calls to root.
5. Claim identical security guarantees between IC canister signatures and
   chain-key signatures.
6. Implement Schnorr or vetKD proof variants.
7. Use production behavior that accepts test keys in production config.

## 7. Proof Primitive

0.76 has one accepted root proof primitive:

```text
root_proof_primitive = chain_key_batch
```

### 7.1 Rejected Pre-0.76 Primitive

`canister_signature` is the current 0.74 design and the reason 0.76 exists:

```text
root timer/update
  -> prepare signature-map leaves
  -> set certified_data
external bridge/provisioner
  -> direct query root for data_certificate-backed proof material
  -> submit proof batch back to root
root update
  -> validate pending metadata
  -> install active proof on issuer
issuer
  -> store active proof
```

This proof shape has an external liveness dependency by design. 0.76 steady
state must not retain it as a runtime mode, fallback, verifier compatibility
path, or manual repair path for delegated auth. Pre-0.76 active proof material
must be replaced with a chain-key batch proof.

### 7.2 Accepted 0.76 Primitive

`chain_key_batch` is the 0.76 target:

```text
root timer/update
  -> build canonical batch
  -> call management canister threshold signing
  -> persist signed batch
  -> install issuer-specific leaf and witness on each issuer
issuer update
  -> verify chain-key root proof
  -> store active proof
verifier
  -> verify token and embedded chain-key root proof locally
```

This primitive must not depend on direct root query certificate material. The
root signature is produced through the IC management canister signing API
callable by canisters.

## 8. New Root Proof Type

Replace the accepted root proof shape with the chain-key batch variant:

```rust
pub enum RootProof {
    IcChainKeyBatchSignatureV1(IcChainKeyBatchSignatureProofV1),
}
```

Suggested DTO shape:

```rust
pub struct IcChainKeyBatchSignatureProofV1 {
    pub header: ChainKeyBatchHeaderV1,
    pub delegation_cert: ChainKeyDelegationCertV1,
    pub issuer_witness: ChainKeyBatchWitnessV1,
    pub signature: ChainKeyRootSignatureV1,
}

pub struct ChainKeyBatchHeaderV1 {
    pub schema_version: u16,
    pub root_canister_id: Principal,
    pub batch_id: [u8; 32],
    pub proof_epoch: u64,
    pub registry_epoch: u64,
    pub registry_hash: [u8; 32],
    pub tree_root: [u8; 32],
    pub not_before_ns: u64,
    pub expires_at_ns: u64,
    pub algorithm: ChainKeyAlgorithm,
    pub key_id: ChainKeyKeyId,
    pub derivation_path_hash: [u8; 32],
    pub key_version: u64,
}

pub struct ChainKeyDelegationCertV1 {
    pub root_canister_id: Principal,
    pub issuer_canister_id: Principal,
    pub proof_epoch: u64,
    pub issuer_proof_algorithm: IssuerProofAlgorithm,
    pub issuer_proof_binding_hash: [u8; 32],
    pub issuer_proof_binding: IssuerProofBinding,
    pub max_token_ttl_ns: u64,
    pub audience: DelegationAudience,
    pub grants: Vec<DelegatedRoleGrant>,
    pub not_before_ns: u64,
    pub expires_at_ns: u64,
    pub registry_epoch: u64,

}

pub struct ChainKeyRootSignatureV1 {
    pub algorithm: ChainKeyAlgorithm,
    pub key_id: ChainKeyKeyId,
    pub derivation_path: Vec<Vec<u8>>,
    /// Transport/debug cache only. Never the verifier trust anchor by itself.
    pub public_key: Vec<u8>,
    pub signature: Vec<u8>,
}

pub struct ChainKeyBatchWitnessV1 {
    pub steps: Vec<ChainKeyBatchWitnessStepV1>,
}

pub enum ChainKeyBatchWitnessStepV1 {
    LeftSibling([u8; 32]),
    RightSibling([u8; 32]),
}
```

The exact Rust names can change during implementation, but the signed contract
must retain equivalent fields.

`ChainKeyDelegationCertV1` is the issuer leaf payload for the first chain-key
batch proof schema. There is no separate `ChainKeyIssuerLeafV1` authority object
with duplicated issuer/audience/grant fields. If implementation introduces a
transport wrapper named `ChainKeyIssuerLeafV1`, it must be a thin wrapper around
`ChainKeyDelegationCertV1` or `delegation_cert_hash`; it must not maintain a
second copy of authorization fields that can drift from the certificate.

Domains are fixed by schema version and type. They should not be runtime
authority fields. If implementation carries a domain field in a DTO for
transport/debugging, the verifier must reject anything except the exact
expected domain bytes.

### 8.1 Root Chain-Key Trust Anchor

The root chain-key signing authority is identified by:

```text
root_canister_id
algorithm
key_id
derivation_path
key_version
public_key
```

The canister that calls `sign_with_ecdsa` must be the root canister whose id is
signed in `ChainKeyBatchHeaderV1.root_canister_id`. In normal operation root
calls the management-canister signing method directly.

The platform contract checked for 0.76 is the official IC management-canister
interface:

- [`ecdsa_public_key`](https://docs.internetcomputer.org/references/ic-interface-spec/management-canister/#ic-ecdsa_public_key)
  is canister-call only, returns a SEC1 compressed ECDSA public key plus chain
  code, defaults `canister_id` to the caller when omitted, and derives the key
  from `canister_id`, `derivation_path`, and `key_id`.
- [`sign_with_ecdsa`](https://docs.internetcomputer.org/references/ic-interface-spec/management-canister/#ic-sign_with_ecdsa)
  is canister-call only, signs the caller-derived key for the supplied
  `derivation_path` and `key_id`, requires a 32-byte `message_hash`, returns
  raw `r || s` for secp256k1, and requires cycles to be explicitly attached.
- If a signing request rejects with `SYS_UNKNOWN` or `CANISTER_ERROR`, the
  signature may still exist in the system. Root must therefore keep signing
  and install transitions idempotent and must not infer that no threshold
  signature was produced.

Root signer configuration must carry the actual derivation path bytes. The
verifier-facing `RootKeyPolicyV1` carries `derivation_path_hash`; the hash alone
is not sufficient for root to call `sign_with_ecdsa`.

A helper signer canister is forbidden unless a future design introduces a
separate `root_signing_canister_id` field and verifier policy explicitly
accepts that signer. Without that explicit field, helper signing changes the
authority key and must be rejected.

The proof may carry `public_key` for transport/debugging, but verifiers must
not trust this field by itself. A proof is accepted only when the embedded
`public_key` exactly matches either:

1. the verifier-configured root chain-key public key; or
2. the public key derived under the verifier's accepted `root_canister_id`,
   `algorithm`, `key_id`, and `derivation_path` policy.

A proof-supplied public key that is not already configured or derivable under
accepted root key policy must be rejected.

### 8.2 RootKeyPolicyV1 And Verifier Config Distribution/Rotation

The verifier trust anchor is local verifier policy, not proof-supplied material.
Every delegated-token verifier must have one or more accepted root key policies:

```rust
pub struct RootKeyPolicyV1 {
    pub root_canister_id: Principal,
    pub proof_mode: RootProofMode,
    pub algorithm: ChainKeyAlgorithm,
    pub key_id: ChainKeyKeyId,
    pub derivation_path_hash: [u8; 32],
    pub public_key: Vec<u8>,
    pub key_version: u64,
    pub min_accepted_key_version: u64,
    pub min_accepted_proof_epoch: u64,
    pub min_accepted_registry_epoch: u64,
    pub max_revocation_latency_ns: u64,
    pub valid_from_ns: u64,
    pub accept_until_ns: u64,
    pub build_network: BuildNetwork,
}
```

0.76 values are:

```text
proof_mode = chain_key_batch
algorithm = threshold_ecdsa_secp256k1
production key_id = key_1
```

`public_key` in this policy is the verifier's trusted root chain-key public key.
The `public_key` carried inside a proof is a transport/debug copy only and is
never a trust anchor. The verifier accepts a proof-supplied public key only when
it exactly matches an active `RootKeyPolicyV1.public_key` or a key derivable
under that policy's accepted root canister id, algorithm, key id, and
derivation path.

`valid_from` and `accept_until` are encoded as `valid_from_ns` and
`accept_until_ns` in protocol structs. Both are nanoseconds since Unix epoch.

`BuildNetwork` is enforced by both signer and verifier. `BuildNetwork::Ic`
must use production `key_1`; a signer must refuse `test_key_1`, and a verifier
must reject `test_key_1` even if the signature verifies. The local PocketIC
fleet also uses key id `key_1`, but with the local root-derived public key from
the management-canister `ecdsa_public_key` result. A future local
`test_key_1` setup is allowed only when `build_network != BuildNetwork::Ic`
and `allow_test_chain_key = true`.

Allowed verifier policy distribution channels:

1. deployment or init configuration installed with the verifier canister;
2. controller/governance update to the verifier canister;
3. root-to-verifier update where `caller == configured root_canister_id` for the
   verifier's current accepted root policy;
4. signed rotation bundle under a currently accepted non-expired root key
   policy.

These are push/provisioning channels. Endpoint token verification must not fetch
or discover root key policy online. If no active policy matches the proof mode,
root canister id, algorithm, key id, derivation path hash, public key,
`BuildNetwork`, and accepted key/epoch floors, verification fails closed.

Strict dual-accept planned rotation:

1. Publish the next `RootKeyPolicyV1` through an allowed channel while the old
   policy is still accepted.
2. The next policy must have a higher `key_version`, an explicit
   `valid_from_ns`, and a bounded `accept_until_ns`. The old policy must also
   have an `accept_until_ns` no later than the intended proof/token drain
   deadline.
3. Verifiers enter dual-accept only for the old and next policy pair. They must
   reject unrelated keys, lower key versions, and proofs below
   `min_accepted_key_version`, `min_accepted_proof_epoch`, or
   `min_accepted_registry_epoch`.
4. Root begins signing with the next key only after policy distribution is
   expected to have reached verifiers. Issuers then install chain-key active
   proofs signed by the next key.
5. After the old proofs and tokens drain, verifiers advance
   `min_accepted_key_version` and/or expire the old policy's `accept_until_ns`.
6. The old key is removed from accepted policy only after it can no longer
   verify any non-expired token for any issuer still in cutover.

Emergency compromise procedure must not rely on a compromised key. A signed
rotation bundle under the compromised key is not an acceptable emergency
recovery channel. Verifiers must receive emergency policy through deployment/init
configuration, controller/governance update, or another preconfigured
non-compromised administrative channel. The emergency update must raise at least
one of `min_accepted_key_version`, `min_accepted_proof_epoch`, or
`min_accepted_registry_epoch`, remove or expire the compromised policy, and
install the replacement root key policy before new tokens are accepted.

### 8.3 Issuer Proof Binding

Root is authorizing an issuer proof mechanism, not merely an issuer principal.
The chain-key certificate leaf must bind the issuer proof algorithm and proof
binding hash. The initial 0.76 shape should reuse the existing
`IssuerProofAlgorithm`, `IssuerProofBinding`, and
`issuer_proof_binding_hash` concepts from delegated auth, where current issuer
tokens use issuer canister signatures.

The verifier must reject a token if the issuer proof material does not match
the root-signed issuer proof algorithm and binding hash.

Root proof expiry also caps issuer token expiry. An issuer may not issue a
delegated token whose `expires_at_ns` exceeds the active root proof
`expires_at_ns` or the root-signed `max_token_ttl_ns`.

### 8.4 Active Proof Storage Key

`chain_key_batch` active proof storage is initially per
issuer/audience/grants set, not a single unqualified proof per issuer. The
safe storage key is:

```rust
pub struct ActiveRootProofKey {
    pub issuer_canister_id: Principal,
    pub audience_hash: [u8; 32],
    pub grants_hash: [u8; 32],
    pub registry_epoch: u64,
    pub key_version: u64,
}
```

The stored value carries the full proof, proof epoch, expiry, refresh time, and
install metadata. A future design may add a single issuer proof containing a
canonical set of all authorized audiences/grants, but 0.76 must not rely on an
ambiguous one-proof-per-issuer state shape.

`audience_hash` and `grants_hash` are SHA-256 over the same canonical audience
and grants encodings used in the signed certificate leaf. They must not be derived
from caller-supplied string formatting or non-canonical vector order.

## 9. Canonical Batch Payload

0.76 must use Merkle batch proofs. A batch of one issuer is valid, but the proof
mode remains `chain_key_batch` and still uses the same leaf, root, and witness
format. Per-issuer per-epoch signing is not an acceptable 0.76 fallback; if a
future design needs that shape, it must introduce a separate
`chain_key_single_issuer` proof mode with its own verifier contract.

Canonical encoding and Merkle construction are part of the signature contract.
They must be frozen with golden fixtures before any production signer code
lands.

The 0.76 encoder should extend the existing delegated-auth canonicalization
style in `crates/canic-core/src/ops/auth/delegated/canonical.rs`: SHA-256,
fixed domain bytes, explicit type tags, u32 big-endian vector lengths, u64
big-endian integers, length-prefixed raw principal bytes, exact UTF-8 strings
under the existing label/project validators, and strictly sorted unique grants
and scopes. The implementation may choose different function names, but it
must not introduce an ad hoc or Candid-dependent signing encoding.

Domain bytes:

```text
DOMAIN_BATCH    = "CANIC_ROOT_DELEGATION_CHAIN_KEY_BATCH_V1"
DOMAIN_BATCH_ID = "CANIC_ROOT_DELEGATION_CHAIN_KEY_BATCH_ID_V1"
DOMAIN_LEAF     = "CANIC_ROOT_DELEGATION_CHAIN_KEY_ISSUER_LEAF_V1"
DOMAIN_REGISTRY = "CANIC_DELEGATED_AUTH_REGISTRY_SNAPSHOT_V1"
```

Canonical field rules:

1. Hash function is SHA-256.
2. Length prefixes are u32 big-endian.
3. Numeric fields are big-endian at their declared integer width.
4. Principals are encoded as length-prefixed raw principal bytes.
5. Bytes are length-prefixed unless the field is fixed `[u8; 32]`.
6. `DelegationAudience` uses explicit variant tags and existing canonical
   audience encoding semantics.
7. `DelegatedRoleGrant` entries and scope vectors must be strictly sorted and
   unique by canonical bytes. Duplicate grants/scopes are rejected.
8. Strings are not normalized at signing time; they must already satisfy the
   existing canonical label/project validators.
9. `registry_hash` is SHA-256 over `DelegatedAuthRegistrySnapshotV1` using the
   dedicated registry snapshot encoder below.
10. `batch_id` is derived only after the Merkle `tree_root` is known. It must
    not be part of any leaf or certificate payload.

Registry snapshot:

```rust
pub struct DelegatedAuthRegistrySnapshotV1 {
    pub schema_version: u16,
    pub root_canister_id: Principal,
    pub registry_epoch: u64,
    pub proof_mode: RootProofMode,
    pub root_key_policy_hash: [u8; 32],
    pub issuer_policies: Vec<DelegatedAuthIssuerPolicySnapshotV1>,
}

pub struct DelegatedAuthIssuerPolicySnapshotV1 {
    pub issuer_canister_id: Principal,
    pub enabled: bool,
    pub preferred_proof_mode: RootProofMode,
    pub allowed_audiences: Vec<DelegationAudience>,
    pub allowed_grants: Vec<DelegatedRoleGrant>,
    pub max_root_proof_ttl_ns: u64,
    pub max_token_ttl_ns: u64,
    pub issuer_proof_algorithm: IssuerProofAlgorithm,
    pub issuer_proof_binding_hash: [u8; 32],
    pub renewal_template_hash: [u8; 32],
}
```

```text
registry_payload =
  canonical_encode(DelegatedAuthRegistrySnapshotV1)

registry_hash =
  SHA256(
    DOMAIN_REGISTRY ||
    len_u32_be(registry_payload) ||
    registry_payload
  )
```

Registry snapshot rules:

1. The registry snapshot has its own canonical encoder; it must not reuse Candid
   or debug serialization.
2. `registry_epoch` is monotonic and advances for every delegated-auth issuer
   policy/template change that can authorize, narrow, revoke, or rotate auth.
3. `issuer_policies` are sorted by raw issuer principal bytes. Duplicate issuer
   principals are rejected.
4. `allowed_audiences` are sorted by canonical audience bytes and duplicate-free.
5. `allowed_grants` and grant scopes are sorted by canonical bytes and
   duplicate-free.
6. Disabled or revoked issuer records that can still affect old proof rejection
   remain represented until all corresponding old proofs/tokens have expired or
   verifier min-epoch policy has invalidated them.
7. The snapshot excludes transient signing state, retry counters, metrics, and
   install-attempt bookkeeping.

Leaf payload:

```text
leaf_payload =
  canonical_encode(ChainKeyDelegationCertV1)

leaf_hash =
  SHA256(
    0x00 ||
    DOMAIN_LEAF ||
    len_u32_be(leaf_payload) ||
    leaf_payload
  )
```

`ChainKeyDelegationCertV1` does not contain `batch_id`. Inclusion under the signed
`tree_root` binds the certificate leaf to the batch. This avoids the circular
definition where `batch_id` depends on leaf hashes while leaf hashes depend on
payloads containing `batch_id`.

Merkle tree:

```text
tree_root =
  merkle_root(sorted leaf_hashes)

internal_node_hash =
  SHA256(0x01 || left_child_hash || right_child_hash)

leaf_sort_key =
  (
    issuer_canister_id_bytes,
    audience_hash,
    grants_hash,
    issuer_proof_binding_hash
  )
```

Merkle rules:

1. Empty batches must not be signed.
2. Leaves must be sorted by `leaf_sort_key`.
3. Duplicate `leaf_sort_key` values must be rejected.
4. Witness steps must encode sibling direction as `LeftSibling` or
   `RightSibling`.
5. Verifier recomputes the leaf hash, applies witness steps in order, and
   compares the result with `ChainKeyBatchHeaderV1.tree_root`.

Batch id:

```text
batch_id =
  SHA256(
    DOMAIN_BATCH_ID ||
    root_canister_id ||
    proof_epoch ||
    registry_epoch ||
    registry_hash ||
    tree_root ||
    not_before_ns ||
    expires_at_ns ||
    algorithm ||
    key_id ||
    derivation_path_hash ||
    key_version
  )
```

Each `batch_id` input uses the same canonical field encoding rules as the batch
header encoder. `batch_id` covers `tree_root`; it does not cover the header
payload that contains `batch_id`.

Batch header:

```text
header_payload =
  canonical_encode(ChainKeyBatchHeaderV1)

canonical_batch_header_bytes =
  DOMAIN_BATCH ||
  len_u32_be(header_payload) ||
  header_payload

batch_header_hash =
  SHA256(canonical_batch_header_bytes)
```

Signature message:

```text
signature_message = batch_header_hash
```

Algorithm-specific message handling is fixed for 0.76: IC threshold ECDSA
secp256k1 uses `signature_message` as the 32-byte `message_hash` passed to
`sign_with_ecdsa`. Schnorr is deferred to a future proof variant and must not
share this verifier contract. The official IC
[`sign_with_schnorr`](https://docs.internetcomputer.org/references/ic-interface-spec/management-canister/#ic-sign_with_schnorr)
method signs the provided message bytes rather than the ECDSA
`message_hash` field, so a Schnorr proof must define a separate canonical
message contract.

Verifier checks:

1. Recompute the certificate leaf hash from embedded `ChainKeyDelegationCertV1`.
2. Verify the Merkle witness from certificate leaf hash to `tree_root`.
3. Recompute `batch_header_hash`.
4. Verify the chain-key signature against the configured root public key or
   derived public key policy.
5. Check the issuer, audience, grants, registry epoch/hash, proof epoch,
   not-before time, expiry, key id, derivation path, algorithm, and key version.

Golden fixtures must cover canonical `ChainKeyDelegationCertV1` certificate leaves,
canonical batch headers, Merkle roots and witnesses, valid and invalid
signatures, wrong key ids, wrong derivation paths, wrong public keys, duplicate
grants, duplicate leaves, and altered leaf ordering.

## 10. ChainKeyDelegationCertV1 Leaf And Header Coherence

The 0.76 wire shape derives the issuer leaf hash from canonical
`ChainKeyDelegationCertV1` bytes. `ChainKeyDelegationCertV1` is the
root-authorized issuer leaf for the first chain-key batch proof schema; there is
no second authority object with duplicated issuer, audience, grant, or
time-window fields. The pre-0.76 `DelegationCert` belongs to the legacy
canister-signature proof path and is not this schema:

```text
chain_key_delegation_cert_payload =
  canonical_encode(ChainKeyDelegationCertV1)

issuer_leaf_hash =
  SHA256(
    0x00 ||
    DOMAIN_LEAF ||
    len_u32_be(chain_key_delegation_cert_payload) ||
    chain_key_delegation_cert_payload
  )
```

`ChainKeyDelegationCertV1` must carry the fields needed by the root-signed leaf,
but it must not carry `batch_id`:

```text
root_canister_id
issuer_canister_id
audience
grants
registry_epoch
registry_hash
proof_epoch
not_before_ns
expires_at_ns
max_token_ttl_ns
issuer_proof_algorithm
issuer_proof_binding_hash
issuer_proof_binding
```

Header-to-certificate coherence is mandatory:

1. `cert.root_canister_id == header.root_canister_id`.
2. `cert.proof_epoch == header.proof_epoch`.
3. `cert.registry_epoch == header.registry_epoch`.
4. `cert.registry_hash == header.registry_hash`.
5. `cert.not_before_ns == header.not_before_ns`.
6. `cert.expires_at_ns == header.expires_at_ns`.
7. `header.batch_id` recomputes from `DOMAIN_BATCH_ID`, header fields, and
   `header.tree_root`.

0.76 uses exact certificate/header validity equality. Per-issuer validity
windows inside one batch are deferred to a future proof variant or version. If a
future version supports per-issuer TTLs in one batch, it must define the weaker
window rule explicitly, such as `cert.not_before_ns >= header.not_before_ns` and
`cert.expires_at_ns <= header.expires_at_ns`.

Certificate self-consistency checks are also mandatory:

1. `grants_hash(cert.grants)` is computed from canonical, sorted,
   duplicate-free grants and scopes.
2. `audience_hash(cert.audience)` is computed from canonical audience bytes.
3. `cert.issuer_proof_binding` hashes to
   `cert.issuer_proof_binding_hash`.
4. `cert.max_token_ttl_ns <= cert.expires_at_ns - cert.not_before_ns`.

If implementation introduces a `ChainKeyIssuerLeafV1` transport wrapper, the
wrapper must contain only `ChainKeyDelegationCertV1` or `delegation_cert_hash`.
A wrapper that duplicates issuer, audience, grants, registry, proof epoch, time
window, max-token-TTL, or issuer-proof-binding fields is rejected by design. If a
future version intentionally duplicates those fields, the verifier must prove
exact equality with `ChainKeyDelegationCertV1` before checking token claims.

Token claims may be a subset of the certificate grants and audience according
to the normal delegated-token authorization rules. That subset rule applies only
between token claims and the already verified certificate. A token cannot widen
the certificate; a token asserting `{admin}` under a certificate that authorizes
only `{read}` is rejected before endpoint authorization.

Negative verifier tests must cover header/certificate mismatches for root
canister id, proof epoch, registry epoch, registry hash, not-before, expiry, and
batch id recomputation. They must also cover malformed certificate grants,
audience, `max_token_ttl_ns`, issuer proof algorithm, issuer proof binding hash,
and issuer proof binding payload.

## 11. Verifier Contract Delta

Current `IcCanisterSignatureV1` verifier obligations include:

1. Verify the IC certificate against the configured raw IC root key.
2. Verify the certified-data path and witness/hash tree.
3. Check the root canister id binding.
4. Check message/seed binding through the existing canister-signature public
   key and proof material.
5. Check audience, grants, subject, issuer, and time bounds through the
   delegated-token material.

Proposed `IcChainKeyBatchSignatureV1` verifier obligations:

1. Check the accepted algorithm, key id, derivation path, key version, and root
   public key policy.
2. Reject proof-supplied public keys that are not configured or derivable under
   accepted root key policy.
3. Recompute the canonical certificate leaf hash from
   `ChainKeyDelegationCertV1`.
4. Verify the issuer Merkle witness to the batch tree root.
5. Verify header-to-certificate coherence before token claim subset checks.
6. Recompute the canonical batch header hash.
7. Verify the root threshold signature over the batch header hash.
8. Check the root canister id is explicitly bound in the signed header and
   certificate leaf.
9. Check certificate issuer canister id, audience, grants, registry epoch,
   registry hash, proof epoch, not-before time, and expiry, and separately
   recompute `header.batch_id`.
10. Check issuer proof algorithm and issuer proof binding hash.
11. Reject delegated tokens whose token expiry exceeds the root proof expiry or
    root-signed `max_token_ttl_ns`.
12. Reject stale proof epochs and key downgrade attempts.
13. Reject legacy `IcCanisterSignatureV1` root proof material unconditionally
    in 0.76 steady state.

Fields the current IC certificate path helps bind, and chain-key mode must bind
explicitly:

1. Root canister identity.
2. Current IC certificate time/freshness is not preserved as the same
   primitive. In chain-key mode, freshness is represented by explicit signed
   validity fields: proof epoch, registry epoch/hash, not-before time, expiry,
   and verifier min-epoch/key-version policy.
3. Key and algorithm identity.
4. Domain separation between root delegation batches and any other root
   signature use.
5. Inclusion of the issuer's authorized delegation certificate leaf.
6. Issuer proof mechanism binding.

## 12. Revocation And Maximum Offline Latency

Delegated proofs and tokens are self-contained. Without a verifier policy
update, a verifier that already has a structurally valid token can verify it
offline until the token or active root proof expires. Therefore offline
revocation latency is bounded by root proof TTL, not by issuer/root online
state.

Define:

```text
max_revocation_latency_ns =
  maximum verifier-accepted offline lifetime for root-delegated authorization

root_proof_ttl_ns =
  ChainKeyBatchHeaderV1.expires_at_ns - ChainKeyBatchHeaderV1.not_before_ns
```

`max_revocation_latency_ns` is verifier and root signing policy. It is not a
proof-supplied trust anchor. Root must refuse to sign a proof whose
`root_proof_ttl_ns > max_revocation_latency_ns`, and verifiers must reject any
proof whose signed root proof TTL exceeds their configured
`max_revocation_latency_ns`.

Issuer token expiry is additionally capped by the active root proof:

```text
delegated_token.claims.expires_at_ns <= active_root_proof.expires_at_ns
delegated_token.claims.expires_at_ns <= issued_at_ns + max_token_ttl_ns
```

Verifiers must not add expiry grace. A token that outlives either the active root
proof expiry or the root-signed `max_token_ttl_ns` is rejected.

Emergency pre-expiry revocation requires an authenticated verifier policy
update. Accepted mechanisms include advancing
`RootKeyPolicyV1.min_accepted_registry_epoch`, advancing
`RootKeyPolicyV1.min_accepted_proof_epoch`, advancing
`RootKeyPolicyV1.min_accepted_key_version`, or replacing/removing the affected
root key policy through one of the allowed config channels. Deleting issuer
state at root or issuer canisters is not sufficient to revoke already issued
self-contained tokens at offline verifiers.

Revocation tests:

1. Root signer refuses to sign a proof whose TTL exceeds
   `max_revocation_latency_ns`.
2. Verifier rejects a proof whose TTL exceeds `max_revocation_latency_ns`, even
   if the signature is otherwise valid.
3. Delegated token expiry after active root proof expiry is rejected.
4. Delegated token expiry beyond root-signed `max_token_ttl_ns` is rejected.
5. Advancing `min_accepted_registry_epoch` rejects otherwise valid older proofs.
6. Advancing `min_accepted_proof_epoch` rejects otherwise valid older proofs.
7. Advancing `min_accepted_key_version` rejects otherwise valid older proofs.
8. Root/issuer-side deletion without verifier policy update does not claim
   emergency pre-expiry revocation in the verifier contract.

## 13. Signing Volume

The target invariant is:

```text
root threshold signatures ~= delegation proof epochs
```

The design must reject these accidental invariants:

```text
root threshold signatures ~= logins
root threshold signatures ~= users
root threshold signatures ~= tokens
root threshold signatures ~= sessions
root threshold signatures ~= endpoint calls
```

Root should sign:

1. Root canister id.
2. Issuer canister id or certificate leaf.
3. Audience.
4. Grants.
5. Registry epoch.
6. Registry hash.
7. Proof epoch or batch id.
8. Not-before time.
9. Expiry time.
10. Algorithm.
11. Key id.
12. Derivation path hash.
13. Key version.

Root must not sign:

1. User principal.
2. Login nonce.
3. Session id.
4. Delegated token claims.
5. Per-login `issued_at`.
6. Per-login `expires_at`.
7. Request id.
8. Endpoint payload.

Expected frequency:

| Mode | Threshold signing volume | Status |
| --- | --- | --- |
| Per login | One signature per login or token issuance. | Design failure by default. |
| Per issuer per epoch | One signature per issuer per proof epoch. | Rejected for 0.76; use a separate future `chain_key_single_issuer` mode if needed. |
| Per batch per epoch | One signature for all issuers included in a batch epoch. | Required 0.76 target. |

### 13.1 Signing Budgets And Metrics

Root must enforce a maximum number of threshold signing requests per proof
epoch and per wall-clock window. Quotas are a safety rail against lazy-repair
bugs, retry storms, or accidental per-login signing.

Required config:

```text
max_signatures_per_epoch
max_signatures_per_day
min_batch_size_for_proactive_signing
allow_single_issuer_batch_repair
```

Required metrics:

```text
chain_key_sign_requests_started
chain_key_sign_requests_succeeded
chain_key_sign_requests_failed
chain_key_sign_requests_deduplicated
chain_key_signed_batches
issuer_install_retries
```

If a signing quota is exceeded, root must return a retryable or fail-closed
state. It must not fall back to per-login signing, legacy bridge proof
acceptance, or token issuance without a valid active root proof.

### 13.2 Cost Model

Implementation must not pin production signing fees in auth policy. The
management-canister interface requires cycles to be explicitly attached to
`sign_with_ecdsa`; production code must derive the amount from the platform/CDK
fee helper at call time. The planning model below is retained only to size the
signing-volume invariant, not to define release defaults or stable USD pricing.

| Operation | Planning cost |
| --- | --- |
| Query call | Free |
| Update message execution base, 13-node subnet | `5_000_000` cycles |
| 1B Wasm instructions, 13-node subnet | `1_000_000_000` cycles |
| Inter-canister call overhead, 13-node subnet | `260_000` cycles plus bytes |
| Ingress reception, 13-node subnet | `1_200_000` cycles plus bytes |
| Production chain-key `sign_with_ecdsa`, `key_1` | planning example: `26_153_846_153` cycles, about `$0.0357` |
| Local/PocketIC chain-key signing | harness-dependent; planning example: `10_000_000_000` cycles, about `$0.0137` |

Let:

```text
L = logins per day
I = issuers
E = proof epochs per day
P = number of independent batches/partitions per epoch
```

Then:

```text
per-login signatures/day            = L
per-issuer per-epoch signatures/day = I * E
per-batch per-epoch signatures/day  = P * E

chain_key_cost/day =
  signatures/day * current_sign_with_ecdsa_cycles

chain_key_usd/day =
  signatures/day * current_sign_with_ecdsa_usd
```

Example assumptions:

```text
L = 100,000 logins/day
I = 100 issuers
E = 24 epochs/day
P = 1 batch/epoch
```

Chain-key ECDSA signing fee only:

| Model | Signatures/day | Cycles/day | Approx USD/day | Assessment |
| --- | ---: | ---: | ---: | --- |
| Per login | 100,000 | 2,615.38T | $3,570.00 | Hard no |
| Per issuer per hour | 2,400 | 62.77T | $85.68 | Possibly acceptable only if issuer count is small |
| Per issuer per day | 100 | 2.62T | $3.57 | Reasonable for small issuer count |
| Per batch per hour | 24 | 0.628T | $0.86 | Good |
| Per batch per 15 min | 96 | 2.51T | $3.43 | Still reasonable |
| Per batch per day | 1 | 0.026T | $0.036 | Very cheap |

The table uses the planning example above. Release acceptance is based on
signatures per epoch, not on a fixed cycle fee.

## 14. Root Renewal State Machine

Use an idempotent and retry-safe state machine:

```text
Idle
  -> Prepared(batch_id, hash)
  -> Signing(batch_id)
  -> Signed(batch_id, signature)
  -> Installing(batch_id, remaining_issuers)
  -> Installed(batch_id)

Any nonterminal state
  -> FailedRetryable(batch_id, reason)
```

Requirements:

1. Duplicate timer ticks must reuse the existing in-flight state.
2. Duplicate signing callbacks must be idempotent by `batch_id` and payload
   hash.
3. Unknown signing results must not trigger a new payload signature. Root
   should retry the same persisted batch hash while the batch remains valid, or
   abandon the batch as stale. It must not advance to a new batch merely because
   the previous signing result was unknown. This matters because the management
   canister may have produced a signature even when root receives an unknown or
   canister-error result.
4. Partial issuer install failure must keep the signed batch and retry only
   remaining issuers until the batch expires or is superseded.
5. Registry changes during signing that revoke, narrow, or rotate delegated-auth
   authority must invalidate the prepared/signed batch and force old
   epoch/hash material to expire through shorter proof validity and/or verifier
   min-epoch policy. Root must not install newly signed proof material that
   carries an old registry epoch after the policy change that made it stale.
6. Stale batches must be rejected before install.
7. State transitions must survive upgrade from persisted stable records.
8. Install success must update the same issuer active-proof state used by the
   current delegated-token login path.

## 15. Lazy Repair Contract

Lazy repair is allowed only if it preserves the signing-volume invariant.

Flow:

```text
issuer login/update
  -> issuer sees missing or stale active root proof
  -> issuer calls root canic_get_or_create_chain_key_delegation_proof update
  -> root returns cached signed batch proof if one is valid
  -> root signs only if no valid batch exists and no signing is in flight
  -> issuer verifies and stores proof
  -> login continues
```

Rules:

1. The root API must be an update, not a query that depends on direct root
   certificate material.
2. Root must derive `issuer_canister_id` from `msg_caller()` rather than from
   caller-supplied issuer identity.
3. Caller must be registered and enabled under root issuer policy.
4. The issuer renewal template's audience/grants must be allowed by root
   policy.
5. The request must be rejected unless the root proof primitive is
   `chain_key_batch`.
6. Root must return cached signed proof material before attempting a new
   management-canister signing call.
7. Concurrent repairs for the same epoch must singleflight into one signing
   attempt.
8. If signing is already in flight, root must return a retryable unavailable
   state or await the same in-flight operation rather than start another
   signing attempt for the same batch scope.
9. Lazy repair must not sign login-specific data.
10. Repeated logins under a fresh active proof must make zero root threshold
   signing calls.
11. If root returns retryable unavailable state, issuer login/update must return
    a retryable auth state rather than issue a token without a valid active root
    proof.

Forbidden lazy-repair behavior:

1. Caller asks root to produce a proof for an arbitrary issuer.
2. Caller supplies arbitrary grants outside root policy.
3. Caller forces fresh signing when a valid cached batch exists.
4. Caller forces a second signing request while one is already in flight.

## 16. Key Lifecycle And Disaster Recovery

Planned rotation:

1. Generate or select the next IC threshold ECDSA key policy with a strictly
   higher `key_version`.
2. Distribute the next `RootKeyPolicyV1` through an allowed verifier config
   channel before root starts signing with it.
3. Enter the strict dual-accept sequence in Section 8.2.
4. Install new chain-key active proofs on issuers.
5. Drain old tokens and proofs.
6. Advance verifier minimum key/proof/registry epochs or expire the old policy.
7. Remove old-key acceptance only after no non-expired token can depend on it.

Emergency compromise response:

1. Stop root signing with the compromised key.
2. Do not accept signed rotation bundles under the compromised key.
3. Push emergency verifier policy through deployment/init config,
   controller/governance update, or another preconfigured non-compromised
   administrative channel.
4. Raise `min_accepted_key_version`, `min_accepted_proof_epoch`, and/or
   `min_accepted_registry_epoch` so already issued compromised proofs fail.
5. Install a replacement `RootKeyPolicyV1`, then reissue issuer active proofs.
6. Verifiers fail closed until replacement policy and fresh active proofs are
   available.

Signing subnet or management signing unavailable behavior:

1. Root retries according to the persisted renewal state machine and signing
   quotas.
2. Issuers continue using existing active chain-key proofs only until those
   proofs expire.
3. Lazy repair returns `Pending`/`RetryAfter` or a retryable auth state while no
   valid proof exists.
4. After active proof expiry, token issuance fails closed.
5. Root must not fall back to bridge mode, canister-signature root proofs,
   per-login signing, or unsigned token issuance.

Root canister id change hazard:

1. `root_canister_id` is part of `RootKeyPolicyV1`, the signed batch header, and
   the header-to-certificate coherence contract.
2. A root canister id change is a new trust anchor, not an ordinary key
   rotation.
3. Verifiers must receive a new root key policy through an allowed config
   channel and must reject proofs signed for the old root once the migration
   advances min-epoch/key policy.
4. A signed bundle from the old root cannot by itself authorize a new root id
   unless verifier policy already allows that migration path.

Upgrade/recovery from persisted in-flight signing state:

1. `Prepared`, `Signing`, `Signed`, `Installing`, and retry states must persist
   enough batch hash, registry epoch/hash, key version, and remaining issuer
   data to resume idempotently after upgrade.
2. Recovery must re-check root key policy, registry epoch, proof TTL,
   `max_revocation_latency_ns`, and signing quotas before installing or retrying
   any persisted state.
3. Unknown management-signing results must retry the same batch hash while valid
   or abandon it as stale; they must not produce a fresh payload signature
   solely because the previous result was unknown.

Fail-closed rules:

1. Expired active root proofs cannot issue new delegated tokens.
2. Missing verifier root key policy rejects delegated tokens.
3. A stale key version, proof epoch, or registry epoch rejects delegated tokens.
4. `chain_key_batch` never automatically falls back to bridge-backed
   `IcCanisterSignatureV1`.

3. Keep historical audit/design docs as archive evidence only; they must not be
   presented as supported 0.76 behavior.
4. Ensure any remaining bridge/provisioning code is unreachable from 0.76 auth
   configuration and cannot install accepted active root proofs.

## 18. Hard-Cut Activation

0.76 has no delegated-auth legacy compatibility mode. The hard cut applies to
the delegated-auth root proof primitive, while historical bridge records may
remain decode-compatible only as archive/upgrade state:

1. `auth.delegated_tokens.root_proof_mode` must be `chain_key_batch`.
2. Delegated-token `RootProof` must expose only the chain-key batch variant in
   active DTO/stable records.
3. Delegated-token verifiers must not dispatch to the legacy root
   canister-signature verifier.
4. Issuer active-proof install must reject legacy root proof material.
5. The bridge/provisioner may remain in source for historical/manual tooling, but
   it must be unreachable from 0.76 delegated-auth liveness and unable to install
   accepted active root proofs.
6. Existing legacy active proofs are not grandfathered. After the 0.76 hard cut,
   missing chain-key active proof material means delegated-token issuance fails
   closed until root installs a valid chain-key batch proof.
7. There is no `LegacyBridge`, `DualCode`, `ChainKeyPreferred`, or `ChainKeyOnly`
   runtime state or passive cutover DTO in the accepted 0.76 delegated-auth
   flow.

Activation order:

1. Ship passive DTO/stable/canonical chain-key batch types and fail-closed
   verifier support.
2. Configure verifiers with `RootKeyPolicyV1` through an allowed config channel.
3. Enable the root chain-key batch signer/state machine.
4. Install chain-key active proofs on issuers.
5. Enable token issuance only when the issuer has a valid chain-key active proof.

Hard-cut tests:

1. Config validation rejects `root_proof_mode = "canister_signature"`.
2. Delegated-token DTO/stable surfaces have no
   `RootProof::IcCanisterSignatureV1` variant.
3. Issuer active-proof install accepts only chain-key batch root proofs.
4. Bridge renewal cannot install an accepted active proof in 0.76 mode.
5. A missing or expired chain-key active proof fails token issuance closed.
6. Active stable/DTO serialization does not expose legacy delegated root proof
   variants or bridge/provisioner renewal fields.

## 20. Tests And Acceptance Criteria

Positive no-external-liveness test:

```text
deploy root + issuer + relying verifier in PocketIC
do not run bridge, CLI, worker, host daemon, external signer, or direct root query
advance time past renewal threshold
root timer signs or reuses a batch
issuer stores active proof
relying verifier accepts issuer auth
```

Lazy repair test:

```text
issuer starts with missing or stale proof
login/update triggers issuer -> root update repair
root returns cached proof or creates one
issuer stores proof
login succeeds
repeated logins do not call sign_with_ecdsa
```

## 22. Pre-Implementation Gates And Closed Decisions

Closed decisions for 0.76:

1. Algorithm is IC threshold ECDSA secp256k1.
2. Schnorr is deferred to a future proof variant.
3. vetKD is rejected for root delegation proofs because it is
   key-derivation/transport material, not a public signature proof primitive.
4. Production key id is `key_1`.
5. The local PocketIC fleet uses `key_1` with the local root-derived public
   key. `test_key_1` may be used only outside `BuildNetwork::Ic` and only when
   `allow_test_chain_key = true`.
6. Signer and verifier both enforce `BuildNetwork`.
7. Production signer refuses `test_key_1`.
8. Production verifier rejects `test_key_1` even if a signature verifies.
9. Signature message is `SHA256(canonical_batch_header_bytes)`.
10. `sign_with_ecdsa.message_hash` is exactly 32 bytes.
11. Signature encoding is exactly 64 bytes `r || s`; each component is a
    32-byte big-endian integer.
12. Stored and verifier-accepted signatures must be low-s. Root signing wrappers
    normalize high-s signatures before persisting. Verifiers reject high-s.
13. `registry_hash` is
    `SHA256(DOMAIN_REGISTRY || len || canonical_encode(DelegatedAuthRegistrySnapshotV1))`.
14. Merkle batching is mandatory. A batch of one issuer is valid.
15. Per-issuer per-epoch signing is not a 0.76 fallback. A future single-issuer
    proof needs a separate `chain_key_single_issuer` mode.
16. The platform-spec pass is closed against the official IC management-canister
    contract for `ecdsa_public_key`, `sign_with_ecdsa`, and `sign_with_schnorr`:
    ECDSA signs a 32-byte message hash, secp256k1 signatures are raw `r || s`,
    public-key derivation is bound to caller/canister id plus derivation path
    plus key id, signing requires explicit cycles, and unknown/canister-error
    rejects cannot be treated as proof that no signature exists.
18. Non-Rust verifier fixtures are not required in the first 0.76 slice because
    this repo currently has no Motoko or TypeScript verifier consumer. Before
    adding a non-Rust verifier, add checked golden canonical leaf/header/root,
    signature, witness, and negative fixtures that the non-Rust implementation
    must pass.

ECDSA is selected over Ed25519/Schnorr for 0.76 because the real threshold ECDSA
signing path can be exercised in local replica/PocketIC CI with the local
PocketIC `key_1` trust anchor. The flagship no-bridge acceptance test must
prove the renewal loop through real signing, not only through a fake signer.
Ed25519/Schnorr support is
therefore deferred to a future proof variant with its own platform coverage and
signature-message contract. That future contract must not reuse the ECDSA
prehash rule: `sign_with_ecdsa` receives a 32-byte `message_hash`, while
`sign_with_schnorr` signs the provided message bytes directly.

Closed acceptance gates since design lock:

1. A live PocketIC public-key probe proves the local fleet trust anchor matches
   the management-canister `ecdsa_public_key` result for root, `key_1`, and
   `["canic", "delegation"]`.
2. A live PocketIC hard-cut test proves retained bridge-backed
   canister-signature provisioning endpoints reject in `chain_key_batch` mode.
3. A live PocketIC no-external-liveness test proves
   timer/update/inter-canister/management-canister signing completes root
   renewal, installs issuer active proof material, and reaches verifier
   acceptance without a bridge, CLI, worker, direct root query, host daemon,
   external signer, or client provisioning step.
4. A live PocketIC lazy-repair test proves a missing-proof issuer token prepare
   reaches root chain-key repair, handles retryable in-flight proof availability,
   installs valid issuer-local active proof material, verifies the delegated
   token, and performs a repeated login with zero additional root management
   signing work.
5. A live PocketIC multi-issuer signing-volume test proves timer renewal can
   batch two due issuer canisters under one root chain-key signing operation.
6. A live PocketIC concurrent lazy-repair signing-volume test proves
   simultaneous missing-proof delegated-token prepare updates either succeed or
   return the bounded retry signal while collapsing to one root chain-key
   signing operation; retrying and then preparing under the fresh stored proof
   adds zero additional threshold signatures.
7. Focused fake-signer state-machine tests prove signing failure retry-after
   recovery, duplicate signing ticks while a batch is already `Signing`,
   stale-registry invalidation before install, expired pre-install pruning,
   partial issuer install retry for only the remaining issuer, and discarding a
   returned management signature if the batch became stale while signing was in
   flight.
8. Focused chain-key verifier tests prove proof-supplied public keys are bound
   to verifier policy, wrong key id/derivation path/root canister/public key
   reject, helper-key signature failure rejects, invalid witnesses reject,
   altered cert grants reject, leaf/header validity mismatch rejects, duplicate
   issuer leaves reject at batch construction, and legacy canister-signature
   root proofs reject.

0.76 bridge-free close-out status:

No design-lock gates remain open. Remaining work is implementation close-out
and any broader validation the maintainer wants before cutting the release.

## 23. Final Recommendation

Replace the current root proof with a chain-key batch proof.

Make `RootProof::IcChainKeyBatchSignatureV1` the only accepted 0.76 steady-state
root proof. Do not retain `RootProof::IcCanisterSignatureV1` as a fallback
verifier path, bridge/manual repair path, or grandfathered active proof. At
design lock, the migration required trust-anchor rules, signer-canister
identity binding, canonical encoders, Merkle fixtures, header-to-certificate
coherence checks, revocation-latency policy, verifier config rotation, key
lifecycle/DR behavior, signing quotas, and negative tests before production
signing logic. The close-out status above records the first 0.76 chain-key
variant's acceptance gates and no-external-liveness/signing-volume coverage as
closed.
