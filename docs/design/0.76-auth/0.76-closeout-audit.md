# 0.76 Closeout Audit: Bridge-Free Root Delegation Renewal

Audit date: 2026-06-30

Baseline note: this closeout audit predates the follow-up 0.76.6 hard-cut
cleanup that removed delegated `RootProof::IcCanisterSignatureV1`,
`RootProofMode::IcCanisterSignature`, and
`RootProofRecord::IcCanisterSignatureV1` entirely. Treat old line-number
evidence for legacy rejection branches as historical.

Scope: repository closeout audit against the 0.76 bridge-free root delegation
renewal design contract, updated after the chain-key batch state-bounds closeout
slice.

## 1. Executive Summary

Is 0.76 ready to close?

**Ready to close.**

Top findings:

1. No P0 bridge-free liveness blocker was found. The accepted delegated-token
   root proof is `RootProof::IcChainKeyBatchSignatureV1`
   (`crates/canic-core/src/dto/auth.rs:36`, `crates/canic-core/src/dto/auth.rs:38`),
   and verifier code rejects legacy canister-signature root proofs before
   signature verification (`crates/canic-core/src/ops/auth/delegated/chain_key.rs:171`,
   `crates/canic-core/src/ops/auth/delegated/chain_key.rs:172`).
2. Root renewal is implemented as root timer/update work that prepares a
   canonical Merkle batch, calls management-canister ECDSA signing, persists the
   signed batch, and installs issuer proofs (`crates/canic-core/src/workflow/runtime/auth/renewal.rs:108`,
   `crates/canic-core/src/ops/auth/delegation/chain_key_batch.rs:195`,
   `crates/canic-core/src/workflow/runtime/auth/provisioning/mod.rs:144`).
3. Old bridge API surfaces are guarded as absent from the active protocol
   (`crates/canic/tests/protocol_surface.rs:779`, `crates/canic/tests/protocol_surface.rs:786`,
   `crates/canic-tests/tests/root_cases/auth_076.rs:86`,
   `crates/canic-tests/tests/root_cases/auth_076.rs:94`).
4. Per-login/session/payload root signing was not found. The signer signs only
   `chain_key_batch_header_hash(input.header)` as `sign_with_ecdsa.message_hash`
   (`crates/canic-core/src/ops/auth/delegated/chain_key_signing.rs:219`,
   `crates/canic-core/src/ops/auth/delegated/chain_key_signing.rs:221`), and
   tests cover cached login reuse and concurrent lazy-repair collapse
   (`crates/canic-tests/tests/root_cases/auth_076.rs:99`,
   `crates/canic-tests/tests/root_cases/auth_076.rs:156`).
5. Signing/state quotas are now enforced for the documented bounds. Each new
   root signing batch is capped at 64 issuers
   (`crates/canic-core/src/ops/auth/delegation/chain_key_batch.rs:52`,
   `crates/canic-core/src/ops/auth/delegation/chain_key_batch.rs:743`), and
   new batch creation fails with resource exhaustion when 128 non-expired,
   non-installed chain-key batches are pending
   (`crates/canic-core/src/ops/auth/delegation/chain_key_batch.rs:53`,
   `crates/canic-core/src/ops/auth/delegation/chain_key_batch.rs:775`,
   `crates/canic-core/src/ops/auth/delegation/chain_key_batch.rs:783`).
6. Docs are aligned with the hard-cut implementation and source layout. The
   operations runbook points to the timer workflow and signing wrapper
   (`docs/operations/root-proof-provisioning.md:16`,
   `docs/operations/root-proof-provisioning.md:17`) and documents the exact
   64-issuer and 128-pending-batch behavior
   (`docs/operations/root-proof-provisioning.md:93`,
   `docs/operations/root-proof-provisioning.md:95`).

Bridge-free invariant: **satisfied by the active delegated-token protocol, based
on source and targeted tests.**

Per-login root signing impossible by code/test: **login-specific signing is
blocked.** Missing-proof lazy repair may trigger one batch-epoch signature, but
the signed payload is the canonical batch header, not login/session/token data.

0.76 steady state rejects legacy canister-signature root proofs: **yes for
delegated-token auth.** Legacy DTO/stable decode remains for historical records,
but delegated-token verification is chain-key-only.

## 3. Current Implementation Map

Current active delegated-token flow is chain-key batch proof renewal. The old
bridge/canister-signature flow remains only in historical records, archive docs,
and role-attestation/non-delegated surfaces.

Historical canister-signature bridge flow, now removed from active 0.76 delegated
auth:

```text
external bridge/CLI/provisioner
  -> query root canic_delegation_renewal_work
  -> direct root query canic_get_delegation_renewal_proof_batch
  -> root reads data_certificate/certified_data
  -> update root canic_install_delegation_proof_batch
  -> issuers receive IcCanisterSignatureV1 proof
```

Bridge/direct root query: yes in historical design. Active 0.76 protocol: no.
Historical stable state is retained for decode only
(`crates/canic-core/src/storage/stable/auth/records.rs:405`,
`crates/canic-core/src/storage/stable/auth/records.rs:419`), and active endpoint
tests assert the old methods are absent (`crates/canic/tests/protocol_surface.rs:779`).

Implemented chain-key batch flow:

```text
root timer/update
  -> prepare_next_chain_key_root_delegation_batch
  -> build cert leaves and deterministic Merkle tree
  -> persist Prepared(batch_id, header_hash)
  -> sign_next_chain_key_root_delegation_batch
  -> management canister ecdsa_public_key + sign_with_ecdsa
  -> verify returned signature and persist Signed(batch_id, signature)
  -> start install plan
  -> root update calls issuer canic_install_active_delegation_proof
  -> issuer verifies proof and stores active proof
```

Root signing: yes, once per batch epoch/header. Bridge/direct root query: no.
Evidence: renewal workflow signs next batch
(`crates/canic-core/src/workflow/runtime/auth/renewal.rs:108`), signer uses
management ECDSA (`crates/canic-core/src/ops/auth/delegated/chain_key_signing.rs:208`,
`crates/canic-core/src/ops/auth/delegated/chain_key_signing.rs:221`), and install
fanout is in provisioning workflow
(`crates/canic-core/src/workflow/runtime/auth/provisioning/mod.rs:116`).

Lazy repair flow:

```text
issuer prepare/login update notices missing/stale active proof
  -> issuer calls root canic_get_or_create_chain_key_delegation_proof update
  -> root binds requested issuer to msg_caller
  -> root checks cache for signed proof
  -> if absent, prepare/reuse/sign one batch
  -> Pending/RetryAfter equivalent if signing already in flight
  -> issuer installs verified active proof
```

Root signing: possible only for missing/stale issuer proof, and only over the
batch header. Bridge/direct root query: no. Evidence: caller binding
(`crates/canic-core/src/api/auth/mod.rs:174`), internal subnet update gate
(`crates/canic/src/macros/endpoints/root.rs:97`), cache/sign path
(`crates/canic-core/src/ops/auth/delegation/chain_key_batch.rs:327`), and
in-flight test (`crates/canic-core/src/ops/auth/delegation/chain_key_batch.rs:2211`).

Login/token flow under fresh active proof:

```text
client -> issuer canic_prepare_delegated_token update
client -> issuer canic_get_delegated_token query
verifier -> verify DelegationCertV1
verifier -> verify RootProof::IcChainKeyBatchSignatureV1 against configured policy
verifier -> verify token claims are subset of cert grants
```

Root signing: no under fresh active proof. Bridge/direct root query: no. Evidence:
token endpoints are nonroot macros (`crates/canic/src/macros/endpoints/nonroot.rs:32`,
`crates/canic/src/macros/endpoints/nonroot.rs:39`), verifier root proof mode is
chain-key only (`crates/canic-core/src/ops/auth/token.rs:388`), and cached login
reuse is tested (`crates/canic-tests/tests/root_cases/auth_076.rs:99`).

`LegacyBridge`, `DualCode`, `ChainKeyPreferred`, or `ChainKeyOnly` cutover state.

  not found for delegated-token auth. Legacy wire/stable decode remains only for

  verifier fail-closed behavior (`crates/canic-core/src/config/validation/auth.rs:99`,

  protocol (`crates/canic/tests/protocol_surface.rs:779`).
- Mixed-issuer tests: not applicable to hard-cut design.
- Global time-window compatibility fallback: not found; config rejects legacy
  root proof mode (`crates/canic-core/src/ops/auth/token.rs:1010`).

If mixed-mode migration is now a product requirement, this is a design-contract
change. Against the current hard-cut design, absence of cutover states is not a
blocker.
