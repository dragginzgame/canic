# Canic 0.69 Design: Blob Storage Protocol

## Status

- Status: draft design note
- Release line: 0.69.x
- Current push target: 0.69.0 protocol preflight
- Decision style: extraction with strict protocol compatibility
- Primary implementation target: backend blob storage protocol foundation
- Current state: 0.69.0 preflight preparation is active. The protocol
  inventory scaffold exists, CI blocks premature implementation, local Toko
  compatibility evidence is captured, and all source modules, features,
  endpoint literals, DTOs, stable records, macros, and lifecycle tests remain
  blocked until
  `docs/contracts/BLOB_STORAGE_INVENTORY.md` is complete.

## 1. Summary

0.69 adds the backend `blob-storage` protocol and lifecycle foundation for the
Caffeine immutable object storage gateway.

Canic-owned names use `blob-storage`, `blob_storage`, and `BlobStorageApi`.
Gateway-owned names stay exactly `_immutableObjectStorage*`. Those method names
and Candid signatures are external protocol, not Canic style.

0.69 emits exactly four non-billing gateway endpoints:

1. `_immutableObjectStorageBlobsAreLive`
2. `_immutableObjectStorageBlobsToDelete`
3. `_immutableObjectStorageConfirmBlobDeletion`
4. `_immutableObjectStorageCreateCertificate`

0.69 inventories all six gateway methods so later work does not guess at the
contract. 0.70 owns Cashier sync and billing endpoints.

0.69 also exposes behavior-level internal Canic APIs so consumer backends do
not need to call `_immutableObjectStorage*` endpoints internally.

No frontend, TypeScript, npm, browser upload workflow, generic metadata
endpoints, or Cashier billing belongs in 0.69.

0.69.0 is the protocol preflight release for the line, not implementation
code.
Until the gateway protocol source is identified and
`docs/contracts/BLOB_STORAGE_INVENTORY.md` is marked complete, the only
permitted 0.69 work is design, inventory, source-provenance capture, gate
maintenance, and tests for the gate itself. After the inventory is complete,
the 0.69 implementation line proceeds through the backend milestones in this
design.

## 8. Protocol DTOs vs Internal Models

Gateway endpoints must accept and return protocol DTOs that match the inventory
exactly. Do not force internal typed wrappers into external gateway Candid.

Internal ops should convert protocol DTOs into typed models before validation
and mutation. `BlobRootHash` is an internal model type, not necessarily an
external DTO field type.

Candid snapshot tests must prove typed Rust wrappers do not alter external
gateway Candid.

### 8.1 BlobRootHash

Internal rules:

1. Accept only `sha256:<64 hex chars>` after protocol DTO conversion.
2. Accept uppercase or lowercase hex input.
3. Store normalized lowercase text.
4. Reject wrong prefix, wrong length, non-hex chars, whitespace, missing
   prefix, and control characters when applicable.
5. Never store malformed hashes.

Stable records may use typed internal models only if serialization remains
upgrade-safe and covered by upgrade round-trip tests.

## 9. Lifecycle State Machine

States:

| State | Meaning |
| --- | --- |
| `absent` | No live blob state and no pending deletion state. |
| `live` | Canonical live blob record exists. |
| `pending_deletion` | Live blob exists and deletion has been requested. |
| `deleted` | Optional tombstone state, only if required by the inventory. |

Allowed transitions:

| Transition | Rule |
| --- | --- |
| `absent -> live` | Certificate creation/registration succeeds. |
| `live -> live` | Allowed only for idempotent registration with identical immutable metadata. |
| `live -> pending_deletion` | Consumer backend requests deletion. |
| `pending_deletion -> absent` | V1 default after gateway confirmation when no tombstone is required. |
| `pending_deletion -> deleted` | Only if the inventory requires a tombstone. |
| `deleted -> live` | Allowed only if inventory or explicit Canic API policy permits re-registration. |

V1 should not create tombstones unless the inventory requires them. Without a
tombstone requirement, confirmed deletion returns the blob to `absent`.

0.69 v1 must use one canonical representation for pending deletion. The
implementation must choose and document one of:

1. `StoredBlobRecord.status = PendingDeletion`.
2. `BlobDeletionPendingRecord` keyed by the same canonical hash.

Do not use both as independent canonical state in v1. If a secondary map or
index is needed for efficient `_immutableObjectStorageBlobsToDelete`, it must
be derived or rebuildable from the canonical representation and tested as an
index, not a second source of truth.

Lifecycle invariants:

1. Malformed hashes are never stored.
2. A live blob has exactly one canonical live record.
3. Pending deletion state must refer to a live blob unless inventory proves
   otherwise.
4. Deletion confirmation removes records by default, or tombstones only if
   required by inventory.
5. Re-registration after deletion is explicit, not accidental.
6. Repeated certificate creation is idempotent only when immutable metadata
   exactly matches.
7. `None` followed by `Some(value)` metadata is a conflict unless the reference
   implementation proves enrichment behavior.

## 10. Consumer-Facing Internal APIs

0.69 exposes behavior-level Canic APIs under `BlobStorageApi`. Consumer
backends must not need to call `_immutableObjectStorage*` endpoints internally.
These APIs use internal typed models after protocol DTO conversion and preserve
the same lifecycle invariants used by the gateway endpoints.

| API | Semantics |
| --- | --- |
| `register_live` | Validates and registers a live blob, or returns an idempotent outcome when the same blob is already live with identical immutable metadata. Conflicting immutable metadata returns `BlobAlreadyLiveWithConflictingMetadata`. |
| `check_live` | Returns protocol-independent live/not-live state for a valid internal `BlobRootHash`. |
| `require_live` | Returns the live record or typed `BlobNotLive`. |
| `mark_pending_delete` | Transitions a live blob to pending deletion and creates the canonical pending-deletion record or index. |
| `confirm_deleted_by_gateway` | Applies the gateway-confirmed deletion transition after gateway-principal authorization. |

These APIs are backend behavior APIs, not generic metadata endpoints. They do
not add `set_blob_info`, `list_blobs`, `delete_blob`, owner query helpers,
client SDKs, or CLI/operator wrappers.

## 11. First-Consumer Compatibility

0.69 stable records and internal APIs must remain compatible with the first
consumer migration target, Toko. This does not mean 0.69 migrates Toko state.
It means 0.69 must not choose record shapes or lifecycle semantics that make it
impossible to map existing Toko live blobs, pending deletions, and gateway
principals into Canic behavior.

The downstream consumer chooses the actual state strategy: no existing state,
read-through, bulk registration, or state move.

Local Toko source evidence has been captured in the inventory. The current
Toko asset canister stores chunk-upload `Asset` and `Chunk` state, while project
instances store `RemoteAsset` projection rows with source asset identity in
`location.asset_id`. No Toko pending-deletion queue, gateway-principal state,
gateway-confirmation lifecycle, or canonical whole-blob root hash mapping was
found in the local source.

Therefore 0.69 must not treat Toko asset ULIDs, text references, optional file
metadata hashes, or per-chunk SHA-256 values as Canic `BlobRootHash` values.
`BlobRootHash` mapping remains an explicit compatibility blocker until the
inventory cites protocol evidence or a maintainer accepts an empty-state,
bulk-registration, or external-mapping migration path.

## 12. Stable Memory and Upgrade Requirements

Use existing Canic stable-record conventions. Do not introduce a different
storage framework.

Required storage planning:

1. Name stable map/logical namespaces for live blobs, pending-deletion state, and
   gateway principals.
2. Use normalized `BlobRootHash` as the key for blob maps.
3. Version serialized record schemas.
4. Bound unbounded fields such as `content_type`.
5. Define request-size and response-size limits for liveness batches and
   deletion candidate sets.
6. Preserve live blobs, pending deletion state, and gateway principals across
   upgrade round trips.
7. Keep record shapes compatible with future downstream migration or document a
   no-state-move path.

## 13. Endpoint Macro and Authorization

Existing Canic endpoint macros use `guard = <access expression>` for a single
consumer guard, for example `canic_emit_icp_refill_endpoints!(guard = ...)`.
0.69 should follow that style:

```rust
canic_emit_blob_storage_endpoints!(
    guard = <access expression>,
);
```

The guard protects only `_immutableObjectStorageCreateCertificate`.

Gateway-only methods check `caller()` against stored gateway principals:

1. `_immutableObjectStorageBlobsToDelete`
2. `_immutableObjectStorageConfirmBlobDeletion`

Unauthorized gateway-only behavior must match the inventory exactly. Test-seeded
gateway principals must not create a public production sync path.
`GatewayPrincipalSource::TestSeeded` must be test/local-only, or explicitly
documented as safe before production builds can use it.

0.69 macro emission:

| Endpoint | Mode | 0.69 auth |
| --- | --- | --- |
| `_immutableObjectStorageBlobsAreLive` | inventory-defined query | inventory-defined public/malformed behavior |
| `_immutableObjectStorageBlobsToDelete` | inventory-defined query | gateway-only stored-principal check |
| `_immutableObjectStorageConfirmBlobDeletion` | inventory-defined update | gateway-only stored-principal check |
| `_immutableObjectStorageCreateCertificate` | inventory-defined update | macro `guard = <access expression>` |

The macro must fail clearly when:

1. `blob-storage` is disabled.
2. `guard = <access expression>` is missing.
3. A 0.69 macro attempt emits deferred billing/sync endpoints.

## 14. Security Requirements

1. Certificate creation must be guard-protected in all production builds.
2. The endpoint macro must require `guard = <access expression>` and fail
   clearly when it is omitted.
3. Gateway-only methods must check only the stored gateway-principal set.
4. Consumer owner/admin guards must not bypass gateway-only authorization for
   `_immutableObjectStorageBlobsToDelete` or
   `_immutableObjectStorageConfirmBlobDeletion`.
5. Test/local seeded gateway principals must not create a public production sync
   path.
6. Production builds must not authorize `TestSeeded` principals unless a later
   design explicitly documents why that is safe.
7. Deletion candidate queries are gateway-facing protocol surface and must not
   leak pending deletion state to unauthorized callers unless the inventory
   requires it.

## 15. Internal Errors and Protocol Mapping

Internal error taxonomy:

1. `MalformedBlobRootHash`
2. `BlobAlreadyLiveWithConflictingMetadata`
3. `BlobNotLive`
4. `BlobAlreadyPendingDeletion`
5. `BlobDeletionNotPending`
6. `UnauthorizedGatewayPrincipal`
7. `GatewayPrincipalSetEmpty`
8. `ProtocolInventoryMissing`
9. `CertificateMaterialUnavailable`

Protocol mapping rules:

1. Internal errors must not automatically become external protocol variants.
2. If inventoried Candid uses `Result`, map errors into the exact existing
   protocol error variant.
3. If the reference traps or rejects, preserve that behavior.
4. Do not invent new externally visible gateway error variants in 0.69.

## 16. Certificate Creation

Certificate response construction and certificate material source must come
directly from `docs/contracts/BLOB_STORAGE_INVENTORY.md`. No placeholder, fake,
or locally invented certificate response or certificate material may be
implemented.

Inventory must decide:

1. Request validation before mutation.
2. Certificate material source expected by the gateway, such as IC update
   response certification, certified data, explicit response field, or another
   reference-defined source.
3. Whether registration happens before certificate material is produced, after
   it is produced, or as part of a single workflow whose certified state must be
   visible to the gateway.
4. Failure behavior if certificate material is unavailable.
5. Rollback or no-rollback behavior if certificate material generation fails
   after validation.
6. Repeated certificate creation response behavior.
7. Logging/caching restrictions for certificate material.

If required certificate material is unavailable, workflow must fail using the
inventoried protocol behavior. Certificate material must not be logged, cached,
or persisted unless the inventory proves that is part of the protocol.

## 17. Query Semantics

### 17.1 `_immutableObjectStorageBlobsAreLive`

Inventory must define:

1. Input ordering.
2. Duplicate hash behavior.
3. Malformed hash behavior.
4. Absent hash behavior.
5. Batch limits.
6. Output ordering.
7. Exact external input encoding.
8. Malformed-length behavior, including whether bad input traps, rejects,
   returns a result error, or maps to not-live.

Implementers must not assume the external input is text or `sha256:<hex>`. If
the reference protocol uses a 32-byte blob input, the endpoint must convert that
32-byte value into internal canonical `BlobRootHash` before lookup.

0.69 tests must lock the inventoried behavior.

### 17.2 `_immutableObjectStorageBlobsToDelete`

Inventory must define:

1. Result ordering.
2. Maximum batch size.
3. Repeat-return behavior until confirmation.
4. Exact non-gateway caller behavior: trap, reject, result error, empty
   response, or another reference-defined behavior.

The implementation must not choose a default for non-gateway callers. Tests
must prove the generated endpoint follows the inventory exactly.

### 17.3 `_immutableObjectStorageConfirmBlobDeletion`

Inventory must define exact unauthorized gateway-principal behavior: trap,
reject, result error, idempotent response, or another reference-defined
behavior.

0.69 tests must lock the inventoried behavior.
