# Canic 0.100 Design: Multi-Subnet Fleet Coordinator and Registry Synchronization

Date: 2026-07-23

## Status

- Status: proposed.
- Scope: multi-subnet topology foundation only.
- Implementation begins with a repository pass to map the canonical
  authorities in this document onto the current root, subnet-state, cascade
  and durable-workflow code.
- This is a hard-cut design. It must not introduce a second competing subnet
  registry, compatibility facade or parallel propagation path.
- Dependency: the [0.99 App/Fleet terminology and identity hard cut](../0.99-app-fleet-terminology/0.99-design.md)
  is complete.
- Compatibility posture: pre-1.0 current schema only. There is no protocol
  version field, old-name mapping layer or multi-schema decoder.

## Terminology Alignment With 0.99

This design uses the 0.99 terms directly:

- `FleetCoordinator` is the separately placed Canister that owns Fleet-wide
  coordination and membership state.
- `AuthoritySubnet` is the special Fleet Subnet where the Coordinator and
  explicitly declared Fleet-authoritative services are hosted. It is outside
  ordinary member lifecycle and workload placement.
- `FleetRegistry` is the complete Fleet-Coordinator-owned member Subnet/root
  membership authority.
- `FleetRegistryMirror` is the complete, read-only copy on each member root.
- `FleetDirectory` is the consumer-shaped projection cascaded from a root to
  ordinary Canisters.
- `FleetBinding { fleet: FleetKey, app: AppId }` is the immutable runtime
  identity established by 0.99.
- `SubnetSlotId` is a logical App configuration position; `SubnetId` is a
  resolved platform identity.

`Authority Subnet` is the canonical replacement for the former `Prime Subnet`
term. The distinguished-subnet concept remains: every Fleet has one Subnet
selected to host its Fleet Coordinator and declared authority services.
Fiduciary is the recommended live placement policy, not the domain name of
that concept.

A Registry is the complete authoritative record set within its scope. A
Directory is a derived, read-only, consumer-shaped projection carrying
Registry provenance. `Index` is not public topology vocabulary.

The 0.99 stable Canister identity, host-state migration and coordinated-upgrade
contracts also remain prerequisites. This design adds multi-subnet Fleet
behavior; it does not weaken or replace those contracts.

## Summary

Canic currently manages topology inside an individual subnet: that subnet’s
root owns the authoritative local state and distributes it to canisters on the
subnet.

What is missing is the layer above that:

Which managed subnets exist, which root belongs to each subnet, and which
Canister coordinates the Fleet-wide topology?

Install one `FleetCoordinator` on an independently selected Authority Subnet,
give it one Fleet-wide `FleetRegistry`, and make it the local root for the
Authority Subnet's declared authority services.

The Authority Subnet defaults to the Fiduciary subnet for public-IC live
deployments, resolves to a test-harness Subnet for local/PocketIC tests, and
may be overridden with a named network profile or explicit `SubnetId`. Every
member Subnet root keeps a durable Registry mirror. Only the Fleet Coordinator
and roots communicate across Subnets for topology synchronization; ordinary
Canisters receive a derived Fleet Directory through the existing same-Subnet
state cascade.

~~~text
Authority Subnet
  Fleet Coordinator
  owns authoritative FleetRegistry
  owns local SubnetRegistry
    authoritative database and other declared authority services
        |
        | infrequent inter-subnet synchronization
        v
each managed subnet root
  stores a complete FleetRegistryMirror
  owns a local SubnetRegistry
    optional local database replica
        |
        | existing same-subnet cascade
        v
local canisters
  store consumer-shaped FleetDirectory and SubnetDirectory
~~~

This gives Canic the topology foundation needed for later features such as:

- one instance of a service on every active subnet;
- resolving the local instance of a service;
- resolving the Fleet-authoritative instance from `FleetDirectory`; and
- resolving a member-local replica from `SubnetDirectory`.

0.100 provisions and publishes the declared Fleet-authoritative instances. The
database snapshot/change protocol, automatic member-replica provisioning and
replica promotion are later work.

## Problem

Per-subnet topology is insufficient once one Canic fleet spans several subnets.

Without one canonical Fleet Registry:

- roots cannot reliably discover other managed subnets;
- Canisters may need a hard-coded Fleet Coordinator principal;
- different roots can disagree about Fleet membership;
- adding or removing a subnet has no authoritative lifecycle;
- later placement policies such as `one_per_active_subnet` cannot be evaluated;
- application code may perform unnecessary inter-subnet discovery calls; and
- there is no consistent answer to “which Canister owns Fleet membership?”;
  and
- there is no canonical discovery path from a local replica to the
  Fleet-authoritative service it must refresh from.

The service-group design considered for VaultDB depends on these facts already
existing. Therefore multi-subnet Fleet Registry management must come first.

## Existing Authority Preserved

0.100 introduces a global authority without replacing local subnet authority.

| State | Authority | Meaning |
| --- | --- | --- |
| Local Subnet registry | Root of that Subnet | Canisters, roles and topology within one Subnet |
| Authority Subnet registry | Fleet Coordinator | Coordinator and declared Fleet-authoritative service children |
| Fleet Registry | Separately placed Fleet Coordinator | Managed member Subnets, their roots and lifecycle |

The exact existing type may no longer be named `SubnetState`. Implementation
must locate and reuse the current canonical local topology and cascade
authorities.

The two state families must remain semantically distinct:

- local topology changes are owned by the local root;
- Fleet membership changes are owned only by the Fleet Coordinator;
- Authority Subnet child topology changes are owned only by the Fleet
  Coordinator's local registry;
- a member root may mirror the Registry but may not mutate it; and
- an ordinary Canister may consume the Fleet Directory projection but may not
  synchronize topology directly with the Fleet Coordinator.

### Fleet Registry Access Contract

0.100 resolves the one-member `canic_fleet_registry` TBD carried from 0.99:

- on the Fleet Coordinator, controller-gated `canic_fleet_registry` is the
  complete administrative/read projection of the authoritative Registry;
- member roots expose their complete copy only as a controller-gated
  `canic_fleet_registry_mirror` diagnostic/backup projection, never as
  canonical authority;
- authenticated root synchronization uses the bounded
  `canic_fleet_registry_head` and `canic_fleet_registry_snapshot` endpoints;
- ordinary Canisters and public topology consumers receive only
  `FleetDirectory`; and
- backup captures the binding, manifest and complete immutable snapshot, but a
  backup file or restored mirror does not acquire mutation authority.

Restore or recovery must verify the exact protected `FleetBinding`, Authority
Subnet, Fleet Coordinator principal, authority epoch and content hash before
activation. Host access, controller visibility and backup possession are
read/recovery capabilities; none make the host, a mirror or a backup the
canonical writer.

## Authority Subnet Selection and Installation

0.100 implements the selector established by 0.99:

~~~rust
pub enum AuthoritySubnetSelector {
    NetworkDefault,
    Profile(SubnetProfileId),
    Explicit(SubnetId),
}

pub struct FleetCoordinatorBinding {
    pub fleet: FleetBinding,
    pub authority_subnet: SubnetId,
    pub coordinator: Principal,
}
~~~

The selector is host deployment input. `canic.toml` does not choose the
Authority Subnet, and runtime code never interprets a profile name.
`canic deploy plan` exposes mutually exclusive
`--authority-subnet-profile <profile>` and
`--authority-subnet <principal>` options. Omitting both produces
`NetworkDefault`.

The default policy is:

| Deployment | Resolved policy |
| --- | --- |
| Public IC live Fleet | Current network `fiduciary` profile |
| Other live network | Its configured authority default, otherwise a planning error |
| PocketIC/local/test Fleet | Authority Subnet supplied by the harness |
| Developer override | Exact named profile or explicit `SubnetId` |

The Fiduciary subnet is suitable as the live default because it is an
application subnet open to user Canister creation and uses a larger committee
than standard application subnets. The stronger fault threshold is valuable
for small, authoritative Fleet metadata. Its higher cycle multiplier is an
explicit tradeoff and applies to the Fleet Coordinator and declared authority
services, not ordinary workloads. A write-heavy or storage-heavy App may
explicitly choose another eligible application Subnet. The Fiduciary Subnet
does not give a Canister special system privileges, free execution or a second
writer. Coordinator authority still comes from the protected
`FleetCoordinatorBinding` and single-writer protocol.

Named profiles are open network metadata, not a protocol enum. The public IC
currently documents Fiduciary and European choices; a Swiss or other future
profile becomes selectable only when the chosen network's trusted metadata
actually advertises an eligible Subnet. The design does not reserve or
hard-code speculative names.

The installation workflow:

1. Resolve `NetworkDefault`, `Profile` or `Explicit` against trusted metadata
   for the exact `CanonicalNetworkId`.
2. Verify that the resolved Subnet exists, accepts arbitrary user Canister
   creation and has sufficient capacity.
3. Persist the selector, exact resolved `SubnetId`, metadata evidence and
   resolution time in the deployment plan.
4. Create the Fleet Coordinator explicitly on that Subnet; subnet
   selection affects creation only and cannot move an existing Canister.
5. Install it with the immutable `FleetBinding`, authority Subnet and initial
   authority epoch.
6. Persist `FleetCoordinatorBinding` in the host receipt, Coordinator and
   every member root's protected bootstrap state.
7. Provision the `[directories.fleet].roles` authority services as same-Subnet
   children of the Coordinator, recording each in its local `SubnetRegistry`.
8. Publish an authority service into `FleetDirectory` only after its install,
   protected Fleet binding, role and expected release evidence are verified.

An unavailable, ineligible, ambiguous or capacity-exhausted default fails the
plan or install. Canic never silently selects a random application Subnet.
The developer can rerun planning with a named profile or explicit Subnet.
System subnets are ineligible because they do not accept arbitrary user
Canister creation.

Changing deployment configuration after installation cannot move or replace
the authority. Authority relocation must fence old history by advancing the
authority epoch through a later explicit recovery design.

## 0.99-to-0.100 Coordinator Extraction

0.99 gives no member privileged status, but its sole current member root still
owns the one-member Fleet-wide state. 0.100 extracts that authority into the
separately placed Fleet Coordinator as one journalled hard cut.

For an existing Fleet:

1. Resolve and persist the Authority Subnet plan before quiescence.
2. Quiesce provisioning, membership mutation, Registry/Directory publication
   and application traffic that depends on changing Fleet topology.
3. Journal the current sole member root, local `SubnetRegistry`, Fleet binding
   and exact one-member `canic_fleet_registry` evidence.
4. Create and install the Fleet Coordinator on the resolved Authority Subnet
   with a staged epoch and Registry containing the current member as `Active`;
   it exposes no current manifest or mutation authority yet.
5. Verify the staged Registry and content hash.
6. Upgrade the existing member root with protected
   `FleetRegistryBootstrap`, remove its canonical Registry writer and activate
   its `FleetRegistryMirror`.
7. Atomically activate the Fleet Coordinator manifest and mutation authority.
8. Pull and verify the exact authority snapshot, then cascade the derived
   Fleet Directory locally.
9. Provision and verify declared authority services, commit and publish their
   Registry bindings, pull that exact revision into the member mirror, cascade
   its `FleetDirectory`, verify the Coordinator, member mirror and host receipt
   agree, and then remove quiescence.

The current root's `canic_fleet_registry` writer and read projection disappear
when that root upgrades. It exposes only `canic_fleet_registry_mirror`
afterward. The new Fleet Coordinator is the only process that ever exposes the
current `canic_fleet_registry` authority.

An interrupted extraction resumes from the host journal and staged/current
manifests. It never restores the member root as writer, creates a second
Coordinator, or reads the old root endpoint as a fallback after that root has
upgraded. Fresh 0.100 deployment installs the Fleet Coordinator and declared
authority services first, then joins member roots through the normal
`Joining` lifecycle.

## Goals

0.100 must provide:

1. One authoritative Fleet Registry of managed Subnets.
2. A unique Canic root for each managed subnet.
3. A separately placed Fleet Coordinator with a protected Authority Subnet and
   Canister binding.
4. Explicit registration, activation, draining and removal lifecycles.
5. Durable synchronization from the Fleet Coordinator to every member root.
6. Atomic activation of complete Registry snapshots.
7. Same-subnet propagation through the existing root cascade.
8. Local APIs for inspecting authority and member-Subnet topology.
9. Fleet-bound provenance and freshness information.
10. Failure-safe operation while the Fleet Coordinator is temporarily
    unavailable.
11. A bounded inter-subnet call pattern.
12. Same-Subnet provisioning and Directory publication for explicitly
    declared Fleet-authoritative services.

## Core Invariants

The implementation must preserve these invariants:

1. Exactly one authority descriptor exists for the `FleetBinding`.
2. The authority descriptor carries that exact immutable binding and identifies
   exactly one Authority Subnet and Fleet Coordinator.
3. Every snapshot, manifest, version, acknowledgement, mirror record,
   projection and hash preimage binds the same `FleetBinding`.
4. The Authority Subnet is part of Fleet topology but not an ordinary
   `FleetRegistryEntry`; its binding cannot be joined, drained or removed
   through the member lifecycle.
5. Every member Subnet ID appears at most once in Registry entries. Explicit
   physical co-location with the Authority Subnet does not merge the
   Coordinator and member-root registry scopes.
6. Every non-removed root principal belongs to at most one Subnet.
7. Only the Fleet Coordinator can commit a new Registry revision.
8. Registry revisions increase monotonically within an authority epoch.
9. A revision identifies exactly one canonical content hash within one
   `FleetBinding` and authority epoch.
10. Evidence from another network, Fleet or App never compares equal or
    activates, even when epoch, revision and entries match.
11. A mirror never activates a partial or unverified snapshot.
12. A mirror never replaces a newer Registry with an older one.
13. A mirror never promotes itself because the Fleet Coordinator is
    unreachable.
14. Ordinary Canisters never need an inter-Subnet call to read Fleet Directory
    state.
15. `Joining`, `Draining` and `Removed` Subnets are not treated as normal
    active placement targets.
16. Registry removal does not falsely claim that the removed root Canister
    has been stopped or deleted.
17. Ordinary workload placement never targets the Authority Subnet.
18. Every declared Fleet-authoritative role resolves to at most one ready
    Canister in `FleetDirectory`.
19. The Coordinator's `SubnetRegistry` is the authority for its local
    authority-service children; the Fleet Registry is not a duplicate
    Canister registry.

## Data Model

~~~rust
pub struct FleetRegistry {
    pub authority: FleetRegistryAuthority,
    pub revision: u64,
    pub members: Vec<FleetRegistryEntry>,
    pub authority_services: Vec<FleetAuthorityServiceBinding>,
}

pub struct FleetRegistryAuthority {
    pub binding: FleetCoordinatorBinding,
    pub epoch: u64,
}

pub struct FleetRegistryEntry {
    pub subnet: SubnetId,
    pub root: Principal,
    pub status: ManagedSubnetStatus,
}

pub struct FleetAuthorityServiceBinding {
    pub role: CanisterRole,
    pub canister: Principal,
    pub release_evidence: ReleaseEvidenceDigest,
}

pub enum ManagedSubnetStatus {
    Joining,
    Active,
    Draining,
    Removed,
}
~~~

Member entries and authority-service bindings must each use one canonical
ordering, preferably ascending canonical Subnet identity and role identity
respectively. An authority role appears at most once. Hashing, equality and
publication must never depend on insertion order.

The content hash is publication metadata rather than a self-referential field
inside the hashed Registry snapshot:

~~~rust
pub struct FleetRegistryManifest {
    pub authority: FleetRegistryAuthority,
    pub revision: u64,
    pub byte_length: u64,
    pub content_hash: [u8; 32],
}
~~~

A complete Registry identity is therefore:

~~~rust
pub struct FleetRegistryVersion {
    pub authority: FleetRegistryAuthority,
    pub revision: u64,
    pub content_hash: [u8; 32],
}
~~~

The hash preimage is domain-separated and includes the full authority
descriptor, revision, canonically encoded member entries and canonically
encoded authority-service bindings. Consequently, otherwise identical
Registries for different Authority Subnets or Coordinators, for local, staging
and mainnet, or for two Fleets of the same App produce different evidence.

## Authority Epoch

The authority epoch fences Registry history belonging to different Fleet
Coordinators within one immutable `FleetBinding`.

0.100 records the epoch but does not implement automatic authority relocation
or replacement.

Every root is provisioned with a protected bootstrap descriptor:

~~~rust
pub struct FleetRegistryBootstrap {
    pub authority: FleetRegistryAuthority,
    pub local_subnet: SubnetId,
}
~~~

A mirror accepts snapshots only from the configured Fleet Coordinator
and only for the configured authority descriptor and exact protected
`FleetBinding`.

Within 0.100:

- a lower epoch is rejected as stale;
- a different Authority Subnet or Coordinator at the same epoch is rejected;
- an unexpected higher epoch is rejected pending an explicit recovery operation;
- no root derives authority from availability or majority; and
- no mirror can promote itself.

A later design may define a protected authority-relocation workflow that
advances the epoch. Merely including the epoch in 0.100 avoids publishing an
unfenceable format.

## Bootstrap Truth

Configuration provides only the minimum facts needed to join the fleet:

- the immutable Fleet binding;
- the expected local subnet identity;
- the Authority Subnet;
- the Fleet Coordinator principal; and
- the expected authority epoch.

It must not contain a second complete list of managed subnets.

These facts are deployment/install input persisted in protected host and root
state. They are not `canic.toml` App configuration. The runtime
`FleetRegistry` becomes the only authority for Fleet membership.

Canic must not claim it has cryptographically discovered which subnet hosts a
root if the Internet Computer does not expose that fact to the canister. The
`(subnet, root)` relationship must come from the existing trusted
deployment/configuration authority and be verified as far as the platform
permits.

## Fleet Coordinator Registry Publication

The Fleet Coordinator stores the authoritative Fleet Registry as an immutable,
revisioned snapshot. It does not own any member's local Canister registry. It
does separately own the Authority Subnet's local `SubnetRegistry`; that local
registry contains the Coordinator and declared authority-service children, not
member Subnet rows.

A Registry mutation must:

1. Load the current canonical Registry.
2. Validate the requested transition.
3. Construct the complete next snapshot.
4. Serialize it deterministically.
5. Enforce entry-count and byte-size limits.
6. Calculate its content hash.
7. Durably store the complete snapshot.
8. Atomically make its manifest current.

`canic_fleet_registry_head` must never expose a manifest whose snapshot is
absent or incomplete.

~~~rust
async fn canic_fleet_registry_head(
) -> Result<FleetRegistryManifest, FleetRegistryError>;

async fn canic_fleet_registry_snapshot(
    version: FleetRegistryVersion,
) -> Result<Vec<u8>, FleetRegistryError>;
~~~

The exact inter-Canister call mode must follow existing Canic and IC
constraints. These are logically read-only operations even if the platform
requires update-call transport.

## Fleet-Authoritative Services and Mutable Data

`[directories.fleet].roles` declares the App roles for which every Fleet has
one authoritative service instance. A mutable IcyDB database is the motivating
case: its Authority Subnet instance accepts canonical writes, while databases
on member Subnets may hold local replicas.

The Fleet Coordinator provisions each declared role as a same-Subnet child:

1. Resolve the role artifact, install arguments, cycle allocation and expected
   release evidence from the deployment plan.
2. Create the Canister on the Coordinator's own Authority Subnet.
3. Install it with the immutable `FleetBinding` and declared role.
4. Verify readiness, role identity and release evidence.
5. Record it in the Coordinator-owned local `SubnetRegistry`.
6. Commit its unique `FleetAuthorityServiceBinding` in the next Fleet Registry
   revision.
7. Publish that binding as the role's entry in `FleetDirectory`.

The binding is not published before readiness. The local `SubnetRegistry`
remains authoritative for Canister lifecycle on the Authority Subnet; the
Fleet Registry binding is authoritative for the Fleet-wide role-to-Canister
choice. Their scopes are different, and neither is a second mutable copy of
the other.

Member roots continue to own their local Canisters. When a member Subnet has a
replica of a Fleet-authoritative role, that local Canister appears in the
member root's `SubnetRegistry` and its consumer binding appears in
`SubnetDirectory`. A consumer can therefore resolve:

- `FleetDirectory[role]` for the canonical writable authority; and
- `SubnetDirectory[role]` for the replica local to its member Subnet.

A mutation targets the Fleet-authoritative Canister. A replica refresh uses
the same Fleet Directory binding to identify its canonical source and its
Subnet Directory binding to identify the local destination. 0.100 supplies
that discovery and provenance path; it does not define IcyDB snapshot/change
transfer, consistency guarantees, conflict handling or automatic replica
creation.

The Authority Subnet deliberately concentrates the Coordinator and canonical
Fleet-wide services in one subnet failure domain. The IC replicates each
Canister within that Subnet, but this remains one logical write authority. If
the Authority Subnet is unavailable, canonical writes and refreshes pause.
Member replicas may continue serving their last verified read state according
to application policy; no replica is automatically promoted. An explicit
fenced promotion or Authority Subnet relocation requires a later design.

This placement is also why ordinary workloads are excluded from the Authority
Subnet. Fiduciary is the safety-oriented live default, but its cycle multiplier
may be inappropriate for a write-heavy or storage-heavy authoritative service.
The developer may select another eligible Authority Subnet in the deployment
plan without changing the topology or Directory contracts.

## Bounded Snapshot Model

0.100 uses complete Registry snapshots, not deltas or an event log.

The Registry is expected to be small and change rarely. The implementation
must define limits such as:

~~~rust
const MAX_MANAGED_SUBNETS: usize = /* bounded value */;
const MAX_FLEET_AUTHORITY_SERVICES: usize = /* bounded value */;
const MAX_FLEET_REGISTRY_SNAPSHOT_BYTES: usize = /* bounded value */;
~~~

A mutation that would exceed either limit fails before changing authoritative state.

Chunk transfer is excluded from 0.100. If a Fleet Registry ever becomes too
large for one bounded response, that should be justified by observed use
before adding a generic transfer protocol.

## Mirror Synchronization

Every member root owns one durable synchronization state:

~~~rust
pub struct FleetRegistryMirrorState {
    pub authority: FleetRegistryAuthority,
    pub active: Option<FleetRegistryVersion>,
    pub last_polled_at: Option<Timestamp>,
    pub last_synchronized_at: Option<Timestamp>,
    pub consecutive_failures: u32,
    pub next_poll_at: Option<Timestamp>,
    pub last_failure: Option<FleetRegistrySyncFailure>,
}
~~~

A timer-driven synchronization pass performs:

1. Call the configured Fleet Coordinator for `canic_fleet_registry_head`.
2. Validate the full Fleet binding and authority descriptor.
3. Return immediately if the exact version is already active.
4. Reject an older epoch or revision.
5. Reject the same epoch and revision with a different hash.
6. Fetch the complete snapshot when behind.
7. Verify its length and hash.
8. Decode and validate every Registry invariant.
9. Durably stage the snapshot.
10. Atomically activate it.
11. Enqueue or resume the existing same-subnet cascade.
12. Record synchronization success.

Revision gaps are permitted because every publication is a complete snapshot.
A mirror can move directly from revision 4 to revision 12.

Polling must be configurable within safe bounds. There is one polling actor per
member Subnet, not one per Canister.

Retries use bounded backoff and the existing Canic timer and durable-work
machinery where applicable.

## Interruption and Uncertain Outcomes

All inter-subnet synchronization calls in 0.100 are observational and safe to repeat.

Local activation and cascade must also be idempotent.

If interruption occurs:

- before snapshot activation, the previous Registry remains active;
- after activation but before the success receipt, restart detects the
  already-active `FleetRegistryVersion`;
- during local cascade, the active root Registry remains valid and cascade
  resumes through its existing durable progress; and
- after receiving a response whose local receipt was not persisted, the root
  repeats the head or snapshot request.

No pending state may be reset merely because the previous process tree became quiescent.

## Subnet Registration

Registration is Fleet-Coordinator-authorized and phased.

~~~text
Absent → Joining → Active
~~~

### Create Joining Entry

An authorized Fleet administrator registers:

- the subnet identity; and
- its expected root principal.

The Fleet Coordinator validates uniqueness and publishes a new Registry revision
with the entry in `Joining`.

A `Joining` subnet:

- is visible for bootstrap and diagnostics;
- is not returned by normal `active_subnets`;
- is not eligible for placement; and
- cannot be treated as a service destination.

### Join Acknowledgement

The joining root:

- starts with its protected bootstrap descriptor;
- pulls the authoritative Fleet Registry;
- verifies that it contains its expected `(subnet, root)` entry;
- activates that snapshot locally; and
- calls the Fleet Coordinator with an idempotent join acknowledgement.

~~~rust
async fn acknowledge_subnet_join(
    applied: FleetRegistryVersion,
) -> Result<JoinStatus, FleetRegistryError>;
~~~

The Fleet Coordinator verifies:

- the caller is the root principal recorded in the `Joining` entry;
- the acknowledged Fleet binding exactly matches protected authority state;
- the reported authority epoch matches;
- the acknowledged revision and hash were published by the Fleet Coordinator;
  and
- the acknowledged snapshot contained that joining entry.

A valid acknowledgement transitions the subnet to Active and publishes the next revision.

Repeating an acknowledgement after activation returns the existing active result.

## Draining and Removal

Removal is also phased:

~~~text
Active → Draining → Removed
~~~

### Draining

Draining means:

- do not choose the subnet for new placement;
- retain it for explicit lookup and evacuation;
- retain its root identity; and
- allow existing workloads to be managed according to their domain policy.

The Registry does not decide when application evacuation has finished. An
authorized administrative action confirms the transition to `Removed`.

### Removed

Removed means:

- exclude the subnet from normal active resolution;
- reject it as a new placement target;
- preserve a tombstone preventing stale state from looking like a new registration; and
- reject Fleet-management authority from that root where Registry membership
  is required.

A removed entry is not immediately deleted from the Registry.

0.100 does not permit reusing a removed Subnet identity as a fresh
registration. Replacement or tombstone compaction requires a later explicit
design.

Registry removal does not stop the root Canister, change controllers or erase
local state. Those are deployment operations outside the Registry. A removed
but still-running root may retain stale local knowledge, but active Fleet
members no longer recognize it as active.

Every member Subnet, including one physically co-located with the Authority
Subnet, follows the same lifecycle and may be drained or removed. Removing a
co-located member entry does not move the Fleet Coordinator, delete its local
authority services or remove the Authority Subnet from the Fleet descriptor.
Coordinator relocation is a separate operation outside 0.100.

## Local Propagation

A root must not create a new direct global channel to every Canister.

After activating a Registry revision, the root uses the existing local
topology/state cascade to distribute the relevant Fleet Directory to:

- existing managed Canisters; and
- Canisters joining or being created later.

0.100 evolves the one canonical `FleetDirectory` envelope introduced in 0.99;
it does not add a second companion projection:

~~~rust
pub struct FleetDirectory {
    pub provenance: FleetDirectoryProvenance,
    pub entries: Vec<DirectoryEntryInput>,
    pub subnets: Vec<FleetDirectorySubnet>,
}

pub struct FleetDirectoryProvenance {
    pub authority: FleetRegistryAuthority,
    pub source_root: Principal,
    pub registry_revision: u64,
    pub registry_content_hash: [u8; 32],
    pub synchronized_at: Timestamp,
}

pub struct FleetDirectorySubnet {
    pub subnet: SubnetId,
    pub root: Principal,
    pub status: ManagedSubnetStatus,
}
~~~

The existing `entries` retain their current consumer lookup meaning. In 0.100
they are the projection of the Registry's verified
`authority_services` bindings. The new `subnets` projection adds member
topology facts; it does not describe local replicas. Member roots derive the
whole projection from their active verified `FleetRegistryMirror`, and the
existing `fleet_directory` state-cascade field carries the current shape as a
deliberate pre-1.0 wire cut. Each root derives `SubnetDirectory` separately
from its own local `SubnetRegistry`, including any local replica bindings.

The semantic rules are fixed:

- the Coordinator's current Registry is the Authority Subnet source, and each
  member root's active verified mirror is its local source;
- ordinary Canisters never fetch the Registry from the Fleet Coordinator;
- projection does not give ordinary Canisters mutation authority;
- new Canisters receive the latest locally active Fleet Directory;
- interrupted distribution resumes rather than starting a second cascade; and
- consumers can observe the Registry revision and freshness they received.

## Consumer API

0.100 exposes topology facts, not service placement or health guesses.

~~~rust
pub fn fleet_registry_authority(
) -> Result<FleetRegistryAuthority, TopologyError>;

pub fn authority_subnet() -> Result<SubnetId, TopologyError>;

pub fn fleet_coordinator() -> Result<Principal, TopologyError>;
~~~

If the current canister needs local subnet identity, that must come from the
current trusted local topology projection rather than being guessed from its
principal.

The API must return explicit errors for:

- missing Fleet Directory state;
- unknown subnet;
- non-active subnet where active membership is required;
- Fleet binding mismatch;
- authority mismatch; and
- corrupt projected state.

0.100 does not add policy-based routing such as:

~~~rust
best_instance_right_now("vault-db")
~~~

Exact role lookup already comes from `FleetDirectory` and `SubnetDirectory`.
Choosing among replicas by health, latency or freshness requires a later
placement/routing design.

## Freshness and Authority Subnet Downtime

If the Fleet Coordinator is temporarily unreachable:

- each root retains its last complete Fleet Registry;
- existing local topology continues operating;
- ordinary Canisters keep their last distributed Fleet Directory;
- local Fleet Directory lookups remain available with stale provenance;
- synchronization retries with bounded backoff;
- subnet registration, activation, draining and removal are unavailable; and
- no mirror becomes authoritative.

Coordinator unavailability alone does not make a separately running authority
service unavailable. If the whole Authority Subnet is unavailable, its
authority services are also unreachable: canonical writes and replica
refreshes pause. Member replicas may keep serving their last verified read
state when their application policy allows it. They never accept canonical
write authority or promote themselves merely because the Coordinator or
Authority Subnet is unreachable.

Canic exposes synchronization age but does not claim a stale Registry or
Directory is fresh.

Callers that require fresh Fleet membership may apply an explicit maximum-age
policy. Generic Directory lookup does not silently reject stale state unless
the operation specifically requires freshness.

## Authorization

The existing Canic authorization authority must protect Fleet Registry
administration.

At minimum:

- only authorized Fleet administration may create, drain or remove Subnet
  entries;
- only the configured Fleet Coordinator may publish authoritative
  Registry snapshots;
- only the root recorded in a `Joining` entry may acknowledge that join;
- mirror synchronization targets the exact configured Fleet Coordinator
  principal;
- every administrative request, root call and acknowledgement binds the exact
  protected `FleetRegistryAuthority`;
- a snapshot’s internal claim to be authoritative is not sufficient;
- member roots expose no canonical Registry mutation endpoint;
- ordinary Canisters cannot mutate root mirror state; and
- removed roots are rejected by operations requiring active Fleet membership.

No new independent administrator list should be created for 0.100.

## Validation Rules

A Fleet Registry snapshot is valid only when:

- its full `FleetBinding` exactly matches protected bootstrap state;
- the authority epoch, Authority Subnet and Fleet Coordinator principal match
  protected bootstrap state;
- its revision is non-zero or follows the chosen canonical genesis rule;
- entries are canonically ordered;
- subnet IDs are unique;
- active root principals are unique;
- every status value is recognized;
- all identities pass existing canonical validation;
- serialized size and entry count are within bounds; and
- the content hash matches the received bytes.

The same `(FleetRegistryAuthority, revision)` arriving with a different hash
is not a routine retry failure. It is authority equivocation or corruption
and must be surfaced as a terminal synchronization failure requiring
investigation.

A different `FleetRegistryAuthority` is always rejected before revision
comparison. Matching revision and content from another canonical network,
Fleet, App, Authority Subnet or Fleet Coordinator is not an idempotent retry.

## Observability

Each root should expose:

- protected Fleet binding;
- local active Registry version;
- configured authority descriptor;
- last successful synchronization time;
- last polling attempt;
- synchronization age;
- consecutive failures;
- next retry time;
- last typed failure;
- local cascade progress for the active revision; and
- whether its own subnet is `Joining`, `Active`, `Draining` or `Removed`.

Metrics should include:

- successful and failed head checks;
- downloaded snapshot count and bytes;
- rejected stale revisions;
- hash or invariant failures;
- synchronization interruption and resume;
- Registry activation duration;
- cascade completion age;
- current Registry revision; and
- current Registry synchronization age.

Failures must remain typed through internal ownership. Human-readable strings
are projections, not the stored error authority.

## Stable-State and Schema Rules

The implementation must identify:

- the canonical stable owner of the authoritative Fleet Registry;
- the canonical stable owner of each mirror;
- the current stable owner of local cascade progress; and
- whether the Fleet Coordinator and mirrors use the same stored snapshot type.

There must not be:

- two stored canonical Subnet lists;
- a configuration list continually dual-written with the runtime Registry;
- both mutable row state and an independently authoritative published snapshot; or
- legacy and new Registry readers running in parallel.

For authority services, the local `SubnetRegistry` row is the lifecycle
authority and `FleetAuthorityServiceBinding` is the Fleet-wide role-selection
authority. The Coordinator must commit their transition through one durable
workflow and must not add a third mutable service map or publish a binding that
does not resolve to its verified local row.

If an existing global-subnet concept already exists, 0.100 must migrate or
replace it as a hard cut rather than wrapping it indefinitely.

Stable snapshots, manifests, mirror state, cascade progress and provenance have
one current shape with the full `FleetRegistryAuthority`. Any migration
required for deployed Canisters must be explicit, bounded and tested through
upgrade. After activation there is no schema tag, old decoder or fallback
reader; pre-1.0 internal structures are replaced directly.

## Implementation Slices

### Slice 1 — Authority and Model Audit

- Locate the current root, local subnet state, subnet registry, configuration
  authority and cascade machinery.
- Record the current source of local subnet and root identity.
- Identify any existing Fleet-wide Subnet list or hard-coded privileged
  root/Subnet principal and remove that assumption.
- Map the 0.99 Authority Subnet selector onto host planning, creation and
  receipts.
- Bind the canonical 0.99 `FleetBinding` into every proposed Registry,
  bootstrap, evidence, mirror and cascade owner.
- Map this design onto existing owners and remove duplicate proposed types.
- Freeze the 0.100 invariants and ownership table.

### Slice 2 — Canonical Fleet Registry

- Add the distinct Fleet Coordinator role and artifact with Authority Subnet
  local-registry ownership but no member-root registry ownership.
- Add the Fleet Registry domain model and typed failures.
- Add deterministic encoding, ordering, hashing and bounds.
- Domain-separate hashes over the full Fleet binding and Registry content.
- Add Fleet-Coordinator-owned stable storage.
- Add atomic immutable snapshot publication.
- Add Fleet Coordinator read endpoints and validation tests.
- Implement the journalled 0.99 authority extraction with no dual writer.

### Slice 3 — Registration Lifecycle

- Add Fleet-Coordinator-authorized `Joining` registration.
- Add idempotent root acknowledgement.
- Add `Joining` → `Active`.
- Add `Active` → `Draining` → `Removed`.
- Reject illegal transitions without reserving a privileged member.
- Preserve removed tombstones.

### Slice 4 — Root Mirror

- Add the protected bootstrap descriptor.
- Add the single durable polling job.
- Add head comparison and complete snapshot retrieval.
- Add hash and invariant verification.
- Add atomic mirror activation.
- Add retry, backoff, reconciliation and administrative force-poll.

### Slice 5 — Local Cascade and API

- Evolve the existing Fleet Directory in place from the active Registry.
- Project it through the existing local cascade.
- Ensure newly created canisters receive the current revision.
- Resume interrupted distribution.
- Add consumer lookup and provenance APIs.
- Remove application-owned hard-coded Coordinator and authority-service lookup
  paths that the Directory APIs replace.

### Slice 6 — Authority-Service Provisioning

- Treat `[directories.fleet].roles` as the explicit authority-service
  provisioning set.
- Create and install each authority service as a same-Subnet Coordinator child.
- Register the child in the Authority Subnet `SubnetRegistry`.
- Verify readiness and release evidence before publishing the unique

- Cascade `FleetDirectory` authority bindings while preserving local

- Qualify restart and uncertain outcomes at every create/install/register/

### Slice 7 — Failure Qualification and Closeout

- Exercise multi-root PocketIC or the closest available multi-subnet environment.
- Inject interruption at every durable boundary.
- Verify upgrade recovery for Fleet Coordinator, authority-service, mirror and
  cascade state.
- Run repository-wide duplicate-authority and stale-path searches.
- Update architecture, operator and recovery documentation.

## Required Tests

### Model and Authority

- public-IC live `NetworkDefault` resolves the current Fiduciary profile;
- local and PocketIC `NetworkDefault` uses the harness-provided Subnet;
- named and explicit developer overrides resolve to the exact selected Subnet;
- unknown profiles, system Subnets, cross-network IDs and unavailable targets
  fail without random fallback;
- plans and receipts bind the selector and exact resolved `SubnetId`;
- Genesis Registry may have no member rows before the first member joins; the
  Authority Subnet is represented by the Coordinator binding rather than a
  member row.
- Authority placement does not make the Authority Subnet eligible for ordinary
  member lifecycle or workload placement.
- Every Registry, manifest and version carries the exact protected
  `FleetRegistryAuthority`.
- Hashes differ when only canonical network, Fleet ID or App ID differs.
- Hashes differ when only Authority Subnet or Fleet Coordinator differs.
- Evidence from another network, Fleet or App is rejected before revision
  comparison.
- Duplicate subnet IDs are rejected.
- Duplicate active root principals are rejected.
- An Authority Subnet or Coordinator mismatch is rejected.
- Non-canonical ordering is rejected or canonicalized before publication.
- duplicate authority roles or principals are rejected;
- every authority-service binding resolves to a verified row in the
  Coordinator's local `SubnetRegistry`;
- Over-limit entry count and serialized size fail before commit.
- Unauthorized Registry mutation is rejected.

### Registration

- A new subnet begins as `Joining`.
- `Joining` is excluded from `active_subnets`.
- Acknowledgement from the wrong principal is rejected.
- Acknowledgement of an unknown snapshot is rejected.
- Valid acknowledgement activates the subnet.
- Lost acknowledgement response can be retried safely.
- Duplicate registration is rejected.

### Draining and Removal

- `Active` → `Draining` → `Removed` succeeds.
- Direct `Active` → `Removed` is rejected unless explicitly justified.
- `Joining` → `Active` is the only normal activation path.
- `Draining` and removed subnets are excluded from normal placement views.
- A removed subnet remains as a tombstone.
- A removed identity cannot silently re-register.
- Every member follows the same lifecycle, including a member co-located with
  the Authority Subnet.
- Removing a co-located member does not alter the Fleet Coordinator binding or
  its Authority Subnet registry.

### Synchronization

- A mirror with no Registry bootstraps from the Fleet Coordinator.
- An up-to-date mirror performs only the bounded head check.
- A mirror skips directly across multiple revisions.
- An older revision cannot replace a newer revision.
- The same revision and hash is an idempotent no-op.
- The same revision with a different hash fails terminally.
- A corrupted or truncated snapshot never activates.
- A snapshot with a different Fleet binding leaves the previous snapshot
  active.
- An authority descriptor mismatch is rejected.
- Unexpected lower and higher authority epochs fail closed.

### Interruption and Recovery

- Interruption before Fleet Coordinator creation leaves the 0.99 root
  authority intact and quiesced.
- Interruption after staging or publication resumes the same extraction
  without creating another Fleet Coordinator.
- Interruption after member-root upgrade never restores its removed canonical
  writer.
- At no durable boundary can both the Fleet Coordinator and member root accept
  Registry mutation.
- Interruption before durable snapshot staging retains the old Registry.
- Interruption after staging but before activation retains the old Registry.
- Interruption after activation but before receipt reconciles as complete.
- Upgrade during polling resumes safely.
- Upgrade during local cascade resumes from durable progress.
- Repeated timeouts do not discard the last valid Registry.
- Fleet Coordinator downtime never promotes a mirror.

### Authority Services

- every `[directories.fleet].roles` role provisions exactly one ready
  Authority Subnet child;
- create, install, local registration and Fleet publication resume
  idempotently after interruption;
- an uninstalled, unready, wrong-role or wrong-release child is never
  published;
- ordinary workload roles cannot be placed on the Authority Subnet;
- the Fleet Directory resolves the canonical authority while a member Subnet
  Directory resolves that member's local replica;
- an Authority Subnet outage blocks canonical writes and refreshes without
  promoting a member replica; and
- a validated developer override can place a write-heavy authority service on
  a non-Fiduciary eligible Subnet without changing Directory semantics.

### Local Distribution

- Existing Canisters receive the new Registry revision in the Fleet Directory
  projection.
- A newly created Canister receives the root’s current revision.
- No ordinary Canister performs an inter-Subnet Registry call.
- Partial cascade is observable and recoverable.
- Provenance survives projection without being replaced by local receipt time.

## Non-Goals

0.100 does not implement:

- IcyDB snapshot, change-stream or application-data replication;
- service groups;
- automatic member-replica or `one_per_active_subnet` provisioning;
- health-, latency- or freshness-based service routing;
- automatic Fleet Coordinator election, replacement or relocation;
- root-to-root consensus;
- multi-writer Registry mutation;
- root-to-root mesh synchronization;
- Registry deltas or change-data capture;
- chunked artifact transfer;
- generic distributed transactions;
- proof that two application databases contain identical data;
- subnet health or “best available subnet” selection; or
- physical removal of a subnet, root or canisters.

## Hard-Cut Completion Criteria

0.100 is complete when:

1. The separately placed Fleet Coordinator is the only canonical writer
   of Fleet Subnet membership.
2. Every authoritative and mirrored record and every evidence hash binds the
   exact canonical network, Fleet and App through `FleetBinding`.
3. Public-IC live deployment defaults to the dynamically resolved Fiduciary
   profile, tests use their harness Subnet, developer overrides are validated,
   and no failure falls back to a random Subnet.
4. The Authority Subnet is a distinguished Fleet topology component bound
   outside ordinary member rows, lifecycle and workload placement.
5. Every member root durably mirrors a verified complete Fleet Registry.
6. Registry changes propagate to ordinary Canisters as a consumer-shaped
   Fleet Directory through the existing local cascade.
7. Registration, activation, draining and removal are explicit and tested.
8. Authority Subnet downtime preserves eligible replica read availability
   without preserving writes or causing promotion.
9. Inter-Subnet topology synchronization is limited to the Fleet Coordinator
   and member roots, bounded and infrequent.
10. Consumers can resolve authority and active-Subnet topology locally with
   Fleet-bound provenance.
11. Every maintained distinguished-subnet surface uses Authority Subnet
    terminology; `Prime Subnet` appears only when identifying the removed
    name.
12. No unexplained competing Subnet list or privileged-member authority
    remains.
13. The 0.99 one-member root's authority is extracted exactly once with no
    dual writer or post-upgrade fallback.
14. Upgrade and interruption tests prove every durable boundary.
15. Declared Fleet-authoritative roles are provisioned on the Authority Subnet,
    registered locally and published uniquely through `FleetDirectory`.
16. Member-local replicas are discoverable through `SubnetDirectory`, while
    their creation and data synchronization remain outside the implementation.

## Expected Follow-On

With 0.100 complete, a later design can safely add replication and
replica-placement policy:

~~~text
0.100
  “Which managed Subnets exist, which roots own them,
   which Fleet Coordinator owns membership, and where is
   each declared Fleet-authoritative service?”

Later replica placement
  “Which member Subnets require a local replica,
   and how is it provisioned?”

Later publication synchronization
  “How does one authoritative service distribute
   application-owned data to its mirrors?”
~~~

0.100 supplies the topology and Directory truth required by that later policy.
Application data remains in a separate authority-service Canister, not inside
the Fleet Coordinator, but the service intentionally shares the Authority
Subnet so the Coordinator can provision and register it as a same-Subnet
child.
