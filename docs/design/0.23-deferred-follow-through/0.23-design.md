# 0.23 Design: Deferred Follow-Through

This document carries the intentionally deferred work from the closed `0.22`
audit, wasm-size, and perf line into a new `0.23` lane.

`0.22` did the heavy cleanup:

- stabilized the `debug` / `fast` / `release` wasm contract
- cleaned the local thin-root install flow
- simplified the shared setup/install path
- reduced the worst managed `wasm_store` bootstrap hotspots
- made PocketIC-heavy test runs and local installer progress more readable

`0.23` should not reopen that line casually. It should finish the remaining
measurable follow-through with a tighter scope.

## Carryover From 0.22

The following work was intentionally left out of `0.22` because it was useful
but not required to close that line:

1. Record the first dated post-cleanup wasm-footprint rerun.
2. Carry the new checked-in per-canister wasm budgets through the first dated rerun and adjust them only with explicit evidence.
3. Run one cold-machine validation pass for the tagged `scripts/install.sh` path.
4. Push more custom auth/root fixture setup into reusable `canic-testkit` helpers.
5. Expand comparable `perf!` coverage across root bootstrap, publication, auth, replay, and scaling flows.
6. Separate “first baseline build” cost from steady-state cached restore cost in heavy PocketIC suites.

## Goals

### 1. Turn Wasm Size Into A Visible Regression Surface

`0.22` reduced and stabilized the build/install path, and `0.23.0` now adds a
checked-in shrunk-wasm budget surface plus the first dated rerun against that
budgeted baseline.

`0.23` should:

- keep the new checked-in budgets and deltas visible in later reports
- decide whether the current `root` budget overrun should be reduced or accepted with an explicit budget move
- treat `root` and other control-plane canisters as purposeful outliers instead
  of the shared floor

### 2. Validate The New Setup Path On A Cleaner Machine

`scripts/install.sh` now bootstraps Rust when needed, installs the pinned
toolchain, adds the required helper tools, and installs `dfx` when it is
missing. That path should get one explicit cold-machine validation so the docs
are backed by a real operator run instead of only shell syntax and local
incremental checks.

`0.23` should:

- run one cold-machine or clean-shell validation pass
- record what still must remain environment-specific
- keep the shared script as the preferred setup path if that validation passes

### 3. Push More Testkit Reuse Downward

`0.22` cleaned up the root-baseline and reconcile-heavy test lanes, but some
auth/root fixture logic still lives in suite-local code instead of reusable
testkit helpers.

`0.23` should:

- move more custom auth/root setup into `canic-testkit`
- keep heavy PocketIC fixture reuse comparable and attributable
- make downstream reuse easier without copying suite-local harness code

### 4. Improve Comparable Perf Coverage

`0.22` opened the instruction/perf line and improved runtime/install
observability, but comparable `perf!` coverage is still incomplete across the
important multi-step flows.

`0.23` should:

- add comparable `perf!` checkpoints to root bootstrap, publication, auth,
  replay, and scaling flows
- keep scenario identity stable enough for reruns to be comparable
- distinguish cold setup cost from cached restore cost in PocketIC-heavy suites

## Non-Goals

`0.23` is not for:

- another broad topology redesign
- replacing `dfx` as the main local install/build boundary
- reopening already-resolved `0.22` installer/build churn unless a real
  regression appears

## Completion Boundary

`0.23.0` now covers the first wasm-size follow-through slice, but the rest of
the `0.23` line still remains open.

The `0.23` line should be considered materially complete once there is:

- one dated post-cleanup wasm-footprint rerun
- one explicit cold-machine validation note for the tagged setup path
- at least one meaningful auth/root fixture extraction into `canic-testkit`
- broader comparable `perf!` coverage across at least one additional heavy flow
