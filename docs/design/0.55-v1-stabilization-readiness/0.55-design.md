# 0.55 Design: V1 Stabilization And Readiness

## Status

- Status: Proposed
- Line: `0.55.x`
- Theme: v1 stabilization and readiness

0.55 is not a new feature line. It is the line where Canic proves the compact
v1 operator story is understandable, testable, and release-ready before adding
more product surface.

## Summary

0.41 through 0.54 built the pre-v1 operator arc:

- deployment truth and authority evidence;
- execution boundaries and receipts;
- artifact promotion and external lifecycle evidence;
- deployment-target identity;
- verified root recovery;
- metadata-driven setup;
- fleet-scoped role lifecycle;
- passive adoption reporting;
- stable evidence envelopes;
- build provenance;
- passive policy gates;
- passive deployment catalog.

0.55 should stabilize that arc instead of adding another major noun. The line
should make the current workflow easy to run and hard to misunderstand:

```text
create fleet -> declare/attach role -> build with provenance -> check
deployment evidence -> gate saved evidence -> inspect known deployments
```

The output of 0.55 should be confidence, not broader authority.

## Why This Line

Recent lines deliberately cut scope:

- 0.52 stopped at source/build/artifact provenance.
- 0.53 stopped at passive policy gates over saved evidence.
- 0.54 stopped at a passive catalog over local deployment-target state.

That restraint is correct for v1. The next useful question is not:

```text
What future platform layer should Canic model next?
```

It is:

```text
Can a real operator follow the compact v1 workflow, understand every boundary,
and get repeatable evidence without hidden mutation?
```

0.55 exists to answer that question.

## Core Rule

0.55 must not expand mutation authority.

It must not add:

- deployment groups;
- promotion lanes;
- saved-evidence catalogs;
- locks;
- signing or attestations;
- provider wrappers;
- artifact registry import;
- `wasm_store` retention or garbage collection;
- active adoption/import;
- controller mutation;
- topology mutation beyond existing role lifecycle commands;
- install or upgrade authority;
- teardown;
- broad live verification.

Those are post-v1 concerns unless a later design proves one narrow user journey
is worth expanding.

## V1 Surface To Prove

The maintained v1 story is:

```text
canic fleet create <fleet>
canic scaffold canister <fleet> <role>
canic fleet role attach <fleet> <role> --subnet <subnet>
canic build <fleet> <role> --provenance <path>
canic deploy check <deployment> --format envelope-json
canic evidence gate --policy <path> --manifest <path>
canic deploy catalog list
canic deploy catalog inspect <deployment>
```

This is not a single magic command. The separate commands are the point:

- fleet and role identity are explicit;
- topology attachment is explicit;
- build provenance is explicit;
- deployment target identity is explicit;
- policy is evaluated against saved evidence;
- catalog output reads known local deployment-target state only.

## Deliverables

0.55 should produce:

1. A v1 readiness checklist that names the exact commands, files, and expected
   outputs for a small managed fleet.
2. A small local smoke path or documented manual validation that exercises the
   compact workflow without requiring networked mainnet state.
3. CLI help and docs alignment for all v1 commands.
4. A stale-surface audit for old command shapes, deprecated wording, and
   pre-hard-cut examples.
5. A release-readiness audit that says whether Canic is ready to call this
   compact surface v1-candidate.

## Proposed Slices

### 0.55.0 - Stabilization Design And Checklist

Define the v1 readiness scope, non-goals, and acceptance checklist.

No source behavior change is required.

### 0.55.1 - V1 Docs And Help Alignment

Audit and align:

- root README;
- installation/getting-started docs;
- architecture index;
- v1 operator walkthrough;
- CLI help text for the v1 command set.

Remove or quarantine stale examples that imply unsupported shortcuts.

### 0.55.2 - Local Smoke Proof

Add the smallest practical local proof for the compact v1 flow.

The proof should prefer existing fixtures and existing local state. It should
not require mainnet, external deployment mutation, controller mutation, or
artifact registry state.

If full install is too heavy, document the exact remaining manual step rather
than pretending the smoke is complete.

### 0.55.3 - V1 Candidate Audit

Produce a closeout audit that answers:

- what a user can safely do today;
- what remains deliberately out of scope;
- which tests prove the compact workflow;
- whether documentation and CLI help agree;
- whether any release-blocking confusion remains.

## Acceptance Criteria

0.55 is complete when:

- the compact v1 command set is documented in one maintained place;
- setup docs explain `fleet`, `role`, and `deployment` without conflating them;
- stale command shapes are absent from active docs;
- policy/evidence/catalog docs describe passive boundaries consistently;
- a local smoke path exists or the missing step is explicit and justified;
- no new stable public DTO family is introduced without an implemented command;
- no new mutation authority is added;
- closeout audit verdict is PASS or PASS WITH FOLLOW-UPS with no blocker/high
  findings.

## Non-Goals

0.55 does not:

- create a one-command deployment pipeline;
- create deployment groups;
- define promotion lanes;
- sign evidence;
- acquire locks;
- import artifacts into `wasm_store`;
- import brownfield deployments;
- mutate controllers;
- perform teardown;
- make deployment catalog entries live or fresh;
- turn policy success into deployment truth;
- promise that every old design document is rewritten.

## Validation

Expected validation should include:

```text
cargo test -p canic-cli --lib --locked
cargo test -p canic-host deployment_catalog --lib --locked
cargo test -p canic --test changelog_governance --locked
cargo test -p canic --test workspace_manifest --locked
cargo fmt --all --check
git diff --check
```

Broader workspace validation should run before release if practical. If a
workspace-wide test is still too heavy or blocked by known local fixtures, the
0.55 closeout audit must record the exact reason and the strongest passing
targeted substitute.
