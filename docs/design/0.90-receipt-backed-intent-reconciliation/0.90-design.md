# Canic 0.90 Design: Receipt-Backed Intent Reconciliation

Date: 2026-07-13

## Status

- Status: `0.90.1` published the hard-cut Canic primitive. The current Canic
  slice adds a focused downstream-adapter conformance proof and handoff. Toko
  adoption follows a compatible published Canic release in the Toko
  repository.
- Release contract:
  [0.90 changelog](../../changelog/0.90.md).
- Implementation tracker: [status.md](status.md).
- 0.90 adds the smallest receipt-backed extension to the existing Canic
  intent system.
- Toko mint is the first planned downstream consumer.
- Multi-step claim orchestration is deferred to a separately accepted future
  design.

## Decision

Canic implements one minimal receipt-backed intent mechanism. Toko implements
mint as its first consumer.

This is deliberately neither a mint subsystem in Canic nor a universal
external-effect framework. The boundary is:

| Canic owns | Toko owns |
| --- | --- |
| `ReceiptBackedIntentRecord` | `MintEffectRequest` |
| `ReceiptBackedIntentState` | `MintReceipt` and `MintOutcome` |
| `OperationId` identity | mint-specific operation-ID derivation |
| opaque payload binding | mint payload construction and hashing |
| resource reservation | mint authorization and business rules |
| begin-or-load idempotency | ledger execution and receipt persistence |
| at-most-once local settlement | receipt validation and retry policy |
| bounded terminal fingerprint | mint cancellation fencing |
| no-TTL pending invariant | typed endpoint responses |

The governing rule is:

> Canic may move one bound reservation from pending to committed or rolled back
> exactly once, using evidence already validated by its consumer.

Canic does not decide whether a ledger response proves that a mint happened.
That decision belongs to the Toko mint workflow and ledger protocol.

The term _receipt-backed_ describes why the consumer is allowed to settle. It
does not mean Canic defines, stores, polls, or interprets a universal receipt.

## Guarantees

### Canic primitive

For one `OperationId` and payload binding, Canic guarantees:

- endpoint re-entry cannot reserve the quantity twice;
- the reservation remains durable while its outcome is uncertain;
- pending receipt-backed work never expires;
- checked resource counters settle at most once;
- matching terminal re-entry is idempotent; and
- contradictory terminal evidence cannot change counters.

### Toko mint integration

The Toko adapter and ledger protocol must additionally guarantee:

- at-most-once mint effect for one caller-scoped mint operation;
- durable applied or no-effect evidence;
- no rollback from a generic call failure; and
- eventual convergence when the ledger evidence is reachable.

Together these properties provide at-most-once external effect and at-most-once
local settlement. They do not provide exactly-once cross-canister execution or
a distributed transaction.

## Why This Is A Hard Cut

The current generic call-intent wrapper cannot safely decide an irreversible
operation. It commits after raw transport success before typed response/domain
decoding, aborts on generic call failure, captures time before the `await`, and
can return remote success after logging a local commit failure.

0.90 removes `CallBuilder::with_intent` completely. It is not renamed and no
compatibility alias preserves automatic reserve/commit/abort behavior.

Consumers use:

- direct local intent operations when the outcome is locally decidable; or
- the receipt-backed intent operations when consumer-validated evidence is
  required.

Existing replay facilities remain replay facilities. Canic reuses the existing
`OperationId` and may reuse the existing payload-hash helper, but it does not
turn replay storage into a generic ledger, resolver, fence, or effect engine.

Any Toko mint replay row that currently acts as a second local settlement
authority must be retired or reduced to non-authoritative request/outcome
cache. Only the Canic receipt-backed intent may change Canic reservation
counters.

## Scope

### Canic core

0.90 adds only:

- one receipt-backed intent record and direct `OperationId` lookup;
- one opaque, versioned payload binding;
- `begin_or_load_receipt_backed_intent`;
- one read-only `load_receipt_backed_intent` view;
- `settle_receipt_backed_intent_if_pending`;
- one bounded terminal evidence fingerprint;
- shared resource accounting with existing local intents;
- a no-TTL invariant for receipt-backed pending work;
- compact permanent terminal rows with a fixed compiled capacity limit; and
- removal of the generic call-builder intent surface.

### Downstream adoption target

The first downstream adoption target is one Toko mint operation:

- one typed mint request;
- one caller-scoped ledger receipt;
- one applied result and one durable no-effect result;
- one receipt validator that produces Canic terminal evidence;
- one immediate targeted observation after ambiguity;
- explicit same-ID retry or cancellation; and
- targeted owner/admin recovery.

That domain implementation is owned and tested in the Toko repository after a
compatible Canic release is published. The Canic repository proves the generic
adapter decisions and persistence boundary without introducing mint types.

### Deferred

0.90 does not add:

- multi-step claim payment/delivery orchestration, which is deferred to a
  separately accepted future design;
- refunds, reversals, or fulfillment policy;
- a generic execute or resolver result;
- `EffectOperationKey`, `EffectKind`, or an effect registry;
- a generic cancellation policy or receipt protocol;
- a transaction, saga, or two-phase commit framework;
- background reconciliation or global pending scans;
- time-based rollback of receipt-backed work; or
- abstractions intended only for hypothetical future consumers.

A second real consumer may justify extracting another shared shape later. 0.90
does not predict that shape.

## Minimal Canic Model

### Operation identity

0.90 reuses the existing 32-byte `OperationId`. Receipt-backed intents have one
global operation-ID namespace per canister.

There is no additional operation kind in the Canic key for this MVP. Toko
derives mint IDs with its own domain separator so its IDs are globally unique
inside that namespace.

Consequences:

- the same ID and same binding is replay;
- the same ID with another binding, resource, or quantity is
  `BindingConflict`;
- every Canic receipt-backed operation accepts the complete `OperationId`; and
- no Canic type models a ledger caller or effect kind.

If a second consumer later makes the single namespace insufficient, that
consumer must bring a concrete namespacing requirement. 0.90 does not add one
pre-emptively.

### Payload binding

Canic stores and compares an opaque fixed-size binding:

```text
PayloadBinding {
    schema_version,
    digest: [u8; 32],
}
```

Canic does not know which fields produced the digest. The consumer is
responsible for:

- authenticating its actor;
- including every field that changes the external effect;
- using one versioned canonical encoding;
- producing the same digest during retry and evidence validation; and
- rejecting unsupported binding versions before begin-or-load.

For Toko mint, Toko may reuse `ReplayPayloadHasher`. Its mint adapter, not the
Canic intent model, specifies the collection, recipient, asset, target ledger,
and field order.

### Terminal evidence

Canic stores only a bounded generic proof reference:

```text
TerminalEvidence {
    source_canister,
    schema_version,
    decision: Committed | RolledBack,
    fingerprint: [u8; 32],
}
```

The fingerprint is a canonical digest chosen and validated by the consumer.
Canic compares it for terminal idempotence but does not inspect the source
receipt or infer business meaning from it.

Domain outcomes remain domain-owned. Toko may keep a bounded `MintOutcome` or
receipt reference as non-authoritative response cache, but that cache cannot
reserve, commit, or release Canic counters.

### Stable record

```text
ReceiptBackedIntentRecord {
    schema_version,
    operation_id: OperationId,
    payload_binding: PayloadBinding,
    resource_key: IntentResourceKey,
    quantity,
    state: ReceiptBackedIntentState,
    revision,
    created_at_ns,
    updated_at_ns,
}

ReceiptBackedIntentState =
    Pending
  | Committed { evidence: TerminalEvidence }
  | RolledBack { evidence: TerminalEvidence }
```

There is intentionally no `Submitted` or `RecoveryRequired` variant in Canic.
Once begin-or-load creates `Pending`, Canic conservatively treats the operation
as potentially submitted. It may not be rolled back without consumer-validated
no-effect evidence.

This loses only an optimization: a crash before the external call may require
Toko to observe or fence an operation that was never submitted. It avoids a
second core state machine and removes a happy-path phase write.

Receipt-backed rows never carry a TTL.

## Begin Or Load

Every consumer entrance uses:

```text
begin_or_load_receipt_backed_intent(
    operation_id,
    payload_binding,
    resource_key,
    quantity,
    reservation_limit,
    now,
) -> BeginReceiptBackedIntentResult

BeginReceiptBackedIntentResult =
    Created { revision }
  | ExistingPending { revision }
  | ExistingCommitted { evidence }
  | ExistingRolledBack { evidence }
  | BindingConflict
  | CapacityExceeded
  | StoreCapacityReached
```

The operation first performs the direct `OperationId` lookup.

- `Created` checks admission, writes one pending record, and increments the
  shared resource aggregate once.
- `ExistingPending` performs no counter write.
- `ExistingCommitted` and `ExistingRolledBack` return the stored evidence
  without changing counters.
- `BindingConflict` performs no state or counter change.
- Capacity checks apply only when the ID is absent. Existing operations remain
  readable and settleable when admission is full.

The function contains no `await`. It precomputes all checked changes before
writing the record and aggregate.

Toko must call this before `execute_mint`. A retried Toko endpoint therefore
cannot create a second reservation even if its previous response was lost.

`load_receipt_backed_intent(operation_id)` is the corresponding exact,
read-only view for targeted recovery. It has no mutation, scan, or domain
interpretation.

## Settlement CAS

Every terminal path uses:

```text
settle_receipt_backed_intent_if_pending(
    operation_id,
    expected_revision,
    expected_payload_binding,
    evidence,
) -> SettleReceiptBackedIntentResult
```

The non-awaiting transition:

1. reloads the record by `OperationId`;
2. compares the expected revision and payload binding;
3. requires `Pending` before changing counters;
4. checks the evidence shape and decision;
5. precomputes checked record and aggregate mutations;
6. moves reserved quantity to committed quantity, or releases it;
7. writes the terminal evidence and increments the revision; and
8. returns without an intervening `await`.

If the record is already terminal:

- the same decision and fingerprint returns idempotently;
- contradictory decision or fingerprint is an invariant failure; and
- neither case changes counters.

This is the only Canic operation allowed to terminally settle a
receipt-backed reservation. The callback, endpoint retry, lazy recovery, and
admin recovery all converge here.

Canic verifies mechanical identity and state only. The caller must validate
the receipt before constructing `TerminalEvidence`.

## One Resource Authority

Local and receipt-backed intents compete for the same capacity through the
existing aggregate, whose stable shape remains unchanged:

```text
IntentResourceTotalsRecord {
    reserved_qty,
    committed_qty,
    pending_count,
}
```

`pending_count` covers both local and receipt-backed reservations. There are no
separate mint counters and no second receipt-backed capacity system.

| Transition | Reserved | Committed | Pending |
| --- | ---: | ---: | ---: |
| receipt-backed begin | `+quantity` | unchanged | `+1` |
| receipt-backed commit | `-quantity` | `+quantity` | `-1` |
| receipt-backed rollback | `-quantity` | unchanged | `-1` |
| local begin | `+quantity` | unchanged | `+1` |
| local terminal | normal delta | normal delta | `-1` |

The existing store metadata already maintains `pending_total` for entries in
the local TTL index. Receipt-backed records never enter that index and never
change that scalar, so they cannot start or keep the cleanup timer alive. No
new aggregate field or migration is required.

The existing local pending index remains local; receipt-backed records use
their direct operation map and never enter the TTL cleanup scan.

The persisted aggregate is authoritative. Expiry releases a local reservation
when its cleanup transition executes; until then the reservation is
conservatively held. Admission reads the exact aggregate and does not scan
local or receipt-backed rows.

## Toko Mint Adapter

### Domain-owned types and storage

Toko owns:

- `MintEffectRequest`;
- `MintReceipt`;
- `MintOutcome`;
- its caller-scoped mint receipt key;
- any bounded original-request or outcome cache needed for recovery/replay;
- mint authorization and deterministic rejection rules; and
- the receipt-store capacity policy.

Canic does not contain collection, recipient, token, asset, ledger-mint, or
mint-rejection fields.

Any Toko request/outcome cache is non-authoritative transport or response
state. The Canic receipt-backed intent is the sole authority for local
reservation counters. The Toko ledger receipt is the sole authority for
whether the external mint occurred.

### Ledger contract

The Toko ledger namespaces a mint receipt by:

- the actual authenticated instance-canister caller; and
- the Toko mint `OperationId`.

That caller is derived from IC message context, never accepted from request
data. One authorized instance cannot occupy another instance's receipt slot.

For a valid authorized typed request, `execute_mint` atomically produces one
of:

- an applied `MintReceipt` written with the mint effect; or
- a durable no-effect `MintReceipt` for deterministic business rejection.

Malformed, unauthorized, or unsupported requests create no attacker-chosen
tombstone. Receipt-capacity refusal occurs before the effect and leaves the
Canic reservation pending.

Toko owns these result variants. They are not promoted into Canic enums.

### Receipt validation

Before calling Canic settlement, the Toko workflow validates:

- actual ledger identity;
- caller-scoped operation ID;
- canonical mint payload binding;
- receipt schema and decision;
- effect reference or no-effect reason; and
- the terminal fingerprint.

Only then does Toko convert the result to `TerminalEvidence`.

Canic never answers, “does this receipt prove mint success?” It answers only,
“is this still the same pending reservation, and may it settle once?”

### Toko-specific observation and fence

Toko may expose domain methods such as:

- `get_mint_receipt(operation_id)`; and
- `cancel_mint_if_absent(original_request)`.

They are not Canic resolver APIs.

An absent-operation cancellation must carry the complete typed original mint
request. The ledger authenticates the actual caller and recomputes its mint
binding before inserting a no-effect receipt. A supplied digest alone is not
enough.

Only the instance canister authorized to execute that mint may fence its absent
slot. A generic system administrator cannot cancel arbitrary ledger
operations.

If execute wins first, the applied receipt wins. If cancellation wins first,
the no-effect receipt wins and a delayed execute cannot mint.

## Toko Flow

### First entrance

```text
1. authenticate and validate the Toko request
2. derive the mint OperationId and payload binding
3. begin_or_load_receipt_backed_intent(...)
4. Created -> call Toko ledger execute_mint(...)
5. validate the returned MintReceipt
6. convert it to TerminalEvidence
7. settle_receipt_backed_intent_if_pending(...)
```

A deterministic no-effect receipt follows the same path and settles
`RolledBack`. No successful or deterministic-negative path performs a second
ledger read.

### Re-entry

| Canic result | Toko behavior |
| --- | --- |
| `Created` | execute once |
| `ExistingPending` | use one bounded Toko recovery action |
| `ExistingCommitted` | return/reconstruct the stored Toko success |
| `ExistingRolledBack` | return/reconstruct the stored Toko no-effect result |
| `BindingConflict` | reject without a ledger call |

The endpoint is therefore idempotent before relying on ledger idempotency.

### Ambiguity

After any post-call transport, decode, callback, or local settlement failure,
the Canic record remains `Pending`.

Toko may:

1. perform at most one immediate targeted receipt observation;
2. settle if that observation yields valid applied or no-effect evidence;
3. otherwise leave the intent pending;
4. replay `execute_mint` once with the same ID when completion is requested; or
5. call `cancel_mint_if_absent` only for explicit cancellation.

No production adapter treats a reject code or error string as proof that the
callee did not receive the message.

### Lazy and admin recovery

- A relevant Toko update may observe at most one named pending mint.
- Lazy recovery never scans all operations and never fences an absent mint.
- Owner/admin recovery addresses one `OperationId`.
- Observation needs only the ID; cancellation additionally requires the typed
  original request whose binding matches the Canic record.
- Upgrade schedules no reconciliation timer.

## Permanent State And Capacity

Because an `OperationId` may be replayed indefinitely, 0.90 does not delete a
terminal receipt-backed intent.

Canic uses:

- one fixed-size record per operation;
- the stable operation map's maintained total record count;
- one fixed compiled maximum for new receipt-backed records; and
- existing resource admission for reserved quantity.

Replays append no row. When the record limit is reached, new IDs fail before
reservation while existing IDs remain loadable and settleable. A later release
may raise the compiled limit, but terminal rows are not deleted merely to
reclaim quota.

Toko independently owns public caller/economic admission and its ledger receipt
capacity. Canic does not add actor-specific or effect-specific quota models for
the MVP.

This is linear durable history, not a bounded total store. Safe reclamation
requires a future replay-horizon design.

## Performance Invariant

### External calls

- normal applied mint: the existing `execute_mint` call only;
- deterministic no-effect: the same call returns the no-effect receipt;
- successful-path receipt reads: zero;
- ambiguous response: at most one immediate targeted observation;
- idle system: zero recovery calls; and
- background recovery calls: zero.

### Canic work

- endpoint entry adds one exact `OperationId` lookup;
- creation writes one bounded record and one exact aggregate;
- settlement writes the same record and aggregate once;
- there is no `Submitted` phase write;
- local intents serialize no receipt-backed fields;
- receipt-backed work enters no TTL index or cleanup scan; and
- no normal operation scans intents, receipts, actors, resources, or callers.

There is no new timer, heartbeat, poller, fleet scan, high-cardinality metric,
generic receipt store, or generic resolver in Canic.

Small fixed hashing, one indexed lookup, and bounded stable writes are expected.
An added happy-path inter-canister call, background invocation, map-wide scan,
unbounded recovery loop, or unexplained material instruction/Wasm increase
blocks 0.90.

## Stable State And Migration

The hard cut uses one new stable type and approved memory allocation 43.
Existing intent allocations 39 through 42 and their bytes are unchanged.

The Canic state transition:

1. preserves supported local intent IDs, resources, quantities, states, totals,
   pending index, metadata, and encodings;
2. treats the maintained resource aggregate as admission authority rather than
   reconstructing logical totals on each read;
3. creates an empty receipt-backed operation map whose existing map header
   maintains the admission count;
4. keeps receipt-backed rows outside local TTL cleanup;
5. removes `CallBuilder::with_intent` and migrates maintained callers to direct
   local intent operations; and
6. adds no legacy reader, alias, fallback, or historical scan.

No historical scan or in-place record migration is required for Slice A.

The Toko integration separately hard-cuts any mint replay row that duplicates
local settlement authority. Toko may retain bounded request/outcome data only
as non-authoritative recovery or response cache.

## Layering

Canic follows `endpoints -> workflow -> policy -> ops -> model`:

- `model/` owns generic receipt-backed records, bindings, evidence, totals, and
  invariants;
- `ops/` owns direct lookup plus checked begin/load and settlement mutations;
- `policy/` owns pure generic admission decisions where needed;
- `workflow/` and endpoints do not interpret domain receipts; and
- the `canic` facade exposes only the minimal owner-facing intent operations.

Toko owns its DTOs, receipt conversion, authorization, policy, ledger calls,
retry/cancellation workflow, and endpoint response mapping.

Workflow does not construct or mutate Canic records. Toko converts validated
domain evidence into the named Canic input and delegates to ops without an
`await` inside settlement.

## Security

- Canic treats the payload binding as opaque and compares it exactly.
- Toko authenticates the actor before deriving that binding.
- Toko includes all effect-changing mint fields in its canonical digest.
- The ledger uses actual IC caller identity for receipt namespacing.
- Only the mint-authorized instance may execute or fence its receipt slot.
- A typed original request, not a caller-provided hash alone, is required to
  fence an absent mint.
- Canic settlement is an internal trusted operation reached only after Toko
  evidence validation.
- Metrics contain no operation IDs, principals, resource keys, payload
  bindings, or receipt bytes.

## Implementation Slices

### Slice A - Minimal Canic primitive

- Freeze focused performance baselines.
- Reuse `OperationId` and add generic payload/evidence types.
- Add `ReceiptBackedIntentRecord` and direct operation storage.
- Add begin-or-load and settlement CAS.
- Reuse the unchanged resource aggregate and existing local timer-idle
  metadata.
- Remove `CallBuilder::with_intent` and migrate maintained local consumers.
- Prove the new allocation and unchanged local intent state through
  focused state-contract and upgrade tests.

No mint request, receipt, resolver, effect kind, or ledger protocol enters
Canic core.

### Slice B - Canic adapter conformance and handoff

- Expose every begin and settlement decision through one test-only adapter.
- Prove changed bindings, capacity, stale revisions, and missing operations.
- Prove committed and rolled-back settlement, terminal replay, and
  contradictory-evidence rejection.
- Preserve a pending operation across a same-Wasm upgrade.
- Document the exact downstream flow and evidence-validation boundary.

No domain request, receipt, resolver, or recovery abstraction enters Canic.

### Downstream adoption - Toko mint

- Define Toko-owned mint ID derivation and payload binding.
- Add caller-scoped idempotent mint receipts and deterministic no-effect
  receipts.
- Add receipt validation and conversion to `TerminalEvidence`.
- Integrate endpoint begin/load, execution, settlement, and bounded recovery.
- Retire co-authoritative Toko mint replay state.
- Complete focused PocketIC, collision, race, capacity, and performance proof.

This work begins in the Toko repository only after the compatible Canic release
is published. It is a downstream adoption gate, not a Canic publication gate.

Multi-step claim orchestration is deferred to a separately accepted future
design. Generalized multi-step orchestration does not enter 0.90.

## Acceptance

The Canic 0.90 line is complete only when:

1. Canic core contains no mint, collection, recipient, token, or ledger-reject
   model;
2. Canic exposes only generic begin/load, read, and settlement operations;
3. `OperationId` is the complete Canic receipt-backed key and its namespace is
   documented;
4. re-entry never reserves the same operation twice;
5. changed binding, resource, or quantity fails without mutation;
6. `Pending` has no TTL and cannot be rolled back from a call error;
7. settlement reloads ID, binding, state, and revision without an `await`;
8. callback/recovery races change counters once;
9. matching terminal evidence is idempotent and contradictory evidence is an
    invariant failure;
10. local and receipt-backed reservations use the same aggregate authority;
11. receipt-backed work cannot start or retain local cleanup;
12. Canic does not define an effect key, resolver, execute result, fence, or
    receipt protocol;
13. no background recovery, global scan, or unbounded retry exists;
14. permanent rows are bounded per record and governed by maintained capacity;
15. `CallBuilder::with_intent` and compatibility aliases are absent;
16. the focused public-facade fixture proves explicit begin decisions,
    compare-and-set conflicts, both terminal decisions, replay, contradiction,
    released rollback capacity, and pending-state upgrade recovery; and
17. focused performance remains within the frozen reviewed budget.

Downstream Toko adoption is complete only when:

1. begin/load occurs before mint execution;
2. Toko validates a typed mint receipt before constructing terminal evidence;
3. ledger receipts are namespaced by actual authorized caller and operation
   ID;
4. deterministic rejection returns durable no-effect evidence in the original
   execute response;
5. fencing an absent mint requires the authenticated typed original request;
6. no post-call failure is classified as known-not-submitted from guesswork;
7. normal applied and deterministic-negative paths add no receipt-read call;
8. ambiguity performs at most one immediate targeted observation; and
9. no Toko replay row remains co-authoritative for local settlement.

## Validation Plan

### Canic unit tests

- begin/load created, pending, committed, rolled-back, capacity, and conflict
  outcomes;
- no second reservation on every replay entrance;
- exact shared aggregate deltas;
- receipt-backed rows never enter TTL cleanup;
- revision CAS, terminal idempotence, and contradiction;
- record-limit behavior without scans; and
- local intent migration and timer-idle accounting.

### Downstream Toko adoption tests

These tests are owned by the Toko repository after the compatible Canic
release is published:

- same ID and same request;
- same ID with changed actor or mint payload;
- same ID from different authorized instance callers;
- unauthorized execute or cancellation;
- applied and deterministic no-effect receipts;
- lost or malformed response followed by one observation;
- same-ID execute retry;
- cancellation winning before delayed execute;
- execute winning before cancellation;
- callback/recovery settlement race;
- receipt-capacity exhaustion preserving pending state; and
- upgrade with pending and terminal operations.

Cross-canister and upgrade tests use PocketIC.

### Performance tests

- normal mint remains one business call;
- deterministic no-effect remains one business call;
- ambiguous handling makes at most one immediate observation;
- addressed-key instrumentation proves no normal map-wide scan;
- receipt-backed pending work schedules no timer; and
- focused before/after instruction and Wasm-size evidence remains within the
  frozen budget.

Run only targeted package checks, warning-denied Clippy, formatting, stable
state ABI, changelog governance, and diff hygiene for each implementation
slice. The maintainer owns broad workspace, deployment, and publish
validation.
