# 0.51 Design: CI/GitOps Provenance And Stable Evidence Envelopes

## Status

- Status: Active
- Line: `0.51.x`
- Theme: CI/GitOps provenance and stable evidence envelopes

This is the active 0.51 line design. The first adoption-report and
deployment-check envelope emitters have landed; later sections still describe
the remaining line direction.

## Summary

0.51 should turn Canic's existing passive evidence artifacts into stable,
automation-friendly outputs without expanding mutation authority.

The line defines a stable evidence envelope, a small exit-class taxonomy,
input fingerprints, command provenance, payload schema identity, and
CI-friendly report semantics. The envelope gives CI, GitOps, and audit systems
one stable outer contract while allowing individual payload DTOs to remain
command-specific unless Canic explicitly promotes them to a stable public
schema.

The core idea:

```text
stable envelope, command-specific payload
```

0.51 is not an execution, import, controller, registry, teardown, or live
verification line. It makes existing evidence easier to consume safely.

## Motivation

0.41-0.50 built a coherent evidence architecture:

- 0.41 introduced deployment truth: plans, inventories, diffs, safety reports,
  receipts, and install gates.
- 0.42 introduced dry-run authority evidence for controller and authority
  drift.
- 0.43 clarified execution boundaries and receipt-shaped operation evidence.
- 0.44 introduced artifact promotion, promotion readiness, provenance, and
  promotion execution receipts.
- 0.45 introduced external lifecycle evidence, proposals, consent evidence,
  verification checks, completion reports, and handoff artifacts.
- 0.46 introduced deployment-target identity and passive comparison of
  archived deployment checks.
- 0.47 introduced verified registered-root recovery through explicit
  root-verification evidence and local-state receipts.
- 0.48 simplified setup so package metadata drives `build!` and `start!`
  identity.
- 0.49 stabilized fleet-scoped role lifecycle: declared-only roles, attached
  roles, and attached-role strict build/deploy/install/truth surfaces.
- 0.50 introduced passive adoption reports with optional evidence inputs,
  authority-gated recommendations, `missing_or_stale_evidence`, and
  `evidence-conflict`.

The result is that Canic has many useful JSON artifacts, but automation still
has no stable outer contract for:

- which command produced an artifact;
- which deployment, fleet, profile, config, or evidence inputs were used;
- whether the command succeeded with warnings, blocked on policy, or failed on
  invalid input;
- which payload schema was emitted;
- which warnings, blockers, missing evidence, and conflicts are safe for CI to
  branch on;
- how to compare reports without binding directly to unstable internal DTO
  details.

0.51 should make Canic's passive evidence suitable for CI and GitOps without
letting automation bypass live-state validation or authority checks.

## Core Rule

0.51 must not make passive commands active.

Stable envelopes must not:

- mutate deployments;
- install or upgrade Wasm;
- attach topology;
- import adoption candidates;
- change controllers;
- register artifacts into a registry;
- delete or garbage-collect artifacts;
- teardown resources;
- imply that stale evidence is fresh;
- turn signed or enveloped output into deployment truth.

An envelope describes how a command produced evidence. It does not make the
evidence authoritative beyond the command and payload semantics.

## Relationship To Existing Output

Canic currently emits raw command payloads for many report surfaces. 0.51
should add stable-envelope output incrementally rather than replacing every
payload at once.

Rules:

- The envelope schema is stable within its declared version.
- The payload is command-specific.
- A payload schema identity must say what payload is inside.
- A command can support both legacy/raw JSON and enveloped JSON during the line
  only if the CLI surface makes the choice explicit.
- Enveloped output must not silently change default process exit behavior.
- Text output remains presentation-oriented and is not the automation
  contract.

## Proposed First Slice

0.51.0 should define the passive model and wire one passive command emitter:

```text
EvidenceEnvelopeV1
ExitClassV1
EvidenceTargetV1
PayloadSchemaRefV1
InputFingerprintV1
CommandProvenanceV1
EvidenceSummaryV1
EvidenceMessageV1
```

Chosen first emitter:

```text
canic fleet adoption report <fleet> --profile <profile> --format envelope-json
```

0.51.0 should not be model-only. A stable envelope is hard to validate without
one real command path, and adoption reports already have the right passive
shape: supplied evidence inputs, warnings, blocked actions, missing/stale
evidence, evidence conflicts, and experimental raw JSON output.

The first slice should also document how commands opt into envelope output and
add focused tests around serialization, exit-class mapping, raw JSON
preservation, input fingerprinting, and no-mutation boundaries.

## Resolved Initial Decisions

0.51 starts with these interface decisions:

1. Envelope output is requested with `--format envelope-json`.
   Existing `--format json` remains the raw command payload.
2. `success_with_warnings` maps to process success by default.
   CI policy may fail warnings using project-specific policy or a later
   `--fail-on-warnings` style option.
3. Paths in envelopes should be relative to the selected project or config root
   wherever practical. Absolute paths should be avoided, redacted, or omitted
   unless required for local diagnostics.
4. `AdoptionReportV1` remains `experimental` when wrapped.
   `DeploymentCheckV1` remains `internal` until separately promoted.
5. The stable contract is the envelope, not the nested payload.
6. 0.51.0 emits envelopes for one passive command first:
   `canic fleet adoption report <fleet> --profile <profile> --format
   envelope-json`.

### `EvidenceEnvelopeV1`

Tentative shape:

```text
EvidenceEnvelopeV1 {
  envelope_schema: PayloadSchemaRefV1,
  canic_version: string,
  command: CommandProvenanceV1,
  target: EvidenceTargetV1,
  generated_at: string,
  source_config: InputFingerprintV1?,
  inputs: Vec<InputFingerprintV1>,
  payload_schema: PayloadSchemaRefV1,
  payload_sha256: string?,
  payload: serde_json::Value,
  summary: EvidenceSummaryV1,
  exit_class: ExitClassV1,
}
```

The envelope should avoid claiming more stability than Canic can honor. For
example, `payload` may contain `DeploymentCheckV1` or `AdoptionReportV1`, but
the envelope stability does not automatically freeze every nested payload field
unless the payload schema is separately documented as stable.

### `EvidenceSummaryV1`

Tentative shape:

```text
EvidenceSummaryV1 {
  warnings: Vec<EvidenceMessageV1>,
  blocked_actions: Vec<EvidenceMessageV1>,
  missing_or_stale_evidence: Vec<EvidenceMessageV1>,
  evidence_conflicts: Vec<EvidenceMessageV1>,
}
```

These fields are structured because 0.51 is an automation contract, not just a
human report wrapper. CI should branch on stable codes and severity, not parse
free-form strings.

### `EvidenceMessageV1`

Tentative shape:

```text
EvidenceMessageV1 {
  code: string,
  message: string,
  severity: "info" | "warning" | "error",
  source: string?,
  related_input: string?,
}
```

Example codes:

```text
adoption.evidence_conflict.artifact_source
adoption.missing_or_stale.inventory_observation
deploy.blocked.safety_report
```

The `message` field remains human-readable. The `code` field is the stable
automation key.

### `CommandProvenanceV1`

Tentative shape:

```text
CommandProvenanceV1 {
  name: string,
  argv_normalized: Vec<String>,
  argv_redactions: Vec<String>,
  format: string,
}
```

The normalized command should be stable enough for audits without preserving
shell-specific quoting, environment noise, or absolute paths when those would
leak local machine details.

Redaction rules:

- redact tokens, credentials, future signing keys, and URLs with embedded
  credentials;
- prefer relative paths for project-local inputs;
- redact or omit paths outside the project/config root unless required for a
  local diagnostic;
- record a redaction reason in `argv_redactions` when an argument is replaced
  with `<redacted>`.

### `EvidenceTargetV1`

Tentative shape:

```text
EvidenceTargetV1 {
  kind: "deployment" | "fleet" | "fleet_adoption" | "artifact" | "unknown",
  deployment: string?,
  fleet: string?,
  role: string?,
  profile: string?,
  network: string?,
}
```

This target identity is for automation routing and audit. It does not replace
the authoritative identities inside deployment-truth payloads.

### `PayloadSchemaRefV1`

Tentative shape:

```text
PayloadSchemaRefV1 {
  id: string,
  version: string,
  stability: "stable" | "experimental" | "internal",
}
```

Examples:

```text
{ id = "canic.evidence_envelope.v1", version = "1", stability = "stable" }
{ id = "canic.deployment_check.v1", version = "1", stability = "internal" }
{ id = "canic.adoption_report.v1", version = "1", stability = "experimental" }
```

0.51 should be explicit when a payload is still experimental. In particular,
0.50 kept adoption report JSON experimental throughout `0.50.x`; 0.51 should
not accidentally promote `AdoptionReportV1` by wrapping it.

### `InputFingerprintV1`

Tentative shape:

```text
InputFingerprintV1 {
  kind: string,
  path: string?,
  path_display: "relative" | "absolute_redacted" | "omitted",
  sha256: string?,
  size_bytes: u64?,
  modified_unix_secs: u64?,
  schema: PayloadSchemaRefV1?,
  note: string?,
}
```

Input fingerprints should cover:

- config files;
- supplied deployment-check artifacts;
- supplied inventory artifacts;
- supplied artifact manifests;
- supplied package metadata evidence;
- supplied Cargo metadata evidence;
- build or artifact files when a command consumes them.

Timestamps alone are not provenance. A timestamp may help explain local
filesystem state, but CI should branch on content hashes and schema identity
where available.

`path_display` records whether `path` is safe to expose. Paths under the
selected project or config root render as relative paths. Absolute paths
outside that root are omitted and marked `absolute_redacted` rather than being
copied into CI artifacts.

## Generated Timestamp Determinism

`generated_at` is useful for audit, but noisy for CI snapshot comparison.

0.51 uses this deterministic test/comparison policy:

- tests fix `generated_at` through an internal deterministic clock or
  fixture-only override;
- future envelope comparison tooling must ignore `generated_at` by default
  unless a
  strict-time mode is explicitly requested;
- stable documentation tells users that timestamp equality is not a
  provenance guarantee.

## Candidate Commands For `0.51.x`

The first command consumers should be passive and already evidence-oriented.

Recommended line sequence:

```text
0.51.0: canic fleet adoption report <fleet> --profile <profile>
0.51.0: canic deploy check <deployment>
```

Why adoption first:

- `canic fleet adoption report <fleet> --profile <profile>` is the newest
  passive evidence surface. It already has explicit supplied evidence inputs,
  missing/stale evidence, evidence conflicts, blocked actions, and an
  experimental JSON payload.
- It lets Canic validate the envelope contract without also stabilizing
  deployment-check semantics in the first patch.

Deployment check should follow once the envelope model has survived one real
output path:

- `canic deploy check <deployment>` is the central deployment-truth report
  surface.
- It already produces `DeploymentCheckV1` evidence and does not need a new
  mutation model.
- `DeploymentCheckV1` should remain `internal` until a separate
  schema-stability review promotes it.

The current repo contains:

```text
canic deploy check demo
canic fleet adoption report demo --profile brownfield
```

Output flag:

```text
--format envelope-json
```

Rejected default for 0.51.0:

```text
make existing --format json silently become enveloped JSON
```

That would break scripts expecting current raw payloads. 0.51 should prefer an
explicit opt-in shape for stable-envelope output.

## Exit Classes

`ExitClassV1` is a stable semantic classification for automation. It is not
identical to the operating-system process exit code.

Initial taxonomy:

```text
success
success_with_warnings
blocked_by_policy
evidence_conflict
missing_required_evidence
invalid_input
execution_failed
internal_error
```

Meanings:

- `success`: command completed and the payload has no warnings, blockers,
  required evidence gaps, or conflicts.
- `success_with_warnings`: command completed and warnings are present, but no
  hard policy blocker or invalid input occurred.
- `blocked_by_policy`: command completed and produced evidence, but safety or
  policy rules block the requested operation or recommendation.
- `evidence_conflict`: supplied or observed evidence contradicts itself.
- `missing_required_evidence`: required evidence for the command or profile was
  absent or stale.
- `invalid_input`: command arguments, paths, JSON, schema, or target identity
  were invalid.
- `execution_failed`: an explicitly active command attempted execution and
  failed. Passive commands should rarely produce this class.
- `internal_error`: unexpected Canic failure.

Process exit-code guidance:

- `success` maps to process success.
- `success_with_warnings` maps to process success so CI can choose whether
  warnings fail policy.
- `blocked_by_policy`, `evidence_conflict`, `missing_required_evidence`,
  `invalid_input`, `execution_failed`, and `internal_error` map to process
  failure unless a command explicitly documents a report-only success mode.
- The envelope should carry `exit_class` even when the process exits non-zero,
  where practical.

Exit-class precedence:

```text
internal_error
execution_failed
invalid_input
evidence_conflict
missing_required_evidence
blocked_by_policy
success_with_warnings
success
```

When multiple conditions exist, the highest-precedence class wins. For
adoption reports, `missing_or_stale_evidence` is not automatically
`missing_required_evidence`: 0.50 intentionally allows partial supplied
evidence. Use `missing_required_evidence` only when the command cannot produce
meaningful output without that evidence.

0.51 should avoid a large taxonomy. More classes can be added in a future
schema version if real automation needs them.

## Input Fingerprints

Input fingerprints let CI answer:

```text
Which exact inputs produced this report?
```

For file inputs, record:

- `kind`;
- normalized path where safe;
- SHA-256 content hash;
- byte size;
- modification timestamp as explanatory metadata;
- schema identity when known.

For non-file inputs, record:

- `kind`;
- stable value digest if available;
- explicit note if no digest is available.

Path handling rules:

- Prefer paths relative to the selected project/config root when practical.
- Avoid leaking unnecessary absolute local paths in stable reports.
- Preserve enough path information for local audit and CI artifacts to be
  understandable.
- Do not emit absolute home-directory or runner workspace paths in snapshotable
  envelope output unless a test or command explicitly opts into local
  diagnostics.
- Apply the same path policy to config roots, working directories, and command
  arguments.

Hashing rules:

- SHA-256 is the default content hash.
- Hashes must be over canonical file bytes, not parsed JSON unless a command
  explicitly documents canonical JSON digest semantics.
- Missing files should become invalid input or missing evidence depending on
  whether the command requires the input.
- The emitted payload should also have a `payload_sha256` computed over the
  serialized payload bytes used in the envelope.

## CI/GitOps Use Cases

0.51 should support these workflows:

- Compare generated envelopes in CI without depending on unstable internal DTO
  layout.
- Fail builds on `evidence_conflict`.
- Warn or fail policy on `missing_or_stale_evidence`.
- Store report artifacts in GitOps pipelines with command, target, and input
  provenance.
- Audit which `DeploymentCheckV1`, inventory, artifact manifest, package
  metadata, or Cargo metadata evidence produced a report.
- Gate promotion or deploy workflows on an envelope's `exit_class` and payload
  schema.
- Preserve raw payloads for deeper inspection while giving automation a stable
  outer contract.

Example CI policy:

```text
if envelope.exit_class == "evidence_conflict":
    fail
if envelope.exit_class == "missing_required_evidence":
    fail
if envelope.exit_class == "success_with_warnings":
    warn_or_fail_by_project_policy
```

## Command Semantics

### Deployment Check Envelope

`canic deploy check <deployment>` envelope output should:

- target `kind = "deployment"`;
- include deployment target and network where known;
- fingerprint the selected config and supplied plan/evidence inputs;
- identify payload schema as `DeploymentCheckV1`;
- carry warnings and hard blockers from the safety report into envelope summary
  fields;
- map safety status to `ExitClassV1`.

The envelope must not make `deploy check` mutate root verification state,
controllers, deployment state, or artifacts.

### Adoption Report Envelope

`canic fleet adoption report <fleet> --profile <profile>` envelope output
should:

- target `kind = "fleet_adoption"`;
- include fleet and profile;
- fingerprint the selected config and all supplied evidence inputs;
- identify payload schema as `AdoptionReportV1` with experimental payload
  stability unless that payload is separately promoted;
- copy report warnings, blocked actions, missing/stale evidence, and evidence
  conflicts into envelope summary fields;
- map adoption report summary to `ExitClassV1`.

The envelope must preserve 0.50's report-only behavior.

## Stable vs Experimental Contract

Stable in 0.51:

- envelope field names and meanings;
- exit class strings;
- payload schema reference shape;
- input fingerprint shape;
- command provenance shape;
- target identity shape.

Not automatically stable:

- every nested field of `DeploymentCheckV1`;
- every nested field of `AdoptionReportV1`;
- text output;
- ordering of non-semantic report sections;
- paths that are inherently local unless normalized by rule.

If a payload becomes stable, it should be documented separately with its own
schema stability statement.

## Safety Rules

### No Mutation Expansion

Adding envelope output to a command must not add new mutation capability.

Tests should guard that envelope-producing paths for passive commands do not
call install, upgrade, controller, topology-attach, artifact-import, teardown,
or deployment-state mutation helpers.

Envelope generation must also not perform live discovery unless the underlying
command already performs that discovery in the same mode without envelope
output.

### No Freshness Laundering

The envelope must preserve whether evidence is fresh, stale, missing, supplied,
or observed. Wrapping a payload must not reset or hide stale evidence status.

### No Truth Substitution

The envelope is not deployment truth. It records command provenance and report
summary. The payload and its validation rules remain the evidence authority.

### No Signing By Default

Unsigned envelopes are useful for CI artifacts and local audit. They are not
attestations. Signing may be a later 0.51 slice, but 0.51.0 must not require
key management.

## Non-Goals

0.51.0 does not implement:

- signing;
- external key management;
- attestation verification;
- artifact registry import;
- `wasm_store` retention or GC;
- active adoption/import;
- controller mutation;
- topology mutation;
- teardown;
- broad live deployment verification;
- GitHub Actions-specific integration;
- mandatory CI usage;
- stable schemas for all existing payload DTOs;
- project manifest or CI lock semantics.

Later 0.51 slices may consider signing, project manifests, or CI locks, but
they should not be prerequisites for the first envelope slice.

## Patch Plan

Patch plan:

- `0.51.0` implemented: `EvidenceEnvelopeV1` foundation, adoption report
  envelope output for `canic fleet adoption report <fleet> --profile <profile>
  --format envelope-json`, deployment-check envelope output for
  `canic deploy check <deployment> --format envelope-json`, and release
  validation fixture hardening for the hard-cut role lifecycle.
- `0.51.1` implemented: input fingerprint hardening with shared file
  fingerprint helpers, payload hash helpers, relative path normalization, and
  absolute path redaction for evidence outside the selected root.
- `0.51.2` implemented: exit-class precedence is shared by envelope emitters,
  deployment-check conflicts are classified as `evidence_conflict` ahead of
  generic policy blocking, and CI policy docs describe warning/blocker/conflict
  handling.
- `0.51.3` implemented: `canic evidence compare --left <path> --right <path>`
  compares the stable envelope contract for CI without comparing the nested
  payload body.
- `0.51.4`: docs, examples, pipeline guidance, and closeout audit.

This plan is intentionally tentative. It should shrink rather than grow if the
first implementation slice finds that a smaller envelope is enough.

## Required Tests

Minimum tests before closing 0.51.0:

- `EvidenceEnvelopeV1` JSON shape is stable and round-trips.
- `ExitClassV1` serializes to documented strings.
- Envelope payload schema reference distinguishes stable, experimental, and
  internal payloads.
- `--format json` remains raw adoption report payload.
- `--format envelope-json` emits `EvidenceEnvelopeV1`.
- Envelope target is `fleet_adoption` and includes fleet and profile.
- Command provenance identifies the adoption report command and output format.
- Payload schema is `canic.adoption_report.v1` with `experimental` stability.
- Warnings, blocked actions, missing/stale evidence, and evidence conflicts are
  copied into envelope summary fields when present.
- Input fingerprints include kind, path, hash, size, and schema where known.
- Payload hash is present or explicitly stubbed with documented rationale.
- Invalid input maps to `invalid_input`.
- Evidence conflict maps to `evidence_conflict`.
- Missing required evidence maps to `missing_required_evidence`.
- `success_with_warnings` maps to process success.
- Snapshotable envelope output does not include absolute home-directory paths
  unless explicitly allowed by the fixture.
- Unsupported commands or formats using `--format envelope-json` error clearly.
- Passive command envelope paths do not mutate config, topology, controllers,
  artifacts, deployment state, or canister lifecycle.

## Documentation Requirements

0.51 docs should explain:

- why the envelope exists;
- what is stable;
- what remains payload-specific or experimental;
- how to consume `exit_class`;
- how to treat warnings, blockers, missing evidence, and conflicts;
- why signing is not part of the first slice;
- why envelope output does not replace live inventory or deployment truth.

The docs should include one deployment-check example and one adoption-report
example once implementation lands.

Initial examples should show the raw-vs-envelope distinction:

```text
canic fleet adoption report demo --profile brownfield --format json
canic fleet adoption report demo --profile brownfield --format envelope-json
```

Docs should warn:

```text
Do not write CI policy against payload fields unless that payload schema is
marked stable. The stable contract in 0.51 is the envelope.
```

## Acceptance Criteria

0.51.0 is acceptable when:

- an envelope model exists;
- exit classes are defined;
- `canic fleet adoption report <fleet> --profile <profile> --format
  envelope-json` can emit an envelope;
- existing adoption `--format json` remains raw payload output;
- envelope output is tested;
- input fingerprints are represented for supplied file evidence or explicitly
  stubbed with documented rationale;
- payload hashing is represented or explicitly stubbed with documented
  rationale;
- payload schema identity is represented;
- the envelope does not change command mutation behavior;
- docs explain what is stable and what remains payload-specific.

## Closeout Wording

Preferred closeout wording:

```text
0.51 adds a stable evidence envelope around selected passive Canic report
outputs. The envelope records command provenance, target identity, payload
schema, input fingerprints, warnings, blockers, missing/stale evidence,
evidence conflicts, and an automation-friendly exit class.

The envelope is stable; nested payload DTOs remain command-specific unless
explicitly promoted. Envelope output does not mutate deployments, controllers,
topology, artifacts, adoption state, or canister lifecycle, and it does not
make stale evidence fresh.
```
