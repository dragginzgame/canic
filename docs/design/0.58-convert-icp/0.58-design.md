# 0.58.0 - ICP-to-Cycles Refill Primitive

## Goal

Add the missing Canic primitive for canister-side ICP to cycles refill. The
core operation is a Canic-managed canister transferring ICP to the CMC top-up
account and then calling `notify_top_up`, so the existing funding chain can
replenish a low funding hub from ICP when it runs dry.

A CLI wrapper may trigger the same primitive for operator convenience, but
0.58.0 is not primarily a new CLI feature and it does not add a second
native identity-funded transfer engine.

Canic already owns the downstream funding chain:

- config-driven auto-topup via `cfg.topup`
- parent `request_cycles`
- replay protection
- grant ledger
- kill switch
- metrics

0.58.0 extends that chain with the missing source refill when the funding
hub itself runs low. This is the runtime primitive; any CLI entrypoint is
secondary.

## Why This Layer

- Canic already owns the funding policy and downstream request path.
- toko already hand-rolled the canister-side CMC / ledger flow and separately
  reimplemented `request_cycles`.
- A shared primitive should let downstream canisters delete bespoke code.
- Native identity-funded conversion would duplicate `icp` behavior and split
  the `cycles` family into passthrough and native halves.

## Existing Canic Surface

The current `cycles` CLI family is still a thin shell over the external
`icp` binary:

```text
canic cycles balance
canic cycles mint
canic cycles transfer
canic cycles topup
```

Do not add a parallel `canic icp convert` command.

Do not overload `canic cycles topup --icp`.

Do not make identity-funded ledger plus notify the primary shape for this
slice.

Any CLI entrypoint that survives should be a thin caller to the canister-side
primitive, not a second funding engine.

## What Is In Scope

- Canister-side ICP to cycles conversion and top up.
- Configurable CMC and ICP ledger IDs for local and custom networks.
- Explicit local-only fabrication for development.
- Integration with the existing funding chain so a funding hub can refill
  itself from ICP when low.
- Minimal durable records and bounded metrics.

## Canonical Flow

1. Resolve source canister and target canister explicitly.
2. Source canister transfers ICP to the CMC top-up account.
3. Capture the ICP ledger `block_index`.
4. Call CMC `notify_top_up`.
5. Persist a compact conversion record.
6. Surface retry and metrics through the existing funding chain.

Supported source modes:

- `canister`: Canic-managed canister-held ICP.
- `fabricated`: local-development-only cycle fabrication.

The caller identity is auth, not the source of funds. Identity-funded
conversion is out of this slice.

Targets remain explicit:

```text
<principal>
<deployment>/root
<deployment>/<role-or-canister>
```

Do not accept bare `api` as an installed deployment target.

## CMC and Ledger Details

- `ICP_LEDGER_CANISTER_ID` and `CMC_CANISTER_ID` need config-overridable
  resolution.
- Local and custom networks must be able to override those IDs without code
  changes.
- On `network=ic`, default to the canonical ICP ledger and CMC IDs and reject
  overrides unless an explicit unsafe/test flag is set.
- Query `icrc1_fee` and `icrc1_decimals` at runtime rather than hardcoding
  them. The fee used in the transfer must be explicit and persisted.
- CMC top-up subaccount is derived from the target canister principal.
- The source account is `{ owner = source_canister_id, subaccount =
  from_subaccount }`.
- The destination account is `{ owner = cmc_canister_id, subaccount =
  cmc_subaccount_for(target_canister) }`.
- `cmc_subaccount_for` uses byte 0 = principal length, bytes 1..N = principal
  bytes, and the remainder zero.
- Use ICRC-1 transfer with memo `TPUP\0\0\0\0`.
- Use direct `notify_top_up`; do not use the legacy ledger `notify()` flow.
- The cycles ledger is not the destination of this flow.
- The authoritative success result is the `Ok` value from `notify_top_up`.
- The same runtime primitive owns transfer, recovery, and notify for this
  slice; Canic does not depend on an external notifier here.

```candid
notify_top_up : (record {
  block_index : nat64;
  canister_id : principal;
}) -> (variant {
  Ok : nat;
  Err : NotifyError;
});
```

## Canister Endpoint

- Add an opt-in canister-side endpoint for Canic-managed canisters.
- It must be behind a macro or bundle feature, not silently included in every
  runtime.
- Auth must be composable: the shared primitive should accept a guard supplied
  by the host canister, so downstream canisters can keep their own session
  capability, controller, or role policy.
- The endpoint is the reusable primitive that toko and similar canisters can
  adopt.
- The source canister performs both transfer and notify so the spend,
  recovery state, authorization, and metrics stay in one runtime primitive.

## Existing Funding Chain Integration

- Do not create a parallel funding policy or a separate refill policy.
- Extend the existing `cfg.topup` and `request_cycles` chain so a funding hub
  can refill itself from ICP when its own balance is low.
- Reuse the existing funding ledger and `cycles_topup` / `cycles_funding`
  metrics families.
- Do not build a second treasury or a separate conversion island.
- Reuse the existing kill switch, cooldown, grant ledger, and target/source
  allowlist hooks; the refill primitive must consume those policy hooks rather
  than bypass them.
- At most one active refill operation may exist for a
  `(source_canister, source_subaccount, target_canister)` tuple unless the
  caller provides an explicit operation ID and policy allows parallel refills.
- Optional refill-specific limits may include `max_refill_e8s_per_call`,
  `max_refill_e8s_per_day`, `allowed_target_set`,
  `allowed_source_subaccounts`, `min_hub_cycles_before_refill`,
  `post_refill_target_cycles`, `cooldown_seconds`, and `kill_switch`.

## Local Modes

- `nns-cmc`: real local flow when the local NNS / CMC is available.
- `fabricated`: explicit local-only shortcut using `provisional_top_up_canister`
  from an operator-side local helper.
- If real local conversion is requested and NNS / CMC is unavailable, fail
  clearly.
- Fabrication is never a substitute for real ICP conversion.
- Fabricated mode does not exercise the canister-side ICP refill endpoint and
  does not count as a production refill.
- If exposed through the CLI, fabricated mode calls `provisional_top_up_canister`
  directly and does not route through the canister-side refill endpoint.

## Dry Run

- Query `get_icp_xdr_conversion_rate` for estimate and rate reporting.
- Report the current rate and estimated cycles.
- No transfer and no notify happen in dry run.
- Estimated cycles never replace the `notify_top_up` result.

## Rate Gate

- `cfg.topup.icp_refill.min_xdr_permyriad_per_icp` is optional.
- If unset, dry-run is reporting only and live mutation does not block on
  rate.
- If set, the same comparison runs in dry-run and live mutation.
- The primitive proceeds only when the current `xdr_permyriad_per_icp` meets
  or exceeds the configured minimum.

## Retry

- Retry belongs to the endpoint and workflow, not to a dedicated CLI
  subcommand in this slice.
- If a record has `ledger_block_index`, retry replays only `notify_top_up`.
- If no `ledger_block_index` exists and the record is `Requested`, retry must
  resubmit the exact same `TransferArg`, including `created_at_time`.
- If no `ledger_block_index` exists and the record is not `Requested`, do not
  silently re-transfer; require explicit operator confirmation and
  source-aware recovery.
- Canister-funded re-transfer must call the source canister endpoint.
- Preserve payloads for `Refunded` and `TransactionTooOld`.

## Idempotency and Recovery

Before awaiting `ledger.icrc1_transfer`, persist a `Requested` record
containing the exact `TransferArg` that will be submitted:

- `operation_id`
- `source_canister`
- `source_subaccount`
- `target_canister`
- `ledger_canister_id`
- `cmc_canister_id`
- `amount`
- `fee`
- `memo`
- `cmc_to_account`
- `created_at_time`
- `status = Requested`

Production transfers must set `created_at_time`.

If retry happens before `ledger_block_index` is known, re-submit the exact same
`TransferArg`. If the ledger returns `Duplicate { duplicate_of }`, store
`duplicate_of` as `ledger_block_index` and continue with `notify_top_up`. Never
construct a fresh `TransferArg` for recovery unless the operator explicitly
confirms a new spend.

The ICRC-1 transfer result is `Nat`. Before calling `notify_top_up`, convert
`Ok(Nat)` and `Duplicate { duplicate_of : Nat }` to `u64` with a checked
conversion. If conversion fails, store `InvalidLedgerBlockIndex` and halt
without retrying transfer.

The workflow must query `icrc1_fee` at runtime, pass it explicitly in
`TransferArg.fee`, and persist the fee used. If `BadFee { expected_fee }` is
returned, record `expected_fee` and allow a policy-controlled retry with the
updated fee only before any `ledger_block_index` exists.

The workflow must also query `icrc1_decimals` at runtime for validation and
operator display; it must not hardcode ledger display assumptions.

## Record Schema

Use a compact record centered on the transfer identity and block index:

```text
struct IcpRefillRecord {
    id: u64,
    operation_id: [u8; 32],

    source_canister: Principal,
    source_subaccount: Option<[u8; 32]>,
    target_canister: Principal,

    ledger_canister_id: Principal,
    cmc_canister_id: Principal,
    cmc_to_account_owner: Principal,
    cmc_to_account_subaccount: Option<[u8; 32]>,

    amount_e8s: u64,
    fee_e8s: u64,
    memo: Vec<u8>,
    created_at_time_ns: u64,

    ledger_block_index: Option<u64>,
    notify_attempts: u32,
    cycles_sent: Option<Nat>,

    status: IcpRefillStatus,
    error_code: Option<IcpRefillErrorCode>,
    error_message: Option<String>,

    refund_block_index: Option<u64>,
    transaction_too_old_min_block_index: Option<u64>,

    created_at_ns: u64,
    updated_at_ns: u64,
}
```

Use `cycles_sent`, not `cycles_minted`, because `notify_top_up` sends cycles
directly to the target canister.

## State Machine

- `Requested`: transfer args are persisted and the ledger call is not yet known
  successful.
- `Transferred`: `ledger_block_index` is known and `notify_top_up` is not yet
  successful.
- `NotifyProcessing`: CMC returned `Processing`; retry is allowed.
- `Completed`: `notify_top_up` returned `Ok(cycles_sent)`.
- `Refunded`: CMC returned `Refunded`; terminal, preserve refund block index if
  present.
- `TransactionTooOld`: CMC returned `TransactionTooOld`; terminal for
  automatic retry.
- `InvalidTransaction`: CMC returned `InvalidTransaction`; terminal unless an
  operator explicitly overrides.
- `Failed`: non-terminal or terminal depending on `error_code`.

Represent the CMC `NotifyError` variants directly in `error_code` instead of
collapsing them too early.

## Records and Metrics

- Persist the compact record above and keep recovery state centered on the ICP
  ledger `block_index`.
- Extend the existing topup tracker surface rather than adding a parallel
  `icp_to_cycles_conversion(s)` query family.
- Reuse existing metrics families and bounded labels.
- Include a bounded `phase` label with values `preflight`, `transfer`,
  `notify`, and `fabricate` so transfer failures and notify failures remain
  distinguishable without a new metric family.
- Do not add a separate six-series metric family.

## CLI Surface

- If retained, `canic cycles convert` should be a thin caller to the
  canister-side primitive and is acceptable as an operator convenience
  trigger.
- Do not add a native identity-funded ledger / CMC path in this slice.
- Do not add a dedicated `retry` CLI verb yet.
- Keep targets explicit; no bare deployment shorthand.
- Any CLI wrapper should preserve the existing `cycles` family mental model
  rather than becoming a second funds engine.

## Layering Plan

Follow the existing Canic layering:

- `dto/`: request, minimal record, status, and error DTOs.
- `model/` and stable storage: authoritative conversion record and ID
  allocation.
- `ops::storage`: deterministic record create, update, and query helpers.
- `infra::ic`: ICP ledger, CMC notify, rate query, subaccount helpers, and
  config-aware ID resolution.
- `ops::ic`: wrappers for transfer, notify, and rate query.
- `policy/`: auth, concurrency, and request-shape decisions only.
- `workflow/`: conversion orchestration, retry orchestration, and dry-run
  preflight.
- `endpoints` and macros: auth boundary and immediate delegation.
- CLI: thin caller only, not a native transfer engine.

Workflow must not construct or mutate persisted records directly. Record
creation and state transitions belong behind ops/storage helpers.

## Implementation Sequence

1. Add passive data shapes first: request DTOs, status/error enums, compact
   refill record, and storage keys. Keep them internal unless a public endpoint
   must expose a request or response.
2. Add deterministic storage ops for creating `Requested` records, storing
   ledger block indices, recording CMC outcomes, updating fee/error recovery
   fields, and querying active operations for concurrency checks.
3. Add `infra::ic` helpers for ledger/CMC ID resolution, CMC top-up subaccount
   derivation, ICRC-1 transfer argument construction, fee/decimals queries,
   ICP/XDR rate query, and direct `notify_top_up`.
4. Add `ops::ic` wrappers around those infra calls so workflow code never
   hand-rolls IC calls, account construction, or `Nat` to `u64` conversion.
5. Add pure policy checks for request shape, source/target allowlists,
   concurrency, optional rate gate, configured limits, cooldown, and kill
   switch.
6. Add workflow orchestration for dry-run, live conversion, recovery retry,
   duplicate transfer recovery, bad-fee retry preparation, CMC processing
   retry, and terminal CMC errors.
7. Add the opt-in endpoint/macro surface that accepts a host-supplied auth
   guard and delegates immediately to workflow.
8. Wire the primitive into the existing funding chain so a configured funding
   hub can refill itself before downstream `request_cycles` demand is denied
   for lack of hub cycles.
9. Add an operator-side local fabrication helper only after the production
   canister-side path is shaped, and keep it explicitly local-only.
10. Add a thin CLI trigger only if it remains useful after the endpoint and
    funding-chain path exist; the CLI must call the primitive or local
    fabrication helper, not implement its own transfer engine.

## Validation Plan

- Unit-test CMC subaccount derivation with principal lengths and zero-padding.
- Unit-test config-aware ledger/CMC ID resolution, including mainnet override
  rejection without the unsafe/test flag.
- Unit-test policy decisions for allowlists, rate gate, per-call/day limits,
  cooldown, kill switch, and active-operation concurrency.
- Unit-test storage transitions so workflow cannot skip `Requested` before
  transfer or mutate records outside `ops::storage`.
- Unit-test checked `Nat` to `u64` conversion for normal, duplicate, and
  overflowed ledger block indices.
- Unit-test `BadFee`, `Duplicate`, `Processing`, `Refunded`,
  `TransactionTooOld`, and `InvalidTransaction` recovery branches.
- Use PocketIC for canister-side endpoint coverage and funding-chain
  integration where canister creation, inter-canister calls, or management
  canister behavior is involved.
- Keep local fabrication tests separate from production conversion tests and
  assert fabrication is rejected outside local networks.
- Keep CLI tests limited to argument parsing and thin delegation once a CLI
  trigger exists.

## Acceptance Criteria

0.58.0 is done when:

- A Canic-managed canister can replace bespoke `top_up_self_with_icp`,
  `notify_top_up`, and `cmc_subaccount_for` mechanics with the Canic primitive
  while keeping its own auth guard.
- CMC and ICP ledger IDs resolve from config or overrides on local and custom
  networks.
- The primitive can be wired into the existing funding chain so a funding hub
  can refill itself when low.
- A successful transfer with failed notify is recoverable from stored
  `ledger_block_index` without re-transferring ICP.
- A transfer created before ledger persistence recovers through `Duplicate {
  duplicate_of }` and stores the recovered block index.
- A `BadFee { expected_fee }` response updates the persisted fee without
  transferring again.
- CMC `Processing` remains retryable.
- CMC `Refunded` is terminal and preserves the refund block index.
- CMC `TransactionTooOld` is terminal for automatic retry.
- Local fabricated mode is explicit and local-only.
- Fabricated mode is rejected outside local networks.
- `network=ic` rejects ledger / CMC ID overrides unless the unsafe/test flag is
  set.
- Metrics reuse existing funding families and bounded labels.
- The design does not introduce a parallel large record / query API or a
  dedicated CLI retry verb.
- Workflow state transitions happen through `ops::storage`, not by mutating
  records directly.

## Out of Scope

Keep these out of 0.58.0:

- A native identity-funded CLI transfer path.
- A parallel `icp` CLI namespace.
- A new standalone auto-topup policy or second funding island.
- Creating new canisters from ICP.
- Minting cycles into the cycles ledger rather than directly topping up a
  canister.
- A parallel `icp_to_cycles_conversion(s)` query family.
- A new six-series metric family.
- Legacy ICP ledger `transfer` with numeric top-up memo.
- Deprecated ledger `notify()` flow.
- UI / dashboard work beyond logs and metrics.

## Reference Anchors

- Existing Canic funding chain: `cfg.topup`, `request_cycles`, replay
  protection, grant ledger, kill switch, and metrics.
- CMC docs: `notify_top_up` and the ICP/XDR rate query.
- DFINITY CMC migration guidance: direct CMC notify is the supported path.
- toko's current hand-rolled ICP-to-cycles flow and separate `request_cycles`
  copy.
- Existing `cycles` CLI family as a thin `icp` wrapper.
