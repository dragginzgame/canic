Slice 58 — ICP → Cycles Conversion via Canic CLI
Goal

Add a Canic-native way to convert ICP into cycles and top up a canister, callable from the CLI, working against both local and ic networks.

The important design point: for ICP held by a user/CLI identity, the CLI can do the transfer + CMC notify itself. For ICP held by a canister, the canister needs an endpoint, because the canister is the ledger account owner and should perform both the ICP transfer and the CMC notification in the same controlled flow. The modern CMC flow checks the transaction initiator during notification, so splitting “transfer by canister” and “notify by CLI” is the wrong abstraction. The current recommended flow is transfer -> cmc_notify, not the old transfer -> ledger_notify; DFINITY has explicitly recommended moving to CMC notify and away from ledger notify.

Canonical flow Canic should wrap

For a top-up, Canic should implement this flow:

Resolve the source of ICP:
identity: the CLI identity’s ICP ledger account.
canister: a Canic-managed canister’s ICP ledger account.
Resolve the target canister to receive cycles:
default: the source canister itself.
optional: another canister ID/name, since any canister can be topped up.
Transfer ICP to the Cycles Minting Canister top-up account:
CMC canister: rkp4c-7iaaa-aaaaa-aaaca-cai.
ICP ledger: ryjl3-tyaaa-aaaaa-aaaba-cai.
target account is controlled by CMC, with the subaccount derived from the target canister principal.
top-up memo is TPUP / 0x50555054; for ICRC memo form, CMC documents the top-up blob memo as "\54\50\55\50\00\00\00\00".
Capture the ICP ledger block_index.
Call CMC:
notify_top_up : (record {
  block_index : nat64;
  canister_id : principal;
}) -> (variant {
  Ok : nat;
  Err : NotifyError;
});

The CMC Candid defines NotifyTopUpArg as block_index plus canister_id, and the successful result is the amount of cycles sent to the canister.

Persist a conversion record for logs/metrics.
If the transfer succeeds but notify fails transiently, allow retry using the same ledger block_index. The CMC-notify flow is specifically better here because CMC tracks processed transactions and lets callers recover lost results by retrying the notify call.
CLI surface

I’d expose this as a cycles convert command rather than ledger top-up, because in Canic the user intent is “convert ICP into cycles for this canister.”

canic cycles convert <canister> --icp 1.25 --network ic

Aliases:

canic cycles top-up <canister> --icp 1.25 --network ic
canic icp convert <canister> --to-cycles --icp 1.25 --network ic

Recommended flags:

canic cycles convert <target-canister> \
  --icp <amount> \
  --network <local|ic|custom> \
  --source <identity|canister> \
  --source-canister <canister-name-or-id> \
  --from-subaccount <hex> \
  --dry-run \
  --yes

Examples:

# Use the CLI identity's ICP to top up a canister on mainnet.
canic cycles convert api --icp 1.0 --network ic

# Use ICP held by the api canister to top itself up.
canic cycles convert api --icp 0.5 --source canister --source-canister api --network ic

# Use ICP held by treasury to top up worker.
canic cycles convert worker --icp 2.0 --source canister --source-canister treasury --network ic

# Local dev using the real local NNS/CMC path.
canic cycles convert api --icp 10 --network local

# Local dev fallback, when local ledger/CMC are not installed.
canic cycles convert api --cycles 10T --network local --fabricate

For local, there should be two modes:

NNS/CMC local mode: use the same ledger → CMC flow if the local network has NNS/system canisters installed. ICP’s docs note that local system canisters can run at the same mainnet IDs when local NNS is enabled.
Fabricate mode: development-only shortcut that adds cycles without actually converting ICP. DFX documents fabricate-cycles as a local-development-only operation, so Canic should label this separately and not count it as real ICP conversion.
Canister endpoint

For canister-owned ICP, add a small Canic funding interface that Canic-managed canisters can include.

Proposed Candid:

type ConvertIcpToCyclesRequest = record {
  amount_e8s : nat64;
  target_canister : opt principal;
  from_subaccount : opt blob;
  expected_min_cycles : opt nat;
  dry_run : opt bool;
};

type ConvertIcpToCyclesStatus = variant {
  Requested;
  Transferred;
  Notified;
  Completed;
  Failed;
  Refunded;
};

type ConvertIcpToCyclesError = variant {
  Unauthorized;
  InvalidAmount;
  InsufficientIcpBalance;
  LedgerTransferFailed : text;
  CmcNotifyFailed : text;
  TransactionTooOld : nat64;
  InvalidTransaction : text;
  Processing;
  Other : text;
};

type ConvertIcpToCyclesRecord = record {
  id : nat64;
  network : text;
  requested_by : principal;
  source_canister : principal;
  source_subaccount : opt blob;
  target_canister : principal;
  amount_e8s : nat64;
  ledger_fee_e8s : nat64;
  ledger_block_index : opt nat64;
  cycles_minted : opt nat;
  status : ConvertIcpToCyclesStatus;
  error : opt text;
  requested_at_ns : nat64;
  completed_at_ns : opt nat64;
};

service : {
  convert_icp_to_cycles : (ConvertIcpToCyclesRequest)
    -> (variant { Ok : ConvertIcpToCyclesRecord; Err : ConvertIcpToCyclesError });

  retry_icp_to_cycles_conversion : (nat64)
    -> (variant { Ok : ConvertIcpToCyclesRecord; Err : ConvertIcpToCyclesError });

  icp_to_cycles_conversion : (nat64)
    -> (opt ConvertIcpToCyclesRecord) query;

  icp_to_cycles_conversions : (record { offset : nat64; limit : nat64 })
    -> (vec ConvertIcpToCyclesRecord) query;
};

Access control should be strict. I’d make convert_icp_to_cycles callable only by controllers, Canic’s manager principal, or a configured funding role. This is a spend operation.

For estimates, Canic can query CMC’s get_icp_xdr_conversion_rate; the CMC exposes this query, and IC docs note that cycles are minted based on the current ICP/XDR rate. The actual result should always come from notify_top_up, because that returns the real minted cycles amount.

Implementation plan
1. Add IC constants and interfaces

Add shared constants:

const ICP_LEDGER_CANISTER_ID: &str = "ryjl3-tyaaa-aaaaa-aaaba-cai";
const CMC_CANISTER_ID: &str = "rkp4c-7iaaa-aaaaa-aaaca-cai";
const CYCLES_LEDGER_CANISTER_ID: &str = "um5iw-rqaaa-aaaaq-qaaba-cai";
const ICP_TRANSFER_FEE_E8S: u64 = 10_000;
const TOP_UP_MEMO_U64: u64 = 0x50555054;

The cycles ledger is relevant for future extension if Canic later wants to mint cycles into a principal-level cycles account rather than immediately topping up a canister; the current slice should focus on notify_top_up. The cycles docs describe both the CMC and cycles ledger roles.

2. Add subaccount helper

Implement:

fn principal_to_subaccount(principal: Principal) -> [u8; 32] {
    let bytes = principal.as_slice();
    let mut subaccount = [0u8; 32];
    subaccount[0] = bytes.len() as u8;
    subaccount[1..1 + bytes.len()].copy_from_slice(bytes);
    subaccount
}

This mirrors the common IC principal-derived subaccount pattern: first byte is principal length, followed by principal bytes, padded with zeroes.

3. Add identity-funded CLI path

The CLI should:

Resolve target canister.
Build CMC top-up account for target.
Transfer ICP from CLI identity to CMC top-up account.
Call notify_top_up.
Write a local and/or manager-side conversion log.
4. Add canister-funded endpoint path

The CLI should:

Resolve source canister and target canister.
Call source_canister.convert_icp_to_cycles.
The canister performs ledger transfer and CMC notify.
The canister stores the conversion record.
CLI prints the record.

This path is required for “any canister that has an ICP balance should have the ability to convert to cycles.”

5. Add retry path
canic cycles convert retry <conversion-id> --source-canister treasury --network ic
canic cycles convert retry --block-index <idx> --target <canister> --network ic

Retry should never re-transfer ICP unless the previous record has no ledger block index. If a block index exists, retry only calls CMC notify.

6. Add dry-run

Dry run should report:

network
source account
target canister
amount ICP/e8s
ledger fee
CMC account/subaccount
current ICP/XDR rate
estimated cycles
whether local mode is real CMC or fabricate

No transfer happens in dry run.

Acceptance criteria

Slice 58 is done when:

canic cycles convert <canister> --icp <amount> --network ic converts ICP from the CLI identity and tops up the target canister.
canic cycles convert <canister> --source canister --source-canister <id> --icp <amount> --network ic causes the source canister to spend its own ICP and top up the target.
The same command works on local when local ledger/CMC are available.
Local fabricate mode exists but is clearly marked as dev-only and logged as source_type=fabricated.
Every conversion produces a durable conversion record.
Failed transfer and failed notify states are distinguishable.
A successful transfer with failed notify can be retried without sending ICP again.
Metrics expose ICP converted, cycles minted, attempts, failures, and pending conversions.
Access control prevents arbitrary callers from spending a canister’s ICP.
Tests cover identity-funded, canister-funded, dry-run, insufficient balance, notify retry, and local fabricate behavior.
Out of scope for this minor slice

I’d keep these out of slice 58:

Automatic threshold-based top-ups.
Creating new canisters from ICP.
Minting cycles into the cycles ledger rather than directly topping up a canister.
UI/dashboard work beyond logs/metrics.
Cross-canister treasury policies beyond explicit source-canister -> target-canister conversion.

Those can become follow-up slices once the basic conversion path is stable.
