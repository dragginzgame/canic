# Slice 58 - ICP to Cycles Conversion via Canic CLI

## Goal

Add a Canic-native way to convert ICP into cycles and top up a canister,
callable from the CLI, working against both `local` and `ic` networks.

The important design point is source ownership:

- For ICP held by a user or CLI identity, the CLI can perform the ICP transfer
  and CMC notification itself.
- For ICP held by a canister, the source canister must expose a controlled
  endpoint and perform both the ICP transfer and the CMC notification itself.

Splitting "transfer by canister" and "notify by CLI" is the wrong abstraction.
The modern CMC flow checks that the notifier matches the transaction initiator,
so the same principal that sends ICP must also notify the CMC. The current flow
Canic should wrap is transfer to the CMC followed by direct CMC notification,
not the old ledger notification flow.

## Existing Canic Surface

Canic already has a `cycles` command family:

```text
canic cycles balance
canic cycles mint
canic cycles transfer
canic cycles topup
```

Slice 58 should extend that family with one new intent:

```text
canic cycles convert
```

`canic cycles topup` currently means "send already-held cycles to a canister."
`canic cycles mint` currently means "convert ICP into cycles for a cycles
account." Slice 58 adds "convert ICP and immediately top up a canister."

Do not add a parallel `canic icp convert` command in this slice. Do not overload
`canic cycles topup --icp` with a second funding meaning. Keeping one spelling
for this spend flow keeps help text, dry-run output, retry, and audit records
much easier to reason about.

## Canonical Flow

For direct canister top-up, Canic should implement:

1. Resolve the source of ICP.
2. Resolve the target canister that will receive cycles.
3. Transfer ICP to the Cycles Minting Canister top-up account for the target.
4. Capture the ICP ledger `block_index`.
5. Call the CMC `notify_top_up` endpoint.
6. Persist a conversion record for logs, retries, and metrics.

### Source Resolution

Supported source types:

- `identity`: the CLI identity's ICP ledger account.
- `canister`: a Canic-managed canister's ICP ledger account.
- `fabricated`: local-development-only cycle fabrication, not real conversion.

For a canister source, the source canister is the ledger account owner and must
execute the transfer plus notify flow from its own endpoint.

### Target Resolution

Targets should use Canic's existing selector shape:

```text
<principal>
<deployment>/root
<deployment>/<role-or-canister>
```

Examples:

```text
canic cycles convert demo/api --icp 1.0 --network ic
canic cycles convert gastn-uqaaa-aaaal-qbymq-cai --icp 1.0 --network ic
```

Canic should not treat a bare word like `api` as an installed deployment target.
Bare words are too easy to confuse with raw canister names, deployment names,
or role names. The existing `<deployment>/<role-or-canister>` selector keeps
the target explicit.

### CMC and Ledger Details

Mainnet constants:

```rust
const ICP_LEDGER_CANISTER_ID: &str = "ryjl3-tyaaa-aaaaa-aaaba-cai";
const CMC_CANISTER_ID: &str = "rkp4c-7iaaa-aaaaa-aaaca-cai";
const CYCLES_LEDGER_CANISTER_ID: &str = "um5iw-rqaaa-aaaaq-qaaba-cai";
const ICP_TRANSFER_FEE_E8S: u64 = 10_000;
const TOP_UP_MEMO_U64: u64 = 0x50555054;
const TOP_UP_MEMO_BLOB: [u8; 8] = *b"TPUP\0\0\0\0";
```

The cycles ledger ID is included only as a known IC constant. Slice 58 should
not mint cycles into a principal-level cycles account; it should directly top
up a canister through `notify_top_up`.

The CMC top-up account is controlled by CMC, with the subaccount derived from
the target canister principal:

```rust
fn principal_to_subaccount(principal: Principal) -> [u8; 32] {
    let bytes = principal.as_slice();
    let mut subaccount = [0u8; 32];
    subaccount[0] = bytes.len() as u8;
    subaccount[1..1 + bytes.len()].copy_from_slice(bytes);
    subaccount
}
```

The direct CMC notify call is:

```candid
notify_top_up : (record {
  block_index : nat64;
  canister_id : principal;
}) -> (variant {
  Ok : nat;
  Err : NotifyError;
});
```

The `Ok` result is the actual number of cycles sent to the target canister.
Estimates must never replace this result in records or output.

## CLI Surface

Primary command:

```text
canic cycles convert <target> --icp <amount> --network <local|ic|custom>
```

Recommended flags:

```text
canic cycles convert <target> \
  --icp <amount> \
  --network <local|ic|custom> \
  --source <identity|canister> \
  --source-canister <deployment>/<role-or-canister-or-principal> \
  --from-subaccount <hex> \
  --dry-run \
  --yes \
  --format <text|json> \
  --output <path>
```

Examples:

```text
# Use the CLI identity's ICP to top up one installed deployment canister.
canic cycles convert demo/api --icp 1.0 --network ic

# Use the CLI identity's ICP to top up a raw canister principal.
canic cycles convert gastn-uqaaa-aaaal-qbymq-cai --icp 1.0 --network ic

# Use ICP held by the api canister to top itself up.
canic cycles convert demo/api \
  --icp 0.5 \
  --source canister \
  --source-canister demo/api \
  --network ic

# Use ICP held by treasury to top up worker.
canic cycles convert demo/worker \
  --icp 2.0 \
  --source canister \
  --source-canister demo/treasury \
  --network ic

# Local dev using local NNS/CMC when available.
canic cycles convert demo/api --icp 10 --network local

# Local dev fallback, explicitly not real conversion.
canic cycles convert demo/api --cycles 10T --network local --fabricate
```

`--fabricate` is valid only for `--network local`. It should be rejected on
`ic` and custom non-local networks. A fabricated top-up must be logged as
`source_type=fabricated` and must not increment real ICP conversion counters.

## Local Modes

Local conversion has two modes:

- `nns-cmc`: use the same ledger to CMC flow if the local network has NNS and
  system canisters installed.
- `fabricated`: development-only shortcut that adds cycles without spending
  ICP.

Canic should detect whether local NNS/CMC is available by probing the expected
system canisters or by checking local network configuration. Dry-run output
must show which mode will be used. If the operator asks for real conversion on
local and NNS/CMC is unavailable, Canic should fail with a clear message rather
than silently fabricate cycles.

## Canister Funding Endpoint

For canister-owned ICP, add a small Canic funding interface that Canic-managed
canisters may opt into.

This endpoint must not be added silently to every shared runtime bundle. It
should be behind an explicit macro/bundle feature so downstream canisters choose
to expose a spend endpoint. The default bundle should remain conservative unless
a maintainer explicitly decides that funding is part of the baseline runtime
surface.

### Proposed Candid

```candid
type ConvertIcpToCyclesRequest = record {
  amount_e8s : nat64;
  target_canister : opt principal;
  from_subaccount : opt blob;
  dry_run : opt bool;
};

type ConvertIcpToCyclesStatus = variant {
  Requested;
  Transferred;
  Notified;
  Completed;
  Failed;
};

type ConvertIcpToCyclesErrorCode = variant {
  Unauthorized;
  InvalidAmount;
  InvalidTarget;
  InvalidSourceSubaccount;
  InsufficientIcpBalance;
  LedgerTransferFailed;
  CmcNotifyFailed;
  TransactionTooOld;
  InvalidTransaction;
  Processing;
  AlreadyProcessed;
  LocalFabricationUnavailable;
  Other;
};

type ConvertIcpToCyclesError = record {
  code : ConvertIcpToCyclesErrorCode;
  message : text;
};

type ConvertIcpToCyclesRecord = record {
  id : nat64;
  requested_by : principal;
  source_canister : principal;
  source_subaccount : opt blob;
  target_canister : principal;
  requested_amount_e8s : nat64;
  ledger_fee_e8s : nat64;
  ledger_block_index : opt nat64;
  cycles_topped_up : opt nat;
  cmc_rate_timestamp_seconds : opt nat64;
  cmc_xdr_permyriad_per_icp : opt nat64;
  status : ConvertIcpToCyclesStatus;
  error_code : opt ConvertIcpToCyclesErrorCode;
  error_message : opt text;
  created_at_ns : nat64;
  updated_at_ns : nat64;
};

service : {
  convert_icp_to_cycles : (ConvertIcpToCyclesRequest)
    -> (variant { Ok : ConvertIcpToCyclesRecord; Err : ConvertIcpToCyclesError });

  retry_icp_to_cycles_conversion : (nat64)
    -> (variant { Ok : ConvertIcpToCyclesRecord; Err : ConvertIcpToCyclesError });

  icp_to_cycles_conversion : (nat64)
    -> (opt ConvertIcpToCyclesRecord) query;

  icp_to_cycles_conversions : (record { offset : nat64; limit : nat64 })
    -> (vec ConvertIcpToCyclesRecord) query;
};
```

`expected_min_cycles` is intentionally not part of the mutation request. The
conversion rate can change before the transfer, and the actual cycles sent to
the canister come only from `notify_top_up`. Preflight/dry-run can report an
estimate, but a "minimum expected cycles" field cannot safely protect a spend
after the ICP transfer has already happened.

Access control must be strict. `convert_icp_to_cycles` should be callable only
by controllers, Canic's manager principal, or a configured funding role. This is
a spend operation.

## Layering Plan

Follow the existing Canic layering:

- `dto/`: passive request, record, status, error, and page DTOs.
- `model/` and stable storage: authoritative `ConvertIcpToCyclesRecord`
  persistence and record ID allocation.
- `ops::storage`: deterministic record create/update/query helpers.
- `infra::ic`: raw ICP ledger and CMC call mechanics.
- `ops::ic`: metrics-ready wrappers for ICP ledger transfer, CMC notify, and
  CMC rate query.
- `policy/`: pure authorization and request-shape decisions only.
- `workflow/`: conversion orchestration, retry orchestration, and dry-run
  preflight.
- `endpoints` and macros: authentication, authorization boundary checks, and
  immediate delegation to workflow.
- CLI: target/source resolution, local evidence/log output, and dispatch to
  identity-funded direct flow or canister-funded endpoint flow.

Workflow must not construct or mutate persisted records directly. Record
creation and state transitions belong behind ops/storage helpers.

## Required IC Interfaces

Slice 58 needs new interfaces beyond the current generic cycles wrappers:

- ICP ledger transfer from the CLI identity path.
- ICP ledger transfer from canister-owned ICP path.
- CMC `notify_top_up`.
- CMC `get_icp_xdr_conversion_rate` for dry-run estimates.
- CMC top-up account/subaccount construction.
- Typed mapping for ledger transfer failures and CMC notify failures.

Canister-owned ICP should use the ledger interface that preserves the canister
as the transaction initiator. If using ICRC transfer, the top-up memo must use
the blob form `TPUP\0\0\0\0`. If using legacy ledger transfer, the top-up memo
must use the equivalent numeric form.

## Retry Semantics

Retry is source-aware.

```text
canic cycles convert retry <conversion-id> \
  --source canister \
  --source-canister demo/treasury \
  --network ic

canic cycles convert retry --block-index <idx> \
  --target demo/api \
  --network ic
```

Rules:

- If a stored record has `ledger_block_index`, retry must not transfer ICP
  again. It only re-runs CMC notify.
- Identity-funded retry must be performed by the same CLI identity that made
  the original transfer. If Canic cannot verify that identity, it must require
  explicit operator confirmation and still use only CMC notify.
- Canister-funded retry must call the source canister's retry endpoint. The CLI
  must not try to notify CMC on behalf of the source canister.
- If a record has no `ledger_block_index`, retry may only re-attempt the
  transfer when the record is clearly still pre-transfer and the operator
  confirms the spend.
- `Processing` is not final failure. Keep the record pending and allow retry.

The CMC notify flow is recoverable because CMC tracks processed transactions
and can recover lost notify results by retrying the notify call with the same
transaction index.

## Dry Run

Dry run should report:

- network
- selected local mode (`nns-cmc` or `fabricated`)
- source type
- source account or source canister
- target canister
- requested ICP and e8s
- ledger fee
- CMC account/subaccount
- current ICP/XDR rate when available
- estimated cycles when available
- exact command/action that would run next

No transfer and no notify happens in dry run.

## Records and Metrics

Every conversion attempt should produce a durable record, including failures.

Minimum record fields:

- `id`
- `source_type`: `identity`, `canister`, or `fabricated`
- `source_principal`
- `source_canister_id`
- `source_subaccount`
- `target_canister_id`
- `requested_amount_e8s`
- `ledger_fee_e8s`
- `ledger_block_index`
- `cmc_cycles_topped_up`
- `cmc_rate_timestamp_seconds`
- `cmc_xdr_permyriad_per_icp`
- `status`
- `error_code`
- `error_message`
- `requested_by`
- `created_at_ns`
- `updated_at_ns`

On-chain records should not rely on a CLI-provided network string as canonical
truth. Store ledger and CMC canister IDs where they matter. The CLI may keep a
network-scoped local evidence/log record for operator ergonomics.

Metric labels must be bounded. Use error codes, not free-form error messages,
as labels.

Recommended metrics:

```text
canic_icp_to_cycles_attempts_total{source_type,status}
canic_icp_to_cycles_converted_e8s_total{source_type}
canic_icp_to_cycles_topped_up_cycles_total{source_type}
canic_icp_to_cycles_failures_total{source_type,error_code}
canic_icp_to_cycles_pending{source_type}
canic_icp_to_cycles_last_success_timestamp_seconds{source_type}
```

Local CLI logs may include `network`. On-chain metric labels should avoid
unbounded or operator-defined network names unless Canic first normalizes them
to a small enum.

## Acceptance Criteria

Slice 58 is done when:

- `canic cycles convert <target> --icp <amount> --network ic` converts ICP from
  the CLI identity and tops up the target canister.
- `canic cycles convert <target> --source canister --source-canister <source>
  --icp <amount> --network ic` causes the source canister to spend its own ICP
  and top up the target.
- Targets use raw principals or `<deployment>/<role-or-canister>` selectors.
- The same command works on local when local ledger/CMC are available.
- Local fabricate mode exists, is explicit, is local-only, and is logged as
  `source_type=fabricated`.
- Every conversion produces a durable conversion record.
- Failed transfer and failed notify states are distinguishable.
- A successful transfer with failed notify can be retried without sending ICP
  again.
- Identity-funded and canister-funded retries preserve the original transaction
  initiator.
- Metrics expose ICP converted, cycles topped up, attempts, failures, and pending
  conversions with bounded labels.
- Access control prevents arbitrary callers from spending a canister's ICP.
- Tests cover identity-funded, canister-funded, dry-run, insufficient balance,
  notify retry, initiator-preserving retry, and local fabricate behavior.

## Out of Scope

Keep these out of slice 58:

- Automatic threshold-based top-ups.
- Creating new canisters from ICP.
- Minting cycles into the cycles ledger rather than directly topping up a
  canister.
- UI/dashboard work beyond logs/metrics.
- Cross-canister treasury policies beyond explicit source-canister to
  target-canister conversion.
- A new top-level `icp` CLI namespace.

Those can become follow-up slices once the basic conversion path is stable.

## Reference Anchors

- IC system canisters: CMC ID, ICP ledger ID, CMC subaccount derivation, and
  ICP/XDR conversion rate.
- ICP/DFX ledger docs: top-up and notify recovery behavior.
- Cycles ledger docs: distinction between cycles ledger account transfers and
  direct canister top-up.
- DFINITY CMC notify migration guidance: direct CMC notify replaces ledger
  notify for this flow.
