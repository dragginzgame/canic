# 0.33 ICP CLI Cutover

- **Status:** draft
- **Scope:** hard cut to `icp-cli`/`icp.yaml`
- **Decision:** remove `dfx` support from Canic's public workflow

## Summary

Canic should move its fleet tooling to `icp-cli`.

The reason is simple: Canic has multiple fleets, and ICP CLI already models
multiple deployable instances as `icp.yaml` environments.

The new shape is:

```text
Canic fleet -> icp.yaml environment -> resolved canister topology
```

This is a hard cut. Public 0.33 workflows must not depend on `dfx`,
`dfx.json`, `.dfx`, or `canister_ids.json`.

## Source Check

The current ICP CLI docs support the model:

- environments combine a network, canister set, and settings
- `local` and `ic` are implicit environments
- canister IDs are stored per environment
- snapshots support create, download, upload, and restore
- snapshot commands use `-e <environment>` for project canister names and
  `-n <network>` for explicit canister IDs

References:

- <https://cli.internetcomputer.org/0.2/guides/managing-environments/>
- <https://cli.internetcomputer.org/0.2/concepts/environments/>
- <https://cli.internetcomputer.org/0.2/guides/canister-snapshots/>

## Goals

- Make `icp-cli` the public host provider for new Canic workflows.
- Install pinned `icp` and `ic-wasm` binaries in dev setup and CI.
- Keep fleet selection positional, for example `canic fleet inspect prod`.
- Resolve fleet topology from `icp.yaml` environments.
- Keep backup manifests and restore journals owned by Canic.
- Parse ICP CLI output into typed Canic receipts.
- Preserve snapshot safety rules from 0.31.
- Reject journals that do not contain the required ICP provider metadata and
  typed receipts.

## Non-Goals

- No `dfx` support.
- No `dfx.json` support.
- No `.dfx` state.
- No `canister_ids.json` reads.
- No public provider selection.
- No full rewrite of the backup manifest model.
- No full rewrite of restore journals.
- No root-canister-owned backup orchestration in 0.33.

## Fleet Profile

A fleet profile identifies one deployable fleet instance.

For now it lives in the fleet's `canic.toml`:

```toml
[fleet]
name = "prod"
provider = "icp"
icp_yaml = "icp.yaml"
environment = "prod"
network = "ic"
root_canister = "root"
```

Required fields:

- `name`
- `provider = "icp"`
- `icp_yaml`
- `environment`

Recommended fields:

- `network`
- `root_canister`
- `identity`
- `expected_identity`
- `expected_topology_hash`

## Resolved Topology

Canic should resolve a fleet into typed data before backup or restore work:

```json
{
  "fleet": "prod",
  "provider": "icp",
  "environment": "prod",
  "network": "ic",
  "members": [
    {
      "name": "root",
      "role": "root",
      "canister_id": "..."
    },
    {
      "name": "app",
      "role": "app",
      "canister_id": "..."
    }
  ]
}
```

Backup manifests and restore plans should store this resolved topology, not raw
`icp.yaml` or raw command output.

## CLI Shape

Use positional fleet names:

```bash
canic fleet list
canic fleet inspect prod
canic fleet topology prod
canic fleet check prod

canic backup create prod --out backups/prod/<run-id>
canic restore prepare prod --backup-dir backups/prod/<run-id> --out-dir restore/prod/<run-id>
canic restore run --journal restore/prod/<run-id>/restore-apply-journal.json --execute --max-steps 1
```

Manual explicit-ID operations may still exist:

```bash
canic snapshot download --network ic --canister <principal> --out backups/manual/<run-id>
```

## ICP CLI Mapping

Snapshot capture:

```bash
icp canister status <canister> -e <environment>
icp canister stop <canister> -e <environment>
icp canister snapshot create <canister> -e <environment>
icp canister snapshot download <canister> <snapshot-id> -o <path> --resume -e <environment>
icp canister start <canister> -e <environment>
```

Snapshot restore:

```bash
icp canister status <target> -e <environment>
icp canister stop <target> -e <environment>
icp canister snapshot upload <target> -i <artifact-path> --resume -e <environment>
icp canister snapshot restore <target> <uploaded-snapshot-id> -e <environment>
icp canister call <target> <method> -e <environment> --query
```

For explicit canister IDs outside project mappings, use `-n <network>` instead
of `-e <environment>`.

## Provider Boundary

Add an ICP provider in `canic-host`.

It should expose typed operations:

```rust
pub trait IcToolProvider {
    fn version(&self) -> Result<ToolVersion>;
    fn resolve_environment(&self, input: ResolveEnvironmentInput)
        -> Result<ResolvedFleetTopology>;
    fn canister_status(&self, input: CanisterStatusInput)
        -> Result<CanisterStatusReceipt>;
    fn stop_canister(&self, input: StopCanisterInput)
        -> Result<CommandReceipt>;
    fn start_canister(&self, input: StartCanisterInput)
        -> Result<CommandReceipt>;
    fn snapshot_create(&self, input: SnapshotCreateInput)
        -> Result<SnapshotCreateReceipt>;
    fn snapshot_download(&self, input: SnapshotDownloadInput)
        -> Result<SnapshotDownloadReceipt>;
    fn snapshot_upload(&self, input: SnapshotUploadInput)
        -> Result<SnapshotUploadReceipt>;
    fn snapshot_restore(&self, input: SnapshotRestoreInput)
        -> Result<SnapshotRestoreReceipt>;
    fn query_call(&self, input: QueryCallInput)
        -> Result<QueryCallReceipt>;
}
```

The backup crate should stay independent of ICP CLI.

## Toolchain

Dev setup and CI install ICP CLI through the official npm packages:

```bash
npm install -g @icp-sdk/icp-cli@0.2.5 @icp-sdk/ic-wasm@0.9.10
icp --version
ic-wasm --version
```

Versions should stay pinned in CI. Do not use `latest` in automation.

## Journal Rules

Journals should store Canic operations and typed receipts.

Good:

```json
{
  "operation": "upload_snapshot",
  "target_canister": "...",
  "artifact_path": "artifacts/root/snapshot",
  "receipt": {
    "provider": "icp",
    "uploaded_snapshot_id": "...",
    "exit_status": 0
  }
}
```

Bad:

```json
{
  "operation": "run_command",
  "command": "icp canister snapshot upload ..."
}
```

Receipts may include rendered commands for auditability, but raw commands must
not be the canonical operation model.

## Safety Rules

Keep the 0.31 snapshot rules:

- source canister must be stopped before snapshot create
- target canister must be stopped before snapshot restore
- restore uses the uploaded target-side snapshot ID
- restore must not reinstall code after snapshot restore
- interrupted download/upload should use explicit resume behavior
- mutating restore requires safe provider metadata and receipts

## Implementation Slices

### 1. Fleet Profiles

- Add `FleetProfile`.
- Parse `provider = "icp"` from `fleets/<name>/canic.toml`.
- Extend `canic fleet list`.
- Add `canic fleet inspect <name>`.

Acceptance:

- Canic lists configured fleets.
- Canic shows provider, `icp.yaml`, environment, and network.
- Bad fleet names fail clearly.

### 2. ICP Provider Skeleton

- Add `IcpCliProvider`.
- Detect the `icp` binary.
- Capture `icp --version`.
- Add typed command receipts.

Acceptance:

- Missing `icp` gives an actionable error.
- Provider metadata is recorded in dry runs and journals.

### 3. Environment Resolution

- Resolve `icp.yaml`.
- Resolve the selected environment.
- Resolve canister mappings.
- Produce `ResolvedFleetTopology`.
- Compute a topology hash.

Acceptance:

- `canic fleet inspect prod` prints resolved members.
- Missing environment and missing mappings fail clearly.

### 4. Backup Capture

- Implement status proof.
- Implement stop/start.
- Implement snapshot create.
- Implement snapshot download.
- Store typed receipts.

Acceptance:

- `canic backup create prod --dry-run` renders the plan.
- Mutating capture enforces stopped-canister proof.
- Snapshot ID and artifact path are recorded.

### 5. Restore

- Implement snapshot upload.
- Capture uploaded snapshot ID.
- Implement snapshot restore.
- Wire provider receipts into restore runner.

Acceptance:

- Restore uses the uploaded target-side snapshot ID.
- Restore fails before mutation if the receipt is missing.
- No reinstall happens after restore.

### 6. Smoke And Docs

- Add `canic fleet check <name>`.
- Add one ICP CLI local smoke path.
- Update docs to use ICP CLI workflows only.

Acceptance:

- Local smoke proves backup and restore end to end.
- New docs use `icp-cli`, `icp.yaml`, and fleet profiles.

## Cutover Requirements

0.33 is complete only when public Canic workflow docs and command help no longer
reference `dfx`.

Required project shape:

1. `icp.yaml` defines the canisters and environments.
2. Each Canic fleet has a `fleets/<name>/canic.toml`.
3. Each Canic fleet points at one ICP CLI environment.
4. Canic resolves canister IDs through ICP CLI environment data.
5. Backup and restore plans use the resolved topology.

Any command that still needs `dfx` is unfinished 0.33 work.

## Open Questions

1. Should direct `--environment` be allowed without a named fleet profile for
   emergency operations?
2. Should fleet name default to the ICP CLI environment name?
3. Should Canic read `.icp` mappings directly, or prefer ICP CLI commands and
   only read files behind a resolver module?
4. Should production fleet profiles require `expected_identity`?
5. Should backup commands run smoke checks by default, with `--no-smoke` for
   automation?

Recommended defaults:

- Require `canic.toml` for named fleets.
- Allow direct `--environment` only for manual emergency commands.
- Prefer ICP CLI commands; isolate direct `.icp` file reads.
- Require expected identity for production profiles.
- Run smoke by default once the smoke path is reliable.
