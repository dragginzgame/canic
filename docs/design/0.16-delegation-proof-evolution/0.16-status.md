## Current Audit Notes

- `2026-03-24`: rerun `layer-violations` scored `1 / 10`; no hard runtime layer break found, and the earlier policy/workflow drift signals did not reproduce on the remediated tree.
- `2026-03-24`: rerun `token-trust-chain` scored `4 / 10`; no trust-chain validation break found. Remaining pressure is mostly `DelegationProof` spread plus continued churn in `api/auth/mod.rs`.
- `2026-03-24`: remediation plan captured in `0.16-trust-chain-remediation.md` so the current audit debt is tracked in the design set, not only in reports.
- `2026-03-24`: first remediation slice landed in code: delegated-token verification now routes through one canonical trust-chain helper and unit regressions assert the canonical `structure -> current_proof -> delegation_signature -> token_signature` order.
- `2026-03-24`: second remediation slice landed in code: verifier proof storage now uses `StoredDelegationProof` inside `ops::storage::auth`, and API/workflow callers convert back to `dto::auth::DelegationProof` only at explicit boundary seams.
- `2026-03-24`: third remediation slice landed in code: `ops/auth` now narrows internal claim handling through `VerifiedTokenClaims`, `TokenGrant`, `TokenAudience`, and `TokenLifetime`, so verifier, signing, and access/session consumers no longer pass `dto::auth::DelegatedTokenClaims` through most internal helpers.
- `2026-03-24`: fourth remediation slice landed in code: `ops/auth/verify.rs` was split into `verify/token_chain.rs`, `verify/proof_state.rs`, and `verify/attestation.rs`, with `verify/mod.rs` keeping the old caller-facing seam while separating proof-state, crypto-chain, and attestation invariants.
- `2026-03-24`: fifth remediation slice landed in code: `api/auth/mod.rs` and `api/auth/session/mod.rs` now delegate proof-reuse, bootstrap audience subset, delegated-session expiry clamping, and verifier-target derivation to `DelegatedTokenOps`, leaving the API layer focused on request wiring and user-facing error mapping.
- `2026-03-25`: issuance and verification now share the same lifetime invariant gate, explicit zero-skew policy, and max-TTL enforcement, closing the malformed/expired/future-token asymmetry found in the auth audit.
- `2026-03-25`: full release-readiness evidence is recorded: `make fmt-check`, `make clippy`, and `make test` all passed.
