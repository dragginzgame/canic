# Canic Design Proposal: Delegation Proof Evolution

- **Version:** `0.16`
- **Status:** `Draft`
- **Author:** `Canic Core`
- **Scope:** deferred Tier 1 delegation Phase 3 work from `0.15`

## 1. Scope Origin

`0.15` completed Tier 1 delegation Phase 1/2 correctness and hardening.
All remaining Phase 3 follow-ups are deferred into this `0.16` design track.

Deferred items originate from:

- `docs/design/0.15-auth-identity-foundation/0.15-design.md` section `12.1`

## 2. Baseline Invariants (Must Not Change)

`0.16` follow-ups must preserve these invariants:

- delegated-token verification stays local and fail-closed
- `authenticated(...)` endpoints remain token-gated (no tokenless mode)
- delegated-session bootstrap remains explicit and verifies token before write
- raw-caller infrastructure predicates remain bound to transport caller
- root provisioning remains the primary readiness path

## 3. Problem Statement

Current verifier proof state is single-active (`set_proof` overwrite model).
That limits multi-shard verifier reliability:

- proof for shard `A` can be overwritten by later proof for shard `B`
- token from `A` can then fail verifier local checks with `ProofMismatch`

Operational recovery also remains mostly issuance-time push-only; there is no
formalized post-issuance repair/prewarm contract in the current design set.

## 4. Goals

1. Define bounded multi-proof verifier storage keyed by shard/cert identity.
2. Define explicit operator/workflow repair-prewarm flow outside auth hot path.
3. Optionally define a bounded single-refresh verifier path only if required.
4. Keep security posture identical to `0.15` fail-closed local verification.
5. Specify rollout/migration and observability requirements for safe adoption.

## 5. Non-Goals

- no weakening of local verification into network-dependent verification
- no network proof fetch/refresh on the `authenticated(...)` hot path
- no implicit delegated-session auth without delegated token
- no rewriting role-attestation or capability trust semantics in this phase
- no relaxation of root-only proof install trust boundary
- no unbounded verifier retry loops or auth-time retry amplification

## 6. Design Tracks

### 6.1 Multi-Proof Verifier State (Required)

- evolve verifier proof storage from `Option<DelegationProof>` to bounded map
- key by verifier-relevant proof identity (issuer shard + cert/proof binding)
- include bounded capacity and deterministic eviction policy
- preserve strict proof match behavior at verify time

### 6.2 Explicit Repair/Prewarm Flow (Required)

- add explicit flow to push/repair verifier material outside request auth path
- scope to operator/workflow driven mechanisms with clear authorization
- add observability for attempts, success, failures, and target coverage

### 6.3 Bounded Refresh on Miss (Optional)

- only considered if production evidence shows explicit repair is insufficient
- single attempt only for selected verifier-miss classes
- must be auditable, bounded, and non-amplifying
- must not convert `authenticated(...)` into unbounded retry behavior

## 7. Rollout and Migration Requirements

- migration must preserve existing proof material safely
- upgrade path must be deterministic and reversible by deployment policy
- metrics must expose cache occupancy/eviction and miss reasons
- integration tests must cover mixed-shard verifier traffic and rollover

## 8. Acceptance Criteria

`0.16` Phase 3 work is complete only when all hold:

- multi-shard verifier traffic does not fail due to single-proof overwrite
- provisioning repair path is explicit, authorized, and observable
- fail-closed local verification behavior is unchanged
- no tokenless path is introduced for `authenticated(...)`
- integration tests prove issuance -> verifier readiness -> bootstrap -> auth

## 9. Open Questions

1. What is the canonical proof-identity key for map storage (issuer shard, cert hash, both)?
2. What bounded capacity/eviction policy is acceptable for stable memory limits?
3. Is optional bounded refresh needed after explicit prewarm/repair is available?
4. Which metrics/alerts are mandatory before enabling multi-proof mode in production?
