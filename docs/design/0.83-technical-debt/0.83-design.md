# Canic 0.83 Design: Technical Debt Audit

Date: 2026-07-08

## Status

- **Status:** complete
- **Release line:** 0.83 technical debt audit
- **Decision style:** debt inventory, release-risk classification, and cleanup
  planning
- **Primary result:** the canonical evidence-backed ledger is
  `docs/audits/0.83-technical-debt/ledger.md`.
- **Closeout state:** the ledger is `pass`; all 37 findings are fixed, no
  findings are deferred, and every recommended slice is complete. The final
  post-v0.83.27 Unreleased batch fixes receipt conflict-key collisions,
  completes promotion execution/staging label ownership, and reconciles audit
  metadata without changing package versions.

## 1. Summary

0.83 is a technical debt audit release.

Its job is not to fix everything. Its job is to make Canic's remaining debt
visible, classified, deduplicated, and actionable before future feature work
continues.

The audit should answer:

- What debt exists?
- Where is the evidence?
- Who owns the boundary?
- What release risk does it create?
- Is it active production risk, test-only debt, docs drift, or archived legacy?
- Should it block release, become a hardening slice, be deferred, or be
  rejected?

0.83 should produce a debt ledger that future work can use to select focused
cleanup slices. It should not silently turn into a broad refactor, module-tree
rename, style pass, or behavior-changing cleanup.

## 2. Hard-Cut Rule

0.83 is an audit and classification release first.

A debt finding does not authorize a code change by itself. Fixes require a
separate accepted finding and a scoped implementation slice.

The audit must not introduce:

- new features
- compatibility shims
- command aliases
- broad refactors
- package renames
- serialized shape changes
- public behavior changes
- opportunistic module moves
- lint suppressions instead of boundary fixes
- cleanup outside the accepted finding set

Discovered issues outside the active audit scope are recorded as follow-ups.

## 3. Goals

- Create a repo-wide technical debt ledger.
- Classify debt by severity, category, owner, affected surface, and release
  risk.
- Separate active production debt from test-only, docs-only, archived, or
  unreachable debt.
- Identify debt introduced or revealed by the 0.78 through 0.82 work.
- Detect duplicated semantics across reports, configs, DTOs, runtime status,
  metrics, state manifests, and CLI renderers.
- Identify half-migrated ownership moves where old paths remain semantic owners
  instead of re-export surfaces.
- Identify broad public visibility that should be narrowed.
- Identify stale compatibility paths and hard-cut violations.
- Identify tests that no longer protect the behavior they claim to protect.
- Produce a debt ledger that can feed future hardening slices without expanding
  the current audit into fixes.

## 4. Non-Goals

0.83 does not:

- fix debt during the audit pass
- rewrite the module tree
- rename packages
- change public CLI behavior
- change endpoint names
- change Candid method names
- change JSON schema fields
- change DTO serialized shapes
- change stable-state layout
- change deployment-truth or evidence/report schemas
- add new runtime, medic, state, or deploy-plan features
- add new lints as hard gates before the audit identifies stable candidates
- replace 0.82 boundary hardening slices
- treat every duplication as harmful
- treat every TODO as release-blocking

## 5. Audit Relationship To 0.82

0.82 is a boundary-hardening implementation line. 0.83 is a debt discovery and
classification line.

0.83 may discover candidates for future hardening slices, but it does not
perform those slices automatically.

Relationship:

- **0.82:** fix a chosen boundary with a scoped no-behavior-change slice.
- **0.83:** inventory debt, classify risk, deduplicate findings, and recommend
  future slices.

Any debt item that requires a code move, visibility change, command removal,
schema migration, or behavior change must become a separate implementation
slice before it is fixed.

## 7. Severity Rubric

### P0

Debt creates or plausibly enables:

- auth bypass
- mutation in a read-only path
- stable-state corruption
- lifecycle upgrade failure
- public exposure of guarded runtime/state data
- unsafe deploy/apply behavior
- incorrect principal/caller/auth binding
- accidental command route that violates a hard-cut design

### P1

- Active production code violates a recent hard-cut design.
- Public CLI, endpoint, Candid, JSON, stable-state, deployment-truth, or
  evidence/report surface can drift or break because ownership is unclear.
- Policy performs mutation, async work, timers, IC calls, storage access, or
  serialization.
- Workflow constructs persisted records directly.
- CLI renderer redefines report status, safety semantics, or source
  attribution.
- Lifecycle restore/timer scheduling order is ambiguous or duplicated.
- Metric labels or counters can drift because dimension ownership is
  duplicated.
- Tests are missing for a safety-critical boundary.

### P2

- Duplicate config parsing, diagnostic codes, value semantics, or report logic
  creates maintainability or drift risk.
- Visibility is broader than needed but not currently externally exposed.
- Old import paths remain as active semantic owners instead of re-export
  surfaces.
- Test ownership is unclear.
- Docs describe old behavior but do not affect active operator commands.
- TODO/FIXME/HACK exists in active code but is not release-blocking.

### P3

- Naming clarity
- Comment drift
- Archived-doc cleanup
- Snapshot readability
- Small visibility polish
- Non-blocking test organization

## 8. Debt Categories

Initial categories:

- `boundary_ownership`
- `diagnostic_ownership`
- `config_ownership`
- `dto_boundary`
- `view_boundary`
- `policy_purity`
- `ops_workflow_ownership`
- `lifecycle`
- `state_migration`
- `runtime_introspection`
- `metrics`
- `visibility`
- `legacy_surface`
- `test_coverage`
- `docs_drift`
- `dependency`
- `dead_code`
- `todo`

### 8.1 Category Meanings

#### `boundary_ownership`

A semantic value, decision, report, or invariant is owned by the wrong layer or
by multiple layers.

#### `diagnostic_ownership`

Diagnostic codes, categories, status derivation, next-action semantics, or
source attribution are duplicated or renderer-owned.

#### `config_ownership`

Config parsing or validation has multiple owners or inconsistent defaults.

#### `dto_boundary`

DTOs own runtime validation, authorization decisions, storage semantics, or
cross-layer behavior instead of remaining passive boundary data.

#### `view_boundary`

Internal read-only projections are confused with API DTOs or persisted records.

#### `policy_purity`

Policy imports storage, runtime, timers, IC APIs, serializers, DTO encoders, or
performs side effects.

#### `ops_workflow_ownership`

Workflow constructs persisted records, directly mutates storage, or calls
platform APIs that should be isolated in ops.

#### `lifecycle`

Install, reinstall, pre-upgrade, post-upgrade, timer scheduling, user hooks, or
invariant restore order is duplicated, ambiguous, or not tested.

#### `state_migration`

State versions, memory IDs, removed-state disposition, migration coverage, or
post-upgrade invariants are missing or unclear.

#### `runtime_introspection`

Runtime status, health, readiness, field visibility, authorization, recent
failures, timer status, or source attribution can leak or drift.

#### `metrics`

Metric dimension values, labels, counters, or projected labels are owned in the
wrong layer or duplicated.

#### `visibility`

`pub`, `pub(crate)`, or module exposure is broader than needed.

#### `legacy_surface`

Hard-cut command, endpoint, module, schema, or compatibility path remains
active.

#### `test_coverage`

Tests no longer protect the behavior they claim to protect, are too far from
the owner, or assert brittle strings instead of typed outcomes.

#### `docs_drift`

Active docs describe behavior that is no longer true or omit current operator
surfaces.

#### `dependency`

Unused, overlapping, or overbroad dependencies/features remain active.

#### `dead_code`

Active code is unreachable, stale, or only retained by accident.

#### `todo`

TODO/FIXME/HACK comments that should be fixed, tracked, or explicitly deferred.

## 9. Affected Surface Classification

Every finding must classify affected surfaces:

- `cli`
- `endpoint`
- `candid`
- `json`
- `dto_serialization`
- `stable_state`
- `deployment_truth`
- `evidence_report`
- `metric_labels`
- `metric_counters`
- `docs`
- `tests`
- `internal_only`

A finding that affects only internal code is lower-risk unless the internal
boundary protects auth, lifecycle, state, metrics, or report semantics.

## 10. Debt Disposition

Each finding must end in one of these decisions:

- `block_release`
- `requires_fix_slice`
- `convert_to_hardening_slice`
- `defer_with_rationale`
- `reject_with_rationale`
- `merge_as_duplicate`
- `document_known_limitation`

Rules:

- P0 findings block release.
- P1 findings block release unless explicitly deferred with owner and
  rationale.
- P2 findings may require a fix slice, convert to later slices, or be deferred.
- P3 findings should not block release unless they create operator confusion.
- Unknown reachability should be marked `unknown`, not silently escalated.
- Duplicate findings should be merged into the earliest or clearest finding.

## 11. Evidence Quality

Every finding must include enough evidence for a reviewer to reproduce the
classification without replaying the whole audit.

Required evidence:

- exact file path
- line number or smallest stable local anchor
- module, function, command, or document section
- command, search, or inspection method used to find the issue
- why the issue is active production, test-only, docs-only, archived, or
  unreachable
- why the observed issue fits the selected category and severity

Findings must not rely only on vague phrasing such as "looks stale" or "seems
duplicated." If reachability cannot be proven, mark the finding `unknown` and
record the missing evidence.

## 12. Audit Tracks

0.83 should run multiple read-only audit tracks. Each track produces findings in
the same ledger format.

### 12.1 Recent Slice Debt

Audit debt introduced or revealed by:

- 0.78 medic
- 0.79 deploy plan
- 0.80 state audit
- 0.81 runtime introspection
- 0.82 boundary hardening

Checks:

- Recent report models have one semantic owner.
- CLI renderers do not redefine report semantics.
- Rejected command forms are not active.
- `--json` conventions are consistent.
- Output does not use deployment-safety language incorrectly.
- Runtime-observed values are not filled from local config.
- State/runtime outputs do not leak persisted values.
- Metric label/counter behavior is preserved after ownership moves.
- Old paths are re-export surfaces only, not duplicate semantic owners.

### 12.2 Boundary Ownership Debt

Checks:

- DTOs remain passive.
- Policy is pure.
- Ops owns state mutation and approved platform side effects.
- Workflow owns orchestration only.
- Model owns authoritative state and invariants.
- Endpoint guards own authentication.
- View types remain internal read-only projections.

### 12.3 Hard-Cut And Legacy Surface Debt

Audit recent hard-cut specs for active rejected forms.

Examples:

- `canic info medic <deployment>`
- `canic medic <deployment>`
- `canic state migrate`
- `canic state repair`
- `canic state explore`
- `canic state dump`
- `canic inspect <ambiguous-target>`
- `canic inspect deployment <deployment> --all`
- `canic topology <deployment>`
- `canic runtime status <deployment>`
- `canic deploy plan <deployment> --apply`
- `canic deploy plan <deployment> --write-truth`

Classify each occurrence:

- active route
- help text only
- test fixture only
- archived doc only
- dead code
- safe delete
- requires separate slice

### 12.4 Config And Diagnostics Debt

Checks:

- Config parsing has one owner per config type.
- Validation is not duplicated across CLI, ops, and report producers.
- Diagnostic codes are producer-owned.
- Status derivation is producer-owned.
- Renderers only render existing reports.
- Text and JSON derive from the same report model.
- Exit-code mapping consumes report status without recomputing safety.

### 12.5 Lifecycle Debt

Checks:

- `canic::start!` remains thin.
- Lifecycle adapters restore synchronously.
- Async work is scheduled only after invariants are restored.
- User hooks run after Canic invariants are restored.
- Install/reinstall/upgrade paths are tested where behavior matters.
- Generic traps below lifecycle boundary are not hiding recoverable errors.
- State restore logic is not duplicated across roles.

### 12.6 Test Debt

Checks:

- Tests assert behavior, not implementation noise.
- Tests use typed errors instead of brittle strings where possible.
- Upgrade behavior uses PocketIC where canister lifecycle matters.
- CLI parser tests cover rejected forms.
- Report JSON tests assert stable fields.
- Metric tests assert labels and counters when metrics are an operator surface.
- Tests live near the owner unless ownership changed.

### 12.7 Docs Debt

Checks:

- Active docs use canonical command surfaces.
- Rejected command forms are not documented as valid.
- Archived docs are clearly historical.
- Design docs match implementation status.
- Operator docs do not claim safety beyond the command contract.
- README examples do not point to removed surfaces.
- Known limitations are explicit.

### 12.8 Dependency And Feature Debt

Checks:

- Unused dependencies are identified.
- Overlapping dependencies are identified.
- Dev/test dependencies are not pulled into production builds.
- Feature flags have clear owners.
- Old feature flags from removed designs are absent or isolated.
- Optional feature sections fail closed when missing.

## 17. Audit Procedure

0.83 should be run in read-only stages:

1. Inventory
2. Classify
3. Deduplicate
4. Severity review
5. Recommend slices
6. Closeout

### 17.1 Stage 1: Inventory

Search for candidate debt and collect evidence.

Do not fix.

### 17.2 Stage 2: Classify

Assign category, owner, affected surfaces, and reachability.

### 17.3 Stage 3: Deduplicate

Merge duplicate findings and record duplicate IDs.

### 17.4 Stage 4: Severity Review

Apply severity rubric. Escalate only with evidence.

### 17.5 Stage 5: Recommend Slices

Group accepted findings into future hardening or feature-specific slices.

### 17.6 Stage 6: Closeout

Produce ledger, deferred list, rejected list, recommended slices, and validation
summary.

## 18. Validation

Because 0.83 is audit-first, validation focuses on current repo health and
evidence quality.

Baseline repo-health commands:

```sh
cargo fmt --all -- --check
cargo check --locked -p canic-core -p canic
cargo test --locked -p canic-cli
cargo test --locked -p canic --test protocol_surface
cargo test --locked -p canic-host deployment_truth --lib
git diff --check
```

These commands establish that the audit is not being written against an
obviously broken workspace. They are not a full release gate unless a maintainer
explicitly promotes the audit to release validation.

Finding-specific commands should be selected based on the accepted findings:

```sh
cargo test --locked -p canic-core state --lib
cargo test --locked -p canic-core runtime --lib
cargo test --locked -p canic-cli medic
cargo test --locked -p canic-cli deploy
cargo test --locked -p canic-cli state
cargo test --locked -p canic-cli inspect
```

Each finding-specific validation entry should name the finding IDs it supports.
If a command cannot run, the ledger should record why.

## 19. Acceptance Criteria

Close 0.83 only when:

- The debt ledger exists.
- Each finding has severity, category, status, owner, evidence, risk, and
  release decision.
- P0 findings are either fixed by a separate slice or block closeout.
- P1 findings are fixed by a separate slice, converted to immediate slices, or
  explicitly deferred with rationale.
- P2/P3 findings are deduplicated and either deferred, rejected, or grouped into
  recommended slices.
- No finding is based only on vague style preference.
- No fixes were made during the audit pass unless a separate accepted fix slice
  was opened.
- Recommended slices are focused and do not combine unrelated boundaries.
- Deferred findings have owners and rationale.
- Rejected findings explain why they are not debt.
- Validation commands and limitations are recorded.
- Finding-specific validation is mapped to finding IDs where applicable.

The final closeout recommends one of:

- proceed to next feature line
- run immediate P0/P1 fix slice
- run focused hardening slice
- repeat targeted audit for a subsystem

## 20. Closed Decisions

- 0.83 is audit-first.
- 0.83 does not introduce a debt CLI command in the first slice.
- Technical debt findings do not authorize code changes by themselves.
- The debt ledger is the primary release artifact.
- Future cleanup should be grouped into focused slices with behavior-impact
  labels.
- Duplicated code is not automatically debt unless it creates ownership,
  correctness, drift, or release-risk problems.
- Archived historical docs are not debt unless active docs link to them as
  current behavior.

## 21. Closeout Decisions

- `ledger.md` is canonical; 0.83 does not require `findings.json`.
- P1 handling remains governed by the release-decision rules in section 10.
  No P1 finding remains at closeout.
- TODO/FIXME/HACK inspection was folded into the initial source inventory and
  recorded as a rejected candidate because it produced no active source match.
- Dependency/feature work was not promoted into an accepted 0.83 finding
  without reproducible evidence. Any future candidate requires a new targeted
  audit rather than an implicit deferred item.
- The closeout recommendation is to proceed to the next feature line after the
  current Unreleased fixes follow the maintainer-owned release flow.
