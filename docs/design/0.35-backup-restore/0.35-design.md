# Canic Backup/Restore Rework

- **Version:** `0.35`
- **Status:** Active implementation
- **Scope:** make fleet backup and restore executable, journaled, and
  topology-aware
- **Supersedes:** `docs/design/0.34-backup-restore/0.34-design.md`

## 1. Summary

`0.35` carries the backup/restore design from the 0.34 draft into the release
line where the operator flow must work end-to-end.

Backup and restore are fleet topology operations, not one-off canister loops:

- backup may target any connected fleet subtree
- root stays running by default and root capture remains explicit maintenance
  work
- selected members are quiesced, snapshotted, restarted, downloaded, verified,
  and finalized through a journal
- restore plans upload artifacts, quiesce target members, load snapshots,
  restart target members, verify status, and retain receipts for recovery

The immediate 0.35 goal is a practical local/operator path using the ICP CLI
runner. Root-mediated control remains the boundary for child-controller cases
where the operator identity cannot directly stop, snapshot, or restore a child.

## 2. Current Implementation Target

The 0.35 command path is:

```bash
canic backup create <fleet>
canic backup verify <backup-ref>
canic restore plan --backup-dir <backup-dir> --require-verified --out restore-plan.json
canic restore apply --plan restore-plan.json --backup-dir <backup-dir> --dry-run --journal-out restore-apply-journal.json
canic restore run --journal restore-apply-journal.json --dry-run
canic restore run --journal restore-apply-journal.json --execute
```

`restore apply` remains a no-mutation renderer. `restore run --execute` owns the
mutating runner, journal locking, command receipts, failed-operation state, and
pending-operation recovery.

## 3. Backup Invariants

Normal backups do not stop root. A backup target must resolve from the live
fleet registry to exactly one connected subtree or to the managed non-root
fleet. Operation evidence is explicit:

- control authority: `Proven | Declared | Unknown`
- snapshot read authority: `Proven | Declared | Unknown`

Dry-run may render declared or unknown authority. Mutating backup execution
requires proven control and snapshot-read authority for every selected target
before any stop or snapshot operation runs.

The consistency point is snapshot creation, not artifact download:

```text
discover topology
validate selected subtree
quiesce selected members
create snapshots for selected members
restart selected members
download snapshot artifacts
verify artifacts
write manifest and final journal state
```

## 4. Restore Invariants

Restore uses the backup manifest as the source of truth and writes a separate
restore plan plus apply journal.

The executable runner must not assume hidden manual steps. A ready restore
journal contains all platform mutations required to complete restore:

```text
upload snapshot artifacts
stop target members
load uploaded snapshots
start target members
verify target members and fleet checks
```

Stop order follows the planned parent-before-child ordering. Start order is the
reverse so descendants are available before parent ingress resumes. Snapshot
load uses the same deterministic member order as planning.

Every mutating operation records a receipt. Failed commands move the operation
to `failed`; interrupted claims can be released with
`canic restore run --unclaim-pending`.

## 5. Ownership Boundaries

`canic-cli` owns command parsing, user-facing output, and invoking host/backup
APIs.

`canic-host` owns ICP CLI process construction, installed-fleet resolution, live
registry parsing, and reusable host/operator primitives.

`canic-backup` owns backup and restore domain state: plans, manifests, journals,
operation ordering, validation, and runner state transitions. It may render
host command descriptions, but it should not know CLI argument parsing rules.

Root-mediated child control belongs behind a host/control adapter, not in
`canic-cli`. If direct ICP CLI control is insufficient for child canisters,
0.35 should add the adapter boundary before duplicating control logic in the
CLI.

## 6. 0.35 Working Checklist

- backup create writes a complete verified backup layout for a local fleet
- restore plan rejects unverified or incomplete backup layouts when requested
- restore apply validates artifacts and emits a ready journal
- restore run previews the exact ICP CLI command it will execute next
- restore run execute can upload, stop, load, start, verify, resume pending
  claims, and fail closed with durable receipts
- docs describe local ICP CLI persistence limits and avoid implying local
  replica restart preserves canisters

## 7. Deferred Work

- root-mediated stop/start/snapshot/restore endpoints for child canisters where
  root is the only controller
- whole-root maintenance backup mode
- richer restore verification beyond status checks
- remote-host artifact transport and retention policy
