# Canic 0.97 Design: Role-Owned Runtime Dependencies And CDK Surface

Date: 2026-07-21

## Status

- Status: proposed with the Cargo contract complete. Implementation still
  requires maintainer acceptance and cannot begin while 0.96 remains active.
- Release line: 0.97, after the active 0.96 receipt line.
- Current Canic anchor: released `v0.96.1`.
- Downstream scope: Toko and other repositories remain read-only from Canic.
  This design defines their migration contract but does not edit them.
- Compatibility posture: pre-1.0 hard cut. Removed Rust paths receive no
  aliases, deprecations, forwarding modules, or fallback imports. Existing
  `canic::cdk` imports intentionally stop compiling and receive one direct
  migration path.

## Decision

The final configured canister role package owns all Canic runtime selection.
Every role package has one direct, unconditional, non-optional normal
dependency on `canic`, and that declaration is the sole owner of the Canic
features used by the role.

There is one supported graph, shown with copyable manifests below. Shared
domain and IC adapter crates are Canic-free; a role never consumes another
role package as a library.

0.97 makes five related decisions:

1. Retain and strengthen the package-graph validator as the enforcement
   authority. Rust visibility is not a dependency-policy mechanism.
2. Reject any normal dependency subtree outside the role's canonical direct
   Canic edge when that subtree reaches a protected Canic framework package.
3. Return the exact deterministic offending dependency path in the existing
   dependency-shape diagnostic.
4. Separate exported-macro plumbing from the human-facing Rust API through the
   existing `#[doc(hidden)] canic::__internal` boundary.
5. Hard-cut the public `canic::cdk` facade. The required inventory decides
   where each current item moves or whether it is deleted; it cannot retain
   the facade, an alias, or a documented compatibility path.

The validator remains necessary after the facade removal. Removing a Rust
module can discourage accidental coupling, but it cannot stop a shared crate
from declaring `canic`, `canic-core`, or another framework package in
`Cargo.toml`.

## Why Multiple Runtime Paths Are Unsafe

Cargo builds a normal dependency with the union of features requested through
the normal runtime graph. Resolver version 2 separates several
target/build/dev cases, but it does not prevent feature union between two
normal runtime paths to the same package.

For example:

```text
root -> canic features: metrics
toko -> canic features: sharding

effective Canic build: metrics + sharding
```

The role's own manifest no longer fully describes its runtime. This matters
because Canic features select endpoint bundles, lifecycle behavior,
authentication capabilities, control-plane support, and stable-memory
allocations. A helper crate must not silently widen those choices.

The issue exists even when Cargo compiles one Canic version only. Two graph
paths create two configuration contributors. Multiple Canic package IDs or
versions are an additional failure, not the only failure.

## Central Invariant

The selected role package is the sole external authority for Canic-owned
framework packages and directly selected Canic capability features. Ordinary
upstream packages may be shared across runtime subtrees and may therefore
receive Cargo features from both Canic and application dependencies.

Let:

- `R` be the selected role package;
- `C` be its one direct normal dependency edge to package `canic`;
- `P` be the protected Canic framework package-name catalog; and
- `D` be any other direct normal dependency edge from `R`.

The required graph property is:

```text
reachable(D) intersect P = empty
```

The canonical subtree below `C` may contain Canic's own internal package
relationships. A sibling subtree below `D` may not contribute another Canic
framework package, regardless of whether it resolves to the same package ID or
version as the canonical subtree.

Consequences:

- the direct role declaration remains the sole Canic feature authority;
- a shared crate cannot bypass the rule through `canic-core` instead of
  `canic`;
- shared paths may still converge on ordinary upstream packages such as
  `ic-cdk`; those packages are outside the protected Canic ownership claim;
- generic `candid`, `ic-cdk`, `ic-cdk-timers`, and
  `ic-stable-structures` dependencies remain allowed;
- build and development dependencies do not become normal Wasm-runtime
  evidence; and
- public or hidden Rust module visibility never substitutes for graph
  validation.

## One Supported Dependency Graph

Developers should not need to reverse-engineer Cargo metadata or run
`cargo tree` to understand the rule. Every configured canister role follows
the same graph:

```text
workspace
├── root-canister [ROLE]
│   ├── canic [DIRECT]
│   ├── toko-domain
│   │   └── serde
│   └── toko-ic-adapter
│       ├── candid
│       └── ic-cdk
├── project-instance [ROLE]
│   ├── canic [DIRECT]
│   ├── toko-domain
│   └── toko-ic-adapter
└── project-ledger [ROLE]
    ├── canic [DIRECT]
    ├── toko-domain
    └── toko-ic-adapter
```

The arrows always point downward:

```text
role -> Canic
role -> shared domain
role -> generic IC adapter

shared domain -X-> Canic
generic IC adapter -X-> Canic
one role package -X-> another role package
```

If two roles need the same DTO, rule, or IC conversion, that code moves to a
Canic-free shared package. A role package must never become another role's
library dependency, because its direct Canic edge would become transitive.

The copyable manifest pattern is:

```toml
# Workspace Cargo.toml
[workspace]
resolver = "2"

[workspace.dependencies]
canic = {
    version = "...",
    default-features = false,
}
```

The workspace declaration owns only Canic identity, source, version, and the
disabled-default baseline. It must not declare any Canic feature. Cargo makes
workspace dependency features additive, so a workspace feature would be an
untracked contributor to every inheriting role.

```toml
# root-canister/Cargo.toml
[dependencies]
canic = { workspace = true, features = ["metrics", "control-plane"] }
toko-domain.workspace = true
toko-ic-adapter.workspace = true

[build-dependencies]
canic.workspace = true
```

This is exactly one direct **normal** Canic edge. The separately permitted
build edge exists only for a `build.rs` that invokes `canic::build!` to compile
the role configuration and enforce the canonical build boundary. Resolver 2
keeps its host/build feature set separate from the normal Wasm dependency.
A role without that build-script use must not carry the build edge.

A non-inherited dependency uses the same explicit baseline:

```toml
[dependencies]
canic = {
    version = "...",
    default-features = false,
    features = ["metrics", "control-plane"],
}
```

```toml
# toko-domain/Cargo.toml
[dependencies]
serde.workspace = true

# No canic, canic-core, canic-control-plane, or canic-macros dependency.
```

```toml
# toko-ic-adapter/Cargo.toml
[dependencies]
candid.workspace = true
ic-cdk.workspace = true

# No protected Canic framework dependency.
```

Every directly selected Canic feature is written on the role's normal
`canic` dependency. Features implied by Canic's own catalog are derived by
Canic and need not be repeated. Role packages do not forward their own
features into `canic/*`, workspace dependencies declare no Canic features,
and shared packages select no Canic feature.

The role declaration always contains a `features` key, even when the explicit
selection is empty. During migration, every feature previously obtained from
Canic defaults—currently including `metrics` where used—is materialized in the
role list before defaults are disabled, so the hard cut does not silently
change runtime behavior.

The package rule at a glance is:

| Package kind | Direct `canic` dependency | Other protected Canic dependency | Upstream IC dependency |
| --- | --- | --- | --- |
| Configured role package | Required exactly once as a normal edge; one purpose-bound build edge is separate | Forbidden outside the canonical Canic subtree | Allowed |
| Shared domain package | Forbidden | Forbidden | Avoid; pass ordinary domain inputs instead |
| Generic IC adapter package | Forbidden | Forbidden | Allowed |
| Another configured role package | Own direct edge only; never consumed as a library | Forbidden through sibling roles | Allowed |

There is no supported “shared Canic helper” category. Canic-specific code is
role code, even when similar adapters exist in more than one role.

## Release Boundary

0.97 owns:

- exact role-owned framework dependency rules;
- the protected Canic framework package catalog;
- deterministic dependency-path discovery and rendering;
- one mandatory role dependency checker reused by every authoritative role
  validation entry point;
- the public `canic::cdk` hard cut;
- hidden CDK paths required only for exported macro expansion;
- migration of Canic's active canister, test, example, and documentation
  callers;
- direct-upstream guidance for generic IC libraries; and
- isolated package fixtures proving supported and rejected graphs.

0.97 does not:

- change fleet topology, `canic.toml`, `icp.yaml`, or role names;
- change `--environment`, backing-network, artifact-environment, or build-
  network behavior;
- alter the Canic feature catalog or role capability requirements;
- change Candid, stable-memory IDs, persisted encodings, lifecycle ordering,
  authorization policy, or runtime behavior;
- create a new standalone `canic-cdk` package;
- introduce a generic dependency-injection framework;
- create compatibility aliases for `canic::cdk`;
- edit Toko or another downstream repository; or
- change Cargo package versions during design or implementation slices.

## Current Implementation Reality

Canic 0.84 introduced the current role-package contract:

- one direct normal dependency on package `canic`;
- unconditional and non-optional selection;
- features declared directly on that dependency;
- no transitive normal path to `canic`; and
- no multiple Canic package IDs or versions.

The current contract permits a renamed Canic dependency and derives direct
feature evidence from the role package's dependency declaration. It does not
prove that inherited workspace features are empty, require disabled Canic
defaults, or freeze the workspace resolver. 0.97 hard-cuts those ambiguities.

The current host validator walks normal Cargo metadata edges. It rejects a
transitive route when another direct dependency can reach the selected Canic
package, but it returns only a generic reason such as:

```text
a transitive normal runtime dependency also contributes Canic
```

It does not retain the package chain that proved the failure. It also protects
the `canic` package name specifically; a sibling dependency reaching
`canic-core` or another framework package is not expressed as the same
role-ownership violation.

The current facade crate publicly re-exports `canic_core::cdk` as
`canic::cdk`. That surface combines:

- upstream `candid` and `ic-cdk` re-exports;
- endpoint and lifecycle attributes;
- stable-structure access;
- Canic serialization and hash helpers;
- Canic-owned types and standard DTOs; and
- paths used by declarative and procedural macro expansion.

The standalone `canic-cdk` package was already removed in 0.43.3. This design
does not restore it.

Canic already has a hidden `canic::__internal` module for macro expansion.
Some macros use that boundary, while other declarative and procedural macros
still generate public `canic::cdk` paths. The public facade is therefore doing
two jobs: supported human API and compiler plumbing. 0.97 separates them
before removing the human-facing path.

## Ownership Rules

The final role crate owns its Canic dependency/features, lifecycle and
endpoint macros, auth adapters, Canic conversions, logging, timers, storage,
control-plane calls, and concrete domain interfaces. Its dependency key is
exactly `canic`; aliases such as `framework = { package = "canic", ... }` are
rejected.

Shared domain crates own entities, rules, calculations, and pure decisions.
They receive authenticated subjects, absolute timestamps, authorization
decisions, and narrow storage inputs from the role instead of reading runtime
globals:

```text
role endpoint authenticates
  -> role adapter reads runtime state
  -> domain input/interface
  -> pure domain decision
  -> role adapter performs the effect
```

Generic IC adapters may use `candid`, `ic-cdk`, timers, and stable-structure
types/algorithms. They may not depend on Canic, select Canic features, own
lifecycle/auth policy, choose a `MemoryId`, open persistent memory, or register
a Canic allocation. Persistent storage is supplied through a role-owned
virtual-memory/capability handle or interface.

Canic-specific integration is a module inside the final role crate, not a
normal helper dependency. A `toko-canic -> canic` package would recreate the
forbidden second path. Mechanically shared source must remain Canic-free and
cannot select role-defining features.

## Protected Framework Package Catalog

0.97 introduces one host-owned catalog for packages that cannot enter through
a sibling normal dependency subtree. The initial catalog is:

| Package | Reason |
| --- | --- |
| `canic` | Public runtime facade and role feature authority. |
| `canic-core` | Runtime, state allocation, lifecycle, authentication, and feature implementation. |
| `canic-control-plane` | Root and Wasm-store runtime behavior selected by Canic features. |
| `canic-macros` | Forbidden compile-time framework coupling outside the role; it is protected for ownership, not because proc-macro code enters the Wasm runtime. |

The catalog is not inferred from a name prefix. Each admitted package has an
explicit owner and reason, so host-only packages such as `canic-cli`,
`canic-host`, and `canic-backup` are not accidentally classified as Wasm
runtime dependencies.

The production constant catalog is the downstream validator authority.
Ordinary consumer validation neither infers new Canic packages from names nor
requires optional catalog packages to be present when their features are
inactive.

A Canic-repository-only release test computes the normal package closure below
`canic` across the supported Canic feature fixtures. Every Canic-owned package
in that closure must be either:

- listed in the protected catalog; or
- explicitly classified by the release test as an allowed Canic-internal
  implementation dependency.

That release test rejects a new unclassified Canic-owned package or a stale
classification. It is not part of downstream role graph validation.

## Canonical Role Cargo Graph Evidence

All validators consume one passive `RoleCargoGraphEvidence` value produced by
one host-owned Cargo evidence function. Callers cannot invoke Cargo metadata
themselves and pass differently resolved graphs into the common checker.

The evidence request freezes:

- exact role manifest path and package identity;
- target `wasm32-unknown-unknown`;
- Cargo metadata format version 1;
- role package defaults enabled and no extra `--features` or
  `--all-features`, exactly matching the maintained role build;
- lock/offline mode matching the caller's promised resolution fidelity; and
- the top-level workspace/package resolver and raw role/workspace Canic
  dependency declarations.

The canonical metadata command shape is:

```text
cargo metadata
  --format-version=1
  --manifest-path <exact-role-manifest>
  --filter-platform wasm32-unknown-unknown
  [--locked]
  [--offline]
```

The exact role manifest selects one current package. A workspace-wide metadata
snapshot is not reused as role evidence. The subsequent Cargo build uses the
same manifest, target, role feature/default selection, and lock/offline policy:

```text
cargo build
  --manifest-path <exact-role-manifest>
  --target wasm32-unknown-unknown
  [--locked]
  [--offline]
```

The Wasm runtime graph contains only `resolve.nodes[*].deps` entries whose
dependency kind is normal. The `packages` array remains unfiltered manifest
evidence and is never mistaken for the selected runtime graph. Inactive
target-specific dependencies excluded by `--filter-platform` do not enter
runtime reachability.

Build dependencies and proc-macro/host evidence are retained separately. They
may prove the permitted `canic::build!` edge or hidden macro plumbing, but they
cannot contribute Canic runtime features under resolver 2.

`RoleCargoGraphEvidence` contains normalized package/edge facts, the selected
normal Canic features, and resolver/default-feature declarations. The checker
may derive `DependencyPathEvidence` from that graph. Raw Cargo package IDs
remain private graph identities.

## Mandatory Role Dependency Checker

The dependency checker is a required product boundary, not optional Medic
advice. One implementation validates every configured role package before
Canic trusts its feature or capability contract.

For each role, the checker must prove all of the following:

1. the top-level Cargo resolver is explicitly version 2; a missing resolver,
   resolver 1, and every unreviewed resolver version are rejected;
2. `[workspace.dependencies].canic`, when used, declares
   `default-features = false` and no Canic features;
3. exactly one direct **normal** dependency key is named `canic` and resolves
   to package `canic`; dependency renaming is rejected;
4. the normal dependency is unconditional, non-optional, and has Canic
   defaults disabled either directly or through the workspace declaration;
5. the role's normal dependency has an explicit `features` key (possibly
   empty), every selected Canic feature is listed there, and the role forwards
   no package feature into `canic/*`;
6. a separate build dependency is permitted only for the role's
   `canic::build!` build script, is not counted as a normal edge, and selects no
   Canic runtime feature;
7. the resolved Canic version matches the running Canic toolchain;
8. no sibling normal dependency subtree reaches a protected Canic framework
   package; and
9. no second Canic package ID or version is reachable.

The same checker runs through:

- `canic build` before Cargo compilation;
- deploy/install paths before build or network mutation;
- `canic medic project --ci` and `--json` as the explicit developer
  preflight;
- state manifest and state audit resolution;
- internal PocketIC role-package builds; and
- release/package proofs that validate configured role packages.

These callers do not implement local graph rules or reinterpret partial
results. They consume one supported result or one blocking typed finding.

The normal developer workflow is:

```text
edit Cargo.toml
  -> run canic medic project --ci
  -> fix the exact reported dependency path
  -> run canic build <fleet> <role>
```

`canic build` always performs the check itself, so skipping Medic cannot admit
an unsupported graph. Medic exists to give developers the same failure sooner
and across all roles in one read-only command.

## Dependency-Path Diagnostics

The current boolean reachability query becomes a deterministic path query.
For every sibling direct normal edge, the validator performs a bounded graph
search over normal edges and retains predecessor information until it reaches
a protected framework package.

The result is a structured, internal `DependencyPathEvidence`, not a formatted
string. Each step retains the dependency alias, package name, version,
normalized source kind, and workspace-relative package path when needed to
disambiguate local packages. Rendering converts that evidence into the
existing finding reason and Medic JSON detail without changing their machine
code or JSON shape.

Path selection is deterministic:

1. shortest edge count wins;
2. ties use dependency alias, package name, version, normalized source kind,
   and then workspace-relative package path; and
3. renamed non-Canic dependencies render as
   `alias (package name version)` when the alias differs from the package name.

Cargo package IDs remain valid internal keys for visited-node identity. They
are not an ordering or display contract because their source representation
may contain checkout-specific paths.

The diagnostic omits absolute manifest and registry paths. A representative
failure is:

```text
error[role_contract_dependency_shape_unsupported]
role `root-canister` must be the only package selecting Canic

found unsupported path:
  root-canister -> toko -> canic

expected:
  root-canister -> canic
  root-canister -> toko-domain

fix:
  1. remove `canic` from `toko/Cargo.toml`
  2. move Canic-specific code into `root-canister`
  3. use `candid` or `ic-cdk` directly for generic IC code
```

An alternate framework entry reports its actual target:

```text
role package `root-canister` reaches protected package `canic-core` outside
its direct Canic subtree:
root-canister -> toko-runtime -> canic-core
```

The example wording is illustrative; Slice B freezes the evidence fields and
meaning, while presentation wording may be refined without changing the
contract.

The stable machine code remains
`role_contract_dependency_shape_unsupported`. The existing typed
`RoleContractFinding::DependencyShapeUnsupported` remains the classification;
the renderer places the structured path into its reason. Multiple resolved
Canic package IDs retain `role_contract_multiple_canic_packages` and carry
the same structured path evidence to the rendering boundary.

## Public CDK Surface

0.97 unconditionally removes `pub use canic_core::cdk` from the `canic`
facade. Before deletion, every exported item and active caller is classified
so the hard cut has a clear migration rather than unexplained breakage:

| Classification | 0.97 disposition |
| --- | --- |
| Upstream-owned type, function, attribute, timer, or stable structure | Consumer imports it from the upstream crate. |
| Required only by Canic macro expansion | Re-export narrowly through `#[doc(hidden)] canic::__internal::cdk`. |
| Canic-owned and intentionally public | Move to its semantic `api`, `dto`, `ids`, `memory`, or other owning module. |
| Internal-only helper | Keep under `canic-core` ownership without a `canic::cdk` public path. |
| Unused or convenience-only surface | Remove completely. |

Hard-cut completion requires all of the following in the same release:

- delete the `canic::cdk` re-export and its public rustdoc module;
- remove generic CDK re-exports from `canic::prelude`;
- migrate maintained source, tests, examples, and current documentation;
- expose macro-only items solely through `#[doc(hidden)] canic::__internal`;
- reject active non-historical `canic::cdk` references with a source guard;
- provide no feature flag, deprecated module, alias, forwarding macro, or
  fallback path; and
- let old human-written imports fail at compile time with the documented
  upstream or semantic-owner migration.

The classification must not create a replacement catch-all such as
`canic::utils` merely to preserve old imports. A Canic-owned item moves only
when a maintained consumer and semantic owner are demonstrated.

The public prelude is part of the audit. Generic CDK items currently re-
exported through the prelude, including caller/self helpers and Candid derives,
must be imported from their upstream owner unless a separate Canic semantic
API intentionally owns them.

`canic-core::cdk` remains Canic's internal implementation substrate in 0.97;
it is not the replacement public facade. Its Rust visibility supports the
published Canic crate graph and is not permission for application or shared
role dependencies. The strengthened package validator rejects those alternate
paths.

## Hidden Macro Plumbing

Canic uses the existing hidden macro namespace rather than adding a competing
`__private` owner:

```rust
#[doc(hidden)]
pub mod __internal {
    pub mod cdk {
        pub use canic_core::cdk::{init, post_upgrade, query, update};
        // Only items required by emitted Canic code are re-exported.
    }
}
```

Declarative macros resolve definition-owned items through `$crate`:

```rust
#[macro_export]
macro_rules! example {
    () => {
        #[$crate::__internal::cdk::query]
        fn generated_query() {}
    };
}
```

`$crate` identifies the crate defining a declarative macro, but it does not
bypass Rust visibility. Items required by cross-crate expansion therefore
remain technically public and `#[doc(hidden)]`; they are compiler plumbing,
not supported human API.

Procedural macro output cannot use declarative `$crate` syntax. Because the
canonical dependency key is now required to be exactly `canic`,
`canic-macros` emits the unambiguous hidden path
`::canic::__internal::cdk`. A renamed Canic dependency fails role validation
before macro expansion. Isolated consumer compilation proves both the
canonical positive case and renamed-dependency rejection.

## Layering And Ownership

| Concern | Owner |
| --- | --- |
| Production protected framework package catalog | `canic-host` role-package validation |
| Repository-only catalog closure/drift proof | Canic release test |
| Canonical Cargo metadata invocation and `RoleCargoGraphEvidence` | `canic-host` package evidence |
| Structured `DependencyPathEvidence` | `canic-host` package evidence |
| Stable finding codes and typed classification | `canic-core` role-contract model |
| Human diagnostic rendering | `canic-host` role-contract presentation |
| Hidden declarative macro paths | `canic` facade `__internal` boundary |
| Procedural macro emitted paths | `canic-macros` |
| Runtime APIs and Canic-owned types | Existing semantic `canic`/`canic-core` owners |
| Generic IC integration | Role crate or Canic-free IC adapter using upstream crates |
| Domain policy | Framework-independent downstream domain crate |

The host evidence producer invokes Cargo once per exact role and returns
passive normalized evidence. The checker does not mutate manifests or rerun
Cargo. Core owns the typed finding code; host presentation renders the
structured path.

## Public Hard Cut And Migration

The public Rust hard cut is source-breaking but runtime-neutral.

| Removed use | Maintained migration |
| --- | --- |
| `canic::cdk::candid::*` | Depend on and import `candid` directly. |
| `canic::cdk::{query, update}` | Prefer Canic's semantic endpoint macros for maintained authenticated endpoints. Use raw `ic_cdk` attributes only for an intentionally handwritten endpoint whose explicit authentication and delegation order is proved. |
| `canic::cdk::{init, post_upgrade}` | Use Canic's maintained lifecycle owner (`start!`) for role lifecycle. Generated Canic lifecycle code uses hidden plumbing; raw lifecycle attributes are not a migration shortcut. |
| `canic::cdk::api::*` | Import the upstream `ic_cdk::api` item, or use a semantic Canic API where one deliberately exists. |
| `canic::cdk::structures::*` | Import `ic-stable-structures` directly unless a Canic memory API owns the operation. |
| `canic::cdk::types::Principal` | Import `candid::Principal`. |
| Canic-owned `cdk::types`, `spec`, `serialize`, or `utils` item | Move to its demonstrated semantic owner or remove it; no forwarding alias remains. |

Active Canic canisters, fixtures, tests, examples, and documentation migrate
in the same release. Historical changelog and archived design text are not
rewritten.

No Candid, stable record, state snapshot, fleet configuration, or environment
migration is required.

## Security Requirements

- Auth remains enforced by role endpoints before shared domain code runs.
- Caller, subject, audience, subnet, parent, and role bindings remain explicit
  inputs at their owning boundary.
- A generic IC adapter cannot make a Canic authorization decision.
- A handwritten raw `ic_cdk` endpoint must authenticate at its entry boundary
  and delegate immediately; maintained Canic-owned endpoint surfaces use the
  semantic Canic macro owner.
- A shared stable adapter cannot select or open persistent memory; role-owned
  allocation evidence supplies its memory/capability handle.
- Hidden macro paths are not a security boundary and are never relied on to
  prevent direct use.
- The dependency validator fails closed when Cargo graph evidence is missing,
  ambiguous, cyclic, or inconsistent with the protected package catalog.
- Diagnostics never expose absolute local paths or registry credentials.

## Implementation Slices

### Slice A: Surface And Graph Inventory

1. Freeze every active `canic::cdk` export and consumer.
2. Classify each item by upstream, macro-only, Canic-owned public, internal,
   or removable ownership.
3. Inventory every role's workspace, normal, build, and feature declaration,
   including why `canic::build!` needs a build edge.
4. Freeze the protected framework package catalog, supported Canic feature
   fixtures, and current canonical Canic subtree.
5. Inventory every build, deploy, Medic, state, internal-build, and release
   caller that currently requests Cargo metadata.
6. Add isolated positive and rejection Cargo fixtures before changing
   validation or public paths.
7. Record every semantic relocation before implementation.

Slice A is audit-only. It determines the destination of maintained items but
cannot retain `canic::cdk`. A genuinely Canic-owned public item moves to one
semantic owner; upstream convenience imports move to upstream crates; unused
items are deleted.

### Slice B: Canonical Cargo Evidence, Enforcement, And Diagnostics

1. Add the one canonical `RoleCargoGraphEvidence` producer and route every
   authoritative caller through it.
2. Enforce resolver 2, empty workspace Canic features, disabled Canic
   defaults, explicit role features, canonical dependency key `canic`, and the
   separately permitted build edge.
3. Add structured deterministic `DependencyPathEvidence` over normal Wasm
   edges and reject sibling paths to every protected framework package.
4. Render the exact offending chain while preserving existing machine codes
   and JSON shape.
5. Add the Canic-repository-only catalog closure/drift test.
6. Preserve build/dev separation and reject renamed Canic dependencies while
   retaining readable aliases for other packages.

### Slice C: Hidden Macro Boundary

1. Add only the CDK exports required by macro expansion to
   `canic::__internal`.
2. Route declarative macros through `$crate::__internal`.
3. Route procedural macro output through the hidden Canic path.
4. Prove isolated consumers do not require the human-facing `canic::cdk`
   module for generated Canic code.
5. Remove public-prelude reliance on generic CDK re-exports.

### Slice D: Public Facade Hard Cut And Closeout

1. Migrate active repository consumers to upstream or semantic owners.
2. Remove the public `canic::cdk` re-export.
3. Update maintained public documentation and examples.
4. Run targeted package, macro, role-contract, rustdoc, and isolated package
   proofs.
5. Record downstream migration instructions without editing downstream
   repositories.

## Validation Plan

### Cargo contract and evidence

- explicit resolver 2 is accepted; resolver 1, a missing resolver, and an
  unreviewed resolver version are rejected;
- workspace-inherited Canic with disabled defaults and no workspace features
  is accepted;

## Risks And Mitigations

| Risk | Mitigation |
| --- | --- |
| Workspace/default features bypass role ownership | Read raw declarations; require resolver 2, disabled defaults, empty workspace features, and explicit role feature roots. |
| Callers resolve different Cargo graphs | Only one host producer invokes metadata with frozen role, target, feature/default, kind, and fidelity inputs. |
| Renamed Canic breaks procedural output | Reject every dependency key except `canic` before expansion. |
| Facade removal creates unexplained churn | Inventory every export; move Canic-owned APIs semantically and upstream conveniences upstream. |
| Hidden plumbing becomes public API | Export only inventory-backed hidden items; omit them from docs and guard the source inventory. |
| Validator misses `canic-core` | Check the explicit protected framework catalog, not only package `canic`. |
| Ordinary IC helpers are rejected | Keep the catalog explicit; upstream IC packages remain allowed. |
| Path output changes by checkout | Use stable display keys; shuffle metadata order and checkout roots in tests. |
| Adapter allocates untracked stable memory | Forbid adapter-owned memory IDs/opening; require role-owned handles and structural checks. |
| Raw endpoint weakens authentication | Prefer semantic Canic endpoints; require direct auth-order proof for any raw endpoint. |
| Shared Canic helper recreates the diamond | Keep Canic integration inside the role; shared packages remain Canic-free. |
| Facade removal is mistaken for enforcement | Keep the graph validator as the non-bypassable authority. |

## Closed Decisions

- The final role package owns Canic runtime features.
- The top-level project explicitly uses resolver 2.
- Workspace Canic defaults are disabled and workspace Canic features are
  empty.
- Every selected Canic feature root is declared on the role's normal edge.

  is separate and selects no runtime feature.

  not solve the diamond.
- One host-owned `RoleCargoGraphEvidence` producer freezes role, Wasm target,
  feature/default selection, dependency kinds, and resolution-fidelity mode.
- Shared domain and generic IC adapter crates do not depend on protected Canic
  packages.

- Declarative macros use `$crate`; procedural macros use the tested hidden
  Canic path.
- Public `canic::cdk` is unconditionally removed; its inventory determines
  destinations, not whether the facade survives.
- The dependency validator is retained and expanded.
- Every authoritative role-validation entry point uses that one checker, and
  build/deploy cannot bypass it.
- Dependency chains remain structured until rendering, and raw package IDs
  are never an ordering/display contract.
- Consumer validation uses the constant protected catalog; only Canic's
  repository release test audits catalog closure and drift.
- Existing finding machine codes are preserved.
- No fleet, environment, Candid, stable-state, or runtime behavior change is
  part of this design.
