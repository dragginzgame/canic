# Canic 0.38 Stable Memory ABI

## Status

Draft started: 2026-05-16

0.38 is the stable-memory ABI hardening minor. It is a deliberate hard ABI cut.
Backward compatibility with implicit owner/label identity is not a goal.

Stable memory IDs are durable canister disk layout. Treat them like filesystem
block ownership, database page ownership, or protocol field numbers. A memory
ID that ever held one logical store must not silently become another logical
store.

## Core Position

The persisted ledger is authoritative. Runtime declarations are claims against
the ledger.

The only allocation identity is:

```text
stable_key -> memory_id forever
```

Owner crate, package name, Rust type path, module path, and local label are
metadata only. They may drift. Allocation identity must not drift.

The ledger itself is also an ABI. Its header, schema version, generation commit
rules, tombstones, reservations, and historical ownership records are durable
storage-engine metadata, not a runtime convenience registry.

## Goals

- Define a durable stable-memory ABI model now, not piecemeal.
- Reserve `0-99` globally and unconditionally for Canic.
- Give applications one clear safe range: `100-254`.
- Define MemoryManager virtual ID validity as `0-254` only.
- Keep ID `255` permanently invalid as the unallocated-bucket sentinel.
- Persist all range and memory ID history in stable memory.
- Distinguish never-allocated, reserved, active, and retired IDs.
- Keep retired IDs permanently poisoned.
- Validate runtime declarations against the ledger before endpoint execution.
- Prevent Canic-managed virtual memories other than the ledger from opening
  before ABI validation succeeds.
- Prevent endpoint dispatch and application/framework state mutation before ABI
  validation succeeds.
- Require explicit stable keys for every Canic-managed stable memory,
  including framework and application memories.
- Separate allocation ABI from schema ABI.
- Reserve space for future allocator, migration, snapshot, and generation
  metadata without implementing dynamic allocation in 0.38.

## Non-Goals

- No compatibility shim for owner/label-derived identity.
- No automatic data migration between memory IDs.
- No app dynamic allocation in 0.38.
- No reuse of retired IDs, ever.
- No "best effort" bootstrap if the ledger detects ABI drift.
- No profile-dependent app ranges.
- No silent ledger schema upgrade during normal bootstrap.

## Global Memory Map

This map is unconditional for Canic-managed canisters:

| Range | Owner | Meaning |
| --- | --- | --- |
| `0` | `canic-memory` | persisted ABI ledger |
| `1-4` | `canic-memory` | metadata expansion budget |
| `5-10` | `canic-control-plane` | root / wasm-store control-plane state |
| `11-99` | `canic-core` | core runtime state and future framework growth |
| `100-254` | applications | downstream explicit memory IDs |
| `255` | none | unallocated-bucket sentinel, permanently invalid |

ID `0` has a canonical self-record:

```text
0 canic.memory.abi_ledger.v1 active
```

The ledger records itself. This is intentional bootstrap metadata, not an
exception to the ABI model.

MemoryManager virtual IDs are valid only for `0-254`. ID `255` is the
unallocated-bucket sentinel value. It is not a usable virtual memory ID for
applications, framework code, diagnostics, migration tooling, or future
allocators. Any attempt to claim ID `255` is a fatal ABI violation.

The policy is permanent:

```text
Canic framework code: 0-99 only
applications: 100-254 only
255: invalid forever
```

The hard external ABI boundary is Canic `0-99` versus applications `100-254`.
The internal Canic subdivisions are framework planning authorities, not
allocation identity. Movement of diagnostic owner metadata within `0-99` is
allowed when the stable key and memory ID remain unchanged.

The Canic budget is:

| Range | Budget |
| --- | --- |
| `0` | ABI ledger |
| `1-4` | ledger-adjacent metadata expansion budget |
| `5-10` | root / wasm-store control-plane state |
| `11-20` | current core runtime stores |
| `21-39` | foreseeable runtime growth |
| `40-59` | migration, snapshot, export, and replay expansion |
| `60-79` | emergency framework reserve |
| `80-99` | long-horizon Canic reserve |
| `100-254` | application-managed explicit stable memories |

Starting application memory IDs at `100` keeps the external contract easy to
remember while reserving enough framework space that a future Canic storage
feature should not require a major ABI boundary change.

Applications and framework code may only use MemoryManager virtual IDs in
`0-254`. ID `255` is permanently invalid.

There is no full-stack versus standalone app range distinction in the Canic
runtime contract. Applications should use `100-254`.

Standalone `canic-memory` consumers that do not use the Canic runtime may choose
their own policy, but Canic framework code and Canic documentation should teach
`100-254` as the safe application range and `255` as invalid.

## Ledger Authority

The ledger is stored at stable memory ID `0`. It is the historical ABI record.

Runtime declarations are not the source of truth. They are current-binary claims
that must be checked against the ledger:

```text
declaration: "I claim stable_key X is memory_id Y"
ledger:      "stable_key X is memory_id Y forever"
```

If the declaration matches the ledger, bootstrap may proceed. If the ledger has
never seen the key or ID, the declaration may allocate a new active record,
subject to range and reservation rules. If the declaration conflicts with
history, bootstrap must fail closed.

The ledger must never be reconstructed from current declarations. Current
declarations can be incomplete, renamed, removed, or reordered. Historical ABI
state must survive those changes.

0.38 does not infer historical ownership from pre-0.38 declarations. Current
declarations can initialize records only in a valid genesis ledger or through a
separate audited migration tool that supplies a full historical manifest.
No historical ownership mutation is inferred from the current binary alone.

Canic assumes a single MemoryManager owns the underlying raw stable memory
space. That is an ABI contract. Canic-managed stores, framework code,
application declarations, diagnostics, and future allocation tooling must all
interpret virtual IDs through that one MemoryManager ownership model. A second
independent MemoryManager over the same raw stable memory is outside the Canic
ABI and risks page-map corruption.

Canic-managed code must not use raw stable-memory APIs, `RestrictedMemory`
carve-outs, or independent MemoryManager instances over the same stable memory
unless a future Canic ABI explicitly reserves and validates that layout. Direct
raw stable-memory access by framework or application code is outside the Canic
ABI and can corrupt the MemoryManager page map even when virtual ID ranges are
otherwise respected.

## Genesis And Existing State

Bootstrap must classify the underlying stable memory state before creating a
ledger:

```text
brand-new canister: no MemoryManager state and no ledger
recognized Canic genesis: MemoryManager state exists only for bootstrap metadata
pre-0.38 Canic state: MemoryManager state exists but no 0.38 ledger exists
foreign/corrupt state: unrecognized or invalid stable memory contents
```

If no ledger exists and the underlying stable memory is otherwise empty or in a
recognized Canic genesis state, bootstrap may initialize the ledger. Genesis
initialization creates the header, the canonical ID `0` self-record, reserved
framework records such as IDs `1-4`, and the first committed generation.

If no ledger exists but stable memory contains non-genesis data, bootstrap must
fail closed unless an explicit controller-initiated destructive format or
audited manifest-based migration command has been provided.

Deploying a 0.38 binary to an existing canister must not silently create a new
ledger that blesses the current binary's declarations. That would convert
current source shape into false history and is a storage ABI violation.

0.38.0 supports genesis initialization and explicit destructive format only.
Adoption of pre-0.38 stable memory into a new ledger is out of 0.38.0 scope
unless a separate audited migration tool supplies a full historical manifest.
Adoption must never infer ownership from current declarations alone.

Foreign or corrupt stable memory state must fail closed. Repair or formatting
must be explicit, controller-initiated, and operator-visible.

## Destructive Format Semantics

A destructive format must destroy the prior Canic stable-memory universe, not
only the ABI ledger. Formatting only virtual memory ID `0` is insufficient
because old virtual-memory pages could later be interpreted as valid state
under new stable keys.

In 0.38.0, destructive format resets the entire underlying stable-memory layout
used by the single MemoryManager and then creates a fresh genesis ledger. No
historical ownership is preserved. No old application or framework store bytes
may remain reachable through the new ledger.

A format implementation must ensure that no old MemoryManager virtual-memory
pages, allocation tables, or store bytes can later be interpreted as valid
state under the new ledger.

## Canonical Genesis Records

Genesis initialization must create canonical range authority records:

```text
0-99     canic.framework   active   Canic framework allocation authority
100-254  applications      active   downstream application allocation authority
```

ID `255` has no range record. It is invalid outside the usable MemoryManager ID
space.

Genesis initialization must create canonical memory ID records:

```text
0 canic.memory.abi_ledger.v1 active
```

Range authority reservation and per-ID reservation are different.

The Canic framework range `0-99` is reserved as an allocation authority range.
This does not require every ID in `0-99` to have an individual memory ID record.
IDs in `0-99` that do not have individual memory ID records may still be
allocated only by Canic framework declarations.

A per-ID `reserved` record is stronger. It assigns a specific ID to a specific
future stable key or purpose before a store is active. Genesis should create a
per-ID reserved record only when the future stable key and purpose are concrete
enough to freeze as ABI.

IDs `1-4` are a `canic-memory` metadata expansion budget, not canonical per-ID
reserved records in 0.38.0. They may be `never_allocated` at the per-ID level,
but only Canic may ever allocate them because the range authority is Canic.
Activating one of these IDs requires an explicit ledger operation or future
design that creates the concrete stable key, purpose, and committed generation.

IDs `5-99` may also be `never_allocated` at the per-ID level. They remain under
Canic range authority.

For a per-ID reserved record, the reserved stable key is already the future
allocation identity. Transitioning `reserved -> active` must preserve both the
stable key and memory ID.

IDs in `100-254` may be allocated only by application declarations with
non-`canic.*` stable keys.

Canonical range authority records are validated on every bootstrap before
runtime declarations are accepted.

Range authority is determined from the stable-key namespace, not from
diagnostic owner metadata. `canic.*` stable keys are framework declarations and
must use IDs `0-99`. Non-`canic.*` stable keys are application declarations and
must use IDs `100-254`.

## Stable Identity

Stable keys are explicit ABI names. Examples:

```text
canic.memory.abi_ledger.v1
canic.control_plane.template_manifest.v1
canic.control_plane.template_chunk_payloads.v1
canic.core.auth_state.v1
canic.core.root_replay.v1
canic.core.cycle_tracker.v1
app.example.orders.v1
app.example.orders_by_customer.v1
```

The key namespace should be descriptive and versioned. Stable keys are
canonical ABI strings with a deliberately boring grammar:

```text
stable_key = segment("." segment)* ".v" nonzero_digit digit*
segment = lowercase_ascii_letter
          (lowercase_ascii_letter_or_digit_or_underscore)*
max length = 128 bytes
```

Validation rules:

- ASCII only.
- Lowercase only.
- No empty segments.
- No leading or trailing dots.
- No whitespace.
- No slashes.
- No hyphens.
- No Unicode normalization or case folding.
- Version suffix is required.
- Version number must be nonzero.

The `vN` suffix is the allocation ABI generation of that logical memory, not
the crate version.

The `canic.*` namespace is reserved for the framework. Applications must not
declare stable keys under:

```text
canic.*
canic.memory.*
canic.core.*
canic.control_plane.*
```

Applications should use an application-owned prefix that will remain stable
across crate and module renames. Recommended forms:

```text
app.<app_name>.<store_name>.vN
<org_or_product>.<app_name>.<store_name>.vN
```

The angle-bracket examples are placeholders. Actual stable-key segments must
use underscores instead of hyphens and must follow the canonical grammar.

Allowed metadata drift:

```text
same stable_key, same memory_id, owner changed
same stable_key, same memory_id, label changed
same stable_key, same memory_id, Rust type moved
same stable_key, same memory_id, Rust type renamed
same stable_key, same memory_id, package renamed
```

Fatal allocation drift:

```text
same stable_key, different memory_id
same memory_id, different stable_key
reserved memory_id claimed by an application
retired memory_id claimed by any stable_key
retired stable_key claimed by any memory_id
range overlaps a historical range owned by another allocation authority
app declaration claims any ID below 100
framework declaration claims any ID above 99
any declaration claims ID 255
application declaration uses canic.*
```

## Runtime Declaration Snapshot

The current runtime declaration snapshot must be internally unambiguous before
it is compared with the ledger.

Within a single runtime declaration snapshot, stable keys and memory IDs must
both be unique. Duplicate declarations are rejected before ledger mutation, even
if the duplicate declarations would individually match historical records.

Current-snapshot conflicts include:

```text
same stable_key declared twice
same memory_id declared twice
same stable_key and same ID declared twice with different metadata
same ID opened through two aliases
exact duplicate declaration repeated in two source locations
```

Exact duplicates are not treated as idempotent. They are rejected because they
can hide source ownership mistakes and create aliasing inside one binary.

## ID Lifecycle States

The ledger must distinguish:

```text
never_allocated
reserved
active
retired
```

`never_allocated` means the ID has no historical ABI meaning. It has no ledger
memory ID record.

`reserved` means the ID is permanently earmarked by the framework or allocator
plan even if no real store is attached yet. Reserved IDs already exist in the
ledger as records. They may not be claimed by applications, and they may not be
used for a different framework purpose.

`active` means a logical store is attached to the ID and the current or a
historical binary may open it under the recorded stable key.

`retired` means the ID had historical ABI meaning and is now permanently
poisoned. A retired ID is not available capacity. It must never be reused.

A previously active memory ID is not retired merely because the current binary
does not declare it. Omitted historical records remain historically owned and
active unless an explicit retirement declaration or explicit ledger operation
marks them retired. Absence may mean rollback, feature gating, partial
framework composition, refactor sequencing, or dead code removal. None of those
events poison the ID.

Retirement is a tombstone, not deletion. Tombstones are permanent because stable
memory contents may still exist, future rollback may reintroduce old binaries,
and tooling must be able to explain historical ownership.

Retirement is irreversible and should be treated as an operator-visible ABI event
because it prevents rollback to binaries that still declare the retired stable
key.

Retirement may occur only through an explicit ledger operation or explicit
retirement declaration that names both the stable key and memory ID. Retirement
must be generation-scoped and committed through the same atomic generation
protocol as allocations.

0.38.0 does not need to expose public retirement tooling unless required by a
framework migration.

Canonical genesis records define the active ID `0` self-record. Other Canic
IDs may remain `never_allocated` at the per-ID level while still protected by
Canic range authority.

Future framework features transition pre-reserved framework IDs from
`reserved -> active`. Pre-reserved framework IDs do not use
`never_allocated -> active`, because their ownership was already assigned by
the ledger at the time the reservation was created.

## Ledger Records

### Header

The ledger header should include:

```text
ledger_magic
ledger_format_id
ledger_schema_version
minimum_reader_schema_version
header_len
header_checksum
superblock_generation
layout_epoch
current_generation
committed_generation_pointer
header_extension_len
header_extension_bytes
created_by_canic_version
last_updated_by_canic_version
```

`ledger_magic` is a fixed byte sequence that identifies virtual memory ID `0`
as a Canic ABI ledger.

`ledger_format_id` identifies the physical header and generation commit format,
such as dual superblocks or append-only commit footers.

`ledger_schema_version` describes the serialization shape and interpretation
rules of the ledger itself. It is a storage ABI version.

`minimum_reader_schema_version` declares the oldest ledger reader that may
interpret the current ledger shape. Older binaries reading newer incompatible
ledger versions must fail closed.

`header_len` bounds the encoded header. `header_checksum` covers the header
fields that must be valid before records are decoded.

`header_checksum` covers the complete encoded header, including extension
bytes, with the checksum field itself zeroed or excluded according to the
physical format. Unknown extension bytes are still protected against corruption
even when they are not interpreted by the current reader.

`superblock_generation` is the physical commit generation used by protected
header slots when the implementation uses dual superblocks.

`layout_epoch` is a deliberate, human-reviewed ABI epoch. It is not a migration
counter and must not change automatically.

`current_generation` identifies the committed runtime layout snapshot only when
it is covered by the protected commit protocol. If it appears in an unprotected
logical header, it is diagnostic.

`committed_generation_pointer` is the pointer used to select the committed
generation after interrupted writes only when protected by the physical commit
protocol.

`header_extension_len` and `header_extension_bytes` reserve explicit header
extension space. Unknown extension bytes must be preserved by compatible
writers and ignored only when the ledger schema version declares that doing so
is safe.

The ledger header must begin with the fixed magic value and versioned physical
format marker. Bootstrap must reject virtual memory ID `0` as foreign or
corrupt unless the header magic, length, schema version, and checksum validate.

The protected commit record, not an unprotected header field, determines the
authoritative generation.

### Range Record

Range records should include:

```text
owner
start
end
status: active | reserved | retired
first_seen_version
last_seen_version
first_generation
last_generation
purpose
```

Ranges are allocation authority boundaries. They do not replace per-ID
stable-key records.

### Memory ID Record

Memory ID records should include:

```text
id
stable_key
owner
label
status: active | reserved | retired
first_seen_version
last_seen_version
first_generation
last_generation
schema_version
schema_fingerprint
purpose
```

Only `stable_key` and `id` are allocation identity. Owner, label, purpose, and
version fields are diagnostic metadata.

`schema_fingerprint` is stored in the ledger as metadata associated with the
memory allocation record. The logical store itself is not required to
understand, persist, or validate the fingerprint.

The memory ID record may expose latest schema metadata for convenience, but the
ledger must preserve enough per-generation declaration metadata to explain
which `schema_version` and `schema_fingerprint` each committed generation
declared.

## Ledger ABI Evolution

The ledger is a durable ABI. Bootstrap must validate the ledger header before
interpreting records beyond the minimum required header fields.

Compatibility rules:

- A binary may read a ledger when its reader supports `ledger_schema_version`
  and satisfies `minimum_reader_schema_version`.
- A binary that sees a newer incompatible `ledger_schema_version` must fail
  closed before opening any Canic-managed virtual memory other than the ledger.
- Ledger schema upgrades must be explicit and controller-initiated.
- Normal bootstrap must not silently rewrite ledger structure versions.
- A compatible writer must preserve reserved header extension bytes and unknown
  compatible extension sections.
- An incompatible writer must refuse to proceed unless an explicit ledger
  upgrade workflow selected that writer.

Forward compatibility is a design constraint. New ledger fields should be added
through reserved header extension space or versioned extension sections before
they become mandatory. Older compatible readers may ignore declared-optional
extensions, but they must not discard them on write.

Failure behavior is fail-closed. If header validation, schema compatibility,
extension parsing, checksum validation, generation pointer validation, or record
decoding cannot prove which generation is authoritative, bootstrap must stop
before endpoint dispatch.

## Layout Epoch Mechanics

`layout_epoch` is compiled into the binary as a constant owned by the Canic ABI
design for that release line.

Bootstrap validates the compiled epoch against the persisted ledger header:

```text
compiled_layout_epoch == ledger.layout_epoch
```

If the values differ, bootstrap fails closed unless an explicit
controller-initiated acknowledgement selected the new epoch and the associated
ledger upgrade or validation path. Epoch changes are operator-visible ABI
events.

`layout_epoch` is not an automatic migration counter. It must not increment
because code moved, because a store schema changed in place, or because a normal
generation was committed. It changes only when the durable layout contract
changes in a way that operators must deliberately acknowledge.

## Allocation ABI vs Schema ABI

0.38 must separate allocation ABI from schema ABI.

Allocation ABI answers:

```text
Which logical memory owns this stable memory ID?
```

Schema ABI answers:

```text
What data shape lives inside that logical memory?
```

The stable key version suffix, such as `v1` or `v2`, is the allocation ABI
generation. It changes only when the logical store needs a new memory
allocation.

`schema_version` is the declared version of the data format inside the same
allocation. It is owned by the logical store, not by the ledger.

In 0.38, `schema_version` is optional declaration metadata. If present, it must
be a nonzero unsigned integer. If omitted, the ledger records `None`.

Examples:

```text
Add optional field to auth record:
  stable_key = canic.core.auth_state.v1
  memory_id = 11
  schema_version 1 -> 2

Rewrite auth storage from one map to incompatible split stores:
  old stable_key = canic.core.auth_state.v1, memory_id = 11, retired later
  new stable_key = canic.core.auth_state.v2, memory_id = new ID
```

`schema_fingerprint` is a stable digest or descriptor of the store schema. It
is stored in the ledger as informational allocation metadata so tooling has a
place to report and compare it. The logical store itself is not required to
understand `schema_fingerprint`.

In 0.38, `schema_fingerprint` is optional opaque ASCII metadata with a maximum
length of 128 bytes. If omitted, the ledger records `None`. The recommended
format is:

```text
<algorithm>:<lowercase_hex_digest>
```

Example:

```text
sha256:abc123...
```

Bootstrap does not enforce algorithm semantics in 0.38, but it must enforce
the optional field's length and ASCII encoding.

For 0.38, schema fingerprints are informational, not fatal during bootstrap.
The fatal checks are allocation checks: key/ID/range/reservation/tombstone
drift. Schema enforcement can become stricter later once schema ownership and
migration hooks are fully designed.

Schema metadata changes are recorded as generation metadata. They are not
allocation drift when the stable key and memory ID remain unchanged. The ledger
should not overwrite the diagnostic history of prior declarations just because
the latest generation declared a new `schema_version` or `schema_fingerprint`.

Example:

```text
generation 10:
  stable_key = canic.core.auth_state.v1
  memory_id = 11
  schema_version = 1
  schema_fingerprint = aaa

generation 11:
  stable_key = canic.core.auth_state.v1
  memory_id = 11
  schema_version = 2
  schema_fingerprint = bbb
```

This is not an allocation conflict. Both declarations should remain visible in
generation history so diagnostics can explain what changed and when.

Possible future fingerprint sources include:

- build-time constants declared next to the store
- hashes of generated schema descriptors
- manual opaque fingerprints supplied by store owners

Future schema migrations should be explicit store-level workflows. They must
not move memory IDs as a side effect.

## Source API

Canic runtime mode requires explicit stable keys for every managed memory,
including application memories. The stable key is the allocation ABI, so the
runtime registration API must force it.

```rust
canic_memory::ic_memory_key!(
    key = "canic.core.auth_state.v1",
    ty = AuthState,
    id = AUTH_STATE_ID,
    schema_version = 1,
    schema_fingerprint = "sha256:...",
)
```

The `schema_version` and `schema_fingerprint` fields are optional in 0.38. If
omitted, the runtime declaration records `None` for that schema metadata.

Framework and application code running under the Canic runtime bootstrap
contract must not use implicit owner/label-derived identity.

Accepted Canic runtime registration API:

```rust
MemoryApi::register_with_key(id, owner, label, stable_key, schema_metadata)
```

Legacy implicit-key APIs are not accepted for Canic-managed canisters:

```rust
ic_memory!(Type, ID)
MemoryApi::register(id, owner, label)
```

They may remain only for standalone `canic-memory` consumers outside the Canic
runtime bootstrap contract. They should be renamed or feature-gated so they are
not accidentally used in Canic-managed code, for example:

```text
ic_memory_legacy!(...)
MemoryApi::register_legacy(...)
feature = "legacy-implicit-stable-keys"
```

For Canic-managed canisters, `register_with_key` is the only accepted runtime
registration path.

## Declaration Collection vs Memory Opening

Stable memory declaration and stable memory opening are separate phases.

`ic_memory_key!` and `MemoryApi::register_with_key(...)` must create or
register declaration descriptors only. They must not open the underlying
MemoryManager virtual memory before bootstrap validation succeeds.

Bootstrap first collects and seals the runtime declaration snapshot. The sealed
snapshot is validated against the ledger. Only after validation succeeds and
the runtime registry snapshot is published may framework or application stores
obtain handles to their declared virtual memories.

In 0.38, runtime registration is not a dynamic allocation API. After the
declaration snapshot is sealed, additional `register_with_key(...)` calls are
rejected unless a future explicit ledger operation defines otherwise. Endpoint
logic must not allocate new memory IDs.

## Fail-Closed Bootstrap

Bootstrap must distinguish raw stable-memory metadata access from
Canic-managed virtual memory access.

MemoryManager initialization may touch raw stable memory metadata required by
`ic-stable-structures` to reconstruct the virtual-memory page map. That is
allowed before ABI validation because the MemoryManager page map is the
substrate needed to reach virtual memory ID `0`.

After MemoryManager initialization:

- no Canic-managed virtual memory except the ABI ledger at ID `0` may be opened
  before validation succeeds
- no raw stable-memory API, `RestrictedMemory` carve-out, or independent
  MemoryManager may be used unless an explicit Canic ABI reserved it
- no declaration macro or registration API may open a virtual memory while
  declarations are being collected
- no framework store may create or retain a handle to its own virtual memory
  before validation succeeds
- no endpoint dispatch may occur before validation succeeds
- no application or framework state mutation may occur before validation
  succeeds
- ledger validation and generation commit are the only allowed pre-dispatch
  Canic ABI mutations

Required ordering:

```text
initialize the single MemoryManager over raw stable memory
open only the ledger virtual memory at ID 0
read or classify the physical ledger header at ID 0
if no valid ledger header exists, classify genesis versus existing state
if genesis or explicit destructive format, initialize canonical ledger records
validate physical ledger header magic, format, length, and checksum
load ledger header and historical records
validate or initialize the canonical ID 0 self-record
validate ledger schema version and layout_epoch
validate committed generation pointer
reserve/validate 0-99 framework ranges
drain runtime range declarations as claims
drain runtime memory declarations as claims
reject duplicate keys, duplicate IDs, and aliasing in the current snapshot
seal the runtime declaration snapshot
validate every claim against historical ledger records
reject all fatal drift
stage a new registry snapshot generation
atomically commit the generation pointer
publish runtime registry snapshot
allow other stable memories to open
allow endpoint execution
```

Opening ID `0` is the only allowed pre-validation Canic-managed virtual memory
open. It is the hardcoded bootstrap root of validation. Once opened, the ledger
must contain or initialize the canonical active self-record for ID `0` before
any other memory ID claim is accepted. Initialization is allowed only for
brand-new or recognized Canic genesis state unless an explicit
controller-initiated operation selected a destructive format or audited
manifest-based migration path.

No framework store should open its own stable memory until its declaration has
been validated against the ledger. If this requires delaying memory handle
creation behind bootstrap, that is part of 0.38's correctness work.

## Generation Semantics

The ledger must model generations explicitly.

A generation is a validated runtime layout snapshot. It records that a specific
binary declared a specific set of ranges and stable-key mappings and that those
claims passed ledger validation.

A new generation is committed during install, upgrade, or an explicit ledger
operation when the validated runtime declaration snapshot differs from the last
committed generation or when the binary/runtime identity should be recorded for
audit. Normal endpoint entry must not create new generations.

An identical successful bootstrap may reuse the prior generation unless an
implementation intentionally records a new audit generation. Such audit
generations still require the same protected commit protocol.

Generation fields support:

- interrupted upgrade diagnosis
- rollback diagnosis
- detecting whether a binary reached post-validation commit
- explaining when a memory was first or last seen
- future migration journal correlation

Stable memory writes are not automatically rolled back on trap. Ledger
generation commits must therefore be atomic by design.

Ledger record mutations must be generation-scoped. A new stable key, memory ID
record, range record, tombstone, reservation, or metadata update becomes
authoritative only when its containing generation commits. Implementations must
not update authoritative records in place before the generation commit point.

An implementation may use stable-structures collections internally, but the
ledger must not behave like an ordinary mutable map of authoritative records.
Any map updates must be staged behind transaction framing so a trap cannot
publish some new ownership facts without the rest of the generation.

Required commit design:

- Write the candidate generation to a scratch or staging region first.
- Include enough length, checksum, schema version, and generation identity data
  to reject torn or partial staged writes.
- Keep the prior committed generation authoritative while staging occurs.
- Protect the authoritative generation pointer with a torn-write recovery
  protocol. The pointer must not be a single unprotected field.
- Commit only after the staged generation and its protected commit metadata are
  fully written and validated.
- Make the protected commit record the only point at which the new generation
  becomes authoritative.
- On bootstrap, recover by reading the protected commit metadata and ignoring
  incomplete staged generations.
- Preserve incomplete staged generation bytes for diagnostics when practical,
  but never treat them as allocation history.

Acceptable physical protocols include:

- Dual superblock/header slots. Each slot carries a generation number, pointer,
  length, checksum, and commit marker. The writer writes the inactive slot
  fully, verifies it, then marks it committed. The reader chooses the
  highest-generation valid committed header.
- Append-only generation records with a checksummed commit footer. The reader
  scans for the highest valid committed generation and ignores any partial
  trailing generation.

The authoritative generation pointer must itself be protected by one of these
recovery protocols or an equivalent design. A partially written pointer must
not be able to select a partially written generation.

Interrupted write expectation:

- If validation fails before generation staging, no new generation is recorded.
- If staging is interrupted before protected commit, the prior committed
  generation remains authoritative.
- If the protected commit completed, the new generation is authoritative even
  if later application startup fails.
- If protected commit metadata cannot prove one authoritative generation,
  bootstrap fails closed.

If validation succeeds but later application startup fails, the committed
generation remains useful evidence that the memory ABI was valid.

Generation commit records ABI validation, not application readiness.
Diagnostics must distinguish:

```text
abi_committed
runtime_ready
```

After generation commit, subsequent startup work must be infallible or
rollback-safe. Any fallible post-commit initialization must not require opening
or mutating stores that have not already passed ABI validation.

Recovery expectation:

- The ledger must be readable even if runtime registry reconstruction fails.
- Diagnostics should be able to report the last committed generation.
- Repair tooling may inspect the ledger, but must not mutate allocation history
  casually.

## Generation History Retention

Implementations may compact generation history only if they preserve all
authoritative allocation facts:

- stable-key-to-ID mappings
- tombstones
- reservations
- range authority history
- first and last generation metadata needed for diagnostics
- latest declaration metadata for every active, reserved, or retired record

Compaction must never reconstruct ownership from current declarations.

## Rollback Semantics

Rollback is validation against historical ownership, not a request to erase
history.

Older binaries may run against ledgers containing newer active, reserved, or
retired IDs when their declarations do not conflict with historical ownership.
Undeclared newer IDs remain historically owned and must be preserved. Their
absence from the older binary's runtime declarations is not retirement.

Rollback is allowed when:

- every stable key declared by the older binary maps to the same historical ID
- every ID declared by the older binary maps to the same historical stable key
- the older binary does not claim a reserved, retired, or historically owned ID
  for a different purpose
- the older binary does not claim an application/framework range it did not
  historically own

Rollback never permits ID reuse. It never erases tombstones. It never converts
historically owned IDs back to `never_allocated`. Generation history and
rollback metadata should remain visible for diagnostics even when current
runtime declarations omit newer IDs.

Rollback validation proves allocation ABI safety only. It does not guarantee
schema compatibility. A rolled-back binary may still be unable to safely read a
newer in-place store schema. Store-level schema compatibility, schema rollback,
and data migration are separate schema ABI concerns and must fail before store
mutation or runtime readiness if incompatible.

## Metadata Expansion Budget

IDs `1-4` are a small `canic-memory` metadata expansion budget. They are part
of the `0-99` Canic range authority, but they are not canonical per-ID reserved
records in 0.38.0.

0.38.0 commits to keeping these IDs unavailable to applications. It does not
commit to exactly four future metadata stores or to concrete stable keys for
those IDs. A future feature may activate one of these IDs only through an
explicit ledger operation that creates the per-ID record and commits it through
the generation protocol.

This avoids freezing vague future store identities while still leaving
ledger-adjacent metadata room outside ID `0`.

## Future Dynamic Allocation

0.38 should not implement dynamic allocation for applications.

Future dynamic allocation, if added, must still be based on:

```text
stable_key -> memory_id forever
```

The allocator may choose a new ID only for a never-allocated stable key. It must
never reclaim retired IDs. It must never compact by moving live IDs. It must
never allocate ID `255`.

## Threat Model

The following are catastrophic ABI violations:

### Same Stable Key, Different ID

This means the current binary is trying to open the logical store through a
different disk slot. The old data would be stranded, and the new slot may be
empty or contain unrelated data.

### Same ID, Different Stable Key

This means the current binary may interpret existing bytes as a different
logical store. That is direct state corruption risk.

### Reserved ID Misuse

Reserved does not mean free. It means "historically earmarked". Application use
of a reserved framework ID, or framework use of a reserved ID for a different
purpose, breaks the allocation plan before the real store is even attached.

### Retired ID Reuse

Retired does not mean free. It means "historically owned, no longer active".
Reusing it destroys the ability to roll back, audit, or safely inspect old
stable memory.

### ID 255 Claim

ID `255` is the unallocated-bucket sentinel. It is outside the usable virtual
ID space. Any claim on it means the binary is violating the numbering protocol.

### Range Overlap

Range overlap means two hard allocation authorities believe they may assign the
same ID. Even if no collision exists today, future additions become unsafe. In
0.38, the hard authority boundary is Canic framework `0-99` versus application
`100-254`; internal Canic subranges are planning labels unless a future ledger
schema promotes them to hard authorities.

### Runtime Declaration Drift After Persistence

Current declarations are claims, not history. If a declaration changes after
the ledger has persisted historical ownership, the change must be judged
against the ledger. It must not rewrite history.

## Enforcement

0.38 should add source, CI, and runtime guardrails. Grep examples are useful for
local investigation, but convention-only enforcement is not enough.

Required enforcement direction:

- Compile-time or CI enforcement for Canic-managed code.
- Canic-managed code may not invoke implicit identity macros.
- Canic-managed application code must use explicit stable-key declarations.
- Implicit-key APIs must be unavailable, renamed as legacy, or feature-gated
  outside Canic runtime mode.
- Declaration macros and registration APIs must not open virtual memory before
  bootstrap validation succeeds.
- Registration after the runtime declaration snapshot is sealed must fail in
  0.38 unless a future explicit ledger operation defines otherwise.
- Canic-managed code must not use raw stable-memory APIs, `RestrictedMemory`
  carve-outs, or independent MemoryManager instances unless an explicit Canic
  ABI reserves and validates that layout.
- Framework stable keys must be unique across the workspace.
- Framework stable keys must match accepted prefixes:
  `canic.core.*.vN`, `canic.control_plane.*.vN`, or `canic.memory.*.vN`.
- Application-owned declarations must not use `canic.*`.
- `canic.*` declarations must stay in `0-99`.
- Non-`canic.*` declarations must stay in `100-254`.
- Application-facing documentation must not tell full Canic apps to use IDs
  below `100` or ID `255`.
- Runtime declaration snapshots must reject duplicate stable keys, duplicate
  memory IDs, repeated exact declarations, and same-ID aliases before ledger
  mutation.
- Runtime validation must test key/ID drift, ID/key drift, reserved misuse,
  tombstones, range overlap, ID `255`, namespace violations, and owner/label
  metadata drift.

Preferred mechanisms:

- proc-macro validation for explicit stable-key declarations
- workspace validation that reads declarations from framework crates
- CI tests that exercise persisted-ledger conflict cases
- runtime bootstrap validation that is independent of source scans

Source scans may remain as developer diagnostics, but they are not the ABI
enforcement mechanism.

- owner, label, type, and package metadata drift succeeds
- duplicate key in current snapshot fails
- duplicate ID in current snapshot fails

- non-`canic.*` ID below `100` fails
- `canic.*` ID above `99` fails
- ID `255` fails

## Runtime Introspection

0.38.0 must include a minimal controller-only ledger diagnostic export capable
of reading virtual memory ID `0` without opening application or framework
memories. Rich formatting and extended tooling may come later.

This diagnostic path may initialize the MemoryManager and open virtual memory
ID `0`, but it must not open any other virtual memory or mutate allocation
history.

The controller-only ledger diagnostic endpoint must be available even when
normal Canic runtime bootstrap fails. It may run a restricted bootstrap path
that initializes the MemoryManager and opens only ID `0`. It must bypass normal
application endpoint dispatch and must not open or mutate application or
framework stores.

The output should include:

```text
ledger header
range records
memory ID records
current generation
generation history
runtime declaration snapshot
validation result
schema fingerprints
rollback-visible historical ownership
```

This is diagnostic output, not a second source of truth.

## Migration Philosophy

0.38 is the ABI-hardening release.

Compatibility shims for unsafe historical conventions are intentionally
avoided. There is no compatibility shim for implicit owner/label identity.
Explicit stable-key ABI is mandatory for every Canic-managed stable memory,
including framework and application memories.

Correctness and permanence are more important than preserving implicit
owner/label behavior. The hard cut is justified because future storage breaks
are much more expensive than a strict minor now.

The 0.38 design should be read as filesystem allocation rules, storage-engine
metadata rules, and protocol numbering rules. It is not a registry built for
runtime convenience.

## First Release Scope

0.38.0 should include the complete registry foundation:

- Ledger at memory ID `0`.
- Global Canic reservation `0-99`.
- Application range `100-254`.
- ID `255` invalid forever.
- Permanent reserved and tombstone model.
- Explicit stable-key identity.
- Reserved `canic.*` namespace.
- Schema metadata fields, stored but not fatal.
- Ledger schema version and compatibility checks.
- Layout epoch validation.
- Atomic staged generation commits.
- Rollback-safe historical ownership validation.
- Explicit stable-key declarations for every Canic-managed memory.
- Framework source, proc-macro, or workspace guardrails.
- Runtime fail-closed validation.
- Minimal controller-only ledger diagnostic export from ID `0`.

Do not split the core ABI model across multiple releases. Later 0.38 patches
may refine tooling, but the irreversible invariants should land in 0.38.0.
