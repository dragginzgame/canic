# 0.48 Design: Adoption Profiles

## Status

TBD.

This is a post-0.46 design stub. It captures adoption paths that are important
for real projects but should not expand the 0.41 through 0.46 deployment truth
foundation.

---

## Goal

0.48 makes Canic usable for brownfield, partial, standalone, leaf-only, hybrid,
and education-oriented projects without weakening deployment truth or authority
rules.

Core operator question:

```text
What Canic features am I adopting, which canisters are managed, and which
mutations are forbidden?
```

---

## Dependency On 0.41-0.47

0.48 depends on:

- 0.41 inventory, safety reports, role artifact manifests, and materialization
  checks;
- 0.42 authority reconciliation and control classification;
- 0.43 executor boundaries where adoption needs to inspect local project roots;
- 0.44 digest-pinned artifact overrides for external wasm roles;
- 0.45 external lifecycle classification;
- 0.46 deployment comparison;
- 0.47 registry metadata where artifact availability or retention matters.

Adoption begins with observation. Mutation remains blocked until inventory and
authority reconciliation prove it is safe.

---

## Core Decision

Adoption profiles are wrappers around the truth and authority model.

They do not bypass inventory, safety reports, or authority reconciliation.

---

## Adoption Profiles

Initial profiles:

- `Brownfield`: existing IC canisters are mapped into a Canic deployment.
- `PartialFleet`: only some roles are Canic-managed.
- `Standalone`: one production canister uses Canic runtime without root fleet
  control plane.
- `LeafOnly`: leaf roles adopt Canic runtime features without root ownership.
- `HybridExternalWasm`: some roles are Motoko, legacy wasm, partner canisters,
  or digest-pinned external wasm.
- `MinimalEducation`: small projects use a reduced config surface and lint
  against platform-only features.

Profiles should be explicit about allowed role management modes and forbidden
mutations.

## Data Model

### AdoptionProfileV1

```text
AdoptionProfileV1 {
  profile,
  allowed_role_management_modes,
  required_inventory,
  allowed_mutations,
  forbidden_mutations,
}
```

### RoleManagementModeV1

```text
RoleManagementModeV1 {
  ManagedByCanic,
  ObservedOnly,
  ExternalControllerRequired,
  UserControlled,
  ImportedPoolCandidate,
}
```

### AdoptionPlanV1

```text
AdoptionPlanV1 {
  target_profile,
  deployment_target,
  role_mappings,
  canister_mappings,
  required_authority_actions,
  forbidden_actions,
  warnings,
}
```

The plan is not permission to mutate. It is an adoption-specific view over the
same deployment truth and authority objects.

### PoolImportModeV1

```text
PoolImportModeV1 {
  ObserveOnly,
  RegisterWithoutReset,
  ResetAndReuseRequiresConfirmation,
}
```

Destructive pool import must never be the default adoption path.

## Commands

Tentative operator surface:

```text
canic fleet adopt <role> <principal>
canic fleet adopt --dry-run
canic fleet create --profile standalone
canic config lint --profile minimal
canic fleet check
```

Command names are tentative. The stable part is the adoption plan and profile
model.

---

## Non-Goals

0.48 should not:

- perform destructive import by default;
- hide controller takeover behind a profile;
- weaken protected-call trust requirements;
- replace 0.42 authority reconciliation;
- make observed-only roles look managed;
- turn minimal profiles into separate deployment semantics.

---

## Implementation Slices

### Slice 1: Profile Model

Define adoption profiles, allowed role management modes, and forbidden
mutations.

### Slice 2: Observe-Only Adoption Plan

Generate read-only adoption plans for existing canisters.

### Slice 3: Non-Destructive Canister Mapping

Allow role-to-canister mappings without reset, reinstall, or controller
takeover.

### Slice 4: Standalone And Leaf-Only Docs

Document runtime-only use, feature expectations, and a root-free feature
matrix.

### Slice 5: Hybrid External-Wasm Roles

Document and model digest-pinned external wasm roles and Candid-only
validation.

### Slice 6: Lint And Check Profiles

Add profile-aware config linting and fleet checks.

---

## Exit Criterion

0.48 is complete when:

```text
A brownfield or small project can see a safe adoption plan before any mutation,
and standalone or leaf-only users can understand which Canic features they are
adopting.
```
