# Canic Snapshot Cleanup And CLI Hardening

- **Version:** `0.31`
- **Status:** Draft
- **Scope:** restore correctness, snapshot ID handling, runner safety, and CLI workflow cleanup after `0.30`
- **Depends on:** [`0.30` fleet snapshot backup design](../0.30-canister-snapshots/0.30-design.md)

## 1. Summary

`0.30` delivered the first working fleet backup/restore control plane: topology
discovery, manifests, journals, validation reports, dry-runs, and guarded
restore runner state.

`0.31` should make that system safer and easier to operate. The main focus is
not a new backup product surface. The focus is correcting restore semantics,
separating backup domain primitives from CLI parsing, and wrapping the existing
low-level machinery in clearer, less error-prone CLI flows.

`0.31` keeps backup/restore authority in the host-side Canic control plane.
Root-aware coordination is acknowledged as a future hybrid capability, but it
is not required for this release.

The highest priority fixes are:

1. snapshot load must not be followed by code reinstall
2. restore must use the uploaded target-side snapshot ID, not the original
   source snapshot ID
3. artifact path validation and runner execution must resolve paths the same
   way
4. restore runners must lock the journal while claiming/executing/completing
   work
5. verification must default to safe query-only checks
6. failed operations must be retryable without losing failure history
7. the CLI parser should move incrementally to `clap`, with backup/restore
   domain logic kept in `canic-backup`

The CLI cleanup is part of the release, not a separate cosmetic refactor.
`0.31` should reduce redundant hand-written parsers, keep help output compact,
and make it harder for operator-facing flags and CI-only guard flags to drift
apart.

## 2. Context

The Internet Computer snapshot lifecycle has two hard operational facts that
Canic must model directly:

- a canister must be stopped before snapshot creation
- a target canister must be stopped before loading/restoring a snapshot

Snapshot load replaces the target canister's current code and data. Therefore,
the restore path must treat snapshot load as the code-and-state restore step,
not as a state restore followed by a separate code reinstall.

In `0.30`, Canic built the correct surrounding safety model: topology hashes,
manifests, journals, preflight reports, restore dry-runs, and guarded runner
state. However, the restore operation model still needs cleanup before it should
be treated as production-grade.

## 3. Problems To Fix

### 3.1 Snapshot Load Followed By ReinstallCode Is Wrong

If snapshot load restores code and data, then emitting `ReinstallCode` after
`LoadSnapshot` is at best redundant and at worst destructive. It may overwrite
the restored code or break the exact snapshot provenance the restore was meant
to preserve.

`0.31` must remove `ReinstallCode` from the normal snapshot restore operation
generation path.

If Canic ever needs code reinstall, it must be a separate explicit fallback mode
that is not mixed into snapshot restore by default.

### 3.2 UploadSnapshot Must Capture The Target Snapshot ID

Snapshot upload creates or returns a target-side snapshot ID. A later
`LoadSnapshot` operation must load that uploaded target snapshot ID.

It must not load the original source snapshot ID from the source fleet. Source
snapshot IDs are provenance; target uploaded snapshot IDs are execution inputs
for restore.

### 3.3 Artifact Paths Must Resolve Identically In Validation And Execution

Restore dry-run validation may prove that an artifact exists under a backup
directory, but the runner must execute the upload command using the same
resolved path.

`0.31` must avoid one code path validating relative paths and another executing
from a different working directory.

### 3.4 Restore Journals Need Cross-Process Locking

`0.30` added claim/pending state, but two runners can still race unless the
journal is protected by an inter-process lock.

The lock must cover the full critical section:

1. read journal
2. claim next operation
3. write pending state
4. execute command
5. write completed or failed state

### 3.5 Verification Must Be Safe By Default

Manifests must not be able to cause arbitrary mutating app calls during
verification by default.

Verification should prefer typed check kinds and query calls. Raw update calls
must require explicit opt-in and should be allowlisted.

### 3.6 Failed Operations Need Better Retry State

Operators need a way to retry failed operations without losing the original
failure context.

Failure records should preserve:

- failed command
- exit status
- stderr when available
- failure reason
- timestamp/update marker
- attempt count

Retry should be explicit, not automatic.

## 4. Goals

### 4.1 Correctness Goals

- Never emit `ReinstallCode` after `LoadSnapshot` in normal snapshot restore.
- Persist uploaded target snapshot IDs in the restore apply journal.
- Make `LoadSnapshot` depend on the uploaded target snapshot ID receipt.
- Resolve backup artifact paths from one typed source of truth.
- Lock restore journals during runner execution.
- Make verification query-only by default.
- Preserve failed operation history across retries.

### 4.2 CLI Goals

- Keep the low-level `snapshot`, `backup`, `manifest`, and `restore` command
  families, but do not preserve legacy aliases for their own sake.
- Use `clap` for command parsing, help rendering, value parsing, and mutually
  exclusive argument groups.
- Keep `canic-backup` independent of CLI parser dependencies. It owns typed
  manifests, plans, journals, validation, and command-preview primitives.
- Keep `canic-cli` responsible for operator UX: `clap` definitions, file IO,
  process execution, and shell-facing errors.
- Convert command families incrementally. Start at the top level, then migrate
  the most redundant restore/backup option parsers behind behavior tests.
- Add safer high-level commands only when they reduce real operator steps.
- Do not add high-level wrappers such as `backup create` or `restore prepare`
  until restore operation generation, uploaded snapshot IDs, artifact path
  resolution, and journal locking are fixed.
- Make the common backup path short and obvious.
- Make the restore path show exactly what is safe, what is dry-run, and what is
  mutating.
- Keep top-level help compact.

### 4.3 Testing Goals

- Add focused unit tests before broad refactors.
- Prefer testing operation generation, journal transitions, locking, path
  resolution, and command rendering.
- Keep full DFX/PocketIC smoke runs separate from fast unit tests.
- Add one real local smoke target that exercises capture, backup smoke, restore
  preparation, runner preview, and a guarded one-step restore path.

### 4.4 Carry-Forward Goals From 0.30

`0.31` must preserve the safety contract that `0.30` established while fixing
the restore operation model.

Carry these forward explicitly:

- topology hash validation remains fail-closed before snapshot creation
- manifest and journal responsibilities remain separate
- design-conformance reports remain available through manifest validation,
  preflight, restore planning, and smoke checks
- restore planning remains no-mutation
- `restore apply` remains dry-run unless a separate, explicit execution mode is
  designed later
- mutation continues through the guarded restore runner
- existing low-level command families remain available for automation, without
  keeping legacy aliases or wrapper scripts
- existing journals remain readable for status/report use where possible

If an existing journal is missing fields required for safe mutation,
the runner must refuse execution and explain how to regenerate or migrate the
journal.

## 5. Non-Goals

- No big-bang CLI rewrite that changes every command at once.
- No broad rename of all commands.
- No object-store backend.
- No cross-network disaster recovery automation.
- No automatic schema migration.
- No generic app quiescence system in this slice, beyond preserving honest
  `crash-consistent` versus `quiesced-unit` labeling and validation.
- No implicit update-call verification.
- No root-canister-owned backup orchestration in this slice.
- No in-canister journal or manifest authority.
- No requirement that root canister understands backup or restore state.

## 6. Hard Snapshot Preconditions

Canic must treat stopped-canister requirements as hard preconditions.

### 6.1 Snapshot Create

Before `dfx canister snapshot create <canister>`, the source canister must be
stopped.

Backup capture must not present stop/resume as a convenience-only option for
mutating capture. Instead, the planner/runner must prove one of:

- Canic stopped the canister as part of the capture workflow.
- The canister was already stopped and Canic recorded that fact.
- The operator explicitly requested a dry-run or validation-only path that does
  not create snapshots.

If a non-dry-run capture would create a snapshot for a running canister, Canic
must fail before snapshot creation.

Final `0.31` acceptance should record stopped-canister proof from both:

- live `dfx canister status` immediately before snapshot creation
- a journal receipt recording who proved or stopped the canister and when

This does not need to block the earliest restore operation-generation slices,
but any mutating snapshot-create path must fail rather than pretending proof
exists.

### 6.2 Snapshot Load

Before loading/restoring a snapshot into a target canister, the target canister
must be stopped.

Restore execution must prove one of:

- Canic stopped the target canister as part of the restore workflow.
- The target canister was already stopped and Canic recorded that fact.
- The command is a dry-run and does not load a snapshot.

If a runner would load a snapshot into a running target, Canic must fail before
the load command.

Final `0.31` acceptance should record stopped-target proof from both:

- live `dfx canister status` immediately before snapshot load
- a journal receipt recording who proved or stopped the target and when

This does not need to block the earliest restore operation-generation slices,
but any mutating snapshot-load path must fail rather than pretending proof
exists.

### 6.3 Resume Semantics

Resume should be explicit and journaled.

For backup capture, Canic may resume source canisters after snapshot creation
when the operator requests it or when the high-level workflow owns the stop.

For restore, resuming target canisters should happen only after the applicable
restore phase has completed and verification policy allows progress.

## 7. Proposed Command Model

The command model should be expressed through `clap` over time. The first
conversion boundary is the top-level command tree and compact help. Follow-up
slices should move individual command families into typed `clap` structs or
builder definitions while preserving the current intended argument names.
Legacy aliases and wrapper scripts should be removed rather than carried
forward.

This migration is also the main redundancy cleanup for `canic-cli`. Commands
should stop carrying repeated `while let Some(arg)` parsing loops and repeated
value extraction helpers. Those mechanics belong in `clap`; Canic-specific
checks belong in typed validation after parsing.

The desired ownership split is:

- `canic-cli`: `clap` command definitions, parsing, path arguments, environment
  defaults, output paths, process execution, and terminal-oriented errors.
- `canic-backup`: typed restore plans, manifests, apply journals, validation
  reports, operation previews, counters, receipts, and path-resolution rules.

This prevents `canic-backup` from becoming a second CLI crate and gives
`canic-cli` one standard parser instead of many ad hoc `match arg.as_str()`
loops.

The existing low-level command families stay:

- `canic list`
- `canic snapshot download`
- `canic backup *`
- `canic manifest validate`
- `canic restore *`

`0.31` may add high-level workflow commands after the correctness fixes land.
These wrappers must wait until at least Slice 4 is complete. The preferred
shape is:

```bash
canic backup create \
  --canister <canister-id> \
  --root <root-canister-id> \
  --recursive \
  --out backups/<run-id> \
  --network local
```

`backup create` would wrap `snapshot download` plus the hard stop/create/resume
preconditions and the standard post-capture checks.

For restore:

```bash
canic restore prepare \
  --backup-dir backups/<run-id> \
  --mapping restore-map.json \
  --out-dir restore/<run-id> \
  --require-design \
  --require-restore-ready
```

`restore prepare` would wrap plan, status, apply dry-run, journal creation, and
runner preview.

Execution should remain explicit:

```bash
canic restore run \
  --journal restore/<run-id>/restore-apply-journal.json \
  --execute \
  --max-steps 1 \
  --network local
```

This keeps mutation behind the existing guarded runner instead of hiding it in a
large all-in-one command.

## 8. Restore Operation Model

### 8.1 Operation Kinds

Normal snapshot restore should generate:

1. `UploadSnapshot`
2. `LoadSnapshot`
3. member verification checks
4. fleet verification checks

It should not generate `ReinstallCode` after `LoadSnapshot`.

If a future non-snapshot fallback mode needs code reinstall, it should use a
separate operation family such as `InstallCodeFallback` and require an explicit
mode flag.

### 8.2 UploadSnapshot Output

`UploadSnapshot` must record a typed output in the apply journal:

```json
{
  "uploaded_snapshot_id": "...",
  "target_canister": "...",
  "artifact_path": "...",
  "attempt": 1
}
```

The exact Rust representation should be typed, not ad hoc JSON assembled in the
CLI layer.

If `dfx` output format is unstable, uploaded snapshot ID parsing must be
isolated behind a small parser with focused tests and clear failure messages.

### 8.3 LoadSnapshot Input

`LoadSnapshot` must not embed the source snapshot ID as the executable load ID.
Instead, it should reference the preceding upload operation.

Conceptually:

```json
{
  "kind": "load-snapshot",
  "target_canister": "...",
  "uploaded_snapshot_from_sequence": 12
}
```

When rendering the command, the runner resolves
`uploaded_snapshot_from_sequence` to the recorded `uploaded_snapshot_id`.

Load command rendering must fail before execution unless the journal contains a
matching `UploadSnapshot` receipt with:

- the same target canister
- the same artifact/source member
- a completed upload operation
- a non-empty uploaded target snapshot ID

If the upload output is missing, failed, stale, from another member, or belongs
to a different target canister, command rendering must fail before execution.

## 9. Journal Model Changes

### 9.1 Operation Attempts

Each operation should track attempts:

```json
{
  "attempt_count": 1,
  "last_attempt": {
    "started_at": "...",
    "finished_at": "...",
    "command": ["dfx", "..."],
    "exit_status": "...",
    "stdout": "...",
    "stderr": "...",
    "reason": "..."
  },
  "previous_failures": []
}
```

The concrete schema can be smaller, but it must preserve enough information for
operators to understand and retry failures safely.

`stdout` and `stderr` may be truncated, but truncation must be deterministic and
recorded. Upload snapshot ID parsing depends on command output, so the receipt
must retain enough stdout context to audit which target snapshot ID was
captured.

The initial receipt shape should store bounded stdout and stderr, for example
the last `16 KiB` to `64 KiB` of each stream, plus:

- `truncated: true|false`
- original byte count
- a hash of the full stream when available

### 9.2 Retry Failed

Add an explicit retry command:

```bash
canic restore retry-failed \
  --journal restore-apply-journal.json \
  --sequence <n> \
  --reason "operator fixed target canister status" \
  --out restore-apply-journal.json
```

Retry should:

- require the operation to be failed
- preserve the old failure in history
- increment or preserve attempt counters consistently
- move the operation back to ready
- not skip earlier ready operations
- not clear uploaded snapshot outputs unless the retried operation owns that
  output and the output is invalidated by retry

Retrying a failed `UploadSnapshot` invalidates downstream `LoadSnapshot`
rendering for that upload until a new completed upload receipt exists. Dependent
load operations should move to blocked, or remain ready-but-unrenderable with a
stable blocking reason, rather than using an old uploaded snapshot ID.

The initial retry command should be per-sequence only. A future
`retry-all-failed` mode can be added after dependency invalidation is well
tested.

### 9.3 Lock File

Use a lock file next to the journal:

```text
restore-apply-journal.json.lock
```

The first implementation can use atomic file creation with create-new
semantics. The lock file should contain metadata:

```json
{
  "pid": 12345,
  "host": "host-name",
  "created_at": "2026-05-05T12:00:00Z",
  "journal": "restore-apply-journal.json"
}
```

The runner must hold the lock across claim, execution, and completion write.

Read-only status and report commands may read the journal while a lock exists,
but mutating commands must not claim, unclaim, retry, mark, or execute while
another runner holds the lock.

Stale lock recovery should be explicit. Do not auto-break locks in the first
implementation.

## 10. Artifact Path Resolution

The apply journal should persist enough path context to render commands from
any working directory.

Preferred model:

```json
{
  "backup_root": "/absolute/path/to/backups/run-id",
  "artifact_path": "artifacts/<member>/snapshot"
}
```

Rules:

- `backup_root` is absolute when the journal is created.
- member artifact paths are relative to `backup_root`.
- validation and execution use the same resolver.
- resolved paths must stay under `backup_root`.
- command rendering uses the resolved absolute path.

If Canic keeps absolute artifact paths only, it must still validate containment
and avoid mixing relative validation with absolute execution.

## 11. Verification Safety

### 11.1 Typed Verification Kinds

Prefer typed verification kinds:

- `canic_ready`
- `query_method`
- `allowlisted_query_method`
- `role_present`
- `topology_hash_matches`
- `wasm_catalog_contains`
- `shard_membership_matches`

The manifest should not be able to request an arbitrary update call by default.

### 11.2 Query Default

Verification calls default to query calls.

Any update verification must require all of:

- explicit manifest declaration
- explicit CLI opt-in
- allowlisted method or typed kind
- audit receipt that a mutating verification was allowed

Proposed flag:

```bash
canic restore run \
  --journal restore-apply-journal.json \
  --execute \
  --allow-update-verification
```

Without that flag, update verification operations must be blocked before command
execution.

  `LoadSnapshot`
- no `ReinstallCode` appears after `LoadSnapshot`

### Slice 2: Uploaded Snapshot ID Receipts

- load command rendering fails before upload receipt exists

- update verification runs only with explicit opt-in

Targets:

- `crates/canic-backup/src/journal/mod.rs`

- failed operation can move back to ready through explicit retry
- failure history is preserved
- attempt count increments or remains auditable

### Slice 7: Clap Parser Boundary And CLI/Backup Separation

Targets:

- `crates/canic-cli/Cargo.toml`
- `crates/canic-cli/src/lib.rs`
- `crates/canic-cli/src/backup/**`
- `crates/canic-cli/src/restore/**`

Rules:

- add `clap` to `canic-cli`, not to `canic-backup`
- keep top-level help compact
- preserve existing command names and flags during migration
- move duplicated value parsing into `clap` value parsers where practical
- keep domain validation in `canic-backup`
- do not move process execution or shell formatting into `canic-backup`

Tests:

- top-level help is generated from the `clap` command tree
- nested command help does not leak internal CI-only assertion flags into the
  top-level help
- removed aliases and wrapper scripts are rejected or absent
- mutually exclusive modes such as dry-run versus execute are represented by
  typed parser constraints where migrated
- `canic-backup` remains free of CLI parser dependencies

Fast unit tests should cover:

- operation generation
- operation counts
- upload/load dependency
- no reinstall after load
- path resolution
- journal lock behavior
- ready/pending/completed transitions
- failed/retry transitions
- verification safety defaults

CLI tests should cover:

- compact help stays compact
- new high-level wrappers map to existing low-level flows
- runner refuses load without uploaded snapshot ID receipt
- runner refuses snapshot create/load without stopped-canister proof in

- existing journals can still be inspected
- unsafe journals cannot be executed without required execution fields

Live DFX smoke should cover:

- stop -> snapshot create -> resume
- stop target -> upload -> load -> verify -> optional resume
- interrupted restore run and retry
- lock contention between two runner invocations
- capture -> `backup smoke` -> `restore prepare` -> runner preview -> one-step
  guarded execution

## 14. Migration Boundary

Existing journals should remain readable where possible.

If a journal lacks fields required for safe execution, Canic should:

- allow status/report inspection
- allow dry-run migration reports
- refuse mutating execution until the missing fields are supplied or the journal
  is regenerated

The command surface should preserve the intended low-level command families,
but not legacy aliases or wrapper scripts. New commands should be first-class
commands, not compatibility shims.

## 15. Journal Schema Compatibility Boundary

Do not introduce a separate public product-version distinction for this work.
The backup/restore product remains the same line. Internally, however, mutating
restore execution requires a journal with the fields needed to execute safely.

Status, report, and dry-run commands may read older journals when they can do so
without inventing missing execution facts.

Mutating runner execution must refuse journals that do not contain the required
execution fields:

- backup root or equivalent artifact path resolution context
- upload operation output fields
- uploaded target snapshot ID receipts for completed uploads
- load operations that reference upload receipts rather than source snapshot IDs
- stopped-canister proof receipts for snapshot create/load steps
- attempt/failure history fields needed for safe retry
- lock-compatible journal metadata

The refusal must be explicit and actionable. It should report whether the
operator needs to regenerate the apply journal, rerun restore preparation, or
run a future migration command.

## 16. Acceptance Criteria

`0.31` is complete when:

- normal snapshot restore generation emits no `ReinstallCode` after
  `LoadSnapshot`
- uploaded snapshot IDs are captured and used for load
- `LoadSnapshot` command rendering fails unless the journal contains a matching
  completed `UploadSnapshot` receipt for the same target canister and
  artifact/source member with a non-empty uploaded target snapshot ID
- artifact paths resolve identically in validation and execution
- restore runner uses a cross-process lock
- verification is query-only by default
- failed operations can be retried explicitly with failure history preserved
- stopped-canister preconditions are enforced before snapshot create and load
- existing journals remain inspectable but are rejected for mutating execution
  with actionable errors when required execution fields are missing
- top-level and migrated command-family parsing is owned by `clap`, while
  `canic-backup` remains parser-independent
- focused unit tests cover the operation model, journal transitions, locking,
  path resolution, and verification defaults

## 17. Future Direction: Root-Aware Coordination

The current `0.31` design keeps manifests, journals, restore planning, and
mutation authority on the host side. This preserves deterministic recovery from
backup artifacts alone.

A future release may add root-aware coordination as an optional hybrid layer.
In that model, the root canister may expose topology, roles, readiness, and
quiescence hooks. However, root awareness should initially be advisory and
coordination-focused, not the source of truth for backup execution.

The intended boundary is:

- Canic CLI owns manifests, journals, restore plans, artifact validation, and
  execution safety.
- Root canister may provide topology, fleet readiness, quiescence, resume hooks,
  and post-restore health signals.
- Backup artifacts must remain restorable even if the original root canister is
  unavailable or incorrect.

This keeps `0.31` focused on correctness while leaving a clean path toward a
hybrid root-assisted backup system.

## 18. Initial Answers And Open Questions

1. High-level `backup create` should run `backup smoke` automatically by
   default. Add `--no-smoke` for automation that wants capture only.
2. Store bounded stdout and stderr, initially the last `16 KiB` to `64 KiB` of
   each stream, with truncation status, original byte count, and full-stream
   hash when available.
3. Start with per-sequence retry only. Add retry-all-failed later after
   dependency invalidation is well tested.
4. Minimum typed verification vocabulary:
   `canic_ready`, `query_method`, `allowlisted_query_method`, `role_present`,
   `topology_hash_matches`, `wasm_catalog_contains`, and
   `shard_membership_matches`.
