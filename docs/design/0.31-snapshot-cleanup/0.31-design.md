# Canic Backup/Restore V1 Cleanup

- **Version:** `0.31`
- **Status:** Draft
- **Scope:** make snapshot backup/restore a small, reliable v1 product
- **Depends on:** [`0.30` fleet snapshot backup design](../0.30-canister-snapshots/0.30-design.md)

## 1. Summary

`0.30` built the first host-side backup/restore machinery: topology discovery,
manifests, journals, restore planning, dry-runs, and a guarded restore runner.
That proved the direction, but the design and command surface grew past the
actual v1 product.

`0.31` narrows the scope to a sane v1:

- track what was backed up, when it was backed up, and where its artifacts live
- back up one canister and its children recursively as a single restore unit
- store backups as portable local directories that can be archived or uploaded
- restore those backups later using snapshot upload/load, not code reinstall
- keep mutating work explicit, journaled, and safe to retry
- use `clap` only in `canic-cli` so backup/restore commands are easier to run
  and maintain

`0.31` keeps backup/restore authority in the host-side Canic control plane.
Root-aware coordination is a future optional capability, not a requirement for
this release.

## 2. V1 Product Boundary

Canic v1 backup/restore is a host-side wrapper around `dfx` snapshot commands.
`dfx` remains the source of truth for live IC operations. Canic adds the missing
operator layer around those commands:

- topology-aware selection
- stopped-canister enforcement
- portable manifests and journals
- artifact checksum verification
- restore mapping and ordering
- uploaded snapshot ID receipts
- locked restore execution

The v1 workflow should feel like this:

```bash
canic snapshot download <canister-id> \
  --root <root-canister-id> \
  --recursive \
  --out backups/<backup-id> \
  --network local

canic backup verify backups/<backup-id>

canic restore plan backups/<backup-id> \
  --map restore-map.json \
  --out restore/<restore-id>

canic restore run restore/<restore-id>/restore-apply-journal.json \
  --execute \
  --max-steps 1
```

High-level aliases such as `canic backup create` are allowed only after the
lower-level flow is correct. They should wrap the same primitives, not create a
second backup system.

## 3. What V1 Must Do

### 3.1 Track Backup History

Every backup directory must record:

- backup ID
- creation timestamp
- source network
- selected/root canister
- included canisters
- parent/child relationships
- topology hash at discovery
- topology hash immediately before snapshot creation
- source snapshot IDs
- artifact paths
- artifact checksums
- download journal state

This is enough for an operator to answer: what did we back up, when, from where,
and are the bytes still valid?

### 3.2 Capture Recursive Units

The common unit is:

```text
selected canister + all discovered descendants
```

Directory hints may select the starting canister, but topology defines
membership. If `--recursive` is used, Canic must discover descendants from the
authoritative topology/registry view and include the whole subtree.

V1 exposes only two backup-unit shapes:

- `single`: the selected canister only
- `subtree`: the selected canister plus discovered descendants

V1 does not need separate product modes for whole-fleet backups,
control-plane-only backups, arbitrary consistency groups, quiescence adapters,
dependency-closure declarations, verification classes, or manual restore
groups. Restore ordering comes from topology: parents before children.

### 3.3 Enforce Stable Topology

Canic must compute a canonical topology hash from sorted topology rows:

```text
(pid, parent_pid, role, module_hash)
```

For mutating snapshot capture:

- compute the hash during discovery
- recompute immediately before snapshot creation
- fail closed if the hashes differ

This prevents a backup manifest from describing a membership set that changed
while the backup was being prepared.

### 3.4 Enforce Stopped-Canister Preconditions

The IC snapshot lifecycle requires stopped canisters:

- source canister must be stopped before snapshot create
- target canister must be stopped before snapshot load

V1 must treat this as a hard precondition. For mutating flows, Canic must either
stop the canister, prove it is already stopped, or refuse to proceed.

Stop/proof/resume decisions must be journaled. Resume should be explicit and
should happen only after the operation that required the stop has completed.

### 3.5 Store Portable Backup Directories

The backup directory is the v1 storage format. It should be self-contained and
safe to archive, copy, or upload with external tools.

Required files:

- `fleet-backup-manifest.json`
- `download-journal.json`
- snapshot artifact files

V1 does not need an object-store abstraction. Uploading a backup can mean
uploading the directory or archive through normal operator infrastructure.

### 3.6 Verify Backup Bytes

`canic backup verify` should validate:

- manifest shape
- manifest/journal agreement
- all expected artifacts exist
- artifact checksums match
- topology hashes are present and agree

This command should be the primary read-only health check. Extra inspection,
provenance, and preflight reports should fold into this command or the explicit
restore dry-run sequence instead of becoming standalone v1 surfaces.

### 3.7 Restore From Uploaded Snapshot Receipts

Restore must use this order:

1. validate backup directory
2. build source-to-target mapping
3. generate parent-before-child restore plan
4. upload snapshot artifact to target environment
5. record uploaded target-side snapshot ID
6. load the uploaded target-side snapshot ID
7. run safe verification checks

`LoadSnapshot` must consume the uploaded target snapshot ID from the restore
journal receipt. It must not use the original source snapshot ID as the load ID.

Snapshot load restores code and state. Normal snapshot restore must not emit
`ReinstallCode` after `LoadSnapshot`.

### 3.8 Keep Restore Execution Journaled And Locked

If Canic executes mutating restore commands, it must use an apply journal.

The journal must record:

- operation sequence
- operation kind
- source member
- target canister
- command preview
- state: ready, pending, completed, failed, blocked
- attempt count
- bounded stdout/stderr
- failure reason
- uploaded snapshot ID receipts

Mutating runners must hold a cross-process lock across claim, pending write,
command execution, and completion/failure write. Read-only status commands may
inspect the journal while locked; mutating commands must not modify it.

### 3.9 Make Retry Explicit

Failed restore operations should be retryable by sequence number. Retry must:

- preserve the old failure reason
- preserve command/stdout/stderr context
- increment or clearly record attempts
- avoid skipping earlier ready work
- invalidate downstream load rendering if a failed upload is retried

V1 should start with per-operation retry only. `retry-all-failed` can wait.

### 3.10 Keep Verification Safe

Verification defaults to non-mutating checks.

Current v1 restore execution starts with one check:

- canister status

Role, topology, and allowlisted query checks can be added later as typed
verification kinds. Manifest-controlled arbitrary method execution is out of
the current v1 path.

## 4. What V1 Should Not Do

These are out of scope for the sane v1:

- root-canister-owned backup orchestration
- in-canister manifest or journal authority
- object-store provider abstraction
- cross-network disaster recovery automation
- automatic schema migration
- arbitrary flat consistency groups as a normal path
- whole-fleet/control-plane subset modes as separate product concepts
- generic app quiescence contracts
- code reinstall after snapshot load
- arbitrary manifest-controlled method execution
- large preflight bundles as the main operator interface
- multiple overlapping status/provenance/inspection report commands

Some of these may come back later, but they should be added only when the v1
flow is reliable and the need is concrete.

## 5. Crate Boundary

`canic-backup` is the domain library. It owns:

- manifest types
- download journals
- restore plans
- apply journals
- artifact verification
- topology hashing
- path resolution
- command previews

`canic-cli` is the operator interface. It owns:

- `clap` command definitions
- help output
- file and path arguments
- terminal output
- environment defaults
- process execution
- shell-facing errors

`canic-backup` must not become a second CLI crate. It should not depend on
`clap`.

Future Canic versions may support cargo-style command registration where
domain crates contribute commands to `canic-cli`. That is not a v1 requirement.
For v1, `canic-cli` is the only command surface and `canic-backup` remains a
plain Rust library.

## 6. CLI Shape

The long-term command surface should be small:

- `canic list`
- `canic snapshot download`
- `canic backup verify`
- `canic restore plan`
- `canic restore apply`
- `canic restore run`
- `canic restore retry`

The public backup command surface is limited to `verify` and `status`.
Preflight, smoke, inspection, and provenance reports are not separate v1
operator subcommands or standalone backup-library products.

No legacy aliases are required. Prefer one obvious command over several
nearly-equivalent commands.

## 7. Implementation Slices

### Slice 1: Restore Correctness

- remove `ReinstallCode` from normal snapshot restore generation
- keep source snapshot ID as provenance only
- make `LoadSnapshot` depend on upload receipt output
- test `UploadSnapshot -> LoadSnapshot -> verification`

### Slice 2: Uploaded Snapshot Receipts

- parse and store uploaded target snapshot IDs
- store bounded stdout/stderr for auditability
- fail load rendering if the receipt is missing or mismatched
- test failed/missing receipt cases

### Slice 3: Path Resolution

- persist an absolute backup root in restore apply journals
- keep artifact paths relative to that root
- render upload commands with resolved absolute paths
- reject traversal outside the backup root

### Slice 4: Stopped-Canister Enforcement

- prove or perform stop before snapshot create
- prove or perform stop before snapshot load
- journal stop/proof/resume events
- refuse mutating flows without proof

### Slice 5: Restore Locking And Retry

- lock restore journals during mutating execution
- allow read-only status while locked
- add per-sequence retry for failed operations
- preserve failure history

### Slice 6: CLI Simplification

- continue moving parsing to `clap`
- remove redundant parser helpers
- trim top-level help
- audit and collapse overlapping backup/restore commands
- keep `canic-backup` parser-independent

### Slice 7: Backup Command Surface Cleanup

- make `backup verify` the primary read-only check
- remove public and unused internal wrappers for `smoke`, `preflight`, `inspect`, and `provenance`
- remove or hide commands that only exist to prove internal design details
- update README examples to the final v1 flow

## 8. Acceptance Criteria

V1 is acceptable when:

- recursive snapshot download produces a portable backup directory
- backup directories record backup time, membership, topology hash, artifacts,
  checksums, and journal state
- topology changes between discovery and snapshot creation fail closed
- mutating snapshot create/load refuses to run without stopped-canister proof
- backup verification validates manifest, journal, artifacts, and checksums
- restore planning is no-mutation and orders parents before children
- restore execution uploads artifacts and loads uploaded target snapshot IDs
- normal restore emits no code reinstall after snapshot load
- restore journals are locked during mutating execution
- failed restore operations can be retried without losing failure history
- verification is query/read-only by default
- the public CLI surface is smaller and `clap` owns parsing

## 9. Future Direction: Root-Aware Coordination

A future release may add root-aware coordination as an optional hybrid layer.
The root canister may provide topology, roles, readiness, quiescence hooks,
resume hooks, and post-restore health signals.

That future root awareness should be advisory and coordination-focused. Backup
artifacts must remain restorable even if the original root canister is
unavailable or incorrect.

The intended boundary remains:

- Canic CLI owns manifests, journals, restore plans, artifact validation, and
  execution safety.
- Root canister may provide topology and readiness signals.
- Backup execution must not depend on in-canister journal authority.
