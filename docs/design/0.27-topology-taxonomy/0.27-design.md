# Canic Topology Taxonomy v1

## Status

Draft v1

## Context

The current codebase already distinguishes several important topology ideas, but
they are expressed through a mix of configuration kind, placement workflow,
registry policy, and backup/recovery language.

Today we already have:

* structural canister kinds such as `root`, `singleton`, `replica`, `shard`,
  and `instance`
* placement workflows for scaling and sharding
* parent/child registry topology
* export-layer singleton lookup views such as `app_directory` and
  `subnet_directory`
* ad hoc application-level directory patterns such as `project_hub ->
  project_instance`

Those are not the same concept, but they are close enough in shape that design
work has started to blur them together. That creates confusion in three places:

1. topology policy and config validation
2. application modeling for new deployment shapes
3. backup and restore planning

`0.28` backup design depends on the topology model being explicit. This design
therefore defines the topology taxonomy first, and backup design should consume
this vocabulary instead of inventing its own.

## Problem Statement

We need one consistent v1 model that answers these separate questions without
mixing them:

1. What kind of node is this canister structurally?
2. How does a parent place, locate, or create children?
3. What ownership shape does this create in the topology graph?

Without that separation:

* the older `tenant` naming remains underdefined
* `project_hub`-style directories stay ad hoc
* sharding and scaling look like special one-off systems instead of members of a
  broader family
* backup units are forced to infer topology semantics from incomplete labels

## Goals

### Primary goals

* Define a topology taxonomy that is simple enough to use in config, policy, and docs.
* Separate structural kind from placement family.
* Make room for directory-managed instance topologies without pretending they
  are scaling or sharding.
* Remove ambiguity around the older `tenant` naming and replace it with
  `instance` immediately.
* Give `0.28` backup design a stable vocabulary for discovery and quiescence.

### Non-goals for v1

* Implement the full config/runtime migration immediately.
* Finalize exact config key names for every future topology family.
* Design the entire backup/restore system here.
* Add new placement families beyond the minimum set required by current use
  cases.

## Design Principles

* **One axis, one question**: structural kind, placement family, and operational
  grouping must not be collapsed into one enum.
* **Topology before workflow**: backup and restore derive from topology, not the
  other way around.
* **Managed and ad hoc must be distinguished**: scaling and sharding are
  Canic-managed placement systems; app directories are not automatically so.
* **One word, one meaning**: `directory` should name the keyed placement family,
  while lookup exports should use `index`.
* **Ownership matters**: recursive descendant ownership is a first-class fact,
  not a side effect.
* **Vocabulary should match operator reality**: use terms that explain how
  canisters are discovered, created, and backed up.

## Proposed v1 Model

v1 uses two primary modeled axes plus one derived topology classification:

1. **Node kind**
2. **Placement family**
3. **Ownership shape** as a derived classification

Operational concepts such as backup units and quiescence modes are then derived
from those modeled facts plus current registry topology.

## Terminology Split: Directory vs Index

The codebase already uses `app_directory` and `subnet_directory` for export-layer
lookup views that list or resolve important canisters. Those are narrower than
the new keyed placement family proposed here.

v1 should reserve:

* `directory` for the placement family where one logical key resolves to at most
  one dedicated instance root
* `index` for lookup/export views over already-known topology

Recommendation:

* `app_directory` becomes `app_index`
* `subnet_directory` becomes `subnet_index`

If config or ABI migration is staged, the old names may survive temporarily for
compatibility, but the design vocabulary should switch now so the concepts do not
collide.

## Axis 1: Node Kind

Node kind answers:

> What sort of canister is this structurally in the topology graph?

v1 should use these concepts:

### `root`

The global authority and coordination anchor for a managed fleet.

Properties:

* no parent
* owns or coordinates fleet-global state
* may create or manage root-owned children

### `singleton`

One named canister under a parent with a stable role.

Properties:

* unique under its parent for a given role
* often acts as a control node, registry owner, or service endpoint
* may itself manage child placement

Examples:

* `app`
* `user_hub`
* `scale_hub`
* `project_hub`

### `worker`

A repeated child node whose identity is operational, not application-keyed.

Properties:

* may have many siblings with the same role
* usually participates in a managed pool
* often replaceable or fungible within policy constraints

Examples:

* scaling workers
* shard canisters

### `instance`

A dedicated child root for one named application object or directory key.

Properties:

* identity is stable and meaningful
* usually corresponds to one configured application key name and value
* typically owns a recursive descendant subtree

Examples:

* `project_instance`
* other per-customer or per-project roots

### Disposition of existing kind names

The current codebase uses `replica`, `shard`, and now `instance`. v1 should
interpret them like this:

* `replica` is an implementation-specific worker kind under `scaling`
* `shard` is an implementation-specific worker kind under `sharding`
* `instance` is the dedicated keyed child-root kind

Recommendation:

* keep `root` and `singleton`
* treat `worker` and `instance` as the conceptual categories
* keep `instance` as the explicit dedicated child-root term
* keep `replica` and `shard` as family-specific implementation labels where
  needed

## Axis 2: Placement Family

Placement family answers:

> How does a parent decide, create, resolve, or assign child membership?

v1 should recognize three real placement families.

### `scaling`

A singleton parent manages a pool of interchangeable workers.

Properties:

* capacity and throughput oriented
* workers are operationally similar
* no stable application key owns a specific worker by default

Current examples:

* `scale_hub -> scale`

### `sharding`

A singleton parent manages a shared shard set and assigns many logical keys
across that set.

Properties:

* partitioned state across a shared worker pool
* many logical keys map onto a reusable shard set
* one shard serves many keys

Current examples:

* `user_hub -> user_shard`

### `directory`

A singleton parent maps one stable configured logical key to at most one dedicated instance
root.

Properties:

* key cardinality is `0..1` for one instance root
* resolved instance identity is stable
* the instance root usually owns a recursive subtree
* operators and app workflows should use the stable directory key name and value
  to find the instance root instead of remembering raw canister ids

Expected examples:

* `project_hub -> project_instance`

Initial v1 constraints:

* directory keys should be representable as stable string-like values in v1

### Existing lookup exports should be called `index`

The older export-layer surfaces are not placement families. They are index
views.

Examples:

* `app_index`: index of important application-facing or app-directory singleton
  roles
* `subnet_index`: index of important subnet-level or root-managed singleton
  roles

Those views may support discovery, but they do not define recursive ownership or
keyed instance placement by themselves.

### What does not count as a placement family

These are important, but they are not additional general placement families:

* root-owned fleets such as publication-store fleets
* plain singleton child lookup
* backup-unit grouping
* subtree recursion itself

Those are topology facts or domain-specific workflows, not peer placement
systems alongside scaling, sharding, and directory.

## Derived Classification: Ownership Shape

Ownership shape answers:

> What graph shape does this node create below it?

Ownership shape is usually not something Canic should ask users to configure as
an independent peer concept. It should be inferred from node kind, placement
family, and the observed parent/child graph.

v1 should still recognize these shapes because backup, discovery, and
quiescence depend on them:

### `flat-child`

The parent owns children directly, but those children are not assumed to own a
meaningful recursive subtree.

Typical for:

* simple singleton children
* plain worker pools

### `managed-pool`

The parent owns a pool of workers created and resolved through one placement
family.

Typical for:

* scaling pools
* shard pools

### `subtree-rooted`

The parent resolves or creates a child that is itself the root of a recursive
descendant subtree.

Typical for:

* project instances
* other dedicated named instances

This ownership shape matters directly for backup discovery and quiescence.

## Shared Topology Graph

All placement families should share one canonical ownership graph.

That means:

* the parent/child topology graph is the authoritative source of recursive
  ownership
* each placement family adds its own family-specific resolution or assignment
  registry on top of that graph

In practice:

* scaling tracks pool membership and worker inventory
* sharding tracks shard inventory plus logical-key assignment
* directory tracks logical-key to instance-root resolution

But recursive descendant discovery should not be reinvented separately for each
family. Once a family resolves the relevant root or worker set, Canic should use
the shared topology graph to answer ownership questions.

## Placement To Child-Kind Mapping

At the conceptual level, the three placement families produce child node kinds
like this:

| Placement family | Conceptual child kind | Current compatibility label |
| --- | --- | --- |
| `scaling` | `worker` | `replica` |
| `sharding` | `worker` | `shard` |
| `directory` | `instance` | `instance` |

This is the intended long-term direction:

* scaling and sharding create managed workers
* directory creates dedicated named instances
* `instance` is the dedicated child-root term for directory placement

## Config Orientation

At the design level, the intended config split is:

* node kind is declared on the child role
* placement family is declared on the parent singleton
* ownership shape is derived

Illustrative sketches:

### Scaling

```toml
[subnets.prime.canisters.scale_hub]
kind = "singleton"

[subnets.prime.canisters.scale_hub.scaling.pools.scales]
canister_role = "scale"
```

Conceptual reading:

* parent node kind: `singleton`
* placement family: `scaling`
* child conceptual kind: `worker`
* current compatibility child label: `replica`

### Sharding

```toml
[subnets.prime.canisters.user_hub]
kind = "singleton"

[subnets.prime.canisters.user_hub.sharding.pools.user_shards]
canister_role = "user_shard"
```

Conceptual reading:

* parent node kind: `singleton`
* placement family: `sharding`
* child conceptual kind: `worker`
* current compatibility child label: `shard`

### Directory

```toml
[subnets.prime.canisters.project_hub]
kind = "singleton"

[subnets.prime.canisters.project_hub.directory.pools.projects]
canister_role = "project_instance"
key_name = "project"

[subnets.prime.canisters.project_instance]
kind = "instance"
```

Conceptual reading:

* parent node kind: `singleton`
* placement family: `directory`
* child conceptual kind: `instance`
* canonical child label: `instance`

The exact config schema for `directory` is still open, but the topology meaning
should follow this shape.

## Derived Operational Concepts

These are downstream concepts derived from topology, not independent topology
families.

### Backup unit

A backup unit is an operational grouping defined from topology facts.

Common shapes:

* `fleet-core`
* `flat`
* `subtree-rooted`

### Quiescence contract

Quiescence should be defined per placement family and ownership shape:

* scaling may freeze worker creation or pool mutation
* sharding may freeze assignment and shard lifecycle mutation
* directory may freeze one directory key and the corresponding instance subtree

### Discovery contract

Discovery should differ by family:

* scaling: enumerate pool members from the scaling family registry, then use the
  shared topology graph for child ownership when needed
* sharding: enumerate shard set and assignments from the sharding family
  registry, then use the shared topology graph for child ownership when needed
* directory: resolve one key to one instance root from the directory family
  registry, then recurse through the shared topology graph

## Mapping Current Canic Shapes

### Existing built-in examples

* `root`: node kind `root`
* `app`: node kind `singleton`
* `scale_hub`: node kind `singleton`, placement family `scaling`
* `scale`: node kind `worker`, owned via `scaling`
* `user_hub`: node kind `singleton`, placement family `sharding`
* `user_shard`: node kind `worker`, owned via `sharding`
* `app_index` / `subnet_index`: export-layer lookup views, not placement
  families

### Toko-style example

* `project_hub`: node kind `singleton`, placement family `directory`
* `project_instance`: node kind `instance`
* project descendants: ownership shape `subtree-rooted`

That means one configured directory key and value resolves to one
`project_instance`, and backup or restore may operate on that instance subtree
without treating the entire hub as one monolithic backup unit.

If Toko currently requires users or operators to remember a raw
`project_instance` canister id, that is the missing directory-layer capability.
The topology model should instead treat one configured directory key name plus
its value as the remembered identifier, with Canic resolving that pair to the
current instance root.

## Legacy `tenant` Terminology

`tenant` was the older name used at the start of this design line for a
dedicated keyed child root. It was underdefined and should not survive this
taxonomy.

Observed state:

* it appeared in schema and registry policy before the `instance` rename
* it required a singleton parent
* it appeared in tests and helper comments
* no checked-in real config used it as a live topology mode

Recommendation:

* treat `tenant` as deprecated historical vocabulary only
* do not build new operator-facing design, config, or docs around `tenant`
* prefer `instance` as the clearer concept for directory-managed dedicated child
  roots
* keep `instance` as the only public term so there is no ambiguity about the
  intended long-term model

Immediate migration target:

* rename `tenant` to `instance`
* remove the old `tenant` spelling from public config and docs as soon as the
  rename lands

The design intent is not neutral here: `tenant` should not survive as a
first-class long-term topology term.

## Why This Is The Right Cut

This taxonomy makes the important distinctions explicit:

* scaling is about fungible capacity
* sharding is about distributing many keys across a shared stateful set
* directory is about stable named instances

Those are strictly different systems.

At the same time:

* `root` and `singleton` remain structural roles, not placement families
* subtree backup is an operational consequence, not a new topology family
* domain-specific fleets do not need to become new global topology primitives

## Implications For `0.28`

The backup/snapshot design should build on this taxonomy as follows:

* backup discovery starts from node kind + placement family + ownership shape
* subtree-rooted instances become natural backup units
* scaling and sharding get Canic-owned quiescence rules
* directory placement gets key-scoped freeze semantics
* `project_hub`-style systems stop being backup exceptions and become one known
  topology family

## Open Questions

1. Should `directory` be a new explicit config subtree alongside `scaling` and
   `sharding`?
2. Which discovery and quiescence hooks belong in Canic core for directory
   placement?
3. Do we want one generic worker kind in config, or preserve family-specific
   kinds such as `replica` and `shard`?
4. Which existing application patterns besides Toko already fit the `directory`
   model?

## Concrete v1 Recommendation

Adopt this taxonomy in design immediately, even if the code migration is staged:

* structural kinds: `root`, `singleton`, `worker`, `instance`
* placement families: `scaling`, `sharding`, `directory`
* ownership shapes: `flat-child`, `managed-pool`, `subtree-rooted`

Then update `0.28` backup design to consume these concepts directly.
