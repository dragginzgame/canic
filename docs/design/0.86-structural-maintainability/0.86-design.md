# Canic 0.86 Design: Structural Maintainability

Date: 2026-07-12

## Status

- Status: complete; the bounded structural pass is published through
  `v0.86.8`.
- Release line: 0.86 structural maintainability.
- Audit source:
  `docs/audits/reports/2026-07/2026-07-11/codebase-health.md`.
- Implementation status: [status.md](status.md).
- Primary rule: split the three identified hub modules by responsibilities
  they already own, without changing behavior or inventing abstractions.

## Decision

0.86 is a mechanical hard cut across three modules, in this order:

1. Split Medic checks into focused children while retaining `medic` as the
   command facade.
2. Split deploy-plan evidence, comparison, diagnostics, and rendering into
   focused children while retaining the current deploy-plan entry points.
3. Split state-manifest resolution, descriptor joining, audit categories, and
   aggregation into focused children while retaining the current host facade.

Functions and private types move directly to their final owner. The old
definitions are deleted in the same slice. There are no forwarding wrappers,
aliases, deprecated paths, or old/new implementations.

## Goals

1. Reduce the edit and review radius of the three audit-identified hubs.
2. Give existing responsibilities clear module owners.
3. Keep public Rust APIs, CLI behavior, exit classification, diagnostic codes,
   JSON shapes, Candid, stable state, and persisted bytes unchanged.
4. Keep tests focused on maintained behavior before and after each move.

## Non-Goals

0.86 does not:

1. Add a check framework, rule engine, plugin system, generic renderer, or
   shared orchestration abstraction.
2. Redesign Medic, deploy planning, or state-contract policy.
3. Move ownership across crates.
4. Add compatibility layers or anti-resurrection tests.
5. Mix unrelated dependency, feature, or user-facing work into the structural
   slices.
6. Change package versions during implementation.

## Slice A: Medic

Move checks according to their existing domains. The parent remains
responsible for selecting and ordering project or deployment checks. Child
modules own only their check construction and domain-specific parsing.

The first move extracts auth-renewal and blob-storage checks. Later Medic moves
must follow the same rule: move a coherent responsibility directly, retain no
wrapper in the parent, and avoid a generalized check trait.

## Slice B: Deploy Plan

Separate the current option/input boundary, evidence collection, pure
comparison, diagnostic construction, and rendering/output responsibilities.
Do not change the order, meaning, or serialization of plan results.

## Slice C: State Manifest

Separate package/config resolution, descriptor joins, audit-category
construction, aggregation, and next-action projection. Allocation authority and
storage descriptors remain with their current owners; this slice only reduces
the host module's structural blast radius.

## Acceptance

Each slice is complete when:

1. The moved responsibility has one production definition in its focused
   module.
2. The former definition and unused imports are removed.
3. Existing public and serialized surfaces are unchanged.
4. Targeted owner tests and package lint pass.
5. The status file records the completed move and the next bounded action.
