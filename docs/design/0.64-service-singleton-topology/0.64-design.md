# Canic 0.64 Topology Model: Service vs Singleton

Status: design note
Target area: Canic 0.64 topology, root bootstrap, SubnetIndex, AppIndex placeholder
Primary decision: add `service` as a first-class canister kind and stop using `singleton` to imply root ownership

## 1. Summary

Canic currently conflates two meanings of `singleton`.

In application topology, a singleton means:

> exactly one child with this role under a given parent.

That definition is correct for nested canisters. For example, Toko can validly have one `project_ledger` child under each `project_instance`.

But canic-core also uses singleton to mean:

> a globally unique root-owned service that should be auto-created and placed into root service indexes.

That second meaning is incorrect. It causes nested singleton children to be treated as root services.

The long-term fix is to separate the concepts by adding a first-class `service` kind.

```text
service   = root-owned/root-scoped canister role, auto-created and SubnetIndex-indexed by default in 0.64
singleton = one child with this role under a parent scope
```

`SubnetIndex` can keep its name. Its semantics should be tightened so it
indexes root services only, not every canister with a matching role across the
registry.

`AppIndex` also keeps its name, but it should not be treated as another default
service index in this slice. The current role-keyed AppIndex is AppIndex V0.
The future multi-subnet projection is AppIndex V1 and is out of scope for this
0.64 correction. Until AppIndex V1 exists, AppIndex V0 entries should be
explicit, service-filtered, and never derived from `singleton`.

## 2. Problem statement

Current derivation is effectively:

```rust
auto_create_roles() = all singleton roles
subnet_index_roles() = all singleton roles
```

This makes every singleton role root-scoped. That is wrong for nested singleton children.

For Toko, the intended topology is:

```text
root
  project_hub          // root service

project_instance A
  project_ledger       // singleton child of A

project_instance B
  project_ledger       // singleton child of B
```

The current behavior treats `project_ledger` as both:

1. a root-owned singleton service; and
2. a per-project singleton child.

That produces an invalid topology:

```text
root
  project_hub
  project_ledger       // wrongly created root-owned ledger

project_instance A
  project_ledger

project_instance B
  project_ledger
```

Then a root service index builder scans the registry by role and sees multiple
`project_ledger` records. Because the index is role-keyed, it throws an error
like:

```text
subnet index role project_ledger appears more than once
```

The failure is not caused by Toko having singleton children. Singleton children are valid. The failure is caused by using singleton as a proxy for root service placement.

## 3. Design goals

The long-term topology model should satisfy these goals:

1. Preserve parent-scoped singleton children.
2. Add an explicit root-service concept.
3. Keep `AppIndex` and `SubnetIndex` as names and public concepts.
4. Make `SubnetIndex` index services only.
5. Stop deriving root bootstrap/index roles from all singleton roles.
6. Prevent nested children from polluting root service indexes.
7. Preserve duplicate detection for root services.
8. Make Toko's intended topology expressible without special cases.
9. Make the model obvious in config, code, tests, and docs.

## 4. Non-goals

This design does not require:

1. Renaming `AppIndex`.
2. Renaming `SubnetIndex`.
3. Renaming `singleton`.
4. Changing `project_ledger` into a replica or many-child role.
5. Removing per-parent singleton semantics.
6. Replacing role-based indexes everywhere.
7. Introducing a full placement DSL before the 0.64 line can move forward.
8. Defining the future multi-subnet `AppIndex` data schema.

A richer placement model can still exist later, but the required fix is smaller:
add `service` and make SubnetIndex service-only.

## 5. Core terminology

### 5.1 Role

A role names what a canister does in an application topology.

Examples:

```text
project_hub
project_instance
project_ledger
user_shard
```

A role alone should not imply global uniqueness. The uniqueness scope depends on the kind.

### 5.2 Service

A `service` is a root-owned/root-scoped canister role that participates in root
bootstrap and the subnet service index by default in the 0.64 model.

Definition:

```text
kind = "service"
```

Semantics:

```text
- direct child of root
- role-unique within its service index scope
- auto-created by root by default in 0.64
- included in SubnetIndex by default in 0.64
- may support future service-only `auto_create = false`
- may support future service-only `subnet_index = false`
- eligible for explicit AppIndex V0 entries until AppIndex V1 exists
- allowed to own directory/scaling/sharding manager pools in the 0.64 model
- resolved from root-level service indexes
```

Examples:

```text
project_hub
user_hub
billing_hub
registry_hub
```

In Toko, `project_hub` should be a service.

### 5.3 Singleton

A `singleton` is a parent-scoped child cardinality rule.

Definition:

```text
kind = "singleton"
```

Semantics:

```text
- one child with this role under a given parent
- not globally unique
- not root auto-created by default
- not placed in SubnetIndex
- not placed in AppIndex by default
- not allowed to own directory/scaling/sharding manager pools in the 0.64 model
- resolved through parent/child topology, not root service indexes
```

Examples:

```text
project_instance -> project_ledger
user_shard       -> user_profile_store
workspace        -> workspace_config
```

In Toko, `project_ledger` should remain a singleton.

### 5.4 AppIndex

`AppIndex` remains a root-owned application-level placeholder for data that
needs a multi-subnet projection.

New clarified semantics:

> AppIndex V0 is the current role-keyed path. It is not populated from canister
> kind by default. In the 0.64 service/singleton correction, AppIndex V0 should
> remain explicit and must not derive entries from `singleton`. AppIndex V0
> entries are service-filtered. AppIndex V1 is the future multi-subnet
> projection and is out of scope for this 0.64 correction.

### 5.5 SubnetIndex

`SubnetIndex` remains a root-owned subnet-level service index.

New clarified semantics:

> `SubnetIndex` indexes root services relevant to subnet-level resolution. It
> does not index every canister on a subnet with a matching singleton role.

The name can stay. The important change is that the indexed population is
services, not all singleton roles. `kind = "service"` implies
`subnet_index = true` by default, and only service roles may appear in
`SubnetIndex`.

## 6. Key invariant

The invariant should appear in code comments, docs, and tests:

```text
Canister kind defines topology semantics.

service:
  root-scoped
  direct child of root
  role-unique within its service index scope
  auto-created by default in 0.64
  included in SubnetIndex by default in 0.64

singleton:
  parent-scoped
  unique by (parent_pid, role)
  never root auto-created from kind alone
  never included in SubnetIndex
  invalid in current role-keyed AppIndex V0 entries

SubnetIndex indexes services only.
Current AppIndex V0 entries are explicit and service-filtered.
Nested children are resolved through parent-scoped lookup.
```

This is the central rule that fixes the bug class.

## 7. Proposed config model

Add `service` to the accepted topology canister kind list under
`[subnets.<name>.canisters.<role>]`.

Top-level `[roles.<role>]` tables continue to describe package/build roles
such as `kind = "canister"`; they are not the topology placement kind.

Current-style config can move from this ambiguous shape:

```toml
[subnets.prime.canisters.project_hub]
kind = "singleton"

[subnets.prime.canisters.project_ledger]
kind = "singleton"
```

to this explicit shape:

```toml
[subnets.prime.canisters.project_hub]
kind = "service"

[subnets.prime.canisters.project_ledger]
kind = "singleton"
parent = "project_instance"
```

The exact field name for `parent` should match existing Canic config conventions. The important point is that `project_ledger` remains a singleton, but its singleton scope is a `project_instance`, not root.

### 7.1 Default service behavior

For the 0.64 line, the simplest rule is:

```text
kind = "service" implies:
  root auto-create = true
  SubnetIndex inclusion = true
  AppIndex V0 inclusion = false unless explicitly configured
  directory/scaling/sharding manager pools are allowed
```

That gives one obvious escape from the current bug:

```text
root-level singleton service -> service
per-parent singleton child   -> singleton
```

If Canic later needs finer control, the model can add optional overrides:

```toml
[subnets.prime.canisters.project_hub]
kind = "service"
auto_create = true
subnet_index = true
```

But the first 0.64 correction should not require new booleans unless existing
applications already need that distinction. The important point is that any
future override is service-only. `auto_create = false` or
`subnet_index = false` should refine service behavior; they should not become
general-purpose flags that make singleton roles root-scoped.

### 7.2 Explicit index lists as an alternative

If Canic already has explicit index lists, they can be kept.

Example:

```toml
services = ["project_hub", "user_hub"]
app_index = ["project_hub"]
subnet_index = ["project_hub", "user_hub"]
auto_create = ["project_hub", "user_hub"]
```

In that variant, every role listed in `subnet_index` or `auto_create` must have
`kind = "service"`. Current role-keyed AppIndex V0 entries should also be
service roles, but AppIndex should not be auto-derived from service roles
because AppIndex V1's long-term purpose is multi-subnet app data.

Validation rule:

```text
A singleton role may not appear in root auto_create, SubnetIndex, or current
role-keyed AppIndex lists.
```

This keeps the existing index names while making the root-service set explicit.

## 8. Downstream example: Toko topology

`project_hub`, `project_instance`, and `project_ledger` are downstream Toko
roles, not built-in Canic roles. They are examples of the topology shape this
Canic change must support.

Toko should express the topology as:

```toml
[subnets.prime.canisters.project_hub]
kind = "service"

[subnets.prime.canisters.project_hub.directory.pools.projects]
canister_role = "project_instance"
key_name = "project"

[subnets.prime.canisters.project_instance]
kind = "instance"

[subnets.prime.canisters.project_ledger]
kind = "singleton"
parent = "project_instance"
```

The `parent = "project_instance"` field is schematic. The exact field name or
placement declaration should match the final Canic config convention for
parent-scoped singleton children.

Intended runtime topology:

```text
root
  project_hub: service

project_hub
  project_instance A
  project_instance B

project_instance A
  project_ledger: singleton

project_instance B
  project_ledger: singleton
```

Root indexes:

```text
SubnetIndex:
  project_hub -> <project_hub_pid>

AppIndex:
  unchanged/empty in this slice unless explicit future multi-subnet app data is configured
```

Root indexes must not contain:

```text
project_ledger -> ...
```

Project ledger resolution should happen through the owning `project_instance` child topology, for example:

```text
resolve_child(parent = project_instance_pid, role = project_ledger)
```

not:

```text
resolve_global(role = project_ledger)
```

## 9. Derivation rules

Replace the old derivation:

```rust
auto_create_roles() = all singleton roles
subnet_index_roles() = all singleton roles
```

with service-based derivation:

```rust
auto_create_roles() = all service roles
subnet_index_roles() = all service roles unless subnet_index is explicitly false
app_index_v0_roles() = explicit app_index entries only; do not derive from kind
```

If there are no explicit per-service `subnet_index` flags yet, this can simply be:

```rust
auto_create_roles() = all service roles
subnet_index_roles() = all service roles
app_index_v0_roles() = config.app_index
```

The key long-term rule is:

```rust
CanisterKind::Singleton must not imply root auto-create or root service index inclusion.
CanisterKind::Service must not imply AppIndex inclusion.
```

### 9.1 Source of truth for topology kind

The normalized topology config is the source of truth for whether a role is a
service, singleton, instance, shard, or replica.

Registry records may store the normalized kind for audit/debugging, but root
index population must authorize inclusion from the current normalized topology
config.

If both registry kind and config kind exist, index builders should use one of
these policies:

1. require them to agree; or
2. trust normalized config and emit a migration warning for stale registry kind.

They must never index a record solely because a stored registry kind says
`Service` or `Singleton`, solely because a role name appears in the registry,
or before checking the current normalized config. This matters during migrations
because stale registry entries may already carry old singleton metadata.

## 10. Root service index builder rules

Even after config derivation is fixed, root service index builders should be defensive.

`RootSubnetIndexBuilder` should only index records that satisfy all of the following:

```text
1. role is configured as a service role
2. record is a direct child of root
3. record is live/active according to existing registry lifecycle rules
```

Conceptual filter:

```rust
entry.kind == CanisterKind::Service
&& entry.parent_pid == Some(root_pid)
&& entry.role in configured_service_roles
```

If the registry record does not store kind, then use role config for kind:

```rust
role_config[entry.role].kind == CanisterKind::Service
&& entry.parent_pid == Some(root_pid)
```

The parent filter is still required. It prevents a bad registry record or future
config regression from polluting root service indexes.

If the current role-keyed `RootAppIndexBuilder` path remains active before the
future multi-subnet AppIndex schema exists, apply the same service/direct-root
filter to explicit `app_index` entries. Do not populate AppIndex from service
defaults.

### 10.1 Duplicate detection

Duplicate detection should remain strict for root services.

This should still be an error:

```text
root
  project_hub
  project_hub
```

This should not be an error:

```text
project_instance A
  project_ledger

project_instance B
  project_ledger
```

The duplicate key for services is:

```text
role
```

because services are direct root children and root-owned.

The duplicate key for singleton children is:

```text
(parent_pid, role)
```

because singleton cardinality is parent-scoped.

### 10.2 Service uniqueness scope

For 0.64, service uniqueness is role-keyed within the index being built.

If `SubnetIndex` is per-subnet, the effective uniqueness scope is:

```text
(subnet_id, role)
```

If the root registry is globally scanned, the builder must first filter to the
target subnet and direct root children, then enforce unique role inside that
filtered set.

This design does not require role uniqueness across unrelated subnet indexes
unless the existing Canic root model already enforces that globally. The goal is
to avoid replacing the singleton bug with a future bug where a valid service
role appears more than once across unrelated subnet scopes.

## 11. Registry lookup rules

Root-level lookup should only be role-keyed for services.

Valid root service lookup:

```text
get_service(role = project_hub) -> project_hub_pid
```

Invalid nested lookup:

```text
get_service(role = project_ledger) -> project_ledger_pid
```

Nested child lookup must include parent scope:

```text
get_child(parent_pid = project_instance_pid, role = project_ledger) -> project_ledger_pid
```

For any role that can appear under multiple parents, a role-only registry query is not a complete identity.

## 12. Validation rules

Config validation should reject ambiguous or invalid topology declarations.

### 12.1 Service validation

A role with `kind = "service"` must satisfy:

```text
- parent is root, implicit root, or omitted in a way that means root
- role may appear in root auto_create
- role appears in SubnetIndex by default unless explicitly disabled
- role may appear in current explicit AppIndex V0 entries
- role may own directory/scaling/sharding manager pools in 0.64
- role may not be declared as child of a non-root role
```

Invalid:

```toml
[subnets.prime.canisters.project_ledger]
kind = "service"
parent = "project_instance"
```

Reason:

```text
A service cannot be nested under project_instance.
```

### 12.2 Singleton validation

A role with `kind = "singleton"` must satisfy:

```text
- uniqueness is scoped to parent
- role is not automatically root-created
- role is not SubnetIndex indexed
- role is not automatically AppIndex indexed
- role may not appear in current role-keyed AppIndex V0 entries
- role may not own directory/scaling/sharding manager pools in 0.64
```

If explicit SubnetIndex/AppIndex V0 flags or lists exist, reject:

```toml
[subnets.prime.canisters.project_ledger]
kind = "singleton"
parent = "project_instance"
subnet_index = true
```

Reason:

```text
A singleton child role cannot be SubnetIndex-indexed unless it is changed to a
root-scoped service.
```

### 12.3 Placement manager validation

For the 0.64 model, active placement manager pools should be service-owned:

```text
- directory pools are allowed on services
- scaling pools are allowed on services
- sharding pools are allowed on services
- directory/scaling/sharding pools are rejected on singleton children
```

This keeps the first service/singleton split easy to reason about: services are
root-reachable managers, while singleton children are parent-scoped managed
canisters. A future nested-manager design can relax this deliberately if a real
use case needs singleton children to manage their own directory, scaling, or
sharding pools.

This restriction only applies to directory/scaling/sharding manager pools. It
does not mean that only services can be parents. An `instance` may still own or
lazily create a parent-scoped singleton child when the topology declares that
child relationship. For example, each downstream Toko `project_instance` may
own exactly one `project_ledger`.

### 12.4 Duplicate validation

Registry/runtime validation should enforce:

```text
- no duplicate service role under root
- no duplicate singleton role under the same parent
- duplicate singleton roles under different parents are valid
```

## 13. Migration plan

### 13.1 Config migration

For existing apps, migrate root-level singleton services to `service`.

Before:

```toml
[subnets.prime.canisters.project_hub]
kind = "singleton"
```

After:

```toml
[subnets.prime.canisters.project_hub]
kind = "service"
```

Nested singleton children stay singleton.

Before:

```toml
[subnets.prime.canisters.project_ledger]
kind = "singleton"
```

After:

```toml
[subnets.prime.canisters.project_ledger]
kind = "singleton"
parent = "project_instance"
```

For roles that were relying on singleton auto-create, the migration should force an explicit choice:

```text
Is this role a root service?
  yes -> kind = "service"
  no  -> kind = "singleton" with parent scope
```

### 13.2 Runtime registry migration

Existing incorrect root-owned singleton children may already exist in some dev deployments.

Migration should identify records where:

```text
role is no longer service
parent_pid == root_pid
```

For Toko, that means a root-owned `project_ledger` should be treated as invalid topology residue.

Possible handling:

```text
- in local/dev: delete or ignore the invalid root-owned child
- in migration code: mark it deprecated/orphaned if deletion is unsafe
- in indexes: exclude it because it is not a service
```

The most important behavior is that root service indexes do not include it.

### 13.3 Compatibility window

A transitional compatibility layer can exist briefly:

```text
- accept old configs with singleton root services
- emit warnings requiring migration to service
- infer service only for known root-owned singleton roles when explicitly listed in old root bootstrap/SubnetIndex config
- keep AppIndex V0 explicit; do not infer AppIndex V0 entries from service roles
```

The compatibility adapter must normalize old configs before derivation. Old
singleton root services may be translated to `service` during config loading,
but no downstream derivation function may continue to interpret raw
`kind = "singleton"` as root auto-create or SubnetIndex inclusion.

But the long-term model should not preserve the old inference:

```text
singleton -> service
```

That inference is the bug.

## 14. Implementation slices

### Slice 1: Spec and type model

Add `service` to the canister kind enum and parser.

Expected shape:

```rust
pub enum CanisterKind {
    Service,
    Singleton,
    Instance,
    // existing variants...
}
```

Document the distinction directly on the enum:

```rust
/// Root-owned service. Direct child of root. Eligible for root bootstrap and SubnetIndex.
Service,

/// Parent-scoped singleton child. Unique under a parent, not globally unique.
Singleton,
```

### Slice 2: Config validation

Add validation that enforces service/root placement and singleton/parent-scoped placement.

Validation should catch:

```text
- service declared under non-root parent
- singleton listed in root auto_create
- singleton listed in SubnetIndex
- singleton listed in current role-keyed AppIndex V0 entries
- singleton owns directory/scaling/sharding pools
- instance/replica/shard parent policy still recognizes service managers
- index lists containing unknown roles
- index lists containing non-service roles
```

### Slice 3: Source-of-truth and derivation changes

Normalize compatibility configs before derivation, and make normalized topology
config the source of truth for service/singleton classification.

Change helpers such as:

```rust
auto_create_roles()
app_index_v0_roles()
subnet_index_roles()
```

so root bootstrap and SubnetIndex derive from service roles instead of
singleton roles, while AppIndex V0 remains explicit and service-filtered.

Old:

```rust
roles.filter(|role| role.kind == Singleton)
```

New:

```rust
roles.filter(|role| role.kind == Service)
```

or, if explicit lists exist:

```rust
configured_auto_create_roles.filter(|role| role.kind == Service)
configured_subnet_index_roles.filter(|role| role.kind == Service)
configured_app_index_v0_roles.filter(|role| role.kind == Service) // explicit entries only
```

### Slice 4: Root service index builder filters

Change `RootSubnetIndexBuilder` to include only direct root children with service
roles. If the current role-keyed `RootAppIndexBuilder` remains active before
the future multi-subnet AppIndex schema exists, apply the same defensive filter
only to explicit `app_index` entries.

Conceptual implementation:

```rust
let root_pid = registry.root_pid()?;

for entry in registry.entries() {
    if entry.parent_pid != Some(root_pid) {
        continue;
    }

    if !topology.role(entry.role).is_service() {
        continue;
    }

    index.insert_unique(entry.role, entry.pid)?;
}
```

Duplicate errors should be raised by `insert_unique` only after the service and parent filters have been applied.

### Slice 5: Child lookup and singleton enforcement

Ensure singleton child enforcement uses parent scope.

Correct key:

```rust
(parent_pid, role)
```

Incorrect key:

```rust
(role)
```

This matters anywhere Canic creates or resolves singleton children.

### Slice 6: Downstream Toko config migration

Change downstream Toko topology config:

```text
project_hub    singleton -> service
project_ledger singleton -> singleton, parent-scoped under project_instance
```

Remove any root auto-create or root service index entries for `project_ledger`.

### Slice 7: Docs and status notes

Update docs/changelog/status notes with the invariant:

```text
service is root-scoped
singleton is parent-scoped
SubnetIndex indexes services only
AppIndex V0 is explicit and service-filtered; AppIndex V1 is future multi-subnet app data
```

## 15. Required tests

### 15.1 Nested singleton does not pollute root service indexes

Topology:

```text
root
  project_hub: service

project_instance A
  project_ledger: singleton

project_instance B
  project_ledger: singleton
```

Expected:

```text
SubnetIndex contains project_hub.
AppIndex is unchanged/empty unless explicit future multi-subnet app data is configured.
SubnetIndex does not contain project_ledger.
No duplicate project_ledger error occurs.
```

### 15.2 Duplicate service under root fails

Topology:

```text
root
  project_hub: service
  project_hub: service
```

Expected:

```text
Root index build fails with duplicate project_hub service.
```

### 15.3 Duplicate singleton under same parent fails

Topology:

```text
project_instance A
  project_ledger: singleton
  project_ledger: singleton
```

Expected:

```text
Child creation or validation fails with duplicate singleton under same parent.
```

### 15.4 Duplicate singleton under different parents succeeds

Topology:

```text
project_instance A
  project_ledger: singleton

project_instance B
  project_ledger: singleton
```

```text
Valid topology.
No root service index duplicate error.
```

Expected:

```text
Config validation fails before index building.
```

```text
Nested record is rejected or ignored before role-key duplicate detection.
```

## 17. Expected old behavior vs new behavior

### Old behavior

```text
kind = singleton
  -> unique globally in root service indexes
  -> root auto-created
  -> SubnetIndex role candidate
  -> AppIndex role candidate when listed in current role-keyed app_index config
```

This breaks nested singleton children.

### New behavior

```text
kind = service
  -> direct child of root
  -> auto-created by root by default in 0.64
  -> SubnetIndex candidate by default
  -> AppIndex candidate only when explicitly listed before the future multi-subnet schema exists
  -> duplicate by role under root is an error

kind = singleton
  -> child under a parent
  -> not root auto-created
  -> not SubnetIndex candidate
  -> not valid in current role-keyed AppIndex V0 entries
  -> duplicate by (parent, role) is an error
```

## 18. Why keeping AppIndex and SubnetIndex is fine

The bug is not the index names. The bug is the population rule.

`SubnetIndex` can remain the right name if it means:

```text
an index of root services available for subnet-level routing/resolution
```

It must not mean:

```text
a role-keyed scan of every canister on the subnet
```

`AppIndex` can remain the right name if it means:

```text
a future placeholder for app-level data that needs a multi-subnet projection
```

It must not mean:

```text
a role-keyed scan of every app canister regardless of parent scope
```

So the naming can stay. The index population and validation rules must change.

## 19. Open questions

These can be resolved during implementation.

### 19.1 Does every service auto-create by default?

Recommended for 0.64:

```text
yes
```

Reason: it keeps the service concept simple and fixes the bug with minimal additional config.

Future optional refinement:

```toml
[subnets.prime.canisters.some_service]
kind = "service"
auto_create = false
```

### 19.2 Do services default into SubnetIndex?

Recommended for 0.64:

```text
yes
```

Reason: in 0.64, `service` means root-owned/root-scoped and root-reachable
through the subnet service index by default. The default should be obvious:

```text
kind = "service" => subnet_index = true
```

If a future service should be intentionally hidden from SubnetIndex, add an
explicit service-only override:

```toml
[subnets.prime.canisters.internal_scheduler]
kind = "service"
subnet_index = false
```

`subnet_index = true` on a non-service role must fail validation.

### 19.3 Are AppIndex and SubnetIndex always identical service sets?

Recommended for 0.64:

```text
no
```

`SubnetIndex` is the service index fixed by this design. AppIndex V0 is the
current explicit role-keyed path; AppIndex V1 is the future placeholder for
app-level data that needs a multi-subnet projection. Keep AppIndex V0 explicit;
do not derive AppIndex V0 entries from service roles.

### 19.4 Should root services be represented as parent = root or parent omitted?

Recommended:

```text
Allow omitted parent for service roles if existing config style expects that.
Internally normalize service parent to root.
```

The runtime registry should still store or resolve a real root parent PID.

### 19.5 Should nested manager roles be allowed later?

Maybe, but not in the first 0.64 correction.

Recommended for 0.64:

```text
directory/scaling/sharding managers are services
singleton children cannot own manager pools
```

A future design can relax this with an explicit nested-manager model if there
is a concrete downstream need.

### 19.6 Should there be a later placement enum?

Maybe.

A more general future model could be:

```rust
enum CanisterKind {
    Service,
    Singleton,
    Instance,
    Shard,
    Replica,
}
```

or:

```rust
enum Cardinality {
    Singleton,
    Many,
}

enum Placement {
    RootService,
    ChildOf(Role),
}
```

For 0.64, adding `service` is simpler and matches the desired direction.

## 20. Final recommendation

Adopt this model for Canic 0.64:

```text
Add kind = "service".
Keep kind = "singleton".
Keep AppIndex and SubnetIndex names.
Make SubnetIndex index services only.
Make root auto-create derive from services only.
Make service default to subnet_index = true.
Keep AppIndex V0 explicit and service-filtered; leave AppIndex V1 future multi-subnet data out of scope.
Allow directory/scaling/sharding manager pools on services only in 0.64.
Make singleton uniqueness parent-scoped.
Make root service index builders filter to direct children of root.
```

For Toko:

```text
project_hub    -> service
project_ledger -> singleton under project_instance
```

The core fix is not a one-off Toko workaround. It is a topology model correction:

```text
Service is root scope.
Singleton is parent scope.
SubnetIndex is a service index.
Current role-keyed AppIndex V0 entries are explicit and service-filtered.
Child singletons are resolved through their parent.
```
