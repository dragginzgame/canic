# 0.43 Design: Backend-Agnostic Execution

## Status

Tentative design notes for the 0.43 line.

0.43 is where Canic should replace the current install execution shape with a
plan-driven executor boundary.

This must come after deployment truth and authority reconciliation. Backend
abstraction before those foundations is a rewrite trap.

Core goal:

```text
Deployment intent no longer depends on dfx, icp, or local filesystem layout.
```

---

## Scope Calibration

Release bar:

- current CLI backend behind the executor boundary;
- no new backend required;
- artifact root decoupling from plan semantics;
- structured operation receipts with partial execution status;
- at least one test or harness path validates the same `DeploymentPlanV1` shape
  used by real deployment execution.

0.43 should include at least one test or harness path that consumes or
validates the same `DeploymentPlanV1` shape as real deployment execution, so
test execution cannot drift from production plan semantics.

Design hook:

- workspace root, ICP root, and artifact root fields recorded so project
  manifests can bind them later;
- executor result shapes preserve per-phase and per-role outcomes for future
  JSON;
- `wasm_store` staging through `DeploymentExecutor::stage_artifact`;
- backend capability model that allows future direct-agent or alternate CLI
  executors;
- optional project manifest shape is anticipated, but its stable public
  contract is deferred to 0.49.

Deferred:

- stable public JSON schema and exit-code contract in 0.49;
- official CI wrappers in 0.49;
- new non-CLI executor backend.

---

## Dependency On Earlier Lines

0.43 assumes:

- 0.41 can produce plan, inventory, diff, role artifact manifests, lightweight
  current-install phase receipts, verified postcondition evidence for those
  phases, and resume safety reports;
- 0.42 can reconcile or explain controller authority;
- `wasm_store` is modeled as a role artifact;
- artifact identity separates source bytes, installed module bytes, and
  embedded config.

Automated resume may be implemented only where the safety report proves phase
skipping is valid. 0.43 must not assume that a receipt alone can drive control
flow.

0.43 should not weaken those checks. It changes how work is executed, not what
the plan means.

---

## Minimal Early Boundaries

0.41 may introduce small internal traits to avoid hardcoding local assumptions:

```text
DeploymentInventorySource
ArtifactMaterializer
```

0.43 introduces the full executor:

```text
DeploymentExecutor {
  create_canister
  canister_status
  update_settings
  install_code
  call
  query
  stage_artifact
}
```

The executor is an implementation detail behind `DeploymentPlanV1`.

The executor result is not a boolean. It is an operation receipt whose status
distinguishes complete execution, failure before mutation, failure after
mutation, and partial application.

---

## Executor Model

### DeploymentExecutor

Tentative operations:

```text
create_canister(request) -> CreatedCanister
canister_status(canister_id) -> CanisterObservation
update_settings(canister_id, settings) -> SettingsReceipt
install_code(canister_id, mode, artifact, init_arg) -> InstallReceipt
call(canister_id, method, payload) -> CallReceipt
query(canister_id, method, payload) -> QueryReceipt
stage_artifact(role_artifact, transport) -> StagingReceipt
```

Each operation must return enough observation data to verify the phase
postcondition without trusting local process state.

### StagingReceiptV1

`stage_artifact` should return typed staging evidence:

```text
StagingReceiptV1 {
  role,
  artifact_identity,
  transport,
  wasm_store_locator,
  prepared_chunk_hashes,
  published_chunk_count,
  verified_postcondition,
}
```

For `ArtifactTransport::WasmStore`, the verified postcondition should prove:

- the store has a manifest for the planned role, template, and version;
- the store has the prepared chunk set with expected chunk hashes;
- the store has all chunks required by the manifest;
- payload hash matches the planned artifact digest.

### Artifact Transport

Artifact transport is separate from artifact roles:

```text
ArtifactTransport::LocalCli
ArtifactTransport::WasmStore
ArtifactTransport::DirectAgent
```

`wasm_store` as a role is bootstrapped by the executor. Later artifact staging
may use `wasm_store` as a transport.

This avoids circular semantics.

### Backends

Initial supported backends:

- current ICP CLI backend;
- PocketIC backend or test specialization that consumes the same plan shape.

Future backends:

- direct IC agent;
- alternate local CLI backend.

Plan schema must not change when backends change.

## Plan-Driven Execution

0.43 should make execution consume a materialized plan:

```text
DeploymentPlanV1 -> SafetyReportV1 -> AuthorityReconciliationPlanV1
-> DeploymentExecutor
```

Execution phases remain aligned with 0.41:

```text
create_root
build_artifacts
materialize_artifacts
install_root
stage_release_set
resume_bootstrap
wait_ready
post_validate
```

The current install mechanics may be extracted behind the executor boundary,
but they should not remain as a second execution path and should not rediscover
independent deployment intent.

### Partial Execution Semantics

Deployment execution is not atomic. If a role or phase succeeds and a later
role or phase fails, the earlier mutation remains applied.

0.43 should make that explicit in execution receipts:

```text
DeploymentExecutionReceiptV1 {
  operation_id,
  plan_id,
  status,
  phase_results,
  role_results,
  final_inventory_id,
}

DeploymentExecutionStatusV1 {
  Complete,
  FailedBeforeMutation,
  FailedAfterMutation,
  PartiallyApplied,
}
```

`FailedBeforeMutation` means retry may be safe after fixing inputs, subject to
normal plan and inventory checks.

`FailedAfterMutation` and `PartiallyApplied` require fresh inventory before any
retry, resume, rollback, or restore plan.

Recovery starts with live re-inventory and resume analysis. Automatic
multi-canister rollback is not part of 0.43.

---

## Artifact Roots

Artifact roots are executor outputs, not plan assumptions.

The plan may express artifact sources and expected digests. The executor or
materializer decides where those artifacts are built or staged.

This avoids baking these paths into deployment intent:

```text
.dfx/local/canisters
.icp/local/canisters
.icp/<network>/canisters
```

The safety report should still display concrete paths when the current backend
uses them.

---

## Automation And Project-Root Design Hooks

0.43 should make executor output structured enough to automate later, but it
does not own the public automation contract.

Executor result structs should be stable internal objects. Stable public JSON
schemas, machine-readable exit-code contracts, and official CI wrappers are
deferred to 0.49.

Deployment receipts and local state should record the roots used by the
executor:

```text
DeploymentExecutionContextV1 {
  workspace_root,
  icp_root,
  artifact_roots,
  backend,
  backend_capabilities,
}
```

This lets later CI and split-repo workflows prove which filesystem context was
used without making that layout part of `DeploymentPlanV1`.

An optional `canic.project.toml` can be anticipated as a design hook for split
repositories. Its stable public contract is deferred to 0.49 unless 0.43 needs
an internal executor context file earlier.

0.43 must not make local filesystem layout part of deployment intent. The plan
expresses artifact sources and expected digests; the executor records where
the current backend materialized them.

---

## PocketIC

PocketIC should consume the same plan shape as real deployments, or a lossless
test specialization.

Goals:

- test plan resolution frequently;
- test authority reconciliation without real IC cost;
- test artifact materialization and embedded config identity;
- prevent real deployments from being the first place plan drift appears.

PocketIC-specific shortcuts must not change the meaning of `DeploymentPlanV1`.

---

## Operation Locking

0.43 should formalize the remote operation boundary if it did not already land
in 0.42.

Executor operations should carry:

- operation ID;
- plan hash;
- previous observed deployment epoch;
- expected root trust anchor.

If the root exposes a deployment operation epoch, executor phases must verify
it before and after mutating phases.

---

## Non-Goals

0.43 should not:

- implement artifact promotion;
- implement user-owned consent flows;
- implement automatic multi-canister rollback;
- invent new authority policy;
- weaken 0.41 safety checks to fit a backend;
- define a stable public JSON schema or exit-code contract, which is deferred
  to 0.49;
- ship official CI actions, which are deferred to 0.49;
- productize the project manifest UX unless needed as internal executor
  context;
- require every backend to support every future operation immediately.

Backend capability discovery is allowed, but missing required capabilities must
be explicit plan validation failures.

---

## Implementation Slices

### Slice 1: Executor Trait And Capability Model

Define executor operations and backend capability reporting.

### Slice 2: Current CLI Backend

Move existing install mechanics behind the executor boundary without changing
operator behavior beyond plan-driven validation.

### Slice 3: Artifact Root Decoupling

Stop treating local CLI artifact layout as plan semantics.

### Slice 4: Retire `wasm_store` Special Materialization

`wasm_store` remains special only in bootstrap ordering.

It must be materialized through the same `RoleArtifactManifest` pipeline as
every other role.

### Slice 5: PocketIC Plan Execution

Make the main test harness consume the same plan shape.

### Slice 6: Current Installer Extraction

Extract the current installer path into the executor boundary and remove any
separate planner behavior. Do not keep retired live-deployment command names.

---

## Exit Criterion

0.43 is complete when:

```text
The same DeploymentPlanV1 can run against the current CLI backend, PocketIC,
or a future direct-agent backend without changing deployment intent.
```
