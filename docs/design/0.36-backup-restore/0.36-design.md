# Canic Backup/Restore V1

- **Version:** `0.36`
- **Status:** Proposed next release focus
- **Scope:** prove, harden, and finish the full operator-working backup/restore
  flow
- **Builds on:** `docs/design/0.35-backup-restore/0.35-design.md`

## 1. Summary

`0.35` established the backup/restore model: topology-aware plans, manifests,
execution journals, restore apply journals, command previews, receipts, and
native runner state transitions.

Much of the execution code already exists: backup execution has a runner,
restore has apply journals and a native runner, and the CLI has dry-run,
execute, pending-recovery, and failure-recording coverage. `0.36` should not
restart this work. It should prove the existing code end-to-end, close the
remaining gaps, and make the workflow trustworthy for a normal installed fleet.

The target happy path is:

```bash
canic backup create <fleet>
canic backup list
canic backup inspect <backup-ref>
canic backup verify <backup-ref>
canic restore prepare <backup-ref> --require-verified --require-restore-ready
canic restore status <backup-ref> --require-no-attention
canic restore run <backup-ref> --execute --max-steps 1 --require-no-attention
canic restore status <backup-ref> --require-complete --require-no-attention
```

The exact CLI spelling may evolve, but the release is only complete when the
operator has an end-to-end command flow with durable state at each boundary.

## 2. Current Implementation Reality

The current codebase already contains the core pieces that 0.36 should validate:

- `canic-backup::plan` builds topology-aware backup plans.
- `canic-backup::execution` models backup execution journals, operations,
  receipts, validation, and resume state.
- `canic-backup::runner` executes ready backup operations through an injected
  executor.
- `canic-backup::manifest`, `persistence`, and `artifacts` validate backup
  layout and artifact integrity.
- `canic-backup::restore::plan` builds restore plans from manifests.
- `canic-backup::restore::apply` renders ready restore apply journals.
- `canic-backup::restore::runner` previews, executes, records receipts, handles
  failed/pending operations, and uses a journal lock.
- `canic-cli::restore` already wires a host command executor into restore run
  execution.

The weak point is therefore not design coverage. It is confidence:

- not enough true end-to-end backup/restore drills against an installed fleet;
- not enough tests proving the CLI runner path uses the same commands as dry-run
  previews;
- backup execution needs the same level of CLI/operator proof that restore run
  already has;
- failure, interruption, resume, and verify behavior needs integration-level
  coverage, not just model-level unit tests.

## 3. Release Principle

The 0.36 line should prioritize executable correctness over breadth.

Prefer:

- one backup path and one in-place restore path that are proven reliable;
- one local/ICP CLI execution path that resumes safely;
- typed receipts that prove what happened;
- clear status output that tells the operator the next valid command.

Defer:

- cross-project restore;
- daemon or scheduler mode;
- remote artifact transport;
- generalized receipt framework beyond the shapes needed by backup/restore v1;
- broad refactors not needed for the executable flow.

## 4. Non-Negotiable Invariants

### 4.1 Backup Is Topology-Aware

Backup targets are resolved from the installed fleet registry. The default
backup scope is the managed non-root fleet. Subtree backup remains valid only
when the selected target resolves to one connected topology region.

Root stays running by default. Root backup remains explicit maintenance work.

### 4.2 Mutation Requires Proven Preconditions

Mutating backup execution must fail before stopping any canister unless every
selected target has:

- proven control authority;
- proven snapshot-read authority;
- an accepted topology preflight;
- an accepted quiescence policy;
- a plan/journal pair with matching identity and operation graph.

Mutating restore execution must fail before loading any snapshot unless:

- backup manifest integrity passes;
- artifact integrity passes;
- restore plan matches the backup manifest and requested target mapping;
- restore apply journal is structurally valid and ready;
- any requested verification gate is accepted.

### 4.3 Journals Are The Resume Boundary

Backup and restore must not depend on hidden terminal state or manual notes.
Every mutating operation has a journal row and every completed mutation has a
receipt.

Interrupted operations move through explicit states:

```text
ready -> pending -> completed
ready -> pending -> failed
pending -> ready      # unclaim interrupted operation
failed -> ready       # explicit retry
```

No operation may execute out of order unless the plan explicitly allows that
operation class to be parallelized. V1 should assume sequential execution.

### 4.4 Restore Must Be Fail-Closed

Restore is more dangerous than backup. V1 must prefer refusing to continue over
guessing.

Fail closed when:

- the manifest and restore plan disagree;
- an artifact checksum is missing or mismatched;
- a journal operation references a missing artifact or target;
- a pending operation belongs to another process;
- the next operation changed between preview and execute;
- a snapshot upload/load receipt cannot be tied to the expected operation.

## 5. 0.36 User Workflows

### 5.1 Backup Create

Default:

```bash
canic backup create <fleet>
```

Expected behavior to prove:

1. Resolve the installed fleet through `canic-host::installed_fleet`.
2. Read the live registry and construct a `BackupPlan`.
3. Prove required authority and preflight gates.
4. Create or resume a `backup-execution-journal.json`.
5. Stop selected targets in topology order.
6. Create snapshots.
7. Start selected targets in reverse topology order.
8. Download artifacts.
9. Verify artifacts.
10. Finalize `backup-manifest.json`.
11. Mark the backup complete.

Required operator output:

- backup id and directory;
- selected target count;
- current phase;
- next operation;
- retry/resume hint on failure;
- final verification summary.

### 5.2 Backup Status

```bash
canic backup status <backup-ref>
```

Expected behavior:

- read manifest, plan, and execution journal when present;
- distinguish `dry-run`, `running`, `failed`, `incomplete`, and `complete`;
- report pending/failed operation counts;
- show the next valid resume command.

### 5.3 Backup Verify

```bash
canic backup verify <backup-ref>
```

Expected behavior:

- validate layout integrity;
- validate manifest integrity;
- validate journal/manifest consistency;
- validate artifact existence and checksums;
- validate topology hash and operation receipts where applicable.

This command must not require the fleet to still be live.

### 5.4 Restore Prepare

```bash
canic restore prepare <backup-ref> --require-verified --require-restore-ready
```

Expected behavior:

1. Resolve backup reference.
2. Verify manifest and artifacts if requested.
3. Build in-place mapping by default.
4. Reject incomplete mappings.
5. Emit a restore plan with exact operation counts and verification summary.
6. Emit the default restore apply journal beside the backup layout.

V1 should focus on in-place restore. Relocated restore can remain planned but
blocked unless all mapping and authority rules are explicit.

### 5.5 Restore Apply

```bash
canic restore apply <backup-ref> --dry-run
canic restore apply --plan <restore-plan> --backup-dir <backup-dir> --dry-run
```

Expected behavior:

- validate plan and artifacts;
- construct a ready restore apply journal;
- render exact runner operations without mutation.

`restore apply` remains no-mutation in 0.36. The normal operator path uses
`restore prepare`; explicit plan-file apply remains available for deeper
inspection and scripted workflows.

### 5.6 Restore Run

```bash
canic restore run <backup-ref> --dry-run
canic restore run <backup-ref> --execute --max-steps 1
canic restore run <backup-ref> --unclaim-pending
canic restore run <backup-ref> --retry-failed
canic restore run --journal <restore-journal> --dry-run
```

Expected execution phases:

```text
upload snapshot artifacts
stop target members
load uploaded snapshots
start target members
verify target members
finalize restore journal
```

Every command preview must match the operation that `--execute` will claim. If
the next operation changes between preview and execute, execution fails.

### 5.7 Restore Status

```bash
canic restore status <backup-ref>
canic restore status --journal <restore-journal>
```

Expected behavior:

- summarize complete/pending/failed/blocked operation counts;
- show the next operation and command;
- show whether the restore is safe to resume;
- show whether `--unclaim-pending` or `--retry-failed` is required.

## 6. Backup Execution Proof Plan

The current `canic-backup::execution` and `canic-backup::runner` code should be
treated as the primary backup runner state machine. 0.36 work should first prove
that this path is connected to CLI execution and only then fill implementation
gaps.

### 6.1 Backup Journal Operations

V1 operation kinds to verify:

- `AcceptPreflightBundle`
- `StopCanister`
- `CreateSnapshot`
- `StartCanister`
- `DownloadSnapshot`
- `VerifyArtifact`
- `FinalizeManifest`

Each operation must include:

- stable sequence number;
- target canister when applicable;
- expected role when available;
- expected artifact path when applicable;
- planned command preview;
- state;
- receipts.

### 6.2 Backup Receipts

Minimum receipt fields to verify:

- operation sequence;
- operation kind;
- target canister;
- provider: `icp-cli` or `local-replica-direct`;
- command display;
- started/finished timestamp;
- success/failure status;
- stdout/stderr tail with bounded size;
- snapshot id for create/download;
- artifact path and checksum for download/verify.

Receipts are not a generic framework in 0.36. They are the concrete proof needed
to resume backup safely.

### 6.3 Backup Resume Rules

Backup execution may resume when:

- journal validates;
- plan hash matches persisted plan;
- no pending operation is held by a live process;
- failed operation has been explicitly retried or skipped only if the operation
  kind allows skip.

V1 should not support skip for mutating operations. Retry only.

### 6.4 Backup Test Gaps

0.36 should add tests or drills for:

- real CLI backup execution against an installed local fleet;
- backup runner `max_steps` partial execution and resume;
- failed command receipt persistence and explicit retry;
- completed backup verify without live fleet access;
- tampered artifact rejection;
- interrupted stop/create/start/download sequence recovery.

## 7. Restore Execution Proof Plan

The existing restore apply journal and native runner remain the restore
execution boundary.

### 7.1 Restore Operation Kinds

V1 operation kinds to verify:

- `UploadSnapshot`
- `StopCanister`
- `LoadSnapshot`
- `StartCanister`
- `VerifyStatus`
- `VerifyFleetCheck`

The current journal model already captures much of this. 0.36 should prove that
rendered dry-run operations and actual host execution stay aligned.

### 7.2 Snapshot Upload And Load Receipts

Upload receipt must record:

- source artifact path;
- checksum;
- target canister;
- uploaded snapshot id;
- command display;
- provider metadata.

Load receipt must record:

- target canister;
- uploaded snapshot id;
- load mode;
- command display;
- success/failure status.

The load operation must not accept a snapshot id from any unrelated receipt.

### 7.3 Restore Verification

V1 verification should include:

- target canister status after start;
- expected target membership count;
- manifest artifact checksum validation;
- optional fleet endpoint checks already represented in the plan.

Richer application-level checks can come later, but the v1 system checks must
be real and visible.

### 7.4 Restore Test Gaps

0.36 should add tests or drills for:

- full restore from a completed backup, not only fixture journals;
- dry-run command preview matching the command executed next;
- upload receipt to load receipt binding;
- interrupted pending operation recovery after process exit;
- failed command receipt persistence and retry;
- restored canister status verification after start;
- manifest/plan/artifact mismatch fail-closed paths.

## 8. Ownership Boundaries

`canic-cli` owns:

- argument parsing;
- command dispatch;
- user-facing progress/status rendering;
- output-file routing;
- translating host/domain errors into operator text.

`canic-host` owns:

- ICP CLI process construction;
- direct local replica read paths;
- installed-fleet resolution;
- registry parsing;
- snapshot command execution helpers;
- bounded command output capture.

`canic-backup` owns:

- backup plans;
- manifests;
- execution journals;
- restore plans;
- restore apply journals;
- operation validation;
- receipt validation;
- progress summaries.

The CLI should not reimplement topology, journal transitions, or registry
parsing. The backup crate should not know CLI argument syntax.

## 9. Milestones

### 0.36.0 - Backup Execution Proven

- `canic backup create <fleet>` real execution is covered by a local installed
  fleet drill or integration test.
- Backup journals resume safely under partial execution and failed command
  cases.
- `backup status`, `inspect`, and `verify` understand real execution layouts.
- Completed backup verification does not require live fleet access.

### 0.36.1 - In-Place Restore Proven

- `restore plan` builds a verified in-place plan from a completed backup.
- `restore apply --dry-run` emits a ready journal.
- `restore run --execute` uploads, stops, loads, starts, and verifies against
  real backup artifacts.

### 0.36.2 - Resume And Failure Hardening

- Interrupted backup and restore runs recover cleanly.
- Failed operations can be retried with clear commands.
- Pending lock behavior is tested for both backup and restore.

### 0.36.3 - Verification And Reporting

- Backup/restore status reports are clear enough for operators.
- Receipts are bounded, stable, and included in verify output.
- Manifest/journal mismatch errors are precise.

### 0.36.4 - Operator Drill And Docs

- Add a documented local drill from fleet install through backup, destructive
  restore, and verification.
- Document local ICP persistence limits.
- Document what v1 does not cover.

## 10. Test Strategy

Unit tests:

- journal state transitions;
- receipt validation;
- manifest/journal integrity;
- restore plan mapping validation;
- command preview stability;
- failed/pending retry behavior.

CLI tests:

- option parsing for execute/status flows;
- backup status for dry-run/running/failed/complete layouts;
- restore status for ready/pending/failed/complete journals;
- output-file behavior where supported.

Integration tests:

- local fleet backup creates downloadable artifacts;
- restore applies artifacts in order;
- interrupted restore can resume;
- failed command records a receipt and blocks completion;
- verify rejects tampered artifacts.

Proof tests should prefer existing runner injection points before adding slow
PocketIC coverage. PocketIC should be used where actual canister
create/snapshot/load behavior matters.

Manual release drill:

```bash
canic replica start --background
canic install demo
canic backup create demo
canic backup verify <backup-ref>
canic restore prepare <backup-ref> --require-verified --require-restore-ready
canic restore run <backup-ref> --dry-run
canic restore run <backup-ref> --execute --max-steps 1 --require-no-attention
canic restore status <backup-ref> --require-complete --require-no-attention
```

## 11. Exit Criteria

0.36 is complete when:

- backup create can complete a real backup for a local installed fleet and the
  path is tested or captured in a repeatable release drill;
- backup verify passes on that backup without live fleet access;
- restore plan/apply/run can complete an in-place restore from that backup and
  the path is tested or captured in a repeatable release drill;
- interruption and retry paths are tested for both backup and restore;
- status commands always identify the next safe operator action;
- dangerous restore ambiguity fails closed;
- docs include one successful operator drill and known limitations.

## 12. Explicit Non-Goals

- cross-project restore by default;
- root-included maintenance backup as the default path;
- cloud artifact storage;
- parallel operation execution;
- background daemon orchestration;
- generalized receipt schema across unrelated Canic features.

## 13. Open Questions

1. Should `restore apply` remain mandatory, or should `restore run` create a
   ready journal from a restore plan when the operator passes `--execute`?
2. Should backup execution keep `backup create <fleet>` as the mutating default,
   or should `--execute` be required for one minor release to match restore?
3. What is the minimum application-level verification hook for v1, beyond
   canister status and artifact integrity?
4. Should root-mediated child snapshot/control be required for v1, or can v1
   target fleets whose controller policy already permits host execution?
