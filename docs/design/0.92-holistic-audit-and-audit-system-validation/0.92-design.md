# Canic 0.92 Design: Holistic Audit and Audit-System Validation

Date: 2026-07-14

## Status

- Status: accepted; Phase A audit-system inventory is complete and Phase B is
  next.
- Published release anchor: `v0.91.6` at
  `5f7a89f9b966ebf2755d5630ddcba0cdf968ebb1`.
- Implementation tracker: [status.md](status.md).
- 0.92 is an audit-and-hardening line, not a feature line and not a 1.0
  readiness claim.

## Decision

Canic is treated as feature complete for this line. 0.92 performs a holistic
audit of the repository and then fixes evidence-backed defects without adding
new product scope.

The existing audit suite is not accepted as an unquestioned source of truth.
0.92 first audits the audit definitions, governance, scripts, retained
evidence, and coverage map. It then improves or removes defective methods,
freezes their identities, and runs the retained improved suite against the
actual repository.

The governing sequence is:

```text
inventory audit system
  -> audit the audits
  -> fix, fingerprint, and freeze audit methods
  -> run the improved holistic suite against one product baseline
  -> deduplicate and classify findings
  -> approve bounded fix slices
  -> compare each slice to its parent and the frozen baseline
  -> publish one evidence-backed closeout
```

Audit-method changes and product changes must not be mixed into the same
baseline. Otherwise a changed measurement could be presented as a product
improvement without the product changing.

## Immutable Identities

0.92 distinguishes four repository identities:

1. **Release anchor:** the published `v0.91.6` commit and tree.
2. **Inventory snapshot:** the immutable snapshot whose audit system Phase A
   inventories.
3. **Frozen-method snapshot:** the commit and tree after accepted audit-method
   fixes, before product audit or product fixes.
4. **Product baseline:** the product tree audited by the frozen methods.

The inventory may use the release anchor when the audit-system path comparison
proves those paths are unchanged. The frozen-method snapshot and product
baseline may share a repository commit only when a path allowlist proves that
method preparation did not change product behavior.

Every run records this identity block:

```text
release_anchor:
source_commit_full:
source_tree_hash:
product_tree_hash:
clean_worktree:
cargo_lock_hash:
rust_toolchain:
target_triple:
feature_set:
audit_method_id:
audit_method_version:
audit_method_fingerprint:
audit_script_hashes:
external_tool_versions:
fixture_or_seed:
environment_class:
started_at:
completed_at:
```

`product_tree_hash` is computed from the canonical product-path allowlist and
the sorted Git object identities for those paths. The allowlist includes
maintained source, manifests, build scripts, tests, canisters, fleets, and
operator scripts; it excludes audit definitions, reports, and the 0.92 design
tracker. The exact allowlist and hashing command are versioned audit-method
inputs.

The full commit hash is mandatory. A short hash is display-only and cannot be
the sole identity in evidence.

## Canonical Authority Precedence

Audits do not silently choose whichever source agrees with the implementation.
Active authorities are reviewed in this order:

1. repository-root `AGENTS.md` and the governance documents it makes
   normative;
2. accepted active designs and finalized release/hard-cut decisions;
3. published public, serialized, stable-state, configuration, and operator
   contracts at the release anchor, unless an accepted design explicitly
   authorizes a hard cut;
4. active operator documentation and package documentation;
5. implementation and generated surfaces as evidence of actual behavior;
6. tests as evidence of enforced behavior;
7. status documents and trackers; and
8. historical reports and archived designs.

This is not permission to break stable state or published data merely because
a prose document is higher in the list. A proposed change to a published or
stable contract requires an explicit accepted hard-cut or migration decision
and its safety evidence.

Conflicting active authorities produce a `governance_conflict` finding. The
audit records all conflicting sources and stops treating the disputed rule as
a pass condition until the maintainer resolves it.

The boundary between a defect and new scope is:

- violation of an existing canonical authority: potential 0.92 defect;
- desired behavior with no current canonical authority: new feature,
  separately designed work, or downstream work.

## Why 0.92 Exists

At the design baseline, Canic contains:

- 15 recurring system audit definitions;
- 7 recurring authentication invariant definitions;
- the `MSH-2.0` module-surface policy and cleanup runner;
- 6 audit-related CI or report scripts;
- more than 1,000 retained report and evidence files; and
- approximately 11 MiB under `docs/audits`, mostly historical reports.

That breadth is valuable, but breadth is not proof of correctness. The current
meta-audit contract is materially smaller than the repository's active rules.
Definitions were written at different times, use different method-label
styles, contain subsystem-specific focus sections, overlap in several areas,
and depend on report conventions that have since changed.

A normal rerun could therefore reproduce stale assumptions, double-count one
issue, omit newer hard-cut and failure-path requirements, or generate large
artifacts without improving confidence. 0.92 makes the audit machinery itself
part of the audited system before relying on its verdicts.

## Goals

0.92 must:

1. prove which audits are current, reproducible, useful, and non-duplicative;
2. align every retained audit with current governance and actual code owners;
3. run the improved set across all maintained crates, canisters, fleets,
   scripts, tests, operator surfaces, and active documentation;
4. trace representative operations end to end rather than relying on static
   counts or status documents;
5. produce one deduplicated finding index with severity, confidence, evidence,
   owner, disposition, and confirming validation;
6. fix clear in-scope defects through bounded finding-backed slices;
7. execute concrete compatibility checks against `v0.91.6` unless a finding
   authorizes a hard cut; and
8. leave the audit suite smaller or clearer where definitions compete for
   authority.

## Non-Goals

0.92 does not:

- claim that Canic is ready for 1.0;
- replace the maintainer's requirement for at least three months of real-world
  use before a 1.0 decision;
- add new Canic product features;
- implement downstream Toko mint or multi-step claim orchestration;
- create a generic effect, receipt, saga, refund, or reconciliation framework;
- introduce compatibility aliases, deprecated wrappers, duplicate command
  paths, or legacy fallback behavior;
- turn every historical audit conclusion into a current requirement;
- rewrite historical findings to make them look current;
- fix speculative or aesthetic issues without reproducible risk;
- require one giant implementation patch; or
- add a general audit CLI or framework merely to automate a one-off check.

## Scope Boundary

The audit covers every maintained repository-owned surface.

| Area | Required coverage |
| --- | --- |
| Runtime and facade | `canic`, `canic-core`, `canic-macros`, macro expansion, generated endpoints, lifecycle glue |
| Control plane and storage | `canic-control-plane`, `canic-wasm-store`, creation, install, upgrade, reconciliation |
| Host and operator | `canic-cli`, `canic-host`, `canic-backup`, commands, JSON/text, diagnostics, exit behavior |
| Test infrastructure | `canic-testing-internal`, `canic-tests`, test canisters, PocketIC ownership, fixtures |
| Fleet examples | maintained demo/test fleets, configuration admission, generated artifacts and Candid truth |
| Build and release | Cargo graph, features, packages, Wasm production, release scripts, environment selection |
| State and durability | stable-memory ownership, migrations, snapshots, backup/restore, journals, atomic publication |
| Security | endpoint auth, caller/subject/audience/subnet binding, capabilities, replay, delegation and attestation |
| Operations | deployment truth, planning, execution, recovery, metrics, diagnostics, operator documentation |
| Governance | active designs, changelogs, status, audit definitions, scripts, report retention and links |

The coverage map must also explicitly include or explicitly exclude with
rationale:

- dependency advisories, licenses, and lockfile integrity;
- `build.rs`, procedural macro, and generated-code trust;
- unsafe code inventory and justification;
- CI workflow permissions and third-party action pinning;
- secret scanning;
- release-artifact provenance and checksums;
- reproducible or explainably non-reproducible Wasm builds; and
- the supported host and target matrix.

Repositories outside the Canic root are read-only and out of mutation scope.
Downstream gaps are recorded as downstream work, not implemented here.

## Separate State Machines

The audit system uses four independent state domains.

```text
definition_disposition:
  retain | revise | merge | split | retire | manual_only | blocked

run_result:
  pass | fail | partial | blocked | not_applicable

result_validity:
  valid | invalid | superseded

finding_status:
  open | accepted | fixed | deferred | rejected | duplicate | blocked

closeout_verdict:
  pass | pass_with_limitations | fail | blocked
```

Rules:

- a definition disposition says what happens to the method; it is not a run
  result;
- result validity records whether a historical run may still support a
  conclusion; invalid and superseded results retain their original run result
  but cannot support closeout;
- `manual_only` is a method classification, never an automatic pass;
- a required invariant audit with `partial` or `blocked` blocks the product
  baseline and 0.92 closeout;
- a conditional audit may be `not_applicable` only with evidence proving its
  trigger is absent;
- an informational or trend audit may remain `blocked` only as an explicit
  limitation, which forces `pass_with_limitations` at best;
- `fail` means the evidence establishes an unacceptable product or method
  defect;
- `blocked` means authoritative evidence cannot be produced, so a pass/fail
  conclusion is unavailable; and
- closeout is `pass` only when no required audit is incomplete and no accepted
  limitation remains.

## Phase A: Audit-System Inventory

The first 0.92 action is read-only inventory. It records the release anchor and
inventory snapshot, then maps:

- every recurring, modular, release-line, and CI-backed audit;
- its owner, purpose, trigger, scope, exclusions, method tag, and output;
- the repository rules and code sinks it claims to enforce;
- required commands and external tools;
- its latest comparable run, if one exists;
- overlap with other audits;
- retained evidence and artifact cost;
- positive and known-invalid fixtures or equivalent negative proof; and
- coverage of the additional holistic areas listed above.

The inventory distinguishes:

- invariant audits from trend or pressure audits;
- mechanical guards from manual architecture reviews;
- reusable definitions from historical reports;
- deterministic failures from narrative checklists;
- current product scope from archived or downstream scope; and
- necessary raw evidence from reproducible generated noise.

Historical report count, size, or repetition is not treated as coverage.
Coverage requires a maintained definition and proof that it reaches the
current code owner.

## Phase B: Audit The Audits

Every candidate method is evaluated against one conformance contract.

### Required definition contract

A retained audit must state:

- stable audit ID and explicit method version;
- owner and intended trigger;
- current scope and exclusions;
- canonical rules or invariants being tested;
- actual source-of-truth files and code sinks;
- deterministic commands where practical;
- manual reasoning required where mechanical evidence is insufficient;
- false-positive exclusions and boundary cases;
- severity, confidence, and overall risk rules;
- required report fields and verification statuses;
- tool prerequisites, cost class, and expected runtime;
- comparability and method-change rules;
- artifact-retention and redaction requirements;
- positive, rejection, boundary, and regression fixtures where executable
  proof is appropriate; and
- follow-up ownership for `fail`, `partial`, or `blocked` results.

### Audit-system review questions

The meta-audit asks:

1. Does the method enforce current `AGENTS.md`, governance, architecture,
   hard-cut, data-shape, lifecycle, testing, security, and release rules?
2. Does it inspect actual reachable source, or can status prose satisfy it?
3. Can renamed layers, new crates, generated code, or stale filenames escape?
4. Does another method already own the same decision or metric?
5. Are severity, confidence, and risk reproducible between reviewers?
6. Are commands valid, safely scoped, and read-only where promised?
7. Can absence of evidence be distinguished from a pass?
8. Are scripts and narrative definitions testing the same contract?
9. Is retained evidence necessary, bounded, hashed, and redacted?
10. Are typed causes and exact failures preserved?
11. Are positive, rejection, boundary, and regression cases covered?
12. Is the method worth retaining relative to runtime, artifact volume, and
    overlap?

### Definition dispositions

Every definition receives exactly one `definition_disposition`:

- `retain` — current and uniquely owned;
- `revise` — useful but method or scope is stale;
- `merge` — overlaps another method whose owner becomes canonical;
- `split` — combines unrelated authorities;
- `retire` — obsolete or supplies no distinct evidence;
- `manual_only` — valuable review that cannot honestly claim a deterministic
  gate; or
- `blocked` — required definition whose method cannot currently be made
  authoritative.

### Manual-only protocol

A `manual_only` definition still requires:

- stable method ID, version, and fingerprint;
- exact commit and files reviewed;
- versioned checklist or trace protocol;
- code references and sampled paths;
- explicit unreviewed boundaries;
- named reviewer;
- a second review or explicit single-review waiver for a P0/P1 finding; and
- disagreement and escalation procedure.

### Retired-method integrity

When methods merge or retire, active indexes move to the canonical method and
no compatibility audit wrapper remains. Before deletion:

- fingerprint the exact definition;
- preserve it in an immutable method archive or an immutable tagged source
  snapshot;
- record old path, audit ID, version, fingerprint, and replacement in a
  retirement manifest;
- scan current-tree historical links; and
- make permitted link-only repairs to point at immutable evidence without
  changing historical findings.

No method may be deleted if that would make the only evidence for a historical
finding unrecoverable.

## Phase C: Improve And Freeze The Audit System

Accepted audit-system findings are fixed before a holistic product baseline is
recorded. Allowed changes include:

- expanding `META-AUDIT.md` to current invariants;
- normalizing metadata, state, severity, confidence, and method-version rules;
- correcting stale scopes, commands, file maps, and subsystem assumptions;
- merging or deleting competing definitions;
- adding small deterministic scripts or fixtures for repeated checks;
- making scripts fail closed with exact diagnostic causes;
- adding missing coverage for a current owner or public contract;
- removing redundant generated formats under the retention policy; and
- adding link, catalog, method-schema, report-manifest, or structure checks
  where they provide durable value.

The line does not build an abstract audit engine merely to host heterogeneous
checks. Plain definitions, focused scripts, Cargo tests, and current CI helpers
remain preferred.

After improvement:

1. each retained definition receives a frozen method ID, version, and content
   fingerprint;
2. referenced scripts, fixtures, allowlists, and trace templates are included
   in that fingerprint or named by exact hash;
3. method tests and dry runs pass;
4. the frozen-method commit and tree are recorded;
5. the product-tree allowlist proves whether product behavior changed during
   method preparation; and
6. no product finding is compared numerically across a method change.

### Post-freeze method-defect protocol

If a method defect is discovered after freeze:

1. set every affected run's `result_validity` to `invalid` in a superseding
   report; do not rewrite its historical evidence or original `run_result`;
2. record an `audit_method_defect` finding;
3. increment the method version and fingerprint;
4. rerun the corrected method against the original product baseline;
5. rerun it against the current fix commit;
6. compare only results produced by the corrected method; and
7. set closeout to `blocked` or explicitly non-comparable when the original
   baseline cannot be reproduced.

Direct end-to-end traces are versioned audit methods and follow the same
freeze and defect protocol. They are not an unversioned path around method
identity.

## Phase D: Run The Improved Holistic Audit

The improved suite runs against one recorded product snapshot before product
fixes begin. Runs may span days and cost classes, but every baseline report
uses the same product identity.

### Trace modes

Every trace declares one mode:

- `code_trace` — follows reachable code, owners, state transitions, errors,
  diagnostics, and public projection without executing the operation;
- `execution_trace` — executes the behavior only in an explicitly permitted,
  disposable environment such as PocketIC or a named test deployment.

### Execution safety contract

“Read-only” means read-only with respect to tracked source and authoritative
environments. Build caches and isolated temporary outputs are allowed.

Every execution run must enforce:

- no production or mainnet mutation;
- no production credentials;
- network disabled by default and enabled only for a named authorized audit;
- a disposable Git worktree or equivalent immutable source snapshot;
- isolated temporary state and `CARGO_TARGET_DIR`;
- pre-run and post-run `git status --porcelain` evidence;
- source-tree mutation causes the run to fail;
- destructive operations require explicit named authorization;
- secrets, tokens, private material, sensitive principals, private paths, and
  environment secrets do not enter retained evidence; and
- cleanup removes disposable runtime state without deleting retained primary
  evidence.

### Audit waves

1. **Repository truth:** module topology, dependencies, visibility, publish
   surface, dead code, duplicate ownership, hard-cut residue, unsafe inventory,
   lockfile and license/advisory posture.
2. **Architecture and data:** layering, DTO/view/record boundaries, policy and
   workflow purity, conversions, errors, diagnostics, canonical state owners.
3. **Security and lifecycle:** endpoint guards, capability and token
   invariants, replay, delegation, secret exposure, bootstrap, upgrade, stable
   memory.
4. **Operator and deployment:** CLI parsing/output, environments, planning,
   manifests, install/upgrade, deployment truth, recovery and projection.
5. **Durability and operations:** Wasm store, control plane, cycles refill,
   intents, backup/restore, publication, failure and resume behavior.
6. **Build, CI, and release:** `build.rs`, macros, generated code, workflow
   permissions, action pinning, artifact provenance, checksums, host/target
   matrix, reproducibility.
7. **Tests and documentation:** positive/rejection/boundary/regression proof,
   examples, READMEs, Candid, changelogs, status, active designs and audits.
8. **Measured runtime cost:** Wasm and instruction evidence through frozen
   methods and bounded raw artifacts.

Expensive PocketIC, Wasm, instruction, or release checks are named audit waves
with progress reporting. Full workspace or broad deployment gates remain
maintainer-owned unless explicitly authorized.

### Mandatory trace IDs

Static scans are insufficient. Each mandatory trace has a stable ID and uses
the frozen trace method:

- `TRACE-DEPLOY-001`: configuration -> build -> release-set manifest ->
  install/upgrade -> deployment truth;
- `TRACE-AUTH-001`: root provisioning -> chain-key/delegation proof -> issuer
  readiness -> token verification;
- `TRACE-CAPABILITY-001`: endpoint guard -> capability/attestation -> replay ->
  workflow -> mutation;
- `TRACE-CYCLES-001`: cycles funding and ICP refill -> ledger/notify outcome ->
  retry identity -> CLI projection;
- `TRACE-INTENT-001`: local and receipt-backed admission -> accounting ->
  settlement -> replay -> upgrade recovery;
- `TRACE-CONTROL-001`: control-plane and Wasm-store publication -> lookup ->
  reconciliation -> conflicting release rejection;
- `TRACE-TOPOLOGY-001`: scaling, sharding, topology, pool, parent and subnet
  binding;
- `TRACE-BLOB-001`: blob request -> billing/cashier -> authorization ->
  persistence -> recovery;
- `TRACE-BACKUP-001`: plan -> journal -> snapshot -> checksum -> manifest ->
  restore preview/execute/recovery; and
- `TRACE-LIFECYCLE-001`: restore -> timer scheduling -> user hooks -> stable
  state and metrics projection.

Every trace report includes:

```text
trace_id:
trace_method_id:
trace_method_version:
trace_method_fingerprint:
mode:
entrypoint:
caller_and_auth_context:
canonical_owner_per_transition:
persistent_state_touched:
external_calls:
retry_and_replay_identity:
rejection_paths:
partial_failure_paths:
upgrade_or_restore_boundary:
diagnostic_projection:
public_projection:
tests_or_evidence:
unreviewed_boundaries:
verdict:
findings:
```

## Finding Identity And Triage

Canonical finding identity is assigned during triage from its canonical owner
and violated invariant, not discovery order. Every reporting audit remains
listed as a source.

```text
finding_id: CANIC-092-<OWNER>-<NNN>
finding_class:
severity: P0 | P1 | P2 | P3
confidence: confirmed | high | medium | low
finding_status:
owner:
current_location:
intended_owner:
affected_surfaces:
source_audit_ids:
source_method_fingerprints:
baseline_commit:
first_observed_at:
evidence:
typed_cause_or_invariant:
risk:
verification_status:
duplicate_of:
recommended_slice:
fix_commit:
validation_commit:
waiver:
disposition:
```

`finding_class` is one of:

- `product_defect`;
- `audit_method_defect`;
- `evidence_gap`;
- `governance_conflict`;
- `operational_risk`; or
- `documentation_drift`.

Severity measures impact. Confidence measures evidence strength. A high-impact
hypothesis may have low confidence; a proven low-impact drift may have high
confidence. Missing critical regression proof is classified as an
`evidence_gap`, even when its impact warrants P1 treatment.

Duplicate discoveries receive `finding_status: duplicate` and `duplicate_of`
the triage-assigned canonical ID. Audit count does not multiply severity.

### Severity policy

- **P0:** confirmed or credibly imminent auth bypass, unsafe mutation,
  stable-state corruption, destructive deployment error, or loss of
  authoritative recovery evidence.
- **P1:** active hard-cut violation, competing authority, fail-open safety
  path, public/serialized contract contradiction, lost typed cause, serious
  operational risk, or critical evidence gap.
- **P2:** concrete drift risk from duplication, excess visibility, stale
  models, unclear ownership, dependency pressure, misleading active docs, or
  weak boundary coverage.
- **P3:** bounded clarity, organization, or low-risk hygiene issue with
  reproducible evidence.

## Phase E: Finding-Backed Fix Slices

The holistic product audit is read-only until its baseline and finding index
are reviewed. A finding does not authorize an unrelated refactor.

Each accepted fix slice states:

- finding IDs;
- authority being corrected;
- behavior-impact label;
- public, Candid, JSON, configuration, and stable-state impact;
- exact deletion list for obsolete paths;
- targeted positive, rejection, boundary, and regression tests;
- affected audits and traces to rerun; and
- explicit non-scope.

Allowed behavior-impact labels are:

- `no_behavior_change`;
- `behavior_change_declared`;
- `serialized_surface_hard_cut`;
- `stable_state_change_declared`; and
- `operator_contract_change_declared`.

Pre-1.0 removals are complete hard cuts. The obsolete path is removed from
code, tests, examples, and active documentation. No alias, shim, fallback,
deprecated API, or duplicate command path remains. A structural regression
guard may be retained only when it enforces the current authority without
preserving or exposing the removed surface.

### P0 and P1 closeout policy

P0 findings always block closeout until fixed.

These P1 categories are non-waivable:

- active fail-open authorization or safety behavior;
- credible destructive deployment behavior;
- stable-state integrity risk;
- unresolved public contract contradiction that can corrupt or misinterpret
  data; and
- inability to produce authoritative recovery evidence.

Any other P1 blocks closeout until fixed or explicitly accepted by the
maintainer with:

```text
approved_by:
accepted_at:
rationale:
compensating_controls:
affected_users_or_operators:
disclosure_location:
revisit_condition:
expires_at_or_release:
```

An unresolved accepted P1 forces `closeout_verdict: pass_with_limitations`.
P2 findings must be fixed, rejected, or explicitly deferred. P3 fixes are
optional and remain bounded.

## Executable v0.91.6 Contract Baseline

Unless an accepted finding authorizes a hard cut, 0.92 compares current
behavior to the published `v0.91.6` release through:

- generated Candid and interface comparison;
- JSON, configuration, and boundary-schema comparison;
- CLI command, exit-code, diagnostic, and structured-output comparison;
- stable-state upgrade tests using state produced by `v0.91.6`;
- backup/restore compatibility or an explicit migration boundary;
- package features and published-surface comparison; and
- changelog and operator migration text for every accepted hard cut.

A declared stable-state change additionally requires:

- pre-change state fixtures;
- migration and post-migration validation;
- interrupted-upgrade behavior;
- rollback or recovery expectations; and
- proof that unsupported rollback cannot silently corrupt state.

## Phase F: Confirming Reruns And Comparison

After each fix slice:

- run targeted tests for the changed owner and behavior;
- rerun affected audits and traces with the same frozen method;
- perform a **causal comparison** against the slice's immediate parent commit;
- perform a **cumulative comparison** against the original product baseline;
- validate the finding against the immediate parent comparison;
- report release-line totals against the original baseline;
- invoke the post-freeze method-defect protocol when a method changes; and
- retain only necessary, hashed, redacted evidence.

Comparing only to the original baseline is insufficient for slice causality.
Comparing only to the parent is insufficient for cumulative closeout.

## Evidence Integrity And Redaction

Every run has one compact evidence manifest containing:

```text
command:
working_directory:
exit_code:
stdout_path:
stderr_path:
baseline_identity:
method_identity:
tool_versions:
timestamps:
artifact_hashes:
retention_class:
redactions_applied:
```

Controls:

- retained evidence uses SHA-256 or an equivalent recorded digest;
- absent stdout/stderr files are recorded as `not_retained`, not omitted;
- logs are scrubbed for credentials, tokens, private material, sensitive
  principals, private paths, and environment secrets;
- the unredacted sensitive source is never committed merely to prove a hash;
- evidence links are checked before pruning;
- the only evidence for a finding is never removed;
- raw artifacts are retained only when the Markdown report is insufficient for
  reproduction or future comparison; and
- merged or retired definitions remain recoverable by immutable fingerprint.

Historical Markdown findings remain append-only. Superseding reports may mark
results invalid, corrected, or obsolete without rewriting the original
conclusion.

## Closeout

0.92 closes with:

- dated primary audit-system and holistic reports under
  `docs/audits/reports/YYYY-MM/YYYY-MM-DD/`;
- required day and month summaries;
- the final deduplicated finding index in the tracker;
- `docs/audits/release-lines/0.92-closeout.md`;
- explicit deferred, rejected, waived, blocked, and not-applicable decisions;
- current definitions and scripts matching the closeout fingerprints; and
- one explicit `closeout_verdict`.

`pass` requires every required audit and trace to complete and no limitation
to remain. `pass_with_limitations` names every waiver or informational block.
`fail` means a known unacceptable defect remains. `blocked` means required
authoritative evidence could not be produced.

## Validation Strategy

Validation is proportional to each audit and finding. The line uses:

- method-conformance, catalog, internal-link, and evidence-manifest checks;
- script syntax and deterministic fixtures where scripts own decisions;
- targeted Cargo check, Clippy, and tests for changed packages;
- focused PocketIC tests for creation, install, upgrade, inter-canister, and
  stable-state findings;
- package and generated-surface checks for publish findings;
- Wasm and instruction measurements only through frozen methods;
- `cargo fmt --check` and `git diff --check` for every implementation slice;
  and
- explicit maintainer authorization before broad workspace, deployment,
  publish, or release-matrix gates.

Tests assert typed causes and observable state at internal boundaries. Exact
text, JSON fields, diagnostics, and exit codes are asserted where those
operator-facing surfaces are documented contracts.

Audit reports preserve exact command failures and causes as `fail` or
`blocked`; unavailable tools or incomplete runs never become `pass`.

## Acceptance Criteria

0.92 is complete only when:

1. every maintained definition has a conformance disposition;
2. retained methods satisfy the current definition contract and map to active
   authority;
3. overlapping authorities are merged, split, or assigned one owner;
4. improved methods, scripts, fixtures, allowlists, and traces are
   fingerprinted and frozen before product baselining;
5. audit-method preparation is proven not to have changed the product tree, or
   those product changes are separately reviewed before baseline;
6. every maintained workspace area and additional holistic topic has explicit
   coverage or a documented manual method;
7. the complete retained improved suite has run against one product baseline;
8. every required run is `pass` or supported by an accepted product finding;
   no required run remains `partial`, `blocked`, or unjustifiably
   `not_applicable`;
9. mandatory traces have run under frozen trace methods;
10. findings are evidence-backed, triage-identified, deduplicated,
    severity-reviewed, confidence-rated, owned, and dispositioned;
11. accepted fixes have causal and cumulative confirming comparisons;
12. no P0 or non-waivable P1 remains unresolved;
13. every waived P1 carries the complete waiver record and forces
    `pass_with_limitations`;
14. `v0.91.6` compatibility is executable evidence or an accepted hard cut
    with migration and operator documentation;
15. public APIs, models, errors, diagnostics, tests, examples, and active docs
    agree at closeout;
16. failure paths are fail-closed, deterministic, and preserve typed causes;
17. obsolete behavior is removed without compatibility resurrection;
18. report and method archives satisfy retention, fingerprint, link, and
    redaction rules;
19. closeout states exact limitations and does not imply 1.0 readiness; and
20. package versions change only through explicit maintainer-owned release
    preparation.

## Closed Design Decisions

- 0.92 audits the audit system before running it.
- Release, inventory, frozen-method, and product identities are separate.
- Method fixes precede and are separated from the product baseline.
- Definition disposition, run result, finding status, and closeout verdict are
  independent state machines.
- Required partial or blocked audits prevent baseline and closeout.
- Code and execution traces are distinct and use versioned methods.
- Post-freeze method defects invalidate and supersede affected results.
- Canonical authority conflicts become findings rather than silent choices.
- Finding identity is assigned during owner/invariant triage, not discovery.
- Each fix receives parent-causal and baseline-cumulative comparisons.
- `v0.91.6` is an executable compatibility baseline.
- Evidence is hashed, bounded, redacted, and linked to immutable method
  identities.
- No compatibility layer preserves removed pre-1.0 behavior.
- No new product feature or downstream Toko work enters this line.
- 0.92 is not a 1.0 readiness or release-candidate claim.
- Real-world use remains a separate prerequisite for a future 1.0 decision.
