# Canic Backup/Restore Rework

- **Version:** `0.34`
- **Status:** Superseded by `docs/design/0.35-backup-restore/0.35-design.md`
- **Scope:** redesign subtree backup and restore around Canic fleet topology
- **Depends on:** `0.33` ICP CLI cutover and fleet-scoped command cleanup

## 1. Summary

The current backup flow is too canister-local. It loops over canisters and runs
stop, snapshot, download, and optional start for each member in sequence. That
worked as a first host-side snapshot wrapper, but it does not match Canic's
hierarchical fleet model.

`0.34` should treat backup and restore as topology operations:

- a backup may target any connected subtree of a fleet
- root normally remains running so it can keep the rest of the fleet alive and
  coordinate selected-child lifecycle and snapshot creation
- selected subtree members are quiesced together, snapshotted together, and then
  restarted before slow artifact downloads happen
- restore uses the same topology contract and applies snapshots through an
  explicit plan and journal

This is a design correction, not a patch to the current loop.

## 2. Problems Found In 0.33

The 0.33 live checks exposed four concrete issues:

- `icp canister call --json` returns an ICP CLI transport wrapper for live
  calls, while the backup parser expected decoded registry JSON.
- human snapshot-create output can contain table units such as `MiB`; snapshot
  ids must be parsed from quiet output or a typed receipt, not scraped from the
  last token of a table.
- fresh snapshot downloads must not pass `--resume`; that flag is only for an
  existing interrupted download.
- child canisters are controlled by root, so direct operator identity calls
  cannot stop, start, or snapshot children unless controller policy is changed.

The fourth issue is the important design signal. Child lifecycle and snapshot
operations need root-aware orchestration. A narrow subtree backup must not stop
root or unrelated fleet branches.

## 3. Invariants

### 3.1 Root Stays Running By Default

Normal backups must not stop root.

Root is the fleet control plane and the natural coordinator for children.
If backing up one subtree required stopping root, the rest of the application
would be affected and root could no longer coordinate child operations.

Root is excluded from normal backup target expansion unless the operator selects
maintenance mode. Root may coordinate child backup operations while remaining
outside the captured subtree.

Root snapshot capture is a separate maintenance mode. It should be explicit and
clearly marked as whole-fleet-impacting.

### 3.2 Backup Scope Is A Connected Subtree

A normal backup target is one of:

- one non-root member
- one member plus all descendants
- the whole managed non-root fleet

Membership comes from the root registry/topology, not from operator-supplied
lists. Manual explicit canister backup may remain as an escape hatch, but it is
outside the fleet-subtree consistency contract.

A selected target must resolve to exactly one live topology node before planning.
Ambiguous role matches fail closed unless the operator supplies a unique role
path or principal.

### 3.3 Quiescence Is Phased

The consistency point is snapshot creation, not artifact download.

For a selected subtree:

```text
discover topology
validate selected subtree
quiesce selected members
create snapshots for selected members
restart selected members
download snapshot artifacts
verify artifacts
write manifest and final journal state
```

Downloads can be slow. They should happen after selected canisters are restarted.

### 3.4 Ordering Is Topology-Aware

Stop order:

```text
selected subtree root -> descendants
```

Start order:

```text
descendants -> selected subtree root
```

The selected subtree root is stopped first to prevent normal ingress from
continuing into the subtree while descendants are being quiesced. It is started
last so descendants are ready before ingress resumes.

Snapshot creation can follow the stopped order or a deterministic stable order
after every selected member is stopped. The manifest must record the actual
operation order.

### 3.5 Authority Evidence Is Explicit

Authority is tracked separately from authority source. Every selected target has
two evidence fields:

- `control_authority: Proven | Declared | Unknown`
- `snapshot_read_authority: Proven | Declared | Unknown`

Evidence means:

- `Proven`: acceptable for mutating backup execution.
- `Declared`: acceptable for dry-run and planning output only.
- `Unknown`: acceptable for plan construction only; execution fails closed.

Planning may produce a useful plan with declared or unknown authority so the
operator can see the intended topology and required phases. Mutating execution
requires every selected target to have proven control authority and proven
snapshot read authority before any stop or snapshot operation runs.

### 3.6 Control Authority Is Proven Before Mutation

For non-root fleet members, lifecycle and snapshot operations must be performed
by an authorized controller path.

The control authority preflight can prove one of:

- root is a controller of each selected member
- the operator identity is a controller of each selected member
- the selected role declares an explicit alternate controller strategy

If no control authority proof exists for every selected member, dry-run planning
may still render the plan, but execution fails before any canister is stopped.

### 3.7 Snapshot Read Authority Is Proven Before Mutation

Creating a snapshot is not enough. The host must also be able to read and
download the snapshot after root creates it.

The snapshot read authority preflight can prove one of:

- the operator identity can read snapshots for each selected member
- each selected member's `snapshot_visibility` allows the operator
- root configures a bounded read path before snapshot creation
- Canic uses an explicit root-mediated artifact transfer protocol

If no read authority proof exists, dry-run planning may still render the plan,
but execution fails before any canister is stopped.

### 3.8 Manifest Is Intent, Journal Is Progress

The backup manifest records what was captured and the topology/compatibility
contract needed for restore.

The journal records execution state and resumability:

- selected targets
- operation phases
- stop/create/start/download/verify receipts
- snapshot ids
- artifact temp paths and final paths
- retry state
- failure state

Do not mix mutable progress into the manifest.

## 4. Operator Shape

The likely public commands should be:

```bash
canic backup create <fleet> --subtree <role-path-or-principal> --recursive
canic backup verify <backup-dir>
canic restore plan <fleet> <backup-dir>
canic restore run <restore-journal>
```

`canic snapshot download` can remain as a lower-level command only if it is
clearly scoped as a manual snapshot primitive. The fleet-aware command should be
`canic backup create`.

Default behavior for `canic backup create <fleet>` should back up the managed
non-root fleet. Early 0.34 slices keep this dry-run only, with optional
`--subtree <role-or-principal>` selection for one connected subtree:

```bash
canic backup create <fleet> --dry-run
canic backup create <fleet> --subtree <role-or-principal> --dry-run
```

Dry-run resolves the installed fleet root, reads the live root registry, writes
`backup-plan.json`, writes the matching `backup-execution-journal.json`, and
verifies the journal is bound to the stored plan. It must not stop, snapshot,
start, download, or verify artifacts.

A root snapshot requires explicit maintenance mode, for example:

```bash
canic backup create <fleet> --include-root --maintenance
```

## 5. Backup Plan Model

Introduce a typed `BackupPlan` in `canic-backup`:

```text
BackupPlan
  fleet
  network: local | ic
  root_canister_id
  selected_subtree_root
  selected_scope_kind: member | subtree | non_root_fleet | maintenance_root
  include_descendants
  root_included
  requires_root_controller
  snapshot_read_authority: { source, evidence }
  quiescence_policy: crash_consistent | root_coordinated | app_quiesced
  topology_hash_before_quiesce
  targets: Vec<BackupTarget>
  phases: Vec<BackupOperation>
```

`BackupTarget` should include:

- canister id
- role
- parent canister id
- depth
- control authority: source plus `Proven | Declared | Unknown` evidence
- snapshot read authority: source plus `Proven | Declared | Unknown` evidence
- identity mode
- expected module hash when available

`BackupOperation` should be explicit:

- `ValidateTopology`
- `ValidateControlAuthority`
- `ValidateSnapshotReadAuthority`
- `ValidateQuiescencePolicy`
- `Stop`
- `CreateSnapshot`
- `Start`
- `DownloadSnapshot`
- `VerifyArtifact`
- `FinalizeManifest`

The current driver trait should move from canister-local methods to
operation/receipt methods:

```text
execute_backup_operation(plan_id, operation) -> BackupReceipt
```

`ValidateControlAuthority` and `ValidateSnapshotReadAuthority` upgrade declared
or unknown authority into target-scoped proven authority receipts. If either
preflight cannot produce proof for every selected target, execution stops before
mutation.

Authority preflight receipts should remain separate from ordinary operation
success receipts:

```text
ControlAuthorityPreflightRequest
  plan_id
  run_id
  fleet
  network
  root_canister_id
  requires_root_controller
  targets: [{ canister_id, role, parent_canister_id, declared_authority }]

SnapshotReadAuthorityPreflightRequest
  plan_id
  run_id
  fleet
  network
  root_canister_id
  targets: [{ canister_id, role, parent_canister_id, declared_authority }]

TopologyPreflightRequest
  plan_id
  run_id
  fleet
  network
  root_canister_id
  selected_subtree_root
  selected_scope_kind
  topology_hash_before_quiesce
  targets: [{ canister_id, parent_canister_id, depth }]

QuiescencePreflightRequest
  plan_id
  run_id
  fleet
  network
  root_canister_id
  selected_subtree_root
  quiescence_policy
  targets: [{ canister_id, role, parent_canister_id }]

ControlAuthorityReceipt
  plan_id
  preflight_id
  target_canister_id
  authority: { source, evidence }
  proof_source: root_coordination | management_status | declaration | unknown
  validated_at
  expires_at

SnapshotReadAuthorityReceipt
  plan_id
  preflight_id
  target_canister_id
  authority: { source, evidence }
  proof_source: root_coordination | snapshot_read_check | declaration | unknown
  validated_at
  expires_at

TopologyPreflightReceipt
  plan_id
  preflight_id
  topology_hash_before_quiesce
  topology_hash_at_preflight
  targets
  validated_at
  expires_at

QuiescencePreflightReceipt
  plan_id
  preflight_id
  quiescence_policy
  accepted
  targets
  validated_at
  expires_at

BackupExecutionPreflightReceipts
  plan_id
  preflight_id
  validated_at
  expires_at
  topology
  control_authority[]
  snapshot_read_authority[]
  quiescence
```

Applying receipts to a plan must fail closed if a receipt targets an unknown
canister, duplicates another receipt, belongs to another plan, is only declared,
belongs to another preflight run, has expired, is not yet valid, or fails to
cover every selected target.

Topology and quiescence receipts remain execution gates rather than plan
mutation. Execution must reject topology hash drift, changed target sets,
quiescence policy mismatches, or a quiescence receipt that is not accepted.

Execution must validate receipts against the current mutation time, not only the
time the receipt was produced. Yesterday's authority or topology proof must not
authorize today's mutation after controller, registry, or snapshot visibility
state may have changed.

This lets the CLI, host, and future root-backed orchestration share one backup
model.

## 6. Root Coordination Contract

Add a small root-owned backup coordination surface rather than exposing generic
management power.

Candidate endpoints:

```text
canic_backup_prepare(BackupPrepareRequest) -> BackupPlanReceipt
canic_backup_stop(BackupOperationRequest) -> BackupOperationReceipt
canic_backup_create_snapshot(BackupOperationRequest) -> SnapshotReceipt
canic_backup_start(BackupOperationRequest) -> BackupOperationReceipt
canic_backup_status(BackupRunId) -> BackupRunStatus
```

These endpoints should:

- authenticate the caller as an operator/root admin
- validate the selected targets against the live registry
- reject attempts to stop root in normal backup mode
- reject targets outside the selected subtree
- call the management canister as root/controller
- return typed receipts with bounded diagnostic text

Root-directed lifecycle and snapshot operations require root to be a controller
of every selected child. If root is not a controller, root coordination is not
available and the plan must use an explicit alternate authority strategy or fail
closed.

Do not model backup or restore around CLI-routed management calls. The stable
boundary is Canic-owned root coordination plus typed receipts.

## 7. Host Artifact Transfer

The host owns artifact bytes and local durability:

- download snapshot artifacts
- resume interrupted transfers
- verify artifact checksums
- write manifests
- write and resume journals
- archive or move backup directories

Root should not own host filesystem concerns. Root coordinates live fleet
mutation and returns receipts; the host persists artifacts and execution
evidence.

The host transfer adapter may use ICP CLI, direct management API calls, or a
future Canic transport, but it must be selected behind the backup driver
boundary. ICP CLI is an adapter, not the backup model.

## 8. Restore Model

Restore should mirror backup:

```text
load backup manifest
resolve target fleet
build restore plan
validate topology and compatibility
quiesce target subtree
upload artifacts
restore uploaded snapshot ids
restart target subtree
run verification
finalize journal
```

Restore must never use source snapshot ids as target restore ids. It must use
uploaded target-side snapshot ids recorded in the restore journal.

Restore ordering:

- stop target subtree top-down
- upload snapshots in deterministic order
- load/restore snapshots according to declared dependency policy
- start descendants before selected subtree root
- run verification from the subtree root outward

Root restore remains maintenance mode. Restoring root can affect fleet
coordination and must be explicitly separated from ordinary subtree restore.
Root restore cannot depend on root-owned coordination endpoints from the same
root instance being restored.

## 9. Compatibility And Verification

The backup manifest should record:

- Canic version
- package/canister version metadata
- role
- parent
- module hash when available
- control authority proof
- snapshot read authority proof
- identity mode: fixed or relocatable
- backup scope and excluded members
- topology hash before quiesce
- topology hash after restart when available

Restore planning should fail closed unless:

- every source member maps to exactly one target member
- fixed-identity members map to the same principal
- relocatable members match role and compatibility metadata
- target topology can represent the source subtree
- required verification checks are known

Verification should be role-aware, not just status-based. `canic_ready` is a
minimum check, not proof of semantic restore correctness.

Default fixed-identity roles:

- root
- hubs and registries
- wasm stores whose principals are stored by other members
- directory/index canisters
- ledger, payment, and accounting canisters
- any canister whose principal is embedded in another member's stable state

Default relocatable roles:

- stateless workers
- shards whose parent registry can update their principal
- cache/read replicas
- ephemeral processing canisters

These defaults are starting policy only. Final identity mode should become an
explicit role contract.

Minimum verification checks:

- all roles: canister running, module hash policy satisfied, `canic_ready`
- root: topology hash matches, selected subtree present, registry consistent
- wasm store: catalog query succeeds, required wasm hashes present
- directory/index: expected member entries present, parent/root link correct
- shard: parent link correct, shard id/range/partition metadata correct
- scaling/worker: registration visible from parent, ready query succeeds

## 10. Journal And Resumability

Backup and restore journals should support safe resume at phase boundaries.

For backup:

- execution journals are built from `BackupPlan.phases`
- mutating operations stay blocked until the execution preflight bundle is
  accepted
- preflight acceptance records `preflight_id` and unblocks mutation without
  calling root/ICP itself
- typed preflight receipt bundle acceptance must verify the bundle `plan_id`
  matches the journal before mutation is unblocked
- operation transitions advance in plan order through
  `ready -> pending -> completed | failed | skipped`
- operation receipts must match `plan_id`, `run_id`, `preflight_id`, operation
  sequence, operation id, kind, and target
- failed operations remain the next transitionable operation until retried
- if stop succeeded but snapshot create failed, start must be attempted before
  returning failure unless the operator requested maintenance mode
- if snapshot create succeeded and download failed, the subtree should already
  be restarted and the journal can resume download
- if artifact verification failed, retry should redownload or reverify according
  to recorded state

The journal must expose `restart_required` whenever one or more selected
canisters have been stopped and the matching start phase has not completed or
been intentionally skipped. This lets the runner and operator see that recovery
is required before normal backup failure handling can stop.

`BackupLayout` should persist the immutable plan as `backup-plan.json` and the
phase journal as `backup-execution-journal.json`, separate from the existing
artifact download journal. The persisted plan is the exact operation graph that
receipts and journals bind to through `plan_id`/`run_id`; the execution journal
owns phase progress and receipts; the download journal continues to own artifact
durability/checksum state until the old snapshot primitive is retired or folded
into the phased execution runner.

Resume must verify the persisted plan and execution journal together before any
operation transition. At minimum, `plan_id`, `run_id`, operation count,
operation sequence, operation id, operation kind, and target canister id must
match. A journal that is valid by itself but belongs to a different plan must
fail before mutation.

For restore:

- uploaded snapshot ids are durable receipts
- failed restore steps preserve receipt context
- retry never skips earlier required work
- restore execution remains locked across mutation claims and writes

## 11. 0.34 Implementation Plan

1. Freeze current `canic snapshot download` as a manual primitive and document
   its limits.
2. Add `BackupPlan`, `BackupTarget`, `BackupOperation`, and receipt DTOs in
   `canic-backup`.
3. Add explicit `Proven | Declared | Unknown` authority evidence plus planning
   preflights for topology, control authority, snapshot read authority, and
   quiescence policy.
4. Change backup execution to phased operations: stop all selected targets,
   create all snapshots, start all selected targets, then download artifacts.
5. Add root-mode policy: normal fleet backups exclude root from stop/snapshot
   unless maintenance mode is explicit.
6. Add root coordination endpoints for selected-child lifecycle and snapshot
   creation.
7. Wire `canic backup create <fleet> --subtree ...` to the new plan model.
8. Rework restore planning around the same subtree model and uploaded snapshot
   receipts.
9. Add PocketIC tests for subtree backup, partial subtree backup, root-stays-up,
   failure restart, restore mapping, and journal resume.
10. Add one local smoke path for `local` and one dry-run/read-only path for
    `ic`.

## 12. Open Questions

- Should normal `canic backup create <fleet>` become shorthand for the managed
  non-root fleet after the explicit subtree path is proven?
- Should root maintenance backup use `--include-root --maintenance`, a separate
  command, or both?
- Should snapshot read authority be configured through
  `snapshot_visibility.allowed_viewers`, root-mediated transfer, or a different
  host management API path?
- Which role verification checks should block restore completion versus report
  warnings?
