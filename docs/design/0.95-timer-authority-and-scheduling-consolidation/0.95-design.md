# Canic 0.95 Design: Timer-System Audit, Scheduling Consolidation, And Receipt Reclamation

Date: 2026-07-20

## Status

- Status: proposed; not accepted and not active.
- Intended release line: 0.95, after the bounded 0.94 backup/restore line closes.
- Planning reference: published `v0.94.1` behavior. The immutable 0.95 release
  anchor remains pending until 0.94 closeout.
- Scope: every autonomous canister timer, lifecycle deferral, retry scheduler,
  and shared timer primitive in this repository. Bounded command-scoped host
  waits are inventoried separately because they do not share canister runtime
  authority.
- Downstream scope: Toko remains read-only from this repository. A Toko
  adoption plan may consume the released contract but cannot define Canic
  storage, scheduling, or reclamation invariants.
- Implementation is not authorized by this proposal. The timer inventory,
  finding dispositions, exact replay horizon, and capacity envelope must be
  accepted before code work begins.

## Decision

0.95 is a system-wide timer audit and consolidation line. It must first prove
what time-based work exists, why each process needs time rather than an event,
how it survives lifecycle changes, and whether it can wake or overlap while
idle. Implementation follows accepted findings; the audit must not assume that
every current interval should survive.

The receipt-backed intent limit is one concrete correctness and capacity
problem inside that wider line. Canic has two independent intent limits, and
only the second needs timer-backed reclamation:

| Limit | Current meaning | 0.95 decision |
| --- | --- | --- |
| `reservation_limit` | Per-resource admitted quantity: committed plus pending plus the new request. The consumer supplies it on begin. | Preserve unchanged. Toko uses the frozen collection supply budget. There is no time-based cleanup of consumed or uncertain quantity. |
| `RECEIPT_BACKED_INTENT_RECORD_LIMIT` | Per-canister count of distinct receipt-backed operation records. It is compiled as `100_000`. | Preserve the fail-closed ceiling while adding an explicit replay horizon, bounded terminal-only reclamation, and capacity observability through the shared due-work scheduler. |

The collection budget is permanent accounting, not temporary cache pressure.
Committed quantity remains consumed. Rolled-back quantity is released during
settlement. Pending quantity remains reserved until the downstream owner proves
either the effect or durable no-effect. No timer changes those rules.

The record ceiling is a technical storage bound. Current application-owned
terminal records are retained indefinitely because deleting one would allow an
old `OperationId` to look new. 0.95 may reclaim a terminal record only after
the complete path rejects that operation as outside its replay window before a
new reservation or external effect can occur.

0.95 does not introduce another fixed cleanup interval for this work. Receipt
reclamation is ordered durable due work: arm one guarded wake-up for the next
eligible deadline, drain a bounded batch after it fires, continue only while
eligible work remains, and disarm when the index is empty.

## Why This Is A Separate Release Line

This work changes Canic's common timer authority, lifecycle scheduling,
runtime status semantics, several subsystem scheduling strategies, the
generic receipt-backed intent contract, stable retention state, capacity
diagnostics, and the downstream adapter's security obligations. It is not a
Toko-only setting and is unrelated to 0.94's backup/restore
operational-readiness claim.

Raising `100_000` alone is not a solution. It delays an unbounded-history
failure without defining when an operation can safely be forgotten. Adding a
TTL alone is unsafe because age does not prove whether an external effect
occurred or whether an old request can be replayed.

The required sequence is:

```text
inventory every timer owner and timed wait
  -> record its durable work predicate and lifecycle behavior
  -> classify it as retain, remove, event-drive, deadline-drive, or consolidate
  -> accept finding-backed dispositions
  -> hard-cut misleading or unused timer surfaces
  -> consolidate scheduling, no-overlap, continuation, and status mechanics
  -> freeze the exact current intent contract
  -> freeze one downstream replay-horizon contract
  -> prove operation identity binds that horizon
  -> reject an expired absent operation before reservation
  -> retain every pending operation regardless of age
  -> index terminal records by safe eligibility time
  -> remove eligible terminal records with the shared bounded due-work runner
  -> prove old requests cannot become new effects
  -> prove resource totals are unchanged by record reclamation
  -> measure idle wake-ups, overlap, and instruction cost after consolidation
  -> publish the Canic contract
  -> leave downstream adoption to Toko
```

## Current Implementation Reality

### Shared timer infrastructure

The current code has one direct IC timer owner, which is the right platform
boundary: production calls to `ic_cdk_timers` are contained in
`ops::runtime::timer`. Above it, `TimerWorkflow`, `TimerApi`, and public macros
repeat or expose scheduling choices.

Static inventory identifies these common-mechanism concerns. They are audit
findings to reproduce and disposition before implementation:

- guarded registration prevents duplicate handles but does not prevent two
  async interval callbacks from overlapping when one invocation is still
  awaiting;
- guarded one-shots depend on each internal callback manually clearing its
  registration slot, while the public guarded macro does not make that
  lifecycle obvious or automatic;
- guarded scheduling and slot-handoff logic is repeated across ops and API
  layers;
- lifecycle deferrals use `TimerApi`, `TimerWorkflow`, and `TimerOps` from
  different adapters despite having the same zero-delay purpose;
- the public facade returns opaque timer handles without a matching public
  cancellation path;
- runtime timer metrics are keyed by scheduling history, while the status view
  currently projects those historical rows as enabled and registered even
  after a one-shot completes or a handle is cleared; and
- timer tasks return `()`, so the common layer cannot truthfully distinguish
  success, no work, bounded continuation, retry, or failure.

### Built-in timer processes

The initial static inventory finds the following production process families.
This table is the starting hypothesis for the executable audit, not permission
to change code without a reproduced finding.

| Process | Current trigger | Durable work predicate | Initial disposition |
| --- | --- | --- | --- |
| Lifecycle and user-hook deferral | Zero-delay one-shot during init or post-upgrade | Lifecycle phase has restored Canic invariants and queued the next phase | Retain the required asynchronous boundary; consolidate all registration and ordering through one lifecycle deferral primitive. |
| Log retention | Start after 30 seconds, then every 10 minutes | Entries exceed count, byte, or optional age policy | Conditional. Count enforcement is currently not fully append-driven, so removal is unsafe. Evaluate append-time count enforcement and next-expiry scheduling; disarm when no policy can have due work. |
| Cycle history sampling | Immediate sample, then every hour | A new balance sample and history-retention decision are due | Retain periodic semantics unless measured evidence supports an adaptive deadline. Separate its result and status from automatic top-up. |
| Automatic cycle top-up | First check after 30 seconds, then coupled to the hourly cycle workflow | Configured canister balance may have crossed its threshold | Retain a bounded recurring check where configured; audit whether cadence, sampling coupling, and retry behavior should be independent. |
| Root pool scheduler | Every 30 minutes plus event-driven reset scheduling | Stable pool state has pending reset work or a blocked reset needs retry | Replace always-on idle polling with one bootstrap check and delayed retries only while durable pending work exists. |
| Pool reset drain | Guarded zero-delay one-shot and bounded continuation | A stable reset batch is pending | Retain event-driven bounded drain; move slot release, non-overlap, continuation, and result reporting into shared mechanics. |
| Local-intent expiry cleanup | Start after 30 seconds, then every hour after any local begin | An intent with a finite TTL has reached its expiry | Replace the fixed scan with an expiry-ordered index and one next-due wake-up. An intent without a TTL must not arm or keep a cleanup timer alive. |
| Root delegated-proof renewal | Start after 30 seconds, then every minute while configured | One or more proofs approach their renewal deadline | Retain time-based renewal but evaluate exact-next-deadline scheduling. Prove that configuration changes can advance an already-armed later wake-up and that async sweeps cannot overlap. |
| Placement receipt acknowledgement | Guarded immediate attempt, then one-minute retries in batches | A terminal placement receipt still awaits authoritative root acknowledgement | Retain event/retry scheduling only while work exists. Replace full receipt-map traversal with a dedicated acknowledgement queue or index if the audit confirms unrelated-row scanning. |
| Receipt-backed terminal reclamation | Not implemented | A terminal application-owned record has passed its immutable replay deadline | Add as ordered due work using the shared scheduler; never add a fixed polling interval. |

The public timer macros are capability surfaces rather than built-in processes.
The initial call-site scan finds in-repository use of unguarded one-shot and
interval macros in the runtime probe, but no production consumer of the
guarded public macros. 0.95 must either prove and document a maintained use for
each public surface or hard-cut it before 1.0. It must not preserve a confusing
surface only because it already exists.

### Receipt-backed intent state

The current code has these properties:

- `BeginReceiptBackedIntentInput::reservation_limit` is a required consumer
  input.
- Admission computes `reserved_qty + committed_qty + requested_quantity` for
  the exact resource key.
- A direct `OperationId` lookup happens before either capacity check. Exact
  retries therefore remain readable at capacity and do not append rows.
- A new ID is rejected with `StoreCapacityReached` once the receipt-backed
  stable map contains `100_000` rows.
- Pending, committed, and rolled-back rows all count toward that total.
- Receipt-backed rows have no TTL and never enter the local-intent expiry
  index.
- Public consumers can begin, load, and settle. They cannot list or remove
  terminal rows.
- Canic's placement workflow has a private exact terminal-removal path. It
  uses that path only after the authoritative root receipt is acknowledged.
  That acknowledgement rule does not automatically apply to Toko.
- The public status surface does not provide the maintained total, next
  terminal eligibility deadline, or remaining receipt-record headroom.

The permanent row currently provides two correctness properties:

1. A pending row fences a duplicate external effect while the outcome may be
   unknown.
2. A terminal row makes a later retry return the already established outcome
   instead of creating a new reservation and effect.

Reconciliation changes `Pending` to `Committed` or `RolledBack`; it does not
make the operation safe to forget. Record reclamation requires the additional
replay-horizon proof defined here.

### Host timed-wait boundary

The host and backup crates contain bounded, command-scoped waits, not
autonomous canister timers. The initial production inventory includes root
readiness polling during install and backup command-lifetime quiescence. These
must appear in the audit so timed behavior is not hidden, but they retain their
own command deadline, cancellation, and process-lifetime ownership.

0.95 must not route host waits through the canister timer scheduler. It may
recommend a host polling helper only if the audit proves repeated production
mechanics. Backup process-death and quiescence changes remain owned by 0.94 and
are not reopened by this line.

## Timer Audit Contract

The audit is exhaustive over production Rust under `crates/`, `canisters/`,
and `fleets/`. It includes direct timer APIs, wrapper APIs, public timer macros,
lifecycle registration, self-rescheduling callbacks, retry delays, and
production `sleep` or timeout loops. Test-only clocks and waits are recorded
only when they define or validate production behavior.

Every timer/process row records:

1. stable timer key and fixed diagnostic label;
2. owning canister role, configuration gate, and lifecycle registration path;
3. one-shot, fixed-interval, after-completion, or next-deadline mode;
4. initial delay, cadence, retry delay, and batch bound;
5. authoritative work predicate and whether it is heap or stable;
6. awaited/external effects and the maximum overlap risk;
7. cursor, continuation, retry, and stop conditions;
8. init and post-upgrade reconstruction behavior;
9. idle wake-up behavior and instruction/stable-read cost;
10. failure outcome, backoff, and operator visibility;
11. public API and test consumers; and
12. accepted disposition: retain, remove, event-drive, deadline-drive,
    consolidate mechanics, or block pending evidence.

The audit report must contain command-backed source evidence and a complete
owner table. It also records negative searches so a new direct CDK timer call,
sleep loop, or bypassing wrapper cannot be omitted accidentally. The report is
stored under the dated audit-report tree and linked from this design before
implementation begins.

The audit evaluates a timer against these decision rules:

- use an event when Canic itself observes every state transition that creates
  work;
- use an exact deadline when stable state identifies the earliest future due
  item;
- use a retry wake-up only while external or fallible work remains pending;
- use a recurring check only when relevant state can change without a local
  event or maintained deadline;
- use a zero-delay one-shot only for a real asynchronous lifecycle or bounded
  continuation boundary;
- disarm when the durable work predicate becomes false; and
- reject a timer whose only justification is that an older timer already
  exists.

Timer removal requires proof that another authority covers every transition,
upgrade, and retry. Timer retention requires a named work predicate, bounded
callback, non-overlap guarantee, lifecycle reconstruction rule, and truthful
status.

## Provisional System Evaluation

The initial inventory supports these system-level hypotheses for the formal
audit to prove or reject:

1. **Required asynchronous boundaries:** lifecycle/user-hook deferrals and
   bounded continuation callbacks remain timers because synchronous lifecycle
   adapters cannot `await` and large drains cannot monopolize one message.
2. **Required recurring observation:** cycle balance checks remain recurring
   where the balance can change without a Canic-local event. Credential renewal
   remains time-based because expiry advances without state mutation.
3. **Unnecessary idle polling:** pool scheduling, local-intent cleanup,
   placement acknowledgement, and receipt reclamation have durable work
   predicates and should have no armed recurring timer while empty.
4. **Conditional polling:** log retention cannot lose its timer until count
   policy is enforced elsewhere and age policy exposes a next deadline. The
   audit chooses from evidence, not aesthetics.
5. **Indexes before timers:** expiry, acknowledgement, and reclamation work
   should use dedicated ordered or queued authority instead of repeatedly
   scanning unrelated primary rows.
6. **Common mechanics, independent policy:** timer registration, generation,
   slot release, after-completion scheduling, bounded continuation, backoff,
   and runtime status should be DRY. Subsystem cadence and domain decisions
   remain with their owners.
7. **No global heartbeat:** one shared minute or hour tick would wake idle
   canisters, couple unrelated failures, obscure per-process status, and make
   the slowest policy everybody's cadence. 0.95 consolidates scheduling
   mechanics, not all work into one callback.

## Central Timer Invariant

Every armed timer must correspond to one necessary, named unit of future work.
It must execute at most one invocation for its timer key at a time, perform a
bounded amount of work, reconstruct from authoritative state after upgrade,
and become unregistered when no work or recurrence remains.

Consequences:

- registering a handle and preventing overlapping executions are separate
  guarantees, and both are required;
- recurring async work schedules its next invocation after the current run
  completes unless a measured fixed-rate requirement proves otherwise;
- a guarded one-shot releases or replaces its slot in common infrastructure,
  not through callback-specific bookkeeping;
- subsystem work predicates and cursors remain authoritative after a timer
  handle disappears during upgrade;
- status distinguishes scheduled, running, idle, retrying, failed, disabled,
  and not registered rather than projecting scheduling history as liveness;
- timer callbacks return a typed outcome that can choose stop, immediate
  bounded continuation, retry after delay, or schedule at a deadline; and
- timer consolidation must not move domain policy into the runtime facade.

## Receipt Reclamation Invariant

Removing a receipt-backed terminal record must never allow the same logical
operation to reserve capacity, execute an external effect, or settle resource
accounting a second time.

Consequences:

- pending records are never removed because of age;
- an absent operation whose replay window is closed is rejected before store
  capacity, resource capacity, reservation, or external-effect admission;
- the replay deadline is immutable and bound to the operation and payload;
- terminal reclamation changes only receipt-history storage and its maintained
  indexes, never `reserved_qty`, `committed_qty`, or `pending_count`;
- a cleanup failure retains the authoritative row and fails closed;
- timers perform no downstream call and infer no domain outcome; and
- a consumer that cannot prove the complete stale-request rejection path is
  not eligible for automatic reclamation.

## Goals

0.95 must:

1. publish an exhaustive, command-backed inventory and disposition for every
   production timer process and bounded host wait;
2. prove which time-based processes are necessary and remove or disarm polling
   where durable events or deadlines cover all work;
3. reduce direct timer ownership to one platform boundary and consolidate
   guarded registration, non-overlap, slot release, continuation, retry, and
   live status mechanics;
4. preserve subsystem-specific policy, cadence, stable work predicates, and
   failure decisions outside the common timer layer;
5. make timer lifecycle reconstruction and actual live state observable;
6. measure idle wake-ups and callback cost before and after timer changes;
7. preserve the current per-resource collection-budget calculation;
8. add one explicit, immutable replay deadline to application-owned
   receipt-backed operations;
9. reject a closed-window operation with no retained row before it can reserve
   or execute again;
10. keep exact retries idempotent while their row is retained;
11. keep old pending rows available for targeted reconciliation indefinitely;
12. reclaim only terminal rows through bounded, upgrade-safe due-work
    scheduling;
13. retain exact compare-and-remove checks for operation identity, payload
   binding, revision, terminal state, and eligibility;
14. maintain constant-time record and state counts without a map-wide scan;
15. expose low-cardinality capacity and cleanup diagnostics before exhaustion;
16. prove cleanup cannot change collection accounting or repeat a mint;
17. preserve Canic-owned placement acknowledgement cleanup; and
18. publish a precise downstream contract that Toko can adopt without moving
    mint policy, receipts, or recovery into Canic.

## Non-Goals

0.95 does not:

- create one global heartbeat for unrelated subsystem work;
- remove a timer without proving the replacement covers lifecycle, retry, and
  externally changing state;
- force exact-deadline scheduling where a measured fixed cadence is the real
  product or operational contract;
- share domain state machines, storage records, or policy merely to reduce
  line count;
- merge command-scoped host waits into canister timer infrastructure;
- reopen 0.94 backup process-lifetime decisions;
- change the meaning of `reservation_limit`;
- introduce pending-intent TTL rollback;
- infer no-effect from timeout, reject text, missing response, or unchanged
  observed state;
- add a generic transaction, saga, effect executor, receipt validator, or
  downstream reconciliation daemon;
- make Canic interpret Toko collection, mint, ledger, cancellation, or receipt
  types;
- add per-user, per-collection, or economic rate limits to Canic;
- make terminal deletion a caller-selected escape from store capacity;
- scan all receipt-backed rows on a timer, upgrade, or ordinary request;
- raise the compiled record limit without a measured capacity justification;
- preserve an old public input through defaults, overloads, aliases, or
  compatibility wrappers;
- invent a `v2` API, DTO, record, or document name before 1.0; or
- edit or implement the downstream Toko repository.

## Terminology

### Resource budget

The authoritative quantity for one `IntentResourceKey`:

```text
accounted_quantity = reserved_qty + committed_qty
```

A new reservation is admitted only when:

```text
accounted_quantity + requested_quantity <= reservation_limit
```

### Replay deadline

An absolute IC timestamp after which an absent operation may no longer be
created. It is chosen by the trusted downstream adapter under a frozen policy,
not by an unauthenticated request field.

### Replay horizon

The maximum permitted distance between admission time and replay deadline.
The exact 0.95 value remains a release-line decision. It must cover expected
client retry and domain recovery behavior while keeping the steady-state row
count safely below the compiled store ceiling.

### Terminal eligibility

A committed or rolled-back record is eligible for reclamation only when its
immutable replay deadline has passed and every exact local consistency check
succeeds.

### Reclamation

Deletion of terminal operation evidence and its eligibility-index entry after
the replay contract makes recreation impossible. Reclamation is not
settlement, rollback, cancellation, or domain reconciliation.

## Shared Scheduling Architecture

0.95 consolidates mechanics below subsystem workflows without creating a
cross-domain scheduler policy. The common runtime owns IC handle registration,
timer-key generations, automatic guarded-slot release, execution non-overlap,
bounded continuation, scheduling outcomes, and live status. Each subsystem
still owns the question "what work is due and what does success mean?"

The maintained internal primitives are:

1. **Lifecycle deferral:** schedule one zero-delay callback after Canic
   invariants are restored. Root, nonroot, and generated user-hook adapters use
   the same primitive and explicit ordering.
2. **Guarded one-shot:** maintain at most one current handle for a timer key,
   allow an earlier deadline to replace a later one, release the slot before
   invoking work, and ignore stale generations.
3. **After-completion recurrence:** begin the next delay only after the current
   async invocation completes. This is the default for recurring work and
   prevents callback overlap.
4. **Bounded due-work drain:** read authoritative stable work, process at most
   the subsystem's frozen batch bound, and return a typed scheduling directive.

The typed directive supports only current behavior:

```text
Stop
ContinueImmediately
RetryAfter { delay }
ScheduleAt { deadline }
RecurAfter { delay }
```

The execution result separately records success, no work, retryable failure,
or invariant failure. Domain errors remain typed subsystem errors; the runtime
does not parse error text to decide retry behavior. A callback cannot silently
return `()` while status claims success.

The shared scheduler must support these state transitions:

```text
not registered
  -> scheduled(deadline, generation)
  -> running(generation)
  -> stopped
     | scheduled(next deadline, next generation)
     | retrying(retry deadline, next generation)
     | failed(no automatic retry)
```

Cancellation or replacement advances the generation so a stale callback
cannot clear or overwrite a newer registration. A timer handle remains heap
state; authoritative due work, cursors, and policies remain in stable subsystem
state where upgrade recovery requires them.

### Truthful timer status

The runtime maintains one live registry entry per fixed timer key, not one row
per historical `(mode, delay, label)` registration. The read-only timer view
reports at least:

- timer key, owning subsystem, mode, and configuration state;
- registered, running, or idle state;
- current generation and next due timestamp when registered;
- last started, completed, and successful timestamps;
- last typed outcome and bounded work count;
- consecutive failures and current retry timestamp; and
- total schedules, executions, successes, failures, and stale callbacks.

Completing or clearing a one-shot changes live registration to false. Disabling
a configured process reports disabled rather than healthy. A process that
should be registered but is absent reports not registered. A scheduled process
that has not ticked is not called healthy merely because its metric row exists.

Metric dimensions are fixed by the timer key and outcome class. Timer labels,
operation IDs, resource keys, callers, deadlines, and raw error text must not
create unbounded metric cardinality.

### Lifecycle reconstruction

Init and post-upgrade adapters restore synchronous invariants, then defer one
common runtime-start workflow. Each subsystem reconstructs scheduling from its
authoritative work predicate:

- configured recurring observation schedules its next after-completion run;
- an ordered due-work owner schedules the earliest stable deadline;
- a queue owner schedules immediately only when stable work exists;
- a retry owner restores the persisted or policy-derived retry deadline; and
- an empty or disabled owner remains unregistered.

Registration must be idempotent within one lifecycle generation. The audit
must prove that root, nonroot, and user hook ordering is unchanged or explicitly
amended, and that repeated configuration/start requests can advance required
work instead of being discarded by an occupied slot.

## Proposed Public Contract

The application-owned begin input becomes a hard-cut required contract with an
absolute replay deadline:

```text
BeginReceiptBackedIntentInput {
    operation_id,
    payload_binding,
    resource_key,
    quantity,
    reservation_limit,
    replay_deadline_ns,
}
```

The maintained begin result gains one typed decision:

```text
ReplayWindowClosed {
    replay_deadline_ns,
}
```

There is no optional deadline and no default. Application adapters must derive
it from trusted immutable request/configuration state. The operation-ID
derivation and payload binding must both cover the exact deadline, so changing
the deadline creates a different logical request rather than extending an old
one.

Canic validates an accepted minimum execution window and maximum replay
horizon. The exact durations must be frozen before implementation. A deadline
in the past, too close for safe effect execution, or beyond the maximum is
invalid input and performs no mutation.

Canic-owned internal placement continues to use its acknowledgement-owned
retention contract. It must not invent a replay deadline merely to share the
application adapter's cleanup path.

## Begin-Or-Load Ordering

Application begin-or-load follows this exact order without `await`:

1. validate the consumer resource namespace, payload-binding schema, quantity,
   and deadline shape;
2. perform the direct `OperationId` lookup;
3. when the row exists, validate the stored replay contract and return its
   exact pending or terminal state without another reservation;
4. when the row is absent and its deadline is closed, return
   `ReplayWindowClosed`;
5. check total receipt-record capacity;
6. check the existing per-resource quantity budget;
7. precompute the primary row, replay metadata, aggregate, and maintained
   counters; and
8. write the complete pending authority without an intervening fallible step.

Existing pending rows remain loadable and settleable after their replay
deadline. The deadline closes creation; it does not decide an uncertain
effect. Existing terminal rows may be returned until bounded reclamation
removes them. After removal, the same expired request receives
`ReplayWindowClosed` and cannot become new work.

## Stable Retention Authority

0.95 adds one current-schema replay-retention allocation for
application-owned receipt-backed operations. It records enough information to
validate the immutable deadline and maintain terminal eligibility without
changing the meaning of the primary receipt row.

The terminal eligibility index is ordered by:

```text
(replay_deadline_ns, operation_id)
```

An index entry carries the exact expected payload binding and terminal
revision required for compare-and-remove. Settlement inserts the eligibility
entry only after consumer-validated terminal evidence is accepted. The owning
ops transition precomputes all changes and has no fallible operation after its
first stable mutation.

Maintained metadata records at least:

```text
total_records
pending_records
terminal_records
next_terminal_deadline_ns
```

The counts are storage invariants, not reconstructed by scanning on status,
upgrade, or cleanup. The next deadline comes from the first ordered index key,
not a map-wide scan.

## Bounded Receipt Reclamation

The receipt workflow owns local terminal-history reclamation only. It uses the
shared guarded one-shot and bounded due-work primitives; it does not own a
fixed interval. It never observes or calls a downstream canister and never
settles an intent.

The workflow is:

```text
read the earliest eligibility key
  -> stop when the index is empty or the earliest deadline is in the future
  -> load the exact primary row
  -> require terminal state
  -> require matching payload binding and revision
  -> require the replay deadline to be closed
  -> precompute record/index/counter removal
  -> remove the exact terminal row and index entry
  -> leave resource totals unchanged
  -> repeat up to the frozen batch bound
  -> schedule the next due deadline or bounded continuation
```

Scheduling requirements:

- process a fixed maximum number of entries per callback;
- use the ordered eligibility index rather than scanning the primary map;
- arm exactly the earliest stable deadline and replace a later wake-up when
  settlement inserts an earlier one;
- rebuild scheduling from stable eligibility state during init and upgrade;
- stop when no terminal work remains;
- retry storage failures with a bounded delay and fixed-label diagnostics;
- retain mismatched, corrupt, pending, or otherwise contradictory rows;
- never delete an entry merely to make a new admission succeed; and
- allow exact existing operations to load and settle while the store is full.

The exact batch size, retry delay, and warning thresholds must be frozen from
focused instruction and stable-memory evidence before implementation closes.
There is no cleanup cadence to freeze: future work schedules at its exact next
deadline, while an immediately eligible backlog uses bounded continuations.

## Pending Recovery

Pending rows remain outside the reclamation index. Their age may be diagnosed,
but it does not authorize rollback or deletion.

Toko continues to own targeted recovery for one `OperationId`:

1. observe the exact ledger receipt or durable no-effect evidence;
2. validate caller, request, collection, payload, source canister, and result;
3. settle the Canic row with exact terminal evidence; or
4. leave it pending and fail closed.

No Canic timer scans Toko operations, calls the ledger, retries a mint, or
constructs terminal evidence. An operator may reconcile named pending work,
but cannot mark it terminal or delete it based only on age.

## Capacity And Diagnostics

0.95 keeps `RECEIPT_BACKED_INTENT_RECORD_LIMIT = 100_000` unless measured
evidence supports a separate explicit amendment. The line must make that
capacity observable before it is exhausted.

One read-only projection reports:

- current total and configured limit;
- pending and terminal counts;
- remaining record headroom;
- next cleanup deadline;
- last cleanup outcome and removed count; and
- whether cleanup retry is scheduled.

Metrics use fixed labels only. Operation IDs, resource keys, collection IDs,
and callers must not become metric labels.

Capacity planning uses unique operations, not minted quantity:

```text
steady_state_records
  ~= unique_operations_within_replay_horizon
   + unresolved_pending_operations
   + bounded_cleanup_backlog
   + Canic-owned receipt records awaiting their own acknowledgement
```

Acceptance requires a documented maximum expected operation rate and enough
headroom for recovery backlog. A collection supply below `100_000` does not by
itself prove safety because rolled-back and rejected-effect operations also
consume IDs while retained.

## Toko Adoption Contract

Toko remains responsible for:

- deriving a globally unique, caller-scoped mint `OperationId`;
- binding the exact replay deadline into the operation ID, payload digest, and
  authenticated request;
- sourcing `reservation_limit` from the frozen collection supply authority;
- rejecting caller attempts to choose or extend the supply limit or replay
  deadline;
- calling Canic begin before `execute_mint`;
- executing the effect only after `Created`;
- validating ledger receipts and durable no-effect evidence;
- targeted recovery of old pending operations;
- returning a typed expired/replay-closed result after the horizon; and
- proving that no endpoint, callback, admin path, or upgrade continuation can
  bypass the same replay-window check.

Toko does not run a timer that directly deletes Canic rows. Canic owns its
stable reclamation index and due-work scheduling. Toko may expose domain diagnostics or
targeted reconciliation, but its code remains downstream-owned.

Before Canic implementation is accepted, the Toko owner must confirm:

1. the maximum legitimate client retry window;
2. the maximum ambiguous-effect recovery window;
3. the ledger's operation-ID idempotency/receipt-retention behavior;
4. the maximum expected unique mint-operation rate per canister; and
5. every entrypoint that can initiate or resume a mint.

## Security Requirements

- Endpoint authentication remains downstream-owned and occurs before adapter
  admission.
- `OperationId`, caller, collection/resource key, quantity, payload, and replay
  deadline are cryptographically bound by the downstream request derivation.
- The replay deadline is never accepted as an unauthenticated caller override.
- An expired absent operation returns before any quantity or store mutation.
- A changed deadline for a retained operation is a binding conflict.
- A stale request cannot select a fresh deadline while retaining the old
  logical operation identity.
- Pending state remains the conservative unknown-effect boundary.
- Terminal evidence still requires consumer validation; time does not create
  evidence.
- Cleanup uses exact revision and payload-binding comparison and never accepts
  a broad resource-prefix deletion request.
- Consumer resource keys remain outside the reserved `canic:` namespace.

## Layering And Ownership

The strict dependency direction remains:

```text
endpoints -> workflow -> policy -> ops -> model
```

- `model::intent` owns the current replay-deadline value, operation results,
  and authoritative state invariants.
- `storage::stable::intent` contains passive persisted shapes ending in
  `*Record`; canonical snapshot shapes end in `*Data`.
- `view::intent` owns the internal read-only capacity/status projection.
- `view::runtime` owns read-only live timer projections. Views do not register,
  clear, or infer timer health.
- `ops::storage::intent` owns exact admission reads, stable retention/index
  mutation, record/data/view conversion, maintained counts, and
  compare-and-remove.
- `ops::runtime::timer` remains the only direct `ic_cdk_timers` owner. It owns
  mechanical handle operations and clock reads, not retries, cadence, domain
  outcomes, or lifecycle ordering.
- Pure deadline and capacity decisions belong in `policy` only if they are
  shared decisions rather than mechanical checked comparisons.
- `workflow::runtime::timer` owns common guarded scheduling, generations,
  non-overlap, typed execution outcomes, bounded continuation, and live
  runtime status over timer ops.
- Each subsystem workflow owns its work predicate, stable cursor/index access,
  batch bound, recurrence or retry policy, and domain outcome mapping.
- `workflow::runtime::intent` owns receipt-reclamation orchestration over
  intent ops and the common due-work scheduler.
- `api::intent` remains a thin local facade.
- Lifecycle adapters and timer macros authenticate or marshal where relevant
  and delegate immediately. They do not reproduce slot or handoff logic.
- Downstream endpoints authenticate, derive Toko identities, validate receipts,
  and delegate immediately.
- Workflow must not construct or mutate stable records directly.

## Public And Stable Hard Cut

The application begin input changes directly. There is no legacy overload,
optional replay deadline, default horizon, deprecated alias, fallback reader,
or compatibility wrapper.

The timer facade changes directly to the accepted current surface. An unused
or misleading guarded macro, duplicate lifecycle entrypoint, opaque handle
without a maintained operation, or fixed-interval helper is removed if the
audit cannot prove a current consumer and complete semantics. The replacement
does not retain aliases or wrappers for old call forms. Current public timer
tests cover only the maintained surface; they do not preserve removed behavior.

Before implementation, an inventory must prove there is no deployed
application-owned receipt-backed consumer requiring the old input contract.
Toko is currently documented as planned rather than adopted. If that inventory
finds an existing consumer or application-owned stable row, 0.95 blocks for an
explicit safety decision; it does not guess a historical deadline or silently
delete the row.

Canic-owned placement records retain their existing acknowledgement-owned
cleanup and must remain upgrade-safe. New stable allocations and snapshot data
use current names and schema version `1` where a version is required. No `V2`
type, `v2` memory key, or dual decoder is introduced merely because behavior
changes before 1.0.

Canonical state export/import must include the retention allocation, ordered
eligibility entries, maintained counters, and any new authoritative expiry or
acknowledgement queues accepted by the audit. Heap timer handles, live
generations, and runtime execution metrics are not stable authority and are
reconstructed. Restore rejects missing, duplicate, contradictory, or invalid
index entries rather than rebuilding authority from assumptions.

## Implementation Slices

Implementation begins only after design acceptance and 0.94 closeout.

### Slice A: Exhaustive Audit And Finding Gate

- publish the complete timer/process and host-wait inventory with source
  evidence, negative searches, call sites, and lifecycle paths;
- reproduce overlap, stale-registration, idle-wake, full-scan, and scheduling
  hypotheses with focused tests or measurements;
- record instruction and wake-up baselines for every built-in process;
- assign and accept one disposition for every inventory row; and
- freeze the maintained public timer surface before code changes.

No timer implementation slice begins with an undispositioned inventory row.
A conditional finding may remain blocked for measurement, but it cannot be
silently treated as permission to keep or remove current behavior.

### Slice B: Common Timer Authority

- introduce fixed timer keys, typed execution outcomes, and typed scheduling
  directives;
- retain one direct IC timer owner and one workflow scheduling authority;
- implement generation-safe guarded one-shots with automatic slot release;
- implement after-completion recurrence and execution non-overlap;
- replace historical-registration health with truthful live timer status;
- consolidate root, nonroot, and user-hook lifecycle deferral; and
- hard-cut duplicate, incomplete, or unused public timer surfaces.

### Slice C: Event- And Deadline-Driven Owners

- replace local-intent full scans with an ordered finite-expiry index and no
  timer for TTL-free intents;
- replace the root pool's always-on idle interval with lifecycle bootstrap,
  event scheduling, and delayed retry only while pending reset work exists;
- move placement acknowledgement to a dedicated durable queue/index if the
  audit confirms unrelated receipt scanning;
- move each bounded drain to the common continuation/non-overlap mechanics;
  and
- change log retention only according to accepted append-time and next-expiry
  evidence, preserving every configured retention invariant.

### Slice D: Necessary Recurring Observation

- separate cycle sampling and top-up outcomes, scheduling, and status while
  preserving their external behavior;
- retain a recurring top-up check only where configuration enables it;
- evaluate delegated-proof renewal against exact-next-deadline scheduling and
  implement the accepted mode;
- prove configuration changes can advance the next required run; and
- freeze cadences and batch bounds from measured cost and deadline evidence.

### Slice E: Receipt Contract And Stable Authority

- freeze the exact replay horizon and minimum execution window;
- inventory every in-repository and known downstream receipt consumer;
- prove no application-owned old-contract row requires compatibility;
- freeze operation-ID and payload-binding vectors including the deadline;
- add the required deadline input and typed closed-window result;
- add current-schema retention metadata, eligibility index, and counters;
- implement exact begin ordering, settlement eligibility, and aggregate
  preservation; and
- extend canonical state validation and exact state round trips.

### Slice F: Receipt Due Work And Capacity

- schedule only the earliest receipt eligibility deadline;
- drain eligible terminal rows through the common bounded due-work primitive;
- rebuild receipt scheduling from stable state on init and upgrade;
- expose read-only capacity and cleanup status with fixed labels; and
- prove saturation does not block existing load, settlement, or cleanup.

### Slice G: System Conformance And Handoff

- run per-owner unit and PocketIC lifecycle/time-advance evidence;
- compare idle wake-ups, overlap, instruction cost, and timer counts with the
  Slice A baseline;
- extend the local intent-authority fixture and prove delayed receipt replay
  cannot execute twice;
- update the Canic downstream adapter handoff; and
- record remaining Toko-owned implementation and proof without editing Toko.

## Risks And Mitigations

### Audit misses an indirect timer owner

A wrapper, macro, or self-rescheduling callback could remain outside the
inventory and bypass the new guarantees. Mitigation: combine direct API and
wrapper searches with reverse call-site ownership, lifecycle-path tracing, and
negative structural checks that fail when an unregistered owner appears.

### Removing polling loses work

An event may be missed across upgrade or an external condition may change
without a local transition. Mitigation: remove polling only when a durable work
predicate plus lifecycle reconstruction covers every producer; retain bounded
retry or recurring observation for genuinely external state.

### One common heartbeat becomes a bottleneck

Unrelated work could share failure, cadence, and instruction limits.
Mitigation: share primitives and status mechanics only. Keep independent timer
keys, deadlines, batches, and domain workflows; do not introduce a global tick.

### After-completion recurrence drifts

Scheduling after execution prevents overlap but moves a nominal wall-clock
cadence by callback duration. Mitigation: use it by default for operational
polling; retain an explicit fixed-rate mode only where the audit proves a
wall-clock requirement and supplies a non-overlap policy.

### Earlier work remains behind a later wake-up

A new expiry or credential deadline may be earlier than the registered handle.
Mitigation: generation-safe guarded one-shots compare deadlines and replace a
later handle atomically in the common scheduling authority.

### Runtime status becomes false after upgrade

Heap registrations disappear while historical metrics or stable work remain.
Mitigation: clear live registry state during restore, reconstruct from subsystem
authority, and report not registered until registration actually succeeds.

### DRY abstraction absorbs domain policy

A generic runner could start deciding which intents expire, receipts retry, or
proofs renew. Mitigation: common code understands only timer keys, execution
outcomes, and scheduling directives. Domain work predicates, errors, indexes,
and retry decisions stay in subsystem workflow/policy/ops layers.

### Replay horizon is too short

Legitimate clients may lose the ability to reconstruct an old result.
Mitigation: freeze the horizon from measured retry/recovery requirements and
retain a minimum execution window before admission.

### Replay horizon is too long

The steady-state record set may still approach `100_000`. Mitigation: require
the operation-rate capacity calculation, warning thresholds, bounded cleanup
evidence, and per-canister headroom before acceptance.

### Downstream changes a deadline

The same logical request could be presented as fresh after reclamation.
Mitigation: bind the deadline into authenticated request data, operation-ID
derivation, and payload binding; reject every bypass path in downstream
conformance proof.

### Timer deletes uncertain work

An external effect could be repeated. Mitigation: pending rows never enter the
eligibility index, and cleanup rechecks terminal state, binding, revision, and
deadline immediately before exact deletion.

### Cleanup changes collection supply

Removing history could incorrectly release or consume capacity. Mitigation:
settlement remains the only aggregate transition; reclamation cannot write the
resource totals allocation.

### Stable index drift

Restore or interrupted development could leave contradictory primary and
eligibility state. Mitigation: mutation is non-awaiting and precomputed;
canonical import validates exact correspondence; runtime mismatch retains the
primary row and reports a typed invariant failure.

## Closed Decisions

- 0.95 begins with an exhaustive timer audit; implementation is finding-backed.
- Timer mechanics are consolidated, but there is no global heartbeat or shared
  domain state machine.
- Direct IC timer access remains owned by one ops module.
- Async recurrence is non-overlapping; after-completion scheduling is the
  default unless a fixed-rate requirement is proven.
- Lifecycle deferrals remain timers and use one common ordering mechanism.
- Event/deadline-driven owners disarm while no durable work exists.
- Receipt reclamation uses next-deadline one-shots and bounded continuations,
  not a fixed cleanup interval.
- Host command waits remain outside canister timer authority.
- `reservation_limit` continues to mean total committed-plus-pending resource
  budget.
- No timer cleans collection quantity.
- Pending receipt-backed intents have no age-based deletion.
- The `100_000` ceiling remains fail closed during the initial design.
- Only application-owned terminal rows with a closed replay window are
  candidates for timed reclamation.
- Canic owns the bounded local timer; Toko owns mint recovery and deadline
  derivation.
- Canic-owned placement retains acknowledgement-owned cleanup.
- The public contract is a pre-1.0 hard cut with no compatibility layer.
- Current versioned names remain version `1` where a version is necessary.
