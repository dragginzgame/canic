# Canic 0.95 Design: Timer Authority And Scheduling Consolidation

Date: 2026-07-20

## Status

- Status: proposed; not accepted and not active.
- Intended release line: 0.95, after the bounded 0.94 backup/restore line
  closes.
- Scope: every autonomous canister timer, lifecycle deferral, retry scheduler,
  and shared timer primitive in this repository.
- Bounded command-scoped host waits are inventoried but remain outside the
  canister timer runtime.
- Receipt replay horizons, receipt-retention state, terminal reclamation,
  capacity engineering, and Toko adoption are explicitly deferred to the
  [0.96 receipt proposal](../0.96-receipt-replay-horizon-and-terminal-reclamation/0.96-design.md).
- Implementation is not authorized until the inventory and every finding
  disposition are accepted.

## Decision

0.95 gives Canic one timer authority and migrates the existing timer owners to
it. The line is complete when every current process has a justified trigger,
cannot overlap itself, reconstructs correctly after lifecycle changes, reports
truthful status, and remains unregistered when event- or deadline-driven work
is absent.

The release may add a generic bounded due-work primitive. It does not add a
receipt-specific record, deadline, eligibility index, capacity counter, API,
or downstream contract.

The dependency between release lines is:

```text
0.95 releases the audited timer authority and bounded due-work scheduler
  -> 0.96 consumes that scheduler for receipt terminal reclamation
```

Unresolved Toko retry, authorization, rate, and entrypoint decisions cannot
block 0.95 closeout.

## Release Boundary

0.95 owns:

- exhaustive timer and timed-wait inventory;
- the one direct IC timer platform boundary;
- guarded one-shot registration and replacement;
- execution non-overlap and running-state arbitration;
- generation handling and stale-callback rejection;
- after-completion recurrence;
- bounded continuation and generic due-work scheduling;
- root, nonroot, and generated user-hook lifecycle reconstruction;
- two-dimensional live timer status and fixed-cardinality metrics;
- removal of unnecessary idle polling;
- migration of existing autonomous process owners; and
- the pre-1.0 public timer-surface hard cut.

0.96 owns:

- application receipt replay deadlines;
- downstream authorization and operation binding;
- receipt replay metadata and eligibility indexes;
- terminal observation grace and reclamation;
- receipt-record and stable-memory capacity proof;
- receipt capacity diagnostics; and
- the Toko adoption contract.

0.95 must not add a placeholder receipt API or stable allocation in
anticipation of 0.96.

## Current Implementation Reality

### Shared timer infrastructure

Production direct calls to `ic_cdk_timers` are contained in
`ops::runtime::timer`, which is the correct platform boundary. Above it,
`TimerWorkflow`, `TimerApi`, and public timer macros repeat or expose scheduling
choices.

The initial static inventory identifies these findings for executable
reproduction:

- guarded registration prevents duplicate handles but does not prevent two
  async interval callbacks from overlapping;
- guarded one-shots depend on callback-specific manual slot clearing;
- guarded scheduling and slot handoff are repeated across ops and API layers;
- lifecycle deferrals call `TimerApi`, `TimerWorkflow`, and `TimerOps` from
  different adapters despite having the same purpose;
- the public facade returns opaque handles without a maintained cancellation
  operation;
- timer metrics are keyed by scheduling history, while status projects those
  historical rows as enabled and registered after a one-shot completes or a
  timer is cleared; and
- timer tasks return `()`, preventing the common layer from distinguishing
  success, no work, retry, continuation, or expected failure.

### Built-in process inventory

The initial inventory finds these production process families. The disposition
column is provisional until the audit reproduces its evidence.

| Process | Current trigger | Authoritative work reason | Initial disposition |
| --- | --- | --- | --- |
| Lifecycle and user-hook deferral | Zero-delay one-shot during init or post-upgrade | Restored Canic invariants permit the next asynchronous phase | Retain the boundary and consolidate registration and ordering. |
| Log retention | Start after 30 seconds, then every 10 minutes | Entries may exceed count, byte, or optional age policy | Conditional: retain until append-time count enforcement and next-age-expiry scheduling are proven. Disarm whenever no configured policy can have future due work. |
| Cycle history sampling | Immediate sample, then every hour | Balance changes without a Canic-local event | Retain necessary recurring observation; measure cadence and separate its result from top-up. |
| Automatic cycle top-up | First check after 30 seconds, then coupled to hourly cycle tracking | Configured balance may cross its threshold externally | Retain only when enabled; evaluate independent cadence and status. |
| Root pool scheduler | Every 30 minutes plus event-driven reset scheduling | Stable reset work is pending or requires delayed retry | Replace always-on idle polling with one bootstrap check and scheduling only while work exists. |
| Pool reset drain | Guarded zero-delay one-shot with bounded continuation | A stable reset batch is pending | Retain event-driven drain and move non-overlap, continuation, and slot handling into the common runtime. |
| Local-intent expiry cleanup | Start after 30 seconds, then every hour after any begin | A finite-TTL intent reaches expiry | Replace the full scan with an expiry-ordered index and next-due scheduling. TTL-free intents must not arm or retain this timer. |
| Root delegated-proof renewal | Start after 30 seconds, then every minute while configured | Proof expiry advances without state mutation | Retain time-based renewal; evaluate exact-next-deadline scheduling and prove configuration can advance a scheduled run. |
| Placement receipt acknowledgement | Immediate guarded attempt, then one-minute retries in batches | Terminal placement receipts await root acknowledgement | Retain event/retry scheduling only while work exists; use a dedicated queue/index if the audit confirms unrelated receipt scanning. |

The public timer macros are capability surfaces, not built-in processes. The
initial call-site scan finds in-repository runtime-probe use of unguarded
one-shot and interval macros but no production consumer of the guarded public
macros. Every retained macro and facade method requires a current consumer and
complete semantics; otherwise 0.95 removes it.

### Host timed-wait boundary

Root-install readiness polling and backup command-lifetime quiescence are
bounded process waits, not autonomous canister timers. Test-only sleep loops do
not become production inventory rows unless they specify production behavior.

The audit records host waits so time-based behavior is not hidden, but it does
not route them through the canister scheduler. Backup process-death and
quiescence work remains owned by 0.94.

## Audit Contract

The audit covers production Rust under `crates/`, `canisters/`, and `fleets/`.
It searches direct timer calls, wrappers, macros, lifecycle registration,
self-rescheduling callbacks, retry delays, sleeps, intervals, and timeout
loops.

Every inventory row records:

1. fixed timer key and diagnostic label;
2. owning role and configuration gate;
3. lifecycle registration and reconstruction path;
4. current scheduling mode, initial delay, cadence, retry delay, and batch
   bound;
5. authoritative work predicate and whether it is heap or stable;
6. awaited or external effects and overlap risk;
7. cursor, continuation, retry, and stop conditions;
8. idle wake-up and instruction/stable-memory cost;
9. expected failure result and operator visibility;
10. public and test consumers; and
11. accepted disposition: retain, remove, event-drive, deadline-drive,
    consolidate, or block for evidence.

The dated audit report contains command-backed source evidence and negative
searches. A new direct timer call, wrapper bypass, or production sleep loop
must make the inventory guard fail until assigned to an owner.

The decision rules are:

- use an event when Canic observes every transition that creates work;
- use an exact deadline when authoritative state identifies the earliest due
  item;
- use a retry wake-up only while external or fallible work remains pending;
- use recurring observation only when relevant state changes without a local
  event or maintained deadline;
- use a zero-delay one-shot only for a lifecycle deferral or bounded
  continuation boundary;
- disarm when the durable work predicate becomes false; and
- reject a timer whose only justification is that an older timer existed.

Timer removal requires proof that lifecycle reconstruction, concurrent
producers, and retries cannot lose work. Timer retention requires a named work
reason, bounded callback, non-overlap guarantee, reconstruction rule, measured
cost, and truthful status.

## Central Timer Invariant

Every armed timer must correspond to a named lifecycle deferral, known due
work, pending retry, or necessary recurring observation.

For every timer key:

- at most one callback invocation runs at a time;
- scheduling while running cannot arm another runnable callback;
- each invocation performs bounded work and yields between continuation
  batches;
- authoritative subsystem state, not the heap handle, reconstructs work after
  upgrade;
- expected failures return typed outcomes;
- no-work and disabled processes become unregistered; and
- status distinguishes registration truth from process condition.

This invariant applies to common scheduling mechanics. It does not move domain
work predicates, storage records, retry policy, or outcome interpretation into
the timer runtime.

## Goals

0.95 must:

1. publish an exhaustive command-backed timer inventory;
2. assign an evidence-backed disposition to every inventory row;
3. preserve one direct IC timer platform owner;
4. consolidate guarded registration, replacement, generation, slot release,
   execution arbitration, continuation, and recurrence;
5. guarantee non-overlap even when new work is scheduled during an active
   callback;
6. reconstruct the exact required timer set after init and upgrade;
7. expose truthful two-dimensional timer status;
8. remove idle polling where events or authoritative deadlines cover all work;
9. retain and measure genuinely necessary recurring observation;
10. migrate every existing built-in timer owner to the common authority;
11. hard-cut unused, incomplete, or duplicate public timer surfaces; and
12. complete cumulative runtime validation across all built-in roles.

## Non-Goals

0.95 does not:

- add receipt replay or reclamation behavior;
- define Toko authorization, retry, rate, or entrypoint contracts;
- add a global heartbeat for unrelated work;
- merge subsystem domain state machines to reduce line count;
- remove polling without proving all work producers and lifecycle paths;
- force exact deadlines where external-state observation requires a measured
  recurrence;
- make host command waits use IC timer infrastructure;
- reopen 0.94 backup semantics;
- promise automatic recovery from traps the IC callback boundary cannot catch;
- retain removed public surfaces through aliases or compatibility wrappers;
  or
- invent unnecessary versioned names before 1.0.

## Shared Scheduling Architecture

### Fixed timer keys

Every built-in process has one fixed internal timer key. Labels and metrics are
derived from that key rather than caller-provided strings. A subsystem may
request scheduling for its key but cannot access another subsystem's handle or
running state.

The common runtime owns:

- IC handle creation and clearing;
- generation and request-sequence counters;
- registration and running arbitration;
- pending requested deadlines;
- after-completion scheduling;
- typed execution outcomes and scheduling directives; and
- live status and fixed-cardinality metrics.

The subsystem owns:

- whether work or observation is required;
- authoritative stable indexes, queues, and cursors;
- batch bounds;
- domain execution;
- expected error classification; and
- the returned next scheduling directive.

### Scheduling directives

Callbacks return one current directive:

```text
Stop
ContinueImmediately
RetryAfter { delay }
ScheduleAt { deadline }
RecurAfter { delay }
```

The execution outcome separately records success, no work, retryable failure,
or invariant failure. Expected domain failures are typed before they reach the
common runtime; error text never decides scheduling.

`ContinueImmediately` means a new zero-delay message after the current bounded
batch completes. It does not loop within the callback.

### Running-state arbitration

Registration non-duplication and execution non-overlap are separate
guarantees. The scheduler uses this exact rule:

```text
scheduled callback begins
  -> clear its consumed handle
  -> mark the timer key running for that generation
  -> while running, record scheduling requests as pending requirements
  -> do not arm any new callback for that key
  -> callback returns one typed outcome and directive
  -> merge the directive with pending requirements
  -> apply disable/cancel precedence
  -> mark the run complete
  -> arm at most one next handle
```

Pending schedule requests are stored as absolute deadlines. Multiple pending
requests merge to the earliest valid deadline.

Precedence at completion is:

1. a disable or cancel command issued during the run clears pending work and
   suppresses the callback's rearm directive;
2. a later explicit enable/schedule command supersedes that earlier
   disable/cancel but still waits for the running invocation to finish;
3. otherwise, an external pending requirement and the callback directive are
   merged and the earliest deadline wins; and
4. when neither requires a future run, the key becomes unregistered.

Commands carry a monotonic request sequence so "later" is deterministic. A
stale generation or earlier request cannot clear, arm, or overwrite a newer
decision. A request for earlier work while running therefore cannot overlap
the active callback, and it cannot be lost when that callback returns `Stop`.

### Guarded one-shots and recurrence

When a timer is scheduled but not running, an earlier requested deadline
replaces the later handle and advances the generation. A later request does not
postpone known earlier work unless an explicit policy command cancels or
reschedules it.

Guarded one-shots release their consumed handle in common infrastructure
before domain work begins. Subsystems do not clear their own slots.

Recurring async observation uses after-completion scheduling by default. Its
next delay begins only after the current invocation returns, preventing
overlap. A fixed-rate mode is permitted only if the audit proves a wall-clock
requirement and defines how missed ticks are coalesced without overlap.

### Generic bounded due work

0.95 may provide a reusable due-work runner with this contract:

```text
load authoritative earliest work
  -> stop or schedule its future deadline when nothing is due
  -> process at most the subsystem batch bound
  -> return typed outcome plus one scheduling directive
```

The primitive knows nothing about intents, receipts, proofs, logs, pools, or
acknowledgements. It does not own a stable record. A subsystem supplies the
ordered/queued authority and maps domain results into directives.

There is no shared minute or hour heartbeat. Independent timer keys retain
independent deadlines, failures, batches, and status.

## Timer Status And Observability

Timer status has two independent dimensions:

| Dimension | Values |
| --- | --- |
| Registration | `unregistered`, `scheduled`, `running` |
| Process condition | `disabled`, `idle`, `active`, `retrying`, `failed`, `missing_registration` |

Examples:

- an enabled queue with no work is `unregistered + idle`;
- a configured recurring observer between runs is `scheduled + active`;
- a retry wake-up is `scheduled + retrying`;
- disabled configuration is `unregistered + disabled`;
- authoritative due work without a handle is
  `unregistered + missing_registration`; and
- an invariant failure with automatic retry suppressed is
  `unregistered + failed`.

The live view reports at least:

- timer key, owner, scheduling mode, and enabled configuration;
- both status dimensions;
- current generation and next due timestamp when scheduled;
- last typed outcome and bounded work count;
- consecutive expected failures and retry deadline; and
- schedules, executions, successes, expected failures, and stale callbacks.

Last-run timestamps, generations, and execution counters are heap-only unless
an accepted finding proves durable operational value. Heap-only values are
explicitly labeled "since current runtime start" and reset on upgrade. They
must not be presented as lifetime history.

An ordered subsystem index remains the authority for its earliest deadline.
Status derives the deadline from the first key or the live handle; it does not
persist a duplicate `next_deadline` field unless measured evidence proves the
first-key read unsuitable.

Metric labels are fixed by timer key, registration state, process condition,
and bounded outcome class. Caller IDs, operation IDs, resource keys,
deadlines, and raw error strings are prohibited labels.

## Unexpected Traps

Typed outcomes cover expected failures only. 0.95 does not claim that the
runtime can catch and classify an unexpected canister trap after domain work
begins.

The contract is:

- expected failures must be returned as typed outcomes;
- an unexpected trap may prevent completion arbitration, failure recording,
  and next-handle registration;
- authoritative stable work remains the recovery source;
- live status may become overdue or `missing_registration` only when that
  condition can be established honestly;
- recovery occurs through a supported runtime restart path or lifecycle
  reconstruction; and
- 0.95 adds no watchdog heartbeat solely to recover trapped timers.

Each migrated owner must minimize the trap surface before its first await and
validate corrupt stable data as a typed invariant failure wherever possible.

## Lifecycle Reconstruction

Init and post-upgrade adapters restore synchronous Canic invariants, then use
one common zero-delay lifecycle deferral. Root, nonroot, and generated user
hooks have one documented ordering path.

Each subsystem reconstructs from its own authority:

- configured recurring observation schedules its next run;
- an ordered due-work owner schedules its earliest stable deadline;
- a queue owner schedules immediately only when stable work exists;
- a retry owner restores the accepted retry requirement; and
- an empty or disabled owner remains unregistered.

Reconstruction is idempotent in one runtime generation. Heap handles,
generations, last-run data, and execution counters are not imported as stable
truth.

Repeated start or configuration requests must merge through running-state
arbitration. An occupied slot cannot silently discard an earlier required run.

## Existing Owner Migration

### Lifecycle deferrals

Retain zero-delay behavior because lifecycle adapters cannot await. Replace
direct ops/workflow/API variation with one lifecycle scheduling facade and one
ordering test matrix.

### Local-intent expiry

Maintain a stable expiry-ordered index for finite-TTL intents. Begin, terminal
transition, abort, import, and removal update the index through intent ops.
TTL-free intents never enter it.

The workflow arms the earliest expiry, drains a bounded due batch, and stops
when the index is empty. It does not scan every pending intent each hour.

### Pool reset

Run one lifecycle bootstrap check. New or recovered pending work schedules the
bounded reset worker immediately. External or blocked work returns an accepted
retry delay. Empty stable state has no pool timer.

### Placement acknowledgement

Use the event/retry pattern only while terminal acknowledgement work exists.
If the audit confirms that paging the receipt-backed primary map repeatedly
scans unrelated consumer records, introduce a placement-owned durable queue or
index. The timer runtime does not interpret acknowledgement results.

### Log retention

Do not remove the interval until all count, byte, and age policies have another
complete authority. The audit must evaluate:

- append-time enforcement for count and per-entry byte limits;
- exact earliest-age-expiry scheduling when maximum age is configured;
- whether an empty log or configuration without age policy can remain
  unregistered; and
- active-work instruction cost compared with the current bounded sweep.

The accepted design may retain a measured periodic sweep if an event/deadline
replacement is more complex or less reliable.

### Cycle sampling and automatic top-up

Cycle balance can change without a local event, so recurring observation is
legitimate. Sampling and top-up expose separate outcomes and process status
even if they retain a shared cadence. Top-up is disabled and unregistered when
configuration does not enable it.

The audit freezes cadence, purge behavior, failure handling, and whether a
shared or independent schedule is simpler and cheaper.

### Delegated-proof renewal

Expiry requires time-based execution. The audit compares the current
one-minute sweep with exact-next-renewal scheduling plus a bounded safety
margin. Configuration changes must advance an already scheduled later wake-up
through common arbitration. Renewal sweeps cannot overlap.

## Layering And Ownership

The strict dependency direction remains:

```text
endpoints -> workflow -> policy -> ops -> model
```

- `model::runtime` owns named internal timer keys and passive execution/status
  values where model placement is needed.
- `view::runtime` owns read-only timer projections.
- `ops::runtime::timer` is the only direct IC timer owner and performs
  mechanical handle operations and clock reads.
- `workflow::runtime::timer` owns common arbitration, generations,
  non-overlap, directives, recurrence, and live status.
- subsystem ops own authoritative stable indexes, queues, cursors, and
  conversion.
- subsystem workflows own work predicates, domain execution, batch bounds,
  and retry/recurrence decisions.
- lifecycle adapters and macros marshal and delegate immediately.
- policy remains pure and does not register timers, access storage, await, or
  inspect runtime status.

Workflow must not construct or mutate stable records. Timer ops must not learn
domain policy.

## Public And Stable Hard Cut

The timer facade changes directly to the accepted current surface. An unused
guarded macro, duplicate lifecycle entrypoint, opaque handle without a
maintained operation, or incomplete fixed-interval helper is removed when the
audit cannot prove a current consumer and complete semantics.

There are no aliases, deprecated wrappers, legacy fallbacks, or
anti-resurrection tests for removed forms. Current tests cover the maintained
surface only.

Schema versioning remains honest:

- a new stable allocation starts at schema version `1` when it requires a
  version;
- a changed existing versioned shape or meaning receives the next schema
  version even before product 1.0; and
- the previous reader is removed in the hard cut rather than retained as a
  compatibility path.

Product version and persisted-schema version are separate concepts. Current
unversioned internal types do not gain version suffixes merely because their
implementation changes.

Canonical export/import includes any accepted new authoritative expiry or
acknowledgement index. Heap timer handles, registry state, generations, and
runtime counters are reconstructed and are not canonical state.

## Implementation Slices

Implementation begins only after design acceptance and 0.94 closeout.

### Slice A: Exhaustive Audit And Finding Gate

- publish the complete timer/process and host-wait inventory;
- reproduce overlap, stale-registration, idle-wake, full-scan, and status
  findings with focused evidence;
- record per-role timer count, idle wake-up, instruction, stable-read/write,
  and external-call baselines;
- assign and accept one disposition for every row; and
- freeze the maintained public timer surface.

No implementation begins with an undispositioned timer owner.

### Slice B: Common Timer Authority

- introduce fixed keys, typed outcomes, and scheduling directives;
- implement guarded one-shots with generation-safe replacement and automatic
  slot release;
- implement the running-state arbitration and command precedence defined here;
- implement after-completion recurrence and bounded due-work mechanics;
- replace historical-registration health with two-dimensional live status;
- consolidate lifecycle deferral; and
- hard-cut incomplete or unused public timer surfaces.

### Slice C: Event- And Deadline-Driven Owners

- migrate local-intent expiry to an ordered finite-expiry index;
- replace the pool's idle interval with bootstrap, events, and pending-only
  retries;
- migrate placement acknowledgement to pending-only scheduling and an accepted
  queue/index authority;
- move all bounded drains to common continuation and arbitration; and
- change log retention only according to accepted measured evidence.

### Slice D: Necessary Observation And Closeout

- migrate cycle sampling, automatic top-up, and delegated-proof renewal;
- freeze their independent/shared cadence and deadline decisions;
- prove configuration can enable, disable, and advance work while a callback
  is running;
- compare every owner with the Slice A idle and active baseline; and
- complete the cumulative validation gate.

## Validation Plan

### Audit and structural evidence

- every production timer/wait call site maps to exactly one inventory row;
- exactly one production ops module calls the direct IC timer API;
- every built-in registration uses the common workflow authority;
- no subsystem duplicates guarded-slot, generation, running arbitration,
  after-completion, or continuation mechanics;
- public call sites justify every retained macro and facade method;
- no removed form has an alias, shim, fallback, or compatibility test; and
- host waits remain outside the canister scheduler.

### Common scheduler unit evidence

- duplicate guarded registration cannot create two handles;
- an earlier deadline replaces a scheduled later deadline;
- stale generations cannot clear or overwrite a newer handle;
- schedule requests made during a run do not arm a callback until completion;
- multiple running-time requests merge to the earliest deadline;
- callback `Stop` cannot discard new work requested during that callback;
- disable/cancel and later enable/schedule follow request-sequence precedence;
- after-completion async recurrence cannot overlap across advanced IC time;
- continuation yields between bounded batches;
- completion and clearing report `unregistered`;
- registration and process-condition dimensions produce every documented
  combination; and
- upgrade resets values labeled "since current runtime start."

### Per-owner unit evidence

- TTL-free intents do not arm expiry work;
- finite expiry entries drain in order and survive import/export exactly;
- empty pool and placement owners remain unregistered after bootstrap;
- new work, bounded backlog, and external failure schedule the accepted next
  action;
- log count, byte, and age invariants hold under the accepted trigger model;
- cycle sampling and top-up expose separate outcomes and cannot overlap;
- proof renewal schedules within its frozen margin and advances for earlier
  work; and
- lifecycle/user hooks execute once in the frozen order.

### PocketIC evidence

- install and upgrade every built-in role and assert the exact timer status
  matrix;
- hold one async callback open, advance time across nominal cadences, request
  earlier work, and prove same-key non-overlap;
- upgrade with queued, future-due, retrying, empty, and disabled work and prove
  exact reconstruction;
- create and drain pool, placement, and local-expiry work and prove their keys
  become `unregistered + idle`;
- enable, disable, and change recurring configuration while work runs;
- inject expected failures and prove typed retry/failure status; and
- exercise the documented unexpected-trap limitation and supported lifecycle
  recovery without claiming an uncaught trap was classified.

### Cost evidence

Measure an idle window long enough to cross every retained cadence for each
role. Compare callback count, instructions, stable reads/writes, and external
calls before and after migration. Measure bounded active workloads so reduced
idle polling does not hide starvation or excessive active cost.

Acceptance requires zero wakes for empty event/deadline owners, no unexplained
increase for recurring observers, bounded backlog progress, and a documented
reason for every remaining idle wake-up.

### Cumulative closeout gate

Because 0.95 changes the system-wide runtime, final closeout requires:

- complete relevant workspace tests;
- all-target strict Clippy for affected workspace packages;
- formatting and diff hygiene;
- layering and public-surface structural guards;
- stable-state and canonical import/export suites;
- all built-in-role PocketIC lifecycle/timer suites; and
- changelog and status governance checks.

This gate does not authorize unrelated CI cleanup or mutation outside Canic.
During implementation, individual batches still use targeted checks until the
accepted final closeout run.

## Risks And Mitigations

### Scheduling during a running callback overlaps

Mitigation: never arm while the key is running. Record pending absolute
deadlines, merge them after completion, apply sequenced disable/cancel
precedence, and arm at most one handle.

### Removing polling loses work

Mitigation: require authoritative work state plus lifecycle reconstruction for
every event/deadline migration. Retain recurring observation when relevant
state changes externally.

### After-completion recurrence drifts

Mitigation: use it for operational polling where non-overlap matters more than
wall-clock phase. Admit fixed-rate mode only with measured need and explicit
missed-tick coalescing.

### A global scheduler couples failures

Mitigation: share mechanics only. Timer keys, deadlines, domain workflows,
batches, and failures remain independent; no global heartbeat is introduced.

### Unexpected traps strand work

Mitigation: return expected failures, preserve stable work authority, expose
honest overdue/missing-registration evidence where possible, and recover only
through supported runtime/lifecycle paths. Do not claim automatic trap catch.

### DRY abstraction absorbs domain policy

Mitigation: the common runtime knows timer keys, requests, outcomes, and
directives only. Subsystem workflows and ops retain all domain decisions and
stable authority.

## Acceptance Criteria

0.95 is complete only when:

1. every timer, retry scheduler, lifecycle deferral, and bounded host wait is
   inventoried and dispositioned;
2. direct IC timer access has one owner;
3. running-state arbitration prevents overlap and lost concurrent work;
4. guarded slots release automatically and stale generations are harmless;
5. event/deadline-driven owners have no timer while their work predicate is
   false;
6. every retained recurring observer has measured cost and justification;
7. init and upgrade reconstruct the exact timer set once;
8. registration state and process condition are reported independently;
9. heap-only history is labeled and resets honestly on upgrade;
10. expected failures are typed and unexpected-trap limitations are explicit;
11. existing built-in timer owners use the common authority;
12. the maintained public surface contains no compatibility path;
13. schema versions follow persisted-shape meaning rather than product-version
    convenience;
14. cumulative runtime validation passes; and
15. no receipt-specific API, storage, capacity, security, or Toko decision has
    entered the release.

## Closed Decisions

- 0.95 is timer-only; receipt reclamation moves to 0.96.
- There is no global heartbeat.
- Direct IC timer access remains in one ops module.
- Scheduling while running records pending requirements and never arms another
  runnable callback.
- Disable/cancel precedence and later enable/schedule are request-sequenced.
- After-completion recurrence is the default for async recurring work.
- Timer status has independent registration and process-condition dimensions.
- Heap execution history is labeled "since current runtime start."
- Unexpected traps are not claimed as typed, automatically recovered failures.
- Ordered indexes own their earliest deadline; status does not persist a
  duplicate without evidence.
- Host command waits remain outside canister timer authority.
- Public removals are pre-1.0 hard cuts without compatibility layers.
- New persisted allocations begin at schema version `1`; changed existing
  versioned meanings receive the next schema version and lose the old reader.
