# Canic 0.96 Design: Receipt Replay Horizon And Terminal Reclamation

Date: 2026-07-20

## Status

- Status: active. Slices A through C are released through `v0.96.3`. Open
  `0.96.4` starts Slice D by freezing terminal observation grace, provisioning
  settlement capacity, and persisting exact terminal eligibility. It does not
  enable reclamation.
- Release line: 0.96, with the current immutable anchor at released
  `v0.96.3`.
- Required platform dependency: the released
  [0.95 timer authority](../0.95-timer-authority-and-scheduling-consolidation/0.95-design.md),
  including guarded one-shots, running arbitration, bounded due work,
  lifecycle reconstruction, and truthful status.
- Downstream scope: Toko remains read-only from this repository. Its signed
  delegated-token expiry is the source-backed deadline ceiling; exact action
  identity, recovery, and numeric capacity conformance remain Toko-owned.
- The terminal observation grace and eligibility capacity are now Canic-owned
  fixed contracts. Reclamation and scheduling remain gated on downstream
  conformance, maintained counters, diagnostics, and bounded cleanup proof.

## Decision

Canic has two independent receipt-intent limits:

| Limit | Meaning | 0.96 decision |
| --- | --- | --- |
| `reservation_limit` | Per-resource admitted quantity: committed plus pending plus the new request | Preserve unchanged. It is permanent collection/resource accounting and has no cleanup timer. |
| `RECEIPT_BACKED_INTENT_RECORD_LIMIT` | Per-canister count of distinct receipt-backed operation records, currently `100_000` | Preserve the fail-closed record ceiling while adding an immutable replay contract, admission-time terminal-capacity reservation, terminal observation grace, ordered eligibility, bounded reclamation, and capacity diagnostics. |

Committed quantity remains consumed. Rolled-back quantity is released during
settlement. Pending quantity remains reserved until the downstream owner proves
the effect or durable no-effect. Time never decides those outcomes.

The record ceiling is a technical retention bound. A terminal record currently
prevents an old `OperationId` from looking new forever. 0.96 may remove that
record only after:

1. the downstream authorization can no longer be replayed as a fresh effect;
2. a frozen terminal observation grace has elapsed;
3. exact record, binding, revision, state, and eligibility checks succeed; and
4. an absent retry is rejected before reservation or external-effect
   admission.

Receipt reclamation consumes the 0.95 bounded due-work scheduler. It does not
add another fixed polling interval.

## Release Boundary

0.96 owns:

- the application-owned receipt replay-deadline contract;
- immutable domain authorization, `OperationId`, payload-binding, and deadline
  relationships;
- exact begin-or-load temporal ordering;
- replay metadata and terminal eligibility authority;
- a frozen terminal observation grace;
- admission-time reservation of worst-case terminal-transition capacity;
- bounded terminal-only reclamation;
- poisoned-earliest-entry failure behavior;
- receipt capacity and cleanup diagnostics; and
- the downstream Toko handoff and conformance gates.

0.96 does not redesign timer mechanics. A timer-runtime defect found during
receipt integration is fixed in the 0.95 authority or a focused follow-up; it
is not reimplemented inside intent workflow.

## Why A Replay Horizon Is Required

Raising `100_000` delays exhaustion but leaves history unbounded. Deleting a
terminal record based only on age is unsafe because the old request might still
be accepted and repeat its external effect.

A safe reclamation contract requires this ordering:

```text
freeze immutable downstream authorization and deadline derivation
  -> reproduce the same operation ID, payload binding, and deadline on retry
  -> reject an expired absent operation before capacity or reservation
  -> retain every pending operation regardless of age
  -> settle only from validated terminal evidence
  -> retain the terminal result through an observation grace
  -> index terminal records by exact safe eligibility time
  -> reserve settlement/index capacity at initial admission
  -> reclaim eligible terminal rows in bounded local batches
  -> reject post-reclamation replay before another external effect
```

## Current Implementation Reality

The current receipt-backed intent path has these properties:

- `BeginReceiptBackedIntentInput::reservation_limit` is required.
- Admission checks
  `reserved_qty + committed_qty + requested_quantity` for the exact resource.
- Direct `OperationId` lookup occurs before record and resource capacity
  checks, so exact retained retries remain readable at capacity.
- A new operation is rejected with `StoreCapacityReached` when the primary
  receipt map contains `100_000` rows.
- Pending, committed, and rolled-back records all count toward the limit.
- Application-owned receipt records have no replay deadline and are retained
  indefinitely.
- Public consumers can begin, load, and settle but cannot remove terminal
  records.
- Canic placement has a private exact removal path after authoritative root
  acknowledgement; that rule does not automatically apply to Toko.
- Status does not expose maintained receipt counts, remaining headroom, or the
  earliest terminal eligibility time.

The retained primary row supplies two safety properties:

1. pending state fences a duplicate external effect while the outcome is
   uncertain; and
2. terminal state returns the established outcome instead of creating a new
   reservation and effect.

Settlement changes pending to committed or rolled back. It does not by itself
make the operation safe to forget.

## Read-Only Toko Baseline

The audit inspected Toko commit
`485f586184651d67739b8e9c0ec489fea6a16b3a` read-only. That snapshot consumes
Canic `0.71.3`, contains no receipt API reference, and therefore cannot contain
old Canic application receipt rows. Its 0.96 adoption is a direct hard cut;
Canic adds no old-row reader or migration for it.

The current maintained path is:

```text
two UI views
  -> one ProjectService/useMintTokens mutation
  -> authenticated project_instance::mint_token_tems(token, token_ids)
  -> project_instance::ops::ledger::mint_token_items
  -> parent-only project_ledger::mint_tokens
  -> atomic supply check and mint
  -> post-await local minted-count increment
```

The current request has no per-action identity, deadline, replay receipt, or
recovery endpoint. The ledger also exposes a parent-only `mint_token_stack`
effect with no maintained caller. That surface is removed or brought under the
same contract during downstream adoption; it cannot remain a bypass.

Delegated-token claims already provide a signed subject, issuer, issue time,
expiry, audience, grants, session nonce, and extension. Toko caps token expiry
at 24 hours and the frontend requests that maximum. The actual deadline can be
shorter because token issuance preserves delegation-proof headroom. Signed
token expiry is therefore the smallest available replay-deadline ceiling, but
the session nonce is not a mint-action nonce: identical mints in one session
can be distinct legitimate actions.

Toko must add one explicit immutable action identity and bind it with the
authenticated user, project/ledger identities, normalized effect payload, and
absolute deadline. A caller may not extend the deadline beyond the currently
verified authorization expiry. Retrying or recovering the same action carries
the original deadline; a refreshed bearer token authorizes observation but
does not create a new action identity.

No source-backed average or burst rate exists yet. `mint_token_tems` has no
explicit batch bound or rate guard. Those limits remain downstream acceptance
inputs rather than values invented by Canic.

## Central Invariant

Removing a receipt-backed terminal record must never allow the same authorized
domain operation to reserve capacity, execute an external effect, or settle
resource accounting a second time.

Consequences:

- pending records are never removed because of age;
- an expired absent operation rejects before store capacity, resource
  capacity, reservation, or external-effect admission;
- retry identity and deadline are immutable products of one domain
  authorization, not current time;
- terminal reclamation changes receipt-history storage only and never changes
  `reserved_qty`, `committed_qty`, or `pending_count`;
- every admitted pending record already owns enough capacity for its worst-case
  terminal transition;
- contradictory eligibility fails closed without deleting or skipping state;
- cleanup performs no downstream call and infers no domain outcome; and
- a consumer that cannot prove all stale-request rejection paths is not
  eligible for automatic reclamation.

## Goals

0.96 must:

1. define one immutable downstream authorization and replay-deadline contract;
2. distinguish domain authorization identity, Canic `OperationId`, payload
   binding, and replay deadline;
3. freeze exact begin-or-load ordering and equality behavior;
4. keep exact retained retries idempotent even after their deadline;
5. reject expired absent operations before mutation or external effects;
6. retain uncertain pending rows indefinitely for targeted reconciliation;
7. retain newly settled terminal results for a frozen observation grace;
8. reserve worst-case terminal-index and stable-memory capacity at admission;
9. index terminal rows by exact safe eligibility time;
10. reclaim only exact eligible terminal rows through the released 0.95
    bounded due-work scheduler;
11. fail without automatic retry on a contradictory earliest eligibility row;
12. maintain exact receipt counts without primary-map scans;
13. derive the next eligibility deadline from the ordered index;
14. expose low-cardinality capacity and cleanup diagnostics;
15. prove reclamation cannot change resource accounting or repeat an effect;
    and
16. publish a precise Toko adoption and conformance contract.

## Non-Goals

0.96 does not:

- change `reservation_limit` semantics;
- add time-based rollback or deletion of pending intents;
- infer no-effect from timeout, reject text, missing response, or unchanged
  observed state;
- add a generic transaction, saga, effect executor, receipt validator, or
  downstream reconciliation daemon;
- make Canic interpret Toko mint, collection, ledger, cancellation, or receipt
  types;
- add economic, per-user, or per-collection rate limits to Canic;
- make deletion a caller-selected escape from capacity;
- scan the full receipt map on a timer, upgrade, or request;
- quarantine or silently skip corrupt eligibility entries;
- add an automatic eligibility-repair subsystem;
- redesign the released 0.95 timer authority;
- retain the old begin input through a default, overload, alias, or wrapper;
  or
- edit the downstream Toko repository.

## Identity And Time Contract

### Domain authorization identity

The downstream domain owns one immutable authorization for the effect. It
identifies the authorized caller and action and fixes every field that may
affect replay, including the replay deadline.

Toko derives this authorization once. It must not calculate
`current_time + horizon` again on retry. Retrying the same authorization always
reproduces the original operation ID, payload binding, and absolute deadline.

A fresh deadline requires a genuinely new authenticated domain authorization.
Whether that new authorization represents a permitted new mint is Toko policy,
not Canic policy.

### Canic `OperationId`

`OperationId` is the deterministic Canic storage key for one immutable domain
authorization and action. It is stable across all retries of that
authorization.

Canic does not require one particular downstream hash layout, but the accepted
Toko vector must prove:

```text
same immutable authorization -> same OperationId
different authorization identity -> different OperationId
different authorized domain action -> different OperationId
```

A new valid authorization receives a new `OperationId` and is a new operation.
The design does not call that case a retained-row binding conflict.

### Payload binding

The payload binding covers at least:

- immutable domain authorization identity;
- authenticated caller/subject;
- collection or resource identity;
- exact effect parameters and quantity;
- Canic `OperationId`; and
- absolute replay deadline.

When a retained `OperationId` is presented with a different payload binding or
deadline, direct lookup finds the row and returns a typed binding conflict.
This detects an altered presentation of the same Canic operation key.

If Toko deliberately derives a different ID from a genuinely new
authorization, Canic treats it as new admission and relies on Toko's domain
authorization policy plus the existing resource budget.

### Replay deadline

The replay deadline is one absolute IC timestamp chosen by the trusted adapter
under a frozen policy. It closes creation of an absent operation; it does not
settle or delete an existing row.

The equality rule is exact:

```text
now >= replay_deadline_ns -> absent operation is closed
```

Existing pending and terminal rows remain loadable after the deadline while
retained.

### Terminal observation grace

The grace is exactly 24 hours, equal to the maximum admitted replay horizon:

```text
terminal_observation_grace_ns = 24 * 60 * 60 * 1_000_000_000
```

This is conservative and source-related rather than an independent arbitrary
duration: a terminal result remains observable for one complete maximum
authorization horizon after the terminal transition. An operation admitted at
the maximum horizon and settled at the final instant can occupy receipt history
for up to 48 hours from admission. Capacity conformance therefore uses 48 hours
as its maximum resolved-operation retention span.

Settlement captures one terminal timestamp. Safe reclamation time is:

```text
eligible_at_ns = max(
    replay_deadline_ns,
    terminal_timestamp_ns + terminal_observation_grace_ns,
)
```

Addition is checked before mutation. The grace is a frozen system constant,
not caller input.

This rule covers pending operations reconciled long after their replay
deadline. A lost settlement response still has a deterministic interval in
which an exact retry can observe the newly established terminal result before
reclamation.

The grace does not authorize replay after the deadline and does not decide a
pending outcome. It is terminal result-retention policy only.

## Public Contract

The application-owned begin input changes directly:

```text
BeginReceiptBackedIntentInput {
    operation_id,
    payload_binding,
    resource_key,
    quantity,
    reservation_limit,
    replay_deadline_ns,
}
```

The maintained typed decisions include:

```text
ReplayWindowClosed {
    replay_deadline_ns,
}

ReplayWindowTooLong {
    remaining_ns,
    maximum_ns,
}
```

There is no optional deadline and no default. The local Canic facade trusts
the authenticated application adapter to supply its frozen derived values;
public downstream endpoints authenticate before calling it.

Canic-owned placement retains its acknowledgement-owned retention contract.
It must not invent an application replay deadline merely to share the 0.96
reclamation path.

## Begin-Or-Load Ordering

Application begin-or-load performs no `await` and uses this exact order:

```text
capture now once
  -> validate structural input and adapter-supplied authenticated bindings
  -> look up OperationId
  -> if present, validate stored binding/deadline and return stored state
  -> if absent and now >= replay_deadline, return ReplayWindowClosed
  -> if absent and remaining window exceeds the maximum,
     return ReplayWindowTooLong
  -> reserve primary plus worst-case terminal-transition capacity
  -> check the existing per-resource reservation limit
  -> precompute primary row, replay metadata, aggregates, and counters
  -> write the complete pending authority without another fallible step
```

Temporal rejection is deliberately after direct lookup. A retained retry can
therefore return its pending or terminal state after deadline. A past deadline
is not rejected as generic structural input before lookup.

The remaining window uses the single captured `now`. Equality at the deadline
is closed and equality at the accepted maximum is admitted. There is no
minimum execution window: admission immediately before expiry is safe because
the endpoint has already authenticated, while a lost outcome remains pending
for targeted recovery. Adding a lower time bound would be a liveness policy
without evidence and is outside the minimal contract.

Record capacity is checked before resource capacity only after expired-absent
rejection. Exact existing operations remain readable and settleable while new
operations fail closed at capacity.

## Stable Retention Authority

0.96 adds current-schema replay metadata for application-owned receipt-backed
operations. The primary record or an exact one-to-one allocation stores the
immutable deadline, payload binding, and revision authority needed by begin,
settlement, and cleanup.

Terminal eligibility is ordered by:

```text
(eligible_at_ns, operation_id)
```

An eligibility entry carries the expected payload binding and terminal
revision needed for exact comparison. Settlement inserts it only after
consumer-validated terminal evidence is accepted.

Maintained metadata includes:

```text
total_records
pending_records
terminal_records
reserved_terminal_capacity
```

The exact representation of `reserved_terminal_capacity` follows the accepted
storage envelope. Counts are invariants, not reconstructed through primary-map
scans.

The next eligibility deadline is derived from the first ordered index key for
status and lifecycle reconstruction. It is not persisted as a duplicate
`next_terminal_deadline_ns` field unless focused evidence proves first-key
access unsuitable and the added drift invariant is separately accepted.

Canonical stable-state contract tests include replay metadata, eligibility
entries, capacity reservations, and maintained counts. Restore/import test
helpers reject missing, duplicate, contradictory, non-terminal, incorrectly
timed, or unreserved entries. 0.96 does not add a public export/import API;
the current snapshot helpers are test-only.

Allocation 43 is also used by Canic-owned placement. The smallest maintained
shape therefore keeps that common primary receipt authority and adds only the
application-owned replay metadata needed for deadline and reclamation. It
does not make placement invent a deadline, create a second primary receipt
store, or add a per-operation reservation record when the exact metadata row
and maintained counters already prove the reservation.

## Settlement Capacity Reservation

The `100_000` primary-record ceiling is not a sufficient capacity proof.
Settlement may need a terminal eligibility entry after the primary store is
full, and stable-memory exhaustion must not strand an admitted pending
operation.

0.96 therefore reserves worst-case terminal-transition capacity when a new
pending record is admitted. Admission accounts for:

- maximum primary record bytes;
- replay metadata;
- one maximum terminal eligibility entry;
- map node, key, value, allocator, and memory-manager overhead;
- maintained counters and reservation metadata;
- maximum unresolved pending backlog;
- canonical export/import representation and bounded working headroom; and
- cleanup working headroom.

The storage implementation must freeze a measured per-record and fixed
capacity envelope. Before the first admission mutation, it ensures that stable
allocation has been provisioned or reserved for both the pending write and its
future terminal entry. If that reservation cannot be made, admission returns
`StoreCapacityReached` without mutating resource totals or primary state.

One terminal-capacity reservation remains attached to every retained
application record:

```text
new pending admission -> reserve terminal slot and bytes
pending settlement    -> consume that reserved slot for eligibility
terminal retention    -> keep reservation represented by the live entry
terminal reclamation  -> release record and reserved capacity accounting
```

Settlement never performs a new capacity admission. Its index insertion uses
the capacity already reserved for that exact operation. The ops transition
precomputes all changes and has no fallible allocation/growth step after its
first stable mutation.

An injected small record-count limit is useful behavior evidence but is not a
stable-memory exhaustion test. Acceptance also requires measured serialized
sizes, allocator overhead, actual stable-growth/provisioning boundaries, and a
test seam that fails physical reservation at each pre-mutation boundary.

The compiled record limit remains `100_000` unless the numeric envelope
supports a separately accepted amendment.

The released `v0.96.0` record shapes, including the released `.1` corrected
totals bound, measured against pinned `ic-stable-structures 0.7.2` and
`ic-memory 0.11.1` have this baseline:

| Current allocation/shape | Exact measurement |
| --- | --- |
| Maximum pending primary value | 441 encoded bytes |
| Maximum terminal primary value | 617 encoded bytes |
| Primary map page size from its retained 1,024-byte bound | 8,855 bytes |
| Primary at 100,000 rows | 2,707 ascending / 1,899 permuted Wasm pages |
| Placement acknowledgement at 100,000 rows | 452 ascending / 317 permuted Wasm pages |
| Resource totals at 100,000 maximum distinct rows | 545 ascending / 469 permuted Wasm pages |
| Three-allocation receipt subtotal through base `MemoryManager`, ascending | 3,969 physical Wasm pages / 248.0625 MiB |

Open `0.96.4` adds the final terminal-eligibility shape:

| Eligibility allocation/shape | Exact measurement |
| --- | --- |
| Maximum eligibility value | 229 encoded bytes |
| Fixed ordered key | 40 bytes |
| Eligibility B-tree page size | 2,362 bytes |
| Eligibility at 100,000 rows | 726 ascending / 626 permuted Wasm pages |
| Four-allocation application subtotal through base `MemoryManager`, ascending | 4,737 physical Wasm pages / 296.0625 MiB |

Admission provisions the same 726-page maximum-live-node envelope at the
100,000-row ceiling. The calculation is tied to pinned
`ic-stable-structures 0.7.2`: minimum degree six gives at least five entries per
non-root node, each 2,362-byte node uses a 16-byte allocator chunk header, and
116 fixed bytes cover the map, allocator, and spare chunk headers. The ignored
capacity probe checks those values against both the isolated allocation and the
shared `MemoryManager`.

The primary value bound remains 1,024 bytes. Shrinking it before adding the
accepted replay metadata would churn the stable contract for a small saving;
the measured current terminal shape leaves 407 bytes of headroom.

The measurement also found that `IntentResourceTotalsRecord` could encode to
69 bytes while declaring a 64-byte maximum. Slice B corrects that declaration
to 69 without changing its encoded fields, stable key, allocation, or schema
meaning. Exact zero totals are removed by the ops owner because absence already
projects the same zero state. This prevents aborted or rolled-back work from
leaving empty stable rows.

Nonzero committed totals remain authoritative after receipt reclamation and
cannot be removed merely to recover capacity. The downstream envelope must
therefore freeze maximum durable resource cardinality as well as operation
rate and unresolved backlog. Admission of a new resource must eventually have
its own measured fail-closed capacity decision; the primary receipt ceiling
cannot be treated as a permanent bound after terminal reclamation begins.

This capacity proof is not permission to build a generic stable-memory
allocator or transaction framework. Use the smallest dedicated provisioning
mechanism supported by measured `ic-stable-structures` behavior. If the
existing record ceiling, bounded encoded shapes, and one dedicated growth
boundary prove the required headroom, no additional abstraction is added.

## Settlement And Eligibility

Pending rows remain outside the eligibility index. Their age may be diagnosed
but never authorizes rollback or deletion.

Settlement:

1. loads and verifies the exact pending operation;
2. validates consumer terminal evidence;
3. captures one terminal timestamp;
4. computes checked `eligible_at_ns` using replay deadline and observation
   grace;
5. verifies the operation's admission-time terminal-capacity reservation;
6. precomputes primary, eligibility, aggregate, count, and reservation changes;
7. performs the terminal transition without `await`; and
8. requests 0.95 scheduling at the new earliest deadline when necessary.

Committed settlement consumes reserved quantity into committed quantity.
Rolled-back settlement releases reserved quantity. Reclamation later changes
neither aggregate.

An eligibility insertion with an earlier deadline than the current handle uses
0.95 guarded replacement. If the cleanup callback is running, 0.95 records the
earlier request and arms it only after completion, preserving non-overlap.

## Bounded Terminal Reclamation

The receipt workflow uses the released 0.95 due-work primitive:

```text
read the earliest eligibility entry
  -> stop when the index is empty
  -> schedule its deadline when it is in the future
  -> load the exact primary row when due
  -> require application-owned terminal state
  -> require matching deadline, eligible time, binding, and revision
  -> require the terminal observation grace and replay deadline to be closed
  -> precompute exact primary/index/count/capacity removal
  -> remove the terminal row and eligibility entry
  -> leave resource totals unchanged
  -> repeat up to the frozen batch bound
  -> continue immediately for due backlog or schedule the next first key
```

The callback performs local stable-state work only. It never calls Toko, the
ledger, or another canister and never settles an intent.

It cannot delete a record to make the admission currently in progress succeed.
New admission remains fail closed until an independently scheduled cleanup has
completed safely.

### Contradictory earliest entry

A contradictory earliest eligibility entry is not retried automatically,
skipped, quarantined, or removed.

The exact policy is:

1. retain the primary row and eligibility entry untouched;
2. return a typed reclamation invariant failure;
3. transition the timer process to `unregistered + failed` through 0.95;
4. do not inspect or reclaim later entries; and
5. expose a fixed-label failure plus an owner-visible exact diagnostic for
   operator-directed investigation.

This prevents an infinite retry storm and avoids silently reclaiming later
history while the ordered authority is contradictory. A subsequent lifecycle
reconstruction may reproduce one failed attempt, but it does not create a
recurring retry loop.

0.96 does not introduce automatic repair. Repair requires an explicitly
reviewed state-restore or future exact operator mechanism; age or capacity
pressure is never repair authority.

## Pending Recovery

Toko continues to own targeted recovery for one `OperationId`:

1. observe the exact ledger receipt or durable no-effect evidence;
2. validate caller, authorization, collection, payload, source canister, and
   result;
3. settle the Canic row with exact terminal evidence; or
4. leave it pending and fail closed.

No Canic timer scans Toko operations, calls the ledger, retries a mint, or
constructs terminal evidence. A late recovered settlement receives the full
terminal observation grace before reclamation.

## Capacity And Diagnostics

One read-only projection reports:

- current total and configured record limit;
- pending and terminal counts;
- remaining logical record headroom;
- reserved terminal slots/bytes and physical stable-memory headroom under the
  frozen envelope;
- earliest eligibility deadline derived from the first index key;
- last cleanup outcome and removed count since runtime start;
- timer registration and process condition from 0.95; and
- invariant-failure presence with protected exact diagnostics.

Operation IDs, resource keys, collection IDs, and callers are prohibited
metric labels. Owner-authenticated diagnostics may carry an exact operation ID
because repair cannot safely target a metric-label hash or broad resource
prefix.

Capacity planning uses unique operations rather than minted quantity:

```text
steady_state_records
  ~= unique operations retained across replay horizon
   + unique operations retained across terminal observation grace
   + unresolved pending operations
   + bounded cleanup backlog
   + Canic-owned receipt records under their separate acknowledgement contract
```

The numeric envelope includes worst-case bytes, not just rows. Toko must
provide maximum expected unique-operation rate, burst rate, ambiguous-effect
backlog, and response-loss/retry behavior per canister. Acceptance requires
headroom below both the logical record ceiling and measured physical capacity.

## Toko Adoption Contract

Toko remains responsible for:

- creating one immutable authenticated domain authorization per permitted
  mint action;
- assigning one action identity distinct from the delegated-session nonce;
- deriving the replay deadline once from that authorization, never from retry
  time;
- reproducing the same `OperationId`, payload binding, and deadline for every
  retry of that authorization;
- creating a genuinely new authorization before any fresh deadline or new
  operation ID is permitted;
- deriving a new `OperationId` for that genuinely new authorization;
- binding caller, authorization identity, collection, quantity, mint payload,
  operation ID, and deadline;
- sourcing `reservation_limit` from the frozen collection supply authority;
- rejecting caller attempts to choose the supply limit or extend the deadline
  beyond the currently verified authorization expiry;
- calling Canic begin before `execute_mint`;
- executing the effect only after a `Created` result;
- validating ledger receipts and durable no-effect evidence;
- targeted recovery of old pending operations;
- returning typed replay-closed and terminal results;
- proving that every endpoint, callback, admin flow, and upgrade continuation
  enforces the same authorization and replay contract; and
- deleting the unused parent-only stack-mint path or routing it through the
  same contract, with no bypass left.

Toko does not delete Canic rows or run receipt cleanup. Canic owns its local
eligibility state and consumes the 0.95 scheduler.

Before implementation is accepted, the Toko owner must freeze:

1. per-action identity fields and authentication vectors;
2. exact operation-ID and payload-binding vectors;
3. signed-expiry deadline rule and evidence that the adapter preserves Canic's
   fixed 24-hour terminal observation contract;
4. maximum batch, average/burst operation rate, ambiguous backlog, and durable
   resource cardinality;
5. ledger operation-ID idempotency and receipt-retention behavior;
6. targeted recovery behavior; and
7. deletion or integration of every current mint initiation surface.

## Security Requirements

- Downstream endpoints authenticate before adapter admission.
- Caller, authorization, operation ID, resource, quantity, payload, and replay
  deadline are cryptographically bound.
- The deadline cannot exceed the currently verified authorization expiry.
- A retry never derives a later deadline from current time.
- An expired absent operation returns before capacity or resource mutation.
- A retained operation with changed binding or deadline returns a typed
  conflict.
- A new authorization/operation ID is admitted only under Toko's domain policy
  and Canic's current resource/capacity rules.
- Pending state remains the conservative unknown-effect boundary.
- Time never creates terminal evidence.
- Cleanup requires exact state, binding, revision, deadline, eligibility, and
  ownership comparison.
- Cleanup never accepts a broad caller- or resource-prefix deletion request.
- Consumer resource keys remain outside the reserved `canic:` namespace.

## Layering And Ownership

The strict dependency direction remains:

```text
endpoints -> workflow -> policy -> ops -> model
```

- `model::intent` owns receipt state, immutable replay values, terminal
  timestamps, typed outcomes, and authoritative invariants.
- stable storage contains passive records ending in `*Record`; canonical
  snapshot shapes end in `*Data`.
- `view::intent` owns read-only capacity and reclamation projections.
- `ops::storage::intent` owns direct lookup, capacity reservation, stable
  index/count mutation, conversion, and exact compare-and-remove.
- pure replay-window or capacity decisions live in policy only when they are
  shared decisions rather than checked storage mechanics.
- `workflow::runtime::intent` owns begin/settlement orchestration and maps
  receipt due work into released 0.95 scheduling directives.
- `api::intent` remains a thin local facade.
- downstream endpoints own authentication and Toko-specific derivation and
  evidence validation.

Workflow does not construct or mutate stable records. The 0.95 timer runtime
does not inspect receipt domain state.

## Public And Stable Hard Cut

The begin contract changes directly. There is no old overload, optional
deadline, default horizon, fallback reader, alias, or compatibility wrapper.

Before implementation, inventory must prove whether any application-owned
receipt consumer or stable row uses the old contract. If one exists, 0.96
blocks for an explicit safety decision; it does not invent a historical
deadline or silently remove the row.

Schema versioning follows persisted meaning:

- a new replay or capacity allocation starts at schema version `1` when it
  requires a version;
- a changed existing versioned shape or meaning receives the next schema
  version even before product 1.0; and
- the previous reader is removed in the hard cut rather than retained through
  compatibility logic.

Product version and persisted-schema version are different concepts. Current

## Risks And Mitigations

### Retry recalculates its deadline

The retained row may be bypassed by a new ID and moving window. Mitigation:
derive authorization, ID, binding, and deadline once; freeze vectors proving
exact retry reproduction; require new authentication for a new deadline.

### Replay horizon or grace is too long

Steady-state rows may approach capacity. Mitigation: include both windows,
bursts, pending backlog, and cleanup service bounds in the numeric envelope.

### Settlement runs out of stable memory

An admitted pending operation could become permanently uncertain. Mitigation:
reserve and provision worst-case terminal-transition bytes during initial
admission and inject physical reservation failures before mutation.

### Lost late-settlement response becomes immediately unobservable

Mitigation: eligibility is the maximum of replay deadline and terminal time
plus frozen observation grace.

### Corrupt first index entry causes retry storm or unsafe skipping

Mitigation: retain it, stop with typed failed/no-retry status, block later
entries, and require explicit operator investigation.

### Stable index or reservation drift

Mitigation: precompute non-awaiting transitions, validate exact canonical
correspondence, reject malformed import, and never rebuild authority from an
assumption.

### Cleanup changes resource supply

Mitigation: settlement is the sole aggregate transition. Reclamation cannot
write resource totals.
