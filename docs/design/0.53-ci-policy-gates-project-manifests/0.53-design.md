# 0.53 Design: CI Policy Gates And Project Evidence Manifests

## Status

- Status: Tentative
- Line: `0.53.x`
- Theme: CI policy gates and project evidence manifests

This is the tentative next line after 0.52. It builds on stable evidence
envelopes and build provenance by giving CI/GitOps a declarative way to decide
whether saved evidence is acceptable for a project policy. It does not add
deployment locks, signing, provider wrappers, artifact registry import,
controller mutation, topology mutation, adoption import, install authority, or
live deployment verification.

The first implementation slices stay narrow: `0.53.1` adds a single-envelope
policy gate over stable envelope fields and payload schema identity, and
`0.53.2` adds optional build-provenance field rules. `0.53.3` adds project
evidence manifests over existing envelope files.

## Summary

0.51 made selected Canic outputs stable enough for automation:

```text
EvidenceEnvelopeV1
ExitClassV1
InputFingerprintV1
PayloadSchemaRefV1
```

0.52 added stable build provenance:

```text
canic.build_provenance.v1
```

0.53 should answer the next CI question:

```text
Does this saved evidence satisfy this project's declared policy?
```

The line should define passive policy evaluation over existing evidence
artifacts. The first slice lets CI fail on unacceptable envelope state: schema
mismatch, disallowed payload schema, blocked exit class, evidence conflicts,
blocked actions, or missing required evidence. The next slice adds
build-provenance-specific rules for dirty source state, missing `Cargo.lock`
hashes, package metadata mismatches, missing gzip Wasm, and invalid artifact
hashes.

Core idea:

```text
stable evidence + project policy -> passive gate report
```

0.53 is not a deployment execution line. It is a policy/reporting line.

## Motivation

After 0.41-0.50, Canic can model deployment truth, authority evidence,
execution boundaries, artifact promotion, external lifecycle evidence,
deployment-target identity, fleet-role lifecycle, and passive adoption reports.

After 0.51, Canic can emit stable evidence envelopes and compare stable
envelope fields.

After 0.52, Canic can record which source, Cargo inputs, package identity, and
build artifacts produced a Canic build output.

The remaining CI/GitOps gap is policy. Projects need a Canic-owned way to
express and evaluate rules such as:

- only accept `success` or explicitly allowed `success_with_warnings`;
- fail on `evidence_conflict`;
- fail on `blocked_by_policy`;
- require build provenance for deployable roles;
- require clean Git source state;
- require a `Cargo.lock` hash;
- require package metadata `fleet.role` to match the target;
- require a gzip Wasm artifact hash;
- require supplied evidence files to match expected schema identities.

Without this line, every CI pipeline must hand-roll policy against Canic JSON.
That risks brittle scripts depending on command-specific payload internals
instead of stable envelope and provenance contracts.

## Core Rule

0.53 policy gates are passive.

The policy evaluator may read:

- policy files;
- project evidence manifests;
- existing `EvidenceEnvelopeV1` files;
- existing build provenance envelopes;
- existing adoption/deployment-check envelope artifacts;
- local file metadata needed to fingerprint supplied evidence.

The policy evaluator must not:

- mutate deployments;
- install or upgrade Wasm;
- attach topology;
- declare, rename, or attach roles;
- import adoption candidates;
- change controllers;
- acquire deployment locks;
- sign or attest evidence;
- register artifacts into `wasm_store` or any artifact registry;
- run builds;
- crawl live deployments;
- turn policy success into deployment truth;
- make stale evidence fresh.

Allowed writes are limited to explicit report output requested by the operator.

## Proposed First Slices

0.53 starts by defining the passive policy-gate vocabulary and implementing one
minimal evaluator over existing envelope files.

Preferred command:

```text
canic evidence gate --policy <path> --envelope <path>
```

Optional output forms:

```text
canic evidence gate --policy <path> --envelope <path> --format json
canic evidence gate --policy <path> --envelope <path> --format envelope-json
canic evidence gate --policy <path> --envelope <path> --output <path>
```

`--format envelope-json` should wrap a `PolicyGateReportV1` payload in
`EvidenceEnvelopeV1`. The raw `--format json` payload should emit
`PolicyGateReportV1`. Text should remain the default human-readable output.
`--format` controls both stdout and `--output <path>` serialization. The
default format is text. CI should request an explicit structured format:

```text
canic evidence gate --policy <path> --envelope <path> --format json --output gate-report.json
canic evidence gate --policy <path> --envelope <path> --format envelope-json --output gate-envelope.json
```

`--output <path>` should write the requested report artifact before returning
a non-zero process status when practical, so CI can preserve failure
artifacts.

The first evaluator should stay narrow:

- read one policy file;
- read one evidence envelope;
- evaluate the envelope schema;
- evaluate payload schema identity and stability;
- evaluate the evaluated envelope exit class;
- evaluate summary conflicts, blocked actions, missing-or-stale evidence, and
  warnings;
- evaluate required input fingerprints by kind and schema against
  `EvidenceEnvelopeV1.inputs`;
- emit stable `PolicyGateReportV1`;
- map the gate result to stable `ExitClassV1`.

The gate report must distinguish the evaluated evidence from the policy
decision. For example, an evaluated envelope may have
`exit_class = "success_with_warnings"` while policy only allows `"success"`.
In that case the report should preserve:

```text
evaluated_envelope_exit_class = "success_with_warnings"
policy_status = "failed"
gate_exit_class = "blocked_by_policy"
```

0.53.1 does not traverse stable `canic.build_provenance.v1` payload fields
beyond checking the payload schema identity and stability. 0.53.2 adds
opt-in build-provenance field rules while preserving the single-envelope
boundary.

Do not require project-wide manifests in 0.53.0. A manifest can be added after
single-envelope policy semantics are clear.

## Policy File

Policy gates use a strict allow-list policy shape:

```toml
schema_version = 1

[envelope]
required_schema = "canic.evidence_envelope.v1"
allowed_payload_schemas = ["canic.build_provenance.v1"]
allowed_payload_stability = ["stable"]

[exit_class]
allowed = ["success"]

[summary]
fail_on_evidence_conflicts = true
fail_on_blocked_actions = true
allow_missing_or_stale_evidence = false

[[required_input]]
kind = "canic_config"
schema = "canic.config.toml"
```

This shape is intentionally declarative. It should not embed shell commands,
provider-specific workflow semantics, or deployment-plan logic.

Required policy sections are:

- `schema_version`;
- `[envelope]`;
- `[exit_class]`.

Optional envelope/summary policy sections are:

- `[summary]`;
- `[[required_input]]`.

Optional build-provenance policy sections are:

- `[build_provenance]`.

Omitted optional sections are not evaluated. Empty allow-lists are invalid
input.

`CiPolicyV1` parsing should be strict. Unknown fields, duplicate sections,
unsupported schema versions, and unknown rule names fail with `invalid_input`.
Policy files are CI surfaces, so silently ignored keys are unsafe.

The smallest valid policy is:

```toml
schema_version = 1

[envelope]
required_schema = "canic.evidence_envelope.v1"

[exit_class]
allowed = ["success"]
```

Recommended CI policies should opt into the summary checks they care about:

```toml
[summary]
fail_on_evidence_conflicts = true
fail_on_blocked_actions = true
allow_missing_or_stale_evidence = false
```

0.53 should not support separate `allow_success_with_warnings` or `deny`
keys. A project that accepts warnings should write:

```toml
[exit_class]
allowed = ["success", "success_with_warnings"]
```

`[[required_input]]` rules are evaluated against `EvidenceEnvelopeV1.inputs`.
A rule passes when at least one input fingerprint matches the requested `kind`
and, when supplied, the requested schema identity. In 0.53.0 the policy
`schema` string matches `InputFingerprintV1.schema.id`. Missing required input
and schema mismatch failures map to `missing_required_evidence`.

Build-provenance rules use a dedicated optional table:

```toml
[build_provenance]
require_clean_source = true
require_cargo_lock = true
require_wasm_gzip = true
require_sha256 = true
require_package_identity_matches_target = true
```

## Policy DTOs

Stable 0.53.0 payloads:

```text
CiPolicyV1
PolicyGateReportV1
PolicyFindingV1
PolicyFindingSeverityV1
PolicyRequirementV1
PolicyEvaluationStatusV1
```

`CiPolicyV1`, `PolicyGateReportV1`, `ProjectEvidenceManifestV1`, and
`ProjectEvidenceGateReportV1` are stable once manifest support lands.

`CiPolicyV1` should model the rules Canic understands.

`PolicyGateReportV1` should report:

- policy schema/version;
- policy file fingerprint;
- evaluated envelope fingerprint;
- evaluated envelope exit class;
- evaluated payload schema;
- evaluated target identity;
- policy status;
- gate exit class;
- findings;
- evaluated warnings, blocked actions, missing evidence, and evidence
  conflicts when policy inspects them.

Suggested report fields:

```text
PolicyGateReportV1 {
  policy_schema_version,
  policy_file_fingerprint,
  evaluated_envelope_fingerprint,
  evaluated_envelope_exit_class,
  evaluated_payload_schema,
  evaluated_target,
  policy_status,
  gate_exit_class,
  requirements,
  findings,
}
```

`PolicyFindingV1` should be structured enough for CI:

```text
PolicyFindingV1 {
  code: string,
  severity: "info" | "warning" | "error",
  message: string,
  requirement_id: string?,
  subject: string?,
  expected: serde_json::Value?,
  actual: serde_json::Value?,
  evidence_path: string?,
  target: EvidenceTargetV1?,
  related_input: string?,
}
```

Finding codes, not prose, should be the stable automation handle.

Examples:

```text
code = "policy.exit_class.disallowed"
expected = ["success"]
actual = "success_with_warnings"

code = "policy.payload_schema.disallowed"
expected = ["canic.build_provenance.v1"]
actual = "canic.adoption_report.v1"
```

Every evaluated `PolicyRequirementV1` should have a stable `requirement_id`,
even when generated from the TOML path:

```text
envelope.required_schema
envelope.allowed_payload_schemas
envelope.allowed_payload_stability
exit_class.allowed
summary.fail_on_evidence_conflicts
summary.fail_on_blocked_actions
summary.allow_missing_or_stale_evidence
required_input.<index>
```

Findings should reference these IDs so manifest-era reports can trace failures
back to the exact rule and evidence entry.

## Policy Gate Envelope Output

`--format envelope-json` should produce a new `EvidenceEnvelopeV1` whose
payload is the policy report:

```text
payload_schema.id = "canic.policy_gate_report.v1"
payload_schema.stability = "stable"
exit_class = gate_exit_class
inputs = [
  fingerprint(policy.toml),
  fingerprint(evaluated-envelope.json),
]
payload = PolicyGateReportV1
```

This creates a second evidence artifact about the evaluated envelope; it does
not replace or rewrite the evaluated envelope.

The gate output target should make policy evaluation first-class. 0.53.0
should extend `EvidenceTargetKindV1` with `policy_gate` for policy gate
envelopes:

```text
target.kind = "policy_gate"
target.profile = null
target.fleet/deployment/role = copied only when unambiguous from the evaluated envelope
```

`PolicyGateReportV1.evaluated_target` must preserve the evaluated envelope
target exactly. Do not use `unknown` for policy gate output unless the
evaluated envelope truly has no target.

## Project Evidence Manifest

0.53 adds a project evidence manifest after the single-envelope gate lands.

Tentative shape:

```toml
schema_version = 1

[project]
name = "demo"
root = "."

[[evidence]]
kind = "build_provenance"
path = "artifacts/canic/app-build-provenance.json"
required = true
payload_schema = "canic.build_provenance.v1"

[evidence.target]
fleet = "demo"
role = "app"

[[evidence]]
kind = "deployment_check"
path = "artifacts/canic/deployment-check-envelope.json"
required = true
payload_schema = "canic.deployment_check.v1"

[evidence.target]
deployment = "demo-staging"
```

Command:

```text
canic evidence gate --policy <path> --manifest <path>
```

Manifest rules:

- paths are resolved relative to the manifest path or explicit project root;
- missing required evidence is a policy failure;
- optional evidence can produce warnings;
- manifest entries are fingerprints and expectations, not deployment plans;
- manifest entries should carry expected target identity in a target table
  shaped like `EvidenceTargetV1`;
- manifests do not acquire locks or authorize mutation.
- manifest gates emit `ProjectEvidenceGateReportV1`;
- `--format envelope-json` wraps `ProjectEvidenceGateReportV1` with
  `payload_schema.id = "canic.project_evidence_gate_report.v1"`.

## Exit Classes

Policy evaluation should reuse `ExitClassV1`.

Suggested mapping:

| Gate result | Exit class | Process exit |
| --- | --- | --- |
| all requirements pass | `success` | 0 |
| requirements pass with allowed warnings | `success_with_warnings` | 0 by default |
| policy denies an envelope state | `blocked_by_policy` | non-zero |
| evaluated evidence contains conflicts | `evidence_conflict` | non-zero |
| required evidence is missing | `missing_required_evidence` | non-zero |
| policy or evidence file is invalid | `invalid_input` | non-zero |
| evaluator cannot complete an unexpected operation | `execution_failed` | non-zero |
| Canic bug/unexpected failure | `internal_error` | non-zero |

If several findings exist, exit-class precedence should follow the shared 0.51
precedence. 0.53 must not invent a separate taxonomy.

The process exit code follows the gate report's `gate_exit_class`, not the
evaluated envelope's `exit_class`.

The gate exit class is computed from the policy evaluation, not copied from
the evaluated envelope. The evaluated envelope exit class remains recorded in
`PolicyGateReportV1.evaluated_envelope_exit_class`.

Example:

```text
evaluated envelope: success_with_warnings
policy: allowed = ["success"]
gate_exit_class: blocked_by_policy
process exit: non-zero
```

If all error-level policy requirements pass but the evaluated envelope
contains warnings or policy-allowed `missing_or_stale_evidence`, the gate exit
class is `success_with_warnings`. A policy that accepts that result must list
`success_with_warnings` in `[exit_class].allowed`; otherwise the gate fails
with `blocked_by_policy`.

Error classification should be deterministic:

| Condition | Exit class |
| --- | --- |
| missing `--policy` file | `invalid_input` |
| missing explicit `--envelope` file | `invalid_input` |
| invalid policy TOML | `invalid_input` |
| invalid envelope JSON | `invalid_input` |
| unsupported policy schema | `invalid_input` |
| unknown policy key | `invalid_input` |
| policy failure | `blocked_by_policy`, `evidence_conflict`, or `missing_required_evidence` |
| missing or schema-mismatched required input | `missing_required_evidence` |
| filesystem read error for user-supplied policy/envelope path | `invalid_input` |
| missing required manifest evidence | `missing_required_evidence` |

Use `execution_failed` only when Canic attempted an operation and the operation
failed unexpectedly. Passive gate path and parse problems are invalid input.

`allow_missing_or_stale_evidence` applies only to
`EvidenceSummaryV1.missing_or_stale_evidence` entries. Those entries are
reportable evidence gaps and may remain warning-level findings when policy
allows them. Missing `[[required_input]]` matches are required-evidence
failures and do not depend on that summary setting.

## Relationship To 0.51 And 0.52

0.53 consumes, but does not replace, the 0.51 and 0.52 contracts.

Rules:

- `EvidenceEnvelopeV1` remains the stable outer automation contract.
- `canic.build_provenance.v1` remains the stable build provenance payload.
- 0.53.0 policy gates should branch on stable envelope fields, payload schema
  identity/stability, input fingerprints, summary fields, and structured
  finding codes.
- Later policy gates can branch on stable build provenance fields.
- Policy gates must not depend on experimental or internal nested payload
  fields unless the policy explicitly opts into that instability.
- Existing evidence emitters must not change output shape just because policy
  gates exist.

## CI/GitOps Use Cases

0.53 should support:

- failing CI when a build provenance envelope reports dirty source state;
- failing CI when `Cargo.lock` evidence is absent;
- failing CI when package metadata `fleet.role` does not match the target;
- requiring gzip Wasm SHA-256 evidence before deployment promotion;
- requiring passive deployment-check envelopes to have acceptable exit classes;
- collecting a policy report as a CI artifact;
- comparing policy results between baseline and current runs through existing
  evidence comparison where practical.

## Non-Goals

0.53 does not implement:

- deployment locks;
- distributed lock storage;
- signing or attestations;
- external key management;
- provider-specific GitHub/GitLab wrappers;
- artifact registry import;
- `wasm_store` retention or garbage collection;
- active adoption/import;
- controller mutation;
- topology mutation;
- install or upgrade authority;
- broad live verification;
- project release orchestration;
- teardown.

These may become later lines, but policy gates should land first so later
active surfaces can consume a tested policy result.

## Patch Plan

Tentative policy implementation sequence. Exact patch numbers may shift if
other 0.53.x cleanup slices land between policy slices.

- Implemented policy slice: single-envelope policy gate with `CiPolicyV1`,
  stable `PolicyGateReportV1`, strict policy parsing, and
  `canic evidence gate --policy <path> --envelope <path>` over stable envelope
  fields only.
- Implemented policy slice: build-provenance policy rules for clean source,
  Cargo lock, package identity, gzip Wasm, and SHA-256 evidence.
- Implemented policy slice: project evidence manifest model and
  `canic evidence gate --policy <path> --manifest <path>`.
- Implemented policy slice: docs and CI examples for policy files, manifest files,
  and expected pass/fail behavior.
- Final policy slice: closeout audit.

This plan should shrink rather than grow. Deployment locks, signing, provider
wrappers, registry import, and mutation authority should remain out of scope.

## Required Tests

Minimum tests before closing 0.53.0:

- policy file parser accepts a minimal allow-list policy;
- minimal policy without optional sections passes a valid success envelope;
- empty `exit_class.allowed` is `invalid_input`;
- missing required `[exit_class]` is `invalid_input`;
- missing required `[envelope]` is `invalid_input`;
- policy allow-list accepts only `success`;
- policy rejects `success_with_warnings` when not allowed;
- policy accepts `success_with_warnings` when explicitly allowed;
- policy file parser rejects unknown/unsupported rule keys;
- unsupported policy schema version fails with `invalid_input`;
- `canic evidence gate --policy <path> --envelope <path>` reads an existing
  envelope;
- valid evidence produces `success`;
- evaluated envelope schema mismatch produces `blocked_by_policy`;
- payload schema mismatch produces `blocked_by_policy`;
- payload stability disallowed produces `blocked_by_policy`;
- disallowed envelope exit class produces `blocked_by_policy`;
- `evidence_conflict` in the evaluated envelope remains a failure;
- summary `evidence_conflicts` fail with `evidence_conflict`;
- summary `blocked_actions` fail with `blocked_by_policy`;
- summary `missing_or_stale_evidence` obeys the policy setting;
- required input passes when a matching input exists;
- required input fails with `missing_required_evidence` when absent;
- required input schema mismatch fails with `missing_required_evidence`;
- warnings produce `success_with_warnings` when allowed;
- warnings produce `blocked_by_policy` when not allowed;
- missing policy file produces `invalid_input`;
- missing explicit envelope file produces `invalid_input`;
- invalid policy file produces `invalid_input`;
- invalid envelope file produces `invalid_input`;
- gate report includes evaluated exit class and gate exit class;
- gate envelope inputs include policy file and evaluated envelope fingerprints;
- default text output is human-readable;
- raw `--format json` emits `PolicyGateReportV1` with structured finding
  codes, requirement IDs, expected values, and actual values;
- `--format envelope-json` wraps `PolicyGateReportV1`;
- `--output` uses the selected `--format` serialization;
- `--output` writes the report even when policy fails;
- gate envelope target kind is `policy_gate`;
- gate report preserves the evaluated target exactly;
- policy gate does not rewrite the evaluated envelope;
- policy gate does not rewrite the evaluated envelope exit class;
- policy gate does not run build, deploy, or discovery helpers;
- policy gate does not mutate the policy file, evidence file, config,
  topology, artifacts, deployment truth, controllers, package manifests, or
  canister lifecycle state.

Additional tests before closing build-provenance policy rules:

- `[build_provenance]` parser accepts known rule keys and rejects unknown keys;
- empty `[build_provenance]` tables are invalid;
- clean source evidence passes `require_clean_source`;
- dirty or unknown source evidence fails `require_clean_source`;
- missing `Cargo.lock` evidence fails with `missing_required_evidence`;
- missing gzip Wasm evidence fails with `missing_required_evidence`;
- invalid artifact SHA-256 evidence fails with `missing_required_evidence`;
- package metadata `fleet.role` mismatch fails with `blocked_by_policy`;
- non-build-provenance payloads fail build-provenance rules without silently
  passing;
- invalid build-provenance payload shape fails clearly.

Additional tests for manifest support:

- missing required manifest evidence fails;
- optional missing manifest evidence warns;
- relative evidence paths resolve from the manifest root;
- manifest target expectations are checked against envelope target fields;
- duplicate evidence entries are rejected or reported deterministically.

## Acceptance Criteria

0.53.0 is acceptable when:

- `CiPolicyV1` exists and parses strictly;
- `PolicyGateReportV1` exists as a stable payload schema;
- `canic evidence gate --policy <path> --envelope <path>` exists;
- the command reads exactly one policy file and one existing
  `EvidenceEnvelopeV1`;
- the command evaluates only stable envelope fields and payload schema
  identity/stability;
- `[[required_input]]` checks evaluate only `EvidenceEnvelopeV1.inputs` kind
  and schema data;
- required policy sections, optional policy sections, omitted-section behavior,
  and empty allow-list behavior are tested;
- policy failures map to `ExitClassV1`;
- the gate report distinguishes evaluated envelope exit class from gate exit
  class;
- text output is tested;
- raw `--format json` emits `PolicyGateReportV1`;
- `--format envelope-json` wraps `PolicyGateReportV1` in
  `EvidenceEnvelopeV1`;
- `--format` controls both stdout and `--output <path>` serialization;
- `--output <path>` writes only the requested report artifact;
- policy gate envelope output uses `target.kind = "policy_gate"`;
- missing or invalid policy and envelope files produce deterministic
  `invalid_input` behavior;
- the command is read-only except explicit report output;
- no config, topology, artifacts, deployment truth, controllers, package
  manifests, evidence inputs, or canister lifecycle state are mutated;
- docs explain that policy success is not deployment truth and does not
  authorize mutation.

The full 0.53 line is acceptable when:

- project policy can gate saved envelope evidence;
- build provenance policy checks are implemented;
- a project evidence manifest can group expected evidence files;
- docs show a minimal CI pipeline using build provenance, deployment-check
  envelopes, and policy gates;
- no active deployment, install, controller, topology, registry, signing, or
  lock authority has been added.

## Recommended Closeout Wording

0.53 adds passive CI policy gates over Canic evidence. A project can declare
which envelope states, payload schemas, build provenance fields, and evidence
files are acceptable, then run `canic evidence gate` to produce a structured
policy report. Policy gates consume stable 0.51 envelopes and stable 0.52 build
provenance; they do not mutate
deployments, install Wasm, attach topology, change controllers, register
artifacts, acquire locks, sign evidence, or turn policy success into
deployment truth.
