# Canic Delegated Auth V2 0.29 Status

- **Design:** [`0.29-design.md`](0.29-design.md)
- **Scope:** implementation state, verification, and known blockers for the `0.29`
  Delegated Auth V2 hard-cut design

## Completed

- Added V2 wire/data structures for `DelegationCertV2`, `DelegationProofV2`,
  `DelegatedTokenClaimsV2`, and `DelegatedTokenV2`.
- Added supporting V2 wire/data structures for root key authority, root key
  certificates, root key sets, role/principal audiences, and shard key binding.
- Added `CanicAuthCanonicalV2` hashing for delegation certificates, token
  claims, root-key certificate payloads, and role hashes.
- Added canonical encoding tests for deterministic sorting/deduplication,
  noncanonical role/scope rejection, domain separation, and root-key certificate
  payload signing boundaries.

## Known Boundaries

- V2 is a bearer-token system. Replay protection is bounded by token TTL only.
- Verifier role configuration is trusted. A verifier with the wrong local role is
  compromised for delegated-auth purposes.
- `RootKeyAuthorityV2` is a critical trust anchor. Compromise is equivalent to
  compromise of the root key hierarchy.
- V2 has no core revocation model. Certificates and tokens remain valid until
  expiry unless a future explicit revocation mechanism is designed.
