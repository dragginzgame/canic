# Canic Delegated Auth V2

- **Version:** `0.29`
- **Status:** `Proposed`
- **Author:** `Canic Core`
- **Scope:** hard-cut delegated authentication replacement for proof-cache based verification
- **Primary property:** delegated tokens are self-validating at every verifier
- **Implementation status:** [`0.29-status.md`](0.29-status.md)

## 1. Hard-Cut Contract

Delegated Auth V2 defines the token, proof, certificate, key, audience, and TTL rules required for a verifier to authenticate a delegated request without consulting root, registry, placement state, or a local proof cache.

V2 verification depends only on:

- the delegated token supplied with the request
- the delegation proof embedded in that token
- the verifier's configured root identity
- the verifier's trusted root key material or a valid embedded root-key fallback
- the shard public key certified by root inside the proof
- the verifier's local role and principal identity
- IC canister time

V2 verification does not depend on:

- proof distribution
- proof history replication
- verifier creation time
- project placement order
- registry snapshots
- topology propagation
- whether root has pushed any state to the verifier

## 2. Trust Model

### 2.1 Root Authority

Root is the only authority allowed to issue a `DelegationCertV2`.

Root signs:

```rust
root_sig = sign_root(canonical_bytes(DelegationCertV2))
```

Root certifies:

- the shard principal that may sign delegated tokens
- the shard public key that must verify token signatures
- the audience where the certificate is valid
- the role hash binding for role-targeted certificates
- the certificate validity window
- the maximum token TTL allowed under the certificate
- the scopes delegated to the shard signer

Root must enforce policy before signing:

```rust
require(cert.expires_at > cert.issued_at);
require(cert.expires_at - cert.issued_at <= policy.max_cert_ttl_secs);
require(cert.max_token_ttl_secs > 0);
require(cert.max_token_ttl_secs <= policy.max_token_ttl_secs);
require(cert.max_token_ttl_secs <= cert.expires_at - cert.issued_at);
require(cert.root_pid == self_root_pid);
require(cert.aud.is_valid_v2());
require(cert.verifier_role_hash == expected_role_hash_for(cert.aud));
require(hash_public_key(cert.shard_public_key_sec1) == cert.shard_key_hash);
require(cert.shard_key_hash == derive_expected_shard_key_hash(
    cert.root_pid,
    cert.shard_pid,
    cert.shard_key_id,
    cert.shard_key_binding,
));
```

Root must not issue certificates that exceed the configured certificate TTL. This is the safety boundary for the V2 no-revocation model.

### 2.2 Shard Signer Authority

A shard signer may mint delegated tokens only under a valid root-signed certificate.

The shard signs:

```rust
shard_sig = sign_shard(canonical_bytes(DelegatedTokenClaimsV2))
```

The shard is not trusted because of `shard_pid` alone. The shard is trusted only when:

- root signed the certificate
- the certificate binds `shard_pid`
- the certificate binds `shard_key_id`
- the certificate binds `shard_public_key_sec1`
- the bound public key verifies the token claims signature
- the shard public key hash matches the expected derivation for that shard identity

### 2.3 Verifier Local State

A verifier stores only stable trust configuration, not proof state:

```rust
pub struct DelegatedAuthVerifierConfigV2 {
    pub expected_root_pid: Principal,
    pub local_principal: Principal,
    pub local_role: Option<CanisterRole>,
    pub root_trust: RootTrustAnchorV2,
}

pub struct RootTrustAnchorV2 {
    pub root_pid: Principal,
    pub trusted_root_keys: RootKeySetV2,
    pub key_authority: Option<RootKeyAuthorityV2>,
}

pub struct RootKeySetV2 {
    pub keys: Vec<RootPublicKeyV2>,
}

pub struct RootPublicKeyV2 {
    pub root_pid: Principal,
    pub key_id: KeyId,
    pub alg: SignatureAlgorithmV2,
    pub public_key_sec1: Vec<u8>,
    pub key_hash: [u8; 32],
    pub not_before: u64,
    pub not_after: Option<u64>,
}
```

`RootKeySetV2` is shared trust state. It is not a proof cache. It is allowed to be cached locally, but a verifier must not require proof replication or proof lookup.

Verifier role configuration is a security boundary. A canister configured with an incorrect `local_role` is considered compromised for delegated-auth purposes and may accept tokens outside its intended authorization domain. V2 does not query registry to prove that the local canister has the configured role.

`RootKeyAuthorityV2` is a root trust anchor. It must be provisioned at verifier install or upgrade through trusted configuration, kept separate from online certificate-signing keys, and treated as an offline or rarely used authority. Compromise of this authority allows an attacker to introduce new root signing keys that can issue valid certificates.

### 3.2 Canonical Encoding

All signed structures use `CanicAuthCanonicalV2`, not implementation-default Candid or serde output.

Rules:

```rust
pub enum CanonicalDomainV2 {
    DelegationCert = 1,
    DelegatedTokenClaims = 2,
    RootKeyCertificate = 3,
}
```

Encoding:

- prepend the ASCII domain separator `CANIC-AUTH-V2\0`
- encode the `CanonicalDomainV2` tag as one `u8`
- encode struct fields in the exact order listed in this document
- encode `u16` as 2-byte big-endian
- encode `u64` as 8-byte big-endian
- encode `[u8; 32]` as exactly 32 bytes
- encode `[u8; 16]` as exactly 16 bytes
- encode `Vec<u8>` as `u32_be length || raw bytes`
- encode `String` as `u32_be length || UTF-8 bytes`
- encode `Principal` as `u32_be length || Principal::as_slice()`
- encode `Option<T>` as `0x00` for `None` or `0x01 || encode(T)` for `Some`
- encode enum variants as `u8 variant_tag || variant_payload`
- encode `Vec<T>` as `u32_be length || item_0 || ... || item_n`

Canonicalization before encoding:

- role names must be non-empty ASCII strings matching `[a-z0-9_:-]+`
- scope strings must be non-empty ASCII strings matching `[a-z0-9_:-]+`
- role, principal, and scope vectors must be sorted by canonical encoded bytes and deduplicated
- strings are compared byte-for-byte after validation
- no Unicode normalization is performed

Any producer that cannot produce this exact encoding must not sign V2 objects. Any verifier that cannot reproduce this exact encoding must reject V2 objects.


`Any` is not part of V2.

### 3.4 Root Key Fallback

```rust
pub struct RootKeyAuthorityV2 {
    pub authority_key_id: KeyId,
    pub authority_alg: SignatureAlgorithmV2,
    pub authority_public_key_sec1: Vec<u8>,
    pub authority_key_hash: [u8; 32],
}

pub struct RootKeyCertificateV2 {
    pub root_pid: Principal,
    pub key_id: KeyId,
    pub alg: SignatureAlgorithmV2,
    pub public_key_sec1: Vec<u8>,
    pub key_hash: [u8; 32],
    pub not_before: UnixSeconds,
    pub not_after: Option<UnixSeconds>,
    pub authority_sig: Vec<u8>,
}
```

An embedded root public key is never trusted merely because it verifies the certificate. It is accepted only if it matches one of:

- an exact trusted key entry already present in `RootKeySetV2`
- a valid `RootKeyCertificateV2` signed by the configured `RootKeyAuthorityV2`

For an unknown `root_key_id`, `root_key_cert` is required. `root_public_key_sec1` alone is not sufficient.

This makes root key rollout more forgiving than proof rollout without turning the embedded key into a self-signed trust anchor.

### 3.5 Shard Key Binding

```rust
pub enum ShardKeyBindingV2 {
    IcThresholdEcdsa {
        key_name_hash: [u8; 32],
        derivation_path_hash: [u8; 32],
    },
}
```

The verifier must be able to recompute:

```rust
expected_shard_key_hash = derive_expected_shard_key_hash(
    cert.root_pid,
    cert.shard_pid,
    cert.shard_key_id,
    cert.shard_key_binding,
);
```

and require:

```rust
require(expected_shard_key_hash == cert.shard_key_hash);
require(hash_public_key(cert.shard_public_key_sec1) == cert.shard_key_hash);
```

If Canic cannot define deterministic shard key derivation for a given key family, that key family is not valid for V2 delegated auth.

### 3.6 Delegation Certificate

```rust
pub struct DelegationCertV2 {
    pub version: u16,
    pub root_pid: Principal,
    pub root_key_id: KeyId,
    pub root_key_hash: [u8; 32],
    pub alg: SignatureAlgorithmV2,

    pub shard_pid: Principal,
    pub shard_key_id: KeyId,
    pub shard_public_key_sec1: Vec<u8>,
    pub shard_key_hash: [u8; 32],
    pub shard_key_binding: ShardKeyBindingV2,

    pub issued_at: UnixSeconds,
    pub expires_at: UnixSeconds,
    pub max_token_ttl_secs: u64,

    pub scopes: Vec<Scope>,
    pub aud: DelegationAudienceV2,
    pub verifier_role_hash: Option<[u8; 32]>, // hash of the canonical role string
}
```

`verifier_role_hash` binds role-targeted certificates to the canonical role string.

Rules:

```rust
match cert.aud {
    DelegationAudienceV2::Principals(_) => {
        require(cert.verifier_role_hash.is_none());
    }
    DelegationAudienceV2::Roles(ref roles) => {
        require(roles.len() == 1);
        require(cert.verifier_role_hash == Some(role_hash(&roles[0])));
    }
    DelegationAudienceV2::RolesOrPrincipals { ref roles, .. } => {
        require(roles.len() <= 1);
        if roles.len() == 1 {
            require(cert.verifier_role_hash == Some(role_hash(&roles[0])));
        } else {
            require(cert.verifier_role_hash.is_none());
        }
    }
}
```

V2 initially allows at most one role per certificate. Multi-principal audiences are allowed. Multi-role certificates are rejected to keep `verifier_role_hash` unambiguous.

### 3.7 Delegation Proof

```rust
pub struct DelegationProofV2 {
    pub cert: DelegationCertV2,
    pub root_sig: Vec<u8>,
    pub root_public_key_sec1: Option<Vec<u8>>, // fallback transport for root key bytes
    pub root_key_cert: Option<RootKeyCertificateV2>,
}
```

`root_public_key_sec1` is a fallback transport for root key bytes.

`root_key_cert` is required when the verifier does not already know `cert.root_key_id`.

### 3.8 Token Claims

```rust
pub struct DelegatedTokenClaimsV2 {
    pub version: u16,
    pub subject: Principal,
    pub issuer_shard_pid: Principal,
    pub cert_hash: [u8; 32],
    pub issued_at: UnixSeconds,
    pub expires_at: UnixSeconds,
    pub aud: DelegationAudienceV2,
    pub scopes: Vec<Scope>,
    pub nonce: [u8; 16],
}
```

`issuer_shard_pid` duplicates `proof.cert.shard_pid` intentionally. Verification requires equality and rejects mismatches. The field exists to make token claims self-describing for logs, traces, and typed failure reporting; it does not create separate authority.

`nonce` is informational entropy only. Core V2 verification does not provide replay protection and does not track nonce use.

V2 is a bearer-token system. A valid token may be replayed until `claims.expires_at`; the core replay bound is TTL only.

If an application requires replay protection, it must define a separate nonce store with:

- storage owner
- retention window
- maximum entries
- eviction policy
- failure behavior when storage is full

That replay system is outside core Delegated Auth V2.

### 3.9 Delegated Token

```rust
pub struct DelegatedTokenV2 {
    pub claims: DelegatedTokenClaimsV2,
    pub proof: DelegationProofV2,
    pub shard_sig: Vec<u8>,
}
```

The token is self-contained. A verifier must not require `proof` to exist in local state.

## 4. Root Issuance Flow

### 4.1 Certificate Issuance

```rust
fn issue_cert_v2(input: IssueDelegationCertInputV2) -> Result<DelegationProofV2, AuthError> {
    let now = ic_time_secs();

    let cert = DelegationCertV2 {
        version: 2,
        root_pid: self_root_pid,
        root_key_id: active_root_key.key_id,
        root_key_hash: active_root_key.key_hash,
        alg: active_root_key.alg,
        shard_pid: input.shard_pid,
        shard_key_id: input.shard_key_id,
        shard_public_key_sec1: input.shard_public_key_sec1,
        shard_key_hash: hash_public_key(input.shard_public_key_sec1),
        shard_key_binding: input.shard_key_binding,
        issued_at: now,
        expires_at: checked_add(now, input.cert_ttl_secs)?,
        max_token_ttl_secs: input.max_token_ttl_secs,
        scopes: canonical_scopes(input.scopes),
        aud: canonical_audience(input.aud),
        verifier_role_hash: expected_role_hash_for(input.aud),
    };

    require(cert.expires_at - cert.issued_at <= policy.max_cert_ttl_secs);
    require(cert.max_token_ttl_secs <= policy.max_token_ttl_secs);
    require(cert.max_token_ttl_secs <= cert.expires_at - cert.issued_at);
    require(cert.aud.is_valid_v2());
    require(cert.verifier_role_hash == expected_role_hash_for(cert.aud));
    require(cert.shard_key_hash == derive_expected_shard_key_hash(
        cert.root_pid,
        cert.shard_pid,
        cert.shard_key_id,
        cert.shard_key_binding,
    ));

    let root_sig = sign_root(canonical_bytes(&cert));

    Ok(DelegationProofV2 {
        cert,
        root_sig,
        root_public_key_sec1: Some(active_root_key.public_key_sec1),
        root_key_cert: active_root_key.certificate_for_transport(),
    })
}
```

Certificate issuance does not push proof state to verifiers.

### 4.2 Token Minting

```rust
fn mint_token_v2(input: MintDelegatedTokenInputV2) -> Result<DelegatedTokenV2, AuthError> {
    let now = ic_time_secs();
    let cert = &input.proof.cert;

    require(now >= cert.issued_at);
    require(now < cert.expires_at);
    require(input.token_ttl_secs > 0);
    require(input.token_ttl_secs <= cert.max_token_ttl_secs);

    let claims = DelegatedTokenClaimsV2 {
        version: 2,
        subject: input.subject,
        issuer_shard_pid: cert.shard_pid,
        cert_hash: hash_canonical(cert),
        issued_at: now,
        expires_at: checked_add(now, input.token_ttl_secs)?,
        aud: canonical_audience(input.aud),
        scopes: canonical_scopes(input.scopes),
        nonce: random_128_bits(),
    };

    require(claims.expires_at <= cert.expires_at);
    require(audience_subset(&claims.aud, &cert.aud));
    require(scope_subset(&claims.scopes, &cert.scopes));

    let shard_sig = sign_shard(canonical_bytes(&claims));

    Ok(DelegatedTokenV2 {
        claims,
        proof: input.proof,
        shard_sig,
    })
}
```

Token minting does not depend on registry topology. A token can target a role or principal that exists later, provided the final verifier's identity satisfies the signed audience.

## 5. Verification Algorithm

```rust
fn verify_delegated_token_v2(
    token: DelegatedTokenV2,
    cfg: DelegatedAuthVerifierConfigV2,
    now: UnixSeconds,
    required_scopes: &[Scope],
) -> Result<VerifiedDelegationV2, AuthError> {
    let cert = &token.proof.cert;
    let claims = &token.claims;

    // Version and root identity.
    require(cert.version == 2);
    require(claims.version == 2);
    require(cert.root_pid == cfg.expected_root_pid);
    require(cfg.root_trust.root_pid == cfg.expected_root_pid);

    // Certificate shape.
    require(cert.expires_at > cert.issued_at);
    require(cert.max_token_ttl_secs > 0);
    require(cert.aud.is_valid_v2());
    require(cert.verifier_role_hash == expected_role_hash_for(cert.aud));
    require(hash_canonical(cert) == claims.cert_hash);

    // Root key resolution.
    let root_key = resolve_root_key(
        cert.root_pid,
        cert.root_key_id,
        cert.root_key_hash,
        cert.alg,
        token.proof.root_public_key_sec1,
        token.proof.root_key_cert,
        cfg.root_trust,
        now,
    )?;

    require(root_key.root_pid == cfg.expected_root_pid);
    require(root_key.key_id == cert.root_key_id);
    require(root_key.key_hash == cert.root_key_hash);
    require(root_key.alg == cert.alg);

    // Root signature.
    verify_signature(
        root_key.public_key_sec1,
        canonical_bytes(cert),
        token.proof.root_sig,
        cert.alg,
    )?;

    // Shard key binding.
    require(hash_public_key(cert.shard_public_key_sec1) == cert.shard_key_hash);
    require(cert.shard_key_hash == derive_expected_shard_key_hash(
        cert.root_pid,
        cert.shard_pid,
        cert.shard_key_id,
        cert.shard_key_binding,
    ));

    // Time bounds.
    require(now >= cert.issued_at);
    require(now < cert.expires_at);
    require(claims.expires_at > claims.issued_at);
    require(duration_lte(
        claims.issued_at,
        claims.expires_at,
        cert.max_token_ttl_secs,
    ));
    require(claims.issued_at >= cert.issued_at);
    require(claims.expires_at <= cert.expires_at);
    require(now >= claims.issued_at);
    require(now < claims.expires_at);

    // Claims authority.
    require(claims.issuer_shard_pid == cert.shard_pid);
    verify_signature(
        cert.shard_public_key_sec1,
        canonical_bytes(claims),
        token.shard_sig,
        cert.alg,
    )?;

    // Audience and role hash.
    require(audience_subset(&claims.aud, &cert.aud));
    require(verifier_is_in_audience(&cfg, &claims.aud));
    require(verifier_is_in_audience(&cfg, &cert.aud));

    if audience_uses_role(&claims.aud) || audience_uses_role(&cert.aud) {
        let local_role = cfg.local_role.ok_or(AuthError::MissingLocalRole)?;
        require(cert.verifier_role_hash == Some(role_hash(&local_role)));
    }

    // Scope.
    require(scope_subset(&claims.scopes, &cert.scopes));
    require(scope_subset(required_scopes, &claims.scopes));

    Ok(VerifiedDelegationV2 {
        subject: claims.subject,
        issuer_shard_pid: claims.issuer_shard_pid,
        scopes: claims.scopes,
        cert_hash: claims.cert_hash,
    })
}
```

No step performs a local proof lookup.

No step checks whether root previously pushed this proof to the verifier.

### 5.1 Root Key Resolution

```rust
fn resolve_root_key(
    root_pid: Principal,
    key_id: KeyId,
    key_hash: [u8; 32],
    alg: SignatureAlgorithmV2,
    embedded_key: Option<Vec<u8>>,
    embedded_key_cert: Option<RootKeyCertificateV2>,
    trust: RootTrustAnchorV2,
    now: UnixSeconds,
) -> Result<RootPublicKeyV2, AuthError> {
    require(root_pid == trust.root_pid);

    if let Some(key) = trust.trusted_root_keys.find(root_pid, key_id, key_hash, alg) {
        return Ok(key);
    }

    if let Some(key_bytes) = embedded_key {
        require(hash_public_key(&key_bytes) == key_hash);

        let key_cert = embedded_key_cert.ok_or(AuthError::UnknownRootKey)?;
        let authority = trust.key_authority.ok_or(AuthError::UnknownRootKey)?;

        require(key_cert.root_pid == root_pid);
        require(key_cert.key_id == key_id);
        require(key_cert.alg == alg);
        require(key_cert.key_hash == key_hash);
        require(key_cert.public_key_sec1 == key_bytes);
        require(key_cert.not_before <= now);
        require(key_cert.not_after.map(|t| now < t).unwrap_or(true));
        require(hash_public_key(&authority.authority_public_key_sec1) == authority.authority_key_hash);

        verify_signature(
            authority.authority_public_key_sec1,
            canonical_bytes(&key_cert.without_sig()),
            key_cert.authority_sig,
            authority.authority_alg,
        )?;

        return Ok(RootPublicKeyV2 {
            root_pid,
            key_id,
            alg,
            public_key_sec1: key_bytes,
            key_hash,
            not_before: key_cert.not_before,
            not_after: key_cert.not_after,
        });
    }

    Err(AuthError::UnknownRootKey)
}
```

An unknown root key is a trust-anchor failure, not a proof miss. The fallback path permits cert validation during key rotation when the verifier has a stable root-key authority but has not yet cached the new online root key.

`root_public_key_sec1` alone is never sufficient for an unknown key. Unknown keys require `root_key_cert`.

### 5.2 Time Model

V2 uses IC canister time as the authoritative verifier time source.

Rules:

- verifiers use `ic_time_secs()` or the equivalent IC system time conversion
- root issuance uses the same IC time source
- shard token minting uses the same IC time source
- no verifier-specific clock-skew allowance is part of V2
- tokens from the future are rejected
- expired certificates and tokens are rejected exactly at their expiry boundary

V2 does not model arbitrary local clock drift because verification runs inside IC canisters.

## 6. Audience Semantics

### 6.1 Verifier Membership

```rust
fn verifier_is_in_audience(
    cfg: &DelegatedAuthVerifierConfigV2,
    aud: &DelegationAudienceV2,
) -> bool {
    match aud {
        DelegationAudienceV2::Roles(roles) => {
            cfg.local_role.map(|role| roles.contains(&role)).unwrap_or(false)
        }
        DelegationAudienceV2::Principals(principals) => {
            principals.contains(&cfg.local_principal)
        }
        DelegationAudienceV2::RolesOrPrincipals { roles, principals } => {
            principals.contains(&cfg.local_principal)
                || cfg.local_role.map(|role| roles.contains(&role)).unwrap_or(false)
        }
    }
}
```

Role membership is local configuration. Verification does not query registry to decide whether a verifier is a role member.

Principal membership is exact principal equality.

### 6.2 Audience Subset Rules

```rust
fn audience_subset(child: &DelegationAudienceV2, parent: &DelegationAudienceV2) -> bool {
    match (child, parent) {
        (Roles(c), Roles(p)) => set(c).is_subset(set(p)),
        (Principals(c), Principals(p)) => set(c).is_subset(set(p)),
        (
            RolesOrPrincipals { roles: cr, principals: cp },
            RolesOrPrincipals { roles: pr, principals: pp },
        ) => set(cr).is_subset(set(pr)) && set(cp).is_subset(set(pp)),

        (Roles(c), RolesOrPrincipals { roles: p, .. }) => set(c).is_subset(set(p)),
        (Principals(c), RolesOrPrincipals { principals: p, .. }) => set(c).is_subset(set(p)),

        (RolesOrPrincipals { roles, principals }, Roles(p)) => {
            principals.is_empty() && set(roles).is_subset(set(p))
        }
        (RolesOrPrincipals { roles, principals }, Principals(p)) => {
            roles.is_empty() && set(principals).is_subset(set(p))
        }

        (Roles(_), Principals(_)) => false,
        (Principals(_), Roles(_)) => false,
    }
}
```

Required edge case:

```rust
cert.aud = RolesOrPrincipals { roles: [A], principals: [] }
claims.aud = Roles([A])
```

This is valid because the claims audience is a subset of the certificate's role branch.

Required rejected edge case:

```rust
cert.aud = Roles([A])
claims.aud = Principals([P])
```

This is rejected because role and principal families do not cross-match.

## 7. Revocation Strategy

V2 uses Option A: no revocation in the core protocol.

Rules:

- certificates are valid until `cert.expires_at`
- tokens are valid until `claims.expires_at`
- root must enforce `policy.max_cert_ttl_secs`
- root must enforce `policy.max_token_ttl_secs`
- verifiers must enforce token expiry, certificate expiry, token TTL, and certificate TTL-derived bounds

Recommended initial limits:

```rust
pub struct DelegatedAuthTtlPolicyV2 {
    pub max_cert_ttl_secs: u64,
    pub max_token_ttl_secs: u64,
}
```

No component may treat absence from a cache, registry, or proof table as revocation.

If revocation is later required, it must be added as an explicit V3 extension or as a separate V2-compatible revocation input with defined consistency, freshness, and failure semantics.

## 8. Key Distribution Model

### 8.1 Root Keys

Verifiers are initialized with:

- expected root principal
- at least one root trust anchor
- optionally, a root key authority for online root key certificates

Root key lookup order:

1. trusted local `RootKeySetV2`
2. embedded `root_public_key_sec1` plus `RootKeyCertificateV2` signed by the configured root key authority

Root key distribution is allowed to be cached and refreshed, but V2 correctness does not depend on per-proof replication.

`RootKeyAuthorityV2` provisioning:

- installed through trusted root/verifier configuration
- expected to be offline or rarely used
- not used to sign ordinary delegation certificates
- rotated only by explicit trusted configuration update
- compromise is equivalent to compromise of the root key hierarchy

### 8.2 Shard Keys

Shard public keys are embedded in `DelegationCertV2`.

Verifiers do not fetch shard keys.

Root certifies the shard public key, and verifiers enforce:

```rust
hash_public_key(cert.shard_public_key_sec1) == cert.shard_key_hash
cert.shard_key_hash == derive_expected_shard_key_hash(...)
```

This prevents `shard_pid` from being accepted independently of the actual shard signing key.

## 9. Removed V1 Concepts

The following V1 mechanisms are removed from the V2 correctness model:

- local verifier proof cache as an acceptance requirement
- proof fanout during token issuance
- proof fanout during topology changes
- creation-time proof catch-up for new verifiers
- exact proof equality matching against local state
- "delegation proof miss" as an authentication result
- implicit revocation by deleting or failing to replicate proof state

These mechanisms are unnecessary because the root-signed certificate and shard-signed claims carry all authority required for verification.

After the dual-validation migration window closes, the old proof syncing code should be deleted rather than left as dormant auth state:

- proof fanout workflow from root to verifier roles
- topology-change proof catch-up for newly created verifiers
- verifier endpoints that accept pushed delegation proofs
- local verifier proof tables used for token acceptance
- storage records whose only purpose is proof replication
- tests whose expected behavior is proof-cache convergence

## 10. Migration

V2 is a hard-cut protocol, but production rollout should avoid signer/verifier skew.

Rollout order:

1. Deploy verifiers with dual validation enabled: accept V1 and V2.
2. Deploy root support for V2 certificate issuance.
3. Deploy shard signers that mint V2 tokens by default while V1 remains accepted.
4. Wait at least the maximum V1 token TTL and maximum V1 certificate/proof TTL.
5. Disable V1 token acceptance on verifiers.
6. Remove V1 proof distribution, proof catch-up, and proof-cache code paths.

If a deployment chooses not to run dual validation, then all verifiers must be upgraded before any signer emits V2 tokens.

## 11. Failure Modes

### 11.1 Unknown Root Key

Cause:

- verifier does not have the root key
- embedded root key is absent
- embedded root key hash is not trusted
- embedded root key certificate is absent or invalid

Result:

```rust
Err(AuthError::UnknownRootKey)
```

This is a root trust-anchor failure. It is mitigated by publishing root key authority configuration before online root key rotation.

### 11.2 Root Principal Mismatch

Cause:

```rust
cert.root_pid != cfg.expected_root_pid
```

Result:

```rust
Err(AuthError::RootPrincipalMismatch)
```

This prevents accepting certificates from another root even if keysets are misconfigured.

### 11.3 Invalid Root Signature

Cause:

- certificate modified
- wrong root key
- corrupted signature

Result:

```rust
Err(AuthError::InvalidRootSignature)
```

### 11.4 Invalid Shard Key Binding

Cause:

- `shard_public_key_sec1` does not hash to `shard_key_hash`
- `shard_key_hash` does not match deterministic derivation for `shard_pid` and `shard_key_id`

Result:

```rust
Err(AuthError::InvalidShardKeyBinding)
```

### 11.5 Invalid Shard Signature

Cause:

- token claims modified
- wrong shard key
- corrupted signature

Result:

```rust
Err(AuthError::InvalidShardSignature)
```

### 11.6 Expired Certificate or Token

Cause:

- `now >= cert.expires_at`
- `now >= claims.expires_at`

Result:

```rust
Err(AuthError::ExpiredCertificate)
Err(AuthError::ExpiredToken)
```

### 11.7 TTL Violation

Cause:

- certificate TTL exceeds root policy
- token TTL exceeds `cert.max_token_ttl_secs`
- token expires after certificate

Result:

```rust
Err(AuthError::TtlViolation)
```

### 11.8 Audience Rejection

Cause:

- verifier principal is not in audience
- verifier role is not in audience
- claims audience is not a subset of certificate audience
- role hash does not match verifier role

Result:

```rust
Err(AuthError::AudienceRejected)
```

### 11.9 Scope Rejection

Cause:

- claims scopes are not a subset of certificate scopes
- endpoint required scopes are not a subset of claims scopes

Result:

```rust
Err(AuthError::ScopeRejected)
```

### 11.10 Replay

Cause:

- a valid token is reused before expiry

Result:

- core V2 accepts the token if all checks pass

Replay protection is not provided by core V2. The TTL policy is the replay bound.

## 12. Security Properties

V2 guarantees:

- certificate integrity through root signature verification
- token claims integrity through shard signature verification
- root identity binding through `cert.root_pid == expected_root_pid`
- root key identity binding through `root_key_hash`
- shard key binding through root-certified key material and deterministic key derivation
- role audience binding through `verifier_role_hash`
- principal audience binding through exact principal equality
- scope attenuation through subset checks
- temporal bounds through certificate and token expiry checks
- deterministic verification independent of topology timing

V2 does not guarantee:

- revocation before expiry
- single-use tokens
- nonce replay detection
- protection from incorrect verifier role configuration
- registry membership freshness
- automatic root trust-anchor rollout
- acceptance of certificates from roots outside the verifier's configured trust anchor

## 13. Implementation Checklist

- Define canonical encoding for all signed V2 structures.
- Add `DelegationCertV2`, `DelegationProofV2`, `DelegatedTokenClaimsV2`, and `DelegatedTokenV2` DTO or ABI types.
- Add `verifier_role_hash` and enforce exact role hash semantics.
- Add `root_pid` binding checks before root key lookup.
- Add `root_public_key_sec1` and `RootKeyCertificateV2` fallback handling.
- Reject embedded root keys that do not chain to configured root trust.
- Embed shard public key in the certificate.
- Enforce shard public key hash and deterministic shard derivation checks.
- Enforce root-side certificate TTL and token TTL policy.
- Enforce verifier-side certificate, token, and TTL-derived bounds.
- Keep nonce informational unless a separate replay subsystem is explicitly designed.
- Remove proof lookup from the verifier acceptance path.
- Roll out dual validation before signers emit V2 tokens.
