# 0.40 Design: Attested Canic Calls

## Status

Implementation-ready design for the 0.40 hard cut.

0.40 starts a Canic-native authenticated internal RPC protocol. It is not just
a caller-role predicate. The protocol gives Canic-to-Canic endpoints a signed,
method-scoped proof of the transport caller's role, carries that proof in a
Canic-owned envelope, and verifies it in the generated endpoint wrapper before
application handler dispatch.

The target developer model is:

```text
Canic internal endpoint authors declare caller role policy.
Canic internal callers use generated internal clients, or CanicCall directly.
Canic obtains, caches, transports, and verifies root-signed proofs.
Application code does not hand-roll proof requests or proof payloads.
Raw calls to protected internal endpoints fail closed.
```

This is a hard cut because endpoint ABI, call encoding, generated wrappers,
attestation payloads, and internal call APIs must change together.

---

## Assessment

This design is sound if "just works" means:

```text
Canic owns an internal RPC envelope and generated endpoint/client wrappers.
```

It is not sound if it means:

```text
inter-canister calls silently gain hidden role metadata.
```

The IC gives the callee `msg_caller()`. It does not give the callee a
root-authorized caller role. That proof must be transported by the caller and
verified by the callee.

The benefit is worth the hard cut:

- sibling app canisters stop trusting stale or partial verifier-local topology
  state as authority
- system endpoints get root-signed subject, role, audience, method, subnet,
  TTL, and epoch checks
- Toko-style `project_hub -> user_hub` calls become a first-class Canic
  pattern
- application authors stop manually passing security payloads
- old raw callers fail before user handler dispatch

The implementation cost is real:

- internal endpoint exported Candid changes to one Canic envelope argument
- direct `Call` cannot transparently wrap arbitrary external services
- root role-attestation issuance must support AppIndex roles as well as subnet
  registry roles
- current macro expansion must split public/raw wrappers from protected
  internal/envelope wrappers
- bootstrap and existing capability endpoints need explicit exceptions

---

## Core Decision

0.40 introduces a protected internal endpoint ABI:

```text
CanicInternalCallEnvelopeV1 {
  header,
  signed_internal_invocation_proof,
  original_candid_args
}
```

Generated internal endpoint wrappers decode the envelope, verify the signed
proof, then decode `original_candid_args` into the user function's declared
parameters.

Application function signatures stay natural:

```rust
#[canic_update(internal, requires(caller::has_role(canister::PROJECT_HUB)))]
async fn system_add_project_to_user(
    user_id: Principal,
    project_pid: Principal,
) -> Result<(), Error> {
    ...
}
```

The exported wire ABI is no longer the raw `(user_id, project_pid)` tuple. The
exported ABI is:

```text
(CanicInternalCallEnvelopeV1) -> Result<_, canic::Error>
```

The macro owns the translation. The user handler is not exported directly.

## Terminology

### AppIndex role

The role assigned to a canonical app canister in the root AppIndex, such as
`project_hub` or `user_hub`.

AppIndex is an issuance authority source for root. It is not a verifier-local
authorization proof.

### subnet registry role

The role stored for a canister in the root subnet registry. Registry roles
cover root-managed topology entries and existing root-capability callers.

The subnet registry is also an issuance authority source for root. It is not a
general sibling-app authorization predicate for protected internal endpoints.

### caller role

The role an endpoint policy requires from the IC transport caller.

Recommended DSL spelling:

```rust
caller::has_role(canister::PROJECT_HUB)
```

`has_role` is clearer than `role`: the predicate authorizes only after an
attested proof is verified. It must not mean "look up this caller in local
AppIndex and accept it."

### attested role

The role contained in a root-signed proof whose `subject` is the IC transport
caller. The verifier never trusts a role string supplied in an unsigned header
or raw argument.

### internal invocation proof

The 0.40 signed object is best understood as a method-scoped role attestation
used as an internal-call capability.

It binds:

```text
subject canister
role
audience canister
audience method
subnet
TTL
epoch
```

That is more specific than a generic role assertion. The implementation may
reuse the existing `SignedRoleAttestation` key family and verifier machinery,
but the wrapper-level protocol should name the object
`SignedInternalInvocationProofV1` or equivalently require
`RoleAttestationScope::InternalMethod`. The important invariant is that
protected internal endpoints require method scope.

### endpoint authorization policy

The policy generated from endpoint attributes, for example:

```rust
#[canic_update(internal, requires(caller::has_role(canister::PROJECT_HUB)))]
```

Policy is local to the verifier and generated by the macro. Root may issue a
proof that a subject has a role, but the endpoint wrapper decides whether that
role is accepted for the current method.

### verifier-local topology state

The verifier's local AppIndex, subnet registry, parent/root/self config, and
cached key state. It can help with structural bootstrap checks and key lookup,
but it is not sufficient authority for sibling app callers.

Key invariant:

```text
Verifiers do not accept a sibling/internal caller merely because local AppIndex
or local registry state says the caller has a role. Root-authorized proof is
transported with the call and verified at the endpoint wrapper.
```

## Current Architecture Notes

The current codebase already has several pieces 0.40 should reuse:

- `RoleAttestation` currently binds `subject`, `role`, optional `subnet_id`,
  `audience`, `issued_at`, `expires_at`, and `epoch`.
- `AuthApi::verify_role_attestation` checks signature, subject equals
  `msg_caller()`, audience equals `canister_self()`, subnet, TTL, epoch floor,
  and key validity.
- Unknown role-attestation keys are refreshed once from
  `canic_attestation_key_set` on update verification paths.
- `auth.role_attestation.min_accepted_epoch_by_role` is the current epoch floor
  model.
- Root issuance currently authorizes role attestations from the subnet registry
  only and copies `req.epoch` into the signed payload. 0.40 must change both.
- The endpoint macro currently exports the user function's raw Candid
  signature and runs access predicates before dispatch. 0.40 protected internal
  endpoints need a different wrapper shape.
- `Call` is intentionally a raw low-level IC call facade. 0.40 should add a
  separate `CanicCall` instead of making `Call` guess which targets are Canic
  internal endpoints.
- There is no mature generated internal client surface today. Generated
  clients are the canonical application-facing direction, but `CanicCall` is
  the necessary first implementation step.

## Non-Goals

0.40 does not:

- attach invisible metadata to IC calls
- rewrite arbitrary `ic_cdk::call::Call` usage
- wrap ledger, NNS, management canister, or other external-service calls
- make public user ingress use role attestations
- remove delegated user tokens
- treat cached outgoing proofs or verifier key caches as authority
- require application code to manually request or pass attestations
- use verifier-local AppIndex or subnet registry as sibling-app authority

## Hard-Cut Rules

### 1. Protected Canic internal endpoints require attested caller roles

Every protected Canic-to-Canic internal endpoint must declare one of:

```rust
caller::has_role(canister::PROJECT_HUB)
caller::has_any_role([canister::PROJECT_HUB, canister::ADMIN_HUB])
```

or one narrow structural bootstrap predicate:

```rust
caller::is_parent()
caller::is_root()
caller::is_same_canister()
caller::is_registered_to_subnet()
```

`caller::is_registered_to_subnet()` remains root-bootstrap-only. It must not be
the final sibling app authorization surface.

### 2. Raw calls to protected internal endpoints fail closed

Calling a protected internal endpoint with the old raw Candid args must fail
before user function dispatch.

The wrapper should return `canic::Error { code: Unauthorized, ... }` for
envelope decode, proof verification, policy mismatch, and caller-binding
failures. It should not trap for expected compatibility failures, and it should
not leak enough detail to become an authorization oracle. Internal logs and
metrics can record the precise rejection reason.

### 3. Canic-owned internal calls use the envelope

Application code should not write:

```rust
let proof = AuthApi::request_role_attestation(...).await?;
Call::unbounded_wait(pid, method)
    .with_args((proof, args...))?
    .execute()
    .await?;
```

The low-level primitive is:

```rust
CanicCall::unbounded_wait(user_hub_pid, "system_add_project_to_user")
    .with_caller_role(canister::PROJECT_HUB)
    .with_args((user_id, project_pid))?
    .execute()
    .await?;
```

The canonical application-facing interface is a generated internal client:

```rust
UserHubInternalClient::new(user_hub_pid)
    .system_add_project_to_user(user_id, project_pid)
    .await?;
```

Handwritten `CanicCall` remains allowed as an escape hatch and transitional
mechanism. Raw `Call` remains for external services and must not auto-wrap.

### 4. AppIndex-only caller predicates are removed

`caller::has_app_role(role)` was useful to diagnose the Toko hardening issue,
but it accepts based on verifier-local AppIndex. The 0.40 protected sibling
surface is now only `caller::has_role(...)` / `caller::has_any_role(...)`,
backed by root-signed internal invocation proofs. `caller::has_app_role(...)`
is rejected by the macro parser.

## Wire Shape

The envelope header contains only transport binding fields. Authorization
identity lives in the signed proof.

```rust
pub struct CanicInternalCallEnvelopeV1 {
    pub version: u16,
    pub header: CanicInternalCallHeaderV1,
    pub proof: SignedInternalInvocationProofV1,
    pub args: Vec<u8>,
}

pub struct CanicInternalCallHeaderV1 {
    pub target_canister: Principal,
    pub target_method: EndpointId,
}

pub struct SignedInternalInvocationProofV1 {
    pub payload: InternalInvocationProofPayloadV1,
    pub signature: Vec<u8>,
    pub key_id: u32,
}

pub struct InternalInvocationProofPayloadV1 {
    pub subject: Principal,
    pub role: CanisterRole,
    pub subnet_id: Option<Principal>,
    pub audience: Principal,
    pub audience_method: EndpointId,
    pub issued_at: u64,
    pub expires_at: u64,
    pub epoch: u64,
}
```

`caller_role` is intentionally not in the unsigned header. The signed proof
already contains `role`; duplicating it outside the proof adds an untrusted
claim and creates avoidable consistency rules.

`nonce` is intentionally not in V1. A field with no required validation
semantics should not be part of the security protocol. 0.40 does not provide
per-invocation replay prevention for same-method calls. If an endpoint needs
single-use command semantics, it must include a request id in its domain DTO
and use existing replay/idempotency machinery.

`args` are not signed by root. The proof is a method-scoped invocation
capability, not a per-argument mutation receipt. Root issuance stays cacheable
across calls to the same method. If a future endpoint needs root-authorized
argument binding, that should be a separate proof family.

The proof authorizes this caller role to invoke this method. It does not
authorize the particular argument payload.

Example:

```text
project_hub may invoke user_hub.system_add_project_to_user
```

does not imply:

```text
project_hub may add any project to any user
```

Object-level, user-level, tenant-level, quota, ownership, and relationship
checks remain domain logic inside the handler/workflow. The generated wrapper
authenticates and authorizes the internal caller role before dispatch; it does
not replace endpoint-specific policy.

The wrapper verifies:

```text
envelope.version is supported
header.target_canister == canister_self()
header.target_method == current exported endpoint name
proof.payload.subject == msg_caller()
proof.payload.audience == canister_self()
proof.payload.audience_method == current exported endpoint name
proof.payload.role matches generated endpoint policy
proof.signature verifies under an accepted root attestation key
proof.payload.expires_at >= now
proof.payload.subnet_id matches verifier subnet when present
proof.payload.epoch >= verifier minimum accepted epoch for role
```

Then, and only then, it decodes `args` into the user function's original
parameters and dispatches.

The method binding must use the exported Candid method name, not necessarily
the Rust function identifier. Current macro code builds some access metadata
from `stringify!(fn_name)` even when `name = "..."` is used; 0.40 must audit and
fix that so proof binding, generated clients, `.did`, and wrapper policy all
use the same exported method id.

## Signing Domain

Current role-attestation signing uses the `CANIC_ROLE_ATTESTATION_V1` domain.

0.40 should use a new signing domain for method-scoped internal invocation
proofs, for example:

```text
CANIC_INTERNAL_INVOCATION_PROOF_V1
```

The implementation can still reuse the role-attestation key family and key-set
publication endpoint. A distinct signing domain prevents a generic
role-attestation payload and an internal-call proof from being confused as the
same protocol object.

If implementation instead evolves `RoleAttestation` with a scope enum, the
domain must still make method-scoped V1 unambiguous:

```rust
pub enum RoleAttestationScope {
    Canister { audience: Principal },
    InternalMethod { audience: Principal, method: EndpointId },
}
```

Protected 0.40 internal endpoints require `InternalMethod`.

## Root Issuance Flow

Caller asks root for an internal invocation proof:

```text
subject = canister_self()
role = local configured canister role
audience = target canister
audience_method = exported target method
ttl = configured internal-call proof ttl
```

Root authorization decision:

```text
subject == msg_caller()
requested role is valid for subject under Root AppIndex or subnet registry
audience is a known Canic canister
audience_method is non-empty
ttl is > 0 and <= auth.role_attestation.max_ttl_secs
root chooses/signs the current authoritative epoch
root signs with the current attestation key
```

Role authority sources:

```text
AppIndex role:
  RootAppIndex[requested_role] == subject_pid

Registry role:
  SubnetRegistry[subject_pid].role == requested_role
```

Root should not trust a caller-provided epoch as authoritative. If the request
keeps an epoch field for compatibility, root must either ignore it and sign the
current epoch or reject it when it is below the current root epoch. The signed
payload must reflect root state, not caller cache state.

Root should validate that the audience principal belongs to the Canic topology
known to root, using AppIndex, subnet index, or subnet registry as appropriate.

Root does not need to validate that `audience_method` exists or that the method
accepts the requested role in V1. Current Canic does not have a root-published
internal method policy manifest. Endpoint compatibility remains verifier-side
macro policy. A future internal-method manifest could let root reject impossible
method/role combinations earlier, but it is not required for the security
invariant.

## Verifier Invariants

The generated verifier wrapper must not trust:

- caller-provided role strings outside the signed proof
- verifier-local AppIndex alone
- verifier-local subnet registry alone
- cached outgoing proofs as authority
- cached public keys without root key identity checks
- raw Candid arguments for protected internal endpoints
- user handler code to perform internal caller authorization

The wrapper may trust:

- IC transport caller from `msg_caller()`
- root signature under an accepted role-attestation/internal-proof key
- signed payload bindings
- local endpoint policy generated by the macro
- local verifier identity from `canister_self()`
- local subnet id for subnet binding
- local epoch floor and accepted key validity windows

Acceptance requires all of these to agree:

```text
msg_caller() == proof.subject
canister_self() == header.target_canister == proof.audience
exported method == header.target_method == proof.audience_method
proof.role satisfies generated endpoint caller policy
```

## Epoch and Revocation Model

Current Canic has a deliberately simple epoch floor:

```toml
[auth.role_attestation.min_accepted_epoch_by_role]
project_hub = 2
```

There is no subject epoch, audience epoch, or key epoch in the current
role-attestation verifier. Attestation keys have `key_id`, `key_name`,
`key_hash`, status, and validity windows.

0.40 V1 keeps a single role epoch in the signed payload:

```text
payload.epoch == root current epoch for payload.role
verifier accepts only epoch >= min_accepted_epoch_by_role[payload.role]
```

### V1 propagation model

V1 uses this model:

```text
short TTL is the hard revocation bound
role epoch cascade is the operational revocation accelerator
role reassignment is not complete until epoch floor propagation succeeds to
known verifiers
```

Root owns the authoritative role epoch map:

```text
RoleEpochs[CanisterRole] -> u64
```

Root includes the current role epoch when it signs a method-scoped internal
invocation proof. Root also publishes minimum accepted role epochs through the
existing cascade path so verifiers can reject older proofs before TTL expiry.
The verifier-local floor is still represented as:

```text
min_accepted_epoch_by_role[payload.role]
```

but for 0.40 it must be operational state distributed by topology/state sync,
not only a static `canic.toml` value.

Role reassignment flow:

```text
1. Root updates AppIndex or subnet-registry role assignment.
2. Root increments RoleEpochs[role].
3. Root emits cascaded state containing the new min accepted epoch floor.
4. Known verifiers acknowledge or are observed with the new floor.
5. The reassignment is considered complete.
```

If cascade is delayed or a verifier is unreachable, the old proof remains valid
there until either:

```text
the verifier receives the epoch floor, or
the proof TTL expires
```

Therefore V1 must keep internal invocation proof TTLs short. The recommended
default is:

```text
60 seconds local/dev
120 seconds production
max 300 seconds unless an endpoint family explicitly justifies more
```

`auth.role_attestation.max_ttl_secs` can remain the upper policy bound, but
`CanicCall` should request a much shorter internal-call TTL by default.

Invalidation behavior:

- AppIndex role reassignment: root bumps the role epoch for that role. Old
  proofs for the previous canonical subject fail once verifiers receive the
  new epoch floor; otherwise TTL remains the hard upper bound.
- Subnet registry role change: root bumps the role epoch for that role. This
  invalidates cached proofs for all subjects with that role.
- Compromised/deauthorized subject: root changes the subject's AppIndex or
  registry assignment and bumps the role epoch. V1 invalidation is broad by
  role, not per subject.
- Audience canister replacement: proof audience is a principal. A replacement
  canister with a new principal rejects old proofs by audience mismatch. An
  upgrade at the same principal relies on method policy, TTL, and epoch.
- Audience method migration: method-scoped proofs fail when the exported method
  name changes.
- Root attestation key rotation: verifiers accept only published keys whose
  key identity, status, and validity window are acceptable. Unknown keys may be
  refreshed once on update paths.
- Emergency invalidation: bump the relevant role epoch and/or expire/remove the
  signing key from the root key set. Role epoch is the safer normal revocation
  lever; key invalidation is broader.
- Stale outgoing attestations: callers may cache them, but callee verification
  remains authoritative and rejects stale epoch/expired/key-invalid proofs.

Known V1 limitation:

```text
Deauthorizing one subject under a shared role requires bumping that role epoch,
which invalidates all cached proofs for that role.
```

That is acceptable for 0.40. Add subject/audience/key epochs only if role-wide
invalidation becomes too disruptive. V1 intentionally does not fetch role epoch
state from root during verifier execution; cascade plus short TTL is the
revocation model.

## Attestation Cache

The outgoing cache is an optimization only. It is not an authority source.

Cache scope:

```text
per subject canister
```

Cache key:

```text
root_pid
root_attestation_key_name
subject_pid
subject_role
audience_pid
audience_method
subnet_id
role_epoch
signing_domain
```

A cached proof can be reused while:

```text
now + refresh_margin < expires_at
local root_pid is unchanged
configured attestation key name is unchanged
local role epoch still equals cache key role_epoch
payload fields match the requested call edge exactly
```

Recommended refresh margin:

```text
min(30 seconds, ttl / 5), never less than one IC timestamp tick
```

On cache miss, `CanicCall` requests a fresh proof from root and continues with
the original call.

On verification failure from the callee:

- if the failure is typed as `AuthMaterialStale` or `AuthKeyUnknown`,
  `CanicCall` invalidates the cache entry, requests a new proof once, and
  retries the internal call once
- if the retry fails, return the callee error
- do not retry on `Unauthorized`, `AuthProofExpired`,
  `InternalRpcMalformed`, role mismatch, audience mismatch, method mismatch,
  subject mismatch, or endpoint policy denial

### Internal error taxonomy

The wrapper needs enough machine-readable error information for Canic internal
clients to distinguish retryable infrastructure drift from authorization
denial. It should not expose detailed proof internals as a user-facing oracle.

V1 protected internal wrappers should map verification failures into this
coarse taxonomy:

| Internal class | Retryable by `CanicCall` | Meaning |
| --- | --- | --- |
| `Unauthorized` | no | Validly decoded proof does not authorize this caller/method, or endpoint policy denies it. |
| `AuthMaterialStale` | yes, once | Proof epoch is below the verifier's accepted role epoch floor. |
| `AuthKeyUnknown` | yes, once | Proof references an unknown root attestation key after verifier-side refresh behavior. |
| `AuthProofExpired` | no | Proof TTL has expired; caller must request a new proof before a new call attempt. |
| `InternalRpcMalformed` | no | Envelope cannot be decoded, has unsupported version, or does not bind to the target canister/method. |

Externally, these may still serialize as stable `canic::Error` values with
`ErrorCode::Unauthorized` or `ErrorCode::InvalidInput` where appropriate, but
generated internal clients need a stable classifier. The classifier can live in
an internal error detail field, a compact error code enum, or a reserved
machine-readable message prefix; the implementation should prefer a typed enum
over string parsing.

Verifier key cache behavior:

- update endpoint wrappers may refresh root key material once on unknown key,
  matching current `AuthApi::verify_role_attestation` behavior
- verifiers never ask the caller for fresh key material

Cache persistence:

- outgoing proof cache may survive upgrade because proofs are time bounded and
  verifier-side checks remain authoritative
- if persisted, store under a new stable auth key such as
  `canic.core.auth.internal_invocation_proof_cache.v1`
- persisted cache entries must be identity-bound by root pid, configured key
  name, subject pid, role, audience, method, epoch, and expiry
- heap-only cache is acceptable for the first implementation if lifecycle
  prewarming covers cold-path performance

Prewarming:

```text
current canister role -> every generated internal client edge it may call
```

Prewarming is optional. Correctness must not depend on it.

## Endpoint ABI Transition

### Protected internal endpoints

For an attested internal endpoint, the exported Candid signature is one
envelope argument:

```did
system_add_project_to_user :
  (CanicInternalCallEnvelopeV1) -> (variant { Ok : T; Err : Error });
```

The user handler keeps the natural Rust signature. The generated wrapper owns:

```text
decode envelope
verify proof and endpoint policy
decode original args
dispatch user handler
```

Internal endpoints remain visible in generated `.did` unless Canic introduces a
separate Candid filtering mechanism. Visibility is not authorization. The
visible protected ABI is the envelope ABI, not the app argument ABI.

Old raw callers fail because their Candid payload cannot decode as
`CanicInternalCallEnvelopeV1`, or because they lack a valid proof. Tests must
assert no user handler state changes on those calls.

### Public endpoints

Public user ingress and external service endpoints keep the current raw Candid
ABI.

### Query endpoints

0.40 V1 does not support protected attested query endpoints.

Reason:

```text
queries cannot perform the root key refresh / material repair path that update
verification can perform
```

Adding cache-only protected queries in the first cut would create a second
verification mode with different failure behavior. That is not worth it for the
0.40 hard cut.

V1 rules:

- `caller::has_role(...)` and `caller::has_any_role(...)` are update-only.
- protected query endpoints fail macro validation.
- current query endpoints must be classified as public/read-only discovery,
  structural exceptions, or converted to update endpoints before protection.
- root-to-wasm-store query endpoints guarded by `caller::is_root()` remain
  structural exceptions until a later protected-query design exists or until
  they are converted to update endpoints.

### Sync functions

Protected attested endpoints should initially require `Result` return types and
the existing macro-supported access shape. Synchronous non-`Result` discovery
or standards queries are not part of the protected internal RPC protocol. They
should be audited and either:

- remain intentionally public/read-only discovery endpoints, or
- be converted to protected async/result endpoints.

### Internal marker cleanup

The current `internal` marker is broader than "privileged Canic-to-Canic RPC".
Examples include metadata, standards, readiness, and bootstrap status queries.
0.40 must audit `internal` endpoints and split them into:

```text
protected internal RPC     -> envelope + proof
structural bootstrap       -> explicit exception
public/read-only discovery -> not a privileged sibling authorization surface
existing capability RPC    -> explicit existing protocol exception or migrated
```

### Endpoint classification audit

Implementation should start with an endpoint inventory, not by mechanically
wrapping every `internal` endpoint.

Initial codebase-aware classification:

| Endpoint/family | Current guard | Purpose | 0.40 classification | Migration action |
| --- | --- | --- | --- | --- |
| `canic_request_role_attestation` | `caller::is_registered_to_subnet()` | Root issues role/internal-call proof material. | Structural bootstrap | Keep structural; strengthen root issuance policy. |
| `canic_attestation_key_set` | `caller::is_registered_to_subnet()` | Root publishes verifier key material. | Structural bootstrap | Keep structural; no sibling app authorization. |
| `canic_request_delegation` | `caller::is_registered_to_subnet()` | Root issues delegated auth material. | Structural bootstrap | Keep structural; audit separately from internal RPC. |
| `canic_sync_state` / `canic_sync_topology` | `caller::is_parent()` | Parent cascades state/topology to children. | Structural bootstrap | Keep parent-only; include role epoch floors in cascaded state. |
| `canic_response_capability_v1` | root: `caller::is_registered_to_subnet()`; nonroot: existing capability envelope | Existing root capability RPC protocol. | Existing capability RPC | Do not double-wrap in initial 0.40; audit/migrate deliberately later. |
| Local wasm-store update endpoints | `caller::is_root()` | Root publishes/manages local wasm-store content. | Protected internal RPC | Convert update endpoints to `caller::has_role(ROOT)` and `CanicCall`. |
| Local wasm-store query endpoints | `caller::is_root()` | Root reads wasm-store catalog/status. | Structural query exception for V1 | Keep structural or convert to update; protected queries are postponed. |
| Root admin/operator endpoints | controller-only, some marked `internal` | Operator/admin control-plane surface. | External/operator raw path | Keep raw controller auth; not sibling internal RPC. |
| `canic_ready`, `canic_bootstrap_status`, metadata/ICRC standards | unguarded `internal` queries | Discovery, readiness, standards metadata. | Public/read-only discovery | Do not envelope-wrap; reconsider whether `internal` marker is misleading. |
| Metrics, env, log, config diagnostics | public or controller guards | Observability/operator diagnostics. | Public/controller raw path | Keep existing auth model unless a specific sibling caller appears. |
| App-defined sibling system endpoints | usually `internal` + caller topology guard | App-to-app internal workflow. | Protected internal RPC | Convert to `caller::has_role` and generated internal clients. |
| External ledger/NNS/management calls | raw `Call`/management APIs | External services. | External/raw call path | Never auto-wrap. |

The audit should become either a companion checked-in migration document or a
test fixture used by the raw-call lint. The important rule is that every
`internal` endpoint has an explicit 0.40 classification before implementation
changes its ABI.

## Bootstrap and Exception Model

Some endpoints exist to establish the proof infrastructure or already carry a
separate capability protocol. They cannot all require a 0.40 envelope on first
contact.

### `canic_request_role_attestation`

Why exempt:

```text
It issues the proof needed to call protected internal endpoints.
```

Who may call:

```text
registered Canic canisters, via root structural guard
```

Current structural guard:

```rust
#[canic_update(internal, requires(caller::is_registered_to_subnet()))]
```

Additional root-side guard:

```text
subject == msg_caller()
role is authorized by root AppIndex or subnet registry
```

Why not a sibling escape hatch:

```text
It returns a proof only for the caller's own subject and authorized role. It
does not execute target endpoint logic.
```

### `canic_attestation_key_set`

Why exempt:

```text
Verifiers need root public key material before they can verify proofs.
```

Who may call:

```text
registered Canic canisters
```

Current structural guard:

```rust
#[canic_update(internal, requires(caller::is_registered_to_subnet()))]
```

Why not a sibling escape hatch:

```text
It publishes verification keys only. It grants no role and carries no target
method authorization.
```

### `canic_request_delegation`

Why exempt:

```text
It is part of root auth material issuance and predates internal invocation
proofs.
```

Who may call:

```text
registered Canic canisters
```

Current structural guard:

```rust
#[canic_update(internal, requires(caller::is_registered_to_subnet()))]
```

Why not a sibling escape hatch:

```text
It issues delegation proof material under root policy. It does not authorize a
sibling app endpoint by local AppIndex.
```

### `canic_sync_state` and `canic_sync_topology`

Why exempt:

```text
They are the cascade path that installs the verifier-local topology and epoch
state needed by other verification paths.
```

Who may call:

```text
the configured parent only
```

Current structural guard:

```rust
#[canic_update(internal, requires(caller::is_parent()))]
```

Why not a sibling escape hatch:

```text
Only the structural parent can push cascade state. Sibling app canisters cannot
use these endpoints to authorize themselves.
```

### `canic_response_capability_v1`

Why exempt initially:

```text
It already carries Canic's root-capability envelope with structural,
role-attestation, and delegated-grant proof modes.
```

Who may call:

```text
root endpoint: registered subnet canisters
nonroot endpoint: callers accepted only by capability envelope verification
```

Current guards:

```rust
#[cfg(canic_is_root)]
#[canic_update(internal, requires(caller::is_registered_to_subnet()))]

#[cfg(not(canic_is_root))]
#[canic_update(internal)]
```

Why not a sibling escape hatch:

```text
The endpoint validates a capability envelope and request hash before executing
capability workflow. It should be audited separately, but it should not be
double-wrapped in 0.40 until the capability protocol is intentionally migrated.
```

### Root and wasm-store structural endpoints

Root-to-wasm-store endpoints currently use `caller::is_root()`. These are
protected Canic internal endpoints and should migrate to
`caller::has_role(canister::ROOT)` where practical for update methods.
Query variants remain structural exceptions in V1 because protected queries
are postponed. Any endpoint that remains structural must be listed as a
bootstrap or query exception with a reason.

## Macro Design

### Endpoint declaration

Preferred spelling:

```rust
#[canic_update(internal, requires(caller::has_role(canister::PROJECT_HUB)))]
async fn system_add_project_to_user(...) -> Result<(), Error> {
    ...
}
```

Multiple accepted roles:

```rust
#[canic_update(internal, requires(caller::has_any_role([
    canister::PROJECT_HUB,
    canister::ADMIN_HUB,
])))]
```

Structural bootstrap predicates remain explicit:

```rust
#[canic_update(internal, requires(caller::is_parent()))]
```

### Macro expansion

For protected internal endpoints, the macro emits:

```text
exported wrapper with one CanicInternalCallEnvelopeV1 argument
renamed user implementation with natural app arguments
generated endpoint policy constant for accepted roles
proof verification before app-args decode and dispatch
```

Public endpoints keep the current raw wrapper shape.

### Compile-time validation

The macro rejects:

- `caller::has_role(...)` on non-`internal` endpoints
- `caller::has_role(...)` and `caller::has_any_role(...)` on query endpoints
- attested role endpoints without `Result` return type
- protected internal endpoints that mix delegated-token first arguments with
  internal invocation envelopes
- unsigned AppIndex role predicates on protected sibling endpoints
- `caller::has_any_role([])`
- unsupported bare aliases

The macro should warn or fail when an `internal` endpoint has no guard and is
not in the explicit discovery/bootstrap exception set.

## Call API Design

### Raw external calls

Existing API remains raw:

```rust
Call::unbounded_wait(pid, "icrc1_balance_of")
```

It does not attach Canic proofs. This keeps ledger, NNS, management, and
third-party canister calls unchanged.

### Low-level internal RPC primitive

`CanicCall` is the low-level protected internal RPC primitive:

```rust
CanicCall::unbounded_wait(pid, method)
    .with_caller_role(role)
    .with_args(args)?
    .execute()
    .await
```

Responsibilities:

- Candid-encode original args into `Vec<u8>`
- resolve local subject pid and configured local role
- obtain or reuse a method-scoped internal invocation proof
- build `CanicInternalCallEnvelopeV1`
- dispatch through the raw lower-level call infrastructure
- invalidate cache and retry once only for `AuthMaterialStale` and
  `AuthKeyUnknown`

### Generated internal clients

Generated clients are the canonical app-facing surface. They know:

```text
target canister role
target exported method
required caller role
argument shape
response shape
```

Current Canic does not yet have a mature generated internal client system, so
0.40 should stage this:

1. implement `CanicCall`
2. migrate Canic built-in internal calls to `CanicCall`
3. generate clients for Canic-owned built-in endpoints
4. expose app-generated internal clients as the default ergonomic path
5. keep handwritten `CanicCall` as an escape hatch

Raw `Call` to known protected Canic methods should be caught by tests or a lint
scan in first-party Canic code.

## Migration Plan

Hard-cut invariants:

- raw calls to attested internal endpoints fail closed
- AppIndex-only sibling caller predicates do not become final surface
- bootstrap exceptions stay narrow and documented
- external ledger/NNS/management calls remain raw

Implementation sequence:

1. Add protocol DTOs:
   `CanicInternalCallEnvelopeV1`, header, payload, signed proof, endpoint id.
2. Add method-scoped proof signing domain and payload hashing.
3. Change root issuance:
   AppIndex + subnet registry authority, root-owned epoch selection, audience
   known-canister check, method non-empty check.
4. Add role epoch state:
   root-owned role epoch map, cascaded verifier floors, reassignment completion
   checks for known verifiers, short default internal-call TTL.
5. Extend verifier ops:
   method binding, endpoint policy role matching, update single-refresh path,
   internal retry/error classifier.
6. Add outgoing proof cache:
   per-subject cache key including method and epoch.
7. Add `CanicCall`:
   envelope encoding, proof acquisition, cache use, typed retry.
8. Update endpoint macro parsing/validation:
   add `caller::has_role`, `caller::has_any_role`, reject unsafe mixes and
   protected query endpoints.
9. Update endpoint macro expansion:
   protected internal wrapper takes one envelope and dispatches only after
   verification.
10. Audit exported method id handling:
   `name = "..."` must bind proof and generated clients to the exported method.
11. Complete endpoint classification audit:
    every `internal` endpoint gets a protected/bootstrap/discovery/capability/
    external classification before its ABI changes.
12. Migrate built-in protected system endpoints:
    root/wasm-store app calls, app hub system endpoints, and other sibling
    Canic internal calls.
13. Preserve bootstrap exceptions:
    root auth material, key set, sync cascade, existing capability endpoint.
14. Add generated internal clients for built-in endpoints, then app endpoints.
15. Migrate test fleets and Toko-style project/user hub calls.
16. Add lint/test scan for raw `Call` to protected internal method names.
17. Deprecate/remove `caller::has_app_role` and any registry-only sibling
    authorization left outside the exception list.

## Test Matrix

### Root issuance

- Root issues AppIndex role proof for canonical app pid.
- Root rejects AppIndex role proof for noncanonical pid.
- Root issues registry role proof for registry-managed child.
- Root rejects `subject != msg_caller()`.
- Root rejects wrong requested role.
- Root rejects unknown audience canister.
- Root rejects empty audience method.
- Root rejects invalid TTL.
- Root signs current epoch rather than trusting stale caller epoch.
- Root defaults internal-call proof TTL to the short configured internal-call
  TTL, not the maximum role-attestation TTL.

### Epoch propagation

- AppIndex reassignment increments root role epoch.
- Subnet registry role change increments root role epoch.
- Cascaded state carries the new minimum accepted epoch floor.
- Known verifier rejects old proof after cascade.
- If cascade is withheld, old proof remains valid only until short TTL expiry.
- Reassignment completion waits for known verifier propagation/observation.

### Verifier wrapper

- Valid envelope dispatches user handler.
- Raw Candid args are rejected before dispatch.
- Malformed envelope returns `ErrorCode::Unauthorized` and does not dispatch.
- Wrong `msg_caller()` is rejected even with otherwise valid proof.
- Proof for method A is rejected on method B.
- Proof for audience canister A is rejected on canister B.
- Role mismatch is rejected.
- Expired proof is rejected.
- Stale epoch is rejected.
- Wrong subnet is rejected.
- Wrong root key / wrong key hash is rejected.
- Unknown key update path refreshes once then succeeds or fails closed.
- Protected query endpoint declarations fail macro validation in V1.
- Endpoint policy uses exported method name when `name = "..."` is present.
- Method-scoped proof does not skip domain argument authorization; handler
  tests still reject invalid object/user/project relationships.

### Revocation

- AppIndex reassignment invalidates old proof after role epoch floor update.
- Registry role change invalidates old proof after role epoch floor update.
- Key rotation accepts previous key only inside validity window.
- Emergency epoch bump invalidates cached outgoing proof.

### Cache and call path

- Outgoing cache key includes audience method.
- Cache is per subject canister.
- Root pid change invalidates cache.
- Attestation key-name change invalidates cache.
- Refresh margin causes pre-expiry refresh.
- `AuthMaterialStale` callee failure invalidates cache and retries issuance
  once.
- `AuthKeyUnknown` callee failure invalidates cache and retries issuance once.
- `Unauthorized`, `AuthProofExpired`, `InternalRpcMalformed`, role/audience/
  method mismatch, and endpoint policy denial do not retry.

### Generated clients and raw calls

- Generated internal client sends `CanicInternalCallEnvelopeV1`.
- Handwritten `CanicCall` sends the same envelope shape.
- Raw `Call` to protected built-in endpoint fails closed.
- Raw external ledger/NNS/management calls remain unwrapped.
- Lint/test scan catches first-party raw `Call` to known protected methods.

### Bootstrap exceptions

- `canic_request_role_attestation` works without 0.40 envelope and remains
  registered-subnet/subject-bound.
- `canic_attestation_key_set` works without 0.40 envelope and remains
  registered-subnet guarded.
- `canic_request_delegation` works without 0.40 envelope and remains
  registered-subnet guarded.
- `canic_sync_state` and `canic_sync_topology` work without 0.40 envelope and
  remain parent-only.
- `canic_response_capability_v1` keeps its capability-envelope checks and does
  not become a general sibling authorization path.

## Resolved Design Questions

### `caller::role(role)` vs `caller::has_role(role)`

Use:

```rust
caller::has_role(role)
```

It reads as a predicate and avoids implying that the endpoint simply reads a
role field from local topology state.

### Generated clients mandatory or staged

Staged.

`CanicCall` is required first as the low-level primitive. Generated clients are
the canonical application-facing API, but the current codebase needs the
primitive before generation can be made comprehensive. Handwritten `CanicCall`
stays as an escape hatch.

### Is `audience_method` required?

Yes for 0.40 protected internal calls.

Generic canister-level role attestations may remain for existing root
capability flows, but protected internal endpoint wrappers require method
scope.

### Lifecycle prewarming

Recommended as an optimization only.

Prewarm generated internal client edges where known, especially root and app
hub hot paths. Correctness must not depend on prewarming.

### Protected query endpoints

Postpone them for 0.40 V1.

The first implementation should support protected update endpoints only.
Current internal queries must be classified as public/read-only discovery,
structural query exceptions, or converted to updates. Query-time proof
verification can be designed later once cache-only failure behavior and key
material refresh semantics are worth the extra protocol surface.

### Epoch propagation

Use short TTL plus cascaded role epoch floors.

Short TTL is the hard revocation bound. Cascaded epoch floors accelerate
revocation before TTL expiry. A root role reassignment is not complete until
the new epoch floor has propagated to known verifiers or those verifiers are
explicitly treated as unavailable.

### Lint/test scan for raw `Call`

Yes.

Add a first-party scan that fails when raw `Call` or `ic_cdk::call::Call` is
used with known protected internal method constants. Keep raw external service
calls allowed.

## Remaining Open Questions

1. Should outgoing internal invocation proof cache be stable in the first
   implementation, or heap-only with lifecycle prewarm? Stable cache is safe if
   identity-bound, but heap-only reduces first-pass complexity.
2. Should root publish an internal-method policy manifest in a later release so
   issuance can reject impossible method/role combinations before the call?

---

## Recommendation

Proceed with the 0.40 hard cut as a real internal RPC protocol:

```text
root-issued method-scoped role proof
Canic-owned internal call envelope
generated endpoint verification
CanicCall low-level primitive
generated internal clients as the canonical app-facing surface
```

The AppIndex-only predicate is a useful temporary diagnosis tool, but the final
model for sibling Canic endpoint authorization is transported, root-signed,
method-scoped proof verified by the generated wrapper.
