# 0.30 Release Audit: Fleet Snapshot Backups

Date: 2026-05-05

This audit summarizes what the `0.30.x` line delivered, how it maps to the
fleet backup/restore design, what operator workflows now exist, and what remains
future work.

## Executive Summary

`0.30.x` turned fleet snapshot backup/restore from a design into a guarded
operator workflow around the `canic` CLI.

The release line now provides:

- topology-based canister selection for snapshot capture
- canonical backup manifests and resumable download journals
- checksum and provenance validation before restore planning
- manifest design-conformance checks for the v1 safety contract
- no-mutation restore planning, status, apply dry-runs, and runner previews
- guarded restore journal execution through `canic restore run`
- compact operator docs and smoke checklists for the backup/restore path

The system still wraps `dfx` for live snapshot and restore operations. Canic
owns the higher-level safety layer: topology selection, manifests, journals,
readiness checks, restore ordering, verification planning, and runner state.

## Main Outcome

Before `0.30`, snapshot work was effectively a low-level per-canister operation
plus design notes.

After `0.30`, the intended operator shape is:

```bash
canic list --network local

canic snapshot download \
  --canister <source-canister> \
  --root <root-canister> \
  --recursive \
  --out backups/<run-id> \
  --stop-before-snapshot \
  --resume-after-snapshot \
  --network local

canic backup smoke \
  --dir backups/<run-id> \
  --out-dir smoke/<run-id> \
  --require-design \
  --require-restore-ready
```

Restore preparation is manifest/journal driven:

```bash
canic restore plan \
  --backup-dir backups/<run-id> \
  --mapping restore-map.json \
  --out restore-plan.json \
  --require-verified \
  --require-design \
  --require-restore-ready

canic restore status \
  --plan restore-plan.json \
  --out restore-status.json

canic restore apply \
  --plan restore-plan.json \
  --status restore-status.json \
  --backup-dir backups/<run-id> \
  --dry-run \
  --out restore-apply-dry-run.json \
  --journal-out restore-apply-journal.json

canic restore run \
  --journal restore-apply-journal.json \
  --dry-run \
  --network local \
  --out restore-run-dry-run.json
```

Guarded execution exists through `canic restore run --execute`, usually with
`--max-steps 1` for cautious incremental runs.

## What Was Implemented

### 1. New Backup And CLI Crates

`0.30.0` and `0.30.1` established the backup line:

- added the backup crate surface
- added the `canic` CLI package
- wired release ordering for the new crates
- added the initial snapshot download command
- added manifest validation and restore planning commands

The CLI command family now includes:

- `canic list`
- `canic snapshot download`
- `canic manifest validate`
- `canic backup status`
- `canic backup inspect`
- `canic backup provenance`
- `canic backup verify`
- `canic backup preflight`
- `canic backup smoke`
- `canic restore plan`
- `canic restore status`
- `canic restore apply`
- `canic restore apply-status`
- `canic restore apply-report`
- `canic restore run`

### 2. Topology-Based Snapshot Capture

`canic snapshot download` now works from registered topology rather than a loose
operator-maintained list.

Implemented behavior:

- select a canister directly
- include direct children with `--include-children`
- include all descendants with `--recursive`
- read live root registry data or saved registry JSON
- compute topology hash at discovery
- recompute topology hash immediately before snapshot creation
- fail closed before snapshot creation if topology changed
- write manifest, journal, artifacts, and checksums into a backup directory

This addresses the key consistency concern that a subtree backup must be based
on one accepted topology view.

### 3. Manifest And Journal Model

The backup layout now separates immutable intent from mutable progress:

- manifest: backup membership, topology, provenance, verification, identity,
  artifact metadata, and restore contract
- download journal: snapshot creation/download/checksum/durable progress
- restore apply journal: planned restore operations and execution state

Implemented journal/readiness improvements:

- monotonic download state tracking
- durable artifact checksum recording
- manifest/journal agreement checks
- topology receipt recording
- backup status and inspection reports
- operation metrics for snapshot/download/checksum/finalization progress

### 4. Backup Integrity And Provenance

The `backup` command family now supports multiple levels of validation:

- `backup status`: resumable journal progress
- `backup inspect`: manifest/journal agreement without reading artifact bytes
- `backup provenance`: source, topology, unit, member, snapshot, code, and
  artifact provenance
- `backup verify`: durable artifact and checksum validation
- `backup preflight`: standard no-mutation validation bundle
- `backup smoke`: post-capture smoke wrapper

Notable safety checks added across the line:

- reject manifest/journal artifact path drift
- reject manifest/journal checksum drift
- reject topology receipt drift
- fail on incomplete journals when required
- write reports before returning nonzero so CI can inspect failure artifacts

### 5. Restore Planning And Readiness

Restore now starts from the manifest, not from ad hoc snapshot directories.

Implemented planning behavior:

- validate manifest and backup layout
- validate optional source-to-target mapping
- reject mapping references not in the manifest
- report identity modes and mapping completeness
- report target parent mapping and topology mismatches
- include snapshot provenance and checksum coverage
- include verification summaries
- include restore operation summaries
- include readiness summaries and stable reason strings
- gate with `--require-verified`
- gate with `--require-restore-ready`
- gate with `--require-design`

Restore readiness became a first-class machine-readable output instead of an
operator inference.

### 6. Restore Apply Dry-Run

`canic restore apply --dry-run` now renders the concrete operations that would
be performed during restore.

Implemented dry-run behavior:

- ordered snapshot upload operations
- ordered snapshot load operations
- code reinstall operations
- member verification operations
- fleet verification operations
- backup directory containment checks
- artifact existence/checksum validation when `--backup-dir` is supplied
- operation count summaries
- initial apply journal generation

The command intentionally still requires `--dry-run`; mutation happens through
the guarded runner path.

### 7. Restore Apply Journal Inspection

The apply journal became a resumable execution boundary.

Maintained tools:

- `restore apply-status`
- `restore apply-report`

Current safeguards:

- operation counts validated against concrete journal operations
- status/report commands expose pending, failed, blocked, and completion state
- mutation is owned by `canic restore run`

The older lower-level apply primitives were removed after `canic restore run`
became the maintained execution path.

### 8. Native Guarded Restore Runner

`canic restore run` became the maintained runner path.

Implemented modes:

- `--dry-run`: preview next runner action without mutation
- `--execute`: claim, execute generated `dfx` command, mark result, persist
  journal
- `--unclaim-pending`: recover one pending operation back to ready

Implemented runner controls:

- `--max-steps`
- `--updated-at`
- run mode / stopped reason / next action gates
- executed count gates
- batch summary gates
- receipt count gates
- receipt kind gates
- stale pending gates
- progress gates
- `--require-no-attention`
- `--require-complete`

Implemented runner output:

- run mode
- stopped reason
- next action
- progress summary
- pending summary
- batch summary
- operation receipts
- operation receipt summary
- operation-kind counts

The old shell runner is now a compatibility wrapper around the native runner.

### 9. Design-Conformance Reporting

`0.30.37` made the design contract enforceable through manifest validation and
restore planning.

`design_conformance` now reports:

- topology hash stability
- accepted topology hash equality
- backup-unit coverage
- flat-unit consistency reasons
- subtree closure declarations
- quiescence mode/strategy coverage
- concrete member verification checks
- fixed vs relocatable identity modes
- snapshot ID/checksum/module/wasm/code provenance coverage
- parent-before-child restore group ordering

The following commands can fail closed on design conformance:

- `canic manifest validate --require-design`
- `canic backup preflight --require-design`
- `canic restore plan --require-design`
- `canic backup smoke --require-design`

### 10. Operator Visibility And Metrics

The 0.30 line also added or tightened runtime visibility adjacent to backup and
restore:

- CanisterOps snapshot/restore calls
- wasm-store metrics
- canister operation metrics
- lifecycle metrics
- directory, scaling, pool, cascade metrics
- sharding and delegated-auth outcome metrics
- auth, replay, intent, platform-call, and provisioning metric families
- local-only query perf samples for audit probes

This matters because snapshot/restore is operationally high risk; operators need
visibility into lifecycle, provisioning, platform calls, auth/replay behavior,
and placement systems when debugging a backup or restore.

### 11. Ingress Payload Limits

`0.30.29` added generated ingress payload limits for `#[canic_update]`.

Default behavior:

- generated `inspect_message` rejects oversized ingress before consensus
- default update payload limit is `16 KiB`
- endpoints can opt into larger payloads with `payload(max_bytes = ...)`

Example:

```rust
#[canic_update(payload(max_bytes = 32 * 1024))]
fn import(payload: String) -> Result<usize, Error> {
    Ok(payload.len())
}
```

This is related to backup/restore because snapshot/bootstrap/upload-style
operator endpoints need explicit payload boundaries.

### 12. CLI Installation And Repo Layout Cleanup

Late 0.30 work made the operator surface easier to install and understand:

- `make install-canic`
- direct checkout install with `cargo install --locked --path crates/canic-cli`
- crates.io install with `cargo install --locked canic-cli --version <version>`
- shared dev installer installs `canic-cli` as `canic`
- `canic list` prints registered canisters as an ASCII tree
- CLI help was trimmed so the first screen shows command families
- `crates/canic-cli/README.md` is now an operator guide rather than a long flag
  dump
- `docs/operations/0.30-backup-restore-smoke.md` is the release smoke checklist
- repo canisters were split into `demo`, `sandbox`, `test`, and `audit`
  directories
- the old shared reference-support crate was removed

## Design Feedback Coverage

### Topology Snapshot Consistency

Covered.

The design required a topology hash invariant. The implementation now records
and checks discovery and pre-snapshot topology hashes, and manifest design
conformance validates the hash inputs and accepted hash equality.

### Quiescence

Partially covered.

The manifest and design conformance now distinguish `crash-consistent` from
`quiesced-unit` and require a declared strategy for quiesced units. However,
role-specific pause/drain/resume adapters are still future work.

Current status: safe labeling and validation exist; broad operational
quiescence implementations are not done.

### Directory Versus Topology Boundary

Covered in the design and reflected in validation language.

The rule is that directories may select a subtree root, but authoritative
topology defines membership. The implementation is oriented around topology
membership and design conformance checks rather than directory-defined backup
membership.

### Restore Ordering

Covered for v1.

Restore planning reports ordering summaries and checks parent-before-child
restore-group ordering. Dry-run/apply journals preserve operation ordering and
guard against out-of-order advancement.

### Verification Model

Mostly covered.

The manifest design-conformance path requires concrete member verification
checks, not only labels. Restore planning and apply dry-runs expand member and
fleet checks into operations.

Remaining future work: richer app-specific verification adapters and any
operator escape hatch such as `--force-no-verify` need careful policy if added.

### Backup Unit Definition

Covered for v1 validation.

Backup-unit topology metadata is now validated. The system checks unit coverage,
flat-unit reasons, subtree closure declarations, and invalid role/filter
references.

### Identity Continuity

Covered.

Restore planning includes identity summaries and distinguishes fixed,
relocatable, mapped, in-place, and remapped members. Design conformance reports
identity coverage.

### Snapshot Provenance

Covered.

Manifest snapshot provenance now includes snapshot IDs, durable artifact
checksums, and optional module/wasm/code metadata coverage.

### Journal Resumability

Covered.

Backup download journals and restore apply journals now have explicit state,
operation counts, claim/pending recovery, durable receipts, and report/status
commands.

### Dry Run

Covered.

Restore planning, status, apply dry-run, runner dry-run, backup preflight, and
backup smoke provide no-mutation validation paths.

### Concurrency

Mostly design-level.

The design defines backup and restore concurrency boundaries, but the delivered
0.30 work is primarily correctness, journaling, and guarded execution. There is
no broad parallel execution feature called out as completed.

## Current Limitations And Open Work

1. Quiescence adapters are not broadly implemented.
   `crash-consistent` backups are supported and must be labeled honestly.
   `quiesced-unit` validation exists, but role-specific stop/pause/drain/resume
   strategies remain future work.

2. Restore execution still wraps `dfx`.
   Canic owns planning, journaling, readiness, and runner state, but live
   snapshot upload/load actions are still performed through generated `dfx`
   commands.

3. `restore apply` is intentionally dry-run only.
   Mutation is routed through `canic restore run --execute` and its journal.

4. Rich semantic verification is still role/application dependent.
   The framework requires concrete checks and expands them into operations, but
   application-specific checks still need to be declared per role/system.

5. Concurrency is specified more than exploited.
   The safety model describes where parallelism is allowed, but the release line
   focused on correctness and resumability first.

6. Object-store backup storage is out of scope for v1.
   The implemented model uses filesystem backup directories.

7. Cross-network disaster recovery automation is still out of scope.
   The system can plan and execute mapped restores, but it is not a full
   cross-network DR orchestrator.

## Overall Assessment

The 0.30 line successfully delivered a v1 fleet backup/restore control plane for
operators. The strongest parts are:

- fail-closed topology hash validation
- manifest/journal separation
- durable checksum/provenance tracking
- no-mutation restore planning and smoke checks
- resumable restore apply journals
- guarded runner execution with audit receipts
- design-conformance reporting

The biggest remaining product gap is real quiescence execution. The system now
labels and validates consistency modes, but the deeper per-role pause/drain
contracts still need to be implemented before Canic can claim coordinated
application-consistent backups for arbitrary fleets.

The recommended next work after 0.30 is:

1. define standard quiescence hooks or role-bound quiescence adapters
2. add a small number of real verification adapters for core roles
3. exercise a real local capture + smoke + one-step guarded restore path
4. add any missing operator examples only after the commands stabilize
5. avoid adding more flags until the current operator flow has been used end to
   end
