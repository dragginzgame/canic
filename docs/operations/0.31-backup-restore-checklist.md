# 0.31 Backup / Restore Checklist

Use this checklist for the current host-side snapshot backup flow. It keeps the
operator path explicit: capture, verify, plan, render, preview, then execute.

## Inputs

Prepare these values first:

* `<source-canister>`: the project or fleet root canister to snapshot
* `<root-canister>`: the Canic root canister used for topology discovery
* `<run-id>`: a unique local backup run directory name

## Capture

```bash
canic snapshot download \
  --canister <source-canister> \
  --root <root-canister> \
  --recursive \
  --out backups/<run-id> \
  --resume-after-snapshot \
  --network local
```

Expected result:

* `backups/<run-id>/manifest.json` exists
* `backups/<run-id>/download-journal.json` exists
* all downloaded artifacts are durable
* discovery and pre-snapshot topology hashes match

## Verify

```bash
canic backup verify \
  --dir backups/<run-id> \
  --out checks/<run-id>/backup-verify.json
```

Expected result:

* manifest and journal backup IDs match
* manifest and journal topology receipts match
* every durable artifact exists and matches its checksum

## Plan

```bash
canic restore plan \
  --backup-dir backups/<run-id> \
  --out checks/<run-id>/restore-plan.json \
  --require-verified \
  --require-restore-ready
```

Expected result:

* restore plan generation succeeds
* `readiness_summary.ready` is `true`

## Render

```bash
canic restore apply \
  --plan checks/<run-id>/restore-plan.json \
  --backup-dir backups/<run-id> \
  --dry-run \
  --out checks/<run-id>/restore-apply-dry-run.json \
  --journal-out checks/<run-id>/restore-apply-journal.json
```

Expected result:

* every expected upload, load, and verification operation is present
* no code reinstall operation is planned after snapshot load
* operation-kind counts match the rendered operations

## Preview

```bash
canic restore run \
  --journal checks/<run-id>/restore-apply-journal.json \
  --dry-run \
  --out checks/<run-id>/restore-run-dry-run.json
```

Expected result:

* runner preview succeeds
* no operation is executed during preview
* the next action is clear from `next_action`

## Execute

Run one or more bounded execution steps after reviewing the preview:

```bash
canic restore run \
  --journal checks/<run-id>/restore-apply-journal.json \
  --execute \
  --max-steps 1 \
  --out checks/<run-id>/restore-run-step.json
```

Expected result:

* at most one operation is claimed and executed
* snapshot load operations first prove the target canister is stopped with
  `dfx canister status`
* upload operations record the target-side uploaded snapshot ID
* later load operations use the uploaded target snapshot ID from the journal
* failures retain command, stderr, stdout tail, timestamp marker, and attempt count
