# 0.94 Command Quiescence And Pending Recovery

Date: 2026-07-19

## Scope

This finding-backed slice follows the released `v0.94.0` journal-lock batch.
It fixes `CANIC-094-PROC-001` at the existing backup/restore executor boundary
and records one directly exposed restore-recovery defect. It adds no daemon,
process scanner, lease service, persisted identity, hidden command, or
compatibility path.

## `CANIC-094-PROC-001`

The runner previously had no restart-visible proof that a child or descendant
was still executing after runner death. The fixed protocol uses one no-follow
kernel lock per journal operation sequence:

- the owner descriptor remains close-on-exec;
- child setup clears close-on-exec only for the intended mutating command;
- descendants retain that open-file description;
- terminal persistence waits for the direct command and a fresh exclusive
  lock, with a bounded 250 ms exit grace; and
- restart returns a typed command-in-flight cause while any member of the old
  command tree still owns the lock.

The direct owner-death test sends `SIGKILL` without unwinding, proves a
descendant retains the lock, and proves reacquisition only after the complete
command tree exits. A host-bound test separately proves the maintained child
setup passes exactly the requested descriptor through an exec and descendant.

Status: `fixed in worktree`.

## `CANIC-094-RESTORE-001`

```text
finding_id: CANIC-094-RESTORE-001
journey_id: CANIC-094-J04
severity: P1
confidence: confirmed
durable_state_before: mutating restore operation Pending with no terminal receipt
external_state_observed: command may be live, committed, failed, or quiescent
old_recovery: public unclaim-pending reset changed Pending to Ready and emitted a synthetic pending-recovered receipt
expected_invariant: unknown mutating work is never represented as retryable or completed
canonical_owner: restore runner execution and pending recovery
affected_surfaces: canic-backup restore API, canic restore run CLI, restore runner JSON
exact_non_scope: no journal editor, supplied receipt, legacy flag, alias, migration shim, or operation-specific reconciliation guess
```

The old reset could erase the distinction between an attempted effect and an
unknown effect. A later execute could then blindly repeat upload, stop, load,
or start. The hard-cut fix removes the CLI flag, public recovery API, response
mode flag, pending-recovered receipt kind/summary, and supporting state-reset
helper.

Current behavior is deterministic:

- a live inherited command lock returns `CommandInFlight`;
- a quiescent mutating `Pending` operation returns `CommandOutcomeUnknown`;
- both preserve the journal, state marker, staging evidence, and receipt set;
  and
- pending read-only verification repeats through the maintained executor and
  may produce its ordinary terminal receipt.

Restore runner JSON is version 2 because its maintained key set changed. No
version-1 output, alias, deprecated flag, fallback reset, or synthetic pending
receipt is retained.

Status: `fixed in worktree`.
