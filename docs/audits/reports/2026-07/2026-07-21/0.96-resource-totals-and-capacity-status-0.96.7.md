## Decision

Terminal receipt reclamation deliberately preserves committed resource totals.
Without a second bound, sequential work on unique resource identities could
therefore release receipt slots while growing the shared totals map forever.

0.96.7 fixes the shared durable resource-total ceiling at 1,000 identities per
canister. This matches the receipt ceiling, fits the pinned six-page maximum
totals measurement, and introduces no new stable allocation or counter.

The limit is shared by local and receipt-backed intent admission because both
authorities already use the same canonical totals map. Existing exact resource
identities remain usable at saturation. Only an admission that would create a
new totals row rejects. When abort or rollback returns the final totals for one
resource to zero, the canonical row is removed and exactly one slot reopens.

## Admission And Lifecycle Ordering

```text
exact retained intent lookup and compatibility checks
  -> validate current totals-map length <= 1,000
  -> if the resource already exists, continue
  -> if a new resource would be row 1,001, reject capacity
  -> compute quantity and aggregate transitions
  -> write primary/index/metadata
  -> persist the already-admitted total
```

Local admission returns the maintained public `ResourceExhausted` envelope.
Receipt-backed admission returns its typed `StoreCapacityReached` result.
Neither path mutates an intent record, pending index, replay record, terminal
reservation, or totals row on rejection.

Lifecycle validates the hard limit before rebuilding the local expiry index.
A pre-existing map above 1,000 is a typed invariant failure; the hard cut does
not add an old-limit reader, migration, truncation, or repair path.

## Guarded Runtime Projection

The internal application-only name is removed. One `ReceiptCapacityView` now
projects the shared authority and feeds the controller-guarded
`canic_runtime_status` endpoint.

`RuntimeReceiptCapacityStatus` reports:

- shared, application, and Canic-owned receipt records;
- pending and terminal application records;
- receipt limit and remaining headroom;
- resource-total count, limit, and remaining headroom;
- fixed warning-headroom threshold;
- terminal reservation slots and pages;
- next terminal eligibility deadline; and
- one fixed `RuntimeCheckStatus` plus source.

The condition is `pass` above 100 remaining rows in both stores, `warn` when
either store has 100 or fewer, and `fail` when either has zero. `warn` degrades
aggregate runtime status. `fail` makes it failing.

If the projection cannot validate canonical state, the field is absent,
aggregate runtime status is failing, and the response prepends a bounded typed
`intent_capacity/receipt_capacity_unavailable` recent-failure projection. The
query does not mutate the heap failure ring.

`canic inspect canister` renders the same values in text and carries the full
record in structured JSON. The endpoint guard and schema version remain `1`;
this is a pre-1.0 hard cut, not a parallel status endpoint or compatibility
surface.
