# Canic 0.96 Receipt Reconciliation Correction

Date: 2026-07-21

## Result

`run_result: pass`

The 0.96.2 receipt deadline adjunct remains fail-closed during lifecycle
reconciliation without performing a cross-map stable-tree point lookup for
every row. One ordered merge pass validates the canonical primary and
application metadata maps, then rebuilds the placement acknowledgement index
only after complete validation.

This is a bounded Slice C correction. It does not change admission,
settlement, stable data, timers, public APIs, or the still-gated terminal
reclamation design.

## Finding

`CANIC-096-RECONCILE-007` — P2 — lifecycle receipt reconciliation amplified
ordered scans with cross-map point lookups.

The released implementation scanned every receipt primary and called an
application-metadata lookup for each row. It then scanned every application
metadata row and looked its primary up in the other stable tree. At the
100,000-row receipt limit with all rows application-owned, this added up to
100,000 metadata point lookups plus 100,000 primary point lookups to the two
complete ordered scans.

This is a static operation-count bound, not a fabricated cycle benchmark. The
cost occurs synchronously in the init/upgrade lifecycle path, where unnecessary
lookup amplification reduces upgrade headroom.

## Fix And Authority

The hard-cut `reconcile_receipt_indexes` operation is now the sole lifecycle
owner. It merge-validates the two ordered stable maps:

```text
primary iterator + application replay iterator
  -> validate primary schema
  -> compare exact operation IDs
  -> validate application schema and embedded identity
  -> reject missing, orphaned, or Canic-owned application metadata
  -> collect terminal placement acknowledgement identities
  -> after both iterators are canonical, rebuild the derived placement index
```

Each canonical map is traversed once. No cross-map `get` occurs inside the
reconciliation loop. Exact begin, load, and settlement operations retain their
point validation because those are bounded single-operation paths.

The prior placement-only method name and diagnostic are removed. Lifecycle
failures now identify receipt-backed index reconciliation, which truthfully
covers both application adjunct validation and placement derived-index repair.

## Failure Semantics

The operation validates all canonical relationships before clearing the
placement acknowledgement index. Missing metadata, orphan metadata, schema
mismatch, embedded identity mismatch, and metadata attached to a Canic-owned
receipt all return the existing typed ops cause. The old placement index is
unchanged on every returned error.

Successful reconciliation derives placement acknowledgement membership only
from terminal placement primaries. Application terminal rows never enter the
placement queue.
