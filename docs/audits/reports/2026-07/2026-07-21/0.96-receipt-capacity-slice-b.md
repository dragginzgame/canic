## Measurement Boundary

The retained ignored unit test constructs the production stable key and value
types directly through pinned `ic-stable-structures` maps. It measures:

- exact CBOR bytes for maximum valid current primary and totals records;
- the persisted B-tree page size selected from each declared `Storable`
  bound;
- virtual Wasm pages after 100,000 ascending insertions;
- virtual Wasm pages after 100,000 deterministic-permuted insertions; and
- the physical subtotal when the primary, acknowledgement, and totals
  allocations share one pinned base `MemoryManager`. Unrelated product and
  memory-governance allocations are excluded.

The expensive probe is ignored by ordinary unit runs and executed explicitly
for capacity work. Its numeric assertions make dependency, encoding, record,
bound, allocator, and memory-manager drift visible when deliberately rerun.

This is a current-allocation baseline, not a fabricated estimate for the
unimplemented eligibility index. The latter is measured only after its exact
key/value shape is accepted.

## Findings And Fixes

### `CANIC-096-CAPACITY-004` — P1 — totals storage bound excluded valid values

`IntentResourceTotalsRecord` contains three `u64` fields and encodes to 69
bytes when each is maximal. Its declared stable maximum was 64. The ops model
can validly hold large committed and reserved values together, so a later
stable insertion could panic after otherwise valid checked arithmetic.

Fixed by declaring the exact 69-byte maximum and retaining a normal regression
test over all three maximal values. The encoded fields, stable key, allocation,
and schema meaning do not change. A focused upgrade test initializes the map
under the prior 64-byte declaration, loads it under the corrected type, then
inserts and reloads maximum 69-byte values. Existing v2 B-tree memory retains
its stored page size without a compatibility reader.

### `CANIC-096-CAPACITY-005` — P2 — zero totals survived terminal release

Local abort and receipt-backed rollback wrote a three-zero totals row. Reads
already interpret an absent row as the same zero value, so the stored row had
no authority but permanently consumed one resource key slot.

Fixed by routing all product totals writes through one ops helper that removes
the exact row when all totals are zero. Pending and nonzero committed totals
remain stored. Focused tests prove local expiry abort and receipt rollback both
compact the row while preserving their public projection.

### `CANIC-096-GATE-006` — gate — committed resource cardinality needs a bound

Terminal receipt reclamation cannot remove nonzero committed totals because
they remain the resource-capacity authority. Once primary rows are reusable,
their 100,000-row ceiling no longer bounds the number of resource-total rows
accumulated over the canister lifetime.

Smallest hard-cut direction: freeze Toko's maximum durable mint-resource
cardinality, measure the final totals headroom, and reject a new resource
before primary mutation when its compiled capacity is exhausted. Existing
resource keys remain readable and usable at capacity. Do not delete committed
totals, couple their lifetime to receipts, or add an unbounded generic
allocator.

## Reservation Consequence

The current primary high-water is finite and measured, but it does not itself
prove future settlement capacity. Settlement will insert an application
eligibility entry that does not exist today. The exact record shape and one
physical provisioning boundary must be measured after the immutable deadline
and grace contract is accepted.

No per-operation reservation DTO, second primary map, or compatibility layer
is justified by this evidence. The next implementation should use the
smallest dedicated stable-growth mechanism capable of ensuring that an
admitted pending operation can later create its one terminal eligibility row.
