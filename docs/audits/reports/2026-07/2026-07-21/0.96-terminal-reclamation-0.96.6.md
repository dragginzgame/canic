## Decision

The 100,000-row receipt ceiling was introduced as a fixed safety bound in
0.90.1 without an operating-rate, retention, or resource-budget derivation.
Earlier 0.96 evidence proved that the value was technically representable; it
did not establish that allowing one canister to retain that many receipt rows
was appropriate.

0.96.6 hard-cuts the shared primary receipt ceiling to 1,000. The value is
Canic-owned, fixed, and deliberately conservative. There is no configuration
override or compatibility path back to 100,000.

## Exact Capacity Evidence

The pinned `ic-stable-structures 0.7.2` and `ic-memory 0.11.1` measurement at
the new ceiling is:

| Allocation | Ascending pages | Permuted pages |
| --- | ---: | ---: |
| Primary receipt records | 27 | 20 |
| Application replay metadata | 5 | 4 |
| Terminal eligibility | 8 | 6 |
| Maximum distinct resource totals | 6 | 5 |

Through the base `MemoryManager`, the four-allocation ascending high-water is
513 Wasm pages, or 32.0625 MiB. The removed 100,000-row ceiling measured 4,737
pages, or 296.0625 MiB. The hard cut therefore reduces this measured maximum by
4,224 pages, approximately 89.2 percent.

Exact admission tests accept records 1 through 1,000, reject record 1,001, and
reserve eight terminal-eligibility pages. The projection now counts both
application and Canic-owned rows against the shared primary limit instead of
overstating application headroom when placement rows exist.

## Reclamation Authority

One existing owner survives:

```text
application settlement
  -> persist terminal eligibility
  -> reconcile TimerKey::IntentCleanup from the earliest exact deadline

intent_cleanup:run
  -> prevalidate the complete due application prefix
  -> remove exact eligibility row
  -> remove exact application replay metadata
  -> remove exact terminal primary
  -> leave resource totals unchanged
  -> use remaining shared batch budget for due local intent expiry
  -> continue immediately for due work or schedule the next exact deadline
```

The shared batch limit is 32 operations per invocation. Application due work
uses the prefix first; local expiry uses only the remaining budget. This keeps
one timer identity and one bounded execution budget rather than adding a
second cleanup loop.

The application prefix is fully validated before the first mutation. Each
candidate must match the eligibility schema and identity, application-owned
terminal primary, replay metadata, payload binding, terminal revision, and
recomputed deadline. There is no `await` or competing mutation between
validation and exact removal. Unexpected post-validation replacement traps
rather than returning partial success.

## Failure And Replay Behavior

- Pending receipts never enter the eligibility prefix and are never deleted
  because of age.
- A future terminal row is left untouched and supplies the next timer
  deadline.
- A contradictory earliest due row rejects before any row in the batch is
  removed. No later eligibility entry is inspected for reclamation.
- The timer reports an invariant failure and stops through the released 0.95
  authority. Guarded recent-failure diagnostics retain the typed exact cause.
- Resource totals are not removed or recomputed. Committed quantity remains
  authoritative after the receipt history is reclaimed.
- Retrying the same authorization after reclamation reaches absent-operation
  temporal admission and returns `ReplayWindowClosed`; it cannot reserve or
  repeat the effect.

## Remaining 0.96 Boundary

Terminal reclamation no longer waits on Toko. Toko must conform to the Canic
contract before adoption, but its mint rate and domain identity do not own
Canic cleanup.

Two Canic-owned closeout items remain:

1. cap durable resource-total cardinality so repeated terminal reclamation
   cannot move long-term exhaustion into the aggregate map; and
2. expose the maintained capacity projection through the existing guarded
   runtime-status authority with fixed low-cardinality warning behavior.

ICP-refill removal, 0.97/0.98 design work, unrelated timer changes, dependency
updates, and downstream repository edits are outside this slice.
