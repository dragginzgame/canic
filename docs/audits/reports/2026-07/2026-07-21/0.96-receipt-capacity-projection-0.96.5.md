## Decision

The replay and eligibility B-trees already maintain exact constant-time lengths:

```text
total_records              = application_replay.len()
terminal_records           = application_eligibility.len()
pending_records            = checked(total_records - terminal_records)
reserved_terminal_slots    = total_records
reserved_terminal_pages    = provisioned eligibility virtual-memory pages
next_eligibility_at_ns     = first validated eligibility key
```

These values are not reconstructed by scanning the common primary receipt map.
No duplicate counter cell, extra allocation, counter schema, repair path,
alias, or fallback exists. This avoids an additional 8 MiB base `MemoryManager` bucket
and preserves the released 4,737-page, 296.0625-MiB high-water subtotal.

## Failure Behavior

- Terminal count exceeding total count rejects through checked subtraction.
- Application or common record count beyond the fixed limit rejects with a
  typed cause.
- The first eligibility entry is accepted only after validating its schema,
  key/value identity, application-owned terminal primary, replay metadata,
  payload binding, revision, and recomputed deadline.
- Lifecycle performs complete index reconciliation before projecting capacity.
- Projection performs no mutation and no receipt deletion.
