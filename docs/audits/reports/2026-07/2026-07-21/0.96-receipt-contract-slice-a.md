# Canic 0.96 Receipt Contract Slice A

Date: 2026-07-21

## Result

`run_result: partial`

The complete Canic-side receipt authority, consumer, storage, timer, test, and
documentation inventory is frozen against released `v0.95.10`. A read-only
trace of the sibling Toko checkout also freezes its current mint entrypoints
and proves it has not adopted the receipt API. Product behavior is unchanged.
The downstream action-authorization shape, rate bound, recovery endpoint, and
final retention policy remain required before implementation.

This is a deliberate gate, not an implementation failure. Adding an arbitrary
deadline or reclamation interval would make replay safety depend on guessed
downstream behavior.

## Complete In-Repository Consumer Inventory

### Application-owned surface

The only direct `ReceiptBackedIntentApi` consumer in this repository is the
test canister at `canisters/test/intent_authority/src/lib.rs`. Its three seed-
based endpoints exercise begin, load, and settlement. The corresponding
PocketIC journey is `crates/canic-tests/tests/pic_intent_race.rs`.

There is no production Toko adapter, mint endpoint, callback, recovery worker,
or authorization derivation in this repository. Toko was inspected read-only
at the exact snapshot above; it likewise contains no receipt API reference.
Its workspace still consumes Canic `0.71.3`, which predates the receipt-backed
API added in Canic 0.90.

### Read-only downstream entrypoint inventory

| Surface | Current authority | 0.96 consequence |
| --- | --- | --- |
| Frontend service/hook | `ProjectService.mintTokens` through `useMintTokens` | Accepts only token IDs; no action identity, deadline, or recovery identity. |
| Maintained UI callers | collection review and full-screen token views | Both converge on the same service/hook; React Query does not configure mutation retry here. |
| Project endpoint | authenticated `mint_token_tems(token, token_ids)` requiring `TokensMint` | One maintained external admission point, but the request carries no immutable mint-action nonce. |
| Project operation | `ops::ledger::mint_token_items(token_ids)` | Re-resolves mutable token/collection state, calls the ledger, then increments local counters after the await. |
| Ledger batch effect | parent-only `mint_tokens(...)` | Performs atomic current-supply checks and mint mutation, but accepts no operation ID and stores no replay receipt. |
| Ledger stack effect | parent-only `mint_token_stack(...)` | Live Candid bypass surface with an unused project helper; it must be removed or covered by the same receipt contract. |

The current delegated token supplies signed subject, issuer, issued-at,
expiry, audience, grants, nonce, and extension claims. Toko caps token lifetime
at 24 hours and the frontend requests that maximum; the actual expiry may be
shorter to stay inside delegation-proof headroom. The signed token expiry is
therefore a concrete upper bound and candidate immutable replay deadline, but
the token nonce identifies the whole session, not one mint action. Repeated
identical mints within one session can be legitimate, so 0.96 still requires a
separate immutable action nonce/ID.

The endpoint has no explicit mint batch bound and no operation-rate limit.
The generic `ensure_limited` helper caps some other endpoint vectors at 1,000
but is not called by `mint_token_tems`. Consequently the source cannot support
a finite average/burst receipt envelope yet.

### Canic-owned placement surface

Scaling, sharding, and directory allocation converge on
`PlacementAllocationWorkflow`. It constructs one deterministic receipt input,
uses private Canic-owned begin and settlement methods, and schedules the
placement acknowledgement drain. Lifecycle rebuilds allocation 45 from
terminal placement rows. The drain acknowledges one root replay receipt and
then invokes the private exact removal operation.

This is a distinct external-receipt release rule. It is not evidence that an
application authorization has expired and must not be reused as the 0.96
application cleanup rule.

## End-To-End Baseline Trace

Application admission currently performs:

```text
public facade
  -> consumer namespace check
  -> capture IC time
  -> validate payload-binding schema
  -> direct OperationId lookup
  -> exact binding/resource/quantity replay or conflict
  -> primary record-count check
  -> dangling placement-index rejection
  -> shared resource-capacity check
  -> insert Pending primary row
  -> update shared resource totals
```

Settlement performs:

```text
public facade
  -> load and consumer namespace check
  -> validate binding and evidence schemas
  -> exact OperationId lookup
  -> binding/state/revision checks
  -> precompute checked aggregate and terminal record
  -> update shared totals
  -> replace the primary row
  -> placement-only: insert acknowledgement index entry
```

There is no `await` inside either path. Exact retained retries are read before
record or resource capacity rejection and remain idempotent at the current
limit. IC message rollback protects traps across the sequence; returned-error
paths validate contradictions before mutation.

## Confirmed Baseline Findings

### `CANIC-096-RECEIPT-001` — P1 — application terminal history is bounded only by eventual refusal

Every distinct application operation leaves its terminal row in allocation
43. The public facade exposes no removal and no application cleanup owner
exists. At 100,000 retained rows, new IDs receive `StoreCapacityReached` even
when every old operation is terminal.

Smallest hard-cut direction: reclaim only application-owned terminal rows
after an immutable downstream replay deadline and fixed observation grace.
Pending rows remain retained. Placement keeps its acknowledgement-owned path.

### `CANIC-096-RECEIPT-002` — P1 — current state cannot prove safe reclamation

`BeginReceiptBackedIntentInput` and `ReceiptBackedIntentRecord` contain no
absolute replay deadline. An old absent authorization would therefore look
like a new operation after deletion. Age, terminal state, capacity pressure,
or root placement acknowledgement cannot prove that a Toko mint authorization
is no longer executable.

Smallest hard-cut direction: require one immutable application deadline bound
into the operation and payload identities, return retained state before time
checks, and reject an absent operation at `now >= deadline` before mutation.
No optional field, default horizon, old reader, or inferred timestamp is
acceptable.

### `CANIC-096-GATE-003` — implementation gate — downstream action identity and rate policy are absent

The read-only Toko trace establishes the authenticated subject/user/capability
boundary, complete current mint call graph, signed token-expiry ceiling, and
absence of old Canic receipt rows. It also proves the remaining gaps: there is
no per-mint action identity, recovery endpoint, ledger replay identity/receipt,
explicit batch bound, or average/burst rate policy. The parent-only stack mint
is a second effect surface that could bypass future batch protection.

This is not fixed in Canic. The Toko owner must provide those facts. Until
then, the action vectors, terminal grace, capacity envelope, and product
implementation remain blocked.

## Hard-Cut And Scope Decisions

- No alias, overload, optional deadline, fallback reader, or migration shim is
  permitted.
- The reviewed Toko snapshot cannot contain old application receipt rows: it
  uses Canic 0.71.3 and has no receipt API consumer. Its eventual adoption may
  therefore hard-cut directly to the 0.96 input without a reader, migration,
  or reinstall for this state. Any unreviewed downstream deployment remains an
  explicit hard-cut boundary and Canic does not invent a historical deadline.
- The common primary map remains the receipt authority. Application replay
  metadata may be an exact adjunct because placement shares allocation 43 and
  must not invent an application deadline.
- No second primary receipt store, generic transaction framework, generic
  stable allocator, Toko domain model, ledger call, or background pending
  resolver is admitted.
- Stable snapshot helpers remain test-only. 0.96 does not add a public
  export/import API merely to test state invariants.

## Gate Verdict

The Canic-side and current Toko entrypoint inventories are complete and product
behavior remains unchanged. 0.96 product implementation is **not yet
admitted**. The next action is to freeze the per-action downstream contract,
remove or cover the stack bypass in Toko's later adoption, and establish a
rate/backlog bound before measuring the smallest stable-capacity mechanism.
