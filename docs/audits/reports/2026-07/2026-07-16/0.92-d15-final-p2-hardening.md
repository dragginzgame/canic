## Dependency Risk Contract

The locked graph still contains four reachable unmaintained transitive
packages. They are not direct Canic dependencies, have no known vulnerability,
and remain introduced by current IC, Candid, and PocketIC owners. Replacing
them locally would require vendoring, forks, disabled maintained features, or
an ecosystem migration, none of which is a bounded P2 fix.

Canic now retains one exact inventory containing advisory ID, warning class,
package name, version, registry checksum, and the complete immediate
introducer-name set. The gate uses pinned `cargo-audit 0.22.2` and rejects:

- any known vulnerability;
- any added or removed informational advisory;
- advisory class, package, version, or checksum drift;
- a warned package becoming a direct workspace dependency;
- a missing, added, or ambiguous immediate dependency introducer; and
- an unavailable or wrong `cargo-audit` version.

The gate runs in CI and in the maintainer patch-release transaction. Its
negative fixtures prove rejection of a vulnerability, an added warning, a
stale inventory entry, and warned-package identity drift.

`CANIC-092-DEPENDENCY-001` therefore moves from `deferred` to `accepted`. The
underlying upstream maintenance limitation is not falsely called fixed; its
exact current boundary is controlled, any graph or advisory change requires
fresh review, and removal remains automatic evidence that the inventory must
be updated.

## Auth Ownership Decomposition

The frozen complexity baseline identified three auth/root-proof production
hubs above 600 logical lines. D15 splits them only along responsibilities that
already existed:

- delegated-token ops retain one small facade/orchestrator, with separate
  verifier-configuration, proof-verification/cache, typed-error, and test
  owners;
- delegated-token preparation retains workflow orchestration and lazy repair,
  while request admission and replay identity/commit encoding have dedicated
  workflow children; and
- chain-key batch preparation retains selection and batch construction, while
  installation planning, outcomes, and persisted install progress belong to
  the existing installation child.

The production logical-line change is:

| Frozen hotspot | Baseline logical LOC | Current parent | Current focused children |
| --- | ---: | ---: | --- |
| delegated-token ops | 1,120 | 215 | config 267; verification 224; typed errors 119 |
| delegated-token prepare | 849 | 510 | replay 300; admission 82 |
| chain-key batch | 607 | 475 | install 259; signing 192; merkle 94; selection 75; batch ID 65 |

Every resulting production owner is below the retained method's 600 logical
line hub threshold. The delegated-token tests are a separate test-only module,
so they do not inflate production ownership. Authorization decisions, replay
semantics, proof formats, call order, persistence, typed causes, diagnostics,
public APIs, configuration, and stable state are unchanged.

This is an ownership reduction, not a claim that cryptographic verification
or auth state machines are intrinsically simple. `CANIC-092-COMPLEXITY-001`
is fixed because the three concrete production gravity wells named by the
finding no longer exist and no competing authority or duplicate flow was
introduced. The frozen method must be rerun on the eventual committed source
identity before release reconciliation.
