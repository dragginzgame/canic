# 0.92 D13 Workspace-Only Release Lock Synchronization

## Verdict

- Result: pass for the accepted D13 affected scope.
- Finding: `CANIC-092-RELEASE-005` is fixed in the current candidate.
- Parent release: `v0.92.11`.
- Product behavior: unchanged.
- Compatibility paths added: none.

Slice E compatibility accounting found that the human-owned 0.92.11 version
bump ran `cargo generate-lockfile`. Although no dependency declaration changed,
that command re-resolved the complete graph and advanced six external packages.
A version-only release operation therefore had a second, unrelated dependency
effect.

D13 replaces complete lockfile regeneration with Cargo's workspace-only,
offline update. The release bump still synchronizes every workspace package
entry, but already locked external identities remain fixed unless a workspace
manifest change makes that impossible. The existing transaction continues to
restore all manifests, the lockfile, and the installed-CLI default when any
later synchronization step fails.

## Fail-Closed Contract

The maintained release sequence is now:

```text
version release gates
  -> cargo set-version --workspace
  -> cargo update --workspace --offline
  -> synchronize non-Cargo version surfaces
  -> reject tag collision
  -> commit the complete transaction or restore every release surface
```

- Offline mode prevents registry refresh during the version transaction.
- Workspace-only mode preserves already locked external packages when the
  version-only manifest change does not require a graph change.
- A required external resolution that is unavailable offline rejects and
  triggers the existing transaction rollback.
- The release-integrity guard requires the workspace-only offline command.
- No second command, compatibility path, or broad dependency-update fallback
  remains in the version bump.
