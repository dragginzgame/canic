## Canonical Package Documentation

The `canic` package README now has one manifest-aligned feature contract for
all 14 maintained non-default feature keys. It identifies `metrics` as the
sole default and documents the narrower store, ICP refill, blob-storage,
sharding, and chain-key/auth selections without inventing alternate roles.

The `canic-control-plane` README now documents both maintained features and
their default relationship: `root-control-plane` implies
`wasm-store-canister`, while a store-only consumer disables defaults and
selects only `wasm-store-canister`. A positive manifest test derives each
feature/default expectation from the owning Cargo manifest and requires the
published README table to match it.

## Hard-Cut CLI Proof

The installed and packaged CLI proofs retain positive assertions for the
current `auth renewal status` command, its JSON schema, and diagnostic
projection. Negative assertions naming the deleted `run-once` and
`provisioner` commands are removed from both the shared shell proof and the
CLI unit test. The active release-probe inventory now states only current
proof boundaries; its obsolete command-form guard block is deleted.

Historical changelogs, archived designs, and frozen reports are unchanged.
They remain historical evidence, not active command authority.

## Public Rustdoc Boundary

The public `RpcApi` documentation describes preservation of typed internal
failures and the single public `Error` conversion boundary without linking to
the crate-private `InternalError` type. Visibility and error conversion code
are unchanged. Warning-as-error `canic-core` rustdoc now passes.

## Packaged-Proof Correction

The affected packaged CLI rerun exposed stale pre-publish dependency patching
in both downstream package proofs. `canic-host` now resolves its local
`canic-control-plane` package explicitly, and the Wasm-store proof packages
the canonical crate with all local Canic-family dependencies available before
that version exists on crates.io.

The canonical and generated Wasm-store scenarios share one disposable Cargo
patch-config renderer. One config sits beside the extracted registry-style
packages for metadata validation; the other sits at the downstream build root
for Cargo's build-time lookup rules. The proof still rejects repository crate
paths and builds only from extracted package archives. This changes validation
infrastructure only; package manifests and product dependency resolution are
unchanged.
