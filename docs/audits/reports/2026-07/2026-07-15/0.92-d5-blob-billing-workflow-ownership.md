# 0.92 D5 Blob-Billing Workflow Ownership

## Verdict

D5 passes focused validation. Blob-storage billing now follows one canonical
path:

```text
authenticated endpoint -> thin API projection -> billing workflow
  -> pure billing policy -> single-step ops/state access -> public DTO
```

This fixes `CANIC-092-LAYERING-001` and moves the current
`TRACE-BLOB-001` rerun from fail to pass. Cashier sequencing, funding-guard
lifetime, retry recovery, gateway synchronization, and readiness observation
are workflow-owned. Policy owns only deterministic validation and decisions;
the API delegates and maps typed workflow failures once.

No billing protocol, price policy, public DTO, Candid shape, stable record,
migration, alias, shim, fallback, deprecated API, or duplicate billing path was
added.

## Canonical Ownership

- Generated endpoint guards continue to authenticate controller and gateway
  callers before the API is reached.
- `BlobStorageApi` delegates immediately and maps typed workflow failures to
  the existing public `ErrorCode` contract.
- `BlobStorageBillingWorkflow` owns configuration sequencing, Cashier calls,
  funding-guard lifetime, reserve/top-up orchestration, gateway replacement,
  readiness observation, and public projection.
- Pure billing policy validates configuration and funding requests and decides
  attachment/readiness outcomes without DTO, storage, async, timer, or IC
  dependencies.
- Ops retains boundary conversion, single Cashier/IC calls, funding-guard
  access, and single-step storage mutation.
- Existing model/view and stable billing records are unchanged.

Source inspection finds no production ops dependency or Cashier sequencing in
the billing API and no API dependency in the workflow or pure policy.

## Failure and Recovery Contract

| Boundary | Existing result | State/effect result |
| --- | --- | --- |
| missing or invalid config | `InvalidInput` | no config mutation or Cashier call |
| zero funding request | `InvalidInput` | no cycles attached |
| reserve violation | successful skipped report | no partial attachment or Cashier call |
| concurrent funding | `Conflict` | second request performs no top-up |
| malformed Cashier balance | `InternalRpcMalformed` | readiness reports malformed balance |
| unavailable Cashier balance | existing internal result | readiness reports unavailable balance |
| transient Cashier failure | existing typed public mapping | funding guard releases for retry |
| invalid gateway list | existing typed public mapping | prior gateway authority remains intact |

Configuration-validation order, blocker/warning order, all-or-nothing reserve
behavior, and top-up error codes match the parent implementation.
