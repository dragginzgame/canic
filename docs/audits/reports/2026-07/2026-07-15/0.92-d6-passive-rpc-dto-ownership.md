# 0.92 D6 Passive RPC DTO Ownership

## Verdict

D6 passes focused validation. Root RPC DTOs now contain boundary data and
neutral constructors only. The maintained capability path is:

```text
authenticated capability endpoint -> passive envelope/request DTO
  -> workflow RootCapability mapping, proof admission, replay, and dispatch
  -> ops wire conversion and single-step calls -> replay model/storage
  -> existing public response
```

This fixes `CANIC-092-LAYERING-002`. One workflow-owned `RootCapability` now
owns request-family classification, capability descriptors, replay payload
identity, and admitted replay metadata. Ops owns the mechanical conversion to
the signed wire payload. The replay-policy completeness guard inspects that
actual runtime command enum rather than a duplicate DTO declaration.

No capability version, proof protocol, request variant or field, Candid shape,
replay hash/domain, public error, metric label, stable record, migration, alias,
shim, fallback, deprecated API, or duplicate command path changed.

## Canonical Ownership

- Endpoint and generated-macro authentication remains before capability
  workflow entry.
- RPC request DTOs retain their exact wire variants and fields. They no longer
  classify capabilities, mutate replay metadata, decide signed payload shape,
  expose upgrade-only accessors, or declare a second command enum.
- Workflow maps the passive request once into `RootCapability`. That internal
  command owns the four families, stable descriptor labels and keys, replay
  payload discriminants, replay identity hashes, and attachment of admitted
  replay metadata.
- Capability envelope verification, proof verification, replay admission, and
  handler dispatch consume the same workflow command instead of rematching a
  boundary DTO through competing helpers.
- Ops extracts source metadata and projects the signed wire payload without
  assigning authorization or replay-policy meaning. The projection strips
  replay-only metadata for every request variant.
- The existing replay model remains the authority for receipt identity,
  lifecycle, persistence, capacity, expiry, conflict, and recovery state.
- The replay-policy manifest now checks the runtime workflow command variants,
  so policy completeness cannot pass because of an unused DTO mirror.

Direct source inspection finds no removed request-family type, duplicate DTO
command, or canonicalization helper in the RPC DTO module. No compatibility
name or fallback was retained.

## Failure and Recovery Contract

| Boundary | Maintained result | State/effect result |
| --- | --- | --- |
| invalid envelope or proof | existing typed public rejection | handler and replay state untouched |
| missing replay metadata | existing non-replayed execution path | no fabricated identity |
| identical request ID and payload | cached prior response | no duplicate side effect |
| same request ID with changed payload or family | replay conflict | conflicting handler is not executed |
| expired or invalid TTL | existing typed replay rejection | no receipt or side effect admitted |
| handler failure before effect | existing abort path | reservation becomes retryable as designed |
| receipt commit failure after effect | recovery required | durable recovery evidence is preserved |

The mapping move changes no error conversion, authorization order, replay
domain separator, variant discriminant, or recovery transition.
