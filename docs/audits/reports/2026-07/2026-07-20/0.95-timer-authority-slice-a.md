## Reproduced Findings

### `CANIC-095-TIMER-001` — P1 — async interval overlap

`TimerOps::set_interval` uses the CDK interval callback to create a new future
on every tick. It has no running bit, generation, or completion arbitration.
The closure borrow ends before the future is awaited, so another tick can
start while the earlier future is suspended. Cycle funding and delegated-proof
renewal can therefore overlap external work even though their handle slot is
guarded.

Smallest hard-cut fix: replace async fixed-rate execution with common
after-completion scheduling and explicit running arbitration. A fixed-rate
mode is not retained because no owner proves that requirement.

### `CANIC-095-TIMER-002` — P1 — stale guarded state and lost reschedule

`TimerOps::set_guarded` stores a handle but deliberately does not remove it
when its one-shot fires. Pool and placement manually clear their slots, while
the public guarded macro does not. A completed public guarded timer therefore
remains registered in its slot and every later request is silently refused.

Smallest hard-cut fix: common infrastructure consumes the handle before domain
work and merges requests made while running. Remove the unused guarded public
forms and all subsystem-owned slot clearing.

### `CANIC-095-TIMER-003` — P2 — false live status

`TimerMetrics::ensure` permanently retains every scheduled `(mode, delay,
label)` row. `api::runtime::timer_statuses` maps every historical row to
`enabled: true` and `registered: true`, including completed one-shots and
cleared timers. Existing tests assert this historical behavior rather than
live registration truth.

Smallest hard-cut fix: project registration and process condition from the
common live registry. Historical counters remain metrics and are not treated
as handles.

### `CANIC-095-TIMER-004` — P2 — unnecessary idle wake-ups

An empty root wakes log retention after 30 seconds and every 10 minutes, and
pool scheduling after 30 seconds and every 30 minutes. Count and byte log
limits are already enforced on append; without an age policy, the log interval
has no future due-work reason. Pool pending records are created through known
producer paths. A TTL-free local intent also arms cleanup once even though
`expirable_pending_total` remains zero.

Smallest hard-cut fix: deadline-drive age retention, event/retry-drive pool
work, and refuse to index or schedule TTL-free intents.

### `CANIC-095-TIMER-005` — P2 — unrelated full scans

Intent expiry iterates the complete pending index before aborting every due
row. Placement acknowledgement pages the shared receipt-backed intent map and
filters placement keys afterward, so one failed acknowledgement can repeatedly
scan unrelated consumers.

Smallest hard-cut fix: an expiry-ordered finite-TTL index and a
placement-owned acknowledgement queue/index, each drained through the generic
bounded due-work contract.

### `CANIC-095-TIMER-006` — P2 — competing mechanical and lifecycle flows

Guarded interval handoff exists in both ops and API. Lifecycle deferral calls
`TimerOps`, `TimerWorkflow`, or `TimerApi` depending on root/nonroot and
init/upgrade path. Public raw CDK timers are also reachable through the CDK
facade, bypassing the measured timer owner.

Smallest hard-cut fix: one workflow-owned scheduler over one direct ops
boundary, one lifecycle facade, and removal of the public raw-timer re-export.

### `CANIC-095-TIMER-007` — P1 — configured root self-refill is unreachable

The cycle workflow contains the maintained timer-driven hub self-refill path,
and the ICP-refill workflow and pure policy explicitly own automatic hub
self-refill. Root lifecycle nevertheless calls `start_standard_only`, whose
mode never invokes `check_hub_self_refill`. A root with enabled ICP refill can
therefore cross its threshold without the configured automatic process ever
running.

Smallest hard-cut fix: give automatic top-up its independent configuration-
gated key and execute the existing root ICP-refill owner on root. Remove the
mode split and dead conditional branches rather than adding another path.

## Frozen Public Surface

The maintained application surface for Slice B is:

- `timer!` for a cancellable one-shot;
- `timer_interval!` for cancellable, after-completion recurrence; and
- one cancellation operation consuming the opaque timer handle.

The surface does not retain `timer_guarded!`, `timer_interval_guarded!`,
`TimerSlot`, direct public raw-CDK timer access, or a fixed-rate async mode.
There are no aliases, deprecated wrappers, or fallback paths. The runtime probe
migrates to the maintained forms and proves current behavior only.

## Frozen Common Decisions

- Built-in keys are a closed internal enum; public application timers use a
  separately owned opaque identity and compile-time label.
- Request and generation sequences are heap `u64` values. Checked overflow
  fails closed rather than wrapping into a current identity.
- Scheduling and cancellation while running are recorded by sequence and
  applied only after completion.
- External requests and callback directives merge to the earliest absolute
  deadline unless a later sequenced cancellation disables the timer.
- Fixed-rate async intervals are removed. Recurrence is after completion.
- Lifecycle deferral, immediate continuation, retry, exact deadline, and
  necessary recurrence are distinct modes.
- Registration and process condition remain separate status dimensions.
- No new stable timer registry is introduced. Subsystem state reconstructs
  timers after lifecycle.
- New intent-expiry and placement-acknowledgement indexes are derived domain
  authority with schema version `1` only if their persisted envelope requires
  a version. No old reader or migration shim is retained.
