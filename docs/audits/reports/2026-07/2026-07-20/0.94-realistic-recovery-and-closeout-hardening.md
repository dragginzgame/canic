## Findings And Hard-Cut Fixes

### `CANIC-094-AUTHORITY-001` — P1

The legacy `canic snapshot download` command and
`canic_backup::snapshot` library independently discovered targets, captured
snapshots, downloaded artifacts, and wrote a different manifest path outside
the durable backup execution journal. That was a competing backup authority
and could bypass 0.94 recovery invariants.

Fix: delete the command, library module, tests, help, dispatch, and active
operator documents. `canic backup create` is the sole capture and resume
owner. No alias, deprecated API, or fallback remains.

### `CANIC-094-PREPARE-001` — P1

`restore prepare` unconditionally replaced `restore-plan.json` and
`restore-apply-journal.json`. Re-running prepare could destroy pending,
failed, completed, or receipt evidence from the only durable restore owner.

Fix: `create_or_adopt_restore_plan` and
`create_or_adopt_restore_apply_journal` create absent documents, adopt only an
exact pristine match, and return typed conflicts without changing an existing
byte. No reset or overwrite path remains.

### `CANIC-094-PRUNE-001` — P1

`canic backup prune --failed` allowed a cleanup command to delete the only
recoverable failure journal and artifacts. Optional keep semantics also made a
broad prune easier than the maintained safety model requires.

Fix: remove `--failed`, require `--keep <count>`, and select only older
`Complete` layouts. Failed, incomplete, invalid, and otherwise recoverable
layouts cannot enter the deletion set. Reported row identities remain those
from the canonical backup list even when statuses are interleaved.

### `CANIC-094-BACKUP-011` — P1

After any selected target had stopped, an execution or preparation failure
could return before the later start operations ran. The real ICP 1.1 inventory
parse failure reproduced this during snapshot preparation and left the
selection unavailable.

Fix: `failure_after_containment` preserves the primary typed cause, reconciles
a possibly committed stop, advances only the canonical start operations for
completed stops, records their receipts, and preserves a second typed cause if
containment fails. Snapshot-phase recovery re-arms stop/start so retry again
quiesces the entire selected set.

### `CANIC-094-RESTORE-002` — P1

All interrupted mutating restore operations previously converged on a generic
unknown-outcome rejection after command quiescence. That was fail-closed but
not resumable and discarded available effect-specific evidence.

Fix: the version-1 apply journal carries an exact pre-upload snapshot
inventory. Restart adopts only one exclusive new upload identity; zero new
identities justify one upload and multiple identities reject. Stop and start
reconcile from exact target status, load repeats the exact snapshot only while
stopped as proven by the capability gate, and read-only verification repeats.
The generic unknown-outcome branch is removed.

### `CANIC-094-VERIFY-001` — P1

Restore verification treated a successful `icp canister status` command as
proof even when output did not establish running state or the expected module.
The runner could therefore report a completed restore without semantic
restored-state evidence.

Fix: verification requires structured JSON, `Running` status, and an exact
normalized source module hash whenever the backup provides one. Missing,
unparseable, stopped, or mismatched evidence becomes a durable failed
operation and receipt.

### `CANIC-094-ICP-001` — P1

ICP CLI 1.1 returns snapshot inventory as `{"snapshots":[...]}`. Canic's 1.0
bare-array parser rejected the real response after targets were stopped.

Fix: both host backup and restore reconciliation use the one strict wrapped
1.1 response model. Runtime support is `>=1.1.0,<2.0.0`; the repository pin is
1.1.0 with exact upstream archive hashes. There is no legacy parser fallback.

## Realistic Journey

The existing six-role `test` fleet was installed through the maintained host
path. The selected backup subtree contained:

- root `t63gs-up777-77776-aaaba-cai`;
- `user_hub` `tf62x-ox777-77776-aaadq-cai`; and
- child `user_shard` `siq6z-b7777-77776-aaaea-cai`.

The two selected test members expose deterministic fixture-only recovery state.
Both were set to and queried as generation `A`. The first maintained backup
attempt durably stopped both members, then rejected the unexpected ICP 1.1
snapshot response. The failure evidence remained and containment restarted
both members. After the parser fix, the same backup plan and journal resumed
instead of creating a new authority. It completed 15 operations with two
snapshot artifacts and passed offline verification.

Both members were changed to and queried as `B`. `restore prepare` created ten
ordered operations. One process executed only the first operation; a second
process resumed the same journal and completed the remaining nine. Direct
application queries returned `A` from both restored members. Verification of
the source backup passed again. Re-running both completed operations executed
zero additional backup or restore operations.

The final live backup journal has 15 completed operations and 11
post-preflight receipts. Its restore journal has ten completed operations and
ten receipts. The disposable network was stopped after the proof.

## Real Upload Effect/Receipt Interruption

The ignored Unix operational test
`real_upload_effect_survives_process_death` uses a test-only child executable
around the existing restore runner/executor boundary. It invokes the real
`canic-host` ICP upload against the disposable canisters; no command result is
faked.

The child reaches an acknowledged barrier only after the real upload returns
success and before the runner persists its terminal receipt. The parent sends
`SIGKILL`. The surviving journal is `Pending`, retains the exact pre-effect
inventory, and has no receipt. Restart observes exactly one new snapshot,
issues zero upload commands, binds that identity to the existing operation,
and writes one completed receipt. The remainder of the real restore then
completed normally.

Result: exact frozen case `CANIC-094-R06` passes with recovery disposition
`reconciled`.

## Authority Map

| Concern | Single owner |
| --- | --- |
| Backup orchestration and retry | `canic_backup::runner` through `canic backup create` |
| Backup plan, execution state, and receipts | `BackupLayout` plus `BackupExecutionJournal` |
| Download/checksum/publication durability | backup artifact journal and canonical artifact directory |
| Final backup truth | exact immutable deployment backup manifest derived from durable artifacts |
| Restore preparation | `canic_backup::restore::persistence` create-or-adopt functions |
| Restore orchestration and retry | `canic_backup::restore::runner` through `canic restore run` |
| Restore state and receipts | one `RestoreApplyJournal` durable replacement |
| External command quiescence | sequence-bound kernel `CommandLifetimeLock` inherited only by the intended tree |
| Cleanup | `canic backup prune --keep`, completed layouts only |
| Verification | offline backup verifier; restore runner structured running/module evidence; final application-state query in the realistic journey |
