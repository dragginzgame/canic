## Finding

### `CANIC-094-RESTORE-004` — P2 — fixed

Malformed or unsuccessful stopped-canister status output already became a
durable failed operation and receipt. An executor I/O error instead released
the command lock and returned its typed cause while leaving the already
claimed load `Pending` with no durable explanation.

The runner now writes the same atomic failed-operation/receipt pair before
returning the original typed `RestoreRunnerError::Io` cause. If that durable
write itself fails, its persistence error wins because no durable-failure
claim can be made. No new document field or error-string parser was added.

## Recovery Evidence

the read-only status executor holds its acknowledged barrier. Restart sees no
receipt, rechecks the same target, proves it stopped, and performs exactly one
load. Malformed status and executor I/O cases persist one matching failure
receipt and issue no load.

The existing atomic terminal-write crash is also the exact effect/receipt
boundary for `R07` through `R11`. Explicit case binding now proves:

- committed stop and start are observed through target status and not issued
  again;
- pending load is replayed exactly once only after stopped-state proof; and
- both verification operations repeat their structured read-only check and
  cannot complete without a matching terminal receipt.

For each `R14` mutating variant, the runner process starts an external command
whose descendant retains the command-lifetime descriptor, then dies under
`SIGKILL`. Closing the direct child's descriptor is insufficient: restart
returns the typed in-flight error without calling the executor or changing a
journal byte. Only after the descendant commits its modeled outcome, exits,
and the complete command tree becomes quiescent may reconciliation begin.
