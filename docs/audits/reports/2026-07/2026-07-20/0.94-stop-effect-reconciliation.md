## Case Executed

`CANIC-094-B05/stop/effect-committed-receipt-missing` uses the maintained
backup runner and its command-lifetime lock. The child durably claims `Stop`,
executes a stateful stop effect, synchronizes the resulting `Stopped` state,
signals the acknowledged effect barrier, and is killed with `SIGKILL` before
the runner can construct or persist a terminal receipt.

After process death:

- the command-lifetime lock is quiescent;
- the execution journal retains one pending stop and no matching receipt; and
- the authoritative external state is `Stopped`.

Restart reads the exact target status, writes one normal completed receipt,
and invokes no second stop. A subsequent replay skips the reconciled stop and
adds no duplicate receipt.

Result: `B05` passes with recovery disposition `reconciled`.

## Finding And Fix

### `CANIC-094-BACKUP-002` - P1 - fixed

The runner previously had no status-observation operation in its executor
contract. Once a stop became pending, restart could prove only that the old
command tree was quiescent and returned `CommandOutcomeUnknown` forever, even
when the target was authoritatively stopped. Maintained operator actions could
not complete the journal without forbidden manual editing.

The executor now returns one typed, identity-bound lifecycle state. Pending
stop recovery applies this exact policy after command quiescence:

- `Stopped`: write the normal completion receipt without another stop;
- `Running`: execute one stop using the retained command-lifetime boundary;
- `Stopping`: halt with a typed unsettled-state cause; and
- observation failure, wrong target identity, or an unknown status label:
  halt with the typed observation cause.

No text parsing, default method, permissive fallback, synthetic operator
command, journal rewrite, or compatibility adapter was added.

## Surface And Scope

- `BackupRunnerExecutor` now requires `canister_status`; the typed return is
