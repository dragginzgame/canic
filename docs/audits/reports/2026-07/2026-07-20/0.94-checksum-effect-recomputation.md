## Case Executed

`CANIC-094-B11/verify-artifact/effect-committed-receipt-missing` starts from an
exact `Downloaded` artifact row and a durable pending `VerifyArtifact`
operation. A child computes the real maintained SHA-256 directory checksum,
synchronizes the resulting test evidence outside the backup layout, and is
killed with `SIGKILL` before it can write either the `ChecksumVerified`
artifact transition or the terminal execution receipt.

After restart, the artifact journal still truthfully says `Downloaded`, the
verification operation is still pending, and no checksum is durably claimed.
The runner repeats the read-only secure traversal, obtains the same checksum
from unchanged staged bytes, writes the normal artifact transition and
receipt, and invokes no external command. Completed replay performs no work.

Result: `B11` passes with recovery disposition `resumed`.

## Finding Review

No product finding was opened. The existing runner already distinguishes
read-only pending work from command-backed mutation: a pending verification
has no command-lifetime lock, so it resumes through the canonical verification
operation. The secure checksum implementation opens the exact artifact with
no-follow descriptor traversal and does not mutate its input.

The required proof needed a deterministic process boundary, not another
production recovery state. The added child-process harness calls the real
checksum implementation and pauses only in test code after calculation. No
production failpoint, alternate checksum helper, fallback path, compatibility
surface, or journal field was introduced.

## Rejection And Boundary Proof

- Removing the exact staged directory before resume returns the existing typed

- Adding a symlink inside the staged directory returns the existing typed
  unsupported-entry cause. Descriptor traversal does not follow the link or
  alter its external sentinel.
- Both failures execute no external command and never create a
  `ChecksumVerified` row.
