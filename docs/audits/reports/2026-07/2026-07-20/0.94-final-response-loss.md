## Case Executed

`CANIC-094-B17/final-successful-response/response-lost-after-persistence`
starts a child at a fresh persisted backup plan. The child executes all six
post-preflight operations, publishes and verifies the final manifest, receives
a complete `BackupRunResponse`, and reaches a deterministic handshake before
that response leaves the child. The parent acknowledges the barrier and sends
`SIGKILL`.

The surviving layout passes complete integrity validation. All ten plan
operations are terminal, while the six post-preflight operations retain their
six normal terminal receipts. Restart returns the same run, plan, backup, and
execution-summary identity with `complete: true`, zero executed operations,
and an empty per-invocation operation list.

Result: B17 passes with recovery disposition `resumed`. The replay invokes no
executor method, preserves the execution journal exactly, and leaves every
layout path, file type, and byte unchanged.

## Finding And Implementation

No product defect was found. The runner derives completed response state from
the validated durable plan and execution journal before considering another
ready operation. A fully terminal journal therefore returns completion
without entering preflight, reconciliation, or effect execution.

B17 adds only a Unix test child at the existing runner boundary. It adds no
production failpoint, response cache, receipt type, CLI switch, environment
control, compatibility path, or alternate completion flow.

The first test draft incorrectly expected one operation receipt for all ten
completed operations. Inspection confirmed the four preflight operations use
their dedicated preflight receipt bundle, while the six post-preflight
operations use terminal operation receipts. The corrected assertion preserves
those two canonical authorities; no product code changed for this observation.
