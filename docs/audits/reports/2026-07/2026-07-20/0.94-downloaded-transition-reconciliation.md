## Cases Executed

The two `CANIC-094-B10/downloaded-artifact-transition/*` cases start from a
durable pending download with synchronized snapshot bytes in the exact private
temporary directory. A child constructs the valid `Downloaded` artifact row
and is killed at one of the persistence handshakes.

### Before durable write

The child is killed after the replacement JSON is synchronized but before its
rename. The canonical artifact journal therefore remains exactly `Created`.
Restart does not infer completion from uncommitted staged bytes: it replaces
the private directory, performs one justified download, publishes
`Downloaded`, records one normal download receipt, and proceeds to checksum
verification.

Result: `before-durable-write` passes with recovery disposition `resumed`.

### After durable write

The child is killed after rename and parent-directory synchronization. The
canonical journal is exactly `Downloaded`, while the execution operation is
still pending and has no terminal receipt. Restart validates the complete
artifact identity and staging directory, reconstructs the same receipt used
by the normal download path, invokes no download command, and proceeds to
checksum verification over the preserved bytes.

Result: `after-durable-write` passes with recovery disposition `reconciled`.

## Finding And Fix

### `CANIC-094-BACKUP-006` - P1 - fixed

At `v0.94.6`, exact `Downloaded` evidence was deliberately preserved but the
B09-only recovery policy rejected it. A process death after the artifact
journal became durable but before the execution receipt therefore left a
valid backup permanently halted.

Pending download recovery now has one canonical state-driven decision:

- exact `Created` returns to the existing fresh-staging download path;
- exact `Downloaded` validates the staging authority and returns the normal
  download receipt; and
- every later or conflicting state remains typed and fail-closed until its
  assigned recovery case is implemented.

The normal and recovered paths share one receipt constructor. Both require the
same target, snapshot identity, canonical artifact path, exact temporary path,
and real non-symlink staging directory. Ordinary download also revalidates the
directory after the external command and before publishing `Downloaded`.

No alternate journal reader, duplicate downloader, fallback state, synthetic
completion flag, compatibility surface, manual journal edit, or operator
escape hatch was added.

## Rejection And Boundary Proof

- A durable `Downloaded` row with a missing staging directory returns the
  typed `ArtifactTempPathMissing` cause before receipt reconstruction.
- Snapshot identity, canonical artifact path, and temporary-path mismatch each
  preserve its existing typed cause, execute no command, and leave the pending
  execution journal unchanged.
- Symlink, file, and special staging roots remain rejected by the shared
  filesystem-entry validation.
- The checksum step uses the existing secure no-follow artifact traversal and
  is not folded into download reconciliation.

## Surface And Scope

- The existing backup runner, execution journal, artifact journal, and secure
  checksum path remain the only authorities involved.
- `BackupRunnerError` gains the public typed `ArtifactTempPathMissing` variant.
  This pre-1.0 change has no legacy variant or compatibility mapping.
- Persisted plan, execution-journal, artifact-journal, receipt, and manifest
  shapes are unchanged. Their genuine version fields remain `1`.
- CLI syntax, JSON output, Candid, and Cargo package versions are unchanged.
- Checksum interruption, checksum-verified publication, durable artifact
  publication, manifest recovery, terminal receipts, and final response loss
  remain assigned to later frozen cases.
