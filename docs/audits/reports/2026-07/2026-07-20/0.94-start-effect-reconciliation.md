## Case Executed

`CANIC-094-B08/start/effect-committed-receipt-missing` runs the maintained
backup runner in a child process after stop and snapshot operations have
completed. The runner durably claims `Start`, commits one stateful start
effect, synchronizes the resulting `Running` state, and is killed with
`SIGKILL` before it can persist the terminal execution receipt.

After command quiescence, restart observes the exact target as `Running`,
writes one normal completed receipt, and invokes no second start. Completed
replay executes no command and appends no receipt.

The retained `B04` after-claim case also proves the inverse boundary: exact
`Stopped` status establishes that the pending start did not commit and

## Finding And Fix

### `CANIC-094-BACKUP-004` - P1 - fixed

At `v0.94.4`, pending start operations did not use the lifecycle-status
authority already maintained for stop recovery. Once a start became pending,
restart returned an unknown-outcome error permanently even when the target was
authoritatively running. A start command that committed before returning a
failure could also enter retry without first reconciling that external state.

Stop and start now use one lifecycle reconciliation policy after command
quiescence:

- the operation's exact completion state reconstructs its normal receipt;
- the opposite stable state justifies one command;
- `Stopping` rejects with the typed unsettled-state cause; and
- observation failure preserves its typed status and message.

Failed lifecycle attempts are reconciled before automatic retry. The same
rule applies after the explicit failed-to-ready transition by consulting the
durable failed receipt. A regression executor rejects any unexpected second
start, proving that neither path blindly repeats a committed effect.

No duplicate start-specific recovery flow, default status, string parser,
compatibility adapter, journal edit, or operator escape hatch was added.
