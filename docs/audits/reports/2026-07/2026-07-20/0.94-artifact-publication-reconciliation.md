## Finding And Fix

`CANIC-094-BACKUP-008` (P1) found that initial publication synchronized the
temporary tree but did not recompute its checksum. The publisher trusted the
earlier `ChecksumVerified` value, renamed the current bytes, advanced the
journal to `Durable`, and built a manifest containing the old hash. Staging
changed between verification and publication could therefore produce a
successful but internally inconsistent backup.

Canonical publication now computes the maintained descriptor-based checksum
while synchronizing either the temporary or canonical tree and verifies it
against the journal before returning success. The same checksum owner and
same no-follow traversal are used for first publication and recovery; no
second publication flow or fallback was added.

A test-only barrier exposes only the existing publication boundary before
rename and after parent-directory synchronization. It adds no production
failpoint, CLI option, environment variable, or hidden recovery command.

## Rejection And Boundary Proof

- Changing checksum-verified private staging before finalization returns the
  typed checksum mismatch, leaves the operation explicitly failed and the
  artifact `ChecksumVerified`, retains private staging, and creates neither a
  canonical directory nor manifest.
- Supplying both private staging and a conflicting canonical directory returns
  the typed path-conflict cause without overwriting either directory.
- An already published canonical directory with changed bytes is rejected by
  the existing runner regression and never advances the journal.
- The `Durable` artifact-journal transition, manifest publication, terminal
  receipt boundaries, and final response loss remain assigned to later frozen
  cases.

## Surface And Scope
