# 0.92 Layer Boundary Baseline

## Verdict

- Run result: `fail`
- Result validity: `invalid`; post-freeze defect discovered by
  `CANIC-STRUCTURE-001/v1`.
- Architecture risk score: **5 / 10** (moderate).
- Static scan confidence: high.
- Manual-review confidence: high for the reported API/DTO owners; medium for
  the broad ops pressure review.

Hard import and policy-purity guards pass, but responsibility does not
consistently follow those imports. Blob-storage billing performs multi-step
business orchestration inside the endpoint API layer, and the root RPC DTO owns
capability-family and signed/replay-payload canonicalization behavior. The
active API documentation and the frozen method also permit direct API-to-ops
delegation while `AGENTS.md` defines a strict endpoint-to-workflow direction.

### Post-run invalidation

The later frozen
[module-structure run](0.92-module-structure-v1.md) proved that 25 production
ops files import `domain::policy`, including grouped imports and direct policy
decision calls. The v1 suggested regex does not match grouped
`use crate::{ domain::policy::... }` forms, and the executable guard contains
no ops-to-policy rule. Therefore the lower-to-higher `PASS`, the 5/10 risk
score, and the result validity are superseded by findings
`CANIC-092-AUDIT-012` and `CANIC-092-LAYERING-005`. The v1 definition is not
silently edited; a corrected versioned method must rerun the immutable
baseline.

## Responsibility and Pressure Review

- `api/blob_storage/billing.rs` (463 lines) is the primary API gravity well.
- `dto/rpc.rs` (405 lines including local tests) is an escalating DTO hub
  because it owns capability/replay behavior as well as transport data.
- `workflow/rpc/request/handler/mod.rs` (307 lines) is a central orchestration
  module but keeps side effects behind ops and is not a finding in this run.
- `workflow/runtime/mod.rs` (80 lines) remains the architecture watchpoint but
  is currently a thin module/adapter surface, pressure score 3/10.
- Auth, blob-storage, Cashier, IC, runtime, and stable-storage ops were sampled
  where the API/DTO traces reached them. No new ops-owned business-policy
  finding was confirmed.
- Public error DTO references remain widespread across explicit public RPC
  boundaries. This run did not prove an additional storage/model error leak.

No enum shock-radius, capability-public-surface, lifecycle, view, public-record,
or policy-side-effect addition is applied to the risk score. DTO fan-in is
expected; DTO-owned capability decisions are not.
