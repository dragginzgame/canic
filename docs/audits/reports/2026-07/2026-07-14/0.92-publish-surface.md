### Feature-documentation pressure

`crates/canic/Cargo.toml` publishes `wasm-store-canister`, `icp-refill`,
`blob-storage`, `blob-storage-billing`, `auth-chain-key-ecdsa`, and
`auth-chain-key-root-sign`, but `crates/canic/README.md`'s stated optional
feature list omits all six. A downstream user can discover them in Cargo
metadata, but the primary package guide presents an incomplete selection.

`crates/canic-control-plane/Cargo.toml` defaults to `root-control-plane` and
`wasm-store-canister`, with the former implying the latter. Its package README
describes both responsibilities but does not explain the feature split or how
lower-level consumers opt out. This is package-contract pressure, not an
unsupported-use violation.

### Architecture documentation mismatch

`crates/canic-core/README.md` and `crates/canic-core/src/lib.rs` describe the
flow as endpoints -> workflow -> domain/decision helpers -> ops ->
storage/model and assign stable schemas to storage. `AGENTS.md` and
`docs/contracts/ARCHITECTURE.md` define the canonical direction as endpoints
-> workflow -> policy -> ops -> model, with authoritative state/storage
invariants owned by model. Public lower-level package docs therefore teach a
competing layer model.
