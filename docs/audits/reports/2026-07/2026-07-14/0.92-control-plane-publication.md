# 0.92 Control-Plane Publication Supporting Trace

## Verdict

- Supporting inspection result: `fail`.
- Evidence validity: valid for the recorded code facts and focused test
  outcomes.
- Mandatory trace validity: not admitted; `TRACE-CONTROL-001` remains blocked
  by `CANIC-092-AUDIT-010`.
- Authorization, exact-release reuse, conflict rejection, and completed
  post-upgrade reconciliation: pass in focused evidence.
- Durable-publication quota/cycle-reserve enforcement: fail.

The root/store publication flow has one canonical workflow and converges from
the target store's exact release catalog back into root-owned manifest state.
Conflicting same-key releases reject before the sampled fleet changes, direct
store updates reject non-root callers, and a completed multi-store publication
binding survives root upgrade.

However, the public publication endpoints are marked `Implemented` with the
`DurablePublish` cost class, quota policy, and cycle-reserve policy while no
publication workflow reserves or carries a `CostGuardPermit`. The root can
therefore send chunks, call management `upload_chunk`, promote manifests, and
mutate root publication state without the declared durable-publication quota
or reserve boundary. Store creation itself uses the separate deployment cost
guard; that does not cover the publication work around it.

## End-to-End Readout

### Admission and authorization

`crates/canic/src/macros/endpoints/root.rs:135-169` places controller guards on
root staging, publication admin, and overview endpoints. The target wasm-store
endpoints use root-caller guards. The focused direct-call test proves the root
can prepare on the store while an anonymous/non-root caller receives a typed
authorization rejection.

### Placement, conflict, and lookup

`fleet/placement.rs:79-111` deterministically orders writable stores and rejects
same template/version material when role, hash, or size differs. Exact material
is reused. Fixed-target publication uses the same snapshot helpers but
`managed.rs:243-251` collapses both conflict and capacity into `cannot fit`.

Publication then resolves source chunks, prepares the target, sends each chunk,
calls management `upload_chunk`, and stages the approved manifest. Only after
the target stage succeeds does `promote.rs:88-91` replace root-owned approved
state. On retry, an exact target release causes root state to be mirrored,
providing a deterministic convergence path.

### Reconciliation and recovery

`release/catalog.rs:56-86` requires at least one exact fleet release and chooses
the existing manifest binding, then preferred binding, then the oldest/binding
ordered candidate. `import_current_store_catalog` replaces only the root-owned
binding projection. The focused upgrade test confirms a completed canary
release remains on its current multi-store binding after root upgrade.

No test interrupts the flow after the target store accepts the manifest but
before root mirroring, and no test makes a later manifest in a multi-manifest
set fail after an earlier new manifest is published. The code has a plausible
convergence mechanism, but the important partial-failure boundaries remain
unproved.

### Declared versus enforced cost boundary

`crates/canic-core/src/replay_policy/endpoint_manifest.rs:105-124` classifies the
root template staging/publication endpoints as monotonic durable publication.
The constructor at lines 249-261 marks each entry `Implemented` and attaches
both durable-publish quota and cycle-reserve policies.

The only durable-publish test at
`crates/canic-core/src/replay_policy/tests/cost.rs:70-104` verifies that those
labels exist. Repository-wide permit use shows no `CostGuardOps::reserve` or
`CostGuardPermit` in the publication workflow. The structural boundary test
checks deployment and value-transfer adapters, not durable publication.
`release/chunks.rs:140` calls management `upload_chunk` directly, and the
root/store publication calls and state mutations likewise take no permit.
