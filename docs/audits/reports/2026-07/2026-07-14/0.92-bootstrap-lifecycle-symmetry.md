The frozen lifecycle invariant holds. Generated `init` and `post_upgrade`
hooks synchronously restore the required config, memory, environment, and
runtime state, then schedule bootstrap and user work through zero-delay timer
boundaries. Root and non-root paths preserve the same high-level phase order,
and failure during synchronous restoration traps before continuation can be
scheduled.

No lifecycle adapter was found executing bootstrap orchestration inline,
mutating stable storage directly, or embedding domain policy. Root lifecycle
still carries more bounded adapter responsibility than non-root lifecycle
because embedded wasm-store sources and the template resolver are registered
around the core restore call, but bootstrap execution remains owned by the
scheduled workflow.

## Contract Checks

### 1. Macro hooks stay thin: pass

- `crates/canic/src/macros/start.rs:7-80` wires non-root init/upgrade to
  synchronous before-bootstrap APIs, then zero-delay Canic and user timers.
- `crates/canic/src/macros/start.rs:205-279` follows the same shape for root.
- `crates/canic/src/macros/start.rs:282-299` runs an optional user init block
  in a lifecycle timer and only then schedules the maintained continuations.
- `crates/canic/src/macros/start.rs:363-386` derives ordinary root/non-root
  dispatch from build metadata cfgs.
- `crates/canic/src/macros/start.rs:400-452` keeps `start_local!` and
  `start_wasm_store!` explicit special modes that reuse the same lifecycle
  timer boundary.

The `.await` matches in macro source occur only inside futures passed to timer
helpers; no IC lifecycle entrypoint awaits them.

### 2. Lifecycle API is delegation-oriented: pass

- `crates/canic-core/src/api/lifecycle/root.rs:10-34` directly delegates root
  before-bootstrap work to lifecycle adapters.
- `crates/canic-core/src/api/lifecycle/nonroot.rs:12-52` delegates non-root
  restore and schedule calls without adding policy or state mutation.
- `crates/canic-control-plane/src/api/lifecycle.rs:18-93` performs the bounded
  root-specific source/resolver registration, delegates core restoration,
  records lifecycle metrics, and places bootstrap workflow calls only inside
  zero-delay timer closures.

### 3. Init and post-upgrade are structurally symmetric: pass

| Flow | Synchronous phases | Async boundary |
| --- | --- | --- |
| Root init | compiled config -> memory/env/runtime init | root bootstrap timer -> user timer |
| Root upgrade | compiled config -> memory registry -> root env -> runtime continuation | root bootstrap timer -> user timer |
| Non-root init | compiled config -> memory/env/index/runtime init | non-root bootstrap timer -> user timer |
| Non-root upgrade | compiled config -> memory registry -> role env -> runtime continuation | non-root bootstrap timer -> user timer |

The upgrade-only memory and environment restoration is the documented
structural difference. The root control-plane scheduling split is explicit:
core owns restoration, while control-plane owns the root bootstrap timer.

### 4. Lifecycle adapters remain synchronous: pass

- `crates/canic-core/src/lifecycle/init/root.rs:12-51` and
  `init/nonroot.rs:17-58` contain no async entrypoint or await.
- `crates/canic-core/src/lifecycle/upgrade/root.rs:12-72` and
  `upgrade/nonroot.rs:20-83` contain no async entrypoint or await.
- Await points in `init/nonroot.rs:61-97`,
  `upgrade/nonroot.rs:87-124`, and control-plane scheduling helpers occur only
  within futures passed to zero-delay timer APIs.

The source guard verifies all six root/non-root before-bootstrap adapters lack
await, async blocks, timer calls, bootstrap workflow calls, and schedule calls.

### 5. Restore precedes bootstrap scheduling: pass

- Root and non-root upgrade adapters initialize the memory registry before
  `EnvOps::restore_root` or `EnvOps::restore_role`, then complete runtime
  continuation before returning.
- The generated macros call the synchronous before-bootstrap adapter before
  invoking any schedule helper.
- Each failed config, memory, environment, or runtime step calls the lifecycle
  trap path and therefore cannot return to the macro continuation.
- The PocketIC fixture proves phase-correct init/post-upgrade failure and
  repeated successful non-root upgrade readiness.

### 6. Lifecycle/workflow layering holds: pass

The lifecycle adapters import no stable-storage schema and no domain policy.
Their approved ops use are environment restoration, bootstrap status, timer
scheduling, and trap timestamping. Bootstrap workflow calls appear only in
scheduled timer closures. Runtime initialization remains in
`workflow::runtime`, and multi-step bootstrap remains in
`workflow::bootstrap`.

### 7. Timer and bootstrap coverage: pass

- `crates/canic-core/tests/lifecycle_boundary_guard.rs` structurally checks all
  root/non-root before-bootstrap adapters and all four schedule helpers.
- `crates/canic-tests/tests/lifecycle_boundary.rs` executes install,
  phase-correct init rejection, phase-correct post-upgrade rejection, and
  three repeated successful non-root upgrades in PocketIC.
- The separately recorded focused root reconciliation execution at the same
  immutable snapshot proves a completed root post-upgrade remains ready and
  preserves its current multi-store release binding.

There is no dedicated PocketIC test whose sole assertion is root lifecycle
timer ordering. That ordering is covered structurally, while real root install
and post-upgrade execution is covered by root integration fixtures. This is a
watchpoint rather than a significant evidence gap under the frozen method.
