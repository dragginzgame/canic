# 0.90 Closeout Audit - 2026-07-13

## Scope

- Snapshot: published `v0.90.2` at `4289dcc7`.
- Reviewed: receipt-backed model and public facade, exact-key storage,
  resource accounting, stable-memory ownership, lifecycle/timer behavior,
  metrics, hard-cut surfaces, focused tests, and downstream handoff wording.
- Excluded: Toko implementation, network calls, full workspace tests, broad
  PocketIC suites, deployment, and a new release-line design.

## Verdict

- Result: **PASS**.
- Residual risk: **2 / 10** inside Canic's owned boundary.
- No `0.90.3` correction is justified by the reviewed evidence.
- Toko's domain receipt and mint guarantees remain deliberately unverified by
  Canic and must be proved in the downstream repository.

## Focused Validation

- `cargo test -p canic-core receipt_backed_`: seven receipt-backed storage and
  canonical-snapshot tests passed; 788 unrelated unit tests were filtered out.
- The published focused PocketIC proof covers overlapping local capacity,
  receipt replay, binding and capacity rejection, missing and stale settlement,
  same-Wasm pending upgrade, commit, rollback, terminal replay, contradictory
  evidence, and released rollback capacity.
- Static scans found no retired call-builder intent surface, no mint-domain
  type leakage, no second receipt-backed store/facade, and no receipt-specific
  timer or scan.
- Canonical state and role-contract sources assign receipt-backed state only to
  memory ID 43.

Full workspace, broad PocketIC, deployment, and release suites were not rerun
under the targeted-validation policy.

## Documentation Correction

The tracker previously said receipt-backed rows did not change “its maintained
pending count,” which could be read as the shared resource aggregate's
`pending_count`. The implementation correctly increments that aggregate count
while pending. The intended invariant is narrower: receipt-backed rows do not
change the intent metadata's local expirable-pending count. The tracker and
compact handoff now state that boundary explicitly.

## Closeout Decision

Close the Canic-owned 0.90 line at `0.90.2`. Do not extend it with downstream
mint types, generic effect abstractions, a receipt protocol, background
reconciliation, or compatibility surfaces.

The next Canic line requires a fresh evidence-backed audit and an approved
bounded design. Toko adoption proceeds independently against the published
Canic contract.
