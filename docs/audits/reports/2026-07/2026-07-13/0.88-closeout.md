# Canic 0.88 Closeout Audit

Date: 2026-07-13

## Verdict

- Result: **PASS**.
- Audited release: `v0.88.2` at `5542f4f2cf2dd99cd3f28a93dd3a26e64ae027c2`.
- Open 0.88 correctness or release blockers: **none**.
- Residual risk: **2 / 10**. Full workspace, PocketIC, deployment, Wasm, and
  publish validation remain maintainer-owned under the targeted-test policy.

0.88 may remain closed. Its three implementation slices match the accepted
design, the removed paths are hard-cut, and the published release surfaces are
internally consistent.

## Scope

The audit checked the final `0.88.0` through `0.88.2` implementation against
the acceptance criteria in
`docs/design/0.88-artifact-durability-and-config-errors/0.88-design.md`.

Reviewed boundaries:

- backup artifact-directory synchronization, publication, recovery, journal
  transition, and completion accounting;
- host and CLI single-file staging, publication, cleanup, parent persistence,
  and the closed output inventory;
- fleet-config error ownership, source retention, rollback, and direct command
  consumers; and
- version, tag, install, changelog, layering, dependency, and lint surfaces.

## Slice A - Durable Backup Artifact Commit

Result: **PASS**.

- Direct snapshot capture and planned backup finalization call the same
  backup-owned artifact commit function.
- Temporary and canonical paths must be distinct siblings. Publication uses
  `renameat` with `NOREPLACE`, so a destination created during publication is
  not replaced.
- Artifact traversal is descriptor-relative. Entries are inspected without
  following symlinks, reopened with `NOFOLLOW`, and checked again after open.
  Only regular files and directories are accepted.
- Every regular file is synchronized before its containing directories;
  nested directories are synchronized bottom-up, followed by the artifact
  root and final parent.
- Recovery is admitted only from `ChecksumVerified` runner state with the
  exact expected temporary path, recorded checksum, absent temporary tree,
  and present canonical tree. The canonical bytes are rehashed and the tree
  and parent are resynchronized before the journal advances.
- Direct capture proposes durable state and the completed metric in a cloned
  journal. A failed durable journal replacement leaves the caller-visible
  journal at `ChecksumVerified` with no completion increment.
- Planned-runner completion is returned only after the download journal has
  durably recorded every selected artifact as durable.

The resumable workflow is the planned backup runner. The direct snapshot
command remains a one-shot capture path, but it uses the same durable commit
and fails below durable state if its journal transition does not complete.
This does not create a false durable claim.

Focused result: eight tests passed across commit-step injection, no-replace
races, unsupported entries, destination conflicts, journal-write failure,
exact post-publication recovery, and checksum rejection.

## Slice B - Publication-Atomic CLI File Output

Result: **PASS**.

- `write_bytes` and `create_new_bytes` delegate to one private commit engine.
- Complete caller-provided bytes are staged in a uniquely created sibling file
  and synchronized before publication.
- Replace mode uses atomic rename. Create-new mode uses atomic `NOREPLACE`; it
  has no existence-check-plus-replacing-rename fallback.
- Handled staging, write, sync, and publication failures remove the temporary
  file. A post-publication parent-sync failure can expose only the complete new
  bytes and still returns an error.
- Replacement creates missing parents one level at a time and synchronizes
  each new directory and the parent entry that introduced it. Create-new
  retains the deployment-plan requirement that its parent already exist.
- The three shared CLI helpers serialize complete bytes before entering host
  durable I/O. `canic deploy plan --out` is the only create-new consumer.
- A production writer scan found no additional in-scope direct final-file
  writer. Scaffold rollback, the cycles pending journal, and subsystem-owned
  persistence remain with their explicitly excluded owners.

Focused result: fourteen tests passed across host commit failures, parent
persistence, no-clobber races, serialization failure, shared helpers, and both
deployment-plan output contracts.

## Slice C - Typed Fleet-Config Failures

Result: **PASS**.

- The configuration subtree returns one concrete `FleetConfigError`; it has no
  boxed dynamic-error signature.
- I/O errors retain operation, path, and `io::Error`. Core configuration and
  TOML errors retain operation and parser source.
- Invalid names and kinds, missing declarations, fleet mismatches, mutation
  conflicts, package-metadata invariants, and role-contract rejection are
  typed variants.
- `RollbackFailed` retains both the original mutation and rollback failure as
  boxed concrete `FleetConfigError` values. Neither cause is flattened.
- Build, fleet, and scaffold command errors retain `FleetConfigError`
  directly. There is no display-text classifier or compatibility conversion.
- The package-manifest note produced after the host has already decided a
  rename can proceed is presentation data, not an error return or later
  classification input.

Focused result: five tests passed for retained I/O/core sources, structural
input and conflict classification, successful rollback, and dual-cause
rollback failure.

## Hard-Cut and Scope Review

Result: **PASS**.

- The old artifact finalization and direct CLI final-file write paths are not
  retained beside the new owners.
- Fleet-config APIs do not expose the former boxed return type through aliases,
  wrappers, or conversions.
- No cross-crate filesystem transaction layer, global error framework, generic
  command projector, or broad visibility rewrite was introduced.
- Persisted backup schemas, checksum format, CLI JSON/text shapes, fleet TOML,
  request formats, and command exits remain unchanged.

## Release-Surface Review

Result: **PASS**.

- `HEAD`, `origin/main`, and `v0.88.2` identify the same release commit.
- Workspace package version, all workspace Canic dependency versions, and all
  Canic package entries in `Cargo.lock` are `0.88.2`.
- The development installer default and published install URL name `0.88.2`.
- Root and detailed changelogs contain finalized entries for `0.88.0`,
  `0.88.1`, and `0.88.2` in descending patch order.
- The 0.88 design and tracker now describe the line as complete and closed.

- three focused shared CLI output tests;
- two focused deployment-plan output tests;

No full workspace, PocketIC, deployment, broad Wasm, or publish suite was run.

## Closeout Decision

0.88 is closed with no correction slice required. The separately proposed
0.89 design may proceed to review without carrying unresolved 0.88 work.
