# 0.91 Role Admission and Manifest Publication Closeout

Date: 2026-07-13

## Result

Pass. The accepted 0.91 scope is implemented without a compatibility path or
a generalized build framework.

The maintained flow is now:

```text
validated config snapshot
  -> admitted package and exact output path per required target
  -> current role-labelled build outputs
  -> exact coverage and path validation
  -> release entries in snapshot order
  -> durable root manifest replacement
```

Filesystem existence is no longer evidence that a normal complete build
finished.

## Ownership Findings

### Role admission

- `canic-core` owns one lowercase ASCII snake_case grammar helper and one
  canister-role policy that adds the 40-byte limit and typed issues.
- Complete config validation, fleet role mutations, and loaded release-set
  manifests use that policy before role-derived artifact use.
- The scaffold parser reuses the grammar helper but does not acquire the
  canister-role length policy.
- No regex, normalization, kebab-case alias, sanitization, or second character
  predicate remains.

### Complete-build proof

- Install-root resolves one validated configuration-backed snapshot before
  root creation or artifact building.
- Each normal role build consumes the package identity and exact artifact
  paths admitted by that snapshot; it does not reload configuration to select
  its role, package, release version, or output location.
- The complete build retains every successful role-labelled output and passes
  that exact collection to the normal manifest writer.
- Root output proves required-target coverage but is not serialized as an
  ordinary release member.
- Missing, duplicate, unexpected, path-mismatched, unreadable, or invalid
  current outputs reject before manifest replacement.
- Successful entry order comes from the snapshot rather than output completion
  or map order.

### Hard cut

The following retired surfaces are absent:

- both readiness-based manifest writers;
- both public configuration-reloading normal manifest writers;
- the configuration-reloading `configured_install_targets` and
  `configured_release_roles` projections;
- `CanisterArtifactBuildOutput::manifest_path`;
- single-role manifest publication; and
- fleet-manifest inclusion in per-role build provenance.

One crate-private current-output writer owns normal complete-build publication.
The deployment-plan writer remains separate because an accepted plan is its
explicit complete artifact source.

## Compatibility Check

The persisted manifest structs, field names, field order, hashing algorithm,
chunking, pretty-JSON serialization, durable writer, staging flow, and receipt
flow are unchanged. On the successful local build path, the refactored entry
builder receives the exact gzip path returned by the builder rather than
reconstructing the same path from disk.

A failed or direct partial build leaves an existing manifest byte-for-byte
unchanged. It may still replace an artifact covered by that old manifest, in
which case the existing observed-file and manifest-digest comparison rejects
the old artifact set. 0.91 does not add artifact rollback, build epochs,
locks, or concurrent-writer guarantees.
