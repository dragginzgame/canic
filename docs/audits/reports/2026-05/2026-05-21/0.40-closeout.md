## Verdict

**CLOSE WITH FOLLOW-UPS.**

The 0.40 implementation matches the intended hard cut: protected Canic-internal updates now use root-signed, method-scoped internal invocation proofs carried in a Canic-owned raw-ingress envelope, and the verifier-local sibling authorization model is no longer part of the protected endpoint surface. Source guards, macro tests, protocol tests, and focused auth tests all passed locally. I found no release-blocking source contradiction. The remaining items are process/documentation closeout tasks: move the active handoff to 0.41.x, keep the intentionally deferred manifest/query/cache work tracked outside 0.40, and commit the release metadata correction made during this audit.

## Release Objective

0.40.x was the "attested Canic calls" hard-cut line. The release objective was to replace verifier-local sibling authorization with root-issued, method-scoped internal invocation proofs transported by `CanicCall` in a Canic-owned internal RPC envelope, while keeping structural query exceptions explicit and narrow.

## Findings

### Blockers

None found.

### Non-Blocking Follow-Ups

1. Move `docs/status/current.md` active line from `0.40.x` to `0.41.x` after closeout.
2. Commit the changelog correction from `ic-memory 0.6.0` to `ic-memory 0.6.1`.
3. Keep root-published internal-method policy manifest, protected attested queries, and stable outgoing proof-cache persistence tracked as later-minor work.
4. Treat the local `main` branch being ahead of `origin/main` as a release-process note: the closeout evidence should land in a pushed commit before considering the audit archived.

### Documentation And Handoff Issues

- `docs/status/current.md` still names `0.40.x` as active. That is accurate until closeout lands, but it should move to `0.41.x` immediately after.
- The root and detailed changelog had stale `ic-memory 0.6.0` text while the manifests used `0.6.1`; this audit corrected the changelog text.

### Deferred Intentional Scope

- Root-published internal-method policy manifest.
- Protected attested queries.
- Stable persistence for outgoing proof-cache entries.

## Risk Assessment

### Security And Auth Risks

Low for 0.40 closeout. The protected update path verifies a root-signed, method-scoped proof before dispatching user arguments. The old verifier-local AppIndex-only sibling predicate is rejected by the macro parser. Retry behavior is intentionally narrow and does not retry malformed, unauthorized, expired, role-mismatch, method-mismatch, or policy-denied calls.

### ABI And API Risks

Low with one caveat. Protected updates use a no-argument Candid surface with the internal envelope carried in raw ingress bytes, and guards cover the wasm-store DID/macro manifest. The `ic-memory` version metadata is now corrected to `0.6.1`.

### Release And Process Risks

Moderate until this report and changelog correction are committed and pushed. The source audit passed locally, but the branch was one commit ahead of origin at the snapshot and this report adds more local documentation changes.

### Downstream And Operator Risks

Low. Structural query exceptions are explicit, and deferred protected queries are documented. Operators should be told that protected attested queries and stable proof-cache persistence are not part of 0.40.

## Recommended Closeout Actions

1. Commit and push this audit report plus the changelog `ic-memory 0.6.1` correction.
2. Update `docs/status/current.md` to make `0.41.x` the active minor after the closeout commit lands.
3. Open or carry explicit follow-up issues for method-policy manifest publication, protected attested queries, and stable proof-cache persistence.
4. Do not reopen 0.40 scope unless a fresh source contradiction appears.
