# Audit: Build Integrity And Reproducibility

## Method Contract

- Audit ID: `CANIC-BUILD-INTEGRITY-001`
- Method version: `2`
- Disposition: `retain`
- Owner: build scripts, procedural macros, generated code, unsafe inventory,
  and Wasm reproducibility
- Kind/profile: build `invariant` plus `measured` reproducibility proof
- Trace mode: `code_trace`; reproducibility uses two isolated local
  `execution_trace` builds
- Cost/runtime: high; 60-180 minutes
- Prerequisites: Git, ripgrep, jq, Rust/Cargo toolchain, Wasm target, build
  helpers, two isolated `CARGO_TARGET_DIR` values, and enough disk for
  duplicate builds
- False-positive boundary: generated output and unsafe blocks are classified by
  reachability and justification; presence alone is not failure
- Shared contract: [AUDIT-HOWTO.md](../../AUDIT-HOWTO.md)

## Purpose

Prove that code executed during build, code generated by macros/scripts, unsafe
blocks, and produced Wasm artifacts have explicit owners and do not create an
unreviewed path around source, feature, environment, or release authority.

This method does not replace Wasm size measurement, package validation, or
runtime correctness tests.

## Invariants

- Every `build.rs` and procedural macro has a narrow, documented output and
  rerun-input boundary.
- Generated code cannot silently widen public/authenticated surface or embed
  environment-specific security configuration without a declared input.
- Unsafe code is inventoried, justified at the owner, and bounded by a testable
  safety invariant.
- Release-profile Wasm builds are byte-reproducible from identical declared
  inputs, or every differing component is explained and excluded from signed
  artifact claims.
- Build helpers fail closed when required shrink, transform, manifest,
  provenance, or output-validation steps fail.

## Scope

- workspace manifests, lockfile, `rust-toolchain.toml`, and Cargo config;
- every `build.rs` and `proc-macro = true` crate;
- `canic-macros`, generated endpoint/config/Candid paths, and build helpers;
- production Rust `unsafe` blocks and FFI/platform boundaries;
- Wasm build, shrink, gzip, manifest, provenance, and checksum paths; and
- build-relevant environment selection.

Exclude third-party crate source outside the repository. Dependency trust is
owned by `CANIC-DEPENDENCY-001`.

## Required Code Trace

```bash
find . -path './target' -prune -o -name build.rs -print
rg -n 'proc-macro\s*=\s*true|include!|include_bytes!|OUT_DIR|cargo:rerun-if|cargo:rustc-env' Cargo.toml crates canisters fleets -g 'Cargo.toml' -g '*.rs'
rg -n '\bunsafe\b' crates canisters fleets -g '*.rs'
rg -n 'CARGO_TARGET_DIR|wasm32-unknown-unknown|ic-wasm|gzip|sha256|provenance|manifest' Makefile scripts crates fleets -g '*.sh' -g '*.rs' -g '*.toml' -g 'Makefile'
```

For each build/generation owner, record inputs, outputs, caller, failure mode,
public/runtime impact, tests, and whether output identity is included in a
release manifest.

For each unsafe block, record location, invariant, safe wrapper, caller set,
and rejection/boundary proof. An unowned or unjustified production unsafe
block is a failure.

## Reproducibility Execution

Use one immutable source snapshot, the same lockfile/toolchain/features, no
network, distinct temporary target and local build-state roots, and the exact
same command twice. Do not reuse produced Wasm between runs.

Record SHA-256 for raw, shrunk, gzip, root bundle, manifest, and provenance
outputs where present. Deterministic gzip must suppress timestamps.

### Provenance Semantic Comparison

Build provenance is compared as parsed JSON, not raw envelope bytes. Normalize
both sides with exactly:

```bash
jq --sort-keys 'del(.generated_at, .payload.generated_at, .payload_sha256)' \
  <build-provenance.json>
```

Only these three fields are excluded:

- envelope `generated_at`, which is observational metadata;
- payload `generated_at`, the same observation in the typed payload; and
- envelope `payload_sha256`, because it is derived from the unnormalized
  payload and therefore changes when payload `generated_at` changes.

The normalized nested payload itself remains in the comparison. Every source,
Cargo, configuration, environment, feature, tool, build-status, artifact path
classification, size, and artifact hash field must compare exactly. A semantic
comparison implementation that removes the payload body or any additional
field is invalid. Reports must retain both raw provenance hashes and the
normalized comparison result.

- Identical decision-bearing artifacts: `pass` for reproducibility.
- Differences limited to the three normalized provenance fields above:
  `pass` when the complete normalized envelopes compare equal.
- Unexplained shipped byte difference: `fail`.
- Missing tool or insufficient disk: `blocked`.

## Fixtures And Boundaries

- Positive: unchanged declared inputs produce identical artifact hashes.
- Rejection: a required build/shrink/manifest step failure stops publication.
- Boundary: an environment or feature change appears in provenance and changes
  comparison identity.
- Regression: generated endpoint/Candid surface remains tied to current source
  and configuration inputs.

## Required Report

Include run identity, build-owner inventory, generated-code trust map, unsafe
ledger, two-build manifest comparison, unexplained differences, verification
readout, findings, unreviewed boundaries, and verdict.
