# Canic 0.50.x Closeout Audit

## Executive Summary

Verdict: **PASS**.

The 0.50.x implementation matches the release theme: adoption reporting is
passive, fleet-scoped, evidence-driven, and does not execute ownership,
topology, artifact, install, controller, or deployment-truth mutation. The core
host model, CLI surface, evidence inputs, text/JSON rendering, and release
docs are present and validated.

0.50.15 resolved the closeout follow-ups found during the audit:

- the numbered 0.50 design doc now records implemented behavior rather than
  tentative planning language;
- JSON adoption report output is explicitly experimental throughout `0.50.x`;
- artifact evidence conflict detection is pinned in both directions;
- observed-only declaration recommendations are pinned across the authority
  matrix;
- explicit artifact-manifest precedence over deployment-check artifact evidence
  is pinned.

No release-blocking implementation defect was found.

## CLI Surface Audit

Implemented command:

```text
canic fleet adoption report <fleet> --profile <profile>
```

Evidence:

- Help/parser definitions are in `crates/canic-cli/src/fleets/mod.rs`:
  `fleet_adoption_command`, `fleet_adoption_report_command`,
  `AdoptionReportOptions::parse`, `parse_adoption_profile`, and
  `parse_adoption_report_format`.
- `run_adoption_report` selects a matching fleet config with
  `selected_fleet_config_path(&options.fleet)`, builds a host report, and
  passes it to `write_adoption_report`.
- `<fleet>` is required and `--profile <profile>` is required.
- Default output is text because `parse_adoption_report_format` uses
  `value.unwrap_or("text")`.
- `--format json` exists and is rendered through
  `output::write_pretty_json`.
- `--output <path>` exists and `write_adoption_report` passes the output path
  to `output::write_text` or `output::write_pretty_json`.
- Parent directories for output files are created by
  `crates/canic-cli/src/output/mod.rs`.
- Evidence options exist:
  - `--inventory <path>`;
  - `--artifact-manifest <path>`;
  - `--package-metadata <path>`;
  - `--deployment-check <path>`;
  - `--cargo-metadata <path>`.

  - `--inventory` plus `--deployment-check` is rejected in

  - `--package-metadata` plus `--cargo-metadata` is rejected in
    `adoption_package_metadata_from_options`.
- `--artifact-manifest` intentionally takes precedence over

## Read-Only Boundary Audit

Adoption reporting does not mutate:

- `canic.toml`;
- package manifests;
- topology;
- local artifact manifests;
- inventories;
- deployment-truth records;
- generated configs;
- deployment plans;
- artifacts;
- controllers;
- canister lifecycle state;
- source fixtures.

Evidence:

- Host builder `adoption_report_from_config_source` accepts strings and
  optional in-memory evidence values. It has no filesystem, network, IC,
  install, controller, or artifact write calls.
- CLI `build_adoption_report_from_config_path` reads config/evidence files,
  constructs `AdoptionReportRequest`, and calls the host builder.
- CLI writing is centralized in `write_adoption_report`; it writes only stdout
  or the requested `--output` path.
- `render_adoption_report` prints `read_only: true`.
- Summary always has `mutating_actions_performed: 0`.
- Smoke test compared `canic.toml` before/after and verified only the requested
  report artifact was written.

## Evidence Input Audit

`--inventory`:

- Reads `DeploymentInventoryV1` via `read_json_file`.
- Invalid/missing files fail through normal IO/serde error propagation.
- Smoke tested valid, missing, and invalid JSON cases.

`--artifact-manifest`:

- Reads `RoleArtifactManifestV1` via `read_json_file`.
- Feeds artifact state/evidence through `artifact_states_by_role` and
  `artifact_evidence_by_role`.
- Explicit artifact manifest takes precedence over deployment-check plan
  artifacts.

`--package-metadata`:

- Reads a JSON array of `AdoptionPackageMetadataV1`.
- Feeds package matching through `package_state`.
- Missing fleet/role or mismatch becomes package-state evidence and may become
  `evidence-conflict`.

`--deployment-check`:

- `read_deployment_check_inventory` extracts the `inventory` field from a saved
  `DeploymentCheckV1` JSON artifact.
- `read_deployment_check_artifact_manifest` extracts `plan.role_artifacts` into
  a synthetic `RoleArtifactManifestV1` when explicit `--artifact-manifest` is
  absent.
- `--deployment-check` and `--inventory` are rejected together.

`--cargo-metadata`:

- `read_cargo_metadata_package_metadata` reads saved
  `cargo metadata --format-version 1` JSON.
- It extracts `[package.metadata.canic]` `fleet` and `role` when present.
- `cargo_metadata_package_path` normalizes package paths against the selected
  fleet config directory.
- `package = "."` and sibling package paths are covered by tests.
- `--cargo-metadata` and `--package-metadata` are rejected together.

Missing/stale evidence:

- `missing_evidence` carries unresolved inventory observations and unresolved
  artifact evidence into `inputs.missing_or_stale_evidence`.
- Text rendering prints the `Missing or stale evidence` section.

## Classification And Recommendation Audit

Verified classifications:

- `managed`: declared and attached role; code inserts
  `AdoptionClassificationV1::Managed` when topology attachment exists.
- `declared-only`: declared role with no topology attachment; tested by
  `adoption_report_preserves_declared_only_as_non_deployable`.
- `observed-only`: observed role/canister without declaration; tested by
  brownfield and observed-only fixtures.
- `user-controlled`: derived from `CanisterControlClassV1::UserControlled`.
- `external-controller-required`: derived from user-controlled or external
  control classes.
- `imported-pool-candidate`: emitted for observed pool entries, with warning
  `pool import is outside 0.50.0`.
- `evidence-conflict`: emitted for package mismatch, duplicate observed roles,
  and artifact manifest/inventory disagreement.

Verified recommendation behavior:

- Declared-only roles receive blocked, unsupported
  `attach_role_later` previews, not execution.
- Observed-only Canic-authorized roles may receive blocked, unsupported
  `canic fleet role declare ...` previews.
- User-controlled observed-only candidates receive
  `review_authority_before_declaration` with no mutating suggested action.
- Leaf-only authority-sensitive roles get warnings and no declaration preview.
- Recommendations render as `Recommendations (report-only; not executed)` and
  `suggested_action_preview`.
- Blocked actions render as `Blocked adoption actions (not executed by report)`.

Closeout coverage:

- `adoption_report_authority_gates_observed_only_declaration_recommendations`
  covers Canic-authorized, user-controlled, external, and unknown observed-only
  authority states.
- Artifact conflict tests cover both artifact manifest/inventory disagreement
  directions. Adoption reporting still has no artifact import or registry
  mutation path.
