# Canic 0.53.x Closeout Audit

Date: 2026-05-31

## Executive Summary

Verdict: **PASS**.

0.53.x delivered the intended CI policy gate and project evidence manifest
line. The line now has a passive `canic evidence gate` command that evaluates
saved `EvidenceEnvelopeV1` files against strict `CiPolicyV1` rules, optional
stable `canic.build_provenance.v1` field checks, project evidence manifests
for grouping existing evidence files, maintained CI guidance, and duplicate
manifest-path hardening.

The highest-risk invariant holds: policy gates consume existing evidence and
produce reports. They do not run builds, discover live state, install or
upgrade Wasm, mutate topology, change controllers, register artifacts, acquire
locks, or turn policy success into deployment truth.

## CLI Surface Audit

### `canic evidence`

Verified.

`crates/canic-cli/src/evidence.rs` defines the command family with only:

```text
gate
compare
```

Smoke output from `cargo run -p canic-cli -- evidence --help` confirmed the
same command list.

### `canic evidence gate --policy <path> --envelope <path>`

Verified.

Evidence:

- `EvidenceGateOptions::parse()` requires `--policy` and exactly one gate
  input through the `gate-input` arg group.
- `evaluate_gate_files()` reads the policy file and one existing
  `EvidenceEnvelopeV1` file.
- `evaluate_policy_gate()` evaluates stable envelope fields, payload schema
  identity/stability, exit class, summary fields, optional build-provenance
  rules, and required input fingerprints.
- `write_gate_report()` supports text, raw JSON, and envelope JSON output.
- `run()` writes output before returning a policy failure when practical.

Smoke output from `cargo run -p canic-cli -- evidence gate --help` confirmed:

```text
Usage: canic evidence gate [OPTIONS] --policy <path> <--envelope <path>|--manifest <path>>
```

### `canic evidence gate --policy <path> --manifest <path>`

Verified.

Evidence:

- `EvidenceGateInput::Manifest(PathBuf)` exists.
- `evaluate_project_evidence_manifest_gate()` parses a strict
  `ProjectEvidenceManifestV1`, fingerprints the manifest, resolves paths from
  the manifest project root, evaluates each existing evidence envelope with the
  same policy engine, and emits `ProjectEvidenceGateReportV1`.
- Missing required files map to `missing_required_evidence`.
- Missing optional files map to `success_with_warnings`.
- Payload schema and target mismatches map to `blocked_by_policy`.
- Duplicate manifest evidence paths are rejected before evaluation.

## Policy Model Audit

Verified.

`crates/canic-host/src/policy_gate.rs` owns the policy model:

- `CiPolicyV1`;
- `PolicyEnvelopeRulesV1`;
- `PolicyExitClassRulesV1`;
- `PolicySummaryRulesV1`;
- `PolicyBuildProvenanceRulesV1`;
- `PolicyRequiredInputRuleV1`;
- `PolicyGateReportV1`;
- `PolicyRequirementV1`;
- `PolicyFindingV1`;
- `PolicyFindingSeverityV1`;
- `PolicyEvaluationStatusV1`;
- `ProjectEvidenceManifestV1`;
- `ProjectEvidenceGateReportV1`.

Strictness:

- policy structs use `#[serde(deny_unknown_fields)]`;
- unsupported schema versions fail;
- empty allow-lists fail;
- empty `[build_provenance]` tables fail;
- unknown build-provenance keys fail;
- manifest entries require non-empty kind, path, payload schema, and at least
  one target selector;
- duplicate manifest paths fail before evidence evaluation.

Stable schema identities:

```text
canic.policy_gate_report.v1
canic.project_evidence_manifest.v1
canic.project_evidence_gate_report.v1
```

## Passive Boundary Audit

Verified.

The policy gate path reads:

- policy files;
- one existing evidence envelope file, or one manifest plus listed evidence
  envelope files;
- local file metadata needed for input fingerprints.

The policy gate path writes only explicit report output requested through
`--output`, or stdout for normal output. No audited code path calls build,
deploy, install, controller, topology, artifact registry, `wasm_store`,
adoption/import, or live discovery helpers.

The architecture guide also states the user-facing boundary clearly: policy
success means saved evidence matched policy at evaluation time; it does not
prove freshness, install state, artifact registry state, controller changes,
topology changes, adoption import, or deployment truth mutation.
