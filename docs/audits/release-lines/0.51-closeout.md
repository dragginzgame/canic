# Canic 0.51.x Closeout Audit

Date: 2026-05-31

## Executive Summary

Verdict: PASS

0.51.x has landed as a coherent CI/GitOps evidence-envelope line. The release
adds a stable `EvidenceEnvelopeV1` outer contract around selected passive
outputs, preserves raw command-specific JSON, adds shared exit-class and
fingerprint behavior, introduces stable envelope comparison, and documents a
passive pipeline shape.

No release-blocking findings were found.

Highest-risk areas checked:

- raw JSON remains raw and is not silently wrapped;
- envelope output remains passive and does not add deployment/controller/topology
  mutation;
- `success_with_warnings` stays process-success by default;
- `evidence_conflict` dominates generic policy blocking;
- input paths are fingerprinted and paths outside the selected root are
  redacted;
- `canic evidence compare` compares stable envelope fields while ignoring
  timestamp/version/payload-body noise.

## CLI Surface Audit

Implemented command surfaces:

```text
canic fleet adoption report <fleet> --profile <profile> --format envelope-json
canic deploy check <deployment> --format envelope-json
canic evidence compare --left <path> --right <path>
canic evidence compare --left <path> --right <path> --format json
```

Findings:

- `canic fleet adoption report` is fleet-scoped, requires `--profile`, and
  accepts `text|json|envelope-json` in `crates/canic-cli/src/fleets/mod.rs:974`.
- Adoption `--output <path>` writes text, raw JSON, or envelope JSON through the
  same output boundary in `crates/canic-cli/src/fleets/mod.rs:1641`.
- `canic deploy check` accepts `json|envelope-json`; deploy-check has no
  dedicated `--output`, so pipeline docs correctly use shell redirection.
- `canic evidence compare` is a read-only compare of two existing
  `EvidenceEnvelopeV1` JSON files. It exits with an error when stable fields
  differ.
- Help tests passed: `command_family_help_returns_ok` and
  `usage_lists_command_families`.

## Stable Envelope Contract Audit

Verified stable envelope fields:

- `envelope_schema`
- `canic_version`
- `command`
- `target`
- `generated_at`
- `source_config`
- `inputs`
- `payload_schema`
- `payload_sha256`
- `payload`
- `summary`
- `exit_class`

The stable contract is the envelope, not the nested payload. This is documented
in `docs/architecture/evidence-envelopes.md:51` and implemented in
`crates/canic-host/src/evidence_envelope.rs:15`.

Payload stability is explicit:

```text
canic.evidence_envelope.v1 = stable
canic.adoption_report.v1   = experimental
canic.deployment_check.v1  = internal
```

The schema constructors in `crates/canic-host/src/evidence_envelope.rs:264`
match the docs and tests.

## Read-Only Boundary Audit

Verified:

- Adoption envelope output writes only the requested report artifact when
  `--output` is present. Test `writes_adoption_report_envelope_json_output_file`
  snapshots the config before/after and asserts no config mutation.
- Deployment-check envelope output prints the current deployment-check payload
  wrapped in an envelope; it does not introduce install, controller, topology,
  or artifact mutation paths.
- Envelope comparison reads two envelope files and writes only stdout.
- Architecture docs state that envelope output must not install/upgrade Wasm,
  attach topology, import candidates, change controllers, register or GC
  artifacts, teardown resources, perform extra live discovery, make stale
  evidence fresh, or turn the envelope itself into deployment truth.

## Exit-Class Audit

`ExitClassV1` implements:

```text
success
success_with_warnings
blocked_by_policy
evidence_conflict
missing_required_evidence
invalid_input
execution_failed
internal_error
```

Precedence is implemented in `ExitClassV1::precedence` and
`evidence_summary_exit_class`. Tests prove:

- snake-case serialization;
- `evidence_conflict` dominates generic blockers;
- `missing_required_evidence` dominates generic blockers where explicitly set;
- warnings and missing/stale evidence classify as `success_with_warnings`.

Process behavior:

- deployment-check warnings remain process-success, tested by
  `deploy_check_status_allows_warning_report`;
- compare returns an error on differences, tested by
  `compare_files_returns_error_for_differences_after_writing_report`.

## Input Fingerprint Audit

Verified:

- `InputFingerprintV1` records `kind`, `path`, `path_display`, `sha256`,
  `size_bytes`, `modified_unix_secs`, optional schema, and optional note.
- File bytes are hashed with SHA-256.
- Paths under the config root render relative.
- Absolute paths outside the selected root are omitted from envelope output and
  command provenance records redactions.
- Adoption envelope fingerprints cover supplied deployment-check, inventory,
  artifact-manifest, cargo-metadata, and package-metadata paths.
- Deployment-check envelope fingerprints source config when available, and
  preserves observed config hashes when deployment-check evidence already has
  them.

## Evidence Comparison Audit

`canic evidence compare` compares:

- envelope schema;
- command provenance;
- target;
- source config;
- inputs;
- payload schema;
- payload SHA-256;
- structured summary;
- exit class.

It intentionally ignores:

- `canic_version`;
- `generated_at`;
- nested `payload`.

This matches docs and tests. The compare command has text and JSON output and
does not inspect live deployments.
