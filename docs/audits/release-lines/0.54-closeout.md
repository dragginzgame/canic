# Canic 0.54.x Closeout Audit

Date: 2026-05-31

## Executive Summary

Verdict: **PASS**.

0.54.x delivered the intended pre-v1 passive deployment catalog line without
expanding Canic into deployment groups, promotion lanes, saved-evidence
catalogs, locks, signing, registry import, teardown, controller mutation,
topology mutation, install authority, or active adoption/import.

The line answers one small operator question:

```text
What deployment targets does this project already know about?
```

The highest-risk invariant holds: catalog commands read local deployment-target
state and produce reports. They do not query live deployments, create
deployment truth, infer deployments from fleet-template names, mutate topology
or controllers, install Wasm, register artifacts, or make saved evidence fresh.

## CLI Surface Audit

Verified active command surface:

```text
canic deploy catalog list
canic deploy catalog inspect <deployment>
```

Supported options:

```text
--format <text|json>
--output <path>
```

Evidence:

- `deploy_catalog_command()`, `deploy_catalog_list_command()`, and
  `deploy_catalog_inspect_command()` in `crates/canic-cli/src/deploy/mod.rs`;
- `DeployCatalogOptions::parse_list()` and `parse_inspect()`;
- tests:
  - `deploy_catalog_options_parse_list_defaults_to_text`;
  - `deploy_catalog_options_parse_inspect_json_output`;
  - `deploy_catalog_rejects_unknown_format`;
  - `deploy_catalog_command_dispatches_list_and_inspect`;
  - `deploy_catalog_help_documents_passive_deployment_target_scope`.

Smoke behavior from `fleets/demo`:

```text
canic deploy catalog list
```

returned an empty catalog with warnings:

```text
catalog.no_deployment_state
catalog.legacy_fleet_state_ignored
```

```text
canic deploy catalog list --format json
```

returned `DeploymentCatalogReportV1` JSON with `entries: []` and the same
warnings.

```text
canic deploy catalog inspect demo-local
```

failed clearly because `demo-local` is not a known deployment target in the
fresh demo fixture.

## Passive Boundary Audit

Verified.

Catalog code reads:

- local deployment-target state under `.canic/<network>/deployments`;
- file bytes for JSON decode and input fingerprinting;
- legacy `.canic/<network>/fleets` existence only to warn that it is ignored.

Catalog code writes only through the explicit output helpers used by
`--output`, or stdout when no output path is provided.

Source guard coverage:

- `deploy_catalog_path_has_no_live_lookup_or_mutation_primitives` scans the
  catalog dispatch path for live/mutating primitives such as install,
  canister creation, `dfx`, deployment-state registration, and root
  verification.

No audited 0.54 path calls live deployment lookup, build, install, controller,
topology, registry, `wasm_store`, adoption/import, signing, lock, teardown, or
deployment-truth mutation helpers.

## Data Model Audit

Verified.

`crates/canic-host/src/deployment_catalog.rs` defines:

```text
DEPLOYMENT_CATALOG_REPORT_SCHEMA_ID = "canic.deployment_catalog_report.v1"
DeploymentCatalogRequest
DeploymentCatalogReportV1
DeploymentCatalogEntryV1
DeploymentCatalogRootVerificationV1
DeploymentCatalogError
```

`DeploymentCatalogReportV1` is intentionally narrow:

```text
schema_version
generated_at
project_root
entries
warnings
```

Entries contain only deployment-target local state evidence:

```text
deployment
fleet
network
root_principal
root_verification
local_state_ref
warnings
```

There are no deployment groups, promotion lanes, latest-evidence pointers,
saved-evidence registries, locks, signing fields, registry fields, teardown
fields, or active adoption/import fields.
