# Canic 0.49.x Closeout Audit

## Follow-Up Resolution

Resolved on 2026-05-31: the 0.49 design doc now reflects the shipped hard-cut
surface. It requires package metadata to include both `fleet` and `role`,
documents `canic scaffold canister <fleet> <role>` without attachment flags,
documents `canic fleet role attach <fleet> <role> --subnet <subnet>
[--kind <kind>]`, removes unshipped scaffold attach and build dev examples,
and points greenfield readers at `canic fleet create <name>` plus the maintained
minimal managed fleet guide.

## Executive Summary

Verdict: PASS.

The 0.49 role lifecycle implementation is release-ready from the audited code
and test evidence. The core model is implemented: roles are declared in
fleet-scoped config, packages declare `[package.metadata.canic] fleet` plus
`role`, `canic::build!` validates package `fleet.role` against declarations and
emits declared/attached state, and visible `canic build <fleet> <role>` rejects
declared-only roles before Cargo artifact builds.

The original highest-risk follow-up was stale design-doc wording around
role-only metadata and unshipped command examples. That follow-up is now
resolved; the implemented 0.49 design doc matches the shipped hard-cut CLI and
metadata surface.

No blocker or high-severity implementation defects were found.

## Role Lifecycle Invariant Audit

- Roles are fleet-scoped by `FleetRoleRefV1 { fleet, role }` in
  `crates/canic-core/src/config/schema/role.rs`, with display form
  `fleet.role`.
- Declarations are represented by `ConfigModel.roles` and
  `RoleDeclaration { kind, package }`.
- Attachment is derived from topology by `ConfigModel::attached_roles`, which
  traverses direct subnet canisters and role-bearing child roles from scaling,
  sharding, and directory pools.
- `ConfigModel::attached_fleet_roles` maps attached local roles into
  `FleetRoleRefV1`.
- Validation allows ordinary declared-only roles and topology-less non-root
  role registries, but root declarations must be attached when topology exists
  and topology roles must be declared.
- `configured_role_lifecycle_from_source` consistently renders declared vs
  attached state and topology labels for direct and pool-derived attachments.
- Declared-only roles are intentionally visible in role list/inspect output.
- Deployment/build/install surfaces use `configured_deployable_roles`, not the
  full lifecycle list.
- Rename updates declaration headers, direct topology headers, exact
  `canister_role` assignments, one-line `app_index`, and matching package
  metadata.

## Findings

### Documentation: 0.49 design doc contained role-only compatibility language

- Severity: Documentation
- Area: Design docs
- Evidence:
  - The original closeout version of
    `docs/design/0.49-role-lifecycle-topology-attachment/0.49-design.md`
    showed a problem example with `[package.metadata.canic] role = "store"`
    only.
  - The same original doc later said `role = "store"` alone was valid when the
    build context resolved unambiguously.
  - Implementation requires `fleet` and `role` in
    `crates/canic/src/build_support/config.rs`.
  - Workspace governance enforces both fields in
    `crates/canic/tests/workspace_manifest.rs`.
- Impact: A contributor reading the design doc could infer a compatibility
  path that does not exist and should not exist under the hard-cut 0.49 model.
- Resolution: Fixed on 2026-05-31. The design doc now removes role-only
  compatibility wording and marks the hard-cut implementation state instead of
  tentative pre-implementation text.
- Suggested regression test: Existing workspace governance already protects
  implementation. No new code test needed; a docs-search guard for active docs
  could be added if docs drift keeps recurring.
