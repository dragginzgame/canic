# Canic 0.92.x Closeout Audit

## Verdict

```text
closeout_verdict: pass_with_limitations
```

0.92 is closed. The audit system was inventoried and corrected before use, all
22 retained definitions have valid active results, all ten mandatory traces
have complete current evidence, and every P0/P1 product, method, governance,
release, and evidence finding is fixed. Executable compatibility accounting
against `v0.91.6` passes with the documented pre-1.0 hard cuts.

The verdict is not plain `pass` because three confirmed P2 watchpoints remain
explicitly deferred. They are bounded dependency-maintenance, trust-path
complexity, and instruction-checkpoint limitations; none establishes an active
correctness or security defect. There are no waivers, blocked current runs,
accepted P1 findings, or unclassified compatibility deltas.

This closeout establishes confidence in the 0.92 line. It does not claim 1.0
readiness; the maintainer's separate requirement for at least three months of
real-world use remains unchanged.

## Audit-System Closeout

- Phase A inventoried the existing audit system and found six P1 method and
  governance defects.
- Phase B fixed those defects, reduced competing audit authority to one
  catalog, and froze method, runner, fixture, trace, and product-scope
  fingerprints before the product baseline.
- Phase C attempted all 22 retained definitions against the immutable
  `v0.92.0` product snapshot. Post-freeze method defects were versioned,
  invalidated, corrected, and rerun without rewriting their historical
  attempts.
- The valid frozen baseline was allowed to fail on accepted product findings;
  it did not close with a partial or blocked required method. Its mandatory
  trace aggregate was six pass and four fail, with the failures mapped to
  canonical findings.
- Finding-backed D1 through D13 slices fixed the accepted defects with causal
  parent comparisons and cumulative baseline accounting. Current affected
  trace evidence is ten pass and zero fail.
- The current audit-method catalog guard proves that all 22 retained
  definitions and the executable/governance inputs still match the frozen
  fingerprints.

Historical invalid, partial, blocked, and failing baseline attempts remain
preserved as evidence. They are not represented as current passing results.

## Product and Authority Outcome

The final implementation has one maintained authority for each corrected
invariant:

- model owns persisted state and state-shaped decision inputs;
- ops owns deterministic access, conversion, and approved single-step effects;
- policy owns pure admission decisions;
- workflow owns multi-step orchestration, capability/replay ownership, root
  proof admission, publication settlement, and blob-billing sequencing; and
- endpoints authenticate, marshal, and delegate immediately.

The current layering guard reports no ops-to-policy dependency. Root-issuer
admission, blob billing, durable Wasm publication, delegated-auth error
projection, and root proof installation each have one maintained flow.
Accidental DTO, error-root, command-proof, and public helper surfaces were
hard-cut instead of retained through aliases or wrappers.

Failure paths in the finding-backed slices reject before unauthorized or
unfunded mutation, preserve specific typed causes, and retain deterministic
retry/recovery identity. No compatibility alias, shim, deprecated API,
duplicate command route, legacy decoder, or fallback path was added.

## `v0.91.6` Compatibility Contract

Executable accounting covers every required contract surface:

- the independently generated 888-line root DID is byte-identical at SHA-256
  `51c9339aefb87d8dcf25951b5f9dfe1f5089711ae52316c0bb8b8adf9dabfc4e`;
- the canonical Wasm-store DID is byte-identical at SHA-256
  `4d92b4ee6b1aba130b8137e5d20e08699aa3cba3cc8652c7b4eda146dccdf84c`;
- production CLI commands, help, exit codes, diagnostics, normalized adoption
  JSON, and configuration ownership agree except for release-version output;
- stable schema trees are identical, and a canister installed from `v0.91.6`
  state upgrades in PocketIC while preserving identity, role, root/network
  bindings, topology, state projection, and readiness;
- backup/restore production ownership and format remain unchanged, and the
  current 195-test domain suite passes; and
- member package features and direct external dependency declarations are
  unchanged.

The accepted 0.92.7 build-provenance hard cut is explicit: old envelopes that
lack transform identity/outcome evidence reject with
`policy.build_provenance.invalid_payload`. Operators rebuild artifacts and
regenerate saved provenance. There is intentionally no defaulted field,
alternate schema, old DTO, compatibility parser, or fallback.

The 0.92.5 and 0.92.6 Rust source cuts likewise require users to remove the
obsolete imports/helpers and use the maintained request and root-auth
surfaces. They do not change Candid or stable state.
