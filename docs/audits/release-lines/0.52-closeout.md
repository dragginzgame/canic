# Canic 0.52.x Closeout Audit

Date: 2026-05-31

## Executive Summary

Verdict: **PASS**.

0.52.x delivered the intended source, build, and artifact provenance line. The
line now has a stable `canic.build_provenance.v1` payload wrapped by the 0.51
`EvidenceEnvelopeV1`, explicit build-time provenance output for
`canic build <fleet> <role> --provenance <path>`, passive report linkage through
`--build-provenance`, and CI/GitOps policy guidance for consuming the evidence.

The highest-risk invariant holds: provenance records evidence about produced
build artifacts, but it does not expand deployment, install, topology,
controller, registry, adoption, or deployment-truth authority. The remaining
future scope is intentionally outside 0.52: signing, CI locks, project manifest
semantics, provider wrappers, artifact registry import, and failed-build
provenance envelopes.

## CLI Surface Audit

### `canic build <fleet> <role> --provenance <path>`

Verified.

Evidence:

- `crates/canic-cli/src/build.rs` defines `BuildOptions { fleet, role, ..., provenance }`.
- The usage string is `canic build [OPTIONS] <fleet> <role>` and examples include `canic build demo app --provenance artifacts/app-provenance.json`.
- `run()` validates the selected role is topology-attached before building, then writes provenance after successful artifact generation through `write_build_provenance_if_requested()`.
- `write_build_provenance_if_requested()` returns immediately when `provenance` is absent, so ordinary builds do not gain a provenance output.
- `build_command_provenance()` records normalized command arguments and redacts paths outside the workspace root.

Relevant tests:

- `build_accepts_explicit_context_paths`;
- `build_rejects_old_role_only_shape`;
- `build_usage_lists_fleet_and_role`;
- `build_command_provenance_redacts_paths_outside_workspace`;
- `build_preflight_rejects_declared_only_role`;
- `build_preflight_accepts_attached_role`;
- `explicit_build_config_must_match_selected_fleet`.

## Provenance Model Audit

Verified.

`crates/canic-host/src/build_provenance.rs` owns the stable build provenance
payload:

- `BuildProvenanceV1`;
- `BuildProvenanceStatusV1`;
- `SourceProvenanceV1`;
- `SourceVcsV1`;
- `SourceDirtyPolicyV1`;
- `CargoProvenanceV1`;
- `BuildScriptInputStateV1`;
- `ArtifactProvenanceV1`;
- `ArtifactProvenanceKindV1`;
- `BuildProvenanceRequest`.

The stable schema identity is:

```text
canic.build_provenance.v1
```

`build_provenance_envelope()` wraps the payload in `EvidenceEnvelopeV1`, uses
target kind `artifact`, includes source config and input fingerprints, computes
`payload_sha256`, and maps warnings to `success_with_warnings`.

Model behavior:

- Git revision and branch are recorded when available.
- Dirty source state is explicit and uses
  `git-status-porcelain-v1-z-sha256`.
- Unknown source state is explicit.
- Cargo.lock and package manifest hashes are recorded when available.
- Package metadata fleet/role is required and must match the requested
  fleet/role.
- `RUSTFLAGS` is digested rather than emitted directly.
- raw Wasm and gzip Wasm are distinct artifact kinds and hashes.
- Candid and metadata artifacts are recorded only if present.
- build-script input collection is explicitly `not_recorded`.

Relevant tests:

- `build_provenance_schema_is_stable`;
- `unknown_source_provenance_is_explicit`;
- `artifact_provenance_records_wasm_and_gzip_separately`;
- `build_provenance_envelope_wraps_stable_payload`.

## Safety Boundary Audit

Verified.

0.52 adds evidence outputs and evidence input fingerprints. It does not add:

- deployment mutation;
- install or upgrade authority;
- controller mutation;
- topology mutation;
- adoption/import mutation;
- artifact registry import;
- `wasm_store` registration;
- CI locks;
- signing or attestation verification;
- provider-specific CI integration.

Evidence:

- `canic build <fleet> <role> --provenance <path>` writes provenance only after
  the existing artifact build succeeds.
- Adoption and deployment-check `--build-provenance` paths are only passed to
  `file_input_fingerprint()` with the stable build provenance schema.
- Docs explicitly state adoption/deployment-check report commands do not re-run
  builds, validate deployment truth from provenance alone, or turn provenance
  into authority.

Non-blocking gaps:
