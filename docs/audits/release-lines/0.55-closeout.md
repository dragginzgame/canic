# Canic 0.55.x Closeout Audit

Date: 2026-06-01

Final post-0.55.4 closeout audit:

```text
docs/audits/release-lines/0.55-final-closeout.md
```

## Executive Summary

Verdict: **PASS**.

0.55.x did what it was supposed to do: it stabilized the compact pre-v1
operator story instead of adding another product layer. The active surface is
small enough to explain:

```text
create fleet -> scaffold role -> attach role -> build with provenance ->
check deployment evidence -> gate saved evidence -> inspect known deployments
```

The highest-risk boundary holds. 0.55 added documentation, help alignment, a
maintained local smoke proof, and this candidate audit. It did not add
deployment groups, signing, locks, registry import, teardown, controller
mutation, active adoption/import, broad live verification, or new deployment
authority.

No blocker or high-severity release-readiness finding was found. The original
operational follow-up from this audit is now resolved by:

```text
scripts/ci/v1-operator-proof.sh
docs/operations/0.55-v1-operator-proof.md
```

That proof builds a real `demo.app` artifact with stable build provenance,
registers an explicit local deployment target in a temporary proof root, and
emits a deployment-check envelope that fingerprints the build provenance.

## Passive / Active Boundary Audit

Verified.

0.55 does not introduce new authority. The active v1 setup commands remain
explicit:

- `fleet create` creates fleet config;
- `scaffold canister` creates source files plus a declared role;
- `fleet role attach` mutates topology explicitly;
- `build` generates artifacts only for attached roles.

The report/evidence/catalog commands stay passive:

- `deploy check` prints or wraps deployment-check evidence;
- `evidence gate` evaluates saved evidence;
- `deploy catalog list/inspect` reads local deployment-target state only.

The local smoke and docs explicitly do not cover or add:

- deployment groups;
- promotion lanes;
- signing;
- locks;
- registry import;
- `wasm_store` retention or garbage collection;
- active adoption/import;
- controller mutation;
- topology mutation beyond `fleet role attach`;
- install or upgrade authority;
- teardown;
- broad live verification.

## Findings

### F-001: Fast smoke does not cover full build/deployment-check evidence

- Severity: Low, resolved in 0.55.4
- Area: Release readiness
- Evidence: `docs/operations/0.55-v1-local-smoke.md` and
  `scripts/ci/v1-readiness-smoke.sh` explicitly skip `canic build <fleet>
  <role> --provenance <path>` and `canic deploy check <deployment> --format
  envelope-json`.
- Impact: The no-network smoke proves the compact local setup/evidence-gate
  subset, but not the heavier artifact/deployment evidence path.
- Resolution: `scripts/ci/v1-operator-proof.sh` now builds `demo.app`, writes
  `build-provenance.json`, registers `demo-local` in a temporary proof root,
  and emits `deployment-check-envelope.json` with the build provenance
  fingerprinted as input evidence.
- Suggested regression test: Keep the operator proof running before final v1
  release. A later post-v1 integration smoke may add live install/root
  verification when replica requirements are acceptable.

### F-002: Historical docs still contain old command names as archive

- Severity: Documentation
- Area: Historical docs/changelog
- Evidence: Search results include historical `0.35.5` changelog text for
  `canic build <role>` and old audit reports mentioning `start_root!`.
- Impact: These are not active user instructions, but broad grep output can be
  noisy during future audits.
- Recommended fix: Keep historical entries as-is. Do not edit old changelog
  truth. Future audits should classify them as archival unless linked from
  current getting-started/setup docs.
- Suggested regression test: None; this is audit hygiene.
