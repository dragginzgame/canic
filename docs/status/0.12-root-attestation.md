## Core Architecture

- [x] Add `IssueRoleAttestation` to `RootCapability`.
- [x] Add `IssueRoleAttestation` request/response wiring in root dispatch path.
- [x] Ensure attestation issuance is capability-gated by policy.
- [x] Ensure 0.11 capability dispatcher behavior remains unchanged.

Current state: root issues signed role attestations with root-generated
`issued_at` / `expires_at` windows and serves an attestation key set endpoint.
Verifier checks resolve by `key_id`, refresh once on unknown keys, and non-root
canisters now warm/refresh key cache on startup and periodic timers. Issuance
now requires explicit audience binding, and verifier paths emit structured
rejection logs plus metrics for failures / unknown keys / epoch rejects. Key
validity windows (`valid_from` / `valid_until`) are now enforced fail-closed.
Verifier minimum-epoch policy now supports per-role floors via
`auth.role_attestation.min_accepted_epoch_by_role`.

## Domain Separation

- [x] Define attestation signing domain separate from delegation domain.
- [x] Ensure attestation key material is distinct from delegation ECDSA key material.
- [x] Ensure attestation keys cannot sign delegation tokens.
- [x] Ensure delegation keys cannot sign attestations.

- [x] Add `RoleAttestation` payload (`subject`, `role`, `subnet_id`, `audience`, `issued_at`, `expires_at`, `epoch`).

- [x] Ensure `issued_at`/`expires_at` are root-generated and time-source consistent.

- [x] Add configuration for `MAX_ATTESTATION_TTL_SECONDS`.

## Issuance Policy

- [x] Enforce role-specific issuance constraints.

## Verification Rules

- [x] Resolve verification key by `key_id`.
- [x] Validate attestation signature before authorization.
- [x] Enforce `payload.subject == ic::caller()` before authorization decisions.
- [x] Enforce expiry check (`expires_at >= now`).
- [x] If `audience` is present, enforce `audience == ic::id()`.
- [x] If `subnet_id` is present, enforce verifier subnet equality.
- [x] Enforce `payload.epoch >= min_accepted_epoch`.
- [x] Ensure verifier fails closed on all verification errors.

## Key Distribution and Cache

- [x] Expose read-only root endpoint `attestation_key_set() -> AttestationKeySet`.
- [x] Include `key_id`, `public_key`, status (`current`/`previous`), validity metadata.
- [x] Treat root canister principal as trust anchor for attestation keys.
- [x] Add certified-data support for key-set endpoint (or document gap if deferred).
- [x] Fetch key set at startup/post-upgrade before mutating attestation auth paths.
- [x] Refresh key set periodically.
- [x] On unknown `key_id`, trigger immediate refresh and retry once.
- [x] Enforce one refresh attempt max per request (no recursion).
- [x] Ensure key-cache replacement is atomic.
- [x] Ensure verifier fails closed when key resolution fails.

Deferred gap note (documented on 2026-03-07):
- `canic_attestation_key_set` is currently served through internal update-RPC flow for
  inter-canister compatibility (`RpcOps::call_rpc_result` / unbounded wait).
- Certified query support for key-set responses is not wired yet; this remains
  a follow-up hardening task.

Security audit note (2026-03-07):
